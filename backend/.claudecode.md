# プロジェクト構成規約 (Spring Boot & Nuxt.js)

## 1. バックエンド (Java / Spring Boot)
src/main/java/com/mannschaft/app/ 配下は「技術レイヤー」ではなく「機能（ドメイン）」ごとにパッケージを分離すること。

### 構成イメージ
```
src/main/java/com/mannschaft/app/
├── config/                          # Spring設定クラス
│   ├── SecurityConfig.java
│   ├── WebSocketConfig.java
│   ├── RedisConfig.java
│   ├── CorsConfig.java
│   └── SwaggerConfig.java
├── common/                          # 共通部品
│   ├── ErrorCode.java               # エラーコード共通インターフェース
│   ├── BusinessException.java       # 業務例外（ErrorCode保持）
│   ├── GlobalExceptionHandler.java  # グローバル例外ハンドラー
│   ├── ApiResponse.java             # 共通レスポンスラッパー（単一用）
│   ├── PagedResponse.java           # ページネーション付きレスポンス
│   ├── ErrorResponse.java           # エラーレスポンス
│   ├── CommonErrorCode.java         # 共通エラーコード Enum
│   └── event/                       # アプリケーションイベント
│       └── ...Event.java
└── [feature]/                       # 例: auth, team, organization
    ├── [Feature]Controller.java     # ※役割ごとに分割可（例: AuthLoginController）
    ├── [Feature]Service.java        # ※役割ごとに分割可（例: AuthTokenService）
    ├── [Feature]Repository.java
    ├── [Feature]Entity.java
    ├── [Feature]ErrorCode.java      # 機能別エラーコード Enum
    ├── [Feature]Mapper.java         # MapStruct マッパー
    └── dto/
        ├── [Feature]Request.java
        └── [Feature]Response.java
```

### ルール
- `com.mannschaft.app.controller` のような全機能共通のレイヤーパッケージは作成禁止。
- **パッケージ宣言:** ファイル移動・作成時は、ディレクトリ構造に合わせた `package` 宣言を正しく記述すること。
- **共通部品:** 複数機能で共有するモデルや基底クラスは `com.mannschaft.app.common` に配置すること。
- **設定クラス:** Spring の設定クラス（Security, WebSocket, Redis, CORS, Swagger 等）は `com.mannschaft.app.config` に配置する。`common` とは分離し、アプリケーション設定専用とする。
- **依存関係:** 異なる機能間のサービスを呼び出す場合は、Repositoryを直接呼ばず、Service経由で呼び出すこと。
- **Service/Controller分割:** 機能が肥大化する場合は、役割ごとにクラスを分割すること。同一 feature パッケージ内に配置する。（例: `AuthService`, `AuthTokenService`, `AuthOAuthService`）

## 2. フロントエンド (TypeScript / Nuxt.js)
Nuxt 3 の標準ディレクトリ構成をベースにしつつ、ロジックを整理すること。

### 構成イメージ
```
server/                          # Nuxt Server API (BFF)
middleware/                      # ルートガード
├── auth.ts                      # ログイン必須チェック
├── role.ts                      # ロール別アクセス制御
└── module-guard.ts              # モジュール有効化チェック
pages/
└── [feature]/                   # 機能ごとのディレクトリ
    └── index.vue
components/
└── [feature]/                   # 機能ごとのコンポーネント
    ├── [Feature]Form.vue
    └── [Feature]List.vue
composables/
├── useApi.ts                    # 共通APIクライアント
├── useErrorHandler.ts           # エラーコード → 表示メッセージ変換
└── use[Feature].ts              # ビジネスロジック・ステート管理
stores/
└── use[Name]Store.ts            # Piniaグローバル状態管理
types/
├── generated/                   # OpenAPI Generator 自動生成型（直接編集禁止）
└── index.ts                     # 手動定義の型・自動生成型の拡張
```

### ルール
- **コンポーネント命名:** `components/user/UserList.vue` のように、フォルダ名とファイル名の重複を許容する（NuxtのAuto-importで `UserList` として呼び出せるようにするため）。
- **ロジックの分離:** API通信やバリデーションロジックは直接 `pages/` に書かず、必ず `composables/` に抽出すること。
- **API通信方式:** 基本は**フロントエンド → Spring Boot API に直接通信**する。Nuxt Server (`server/api/`) を BFF として利用するのは、APIキーの秘匿が必要な場合や複雑なデータ加工が必要な場合のみとする。
- **自動生成型:** `types/generated/` のファイルは直接編集禁止。拡張が必要な場合は `types/` 内の別ファイルで再定義する。
- **認証・認可ガード:** バックエンドは Spring Security で API を厳格に保護し、フロントエンドは `middleware/` で画面へのアクセスを制御して適切な画面遷移を促す。

## 3. エラーハンドリングとAPIレスポンス設計

### 3.1 レスポンス形式
HTTPステータスコードで成功/失敗を判断する。`success` フィールドは持たない。
成功時は常に `data` フィールドでラップし、一貫性を保つ。
すべてのコントローラーは `ApiResponse<T>` または `PagedResponse<T>` を返り値とする。

**共通レスポンスクラス:**
```java
// 単一レスポンス: ApiResponse<T>
// フィールド: data (T)
ApiResponse<UserResponse> → { "data": { "id": 1, "name": "..." } }

// ページネーションレスポンス: PagedResponse<T> extends ApiResponse<List<T>>
// フィールド: data (List<T>), meta (PageMeta)
PagedResponse<UserResponse> → { "data": [...], "meta": { "total": 100, "page": 1, "size": 20, "totalPages": 5 } }

// エラーレスポンス: ErrorResponse
// フィールド: error (ErrorDetail: code, message, fieldErrors?)
ErrorResponse → { "error": { "code": "AUTH_001", "message": "...", "fieldErrors": [...] } }
```

**バリデーションエラー時の `fieldErrors`:**
```json
{
  "error": {
    "code": "COMMON_001",
    "message": "入力内容に不備があります",
    "fieldErrors": [
      { "field": "email", "message": "メールアドレスは必須です" },
      { "field": "password", "message": "8文字以上で入力してください" }
    ]
  }
}
```
バリデーションメッセージは `src/main/resources/messages.properties` に集約し、`GlobalExceptionHandler` でフィールド名＋メッセージを変換する。

### 3.2 エラーコード設計（Enum・機能別分割）
- `com.mannschaft.app.common` に共通インターフェース `ErrorCode` を配置する。
- 各機能パッケージ内に `[Feature]ErrorCode` Enum を作成し、`ErrorCode` を実装する。

```java
// com.mannschaft.app.common.ErrorCode
public interface ErrorCode {
    String getCode();
    String getMessage();
    HttpStatus getHttpStatus();
}

// com.mannschaft.app.auth.AuthErrorCode
public enum AuthErrorCode implements ErrorCode {
    AUTH_001("AUTH_001", "メールアドレスまたはパスワードが正しくありません", HttpStatus.UNAUTHORIZED),
    AUTH_002("AUTH_002", "トークンの有効期限が切れています", HttpStatus.UNAUTHORIZED);
    // ...
}
```

### 3.3 エラーコード命名規則
`{機能プレフィックス}_{連番3桁}` の形式で命名する。

| プレフィックス | 機能 | 例 |
| :--- | :--- | :--- |
| `COMMON` | 共通（バリデーション、権限等） | `COMMON_001` |
| `AUTH` | 認証・認可 | `AUTH_001` |
| `TEAM` | チーム管理 | `TEAM_001` |
| `ORG` | 組織管理 | `ORG_001` |

新しい機能を追加する際は、対応するプレフィックスを定義してからエラーコードを作成すること。

### 3.4 メッセージのハイブリッド方式
- **バックエンド**: Enum にデフォルトメッセージを保持し、レスポンスに含めて返す。バックエンド単体でもエラー内容が完結するようにする。
- **フロントエンド**: エラーコード (`code`) をキーとして表示文言を制御できる。サーバーのメッセージをそのまま表示しても、コードに基づいて独自の文言に差し替えてもよい。
- この方式により、フロントエンドの実装が未完了でもエラー内容が把握でき、将来のi18n対応にもコードベースで移行しやすい。

### 3.5 グローバル例外ハンドラー
`com.mannschaft.app.common` に以下のクラスを配置する。

- **`BusinessException`**: `RuntimeException` を継承し、`ErrorCode` を保持する業務例外クラス。Service層からスローする。
- **`GlobalExceptionHandler`** (`@RestControllerAdvice`): 以下の例外をキャッチし、共通レスポンス形式に変換する。
    1. `BusinessException` → 保持する `ErrorCode` に基づきレスポンス生成
    2. `MethodArgumentNotValidException` 等（Spring標準バリデーション） → `COMMON_001` + `fieldErrors` として変換
    3. その他の未定義例外 → `COMMON_999`（システムエラー）として一律処理

### 3.6 フロントエンドのエラーハンドリング
- **`composables/useErrorHandler.ts`** を作成し、エラーコードから表示メッセージへの変換ロジックを集約する。
- **共通インターセプター**（`useApi.ts` 内）で、認証切れ（401）やシステムエラー（5xx）を自動キャッチし、トースト通知やエラーページへの遷移を行う。
- **フォームバリデーションエラー**: `fieldErrors` 配列を受け取り、各画面の VeeValidate と連携して項目別にメッセージを表示する。

## 4. APIバージョニング
- **方式**: URLパスベースのバージョニング（`/api/v1/...`）を採用する。
- **バージョンアップ基準**: 破壊的変更が必要な場合のみメジャーバージョン（v2, v3...）を上げる。フィールドの追加など後方互換のある変更は同一バージョン内で対応する。
- **並行稼働**: 新バージョンリリース後も、旧バージョンは最低3ヶ月間は並行稼働させる。

## 5. イベント駆動設計
機能間の連携（例: 予約作成 → 通知送信、マッチング成立 → スケジュール登録）が多いため、Service 間の直接呼び出しによる密結合を避ける。

- **方針**: Spring の `ApplicationEvent` + `@EventListener` / `@Async` を活用したイベント駆動で機能間の非同期連携を行う。
- **イベントクラス配置**: `com.mannschaft.app.common.event/` にイベントクラスを配置する。
- **適用基準**: 同一機能内の処理は Service 直接呼び出し、**異なる機能間の非同期連携**にイベントを使用する。

```java
// 例: 予約作成時に通知を送信
// reservation パッケージ
applicationEventPublisher.publishEvent(new ReservationCreatedEvent(reservation));

// notification パッケージが購読
@EventListener
public void onReservationCreated(ReservationCreatedEvent event) {
    // プッシュ通知送信
}
```

## 6. 新モジュール追加ガイドライン
選択式モジュールを新規追加する際の手順:

1. **エラーコードプレフィックス**を定義する（例: `RESERVE`）
2. **Flyway マイグレーション**で `module_definitions` テーブルにレコード追加
3. **`module_level_availability`** でレベル別適用（組織/チーム/個人）を設定
4. **feature パッケージ**を新規作成（本ファイルの構成イメージに従う）
5. **テンプレートプリセット**への追加（必要に応じて `template_modules` を更新）
6. **フロントエンド**: `pages/`, `components/`, `composables/` に対応するファイルを追加
