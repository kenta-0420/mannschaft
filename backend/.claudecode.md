# プロジェクト構成規約 (Spring Boot & Nuxt.js)

## 1. バックエンド (Java / Spring Boot)
src/main/java/com/mannschaft/app/ 配下は「技術レイヤー」ではなく「機能（ドメイン）」ごとにパッケージを分離すること。

### 構成イメージ
```
src/main/java/com/mannschaft/app/
├── config/                          # Spring設定クラス
│   ├── SecurityConfig.java
│   ├── WebSocketConfig.java
│   ├── RedisConfig.java
│   ├── CorsConfig.java
│   └── SwaggerConfig.java
├── common/                          # 共通部品
│   ├── ErrorCode.java               # エラーコード共通インターフェース
│   ├── BusinessException.java       # 業務例外（ErrorCode保持）
│   ├── GlobalExceptionHandler.java  # グローバル例外ハンドラー
│   ├── ApiResponse.java             # 共通レスポンスラッパー（単一用）
│   ├── PagedResponse.java           # ページネーション付きレスポンス
│   ├── ErrorResponse.java           # エラーレスポンス
│   ├── CommonErrorCode.java         # 共通エラーコード Enum
│   ├── CursorPagedResponse.java     # カーソルベースページネーションレスポンス
│   ├── CommonConstants.java         # プロジェクト横断的なシステム定数
│   ├── DomainEventPublisher.java    # イベント発行抽象インターフェース
│   └── event/                       # 共通イベント基底クラス
│       ├── DomainEvent.java         # マーカーインターフェース
│       └── BaseEvent.java
└── [feature]/                       # 例: auth, team, organization
    ├── [Feature]Controller.java     # ※役割ごとに分割可（例: AuthLoginController）
    ├── [Feature]Service.java        # ※役割ごとに分割可（例: AuthTokenService）
    ├── [Feature]Repository.java
    ├── [Feature]Entity.java
    ├── [Feature]ErrorCode.java      # 機能別エラーコード Enum
    ├── [Feature]Mapper.java         # MapStruct マッパー
    ├── [Feature]CreatedEvent.java   # 機能固有のドメインイベント
    └── dto/
        ├── [Feature]Request.java
        └── [Feature]Response.java
```

### ルール
- `com.mannschaft.app.controller` のような全機能共通のレイヤーパッケージは作成禁止。
- **パッケージ宣言:** ファイル移動・作成時は、ディレクトリ構造に合わせた `package` 宣言を正しく記述すること。
- **共通部品:** 複数機能で共有するモデルや基底クラスは `com.mannschaft.app.common` に配置すること。
- **設定クラス:** Spring の設定クラス（Security, WebSocket, Redis, CORS, Swagger 等）は `com.mannschaft.app.config` に配置する。`common` とは分離し、アプリケーション設定専用とする。
- **依存関係:** 異なる機能間のサービスを呼び出す場合は、Repositoryを直接呼ばず、Service経由で呼び出すこと。
- **Service/Controller分割:** 機能が肥大化する場合は、役割ごとにクラスを分割すること。同一 feature パッケージ内に配置する。（例: `AuthService`, `AuthTokenService`, `AuthOAuthService`）

## 2. フロントエンド (TypeScript / Nuxt.js)
Nuxt 3 の標準ディレクトリ構成をベースにしつつ、ロジックを整理すること。

### 構成イメージ
```
server/                          # Nuxt Server API (BFF)
middleware/                      # ルートガード
├── auth.ts                      # ログイン必須チェック
├── role.ts                      # ロール別アクセス制御
└── module-guard.ts              # モジュール有効化チェック
pages/
└── [feature]/                   # 機能ごとのディレクトリ
    └── index.vue
components/
└── [feature]/                   # 機能ごとのコンポーネント
    ├── [Feature]Form.vue
    └── [Feature]List.vue
composables/
├── useApi.ts                    # 共通APIクライアント
├── useErrorHandler.ts           # エラーコード → 表示メッセージ変換
└── use[Feature].ts              # ビジネスロジック・ステート管理
stores/
└── use[Name]Store.ts            # Piniaグローバル状態管理
types/
├── generated/                   # OpenAPI Generator 自動生成型（直接編集禁止）
└── index.ts                     # 手動定義の型・自動生成型の拡張
```

### ルール
- **コンポーネント命名:** `components/user/UserList.vue` のように、フォルダ名とファイル名の重複を許容する（NuxtのAuto-importで `UserList` として呼び出せるようにするため）。
- **ロジックの分離:** API通信やバリデーションロジックは直接 `pages/` に書かず、必ず `composables/` に抽出すること。
- **API通信方式:** 基本は**フロントエンド → Spring Boot API に直接通信**する。Nuxt Server (`server/api/`) を BFF として利用するのは、APIキーの秘匿が必要な場合や複雑なデータ加工が必要な場合のみとする。
- **SSR と認証の制約:** JWT は `localStorage` に保存するため、SSR（サーバーサイドレンダリング）時にはアクセスできない。よって **SSR で実行されるリクエストに認証トークンは付与しない**。SSR は公開ページ（SEO目的）にのみ使用し、認証が必要なページは SPA モードで動作させること。
- **自動生成型:** `types/generated/` のファイルは直接編集禁止。拡張が必要な場合は `types/` 内の別ファイルで再定義する。
- **認証・認可ガード:** バックエンドは Spring Security で API を厳格に保護し、フロントエンドは `middleware/` で画面へのアクセスを制御して適切な画面遷移を促す。

## 3. エラーハンドリングとAPIレスポンス設計

### 3.1 レスポンス形式
HTTPステータスコードで成功/失敗を判断する。`success` フィールドは持たない。
成功時は常に `data` フィールドでラップし、一貫性を保つ。
すべてのコントローラーは `ApiResponse<T>` または `PagedResponse<T>` を返り値とする。

**共通レスポンスクラス:**
```java
// 単一レスポンス: ApiResponse<T>
// フィールド: data (T)
ApiResponse<UserResponse> → { "data": { "id": 1, "name": "..." } }

// ページネーションレスポンス: PagedResponse<T> extends ApiResponse<List<T>>
// フィールド: data (List<T>), meta (PageMeta)
PagedResponse<UserResponse> → { "data": [...], "meta": { "total": 100, "page": 1, "size": 20, "totalPages": 5 } }

// エラーレスポンス: ErrorResponse
// フィールド: error (ErrorDetail: code, message, fieldErrors?)
ErrorResponse → { "error": { "code": "AUTH_001", "message": "...", "fieldErrors": [...] } }
```

**APIレスポンス形式の確定:**
- すべてのAPIレスポンスは `ApiResponse<T>` でラップし、一貫したデータアクセス構造を提供する。
- JSON内に冗長な `success` フラグやHTTPステータスは含めず、HTTPプロトコル側のステータスコードを活用する。
- フロントエンドにおけるレスポンス構造の統一を優先し、`data` フィールドによる一貫したデータアクセスパターンを確立する。

**バリデーションエラー時の `fieldErrors`:**
```json
{
  "error": {
    "code": "COMMON_001",
    "message": "入力内容に不備があります",
    "fieldErrors": [
      { "field": "email", "message": "メールアドレスは必須です" },
      { "field": "password", "message": "8文字以上で入力してください" }
    ]
  }
}
```
バリデーションメッセージは `src/main/resources/messages.properties` に集約し、`GlobalExceptionHandler` でフィールド名＋メッセージを変換する。

### 3.2 エラーコード設計（Enum・機能別分割）
- `com.mannschaft.app.common` に共通インターフェース `ErrorCode` を配置する。
- 各機能パッケージ内に `[Feature]ErrorCode` Enum を作成し、`ErrorCode` を実装する。
- `ErrorCode` はHTTPステータスに依存しない。HTTPステータスとの紐付けは `GlobalExceptionHandler` で行う。

```java
// com.mannschaft.app.common.ErrorCode
public interface ErrorCode {
    String getCode();
    String getMessage();
    Severity getSeverity();

    enum Severity { ERROR, WARN, INFO }
}

// com.mannschaft.app.auth.AuthErrorCode
public enum AuthErrorCode implements ErrorCode {
    AUTH_001("AUTH_001", "メールアドレスまたはパスワードが正しくありません", Severity.WARN),
    AUTH_002("AUTH_002", "トークンの有効期限が切れています", Severity.WARN);
    // ...
}

// GlobalExceptionHandler 内で ErrorCode → HttpStatus のマッピングを管理
// Severity ベースのデフォルト + 個別オーバーライドの2層構造
private static final Map<ErrorCode, HttpStatus> HTTP_STATUS_MAP = Map.of(
    AuthErrorCode.AUTH_001, HttpStatus.UNAUTHORIZED,
    AuthErrorCode.AUTH_002, HttpStatus.UNAUTHORIZED
    // ... 個別マッピングが必要なもののみ登録
);

private HttpStatus resolveHttpStatus(ErrorCode errorCode) {
    // 1. 個別マッピングがあればそれを使用
    // 2. なければ Severity からデフォルト解決（ERROR→500, WARN→400, INFO→200）
    return HTTP_STATUS_MAP.getOrDefault(errorCode, defaultFromSeverity(errorCode.getSeverity()));
}
```

**ErrorCode → HttpStatus マッピング方針:**
- `GlobalExceptionHandler` 内に `Map<ErrorCode, HttpStatus>` で個別マッピングを保持する。
- マッピングが未登録のエラーコードは、`Severity` からデフォルトの HttpStatus を解決する（ERROR→500, WARN→400, INFO→200）。
- これによりエラーコードが増えてもマッピング漏れが起きず、個別のオーバーライドも可能。

**設計理由:** エラー定義を特定のプロトコル（HTTP）から独立させ、バッチ処理やメッセージング処理など、非HTTPコンテキストでも再利用可能にするため。

### 3.3 エラーコード命名規則
`{機能プレフィックス}_{連番3桁}` の形式で命名する。

| プレフィックス | 機能 | 例 |
| :--- | :--- | :--- |
| `COMMON` | 共通（バリデーション、権限等） | `COMMON_001` |
| `AUTH` | 認証・認可 | `AUTH_001` |
| `TEAM` | チーム管理 | `TEAM_001` |
| `ORG` | 組織管理 | `ORG_001` |

新しい機能を追加する際は、対応するプレフィックスを定義してからエラーコードを作成すること。

### 3.4 メッセージのハイブリッド方式
- **バックエンド**: Enum にデフォルトメッセージを保持し、レスポンスに含めて返す。バックエンド単体でもエラー内容が完結するようにする。
- **フロントエンド**: エラーコード (`code`) をキーとして表示文言を制御できる。サーバーのメッセージをそのまま表示しても、コードに基づいて独自の文言に差し替えてもよい。
- この方式により、フロントエンドの実装が未完了でもエラー内容が把握でき、将来のi18n対応にもコードベースで移行しやすい。

### 3.5 グローバル例外ハンドラー
`com.mannschaft.app.common` に以下のクラスを配置する。

- **`BusinessException`**: `RuntimeException` を継承し、`ErrorCode` を保持する業務例外クラス。Service層からスローする。
- **`GlobalExceptionHandler`** (`@RestControllerAdvice`): 以下の例外をキャッチし、共通レスポンス形式に変換する。
    1. `BusinessException` → 保持する `ErrorCode` に基づきレスポンス生成。`ErrorCode` → `HttpStatus` のマッピングもここで管理する。
    2. `MethodArgumentNotValidException` 等（Spring標準バリデーション） → `COMMON_001` + `fieldErrors` として変換
    3. その他の未定義例外 → `COMMON_999`（システムエラー）として一律処理

### 3.6 フロントエンドのエラーハンドリング
- **`composables/useErrorHandler.ts`** を作成し、エラーコードから表示メッセージへの変換ロジックを集約する。
- **共通インターセプター**（`useApi.ts` 内）で、認証切れ（401）やシステムエラー（5xx）を自動キャッチし、トースト通知やエラーページへの遷移を行う。
- **フォームバリデーションエラー**: `fieldErrors` 配列を受け取り、各画面の VeeValidate と連携して項目別にメッセージを表示する。

## 4. APIバージョニング
- **方式**: URLパスベースのバージョニング（`/api/v1/...`）を採用する。
- **バージョンアップ基準**: 破壊的変更が必要な場合のみメジャーバージョン（v2, v3...）を上げる。フィールドの追加など後方互換のある変更は同一バージョン内で対応する。
- **並行稼働**: 新バージョンリリース後も、旧バージョンは最低3ヶ月間は並行稼働させる。

## 5. リソーススコープ（取得範囲）の命名規約
- **個人スコープ**: ログインユーザー自身のデータを取得する場合は `/resources/me`（例: `/todos/me`）というパスを使用する。
- **所属組織スコープ**: チームや組織に紐づくデータを取得する場合は、`/teams/{teamId}/resources` のように、親リソースのIDをパスに含めることを原則とする。
- **一貫性の保持**: `GET /todos` のようにスコープが曖昧なエンドポイントは避け、必ず「誰の・何の」範囲であるかをパス構造で定義する。
- **例外**: 検索APIなどで動的に範囲を切り替える場合は、`scope=mine|team|all` といったクエリパラメータを標準化して使用する。
- **README.md との関係**: README.md の API 設計セクションは機能一覧としての概要表記であり、スコープを省略した短縮形で記載されている。実装時は本規約に準拠し、すべてのエンドポイントに明示的なスコープ（`/me`, `/teams/{teamId}/...` 等）を付与すること。

## 6. イベント駆動設計
機能間の連携（例: 予約作成 → 通知送信、マッチング成立 → スケジュール登録）が多いため、Service 間の直接呼び出しによる密結合を避ける。

- **方針**: イベント発行を `DomainEventPublisher` インターフェースで抽象化し、初期実装は Spring の `ApplicationEvent` + `@TransactionalEventListener` を使用する。将来的な外部メッセージキュー（RabbitMQ/Kafka等）への移行に備える。
- **イベントクラス配置**:
    - `com.mannschaft.app.common.event/` には共通のマーカーインターフェース `DomainEvent` と基底クラス（`BaseEvent` 等）のみを配置する。
    - 各機能固有のイベントクラスは、**発行元の feature パッケージ内**に配置し、`DomainEvent` を実装する（例: `com.mannschaft.app.reservation.ReservationCreatedEvent`）。
- **適用基準**: 同一機能内の処理は Service 直接呼び出し、**異なる機能間の非同期連携**にイベントを使用する。
- **トランザクション境界**: イベントリスナーには `@TransactionalEventListener(phase = AFTER_COMMIT)` を使用し、**発行元のトランザクションが正常にコミットされた後にのみ実行**されるようにする。これにより、発行元がロールバックされたのにリスナーが動作する不整合を防止する。

```java
// com.mannschaft.app.common.event.DomainEvent（マーカーインターフェース）
public interface DomainEvent {
    Instant occurredAt();
}

// com.mannschaft.app.common.DomainEventPublisher（抽象インターフェース）
public interface DomainEventPublisher {
    void publish(DomainEvent event);  // Object ではなく DomainEvent で型安全性を確保
}

// com.mannschaft.app.config.SpringEventPublisher（初期実装）
@Component
@RequiredArgsConstructor
public class SpringEventPublisher implements DomainEventPublisher {
    private final ApplicationEventPublisher applicationEventPublisher;
    @Override
    public void publish(DomainEvent event) {
        applicationEventPublisher.publishEvent(event);
    }
}

// 例: 予約作成時に通知を送信
// reservation パッケージ内にイベントクラスを配置
// com.mannschaft.app.reservation.ReservationCreatedEvent implements DomainEvent
domainEventPublisher.publish(new ReservationCreatedEvent(reservation));

// notification パッケージが購読（トランザクションコミット後に実行）
@TransactionalEventListener(phase = AFTER_COMMIT)
@Async
public void onReservationCreated(ReservationCreatedEvent event) {
    // プッシュ通知送信
}
```

- **非同期イベント失敗時の方針**: `@Async` リスナーで例外が発生した場合は、ログ出力（ERROR レベル）+ 監査ログ記録を必須とする。即時リトライは行わず、運用監視で検知・対応する。将来的にリトライ機構（Spring Retry / メッセージキューの DLQ 等）を導入する場合は、本セクションを更新すること。

## 7. 新モジュール追加ガイドライン
選択式モジュールを新規追加する際の手順:

1. **エラーコードプレフィックス**を定義する（例: `RESERVE`）
2. **Flyway マイグレーション**で `module_definitions` テーブルにレコード追加
3. **`module_level_availability`** でレベル別適用（組織/チーム/個人）を設定
4. **feature パッケージ**を新規作成（本ファイルの構成イメージに従う）
5. **テンプレートプリセット**への追加（必要に応じて `template_modules` を更新）
6. **ユニットテスト・統合テストの作成**
    - 新しいモジュール（Service, Controller等）を作成した際は、必ず対応するテストコードをセットで作成すること。
    - **テスト範囲**: ユニットテスト（境界値や例外系を含むビジネスロジックの検証）、統合テスト（DBアクセスや依存コンポーネントとの連携検証）。
    - **完了定義**: テストがすべてパスし、プロジェクトで規定されたカバレッジ基準を満たしていることを確認して完了とする。
7. **フロントエンド**: `pages/`, `components/`, `composables/` に対応するファイルを追加

## 8. データ保持・削除規約
- **個人情報の取り扱い**: 退会申請後 **30日間** を猶予期間（論理削除状態）とし、その後個人を特定できる情報（氏名、メールアドレス等）を物理削除する。
- **法的強制力のあるデータ**: 決済履歴や領収書データ等は、税法等の法令遵守のため、退会後も **7年間** 保持する。
- **データの匿名化**: 投稿内容や行動ログを統計的に利用する場合は、個人との紐付けを完全に遮断（アノニマイズ）した状態で保持する。
- **アカウント凍結時**: 規約違反等による凍結時は、再登録防止や法的対応のため、凍結解除または一定期間が経過するまで情報を保持し続ける。

## 9. サブスクリプション設計指針
- **DB構造の柔軟化**: `subscription_plans` テーブルに `plan_type` カラム（`FLAT` / `METERED`）を定義し、定額・従量の両方に対応可能なスキーマ設計を Phase 1 から検討する。
- **ビジネスロジックの分離**: 料金計算ロジックは将来の「従量課金」導入を見据え、特定の計算式に依存しないよう `BillingStrategy` 等で抽象化して実装する。
- **決定時期**: 具体的な課金ロジックの実装は Phase 8 までに確定させるが、DBマイグレーションのコストを最小化するため、器（テーブル構造）の準備は現段階で行う。

## 10. データベース命名規約
- **テーブル名**: 小文字の `snake_case` かつ **複数形** とする（例: `task_groups`）。
- **カラム名**: 小文字の `snake_case` とする（例: `due_date`）。
- **主キー**: 原則として `id`（Long/UUID）とする。
- **論理値 (Boolean)**: `is_` または `has_` で始める命名とする（例: `is_completed`）。
- **日時 (Timestamp)**: `created_at`, `updated_at`, `deleted_at` 等、`_at` を末尾に付与する。
- **日付 (Date)**: 日時を含まない場合は `_on` を末尾に付与する（例: `birthday_on`）。
- **外部キー**: `[相手先テーブルの単数形]_id` とする（例: `user_id`）。

## 11. エンティティ共通基底規約

### BaseEntity
すべての主要エンティティが継承する `@MappedSuperclass` を `com.mannschaft.app.common` に配置する。

```java
@MappedSuperclass
@Getter
public abstract class BaseEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(nullable = false)
    private LocalDateTime updatedAt;

    // 論理削除対象エンティティのみ deletedAt を追加（§14 参照）
    // BaseEntity には含めず、SoftDeletableEntity 等のサブクラスで拡張する

    @PrePersist
    protected void onCreate() {
        this.createdAt = LocalDateTime.now();
        this.updatedAt = this.createdAt;
    }

    @PreUpdate
    protected void onUpdate() {
        this.updatedAt = LocalDateTime.now();
    }
}
```

- ログ系テーブル（`audit_logs`, `notifications` 等）や中間テーブルは `BaseEntity` を継承せず、個別に定義してよい。
- 論理削除が必要なエンティティは `deletedAt` フィールドを個別に追加する（本ファイル §14 参照）。

## 12. ページネーション規約
- **標準方式**: オフセットベース。クエリパラメータは `?page=0&size=20&sort=createdAt,desc` を標準とする。
    - `page`: 0始まりのページ番号
    - `size`: 1ページあたりの件数（デフォルト20、最大100）
    - `sort`: ソート対象カラムと方向（カンマ区切り）
- **カーソル方式**: チャットメッセージやタイムラインなど、リアルタイムにデータが追加されるリストにはカーソルベースを使用する。`?cursor={lastId}&limit=20` の形式。
- **レスポンス**:
    - オフセットベース: `PagedResponse<T>` の `meta` フィールドで返す（§3.1 参照）。
    - カーソルベース: `CursorPagedResponse<T>` を使用する。

```java
// com.mannschaft.app.common.CursorPagedResponse<T>
// フィールド: data (List<T>), meta (CursorMeta)
// CursorMeta: nextCursor (String, nullable), hasNext (boolean), limit (int)
CursorPagedResponse<ChatMessageResponse> → {
    "data": [...],
    "meta": { "nextCursor": "msg_12345", "hasNext": true, "limit": 20 }
}
```

## 13. APIフィルタリング規約
- **基本形式**: フィルタリング条件は、対象フィールド名をそのままクエリパラメータのキーとして使用する（例: `?status=active`）。
- **範囲指定の命名**: 日付や数値の範囲検索には、フィールド名に接頭辞/接尾辞を付与する。
    - 開始: `_from` または `_min`（例: `?price_min=1000`）
    - 終了: `_to` または `_max`（例: `?price_max=5000`）
- **複数値（IN句）の指定**: 同一キーの複数指定（`?tags=java&tags=spring`）またはカンマ区切りを許容するよう実装を統一する。
- **部分一致（LIKE検索）**: 文字列検索で部分一致をサポートする場合は、パラメータ名に `_like` を付与する（例: `?title_like=keyword`）。

## 14. 論理削除規約
- **実装方式**: 論理削除が必要なテーブルには `deleted_at DATETIME NULL` カラムを追加する。`NULL` = 有効、値あり = 削除済み。`is_deleted` フラグ方式は使用しない。
- **適用テーブル**: 主要エンティティ（`users`, `teams`, `organizations`, `blog_posts`, `chat_messages` 等）に適用する。ログ系テーブル（`audit_logs`, `notifications`）や中間テーブルには原則適用しない。
- **クエリ**: Repository のデフォルトクエリは `WHERE deleted_at IS NULL` を条件に含める。削除済みデータを含む検索は管理者機能でのみ許可する。
- **物理削除**: 本ファイル §8「データ保持・削除規約」に定める猶予期間経過後に、バッチ処理で物理削除を実行する。

## 15. 非同期ジョブ実行規約
- **基本方式**: `@Async` アノテーションを使用した Spring Boot 標準の非同期実行を採用する。
- **スレッドプール管理**: `ThreadPoolTaskExecutor` を適切に構成し、APIリクエスト用のスレッドを圧迫しないよう専用のプールを割り当てる。用途ごとにプールを分離し、長時間ジョブ（データエクスポート等）がイベントリスナー（通知送信等、§6 参照）を圧迫しないようにする。
    - **イベント用プール**: 軽量・短時間処理向け（例: `event-pool`）
    - **ジョブ用プール**: 重量・長時間処理向け（例: `job-pool`）
    - `@Async("event-pool")`, `@Async("job-pool")` のように明示的にプール名を指定すること。
- **状態管理**: ジョブの進捗状況（ステータス、完了率、生成ファイルURL等）は、`Redis` または DB のジョブ管理テーブルで保持する。
- **拡張性**: 処理の肥大化や分散処理が必要になった段階で、Spring Batch や外部メッセージキュー（RabbitMQ/Kafka）への移行を検討する。

### AI (Claude Code) 向けの制約（必須）
新しい機能の実装に取り掛かる際は、勝手にコードを書き始めず、必ず以下の手順を踏むこと：

1. **要件の確認とすり合わせ**: 実装する機能の詳細と、必要なテーブル定義（DDL）の案を提示し、ユーザーの承認を得ること。

3. **承認後の実装**: ユーザーから「Goサイン」が出た後に、初めて実際のコーディングを開始すること。

AIによる予期せぬ大規模な破壊的変更を防止し、設計の意図を正確に反映させるための制約。

### コーディングスタイルの原則（AI・人間共通）

本プロジェクトのコードは **実務経験3〜5年のプログラマが無理なく読めるレベル** を基準とする。
以下の原則をすべてのコーディング作業で遵守すること。

#### 原則1: 素直で標準的な書き方を選ぶ
- Java / TypeScript の **標準API・定番パターン** を優先して使う。ニッチなライブラリメソッドや最先端の言語機能よりも、広く知られた書き方を選ぶこと。
- **Stream・ラムダ式・インターフェースは積極的に使ってよい**。処理効率や可読性が向上する場面では for ループより Stream を優先する。ただし、`reduce` の多重ネストや `Collectors.teeing` のような使用頻度の低いメソッドは避け、`filter`, `map`, `collect`, `forEach` 等の定番オペレーションを中心に組み立てる。
- 例（TypeScript/Vue）: 高度な型パズル（Conditional Types のネスト、Template Literal Types の乱用）を避け、型が追いやすい素直な定義にする。
- **判断基準**: 「このコードを見た人が、ドキュメントやGitHub Copilotの補助なしで意味を理解できるか？」を常に自問する。

#### 原則2: コードで仕様の意図を語る
- 変数名・メソッド名・クラス名を **仕様書の用語と一致** させる。コードを読むだけで「この処理は仕様書のどの部分を実装しているか」が分かることを目指す。
- 略語は業界標準のもの（`id`, `url`, `dto` 等）に限定し、プロジェクト固有の略語は作らない。
- Boolean を返すメソッドは `is〜` / `has〜` / `can〜` で始め、条件の意味を名前だけで伝える。
- Enum の値名は仕様書の区分名をそのまま反映する。

#### 原則3: 複雑な箇所にはコメントを残す
以下に該当する箇所には **必ず日本語コメントを記載** すること:
- **ビジネスルールの分岐**: なぜその条件で分岐するのか（Why）を説明する。
  ```java
  // 安否確認は「全員回答済み」で自動クローズするが、
  // 24時間経過しても未回答者がいる場合は管理者にリマインド通知を送る
  if (response.allAnswered()) {
      safetyCheck.autoClose();
  } else if (safetyCheck.isOverdue()) {
      notifyAdminReminder(safetyCheck);
  }
  ```
- **複数テーブルを跨ぐクエリや集計ロジック**: 何のデータをどう結合しているかの全体像。
- **パフォーマンス上の理由で直感的でない書き方をしている箇所**: なぜこの書き方を選んだか。
- **外部サービス連携の前提条件や制約**: API仕様のバージョンや制限値。
- ただし、**コードを読めば自明なこと**にはコメントを付けない（`// ユーザーを取得する` のような無意味なコメントは禁止）。

#### 原則4: メソッドの構造を予測可能にする
- メソッドの冒頭で **バリデーション → データ取得 → ビジネスロジック → 結果返却** の流れを一貫させる。
- 1つのメソッドが1つの仕事だけをする。読み手が「このメソッドは何をするか」をメソッド名だけで理解でき、中身を読んで裏切られないこと。
- 途中で処理の目的が変わる場合はメソッドを分割する。

### 人間（開発者）向けの運用
- 軽微なバグ修正、リファクタリング、ドキュメント更新については、事前の設計承認をスキップして直接実装・コミットすることを許可する。

## 18. Flyway マイグレーション SQL スタイル規約

ファイル命名は `V{phase}.{連番}__{説明}.sql`（既存ルール）。SQL の書き方を以下に統一する。

### 基本スタイル
- **キーワードは大文字**: `CREATE TABLE`, `ALTER TABLE`, `NOT NULL`, `DEFAULT`, `FOREIGN KEY` 等
- **テーブル名・カラム名は小文字 snake_case**（§10 DB命名規約と統一）
- **1ファイル = 1つの論理的変更**: テーブル1つの作成、または関連する ALTER のまとめ

### コメント
- テーブル作成時は冒頭に作成意図のコメントを記載する:
```sql
-- 掲示板のスレッドを管理するテーブル
CREATE TABLE bulletin_threads (
    ...
);
```
- DELETE / DROP / ALTER TABLE ... DROP COLUMN を含む破壊的マイグレーションはファイル冒頭に警告を記載する:
```sql
-- ⚠ 破壊的変更: teams.organization_id を削除し team_org_memberships へ移行
ALTER TABLE teams DROP COLUMN organization_id;
```

### 制約・インデックスの命名
- **外部キー制約**: `fk_{テーブル名}_{参照先テーブル名}` 形式で明示的に名前を付ける:
```sql
CONSTRAINT fk_bulletin_threads_team
    FOREIGN KEY (team_id) REFERENCES teams(id)
```
- **インデックス**: `idx_{テーブル名}_{カラム名}` 形式:
```sql
CREATE INDEX idx_bulletin_threads_team_id_created_at
    ON bulletin_threads(team_id, created_at);
```
- **ユニーク制約**: `uq_{テーブル名}_{カラム名}` 形式

### 禁止事項
- `IF NOT EXISTS` は付けない（Flyway がバージョン管理するため二重実行は起きない。付けるとミスの検知が遅れる）
- `IF EXISTS`（DROP 時）も同様に禁止。存在しないオブジェクトの DROP はマイグレーション設計のミスであり、明示的にエラーにすべき

## 19. DTO 粒度ルール

### Request（リクエスト）
- **Create と Update は原則として分ける**:
    - `Create[Feature]Request` → 作成時の必須フィールドのみ（例: email, password, name）
    - `Update[Feature]Request` → 変更可能フィールドのみ（例: name, displayName）
- **理由**: バリデーションルールが異なるため、共用すると `@NotBlank` と nullable が混在して複雑になる

### Response（レスポンス）
- **一覧用と詳細用を分ける**:
    - `[Feature]SummaryResponse` → 一覧表示に必要な最小限（id, name, avatarUrl 等）
    - `[Feature]DetailResponse` → 詳細画面用（全フィールド + リレーション情報）
    - `[Feature]Response` → 上記の区別が不要な場合（小さいエンティティ）

### 共通ルール
- DTO は feature パッケージ内の `dto/` サブパッケージに配置する
- **エンティティを直接レスポンスに返すことは禁止**（必ず DTO に変換する）
- DTO のフィールドは不変（`final` + `@Getter`、`@Setter` なし）を推奨する
- ネストが必要な場合は Inner class ではなく別ファイルに切り出す

## 20. 日時の扱い統一

### DB（MySQL 8.0）
- 日時カラムは **`DATETIME`** を使用する。`TIMESTAMP` は使わない（2038年問題 + MySQL の暗黙タイムゾーン変換を避けるため）
- 保存する値は **日本時間（JST / Asia/Tokyo）** で統一する
    - 国内向けサービスのため UTC 変換のコストを避ける
    - 将来の多国展開時は UTC 化マイグレーションを検討する
- 日付のみのカラムは `DATE` 型を使用する

### Java（Spring Boot）
- エンティティの日時フィールドは **`LocalDateTime`** を使用する（BaseEntity と統一）
- タイムゾーン変換が必要な場合のみ `ZonedDateTime` を使用する
- `Instant` は使わない（DB との変換が煩雑になるため）
- JVM のデフォルトタイムゾーンを起動時に `Asia/Tokyo` に設定する:
```java
@PostConstruct
public void initTimeZone() {
    TimeZone.setDefault(TimeZone.getTimeZone("Asia/Tokyo"));
}
```

### フロントエンド（TypeScript / Nuxt 3）
- API レスポンスの日時は **ISO 8601 文字列**で受け取る: `"2026-03-13T21:00:00"`（タイムゾーンオフセットなし = JST 前提）
- 表示時の日時フォーマットは **dayjs** で統一する（プロジェクト内で1つのライブラリに絞る）
