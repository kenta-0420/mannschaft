# F11.3: UI 国際化（i18n）

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 11
> **最終更新**: 2026-03-30
> **モジュール種別**: デフォルト機能（全アカウント対象）
> **関連ドキュメント**: F11.2 多言語コンテンツ（ユーザー投稿コンテンツの翻訳）、F01.1 認証（users.locale フィールド）

---

## 1. 概要

F11.2「多言語コンテンツ」がユーザー投稿コンテンツ（ブログ・お知らせ等）の翻訳を扱うのに対し、本機能はアプリケーション自体の UI テキスト（バリデーションメッセージ、ダイアログ、エラーメッセージ、ラベル、ボタン等）をアカウント登録時に指定した言語で表示する機能を提供する。

**解決する問題**:
- バリデーションエラーが `"required"` や `"must not be blank"` 等の英語で表示される
- エラーダイアログのメッセージが言語不統一
- ユーザーの `locale` 設定がバックエンドに保存されているが、UI に反映されていない
- フロントエンドのラベル・ボタン・プレースホルダーが日本語固定

**対象メッセージ種別**:
- Bean Validation メッセージ（`@NotBlank`, `@Size`, `@Email` 等のエラー文言）
- 業務エラーメッセージ（`BusinessException` のエラー文言）
- API レスポンスの `message` フィールド
- フロントエンドのラベル・ボタン・ダイアログ・プレースホルダー・バリデーション文言

```
1. ブラウザの Accept-Language ヘッダー → 対応言語に含まれれば採用
2. 対応外の言語コードの場合 → ja（日本語）にフォールバック
```

## 3. DB設計

### 新規テーブル：なし

既存の `users.locale`（VARCHAR(5)）をそのまま使用する。

### 既存テーブルの関連フィールド

#### `users`

| カラム名 | 型 | 許容値 | 説明 |
|---------|---|--------|------|
| `locale` | VARCHAR(5) | `ja` / `en` / `zh` / `ko` / `es` / `de` | 言語コード（ISO 639-1） |
| `timezone` | VARCHAR(50) | 例: `Asia/Tokyo` | タイムゾーン |

**`users.locale` の NULL 対応**: NULL の場合はデフォルト `ja` として扱う。

## 4. バックエンド設計

### 4.1 アーキテクチャ概要

```
HTTP リクエスト受信
    ↓
UserLocaleFilter（OncePerRequestFilter）
    - JWT から userId を取得（エラーは無視）
    - users.locale を参照（NULL の場合 "ja"）
    - 未ログイン時は Accept-Language ヘッダーから判定
    - LocaleContextHolder にセット
    ↓
Controller / Service 処理
    ↓
バリデーションエラー発生時
    → ValidationMessages_{locale}.properties から文言取得
業務エラー発生時
    → messages_{locale}.properties から文言取得
    ↓
ApiErrorResponse の message フィールドに言語別文言をセット
```

### 4.5 UserLocaleFilter（実装詳細）

```java
// src/main/java/com/mannschaft/app/common/i18n/UserLocaleFilter.java
@Component
@RequiredArgsConstructor
public class UserLocaleFilter extends OncePerRequestFilter {
    // 処理順:
    // 1. Authorization ヘッダーから JWT をパース（エラーは無視）
    // 2. userId が取得できた場合: users.locale を DB から取得（キャッシュ推奨）
    // 3. locale 文字列 → Locale オブジェクトに変換（Locale.forLanguageTag）
    // 4. 対応言語リストに含まれない場合は ja にフォールバック
    // 5. 未ログイン時: Accept-Language ヘッダーから parseAcceptLanguage で決定
    // 6. LocaleContextHolder.setLocaleContext(new SimpleLocaleContext(locale)) でセット
    // 7. 処理後: LocaleContextHolder.resetLocaleContext() でクリア（スレッドセーフ）

    private static final Set<String> SUPPORTED_LOCALES =
        Set.of("ja", "en", "zh", "ko", "es", "de");
    private static final Locale DEFAULT_LOCALE = Locale.JAPANESE;
}
```

### 4.6 ErrorCode との統合（移行戦略）

現在の `ErrorCode` インターフェースは `getMessage()` で日本語文字列をハードコードして返している。

**Phase 11 での対応（方式C: 最小変更）**:
- `GlobalExceptionHandler` で `BusinessException` をキャッチした際、`MessageSource` を参照して多言語メッセージを返す
- `ErrorCode.getCode()` をメッセージキーの代わりに使う（例: `TEAM_001` → `team.001` としてプロパティに定義）
- 既存の `ErrorCode.getMessage()` は互換性のため残す

**Phase 12 での完全移行（方式A）**:
- `ErrorCode` インターフェースに `getMessageKey()` を追加
- 各 ErrorCode Enum に `messageKey` フィールドを追加
- `getMessage()` を deprecated にして段階廃止

### 5.3 locale の取得・切り替えフロー

```
1. ログイン成功 → API レスポンスの locale フィールドを取得
2. localStorage / Cookie に locale を保存
3. i18next の changeLanguage(locale) を呼び出して即時切り替え
4. プロフィール設定画面で言語変更 → PUT /api/auth/profile { locale: "en" }
   → フロント側も即時 changeLanguage で反映
5. ページリロード時は localStorage の値で復元
```

## 6. API 設計

### 6.1 言語設定変更 API

既存の「プロフィール更新」API（`PUT /api/auth/profile`）の `locale` フィールドで対応。新規エンドポイントは不要。

```json
// PUT /api/auth/profile
{
  "locale": "en"   // "ja" / "en" / "zh" / "ko" / "es" / "de"
}
```

**バリデーション追加**: `locale` フィールドの値が対応言語コード6種類のいずれかであることを検証する。

### 6.2 対応言語一覧取得 API（新規）

フロントエンドの言語選択UIで使用する。

```
GET /api/i18n/supported-locales

Response 200:
{
  "data": [
    { "code": "ja", "name": "日本語",    "nativeName": "日本語" },
    { "code": "en", "name": "English",   "nativeName": "English" },
    { "code": "zh", "name": "中文",      "nativeName": "中文（简体）" },
    { "code": "ko", "name": "한국어",    "nativeName": "한국어" },
    { "code": "es", "name": "Español",   "nativeName": "Español" },
    { "code": "de", "name": "Deutsch",   "nativeName": "Deutsch" }
  ]
}
```

### 6.3 エラーレスポンスの変化

**現在（日本語ハードコード）:**

## 8. 懸念事項

1. **`users.locale` の初期値**: 新規登録フォームで言語選択が必須か任意かを決める必要がある。NULL の場合は `ja` デフォルトとする。
2. **翻訳精度**: 中国語・韓国語・スペイン語・ドイツ語の翻訳文言は AI 生成 → ネイティブチェック推奨。Phase 11 では AI 生成で実装し、後でネイティブレビューする方針とする。
3. **フロント翻訳ファイルの分量**: 全画面を6言語対応する翻訳ファイルは相当量になる。優先ページ（ログイン・登録・主要ダッシュボード）から段階的に整備する。
4. **ErrorCode 変更の破壊的影響**: 既存テストが日本語文言をハードコードでアサートしている箇所があるため、段階的移行（Phase 11 は方式C）を維持する。
5. **RTL 言語**: 今回の6言語はすべて LTR（左書き）のため RTL 対応は不要。
