# F11.3: UI 国際化（i18n）

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 11
> **最終更新**: 2026-03-30
> **モジュール種別**: デフォルト機能（全アカウント対象）
> **関連ドキュメント**: F11.2 多言語コンテンツ（ユーザー投稿コンテンツの翻訳）、F01.1 認証（users.locale フィールド）

---

## 1. 概要

F11.2「多言語コンテンツ」がユーザー投稿コンテンツ（ブログ・お知らせ等）の翻訳を扱うのに対し、本機能はアプリケーション自体の UI テキスト（バリデーションメッセージ、ダイアログ、エラーメッセージ、ラベル等）をアカウント登録時に指定した言語で表示する機能を提供する。

**解決する問題**:
- バリデーションエラーが `"required"` や `"must not be blank"` 等の英語で表示される
- エラーダイアログのメッセージが言語不統一
- ユーザーの `locale` 設定（例: `en`, `ja`, `ko`）がバックエンドに保存されているが、メッセージに反映されていない

**対象メッセージ種別**:
- Bean Validation メッセージ（`@NotBlank`, `@Size`, `@Email` 等のエラー文言）
- 業務エラーメッセージ（`BusinessException` のエラー文言）
- API レスポンスの `message` フィールド
- （将来対応）フロントエンドのラベル・ボタン・ダイアログ

## 2. スコープ

全ロール対象（自分のアカウント言語設定に基づいて表示）

|-----------|--------|-------|
| `ja` | 日本語 | ◎ 必須（デフォルト） |
| `en` | 英語 | ◎ 必須 |
| `ko` | 韓国語 | △ 将来対応 |
| `zh` | 中国語（簡体） | △ 将来対応 |
| `pt` | ポルトガル語 | △ 将来対応 |

## 4. バックエンド設計

### 4.1 アーキテクチャ概要

```
HTTP リクエスト受信
    ↓
LocaleResolverFilter（カスタムフィルター）
    - JWT から userId を取得
    - users.locale を参照
    - LocaleContextHolder にセット
    ↓
Controller / Service 処理
    ↓
バリデーションエラー発生時
    → ValidationMessages_ja.properties / ValidationMessages_en.properties から文言取得
業務エラー発生時
    → messages_ja.properties / messages_en.properties から文言取得
    ↓
ApiErrorResponse の message フィールドに言語別文言をセット
```

### 4.3 LocaleResolverFilter

```java
// src/main/java/com/mannschaft/app/common/i18n/UserLocaleFilter.java
// OncePerRequestFilter として実装
// 処理順:
// 1. Authorization ヘッダーから JWT をパース（エラーは無視）
// 2. userId が取得できた場合: UserRepository.findLocaleByUserId(userId) でlocale取得
// 3. 取得できた locale を Locale オブジェクトに変換
// 4. LocaleContextHolder.setLocale(locale) でセット
// 5. 取得できなかった場合: Accept-Language ヘッダーをフォールバックとして使用
// 6. Accept-Language も不明の場合: デフォルト Locale.JAPANESE
```

### 4.4 MessageService（業務エラーの多言語化）

```java
// src/main/java/com/mannschaft/app/common/i18n/MessageService.java
// @Service
// MessageSource を注入して現在のLocale でメッセージを取得するラッパー
// 使用例:
// messageService.get("team.not_found")
// → LocaleContextHolder の Locale に応じて ja/en から返す
```

### 4.5 ErrorCode との統合

現在の `ErrorCode` インターフェースは `getMessage()` でハードコードされた日本語文字列を返している。

**変更方針**:

| 方式 | メリット | デメリット |
|------|---------|-----------|
| **A: ErrorCode のキーを properties に移す** | 全エラーが一元管理される | ErrorCode 全件修正が必要（工数大） |
| **B: GlobalExceptionHandler で翻訳** | ErrorCode 修正不要。段階的に対応可能 | ErrorCode に独自キー定義が必要 |
| **C: properties に既存日本語をそのまま登録しつつ英語を追加** | 最小変更。既存動作を保ちながら英語追加 | messages_ja は既存と同じ内容の重複 |

**推奨: 方式C（フェーズ11は C で着手、将来的に A へ移行）**

```
Phase 11: messages_ja に既存文言を登録 + messages_en に英語追加
           GlobalExceptionHandler で LocaleContextHolder を参照して返却
Phase 12: ErrorCode にメッセージキー定義を追加して A 方式に完全移行
```

## 5. フロントエンド設計

> **注意**: フロントエンドのi18nはフロントエンドリポジトリ側で対応が必要。本ドキュメントはバックエンドの設計書だが、フロント側の要点も記載する。

### 5.1 フロントエンド側で必要な対応

- `react-i18next` / `next-i18next` 等のi18nライブラリ導入
- `en.json` / `ja.json` の翻訳ファイル作成
- ユーザーの `locale` 設定をAPIから取得し、ライブラリにセット
- バリデーションライブラリ（Zod/Yup等）のエラーメッセージを i18n キーに差し替え

### 5.2 バックエンドからフロントへの locale 通知

- ログインレスポンス / プロフィール取得 API に `locale` フィールドを含める（既に `users.locale` フィールドあり）
- フロントはこの値を元に表示言語を切り替える

## 6. API 設計

### 6.1 言語設定変更 API

既存の「プロフィール更新」API（`PUT /api/auth/profile`）の `locale` フィールドで対応。新規エンドポイントは不要。

```json
// PUT /api/auth/profile
{
  "locale": "en"
}
```

### 6.2 エラーレスポンスの変化

**現在（日本語ハードコード）:**
```json
{
  "success": false,
  "message": "入力内容を確認してください",
  "errors": [
    { "field": "email", "message": "must not be blank" }
  ]
}
```

**F11.3 実装後（locale=en の場合）:**
```json
{
  "success": false,
  "message": "Please check your input",
  "errors": [
    { "field": "email", "message": "This field is required" }
  ]
}
```

## 7. 実装計画

### フェーズ分け

| フェーズ | 内容 | 工数感 |
|---------|------|-------|
| **7.1 バリデーションメッセージのみ** | `ValidationMessages_ja/en.properties` 作成 + LocaleContextHolder でフィルタ設定 | 小 |
| **7.2 業務エラーメッセージ** | `messages_ja/en.properties` 作成 + GlobalExceptionHandler で多言語化 | 中 |
| **7.3 全 ErrorCode 網羅** | 全機能のエラーコードの英語訳を properties に追加 | 中〜大 |
| **7.4 フロントエンド** | フロントリポジトリ側の対応（別途） | 別途見積もり |

### 優先着手推奨: フェーズ 7.1

最小変更でユーザーが目にする「required」「must not be blank」問題をすぐに解消できる。

## 8. 懸念事項・確認事項

1. **フロントエンドのリポジトリ場所**: バックエンドのAPIレスポンスを日本語化しても、フロント側のラベル・ボタン・バリデーションが別ライブラリを使っている場合は別途フロント対応が必要。
2. **`users.locale` の初期値**: 現在 NULL の場合のデフォルト言語は `ja` とする。
3. **未ログイン時の言語**: `Accept-Language` ヘッダーをフォールバックとして使用する。
4. **ErrorCode の message 変更の破壊的影響**: 既存テストが日本語文言をハードコードでアサートしている箇所が多数あるため、段階的移行（方式C）を推奨。
