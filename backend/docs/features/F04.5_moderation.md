# F04.5: 通報・モデレーション

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 4
> **最終更新**: 2026-03-14

---

## 1. 概要

タイムライン・ブログ・ソーシャルプロフィールに対する不適切コンテンツの通報と、SYSTEM_ADMIN によるモデレーション（レビュー・対応・実名開示）を管理する。デフォルト機能 #10「通報・モデレーション」の基盤として Phase 4 から導入し、以降のフェーズで対象コンテンツタイプを拡張する。通報があった場合のみ SYSTEM_ADMIN がソーシャルプロフィールの実名を確認でき、実名開示は `audit_logs` に記録される。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全通報の閲覧・レビュー・対応（ステータス変更・実名開示・コンテンツ削除・プロフィール凍結） |
| ADMIN | 通報の作成（自チーム/組織のコンテンツに対して） |
| DEPUTY_ADMIN | 通報の作成 |
| MEMBER | 通報の作成 |
| SUPPORTER | 通報の作成 |
| GUEST | 通報不可 |

### 対象レベル
- [x] 組織 (Organization) — 組織内コンテンツの通報
- [x] チーム (Team) — チーム内コンテンツの通報
- [x] パブリック (Public) — パブリックタイムライン・ブログの通報

## 4. API設計

### エンドポイント一覧
| メソッド | パス | 認証 | 説明 |
|---------|-----|------|------|
| POST | `/api/v1/reports` | 必要 | コンテンツ通報 |
| GET | `/api/v1/admin/reports` | 必要 | 通報一覧（SYSTEM_ADMIN） |
| GET | `/api/v1/admin/reports/{id}` | 必要 | 通報詳細（SYSTEM_ADMIN） |
| PATCH | `/api/v1/admin/reports/{id}` | 必要 | 通報対応（ステータス変更・実名開示） |

### リクエスト／レスポンス仕様

#### `POST /api/v1/reports`

**認可**: ログイン済みユーザー（GUEST 除く）

コンテンツを通報する。同一対象への重複通報は許容（複数ユーザーからの通報を想定）。

**リクエストボディ**
```json
{
  "target_type": "TIMELINE_POST",
  "target_id": 250,
  "reporter_type": "SOCIAL_PROFILE",
  "reporter_id": 5,
  "reason": "HARASSMENT",
  "description": "特定のユーザーに対する誹謗中傷が含まれています"
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `target_type` | String | Yes | `BLOG_POST` / `TIMELINE_POST` / `SOCIAL_PROFILE` |
| `target_id` | Long | Yes | 通報対象の ID |
| `reporter_type` | String | Yes | `USER` / `SOCIAL_PROFILE`（通報者の人格） |
| `reporter_id` | Long | Yes | 通報者の ID |
| `reason` | String | Yes | `SPAM` / `HARASSMENT` / `INAPPROPRIATE` / `COPYRIGHT` / `OTHER` |
| `description` | String | No | 詳細説明（最大1000文字） |

**レスポンス（201 Created）**
```json
{
  "data": {
    "id": 1,
    "target_type": "TIMELINE_POST",
    "target_id": 250,
    "reason": "HARASSMENT",
    "status": "PENDING",
    "created_at": "2026-03-14T11:00:00"
  }
}
```

**補足**
- `reporter_type = SOCIAL_PROFILE` の場合: `reporter_id` は現在のユーザーが所有するソーシャルプロフィールの ID であること
- `reporter_type = USER` の場合: `reporter_id` は現在のユーザーの `users.id` であること
- 通報者の情報はレスポンスに含めない（通報の匿名性を担保）
- 同一ユーザーによる同一対象の重複通報はアプリ層で防止（UNIQUE KEY は設けず、サービス層で最新1件のみ有効とする）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | target_type / reason が不正 / reporter_id が自分のものでない / description が1000文字超過 |
| 403 | GUEST |
| 404 | target_id が存在しない |

---

#### `GET /api/v1/admin/reports`

**認可**: SYSTEM_ADMIN のみ

通報一覧をカーソルベースで取得する。

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `status` | String | No | `PENDING` / `REVIEWING` / `RESOLVED` / `DISMISSED` でフィルタ |
| `target_type` | String | No | `BLOG_POST` / `TIMELINE_POST` / `SOCIAL_PROFILE` でフィルタ |
| `cursor` | Long | No | カーソル |
| `limit` | Integer | No | 取得件数（デフォルト: 20） |

**レスポンス（200 OK）**
```json
{
  "data": [
    {
      "id": 1,
      "target_type": "TIMELINE_POST",
      "target_id": 250,
      "target_summary": {
        "content_preview": "問題のある投稿の冒頭100文字...",
        "posted_by": {
          "type": "SOCIAL_PROFILE",
          "handle": "coffee_cat",
          "display_name": "コーヒー好きのねこ"
        }
      },
      "reporter_count": 3,
      "reason": "HARASSMENT",
      "status": "PENDING",
      "identity_disclosed": false,
      "created_at": "2026-03-14T11:00:00"
    }
  ],
  "meta": {
    "next_cursor": null,
    "limit": 20,
    "has_next": false,
    "pending_count": 5,
    "reviewing_count": 2
  }
}
```

**補足**
- `target_summary`: 通報対象のコンテンツプレビュー（最大100文字）+ 投稿者情報
- `reporter_count`: 同一対象に対する通報者数（個別の通報者は一覧では非表示）
- `pending_count` / `reviewing_count`: 未対応件数のサマリー（ダッシュボード表示用）
- `identity_disclosed = false` の段階では `posted_by` にソーシャルプロフィール情報のみ表示（実名は非表示）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | SYSTEM_ADMIN でない |

#### `GET /api/v1/admin/reports/{id}`

**認可**: SYSTEM_ADMIN のみ

通報の詳細を取得する。`identity_disclosed = true` の場合のみ実名情報を含む。

**レスポンス（200 OK）**
```json
{
  "data": {
    "id": 1,
    "target_type": "TIMELINE_POST",
    "target_id": 250,
    "target_detail": {
      "content": "問題のある投稿の全文...",
      "posted_by": {
        "type": "SOCIAL_PROFILE",
        "handle": "coffee_cat",
        "display_name": "コーヒー好きのねこ"
      },
      "real_identity": null,
      "attachments": []
    },
    "reports": [
      {
        "reporter_type": "USER",
        "reason": "HARASSMENT",
        "description": "特定のユーザーに対する誹謗中傷が含まれています",
        "created_at": "2026-03-14T11:00:00"
      }
    ],
    "status": "PENDING",
    "identity_disclosed": false,
    "reviewed_by": null,
    "review_note": null,
    "resolved_at": null,
    "created_at": "2026-03-14T11:00:00"
  }
}
```

**補足**
- `target_detail.real_identity`: `identity_disclosed = true` の場合のみ `{ "user_id": 10, "display_name": "田中太郎", "email": "tanaka@example.com" }` を返却。`false` の場合は `null`
- `reports`: 同一対象に対する全通報のリスト（通報者の個人情報は含めない）

#### `PATCH /api/v1/admin/reports/{id}`

**認可**: SYSTEM_ADMIN のみ

通報のステータス変更・実名開示・対応メモを記録する。

**リクエストボディ**
```json
{
  "status": "RESOLVED",
  "identity_disclosed": true,
  "review_note": "投稿内容を確認し、利用規約違反と判断。投稿を削除し、プロフィールを凍結。",
  "actions": ["DELETE_CONTENT", "FREEZE_PROFILE"]
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `status` | String | Yes | `REVIEWING` / `RESOLVED` / `DISMISSED` |
| `identity_disclosed` | Boolean | No | 実名開示フラグ（true に変更した場合、`audit_logs` に記録） |
| `review_note` | String | No | 対応メモ（最大1000文字） |
| `actions` | String[] | No | 実行するアクション: `DELETE_CONTENT`（投稿論理削除）/ `FREEZE_PROFILE`（プロフィール凍結） |

**レスポンス（200 OK）**
```json
{
  "data": {
    "id": 1,
    "status": "RESOLVED",
    "identity_disclosed": true,
    "reviewed_by": {
      "id": 1,
      "display_name": "システム管理者"
    },
    "review_note": "投稿内容を確認し、利用規約違反と判断。投稿を削除し、プロフィールを凍結。",
    "actions_executed": ["DELETE_CONTENT", "FREEZE_PROFILE"],
    "resolved_at": "2026-03-14T12:00:00"
  }
}
```

**補足**
- `identity_disclosed` を `false` → `true` に変更した場合:
  - `audit_logs` に `REPORT_IDENTITY_DISCLOSED` イベントを記録（actor_id, report_id, target_social_profile_id）
  - 以降、このレポートの詳細 API で `real_identity` が返却される
- `actions`:
  - `DELETE_CONTENT`: `target_type = TIMELINE_POST / BLOG_POST` の場合、対象コンテンツを論理削除
  - `FREEZE_PROFILE`: `target_type = SOCIAL_PROFILE` の場合、または投稿者のソーシャルプロフィールを凍結（`is_active = false`）
- `status = RESOLVED / DISMISSED` の場合は `resolved_at` を自動設定
- `reviewed_by` は現在の SYSTEM_ADMIN の `users.id` を自動記録

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | status が不正 / review_note が1000文字超過 / actions が不正 |
| 403 | SYSTEM_ADMIN でない |
| 404 | 通報 ID が存在しない |
| 409 | 既に RESOLVED / DISMISSED 済み（再対応は不可。新規通報として扱う） |

## 5. ビジネスロジック

### 主要フロー

#### 通報の作成
```
1. ユーザーが投稿・ブログ・プロフィールの「通報」ボタンをタップ
2. POST /api/v1/reports を呼び出し
3. バックエンドが以下を実行:
   a. target_type + target_id の存在確認
   b. reporter_id が自分のものであることを確認
   c. content_reports に INSERT（status = PENDING）
4. 監査ログには記録しない（一般ユーザーの操作のため）
5. SYSTEM_ADMIN ダッシュボードの未対応通報カウンターを WebSocket で更新
```

#### 通報のレビュー・対応
```
1. SYSTEM_ADMIN が管理画面で通報一覧を確認
2. GET /api/v1/admin/reports/{id} で詳細を確認
3. 必要に応じて PATCH で以下を実行:
   a. status = REVIEWING に変更（着手を記録）
   b. identity_disclosed = true で実名を確認（audit_logs に記録）
   c. actions で DELETE_CONTENT / FREEZE_PROFILE を実行
   d. status = RESOLVED / DISMISSED で完了
4. 対応アクション実行時:
   a. DELETE_CONTENT:
      - timeline_posts.deleted_at = NOW() を設定
      - Redis フィードキャッシュから ZREM + DEL
      - audit_logs に TIMELINE_POST_DELETED イベントを記録
   b. FREEZE_PROFILE:
      - user_social_profiles.is_active = false を設定
      - audit_logs に SOCIAL_PROFILE_FREEZE イベントを記録
5. 全アクションは単一トランザクションで実行
```

### 重要な判定ロジック

- **通報の重複**: 同一ユーザーから同一対象への重複通報はサービス層で最新1件に制限（DB レベルの UNIQUE KEY は設けない。複数ユーザーからの同一対象通報は正常動作）
- **実名開示の不可逆性**: `identity_disclosed` を一度 `true` にしたら `false` に戻せない。監査ログで開示履歴を追跡
- **対応完了の不可逆性**: `RESOLVED` / `DISMISSED` は最終ステータス。再対応が必要な場合は新規通報として扱う
- **同一対象の複数通報**: 同一 `target_type + target_id` に対する複数通報は `reporter_count` として集約表示。対応は1件のレポートに対して行い、同一対象の他の通報も自動的に `RESOLVED` にする
- **target_type の拡張**: Phase 4 では `BLOG_POST`, `TIMELINE_POST`, `SOCIAL_PROFILE` をサポート。Phase 5 以降で `CHAT_MESSAGE`, `BULLETIN_THREAD` 等を追加可能（ENUM に値を追加するだけ）

## 6. セキュリティ考慮事項

- **認可チェック**: 通報作成は GUEST 以外の全ユーザー。通報一覧・詳細・対応は SYSTEM_ADMIN のみ。各 API でロールを必ず検証
- **実名開示の監査**: `identity_disclosed = true` への変更は必ず `audit_logs` に記録。記録内容: `actor_id`（SYSTEM_ADMIN の user_id）、`report_id`、`target_social_profile_id`、`timestamp`
- **通報者の匿名性**: 通報者の詳細情報（`reporter_id` の逆引き）は SYSTEM_ADMIN にも表示しない。通報理由と説明文のみ閲覧可能
- **レートリミット**: 通報 API は Bucket4j で 1時間に10回（ユーザー単位）。通報のスパム防止
- **XSS 対策**: `description` / `review_note` はレスポンス時にサニタイズ
- **通報対象の削除**: 通報対象が論理削除されても通報レコードは残る（対応履歴として保持）
