# F03.7: 順番待ち管理

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 3
> **最終更新**: 2026-03-18
> **関連ドキュメント**: F02.1 QR会員証（QRコード読み取り・チェックイン機構流用）、F03.4 予約管理（時間枠予約との差異整理）、F04.3 プッシュ通知（呼び出し通知）、F10.2 モデレーション（無断キャンセル累積によるペナルティ）

---

## 1. 概要

病院・クリニック・市役所・銀行窓口・飲食店等において、来場者がQRコード読み取りまたはオンライン予約で順番を取得し、待ち人数・おおよその呼び出しまでの待ち時間をリアルタイムで確認できる順番待ち管理機能。窓口（カウンター）ごとに受付方式（QRコード現地受付のみ / ネット予約のみ / 両方）を ADMIN が設定でき、窓口ごとの平均対応時間に基づいて待ち時間の自動推定を行う。無断キャンセル（呼び出し後の不在）の累積カウントにより、閾値超過時に一定期間のネット予約停止ペナルティを自動適用する。

**F03.4 予約管理との差異**:
- F03.4: 事前に日時・スタッフを指定して時間枠を予約する「アポイントメント型」
- F03.7: 来場時（or 当日オンライン）にリアルタイムで番号を取得する「順番待ち型」。時間枠の概念はなく、到着順に処理される

### テーブル定義

#### `queue_categories`

窓口カテゴリの定義。病院の「内科」「外科」、市役所の「住民票」「税務」等。待ち行列の方式（SHARED/INDIVIDUAL）とチケット番号のプレフィックスをカテゴリ単位で管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | TEAM / ORGANIZATION |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id / organizations.id |
| `name` | VARCHAR(50) | NO | — | カテゴリ名（例: '内科', '外科', '住民票', '税務', '会計'） |
| `queue_mode` | VARCHAR(10) | NO | 'INDIVIDUAL' | 待ち行列方式。`INDIVIDUAL`（窓口ごとに独立した待ち行列）/ `SHARED`（カテゴリ内の全窓口で1つの待ち行列を共有） |
| `prefix_char` | VARCHAR(5) | YES | NULL | チケット番号のプレフィックス（例: 'A', 'B', '内'）。NULL の場合はプレフィックスなし（連番のみ） |
| `max_queue_size` | SMALLINT UNSIGNED | NO | 50 | カテゴリ全体の最大待ち人数（`SHARED` の場合はカテゴリ合計、`INDIVIDUAL` の場合は各窓口の個別上限。0 = 無制限） |
| `display_order` | SMALLINT UNSIGNED | NO | 0 | 表示順 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
UNIQUE KEY uq_qcat_scope_name (scope_type, scope_id, name, deleted_at)     -- 同一スコープ内のカテゴリ名重複防止
UNIQUE KEY uq_qcat_scope_prefix (scope_type, scope_id, prefix_char, deleted_at) -- 同一スコープ内のプレフィックス重複防止
INDEX idx_qcat_scope_order (scope_type, scope_id, display_order)             -- スコープ別のカテゴリ一覧
```

**制約・備考**
- `queue_mode` がカテゴリレベルにあることで、同カテゴリ内の全窓口が同一の待ち行列方式で動作することを保証（窓口レベルでの SHARED/INDIVIDUAL 混在を根本的に防止）
- `prefix_char` はスコープ内で一意（UNIQUE KEY で保証）。重複するプレフィックスを設定すると日ごとのチケット番号が衝突する
- **SHARED モード**: カテゴリ内の全窓口が1つの待ち行列を共有。患者は「内科」で番号を取り、空いた窓口から呼ばれる。`max_queue_size` はカテゴリ全体で判定
- **INDIVIDUAL モード**: 各窓口が独立した待ち行列を持つ。`max_queue_size` は各窓口の `queue_counters.max_queue_size` で個別に判定（カテゴリの `max_queue_size` は INDIVIDUAL モードでは無視）
- カテゴリなしの窓口（飲食店のレジ等）: カテゴリを1つ作成し `queue_mode = 'INDIVIDUAL'`、`prefix_char = NULL` で運用

#### `queue_counters`

窓口（カウンター）の定義。病院の「内科1番」「内科2番」、市役所の「住民票窓口」「税務窓口」等。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `category_id` | BIGINT UNSIGNED | NO | — | FK → queue_categories。ON DELETE RESTRICT。カテゴリ経由で scope_type/scope_id を参照 |
| `name` | VARCHAR(100) | NO | — | 窓口名（例: '内科1番', '住民票窓口', 'レジA'） |
| `description` | VARCHAR(500) | YES | NULL | 窓口の説明（対応業務・注意事項等） |
| `accept_mode` | VARCHAR(20) | NO | 'BOTH' | 受付方式。`QR_ONLY`（QRコード現地受付のみ）/ `ONLINE_ONLY`（ネット予約のみ）/ `BOTH`（両方） |
| `avg_service_minutes` | SMALLINT UNSIGNED | NO | 10 | 1件あたりの平均対応時間（分）。待ち時間推定の基礎値 |
| `avg_service_minutes_manual` | BOOLEAN | NO | FALSE | TRUE = ADMIN が手動設定した値を保持（自動調整バッチで上書きしない） |
| `max_queue_size` | SMALLINT UNSIGNED | NO | 50 | 窓口個別の最大待ち人数（INDIVIDUAL モードでのみ有効。SHARED モードでは `queue_categories.max_queue_size` が適用。0 = 無制限） |
| `is_active` | BOOLEAN | NO | TRUE | 窓口の有効/無効（FALSE = 完全休止。受付も対応も不可） |
| `is_accepting` | BOOLEAN | NO | TRUE | 新規受付の可否（FALSE = 新規受付停止。既存チケットの呼び出し・対応は継続。昼休み等の一時停止用） |
| `operating_time_from` | TIME | YES | NULL | 営業開始時刻（NULL = 24時間受付） |
| `operating_time_to` | TIME | YES | NULL | 営業終了時刻 |
| `display_order` | SMALLINT UNSIGNED | NO | 0 | 表示順 |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
UNIQUE KEY uq_qc_category_name (category_id, name, deleted_at)       -- 同一カテゴリ内の窓口名重複防止
INDEX idx_qc_category_active (category_id, is_active, display_order)  -- カテゴリ別の窓口一覧
```

**制約・備考**
- `accept_mode` の使い分け:
  - `QR_ONLY`: 現地にポスター/タブレットでQRコードを掲示。来場者がスマホで読み取って順番取得。事前のネット予約は不可
  - `ONLINE_ONLY`: Webアプリから当日のネット予約のみ。現地QRコードなし（完全予約制の窓口向け）
  - `BOTH`: QRコード + ネット予約の両方で順番取得可能。ネット予約者と現地受付者は到着順で統合管理
- `category_id` 経由で `queue_categories.scope_type` / `scope_id` を参照。窓口テーブル自体にスコープカラムは持たない（正規化）
- `avg_service_minutes` は ADMIN が実績に基づいて調整。待ち時間推定は `queue_categories.queue_mode` に応じて計算方法が変わる（詳細はビジネスロジック「待ち時間の推定計算」を参照）
- `max_queue_size`（INDIVIDUAL モード時）: `queue_categories.max_queue_size`（カテゴリ全体上限）とは独立。INDIVIDUAL モードでは各窓口の `max_queue_size` で個別制限
- **is_active vs is_accepting**: `is_active = FALSE` は窓口の完全停止（受付も対応も不可。ディスプレイに「休止中」表示）。`is_accepting = FALSE` は新規受付のみ停止（既存チケットの呼び出し・対応は継続。昼休み・午前の受付終了等で使用。ディスプレイに「受付停止中」表示）
- `operating_time_from` / `operating_time_to` が両方 NULL の場合は24時間受付可能。営業時間外のネット予約は拒否（422。当日の順番待ちに特化した設計）
- `avg_service_minutes_manual`: ADMIN が手動で `avg_service_minutes` を設定した場合 `TRUE` に設定。`auto_adjust_service_minutes = TRUE` でもバッチによる自動上書きを抑制。ADMIN が「自動調整に戻す」操作をすると `FALSE` にリセット

#### `queue_tickets`

順番待ちチケット（番号札）。来場者1人に対して1チケットを発行する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `category_id` | BIGINT UNSIGNED | NO | — | FK → queue_categories。ON DELETE RESTRICT。非正規化（`counter_id` 経由でも参照可能だが、SHARED モードの position 操作・待ち人数集計で JOIN を回避するために保持） |
| `counter_id` | BIGINT UNSIGNED | NO | — | FK → queue_counters。ON DELETE RESTRICT。SHARED モードの場合、発券時はカテゴリ内の代表窓口をセットし、呼び出し時に実際の窓口に付け替え |
| `ticket_number` | VARCHAR(20) | NO | — | 番号札番号（プレフィックス + 連番。例: 'A-001', 'B-015'。日ごとにリセット） |
| `user_id` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL。会員の場合のユーザーID |
| `guest_phone_encrypted` | VARBINARY(256) | YES | NULL | 非会員（GUEST）の電話番号（AES-256 暗号化）。呼び出し通知用（SMS 送信時に復号）。会員の場合は NULL |

| `party_size` | SMALLINT UNSIGNED | NO | 1 | 人数（組単位。飲食店の場合: 1名, 2名, 4名等） |

| `status` | VARCHAR(20) | NO | 'WAITING' | ステータス。`WAITING`（待ち中）/ `CALLED`（呼び出し中）/ `SERVING`（対応中）/ `COMPLETED`（完了）/ `CANCELLED`（キャンセル）/ `NO_SHOW`（不在） |

| `estimated_wait_minutes` | SMALLINT UNSIGNED | YES | NULL | 推定待ち時間（分。発券時に計算。リアルタイム更新は API レスポンスで動的算出） |
| `called_at` | DATETIME | YES | NULL | 呼び出し日時（CALLED 遷移時） |
| `serving_at` | DATETIME | YES | NULL | 対応開始日時（SERVING 遷移時） |
| `completed_at` | DATETIME | YES | NULL | 完了日時（COMPLETED 遷移時） |
| `cancelled_at` | DATETIME | YES | NULL | キャンセル日時 |
| `cancelled_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL。キャンセル実行者（本人 or ADMIN） |
| `no_show_at` | DATETIME | YES | NULL | NO_SHOW 判定日時 |
| `hold_until` | DATETIME | YES | NULL | 一時離席の延長期限。CALLED 状態で利用者が「一時離席」を要求した場合に設定。この時刻まで NO_SHOW タイムアウトを延長。1チケットにつき1回のみ使用可能 |
| `hold_used` | BOOLEAN | NO | FALSE | 一時離席の使用済みフラグ（TRUE = 既に1回使用済み。再度の離席は不可） |
| `actual_service_minutes` | SMALLINT UNSIGNED | YES | NULL | 実際の対応時間（分。COMPLETED 時に `completed_at - serving_at` で自動計算） |
| `note` | VARCHAR(300) | YES | NULL | 備考（ADMIN メモ。例: '車椅子', '通訳必要', '再来院'） |

**インデックス**
```sql
INDEX idx_qt_category_date_status (category_id, issued_date, status, position)  -- カテゴリ別の待ち行列（SHARED モード用）
INDEX idx_qt_counter_date_status (counter_id, issued_date, status, position)  -- 窓口別の当日待ち行列（INDIVIDUAL モード用）
INDEX idx_qt_user_date (user_id, issued_date DESC)                            -- ユーザーの順番待ち履歴
INDEX idx_qt_counter_ticket (counter_id, issued_date, ticket_number)           -- チケット番号検索
INDEX idx_qt_status_called (status, called_at)                                -- NO_SHOW 検出用
INDEX idx_qt_guest_phone_hash (guest_phone_hash, issued_date, status)         -- GUEST の同時チケット制限・NO_SHOW 追跡
```

**制約・備考**
- `user_id` と `guest_phone_encrypted` は XOR ではない（会員が電話番号を併記することも可）。両方 NULL は不可（アプリ層で検証）
- **電話番号の暗号化**: F09.3 駐車場の `plate_number` と同パターン。保存時に正規化（ハイフン・スペース除去 → `09012345678` 形式）→ AES-256 暗号化 → `guest_phone_encrypted` に格納。同時に SHA-256 ハッシュ → `guest_phone_hash` に格納。検索（同時チケット制限・NO_SHOW 追跡）は `guest_phone_hash` で行い、表示（ADMIN の呼び出し画面・SMS 送信）は復号して使用
- **GUEST の NO_SHOW 追跡**: GUEST は `user_id` がないため `guest_phone_hash` で NO_SHOW 回数を追跡する。ただしペナルティは適用しない（GUEST はネット予約不可のため停止対象がない。QR 受付は常に許可）。ADMIN の統計画面で「NO_SHOW 常習の電話番号」を確認可能にし、現地対応に委ねる
- **GUEST 個人情報の保持期間**: `guest_phone_encrypted` / `guest_phone_hash` / `guest_name` は発券日から90日後にバッチで NULL に上書き（個人情報の最小化）。チケットの統計データ（`status`, `actual_service_minutes` 等）は保持
- **チケット番号の原子的採番**: `ticket_number` は `issued_date` ごとにリセット。窓口の `category`（or `counter`）ごとにプレフィックス + 連番（例: 内科=A, 外科=B → A-001, A-002, B-001）。プレフィックスなしの連番も設定可能。採番は `SELECT MAX(ticket_number) ... FOR UPDATE` で排他制御し、同時発券時の番号重複を防止
- **position の原子的割当**: `position` は `SELECT MAX(position) ... FOR UPDATE` で同一スコープの WAITING チケットの最大値 + 1 を割当。SHARED モードは `category_id` で、INDIVIDUAL モードは `counter_id` で絞り込み。前詰めの SQL:
  - INDIVIDUAL: `UPDATE queue_tickets SET position = position - 1 WHERE counter_id = :id AND issued_date = :date AND status = 'WAITING' AND position > :removedPosition`
  - SHARED: `UPDATE queue_tickets SET position = position - 1 WHERE category_id = :catId AND issued_date = :date AND status = 'WAITING' AND position > :removedPosition`
- `party_size` は飲食店の席案内で使用。病院・市役所では通常1名
- **NO_SHOW 判定**: `CALLED` 状態で一定時間（`queue_settings.no_show_timeout_minutes`）経過しても `SERVING` に遷移しない場合、自動的に `NO_SHOW` に遷移（リアルタイムチェック or 定期バッチ）
- **ステータス遷移**: `WAITING` → `CALLED`（ADMIN が呼び出し）→ `SERVING`（来場確認）→ `COMPLETED`（対応完了）。`WAITING` / `CALLED` → `CANCELLED`（本人 or ADMIN）。`CALLED` → `NO_SHOW`（タイムアウト）。`SERVING` → `CANCELLED`（ADMIN のみ。対応中断。`actual_service_minutes` は `serving_at` 〜 `cancelled_at` で記録し、統計の `completed_count` には含めない）
- **営業時間終了時の WAITING チケット処理**: 営業終了時刻（`operating_time_to`）を過ぎた時点で残っている `WAITING` チケットは自動キャンセルしない（ADMIN が個別にキャンセル or 翌日0時のバッチで一括キャンセル）。理由: 窓口が延長営業する場合があるため、自動キャンセルは業務を妨げる。翌日バッチで `issued_date < TODAY AND status = 'WAITING'` を `CANCELLED`（`cancellation_reason = '営業時間終了による自動キャンセル'`）に一括更新
- **翌日バッチの処理**: (1) 前日以前の `WAITING` → `CANCELLED`（自動キャンセル）。(2) 前日以前の `CALLED` → `NO_SHOW`。(3) `queue_daily_stats` の前日分を集計。(4) 発券日から90日超の GUEST 個人情報を NULL 化

#### `queue_settings`

スコープごとの順番待ち運用設定。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | TEAM / ORGANIZATION |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id / organizations.id |
| `no_show_timeout_minutes` | SMALLINT UNSIGNED | NO | 5 | 呼び出し後の NO_SHOW 判定タイムアウト（分。CALLED → NO_SHOW の自動遷移時間） |

| `no_show_penalty_threshold` | SMALLINT UNSIGNED | NO | 3 | ペナルティ発動の閾値（直近30日間の NO_SHOW 回数） |
| `no_show_penalty_days` | SMALLINT UNSIGNED | NO | 14 | ペナルティ期間（日数。閾値超過後、ネット予約が停止される期間。QRコード現地受付は引き続き可能） |
| `max_active_tickets_per_user` | SMALLINT UNSIGNED | NO | 1 | 1ユーザーあたりの同時アクティブチケット上限（全窓口合計。1 = 同時に1窓口のみ） |

| `reservation_priority_enabled` | BOOLEAN | NO | FALSE | F03.4 予約統合の有効/無効（TRUE = 事前予約者が QR 受付時にキュー先頭付近に優先挿入される） |

| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_qsettings_scope (scope_type, scope_id)  -- 1スコープ1設定
```

**制約・備考**
- 1スコープ1設定（UNIQUE KEY で保証）。設定未作成時はデフォルト値を Service 層で適用
- **ペナルティの対象範囲**: ネット予約（`source = 'ONLINE'`）のみ停止。QRコード現地受付（`source = 'QR'`）は常に許可（来場者を門前払いにしない配慮）
- `max_active_tickets_per_user = 1` で「同時に1つの窓口でしか待てない」を実現。飲食店の複数店舗同時待ちを防止
- **GUEST の同時チケット制限**: GUEST は `user_id` がないため、同一スコープ内で同一 `guest_phone_hash` のアクティブチケット（`WAITING` / `CALLED`）が既に存在する場合は 409 エラー。電話番号を正規化（`090-1234-5678` → `09012345678`）→ SHA-256 ハッシュで照合

#### `queue_qr_codes`

カテゴリ or 窓口の受付用QRコード。ポスターやタブレットに掲示し、来場者がスマホで読み取って順番を取得する。SHARED モードではカテゴリ単位、INDIVIDUAL モードでは窓口単位で QR を発行する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `category_id` | BIGINT UNSIGNED | YES | NULL | FK → queue_categories。ON DELETE CASCADE。SHARED モードのカテゴリ QR 用 |
| `counter_id` | BIGINT UNSIGNED | YES | NULL | FK → queue_counters。ON DELETE CASCADE。INDIVIDUAL モードの窓口 QR 用 |
| `qr_token` | VARCHAR(64) | NO | — | QRコードに埋め込むトークン（SecureRandom 生成。URL に含めて読み取り → 受付API を呼び出し） |
| `is_active` | BOOLEAN | NO | TRUE | QRコードの有効/無効（FALSE = 無効化。再発行時に旧トークンを無効化） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_qqr_token (qr_token)              -- トークンの一意性
INDEX idx_qqr_category (category_id, is_active)   -- カテゴリの有効QRコード検索
INDEX idx_qqr_counter (counter_id, is_active)     -- 窓口の有効QRコード検索
```

**制約・備考**
- `category_id` と `counter_id` は XOR（CHECK 制約。どちらか一方のみ非 NULL）
- **SHARED モード**: `category_id` に紐付く QR を発行。「内科」のポスターに1つの QR を掲示 → 読み取りでカテゴリの共有キューに参加。発券時に `counter_id` はカテゴリ内の代表窓口（`display_order` 最小の稼働中窓口）を自動セット
- **INDIVIDUAL モード**: `counter_id` に紐付く QR を発行。各窓口に個別の QR を掲示 → 読み取りでその窓口のキューに参加
- QRコードの URL 形式: `https://{domain}/queue/join/{qr_token}`。スマホブラウザで開くと順番取得画面（会員: ログイン後に自動発券、非会員: 電話番号入力フォーム）を表示
- QR 読み取り API（`POST /queue/join/{qrToken}`）で、トークンが `category_id` を持つか `counter_id` を持つかでカテゴリ QR / 窓口 QR を自動判別
- QRコード再発行: 旧トークンを `is_active = FALSE` に更新 → 新トークンを生成。セキュリティ上、定期的な再発行を推奨
- F02.1 QR会員証の `checkin_locations` とは独立。会員証のチェックインとは別機能

#### `GET /api/v1/teams/{teamId}/queue/status`

**認可**: SUPPORTER 以上 / 認証不要（ADMIN が公開設定した場合）

全窓口のリアルタイム待ち状況を取得する。待合室のディスプレイ表示やWebページ埋め込み用。

**レスポンス（200 OK）**
```json
{

      {

        "last_called_ticket": "A-004",
        "accept_mode": "BOTH",
        "operating_time_from": "09:00",
        "operating_time_to": "17:00"
      },
      {
        "id": 2,
        "name": "内科2番",
        "category": "内科",
        "is_active": true,
        "waiting_count": null,
        "estimated_wait_minutes": null,
        "current_ticket": "A-012",
        "last_called_ticket": "A-011",
        "accept_mode": "QR_ONLY",
        "operating_time_from": "09:00",
        "operating_time_to": "17:00"
      }
    ],
    "categories": [
      {
        "category": "内科",
        "queue_mode": "SHARED",
        "total_waiting": 13,
        "estimated_wait_minutes": 33,
        "counter_ids": [1, 2]
      }
    ],
    "total_waiting": 13,
    "updated_at": "2026-03-18T10:30:00"
  }
}
```

**補足**
- `categories`: SHARED モードのカテゴリ別集計。INDIVIDUAL モードの窓口は `categories` に含まない（`counters` 配列のみで表示）
- `waiting_count`: INDIVIDUAL モードでは窓口ごとの `WAITING` チケット数。**SHARED モードの場合は各窓口の `waiting_count` / `estimated_wait_minutes` は `null`**（カテゴリ全体の待ち人数は `categories` 配列の `total_waiting` で表示。個別窓口の内訳は意味をなさないため）
- `estimated_wait_minutes`: `waiting_count × avg_service_minutes`（同カテゴリの複数窓口がある場合は窓口数で分散を考慮）
- `current_ticket`: 現在対応中（`SERVING`）のチケット番号。なければ最後に呼び出したチケット
- **リアルタイム更新方式**: WebSocket（STOMP）を採用。チケットのステータス遷移時に `QueueStatusChangedEvent` を発行し、以下のトピックにブロードキャスト:
  - `/topic/queue/{scopeType}/{scopeId}/status` — スコープ全体の待ち状況（待合室全体ディスプレイ用）
  - `/topic/queue/{scopeType}/{scopeId}/counter/{counterId}` — 特定窓口の変更（窓口別ディスプレイ用）
  - `/topic/queue/{scopeType}/{scopeId}/ticket/{ticketId}` — 特定チケットのステータス変更（利用者のスマホ画面用。自分のチケットのみ購読）
  - WebSocket 非対応環境向けのフォールバックとして30秒間隔のポーリングも併用

- `waiting_ahead`: 自分の前にいる待ち人数（`position - 1` ではなく、実際の WAITING + CALLED 数で動的算出）
- `estimated_wait_minutes`: `waiting_ahead × avg_service_minutes`（リアルタイム計算）
- `estimated_call_time`: `NOW() + estimated_wait_minutes` で動的算出。「おおよその呼び出し時刻」として表示（例: 「11:10頃」）。外出判断の目安に使用

#### `PATCH /api/v1/teams/{teamId}/queue/tickets/{id}/call`

**認可**: ADMIN / DEPUTY_ADMIN（`MANAGE_QUEUE`）

特定のチケットを順番を飛ばして呼び出す。通常は `call-next`（先頭呼び出し）を使用するが、緊急患者の優先対応等で特定チケットを直接呼び出す場合に使用。

**リクエストボディ**: なし

**処理の流れ**
1. `status = WAITING` を検証（それ以外は 409）
2. `status` → `CALLED`、`called_at` = NOW()
3. 呼び出されたチケットの `position` より前にいた `WAITING` チケットの `position` はそのまま維持（順番は変わらない。呼び出されたチケットだけがキューから抜ける）
4. 通知送信（`call-next` と同一）
5. **監査ログに「順番外呼び出し」を記録**（通常の `call-next` と区別可能にする）

**レスポンス（200 OK）**: `call-next` と同一構造

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | 権限不足 |
| 404 | チケットが存在しない |
| 409 | status が WAITING でない |

#### `PATCH /api/v1/teams/{teamId}/queue/tickets/call-next`

**認可**: ADMIN / DEPUTY_ADMIN（`MANAGE_QUEUE`）

指定窓口の待ち行列先頭を自動呼び出し。

**リクエストボディ**
```json
{
  "counter_id": 1
}
```

**レスポンス（200 OK）**
```json
{
  "data": {
    "ticket_id": 42,
    "ticket_number": "A-015",
    "status": "CALLED",
    "called_at": "2026-03-18T11:00:00",
    "user_display_name": "田中太郎",
    "party_size": 1
  }
}
```

**処理の流れ**
1. **INDIVIDUAL モード**: `counter_id` の `WAITING` チケットを `position ASC` で先頭を取得
   **SHARED モード**: リクエストの `counter_id` が属するカテゴリ内の全 `WAITING` チケットを `position ASC` で先頭を取得。チケットの `counter_id` を呼び出し窓口の ID に付け替え
2. `status` → `CALLED`、`called_at` = NOW()
3. 会員の場合: プッシュ通知 `QUEUE_CALLED`（「番号 A-015 でお呼びしています（{窓口名}）」）を送信
4. GUEST の場合: SMS 通知を `guest_phone` に送信（F04.3 の SMS 送信機能を使用）
5. NO_SHOW タイマー開始: `no_show_timeout_minutes` 後に自動 NO_SHOW 遷移をスケジュール（`ScheduledExecutorService` で遅延タスクを登録。アプリ再起動時は `CALLED` チケットの `called_at` + タイムアウト時間をチェックして再スケジュール）
6. **「もうすぐ呼び出し」通知**: 呼び出し後、待ち行列内の `position` が `almost_ready_threshold` 以下になったチケットの所有者に `QUEUE_ALMOST_READY` 通知を送信（既送信チェック付き）

### NO_SHOW 検出（リアルタイム）

```java
@Scheduled(fixedRate = 60000)  // 1分ごとにチェック
public void detectQueueNoShows() {
    // スコープごとに NO_SHOW タイムアウトが異なるため、CALLED チケットをスコープ別にグルーピング
    List<QueueTicket> calledTickets = ticketRepository
        .findByStatusAndIssuedDate(TicketStatus.CALLED, LocalDate.now());

    for (QueueTicket ticket : calledTickets) {
        QueueCounter counter = ticket.getCounter();
        QueueCategory category = counter.getCategory();  // カテゴリ経由でスコープ・キューモードを参照
        QueueSettings settings = getSettings(category.getScopeType(), category.getScopeId());
        // 一時離席中の場合は hold_until で判定（通常の timeout より延長）
        LocalDateTime timeoutAt = ticket.getHoldUntil() != null
            ? ticket.getHoldUntil()
            : ticket.getCalledAt().plusMinutes(settings.getNoShowTimeoutMinutes());

        if (LocalDateTime.now().isAfter(timeoutAt)) {
            ticket.markAsNoShow();
            // 後続チケットの position を前詰め（SHARED モードの場合はカテゴリ内全窓口を対象）
            if ("SHARED".equals(category.getQueueMode())) {
                ticketRepository.decrementPositionsByCategoryAfter(
                    ticket.getCategoryId(), ticket.getIssuedDate(), ticket.getPosition());
            } else {
                ticketRepository.decrementPositionsAfter(
                    ticket.getCounterId(), ticket.getIssuedDate(), ticket.getPosition());
            }

            // ペナルティカウント更新（会員のみ。GUEST はペナルティ対象外）
            if (ticket.getUserId() != null) {
                penaltyService.recordNoShow(ticket.getUserId(),
                    category.getScopeId(), category.getScopeType());
            }

            // WebSocket ブロードキャスト
            eventPublisher.publishEvent(new QueueStatusChangedEvent(
                category.getScopeType(), category.getScopeId()));
        }
    }
}
```

### NO_SHOW ペナルティチェック

```java
public void validateNoShowPenalty(Long userId, String scopeType, Long scopeId, String source) {
    // QRコード受付はペナルティ対象外（来場者を門前払いにしない）
    if ("QR".equals(source)) return;

    QueueSettings settings = getSettings(scopeType, scopeId);
    if (!settings.getNoShowPenaltyEnabled()) return;

    LocalDate thirtyDaysAgo = LocalDate.now().minusDays(30);
    long noShowCount = ticketRepository.countNoShowByUserAndDateAfter(
        userId, thirtyDaysAgo);

    if (noShowCount >= settings.getNoShowPenaltyThreshold()) {
        LocalDate lastNoShowDate = ticketRepository
            .findLatestNoShowDate(userId, scopeId, scopeType);
        LocalDate penaltyEndDate = lastNoShowDate
            .plusDays(settings.getNoShowPenaltyDays());

        if (!LocalDate.now().isAfter(penaltyEndDate)) {
            throw new UnprocessableEntityException("QUEUE_NO_SHOW_PENALTY",
                String.format("無断キャンセル（直近30日間で%d回）により%sまでネット予約できません。" +
                    "現地QRコードでの受付は引き続きご利用いただけます。",
                    noShowCount, penaltyEndDate));
        }
    }
}
```

### 待ち時間の推定計算

```java
public int estimateWaitMinutes(Long counterId, int position) {
    QueueCounter counter = counterRepository.findById(counterId).orElseThrow();
    QueueCategory category = counter.getCategory();

    if ("SHARED".equals(category.getQueueMode())) {
        // SHARED モード: カテゴリ内の稼働中窓口数で分散を考慮
        long activeCounters = counterRepository
            .countByCategoryIdAndIsActive(category.getId(), true);
        // カテゴリ内窓口の平均 avg_service_minutes を使用
        int avgMinutes = counterRepository
            .avgServiceMinutesByCategoryIdAndIsActive(category.getId(), true);
        int effectivePosition = (int) Math.ceil(
            (double) position / Math.max(activeCounters, 1));
        return effectivePosition * avgMinutes;
    } else {
        // INDIVIDUAL モード: 窓口単独の avg_service_minutes をそのまま使用
        return position * counter.getAvgServiceMinutes();
    }
}
```

public void holdTicket(Long ticketId, Long currentUserId) {
    QueueTicket ticket = ticketRepository.findById(ticketId).orElseThrow();

    if (!TicketStatus.CALLED.equals(ticket.getStatus()))
        throw new ConflictException("CALLED 状態のチケットのみ一時離席可能です");
    if (ticket.getHoldUsed())

    if (!currentUserId.equals(ticket.getUserId()))

    ticketRepository.save(ticket);

    eventPublisher.publishEvent(new QueueStatusChangedEvent(...));
}

## 6. セキュリティ考慮事項

- **認可チェック**: 窓口管理操作は ADMIN / DEPUTY_ADMIN（`MANAGE_QUEUE`）限定
- **スコープ分離**: 全 API で `teamId` パスパラメータとスコープメンバーシップを検証
- **GUEST の電話番号保護**: `guest_phone_encrypted` は AES-256 で暗号化保存。ADMIN のみ復号した電話番号を閲覧可能。MEMBER のチケット一覧や待ち状況 API には含めない。`guest_phone_hash` で検索（同時チケット制限・NO_SHOW 追跡）し、復号は SMS 送信・ADMIN 表示時のみ実行
- **QRトークンのセキュリティ**: `qr_token` は SecureRandom で64文字生成。推測不能。定期的な再発行を推奨
- **レート制限**: チケット発行 API は1ユーザーあたり1分間に5回。QRコード受付は1トークンあたり1分間に10回（同一イベントの複数人受付を考慮）
- **SMS 送信制限**: GUEST への SMS 通知は1電話番号あたり1日10通まで（スパム防止）
- **個人情報のマスキング**: 待ち状況表示では他人のチケット情報は番号のみ表示。名前・電話番号は非表示
- **GUEST 個人情報の最小化**: `guest_phone_encrypted` / `guest_phone_hash` / `guest_name` は発券日から90日後に NULL 化バッチで削除。統計データ（ステータス・対応時間）は保持
- **SMS 送信プロバイダ**: AWS SNS（SMS）を使用。1通あたり約0.07〜0.1ドル（日本向け）。コスト制御のため `QUEUE_CALLED` 通知のみ SMS 送信し、`QUEUE_ALMOST_READY` はプッシュ通知のみ（会員限定）
- **一時離席の濫用防止**: `hold_used` フラグで1チケット1回のみに制限。GUEST は一時離席不可（SMS でのみ通知されるため、アプリ操作ができない）
- **転送のスコープ検証**: 転送先カテゴリが同一スコープに属することを検証（異なるチーム/組織への転送は不可）
- **予約統合の IDOR 防止**: `reservation_id` が当日の予約かつ対象ユーザーのものであることを検証
- **ディスプレイボードの情報制限**: `display-board` API は番号・窓口名のみ返却。利用者の名前・電話番号は含めない
- **監査ログ**: `QUEUE_TICKET_CALLED` / `QUEUE_TICKET_NO_SHOW` / `QUEUE_TICKET_TRANSFERRED` / `QUEUE_TICKET_HOLD` / `QUEUE_SETTINGS_UPDATED` / `QUEUE_COUNTER_CREATED` / `QUEUE_COUNTER_DELETED` / `QUEUE_COUNTER_ACCEPTING_CHANGED` 等を記録

- [x] **チケット番号・position の原子的採番**: `SELECT MAX(...) FOR UPDATE` で排他制御。同時発券時の番号重複・position 競合を防止
- [x] **GUEST の NO_SHOW 追跡とプライバシー**: `guest_phone` ハッシュで追跡するがペナルティは適用しない（GUEST にネット予約停止の対象がないため）。電話番号は90日後にバッチで NULL 化（個人情報最小化）
- [x] **リアルタイム更新方式**: WebSocket（STOMP）を採用。`/topic/queue/{scopeType}/{scopeId}/status` にブロードキャスト。非対応環境は30秒ポーリングでフォールバック
- [x] **営業時間終了時の WAITING チケット**: 自動キャンセルしない（延長営業考慮）。翌日0時バッチで一括キャンセル
- [x] **「もうすぐ呼び出し」通知トリガー**: `queue_settings.almost_ready_threshold`（デフォルト3人前）で設定。`call-next` 実行後に閾値以下のチケット所有者に通知
- [x] **待ち状況の一般公開設定**: `queue_settings.display_board_public` で認証不要アクセスを制御
- [x] **SMS プロバイダとコスト制御**: AWS SNS を使用。SMS は `QUEUE_CALLED` のみ（GUEST 向け）。`QUEUE_ALMOST_READY` はプッシュ通知限定（会員のみ）
