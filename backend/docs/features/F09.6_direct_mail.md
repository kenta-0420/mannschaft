## 1. 概要

ADMIN / DEPUTY_ADMIN が組織・チーム内のメンバーに対して、リッチテキストメールを一斉・個別送信する機能。即時送信と予約送信に対応。

**主要機能**:
- **配信対象選択**: 全メンバー一斉 / グループ指定 / ロール指定 / 個別選択
- **リッチテキスト**: Markdown 形式で本文作成 → HTML メールに変換して送信
- **予約送信**: 日時指定で送信予約。予約のキャンセルも可能
- **開封率トラッキング**: トラッキングピクセルで開封を検知
- **送信制限**: チーム/組織単位で**週3回まで無料**。翌週月曜 00:00 にリセット
- **配信履歴**: 件名・送信日時・送信数・開封率を ADMIN が確認

### メール配信基盤

F04.3 の `NotificationChannel` フレームワークに `EmailChannel` を追加し、プラットフォーム全体のメール配信基盤として機能させる。ダイレクトメールはこの `EmailChannel` を利用する最初のユースケースだが、将来的にアカウント認証メール・パスワードリセット・異議申立て（F10.2）等にも同一基盤を再利用。

> **メール配信サービス**: AWS SES（Simple Email Service）を採用。理由: (1) 既存の AWS インフラ（S3・CloudFront 等）との親和性。(2) バウンス/苦情の SNS Webhook が標準装備。(3) 従量課金（1,000通あたり $0.10）でスタートアップに最適。
>
> ```yaml
> mannschaft.email:
>   provider: ses                          # ses / smtp（開発環境用）
>   from-address: noreply@mannschaft.app          # トランザクションメール用（パスワードリセット等）
>   marketing-from-address: mail@mannschaft.app   # マーケティングメール用（ダイレクトメール・プロモーション）
>   from-name: Mannschaft
>   region: ap-northeast-1
>   weekly-send-limit: 3                   # チーム/組織あたり週次送信上限
>   max-recipients-per-send: 500           # 1回の送信の最大受信者数
>   tracking-enabled: true                 # 開封トラッキング有効化
> ```

### テーブル定義

#### `direct_mail_logs`

送信セッション記録。1レコード = 1回の送信アクション。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | `TEAM` / `ORGANIZATION` |
| `scope_id` | BIGINT UNSIGNED | NO | — | FK → teams / organizations |
| `sender_id` | BIGINT UNSIGNED | NO | — | FK → users（送信者。ADMIN / DEPUTY_ADMIN） |
| `subject` | VARCHAR(200) | NO | — | 件名 |
| `body_markdown` | TEXT | NO | — | 本文（Markdown 形式。最大10,000文字） |
| `body_html` | TEXT | NO | — | 本文（Markdown → HTML 変換済み。送信時に生成） |
| `recipient_type` | VARCHAR(20) | NO | — | `ALL`（全メンバー）/ `ROLE`（ロール指定）/ `GROUP`（グループ指定）/ `INDIVIDUAL`（個別選択） |
| `recipient_filter` | JSON | YES | NULL | 配信条件（例: `{"roles":["MEMBER","SUPPORTER"]}` / `{"group_ids":[1,2]}` / `{"user_ids":[5,6,7]}`） |
| `estimated_recipients` | INTEGER | YES | NULL | 予定配信数（SCHEDULE 時に概算取得。UI 表示用。実際の送信数は `total_recipients` に送信時に確定） |
| `total_recipients` | INTEGER | NO | 0 | 実配信先の総数（送信実行時に確定。SCHEDULED 中は 0） |
| `sent_count` | INTEGER | NO | 0 | 送信成功数（デノーマライズ） |
| `opened_count` | INTEGER | NO | 0 | 開封数（デノーマライズ） |
| `bounced_count` | INTEGER | NO | 0 | バウンス数（デノーマライズ） |
| `status` | VARCHAR(20) | NO | 'DRAFT' | `DRAFT`（下書き）/ `SCHEDULED`（予約済み）/ `SENDING`（送信中）/ `SENT`（送信完了）/ `FAILED`（送信失敗）/ `CANCELLED`（キャンセル） |
| `scheduled_at` | DATETIME | YES | NULL | 予約送信日時（NULL = 即時送信。未来の日時のみ） |
| `error_message` | TEXT | YES | NULL | FAILED 時のエラー詳細（タイムアウト・配信対象0名等） |
| `sent_at` | DATETIME | YES | NULL | 実際の送信完了日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_dml_scope (scope_type, scope_id, status, created_at DESC)  -- スコープ別の送信履歴
INDEX idx_dml_scheduled (status, scheduled_at)                        -- 予約送信バッチ用
INDEX idx_dml_sender (sender_id, created_at DESC)                     -- 送信者別の履歴
```

**制約・備考**
- `body_markdown` → `body_html` の変換は保存時（DRAFT 保存・編集時）に実行（Commonmark ライブラリ。XSS 対策のサニタイズ含む）。`body_html` はプレースホルダ未置換のテンプレート版。送信ジョブでは `body_html` をベースに受信者ごとのプレースホルダ置換 + トラッキングピクセル挿入を行う（Markdown→HTML の再変換は不要）
- `DRAFT` → `SCHEDULED`（予約）or `SENDING`（即時送信）→ `SENT` / `FAILED`
- `SCHEDULED` → `CANCELLED`（予約キャンセル。送信前のみ）
- `recipient_filter`: `recipient_type` に応じた JSON 構造。`ALL` の場合は NULL
- 週次送信上限チェック: `status IN ('SENT', 'SENDING', 'SCHEDULED')` かつ当週（月曜〜日曜）の件数で判定

#### `direct_mail_recipients`

受信者ごとの配信ステータス。開封トラッキング・バウンス管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `mail_log_id` | BIGINT UNSIGNED | NO | — | FK → direct_mail_logs（ON DELETE RESTRICT。配信ログの削除時に受信者データが消えないよう保護。DRAFT 削除時のみアプリ層で先に recipients を物理削除） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（受信者） |
| `email` | VARCHAR(255) | NO | — | 送信先メールアドレス（送信時点のスナップショット） |
| `status` | VARCHAR(20) | NO | 'PENDING' | `PENDING`（未送信）/ `SENT`（送信成功）/ `FAILED`（送信失敗）/ `BOUNCED`（バウンス） |
| `ses_message_id` | VARCHAR(200) | YES | NULL | AWS SES Message ID（配信追跡用） |
| `opened_at` | DATETIME | YES | NULL | 開封日時（トラッキングピクセルで検知。最初の開封のみ記録） |
| `bounce_type` | VARCHAR(20) | YES | NULL | バウンス種別（`HARD` / `SOFT`。SES SNS Webhook で取得） |
| `error_message` | TEXT | YES | NULL | 送信失敗時のエラー詳細 |
| `retried` | BOOLEAN | NO | FALSE | 自動リトライ済みフラグ（TRUE = リトライ実行済み。1回のみ） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_dmr_log (mail_log_id, status)                    -- メール別の配信ステータス一覧
INDEX idx_dmr_user (user_id, created_at DESC)              -- ユーザー別の受信履歴
INDEX idx_dmr_ses (ses_message_id)                          -- SES Webhook からの検索
INDEX idx_dmr_retry (mail_log_id, status, retried)         -- 自動リトライ対象の検索
```

**制約・備考**
- `email`: 送信時点の `users.email` をスナップショット保存（ユーザーがメールアドレスを変更しても送信記録は不変）
- `opened_at`: HTML メール内のトラッキングピクセル（1x1 透明 GIF）の読み込みで検知。`GET /api/v1/mail-tracking/{token}` → 1x1 GIF を返却 + `opened_at = NOW()` を非同期更新
- `bounce_type = HARD`（恒久的な不達。メールアドレスが存在しない等）の場合、`users.email_bounced = true` を設定し、以降の配信対象から自動除外
- SES バウンス/苦情通知: SNS → SQS → Spring `@SqsListener` で受信

#### `direct_mail_image_uploads`

メール用画像の S3 管理。F06.1 `blog_image_uploads` と同一パターン。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams |
| `mail_log_id` | BIGINT UNSIGNED | YES | NULL | FK → direct_mail_logs（紐付け前は NULL） |
| `s3_key` | VARCHAR(500) | NO | — | S3 オブジェクトキー |
| `file_name` | VARCHAR(200) | NO | — | 元ファイル名 |
| `file_size` | INTEGER | NO | — | ファイルサイズ（バイト。最大5MB） |
| `content_type` | VARCHAR(50) | NO | — | MIMEタイプ（`image/jpeg` / `image/png` / `image/gif`） |
| `uploaded_by` | BIGINT UNSIGNED | NO | — | FK → users |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_dmiu_mail (mail_log_id)  -- メール別の画像一覧
INDEX idx_dmiu_orphan (mail_log_id, created_at)  -- 孤児画像クリーンアップ用
```

**制約・備考**
- アップロード: `POST /direct-mails/images` で S3 Pre-signed URL を取得 → クライアントが直接 S3 にアップロード → CDN URL（`https://cdn.mannschaft.app/dm/{s3_key}`）を Markdown 内で `![](URL)` として参照
- メール保存時: `body_markdown` 内の S3 URL をスキャンし、該当する画像の `mail_log_id` を紐付け
- 孤児削除: `mail_log_id IS NULL AND created_at < NOW() - INTERVAL 72 HOUR` を日次バッチで物理削除 + S3 オブジェクト削除
- 1メールあたり最大10画像、1画像あたり最大5MB

#### `POST /api/v1/teams/{teamId}/direct-mails`

**認可**: ADMIN / DEPUTY_ADMIN（`SEND_MAIL` 権限）

メールを作成する。`action` で下書き保存 / 即時送信 / 予約送信を指定。

**リクエストボディ**
```json
{
  "subject": "4月の休診日のお知らせ",
  "body_markdown": "# 4月の休診日のお知らせ\n\n{member_name} 様\n\n以下の日程で休診いたします...",
  "recipient_type": "ALL",
  "recipient_filter": null,
  "action": "SEND",
  "scheduled_at": null,
  "template_id": null
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `subject` | String | Yes | 件名（最大200文字） |
| `body_markdown` | String | Yes | 本文 Markdown（最大10,000文字） |
| `recipient_type` | String | Yes | `ALL` / `ROLE` / `GROUP` / `INDIVIDUAL` |
| `recipient_filter` | JSON | 条件付き | `ROLE`: `{"roles":["MEMBER"]}` / `GROUP`: `{"group_ids":[1]}` / `INDIVIDUAL`: `{"user_ids":[5,6]}` |
| `action` | String | Yes | `DRAFT`（下書き保存）/ `SEND`（即時送信）/ `SCHEDULE`（予約送信） |
| `scheduled_at` | DateTime | SCHEDULE 時必須 | 予約送信日時（未来のみ） |
| `template_id` | Long | No | テンプレートから作成する場合の ID |

**レスポンス（201 Created）**
```json
{
  "data": {
    "id": 1,
    "subject": "4月の休診日のお知らせ",
    "status": "SENDING",
    "total_recipients": 45,
    "created_at": "2026-03-18T10:00:00+09:00"
  },
  "quota": { "used": 1, "limit": 3, "remaining": 2 }
}
```

**補足**
- `action = SEND`: 週次上限チェック → `status = SENDING` → 非同期でメール送信 → `SENT` / `FAILED`
- `action = DRAFT`: `status = DRAFT` で保存。送信しない
- `action = SCHEDULE`: `scheduled_at` を設定して `status = SCHEDULED`。バッチで送信
- `template_id` 指定時: テンプレートの `subject` / `body_markdown` を初期値としてコピー。リクエストの `subject` / `body_markdown` が指定されていればそちらを優先
- 週次上限: `SEND` / `SCHEDULE` 時にチェック。`DRAFT` はカウント対象外

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | subject が空 / body_markdown が10,000文字超過 / recipient_type に対する filter 不正 / scheduled_at が過去 |
| 403 | 権限不足 |
| 409 | 週次送信上限（3回）に到達 |
| 422 | 配信対象が0名（全員がメール未設定 or バウンス済み） |

## 5. ビジネスロジック

### 主要フロー

#### 即時送信フロー
```
1. ADMIN が POST /direct-mails で action = SEND を指定
2. バックエンドがバリデーション:
   a. 週次送信上限チェック（Redis: dm:weekly:{scopeType}:{scopeId}:{isoYearWeek}。TTL 8日）
   b. 配信対象の確定:
      - recipient_type に基づいてメンバーリストを取得
      - INDIVIDUAL の場合: `user_ids` がスコープ内メンバーであることを検証（スコープ外は 400）
      - email が未設定 or email_bounced = true のユーザーを除外
      - 残りが0名 → 422
   c. max_recipients_per_send（デフォルト500名）超過チェック → 超過は 400
3. direct_mail_logs に INSERT（status = SENDING）
4. direct_mail_recipients に対象メンバー分を一括 INSERT（status = PENDING）
5. Redis の週次カウンターをインクリメント
6. 非同期ジョブをキューに投入:
   a. 保存済みの body_html（テンプレート版）を取得
   b. 各受信者に対して body_html のプレースホルダ置換（HTML エスケープ済みの値で）:
      - {member_name} → users.display_name
      - {team_name} / {org_name} → スコープ名
   c. トラッキングピクセルを HTML 末尾に挿入（受信者固有の HMAC トークン付き）
   d. AWS SES SendEmail API で送信（バッチ: 50通/秒。SES のデフォルト送信レート）
   e. 各受信者の status を SENT に更新。ses_message_id を記録。soft_bounce_count > 0 のユーザーは 0 にリセット
   f. 全送信完了後: direct_mail_logs.status = SENT, sent_at = NOW()
   g. 一部失敗がある場合: status = SENT（部分成功として扱う。failed は recipients 側で追跡）
7. 送信者にプッシュ通知（「メールの送信が完了しました（{sent_count}/{total_recipients}通）」）
```

#### 予約送信フロー
```
1. ADMIN が POST /direct-mails で action = SCHEDULE + scheduled_at を指定
2. バリデーション:
   - 週次上限チェック（即時送信と同様）
   - recipient_type / recipient_filter の形式チェック
   - INDIVIDUAL の場合: `user_ids` がスコープ内メンバーであることを検証（スコープ外は 400）
   - 概算配信数を取得（`estimated_recipients`。email 未設定 / bounced を除外した現時点のカウント）
3. direct_mail_logs に INSERT（status = SCHEDULED, scheduled_at, estimated_recipients）
   - direct_mail_recipients は INSERT しない（送信時にリストを再取得するため）
4. Redis の週次カウンターをインクリメント（予約時点でカウント。キャンセル時にデクリメント）
5. Spring @Scheduled バッチが1分ごとに実行:
   a. status = SCHEDULED AND scheduled_at <= NOW() のレコードを検索
   b. recipient_type + recipient_filter に基づいてメンバーリストを再取得（送信時点の最新メンバー）
      - email 未設定 / email_bounced = true を除外
      - 対象0名の場合: status = FAILED（error_message = '配信対象が0名'）+ 送信者に通知
   c. direct_mail_recipients に一括 INSERT
   d. status = SENDING に遷移
   e. 即時送信フローの手順6以降を実行
```

#### SES バウンス/苦情 Webhook フロー
```
1. AWS SES → SNS Topic → SQS Queue → Spring @SqsListener
2. メッセージタイプの判定:
   a. Bounce:
      - ses_message_id で direct_mail_recipients を検索
      - status = BOUNCED, bounce_type = HARD or SOFT を設定
      - direct_mail_logs.bounced_count をインクリメント
      - HARD バウンスの場合: users.email_bounced = true（以降の全メール配信から除外）
      - SOFT バウンスの場合: users.soft_bounce_count をインクリメント。3以上で email_bounced = true に自動昇格
   b. Complaint（苦情）:
      - ユーザーの notification_preferences で**マーケティングメール（DIRECT_MAIL / PROMOTION）のみ自動 OFF**
        （トランザクションメール（パスワードリセット・異議申立て等）は OFF にしない。特定電子メール法上、取引関連メールは配信停止対象外）
      - 監査ログ: EMAIL_COMPLAINT を記録
3. SNS 署名検証: AWS SDK の `SnsMessageVerifier` で署名を検証（偽装リクエスト防止）
```

#### SENDING タイムアウト検知
```
1. 予約送信バッチ（1分ごと実行）の冒頭で、タイムアウト検知を実行:
   - status = 'SENDING' AND updated_at < NOW() - INTERVAL 30 MINUTE のレコードを検索
   - status = 'FAILED', error_message = 'Sending timed out' に遷移
   - 送信者に DIRECT_MAIL_FAILED 通知を送信
2. 理由: 非同期ジョブがクラッシュした場合に SENDING のまま残るのを防止。
   500名送信でも SES 50通/秒で最大10秒。30分は十分な猶予
```

```
2. direct_mail_image_uploads から以下を検索:
```

#### 未開封者再送信フロー
```
1. ADMIN が POST /direct-mails/{id}/resend-to-unopened を実行
2. バリデーション:

   b. 週次送信上限チェック（新規メールとして1回消費）
   c. 未開封者リストの取得:
```

#### FAILED 受信者の自動リトライ
```
1. 送信ジョブ完了時（即時送信フロー手順6g の後）:
   a. direct_mail_recipients WHERE mail_log_id = {id}
      AND status = 'FAILED' AND retried = FALSE を検索
   b. error_message が SES スロットリング or タイムアウト系の場合のみ対象
      （HARD バウンス・宛先不明・認証エラーはリトライしない）
   c. 対象レコードの retried = TRUE に更新
   d. Redis キューに投入: dm:retry:{mail_log_id}（TTL 10分）
2. 5分後のリトライバッチ（Spring @Scheduled、1分ごと実行）:
   a. dm:retry:* キーを検索（投入から5分以上経過したもの）
   b. 対象受信者に対して SES SendEmail API で再送信
   c. 成功 → status = SENT, ses_message_id 更新。direct_mail_logs.sent_count インクリメント
   d. 失敗 → status = FAILED のまま（2回目の失敗。これ以上リトライしない）
3. リトライは1回のみ。retried = TRUE のレコードは以降のリトライ対象外
4. リトライ対象: SES Throttling / ServiceUnavailable / RequestTimeout のみ
   対象外: MessageRejected / AccountSendingPaused / MailFromDomainNotVerified
```

#### バウンス率アラート
```
1. 送信ジョブ完了時（即時送信フローの手順7の後）に判定:
   a. 直近30日間の集計:
      SELECT SUM(bounced_count) as total_bounced,
             SUM(total_recipients) as total_recipients
      FROM direct_mail_logs
      WHERE scope_type = ? AND scope_id = ?
        AND status = 'SENT'
        AND sent_at > NOW() - INTERVAL 30 DAY
   b. bounce_rate = total_bounced / total_recipients
2. 閾値判定:
   a. bounce_rate > 5%:
      - ADMIN にプッシュ通知（DIRECT_MAIL_BOUNCE_WARNING）
      - メール配信画面に警告バナー: 「バウンス率が {rate}% に達しています。
        メンバーのメールアドレスを確認してください」
   b. bounce_rate > 10%:
      - SYSTEM_ADMIN にプッシュ通知（SYSTEM_BOUNCE_ALERT）
      - 理由: SES アカウント停止リスク（AWS は 5% 超でレビュー、10% 超で停止の可能性）
3. アラートの重複防止:
   - Redis: dm:bounce_alert:{scopeType}:{scopeId}:{level}（TTL 7日）
   - 同一スコープ・同一レベルのアラートは7日間に1回のみ送信
```

#### 送信ログアーカイブバッチ
```
1. Spring @Scheduled バッチが日次で実行（05:00 JST）
2. アーカイブ対象の検索:
   SELECT id FROM direct_mail_logs
   WHERE status IN ('SENT', 'CANCELLED', 'FAILED')
     AND COALESCE(sent_at, updated_at) < NOW() - INTERVAL 6 MONTH
   LIMIT 1000
3. バッチサイズ 1000件/回で処理（ロック時間抑制）:
   a. 対象の direct_mail_logs → direct_mail_logs_archive に INSERT
   b. 対応する direct_mail_recipients → direct_mail_recipients_archive に INSERT
   c. 元テーブルから DELETE（recipients → logs の順。FK RESTRICT のため）
   d. 1000件処理後に残りがあれば次回バッチに持ち越し
4. トランザクション: ログ1件 + その受信者全件を1トランザクションで処理
   （1件のログに紐づく受信者が最大500件のため、トランザクションサイズは適切）
5. 実行ログ: 移動件数（ログ数 + 受信者数）をアプリケーションログに記録
6. DRAFT / SCHEDULED / SENDING はアーカイブ対象外
   （DRAFT は手動削除、SCHEDULED は送信待ち、SENDING はタイムアウト検知で処理）
```

#### 週次送信制限の管理
```
Redis キー: dm:weekly:{scopeType}:{scopeId}:{isoYearWeek}
  例: dm:weekly:TEAM:5:2026-W12
TTL: 8日（月曜 00:00 を跨いで余裕を持たせる）
値: 今週の送信回数（SEND + SCHEDULE した件数）

- SEND / SCHEDULE 時: INCR → 3 超過なら 409
- CANCEL 時: DECR（予約キャンセルで枠を返却）。ただし `MAX(DECR result, 0)` で負数にならないようガード（週をまたいだキャンセルで TTL 切れ後に DECR すると -1 になるため）
- DRAFT は カウント対象外
```

### 重要な判定ロジック

- **週次送信制限**: Redis カウンターで管理。`SEND` / `SCHEDULE` 時にインクリメント、`CANCEL` 時にデクリメント。`DRAFT` はカウント外。月曜 00:00 にリセット（Redis TTL 8日で自動期限切れ）
- **バウンスメールの自動除外**: `users.email_bounced = true` のユーザーは配信対象リストから自動除外。HARD バウンスは即座に `email_bounced = true`。SOFT バウンスは `users.soft_bounce_count` をインクリメントし、3回連続で `email_bounced = true` に自動昇格。送信成功（`SENT`）時に `soft_bounce_count = 0` にリセット
- **email_bounced の自動リカバリ**: ユーザーがメールアドレスを変更した場合（`PUT /users/me/profile`）、`email_bounced = false` + `soft_bounce_count = 0` に自動リセット（`ApplicationEvent` で連携）。新メアドで再びバウンスすれば再度フラグが立つためリスクは低い
- **送信元の分離（SES レピュテーション保護）**: マーケティングメール（ダイレクトメール・プロモーション）は `mail@mannschaft.app`、トランザクションメール（パスワードリセット・異議申立て等）は `noreply@mannschaft.app` で送信元を分離。SES の Configuration Set を分けてレピュテーションを独立管理。マーケティングの苦情率がトランザクションの到達率に影響しない
- **開封トラッキングの精度**: HTML メール内のトラッキングピクセルに依存。テキストメールクライアント・画像ブロック環境では検知不可。`open_rate` は実際の開封率の下限値として扱う
- **プレースホルダ置換のセキュリティ**: `{member_name}` 等の置換値は HTML エスケープしてから挿入（XSS 防止）
- **予約送信のキャンセル**: `SCHEDULED` → `CANCELLED` は送信前のみ。`SENDING` 以降はキャンセル不可。キャンセル時に Redis 週次カウンターをデクリメント
- **大量配信のスロットリング**: SES の送信レート（デフォルト50通/秒）に合わせてバッチ送信。500名超過の場合は API レベルで拒否（`max_recipients_per_send`）
- **メールアドレスのスナップショット**: `direct_mail_recipients.email` は送信時点の `users.email` をコピー。送信後にユーザーがメアドを変更しても送信記録は不変
- **SENDING タイムアウト**: `updated_at` から30分超過した `SENDING` レコードは予約送信バッチの冒頭で `FAILED` に遷移。送信者に通知。500名×50通/秒 = 最大10秒のため30分は十分な猶予
- **テスト送信**: `POST /test-send` で送信者自身 or 指定メアド宛にテスト1通を送信。週次上限カウント対象外。`direct_mail_recipients` に記録しない。プレースホルダは送信者名で置換。トラッキングピクセルは含めない。レートリミット: 1時間5回
- **画像アップロード**: `POST /direct-mails/images` で S3 Pre-signed URL を取得。Markdown 内で `![](CDN URL)` として参照。メール保存時に画像を紐付け。孤児画像（72時間未紐付け）は日次バッチで削除。1メールあたり最大10画像、1画像最大5MB
- **予約送信のリスト再取得**: 予約送信では `SCHEDULE` 時にリストを確定せず、`SENDING` 遷移時（バッチ実行時）に `recipient_type` + `recipient_filter` で最新メンバーを再取得。予約〜送信間のメンバー入退会に対応。対象0名なら `FAILED` に遷移して送信者に通知
- **配信停止のスコープ別制御**: メールフッターの配信停止リンクで「このチーム/組織からのメールを停止」or「全マーケティングメールを停止」を選択可能。`notification_preferences` のスコープ別設定に連携。特定チームの DM だけが不要な場合に他チームの DM まで止まることを防止
- **送信ログのアーカイブ**: 6ヶ月以上前の `SENT` / `CANCELLED` / `FAILED` のログ + 受信者データを `direct_mail_logs_archive` / `direct_mail_recipients_archive` テーブルに移動する日次バッチ。F10.1 の `content_reports_archive` と同一パターン。通常の stats API 集計対象（7d/30d/90d）は6ヶ月以内に収まるためアーカイブを参照する必要なし
- **配信対象からの GUEST 除外**: GUEST ロールはメール配信対象外。`recipient_type = ALL` でも GUEST は含まれない
- **未開封者再送信**: SENT 状態のメールに対して `opened_at IS NULL AND status = 'SENT'` の受信者のみに再送信。新規 `direct_mail_logs` として作成（`recipient_type = 'INDIVIDUAL'`）。週次上限を1回消費。BOUNCED / FAILED の受信者は対象外。元メール送信から24時間未満の場合は warning を返却（開封トラッキング反映待ち推奨）
- **予約日時の変更**: `PATCH /schedule` で SCHEDULED 状態のメールの `scheduled_at` を変更可能。週次カウンターは変動しない（予約時点でカウント済み）。週をまたぐ変更でもカウンターの週は元の予約週で維持
- **配信対象プレビュー**: `POST /preview-recipients` でメール作成前に配信対象を確認。`direct_mail_logs` を作成しない純粋な確認用 API。bounced / no_email / GUEST の除外数を内訳表示。sample は先頭5名（部分マスク）
- **送信統計**: `GET /stats` で直近 7d/30d/90d の配信効果を集計。加重平均の開封率・バウンス率を算出。アーカイブテーブルは参照しない（最大90dのため6ヶ月以内）
- **受信者ステータス一覧**: `GET /recipients` で個別受信者の配信状況を確認。status / opened でフィルタ可能。メアドは部分マスク。DRAFT / SCHEDULED は recipients 未作成のため空配列
- **FAILED 自動リトライ**: SES スロットリング・タイムアウト系の FAILED 受信者のみ5分後に1回自動リトライ。HARD バウンス・宛先不明はリトライしない（レピュテーション保護）。`retried` フラグで2回目以降を防止
- **バウンス率アラート**: 送信完了時に直近30日のバウンス率を判定。5%超で ADMIN に警告通知 + UI バナー。10%超で SYSTEM_ADMIN に通知（SES アカウント停止リスク）。Redis で7日間の重複防止

## 6. セキュリティ考慮事項

- **SES 認証**: IAM ロールベースの認証（EC2 / ECS のインスタンスプロファイル）。API キーを環境変数に保存しない
- **SNS Webhook 署名検証**: `SnsMessageVerifier` で AWS 署名を検証。偽装リクエストを拒否
- **トラッキングトークンの改ざん防止**: `mail-tracking/{token}` の `token` は `HMAC-SHA256(recipient_id, secret_key)` で生成。recipient_id の推測によるトラッキング偽装を防止
- **HTML メールの XSS 防止**: Markdown → HTML 変換時に危険なタグ（`<script>` / `<iframe>` / `on*` 属性等）を除去。プレースホルダ置換値も HTML エスケープ
- **配信リストのプライバシー**: 受信者リスト（`direct_mail_recipients`）は送信者（ADMIN）のみ閲覧可能。受信者同士は互いの受信状況を確認できない
- **レートリミット**: 週3回の送信制限 + SES の50通/秒スロットリング + API レベルで500名/回の上限
- **苦情対応（CAN-SPAM / 特定電子メール法）**: メールフッターに配信停止リンクを自動挿入。リンク先ページでは「このチーム/組織からのメールを停止」と「すべてのマーケティングメールを停止」の2つの選択肢を提供（`notification_preferences` のスコープ別設定に連携）。CAN-SPAM の「送信者からのメールを停止」要件を満たしつつ、他チームの DM は維持可能
- **監査ログ**: `DIRECT_MAIL_SENT` / `DIRECT_MAIL_SCHEDULED` / `DIRECT_MAIL_CANCELLED` / `DIRECT_MAIL_RESENT` / `EMAIL_BOUNCED` / `EMAIL_COMPLAINT` / `DIRECT_MAIL_BOUNCE_WARNING` を記録
