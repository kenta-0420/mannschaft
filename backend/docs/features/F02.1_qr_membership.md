## 1. 概要

個人アカウントが所属するチーム・組織ごとにデジタル会員証を発行し、QRコードでメンバー認証を行う機能。メンバーはスマートフォンに会員証QRを表示し、店舗・施設側がスキャンすることで来店確認・本人確認を実施する。QRコードは署名付きトークンから動的に生成し、画像としては保存しない。スキャン時にサーバーで署名検証 + メンバーシップ確認を行い、不正利用を防止する。

**主要ユースケース**:
- 整骨院・美容室・ジム: 来店時にQRをスキャンし、会員確認 + サービス履歴自動紐付け
- マンション管理組合: 居住者証明として共用施設利用時にスキャン
- 学校・サークル: メンバー確認・イベント入場管理
- 飲食店: 会員割引の適用確認

**2つのチェックイン方式**:
| 方式 | 操作 | 適合業種 |
|------|------|---------|
| **スタッフスキャン型** | 会員がQR表示 → スタッフがカメラで読取 | 整骨院・美容室（1対1対応。確実な本人確認） |
| **セルフチェックイン型** | 店舗がQRポップを掲示 → 会員が自分のスマホで読取 | ジム・コワーキング・イベント会場（大量入場。スタッフ不要） |

**招待QRとの棲み分け**:
| 機能 | 用途 | 管理元 |
|------|------|--------|
| **F02.1 QR会員証（本機能）** | 所属済みメンバーの本人認証・来店確認 | 個人（表示）+ チーム（スキャン） |
| F01.2 招待トークン（`invite_tokens`） | 未所属ユーザーの新規加入用QR | チーム/組織（発行） |

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全会員証の参照・無効化（モデレーション） |
| ADMIN | チーム/組織の会員証一覧参照・スキャン認証・会員証の無効化/再発行・チェックイン履歴閲覧 |
| DEPUTY_ADMIN | `MANAGE_MEMBERS` 権限を持つ場合: ADMIN と同等 |
| MEMBER | 自分の会員証表示・QR再生成 |
| SUPPORTER | 自分の会員証表示（QR再生成不可） |

### 対象レベル
- [x] 組織 (Organization) — 組織メンバーとしての会員証
- [x] チーム (Team) — チームメンバーとしての会員証
- [x] 個人 (Personal) — プラットフォーム会員証（チーム/組織非所属でも保有）

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `member_cards` | 会員証マスター（1ユーザー×1スコープ = 1枚） | あり（`deleted_at`） |

### テーブル定義

#### `member_cards`

デジタル会員証。ユーザーがチームまたは組織に所属した時点で自動発行される。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`PLATFORM` / `TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | YES | NULL | FK → teams.id or organizations.id。`PLATFORM` の場合は NULL |
| `card_code` | CHAR(36) | NO | — | 会員証コード（UUID v4。QR コード生成のソース） |
| `card_number` | VARCHAR(20) | NO | — | 表示用会員番号（例: `M-000001`）。スコープ内で連番 |
| `display_name` | VARCHAR(100) | NO | — | 会員証に表示する名前（`users.display_name` のスナップショット。変更時に同期） |
| `status` | VARCHAR(20) | NO | 'ACTIVE' | ステータス（`ACTIVE` / `SUSPENDED` / `REVOKED`） |
| `issued_at` | DATETIME | NO | CURRENT_TIMESTAMP | 発行日時 |
| `suspended_at` | DATETIME | YES | NULL | 一時停止日時 |
| `revoked_at` | DATETIME | YES | NULL | 無効化日時 |
| `last_checkin_at` | DATETIME | YES | NULL | 最終チェックイン日時（denormalize） |
| `checkin_count` | INT UNSIGNED | NO | 0 | チェックイン回数（denormalize） |
| `total_spend` | DECIMAL(12,2) | NO | 0.00 | 累計売上額（denormalize。将来のレジ/POS 連携用。Phase 8 で使用開始） |
| `qr_secret` | VARCHAR(64) | NO | — | QR署名用シークレット（カードごとにユニーク。HMAC-SHA256 署名に使用） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除（SoftDeletableEntity 適用） |
| `active_unique_key` | TINYINT | — | GENERATED | `GENERATED ALWAYS AS (IF(deleted_at IS NULL, 1, NULL)) VIRTUAL`。パーシャルユニーク用 |

**インデックス**
```sql
-- 1ユーザー×1スコープに1枚（生成カラム方式パーシャルユニーク。F03.4 と同方式）
-- member_cards に `active_unique_key TINYINT GENERATED ALWAYS AS (IF(deleted_at IS NULL, 1, NULL)) VIRTUAL` を追加
UNIQUE KEY uq_mc_user_scope_active (user_id, scope_type, scope_id, active_unique_key)
UNIQUE KEY uq_mc_card_code (card_code)                                    -- カードコードの一意性
INDEX idx_mc_scope (scope_type, scope_id, status)                         -- スコープ別会員証一覧
INDEX idx_mc_card_number (scope_type, scope_id, card_number)              -- 会員番号検索
INDEX idx_mc_last_checkin (scope_type, scope_id, last_checkin_at DESC)    -- 最終来店順
```

**制約・備考**
- **自動発行**: `TeamMemberJoinedEvent` / `OrganizationMemberJoinedEvent` を購読し、メンバー加入時に自動で `member_cards` を INSERT。`PLATFORM` スコープの会員証はユーザー登録時に自動発行
- `card_code`: UUID v4。QR コードのペイロードに含まれる。QR コードの「実体」はこの UUID であり、画像はフロントエンドで動的生成する（サーバーに画像を保存しない）
- `card_number`: スコープ内で連番を自動採番。フォーマット: `{prefix}-{6桁連番}`。prefix はチーム/組織の設定（`team_feature_settings` または `organization_settings` の `member_card_prefix` キー。デフォルト: `M`）で変更可能。例: `M-000001`, `GYM-000001`
- `display_name`: `users.display_name` が変更された場合に `UserProfileUpdatedEvent` で同期。リアルタイム同期ではなくイベント駆動で更新
- `status` のライフサイクル:
  - `ACTIVE`: 有効。QR スキャン認証が可能
  - `SUSPENDED`: 一時停止。ADMIN が手動で設定。スキャン時に「この会員証は一時停止中です」表示。メンバーシップは維持
  - `REVOKED`: 無効化。チーム/組織から脱退した場合に自動設定。再発行には再加入が必要
- `qr_secret`: カード発行時に `SecureRandom` で64文字のランダム文字列を生成。QR トークン生成時に HMAC-SHA256 の秘密鍵として使用。再生成時に `qr_secret` を更新すると旧QRが無効になる（セキュリティリセット）
- `uq_mc_user_scope_active`: 生成カラム `active_unique_key`（有効時 = 1、削除時 = NULL）を使用したパーシャルユニーク。F03.4 `reservations` と同方式。論理削除後の再発行が可能
- チーム/組織から脱退した場合: `TeamMemberLeftEvent` を購読し、`status = 'REVOKED'`、`revoked_at = NOW()` を自動設定

#### `member_card_checkins`

QRスキャンによるチェックイン（来店・入場）の履歴。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `member_card_id` | BIGINT UNSIGNED | NO | — | FK → member_cards。ON DELETE CASCADE |
| `checkin_type` | VARCHAR(20) | NO | 'STAFF_SCAN' | チェックイン種別（`STAFF_SCAN` = スタッフスキャン / `SELF` = セルフチェックイン） |
| `checked_in_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL（`STAFF_SCAN` 時のスキャンスタッフ。`SELF` 時は NULL） |
| `checkin_location_id` | BIGINT UNSIGNED | YES | NULL | FK → checkin_locations。ON DELETE SET NULL（`SELF` 時の拠点。`STAFF_SCAN` 時は NULL） |
| `checked_in_at` | DATETIME | NO | CURRENT_TIMESTAMP | チェックイン日時 |
| `location` | VARCHAR(200) | YES | NULL | チェックイン場所（`STAFF_SCAN`: 手入力。`SELF`: `checkin_locations.name` を自動設定） |
| `note` | VARCHAR(500) | YES | NULL | ADMIN 向けメモ |
| `reservation_id` | BIGINT UNSIGNED | YES | NULL | FK → reservations（予約と紐付け。F03.4 連携。Phase 3 で FK 追加） |
| `service_record_id` | BIGINT UNSIGNED | YES | NULL | FK → service_records（サービス履歴と紐付け。F07.1 連携。Phase 7 で FK 追加） |
| `transaction_id` | BIGINT UNSIGNED | YES | NULL | FK → transactions（将来のレジ/POS 連携用。Phase 8 で FK 追加。チェックイン時の売上データ紐付け） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_mcc_card (member_card_id, checked_in_at DESC)              -- カード別チェックイン履歴 + 二重スキャン防止（LIMIT 1 で最新取得）
INDEX idx_mcc_checked_in_by (checked_in_by, checked_in_at DESC)       -- スキャン者別履歴
INDEX idx_mcc_reservation (reservation_id)                            -- 予約連携
INDEX idx_mcc_location (checkin_location_id, checked_in_at DESC)      -- 拠点別チェックイン履歴
```

**制約・備考**
- 同一カードの連続チェックイン制限: 同一 `member_card_id` で直前のチェックインから5分以内の再スキャンは 409 エラー（二重スキャン防止）
- `reservation_id`: スキャン時にその日の予約を自動検索し、該当する `CONFIRMED` 予約があれば紐付け + `COMPLETED` に自動遷移（F03.4 連携。Phase 3 で FK 制約を ALTER TABLE で追加）
- `service_record_id`: チェックイン後に ADMIN がサービス履歴を作成した際にリンク（Phase 7 で FK 制約追加）
- チェックイン履歴は物理削除しない（監査証跡として保持）

#### `checkin_locations`

セルフチェックイン用の拠点定義。店舗の受付やジムの入口等にQRポップとして掲示する。1拠点 = 1つの固定QRコード。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | FK → teams.id or organizations.id |
| `name` | VARCHAR(100) | NO | — | 拠点名（例: 「受付」「第2スタジオ」「正面入口」） |
| `location_code` | CHAR(36) | NO | — | 拠点コード（UUID v4。QR に埋め込む） |
| `location_secret` | VARCHAR(64) | NO | — | QR署名用シークレット（`SecureRandom` で生成。HMAC-SHA256 署名に使用） |
| `is_active` | BOOLEAN | NO | TRUE | 有効フラグ。FALSE = このQRでチェックイン不可 |
| `auto_complete_reservation` | BOOLEAN | NO | TRUE | セルフチェックイン時に当日予約を自動 COMPLETED にするか |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
UNIQUE KEY uq_cl_location_code (location_code)                        -- 拠点コードの一意性
INDEX idx_cl_scope (scope_type, scope_id, is_active)                   -- スコープ別拠点一覧
```

**制約・備考**
- 1チーム/組織あたり最大20拠点（Service 層バリデーション）
- `location_code`: UUID v4。店舗設置QRの「実体」。QRポップに印刷する内容はこのコードから生成した署名付きトークン
- `location_secret`: 拠点ごとにユニーク。セルフチェックイン時に署名を検証。再生成すると掲示中の QR が無効になるため、再印刷が必要
- `is_active = FALSE`: QR をスキャンしても「この拠点は現在利用できません」エラー。物理的にポップを撤去せずとも即座に無効化可能
- **スタッフスキャン型との違い**: 店舗設置QRは**長期間有効**（`location_secret` が変更されない限り有効。会員証QRの5分有効期限とは異なる設計）。代わりに、チェックイン時にサーバーで「このユーザーはこのスコープに所属しているか」をリアルタイム検証する
- **セキュリティ**: QR画像自体は静的だが、スキャン時にサーバーでメンバーシップ＋カードステータスを検証するため、非メンバーや SUSPENDED/REVOKED の会員はチェックインできない
- `scope_type + scope_id`: ポリモーフィック FK。DB レベルの FK 制約は設けない。Service 層で `scope_type` に応じた存在チェックを実行（`surveys` テーブルと同方式）
- **`location_secret` の再生成**: QR漏洩時は拠点を DELETE → 再作成する運用。`location_secret` のみの再生成 API は提供しない（再生成すると掲示中のポップが無効になり再印刷が必要なため、拠点再作成と同じ手間。API を分ける実益がない）

**店舗設置QRトークンの構造**:
```
{location_code}.{signature}
```
- `location_code`: UUID（36文字）
- `signature`: `HMAC-SHA256(location_code, location_secret)` の Base64URL
- 会員証QRと異なり有効期限を埋め込まない（長期掲示用。セキュリティはサーバー側のメンバーシップ検証で担保）

#### `GET /api/v1/member-cards/{id}`

会員証詳細。本人は自分の会員証のみ、ADMIN+ はスコープ内の会員証を参照可能。

**エラーレスポンス**
| ステータス | 条件 |

#### `GET /api/v1/organizations/{orgId}/member-cards`

組織の会員証一覧。`GET /teams/{teamId}/member-cards` と同形式のリクエスト/レスポンス。

会員証の QR コード表示に必要なトークンを取得する。フロントエンドがこのトークンから QR コード画像を動的生成する。

**レスポンス（200 OK）**
```json
{
  "data": {
    "member_card_id": 2,
    "qr_token": "eyJjYXJkX2NvZGUiOiIxMjM0NTY3OC0...",
    "expires_in": 300,
    "card_number": "M-000012",
    "display_name": "田中太郎",
    "scope_type": "TEAM",
    "scope_name": "〇〇整骨院"
  }
}
```

- `qr_token`: `card_code` + 発行日時 + 有効期限を `qr_secret` で HMAC-SHA256 署名したトークン。Base64URL エンコード
- `expires_in`: QR トークンの有効期限（秒）。デフォルト: 300秒（5分）。期限切れ後はフロントエンドが自動で再取得
- フロントエンドは `qr_token` を QR コードライブラリ（`qrcode.js` 等）に渡して画像をレンダリング

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 401 | 未認証 |
| 403 | 他人の会員証 |
| 404 | 会員証不存在 |
| 409 | 会員証が `ACTIVE` でない（SUSPENDED / REVOKED） |

**QR トークンの構造**:
```
{card_code}.{issued_at_epoch}.{expires_at_epoch}.{signature}
```
- `card_code`: UUID（36文字）
- `issued_at_epoch`: トークン発行時刻（Unix秒）
- `expires_at_epoch`: 有効期限（Unix秒）
- `signature`: `HMAC-SHA256(card_code + "." + issued_at + "." + expires_at, qr_secret)` の Base64URL

**レスポンス（200 OK）— 認証失敗**
```json
{
  "data": {
    "verified": false,
    "reason": "EXPIRED",
    "message": "QRコードの有効期限が切れています。会員証画面を再表示してください。"
  }
}
```

| `reason` | 説明 |
|----------|------|
| `EXPIRED` | QR トークンの有効期限切れ |
| `INVALID_SIGNATURE` | 署名が不正（改ざん or 再生成後の旧QR） |
| `CARD_NOT_FOUND` | card_code に該当する会員証がない |
| `SUSPENDED` | 会員証が一時停止中 |
| `REVOKED` | 会員証が無効化済み |

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `qr_token` 未指定 / フォーマット不正 |
| 401 | 未認証 |
| 403 | スキャン権限なし（ADMIN+ でない）/ スキャンした ADMIN が当該カードのスコープに非所属 |
| 409 | 同一カードの5分以内の再スキャン（二重チェックイン防止） |

**認証処理フロー**:
1. `qr_token` を `.` で分割 → `card_code`, `issued_at`, `expires_at`, `signature` を取得
2. `expires_at` が現在時刻より前 → `EXPIRED` で返却
3. `card_code` で `member_cards` を検索 → 不存在なら `CARD_NOT_FOUND`
4. `qr_secret` で HMAC-SHA256 を再計算し、`signature` と比較 → 不一致なら `INVALID_SIGNATURE`
5. スコープ権限チェック → スキャンした ADMIN が当該カードのスコープ（チーム/組織）に所属しているか検証。`PLATFORM` スコープの場合は SYSTEM_ADMIN のみスキャン可能。他チームの会員証をスキャンしようとした場合は 403 エラー
6. `status` チェック → `SUSPENDED` / `REVOKED` ならそれぞれの reason で返却
7. 認証成功 → `member_card_checkins` に INSERT
8. `member_cards.last_checkin_at` / `checkin_count` をアトミック更新
9. `auto_complete_reservation = true` の場合: 当日の `CONFIRMED` 予約を検索し、該当があれば `COMPLETED` に遷移 + `reservation_id` を設定
10. ADMIN にチェックイン通知を送信（`MEMBER_CHECKED_IN`）

|----------|------|

| `LOCATION_INACTIVE` | 拠点が無効化されている（`is_active = FALSE`） |
| `NOT_MEMBER` | ログインユーザーが当該スコープに所属していない |
| `CARD_SUSPENDED` | 会員証が一時停止中 |
| `CARD_REVOKED` | 会員証が無効化済み |

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `location_qr_token` 未指定 / フォーマット不正 |
| 401 | 未認証 |

**セルフチェックイン処理フロー**:
1. `location_qr_token` を `.` で分割 → `location_code`, `signature` を取得
2. `location_code` で `checkin_locations` を検索 → 不存在なら `INVALID_LOCATION`
3. `location_secret` で HMAC-SHA256 を再計算し、`signature` と比較 → 不一致なら `INVALID_LOCATION`
4. `is_active` チェック → `FALSE` なら `LOCATION_INACTIVE`
5. ログインユーザーの `member_cards` を `scope_type + scope_id` で検索 → 不存在なら `NOT_MEMBER`
6. `status` チェック → `SUSPENDED` / `REVOKED` ならそれぞれの reason で返却
7. 二重チェックイン防止（5分以内の再スキャン → 409）
8. チェックイン成功 → `member_card_checkins` に INSERT（`checkin_type = 'SELF'`、`checkin_location_id` を設定、`location` に拠点名を自動設定）
9. `member_cards.last_checkin_at` / `checkin_count` をアトミック更新
10. `auto_complete_reservation = TRUE`（拠点設定）の場合: 当日の `CONFIRMED` 予約を検索 → `COMPLETED` に遷移
11. ADMIN に `MEMBER_CHECKED_IN` 通知を送信（スタッフスキャン型と同じ通知タイプ。`checkin_type = 'SELF'` で区別可能）

## 5. ビジネスロジック

### 主要フロー

#### 会員証自動発行フロー
```
1. ユーザーがチーム/組織に加入（TeamMemberJoinedEvent / OrganizationMemberJoinedEvent）
2. member_cards を自動 INSERT:
   a. card_code = UUID v4 を生成
   b. card_number = スコープ内の最大連番 +1（例: M-000013）
   c. display_name = users.display_name をコピー
   d. qr_secret = SecureRandom で64文字のランダム文字列を生成
   e. status = ACTIVE
3. PLATFORM スコープの会員証はユーザー登録時（UserCreatedEvent）に自動発行
```

#### QRコード表示フロー
```
1. メンバーが会員証画面を開く（GET /member-cards/{id}/qr）
2. サーバーが qr_token を生成:
   a. payload = card_code + "." + issued_at_epoch + "." + expires_at_epoch
   b. signature = HMAC-SHA256(payload, qr_secret)
   c. qr_token = payload + "." + Base64URL(signature)
3. フロントエンドが qr_token を QR コード画像に変換（qrcode.js）
4. 画面に QR コード + 会員番号 + 名前を表示
5. 有効期限（5分）経過後、フロントエンドが自動で再取得（setInterval）
```

#### QRスキャン認証フロー
```
1. ADMIN/スタッフがスキャン画面を開く（カメラ起動）
2. メンバーの QR コードをスキャン → qr_token を取得
3. POST /member-cards/verify に送信
4. サーバーが認証処理:
   a. トークン分割 → 期限チェック → DB 検索 → 署名検証 → ステータスチェック
   b. 認証成功: member_card_checkins に INSERT
   c. 当日の CONFIRMED 予約を自動検索 → 該当あれば COMPLETED に遷移
   d. denormalize 更新（last_checkin_at, checkin_count）
5. 認証結果を画面に表示（会員情報 + 予約情報）
```

#### セルフチェックインフロー
```
1. 会員がスマホで店舗設置QRをスキャン
2. フロントエンドが location_qr_token を取得
3. POST /member-cards/self-checkin に送信（ログイン中のユーザーとして自動認証）
4. サーバーが処理:
   a. 拠点QR検証（signature チェック → is_active チェック）
   b. ログインユーザーの member_cards を当該スコープで検索
   c. カードステータス検証（ACTIVE か）
   d. 二重スキャン防止（5分以内）
   e. member_card_checkins に INSERT（checkin_type = 'SELF'）
   f. denormalize 更新（last_checkin_at, checkin_count）
   g. 予約自動紐付け（auto_complete_reservation 設定に依存）
5. 結果画面を表示（チェックイン成功 + 予約情報）
```

#### チーム脱退時の自動無効化
```
1. TeamMemberLeftEvent / OrganizationMemberLeftEvent を購読
2. 該当スコープの member_cards.status = REVOKED、revoked_at = NOW()
3. メンバーに「会員証が無効化されました」通知を送信
```

### 来店ポイント / スタンプカード

チーム設定の `reward_checkin_threshold`（デフォルト: 0 = 無効。例: 10）でスタンプカード機能を有効化。

```
1. チェックイン成功時に checkin_count を確認
2. checkin_count % reward_checkin_threshold == 0 の場合:
   a. メンバーに MEMBER_REWARD_EARNED 通知を送信
     （「スタンプカードが完了しました！スタッフにお声がけください。」）
   b. audit_logs にリワード達成を記録
3. フロントエンド:
   a. 会員証画面にスタンプカード UI を表示
     （checkin_count % threshold で現在のスタンプ数を計算）
   b. 完了時: 🎉 アニメーション + 「スタッフに見せてください」メッセージ
```

- リワードの内容（割引・無料サービス等）はプラットフォームでは管理しない（店舗運用に委ねる）
- `reward_checkin_threshold = 0` でスタンプカード UI を非表示

### チェックインのトリガーアクション

`MemberCheckedInEvent`（ApplicationEvent）を発行し、他機能がリスナーで購読する拡張ポイント:

| リスナー | トリガーアクション | Phase |
|---------|-----------------|-------|
| `ReservationService` | 当日の CONFIRMED 予約を自動 COMPLETED（実装済み） | Phase 3 |
| `ServiceRecordService` | チェックイン時にサービス記録のドラフトを自動作成。ADMIN 画面に「サービス記録を追加しますか？」プロンプト表示 | Phase 7 |
| `ChartRecordService` | 前回のカルテ情報をスタッフ画面にポップアップ表示（整骨院・美容室で施術前に前回の経過を確認） | Phase 7 |
| `TransactionService` | チェックイン時に会計セッションを開始（将来のレジ/POS 連携。後述） | Phase 8+ |

### 重要な判定ロジック

- **二重スキャン防止**: 同一 `member_card_id` の直前の `member_card_checkins.checked_in_at` から `checkin_cooldown_minutes`（チーム設定。デフォルト: 5、範囲: 1〜60）分以内の再スキャンは 409 エラー。判定: `SELECT checked_in_at FROM member_card_checkins WHERE member_card_id = ? ORDER BY checked_in_at DESC LIMIT 1`
- **`checkin_count` / `last_checkin_at` のアトミック更新**: `UPDATE member_cards SET checkin_count = checkin_count + 1, last_checkin_at = NOW() WHERE id = :cardId` の1 SQL で実行
- **予約自動紐付け**: スキャン時に `reservations` テーブルから `user_id + team_id + slot_date = TODAY + status = CONFIRMED` の予約を検索。複数該当する場合は `start_time` が最も近い予約を紐付け
- **QR 有効期限**: トークン自体に有効期限を埋め込み、5分間のみ有効。スクリーンショットの転送による不正利用を時間制限で緩和
- **会員番号の採番**: `SELECT MAX(CAST(SUBSTRING_INDEX(card_number, '-', -1) AS UNSIGNED)) FROM member_cards WHERE scope_type = ? AND scope_id = ? FOR UPDATE` で最大番号を取得し +1。`SUBSTRING_INDEX(card_number, '-', -1)` でハイフン以降の数値部分を取得するため、プレフィックスの長さに依存しない（`M-000001` でも `GYM-000001` でも対応可能）。`FOR UPDATE` で同一スコープの同時採番時のレースコンディションを防止。論理削除されたカードの番号は再利用しない

### F04.3 通知基盤との連携

| 通知タイプ | トリガー | 送信先 |
|-----------|---------|--------|
| `MEMBER_CARD_ISSUED` | 会員証自動発行時 | 対象メンバー |
| `MEMBER_CARD_SUSPENDED` | 一時停止時 | 対象メンバー |
| `MEMBER_CARD_REACTIVATED` | 一時停止解除時 | 対象メンバー |

## 6. セキュリティ考慮事項

- **QR トークンの署名検証**: HMAC-SHA256 でトークンの完全性を検証。`qr_secret` はカードごとに異なるため、1枚のシークレット漏洩が他カードに影響しない
- **QR トークンの時間制限**: 有効期限5分。スクリーンショットの転送による他人利用を時間的に制限。完全な防止は不可能だが、リスクを大幅に軽減
- **qr_secret の再生成**: 紛失・漏洩が疑われる場合に `POST /regenerate` で `qr_secret` をリセット。旧 QR は即座に無効化
- **セルフチェックインのセキュリティ**: 店舗QRは長期有効のため、QR画像が外部流出した場合に非メンバーがスキャンする可能性がある。ただしサーバー側でログインユーザーのメンバーシップを検証するため、非メンバーはチェックイン不可。代理チェックイン（他人のスマホで自分のアカウントにログインしてスキャン）は認証情報の共有であり、プラットフォーム利用規約で禁止する運用対応
- **レートリミット**: verify / self-checkin API は `Bucket4j` で1デバイス（IP）あたり1分間30回まで。ブルートフォース攻撃防止
- **`qr_secret` の保管**: DB に平文で保存。DB アクセス権限の適切な管理で保護する。`qr_secret` は API レスポンスに一切含めない（DB カラム → Java Entity のマッピングはするが、DTO 変換時に除外）。将来的に AES-256 暗号化（Phase 10-11 セキュリティ監査時）を検討
- **IDOR 防止**: `GET /member-cards/{id}/qr` は本人のみ。他人の QR トークンは取得不可
- **スキャン時のスコープ検証**: `POST /verify` は、スキャンした ADMIN が当該カードのスコープ（チーム/組織）に所属していることを必ず検証。他チームの会員証を不正スキャンできないようにする
- **入力サニタイズ**: `location`・`note` は XSS 対策として HTML タグを除去
- **監査ログ**: 会員証の発行・一時停止・無効化・再生成・QR スキャン認証（成功/失敗）・セルフチェックイン（成功/失敗）・拠点の作成/削除を全て `audit_logs` に記録。特にスキャン失敗（`INVALID_SIGNATURE`）やセルフチェックイン失敗（`NOT_MEMBER`）は不正利用の兆候として監視対象
- **チェックイン履歴のアーカイブ**: `checked_in_at` が12ヶ月以上前の `member_card_checkins` は `member_card_checkins_archive` に移行可能。Phase 10 のアーカイブバッチで対応。統計 API は必要に応じてアーカイブテーブルも UNION で参照

### 解決済み事項

| 事項 | 決定 | 理由 |
|------|------|------|
| QR コード画像の保存方式 | 画像は保存しない。`card_code` からフロントエンドで動的生成 | ストレージ不要。署名付きトークンの方がセキュリティが高い |
| QR の有効期限 | 5分間（トークン内に埋め込み） | スクリーンショット転送対策。短すぎると UX 悪化、長すぎるとセキュリティリスク |

| オフラインスキャン | Phase 2 スコープ外 | オフライン時はサーバー検証不可。将来 JWT 方式（公開鍵検証）を検討 |
