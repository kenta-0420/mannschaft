# F01.3: テンプレート・モジュール管理

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 1
> **最終更新**: 2026-03-13

---

## 1. 概要

業種別テンプレート（スポーツ / 整骨院 / 学校 / 会社 / 飲食店 / 美容室 / ジム / サークル / 町内会 / マンション 等）と、モジュール式機能構成を管理するシステム基盤。チーム作成時にテンプレートを選択すると推奨モジュールが自動有効化され、管理者が後からON/OFF変更できる。

機能は**デフォルト機能**（28個・全チーム常時有効）と**選択式モジュール**（カタログから選択・11個）の2種類に分かれる。選択式モジュールは10個まで無料、11個目以降は有料プランが必要。SYSTEM_ADMINがテンプレート・モジュール定義・レベル別適用設定を一元管理し、運営側が随時新モジュールを追加できる。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | テンプレート CRUD、モジュール定義 CRUD、レベル別適用設定、テンプレート×モジュール紐付け管理 |
| ADMIN | チームのテンプレート選択、選択式モジュールの有効化/無効化、モジュール一覧閲覧 |
| DEPUTY_ADMIN | `MANAGE_MODULES` 権限を持つ場合: モジュール一覧閲覧、有効化/無効化 |
| MEMBER | 有効化済みモジュールの利用のみ（管理操作不可） |
| SUPPORTER | 有効化済みモジュールの利用のみ（管理操作不可） |
| GUEST | 有効化済みモジュールのうち GUEST 閲覧可のもののみ |

### 対象レベル
- [x] 組織 (Organization) — 組織レベルのモジュール適用
- [x] チーム (Team) — テンプレート選択・モジュール有効化/無効化
- [x] 個人 (Personal) — 個人レベルのモジュール適用

### テーブル定義

#### `team_templates`

SYSTEM_ADMIN が管理する業種別テンプレート。チーム作成時にユーザーが選択し、推奨モジュールが自動有効化される。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `name` | VARCHAR(100) | NO | — | テンプレート名（例: スポーツチーム、美容室、マンション管理組合）|
| `slug` | VARCHAR(100) | NO | — | URL/API 識別子（例: `sports`, `salon`, `condominium`）|
| `description` | TEXT | YES | NULL | テンプレートの説明文（選択画面で表示）|
| `icon_url` | VARCHAR(500) | YES | NULL | アイコン画像の S3 キー |
| `category` | VARCHAR(50) | YES | NULL | カテゴリ（例: ビジネス、教育、コミュニティ、不動産）|
| `sort_order` | INT | NO | 0 | 選択画面での表示順 |
| `is_active` | BOOLEAN | NO | TRUE | FALSE = 選択画面に表示しない（既存チームは影響なし）|
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（SYSTEM_ADMIN; SET NULL on delete）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
UNIQUE KEY uq_tt_slug (slug)                    -- slug は一意
INDEX idx_tt_category_sort (category, sort_order) -- カテゴリ別の一覧取得用
INDEX idx_tt_active_sort (is_active, sort_order)  -- アクティブテンプレートの一覧取得用
```

**制約・備考**
- 論理削除: `deleted_at DATETIME nullable`（SoftDeletableEntity 適用）。削除しても既にテンプレートを選択済みのチームには影響しない
- `slug` は英数字・ハイフンのみ許容（アプリ層で正規表現バリデーション: `^[a-z0-9-]+$`）
- テンプレートは SYSTEM_ADMIN のみ作成・編集・削除可能
- 「その他」テンプレート（slug: `custom`）をデフォルトで用意し、推奨モジュールなし（ユーザーが自由に選択）

#### `template_modules`

テンプレートに紐付く推奨選択式モジュール。テンプレート選択時に自動有効化される。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `template_id` | BIGINT UNSIGNED | NO | — | FK → team_templates（ON DELETE CASCADE）|
| `module_id` | BIGINT UNSIGNED | NO | — | FK → module_definitions（ON DELETE RESTRICT）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_tm_template_module (template_id, module_id)   -- テンプレート×モジュールで一意
INDEX idx_tm_module (module_id)                               -- モジュール逆引き用
```

**制約・備考**
- デフォルト機能（28個）はテンプレートに関係なく全チーム常時有効のため、この中間テーブルには含めない。選択式モジュールのみが対象
- `template_id` ON DELETE CASCADE: テンプレート物理削除時にカスケード削除（テンプレート側の論理削除時は影響しない）
- `module_id` ON DELETE RESTRICT: モジュール定義は物理削除しない（論理削除で対応）。RESTRICT により `team_enabled_modules` と一貫した保護方針

---

#### `template_fields`

業種テンプレートに含まれるカスタム項目の定義。テンプレートレベルの設定として運営が業種別プリセットとして管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `template_id` | BIGINT UNSIGNED | NO | — | FK → team_templates（ON DELETE CASCADE）|
| `module_id` | BIGINT UNSIGNED | YES | NULL | FK → module_definitions（ON DELETE SET NULL; NULL = テンプレート共通項目）|
| `field_name` | VARCHAR(100) | NO | — | フィールド名（例: 背番号、担当コース）|
| `field_type` | ENUM('TEXT', 'NUMBER', 'DATE', 'SELECT', 'MULTISELECT', 'CHECKBOX') | NO | — | フィールド型 |
| `description` | VARCHAR(500) | YES | NULL | フィールドの説明 |
| `options` | JSON | YES | NULL | SELECT / MULTISELECT 型の選択肢リスト |
| `is_required` | BOOLEAN | NO | FALSE | 必須フィールドか |
| `sort_order` | INT | NO | 0 | 表示順 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_tf_template_sort (template_id, sort_order)          -- テンプレート内のフィールド一覧用
INDEX idx_tf_template_module (template_id, module_id)         -- モジュール別フィールド取得用
```

**制約・備考**
- テンプレート選択時に、各モジュール固有のカスタムフィールド初期値としてチーム側にコピーされる（テンプレートの変更は既存チームに遡及しない）
- `module_id = NULL` のレコードはテンプレート共通のカスタム項目（モジュールに依存しない）
- `module_id` ON DELETE SET NULL: モジュール定義は物理削除しない運用（RESTRICT が他テーブルで先にブロック）だが、万が一の場合はテンプレート共通項目に降格（nullable FK の自然な動作）
- SYSTEM_ADMIN のみ管理可能

#### `module_definitions`

デフォルト機能（28個）と選択式モジュール（11個）の統合マスター定義。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `name` | VARCHAR(100) | NO | — | モジュール名（例: ダッシュボード、QR会員証）|
| `slug` | VARCHAR(100) | NO | — | システム識別子（例: `dashboard`, `qr-member-card`）|
| `description` | TEXT | YES | NULL | モジュールの説明文 |
| `icon_url` | VARCHAR(500) | YES | NULL | アイコン画像の S3 キー |
| `module_type` | ENUM('DEFAULT', 'OPTIONAL') | NO | — | `DEFAULT` = デフォルト機能、`OPTIONAL` = 選択式モジュール |
| `module_number` | INT | NO | — | モジュール番号（デフォルト: 1〜28、選択式: 1〜11。module_type 内で一意）|
| `requires_paid_plan` | BOOLEAN | NO | FALSE | TRUE = 有料プラン必須（例: カルテ = 選択式11個目） |
| `feature_flag` | VARCHAR(100) | YES | NULL | フィーチャーフラグ名（例: `FEATURE_V9_ENABLED`。NULL = フラグなし = 常時有効）|
| `onboarding_url` | VARCHAR(500) | YES | NULL | 初回有効化時に表示するガイドページの URL |
| `trial_days` | INT | YES | NULL | トライアル日数（NULL = トライアル不可）。`requires_paid_plan = true` のモジュールに設定 |
| `sort_order` | INT | NO | 0 | カタログ表示順 |
| `is_active` | BOOLEAN | NO | TRUE | FALSE = カタログに表示しない |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
UNIQUE KEY uq_md_slug (slug)                              -- slug は一意
UNIQUE KEY uq_md_type_number (module_type, module_number) -- 種別内でモジュール番号は一意
INDEX idx_md_type_sort (module_type, sort_order)          -- 種別別の一覧取得用
INDEX idx_md_type_active (module_type, is_active)         -- アクティブモジュールのフィルタ用
```

**制約・備考**
- 論理削除: `deleted_at DATETIME nullable`（SoftDeletableEntity 適用）
- `slug` は英数字・ハイフンのみ許容（アプリ層で正規表現バリデーション: `^[a-z0-9-]+$`）
- `slug` の UNIQUE 制約は論理削除レコードを含む。論理削除済みの slug を再利用する場合は物理削除後に新規作成するか、slug を変更する運用（`team_templates.slug` も同様）
- `module_type = 'DEFAULT'` のモジュールはチームの有効化/無効化の対象外（常時有効）
- `feature_flag`: 広告表示・コルクボード等、特定フェーズまで非表示にする機能のフラグ名を格納。アプリ起動時に環境変数と照合し、`false` のモジュールは API・UI で完全非表示
- 初期データは Flyway のシードマイグレーションで投入（デフォルト28個 + 選択式11個 = 計39レコード）

#### `module_field_definitions`

各モジュールが提供する汎用フィールドの定義。モジュール固有の入力項目スキーマ。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `module_id` | BIGINT UNSIGNED | NO | — | FK → module_definitions（ON DELETE CASCADE）|
| `field_name` | VARCHAR(100) | NO | — | フィールド名 |
| `field_type` | ENUM('TEXT', 'NUMBER', 'DATE', 'SELECT', 'MULTISELECT', 'CHECKBOX') | NO | — | フィールド型 |
| `description` | VARCHAR(500) | YES | NULL | フィールドの説明 |
| `options` | JSON | YES | NULL | SELECT / MULTISELECT 型の選択肢リスト |
| `is_required` | BOOLEAN | NO | FALSE | 必須フィールドか |
| `sort_order` | INT | NO | 0 | 表示順 |
| `version` | INT | NO | 1 | フィールド定義のバージョン番号（SYSTEM_ADMIN がフィールドを更新するたびにインクリメント） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_mfd_module_sort (module_id, sort_order)   -- モジュール内のフィールド一覧用
```

**制約・備考**
- SYSTEM_ADMIN のみ管理可能
- モジュール有効化時に、そのモジュールのフィールド定義がチーム側のカスタムフィールドに初期値としてコピーされる（スキーマのテンプレート的役割）。コピー時に `source_version`（コピー元の `version` 値）をチーム側フィールドに記録
- モジュール定義側のフィールド変更は既存チームに遡及しない。ただし `version` が進んでいる場合、チーム ADMIN のモジュール管理画面に「新しいフィールド定義が利用可能です」と通知し、任意で最新定義を適用可能（既存データへの影響を ADMIN が確認して判断）

#### `module_level_availability`

モジュールを組織・チーム・個人のどのレベルで利用可能にするかを制御する。SYSTEM_ADMIN がシステム管理画面から ON/OFF を設定。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `module_id` | BIGINT UNSIGNED | NO | — | FK → module_definitions（ON DELETE CASCADE）|
| `level` | ENUM('ORGANIZATION', 'TEAM', 'PERSONAL') | NO | — | 適用レベル |
| `is_available` | BOOLEAN | NO | TRUE | このレベルで利用可能か |
| `note` | VARCHAR(500) | YES | NULL | 備考（例: 「チーム: 枠管理 / 個人: 予約のみ」）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_mla_module_level (module_id, level)   -- モジュール×レベルで一意
INDEX idx_mla_level (level, is_available)            -- レベル別の有効モジュール取得用
```

**制約・備考**
- 各モジュールにつき最大3レコード（ORGANIZATION / TEAM / PERSONAL）
- `is_available = false` の場合、そのレベルでモジュールの API・UI を非表示にする
- 初期データは Flyway のシードマイグレーションで投入（README のレベル別適用テーブルに基づく）
- SYSTEM_ADMIN のみ変更可能。変更は即時反映（Caffeine インメモリキャッシュを ApplicationEvent で全インスタンス無効化。レベル情報は Redis `team:modules:{teamId}` の slug リストには含まれないため Redis 無効化は不要）

#### `team_enabled_modules`

チームが有効化した選択式モジュールの状態を管理する。デフォルト機能はこのテーブルに含めない（常時有効のため）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams（ON DELETE CASCADE）|
| `module_id` | BIGINT UNSIGNED | NO | — | FK → module_definitions（ON DELETE RESTRICT）|
| `is_enabled` | BOOLEAN | NO | TRUE | 有効/無効 |
| `enabled_at` | DATETIME | NO | CURRENT_TIMESTAMP | 最初に有効化した日時（再有効化時も更新しない。直近の操作日時は `updated_at` で判定）|
| `disabled_at` | DATETIME | YES | NULL | 無効化した日時（再有効化で NULL に戻す）|
| `enabled_by` | BIGINT UNSIGNED | YES | NULL | FK → users（有効化した ADMIN; SET NULL on delete）|
| `trial_expires_at` | DATETIME | YES | NULL | トライアル有効期限。NULL = トライアルなし（通常有効化 or 有料プラン契約済み）|
| `trial_used` | BOOLEAN | NO | FALSE | TRUE = 過去にトライアルを利用済み（再トライアル防止用） |
| `scheduled_change_at` | DATETIME | YES | NULL | 予約された有効化/無効化の実行日時 |
| `scheduled_enabled` | BOOLEAN | YES | NULL | 予約された変更後の状態（TRUE = 有効化予約、FALSE = 無効化予約）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_tem_team_module (team_id, module_id)   -- チーム×モジュールで一意
INDEX idx_tem_team_enabled (team_id, is_enabled)      -- チームの有効モジュール一覧用
INDEX idx_tem_module (module_id)                       -- モジュール利用チーム数集計用
INDEX idx_tem_trial_expires (trial_expires_at)         -- トライアル期限切れバッチ用
INDEX idx_tem_scheduled (scheduled_change_at)          -- 予約変更バッチ用
```

**制約・備考**
- テンプレート選択時に `template_modules` の推奨モジュールが一括 INSERT される
- ADMIN が後から有効化/無効化を切り替え可能
- 無効化しても既存データ（サービス履歴、備品等）は削除しない。モジュールの API・UI を非表示にするのみ
- **フリープラン制限**: `is_enabled = true` のレコード数が10を超える場合、11個目以降の有効化には有料プランが必要（アプリ層で `team_subscriptions` と照合して検証）
- **Redis キャッシュ**: `team:modules:{teamId}` に DEFAULT + OPTIONAL の「実効有効モジュール」slug リストをキャッシュ（TTL: 1時間）。有効化/無効化時にキャッシュを即時無効化
- `module_id` ON DELETE RESTRICT: モジュール定義は物理削除しない（論理削除で対応）
- **トライアル**: `requires_paid_plan = true` のモジュールを有料プラン未契約でも期限付きで試用可能。1モジュールにつき1チーム1回のみ（`trial_used` で制御）。トライアル日数は SYSTEM_ADMIN が `module_definitions` 側で設定。期限切れは日次バッチで `is_enabled = false` に変更
- **スケジューリング**: `scheduled_change_at` + `scheduled_enabled` で予約された変更を日次バッチで処理。処理後にフィールドを NULL クリア。モジュール管理画面に「予約済み変更」バッジ表示

#### `module_recommendations`

モジュール間の推奨関係を定義する。「このモジュールを有効化するなら一緒に使うと便利」なソフト推奨。SYSTEM_ADMIN が管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `module_id` | BIGINT UNSIGNED | NO | — | FK → module_definitions（ON DELETE CASCADE）推奨元モジュール |
| `recommended_module_id` | BIGINT UNSIGNED | NO | — | FK → module_definitions（ON DELETE CASCADE）推奨先モジュール |
| `reason` | VARCHAR(200) | YES | NULL | 推奨理由（例: 「施術記録と組み合わせて顧客管理を強化」）|
| `sort_order` | INT | NO | 0 | 推奨表示順 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_mr_module_rec (module_id, recommended_module_id)  -- 同一ペアの重複防止
INDEX idx_mr_recommended (recommended_module_id)                 -- 逆引き用
```

**制約・備考**
- 推奨は一方向（A→B は「A を有効化する際に B を推奨」。B→A の逆方向が必要なら別レコードで登録）
- 強制ではなくサジェスト。ユーザーは無視可能
- SYSTEM_ADMIN のみ管理可能。モジュール定義の作成・更新 API で同時に設定

---

#### `team_module_snapshots`

モジュール設定変更前のスナップショットを自動保存する。ADMIN がロールバック操作で直前の状態に戻せる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams（ON DELETE CASCADE）|
| `snapshot_data` | JSON | NO | — | 変更前の有効モジュール ID リスト + 各モジュールの状態 |
| `trigger_action` | VARCHAR(50) | NO | — | スナップショットのトリガー（`TOGGLE` / `BULK_UPDATE` / `TEMPLATE_CHANGE` / `SCHEDULED` / `TRIAL_EXPIRED`）|
| `triggered_by` | BIGINT UNSIGNED | YES | NULL | FK → users（SET NULL on delete）。バッチ処理の場合は NULL |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_tms_team_created (team_id, created_at DESC)   -- チームの最新スナップショット取得用
```

**制約・備考**
- モジュール有効化/無効化・一括更新・テンプレート変更・スケジュール実行・トライアル期限切れの各操作前に自動保存
- チームごとに直近5件を保持。6件目以降は最古のスナップショットを物理削除（バッチまたは INSERT 時）
- `snapshot_data` の JSON 構造: `{"modules": [{"module_id": 30, "is_enabled": true, "trial_expires_at": null}, ...]}`
- ロールバック実行時は snapshot_data をもとに `team_enabled_modules` を一括 UPDATE し、Redis キャッシュを無効化

### 他テーブルとの依存関係

- **`teams.template_id`**: `teams` テーブルに `template_id BIGINT UNSIGNED NULL FK → team_templates（SET NULL on delete）` カラムが必要。テンプレート変更フロー（`PUT /teams/{teamId}/template`）で更新される。NULL = テンプレート未選択（「その他」テンプレートで作成後にテンプレートが論理削除された場合等）。このカラムの追加は `teams` テーブルの Flyway マイグレーション（Phase 1 前半）に含める
- **`team_subscriptions`**: フリープラン制限・有料プラン必須チェックで参照。プラン・サブスクリプション（Phase 8）のテーブルが未作成の Phase 1〜7 では、アプリ層でフリープラン（10個上限）として扱う

## 5. ビジネスロジック

### 主要フロー

#### チーム作成時のテンプレート適用
```
1. ユーザーがチーム作成画面でテンプレートを選択（省略時は「その他」テンプレート）
2. チーム作成 API がチームレコードを INSERT
3. テンプレートの推奨モジュール（template_modules）を取得
4. 推奨モジュールを team_enabled_modules に一括 INSERT（is_enabled = true）
5. テンプレートのカスタムフィールド（template_fields）をチーム固有のフィールド定義にコピー
6. Redis キャッシュ team:modules:{teamId} に有効モジュール ID リストをセット
7. ApplicationEvent（TeamCreatedEvent）を発行
```

#### モジュール有効化/無効化
```
1. ADMIN がモジュール管理画面でトグルを操作
2. バックエンドが以下を検証:
   a. デフォルト機能でないこと（DEFAULT は変更不可）
   b. module_level_availability でチームレベルが available であること
   c. 有効化の場合: フリープラン制限チェック（有効モジュール数 ≤ 10 or 有料プラン契約済み）
   d. 有効化の場合: requires_paid_plan = true なら team_subscriptions で有料プラン確認
   e. 有効化の場合: feature_flag が設定されている場合、環境変数で有効かチェック
3. team_enabled_modules を UPSERT（INSERT ON DUPLICATE KEY UPDATE）
4. Redis キャッシュ team:modules:{teamId} を無効化
5. ApplicationEvent（ModuleToggledEvent）を発行
6. 有効化時:
   a. そのモジュールのフィールド定義がチーム側に未コピーなら初期値としてコピー
   b. module_recommendations を参照し、未有効化の推奨モジュールがあればレスポンスに含める
      （「サービス履歴と一緒にカルテも有効化すると便利です」等のサジェスト）
   c. module_definitions.onboarding_url が設定されている場合、レスポンスに含める
      （フロントエンドがガイドページへのリンクを表示）
7. 無効化時: 既存データは保持。API・UI を非表示にするのみ
```

#### テンプレート変更
```
1. ADMIN がチーム設定画面でテンプレートを変更
2. バックエンドがテンプレートの推奨モジュールを取得
3. 既存の有効モジュールは維持したまま、新テンプレートの推奨モジュールを追加有効化
   （既に有効なモジュールはスキップ）
4. フリープラン制限チェック:
   - 制限内: 全推奨モジュールを有効化
   - 制限超過: 制限内のモジュールのみ有効化し、レスポンスに警告を含める
5. Redis キャッシュを無効化
6. チームの template_id を更新
```

#### モジュール有効化チェック（各機能 API の入り口）
```
1. Controller メソッドに @RequiresModule(slug = "reservation", level = ModuleLevel.TEAM) を付与
2. AOP が以下の 3 層チェックを実行:
   a. [Redis] team:modules:{teamId} からチームの「実効有効モジュール」slug リストを取得
      （キャッシュミス時は以下のクエリで取得し Redis にセット）
      ※ キャッシュ内容 = DEFAULT（is_active=true, feature_flag有効, deleted_at IS NULL）
        + OPTIONAL（上記に加え team_enabled_modules.is_enabled=true）の slug リスト
      ※ DEFAULT モジュールはチーム横断で同一だが、OPTIONAL と合わせて1回の Redis GET で
        完結させるため、チームキーに含める（JOIN 1回で構築可能）
   b. 対象 slug がリストに含まれない場合: 403 Forbidden
   c. [インメモリ] module_level_availability を Spring @Cacheable（Caffeine, TTL: 10分）で取得
      アノテーションの level と照合し、is_available = false なら 403 Forbidden
      （マスターデータのため Redis ではなくローカルキャッシュで十分。
       SYSTEM_ADMIN 変更時は ApplicationEvent で全インスタンスのキャッシュを無効化）
3. 全チェック通過後、Controller メソッドを実行
```

> **キャッシュ 2 層戦略**:
> - **Redis `team:modules:{teamId}`**: DEFAULT + チーム固有 OPTIONAL の「実効有効モジュール」slug リスト。`is_active` / `feature_flag` / `is_enabled` を考慮済み
> - **Caffeine（インメモリ）**: `module_level_availability` マスターデータ（TTL: 10分）
> - **無効化ルール**:
>   - `is_active` / `feature_flag` 変更 → **Redis `team:modules:*` 一括無効化**（slug リストの再計算が必要）
>   - `module_level_availability` 変更 → **Caffeine のみ無効化**（ApplicationEvent で全インスタンスに伝播。Redis の slug リストはレベル情報を含まないため無効化不要）
>   - チーム個別のモジュール有効化/無効化 → **Redis `team:modules:{teamId}` のみ無効化**

#### SYSTEM_ADMIN によるモジュール追加
```
1. SYSTEM_ADMIN が新モジュールを module_definitions に作成
2. module_level_availability に各レベルの適用設定を INSERT
3. 必要に応じて module_field_definitions にフィールド定義を追加
4. 既存テンプレートに推奨モジュールとして紐付け（template_modules）
5. 既存チームへの影響:
   - 自動有効化はしない（チーム ADMIN が明示的に有効化する運用）
   - ダッシュボードの「新しいモジュールが利用可能です」通知で周知
```

#### SYSTEM_ADMIN によるモジュール非アクティブ化
```
1. SYSTEM_ADMIN が module_definitions.is_active を false に変更
2. バックエンドが以下を実行:
   a. 該当モジュールを有効化中のチーム数を取得し、確認ダイアログで表示
      「このモジュールは現在 N チームで有効化されています。非アクティブ化しますか？」
   b. SYSTEM_ADMIN が確認後、is_active = false に更新
   c. Redis キャッシュ team:modules:* を一括無効化
3. 既存チームへの影響:
   - team_enabled_modules のレコードは変更しない（is_enabled = true のまま）
   - Redis キャッシュ再構築時に is_active = false のモジュールが slug リストから除外される
     → @RequiresModule の AOP が slug 不在で 403 を返し、API・UI が自動的に非表示になる
   - チーム ADMIN のモジュール管理画面には「運営により一時停止中」ラベルを表示
     （availability_status = SUSPENDED_BY_ADMIN）
4. モジュールを再アクティブ化（is_active = true）すると、
   Redis キャッシュ再構築で slug が復帰し、team_enabled_modules が保持されているため
   即座に元の状態に復帰
```

#### SYSTEM_ADMIN によるレベル別適用変更
```
1. SYSTEM_ADMIN が module_level_availability の is_available を変更
   （例: 予約管理の PERSONAL レベルを true → false）
2. バックエンドが以下を実行:
   a. Caffeine インメモリキャッシュを無効化（ApplicationEvent で全インスタンスに伝播）
      ※ Redis の team:modules:{teamId} はレベル情報を含まないため無効化不要
3. 既存チームへの影響:
   - team_enabled_modules のレコードは変更しない（is_enabled = true のまま）
   - @RequiresModule の AOP 内で module_level_availability を検証し、
     該当レベルの API を 403 Forbidden で拒否
   - チーム ADMIN のモジュール管理画面には
     「このモジュールは現在このレベルでは利用できません（運営設定）」ラベルを表示
4. レベル復活（is_available = true に戻す）時、
   team_enabled_modules が保持されているため即座に元の状態に復帰
```

#### 「その他」テンプレートからの推奨テンプレート提案
```

```

```

#### ロールバック実行
```
1. ADMIN が「モジュール設定を元に戻す」を選択
2. POST /teams/{teamId}/modules/rollback で snapshot_id を送信
3. バックエンドが以下を検証:

   b. snapshot_data から各モジュールの is_enabled を復元:

### 重要な判定ロジック
- **デフォルト機能の常時有効**: `module_type = 'DEFAULT'` のモジュールは `team_enabled_modules` に関係なく常に有効。Redis キャッシュ `team:modules:{teamId}` にも DEFAULT の slug を含めて格納するため、`@RequiresModule` AOP は OPTIONAL と同一フローで検証可能（DEFAULT 専用の分岐不要）
- **フリープラン制限**: `team_enabled_modules` で `is_enabled = true` かつ `module_definitions.module_type = 'OPTIONAL'` のレコード数をカウント。10以下なら無料、11以上は `team_subscriptions` で有料プラン確認
- **有料プラン必須モジュール**: `requires_paid_plan = true`（例: カルテ）はフリープラン制限とは別に、単体で有料プラン契約が必要
- **フィーチャーフラグ制御**: `feature_flag` が設定されているモジュールは、対応する環境変数が `true` でない限り API・UI で非表示。`FEATURE_V9_ENABLED=false` の場合、広告表示・コルクボードは完全非表示
- **モジュール無効化時のデータ保持**: モジュールを無効化してもデータは削除しない。再有効化すれば以前のデータがそのまま利用可能
- **レベル別適用**: モジュール有効化チェック時に `module_level_availability` のレベルも検証する。例: 予約管理がチームレベルで有効でも、個人レベルで `is_available = false` なら個人向け API は 403。レベル変更時も `team_enabled_modules` は変更しない（ソフトブロック方式。`is_active = false` の非アクティブ化フローと同一パターン）
- **テンプレート変更の非破壊性**: テンプレート変更は追加のみ（既存有効モジュールは無効化しない）。不要なモジュールは ADMIN が手動で無効化する
- **組織レベルのモジュール管理**: 組織 ADMIN はチームのモジュール有効化/無効化を制御しない（チーム個別管理のみ）。組織→チームへの設定カスケードは行わない。チームが複数組織に所属する多対多構造のため、競合回避とチーム ADMIN の自律性を優先
- **ダッシュボードウィジェットとモジュールの連動**: `dashboard_widget_settings` はモジュール無効化時に書き換えない。フロントエンド側の `WIDGET_MODULE_MAP` でフィルタリングし、モジュール再有効化時にウィジェット設定が自動復活する設計。無効モジュールのウィジェットはグレーアウト（完全非表示ではなく disabled）
- **モジュール推奨のサジェスト**: 有効化時に `module_recommendations` を参照し、未有効化の推奨モジュールをレスポンスに含める。強制ではなくサジェスト（ユーザーが dismiss 可能）。推奨理由テキストを表示し、なぜ便利なのかを明示
- **オンボーディングガイド**: `module_definitions.onboarding_url` が設定されているモジュールは、初回有効化時にガイドページへのリンクをレスポンスに含める。2回目以降の再有効化では省略（`team_enabled_modules.created_at` と `enabled_at` の一致で初回判定）
- **フィールド定義バージョン管理**: `module_field_definitions.version` と チーム側の `source_version` を比較し、最新定義が利用可能な場合に通知。適用は ADMIN の任意操作（強制アップデートなし）。適用時は差分プレビューを表示し、既存データへの影響がないことを確認してから実行
- **テンプレート利用レコメンド**: 同テンプレート利用チームの傾向に基づく推奨。プライバシー保護のため集計値（割合）のみ。母数 10 チーム未満では非表示。日次バッチで集計し Redis にキャッシュ
- **「その他」テンプレートからの提案**: Jaccard 類似度 ≥ 0.5 のテンプレートを提案。初回のみ通知、dismiss で再表示しない
- **トライアル制御**: `requires_paid_plan = true` かつ `trial_days IS NOT NULL` のモジュールのみトライアル可能。1チーム1モジュールにつき1回限り（`trial_used = true` で再トライアル防止）。トライアル期間中は通常有効化と同等のアクセス権限。`trial_expires_at <= NOW()` のレコードは日次バッチで `is_enabled = false` に変更（即時ではなくバッチ処理。最大24時間の猶予）
- **スケジューリング制約**: 1モジュール1チームにつき予約は1件のみ（`scheduled_change_at` / `scheduled_enabled` が NULL でない場合は上書き）。予約日時は現在時刻より24時間以上先であること。即時変更は通常のトグル API を使用。予約中のモジュールは管理画面に「{日時}に{有効化/無効化}予定」バッジを表示
- **ロールバックの安全性**: ロールバック実行時、現在の状態もスナップショットとして保存するため、ロールバックの取り消し（再ロールバック）が可能。スナップショットに含まれるモジュール ID が module_definitions から物理削除されている場合は無視（論理削除のみの運用のため通常は発生しない）。ロールバックは is_enabled の復元のみで、trial_expires_at / scheduled_change_at 等の付随情報はクリアされる
- **モジュール構成コピーの権限**: コピー先チームの ADMIN 権限 + コピー元チームの MEMBER 以上の権限が必要。同一ユーザーが両チームに所属している場合のみ実行可能。コピーは is_enabled の状態のみ（trial / schedule 情報はコピーしない）
- **利用モニタリングの通知制御**: 同一モジュールについての「未使用通知」は月1回まで。ADMIN が明示的に無効化アクションを取るか、モジュールの利用が再開されると通知はリセット。通知はサジェストであり強制無効化はしない

## 6. セキュリティ考慮事項

- **認可チェック**: モジュール有効化/無効化 API は ADMIN / DEPUTY_ADMIN（`MANAGE_MODULES`）のみ。テンプレート変更は ADMIN のみ
- **SYSTEM_ADMIN 専用 API の保護**: `/system-admin/**` エンドポイントは SYSTEM_ADMIN ロールを厳密に検証。JWT のロール情報と DB のロールを二重チェック
- **公開エンドポイントの保護**: `GET /templates`・`GET /templates/{id}`・`GET /templates/{id}/modules` は認証不要だが、IP ベースのレートリミットを適用（Bucket4j で 1分間に30回）。テンプレート一覧はマスターデータのため情報漏洩リスクは低いが、大量リクエストによる負荷を防止
- **モジュール有効化チェックの一貫性**: 各機能 API の入り口で必ずモジュール有効化チェックを実行。チェック漏れを防ぐため `@RequiresModule(slug = "reservation", level = ModuleLevel.TEAM)` アノテーションを Controller メソッドに付与し、AOP で一括検証する設計を推奨。`level` パラメータで `module_level_availability` のレベルチェックも同時実行
- **キャッシュ整合性（2層構成）**: (1) Redis `team:modules:{teamId}` は DEFAULT + OPTIONAL の「実効有効モジュール」slug リスト（`is_active` / `feature_flag` / `is_enabled` 考慮済み）を格納。TTL 1時間。(2) Caffeine は `module_level_availability` マスターデータを格納（TTL 10分）。無効化ルール: `is_active`/`feature_flag` 変更 → Redis `team:modules:*` 一括無効化、`module_level_availability` 変更 → Caffeine のみ無効化（ApplicationEvent 伝播）、チーム個別モジュール変更 → Redis `team:modules:{teamId}` のみ
- **レートリミット**: テンプレート・モジュール管理 API は管理者のみ使用するため、レートリミットは標準設定（Bucket4j で 1分間に60回）
- **監査ログ**: SYSTEM_ADMIN によるテンプレート CRUD・モジュール定義 CRUD・レベル別適用設定変更はすべて `audit_logs` に記録する。チーム ADMIN によるモジュール有効化/無効化・テンプレート変更も記録対象
- **XSS 対策**: テンプレート・モジュールの `name`・`description` はレスポンス時にサニタイズする。SYSTEM_ADMIN が設定する値であってもユーザーに表示されるため対策必須
- **プレビュー・シミュレーション API の保護**: `GET /templates/{id}/preview` は認証不要だが公開レートリミット（1分30回）を適用。`GET /system-admin/modules/{id}/deactivation-impact` は SYSTEM_ADMIN 専用（JWT 二重チェック）
- **トライアル API の不正利用防止**: `POST /teams/{teamId}/modules/{moduleId}/trial` は ADMIN のみ。`trial_used = true` を DB レベルで検証し、API の連続呼び出しによる二重トライアルを防止。トライアル開始を audit_logs に記録
- **スケジュール API の保護**: `POST /teams/{teamId}/modules/{moduleId}/schedule` は ADMIN のみ。予約日時のバリデーション（過去日時拒否、24時間以上先）をアプリ層で実施。予約の連続操作はレートリミット（1分10回）を適用
- **ロールバック API の保護**: `POST /teams/{teamId}/modules/rollback` は ADMIN のみ。連続ロールバック防止のため、直前のロールバックから5分以上経過していること（`team_module_snapshots` の最新 `trigger_action = 'ROLLBACK'` の `created_at` で判定）
- **コピー API のクロスチーム権限検証**: `POST /teams/{teamId}/modules/copy-from` は、コピー先（ADMIN 権限）とコピー元（MEMBER 以上）の両方の権限を JWT + DB で検証。異なる組織間のコピーも許容するが、両チームへの所属が必須
