# F12.5: フロントエンドエラー追跡

> **ステータス**: 🟢 設計確定
> **実装フェーズ**: Phase 12
> **最終更新**: 2026-04-02
> **関連ドキュメント**: FRONTEND_CODING_CONVENTION.md §13（エラーレポート）、F10.3 監査ログ、F04.3 プッシュ通知

---

## 1. 概要

ユーザーが黙って離脱しているフロントエンドのクラッシュ・API通信失敗を検知し、開発チームが迅速に対応できる仕組みを構築する。

### 解決する課題

| 課題 | 現状 | 本機能で解決 |
|------|------|------------|
| フロントエンドクラッシュの検知 | サーバーログに残らず把握不能 | Vue errorHandler + ErrorBoundary で捕捉・送信 |
| API通信失敗の把握 | 4xx/5xx はサーバー側ログのみ | クライアント側の文脈（URL・操作中の画面）付きで報告 |
| ユーザーフィードバック収集 | お問い合わせフォームしかない | エラー発生時にその場でコメントを添えて送信 |
| エラーの影響範囲特定 | ログから userId を手動で紐付け | user_id・organization_id を自動付与 |

### 設計方針

- **外部SaaS（Sentry等）は導入しない**: 初期フェーズではバックエンドAPIへの直送で十分。運用負荷とコストを抑える。将来 Sentry/GlitchTip 導入時はバックエンドの受信エンドポイントをアダプター化すれば移行可能
- **ユーザー体験を阻害しない**: エラーレポートモーダルは非ブロッキング。送信は任意。閉じてそのまま操作続行可能
- **エラー連鎖を防止**: レポート送信自体の失敗が新たなエラーモーダルを呼ばない
- **FRONTEND_CODING_CONVENTION.md §13 準拠**: フロントエンド側の実装仕様は §13 に既定済み。本設計書ではバックエンド側の受信・保存・閲覧を定義する

### 3.4 重複集約ロジック

同一エラー（`error_hash` が一致）が既に `status = NEW`、`INVESTIGATING`、または `REOPENED` で存在する場合は、新規レコードを作成せずに既存レコードを更新する:

- `occurrence_count` をインクリメント
- `last_occurred_at` を更新
- `affected_user_count` を更新（後述の影響ユーザー数追跡を参照）
- `user_comment` があれば `latest_user_comment` を上書き（直近1件のみ保持。`admin_note` は管理者専用のため自動追記しない。全コメント履歴が必要になった場合は将来 `error_report_comments` テーブルに分離する）

これにより同一エラーの大量発生時にテーブルが肥大化することを防ぐ。`error_hash` は `SHA-256(normalize(error_message) + "|" + page_url のパス部分)` で計算する（クエリパラメータは除外）。

#### リグレッション検知（RESOLVED → REOPENED）

同一 `error_hash` のレコードが `status = RESOLVED` で存在する場合、新規レコードを作成せずに既存レコードを `REOPENED` に変更する。これにより過去の調査メモ（`admin_note`）や発生履歴との紐付けが維持される。

```java
// ErrorReportService 内の分岐
Optional<ErrorReportEntity> existing = errorReportRepository.findByErrorHash(hash);
if (existing.isPresent()) {
    ErrorReportEntity report = existing.get();
    if (report.getStatus() == ErrorReportStatus.RESOLVED) {
        // リグレッション: REOPENED に変更し、管理者に再発通知
        report.reopen(occurredAt);
        errorReportNotifier.notifyRegression(report);
        return report;
    } else if (report.getStatus() != ErrorReportStatus.IGNORED) {
        // 通常の重複集約
        errorReportRepository.incrementOccurrence(hash, occurredAt, userComment);
        ...
    }
}
// IGNORED または該当なし → 新規作成
```

**再発通知**: REOPENED 時は severity に関わらず必ず Slack + SYSTEM_ADMIN プッシュ通知を送る。「直したはずが再発した」は高優先度の情報であり、通知閾値を無視してでも管理者に伝える必要がある。

#### 影響ユーザー数の追跡

`occurrence_count` は同一ユーザーのリロードも含む延べ回数。影響範囲を正確に把握するため、ユニークユーザー数を `affected_user_count` で追跡する。

**実装方式**: Valkey の SET 構造でエラーハッシュごとのユーザーIDセットを管理する。

```java
// ErrorReportService 内
private int trackAffectedUser(String errorHash, Long userId) {
    if (userId == null) return -1; // 未ログインユーザーはカウント不能
    String key = "error-report:affected:" + errorHash;
    redisTemplate.opsForSet().add(key, userId.toString());
    redisTemplate.expire(key, Duration.ofDays(90)); // クリーンアップと同期
    return redisTemplate.opsForSet().size(key).intValue();
}
```

`affected_user_count` はDB上のカラムとしても保持し、`incrementOccurrence` 実行後にService層で更新する。Valkey が使えない状況のフォールバックとして、DB カラムの値を管理画面に表示する。

#### error_message の正規化（ハッシュ計算前）

同一バグでもエラーメッセージに動的値（プロパティ名、行番号、ID等）が含まれると別ハッシュになり重複集約が効かない。ハッシュ計算前に以下の正規化を適用する:

```java
/**
 * エラーメッセージを正規化してハッシュの重複検知精度を高める。
 * 元の error_message はそのまま DB に保存し、正規化はハッシュ計算時のみ使用する。
 */
private String normalizeForHash(String errorMessage) {
    return errorMessage
        .replaceAll("'[^']*'", "'?'")           // 'name' → '?'
        .replaceAll("\"[^\"]*\"", "\"?\"")       // "name" → "?"
        .replaceAll("\\b\\d+\\b", "N")           // 42 → N
        .replaceAll("[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}", "UUID");
}
```

例: `Cannot read properties of undefined (reading 'teamId')` → `Cannot read properties of undefined (reading '?')`

#### 同時更新の競合制御

同一 `error_hash` に対して複数リクエストが同時到着した場合のrace conditionを防ぐため、ネイティブSQLで原子的に更新する。JPA楽観ロック（`@Version`）は高頻度エラー時にリトライが多発するため採用しない。

```java
@Modifying(clearAutomatically = true)
@Query(value = """
    UPDATE error_reports
    SET occurrence_count = occurrence_count + 1,
        last_occurred_at = :occurredAt,
        latest_user_comment = COALESCE(:userComment, latest_user_comment),
        severity = CASE
            WHEN occurrence_count + 1 >= 50 THEN 'CRITICAL'
            WHEN occurrence_count + 1 >= 10 THEN 'HIGH'
            ELSE severity
        END,
        updated_at = CURRENT_TIMESTAMP
    WHERE error_hash = :hash AND status IN ('NEW', 'INVESTIGATING', 'REOPENED')
    """, nativeQuery = true)
int incrementOccurrence(@Param("hash") String hash,
                        @Param("occurredAt") LocalDateTime occurredAt,
                        @Param("userComment") String userComment);
```

`COALESCE(:userComment, latest_user_comment)` により、コメントが null の場合は既存値を維持する。`clearAutomatically = true` によりネイティブSQL実行後に永続化コンテキストを自動クリアし、後続の `findByErrorHash()` で最新のDB状態を取得できる。

影響行数が 0 の場合は新規 INSERT にフォールバックする。

severity がCASE式で原子的に昇格するため、Service層では更新前後のseverityを比較し、昇格があった場合のみ `ErrorReportNotifier` を呼び出す（§7.3 参照）。

### 3.5 GDPR（F12.3）との整合性

`error_reports` テーブルには `ip_address`、`user_agent`、`user_id` という個人データが含まれる。

- `ErrorReportEntity` に `@PersonalData(category = "error_reports")` を付与する。F12.3 の `PersonalDataCoverageValidator` が起動時に網羅性を検証する
- `user_id` は FK制約 `ON DELETE SET NULL` により、users 物理DELETE時に自動で NULL 化される
- `ip_address` と `user_agent` は FK制約がないため、F12.3 の `AccountPurgeService` Phase 2 で明示的に匿名化する: `UPDATE error_reports SET ip_address = NULL, user_agent = NULL WHERE user_id = ?`（`user_agent` もブラウザフィンガープリントとして個人データに該当）
- エラーレポート自体は匿名化のみで物理削除しない（統計・障害分析に継続利用）

## 5. API 仕様

### 5.1 公開API（認証不要）

#### `POST /api/v1/error-reports`

エラーレポートを受信する。認証不要だが、IPベースのレート制限を適用する。

**レート制限**: 同一IPから10回/分。超過時は `429 Too Many Requests`。Bucket4j + Valkey（Redis）でFilter層に実装する（§5.6 参照）。

**リクエストボディ**:
```json
{
  "error_message": "Cannot read properties of undefined (reading 'name')",
  "stack_trace": "TypeError: Cannot read properties...\n    at TeamList.vue:42\n    ...",
  "page_url": "https://app.mannschaft.com/teams/123",
  "user_agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36",
  "user_comment": "チーム一覧を開いたら画面が真っ白になりました",
  "user_id": 42,
  "occurred_at": "2026-04-02T14:30:00",
  "request_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

**バリデーション**:

| フィールド | 制約 |
|-----------|------|
| `error_message` | @NotBlank, @Size(max=1000) |
| `stack_trace` | サイズ制限なし（DTOでは制限しない。Service層で先頭2,000文字に切り捨ててから保存） |
| `page_url` | @NotBlank, @Size(max=2048) |
| `user_agent` | @Size(max=500) |
| `user_comment` | @Size(max=1000) |
| `user_id` | null許可。存在チェックは行わない（認証なしのため信頼しない） |
| `occurred_at` | @NotNull, ISO 8601 形式 |
| `request_id` | null許可, @Size(max=36)。API呼び出し起因の場合に `X-Request-Id` レスポンスヘッダーの値を送信 |

**レスポンス**: `201 Created`（新規作成時）/ `200 OK`（重複集約時）
```json
{
  "id": 1,
  "status": "NEW"
}
```

**`organization_id` の自動付与**:

リクエストボディに `organization_id` は含まない。`user_id` が送られてきた場合、バックエンドでルックアップして自動付与する。

既存の `TeamOrgMembershipRepository` には user→org の直接ルックアップメソッドがないため、`ErrorReportRepository`（または専用リポジトリ）にネイティブSQLで1クエリに集約する。user→member→team→team_org_membership→organization の2段階結合が必要。

```java
// ErrorReportRepository に定義するネイティブクエリ
@Query(value = """
    SELECT DISTINCT tom.organization_id
    FROM team_org_memberships tom
    JOIN members m ON m.team_id = tom.team_id
    WHERE m.user_id = :userId AND tom.status = 'ACTIVE'
    LIMIT 1
    """, nativeQuery = true)
Optional<Long> findOrganizationIdByUserId(@Param("userId") Long userId);
```

```java
// ErrorReportService 内
private Long resolveOrganizationId(Long userId) {
    if (userId == null) return null;
    return errorReportRepository
        .findOrganizationIdByUserId(userId)
        .orElse(null);
}
```

- FRONTEND_CODING_CONVENTION.md §13 の送信データ仕様に `organization_id` は定義されていないため、フロントエンド側の変更は不要
- `user_id` 自体が自己申告で信頼しない前提なので、`organization_id` も「参考値」の位置付け
- 複数組織に所属する場合は最初に見つかった1件を使用する（影響範囲の推定には十分）

**セキュリティ考慮**:
- `user_id` はあくまで参考値。認証なしのため改竄可能。管理画面では「自己申告」として表示する
- `stack_trace` と `error_message` にユーザー入力が混入する可能性あり。管理画面での表示時はHTMLエスケープ必須
- `page_url` は内部URLのみ想定。外部URL（`https://` で始まるが自ドメインでない）は `ip_address` とともにログに記録し、不正利用監視に活用する

### 5.4 既知の不具合API（公開）

#### `GET /api/v1/active-incidents`

CRITICAL または HIGH のエラーが `status = NEW / INVESTIGATING / REOPENED` で存在する場合、該当ページのパスパターンを返す。フロントエンドのレイアウトで定期ポーリング（60秒間隔）し、一致するページにバナーを表示する。

**認証**: 不要（`permitAll()`）。レート制限は不要（GETで軽量、Valkey キャッシュで応答）。

**レスポンス**: `200 OK`
```json
{
  "incidents": [
    {
      "page_pattern": "/teams/*",
      "message": "一部の画面で不具合が発生しています。現在対応中です。",
      "severity": "CRITICAL",
      "since": "2026-04-02T10:00:00"
    }
  ]
}
```

**キャッシュ**: Valkey に TTL 1分でキャッシュ。エラーレポートのステータス変更時に `@CacheEvict` で無効化。

**`page_pattern` の生成**: `page_url` からパスを抽出し、末尾の動的セグメント（数値・UUID）をワイルドカード `*` に置換する。例: `/teams/123/members` → `/teams/*/members`。

> **注意**: フロントエンドでは `useActiveIncidents` コンポーザブルを作成し、現在の `route.path` が `incidents[].page_pattern` にマッチするかチェックする。マッチした場合にレイアウト上部にバナーを表示する。

### 5.5 解決通知

エラーレポートが `RESOLVED` に変更された際、`user_id` が非NULLのレポートに対して報告者にプッシュ通知を送信する。

```java
// ErrorReportService 内（ステータス更新時）
if (newStatus == ErrorReportStatus.RESOLVED && report.getUserId() != null) {
    notificationService.createNotification(
        report.getUserId(), "ERROR_REPORT_RESOLVED", NotificationPriority.NORMAL,
        "ご報告いただいた不具合が解決しました",
        String.format("エラー「%s」への対応が完了しました。ご報告ありがとうございました。",
                truncate(report.getErrorMessage(), 50)),
        "ERROR_REPORT", report.getId(),
        NotificationScopeType.PERSONAL, null,
        null, null
    );
}
```

> **注意**: `user_id` は自己申告値のため、通知先が正確でない可能性がある。悪意ある user_id 指定で他人に通知が飛ぶリスクがあるが、通知内容は「不具合が解決した」という無害な情報のみであるため、セキュリティ上の問題は低い。

### 5.6 レート制限の実装

Bucket4j + Valkey（Redis）をFilter層で実装する。Controller層のアノテーションではなく、Filter層でブロックすることで不正リクエストがService層に到達しない。

**依存追加（build.gradle.kts）**:
```kotlin
implementation("com.bucket4j:bucket4j_jcache:8.10.1")
implementation("com.bucket4j:bucket4j-redis:8.10.1")
```

```java
/**
 * エラーレポートAPIのIPベースレート制限フィルタ。
 * /api/v1/error-reports のみに適用し、他APIに影響しない。
 */
@Component
@RequiredArgsConstructor
public class ErrorReportRateLimitFilter extends OncePerRequestFilter {

    private final ProxyManager<String> proxyManager; // Bucket4j Valkey連携

    @Override
    protected boolean shouldNotFilter(HttpServletRequest request) {
        // getServletPath() を使用。getRequestURI() はコンテキストパスを含むため不可。
        return !"/api/v1/error-reports".equals(request.getServletPath())
                || !"POST".equalsIgnoreCase(request.getMethod());
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain chain) throws ServletException, IOException {
        String key = "error-report-rate:" + request.getRemoteAddr();
        Bucket bucket = proxyManager.builder()
                .build(key, () -> BucketConfiguration.builder()
                        .addLimit(Bandwidth.simple(10, Duration.ofMinutes(1)))
                        .build());
        if (bucket.tryConsume(1)) {
            chain.doFilter(request, response);
        } else {
            response.setStatus(HttpStatus.TOO_MANY_REQUESTS.value());
        }
    }
}
```

**却下した代替案**:
- Resilience4j `@RateLimiter`: インメモリのみ。複数インスタンス環境で正しく動作しない
- Spring Cloud Gateway: モノリスアーキテクチャに対してオーバーキル

## 6. 自動重要度判定

エラーレポート受信時に、以下のルールで `severity` を自動設定する。管理者は後から手動で変更可能。

#### 新規作成時に適用するルール

| 条件 | severity | 理由 |
|------|----------|------|
| `page_url` が `/checkout` または `/payment` を含む | HIGH | 決済フローのエラーは即対応 |
| `error_message` が `ChunkLoadError` を含む | LOW | デプロイ直後の一時的な問題 |
| 上記以外 | MEDIUM | デフォルト |

#### 重複集約時に適用するルール（`incrementOccurrence` SQL 内 CASE式）

| 条件 | severity | 理由 |
|------|----------|------|
| `occurrence_count` が 50 以上に到達 | CRITICAL | 大量発生 = 広範囲に影響 |
| `occurrence_count` が 10 以上に到達 | HIGH | 複数ユーザーに影響 |

#### affected_user_count による補正（Service層で実行）

SQL内CASE式での severity 昇格後、Service層で `affected_user_count` を考慮した補正を行う:

| 条件 | 補正 | 理由 |
|------|------|------|
| `occurrence_count >= 50` かつ `affected_user_count <= 1` | CRITICAL → HIGH に据え置き | 1人のユーザーが大量リロードしているだけ |
| `affected_user_count >= 20` かつ severity が MEDIUM | MEDIUM → HIGH に昇格 | ユニークユーザー20人以上は発生回数に関わらず影響大 |

> この補正はDB原子操作ではなくService層で実施するため、一瞬CRITICALになった後にHIGHに下がるケースがある。通知はService層補正後の最終severityに基づいて送信する。

> **注意**: 管理者が手動で severity を変更済みの場合、CASE式の `ELSE severity` により手動設定が維持される。ただし occurrence_count 閾値を超えた場合は自動昇格が優先される。手動で CRITICAL に設定したものが MEDIUM に降格されることはない（CASE式は昇格方向のみ）。

## 7. 通知

### 7.1 Slack Webhook（任意設定）

`application.yml` に Slack Webhook URL が設定されている場合のみ動作する。

```yaml
mannschaft:
  error-report:
    slack-webhook-url: ${MANNSCHAFT_ERROR_REPORT_SLACK_WEBHOOK:}
    notify-threshold: HIGH  # HIGH以上で通知（デフォルト）
```

通知対象: `severity` が閾値以上のエラーレポート。通知タイミングの詳細は §7.3 を参照。

**実装方式**: `RestClient`（Spring 6.1+）で直接送信する。F09.9 Webhook 基盤（ユーザー定義のOutgoing Webhook）とは用途が異なるため統合しない。通知は `@Async("event-pool")` 内で実行するため非同期性は確保済み。

```java
@Async("event-pool")
public void notifySlack(ErrorReportEntity report) {
    if (slackWebhookUrl == null || slackWebhookUrl.isBlank()) return;
    try {
        // String.format だと errorMessage 内の " や改行で JSON が壊れるため ObjectMapper を使う
        String text = String.format(":rotating_light: *[%s] フロントエンドエラー*\n> %s\nページ: %s\n発生回数: %d",
                report.getSeverity(), report.getErrorMessage(),
                report.getPageUrl(), report.getOccurrenceCount());
        String payload = objectMapper.writeValueAsString(Map.of("text", text));
        restClient.post().uri(slackWebhookUrl)
            .contentType(MediaType.APPLICATION_JSON)
            .body(payload)
            .retrieve().toBodilessEntity();
    } catch (Exception e) {
        log.warn("Slack通知送信失敗: errorReportId={}", report.getId(), e);
    }
}
```

### 7.2 SYSTEM_ADMIN プッシュ通知

CRITICAL のエラーレポートが作成された場合、全 `SYSTEM_ADMIN` にプッシュ通知を送信する（F04.3 通知基盤を活用）。

```java
notificationService.createNotification(
    adminUserId, "ERROR_REPORT_CRITICAL", NotificationPriority.HIGH,
    "フロントエンドエラー（CRITICAL）",
    String.format("エラー「%s」が %d 回発生しています", errorMessage, count),
    "ERROR_REPORT", errorReportId,
    NotificationScopeType.SYSTEM, null,
    "/system-admin/error-reports/" + errorReportId, null
);
```

### 7.3 severity 昇格時の通知

重複集約で `occurrence_count` が閾値を超えて severity が昇格した場合にも通知を送る。「新規作成時のみ通知」だと、MEDIUM で作成されたエラーが CRITICAL に昇格しても誰にも通知されない問題を防ぐ。

**通知タイミング**:
1. **新規作成時**: severity が `notify-threshold` 以上なら即通知（Slack + プッシュ通知）
2. **severity 昇格時**: `incrementOccurrence()` 実行後、Service層で更新前後の severity を比較し、昇格があった場合のみ通知

```java
// ErrorReportService 内
ErrorReportSeverity oldSeverity = existing.getSeverity();
errorReportRepository.incrementOccurrence(hash, occurredAt, userComment);
// @Modifying(clearAutomatically = true) により永続化コンテキストは自動クリア済み
ErrorReportEntity updated = errorReportRepository.findByErrorHash(hash).orElseThrow();
ErrorReportSeverity newSeverity = updated.getSeverity();

if (newSeverity.ordinal() > oldSeverity.ordinal()) {
    errorReportNotifier.notifyEscalation(updated, oldSeverity, newSeverity);
}
```

**重複通知の防止**: 同一 `error_hash` の各閾値（10回→HIGH、50回→CRITICAL）につき通知は1回のみ。`occurrence_count` が10→11→...→49 の間は HIGH 通知を再送しない（severity が変化しないため）。

### 7.4 リグレッション（再発）通知

RESOLVED → REOPENED に変更された場合、severity に関わらず必ず Slack + SYSTEM_ADMIN プッシュ通知を送る。

```java
@Async("event-pool")
public void notifyRegression(ErrorReportEntity report) {
    // Slack 通知（閾値無視で必ず送信）
    String text = String.format(":warning: *[再発] フロントエンドエラー*\n> %s\nページ: %s\n前回解決: %s",
            report.getErrorMessage(), report.getPageUrl(), report.getResolvedAt());
    // ... Slack 送信

    // SYSTEM_ADMIN プッシュ通知
    for (Long adminUserId : userRoleRepository.findSystemAdminUserIds()) {
        notificationService.createNotification(
            adminUserId, "ERROR_REPORT_REGRESSION", NotificationPriority.HIGH,
            "解決済みエラーが再発しました",
            String.format("エラー「%s」が再発しました。前回の admin_note を確認してください。",
                    truncate(report.getErrorMessage(), 50)),
            "ERROR_REPORT", report.getId(),
            NotificationScopeType.SYSTEM, null,
            "/system-admin/error-reports/" + report.getId(), null
        );
    }
}
```

> **重要**: 再発通知は `notify-threshold` 設定を無視する。「直したはずが再発」は severity が LOW でも高優先度の情報。

## 8. データ保持・クリーンアップ

### 8.1 クリーンアップバッチ

| 対象 | 保持期間 | 処理 |
|------|---------|------|
| `status = RESOLVED` or `IGNORED` | 90日 | 物理削除 + Valkey の `error-report:affected:{hash}` キーも削除 |
| `status = NEW` or `REOPENED` で `last_occurred_at` が 180日以上前 | — | `IGNORED` に自動変更 |
| `status = INVESTIGATING` で `updated_at` が 180日以上前 | — | `IGNORED` に自動変更 + `admin_note` に「180日間更新なしのため自動クローズ」追記 |

クリーンアップバッチは `@Scheduled(cron = "0 0 3 * * SUN", zone = "Asia/Tokyo")` で毎週日曜 AM3:00（JST）に実行。`@SchedulerLock` + `@Transactional` で分散ロック。既存の `AuthCleanupBatchService` と同パターン。

### 8.2 週次サマリーメール

毎週月曜 AM9:00（JST）に全 SYSTEM_ADMIN へエラーレポートの週次サマリーメールを自動送信する。

```java
@Scheduled(cron = "0 0 9 * * MON", zone = "Asia/Tokyo")
@SchedulerLock(name = "errorReportWeeklySummary", lockAtMostFor = "PT10M", lockAtLeastFor = "PT1M")
```

**メール内容**:

| 項目 | 内容 |
|------|------|
| 先週の新規エラー | 件数 |
| 先週の CRITICAL / HIGH | 件数 |
| 現在の未対応（NEW + INVESTIGATING + REOPENED） | 件数 |
| 先週の再発（REOPENED） | 件数 |
| TOP5 エラー | エラーメッセージ・ページ・発生回数・影響ユーザー数 |

**送信条件**: 先週の新規エラーが0件かつ未対応が0件の場合は送信しない（「何もなかった週」のメールは不要）。

**実装**: `ErrorReportRepository` の集計クエリ + `EmailService.sendEmail()` でHTML送信。テンプレートは `OverdueInvoiceBatchService.buildOverdueEmailHtml()` と同パターンのインラインHTML。

## 10. フロントエンド実装（概要）

FRONTEND_CODING_CONVENTION.md §13 に詳細定義済み。ここではバックエンド視点での要点のみ記載。

> **重要**: FRONTEND_CODING_CONVENTION.md §13 にバックエンド設計書（本ドキュメント）への相互リンクを追加すること。フロントエンド実装者がバックエンド仕様（レート制限、正規化、重複集約、リグレッション検知）を把握できるようにする。

| コンポーネント | ファイル | 役割 |
|-------------|--------|------|
| グローバルエラーハンドラー | `plugins/errorHandler.client.ts` | Vue errorHandler + unhandledrejection を捕捉 |
| エラーレポート composable | `composables/useErrorReport.ts` | capture() / submit() API |
| エラーレポートモーダル | `components/ErrorReportModal.vue` | ユーザー向けUI（入力→送信中→完了→追跡番号表示） |
| 既知の不具合バナー | `composables/useActiveIncidents.ts` | `GET /api/v1/active-incidents` を60秒ポーリング |
| 既知の不具合バナー | `components/ActiveIncidentBanner.vue` | レイアウト上部にバナー表示 |

### フロントエンド側の注意事項

- 開発環境（`NODE_ENV === 'development'`）ではモーダル非表示オプションを設ける（CONVENTION §13）
- `occurred_at` は `dayjs().format('YYYY-MM-DDTHH:mm:ss')` で JST 前提のタイムゾーンオフセットなし ISO 8601（.claudecode.md §20 準拠）
- `stack_trace` は先頭 2,000 文字で切り捨て
- 送信失敗時は `console.error` に留め、再帰的モーダル表示を行わない

### X-Request-Id の送信

API 呼び出し起因のエラーの場合、`ofetch` の `onResponseError` で `X-Request-Id` レスポンスヘッダーを取得し、エラーレポートの `request_id` フィールドとして送信する。これによりフロントエンドエラーとバックエンドログ（`RequestLoggingFilter` が MDC に設定する `requestId`）を1対1で紐付けできる。

```ts
// composables/useApi.ts の onResponseError 内
const requestId = response.headers.get('X-Request-Id')
// useErrorReport.capture() に requestId を渡す
```

### 送信完了後の追跡番号表示

エラーレポート送信後の完了画面に、レスポンスの `id` を追跡番号として表示する。ログイン中のユーザーには「対応完了時にお知らせで通知します」のメッセージも表示する。

```
送信完了画面:
  「ご報告ありがとうございます。開発チームが確認します。」
  「追跡番号: #1234」
  「※ 対応完了時にお知らせで通知します」 ← user_id 非NULLの場合のみ表示
```

### 既知の不具合バナー

`useActiveIncidents` コンポーザブルが `GET /api/v1/active-incidents` を60秒間隔でポーリングし、現在の `route.path` が `incidents[].page_pattern` にマッチする場合にレイアウト上部にバナーを表示する。

```ts
// composables/useActiveIncidents.ts
const { data: incidents } = useAsyncData('active-incidents',
  () => $fetch('/api/v1/active-incidents'),
  { server: false, watch: false }
)

// 60秒ポーリング
const { pause, resume } = useIntervalFn(() => refresh(), 60_000)
```

バナーはページ遷移をブロックしない。`<Teleport to="body">` で最上部に固定表示し、閉じるボタンで非表示にできる（セッション中は再表示しない）。

| 14 | リグレッション検知 | RESOLVED → REOPENED ステータス遷移 + 再発通知 | 過去の調査メモとの紐付け維持。再発は閾値無視で必ず通知（§3.4, §7.4） |

| 16 | 影響ユーザー数追跡 | `affected_user_count` カラム + Valkey SET で重複排除 | occurrence_countだけでは1人の大量リロードと100人×1回を区別できない（§3.4, §6） |
| 17 | 一括操作API | `PATCH /bulk` で複数レポートを一括 RESOLVE/IGNORE | ChunkLoadError等の一括処理で管理者の手間を削減（§5.2） |
| 18 | 既知の不具合バナー | `GET /api/v1/active-incidents` + FEバナーコンポーネント | ユーザーの不安を軽減し問い合わせを削減（§5.4, §10） |
| 19 | 解決通知 | RESOLVED 時に報告者（user_id非NULL）にプッシュ通知 | 報告者に対応完了を伝え、報告する動機を維持（§5.5） |
| 20 | 週次サマリーメール | 毎週月曜AM9:00にSYSTEM_ADMINへエラー傾向をメール送信 | 管理画面を開かずに傾向を把握。0件週は送信しない（§8.2） |
| 21 | CONVENTION §13 相互リンク | §13 に本設計書へのリンクを追加 | FE実装者がBE仕様（レート制限・正規化・重複集約）を把握できるようにする（§10） |
