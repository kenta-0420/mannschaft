# F05.2: 回覧板

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 5
> **最終更新**: 2026-03-14
> **関連ドキュメント**: F05.3 電子印鑑（押印連携）、F04.3 プッシュ通知（リマインド通知）

---

## 1. 概要

文書・お知らせをチーム・組織内のメンバーに回覧し、全員の確認（電子印鑑の押印）をもって完結する回覧板機能。**順番回覧**（前の人が押印後に次の人へ通知）、**ハイブリッド回覧**（上位者は順番、以降は一斉配布）、**一斉回覧**（全員に同時配布）の3方式を提供する。確認は電子印鑑機能（F05.3）と連携し、押印のたびに証跡ログが記録される。押印の表示スタイルとして日本のビジネス慣習に基づく**お辞儀ハンコ**（上位者に向かって傾けて押印）にも対応し、スコープ別・回覧別に設定可能。未確認者への自動リマインド通知で確認漏れを防止し、全員の押印完了時に自動アーカイブする。添付ファイル（PDF・画像等）に対応し、稟議書・安全点検記録・月次報告などの回覧ワークフローを電子化する。

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全組織・チームの回覧参照 |
| ADMIN | 回覧作成・編集・削除・回覧先メンバー管理・回覧ステータス管理・催促送信・強制完了 |
| DEPUTY_ADMIN | `MANAGE_CONTENT` 権限を持つ場合: 回覧作成・編集・回覧先管理・催促送信 |
| MEMBER | 回覧作成（チーム設定 `allow_member_circulation` で制御）・回覧の押印（確認）・自分宛の回覧一覧・コメント投稿 |
| SUPPORTER | 回覧の押印（確認）・自分宛の回覧一覧（回覧作成は不可） |
| GUEST | 対象外 |

### 対象レベル
- [x] 組織 (Organization)
- [x] チーム (Team)
- [ ] 個人 (Personal)

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `circulation_documents` | 回覧文書（タイトル・本文・回覧方式・ステータス・表示スタイル） | あり |
| `circulation_recipients` | 回覧先メンバー（押印状態・順番・傾き角度） | なし |

| `circulation_comments` | 回覧に対するコメント（押印時の一言等） | あり |

### テーブル定義

#### `circulation_documents`

回覧文書。1文書に対して複数のメンバーが押印（確認）する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users（作成者）。ON DELETE RESTRICT |
| `title` | VARCHAR(200) | NO | — | 回覧タイトル |
| `body` | TEXT | YES | NULL | 本文（Markdown 対応。添付ファイルのみの回覧では NULL 許可） |
| `circulation_mode` | VARCHAR(20) | NO | — | 回覧方式。`SEQUENTIAL`（順番回覧）/ `HYBRID`（途中まで順番＋以降一斉）/ `SIMULTANEOUS`（一斉回覧） |
| `sequential_count` | SMALLINT | YES | NULL | HYBRID モードで順番回覧する人数。先頭 N 人が順番、残りが一斉。SEQUENTIAL / SIMULTANEOUS の場合は NULL |
| `status` | VARCHAR(20) | NO | 'DRAFT' | 回覧ステータス（`DRAFT` / `IN_PROGRESS` / `COMPLETED` / `CANCELLED`） |
| `priority` | VARCHAR(10) | NO | 'NORMAL' | 緊急度（`URGENT` / `NORMAL`）。URGENT は通知時に強調表示 |
| `due_date` | DATE | YES | NULL | 確認期限日（任意。期限超過時にリマインド強化） |
| `reminder_enabled` | BOOLEAN | NO | TRUE | 自動リマインド通知の有効/無効 |
| `reminder_interval_hours` | INT | NO | 24 | リマインド通知の間隔（時間単位。デフォルト: 24時間） |
| `stamp_display_style` | VARCHAR(20) | NO | 'STANDARD' | 押印表示スタイル。`STANDARD`（通常）/ `OJIGI`（お辞儀ハンコ：上位者に向かって傾斜） |
| `total_recipient_count` | SMALLINT | NO | 0 | 回覧先の総人数（非正規化） |
| `stamped_count` | SMALLINT | NO | 0 | 押印済みの人数（非正規化。全員完了で自動 COMPLETED） |
| `completed_at` | DATETIME | YES | NULL | 全員押印完了日時 |
| `attachment_count` | SMALLINT | NO | 0 | 添付ファイル数（非正規化） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_cd_scope (scope_type, scope_id, status, created_at DESC)   -- スコープ別回覧一覧
INDEX idx_cd_created_by (created_by, created_at DESC)                -- 自分が作成した回覧一覧
```

**制約・備考**
- `scope_type` + `scope_id` はポリモーフィック参照。FK 制約なし（アプリ層で検証）
- `status` 遷移: `DRAFT` → `IN_PROGRESS`（回覧開始）→ `COMPLETED`（全員押印完了）。`DRAFT` / `IN_PROGRESS` → `CANCELLED`（取消）。`COMPLETED` からの遷移は不可
- 完了判定: `circulation_recipients` に `status = PENDING` が0件になった時点で `status = COMPLETED` + `completed_at = NOW()` を自動更新（SKIPPED を考慮するため stamped_count ではなく PENDING 件数で判定）
- `due_date` は表示・通知用であり、期限超過後も押印操作は可能（強制締切ではない）。自動強制完了は実装しない（安全点検等の法的証跡で「未確認なのに完了扱い」となるリスクを回避）
- DRAFT の回覧は作成者のみ閲覧可能。IN_PROGRESS で回覧先メンバーに公開
- `created_by` ON DELETE RESTRICT: 作成者が退会すると回覧が孤立するため、退会前に ADMIN への引継ぎ or 完了/取消を要求
- 論理削除: DRAFT / CANCELLED の回覧のみ削除可能。IN_PROGRESS / COMPLETED は削除不可（証跡保持）
- `sequential_count` バリデーション: HYBRID の場合 `1 ≤ sequential_count < total_recipient_count`。SEQUENTIAL / SIMULTANEOUS の場合は NULL であること
- `stamp_display_style`: スコープ別のデフォルト値はチーム設定（`default_stamp_display_style`）で管理。回覧作成時に上書き可能

---

#### `circulation_recipients`

回覧先メンバー。`sort_order` で押印グループを管理する。同じ `sort_order` のメンバーは同時に押印可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `circulation_id` | BIGINT UNSIGNED | NO | — | FK → circulation_documents（ON DELETE CASCADE） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE RESTRICT） |
| `sort_order` | SMALLINT | NO | 0 | 押印グループ順序。同じ値のメンバーは同時押印可能。SEQUENTIAL: 全員異なる値(0,1,2,...)。HYBRID: 先頭N人は異なる値、残りは同じ値。SIMULTANEOUS: 全員0 |
| `status` | VARCHAR(20) | NO | 'PENDING' | 確認状態（`PENDING` / `STAMPED` / `SKIPPED`）。SKIPPED = ADMIN が強制スキップ |
| `stamped_at` | DATETIME | YES | NULL | 押印日時 |
| `stamp_log_id` | BIGINT UNSIGNED | YES | NULL | FK → seal_stamp_logs（押印時に記録される証跡ログの ID） |
| `tilt_angle` | SMALLINT | NO | 0 | 押印傾き角度（度単位、-30〜+30）。お辞儀スタイル時に使用。正値=左傾き（上位者方向へお辞儀） |
| `is_flipped` | BOOLEAN | NO | FALSE | ランダム逆さまイースターエッグ（押印時に約0.1%の確率で TRUE） |
| `corrected_at` | DATETIME | YES | NULL | 逆さまハンコの訂正日時（訂正押印時にセット） |
| `correction_stamp_log_id` | BIGINT UNSIGNED | YES | NULL | FK → seal_stamp_logs（訂正押印の証跡ログ ID） |
| `last_reminded_at` | DATETIME | YES | NULL | 最後にリマインド通知を送った日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_cr_doc_user (circulation_id, user_id)              -- 同一回覧への重複登録防止
INDEX idx_cr_user (user_id, status)                               -- 自分宛の未確認回覧一覧
INDEX idx_cr_doc_order (circulation_id, sort_order)               -- 順番回覧の次の確認者取得
```

**制約・備考**
- `user_id` ON DELETE RESTRICT: 回覧対象者が退会する場合、未押印の回覧があればエラー。ADMIN が先に SKIPPED に変更するか回覧を完了させる必要がある
- `sort_order` グループ管理: 同じ `sort_order` を持つメンバーは同時に押印可能。前のグループ（より小さい `sort_order`）に PENDING がなくなった時点で次のグループがアンロックされる
  - SEQUENTIAL: `sort_order` = 0, 1, 2, 3, ...（全員異なる → 1人ずつ順番）
  - HYBRID（`sequential_count=2`）: `sort_order` = 0, 1, 2, 2, 2, ...（先頭2人は順番、残りは一斉）
  - SIMULTANEOUS: `sort_order` = 0, 0, 0, ...（全員同じ → 全員同時）
- `stamp_log_id` は F05.3 の `seal_stamp_logs` への FK。押印操作は `CirculationService` 経由で `SealStampService.stamp()` を呼び出し、返却された `stamp_log_id` を保存
- SKIPPED: ADMIN が特定のユーザーを強制スキップ（長期不在等）。`stamped_count` にはカウントしないが、完了判定は `PENDING が0件` で行う（STAMPED + SKIPPED = total_recipient_count）
- `tilt_angle`: `stamp_display_style = 'OJIGI'` の場合に自動計算。デフォルト = `min(sort_order × 8, 30)` 度。ユーザーが押印前にフロントエンドで -30〜+30 の範囲で調整可能。STANDARD スタイルの場合は常に 0
- `is_flipped`: 押印時にバックエンドが `SecureRandom` で 0.1%（1/1000）の確率で TRUE を決定。TRUE の場合、フロントエンドでハンコを 180° 回転して表示（逆さまハンコ）。ユーザーは訂正 API で「二重線 + 押し直し」が可能
- `correction_stamp_log_id`: 訂正押印時に新たな `seal_stamp_logs` レコードを作成し、その ID を格納。フロントエンドでは元のハンコ（二重線付き）+ 訂正ハンコ（隣接配置）の両方を表示

#### `circulation_attachments`

回覧文書の添付ファイル。PDF・画像等をサポート。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `circulation_id` | BIGINT UNSIGNED | NO | — | FK → circulation_documents（ON DELETE CASCADE） |
| `file_key` | VARCHAR(500) | NO | — | S3 オブジェクトキー |
| `original_filename` | VARCHAR(255) | NO | — | アップロード時の元ファイル名 |
| `file_size` | INT | NO | — | ファイルサイズ（バイト） |
| `mime_type` | VARCHAR(100) | NO | — | MIMEタイプ（`application/pdf`, `image/webp` 等） |
| `sort_order` | SMALLINT | NO | 0 | 表示順 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_ca_doc (circulation_id, sort_order)   -- 回覧別添付ファイル一覧
```

**制約・備考**
- `circulation_id` ON DELETE CASCADE: 回覧文書削除時にカスケード
- 対応ファイル形式: PDF, JPEG, PNG, WebP, GIF（画像は Lambda で WebP 変換）
- ファイルサイズ上限: 1ファイル20MB、1回覧あたり合計50MB
- S3 配信: CloudFront 経由の Pre-signed URL（有効期限: 1時間）
- 最大10ファイル/回覧

#### `circulation_comments`

回覧に対するコメント。押印時の一言や質問・確認事項を記録する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `circulation_id` | BIGINT UNSIGNED | NO | — | FK → circulation_documents（ON DELETE CASCADE） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE RESTRICT） |
| `body` | TEXT | NO | — | コメント本文（最大1,000文字） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_cc_doc (circulation_id, created_at)   -- 回覧別コメント一覧
```

**制約・備考**
- 押印時にコメントを付けることが可能（任意）
- 押印せずにコメントのみ投稿することも可能（質問・確認用）
- コメントの編集は不可（証跡性のため）。論理削除のみ

### リクエスト／レスポンス仕様

#### `GET /api/v1/circulation`

**認可**: 対象スコープの SUPPORTER 以上

スコープ別の回覧一覧をカーソルベースのページネーションで取得する。

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `scope_type` | String | Yes | `TEAM` / `ORGANIZATION` |
| `scope_id` | Long | Yes | 対象スコープの ID |
| `status` | String | No | `IN_PROGRESS` / `COMPLETED` / `CANCELLED`（フィルタ。デフォルト: 全件） |
| `keyword` | String | No | タイトル部分一致検索（`title LIKE '%keyword%'`） |
| `overdue` | Boolean | No | `true` の場合、期限超過の IN_PROGRESS 回覧のみ（`due_date < NOW()` かつ `status = IN_PROGRESS`） |
| `cursor` | Long | No | カーソル |
| `limit` | Integer | No | 取得件数（デフォルト: 20、最大: 50） |

```json
      ],

  ],
```

**補足**
- `current_recipients`: 現在の確認待ちユーザー情報の配列。SEQUENTIAL: 1人。HYBRID: 順番部分では1人、一斉部分では残り全員。SIMULTANEOUS: 全 PENDING ユーザー
- `due_date_status`: `ON_TRACK`（期限内）/ `DUE_SOON`（期限3日以内）/ `OVERDUE`（期限超過）/ `null`（期限未設定）

**エラーレスポンス**
| ステータス | 条件 |

| 403 | 対象スコープのメンバーでない |

#### `POST /api/v1/circulation`

**認可**: ADMIN / DEPUTY_ADMIN（`MANAGE_CONTENT`）。チーム設定 `allow_member_circulation = true` の場合は MEMBER も作成可能

回覧文書を DRAFT ステータスで作成する。

**リクエスト（multipart/form-data）**
| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `scope_type` | String | Yes | `TEAM` / `ORGANIZATION` |
| `scope_id` | Long | Yes | 対象スコープの ID |
| `title` | String | Yes | タイトル（1〜200文字） |
| `body` | String | No | 本文（Markdown、最大10,000文字） |
| `circulation_mode` | String | Yes | `SEQUENTIAL` / `HYBRID` / `SIMULTANEOUS` |
| `sequential_count` | Integer | Conditional | HYBRID の場合のみ必須。先頭から何人を順番回覧にするか（1 ≤ N < recipient_ids.length） |
| `priority` | String | No | `URGENT` / `NORMAL`（デフォルト: `NORMAL`） |
| `due_date` | String | No | 確認期限日（ISO 8601 date） |
| `reminder_enabled` | Boolean | No | デフォルト: `true` |
| `reminder_interval_hours` | Integer | No | デフォルト: `24` |
| `stamp_display_style` | String | No | `STANDARD` / `OJIGI`（デフォルト: スコープ設定値。未設定時は `STANDARD`） |
| `recipient_ids` | Long[] | Yes | 回覧先ユーザー ID の配列（SEQUENTIAL / HYBRID の場合は配列順が回覧順序） |
| `files` | MultipartFile[] | No | 添付ファイル（最大10ファイル、各20MB以下） |

**レスポンス（201 Created）**
```json
{
```

#### `POST /api/v1/circulation/{id}/stamp`

**認可**: 回覧先メンバー（本人）

回覧に押印（確認）する。電子印鑑（F05.3）を使用して押印証跡を記録する。

|-----------|---|------|------|
| `seal_id` | Long | No | 使用する印鑑 ID。省略時はスコープ別デフォルト印鑑（F05.3 `seal_scope_defaults`） |
| `tilt_angle` | Integer | No | 押印傾き角度（-30〜+30度）。`stamp_display_style = 'OJIGI'` 時のみ有効。省略時は自動計算（`min(sort_order × 8, 30)`） |
| `comment` | String | No | 押印時のコメント（最大1,000文字） |

**レスポンス（200 OK）**
```json
{
  "data": {
    "stamp_log_id": 42,
    "seal_variant": "LAST_NAME",
    "seal_svg": "<svg>...</svg>",
    "tilt_angle": 8,
    "is_flipped": false,

      { "user_id": 23, "display_name": "山本花子" },
      { "user_id": 24, "display_name": "伊藤三郎" }

}
```

**補足**
- `next_recipients`: 次にアンロックされる確認者の配列。SEQUENTIAL: 1人。HYBRID の一斉部分突入時: 残り全員。SIMULTANEOUS: 他の PENDING 全員。最後の確認者が押印した場合は空配列

| 400 | seal_id が自分の印鑑でない / tilt_angle が -30〜+30 の範囲外 |

## 5. ビジネスロジック

### 主要フロー

#### 回覧の作成と開始
```
1. ADMIN / DEPUTY_ADMIN（または allow_member_circulation = true の MEMBER）が回覧作成画面で入力
2. POST /api/v1/circulation を呼び出し（DRAFT で保存）
3. バックエンドが以下を実行:
   a. スコープバリデーション（scope_type + scope_id + メンバーシップ確認）
   b. MEMBER の場合: チーム設定 allow_member_circulation を確認
   c. recipient_ids の全員がスコープメンバーであることを検証
   d. HYBRID の場合: sequential_count のバリデーション（1 ≤ N < recipient_ids.length）
   e. circulation_documents に INSERT（status = DRAFT）
   f. circulation_recipients に INSERT（recipient_ids の各ユーザー）
      - SEQUENTIAL: 配列の順序を sort_order に反映（0, 1, 2, ...）
      - HYBRID: 先頭 sequential_count 人は 0, 1, ..., N-1。残りは全員 sort_order = N
      - SIMULTANEOUS: 全員 sort_order = 0
   g. circulation_attachments に INSERT（添付がある場合）
4. 作成者が内容を確認後、POST /circulation/{id}/start を呼び出し
5. バックエンドが以下を実行:
   a. status = IN_PROGRESS に更新
   b. sort_order = 0 のユーザー全員に通知送信:
      - SEQUENTIAL: 1人のみ
      - HYBRID: 1人のみ（先頭の順番回覧者）
      - SIMULTANEOUS: 全員
   c. ApplicationEvent（CirculationStartedEvent）を発行
```

#### 押印（確認）フロー
```
1. メンバーが回覧詳細画面で「押印する」ボタンを押下
   - お辞儀スタイルの場合: 印鑑プレビューに傾き角度スライダーを表示（初期値は自動計算値）
2. POST /api/v1/circulation/{id}/stamp を呼び出し
3. バックエンドが以下を検証:
   a. 回覧が IN_PROGRESS であること
   b. 自分が回覧先メンバーであること
   c. 自分がまだ PENDING であること
   d. 自分の sort_order グループがアンロック済みであること:
      「自分の sort_order 未満の sort_order を持つレコードに PENDING が0件」
4. 以下を単一トランザクションで実行:
   a. SealStampService.stamp() を呼び出し:
      - seal_id を指定（省略時はスコープ別デフォルト印鑑）
      - target_type = 'CIRCULATION', target_id = circulation_id
      - seal_stamp_logs に INSERT → stamp_log_id を返却
   b. お辞儀角度の決定:
      - stamp_display_style = 'OJIGI' の場合:
        - tilt_angle がリクエストで指定されていればその値（-30〜+30 バリデーション）
        - 未指定の場合: min(sort_order × 8, 30)
      - stamp_display_style = 'STANDARD' の場合: 0
   c. circulation_recipients を更新（status = STAMPED, stamped_at = NOW(), stamp_log_id, tilt_angle）
   d. circulation_documents.stamped_count をアトミック更新（+1）
   e. comment がある場合: circulation_comments に INSERT
   f. 完了判定:
      - PENDING が0件 → status = COMPLETED, completed_at = NOW()
   g. 未完了の場合:
      - 次のアンロック対象グループ（現在の最小 PENDING sort_order）のユーザー全員に通知送信
5. ApplicationEvent（CirculationStampedEvent）を発行
   - 作成者に WebSocket 通知（CIRCULATION_STAMPED: 「佐藤花子が押印しました（3/8）」）
   - COMPLETED の場合: 作成者に WebSocket 通知（CIRCULATION_COMPLETED）
```

```
2. 通知対象を決定:

   c. 投稿者自身を除外

      閲覧は可能だがコメント通知は除外（ノイズ回避）
3. F04.3 プッシュ通知で送信:
   - 「[回覧板] {commenter_name} がコメントしました: {title}」
```

#### 自動リマインド通知（バッチ）
```
1. Spring @Scheduled バッチが1時間間隔で実行
2. circulation_documents から以下の条件で取得:
   - status = IN_PROGRESS
   - reminder_enabled = true
3. 各回覧について、現在アンロック済みのグループの PENDING メンバーを取得:
   - 最小の PENDING sort_order を持つグループのメンバー
4. 各未確認メンバーについて:
   a. last_reminded_at が NULL、または NOW() - last_reminded_at >= reminder_interval_hours の場合:

   （前のグループの完了を待つ設計のため）
```

#### 回覧の取消
```
1. 作成者 / ADMIN が POST /circulation/{id}/cancel を呼び出し
2. バックエンドが以下を検証:
   a. status が DRAFT または IN_PROGRESS であること
3. status = CANCELLED に更新
4. ApplicationEvent（CirculationCancelledEvent）を発行
   - IN_PROGRESS から取消の場合: 全回覧先メンバーに通知（「回覧が取り消されました」）
5. 押印済みの seal_stamp_logs は取消しない（証跡保持。回覧自体の取消と押印の取消は独立）
```

### 重要な判定ロジック

- **sort_order グループによる押印順序管理**: 同じ `sort_order` を持つメンバーは同時に押印可能。前のグループ（より小さい `sort_order`）に PENDING が0件になった時点で次のグループがアンロックされる。この統一ルールが3モード全てを実現:
  - SEQUENTIAL: 全員異なる sort_order → 1人ずつ順番
  - HYBRID: 先頭 N 人が異なる sort_order + 残り全員が同じ sort_order → 途中まで順番、以降一斉
  - SIMULTANEOUS: 全員 sort_order = 0 → 全員同時
- **HYBRID の sort_order 割当**: `recipient_ids = [A, B, C, D, E]` で `sequential_count = 2` の場合: A=0, B=1, C=2, D=2, E=2。A→B は順番、B の押印完了後に C/D/E が同時アンロック
- **完了判定**: `circulation_recipients` に `status = PENDING` のレコードが0件になった時点で完了。`stamped_count` ではなく `PENDING が0件` で判定する（SKIPPED を考慮）
- **閲覧と押印の分離**: SEQUENTIAL / HYBRID モードでも回覧先メンバー全員が回覧内容を閲覧可能。押印操作のみ sort_order による順番制約を適用。理由: 紙の回覧板でも「先に内容だけ確認しておく」ことは一般的であり、急ぎの周知事項が順番待ちでブロックされることを防ぐ
- **DRAFT の編集制限**: DRAFT の回覧のみ編集可能。IN_PROGRESS 以降は title / body / recipients の変更不可（押印済みの内容を事後変更すると証跡の信頼性が損なわれるため）
- **削除制限**: DRAFT と CANCELLED のみ論理削除可能。IN_PROGRESS / COMPLETED は削除不可（証跡保持）
- **強制完了**: ADMIN が `force-complete` を実行すると、全 PENDING ユーザーを SKIPPED に変更し、`status = COMPLETED` に遷移。監査ログに記録
- **ユーザー退会時の制約**: 回覧先メンバーが退会する場合、PENDING 状態の回覧があると退会をブロック。ADMIN が先に SKIPPED に変更するか回覧を完了/取消する必要がある
- **デフォルト印鑑の解決**: 押印時に `seal_id` が省略された場合: (1) `seal_scope_defaults` でスコープ別デフォルトを検索 → (2) なければグローバルデフォルト（`scope_type = DEFAULT`）→ (3) なければ `LAST_NAME` バリアントの印鑑
- **お辞儀ハンコの角度計算**: `stamp_display_style = 'OJIGI'` の場合、デフォルト傾き角度 = `min(sort_order × 8, 30)` 度。sort_order = 0（最上位者）は 0°（直立）、1 は 8°、2 は 16°、3 は 24°、≥ 4 は 30°（上限）。ユーザーはフロントエンドのスライダーで -30°〜+30° の範囲内で自由に調整可能
- **お辞儀の傾き方向**: 正値 = 左傾き（日本の慣習で、上位者の印鑑に向かってお辞儀する方向）。フロントエンドで `transform: rotate({tilt_angle}deg)` を適用。STANDARD スタイルの場合は常に 0°
- **リマインド通知の制御**: `reminder_enabled = false` の回覧は自動リマインド対象外。手動リマインド（POST /remind）は `reminder_enabled` の値に関わらず実行可能。手動リマインドは1回覧あたり1時間に1回のレートリミット
- **MEMBER の回覧作成権限**: チーム設定 `allow_member_circulation`（デフォルト: `false`）が `true` の場合のみ MEMBER が回覧を作成可能。組織レベルの回覧は ADMIN / DEPUTY_ADMIN のみ
- **コメント通知のスレッド参加者パターン**: コメント通知は回覧作成者 + 過去にコメント済みのユーザーに送信。回覧先全員には送信しない（ノイズ回避）
- **期限超過回覧の管理**: 自動強制完了は実装しない。期限超過後もリマインド通知が継続し、ADMIN が `overdue=true` フィルタで期限超過回覧を一覧確認して手動で `force-complete` を判断する

## 6. セキュリティ考慮事項

- **認可チェック**: 回覧作成は ADMIN / DEPUTY_ADMIN（`MANAGE_CONTENT`）+ チーム設定による MEMBER 許可。押印は回覧先メンバーのみ。回覧詳細の閲覧は回覧先メンバー + 作成者 + ADMIN に限定。スコープメンバーシップは全 API で検証
- **押印の不可逆性**: 押印操作は取消不可（`seal_stamp_logs` の `revoke` は F05.3 側の機能であり、回覧板では非推奨。押印取消が必要な場合は回覧自体を取消して再作成する運用）
- **DRAFT 編集の証跡**: IN_PROGRESS 以降の編集を禁止することで、「押印した時点の内容」と「現在の内容」の乖離を防止。回覧内容の事後改ざんリスクを排除
- **レートリミット**: 回覧作成 API は Bucket4j で 1分間に5回。押印 API は 1分間に10回。手動リマインドは 1回覧あたり1時間に1回。コメント投稿は 1分間に20回
- **添付ファイルのバリデーション**: ファイル形式は Content-Type + マジックバイトの二重チェック。PDF は悪意のある JavaScript 埋め込みチェック（PDF.js ライブラリで事前パース）
- **Pre-signed URL**: 添付ファイルは CloudFront 経由の署名付き URL で配信（有効期限: 1時間）
- **PDF エクスポートのセキュリティ**: 生成 PDF にはフッターに検証用ハッシュ（全 stamp_log_id の SHA-256 連結ハッシュ）を付与。改ざん検出に使用。PDF 生成は非同期ジョブで実行し、DDoS リスクを軽減（1回覧あたり1分間に1回のレートリミット）
- **お辞儀角度のバリデーション**: `tilt_angle` は -30〜+30 の範囲でバックエンド側で強制クランプ。範囲外の値は拒否（400 Bad Request）
- **監査ログ**: 強制完了（`CIRCULATION_FORCE_COMPLETED`）、取消（`CIRCULATION_CANCELLED`）を `audit_logs` に記録。通常の押印は `seal_stamp_logs` が証跡を担うため `audit_logs` には記録しない
