## 1. 概要

チャット・投稿・掲示板・ファイルなど、流れていく情報を「新聞の切り抜き」感覚でカード化し、ドラッグ＆ドロップで自由に配置・整理できる視覚型ストックボード。個人/チーム/組織の3スコープ対応。

**主要機能**:
- **参照カード**: チャットメッセージ・タイムライン投稿・掲示板スレッド/返信・ブログ記事・ファイルをストック（`content_snapshot` で元削除後も内容保持）
- **メモカード**: ボード専用の独立テキストメモ
- **URLカード**: 外部URL + OGPサムネイル表示
- **見出しカード**: 横長タイトルバー（視覚的な仕切り線）
- **セクション**: 折りたたみ可能な名前付きコンテナ。カードをグループ化して整理
- **自由配置**: X/Y座標をDB保存、カラーラベリング（6色）、3段階サイズ
- **コンテキスト保持**: 参照カードクリック → サイドパネルで元コンテンツの前後5件を表示
- **付箋メモ**: 各カードに `user_note` でメモ追記
- **自動アーカイブ**: カードごとに `auto_archive_at` で期限設定

**フィーチャーフラグ制御**: Phase 1〜8 は `FEATURE_V9_ENABLED=false` により UI・API を完全非表示（503 Service Unavailable を返却）。DB テーブルは Phase 1 で先行作成。

## 2. スコープ

### 対象ロール

| ロール | 操作 |
|--------|------|
| ADMIN | 全ボード管理（個人・チーム・組織）。チーム/組織共有ボードの作成・削除・編集権限設定 |
| DEPUTY_ADMIN | `MANAGE_CORKBOARD` 権限保持時: チーム/組織共有ボードの作成・編集 |
| MEMBER | 個人ボードの CRUD。チーム/組織共有ボードの閲覧 + カード追加（編集権限が付与されている場合） |
| SUPPORTER | チーム/組織共有ボードの閲覧のみ（カード追加不可） |
| GUEST | 対象外 |

### 対象レベル

- [x] 組織 (Organization) — 組織共有ボード
- [x] チーム (Team) — チーム共有ボード
- [x] 個人 (Personal) — 個人ボードのみ作成・管理

### 共有ボードの編集権限モデル

チーム/組織の共有ボードは、ADMIN がボードごとに**編集権限** (`edit_policy`) を設定する:

| edit_policy | 説明 |
|-------------|------|
| `ADMIN_ONLY` | ADMIN / DEPUTY_ADMIN のみ編集可（デフォルト） |
| `ALL_MEMBERS` | MEMBER 以上の全員がカード追加・移動・削除可 |

> SUPPORTER は `edit_policy` に関わらず閲覧のみ。GUEST はアクセス不可。

## 3. DB設計

### テーブル一覧

| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `corkboards` | ボードマスター（スコープ・背景・デフォルト設定） | あり（`deleted_at`） |
| `corkboard_cards` | カード（位置・色・種別・参照先・スナップショット） | あり（`deleted_at`） |
| `corkboard_groups` | セクション（折りたたみ可能な名前付きコンテナ） | なし（物理削除） |
| `corkboard_card_groups` | カードとセクションの中間テーブル | なし（物理削除） |

### テーブル定義

#### `corkboards`

ボードマスター。個人・チーム・組織それぞれに複数ボードを作成可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | `PERSONAL` / `TEAM` / `ORGANIZATION` |
| `scope_id` | BIGINT UNSIGNED | YES | NULL | `TEAM` → FK teams（ON DELETE RESTRICT）/ `ORGANIZATION` → FK organizations（ON DELETE RESTRICT）/ `PERSONAL` → NULL。チーム/組織削除前にボードの論理削除が必要 |
| `owner_id` | BIGINT UNSIGNED | YES | NULL | `PERSONAL` の場合: FK → users（ON DELETE SET NULL）。ユーザー物理削除後は NULL → 孤立した個人ボード（`scope_type = PERSONAL` かつ `owner_id IS NULL`）はユーザー削除バッチで物理削除。チーム/組織の場合: NULL |
| `name` | VARCHAR(100) | NO | — | ボード名（例: 「来週の準備」「重要連絡まとめ」） |
| `background_style` | VARCHAR(10) | NO | 'CORK' | `CORK`（コルク風テクスチャ）/ `WHITE` / `DARK` |
| `edit_policy` | VARCHAR(20) | NO | 'ADMIN_ONLY' | `ADMIN_ONLY` / `ALL_MEMBERS`（PERSONAL ボードでは無視） |
| `is_default` | BOOLEAN | NO | FALSE | スコープごとに最大1つ。初回アクセス時に自動表示するボード |
| `version` | BIGINT | NO | 0 | 楽観的ロック |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_corkboards_personal (owner_id, deleted_at)                         -- 個人ボード一覧
INDEX idx_corkboards_scope (scope_type, scope_id, deleted_at)                -- チーム/組織ボード一覧
```

**制約・備考**
- `scope_type = PERSONAL` の場合: `owner_id` 必須、`scope_id` = NULL
- `scope_type = TEAM / ORGANIZATION` の場合: `scope_id` 必須、`owner_id` = NULL
- `is_default` の排他制御: **アプリ層のみで制御**（INSERT/UPDATE 時にトランザクション内で既存デフォルトを `false` に更新してから設定）。MySQL の UNIQUE INDEX は `scope_id` / `owner_id` が NULL を含むためユニーク制約として機能しない（MySQL では `NULL != NULL`）ため、DB 制約は設けない
- 個人ボード数: 1ユーザーあたり最大20ボード
- チーム/組織ボード数: 1スコープあたり最大10ボード

#### `corkboard_cards`

カード。ボードに配置される個々のアイテム。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `board_id` | BIGINT UNSIGNED | NO | — | FK → corkboards（ON DELETE CASCADE。論理削除ではなく物理削除時のみ発動。通常運用では corkboards は論理削除のためカードは残存し、ボードの deleted_at 判定で間接的に非表示になる） |
| `card_type` | VARCHAR(20) | NO | — | `REFERENCE` / `MEMO` / `URL` / `SECTION_HEADER` |
| `ref_type` | VARCHAR(30) | YES | NULL | `CHAT_MESSAGE` / `TIMELINE_POST` / `BULLETIN_THREAD` / `BLOG_POST` / `FILE`（`card_type = REFERENCE` のみ。新機能追加時に拡張可能） |
| `ref_id` | BIGINT UNSIGNED | YES | NULL | 参照先レコードの ID（`card_type = REFERENCE` のみ） |
| `content_snapshot` | TEXT | YES | NULL | 参照元の内容スナップショット（最大5,000文字。元が削除されても残る） |
| `ref_url` | VARCHAR(2048) | YES | NULL | `card_type = URL` の外部URL |
| `ogp_title` | VARCHAR(200) | YES | NULL | OGP タイトル（URL カード用） |
| `ogp_image_url` | VARCHAR(2048) | YES | NULL | OGP サムネイル URL（URL カード用） |
| `title` | VARCHAR(200) | YES | NULL | カードタイトル（MEMO / URL / SECTION_HEADER で使用） |
| `memo_content` | TEXT | YES | NULL | メモ本文（`card_type = MEMO` のみ。最大3,000文字） |
| `pos_x` | INT | NO | 0 | X座標（ピクセル） |
| `pos_y` | INT | NO | 0 | Y座標（ピクセル） |
| `card_width` | INT | NO | 200 | カード幅（ピクセル） |
| `card_height` | INT | NO | 150 | カード高さ（ピクセル） |
| `card_size` | VARCHAR(10) | NO | 'MEDIUM' | `SMALL` / `MEDIUM` / `LARGE`（プリセットサイズ。card_width/card_height と連動） |
| `color` | VARCHAR(10) | NO | 'WHITE' | `WHITE` / `YELLOW` / `RED` / `BLUE` / `GREEN` / `PURPLE` |
| `user_note` | TEXT | YES | NULL | 付箋メモ（ユーザーが追記するメモ。最大1,000文字） |
| `is_pinned` | BOOLEAN | NO | FALSE | ピン留め。`true` の場合、編集権限を持つユーザーでも位置移動不可（ADMIN のみピン留め設定/解除可） |
| `is_ref_deleted` | BOOLEAN | NO | FALSE | 参照先削除済みフラグ（`card_type = REFERENCE` のみ。デッドリファレンス検知バッチで設定） |
| `ref_updated_at` | DATETIME | YES | NULL | 参照カード作成時の元コンテンツ `updated_at`（参照元の更新検知用。現在の元コンテンツの `updated_at` と比較して差異があれば「元の内容が更新されています」バッジを表示） |
| `auto_archive_at` | DATETIME | YES | NULL | 自動アーカイブ日時（NULL = アーカイブしない） |
| `is_archived` | BOOLEAN | NO | FALSE | アーカイブ済みフラグ |
| `sort_order` | INT | NO | 0 | セクション内での表示順（セクション未所属の場合は 0） |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL。カード作成者。ユーザー物理削除後は NULL → フロントエンドで「退会済みユーザー」表示） |
| `version` | BIGINT | NO | 0 | 楽観的ロック（位置更新 PATCH は LWW のため version 不使用。内容更新 PUT のみ使用） |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_cc_board (board_id, is_archived, deleted_at)        -- ボード内カード一覧
INDEX idx_cc_ref (ref_type, ref_id)                           -- 参照先からの逆引き（元コンテンツ削除時のスナップショット確認）
INDEX idx_cc_archive (is_archived, auto_archive_at)           -- 自動アーカイブバッチ用
INDEX idx_cc_ref_deleted (card_type, is_ref_deleted, deleted_at)  -- デッドリファレンス検知バッチ用
INDEX idx_cc_created_by (created_by)                          -- ユーザー別のカード一覧
FULLTEXT INDEX ft_cc_memo (memo_content)                      -- F04.6 検索連携: メモカードの全文検索用
```

**制約・備考**
- `SECTION_HEADER` カード: `title` のみ使用し、`ref_type` / `ref_id` / `content_snapshot` / `memo_content` / `ref_url` は NULL。横長タイトルバーとして描画
- `REFERENCE` カード: `ref_type` + `ref_id` 必須。`content_snapshot` はカード作成時に元コンテンツの先頭5,000文字を保存
- `URL` カード: `ref_url` 必須。OGP 情報はカード作成時に非同期取得
- `MEMO` カード: `memo_content` に本文を保存
- `card_size` プリセット: `SMALL`=150x100, `MEDIUM`=200x150, `LARGE`=300x200。ユーザーがサイズを選択すると `card_width` / `card_height` も連動更新
- 1ボードあたりカード上限: 200枚（アーカイブ済み含まず）
- `content_snapshot` は作成時に1回だけ取得。元コンテンツが更新されても同期しない（スナップショットの性質）
- `pos_x` / `pos_y` は 0 以上の値のみ許可（バリデーション。負の座標はボード描画領域外になるため拒否）
- `SECTION_HEADER` カードはセクションへの所属を禁止（`corkboard_card_groups` に INSERT 不可。見出しカードはセクションの外で独立して配置するもの）
- `created_by = NULL`（ユーザー物理削除後）のカードはフロントエンドで「退会済みユーザー」と表示。カード自体は削除しない（共有ボードのコンテンツはボードに帰属）
- `is_pinned = true` のカードは位置更新 API（`PATCH .../position` / `batch-position`）を拒否。ピン留めの設定/解除は ADMIN のみ（共有ボードで重要カードの誤移動を防止）
- `is_ref_deleted`: デッドリファレンス検知バッチ（日次）が `ref_type + ref_id` の存在を確認し、元が論理削除済みの場合に `true` を設定。フロントエンドで「元コンテンツ削除済み」バッジを一覧段階で表示（クリック前に判断可能）
- `ref_updated_at`: 参照カード作成時に元コンテンツの `updated_at` を保存。フロントエンドがコンテキスト API 呼び出し時に現在の元コンテンツの `updated_at` と比較し、差異があれば「元の内容が更新されています」バッジを表示（スナップショットの鮮度をユーザーが認識可能）

#### `corkboard_groups`

セクション（折りたたみ可能な名前付きコンテナ）。カードをグループ化して整理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `board_id` | BIGINT UNSIGNED | NO | — | FK → corkboards（ON DELETE CASCADE） |
| `name` | VARCHAR(100) | NO | — | セクション見出し（例: 「来週の議題」「決定事項まとめ」） |
| `pos_x` | INT | NO | 0 | X座標 |
| `pos_y` | INT | NO | 0 | Y座標 |
| `group_width` | INT | NO | 400 | セクション幅 |
| `group_height` | INT | NO | 300 | セクション高さ |
| `color` | VARCHAR(20) | NO | 'TRANSPARENT' | `TRANSPARENT` / `LIGHT_YELLOW` / `LIGHT_BLUE` / `LIGHT_GREEN` / `LIGHT_PURPLE` |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_cg_board (board_id)  -- ボード内セクション一覧
```

**制約・備考**
- 1ボードあたりセクション上限: 20個
- セクション削除時: 所属カードの `corkboard_card_groups` レコードを物理削除（カードはセクション未所属に戻る）
- セクションはネスト不可（フラットな1階層のみ）
- 折りたたみ状態はフロントエンドの localStorage で管理（`corkboard:collapse:{boardId}` → `{ "sectionId": true/false }`）。DB に保存しない（ユーザーごとの表示設定であり、ビジネスロジックに影響しないため）

---

#### `corkboard_card_groups`

カードとセクションの中間テーブル。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `card_id` | BIGINT UNSIGNED | NO | — | FK → corkboard_cards（ON DELETE CASCADE） |
| `group_id` | BIGINT UNSIGNED | NO | — | FK → corkboard_groups（ON DELETE CASCADE） |

**制約**
```sql
PRIMARY KEY (card_id)  -- 1カードは最大1セクションに所属
INDEX idx_ccg_group (group_id)  -- セクション内カード一覧用（FK インデックスの明示化）
CONSTRAINT fk_ccg_card FOREIGN KEY (card_id) REFERENCES corkboard_cards(id) ON DELETE CASCADE
CONSTRAINT fk_ccg_group FOREIGN KEY (group_id) REFERENCES corkboard_groups(id) ON DELETE CASCADE
```

**備考**
- 1枚のカードは最大1つのセクションに所属（`card_id` を PK にすることで強制）
- カードをセクション間で移動: 旧レコードを UPDATE（DELETE + INSERT ではなく）
- カードをセクションから外す: レコードを DELETE

## 5. ビジネスロジック

### 主要フロー

#### 参照カード作成フロー
```
1. ユーザーがチャット / タイムライン / 掲示板 / ブログ / ファイルで「コルクボードに保存」を選択
2. フロントエンドがボード選択ダイアログを表示（個人ボード一覧 + 所属チーム/組織の共有ボード）
3. POST /corkboards/{id}/cards に ref_type + ref_id を送信
4. バックエンド処理:
   a. ボードの編集権限チェック
   b. カード数上限チェック（200枚）
   c. ref_type + ref_id の存在確認 + アクセス権チェック
      - CHAT_MESSAGE: チャットチャネルのメンバーか
      - TIMELINE_POST: 投稿の visibility に基づくアクセス権
      - BULLETIN_THREAD: スレッドの visibility + スコープメンバーシップ
      - BLOG_POST: 公開記事 or 下書きの作成者
      - FILE: ファイルのスコープへのアクセス権
   d. content_snapshot を生成:
      - CHAT_MESSAGE: メッセージ本文の先頭5,000文字
      - TIMELINE_POST: 投稿本文の先頭5,000文字
      - BULLETIN_THREAD: スレッドタイトル + 本文の先頭5,000文字
      - BLOG_POST: 記事タイトル + 本文の先頭5,000文字
      - FILE: ファイル名 + 説明文
   e. corkboard_cards に INSERT
5. 201 Created を返却
```

#### URLカード作成フロー（OGP取得）
```
1. ユーザーがURLカードを作成
2. POST /corkboards/{id}/cards に ref_url を送信
3. バックエンド処理:
   a. URL のバリデーション（形式チェック + プライベート IP / localhost 拒否）
   b. corkboard_cards に INSERT（ogp_title / ogp_image_url は NULL）
   c. 非同期で OGP 情報を取得:
      - HTTP HEAD → Content-Type チェック（text/html のみ処理）
      - HTTP GET（最大1MB、タイムアウト5秒）
      - <meta property="og:title"> / <meta property="og:image"> をパース
      - ogp_title / ogp_image_url を UPDATE
      - 取得失敗時: ログ出力のみ（カードは作成済み。タイトル・サムネイルなしで表示）
      - 取得成功時: 共有ボードの場合は WebSocket で CARD_UPDATED イベントをブロードキャスト（OGP 情報付き）
4. 201 Created を返却（OGP 未取得状態）
5. フロントエンド:
   - 共有ボード: WebSocket の CARD_UPDATED イベントで OGP 情報を自動反映
   - 個人ボード: 5秒後にカード情報を再取得（ポーリング1回）して OGP を反映
```

#### コンテキスト表示フロー（参照カード）
```
1. ユーザーが参照カードをクリック
2. フロントエンドがサイドパネルを表示
3. ref_type に応じたコンテキスト API を呼び出し:
   - CHAT_MESSAGE: GET /chat/messages/{ref_id}/context?count=5
   - TIMELINE_POST: GET /timeline-posts/{ref_id}/context?count=5
   - BULLETIN_THREAD: GET /bulletin-threads/{ref_id}/context?count=5
   - BLOG_POST / FILE: コンテキスト不要（元コンテンツへのリンクを表示）
4. バックエンド: ref_id の前後5件を created_at 順で取得
5. レスポンス: 元コンテンツ + 前後のコンテンツリスト + 元コンテンツへのディープリンク
6. 元コンテンツが削除済みの場合:
   - コンテキスト API は 404 を返却
   - フロントエンドは content_snapshot をサイドパネルに表示 + 「元のコンテンツは削除されました」を表示
```

#### 自動アーカイブバッチ
```
1. Spring @Scheduled バッチが毎時0分に実行
2. corkboard_cards から以下を検索:
   - is_archived = false
   - auto_archive_at IS NOT NULL
   - auto_archive_at <= NOW()
   - deleted_at IS NULL
3. 該当カードの is_archived = true に一括 UPDATE
4. バッチサイズ: 500件ずつ処理
5. 処理件数をアプリケーションログに記録
6. 共有ボードのカードがアーカイブされた場合: WebSocket で CARD_ARCHIVED イベントをブロードキャスト（ボードを開いているユーザーに即座に反映）
```

#### カード複製フロー
```
1. ユーザーがカードの「複製」を選択
2. POST /corkboards/{id}/cards/{cardId}/duplicate を呼び出し
3. バックエンド処理:
   a. カード数上限チェック（200枚）
   b. 元カードの全フィールドをコピー（以下を除く）:
      - id: 新規採番
      - pos_x / pos_y: 元カードから +20px オフセット
      - is_pinned: false（複製カードはピン留めしない）
      - is_archived: false
      - version: 0
      - created_by: 複製操作者
      - created_at / updated_at: 現在時刻
   c. corkboard_card_groups: 元カードがセクションに所属していた場合、複製カードも同じセクションに追加
   d. corkboard_cards に INSERT
4. 201 Created を返却
5. 共有ボード: WebSocket で CARD_CREATED イベントをブロードキャスト
```

#### デッドリファレンス検知バッチ
```
1. Spring @Scheduled バッチが日次（03:00 JST）で実行
2. corkboard_cards から card_type = 'REFERENCE' AND is_ref_deleted = false AND deleted_at IS NULL を検索
3. 各カードの ref_type + ref_id に対して元テーブルを確認:
   - CHAT_MESSAGE: chat_messages WHERE id = ref_id AND deleted_at IS NULL
   - TIMELINE_POST: timeline_posts WHERE id = ref_id AND deleted_at IS NULL
   - BULLETIN_THREAD: bulletin_threads WHERE id = ref_id AND deleted_at IS NULL
   - BLOG_POST: blog_posts WHERE id = ref_id AND deleted_at IS NULL
   - FILE: shared_files WHERE id = ref_id AND deleted_at IS NULL
4. 元が論理削除済み or 物理削除済みの場合: is_ref_deleted = true に UPDATE
5. バッチサイズ: 500件ずつ処理（ref_type ごとに IN 句でバッチ確認）
6. 処理件数をアプリケーションログに記録
```

#### 論理削除カード物理削除バッチ
```
1. Spring @Scheduled バッチが日次（04:00 JST）で実行
2. corkboard_cards から deleted_at < NOW() - INTERVAL 90 DAY を検索
3. corkboard_card_groups の紐付けを物理削除（FK CASCADE で自動）
4. corkboard_cards を物理削除
5. バッチサイズ: 500件ずつ DELETE
6. 処理件数をアプリケーションログに記録
7. 同様に corkboards の deleted_at < NOW() - INTERVAL 90 DAY も物理削除
   （CASCADE により所属カード・セクション・中間テーブルが連動削除）
```

#### ボード削除フロー
```
1. ADMIN が DELETE /corkboards/{id} を呼び出し
2. バックエンド:
   a. 権限チェック（個人ボード: 本人のみ / 共有ボード: ADMIN のみ。DEPUTY_ADMIN はボード削除不可 — README.md「コンテンツ削除のデフォルト非付与」ポリシーに準拠）
   b. corkboards.deleted_at = NOW() に更新（論理削除）
   c. 所属カード・セクションは CASCADE で論理削除しない
      （ボードが論理削除されている場合、カード/セクションの API は 404 を返す）
   d. 共有ボードの場合: WebSocket で BOARD_DELETED イベントをブロードキャスト
      → フロントエンドはボード画面を閉じ「このボードは削除されました」を表示
3. 200 OK（deleted_at 含む）を返却
```

### 重要な判定ロジック

- **フィーチャーフラグ制御**: `FEATURE_V9_ENABLED=false` の場合、全 corkboard API は 503 Service Unavailable を返却。`@ConditionalOnProperty` でコントローラー自体を無効化するか、フィルターで制御
- **編集権限チェック（共有ボード）**: `edit_policy = ADMIN_ONLY` → ADMIN / DEPUTY_ADMIN のみ。`ALL_MEMBERS` → MEMBER 以上。SUPPORTER は常に閲覧のみ。GUEST はアクセス不可。**編集権限があるユーザーはボード内の全カードを編集・移動・削除可能**（自分が作成したカードに限らない。共有ホワイトボードのメタファー）。カード単位の所有権チェックは行わない
- **個人ボードのアクセス制御**: `scope_type = PERSONAL` のボードは `owner_id` = ログインユーザーのみアクセス可能。SYSTEM_ADMIN は全ボード参照可
- **参照カードの重複許可**: 同一 `ref_type + ref_id` のカードを同一ボードに複数作成可能（異なるメモや色で区別するユースケース）
- **content_snapshot の不変性**: 一度保存した `content_snapshot` は更新しない。元コンテンツが編集されてもスナップショットは作成時の内容を保持する
- **URL カードの SSRF 対策**: `ref_url` のホスト部分を検証し、プライベート IP（10.x.x.x / 172.16-31.x.x / 192.168.x.x）/ localhost / link-local アドレスへのリクエストを拒否。DNS リバインド対策として、名前解決後の IP も検証
- **カードサイズのプリセット連動**: `card_size` を変更すると `card_width` / `card_height` が自動設定される。`SMALL`=150x100, `MEDIUM`=200x150, `LARGE`=300x200。カード種別ごとのデフォルトサイズ: `REFERENCE`→MEDIUM, `MEMO`→LARGE, `URL`→SMALL, `SECTION_HEADER`→幅はボード幅の80%固定。フリーリサイズは対応しない（リサイズハンドル・最小/最大制約・重なり検知の実装コストに対し、プリセット3種で主要ユースケースをカバーできるため。DB の `card_width`/`card_height` カラムは存在するため、将来フリーリサイズ対応時もマイグレーション不要）
- **is_default の排他制御**: `is_default = true` に設定する際、同一スコープ（`scope_type + scope_id + owner_id` の組み合わせ）の既存デフォルトを `false` に更新してから設定。トランザクション内で実行
- **アーカイブカードの扱い**: アーカイブ済みカードはデフォルトの一覧から除外。`is_archived = true` のカードはボード上に表示しないが、フィルタで閲覧可能。アーカイブ解除で復帰
- **ピン留めカードの保護**: `is_pinned = true` のカードは位置更新（単体・一括）を拒否（`CORK_010`）。内容更新（PUT）は可能。ピン留め設定/解除は ADMIN のみ。共有ボードで重要カードの誤移動を防止。一括操作（batch）ではスキップして `skipped_count` で通知
- **デッドリファレンス検知**: 日次バッチで `is_ref_deleted` フラグを自動設定。フロントエンドはカード一覧で「元コンテンツ削除済み」バッジを表示。ユーザーはアーカイブするか、`content_snapshot` のまま残すかを判断可能
- **参照元の更新検知**: `ref_updated_at`（カード作成時の元 `updated_at`）をコンテキスト API の応答と比較。差異がある場合「元の内容が更新されています」バッジを表示。自動同期はしない（スナップショットの不変性を維持）
- **一括操作（batch）の WebSocket**: 一括操作は個別イベント（CARD_ARCHIVED × N 件等）としてブロードキャスト。専用のバッチイベントは作らない（フロントエンドの既存ハンドラーを流用し、実装コストを抑制）
- **論理削除カードの物理削除**: 日次バッチで `deleted_at` から90日経過したカード・ボードを物理削除。F05.1 掲示板の既読データ削除と同一パターン。インデックス効率とテーブルサイズの肥大化を抑制
- **セクション折りたたみ（localStorage）**: 折りたたみ状態はフロントエンドの localStorage（`corkboard:collapse:{boardId}`）で管理。DB に保存しない。理由: (1) 純粋な表示設定でありビジネスロジックに影響しない (2) トグル操作が頻繁で毎回 API コールすると UX 悪化 (3) クロスデバイス同期の価値が低い。折りたたみ時はタイトルバーのみ描画、所属カードは非表示（データは保持）
- **他機能からの連携**: F04.1 タイムライン / F04.2 チャット / F05.1 掲示板の各 UI に「コルクボードに保存」ボタンを追加。ApplicationEvent は不要（ユーザー操作トリガーのため API 直接呼び出し）

### リアルタイム同期（WebSocket / STOMP）

共有ボード（`scope_type = TEAM / ORGANIZATION`）で複数ユーザーが同時編集する場合のリアルタイム同期。F04.2 チャット / F04.1 タイムラインで構築済みの STOMP + Redis Pub/Sub 基盤を再利用する。

> **個人ボード（`scope_type = PERSONAL`）は WebSocket 不要**（単一ユーザーのため）。

#### STOMP Destination

```
/topic/corkboard/{boardId}    → 共有ボードの変更ブロードキャスト（SUBSCRIBE）
```

#### イベント種別

| イベント | ペイロード | 発火タイミング |
|---------|-----------|--------------|
| `CARD_CREATED` | `{ card: { id, card_type, pos_x, pos_y, ... }, createdBy: { id, display_name } }` | カード作成後 |
| `CARD_MOVED` | `{ cardId, pos_x, pos_y, movedBy: { id, display_name } }` | 位置更新後 |
| `CARD_UPDATED` | `{ card: { id, ... } }` | カード内容更新後 |
| `CARD_DELETED` | `{ cardId }` | カード削除後 |
| `CARD_ARCHIVED` | `{ cardId }` | カードアーカイブ後 |
| `SECTION_CREATED` | `{ section: { id, name, pos_x, pos_y, ... } }` | セクション作成後 |
| `SECTION_UPDATED` | `{ section: { id, ... } }` | セクション更新後 |
| `SECTION_DELETED` | `{ sectionId }` | セクション削除後 |
| `CARD_SECTION_CHANGED` | `{ cardId, oldSectionId, newSectionId }` | カードのセクション移動後 |
| `BOARD_DELETED` | `{ boardId }` | ボード論理削除後（閲覧中ユーザーにボード閉鎖を通知） |

#### ブロードキャスト方式

```
1. API リクエスト成功後、Service 層で ApplicationEvent を発行
2. @TransactionalEventListener(phase = AFTER_COMMIT) でコミット後にリスナーが起動
3. SimpMessagingTemplate を使用して /topic/corkboard/{boardId} にブロードキャスト
4. Redis Pub/Sub 経由でマルチインスタンス間を伝搬（F04.2 と同一基盤）
5. 操作者自身にはメッセージを送らない（sessionId で除外。自分の操作は API レスポンスで即座に反映済み）
```

#### 切断時の復帰

```
1. WebSocket 切断を検知
2. 指数バックオフで再接続（1s → 2s → 4s → 8s → max 30s。F04.2 と同一）
3. 再接続成功後: GET /corkboards/{id} で全データ再取得（差分同期ではなくフルリロード）
   - チャットと異なり、ボード全体が1画面に収まるため cursor-based catchup は不要
   - 最大200カード + 20セクションのレスポンスサイズは十分小さい
4. フロントエンド: 取得データで画面を再描画
```

#### Redis キー

```
mannschaft:ws:corkboard:{boardId}:viewers    -- 現在閲覧中のユーザー数（SET）
TTL: なし（接続時に SADD、切断時に SREM）
用途: 「N人が閲覧中」表示（将来の拡張。Phase 9 初版では未使用）
注意: アプリクラッシュ時に SREM が呼ばれず SET が肥大化するリスクあり。
      利用開始時は日次バッチで全 viewers SET を SCAN → WebSocket セッション不在の userId を SREM するクリーンアップを追加すること
```

## 6. セキュリティ考慮事項

- **SSRF 防止（URL カード）**: 外部 URL の OGP 取得時にプライベート IP / localhost / link-local アドレスへのアクセスを拒否。DNS リバインド対策として名前解決後の IP も検証。HTTP リダイレクト先も検証（最大3ホップ）
- **ユーザー入力フィールドの XSS 防止**: `content_snapshot` / `memo_content` / `user_note` / `title` は全てプレーンテキストとして保存（HTML タグ除去）。フロントエンドでの表示時は `v-text`（Vue の自動エスケープ）を使用。`v-html` での表示は禁止
- **ref_url の表示**: URLカードの URL はユーザー入力。表示時は `<a>` タグの `href` にそのまま設定せず、`javascript:` / `data:` スキームを拒否するサニタイズを実施
- **アクセス制御**: ボードの `scope_type` に基づくメンバーシップ検証。個人ボードは `owner_id` 一致のみ。チーム/組織ボードはスコープのメンバーシップを検証
- **カード作成時の参照先権限**: `ref_type + ref_id` の参照先に対するユーザーのアクセス権を確認。アクセス権のないコンテンツをカード化できないようにする
- **OGP 取得のリソース制限**: タイムアウト5秒、レスポンスボディ最大1MB、Content-Type が `text/html` の場合のみパース。リダイレクトは最大3ホップ。URL カード作成のレートリミット: 1ユーザー1分あたり10回（OGP 取得の外部 HTTP リクエストによるサーバー負荷を抑制）
- **フィーチャーフラグ**: `FEATURE_V9_ENABLED=false` 時は API 自体を無効化。テーブルが存在しても一切のデータ操作を許可しない
- **WebSocket 認可**: STOMP SUBSCRIBE 時に `ChannelInterceptor` でボードの閲覧権限を検証。スコープ外ユーザーの購読は ERROR フレームで拒否（F04.2 チャットと同一パターン）
- **WebSocket レートリミット**: メッセージ受信レート 5/秒（Bucket4j。F04.2 と同一設定）
- **監査ログ**: `CORKBOARD_CREATED` / `CORKBOARD_DELETED` / `CORKBOARD_CARD_CREATED` を記録（共有ボードの操作のみ。個人ボードは記録しない）
- **ストレージクォータ**: `content_snapshot`（最大5,000文字 × 200枚 = 最大約1MB/ボード）は `docs/cross-cutting/storage_quota.md` の対象外（テキストデータのみでファイルストレージを消費しないため）。個人ボード20個 × 1MB = 最大約20MB/ユーザー。Phase 9 時点では十分小さいが、将来的にクォータ対象への組み込みを検討する場合は storage_quota.md に追記すること
