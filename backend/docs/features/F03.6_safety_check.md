# F03.6: 緊急安否確認

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 3
> **最終更新**: 2026-03-13

---

## 1. 概要

災害・緊急事態発生時に、管理者がワンクリックで所属メンバー全員に安否確認を一斉送信し、メンバーの回答をリアルタイムに集計・表示するシステム。プッシュ通知 + WebSocket でブロッキングモーダルを表示し、GPS 位置情報の任意送信やプリセット例文による迅速な回答を可能にする。自動的に掲示板の専用スレッドを生成し、安否ステータスをリアルタイム掲載する。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | プリセット例文の管理（CRUD） |
| ADMIN | 安否確認の実行・クローズ・結果閲覧・リマインド設定 |
| DEPUTY_ADMIN | `MANAGE_SAFETY_CHECK` 権限を持つ場合: 安否確認の実行・クローズ・結果閲覧・リマインド |
| MEMBER | 安否回答（個別・一括）、自分の回答履歴閲覧 |
| SUPPORTER | 安否回答（個別・一括）、自分の回答履歴閲覧 |
| GUEST | 対象外（安否確認の送信・回答いずれも不可） |

### 対象レベル
- [x] 組織 (Organization) — 組織全体への安否確認
- [x] チーム (Team) — チーム単位での安否確認
- [x] 個人 (Personal) — 個人としての回答（複数スコープへの一括回答含む）

### テーブル定義

#### `safety_checks`

安否確認セッション。ADMIN が実行するたびに 1 レコード生成され、対象スコープ内の全メンバーに通知が配信される。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION', 'GROUP') | NO | — | 安否確認の対象スコープ種別 |
| `scope_id` | BIGINT UNSIGNED | NO | — | 対象スコープの ID（teams.id / organizations.id / groups.id） |
| `title` | VARCHAR(200) | YES | NULL | 安否確認のタイトル（例: 「地震発生に伴う安否確認」。NULL の場合は「安否確認」） |
| `message` | VARCHAR(1000) | YES | NULL | 管理者からの補足メッセージ（例: 「震度5の地震が発生しました。全員の安否を確認します」）|
| `is_drill` | BOOLEAN | NO | FALSE | TRUE = 訓練モード。通知・モーダルに「【訓練】」を明示表示 |
| `status` | ENUM('ACTIVE', 'CLOSED') | NO | 'ACTIVE' | セッション状態。CLOSED = 回答受付終了 |
| `reminder_interval_minutes` | INT | YES | NULL | 自動リマインド間隔（分）。NULL = リマインドなし |
| `last_reminder_at` | DATETIME | YES | NULL | 最後にリマインドを送信した日時（リマインドバッチが参照） |
| `total_target_count` | INT | NO | 0 | 実行時の対象メンバー数（スナップショット。後からメンバーが増減しても変わらない） |
| `admin_24h_notified` | BOOLEAN | NO | FALSE | 24時間経過リマインドを ADMIN に送信済みか（1回のみ送信保証用） |
| `bulletin_thread_id` | BIGINT UNSIGNED | YES | NULL | FK → bulletin_threads（ON DELETE SET NULL）自動生成された専用掲示板スレッド |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）実行した ADMIN |
| `closed_at` | DATETIME | YES | NULL | クローズした日時 |
| `closed_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）クローズした ADMIN |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_sc_scope (scope_type, scope_id, created_at DESC)   -- スコープ別の安否確認一覧用
INDEX idx_sc_status (status)                                  -- ACTIVE セッションの抽出用
INDEX idx_sc_reminder (status, reminder_interval_minutes, last_reminder_at)  -- リマインドバッチ用
```

**制約・備考**
- `scope_type` + `scope_id` でポリモーフィック参照。FK 制約は設けず、アプリ層でバリデーション（teams / organizations / groups テーブルの存在確認）
- `total_target_count` は実行時点のメンバー数をスナップショットとして保存。回答率の分母として使用。後からスコープにメンバーが加入しても変動しない
- `bulletin_thread_id` ON DELETE SET NULL: 掲示板スレッドが削除されても安否確認レコードは残す
- 物理削除・論理削除ともに行わない。安否確認は監査記録として恒久的に保持する
- 同一スコープで ACTIVE なセッションは同時に複数存在可能（連続して地震が来る等のケース）

#### `safety_responses`

各メンバーの安否回答。1つの安否確認に対し、1ユーザー1回答（UPSERT で上書き可能）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `safety_check_id` | BIGINT UNSIGNED | NO | — | FK → safety_checks（ON DELETE CASCADE） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE） |
| `status` | ENUM('SAFE', 'NEED_SUPPORT', 'OTHER') | NO | — | 安否ステータス |
| `message` | VARCHAR(200) | YES | NULL | 任意メッセージ（プリセット選択 or 自由入力） |
| `message_source` | ENUM('PRESET', 'CUSTOM') | YES | NULL | メッセージの出典。NULL = メッセージなし |
| `gps_shared` | BOOLEAN | NO | FALSE | GPS 位置情報を送信したか |
| `gps_latitude` | DECIMAL(10,7) | YES | NULL | 緯度（`gps_shared = true` の場合のみ値あり） |
| `gps_longitude` | DECIMAL(10,7) | YES | NULL | 経度（`gps_shared = true` の場合のみ値あり） |
| `responded_at` | DATETIME | NO | CURRENT_TIMESTAMP | 回答日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_sr_check_user (safety_check_id, user_id)   -- 1ユーザー1回答
INDEX idx_sr_user (user_id)                                -- ユーザー別回答履歴用
INDEX idx_sr_status (safety_check_id, status)              -- ステータス別集計用
```

**制約・備考**
- `safety_check_id` ON DELETE CASCADE: 安否確認セッション削除時にカスケード（通常は物理削除しない運用だが、万が一の場合に対応）
- `user_id` ON DELETE CASCADE: ユーザー退会時に回答レコードも削除（集計値は `total_target_count` で分母を保持しているため、退会による集計値のずれは許容）
- UPSERT 対応: 回答修正は `ON DUPLICATE KEY UPDATE` で `status`, `message`, `message_source`, `gps_shared`, `gps_latitude`, `gps_longitude`, `responded_at` を上書き
- GPS 座標は DECIMAL(10,7) で小数点以下7桁（約1.1cm精度）。十分な精度を持つ
- `gps_shared = false` かつ `gps_latitude`/`gps_longitude` が NOT NULL というデータ不整合を防ぐため、アプリ層で `gps_shared = false` の場合は座標を NULL に強制

#### `safety_check_message_presets`

安否回答時にモーダルに表示するプリセット例文。SYSTEM_ADMIN が管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `body` | VARCHAR(200) | NO | — | プリセット例文（例: 「無事です。特に被害はありません」）|
| `sort_order` | INT | NO | 0 | 表示順 |
| `is_active` | BOOLEAN | NO | TRUE | FALSE = モーダルに表示しない |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_scmp_active_sort (is_active, sort_order)   -- アクティブなプリセットの表示順取得用
```

**制約・備考**
- SYSTEM_ADMIN のみ管理可能
- 初期データ（Flyway シード）として以下の 5 件を投入:
  1. 無事です。特に被害はありません
  2. 自宅にいます。問題ありません
  3. 避難中です。後ほど詳細を連絡します
  4. 軽微な被害がありますが無事です
  5. 現在移動中です。安全を確認しています
- 物理削除は行わない。`is_active = false` で非表示にする運用
- 最大200文字（VARCHAR(200)）。自由入力と同じ上限

#### `safety_check_templates`

安否確認の実行テンプレート。災害種別ごとに事前設定し、緊急時にワンタップで実行可能にする。SYSTEM_ADMIN がシステムデフォルトテンプレートを管理し、ADMIN がスコープ単位でカスタムテンプレートを追加できる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION', 'GROUP') | YES | NULL | NULL = システムデフォルト（全スコープで利用可能） |
| `scope_id` | BIGINT UNSIGNED | YES | NULL | NULL = システムデフォルト |
| `template_name` | VARCHAR(100) | NO | — | テンプレート名（例: 「地震」「台風」「火災」「不審者」） |
| `title` | VARCHAR(200) | YES | NULL | 安否確認タイトルのテンプレート |
| `message` | VARCHAR(1000) | YES | NULL | 補足メッセージのテンプレート |
| `reminder_interval_minutes` | INT | YES | NULL | リマインド間隔のデフォルト値 |
| `is_system_default` | BOOLEAN | NO | FALSE | TRUE = SYSTEM_ADMIN 管理のシステムデフォルト |
| `sort_order` | INT | NO | 0 | 表示順 |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_sct_scope (scope_type, scope_id, sort_order)             -- スコープ別テンプレート一覧用
INDEX idx_sct_system_default (is_system_default, sort_order)       -- システムデフォルト一覧用
```

**制約・備考**
- システムデフォルト（`is_system_default = true`, `scope_type = NULL`, `scope_id = NULL`）は SYSTEM_ADMIN のみ管理可能
- スコープ固有テンプレート（`is_system_default = false`, `scope_type`・`scope_id` 指定）は当該スコープの ADMIN / DEPUTY_ADMIN が管理
- テンプレート選択時、title / message / reminder_interval_minutes がフォームにプリセットされ、ADMIN は内容を編集してから実行可能（テンプレートはあくまで初期値）
- 初期データ（Flyway シード）としてシステムデフォルト4件を投入:
  1. 地震（タイトル: 「地震発生に伴う安否確認」、メッセージ: 「地震が発生しました。全員の安否を確認します。」、リマインド: 30分）
  2. 台風（タイトル: 「台風接近に伴う安否確認」、メッセージ: 「台風が接近しています。全員の安否を確認します。」、リマインド: 60分）
  3. 火災（タイトル: 「火災発生に伴う安否確認」、メッセージ: 「火災が発生しました。全員の安否を確認します。」、リマインド: 15分）
  4. 不審者（タイトル: 「不審者情報に伴う安否確認」、メッセージ: 「不審者の目撃情報があります。全員の安否を確認します。」、リマインド: 30分）

#### `safety_response_followups`

要支援（NEED_SUPPORT）回答への対応追跡。ADMIN が各要支援メンバーへの対応状況を記録・管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `safety_response_id` | BIGINT UNSIGNED | NO | — | FK → safety_responses（ON DELETE CASCADE） |
| `followup_status` | ENUM('PENDING', 'IN_PROGRESS', 'COMPLETED', 'UNREACHABLE') | NO | 'PENDING' | 対応ステータス |
| `assigned_to` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）対応担当者 |
| `note` | VARCHAR(500) | YES | NULL | 対応メモ（例: 「電話確認済み、軽傷あり」） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_srf_response (safety_response_id)                    -- 1回答に対し1対応追跡
INDEX idx_srf_status (followup_status)                              -- ステータス別フィルタ用
INDEX idx_srf_assigned (assigned_to)                                -- 担当者別一覧用
```

**制約・備考**
- `safety_response_id` ON DELETE CASCADE: 回答レコード削除時にカスケード
- NEED_SUPPORT の回答が登録されると、自動で `followup_status = PENDING` のレコードが生成される（SafetyResponseReceivedEvent ハンドラ内）
- SAFE / OTHER の回答にはフォローアップレコードを生成しない
- assigned_to は省略可（未アサイン状態を許容。複数 ADMIN がいる場合の担当分けに使用）
- NEED_SUPPORT → SAFE に回答変更（UPSERT）された場合、対応するフォローアップレコードを自動削除

### 他テーブルとの依存関係

- **`bulletin_threads`**: 安否確認実行時に自動生成する専用スレッド。`safety_checks.bulletin_thread_id` で参照。掲示板機能（Phase 5）のテーブルが先行して作成されている前提。Phase 3 時点で掲示板テーブルが未作成の場合は `bulletin_thread_id` を NULL のままにし、Phase 5 完了後に連携を有効化
- **`notifications`**: 安否確認の通知レコード。F04.3 プッシュ通知基盤が管理。通知の「既読」と安否の「回答済み」は別概念として分離管理
- **`teams` / `organizations` / `groups`**: ポリモーフィック参照先。FK 制約なし（アプリ層バリデーション）

### ER図（テキスト形式）
```
safety_checks (1) ──── (N) safety_responses
safety_responses (1) ──── (0..1) safety_response_followups
safety_check_templates (N) ──── (0..1) scope [scope_type + scope_id]
users (1) ──── (N) safety_responses [user_id]
users (1) ──── (N) safety_checks [created_by]
users (1) ──── (N) safety_checks [closed_by]
users (1) ──── (N) safety_response_followups [assigned_to]
bulletin_threads (1) ──── (0..1) safety_checks [bulletin_thread_id]
```

### リクエスト／レスポンス仕様

#### `POST /api/v1/safety-checks`

**認可**: ADMIN / DEPUTY_ADMIN（`MANAGE_SAFETY_CHECK`）

安否確認を実行する。対象スコープ内の全メンバー（GUEST 除く）にプッシュ通知 + WebSocket `SAFETY_CHECK_ALERT` メッセージを送信し、専用掲示板スレッドを自動生成する。

**リクエストボディ**
```json
{
  "scope_type": "TEAM",
  "scope_id": 1,
  "template_id": 1,
  "title": "地震発生に伴う安否確認",
  "message": "震度5の地震が発生しました。全員の安否を確認します。",
  "reminder_interval_minutes": 30,
  "is_drill": false
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `scope_type` | String | Yes | `TEAM` / `ORGANIZATION` / `GROUP` |
| `scope_id` | Long | Yes | 対象スコープの ID |
| `template_id` | Long | No | テンプレート ID。指定時は title / message / reminder_interval_minutes の未指定フィールドにテンプレートの値をデフォルト適用 |
| `title` | String | No | タイトル（最大200文字。省略時は「安否確認」） |
| `message` | String | No | 補足メッセージ（最大1000文字） |
| `reminder_interval_minutes` | Integer | No | 自動リマインド間隔（分）。15〜1440（1日）。NULL = リマインドなし |
| `is_drill` | Boolean | No | 訓練モード（デフォルト: false）。true の場合、通知・モーダルに「【訓練】」を明示表示 |

**レスポンス（201 Created）**
```json
{
  "data": {
    "id": 1,
    "scope_type": "TEAM",
    "scope_id": 1,
    "title": "地震発生に伴う安否確認",
    "message": "震度5の地震が発生しました。全員の安否を確認します。",
    "status": "ACTIVE",
    "is_drill": false,
    "reminder_interval_minutes": 30,
```

## 5. ビジネスロジック

### 主要フロー

#### 安否確認の実行
```
1. ADMIN が管理画面で「安否確認を実行」ボタンを押下（テンプレート選択 or 手入力）
2. バックエンドが以下を実行:
   a. template_id 指定時: テンプレートの title / message / reminder_interval_minutes を未指定フィールドのデフォルト値として適用
   b. scope_type + scope_id のバリデーション（スコープの存在確認 + ADMIN 権限確認）
   c. 対象スコープのメンバー一覧を取得（GUEST 除外）
   d. total_target_count にメンバー数をスナップショット保存
   e. safety_checks レコードを INSERT（is_drill フラグ含む）
   f. 掲示板の専用スレッドを自動生成（bulletin_threads テーブルが存在する Phase 5 以降のみ。Phase 3 時点では bulletin_thread_id = NULL のまま、掲示板連携はスキップ）:
      - タイトル: 「【安否確認】{title ?? "安否確認"} — {実行日時}」
      - 優先度: HIGHEST
      - スコープ: 安否確認と同一スコープ
      - bulletin_thread_id に保存
   g. notifications テーブルに一括 INSERT（type: SAFETY_CHECK, priority: URGENT）
   h. WebSocket で SAFETY_CHECK_ALERT メッセージを全対象メンバーに配信
      - is_drill = true の場合、ペイロードに drill: true を含める
   i. プッシュ通知を全チャネルで強制配信（通知設定に関わらず配信）
      - is_drill = true の場合、タイトルに「【訓練】」プレフィックスを付与
      - アクション付き通知として「無事です」「要支援」ボタンを含める（Actionable Notification）
        → 「無事です」タップ: status=SAFE, message=NULL で自動回答（アプリ起動不要）
        → 「要支援」タップ: status=NEED_SUPPORT, message=NULL で自動回答
        → 通知本文タップ: アプリを開いてモーダルから詳細回答
      - F04.3 側で SAFETY_CHECK 用の通知カテゴリ（iOS）/ チャネル（Android）を定義
   j. ApplicationEvent（SafetyCheckCreatedEvent）を発行
3. 監査ログに SAFETY_CHECK_CREATED イベントを記録
```

#### メンバーの安否回答（個別）
```
1. メンバーがブロッキングモーダルで回答を入力
2. POST /safety-checks/{id}/respond を呼び出し
3. バックエンドが以下を検証:
   a. 安否確認が ACTIVE であること
   b. ユーザーが対象スコープのメンバーであること（GUEST でないこと）
   c. send_gps = true の場合、gps_latitude / gps_longitude が提供されていること
4. safety_responses に UPSERT（INSERT ON DUPLICATE KEY UPDATE）
5. NEED_SUPPORT 回答の場合:
   a. safety_response_followups に followup_status = PENDING のレコードを自動生成
   b. 回答変更（UPSERT）で NEED_SUPPORT → SAFE/OTHER に変わった場合は対応するフォローアップレコードを削除
6. 以下を並行実行:
   a. WebSocket で管理者の集計画面をリアルタイム更新
   b. 掲示板スレッドの集計情報をリアルタイム更新
   c. ApplicationEvent（SafetyResponseReceivedEvent）を発行
   d. 回答率が 50% / 80% に到達した場合、ADMIN にプッシュ通知:
      「回答率 {rate}% に到達しました（{responded}/{total}）。未回答 {remaining}名」
      （閾値ごとに1回のみ。50%/80%到達をイベントハンドラ内で判定）
7. レスポンスを返却
```

#### 一括回答（複数スコープ）
```
1. フロントエンドが Pinia safetyCheckStore.pendingChecks から未回答リストを表示
2. ユーザーがチェックボックスで対象を選択（デフォルト: 全選択）
3. POST /safety-checks/bulk-respond を呼び出し
4. バックエンドが以下を実行:
   a. 各 safety_check_id について:
      - ACTIVE であること確認（CLOSED はスキップ → skipped_ids に追加）
      - ユーザーがそのスコープのメンバーであること確認（非メンバーはスキップ）
   b. 回答可能な全件に対して safety_responses を一括 UPSERT
   c. SafetyCheckBulkRespondedEvent を発行
      → 通知基盤が対応する SAFETY_CHECK 通知を一括既読化
   d. 各スコープの管理者画面に WebSocket で集計更新を配信
5. フロントエンドがモーダルを閉じる
```

#### 自動リマインドバッチ
```
1. 定期バッチ（@Scheduled, fixedDelay = 60000 = 1分間隔）で以下を実行:
   a. safety_checks で status = ACTIVE かつ reminder_interval_minutes IS NOT NULL の
      レコードを取得
   b. 各レコードについて:
      - last_reminder_at が NULL（初回リマインドは created_at + interval で判定）
        または NOW() - last_reminder_at >= reminder_interval_minutes
      - 上記条件を満たす場合:
        i.  safety_responses に回答がない対象メンバーを取得
        ii. プッシュ通知を送信（priority: URGENT）
        iii. last_reminder_at を NOW() に更新
   c. リマインド対象が 0 人の場合はスキップ（全員回答済み）
2. パフォーマンス考慮: リマインド対象チェックは1回の SQL で未回答者を特定
   （safety_checks のスコープメンバー LEFT JOIN safety_responses で NULL のユーザー）
```

#### ログイン時の未回答チェック（フロントエンド）
```
1. ユーザーがログイン完了
2. フロントエンドが GET /api/v1/safety-checks/pending を呼び出し
3. 未回答の安否確認が 1 件以上ある場合:
   a. Pinia safetyCheckStore.pendingChecks にセット
   b. ブロッキングモーダルを自動表示（is_drill = true の安否確認はヘッダーに「【訓練】」を明示）
   c. 背景をオーバーレイでブロック（回答するまで閉じられない）
4. 未回答が 0 件の場合: 何もしない（通常のダッシュボードを表示）
5. ログイン中に新たな SAFETY_CHECK_ALERT を受信した場合:
   a. pendingChecks に追加
   b. モーダルが表示されていなければ表示、表示中なら一覧に追加
```

#### 掲示板スレッドのリアルタイム更新（Phase 5 以降。Phase 3 時点では bulletin_thread_id = NULL のセッションはスキップ）
```
1. safety_responses に新しい回答が INSERT/UPDATE されるたびに（bulletin_thread_id が NULL でない場合のみ）:
   a. 掲示板スレッドの集計データを再計算:
      - 安全: N名 / 要支援: N名 / その他: N名 / 未回答: N名
   b. WebSocket で掲示板スレッドを購読中のクライアントに集計更新を配信
   c. NEED_SUPPORT の回答があった場合:
      - 掲示板スレッドに自動投稿:「{ユーザー名}さんが要支援と回答しました: {メッセージ}」
      - ADMIN に個別プッシュ通知（即座に把握できるよう）
2. 掲示板スレッドの自動投稿は safety_check 機能から bulletin 機能への
   ApplicationEvent（SafetyResponseNeedSupportEvent）で連携
```

#### 安否確認のクローズ（手動 / 自動）
```
【手動クローズ】
1. ADMIN がクローズを実行
2. バックエンドが以下を実行:
   a. status を CLOSED に更新
   b. closed_at = NOW(), closed_by = currentUser を設定
   c. 自動リマインドを停止（status = CLOSED で自動的にバッチ対象外になる）
   d. 掲示板スレッドに最終集計結果を自動投稿（bulletin_thread_id が非 NULL の場合のみ）:
      「安否確認が終了しました。回答率: {rate}%（安全: N / 要支援: N / その他: N / 未回答: N）」
   e. ApplicationEvent（SafetyCheckClosedEvent）を発行
3. クローズ後も結果の閲覧は可能（GET /safety-checks/{id}/results）
4. 未回答者のモーダルは次回 pending チェック時に表示されなくなる

【回答率100%自動クローズ】
1. SafetyResponseReceivedEvent ハンドラ内で回答率を計算
   （safety_responses のレコード数 / total_target_count）
2. 回答率が 100% に到達した場合:
   a. status を CLOSED、closed_at = NOW() に更新（closed_by = NULL: システム自動クローズ）
   b. 掲示板スレッドに最終集計結果を自動投稿
   c. ADMIN に通知:「全員の安否が確認されました。安否確認を自動クローズしました」

【24時間経過リマインド】
1. 専用バッチ（@Scheduled, cron: "0 */15 * * * *" = 15分間隔）で以下を実行:
   a. safety_checks で status = ACTIVE かつ admin_24h_notified = FALSE かつ
      created_at + 24時間 <= NOW() のレコードを取得
   b. 各レコードについて:
      - ADMIN に通知:「安否確認が ACTIVE のまま24時間経過しました。クローズしますか？」
      - admin_24h_notified = TRUE に更新
2. reminder_interval_minutes の設定に関わらず送信する（リマインドなし設定のセッションも対象）
3. 時間ベースの自動クローズは行わない（災害の規模により72時間以上の追跡が必要な場合がある）
```

#### 過去の安否確認から再実行（フロントエンド）
```
1. ADMIN が GET /api/v1/safety-checks で過去の安否確認一覧を閲覧
2. 任意のレコードの「同じ設定で再実行」ボタンを押下
3. フロントエンドが title / message / reminder_interval_minutes をコピーし、
   安否確認実行フォームにプリセット
4. ADMIN が内容を確認・編集し、POST /api/v1/safety-checks で実行
※ API 変更不要。フロントエンドの UI 操作のみで完結
```

#### フォローアップ管理
```
1. NEED_SUPPORT の回答が登録されると、safety_response_followups に
   followup_status = PENDING のレコードが自動生成される
2. ADMIN が results 画面で要支援メンバーの一覧を確認
3. 各メンバーに対して以下を操作:
   a. 「対応中」に変更（assigned_to に自分を設定）→ 他の ADMIN に対応中であることを共有
   b. 電話・現地確認後に「対応完了」に変更（note に対応内容を記録）
   c. 連絡が取れない場合は「連絡不可」に変更
4. PUT /safety-checks/{id}/results/followups/{responseId} で状態更新
5. results API のレスポンスに followup 情報が含まれるため、
   複数 ADMIN 間で対応の重複を防止
```

#### 応答時間統計の算出
```
1. results API のリクエスト時に response_time_stats を動的に算出:
   a. safety_responses.responded_at - safety_checks.created_at で各回答の応答時間を計算
   b. avg_response_time_seconds: 全回答の平均応答時間（秒）
   c. median_response_time_seconds: 中央値（SQL の PERCENTILE_CONT or アプリ層で算出）
   d. first_response_at: 最初の回答日時
   e. last_response_at: 最後の回答日時
2. DB追加不要（既存の created_at / responded_at から導出）
3. 訓練モード（is_drill = true）の安否確認では、応答時間統計を
   訓練の振り返り指標として活用（「前回訓練より平均2分改善」等）
```

### 重要な判定ロジック

- **対象メンバーの特定**: scope_type に応じて members / supporters テーブルを参照。GUEST は除外。GROUP スコープの場合は group_members から取得。安否確認実行時点のメンバーリストをスナップショットとして `total_target_count` に保存し、回答率の分母にする
- **GUEST 除外の理由**: GUEST は組織・チームへの一時的なアクセス権限であり、安否確認の対象とするのは過剰。GUEST の安全確認は招待元が個別に行う想定
- **GROUP スコープの権限モデル**: グループには独自の ADMIN ロールがないため、グループが属する親チーム/組織の ADMIN / DEPUTY_ADMIN（`MANAGE_SAFETY_CHECK`）が実行権限を持つ。権限チェックは `groups.team_id`（or `groups.organization_id`）を取得し、その親スコープの ADMIN 権限を検証。ユースケース: 「サッカー部3年生だけに安否確認」のようにチーム全体より狭い範囲をターゲットにしたい場合
- **回答率100%自動クローズ**: 全員回答済みの安否確認は自動的に CLOSED に変更（`closed_by = NULL`）。ACTIVE のまま残す意味がなく、自動リマインドバッチの無駄な走査も防止。時間ベースの自動クローズは行わない（災害規模により長期追跡が必要な場合があるため、判断は ADMIN に委ねる）。一括回答で同時に複数の回答が処理される場合の競合を防ぐため、自動クローズ判定は `@Transactional` 内で `safety_responses` の COUNT と `total_target_count` を比較し、楽観ロック（`safety_checks.status` の CAS チェック）で二重クローズを防止
- **スヌーズ機能**: ブロッキングモーダルに「後で回答する」リンクを提供（15分固定、最大3回 = 計45分）。回数は `localStorage` でキー `safety_check_snooze_{safety_check_id}` として管理（安否確認ごとに独立カウント）。3回使い切り後はスヌーズ不可。運転中・会議中等の一時的に回答困難な状況への配慮と、緊急性の維持を両立
- **CSV エクスポート**: Phase 3 では実装しない。Phase 10 の管理者ダッシュボードで統一的なエクスポート基盤を構築し、安否確認・メンバー一覧・出欠管理等に横断的に適用する。当面は results API のレスポンスをフロントエンドでコピー / ブラウザ印刷で代替
- **回答の上書き**: 同一ユーザーが同一安否確認に2回以上回答した場合は UPSERT で最新の回答に上書き。「SAFE で回答した後に状況が変わり NEED_SUPPORT に変更」等のケースに対応
- **一括回答の部分成功**: bulk-respond は部分成功を許容。クローズ済みや権限なしの安否確認はスキップし、回答可能な分のみ処理。全件スキップの場合のみ 403 を返す
- **ブロッキングモーダルの強制表示**: フロントエンドの実装で、モーダルの外側クリックや ESC キーでの閉鎖を無効化する。回答ボタンが主な閉じ方。ただし「後で回答する」スヌーズリンクを提供（15分固定、最大3回 = 計45分）。スヌーズ回数は `localStorage` で管理。3回使い切った後はスヌーズボタン非表示で回答を強制。スヌーズ中もヘッダーに「未回答の安否確認があります」バナーを常時表示し、任意のタイミングで回答可能
- **message / message_source のクロスバリデーション**: `message` が NULL の場合 `message_source` も NULL であること。`message` が非 NULL の場合 `message_source` は `PRESET` または `CUSTOM` のいずれかが必須。`message_source = PRESET` の場合、`message` の値がアクティブなプリセット例文の `body` に一致することをバリデーション（一致しない場合は 400 エラー）。`message_source = CUSTOM` の場合は自由入力のため内容チェックなし
- **GPS 位置情報の任意性**: GPS 送信はデフォルト OFF のオプトイン。位置情報を送信しなくても安否回答は可能。GPS が OFF でもブラウザの Geolocation API がブロックされていても問題なし
- **通知の既読と安否の回答済みの分離**: 通知テーブルの `is_read` と安否確認の回答状態は独立。通知を既読にしても安否が回答済みにはならない。安否を回答すると対応する通知は自動既読になる（逆方向のみ連携）
- **リマインド送信間隔**: 最小15分。管理者が設定した interval に基づきバッチが自動送信。手動リマインドは間隔制限なし（ただし 429 で5分間隔の制限あり）
- **NEED_SUPPORT 回答の即時通知**: 要支援の回答は ADMIN に個別プッシュ通知を送信。安全の回答は集計画面のリアルタイム更新のみ（個別通知はしない。大量の SAFE 通知は不要）
- **同一スコープでの複数 ACTIVE セッション**: 許容する。連続して災害が発生するケース（地震の後に津波警報等）に対応。pending API は全 ACTIVE セッションの未回答を返す
- **デフォルト機能**: 安否確認はデフォルト機能 #23（module_type = DEFAULT）。全チーム常時有効であり、`@RequiresModule` チェックは不要だが、コード一貫性のため `@RequiresModule(slug = "safety-check", level = ModuleLevel.TEAM)` を付与する（DEFAULT モジュールは Redis キャッシュに含まれるため常にパス）
- **24時間経過リマインドの判定**: 自動リマインドバッチ内で `created_at + 24h <= NOW()` を検出するが、`reminder_interval_minutes = NULL`（リマインドなし設定）のセッションにも24時間経過通知は送信する。これはリマインドではなくクローズ忘れ防止の ADMIN 向け通知であり、メンバーへの再送信とは別の目的。判定フラグとして safety_checks に `admin_24h_notified BOOLEAN NOT NULL DEFAULT FALSE` を使用し、1回のみ送信を保証する
- **訓練モード（ドリル）**: `is_drill = true` の安否確認は通知・モーダル・掲示板スレッドすべてに「【訓練】」を明示。回答データは通常の安否確認と同じテーブルに保存されるが、一覧 API の `is_drill` フィルタパラメータで分離閲覧可能。訓練では応答時間統計（`response_time_stats`）を振り返り指標として活用し、「前回訓練より平均N分改善」等の定量評価を支援
- **テンプレート適用の優先順位**: POST /safety-checks で `template_id` と個別フィールド（title / message / reminder_interval_minutes）を同時指定した場合、個別フィールドの値が優先。template_id はあくまで未指定フィールドのデフォルト値として使用
- **アクション付きプッシュ通知（Actionable Notification）**: iOS の UNNotificationCategory / Android の NotificationCompat.Action を使用し、プッシュ通知から直接「無事です」「要支援」を回答可能。アプリ起動不要でワンタップ回答を実現。詳細入力が必要な場合は通知本文タップでアプリを開きモーダルから回答。F04.3 側で SAFETY_CHECK カテゴリ/チャネルを定義
- **フォローアップ管理**: NEED_SUPPORT 回答に対して自動で `safety_response_followups`（status=PENDING）を生成。ADMIN が対応状況を IN_PROGRESS / COMPLETED / UNREACHABLE に更新し、assigned_to で担当者を明示。複数 ADMIN 間の対応重複を防止。回答が NEED_SUPPORT → SAFE/OTHER に変更された場合はフォローアップレコードを自動削除
- **回答率の中間閾値アラート**: SafetyResponseReceivedEvent ハンドラ内で回答率を算出し、50% / 80% 到達時に ADMIN にプッシュ通知を送信。閾値は固定（設定可能にしない。シンプルさを優先）。各閾値につき1回のみ送信（イベントハンドラ内で現在の回答数と閾値を比較し、ちょうど到達した場合のみ通知）
- **応答時間統計**: results API の `response_time_stats` は DB 追加不要。`safety_checks.created_at` と `safety_responses.responded_at` の差分から動的に算出。中央値の算出は MySQL 8.0 のウィンドウ関数（`PERCENTILE_CONT`）またはアプリ層で計算
- **過去の安否確認から再実行**: フロントエンド UI のみで完結（API 変更不要）。一覧画面の各レコードに「同じ設定で再実行」ボタンを配置し、title / message / reminder_interval_minutes を実行フォームにプリセット。テンプレート機能とは別に、直近の実績ベースで素早く再実行したい場面に対応

## 6. セキュリティ考慮事項

- **認可チェック**: 安否確認の実行・クローズ・結果閲覧は ADMIN / DEPUTY_ADMIN（`MANAGE_SAFETY_CHECK`）のみ。回答は対象スコープの MEMBER / SUPPORTER 以上
- **GPS 位置情報の保護**: GPS 座標は個人情報。results API で GPS 座標を返却する際は ADMIN のみアクセス可能。掲示板スレッドには GPS 情報を含めない（集計画面専用）。GPS データの保持期間は安否確認レコードと同じ（恒久保持）。将来的に GDPR 対応で保持期間を設定する場合は `safety_responses` の GPS カラムを NULL クリアするバッチを追加
- **SAFETY_CHECK_ALERT の配信制御**: WebSocket メッセージのペイロードにスコープ名と日時のみを含め、メンバーリスト等の機密情報は含めない。ペイロードは 1KB 以下の軽量設計
- **レートリミット**: 安否確認の実行 API（POST /safety-checks）は Bucket4j で 1分間に5回（ユーザー単位）。緊急時の連打防止と、悪意あるスパム送信防止を両立。手動リマインド API（POST /remind）は 5分間に1回。回答 API（POST /respond, POST /bulk-respond）は 1分間に30回（緊急時の高頻度アクセスを許容しつつ、自動化攻撃を抑止）
- **一括回答の ID 検証**: bulk-respond で送信された `safety_check_ids` のうち、ユーザーがメンバーでないスコープの安否確認は無視（他スコープの安否確認 ID を推測して回答するのを防止）
- **SYSTEM_ADMIN プリセット管理の保護**: `/system-admin/**` エンドポイントは SYSTEM_ADMIN ロールを JWT + DB で二重チェック
- **XSS 対策**: 安否確認の `title`・`message`・回答の `message` はレスポンス時にサニタイズ。プリセット例文も SYSTEM_ADMIN 入力だが同様にサニタイズ
- **監査ログ**: 安否確認の実行・クローズ・リマインド送信は `audit_logs` に記録。個別の回答は安否確認固有テーブル（safety_responses）に記録するため audit_logs には含めない
- **GPS 座標のバリデーション**: `gps_latitude` は -90.0〜90.0、`gps_longitude` は -180.0〜180.0 の範囲をアプリ層で検証。範囲外の値は 400 エラーで拒否。`send_gps = true` なのに座標が未提供の場合も 400 エラー
