# F03.6: 緊急安否確認

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 3
> **最終更新**: 2026-03-13

---

## 1. 概要

災害・緊急事態発生時に、管理者がワンクリックで所属メンバー全員に安否確認を一斉送信し、メンバーの回答をリアルタイムに集計・表示するシステム。プッシュ通知 + WebSocket でブロッキングモーダルを表示し、GPS 位置情報の任意送信やプリセット例文による迅速な回答を可能にする。自動的に掲示板の専用スレッドを生成し、安否ステータスをリアルタイム掲載する。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | プリセット例文の管理（CRUD） |
| ADMIN | 安否確認の実行・クローズ・結果閲覧・リマインド設定 |
| DEPUTY_ADMIN | `MANAGE_SAFETY_CHECK` 権限を持つ場合: 安否確認の実行・結果閲覧 |
| MEMBER | 安否回答（個別・一括）、自分の回答履歴閲覧 |
| SUPPORTER | 安否回答（個別・一括）、自分の回答履歴閲覧 |
| GUEST | 対象外（安否確認の送信・回答いずれも不可） |

### 対象レベル
- [x] 組織 (Organization) — 組織全体への安否確認
- [x] チーム (Team) — チーム単位での安否確認
- [x] 個人 (Personal) — 個人としての回答（複数スコープへの一括回答含む）

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `safety_checks` | 安否確認セッション（実行単位） | なし |
| `safety_responses` | メンバーの安否回答 | なし |
| `safety_check_message_presets` | 回答時のプリセット例文（運営管理） | なし |

### テーブル定義

#### `safety_checks`

安否確認セッション。ADMIN が実行するたびに 1 レコード生成され、対象スコープ内の全メンバーに通知が配信される。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION', 'GROUP') | NO | — | 安否確認の対象スコープ種別 |
| `scope_id` | BIGINT UNSIGNED | NO | — | 対象スコープの ID（teams.id / organizations.id / groups.id） |
| `title` | VARCHAR(200) | YES | NULL | 安否確認のタイトル（例: 「地震発生に伴う安否確認」。NULL の場合は「安否確認」） |
| `message` | TEXT | YES | NULL | 管理者からの補足メッセージ（例: 「震度5の地震が発生しました。全員の安否を確認します」）|
| `status` | ENUM('ACTIVE', 'CLOSED') | NO | 'ACTIVE' | セッション状態。CLOSED = 回答受付終了 |
| `reminder_interval_minutes` | INT | YES | NULL | 自動リマインド間隔（分）。NULL = リマインドなし |
| `last_reminder_at` | DATETIME | YES | NULL | 最後にリマインドを送信した日時（リマインドバッチが参照） |
| `total_target_count` | INT | NO | 0 | 実行時の対象メンバー数（スナップショット。後からメンバーが増減しても変わらない） |
| `bulletin_thread_id` | BIGINT UNSIGNED | YES | NULL | FK → bulletin_threads（ON DELETE SET NULL）自動生成された専用掲示板スレッド |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）実行した ADMIN |
| `closed_at` | DATETIME | YES | NULL | クローズした日時 |
| `closed_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）クローズした ADMIN |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_sc_scope (scope_type, scope_id, created_at DESC)   -- スコープ別の安否確認一覧用
INDEX idx_sc_status (status)                                  -- ACTIVE セッションの抽出用
INDEX idx_sc_reminder (status, reminder_interval_minutes, last_reminder_at)  -- リマインドバッチ用
```

**制約・備考**
- `scope_type` + `scope_id` でポリモーフィック参照。FK 制約は設けず、アプリ層でバリデーション（teams / organizations / groups テーブルの存在確認）
- `total_target_count` は実行時点のメンバー数をスナップショットとして保存。回答率の分母として使用。後からスコープにメンバーが加入しても変動しない
- `bulletin_thread_id` ON DELETE SET NULL: 掲示板スレッドが削除されても安否確認レコードは残す
- 物理削除・論理削除ともに行わない。安否確認は監査記録として恒久的に保持する
- 同一スコープで ACTIVE なセッションは同時に複数存在可能（連続して地震が来る等のケース）

---

#### `safety_responses`

各メンバーの安否回答。1つの安否確認に対し、1ユーザー1回答（UPSERT で上書き可能）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `safety_check_id` | BIGINT UNSIGNED | NO | — | FK → safety_checks（ON DELETE CASCADE） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE） |
| `status` | ENUM('SAFE', 'NEED_SUPPORT', 'OTHER') | NO | — | 安否ステータス |
| `message` | VARCHAR(200) | YES | NULL | 任意メッセージ（プリセット選択 or 自由入力） |
| `message_source` | ENUM('PRESET', 'CUSTOM') | YES | NULL | メッセージの出典。NULL = メッセージなし |
| `gps_shared` | BOOLEAN | NO | FALSE | GPS 位置情報を送信したか |
| `gps_latitude` | DECIMAL(10,7) | YES | NULL | 緯度（`gps_shared = true` の場合のみ値あり） |
| `gps_longitude` | DECIMAL(10,7) | YES | NULL | 経度（`gps_shared = true` の場合のみ値あり） |
| `responded_at` | DATETIME | NO | CURRENT_TIMESTAMP | 回答日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_sr_check_user (safety_check_id, user_id)   -- 1ユーザー1回答
INDEX idx_sr_user (user_id)                                -- ユーザー別回答履歴用
INDEX idx_sr_status (safety_check_id, status)              -- ステータス別集計用
```

**制約・備考**
- `safety_check_id` ON DELETE CASCADE: 安否確認セッション削除時にカスケード（通常は物理削除しない運用だが、万が一の場合に対応）
- `user_id` ON DELETE CASCADE: ユーザー退会時に回答レコードも削除（集計値は `total_target_count` で分母を保持しているため、退会による集計値のずれは許容）
- UPSERT 対応: 回答修正は `ON DUPLICATE KEY UPDATE` で `status`, `message`, `message_source`, `gps_shared`, `gps_latitude`, `gps_longitude`, `responded_at` を上書き
- GPS 座標は DECIMAL(10,7) で小数点以下7桁（約1.1cm精度）。十分な精度を持つ
- `gps_shared = false` かつ `gps_latitude`/`gps_longitude` が NOT NULL というデータ不整合を防ぐため、アプリ層で `gps_shared = false` の場合は座標を NULL に強制

#### `safety_check_message_presets`

安否回答時にモーダルに表示するプリセット例文。SYSTEM_ADMIN が管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `body` | VARCHAR(200) | NO | — | プリセット例文（例: 「無事です。特に被害はありません」）|
| `sort_order` | INT | NO | 0 | 表示順 |
| `is_active` | BOOLEAN | NO | TRUE | FALSE = モーダルに表示しない |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_scmp_active_sort (is_active, sort_order)   -- アクティブなプリセットの表示順取得用
```

**制約・備考**
- SYSTEM_ADMIN のみ管理可能
- 初期データ（Flyway シード）として以下の 5 件を投入:
  1. 無事です。特に被害はありません
  2. 自宅にいます。問題ありません
  3. 避難中です。後ほど詳細を連絡します
  4. 軽微な被害がありますが無事です
  5. 現在移動中です。安全を確認しています
- 物理削除は行わない。`is_active = false` で非表示にする運用
- 最大200文字（VARCHAR(200)）。自由入力と同じ上限

### 他テーブルとの依存関係

- **`bulletin_threads`**: 安否確認実行時に自動生成する専用スレッド。`safety_checks.bulletin_thread_id` で参照。掲示板機能（Phase 5）のテーブルが先行して作成されている前提。Phase 3 時点で掲示板テーブルが未作成の場合は `bulletin_thread_id` を NULL のままにし、Phase 5 完了後に連携を有効化
- **`notifications`**: 安否確認の通知レコード。F04.3 プッシュ通知基盤が管理。通知の「既読」と安否の「回答済み」は別概念として分離管理
- **`teams` / `organizations` / `groups`**: ポリモーフィック参照先。FK 制約なし（アプリ層バリデーション）

### ER図（テキスト形式）
```
safety_checks (1) ──── (N) safety_responses
users (1) ──── (N) safety_responses [user_id]
users (1) ──── (N) safety_checks [created_by]
users (1) ──── (N) safety_checks [closed_by]
bulletin_threads (1) ──── (0..1) safety_checks [bulletin_thread_id]
```

### リクエスト／レスポンス仕様

#### `POST /api/v1/safety-checks`

**認可**: ADMIN / DEPUTY_ADMIN（`MANAGE_SAFETY_CHECK`）

安否確認を実行する。対象スコープ内の全メンバー（GUEST 除く）にプッシュ通知 + WebSocket `SAFETY_CHECK_ALERT` メッセージを送信し、専用掲示板スレッドを自動生成する。

**リクエストボディ**
```json
{
  "scope_type": "TEAM",
  "scope_id": 1,
  "title": "地震発生に伴う安否確認",
  "message": "震度5の地震が発生しました。全員の安否を確認します。",
  "reminder_interval_minutes": 30
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `scope_type` | String | Yes | `TEAM` / `ORGANIZATION` / `GROUP` |
| `scope_id` | Long | Yes | 対象スコープの ID |
| `title` | String | No | タイトル（最大200文字。省略時は「安否確認」） |
| `message` | String | No | 補足メッセージ（最大1000文字） |
| `reminder_interval_minutes` | Integer | No | 自動リマインド間隔（分）。15〜1440（1日）。NULL = リマインドなし |

**レスポンス（201 Created）**
```json
{
  "data": {
    "id": 1,
    "scope_type": "TEAM",
    "scope_id": 1,
    "title": "地震発生に伴う安否確認",
    "message": "震度5の地震が発生しました。全員の安否を確認します。",
    "status": "ACTIVE",
    "reminder_interval_minutes": 30,
    "total_target_count": 42,
    "bulletin_thread_id": 501,
    "created_by": {
      "id": 10,
      "display_name": "田中太郎"
    },
    "created_at": "2026-03-13T14:30:00"
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | バリデーションエラー（scope_type 不正、reminder_interval 範囲外等） |
| 403 | 対象スコープの ADMIN / DEPUTY_ADMIN（`MANAGE_SAFETY_CHECK`）でない |
| 404 | scope_id が存在しない |

#### `PATCH /api/v1/safety-checks/{id}/close`

```json
    "summary": {

      "need_support_count": 3,
      "other_count": 1,

      "response_rate": 92.9
    }
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | 対象スコープの ADMIN / DEPUTY_ADMIN でない |
| 404 | 安否確認 ID が存在しない |

#### `POST /api/v1/safety-checks/{id}/respond`

**認可**: 対象スコープの MEMBER / SUPPORTER 以上

個別の安否回答を送信する。既に回答済みの場合は上書き（UPSERT）。

**リクエストボディ**
```json
{
  "status": "SAFE",
  "message": "無事です。特に被害はありません",
  "message_source": "PRESET",
  "send_gps": true,
  "gps_latitude": 35.6812362,
  "gps_longitude": 139.7671248
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `status` | String | Yes | `SAFE` / `NEED_SUPPORT` / `OTHER` |
| `message` | String | No | 任意メッセージ（最大200文字） |
| `message_source` | String | No | `PRESET` / `CUSTOM` / null |
| `send_gps` | Boolean | No | GPS 位置情報を送信するか（デフォルト: false） |
| `gps_latitude` | Decimal | No | 緯度（`send_gps = true` の場合必須） |
| `gps_longitude` | Decimal | No | 経度（`send_gps = true` の場合必須） |

**レスポンス（200 OK）**
```json
{
  "data": {
    "safety_check_id": 1,
    "status": "SAFE",
    "message": "無事です。特に被害はありません",
    "gps_shared": true,
    "responded_at": "2026-03-13T14:32:15"
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | バリデーションエラー（status 不正、send_gps=true なのに座標なし等） |
| 403 | 対象スコープのメンバーでない / GUEST |
| 404 | 安否確認 ID が存在しない |
| 409 | 安否確認がクローズ済み |

---

#### `POST /api/v1/safety-checks/bulk-respond`

**認可**: ログイン済みユーザー（各安否確認の対象スコープメンバーであること）

複数スコープからの安否確認に一括回答する。地震等の災害時に複数のチーム・組織から同時に届いた安否確認にまとめて応答するための API。

**リクエストボディ**
```json
{
  "safety_check_ids": [1, 2, 3],
  "status": "SAFE",
  "message": "自宅にいます。問題ありません",
  "message_source": "PRESET",
  "send_gps": false
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `safety_check_ids` | Long[] | Yes | 回答対象の安否確認 ID リスト（最大20件） |
| `status` | String | Yes | `SAFE` / `NEED_SUPPORT` / `OTHER` |
| `message` | String | No | 任意メッセージ（全件共通、最大200文字） |
| `message_source` | String | No | `PRESET` / `CUSTOM` / null |
| `send_gps` | Boolean | No | GPS 位置情報を送信するか（デフォルト: false） |

**レスポンス（200 OK）**
```json
{
  "data": {
    "responded_count": 3,
    "skipped_ids": [],
    "responded_at": "2026-03-13T14:32:15"
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | safety_check_ids が空 / 20件超過 / status 不正 |
| 403 | いずれかの安否確認の対象スコープメンバーでない（該当 ID はスキップし、回答可能な分のみ処理。全件不可の場合は 403） |

**補足**: クローズ済みの安否確認 ID が含まれている場合はスキップし、`skipped_ids` に含める（エラーにはしない）。対応する SAFETY_CHECK 通知を一括既読にする（`SafetyCheckBulkRespondedEvent` → 通知基盤が `notifications.is_read = true` に更新）。

#### `GET /api/v1/safety-checks/{id}/results`

**認可**: ADMIN / DEPUTY_ADMIN（`MANAGE_SAFETY_CHECK`）

安否確認の集計結果・回答一覧・GPS 位置情報を返す。管理者向けのリアルタイム集計画面で使用。

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|

## 5. ビジネスロジック

### 主要フロー

#### 安否確認の実行
```
1. ADMIN が管理画面で「安否確認を実行」ボタンを押下
2. バックエンドが以下を実行:
   a. scope_type + scope_id のバリデーション（スコープの存在確認 + ADMIN 権限確認）
   b. 対象スコープのメンバー一覧を取得（GUEST 除外）
   c. total_target_count にメンバー数をスナップショット保存
   d. safety_checks レコードを INSERT
   e. 掲示板の専用スレッドを自動生成:
      - タイトル: 「【安否確認】{title ?? "安否確認"} — {実行日時}」
      - 優先度: HIGHEST
      - スコープ: 安否確認と同一スコープ
      - bulletin_thread_id に保存
   f. notifications テーブルに一括 INSERT（type: SAFETY_CHECK, priority: URGENT）
   g. WebSocket で SAFETY_CHECK_ALERT メッセージを全対象メンバーに配信
   h. プッシュ通知を全チャネルで強制配信（通知設定に関わらず配信）
   i. ApplicationEvent（SafetyCheckCreatedEvent）を発行
3. 監査ログに SAFETY_CHECK_CREATED イベントを記録
```

#### メンバーの安否回答（個別）
```
1. メンバーがブロッキングモーダルで回答を入力
2. POST /safety-checks/{id}/respond を呼び出し
3. バックエンドが以下を検証:
   a. 安否確認が ACTIVE であること
   b. ユーザーが対象スコープのメンバーであること（GUEST でないこと）
   c. send_gps = true の場合、gps_latitude / gps_longitude が提供されていること
4. safety_responses に UPSERT（INSERT ON DUPLICATE KEY UPDATE）
5. 以下を並行実行:
   a. WebSocket で管理者の集計画面をリアルタイム更新
   b. 掲示板スレッドの集計情報をリアルタイム更新
   c. ApplicationEvent（SafetyResponseReceivedEvent）を発行
6. レスポンスを返却
```

#### 一括回答（複数スコープ）
```
1. フロントエンドが Pinia safetyCheckStore.pendingChecks から未回答リストを表示
2. ユーザーがチェックボックスで対象を選択（デフォルト: 全選択）
3. POST /safety-checks/bulk-respond を呼び出し
4. バックエンドが以下を実行:
   a. 各 safety_check_id について:
      - ACTIVE であること確認（CLOSED はスキップ → skipped_ids に追加）
      - ユーザーがそのスコープのメンバーであること確認（非メンバーはスキップ）
   b. 回答可能な全件に対して safety_responses を一括 UPSERT
   c. SafetyCheckBulkRespondedEvent を発行
      → 通知基盤が対応する SAFETY_CHECK 通知を一括既読化
   d. 各スコープの管理者画面に WebSocket で集計更新を配信
5. フロントエンドがモーダルを閉じる
```

#### 自動リマインドバッチ
```
1. 定期バッチ（@Scheduled, fixedDelay = 60000 = 1分間隔）で以下を実行:
   a. safety_checks で status = ACTIVE かつ reminder_interval_minutes IS NOT NULL の
      レコードを取得
   b. 各レコードについて:
      - last_reminder_at が NULL（初回リマインドは created_at + interval で判定）
        または NOW() - last_reminder_at >= reminder_interval_minutes
      - 上記条件を満たす場合:
        i.  safety_responses に回答がない対象メンバーを取得
        ii. プッシュ通知を送信（priority: URGENT）
        iii. last_reminder_at を NOW() に更新
   c. リマインド対象が 0 人の場合はスキップ（全員回答済み）
2. パフォーマンス考慮: リマインド対象チェックは1回の SQL で未回答者を特定
   （safety_checks のスコープメンバー LEFT JOIN safety_responses で NULL のユーザー）
```

#### ログイン時の未回答チェック（フロントエンド）
```
1. ユーザーがログイン完了
2. フロントエンドが GET /api/v1/safety-checks/pending を呼び出し
3. 未回答の安否確認が 1 件以上ある場合:
   a. Pinia safetyCheckStore.pendingChecks にセット
   b. ブロッキングモーダルを自動表示
   c. 背景をオーバーレイでブロック（回答するまで閉じられない）
4. 未回答が 0 件の場合: 何もしない（通常のダッシュボードを表示）
5. ログイン中に新たな SAFETY_CHECK_ALERT を受信した場合:
   a. pendingChecks に追加
   b. モーダルが表示されていなければ表示、表示中なら一覧に追加
```

#### 掲示板スレッドのリアルタイム更新
```
1. safety_responses に新しい回答が INSERT/UPDATE されるたびに:
   a. 掲示板スレッドの集計データを再計算:
      - 安全: N名 / 要支援: N名 / その他: N名 / 未回答: N名
   b. WebSocket で掲示板スレッドを購読中のクライアントに集計更新を配信
   c. NEED_SUPPORT の回答があった場合:
      - 掲示板スレッドに自動投稿:「{ユーザー名}さんが要支援と回答しました: {メッセージ}」
      - ADMIN に個別プッシュ通知（即座に把握できるよう）
2. 掲示板スレッドの自動投稿は safety_check 機能から bulletin 機能への
   ApplicationEvent（SafetyResponseNeedSupportEvent）で連携
```

#### 安否確認のクローズ
```
1. ADMIN がクローズを実行
2. バックエンドが以下を実行:
   a. status を CLOSED に更新
   b. closed_at = NOW(), closed_by = currentUser を設定
   c. 自動リマインドを停止（status = CLOSED で自動的にバッチ対象外になる）
   d. 掲示板スレッドに最終集計結果を自動投稿:
      「安否確認が終了しました。回答率: {rate}%（安全: N / 要支援: N / その他: N / 未回答: N）」
   e. ApplicationEvent（SafetyCheckClosedEvent）を発行
3. クローズ後も結果の閲覧は可能（GET /safety-checks/{id}/results）
4. 未回答者のモーダルは次回 pending チェック時に表示されなくなる
```

### 重要な判定ロジック

- **対象メンバーの特定**: scope_type に応じて members / supporters テーブルを参照。GUEST は除外。GROUP スコープの場合は group_members から取得。安否確認実行時点のメンバーリストをスナップショットとして `total_target_count` に保存し、回答率の分母にする
- **GUEST 除外の理由**: GUEST は組織・チームへの一時的なアクセス権限であり、安否確認の対象とするのは過剰。GUEST の安全確認は招待元が個別に行う想定
- **回答の上書き**: 同一ユーザーが同一安否確認に2回以上回答した場合は UPSERT で最新の回答に上書き。「SAFE で回答した後に状況が変わり NEED_SUPPORT に変更」等のケースに対応
- **一括回答の部分成功**: bulk-respond は部分成功を許容。クローズ済みや権限なしの安否確認はスキップし、回答可能な分のみ処理。全件スキップの場合のみ 403 を返す
- **ブロッキングモーダルの強制表示**: フロントエンドの実装で、モーダルの外側クリックや ESC キーでの閉鎖を無効化する。回答ボタン押下が唯一の閉じ方。ただし「後で回答する」リンク（30分後に再表示するスヌーズ）は提供可能（→ 未解決事項）
- **GPS 位置情報の任意性**: GPS 送信はデフォルト OFF のオプトイン。位置情報を送信しなくても安否回答は可能。GPS が OFF でもブラウザの Geolocation API がブロックされていても問題なし
- **通知の既読と安否の回答済みの分離**: 通知テーブルの `is_read` と安否確認の回答状態は独立。通知を既読にしても安否が回答済みにはならない。安否を回答すると対応する通知は自動既読になる（逆方向のみ連携）
- **リマインド送信間隔**: 最小15分。管理者が設定した interval に基づきバッチが自動送信。手動リマインドは間隔制限なし（ただし 429 で5分間隔の制限あり）
- **NEED_SUPPORT 回答の即時通知**: 要支援の回答は ADMIN に個別プッシュ通知を送信。安全の回答は集計画面のリアルタイム更新のみ（個別通知はしない。大量の SAFE 通知は不要）
- **同一スコープでの複数 ACTIVE セッション**: 許容する。連続して災害が発生するケース（地震の後に津波警報等）に対応。pending API は全 ACTIVE セッションの未回答を返す
- **デフォルト機能**: 安否確認はデフォルト機能 #23（module_type = DEFAULT）。全チーム常時有効であり、`@RequiresModule` チェックは不要だが、コード一貫性のため `@RequiresModule(slug = "safety-check", level = ModuleLevel.TEAM)` を付与する（DEFAULT モジュールは Redis キャッシュに含まれるため常にパス）

## 6. セキュリティ考慮事項

- **認可チェック**: 安否確認の実行・クローズ・結果閲覧は ADMIN / DEPUTY_ADMIN（`MANAGE_SAFETY_CHECK`）のみ。回答は対象スコープの MEMBER / SUPPORTER 以上
- **GPS 位置情報の保護**: GPS 座標は個人情報。results API で GPS 座標を返却する際は ADMIN のみアクセス可能。掲示板スレッドには GPS 情報を含めない（集計画面専用）。GPS データの保持期間は安否確認レコードと同じ（恒久保持）。将来的に GDPR 対応で保持期間を設定する場合は `safety_responses` の GPS カラムを NULL クリアするバッチを追加
- **SAFETY_CHECK_ALERT の配信制御**: WebSocket メッセージのペイロードにスコープ名と日時のみを含め、メンバーリスト等の機密情報は含めない。ペイロードは 1KB 以下の軽量設計
- **レートリミット**: 安否確認の実行 API（POST /safety-checks）は Bucket4j で 1分間に5回。緊急時の連打防止と、悪意あるスパム送信防止を両立。手動リマインド API（POST /remind）は 5分間に1回
- **一括回答の ID 検証**: bulk-respond で送信された `safety_check_ids` のうち、ユーザーがメンバーでないスコープの安否確認は無視（他スコープの安否確認 ID を推測して回答するのを防止）
- **SYSTEM_ADMIN プリセット管理の保護**: `/system-admin/**` エンドポイントは SYSTEM_ADMIN ロールを JWT + DB で二重チェック
- **XSS 対策**: 安否確認の `title`・`message`・回答の `message` はレスポンス時にサニタイズ。プリセット例文も SYSTEM_ADMIN 入力だが同様にサニタイズ
- **監査ログ**: 安否確認の実行・クローズ・リマインド送信は `audit_logs` に記録。個別の回答は安否確認固有テーブル（safety_responses）に記録するため audit_logs には含めない

- [ ] GROUP スコープでの安否確認の権限モデル。グループには独自の ADMIN ロールがないため、グループ作成者 or 親チーム/組織の ADMIN が実行権限を持つのか
- [ ] 安否確認結果の CSV エクスポート機能を提供するか。災害報告書作成等の用途で必要になる可能性がある
