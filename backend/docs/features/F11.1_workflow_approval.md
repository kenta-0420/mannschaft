# F11.1: 汎用ワークフロー・承認エンジン

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 11
> **最終更新**: 2026-03-18
> **モジュール種別**: デフォルト機能 #29（全チーム・組織で利用可能）
> **関連ドキュメント**: F05.3 電子印鑑（承認時の押印連携）、F04.3 プッシュ通知（承認待ち・完了通知）、F05.2 回覧板（既存承認パターンの参考）

---

## 1. 概要

組織・チーム内のあらゆる申請・承認フローを汎用的に処理するワークフローエンジン。ADMIN が承認テンプレート（1〜3段階の承認ステップ）を定義し、MEMBER 以上が申請を作成すると、定義されたルーティングに従って承認者へ通知・判定が進行する。カスタムフィールド（テキスト/数値/日付/添付ファイル）に対応し、備品購入申請・経費精算・施設利用申請など業種を問わない汎用 API を提供する。承認時には F05.3 電子印鑑との連携（押印オプション）に対応し、証跡を記録する。

**主要ユースケース**:
- マンション管理組合: 理事会への施設利用申請、修繕工事承認
- 企業: 経費精算、備品購入稟議、休暇申請
- スポーツチーム: 遠征企画承認、予算申請
- 学校/PTA: 行事企画承認、支出承認

### テーブル定義

#### `workflow_templates`

承認テンプレート。ADMIN がスコープ（チーム/組織）ごとに作成する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `name` | VARCHAR(100) | NO | — | テンプレート名（例: 「経費精算申請」「施設利用申請」） |
| `description` | VARCHAR(500) | YES | NULL | テンプレートの説明 |
| `icon` | VARCHAR(30) | YES | NULL | アイコン（emoji or アイコン名） |
| `color` | VARCHAR(7) | YES | NULL | テーマカラー（hex。例: `#3B82F6`） |
| `is_seal_required` | BOOLEAN | NO | FALSE | 承認時に電子印鑑の押印を必須とするか（F05.3 連携） |
| `is_active` | BOOLEAN | NO | TRUE | テンプレートが有効か（無効化すると新規申請不可。既存申請は継続処理可能） |
| `sort_order` | INT | NO | 0 | 表示順 |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。作成した ADMIN。ON DELETE RESTRICT |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_wt_scope (scope_type, scope_id, is_active, sort_order)  -- スコープ別の有効テンプレート一覧
```

**制約・備考**
- `scope_type` + `scope_id` はポリモーフィック参照。FK 制約なし（アプリ層で検証）
- 1スコープあたり最大 **30 テンプレート**（`is_active` に関わらず論理削除されていないもの）
- テンプレートを論理削除しても、処理中（`PENDING`）の申請は継続処理可能
- `is_active = FALSE`: 既存の処理中申請は継続可能だが、新規申請は不可
- `is_seal_required = TRUE`: 各承認ステップの承認者が判定時に電子印鑑の押印を行う。`SealStampService.stamp()` を内部呼び出し。`target_type = 'WORKFLOW_APPROVAL'`

#### `workflow_template_steps`

テンプレートの承認ステップ定義。1〜3段階。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `template_id` | BIGINT UNSIGNED | NO | — | FK → workflow_templates。ON DELETE CASCADE |
| `step_order` | SMALLINT | NO | — | ステップ順序（1, 2, 3） |
| `name` | VARCHAR(100) | NO | — | ステップ名（例: 「課長承認」「部長承認」「経理部確認」） |
| `approval_type` | VARCHAR(20) | NO | — | 承認方式（`ALL`=全員承認 / `ANY`=誰か1人が承認） |
| `approver_type` | VARCHAR(20) | NO | — | 承認者指定方式（`ROLE`=ロールで指定 / `USER`=個人で指定） |
| `approver_role` | VARCHAR(30) | YES | NULL | `approver_type = 'ROLE'` の場合のロール名（`ADMIN` / `DEPUTY_ADMIN`） |
| `approver_user_ids` | JSON | YES | NULL | `approver_type = 'USER'` の場合の承認者 user_id 配列（例: `[1, 5, 12]`） |
| `auto_approve_days` | SMALLINT | YES | NULL | 自動承認期限（日数）。指定日数経過後に未判定なら自動承認。NULL=期限なし |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_wts_template_order (template_id, step_order)  -- テンプレート内でステップ順序一意
```

**制約・備考**
- 1テンプレートあたり最大 **3 ステップ**
- `step_order` は 1 始まりの連番。ステップは順次実行（ステップ1が完了してからステップ2へ進む）
- `approval_type = 'ALL'`: 全承認者が承認するまでステップ未完了。1人でも却下すれば申請全体が却下
- `approval_type = 'ANY'`: 承認者のうち1人が承認すればステップ完了。全員が却下した場合のみ申請却下
- `approver_type = 'ROLE'`: 申請作成時に、スコープ内の該当ロールメンバーが自動的に承認者として `workflow_request_approvers` に登録される
- `approver_type = 'USER'`: `approver_user_ids` に指定されたユーザーが承認者として登録される
- `auto_approve_days`: 長期不在で承認が停滞するのを防止。自動承認時は `workflow_request_approvers.decision = 'AUTO_APPROVED'` として記録。日次バッチで判定

テンプレートのカスタムフィールド定義。申請者が入力する項目を ADMIN が定義する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `template_id` | BIGINT UNSIGNED | NO | — | FK → workflow_templates。ON DELETE CASCADE |
| `field_key` | VARCHAR(50) | NO | — | フィールドキー（英数字・アンダースコア） |
| `field_label` | VARCHAR(100) | NO | — | 表示ラベル（例: 「申請金額」「利用希望日」「申請理由」） |
| `field_type` | VARCHAR(20) | NO | — | フィールド型（`TEXT` / `NUMBER` / `DATE` / `SELECT` / `CHECKBOX` / `TEXTAREA` / `FILE`） |
| `is_required` | BOOLEAN | NO | FALSE | 必須フィールドか |
| `options_json` | JSON | YES | NULL | `SELECT` 型の選択肢（例: `["備品購入", "消耗品購入", "その他"]`） |
| `placeholder` | VARCHAR(200) | YES | NULL | プレースホルダーテキスト |
| `unit` | VARCHAR(20) | YES | NULL | 単位（例: 「円」「日」。`NUMBER` 型の場合に表示） |
| `sort_order` | INT | NO | 0 | 表示順 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_wtf_template (template_id, sort_order)      -- テンプレート別のフィールド一覧
UNIQUE KEY uq_wtf_key (template_id, field_key)         -- テンプレート内でフィールドキー一意
```

**制約・備考**
- 1テンプレートあたり最大 **15 フィールド**
- `field_type` ごとの保存形式（F06.4 activity_records と同一方式）:
  - `TEXT`: 文字列（最大 200 文字）
  - `NUMBER`: 数値（DECIMAL(10,2)）
  - `DATE`: 日付（YYYY-MM-DD）
  - `SELECT`: 選択肢の値（`options_json` から選択）
  - `CHECKBOX`: 真偽値（true/false）
  - `TEXTAREA`: 長文（最大 2,000 文字）
  - `FILE`: ファイルキー（S3 の `file_key`。`workflow_request_attachments` と紐付け）
- フィールド定義の変更ルール:
  - **追加**: 許可。既存申請には新フィールドの値がない → 表示時に「未記録」と表示
  - **削除**: 許可。既存申請に旧フィールドの値が残る → 表示時に「（削除済みフィールド）」として値を表示
  - **`field_type` 変更**: 禁止
  - **`field_key` 変更**: 禁止

#### `workflow_requests`

申請レコード。申請者が作成し、承認ステップを経て完了する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `template_id` | BIGINT UNSIGNED | NO | — | FK → workflow_templates。使用テンプレート。ON DELETE RESTRICT |
| `title` | VARCHAR(200) | NO | — | 申請タイトル（例: 「2026年4月 出張旅費精算」） |
| `field_values` | JSON | NO | '{}' | カスタムフィールドの値（`{ "amount": 15000, "reason": "出張旅費", "date": "2026-04-15" }`） |
| `status` | VARCHAR(20) | NO | 'DRAFT' | ステータス（`DRAFT` / `PENDING` / `APPROVED` / `REJECTED` / `RETURNED` / `CANCELLED`） |
| `current_step_order` | SMALLINT | YES | NULL | 現在の承認ステップ順序。DRAFT の場合 NULL。APPROVED/REJECTED/RETURNED/CANCELLED の場合は最後に処理されたステップ |
| `source_type` | VARCHAR(50) | YES | NULL | 他機能からの呼び出し元種別（例: `EXPENSE`, `EQUIPMENT_REQUEST`, `FACILITY_USAGE`） |
| `source_id` | BIGINT UNSIGNED | YES | NULL | 呼び出し元の PK |
| `requested_by` | BIGINT UNSIGNED | NO | — | FK → users（申請者）。ON DELETE RESTRICT |
| `requested_at` | DATETIME | YES | NULL | 申請日時（`DRAFT` → `PENDING` に遷移した日時） |
| `completed_at` | DATETIME | YES | NULL | 完了日時（`APPROVED` / `REJECTED` / `CANCELLED` に遷移した日時） |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_wr_scope (scope_type, scope_id, status, requested_at DESC)      -- スコープ別の申請一覧
INDEX idx_wr_requested_by (requested_by, status, requested_at DESC)        -- 申請者別の申請一覧
INDEX idx_wr_template (template_id, status, requested_at DESC)             -- テンプレート別の申請一覧
INDEX idx_wr_source (source_type, source_id)                               -- 呼び出し元からの逆引き
```

**制約・備考**
- `status` 遷移:
  - `DRAFT` → `PENDING`（申請提出）
  - `PENDING` → `APPROVED`（全ステップ承認完了）
  - `PENDING` → `REJECTED`（いずれかのステップで却下）
  - `PENDING` → `RETURNED`（差し戻し。申請者が修正して再提出可能）
  - `DRAFT` / `PENDING` → `CANCELLED`（申請者またはADMINが取消）
  - `RETURNED` → `PENDING`（修正後の再提出）
  - `APPROVED` / `REJECTED` / `CANCELLED` からの遷移は不可
- `source_type` + `source_id`: 他機能がワークフローエンジンを利用する場合の呼び出し元追跡。NULL の場合は汎用申請（ワークフロー単体利用）
- `field_values`: `workflow_template_fields` で定義されたフィールドの値を JSON で保存。Service 層でバリデーション
- `requested_by` ON DELETE RESTRICT: 申請者の退会時、処理中の申請がある場合はエラー。ADMIN が先に `CANCELLED` にするか申請を完了させる
- 論理削除: `DRAFT` / `CANCELLED` の申請のみ削除可能。`PENDING` / `APPROVED` / `REJECTED` / `RETURNED` は削除不可（証跡保持）

#### `workflow_request_steps`

申請の各承認ステップ実行状態。申請提出時に `workflow_template_steps` から生成される。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `request_id` | BIGINT UNSIGNED | NO | — | FK → workflow_requests。ON DELETE CASCADE |
| `step_order` | SMALLINT | NO | — | ステップ順序（テンプレートからコピー） |
| `name` | VARCHAR(100) | NO | — | ステップ名（テンプレートからコピー） |
| `approval_type` | VARCHAR(20) | NO | — | 承認方式（テンプレートからコピー。`ALL` / `ANY`） |
| `status` | VARCHAR(20) | NO | 'WAITING' | ステップ状態（`WAITING` / `IN_PROGRESS` / `APPROVED` / `REJECTED` / `RETURNED` / `SKIPPED`） |
| `started_at` | DATETIME | YES | NULL | ステップ処理開始日時 |
| `completed_at` | DATETIME | YES | NULL | ステップ完了日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_wrs_request_order (request_id, step_order)  -- 申請内でステップ順序一意
INDEX idx_wrs_status (status, started_at)                   -- 処理中ステップの一覧
```

**制約・備考**
- `status` 遷移:
  - `WAITING` → `IN_PROGRESS`（前のステップが完了して順番が来た）
  - `IN_PROGRESS` → `APPROVED` / `REJECTED` / `RETURNED`（承認者の判定結果）
  - `WAITING` → `SKIPPED`（前のステップで却下/差し戻しされた場合、後続ステップをスキップ）
- テンプレートからのスナップショット: ステップ名・承認方式はテンプレートから申請提出時にコピー。提出後にテンプレートを変更しても処理中の申請には影響しない

#### `workflow_request_approvers`

各ステップの承認者と判定結果。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `request_step_id` | BIGINT UNSIGNED | NO | — | FK → workflow_request_steps。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（承認者）。ON DELETE RESTRICT |
| `decision` | VARCHAR(20) | YES | NULL | 判定結果（NULL=未判定 / `APPROVED` / `REJECTED` / `RETURNED` / `AUTO_APPROVED`） |
| `comment` | TEXT | YES | NULL | 判定時のコメント（最大 2,000 文字） |
| `stamp_log_id` | BIGINT UNSIGNED | YES | NULL | FK → seal_stamp_logs。電子印鑑の押印記録（`is_seal_required = TRUE` の場合）。ON DELETE SET NULL |
| `decided_at` | DATETIME | YES | NULL | 判定日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_wra_step_user (request_step_id, user_id)       -- ステップ内で承認者一意
INDEX idx_wra_user (user_id, decision)                         -- 承認者別の未判定一覧
INDEX idx_wra_stamp (stamp_log_id)                             -- 押印記録からの逆引き
```

**制約・備考**
- 申請提出時にテンプレートの `approver_type` に応じて承認者を展開して INSERT
- `user_id` ON DELETE RESTRICT: 承認者が退会する場合、未判定の承認がある場合はエラー。ADMIN が先に申請を却下/取消するか、別の承認者に差し替える
- `stamp_log_id`: テンプレートの `is_seal_required = TRUE` の場合に、承認時に `SealStampService.stamp()` を呼び出し、返却された stamp_log_id を保存。`target_type = 'WORKFLOW_APPROVAL'`、`target_id = workflow_request_approvers.id`
- `AUTO_APPROVED`: `auto_approve_days` 超過時に日次バッチで自動設定。`comment` に「自動承認（期限超過）」を自動記入

| `file_size` | INT UNSIGNED | NO | — | ファイルサイズ（バイト。上限 20MB） |
| `mime_type` | VARCHAR(100) | NO | — | MIMEタイプ |
| `sort_order` | SMALLINT | NO | 0 | 表示順 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_wratt_request (request_id, sort_order)   -- 申請別添付ファイル一覧
```

**制約・備考**
- 対応ファイル形式: PDF, JPEG, PNG, WebP, GIF, XLSX, DOCX, CSV
- ファイルサイズ上限: 1ファイル 20MB、1申請あたり合計 50MB
- 最大 10 ファイル/申請
- S3 配信: Pre-signed URL（有効期限: 1時間）

## 5. ビジネスロジック

### 申請提出フロー

```
1. 申請者が DRAFT 状態の申請を作成（または RETURNED の申請を修正）
2. 申請者が submit API を呼び出し
3. Service 層で以下を実行:
   a. field_values のバリデーション（workflow_template_fields の定義に基づき型チェック・必須チェック）
   b. status を PENDING に更新、requested_at をセット
   c. workflow_template_steps からスナップショットを生成 → workflow_request_steps に INSERT
   d. 各ステップの承認者を展開:
      - approver_type = 'ROLE': スコープ内の該当ロールメンバーを取得して INSERT
      - approver_type = 'USER': approver_user_ids のユーザーを INSERT
   e. ステップ1の status を IN_PROGRESS に更新
   f. current_step_order = 1 に更新
   g. ApplicationEvent (WorkflowRequestSubmittedEvent) を発行
4. 通知サービスがイベントを受信:
   a. ステップ1の全承認者にプッシュ通知（「新しい承認依頼があります」）
   b. 申請者に提出完了通知
```

### 承認判定フロー

```
1. 承認者が approve / reject / return API を呼び出し
2. Service 層で以下を実行:
   a. 楽観的ロック（version）チェック
   b. 承認者が現在のステップの承認者であることを検証
   c. 該当 workflow_request_approvers の decision, comment, decided_at を更新
   d. is_seal_required = TRUE の場合:
      - SealStampService.stamp() を呼び出し（target_type = 'WORKFLOW_APPROVAL'）
      - stamp_log_id を保存
   e. ステップの完了判定（後述）
   f. ApplicationEvent (WorkflowDecisionEvent) を発行
3. 通知サービスがイベントを受信:
   a. 承認: 申請者に通知。全ステップ完了時は「申請が承認されました」
   b. 却下: 申請者に通知（「申請が却下されました」+ コメント）
   c. 差し戻し: 申請者に通知（「申請が差し戻されました」+ コメント）
```

### ステップ完了判定ロジック

```
■ approval_type = 'ALL'（全員承認）の場合:
  1. 全承認者の decision を確認
  2. 1人でも REJECTED → ステップ: REJECTED → 申請: REJECTED（後続ステップは SKIPPED）
  3. 1人でも RETURNED → ステップ: RETURNED → 申請: RETURNED（後続ステップは SKIPPED）
  4. 全員 APPROVED / AUTO_APPROVED → ステップ: APPROVED → 次ステップへ進行

■ approval_type = 'ANY'（誰か1人が承認）の場合:
  1. 1人でも APPROVED / AUTO_APPROVED → ステップ: APPROVED → 次ステップへ進行
  2. 全員 REJECTED → ステップ: REJECTED → 申請: REJECTED
  3. 1人でも RETURNED（かつ全員未承認）→ ステップ: RETURNED → 申請: RETURNED
  4. 一部が判定済み、残りが未判定 → ステップ: IN_PROGRESS（継続）

■ 次ステップへの進行:
  1. 現在のステップが APPROVED
  2. 次の step_order のステップが存在する → 次ステップの status を IN_PROGRESS に更新
  3. 次ステップの承認者に通知
  4. current_step_order を更新

■ 全ステップ完了:
  1. 最後のステップが APPROVED → 申請 status = APPROVED、completed_at をセット
  2. source_type がある場合、ApplicationEvent (WorkflowApprovedEvent) を発行
     呼び出し元機能がイベントを購読して後続処理（例: 施設利用枠の確定）
```

### 自動承認バッチ

```
1. 日次バッチ（00:30 JST）で実行
2. workflow_request_steps.status = 'IN_PROGRESS' のステップを取得
3. 各ステップに紐づく workflow_template_steps.auto_approve_days を確認
4. workflow_request_steps.started_at + auto_approve_days < 現在日時 の場合:
   a. 未判定（decision IS NULL）の全承認者を AUTO_APPROVED に更新
   b. comment に「自動承認（期限超過）」を自動記入
   c. ステップ完了判定を実行
   d. 監査ログに記録（WORKFLOW_AUTO_APPROVED）
5. 自動承認が発動した場合、申請者と元の承認者に通知
```

### 差し戻し後の再提出フロー

```
1. 承認者が return API で差し戻し
2. 申請 status = RETURNED、全ステップの status をリセット:
   - RETURNED / IN_PROGRESS → WAITING に戻す
   - 後続の WAITING はそのまま
3. 全承認者の decision をリセット（NULL に戻す）
4. 申請者が申請内容（field_values / title / attachments）を修正
5. 申請者が submit API で再提出
6. 再度ステップ1から承認フロー開始
```

### 他機能連携パターン

```
■ 他機能からワークフローを利用する場合のシーケンス:

1. 他機能の Controller/Service が external API を呼び出し:
   POST /workflows/templates/{templateId}/requests/external
   - source_type: "FACILITY_USAGE"
   - source_id: 15 (facility_reservations.id)
   - auto_submit: true

2. ワークフローエンジンが申請を作成 → 承認フロー開始

3. 承認完了時:
   WorkflowApprovedEvent { requestId, sourceType, sourceId } を発行

4. 他機能（例: FacilityReservationService）がイベントを購読:
   @TransactionalEventListener(phase = AFTER_COMMIT)
   public void onWorkflowApproved(WorkflowApprovedEvent event) {
       if ("FACILITY_USAGE".equals(event.getSourceType())) {
           facilityReservationService.confirmReservation(event.getSourceId());
       }
   }

5. 却下時: WorkflowRejectedEvent を発行 → 他機能が予約キャンセル等
```

### 重要な判定ロジック

- **承認者の権限**: 承認者として `workflow_request_approvers` に登録されているユーザーのみが判定可能。ADMIN であってもステップの承認者でなければ判定不可（ただし ADMIN は申請の強制却下/取消は可能）
- **申請の可視性**: ADMIN は全申請を閲覧可能。MEMBER は自分の申請 + 自分が承認者として指定された申請のみ
- **DRAFT 状態での編集**: `field_values` / `title` / 添付ファイルのすべてを編集可能
- **RETURNED 状態での編集**: `field_values` / `title` / 添付ファイルのすべてを編集可能（差し戻し理由に基づき修正）
- **PENDING 状態での編集**: 不可（承認フロー進行中の内容変更を防止）
- **申請取消**: 申請者は `DRAFT` / `PENDING` の申請を取消可能。ADMIN は `PENDING` の申請を強制取消可能
- **テンプレート変更の影響**: 処理中の申請はテンプレートからのスナップショットで動作するため、テンプレート変更の影響を受けない
- **承認者の退会**: 未判定の承認がある場合は退会不可。ADMIN が先に申請を却下/取消するか、テンプレートの承認者設定を変更して新しい申請で対応

### 通知トリガー一覧

| トリガー | 通知先 | 条件 |
|---------|-------|------|
| 申請提出 | ステップ1の全承認者 | 申請が PENDING に遷移 |
| 承認（ステップ進行） | 次ステップの全承認者 + 申請者 | 現ステップが APPROVED で次ステップへ進行 |
| 承認（全ステップ完了） | 申請者 | 最終ステップが APPROVED |
| 却下 | 申請者 | いずれかのステップで REJECTED |
| 差し戻し | 申請者 | いずれかのステップで RETURNED |
| 取消 | 全承認者 | 申請が CANCELLED |
| 自動承認 | 申請者 + 元の承認者 | auto_approve_days 超過 |
| コメント投稿 | 申請者 + 全承認者（投稿者自身を除く） | コメント追加時 |
| 承認リマインド | 未判定の承認者 | IN_PROGRESS ステップで3日経過後、毎日1回 |

## 6. セキュリティ考慮事項

- **認可チェック**: WorkflowService の入り口で `scope_type` / `scope_id` と `currentUser` の所属を必ず検証する。申請の閲覧は申請者本人・承認者・ADMIN に限定
- **レートリミット**: 申請作成 API に `Bucket4j` で 1時間に20回の制限を適用（大量申請によるスパム防止）
- **楽観的ロック**: `workflow_requests` と `workflow_templates` に `version` カラムを適用。承認操作時に `version` の一致を検証し、同時判定による不整合を防止
- **カスタムフィールドのバリデーション**: `field_values` JSON の値を `workflow_template_fields` の定義に基づき厳密にバリデーション（型チェック・必須チェック・文字数チェック・選択肢チェック）
- **添付ファイル**: Pre-signed URL 方式でアップロード。`contentType` ホワイトリスト検証 + `fileSize` 上限チェック
- **証跡の不変性**: 承認判定（`workflow_request_approvers.decision`）は一度記録すると変更不可。差し戻しで再提出された場合は新しいステップインスタンスが生成される
- **ADMIN 強制取消の監査**: ADMIN が申請を強制取消した場合、監査ログに `WORKFLOW_FORCE_CANCELLED` として記録し、申請者に通知
