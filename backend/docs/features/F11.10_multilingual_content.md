# F11.10: 多言語コンテンツサポート

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 11
> **最終更新**: 2026-03-19
> **モジュール種別**: 選択式モジュール #18
> **関連ドキュメント**: F06.1 CMS（ブログ・お知らせ）、F04.1 タイムライン（ANNOUNCEMENT 投稿）、F11.4 ナレッジベース（内部 Wiki）

## 1. 概要

UI 翻訳（デフォルト機能 #14: 多言語対応 i18n）が画面ラベルやメッセージの言語切替を担うのに対し、本モジュールはユーザーが作成するコンテンツ自体（ブログ記事・お知らせ・ナレッジベースページ等）を多言語で管理する機能を提供する。1つの原文コンテンツに対して複数言語の翻訳版を紐付け、読者の `users.locale` 設定に応じて自動的に適切な言語版を表示する。原文が更新された場合は翻訳版のステータスを「要更新」に自動変更し、翻訳の鮮度を管理する。

**ユースケース**:
- **国際コミュニティ**: 在日外国人コミュニティの多言語お知らせ配信
- **国際スポーツチーム**: 外国人選手向けの練習スケジュール・連絡事項の多言語化
- **インターナショナルスクール**: 保護者向け通知の日英併記
- **多国籍企業の日本支社**: 社内通知・マニュアルの日英中韓対応

**Phase 11 の範囲**: 手動翻訳のみ。将来的な機械翻訳 API 連携（Claude / DeepL）のフックを設計に含めるが、実装は Phase 11 では行わない。

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全組織・チームの翻訳設定・翻訳データを参照。プラットフォームデフォルト対応言語の管理 |
| ADMIN | 翻訳対象言語の有効化/無効化、翻訳の作成・編集・削除、翻訳ステータス管理、翻訳者の指定 |
| DEPUTY_ADMIN | `MANAGE_TRANSLATIONS` 権限を持つ場合: 翻訳の作成・編集・削除、翻訳ステータス管理 |
| MEMBER | 翻訳済みコンテンツの閲覧（自動言語選択）。`WRITE_TRANSLATION` 権限を持つ場合: 割り当てられたコンテンツの翻訳作成・編集 |
| SUPPORTER | 翻訳済みコンテンツの閲覧（自動言語選択） |
| GUEST | 翻訳済み公開コンテンツの閲覧（自動言語選択） |

### 対象レベル
- [x] 組織 (Organization)
- [x] チーム (Team)
- [ ] 個人 (Personal)

**個人を対象外とする理由**: 個人ブログの翻訳需要は低く、翻訳管理の運用コスト（翻訳者の割り当て・ステータス管理等）が個人には過大であるため。個人ブログの多言語化が必要な場合は、チーム/組織レベルでモジュールを有効化し、チーム内の翻訳ワークフローを利用する。

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `translation_configs` | スコープごとの多言語設定（対応言語・デフォルト言語） | なし |
| `content_translations` | コンテンツの翻訳版（ポリモーフィック参照） | あり |
| `translation_assignments` | 翻訳者のアサイン管理 | なし |

### テーブル定義

#### `translation_configs`

スコープ（組織/チーム）ごとに使用する言語の設定を管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別。値: `ORGANIZATION`, `TEAM` |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID（organizations.id / teams.id） |
| `primary_language` | VARCHAR(5) | NO | 'ja' | 原文の言語コード（ISO 639-1）。値: `ja`, `en`, `ko`, `zh`, `pt`, `de` |
| `enabled_languages` | JSON | NO | '[]' | 有効な翻訳対象言語コードの配列。例: `["en","ko"]`。primary_language は含まない |
| `is_auto_detect_reader_language` | BOOLEAN | NO | TRUE | 読者の `users.locale` から自動言語選択を行うか。FALSE の場合は常に原文を表示 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_tc_scope (scope_type, scope_id)    -- 1スコープにつき1設定
```

**制約・備考**
- `scope_type + scope_id`: ポリモーフィック FK。DB レベルの FK 制約は設けない。Service 層で `scope_type` に応じた存在チェックを実行
- `enabled_languages`: JSON 配列として保存。Java 側では `List<String>` にマッピング（JPA `@Convert` + Jackson）
- `primary_language` は `enabled_languages` に含めない（原文は翻訳不要のため）
- ADMIN がスコープごとに `enabled_languages` を設定。プラットフォームでサポートする言語は `ja`, `en`, `ko`, `zh`, `pt`, `de` の6言語固定（SYSTEM_ADMIN がコード変更なしに追加することはできない）
- `is_auto_detect_reader_language = FALSE` の場合、翻訳版は存在しても自動選択されない。読者が手動で `?lang=en` を指定した場合のみ翻訳版を返却する

#### `content_translations`

原文コンテンツに対する翻訳版を保存する。ポリモーフィック参照で複数コンテンツタイプに対応。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別。値: `ORGANIZATION`, `TEAM` |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `source_type` | VARCHAR(30) | NO | — | 原文コンテンツの種別。値: `BLOG_POST`, `ANNOUNCEMENT`, `KNOWLEDGE_BASE` |
| `source_id` | BIGINT UNSIGNED | NO | — | 原文コンテンツの ID（blog_posts.id / kb_pages.id 等） |
| `language` | VARCHAR(5) | NO | — | 翻訳先の言語コード（ISO 639-1） |
| `translated_title` | VARCHAR(200) | NO | — | 翻訳済みタイトル |
| `translated_body` | TEXT | NO | — | 翻訳済み本文（Markdown 形式） |
| `translated_excerpt` | VARCHAR(500) | YES | NULL | 翻訳済み要約・抜粋（原文に excerpt がある場合） |
| `status` | VARCHAR(20) | NO | 'DRAFT' | 翻訳ステータス。値: `DRAFT`（未翻訳/作成中）, `IN_REVIEW`（レビュー中）, `PUBLISHED`（翻訳完了・公開済み）, `NEEDS_UPDATE`（原文更新により要更新） |
| `translator_id` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）翻訳者 |
| `reviewer_id` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）レビュアー |
| `source_updated_at` | DATETIME | NO | — | 翻訳作成/更新時点の原文 `updated_at` スナップショット。原文の現在の `updated_at` と比較して要更新判定に使用 |
| `published_at` | DATETIME | YES | NULL | 翻訳の公開日時（PUBLISHED に遷移した日時） |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
-- 1つの原文コンテンツ × 1言語につき翻訳は1件（論理削除対応）
UNIQUE KEY uq_ct_source_lang (source_type, source_id, language, deleted_at)
INDEX idx_ct_scope_status (scope_type, scope_id, status)                     -- スコープ別翻訳ステータス一覧
INDEX idx_ct_source (source_type, source_id, deleted_at)                     -- 原文コンテンツの翻訳一覧取得
INDEX idx_ct_translator (translator_id, status)                               -- 翻訳者別タスク一覧
INDEX idx_ct_needs_update (status, source_type, scope_type, scope_id)        -- 要更新翻訳の検索
```

**制約・備考**
- 論理削除: `deleted_at DATETIME nullable`（SoftDeletableEntity 適用）
- `source_type + source_id`: ポリモーフィック FK。DB レベルの FK 制約は設けない。Service 層で `source_type` に応じた存在チェックを実行
- `UNIQUE (source_type, source_id, language, deleted_at)`: MySQL の `NULL != NULL` 特性を利用した論理削除対応パターン。未削除レコードは1つの原文×1言語に1件のみ保証
- `source_updated_at`: 翻訳作成・更新時に原文の `updated_at` をスナップショットとして保存。バッチまたは原文更新イベントで `source_updated_at != 原文.updated_at` を検知し、`status = 'NEEDS_UPDATE'` に自動更新
- `ANNOUNCEMENT` は `blog_posts` テーブルの `post_type = 'ANNOUNCEMENT'` のレコードを指す。`source_type` は `BLOG_POST` ではなく `ANNOUNCEMENT` として区別する（管理画面でのフィルタリング用途）
- `KNOWLEDGE_BASE` は `kb_pages` テーブル（F11.4）のレコードを指す
- 原文テーブル（`blog_posts`, `kb_pages` 等）は変更しない。翻訳テキストは本テーブルに保存
- `translated_body` のサイズ制限は原文と同一（50,000文字）
- 楽観的ロック: `version` カラムで同時編集を検知

#### `translation_assignments`

翻訳者のアサイン管理。ADMIN が特定コンテンツの特定言語への翻訳を MEMBER に割り当てる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別。値: `ORGANIZATION`, `TEAM` |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE）翻訳者 |
| `language` | VARCHAR(5) | NO | — | 担当言語コード |
| `is_active` | BOOLEAN | NO | TRUE | アサインが有効か |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_ta_scope_user_lang (scope_type, scope_id, user_id, language)   -- 1ユーザー×1スコープ×1言語に1アサイン
INDEX idx_ta_scope_lang (scope_type, scope_id, language, is_active)           -- スコープ×言語別の有効翻訳者一覧
INDEX idx_ta_user (user_id, is_active)                                        -- ユーザー別アサイン一覧
```

**制約・備考**
- `WRITE_TRANSLATION` 権限を持つ MEMBER は、自分がアサインされた言語のコンテンツのみ翻訳作成・編集が可能
- ADMIN/DEPUTY_ADMIN は `translation_assignments` に依存せず全言語の翻訳を作成・編集可能
- 1ユーザーが複数言語にアサインされることも可能（レコードを複数作成）
- `is_active = FALSE` でアサインを無効化（履歴保持のため物理削除しない）

### ER図（テキスト形式）
```
translation_configs (1) ──── (N) content_translations
  │ scope_type + scope_id            │ scope_type + scope_id
  │                                  │
  │                                  ├── source_type = 'BLOG_POST' ──── (1) blog_posts
  │                                  ├── source_type = 'ANNOUNCEMENT' ─ (1) blog_posts (post_type='ANNOUNCEMENT')
  │                                  └── source_type = 'KNOWLEDGE_BASE' (1) kb_pages
  │
  └──── (N) translation_assignments
         │ scope_type + scope_id
         │
         └── user_id ──── (1) users
```

#### `PATCH /api/v1/teams/{teamId}/translations/{id}/status`

**リクエストボディ**
```json
{
  "status": "PUBLISHED",
  "version": 1
}
```

**レスポンス（200 OK）**
```json
{
  "data": {
    "id": 1,
    "status": "PUBLISHED",
    "publishedAt": "2026-03-19T12:00:00",
    "version": 2,
    "updatedAt": "2026-03-19T12:00:00"
  }
}
```

**ステータス遷移ルール**:
- `DRAFT` → `IN_REVIEW` / `PUBLISHED`（ADMIN/DEPUTY_ADMIN は直接 PUBLISHED 可。MEMBER は IN_REVIEW まで）
- `IN_REVIEW` → `PUBLISHED` / `DRAFT`（差し戻し）
- `PUBLISHED` → `DRAFT`（非公開に戻す）
- `NEEDS_UPDATE` → `DRAFT`（再翻訳開始）/ `PUBLISHED`（更新不要と判断した場合）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | 無効なステータス遷移 |
| 403 | 権限不足（MEMBER が PUBLISHED に遷移しようとした場合） |
| 404 | 翻訳が存在しない |
| 409 | 楽観的ロック競合 |

#### `GET /api/v1/teams/{teamId}/translations`

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `status` | String | いいえ | フィルタ: `DRAFT`, `IN_REVIEW`, `PUBLISHED`, `NEEDS_UPDATE` |
| `language` | String | いいえ | フィルタ: 言語コード |
| `sourceType` | String | いいえ | フィルタ: `BLOG_POST`, `ANNOUNCEMENT`, `KNOWLEDGE_BASE` |
| `page` | Integer | いいえ | ページ番号（0始まり、デフォルト0） |
| `size` | Integer | いいえ | 1ページあたり件数（デフォルト20、最大100） |
| `sort` | String | いいえ | ソート（デフォルト `updatedAt,desc`） |

**レスポンス（200 OK）**
```json
{
```

#### 既存 API への翻訳機能拡張

既存のコンテンツ取得 API に言語選択パラメータを追加する。**既存 API のリクエスト/レスポンス構造は変更しない**。

| 既存 API | 追加パラメータ | 動作 |
|----------|-------------|------|
| `GET /api/v1/blog/posts/{slug}` | `Accept-Language` ヘッダー or `?lang=en` | 翻訳版が存在し PUBLISHED なら翻訳テキストを返却。なければ原文を返却 |
| `GET /api/v1/blog/posts` | `?lang=en` | 一覧の title/excerpt を翻訳版で返却（PUBLISHED のみ） |
| `GET /api/v1/kb/pages/{id}` | `Accept-Language` ヘッダー or `?lang=en` | 同上 |

**言語選択の優先順位**:
1. `?lang=en` クエリパラメータ（最優先）
2. `Accept-Language` ヘッダーの最優先言語
3. 認証済みユーザーの `users.locale` 設定
4. `translation_configs.primary_language`（フォールバック: 原文）

**レスポンスの追加フィールド**（翻訳版が返却された場合）:
```json
{
  "data": {
    "id": 123,
    "title": "Monthly Activity Report",
    "body": "## Summary\n...",
    "excerpt": "Monthly activity summary",
    "_translation": {
      "language": "en",
      "originalLanguage": "ja",
      "translationId": 1,
      "translatedAt": "2026-03-19T12:00:00",
      "availableLanguages": ["ja", "en", "ko"]
    }
  }
}
```

`_translation` フィールドは翻訳版が返却された場合のみ付与。原文が返却された場合は `_translation` は `null`（ただし `availableLanguages` のみ返却）:
```json
{

    "id": 123,
    "title": "月次活動報告",
```

## 5. ビジネスロジック

### 主要フロー

#### 翻訳作成フロー
```
1. ADMIN/翻訳者が翻訳対象コンテンツと言語を選択
2. バックエンドが以下を検証:
   a. モジュールが有効か（team_enabled_modules で確認）
   b. 指定言語が translation_configs.enabled_languages に含まれるか
   c. 同一原文×同一言語の翻訳が存在しないか
   d. MEMBER の場合: WRITE_TRANSLATION 権限 + 該当言語のアサインがあるか
3. 原文コンテンツの現在の updated_at を source_updated_at として保存
4. content_translations にレコード作成（status = DRAFT）
5. ApplicationEvent（TranslationCreatedEvent）を発行
6. 通知サービスがイベントを受信し、レビュアー（ADMIN）にアプリ内通知
```

#### 原文更新時の翻訳ステータス自動更新フロー
```
1. 原文コンテンツ（blog_posts / kb_pages）が更新される
2. 原文 Service が BlogPostUpdatedEvent / KbPageUpdatedEvent を発行
3. TranslationSyncListener がイベントを受信（@TransactionalEventListener(AFTER_COMMIT)）
4. 該当原文の全翻訳（status = PUBLISHED）を検索
5. 各翻訳の source_updated_at と原文の新 updated_at を比較
6. 不一致の場合、status を NEEDS_UPDATE に更新
7. 翻訳者に「原文が更新されました。翻訳の見直しが必要です」通知を送信
```

#### 読者への自動言語選択フロー
```
1. 読者がコンテンツ取得 API をリクエスト
2. 言語選択の優先順位に基づいて対象言語を決定:
   a. ?lang パラメータ → b. Accept-Language ヘッダー → c. users.locale → d. primary_language
3. スコープの translation_configs を確認:
   a. is_auto_detect_reader_language = FALSE → 原文を返却（?lang 指定時は除く）
   b. モジュール未有効 → 原文を返却
4. 決定された言語の翻訳が content_translations に存在し status = PUBLISHED か確認
5. 存在する場合: translated_title / translated_body / translated_excerpt でレスポンスの title / body / excerpt を差し替え。_translation メタ情報を付与
6. 存在しない場合: 原文を返却。_translation.availableLanguages のみ返却
```

#### 翻訳ステータス遷移図
```
                    ┌─────────────────────────────┐
                    │                             │
                    ▼                             │
               ┌─────────┐    レビュー提出    ┌──────────┐    承認     ┌───────────┐
  作成 ────>   │  DRAFT  │ ──────────────> │ IN_REVIEW │ ────────> │ PUBLISHED │
               └─────────┘                 └──────────┘           └───────────┘
                    ▲                          │                       │
                    │         差し戻し          │                       │
                    └──────────────────────────┘                       │
                    ▲                                                  │
                    │              非公開化                              │
                    └──────────────────────────────────────────────────┘
                    ▲
                    │              原文更新検知
               ┌──────────────┐ ──────────────────┐
               │ NEEDS_UPDATE │                    │
               └──────────────┘                    │
                    │     再翻訳開始(→DRAFT)        │
                    │     更新不要判断(→PUBLISHED)   │
                    └──────────────────────────────┘
```

### 重要な判定ロジック

- **言語コードバリデーション**: サポート言語は `ja`, `en`, `ko`, `zh`, `pt`, `de` の6つに限定。それ以外の言語コードは 400 エラー
- **MEMBER の翻訳権限**: `WRITE_TRANSLATION` 権限 + `translation_assignments` で該当スコープ・言語にアサインされていること。両方を満たさない場合は 403 エラー
- **MEMBER のステータス遷移制限**: MEMBER は `DRAFT` → `IN_REVIEW` のみ可能。`PUBLISHED` への遷移は ADMIN/DEPUTY_ADMIN のみ
- **原文削除時**: 原文が論理削除された場合、紐付く翻訳も論理削除する（ApplicationEvent 経由）
- **翻訳設定の言語削除時**: `enabled_languages` から言語を削除した場合、その言語の翻訳は削除されない（非表示になるだけ）。再度有効化すれば既存翻訳が復活する
- **翻訳の NEEDS_UPDATE 判定**: `content_translations.source_updated_at` と原文テーブルの `updated_at` を比較。厳密な文字列比較ではなく、1秒以上の差がある場合に NEEDS_UPDATE とする（DB の `DATETIME` 精度による微小差を無視）

### SSR 対応（SEO）

公開ブログ記事の SSR ページで以下の `<link>` タグを出力し、検索エンジンに多言語ページの存在を通知する:

```html
<link rel="alternate" hreflang="ja" href="https://example.com/teams/42/blog/monthly-report" />
<link rel="alternate" hreflang="en" href="https://example.com/teams/42/blog/monthly-report?lang=en" />
<link rel="alternate" hreflang="ko" href="https://example.com/teams/42/blog/monthly-report?lang=ko" />
<link rel="alternate" hreflang="x-default" href="https://example.com/teams/42/blog/monthly-report" />
```

- `hreflang` タグは翻訳版が `PUBLISHED` の言語のみ出力
- `x-default` は原文の URL
- SSR 時に `Accept-Language` ヘッダーまたは `?lang` パラメータに基づいて適切な翻訳版をレンダリング

## 6. セキュリティ考慮事項

- **認可チェック**: TranslationService の入り口で `scope_type + scope_id` と `currentUser` の所属を必ず検証する。MEMBER は `WRITE_TRANSLATION` 権限 + アサインの両方を検証
- **XSS 対策**: `translated_body` は Markdown 形式で保存。フロントエンドでのレンダリング時に `markdown-it` + `DOMPurify` でサニタイズ（F06.1 と同一方式）
- **翻訳インジェクション防止**: 翻訳テキストに `<script>` タグ等の危険なコンテンツが含まれていないことを、保存時にバックエンドでも簡易チェック（HTML タグ完全除去ではなく、Markdown として妥当な範囲のみ許可）
- **レートリミット**: 翻訳作成 API には `Bucket4j` で 1分間に10回の制限を適用（大量翻訳の一括登録による負荷防止）
- **原文アクセス権の継承**: 翻訳の閲覧権限は原文の `visibility` 設定に従う。原文が `MEMBERS_ONLY` であれば翻訳版も MEMBER 以上のみ閲覧可能

- [x] ~~翻訳対象コンテンツの追加（チャットメッセージ等）~~ → Phase 11 では `BLOG_POST`, `ANNOUNCEMENT`, `KNOWLEDGE_BASE` の3種別のみ対応。チャットメッセージはリアルタイム性が求められるため手動翻訳には不向き。将来の機械翻訳導入時に検討。`source_type VARCHAR(30)` のため新種別の追加は容易
- [x] ~~翻訳版の独自 slug/URL~~ → 翻訳版は独自の slug を持たない。原文の slug + `?lang=en` で一意にアクセス可能。翻訳版ごとの slug 管理は運用コストが高いため不採用
- [x] ~~ANNOUNCEMENT の source_type を BLOG_POST と分けるか~~ → 分ける。`source_type = 'ANNOUNCEMENT'` として区別する。理由: 管理画面でお知らせの翻訳状況を独立してフィルタ・集計したいため。DB 上は `blog_posts` テーブルの同一レコードを参照するが、`source_type` で論理的に区分する
