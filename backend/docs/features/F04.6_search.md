# F04.6: グローバル検索

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 4
> **最終更新**: 2026-03-18
> **モジュール種別**: デフォルト機能 #9（検索）
> **関連ドキュメント**: F04.1 タイムライン（投稿検索）、F04.2 チャット（メッセージ検索）、F05.1 掲示板（スレッド検索）、F05.5 ファイル共有（ファイル検索）、F06.1 CMS・ブログ（記事検索）、F01.1 認証（`users.is_searchable`）

---

## 1. 概要

プラットフォーム内のコンテンツを横断的に検索するグローバル検索機能。タイムライン投稿・チャットメッセージ・掲示板スレッド・ブログ記事・ファイル・メンバー・チーム/組織を統一的な検索 API で一括検索し、コンテンツ種別フィルタ・日付範囲・スコープ絞り込み・キーワードハイライトを提供する。MySQL 8.0 の FULLTEXT インデックス（N-gram パーサー + BOOLEAN MODE）を活用し、Elasticsearch 等の外部検索エンジンを導入せずにシンプルな構成で実現する。

各フィーチャーが個別に持つ検索 API（`GET /timeline/search`、`GET /chat/channels/{id}/messages/search` 等）は機能内の詳細検索として残し、本機能はそれらを**横断的に統合する入口**として位置づける。

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全コンテンツを検索可能（モデレーション用） |
| ADMIN | 自スコープの全コンテンツ（チーム/組織内の全投稿・メッセージ・ファイル等） |
| DEPUTY_ADMIN | ADMIN と同等（検索は読み取り操作のため追加パーミッション不要） |
| MEMBER | 自分がアクセス可能なコンテンツ（所属チーム/組織・参加チャンネル・公開コンテンツ） |
| SUPPORTER | 閲覧権限のあるコンテンツ（公開投稿・SUPPORTERS_AND_ABOVE 設定のコンテンツ） |
| GUEST | 対象外（検索不可） |

### 対象レベル
- [x] 組織 (Organization) — 組織内コンテンツの横断検索
- [x] チーム (Team) — チーム内コンテンツの横断検索
- [x] 個人 (Personal) — 自分が閲覧可能な全コンテンツの横断検索

## 3. 検索対象コンテンツ

| コンテンツ種別 | `type` パラメータ | 検索対象テーブル | 検索対象カラム | FULLTEXT インデックス | 認可制御 |
|--------------|-----------------|----------------|-------------|---------------------|---------|
| タイムライン投稿 | `POST` | `timeline_posts` | `content` | `ft_tp_content` (F04.1 定義済み) | 投稿の `visibility` + スコープメンバーシップ |
| チャットメッセージ | `MESSAGE` | `chat_messages` | `body` | `ft_cm_body`（本機能で追加） | チャンネルメンバーシップ（参加チャンネルのみ） |
| 掲示板スレッド | `THREAD` | `bulletin_threads` | `title` | `ft_bulletin_threads_title` (F05.1 定義済み) | スレッドの `visibility` + スコープメンバーシップ |
| ブログ記事 | `ARTICLE` | `blog_posts` | `title`, `body` | `ft_bp_title_body`（本機能で追加） | 記事の `visibility` + `status = PUBLISHED` のみ |
| ファイル | `FILE` | `shared_files` | `file_name`, `description` | `ft_sf_name_desc`（本機能で追加） | ファイルの `visibility` + スコープメンバーシップ |
| メンバー | `USER` | `users` | `last_name`, `first_name`, `display_name`, `nickname2` | `ft_users_name`（本機能で追加） | `is_searchable = TRUE` のユーザーのみ。同一スコープに所属するメンバー |
| チーム | `TEAM` | `teams` | `name` | LIKE 検索（レコード数が少ないため FULLTEXT 不要） | `is_public = TRUE` or 自分が所属するチーム |
| 組織 | `ORGANIZATION` | `organizations` | `name` | LIKE 検索 | `is_public = TRUE` or 自分が所属する組織 |
| 活動記録 | `ACTIVITY` | `activity_results` | `title` | LIKE 検索 | 活動の `visibility` + スコープメンバーシップ |

### リクエスト／レスポンス仕様

#### `GET /api/v1/search`

**認可**: MEMBER 以上

グローバル横断検索。全コンテンツ種別を一括検索し、種別ごとにグルーピングして返却する。

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `q` | String | Yes | 検索キーワード（最小2文字、最大100文字） |
| `type` | String | No | コンテンツ種別フィルタ。カンマ区切りで複数指定可（例: `POST,THREAD,ARTICLE`）。未指定: 全種別 |
| `scope_type` | String | No | スコープ種別フィルタ（`TEAM` / `ORGANIZATION`）。未指定: 全スコープ |
| `scope_id` | Long | No | スコープ ID（`scope_type` と併用） |
| `date_from` | String | No | 期間開始日（YYYY-MM-DD） |
| `date_to` | String | No | 期間終了日（YYYY-MM-DD） |
| `author_id` | Long | No | 投稿者/作成者 ID でフィルタ |
| `page` | int | No | ページ番号（デフォルト: 1） |
| `per_page` | int | No | 件数（デフォルト: 20、最大: 50） |

**レスポンス（200 OK）**

**補足**
- `snippet`: 検索キーワード周辺のテキスト抜粋（最大200文字）。キーワード部分を `<mark>` タグでハイライト。フロントエンドで太字表示
- `type_counts`: 各コンテンツ種別のヒット件数。フロントエンドの種別タブに件数バッジを表示（種別を切り替えて絞り込み可能）
- `url`: フロントエンドのルーティングパス。検索結果クリックで該当コンテンツに遷移
- **`type` フィルタ未指定時（概要ビュー）**: 全種別の `type_counts` + 各種別の上位5件ずつを返却。種別混在のマージソートページネーションは行わない（9テーブルの正確なマージソートはパフォーマンス・実装コストともに問題が大きいため）。ユーザーは `type_counts` のバッジを見て興味のある種別タブをクリックし、`type` 指定の詳細検索に遷移する
- **`type` 指定時（詳細ビュー）**: 該当種別の単一テーブルに対して `OFFSET + LIMIT` で正確にページネーション。`page` / `per_page` はこのモードでのみ有効
- `meta.view_mode`: `OVERVIEW`（type 未指定。各種別上位5件。`page` / `per_page` / `total_pages` は返却しない）or `DETAIL`（type 指定。単一種別のページネーション付き）
- `timed_out_types`: 3秒タイムアウトした種別の配列（例: `["MESSAGE"]`）。フロントエンドは該当タブに「検索がタイムアウトしました。日付範囲を絞って再検索してください」と表示。空配列 = 全種別が正常応答
- **検索履歴の自動保存**: 検索実行時に `search_histories` に自動保存（ユーザーごと最大50件。超過分は古い順に削除）
- **認可フィルタリング**: 検索結果は必ずユーザーのアクセス権限でフィルタリング。他チームの非公開投稿・未参加チャンネルのメッセージ等は結果に含まれない

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `q` が2文字未満 / 100文字超 / 空 |
| 403 | GUEST（検索不可） |

#### `GET /api/v1/search/suggestions`

**認可**: MEMBER 以上

検索オートコンプリート。入力途中のキーワードに対して候補を返却する。

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `q` | String | Yes | 入力中のキーワード（最小1文字） |
| `limit` | int | No | 候補数（デフォルト: 5、最大: 10） |

**レスポンス（200 OK）**
```json
{
  "data": {
    "suggestions": [
      { "text": "グラウンド利用", "type": "KEYWORD" },
      { "text": "グラウンド整備", "type": "KEYWORD" },
      { "text": "田中太郎", "type": "USER", "user_id": 10, "avatar_url": "https://..." },
      { "text": "サッカー部", "type": "TEAM", "team_id": 5 }
    ]
  }
}
```

**補足**
- `KEYWORD`: 自分の過去の検索キーワード（`search_histories` から前方一致検索）から候補を生成。`count` は省略（過去の検索では件数を保持していないため `null`）。将来的にプラットフォーム全体の検索頻度を集計するテーブルを追加して人気キーワード機能に拡張可能
- `USER`: `display_name` / `last_name` + `first_name` の前方一致（`is_searchable = TRUE` のみ）
- `TEAM` / `ORGANIZATION`: `name` の前方一致
- フロントエンドは debounce 300ms で API を呼び出し、ドロップダウンで候補表示

#### `GET /api/v1/search/recent`

**認可**: MEMBER 以上

自分の検索履歴を取得する。

**レスポンス（200 OK）**
```json
{
  "data": [
    { "id": 1, "query": "グラウンド", "searched_at": "2026-03-18T10:30:00" },
    { "id": 2, "query": "練習試合 スケジュール", "searched_at": "2026-03-18T09:00:00" }
  ]
}
```

**補足**
- 最新20件を返却（`searched_at DESC`）
- 同一キーワードの再検索時: 既存レコードの `searched_at` を更新（重複レコードは作らない）

## 5. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `search_histories` | ユーザーの検索キーワード履歴 | なし |

### テーブル定義

#### `search_histories`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `query` | VARCHAR(100) | NO | — | 検索キーワード |
| `searched_at` | DATETIME | NO | CURRENT_TIMESTAMP | 最終検索日時 |

**インデックス**
```sql
UNIQUE KEY uq_sh_user_query (user_id, query)  -- 同一ユーザー×キーワードの重複防止
INDEX idx_sh_user_date (user_id, searched_at DESC)  -- ユーザーの検索履歴一覧
```

**制約・備考**
- 1ユーザーあたり最大50件。超過時は `searched_at` 最古のレコードを物理削除
- 同一キーワードの再検索: UPSERT で `searched_at` を更新（INSERT ON DUPLICATE KEY UPDATE）
- **クエリ正規化**: 保存前に `TRIM()` + 連続スペースの単一化 + 全角スペース→半角変換を実行。「 グラウンド 」と「グラウンド」を同一キーワードとして扱う
- 物理削除（論理削除不要。履歴データに保全義務なし）

### 追加 FULLTEXT インデックス

既存テーブルに FULLTEXT インデックスを追加（各フィーチャーのテーブルに ALTER TABLE で追加）:

```sql
-- 全 FULLTEXT インデックスは N-gram パーサーを使用（日本語のスペースなしトークナイズに対応）
-- my.cnf に ngram_token_size = 2 を設定（バイグラム。「整骨院」→「整骨」「骨院」に分割してヒット）

-- F04.2 chat_messages（チャットメッセージ検索用。DM は除外して検索するためインデックス自体は全メッセージに付与）
ALTER TABLE chat_messages ADD FULLTEXT INDEX ft_cm_body (body) WITH PARSER ngram;

-- F06.1 blog_posts（ブログ記事検索用）
ALTER TABLE blog_posts ADD FULLTEXT INDEX ft_bp_title_body (title, body) WITH PARSER ngram;

-- F05.5 shared_files（ファイル検索用）
ALTER TABLE shared_files ADD FULLTEXT INDEX ft_sf_name_desc (file_name, description) WITH PARSER ngram;

-- F01.1 users（メンバー検索用）
ALTER TABLE users ADD FULLTEXT INDEX ft_users_name (last_name, first_name, display_name, nickname2) WITH PARSER ngram;
```

**N-gram パーサーの設定（my.cnf / Docker Compose）**
```ini
[mysqld]
ngram_token_size = 2
# innodb_ft_min_token_size は N-gram 使用時は不要（ngram_token_size が優先）
# innodb_ft_enable_stopword = OFF（日本語にストップワードは不要）
```

**既存 FULLTEXT インデックスの N-gram 移行**
- `timeline_posts.content`（ft_tp_content、F04.1 定義済み）と `bulletin_threads.title`（ft_bulletin_threads_title、F05.1 定義済み）は既にデフォルトパーサーで作成済み。本機能のマイグレーションで DROP INDEX → 再作成 WITH PARSER ngram を実行

```sql
-- V4.025: 既存 FULLTEXT インデックスの N-gram パーサー移行
ALTER TABLE timeline_posts DROP INDEX ft_tp_content,
  ADD FULLTEXT INDEX ft_tp_content (content) WITH PARSER ngram;

ALTER TABLE bulletin_threads DROP INDEX ft_bulletin_threads_title,
  ADD FULLTEXT INDEX ft_bulletin_threads_title (title) WITH PARSER ngram;
```

## 6. ビジネスロジック

### 検索実行フロー

```
1. ユーザーが検索キーワードを入力し検索実行
2. キーワードのバリデーション（2〜100文字、特殊文字エスケープ）
3. 検索履歴に UPSERT（search_histories）
4. type フィルタに応じて検索対象テーブルを決定（未指定: 全種別）
5. **SYSTEM_ADMIN の認可バイパス**: SYSTEM_ADMIN の場合は認可フィルタを適用しない（全コンテンツを検索可能。モデレーション用途）。`deleted_at IS NULL` のみ適用
6. **scope_type / scope_id フィルタ**: 指定されている場合は追加 WHERE 条件で対象スコープを限定（例: `team_id = :scopeId`）
7. **date_from / date_to フィルタ**: 指定されている場合は `created_at BETWEEN :dateFrom AND :dateTo` を追加
8. **author_id フィルタ**: 指定されている場合は投稿者/作成者の `user_id = :authorId`（or `created_by = :authorId`）を追加
9. 各テーブルに対して並列クエリを実行（CompletableFuture）:
   a. FULLTEXT 対応テーブル: MATCH(column) AGAINST(:query IN BOOLEAN MODE)
   b. LIKE 対応テーブル（teams, organizations, activity_results）: name LIKE CONCAT('%', :query, '%')
   c. 各クエリに認可フィルタ + 上記フィルタを WHERE 条件として付加
10. type_counts を算出（各テーブルの COUNT を並列取得）
11. **結果の取得**:
    a. type 未指定（概要ビュー）: 各種別から `created_at DESC` で上位5件ずつ取得。ページネーションなし
    b. type 指定（詳細ビュー）: 該当種別の単一テーブルに OFFSET + LIMIT を適用
12. snippet 生成:
    a. コンテンツを HTML エスケープ
    b. キーワード（スペース区切りで複数語がある場合は各語を個別に処理）の周辺100文字を抜粋
    c. 各キーワード部分を `<mark>` タグで囲む（複数語の場合は全語をハイライト）
13. レスポンス返却
```

### 認可フィルタリング（各テーブル別）

```java
// 各コンテンツ種別の認可 WHERE 条件を動的構築
// ※ 全テーブルに deleted_at IS NULL を付加（論理削除済みコンテンツの除外）
// ※ SUPPORTER の場合は visibility フィルタを追加（MEMBERS_ONLY を除外）

String visibilityFilter = isSupporterRole
    ? "AND {table}.visibility IN ('PUBLIC', 'SUPPORTERS_AND_ABOVE')"
    : "";  // MEMBER 以上は visibility 制限なし

Map<String, String> authFilters = Map.of(
    "POST",    "(tp.team_id IN (:userTeamIds) OR tp.organization_id IN (:userOrgIds) OR tp.visibility = 'PUBLIC') AND tp.deleted_at IS NULL " + visibilityFilter.replace("{table}", "tp"),
    "MESSAGE", "cm.channel_id IN (SELECT ccm.channel_id FROM chat_channel_members ccm JOIN chat_channels cc ON cc.id = ccm.channel_id WHERE ccm.user_id = :userId AND cc.channel_type != 'DIRECT') AND cm.deleted_at IS NULL",
    "THREAD",  "(bt.team_id IN (:userTeamIds) OR bt.organization_id IN (:userOrgIds)) AND bt.deleted_at IS NULL " + visibilityFilter.replace("{table}", "bt"),
    "ARTICLE", "bp.status = 'PUBLISHED' AND (bp.visibility = 'PUBLIC' OR bp.team_id IN (:userTeamIds) OR bp.organization_id IN (:userOrgIds)) AND bp.deleted_at IS NULL",
    "FILE",    "(sf.team_id IN (:userTeamIds) OR sf.organization_id IN (:userOrgIds)) AND sf.deleted_at IS NULL",
    "USER",    "u.is_searchable = TRUE AND u.deleted_at IS NULL AND u.id IN (SELECT ur.user_id FROM user_roles ur WHERE ur.team_id IN (:userTeamIds) OR ur.organization_id IN (:userOrgIds))",
    "TEAM",    "(t.is_public = TRUE OR t.id IN (:userTeamIds)) AND t.deleted_at IS NULL",
    "ORGANIZATION", "(o.is_public = TRUE OR o.id IN (:userOrgIds)) AND o.deleted_at IS NULL",
    "ACTIVITY", "(ar.team_id IN (:userTeamIds) OR ar.organization_id IN (:userOrgIds)) AND ar.deleted_at IS NULL " + visibilityFilter.replace("{table}", "ar")
);
```

**認可フィルタの設計方針**
- **SYSTEM_ADMIN バイパス**: SYSTEM_ADMIN の場合は上記の認可フィルタを適用せず、`deleted_at IS NULL` のみで全コンテンツを検索可能（モデレーション用。監査ログに `ADMIN_SEARCH_EXECUTED` を記録）
- **`deleted_at IS NULL`**: 全テーブルに付加。論理削除済みコンテンツを検索結果から除外
- **組織スコープ**: `team_id IN (:userTeamIds)` に加えて `organization_id IN (:userOrgIds)` を OR 条件で追加。組織スコープのブログ・掲示板・ファイルを漏れなく検索
- **DM 除外**: チャットメッセージは `channel_type != 'DIRECT'` を付加。DM のプライベートな内容はグローバル検索に含めない（チャンネル内検索 `GET /chat/channels/{id}/messages/search` でのみ検索可能）
- **SUPPORTER の visibility**: SUPPORTER は `PUBLIC` / `SUPPORTERS_AND_ABOVE` のコンテンツのみ検索可能。`MEMBERS_ONLY` は除外

### 検索サジェスト生成

```
1. q の前方一致で search_histories を検索（自分の履歴のみ、上位3件）
2. q の前方一致で users.display_name / last_name + first_name を検索（is_searchable = TRUE、上位2件）
3. q の前方一致で teams.name / organizations.name を検索（上位2件）
4. 結果をマージして limit 件に制限
5. 応答時間目標: P95 < 100ms（Redis キャッシュ不要。LIKE 'prefix%' はインデックスを活用して高速）
```

### パフォーマンス戦略

```
■ MySQL FULLTEXT + N-gram パーサー:
  - 全 FULLTEXT インデックスに `WITH PARSER ngram` を指定（日本語のスペースなしテキストを正しくトークナイズ）
  - ngram_token_size = 2（バイグラム）: 「整骨院」→「整骨」「骨院」に分割。「整骨」でも「骨院」でも「整骨院」でもヒット
  - BOOLEAN MODE を使用（NATURAL LANGUAGE MODE より柔軟。部分一致に +prefix* 構文が使える）
  - innodb_ft_enable_stopword = OFF（日本語にストップワードは不要）
  - N-gram の注意: 2文字のバイグラムは短い一般的な語（「の」「は」等）でも大量にヒットするため、1文字検索は 400 エラーで拒否（最小2文字）

■ 並列クエリ:
  - 各テーブルへの検索クエリを CompletableFuture.supplyAsync で並列実行
  - タイムアウト: 各クエリ3秒。タイムアウトしたテーブルの結果はスキップ（部分結果を返却）
  - レスポンスに timed_out_types を含めて、どの種別がタイムアウトしたかをフロントに通知

■ キャッシュ:
  - 検索結果のキャッシュは行わない（認可フィルタがユーザーごとに異なるため）
  - type_counts のキャッシュ: Redis TTL 30秒（キー: search:{userId}:{queryHash}:counts）
  - サジェスト候補のキャッシュ: Redis TTL 5分（キー: search:suggest:{prefix}）

■ 大量データ対策:
  - chat_messages は日付インデックスとの複合条件で範囲を限定（デフォルト: 直近6ヶ月）
  - 6ヶ月超の検索は date_from パラメータの明示指定が必要（400 エラーにはしない。警告メッセージを返却）
```

## 7. セキュリティ考慮事項

- **認可フィルタリング**: 全検索クエリに WHERE 条件でユーザーのアクセス権限を付加。他チームの非公開コンテンツ・未参加チャンネルのメッセージは結果に含まれない
- **`is_searchable` の尊重**: `users.is_searchable = FALSE` のユーザーは `USER` 種別の検索結果に含まれない。ただしメンション（`@` 入力時の候補）は別機能のため引き続き表示
- **FULLTEXT インジェクション防止**: `MATCH() AGAINST()` のクエリはパラメータバインド。ユーザー入力の特殊文字（`+`, `-`, `*`, `~`, `"`, `(`, `)`）はエスケープ処理。`BOOLEAN MODE` の演算子をユーザーが意図せず使用することを防止
- **複数語クエリの処理**: スペース区切りのキーワード（例: 「練習試合 スケジュール」）は各語を AND 条件に変換（`+練習試合 +スケジュール`）。全語を含むコンテンツのみヒット。OR 検索は Phase 4 では未対応（将来的に `OR` 演算子のサポートを検討）
- **snippet の XSS 防止**: snippet 生成時にコンテンツを HTML エスケープした後に `<mark>` タグを挿入。元コンテンツに含まれる HTML タグは無害化
- **検索履歴のプライバシー**: 検索履歴は本人のみ閲覧・削除可能。ADMIN も他人の検索履歴にアクセスできない
- **レート制限**: 検索 API は1ユーザーあたり1分間に30回。サジェスト API は1ユーザーあたり1分間に60回（debounce 前提でも高頻度のため緩め）
- **クエリ長制限**: 最大100文字。過度に長いクエリによる FULLTEXT の性能劣化を防止
- **監査ログ**: 検索操作自体は監査ログに記録しない（通常の読み取り操作のため。SYSTEM_ADMIN のモデレーション検索のみ `ADMIN_SEARCH_EXECUTED` として記録）

**マイグレーション上の注意点**
- V4.021〜V4.024 は ALTER TABLE で FULLTEXT インデックス（`WITH PARSER ngram`）を追加。既存データに対する FULLTEXT インデックス構築は大量レコードがある場合に時間がかかるため、メンテナンスウィンドウでの実行を推奨
- V4.025 は既存の FULLTEXT インデックス（F04.1 の `ft_tp_content`、F05.1 の `ft_bulletin_threads_title`）を DROP → `WITH PARSER ngram` で再作成。既存データの再インデックスが発生するため V4.021〜V4.024 と同じメンテナンスウィンドウで実行
- **MySQL 設定（my.cnf）**: `ngram_token_size = 2` と `innodb_ft_enable_stopword = OFF` を設定。設定変更後は MySQL の再起動が必要。Docker Compose の MySQL サービス設定で `command: --ngram-token-size=2 --innodb-ft-enable-stopword=OFF` でも指定可能
