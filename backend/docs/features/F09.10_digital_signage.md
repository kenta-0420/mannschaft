## 1. 概要

TV・タブレット・大型ディスプレイに表示する専用全画面モード。クラブハウス、マンションエントランス、店舗待合室、学校の廊下等に設置したディスプレイに、お知らせ・スケジュール・活動記録ハイライト・ランキング等をスライドショー形式で自動表示する。認証不要のトークンベース URL でブラウザ全画面表示し、WebSocket によるリアルタイム更新と緊急メッセージの即時全画面表示を備える。

**主要ユースケース**:
- スポーツチーム: クラブハウスに今週の練習予定・試合結果・月間 MVP を表示
- マンション: エントランスに理事会からのお知らせ・ゴミ出し日・修繕工事情報を掲示
- 美容室・整骨院: 待合室にお知らせ・キャンペーン情報を表示
- 学校: 廊下のディスプレイに行事予定・部活動結果を表示

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全スコープのサイネージ画面・アクセストークンを閲覧・強制削除 |
| ADMIN | サイネージ画面の CRUD、スロット構成の編集、アクセストークンの発行・無効化、表示スケジュール設定、緊急メッセージ送信 |
| DEPUTY_ADMIN | `SIGNAGE_MANAGE` パーミッションが付与されている場合: ADMIN と同等。付与されていない場合: 閲覧不可 |
| MEMBER | 閲覧不可（サイネージ管理画面へのアクセス権なし） |
| SUPPORTER | 閲覧不可 |
| GUEST | 閲覧不可 |

※ サイネージ表示画面（`/signage/{token}`）はトークンベースアクセスのため、ロールに依存しない。トークンを知っている（URL にアクセスできる）端末であれば誰でも表示可能。

### テーブル定義

#### `signage_screens`

サイネージ画面の定義。1スコープ（チーム/組織）に対して複数の画面を作成可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID（team_id or organization_id） |
| `name` | VARCHAR(100) | NO | — | 画面名（例: 「エントランス用」「クラブハウス TV」） |
| `description` | VARCHAR(500) | YES | NULL | 説明 |
| `layout` | VARCHAR(20) | NO | 'LANDSCAPE' | レイアウト種別。値: `LANDSCAPE`（16:9）, `PORTRAIT`（9:16）, `STANDARD`（4:3） |
| `default_slide_duration` | INT | NO | 10 | スライド1枚のデフォルト表示秒数（3〜120 秒） |
| `transition_effect` | VARCHAR(20) | NO | 'FADE' | スライド切替エフェクト。値: `FADE`, `SLIDE`, `NONE` |
| `is_active` | BOOLEAN | NO | TRUE | 画面の有効/無効 |
| `is_clock_shown` | BOOLEAN | NO | TRUE | 画面上部に時計を常時表示するか |
| `is_weather_shown` | BOOLEAN | NO | FALSE | 天気予報を常時表示するか |
| `weather_location` | VARCHAR(100) | YES | NULL | 天気予報の地域（OpenWeatherMap の city name。例: `Tokyo,JP`） |
| `background_color` | VARCHAR(7) | NO | '#1F2937' | 背景色（HEX） |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。作成した ADMIN。ON DELETE RESTRICT |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
INDEX idx_ss_scope (scope_type, scope_id, is_active)  -- スコープ別画面一覧取得
INDEX idx_ss_created_by (created_by)                    -- 作成者別一覧
```

**制約・備考**
- `scope_type` の CHECK 制約:
  ```sql
  CONSTRAINT chk_ss_scope_type
    CHECK (scope_type IN ('TEAM', 'ORGANIZATION'))
  ```
- `layout` の CHECK 制約:
  ```sql
  CONSTRAINT chk_ss_layout
    CHECK (layout IN ('LANDSCAPE', 'PORTRAIT', 'STANDARD'))
  ```
- `transition_effect` の CHECK 制約:
  ```sql
  CONSTRAINT chk_ss_transition_effect
    CHECK (transition_effect IN ('FADE', 'SLIDE', 'NONE'))
  ```
- 論理削除: `deleted_at DATETIME nullable`（SoftDeletableEntity 適用）
- 楽観的ロック: `version` カラムで同時編集を検知
- `layout` は表示端末のアスペクト比に応じて選択。フロントエンドがレイアウトに応じた CSS を適用
- `is_weather_shown = TRUE` の場合、`weather_location` は必須（アプリケーション層でバリデーション）
- `default_slide_duration` は各スロットで個別に上書き可能

#### `signage_slots`

画面内のコンテンツスロット。1つの画面に複数のスロットを持ち、表示順にスライドショーで切替。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `screen_id` | BIGINT UNSIGNED | NO | — | FK → signage_screens |
| `slot_order` | INT | NO | — | 表示順（1始まり。画面内で一意） |
| `slot_type` | VARCHAR(30) | NO | — | スロット種別。値: `ANNOUNCEMENT`, `SCHEDULE`, `ACTIVITY_HIGHLIGHT`, `RANKING`, `FREE_TEXT`, `FREE_IMAGE`, `WEATHER`, `CLOCK` |
| `title` | VARCHAR(100) | YES | NULL | スロットのタイトル（画面上部に表示。NULL の場合はデフォルトタイトル） |
| `slide_duration` | INT | YES | NULL | このスロットの表示秒数（NULL の場合は画面のデフォルトを使用。3〜120 秒） |
| `content_config` | JSON | YES | NULL | スロット種別ごとのコンテンツ設定（下記参照） |
| `is_active` | BOOLEAN | NO | TRUE | スロットの有効/無効 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_sslot_screen_order (screen_id, slot_order)      -- 画面別スロット順序取得
UNIQUE KEY uq_sslot_screen_order (screen_id, slot_order)  -- 画面内で表示順は一意
```

**制約・備考**
- `screen_id` は `signage_screens.id` への FK（CASCADE DELETE: 画面削除時にスロットも削除）
- `slot_type` の CHECK 制約:
  ```sql
  CONSTRAINT chk_sslot_type
    CHECK (slot_type IN ('ANNOUNCEMENT', 'SCHEDULE', 'ACTIVITY_HIGHLIGHT', 'RANKING', 'FREE_TEXT', 'FREE_IMAGE', 'WEATHER', 'CLOCK'))
  ```
- `content_config` は JSON 型で、`slot_type` に応じた設定を格納:
  - `ANNOUNCEMENT`: `{ "max_items": 5, "post_type": "ANNOUNCEMENT", "priority_filter": ["NORMAL", "IMPORTANT", "CRITICAL"] }`
  - `SCHEDULE`: `{ "days_ahead": 7, "show_past_today": false }`
  - `ACTIVITY_HIGHLIGHT`: `{ "max_items": 5, "visibility": "PUBLIC" }`
  - `RANKING`: `{ "period": "MONTHLY", "display_count": 10 }`
  - `FREE_TEXT`: `{ "text": "本日の営業時間: 9:00〜18:00", "font_size": "LARGE", "text_color": "#FFFFFF", "background_color": "#DC2626" }`
  - `FREE_IMAGE`: `{ "image_url": "https://s3.../signage/image.jpg", "file_key": "signage/teams/7/abc123.jpg" }`
  - `WEATHER`: `{ "location": "Tokyo,JP", "show_forecast": true }` （画面レベルの天気と独立。スライドとして天気詳細を表示する場合に使用）
  - `CLOCK`: `{ "format": "24h", "show_date": true }`
- `ANNOUNCEMENT` スロットはスコープ内の `blog_posts` から `post_type = 'ANNOUNCEMENT'` かつ `status = 'PUBLISHED'` のものを自動取得
- `SCHEDULE` スロットはスコープ内の `schedules` から今日〜N日後のイベントを自動取得
- `ACTIVITY_HIGHLIGHT` スロットはスコープ内の `activity_results` から直近の `PUBLIC` 記録を取得
- `RANKING` スロットは F04.7 の `ranking_snapshots` から該当期間のランキングを取得。ゲーミフィケーションモジュールが無効の場合はスロットをスキップ

#### `signage_schedules`

曜日・時間帯別の表示スケジュール。特定の時間帯に表示するスロットの組み合わせを定義。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `screen_id` | BIGINT UNSIGNED | NO | — | FK → signage_screens |
| `name` | VARCHAR(100) | NO | — | スケジュール名（例: 「平日・営業時間」「週末モード」） |
| `day_of_week` | VARCHAR(20) | NO | — | 曜日パターン。値: `WEEKDAY`（月〜金）, `WEEKEND`（土日）, `EVERYDAY`, `MON`, `TUE`, `WED`, `THU`, `FRI`, `SAT`, `SUN` |
| `start_time` | TIME | NO | — | 開始時刻（例: `09:00:00`） |
| `end_time` | TIME | NO | — | 終了時刻（例: `18:00:00`） |
| `slot_ids` | JSON | NO | — | この時間帯に表示するスロット ID の配列（表示順も兼ねる）。例: `[1, 3, 5]` |
| `priority` | INT | NO | 0 | スケジュールの優先度（高い値が優先。時間帯が重複する場合の解決用） |
| `is_active` | BOOLEAN | NO | TRUE | スケジュールの有効/無効 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_ssch_screen (screen_id, is_active, priority)  -- 画面別アクティブスケジュール取得
```

**制約・備考**
- `screen_id` は `signage_screens.id` への FK（CASCADE DELETE）
- `day_of_week` の CHECK 制約:
  ```sql
  CONSTRAINT chk_ssch_day_of_week
    CHECK (day_of_week IN ('WEEKDAY', 'WEEKEND', 'EVERYDAY', 'MON', 'TUE', 'WED', 'THU', 'FRI', 'SAT', 'SUN'))
  ```
- `slot_ids` は `signage_slots.id` の配列。アプリケーション層で存在チェック
- スケジュール未定義の時間帯は、画面に紐づく全アクティブスロットを表示順に全件スライド
- `start_time` < `end_time` を CHECK 制約で保証:
  ```sql
  CONSTRAINT chk_ssch_time_range CHECK (start_time < end_time)
  ```
- 深夜跨ぎ（例: 22:00〜02:00）は2レコードに分割して設定（22:00〜23:59, 00:00〜02:00）
- 時間帯が重複する場合は `priority` が高いスケジュールを採用

#### `signage_access_tokens`

サイネージ表示画面へのトークンベースアクセス。認証不要で URL にアクセスできる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `screen_id` | BIGINT UNSIGNED | NO | — | FK → signage_screens |
| `token` | VARCHAR(36) | NO | — | UUID v4 トークン（URL パスに使用） |
| `name` | VARCHAR(100) | NO | — | トークン名（例: 「エントランスTV」「クラブハウスiPad」） |
| `is_active` | BOOLEAN | NO | TRUE | トークンの有効/無効 |
| `allowed_ips` | JSON | YES | NULL | IP アドレス制限（JSON 配列。例: `["192.168.1.100", "10.0.0.0/24"]`。NULL = 制限なし） |
| `last_accessed_at` | DATETIME | YES | NULL | 最終アクセス日時（端末の死活監視用） |
| `last_accessed_ip` | VARCHAR(45) | YES | NULL | 最終アクセス元 IP アドレス |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。発行した ADMIN。ON DELETE RESTRICT |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_sat_token (token)                        -- トークンは一意
INDEX idx_sat_screen (screen_id, is_active)            -- 画面別アクティブトークン一覧
```

**制約・備考**
- `screen_id` は `signage_screens.id` への FK（CASCADE DELETE）
- `token` は UUID v4 で生成。有効期限なし（`is_active = FALSE` で明示的に無効化するまで有効）
- 1つの画面に対して複数のトークンを発行可能（端末ごとに異なるトークンを割り当て可能）
- `allowed_ips` が設定されている場合、リクエスト元 IP がリストに含まれない場合は 403 を返却。CIDR 表記対応
- `last_accessed_at` / `last_accessed_ip` はサイネージ画面のポーリング時に更新（管理画面で端末の接続状態を確認可能）
- 論理削除は適用しない（無効化は `is_active = FALSE` で管理。削除時は物理削除）

#### `signage_emergency_messages`

緊急メッセージの送信履歴。WebSocket で即時配信。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `screen_id` | BIGINT UNSIGNED | NO | — | FK → signage_screens |
| `message` | VARCHAR(500) | NO | — | 緊急メッセージ本文 |
| `background_color` | VARCHAR(7) | NO | '#DC2626' | 背景色（HEX。デフォルト: 赤） |
| `text_color` | VARCHAR(7) | NO | '#FFFFFF' | テキスト色（HEX。デフォルト: 白） |
| `is_active` | BOOLEAN | NO | TRUE | 表示中かどうか（FALSE で解除） |
| `sent_by` | BIGINT UNSIGNED | NO | — | FK → users。送信した ADMIN。ON DELETE RESTRICT |
| `dismissed_at` | DATETIME | YES | NULL | 解除日時（ADMIN が手動解除した日時） |
| `dismissed_by` | BIGINT UNSIGNED | YES | NULL | FK → users。解除した ADMIN。ON DELETE SET NULL |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_sem_screen_active (screen_id, is_active)  -- 画面別アクティブ緊急メッセージ
INDEX idx_sem_screen_created (screen_id, created_at) -- 履歴取得用
```

**制約・備考**
- `screen_id` は `signage_screens.id` への FK（CASCADE DELETE）
- 1画面につき同時にアクティブな緊急メッセージは1件のみ。新規送信時に既存のアクティブメッセージは自動解除（`is_active = FALSE`, `dismissed_at = NOW()`）
- ログ系テーブルのため論理削除は適用しない
- `is_active = TRUE` のメッセージが存在する間、サイネージ画面は通常スライドを中断して緊急メッセージを全画面表示

## 5. ビジネスロジック

### 主要フロー

#### サイネージ画面作成フロー
```
1. ADMIN が管理画面からサイネージ画面を作成
2. バックエンドがモジュール有効化チェック + 権限検証
3. signage_screens にレコード挿入
4. ADMIN がスロットを追加（種別・表示順・表示秒数を設定）
5. ADMIN がアクセストークンを発行
6. サイネージ端末で `/signage/{token}` を全画面表示
```

#### サイネージ表示フロー（フロントエンド中心）
```
1. ブラウザが `/signage/{token}` にアクセス
2. Nuxt 3 がサイネージ専用レイアウト（signage.vue）を適用
3. GET /api/v1/signage/{token}/screen で画面データ・スロット一覧を取得
4. 各スロットの実コンテンツを GET /api/v1/signage/{token}/content/{slotId} で取得
5. スライドショーコンポーネントが slot_order 順にスロットを表示
6. 各スロットを slide_duration 秒間表示し、transition_effect で次のスロットに切替
7. 全スロット表示完了後、先頭に戻ってループ
8. 5分間隔のポーリングで各スロットのコンテンツを更新
9. WebSocket（/topic/signage/{screen_id}）でリアルタイム更新を監視
```

#### WebSocket リアルタイム更新フロー
```
1. ADMIN がスロットを編集・追加・削除
2. バックエンドが DB を更新
3. バックエンドが WebSocket メッセージを送信: /topic/signage/{screen_id}
4. メッセージ種別:
   a. SLOT_UPDATED → 該当スロットのコンテンツを再取得
   b. SCREEN_UPDATED → 画面データ全体を再取得
   c. EMERGENCY → 緊急メッセージを全画面表示
   d. EMERGENCY_DISMISSED → 緊急メッセージを解除し通常スライドに復帰
5. サイネージ端末が受信してリアルタイムに表示を更新
```

#### 緊急メッセージフロー
```
1. ADMIN が管理画面から緊急メッセージを送信
2. バックエンドが既存のアクティブ緊急メッセージを自動解除
3. signage_emergency_messages に新規レコード挿入
4. WebSocket で EMERGENCY イベントを /topic/signage/{screen_id} に送信
5. サイネージ画面が通常スライドを即時中断
6. 緊急メッセージを全画面表示（背景色・テキスト色は設定値を使用）
7. ADMIN が解除するまで緊急表示を継続
8. ADMIN が解除 → WebSocket で EMERGENCY_DISMISSED を送信
9. サイネージ画面が通常スライドショーに復帰
```

#### 表示スケジュール解決フロー
```
1. サイネージ端末が現在の曜日・時刻を取得
2. signage_schedules から現在時刻に該当するスケジュールを検索
3. 該当スケジュールがある場合:
   a. 複数該当 → priority が最も高いスケジュールを採用
   b. 採用されたスケジュールの slot_ids のみをスライドショー対象に
4. 該当スケジュールがない場合:
   a. 画面に紐づく全アクティブスロットを表示順に全件スライド
5. スケジュール切替タイミング（開始/終了時刻）でスロット構成を自動更新
```

#### 省電力モード
```
1. signage_schedules に「表示時間帯」を定義
2. 全スケジュールの時間帯外 = 省電力モード
3. 省電力モード中はサイネージ画面を黒画面に切替
4. ブラウザが Visibility API で画面非表示を検知し、ポーリングを停止
5. 次の表示時間帯の開始時に自動的に通常表示に復帰
6. 緊急メッセージは省電力モード中でも強制表示
```

### 重要な判定ロジック

- **トークン検証**: リクエスト受信時に `signage_access_tokens` から `token` を検索。`is_active = TRUE` かつ紐づく `signage_screens.deleted_at IS NULL` かつ `signage_screens.is_active = TRUE` を全て満たす場合のみ有効
- **IP制限チェック**: `allowed_ips` が非 NULL の場合、リクエスト元 IP がリストに含まれるか検証。CIDR 表記はサブネットマスクで判定
- **スロット有効性チェック**: `is_active = TRUE` のスロットのみ表示対象。`RANKING` スロットはゲーミフィケーションモジュールが有効な場合のみ表示（無効時はスキップ）
- **コンテンツ取得の空ハンドリング**: 該当コンテンツが0件の場合（お知らせが1件もない等）、そのスロットをスキップして次のスロットに遷移
- **天気予報**: OpenWeatherMap API を BFF（`server/api/weather.ts`）経由で呼び出し。API キーはサーバーサイドで秘匿。レスポンスは Valkey にキャッシュ（TTL: 30分）

## 6. セキュリティ考慮事項

- **トークンベースアクセス**: サイネージ表示画面は JWT 認証を使わず、UUID v4 トークンで認証する。トークンは推測困難（122bit のエントロピー）だが、URL を知っている人は誰でもアクセス可能。機密情報は表示しない前提で設計
- **IP アドレス制限**: `signage_access_tokens.allowed_ips` でアクセス元を制限可能。固定 IP のネットワーク環境（オフィス、マンション共用 Wi-Fi 等）で推奨
- **管理 API の認可**: サイネージ管理系 API は全て ADMIN / DEPUTY_ADMIN（`SIGNAGE_MANAGE` パーミッション）のみ。チーム/組織の所属チェックを Service 入り口で実施
- **WebSocket 認証**: サイネージ端末の WebSocket 接続はトークンベース。接続時に `token` をクエリパラメータで渡し、バックエンドが `signage_access_tokens` を検証
- **レートリミット**: サイネージ表示 API（`/api/v1/signage/{token}/...`）には `Bucket4j` で 1分間に60回の制限を適用（ポーリング + WebSocket fallback を想定）
- **緊急メッセージの認可**: 緊急メッセージ送信は ADMIN / DEPUTY_ADMIN のみ。誤送信防止のためフロントエンドで確認ダイアログを表示
- **天気予報 API キー秘匿**: OpenWeatherMap API キーはバックエンドの BFF（Nuxt 3 の `server/api/`）経由でのみ使用。クライアントには公開しない
- **XSS 防止**: `FREE_TEXT` スロットの `text` は HTML タグを許可しない（プレーンテキストのみ）。フロントエンドで `v-text` ディレクティブを使用し、HTML インジェクションを防止

## 7. フロントエンド設計

### レイアウト設計

#### サイネージ専用レイアウト（`layouts/signage.vue`）

通常のアプリケーションレイアウト（ヘッダー・サイドバー・フッター）を一切含まない全画面レイアウト。

```
┌──────────────────────────────────────────────┐
│ [時計]                            [天気アイコン] │  ← オーバーレイバー（is_clock_shown / is_weather_shown）
├──────────────────────────────────────────────┤
│                                              │
│                                              │
│           スライドショーエリア                    │
│         （各スロットのコンテンツを表示）            │
│                                              │
│                                              │
│                                              │
├──────────────────────────────────────────────┤
│  ● ○ ○ ○ ○                                  │  ← スライドインジケーター
└──────────────────────────────────────────────┘
```

- **全画面**: `position: fixed; inset: 0;` でビューポート全体を占有
- **カーソル非表示**: `cursor: none` でマウスカーソルを隠す
- **スクロール無効**: `overflow: hidden` でスクロールを防止
- **ウェイクロック**: Screen Wake Lock API（`navigator.wakeLock`）でディスプレイのスリープを防止
- **オーバーレイバー**: 半透明の固定バーに時計・天気を常時表示（`is_clock_shown` / `is_weather_shown` 設定に応じて表示/非表示）

#### レスポンシブレイアウト

| レイアウト | アスペクト比 | 用途 | CSS 制御 |
|-----------|:----------:|------|---------|
| `LANDSCAPE` | 16:9 | TV・横置きモニター | デフォルト。横長レイアウト |
| `PORTRAIT` | 9:16 | 縦置きディスプレイ | テキストサイズ拡大、縦スクロール的なスライド切替 |
| `STANDARD` | 4:3 | タブレット | 中間レイアウト。余白を調整 |

レイアウト切替は `signage_screens.layout` の値に応じて CSS クラスを動的に適用。

### スライドショーロジック（`useSignagePlayer.ts`）

```
初期化:
1. GET /api/v1/signage/{token}/screen で画面データ取得
2. current_schedule に基づいて表示対象スロットを決定
3. 各スロットのコンテンツを GET /api/v1/signage/{token}/content/{slotId} で取得
4. スライドショーを開始

スライドループ:
1. 現在のスロットを slide_duration 秒間表示
2. transition_effect に基づくアニメーションで次のスロットに切替
3. 最後のスロットの後は先頭に戻る
4. コンテンツが空のスロットはスキップ

ポーリング（5分間隔）:
1. 各スロットの GET /api/v1/signage/{token}/content/{slotId} を呼び出し
2. レスポンスが前回と異なる場合のみ表示を更新（差分検知）
3. ポーリング失敗時はリトライ（30秒後に再試行、最大3回）

スケジュール切替:
1. 1分間隔で現在時刻と signage_schedules を照合
2. スケジュールが切り替わったら表示対象スロットを再計算
3. スライドショーを先頭から再開始
```

### WebSocket ハンドリング（`useSignageWebSocket.ts`）

```
接続:
1. STOMP over WebSocket で /ws に接続
2. クエリパラメータに token を含める: /ws?signage_token={token}
3. 購読: /topic/signage/{screen_id}

メッセージハンドリング:
- type: "SLOT_UPDATED"
  → payload.slot_id のコンテンツを再取得
  → 現在表示中のスロットの場合は即時更新

- type: "SCREEN_UPDATED"
  → GET /api/v1/signage/{token}/screen で全体を再取得
  → スライドショーの構成をリビルド

- type: "EMERGENCY"
  → payload.message / background_color / text_color を使用
  → SignageEmergency.vue を表示（通常スライドを即時中断）
  → スライドタイマーを一時停止

- type: "EMERGENCY_DISMISSED"
  → SignageEmergency.vue を非表示
  → 通常スライドショーを再開（中断地点から）

再接続:
- 接続断 → 5秒後にリトライ（指数バックオフ: 5, 10, 20, 40秒、最大60秒）
- 再接続成功時にコンテンツを全件再取得（オフライン中の更新を反映）
- WebSocket が接続不能の場合、ポーリング間隔を5分→1分に短縮（フォールバック）
```

## 9. 未解決事項

- ~~トークンの有効期限をどうするか~~ → 有効期限なし。`is_active` フラグで明示的に無効化する運用。理由: サイネージ端末は常設のため、定期的なトークン更新は運用負荷が高い

- ~~天気予報の外部 API をバックエンドとフロントエンドのどちらで呼ぶか~~ → Nuxt 3 の BFF（`server/api/weather.ts`）で呼び出す。API キーをクライアントに公開しないため
- ~~コンテンツ更新のリアルタイム性とポーリングのバランス~~ → WebSocket をメインとし、5分間隔のポーリングをフォールバックとして併用。WebSocket 断の場合はポーリング間隔を1分に短縮
- ~~複数トークンが同じ画面を参照する場合の WebSocket サブスクリプション~~ → 全トークンが同一の `/topic/signage/{screen_id}` を購読。screen_id 単位でメッセージをブロードキャスト
- ~~深夜跨ぎスケジュールの扱い~~ → 2レコードに分割（22:00〜23:59 と 00:00〜02:00）。`start_time < end_time` の CHECK 制約を維持しつつ、運用ガイドで案内
