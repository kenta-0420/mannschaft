# F08.1: マッチング・対外交流

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 8
> **最終更新**: 2026-03-15
> **関連ドキュメント**: F03.1 スケジュール共有（成立後の自動スケジュール追加）、F04.2 チャット（成立後のグループチャット自動作成）、F04.3 プッシュ通知（応募通知・成立通知）、F04.5 モデレーション（NGチーム・通報）、F01.3 テンプレート・モジュール（選択式モジュール #8）

---

## 1. 概要

スポーツチーム・文化サークル・地域コミュニティ等が、地域・レベル・活動種別に応じた対戦相手・練習相手・交流先を募集・応募できる機能。チームAが「対戦相手募集」を投稿し、チームBが応募、ADMIN が承諾すると**マッチング成立**となり、双方のチームスケジュール（F03.1）に自動で予定が追加される。成立後の相互レビューでチームの信頼スコアが蓄積される。NGチーム設定により、トラブル防止のために特定チームを非表示・応募不可にする機能も提供する。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全チームの募集・応募・レビューの参照。不正コンテンツの削除 |
| ADMIN | 自チームの募集投稿 CRUD、応募の承諾・拒否、NGチーム管理、レビュー投稿 |
| DEPUTY_ADMIN | `MANAGE_MATCHING` 権限を持つ場合: 募集投稿 CRUD、応募の承諾・拒否 |
| MEMBER | 募集一覧・詳細の閲覧（自チームの募集・他チームの公開募集）、応募なし（ADMIN/DEPUTY_ADMIN が操作） |
| SUPPORTER | 対象外 |
| GUEST | 対象外 |

> **設計判断**: 応募・承諾はチーム代表者（ADMIN/DEPUTY_ADMIN）が行う。MEMBERは閲覧のみ。「チーム同士の正式な交流」という性質上、個人メンバーが勝手に他チームへ応募できる仕様は混乱を招く。

### 対象レベル
- [ ] 組織 (Organization) — 対象外（チーム間交流のため）
- [x] チーム (Team) — 募集・応募・NGチーム管理
- [ ] 個人 (Personal) — 対象外（個人間マッチングは別途検討）

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `match_requests` | 対外交流の募集投稿（チームが作成・公開） | あり（`deleted_at`） |
| `match_proposals` | 募集への応募（他チームが申し込み） | なし（`status` で管理） |
| `match_reviews` | マッチング成立後の相互レビュー | なし |
| `ng_teams` | NGチーム設定（相互非表示・応募ブロック） | なし |

### テーブル定義

#### `match_requests`

チームが対外交流の相手を募集する投稿。公開されると他チームの ADMIN に表示される。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。募集チーム |
| `title` | VARCHAR(100) | NO | — | 募集タイトル（例: 「練習試合相手募集@大阪」） |
| `description` | TEXT | YES | NULL | 詳細説明（チームの特徴・要望等） |
| `activity_type` | ENUM('MATCH', 'PRACTICE', 'EXCHANGE', 'TRAINING', 'OTHER') | NO | — | 活動種別（試合 / 練習 / 交流戦 / 合同練習 / その他） |
| `visibility` | ENUM('PLATFORM', 'ORGANIZATION') | NO | 'PLATFORM' | 公開範囲（PLATFORM = Mannschaft 全チーム公開 / ORGANIZATION = 同一組織内チームのみ）|
| `prefecture` | VARCHAR(10) | NO | — | 都道府県コード（例: 'osaka', 'tokyo'）。地域検索用 |
| `venue_name` | VARCHAR(200) | YES | NULL | 希望会場名（自由記述。NULL = 相談） |
| `preferred_date_from` | DATE | YES | NULL | 希望日程（開始）。NULL = 未定 |
| `preferred_date_to` | DATE | YES | NULL | 希望日程（終了）。NULL = 未定 |
| `preferred_time_from` | TIME | YES | NULL | 希望時間帯（開始）。NULL = 未定 |
| `preferred_time_to` | TIME | YES | NULL | 希望時間帯（終了）。NULL = 未定 |
| `level` | ENUM('BEGINNER', 'INTERMEDIATE', 'ADVANCED', 'ANY') | NO | 'ANY' | 希望相手レベル |
| `min_participants` | SMALLINT UNSIGNED | YES | NULL | 最低参加人数（NULL = 不問） |
| `max_participants` | SMALLINT UNSIGNED | YES | NULL | 最大参加人数（NULL = 不問） |
| `status` | ENUM('OPEN', 'MATCHED', 'CANCELLED', 'EXPIRED') | NO | 'OPEN' | ステータス |
| `proposal_count` | INT UNSIGNED | NO | 0 | 応募数（非正規化カウント。atomic UPDATE で管理） |
| `expires_at` | DATETIME | YES | NULL | 募集期限（NULL = 無期限。期限超過で自動 EXPIRED）|
| `matched_proposal_id` | BIGINT UNSIGNED | YES | NULL | FK → match_proposals（成立した応募。成立後に設定）|
| `cancel_count` | SMALLINT UNSIGNED | NO | 0 | 成立後キャンセル回数（この募集に関するキャンセル累計。atomic UPDATE で管理）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
INDEX idx_mr_team (team_id, status, created_at)                        -- 自チームの募集一覧
INDEX idx_mr_search (visibility, prefecture, activity_type, status, expires_at)  -- 地域・種別での募集検索
INDEX idx_mr_level (level, status)                                     -- レベル絞り込み
INDEX idx_mr_expires (expires_at, status)                              -- 期限切れバッチ処理用
```

**制約・備考**
- `preferred_date_from <= preferred_date_to` は Service 層でバリデーション
- `proposal_count` は応募作成/削除時に `UPDATE match_requests SET proposal_count = proposal_count ± 1` で atomic 更新
- `status = MATCHED` 後は内容の編集不可。`matched_proposal_id` が設定される
- `status = EXPIRED` は毎晩バッチ（`expires_at < NOW() AND status = 'OPEN'`）で自動更新
- `visibility = 'PLATFORM'`: OPEN 中の募集は NGチームを除く全認証済みチームから閲覧可能
- `visibility = 'ORGANIZATION'`: OPEN 中の募集は同一組織内チームのみ閲覧可能（`team_members` テーブルの `organization_id` で判定）
- キャンセル後は自動で `status = 'OPEN'` に復元（再募集可）。ただし再公開通知は送信しない（スパム防止）

#### `match_proposals`

他チームの募集に対する応募レコード。1募集に対して複数チームが応募可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `request_id` | BIGINT UNSIGNED | NO | — | FK → match_requests。ON DELETE CASCADE |
| `proposing_team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。応募チーム |
| `message` | TEXT | YES | NULL | 応募メッセージ（自己紹介・希望条件等） |
| `proposed_date` | DATE | YES | NULL | 具体的に提案する日付（NULL = 募集条件に従う）|
| `proposed_time_from` | TIME | YES | NULL | 提案時間帯（開始） |
| `proposed_time_to` | TIME | YES | NULL | 提案時間帯（終了） |
| `proposed_venue` | VARCHAR(200) | YES | NULL | 提案会場（NULL = 募集条件に従う）|
| `status` | ENUM('PENDING', 'ACCEPTED', 'REJECTED', 'CANCELLED', 'WITHDRAWN') | NO | 'PENDING' | 応募ステータス |
| `rejection_reason` | VARCHAR(500) | YES | NULL | 拒否理由（`status = REJECTED` の場合。任意）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_mp_request_team (request_id, proposing_team_id)  -- 同一チームの二重応募防止
INDEX idx_mp_proposing_team (proposing_team_id, status)        -- 自チームの応募一覧
INDEX idx_mp_request_status (request_id, status)               -- 募集ごとの応募一覧
```

**制約・備考**
- 1チームが同一募集に複数回応募することを UNIQUE KEY で防止
- 自チームの募集に自チームが応募することは Service 層で禁止（400 エラー）
- `status = ACCEPTED` 時: 同一 `request_id` の他の PENDING 応募は自動で `REJECTED` に更新。`match_requests.status = MATCHED`、`match_requests.matched_proposal_id` を設定
- `status = WITHDRAWN`: 応募チームが成立前に取り下げた場合
- `status = CANCELLED`: 成立後に一方がキャンセルした場合（スケジュールからも削除）

#### `match_reviews`

マッチング成立・実施後の相互レビュー。成立した応募（`match_proposals.status = ACCEPTED`）に対して、双方のチームが相手を評価する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `proposal_id` | BIGINT UNSIGNED | NO | — | FK → match_proposals |
| `reviewer_team_id` | BIGINT UNSIGNED | NO | — | FK → teams。レビューするチーム |
| `reviewee_team_id` | BIGINT UNSIGNED | NO | — | FK → teams。評価されるチーム |
| `rating` | TINYINT UNSIGNED | NO | — | 評価（1〜5）|
| `comment` | VARCHAR(1000) | YES | NULL | コメント（任意）|
| `is_public` | BOOLEAN | NO | TRUE | コメントの公開/非公開（FALSE = 評価値のみ公開）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_mr_proposal_reviewer (proposal_id, reviewer_team_id)  -- 同一マッチングへの二重レビュー防止
INDEX idx_mr_reviewee (reviewee_team_id, created_at)                -- チームの評価履歴取得
```

**制約・備考**
- レビュー可能期間: マッチング成立から **30日以内**（`match_proposals.updated_at + 30日` で判定）
- 成立した `proposal_id` に対して `reviewer_team_id` は 2 チーム（募集側・応募側）のみ
- `rating` は 1〜5 の整数。Service 層でバリデーション
- レビューの編集・削除は不可（不正なレビュー操作防止）
- `is_public = FALSE` の場合、コメント文字列は API レスポンスから除外（評価値のみ公開）

---

#### `ng_teams`

特定チームとの接触を一方的にブロックする設定。NGチーム同士は互いの募集が非表示になり、応募も不可になる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。設定するチーム |
| `blocked_team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。ブロック対象チーム |
| `reason` | VARCHAR(500) | YES | NULL | 設定理由（内部メモ。管理画面のみ表示）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_ng_pair (team_id, blocked_team_id)               -- 同一ペアの重複防止
INDEX idx_ng_team (team_id)                                    -- 自チームのNGリスト取得
INDEX idx_ng_blocked (blocked_team_id)                         -- 逆引き（募集検索フィルタ用）
```

**制約・備考**
- 非対称: チームAがチームBをブロックしても、チームBはチームAをブロックしていない限り自分のNGリストには表示されない
- **表示抑制は双方向**: A→B のNG設定があれば、AからもBからも互いの募集は非表示になる（一方的なブロックでも相互に非表示）
- 自チームを NG 設定することは Service 層で禁止（400 エラー）

### ER図（テキスト形式）
```
teams (1) ──── (N) match_requests
teams (1) ──── (N) match_proposals
match_requests (1) ──── (N) match_proposals
match_proposals (1) ──── (2) match_reviews   ← 双方向レビュー（最大2件）
teams (1) ──── (N) match_reviews (reviewer_team_id)
teams (1) ──── (N) match_reviews (reviewee_team_id)
teams (1) ──── (N) ng_teams (team_id)
teams (1) ──── (N) ng_teams (blocked_team_id)
```

#### `PATCH /api/v1/matching/proposals/{id}/accept`

応募の承諾（マッチング成立）。承諾する側は募集チームの ADMIN/DEPUTY_ADMIN のみ。

**リクエストボディ（任意）**
```json
{
  "schedule_title": "FC Osaka East との練習試合",
  "confirmed_date": "2026-04-12",
  "confirmed_time_from": "13:00:00",
  "confirmed_time_to": "16:00:00",
  "confirmed_venue": "大阪市中央公園グラウンド"
}
```

**レスポンス（200 OK）**
```json
{
  "data": {
    "proposal_id": 55,
    "request_id": 1,
    "status": "ACCEPTED",
    "schedule_created": true,
    "schedule_ids": {
      "requesting_team": 301,
      "proposing_team": 302
    }
  }
}
```

**処理の流れ**
1. `match_proposals.status` → `ACCEPTED`
2. 同一 `request_id` の他の PENDING 応募 → 一括 `REJECTED`（拒否理由: `"別の応募が承諾されました"`）
3. `match_requests.status` → `MATCHED`、`matched_proposal_id` 設定
4. 双方のチームスケジュール（F03.1）へ予定を自動追加（`MatchingAcceptedEvent` 発行）
5. 双方チームの ADMIN をメンバーとするグループチャットを自動作成（`MatchingAcceptedEvent` → F04.2 `ChatService` が受信、`source_type = 'MATCHING'`、`source_id = proposalId`）※Phase 9 で実装
6. 応募チームの ADMIN に承諾通知（F04.3）
7. 拒否された他チームの ADMIN に通知（F04.3）

5. 相手チームの ADMIN にキャンセル通知（F04.3）
6. **再公開通知は送信しない**（スパム防止。募集は静かに OPEN に戻るのみ）

## 5. ビジネスロジック

### マッチング成立フロー

```
1. チームAの ADMIN が募集を投稿 (status: OPEN)
2. チームBの ADMIN が募集を発見・応募 (status: PENDING)
3. チームAの ADMIN に応募通知（F04.3）
4. チームAの ADMIN が応募を承諾
   ├─ match_proposals.status → ACCEPTED
   ├─ 同一 request への他の PENDING 応募 → 一括 REJECTED
   ├─ match_requests.status → MATCHED
   ├─ MatchingAcceptedEvent 発行
   │   └─ F03.1 スケジュールサービスが受信 → 双方のチームスケジュールに予定追加
   └─ チームBに承諾通知、拒否された他チームに通知
5. 活動実施後（任意）: 双方が 30日以内にレビュー投稿
```

### 公開範囲フィルタリング

募集一覧 API で `visibility` に応じた閲覧可否を判定:
```sql
-- PLATFORM: NG除外のみ
-- ORGANIZATION: さらに同一組織チームに絞る
WHERE mr.visibility = 'PLATFORM'
   OR (
     mr.visibility = 'ORGANIZATION'
     AND mr.team_id IN (
       SELECT tm.team_id FROM team_members tm
       WHERE tm.organization_id = (
         SELECT tm2.organization_id FROM team_members tm2
         WHERE tm2.team_id = :currentTeamId LIMIT 1
       )
     )
   )
```

### NGチームによる表示抑制

募集一覧 API のクエリ時、以下の条件で除外:
```sql
-- NG除外条件（双方向）
WHERE team_id NOT IN (
  SELECT blocked_team_id FROM ng_teams WHERE team_id = :currentTeamId
  UNION
  SELECT team_id FROM ng_teams WHERE blocked_team_id = :currentTeamId
)
```

応募時は Service 層で同様の NG チェックを実行（409 ではなく 403 を返す）。

### チーム評価スコアの算出

平均評価はクエリ時に動的集計（`AVG(rating)`）。`match_reviews` テーブルに `reviewee_team_id` インデックスがあるため高速。denormalized カウントは設けない（更新頻度が低いため）。

```sql
SELECT
  reviewee_team_id,
  ROUND(AVG(rating), 1) AS average_rating,
  COUNT(*) AS review_count
FROM match_reviews
WHERE reviewee_team_id = :teamId
GROUP BY reviewee_team_id
```

**スコア非表示ルール**: `review_count < 5` の場合、API レスポンスの `average_rating` を `null` として返す（「評価なし」表示）。閾値はアプリケーション定数（`MatchingProperties.minReviewCountForScore = 5`）で管理し、後から変更可能にする。

### チームのキャンセル回数表示

成立後キャンセル回数はチーム単位で集計し、募集カード・チームプロフィールに公開表示する（社会的抑止力として機能）:
```sql
SELECT SUM(cancel_count) AS total_cancel_count
FROM match_requests
WHERE team_id = :teamId
```

`cancel_count` は `match_requests` レコードに付随するため、キャンセルした側・された側の双方について累計が取れる。API レスポンスの `team.cancel_count` に集計値を含める。

### 募集期限切れバッチ

毎晩 0:00（システム時刻）にスケジュールバッチ（`@Scheduled`）が実行:
```sql
UPDATE match_requests
SET status = 'EXPIRED'
WHERE status = 'OPEN'
  AND expires_at IS NOT NULL
  AND expires_at < NOW()
```

EXPIRED になった募集の PENDING 応募は `CANCELLED` に更新（`MatchingExpiredEvent` 発行 → 応募チームへ通知）。

### スケジュール自動追加（F03.1 連携）

`MatchingAcceptedEvent` のペイロード:
```java
record MatchingAcceptedEvent(
    Long requestId,
    Long proposalId,
    Long requestingTeamId,
    Long proposingTeamId,
    LocalDate confirmedDate,
    LocalTime confirmedTimeFrom,
    LocalTime confirmedTimeTo,
    String confirmedVenue,
    String scheduleTitle
) {}
```

F03.1 の `ScheduleService` がイベントを受信し、双方のチームスケジュールに `source_type = 'MATCHING'`、`source_id = proposalId` で予定を作成。キャンセル時は `source_type` と `source_id` で特定して削除。

### チャット自動作成（F04.2 連携）※Phase 9

`MatchingAcceptedEvent` を F04.2 の `ChatService` でも受信し、以下のグループチャットを自動作成:
- **参加メンバー**: 募集チームの ADMIN + 応募チームの ADMIN
- **チャット名**: `"[マッチング] {request.title}"`
- **識別子**: `source_type = 'MATCHING'`、`source_id = proposalId`（重複作成防止・キャンセル時の特定に使用）
- **キャンセル時**: チャット自体は残す（連絡用として有効）。システムメッセージ「このマッチングはキャンセルされました」を自動投稿

## 6. セキュリティ考慮事項

- **認可チェック**: 募集の操作（編集・削除・応募承諾）は `match_requests.team_id = currentUser.teamId` を Service 層入り口で検証
- **NGチーム迂回防止**: 応募 API でも募集一覧と同じ NG チェックを実施。URL 直打ちによる NG 迂回を防止
- **チームID推測対策**: 全 API で `teamId` の存在確認 + `currentUser` が当該チームに所属しているか確認
- **レート制限**: 募集投稿・応募に `Bucket4j` を適用（1チームあたり: 投稿 10件/日、応募 20件/日）。スパム募集・大量応募を抑制
- **レビュー不正防止**:
  - 成立していない（`status != ACCEPTED`）proposal へのレビュー投稿は 403
  - 成立に無関係なチームからのレビュー投稿は 403（`reviewer_team_id` が `requesting_team_id` / `proposing_team_id` のいずれかであることを検証）
  - レビューの編集・削除を禁止（不正操作防止）
- **個人情報**: `ng_teams.reason` は内部メモ（ADMIN 管理画面のみ表示。公開 API では返却しない）
- **募集内容の公開範囲**: `visibility = 'PLATFORM'` は NGチームを除く全認証済みチームが閲覧可能。`visibility = 'ORGANIZATION'` は同一組織内チームのみ。MATCHED / CANCELLED / EXPIRED は自チームのみ閲覧可能

## 7. Flywayマイグレーション

```
V8.001__create_match_requests_table.sql        -- match_requests テーブル作成
V8.002__create_match_proposals_table.sql       -- match_proposals テーブル作成
V8.003__create_match_reviews_table.sql         -- match_reviews テーブル作成

```

**マイグレーション上の注意点**
- `match_requests` → `match_proposals` → `match_reviews` の順で作成（FK 依存順）
- `ng_teams` は `teams` テーブルへの自己参照 FK が2本（`team_id`、`blocked_team_id`）
- `match_requests.matched_proposal_id` は `match_proposals` 作成後に FK を追加（循環 FK を避けるため ALTER TABLE で後付け）:
  ```sql
  -- V8.001 で matched_proposal_id は FK なし（BIGINT UNSIGNED NULL）で作成
  -- V8.002 の末尾で ALTER TABLE match_requests ADD CONSTRAINT ...
  ALTER TABLE match_requests
    ADD CONSTRAINT fk_mr_matched_proposal
    FOREIGN KEY (matched_proposal_id) REFERENCES match_proposals(id) ON DELETE SET NULL;
  ```
