# F08.1: マッチング・対外交流

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 8
> **最終更新**: 2026-03-15
> **関連ドキュメント**: F03.1 スケジュール共有（成立後の自動スケジュール追加）、F04.2 チャット（成立後のグループチャット自動作成）、F04.3 プッシュ通知（応募通知・成立通知）、F04.5 モデレーション（NGチーム・通報）、F01.3 テンプレート・モジュール（選択式モジュール #8）

---

## 1. 概要

スポーツチーム・文化サークル・地域コミュニティ等が、地域・カテゴリ（小学生〜シニア）・レベル・活動大分類に応じた対戦相手・練習相手・交流先、さらにメンバー・審判・コーチ等の人材を汎用的に募集・応募できる機能。チームAが「対戦相手募集」を投稿し、チームBが応募、ADMIN が承諾すると**マッチング成立**となり、双方のチームスケジュール（F03.1）に自動で予定が追加される。都道府県・市区町村単位の地域フィルタと年齢カテゴリフィルタにより、近隣かつ同レベルの相手を効率的に発見できる。成立後の相互レビューでチームの信頼スコアが蓄積される。NGチーム設定により、トラブル防止のために特定チームを非表示・応募不可にする機能も提供する。

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全チームの募集・応募・レビューの参照。不正コンテンツの削除 |
| ADMIN | 自チームの募集投稿 CRUD、応募の承諾・拒否、NGチーム管理、レビュー投稿 |
| DEPUTY_ADMIN | `MANAGE_MATCHING` 権限を持つ場合: 募集投稿 CRUD、応募の承諾・拒否 |
| MEMBER | 募集一覧・詳細の閲覧（自チームの募集・他チームの公開募集）、応募なし（ADMIN/DEPUTY_ADMIN が操作） |
| SUPPORTER | 対象外 |
| GUEST | 対象外 |

> **設計判断**: 応募・承諾はチーム代表者（ADMIN/DEPUTY_ADMIN）が行う。MEMBERは閲覧のみ。「チーム同士の正式な交流」という性質上、個人メンバーが勝手に他チームへ応募できる仕様は混乱を招く。

### 対象レベル
- [ ] 組織 (Organization) — 対象外（チーム間交流のため）
- [x] チーム (Team) — 募集・応募・NGチーム管理
- [ ] 個人 (Personal) — 対象外（個人間マッチングは別途検討）

### テーブル定義

#### `match_requests`

チームが対外交流の相手を募集する投稿。公開されると他チームの ADMIN に表示される。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。募集チーム |
| `title` | VARCHAR(100) | NO | — | 募集タイトル（例: 「練習試合相手募集@大阪」） |
| `description` | TEXT | YES | NULL | 詳細説明（チームの特徴・要望等） |
| `activity_type` | ENUM('COMPETITION', 'PRACTICE', 'EXCHANGE', 'RECRUIT', 'OTHER') | NO | — | 活動大分類（対戦 / 練習 / 交流 / 人材募集 / その他） |
| `activity_detail` | VARCHAR(50) | YES | NULL | 活動の具体内容（自由記述。例: '練習試合', '審判募集', 'マネージャー募集', '合同練習'）。検索・表示用 |
| `category` | ENUM('ELEMENTARY', 'JUNIOR_HIGH', 'HIGH_SCHOOL', 'UNIVERSITY', 'ADULT', 'SENIOR', 'ANY') | NO | 'ANY' | 対象カテゴリ（小学生 / 中学生 / 高校生 / 大学生 / 一般 / シニア / 不問） |
| `visibility` | ENUM('PLATFORM', 'ORGANIZATION') | NO | 'PLATFORM' | 公開範囲（PLATFORM = Mannschaft 全チーム公開 / ORGANIZATION = 同一組織内チームのみ）|
| `prefecture_code` | CHAR(2) | NO | — | JIS X 0401 都道府県コード（例: '27' = 大阪府）。FK → prefectures。地域検索用 |
| `city_code` | CHAR(5) | YES | NULL | 全国地方公共団体コード（例: '27128' = 大阪市中央区）。FK → cities。NULL = 都道府県のみで絞り込み |
| `venue_name` | VARCHAR(200) | YES | NULL | 希望会場名（自由記述。NULL = 相談） |
| `preferred_date_from` | DATE | YES | NULL | 希望日程（開始）。NULL = 未定 |
| `preferred_date_to` | DATE | YES | NULL | 希望日程（終了）。NULL = 未定 |
| `preferred_time_from` | TIME | YES | NULL | 希望時間帯（開始）。NULL = 未定 |
| `preferred_time_to` | TIME | YES | NULL | 希望時間帯（終了）。NULL = 未定 |
| `level` | ENUM('BEGINNER', 'INTERMEDIATE', 'ADVANCED', 'ANY') | NO | 'ANY' | 希望相手レベル |
| `min_participants` | SMALLINT UNSIGNED | YES | NULL | 最低参加人数（NULL = 不問） |
| `max_participants` | SMALLINT UNSIGNED | YES | NULL | 最大参加人数（NULL = 不問） |
| `status` | ENUM('OPEN', 'MATCHED', 'CANCELLED', 'EXPIRED') | NO | 'OPEN' | ステータス |
| `proposal_count` | INT UNSIGNED | NO | 0 | 応募数（非正規化カウント。atomic UPDATE で管理） |
| `expires_at` | DATETIME | YES | NULL | 募集期限（NULL = 無期限。期限超過で自動 EXPIRED）|
| `matched_proposal_id` | BIGINT UNSIGNED | YES | NULL | FK → match_proposals（成立した応募。成立後に設定）|
| `cancel_count` | SMALLINT UNSIGNED | NO | 0 | 成立後キャンセル回数（この募集に関するキャンセル累計。atomic UPDATE で管理）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
INDEX idx_mr_team (team_id, status, created_at)                        -- 自チームの募集一覧
INDEX idx_mr_search (visibility, prefecture_code, city_code, activity_type, status, expires_at)  -- 地域・種別での募集検索
INDEX idx_mr_category (category, status)                               -- カテゴリ絞り込み
INDEX idx_mr_level (level, status)                                     -- レベル絞り込み
INDEX idx_mr_expires (expires_at, status)                              -- 期限切れバッチ処理用
FULLTEXT INDEX ft_mr_search (title, activity_detail)                   -- タイトル+活動詳細のキーワード検索
```

**制約・備考**
- `preferred_date_from <= preferred_date_to` は Service 層でバリデーション
- `preferred_time_from <= preferred_time_to` は Service 層でバリデーション（同日内の時間帯整合性）
- `min_participants <= max_participants` は Service 層でバリデーション（両方が非 NULL の場合のみ）
- `proposal_count` は応募作成時に `+1`、応募取り下げ（WITHDRAWN）時に `-1` で atomic 更新。REJECTED / ACCEPTED はカウント変更しない（募集側が「何件の有効応募があるか」を把握する指標）。募集 EXPIRED 時の一括 CANCELLED でもカウントは変更しない（履歴的意味を保持）
- `status = MATCHED` 後は内容の編集不可。`matched_proposal_id` が設定される
- `status = EXPIRED` は毎晩バッチ（`expires_at < NOW() AND status = 'OPEN'`）で自動更新。**最大24時間の遅延**あり（リアルタイム不要の設計判断。API 側では `expires_at < NOW()` の OPEN 募集を非表示にすることでユーザー体感上の遅延を解消）
- `visibility = 'PLATFORM'`: OPEN 中の募集は NGチームを除く全認証済みチームから閲覧可能
- `visibility = 'ORGANIZATION'`: OPEN 中の募集は同一組織内チームのみ閲覧可能（`teams` テーブルの `organization_id` で判定）。`prefecture_code` で同一都道府県内チームへの限定公開にも対応可能
- キャンセル後は自動で `status = 'OPEN'` に復元（再募集可）。ただし再公開通知は送信しない（スパム防止）

#### `match_proposals`

他チームの募集に対する応募レコード。1募集に対して複数チームが応募可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `request_id` | BIGINT UNSIGNED | NO | — | FK → match_requests。ON DELETE CASCADE |
| `proposing_team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。応募チーム |
| `message` | TEXT | YES | NULL | 応募メッセージ（自己紹介・希望条件等） |
| `proposed_venue` | VARCHAR(200) | YES | NULL | 提案会場（NULL = 募集条件に従う）|
| `status` | ENUM('PENDING', 'ACCEPTED', 'REJECTED', 'CANCELLED', 'WITHDRAWN') | NO | 'PENDING' | 応募ステータス |
| `status_reason` | VARCHAR(500) | YES | NULL | ステータス変更理由（REJECTED: 拒否理由 / CANCELLED: キャンセル理由 / WITHDRAWN: 取り下げ理由。任意）|
| `cancelled_by_team_id` | BIGINT UNSIGNED | YES | NULL | FK → teams。ON DELETE SET NULL。キャンセル実行チーム（`status = CANCELLED` 時のみ設定）|
| `cancellation_type` | ENUM('UNILATERAL', 'MUTUAL_PENDING', 'MUTUAL') | YES | NULL | キャンセル種別（`status = CANCELLED` 時のみ設定。UNILATERAL=一方的 / MUTUAL_PENDING=合意待ち / MUTUAL=双方合意済み）|
| `mutual_agreed_at` | DATETIME | YES | NULL | 相手チームが合意キャンセルを承認した日時（`cancellation_type = MUTUAL` 時のみ設定）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_mp_request_team (request_id, proposing_team_id)  -- 同一チームの二重応募防止
INDEX idx_mp_proposing_team (proposing_team_id, status)        -- 自チームの応募一覧
INDEX idx_mp_request_status (request_id, status)               -- 募集ごとの応募一覧
```

**制約・備考**
- 日程候補は `match_proposal_dates` 子テーブルで管理（1応募あたり最大5件の複数日程候補に対応）
- 1チームが同一募集に複数回応募することを UNIQUE KEY で防止
- 自チームの募集に自チームが応募することは Service 層で禁止（400 エラー）
- `status = ACCEPTED` 時: 同一 `request_id` の他の PENDING 応募は自動で `REJECTED` に更新。`match_requests.status = MATCHED`、`match_requests.matched_proposal_id` を設定
- `status = WITHDRAWN`: 応募チームが成立前に取り下げた場合。`status_reason` に取り下げ理由を保存（任意）
- `status = CANCELLED`: 成立後に一方がキャンセルした場合（スケジュールからも削除）。`cancelled_by_team_id` にキャンセル実行チーム、`status_reason` にキャンセル理由を保存。`cancellation_type` でキャンセル種別を管理:
  - `UNILATERAL`: 一方的キャンセル（`cancel_count` に加算）
  - `MUTUAL_PENDING`: 合意キャンセル申請中（相手の承認待ち。承認されるまでは `cancel_count` に加算）
  - `MUTUAL`: 双方合意キャンセル（**`cancel_count` に加算しない**）。`mutual_agreed_at` に承認日時を記録

#### `match_reviews`

マッチング成立・実施後の相互レビュー。成立した応募（`match_proposals.status = ACCEPTED`）に対して、双方のチームが相手を評価する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `proposal_id` | BIGINT UNSIGNED | NO | — | FK → match_proposals。ON DELETE CASCADE |
| `reviewer_team_id` | BIGINT UNSIGNED | NO | — | FK → teams。レビューするチーム |
| `reviewee_team_id` | BIGINT UNSIGNED | NO | — | FK → teams。評価されるチーム |
| `rating` | TINYINT UNSIGNED | NO | — | 評価（1〜5）|
| `comment` | VARCHAR(1000) | YES | NULL | コメント（任意）|
| `is_public` | BOOLEAN | NO | TRUE | コメントの公開/非公開（FALSE = 評価値のみ公開）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_mr_proposal_reviewer (proposal_id, reviewer_team_id)  -- 同一マッチングへの二重レビュー防止
INDEX idx_mr_reviewee (reviewee_team_id, created_at)                -- チームの評価履歴取得
```

**制約・備考**
- レビュー可能期間: マッチング成立から **30日以内**（`match_proposals.updated_at + 30日` で判定）
- 成立した `proposal_id` に対して `reviewer_team_id` は 2 チーム（募集側・応募側）のみ
- `rating` は 1〜5 の整数。Service 層でバリデーション
- レビューの編集・削除は不可（不正なレビュー操作防止）
- `is_public = FALSE` の場合、コメント文字列は API レスポンスから除外（評価値のみ公開）

#### `ng_teams`

特定チームとの接触を一方的にブロックする設定。NGチーム同士は互いの募集が非表示になり、応募も不可になる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。設定するチーム |
| `blocked_team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。ブロック対象チーム |
| `reason` | VARCHAR(500) | YES | NULL | 設定理由（内部メモ。管理画面のみ表示）|
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_ng_pair (team_id, blocked_team_id)               -- 同一ペアの重複防止
INDEX idx_ng_team (team_id)                                    -- 自チームのNGリスト取得
INDEX idx_ng_blocked (blocked_team_id)                         -- 逆引き（募集検索フィルタ用）
```

**制約・備考**
- 非対称: チームAがチームBをブロックしても、チームBはチームAをブロックしていない限り自分のNGリストには表示されない
- **表示抑制は双方向**: A→B のNG設定があれば、AからもBからも互いの募集は非表示になる（一方的なブロックでも相互に非表示）
- 自チームを NG 設定することは Service 層で禁止（400 エラー）

**インデックス**
```sql
UNIQUE KEY uq_mnp_team (team_id)                                     -- 1チーム1設定
```

**制約・備考**
- 1チーム1設定（UNIQUE KEY で保証）

- `is_enabled = FALSE` で通知を一時停止可能（設定を消さずにオフにできる）

#### `cities`

市区町村マスタ（全国地方公共団体コード、JIS X 0402）。約1,900件の初期データを Flyway で投入。

| カラム名 | 型 | NULL | デフォルト | 説明 |

#### `PATCH /api/v1/matching/proposals/{id}/accept`

応募の承諾（マッチング成立）。承諾する側は募集チームの ADMIN/DEPUTY_ADMIN のみ。

**リクエストボディ（任意）**
```json
{
  "schedule_title": "FC Osaka East との練習試合",
  "confirmed_date": "2026-04-12",
  "confirmed_time_from": "13:00:00",
  "confirmed_time_to": "16:00:00",
  "confirmed_venue": "大阪市中央公園グラウンド"
}
```

**レスポンス（200 OK）**
```json
{
  "data": {
    "proposal_id": 55,
    "request_id": 1,
    "status": "ACCEPTED",
    "schedule_created": true,
    "schedule_ids": {
      "requesting_team": 301,
      "proposing_team": 302
    }
  }
}
```

**処理の流れ**
1. **悲観ロック取得**: `SELECT ... FOR UPDATE` で `match_requests` を行ロック（同一募集への同時 accept を排他制御）
2. `match_requests.status = OPEN` を検証（OPEN 以外なら 409 Conflict: `"この募集は既にマッチング成立済みです"`）
3. `match_proposals.status` → `ACCEPTED`
4. 同一 `request_id` の他の PENDING 応募 → 一括 `REJECTED`（`status_reason`: `"別の応募が承諾されました"`）
5. `match_requests.status` → `MATCHED`、`matched_proposal_id` 設定
6. `confirmed_date` が設定されている場合のみ、双方のチームスケジュール（F03.1）へ予定を自動追加（`MatchingAcceptedEvent` 発行）。`confirmed_date = NULL` の場合はスケジュール作成をスキップ（RECRUIT 等の日程未定マッチングに対応）
7. 双方チームの ADMIN をメンバーとするグループチャットを自動作成（`MatchingAcceptedEvent` → F04.2 `ChatService` が受信、`source_type = 'MATCHING'`、`source_id = proposalId`）※Phase 9 で実装
8. 応募チームの ADMIN に承諾通知（F04.3）
9. 拒否された他チームの ADMIN に通知（F04.3）

#### `PATCH /api/v1/matching/proposals/{id}/cancel`

マッチング成立後のキャンセル。募集側・応募側のどちらの ADMIN も実行可能。`mutual: true` で合意キャンセルを申請できる。

**リクエストボディ**
```json
{
  "reason": "会場が確保できなくなりました。",
  "mutual": false
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `reason` | string | 任意 | キャンセル理由 |
| `mutual` | boolean | 任意 | `true` = 合意キャンセルを申請（相手チームの承認で `cancel_count` 非加算）。デフォルト `false`（一方的キャンセル）|

**処理の流れ**
1. `match_proposals.status` → `CANCELLED`、`cancelled_by_team_id` に実行チームの ID を設定、`status_reason` にキャンセル理由を保存
2. `cancellation_type` を設定:
   - `mutual = false` → `UNILATERAL`
   - `mutual = true` → `MUTUAL_PENDING`（相手チームの承認待ち）
3. `match_requests.status` → `OPEN`（再募集可能に戻す）、`matched_proposal_id` → NULL
4. `match_requests.cancel_count` を `+1`（atomic UPDATE。MUTUAL 確定時に `-1` で戻す）
5. 双方のスケジュールから自動追加した予定を削除（`MatchingCancelledEvent` 発行）
6. 相手チームの ADMIN にキャンセル通知（F04.3）。`MUTUAL_PENDING` の場合は合意承認リンク付きで通知
7. **再公開通知は送信しない**（スパム防止。募集は静かに OPEN に戻るのみ）

---

#### `PATCH /api/v1/matching/proposals/{id}/agree-cancel`

合意キャンセルの承認。`cancellation_type = MUTUAL_PENDING` の場合に、キャンセルを実行していない側のチームの ADMIN が操作。承認すると双方の `cancel_count` に加算されなくなる。

**レスポンス（200 OK）**
```json
{
  "data": {
    "proposal_id": 55,
    "cancellation_type": "MUTUAL",
    "mutual_agreed_at": "2026-04-13T10:30:00"
  }
}
```

**処理の流れ**
1. `cancellation_type = MUTUAL_PENDING` であることを検証（それ以外は 422）
2. 操作チームが `cancelled_by_team_id` と**異なる**チームであることを検証（キャンセル実行者自身は承認不可。403）
3. `cancellation_type` → `MUTUAL`、`mutual_agreed_at` → `NOW()`
4. `match_requests.cancel_count` を `-1`（atomic UPDATE。合意成立により加算を取り消し）
5. キャンセル実行チームの ADMIN に合意完了通知（F04.3）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | 権限不足 / キャンセル実行チーム自身が承認しようとした場合 |
| 404 | proposal が存在しない / CANCELLED でない |
| 422 | `cancellation_type` が `MUTUAL_PENDING` でない（既に MUTUAL / UNILATERAL 確定済み）|

**合意期限**: MUTUAL_PENDING のまま **7日間** 経過した場合、期限切れバッチで自動的に `UNILATERAL` に確定する（`cancel_count` は既に加算済みのため追加処理不要）

**リクエストボディ**
```json
{
```

## 5. ビジネスロジック

### マッチング成立フロー

```
1. チームAの ADMIN が募集を投稿 (status: OPEN)
2. チームBの ADMIN が募集を発見・応募 (status: PENDING)
3. チームAの ADMIN に応募通知（F04.3）
4. チームAの ADMIN が応募を承諾
   ├─ match_proposals.status → ACCEPTED
   ├─ 同一 request への他の PENDING 応募 → 一括 REJECTED
   ├─ match_requests.status → MATCHED
   ├─ MatchingAcceptedEvent 発行
   │   └─ F03.1 スケジュールサービスが受信 → 双方のチームスケジュールに予定追加
   └─ チームBに承諾通知、拒否された他チームに通知
5. 活動実施後（任意）: 双方が 30日以内にレビュー投稿
```

### 公開範囲フィルタリング

募集一覧 API で `visibility` に応じた閲覧可否を判定:
```sql
-- PLATFORM: NG除外のみ
-- ORGANIZATION: さらに同一組織チームに絞る
WHERE mr.visibility = 'PLATFORM'
   OR (
     mr.visibility = 'ORGANIZATION'
     AND mr.team_id IN (
       SELECT t.id FROM teams t
       WHERE t.organization_id = (
         SELECT t2.organization_id FROM teams t2
         WHERE t2.id = :currentTeamId
       )
     )
   )
```

### NGチームによる表示抑制

募集一覧 API のクエリ時、以下の条件で除外:
```sql
-- NG除外条件（双方向）
WHERE team_id NOT IN (
  SELECT blocked_team_id FROM ng_teams WHERE team_id = :currentTeamId
  UNION
  SELECT team_id FROM ng_teams WHERE blocked_team_id = :currentTeamId
)
```

応募時は Service 層で同様の NG チェックを実行（409 ではなく 403 を返す）。

### チーム評価スコアの算出

平均評価はクエリ時に動的集計（`AVG(rating)`）。**直近2年間のレビューのみ**を対象とする（古いレビューは集計から除外。意図しない事情での低評価がずっと残り続けることを防止し、チームの改善努力を反映する）。`match_reviews` テーブルに `reviewee_team_id` + `created_at` の複合インデックスがあるため高速。denormalized カウントは設けない（更新頻度が低いため）。

```sql
SELECT
  reviewee_team_id,
  ROUND(AVG(rating), 1) AS average_rating,
  COUNT(*) AS review_count
FROM match_reviews
WHERE reviewee_team_id = :teamId
  AND created_at >= DATE_SUB(NOW(), INTERVAL 2 YEAR)
GROUP BY reviewee_team_id
```

**スコア非表示ルール**: `review_count < 5` の場合、API レスポンスの `average_rating` を `null` として返す（「評価なし」表示）。閾値はアプリケーション定数（`MatchingProperties.minReviewCountForScore = 5`）で管理し、後から変更可能にする。

> **2年失効の設計判断**: レビューレコード自体は物理削除しない（監査用に保持）。集計クエリの `WHERE created_at >= DATE_SUB(NOW(), INTERVAL 2 YEAR)` で除外するのみ。失効期間はアプリケーション定数（`MatchingProperties.reviewRetentionYears = 2`）で管理。キャンセル履歴も同一の2年ルールを適用し、レビュー・キャンセルともに「直近2年間の行動」でチームを評価する統一基準とする。

### チームのキャンセル回数表示

キャンセル情報は **2つの粒度** で集計・表示する:

#### リクエスト単位（募集カード表示用）

`match_requests.cancel_count` は「この募集が過去何回キャンセルされたか」を示す（誰が実行したかは問わない）。募集カードに表示することで、頻繁にキャンセルが発生する募集を応募者が判断できる。

#### チーム単位（チームプロフィール・社会的抑止力）

チームプロフィールに表示する `cancel_count` は **そのチームが自ら実行したキャンセルのみ** を集計する。`cancelled_by_team_id` を使い、他チーム起因のキャンセルは含めない（被害者にペナルティを負わせない）:

```sql
-- このチームが実行した一方的キャンセル数（直近2年間のみ）
SELECT COUNT(*) AS initiated_cancel_count
FROM match_proposals
WHERE cancelled_by_team_id = :teamId
  AND status = 'CANCELLED'
  AND cancellation_type != 'MUTUAL'
  AND updated_at >= DATE_SUB(NOW(), INTERVAL 2 YEAR)
```

API レスポンスの `team.cancel_count` にはこの値を含める。**双方合意キャンセル（`MUTUAL`）は加算しない**。**2年経過したキャンセル履歴は集計対象外**（意図しない事情でのキャンセルがずっと残り続けることを防止。レビューと同一の失効期間）。

> **設計判断**: `match_requests.cancel_count`（denormalized）はリクエスト単位の信頼度指標として維持し、MUTUAL 確定時に `-1` するロジックも継続。チームプロフィール用の集計は `cancelled_by_team_id` ベースの動的クエリで行い、用途を明確に分離する。

#### キャンセル履歴の閲覧

チームプロフィールのキャンセル回数はクリック可能で、キャンセル履歴一覧を表示する。各履歴にはキャンセル理由（`status_reason`）と日時を含め、透明性を確保する。

### 募集期限切れバッチ

毎晩 0:00（システム時刻）にスケジュールバッチ（`@Scheduled`）が実行:
```sql
UPDATE match_requests
SET status = 'EXPIRED'
WHERE status = 'OPEN'
  AND expires_at IS NOT NULL
  AND expires_at < NOW()
```

EXPIRED になった募集の PENDING 応募は `REJECTED` に更新（`status_reason`: `"募集が期限切れになりました"`）。`MatchingExpiredEvent` 発行 → 応募チームへ通知。

> **設計判断**: PENDING → CANCELLED ではなく REJECTED を使用する。CANCELLED は「成立後キャンセル」の意味で `cancellation_type` / `cancelled_by_team_id` を伴うため、期限切れによる失効とは意味が異なる。REJECTED は「募集側の事情による不成立」と解釈でき、期限設定も募集側の事情に該当する。

同バッチで**合意キャンセル期限切れ**も処理:
```sql
UPDATE match_proposals
SET cancellation_type = 'UNILATERAL'
WHERE status = 'CANCELLED'
  AND cancellation_type = 'MUTUAL_PENDING'
  AND updated_at < DATE_SUB(NOW(), INTERVAL 7 DAY)
```
7日間相手チームの承認がなかった `MUTUAL_PENDING` は自動的に `UNILATERAL`（一方的キャンセル）に確定。`cancel_count` は既に加算済みのため追加処理不要。

### スケジュール自動追加（F03.1 連携）

`MatchingAcceptedEvent` のペイロード:
```java
record MatchingAcceptedEvent(
    Long requestId,
    Long proposalId,
    Long requestingTeamId,
    Long proposingTeamId,
    LocalDate confirmedDate,
    LocalTime confirmedTimeFrom,
    LocalTime confirmedTimeTo,
    String confirmedVenue,
    String scheduleTitle
) {}
```

F03.1 の `ScheduleService` がイベントを受信し、`confirmedDate != null` の場合のみ双方のチームスケジュールに `source_type = 'MATCHING'`、`source_id = proposalId` で予定を作成。`confirmedDate = null` の場合はスケジュール作成をスキップ（`activity_type = RECRUIT` 等の日程未定マッチングに対応）。キャンセル時は `source_type` と `source_id` で特定して削除。

### チャット自動作成（F04.2 連携）※Phase 9

`MatchingAcceptedEvent` を F04.2 の `ChatService` でも受信し、以下のグループチャットを自動作成:
- **参加メンバー**: 募集チームの ADMIN + 応募チームの ADMIN
- **チャット名**: `"[マッチング] {request.title}"`
- **識別子**: `source_type = 'MATCHING'`、`source_id = proposalId`（重複作成防止・キャンセル時の特定に使用）
- **キャンセル時**: チャット自体は残す（連絡用として有効）。システムメッセージ「このマッチングはキャンセルされました」を自動投稿

### マッチング推薦通知（新規募集の自動通知）

新規募集が投稿されたとき、`match_notification_preferences` テーブルを参照して条件が合致するチームの ADMIN に通知を送信する:

```sql
SELECT mnp.team_id
FROM match_notification_preferences mnp
WHERE mnp.is_enabled = TRUE
  AND mnp.team_id != :postingTeamId
  AND (mnp.prefecture_code IS NULL OR mnp.prefecture_code = :prefectureCode)
  AND (mnp.city_code IS NULL OR mnp.city_code = :cityCode)
  AND (mnp.activity_type IS NULL OR mnp.activity_type = :activityType)
  AND (mnp.category IS NULL OR mnp.category = :category OR :category = 'ANY')
  AND mnp.team_id NOT IN (/* NG チーム除外 */)
```

`MatchRequestCreatedEvent` 発行 → 通知サービスが受信し、合致チームの ADMIN にプッシュ通知。通知頻度抑制として 1チームあたり最大 **5件/日** に制限（`Bucket4j` でチーム単位のレートリミット）。

### 期限切れ前リマインダー

募集期限切れバッチ（毎晩 0:00）で、翌日に期限を迎える募集の ADMIN にリマインダー通知を送信:

```sql
SELECT id, team_id, title, proposal_count
FROM match_requests
WHERE status = 'OPEN'
  AND expires_at IS NOT NULL
  AND expires_at BETWEEN NOW() AND DATE_ADD(NOW(), INTERVAL 24 HOUR)
  AND deleted_at IS NULL
```

通知内容: 「募集『{title}』が明日期限を迎えます。応募 {proposal_count} 件。」応募が0件の場合は「条件を変更して再投稿しませんか？」を追記。

## 6. セキュリティ考慮事項

- **認可チェック**: 募集の操作（編集・削除・応募承諾）は `match_requests.team_id = currentUser.teamId` を Service 層入り口で検証
- **NGチーム迂回防止**: 応募 API でも募集一覧と同じ NG チェックを実施。URL 直打ちによる NG 迂回を防止
- **チームID推測対策**: 全 API で `teamId` の存在確認 + `currentUser` が当該チームに所属しているか確認
- **レート制限**: 募集投稿・応募に `Bucket4j` を適用（1チームあたり: 投稿 10件/日、応募 20件/日）。スパム募集・大量応募を抑制
- **レビュー不正防止**:
  - 成立していない（`status != ACCEPTED`）proposal へのレビュー投稿は 403
  - 成立に無関係なチームからのレビュー投稿は 403（`reviewer_team_id` が `requesting_team_id` / `proposing_team_id` のいずれかであることを検証）
  - レビューの編集・削除を禁止（不正操作防止）
- **個人情報**: `ng_teams.reason` は内部メモ（ADMIN 管理画面のみ表示。公開 API では返却しない）
- **募集内容の公開範囲**: `visibility = 'PLATFORM'` は NGチームを除く全認証済みチームが閲覧可能。`visibility = 'ORGANIZATION'` は同一組織内チームのみ。MATCHED / CANCELLED / EXPIRED は自チームのみ閲覧可能
