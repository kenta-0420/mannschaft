# F11.8: ゲーミフィケーション（ポイント・バッジ・ランキング）

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 11
> **最終更新**: 2026-03-18
> **モジュール種別**: 選択式モジュール #17（ゲーミフィケーション）
> **関連ドキュメント**: F04.1 タイムライン（投稿・リアクションがポイント源）、F06.4 活動記録（参加がポイント源）、F03.1 スケジュール・出欠管理（出欠回答がポイント源）、F02.2 ダッシュボード（ウィジェット連携）

---

## 1. 概要

メンバーの日常的な活動（タイムライン投稿・活動記録への参加・出欠回答・コメント・ログイン等）に対してポイントを自動付与し、バッジ・ランキングで可視化するエンゲージメント促進機能。ADMIN がカスタムポイントルールやカスタムバッジを作成可能で、組織・チームの特性に応じた報酬設計を実現する。

**主要ユースケース**:
- スポーツチーム: 練習参加率の可視化、月間 MVP 自動選出
- 学校: 活動参加ポイントで表彰対象を明確化
- コミュニティ: 投稿・イベント参加の活性化指標
- 企業: 社内コミュニケーション促進・貢献度の可視化

### 本ドキュメントのスコープ

| 機能 | 説明 |
|------|------|
| ポイントシステム | アクション別にポイントを自動付与。ADMIN によるカスタムルール追加 |
| バッジシステム | 条件達成でバッジを自動付与。ADMIN によるカスタムバッジ作成 |
| ランキング | 週次/月次/年次のポイントランキング。チーム内・組織内の表示 |
| プライバシー設定 | メンバーが自身のランキング表示を非公開に設定可能 |
| ダッシュボード連携 | F02.2 に「今月のポイント」「獲得バッジ」ウィジェットを追加 |

**スコープ外**:
- ポイントの金銭的交換・商品交換 → 将来検討
- チーム間のランキング比較 → 将来検討
- MVP 投票機能（手動投票） → F05.4 アンケート・投票で代替可能

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全スコープのゲーミフィケーション設定閲覧。システムデフォルトポイントルールの管理 |
| ADMIN | ゲーミフィケーション設定の有効化/無効化。カスタムポイントルール・カスタムバッジの CRUD。ランキングの閲覧。年次リセットの実行 |
| DEPUTY_ADMIN | `GAMIFICATION_MANAGE` パーミッションが付与されている場合: ADMIN と同等の操作。付与されていない場合: MEMBER と同等 |
| MEMBER | 自分のポイント・バッジ・ランキングの閲覧。プライバシー設定（ランキング非表示）の変更 |
| SUPPORTER | 自分のポイント・バッジの閲覧（ランキングには表示されない） |
| GUEST | 閲覧不可 |

### 対象レベル
- [x] 組織 (Organization) — 組織全体のポイントルール・ランキング
- [x] チーム (Team) — チーム内のポイントルール・ランキング
- [ ] 個人 (Personal) — ポイント・バッジの閲覧のみ（設定変更不可）

### テーブル定義

#### `gamification_configs`

スコープ別のゲーミフィケーション有効化設定。モジュールが有効化されたスコープごとに1レコード。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `is_enabled` | BOOLEAN | NO | TRUE | ゲーミフィケーション有効/無効 |
| `is_ranking_enabled` | BOOLEAN | NO | TRUE | ランキング表示の有効/無効 |
| `ranking_display_count` | INT | NO | 10 | ランキング表示人数（上位 N 名。最小 3、最大 100） |
| `point_reset_month` | SMALLINT | YES | NULL | 年次ポイントリセット月（1〜12。NULL = リセットしない） |
| `version` | BIGINT | NO | 0 | 楽観的ロック |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_gc_scope (scope_type, scope_id)  -- スコープごとに1レコード
```

**制約・備考**
- モジュール有効化時に Service 層で自動作成（デフォルト値で INSERT）
- `is_enabled = FALSE` の場合、ポイント付与・バッジ判定・ランキング集計をすべて停止
- `point_reset_month`: 年次リセット月。指定月の 1 日 00:00 JST にバッチで `point_transactions` に `RESET` タイプのトランザクションを挿入し、累計を 0 にする。NULL の場合はリセットしない（累積方式）
- `ranking_display_count`: フロントエンドのランキング表示件数を制御。Redis Sorted Set の `ZREVRANGE` の `count` パラメータにも使用
- `version`: 楽観的ロック。複数 ADMIN が同時に設定を変更した場合の競合を検知（.claudecode.md §22 準拠）

#### `point_rules`

ポイント付与ルール定義。システムデフォルトルール（`is_system = TRUE`）と ADMIN カスタムルール（`is_system = FALSE`）の2種類。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `action_type` | VARCHAR(50) | NO | — | アクション種別（`ACTIVITY_PARTICIPATION` / `TIMELINE_POST` / `ATTENDANCE_RESPONSE` / `COMMENT_POST` / `DAILY_LOGIN` / `CUSTOM`） |
| `name` | VARCHAR(100) | NO | — | ルール名（例: 「活動参加」「投稿ボーナス」） |
| `description` | VARCHAR(500) | YES | NULL | ルールの説明 |
| `points` | INT | NO | — | 付与ポイント数（正の整数。1〜1000） |
| `daily_limit` | INT | YES | NULL | 1日あたりの付与上限回数（NULL = 無制限） |
| `is_system` | BOOLEAN | NO | FALSE | システムデフォルトルールか。TRUE の場合は削除不可（ポイント値の変更は可能） |
| `is_active` | BOOLEAN | NO | TRUE | ルールの有効/無効 |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）。作成者。システムルールは NULL |
| `version` | BIGINT | NO | 0 | 楽観的ロック |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_pr_scope (scope_type, scope_id, is_active, deleted_at)       -- スコープ別の有効ルール一覧
-- ※ スコープ内でアクション種別の重複防止（CUSTOM 以外）はアプリ層で `deleted_at IS NULL` 条件付きバリデーションで実施。
-- MySQL の UNIQUE KEY に deleted_at を含めても NULL 同士は DISTINCT 扱いのため DB レベルでの重複防止が機能しない。
```

**制約・備考**
- システムデフォルトルールはゲーミフィケーション有効化時に自動作成:
  - `ACTIVITY_PARTICIPATION`: +10pt（活動記録への参加）
  - `TIMELINE_POST`: +5pt（タイムライン投稿）
  - `ATTENDANCE_RESPONSE`: +3pt（出欠回答）
  - `COMMENT_POST`: +2pt（コメント投稿）
  - `DAILY_LOGIN`: +1pt（日次ログイン。daily_limit = 1）
- `action_type = CUSTOM` のルールは複数作成可能（UNIQUE 制約は `CUSTOM` 以外に適用。アプリ層で `CUSTOM` の場合はユニーク検証をスキップ）
- `daily_limit`: `DAILY_LOGIN` はデフォルト 1。`COMMENT_POST` はデフォルト 10。他は NULL（無制限）
- ADMIN はシステムルールの `points` と `daily_limit` を変更可能。`is_active = FALSE` で無効化も可能。削除は不可（`is_system = TRUE` の場合）
- 論理削除: カスタムルール（`is_system = FALSE`）のみ削除可能
- `version`: 楽観的ロック。ADMIN が同時にルールを変更した場合の競合を検知

#### `point_transactions`

ポイント付与/リセットのトランザクションログ。ポイント残高は集計で算出（非正規化カラムは持たない）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE）。ポイント対象者 |
| `point_rule_id` | BIGINT UNSIGNED | YES | NULL | FK → point_rules（ON DELETE SET NULL）。ルール削除後もトランザクション履歴は残す |
| `transaction_type` | VARCHAR(20) | NO | — | トランザクション種別（`EARN` / `RESET` / `ADMIN_ADJUST`） |
| `points` | INT | NO | — | ポイント増減値。`EARN`: 正の値。`RESET`: 負の値（累計を相殺）。`ADMIN_ADJUST`: 正/負どちらも可 |
| `action_type` | VARCHAR(50) | NO | — | 発生元アクション種別（`point_rules.action_type` と同値。`RESET` / `ADMIN_ADJUST` の場合はそれぞれ固定値） |
| `reference_type` | VARCHAR(30) | YES | NULL | 参照元エンティティ種別（`TIMELINE_POST` / `ACTIVITY_RESULT` / `SCHEDULE_ATTENDANCE` / `CHAT_MESSAGE` / `LOGIN` 等） |
| `reference_id` | BIGINT UNSIGNED | YES | NULL | 参照元エンティティ ID。ログイン等の場合は NULL |
| `description` | VARCHAR(200) | YES | NULL | 補足説明（ADMIN_ADJUST 時の理由等） |
| `earned_on` | DATE | NO | — | ポイント付与日（JST。日次集計・daily_limit 判定に使用） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_pt_user_scope (user_id, scope_type, scope_id, earned_on DESC)                 -- ユーザー別ポイント履歴
INDEX idx_pt_scope_date (scope_type, scope_id, earned_on DESC)                            -- スコープ別日次集計
INDEX idx_pt_daily_limit (user_id, scope_type, scope_id, action_type, earned_on)         -- daily_limit 判定用
INDEX idx_pt_ranking (scope_type, scope_id, user_id, transaction_type, points)            -- ランキング集計用
```

**制約・備考**
- ログ系テーブルのため `BaseEntity` を継承しない。`updated_at` なし（INSERT only）
- ポイント残高 = `SUM(points) WHERE scope_type = ? AND scope_id = ? AND user_id = ?`
- 期間別ポイント = `SUM(points) WHERE earned_on BETWEEN ? AND ?`
- `transaction_type`:
  - `EARN`: 通常のポイント付与（イベントリスナー経由）
  - `RESET`: 年次リセット（バッチ処理。現在の累計を負の値で打ち消す）
  - `ADMIN_ADJUST`: 管理者による手動調整（加算/減算）
- `reference_type` + `reference_id`: ポイントの発生元を追跡。同一参照元への二重付与防止にも使用（UNIQUE 制約ではなくアプリ層で検証。削除→再作成のケースがあるため）
- `earned_on`: `DATE` 型。日次集計と `daily_limit` 判定の効率化のため日時ではなく日付で保持
- データ保持期間: 2年。バッチで `earned_on` が2年超のレコードを物理削除（集計済みスナップショットは `ranking_snapshots` に保持）

#### `badges`

バッジ定義。システムデフォルトバッジ（`is_system = TRUE`）と ADMIN カスタムバッジ（`is_system = FALSE`）の2種類。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `name` | VARCHAR(100) | NO | — | バッジ名（例: 「皆勤賞」「MVP」「投稿マスター」） |
| `description` | VARCHAR(500) | YES | NULL | バッジの説明 |
| `icon_key` | VARCHAR(500) | YES | NULL | アイコン画像の S3 オブジェクトキー（カスタムバッジの場合）。NULL の場合はデフォルトアイコンを使用 |
| `icon_emoji` | VARCHAR(10) | YES | NULL | 絵文字アイコン（システムバッジのデフォルト。例: `🏆`, `🌟`, `📝`） |
| `badge_type` | VARCHAR(30) | NO | — | バッジ種別（`PERFECT_ATTENDANCE` / `MVP` / `POST_MASTER` / `STREAK` / `CUSTOM`） |
| `condition_type` | VARCHAR(30) | NO | — | 条件種別（`ATTENDANCE_RATE` / `MONTHLY_RANK` / `CUMULATIVE_COUNT` / `CONSECUTIVE_DAYS` / `MANUAL`） |
| `condition_value` | INT | NO | — | 条件の閾値（例: 出欠率 100、ランキング 1位、投稿累計 100件、連続ログイン 30日） |
| `condition_period` | VARCHAR(20) | YES | NULL | 条件の評価期間（`WEEKLY` / `MONTHLY` / `YEARLY` / `ALL_TIME`）。`MANUAL` の場合は NULL |
| `is_system` | BOOLEAN | NO | FALSE | システムデフォルトバッジか |
| `is_active` | BOOLEAN | NO | TRUE | バッジの有効/無効 |
| `is_repeatable` | BOOLEAN | NO | FALSE | 複数回獲得可能か（例: 月間 MVP は毎月獲得可能。TRUE の場合、同一期間内の重複は不可） |
| `sort_order` | INT | NO | 0 | 表示順 |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）。作成者 |
| `version` | BIGINT | NO | 0 | 楽観的ロック |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_badges_scope (scope_type, scope_id, is_active, sort_order)  -- スコープ別バッジ一覧
```

**制約・備考**
- システムデフォルトバッジはゲーミフィケーション有効化時に自動作成:
  - `PERFECT_ATTENDANCE`（皆勤賞）: `condition_type = ATTENDANCE_RATE`, `condition_value = 100`, `condition_period = MONTHLY`。月間出欠回答率 100%。`is_repeatable = TRUE`
  - `MVP`: `condition_type = MONTHLY_RANK`, `condition_value = 1`, `condition_period = MONTHLY`。月間ポイント 1 位。`is_repeatable = TRUE`
  - `POST_MASTER`（投稿マスター）: `condition_type = CUMULATIVE_COUNT`, `condition_value = 100`, `condition_period = ALL_TIME`。タイムライン投稿累計 100 件。`is_repeatable = FALSE`
- `condition_type` の評価方法:
  - `ATTENDANCE_RATE`: `schedule_attendances` から期間内の回答率を算出
  - `MONTHLY_RANK`: `point_transactions` から期間内のポイント合計をランキングし、順位が `condition_value` 以内
  - `CUMULATIVE_COUNT`: `point_transactions` の `action_type` 別トランザクション件数が `condition_value` 以上
  - `CONSECUTIVE_DAYS`: `point_transactions` の `earned_on` から連続日数を算出（`DAILY_LOGIN` アクション対象）
  - `MANUAL`: ADMIN が手動でバッジを付与。バッチ判定対象外
- カスタムバッジの `badge_type = CUSTOM` は複数作成可能
- 1スコープあたり最大 **50 バッジ**（システム + カスタム合計）
- `icon_key`: カスタムアイコンのアップロードは Pre-signed URL フロー（.claudecode.md §26）に従う。許可 MIME タイプ: `image/png`, `image/webp`。最大 500KB
- 論理削除: カスタムバッジのみ削除可能。獲得済みバッジは `user_badges` に残る

#### `user_badges`

ユーザーが獲得したバッジの記録。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `badge_id` | BIGINT UNSIGNED | NO | — | FK → badges（ON DELETE CASCADE） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE）。バッジ獲得者 |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（非正規化。badges テーブルとの JOIN なしでスコープ別取得を効率化） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID（非正規化） |
| `earned_on` | DATE | NO | — | 獲得日（JST） |
| `period_label` | VARCHAR(20) | YES | NULL | 獲得対象期間のラベル（例: `2026-03`（月次）, `2026`（年次））。`is_repeatable = TRUE` のバッジの重複判定に使用 |
| `awarded_by` | VARCHAR(20) | NO | 'SYSTEM' | 付与元（`SYSTEM` / `ADMIN`）。ADMIN による手動付与の場合は `ADMIN` |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_ub_user_scope (user_id, scope_type, scope_id, earned_on DESC)          -- ユーザー別バッジ一覧
INDEX idx_ub_badge (badge_id, earned_on DESC)                                      -- バッジ別の獲得者一覧
UNIQUE KEY uq_ub_badge_user_period (badge_id, user_id, period_label)              -- 同一期間の重複獲得防止
```

**制約・備考**
- ログ系テーブルのため `updated_at` なし（INSERT only）
- `period_label`: `is_repeatable = TRUE` のバッジで使用。月次バッジは `YYYY-MM` 形式、年次バッジは `YYYY` 形式、`ALL_TIME` バッジは `ALL_TIME` 固定値。`is_repeatable = FALSE` の場合は `ALL_TIME`
- `uq_ub_badge_user_period`: 同一ユーザーが同一期間に同じバッジを重複獲得することを防止
- バッジ獲得時に `notifications` テーブルに通知レコードを INSERT（`notification_type = 'BADGE_EARNED'`）

#### `ranking_snapshots`

ランキングの日次スナップショット。Redis Sorted Set でリアルタイムランキングを提供し、日次バッチで DB にスナップショットを保存。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `period_type` | VARCHAR(20) | NO | — | 期間種別（`WEEKLY` / `MONTHLY` / `YEARLY`） |
| `period_label` | VARCHAR(20) | NO | — | 期間ラベル（`2026-W12`（週次）/ `2026-03`（月次）/ `2026`（年次）） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE） |
| `total_points` | INT | NO | — | 期間内の合計ポイント |
| `rank_position` | INT | NO | — | 順位（1 始まり。同率の場合は同一順位） |
| `snapshot_on` | DATE | NO | — | スナップショット取得日 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | 日次バッチで INSERT ... ON DUPLICATE KEY UPDATE 時に更新 |

**インデックス**
```sql
INDEX idx_rs_scope_period (scope_type, scope_id, period_type, period_label, rank_position)  -- スコープ・期間別ランキング表示
INDEX idx_rs_user (user_id, scope_type, scope_id, period_type, snapshot_on DESC)             -- ユーザー別ランキング推移
UNIQUE KEY uq_rs_snapshot (scope_type, scope_id, period_type, period_label, user_id)         -- 同一期間・ユーザーの重複防止
```

**制約・備考**
- 日次バッチで `INSERT ... ON DUPLICATE KEY UPDATE` により期間ごとに最新化されるため、`updated_at` を保持する
- `period_label` フォーマット:
  - `WEEKLY`: ISO 8601 週番号（`YYYY-Www`。例: `2026-W12`）
  - `MONTHLY`: `YYYY-MM`（例: `2026-03`）
  - `YEARLY`: `YYYY`（例: `2026`）
- 日次バッチ（03:30 JST）で `point_transactions` を集計し、週次/月次/年次のスナップショットを INSERT ... ON DUPLICATE KEY UPDATE で更新
- データ保持期間: 3年。バッチで `snapshot_on` が3年超のレコードを物理削除
- `rank_position`: 同率の場合は同一順位を付与（例: 1位, 1位, 3位）。ユーザー ID の昇順でタイブレーク（表示順のみ。順位は同一）

| カラム名 | 型 | NULL | デフォルト | 説明 |

| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE） |

**インデックス**

**制約・備考**
- 初回アクセス時に Service 層でデフォルト値で自動作成（Lazy initialization）
- `show_in_ranking = FALSE` の場合:

## 5. ビジネスロジック

### 主要フロー

#### ポイント自動付与フロー

```
1. 各機能がアクション完了時に DomainEvent を発行
   - TimelinePostCreatedEvent（F04.1）
   - ActivityParticipantAddedEvent（F06.4）
   - ScheduleAttendanceRespondedEvent（F03.1）
   - CommentCreatedEvent（各機能共通）
   - UserDailyLoginEvent（F01.1 認証）

2. GamificationPointListener が @TransactionalEventListener(phase = AFTER_COMMIT) で購読
   └── @Async("event-pool") で非同期実行

3. ポイント付与判定:
   a. ゲーミフィケーション有効チェック（gamification_configs.is_enabled）
   b. スコープの point_rules から該当アクションのルールを取得
   c. ルールが is_active = TRUE かチェック
   d. daily_limit チェック（point_transactions の当日件数を確認）
   e. 二重付与チェック（同一 reference_type + reference_id の既存トランザクション確認）

4. 判定 OK の場合:
   a. point_transactions に INSERT
   b. Redis Sorted Set を ZINCRBY で更新:
      - mannschaft:gamification:ranking:{scopeType}:{scopeId}:weekly:{periodLabel}
      - mannschaft:gamification:ranking:{scopeType}:{scopeId}:monthly:{periodLabel}
      - mannschaft:gamification:ranking:{scopeType}:{scopeId}:yearly:{periodLabel}
   c. ログ出力（INFO レベル。userId, points, actionType）

5. 判定 NG の場合:
   - daily_limit 超過 / 二重付与 → スキップ（WARN ログ出力）
   - ゲーミフィケーション無効 / ルール無効 → スキップ（DEBUG ログ出力）
```

#### バッジ自動判定フロー（日次バッチ）

```
1. 日次バッチ（03:00 JST）が起動
   - @Scheduled(cron = "0 0 3 * * *")
   - スレッドプール: @Async("job-pool")

2. gamification_configs から is_enabled = TRUE の全スコープを取得

3. 各スコープの badges（is_active = TRUE, condition_type != MANUAL）を取得

4. 各バッジの条件を評価:
   a. ATTENDANCE_RATE:
      - 対象期間の schedule_attendances を集計
      - 回答率 = 回答済み件数 / 対象スケジュール件数 × 100
      - condition_value（100）以上 → バッジ付与
   b. MONTHLY_RANK:
      - point_transactions の対象期間ポイント合計でランキング算出
      - 順位が condition_value 以内 → バッジ付与
   c. CUMULATIVE_COUNT:
      - point_transactions の該当 action_type の件数を集計
      - condition_value 以上 → バッジ付与
   d. CONSECUTIVE_DAYS:
      - point_transactions の DAILY_LOGIN の earned_on から連続日数を算出
      - condition_value 以上 → バッジ付与

5. 新規獲得バッジの場合:
   a. user_badges に INSERT（period_label による重複チェック済み）
   b. notifications に通知レコードを INSERT
   c. WebSocket でリアルタイム通知を送信
   d. ログ出力（INFO レベル。userId, badgeName）

6. バッチ完了後、処理件数をログ出力（INFO レベル）
```

#### ランキング集計フロー（日次バッチ）

```
1. 日次バッチ（03:30 JST）が起動（バッジ判定完了後）
   - @Scheduled(cron = "0 30 3 * * *")

2. gamification_configs から is_enabled = TRUE かつ is_ranking_enabled = TRUE の全スコープを取得

3. 各スコープ × 各期間種別（WEEKLY / MONTHLY / YEARLY）で:
   a. point_transactions から期間内のユーザー別合計ポイントを算出
   b. 合計ポイントの降順で順位を付与（同率は同一順位）
   c. ranking_snapshots に INSERT ... ON DUPLICATE KEY UPDATE でスナップショットを保存

4. Redis Sorted Set を point_transactions から再構築（DB が正。Redis は高速参照用キャッシュ）
   - 既存キーを DEL → ZADD で一括再構築
   - TTL: 48時間（次回バッチで更新されるため、バッチ失敗時の安全マージン）

5. バッチ完了ログ出力
```

#### 年次ポイントリセットフロー

```
1. 日次バッチ（04:00 JST）で gamification_configs.point_reset_month を確認
   - 当月 = point_reset_month かつ 日 = 1 の場合にリセット実行

2. 対象スコープの全ユーザーの現在ポイント残高を算出

3. 各ユーザーの残高を相殺する RESET トランザクションを INSERT:
   - transaction_type = 'RESET'
   - points = -(現在残高)
   - action_type = 'RESET'

4. Redis Sorted Set をクリア（DEL）

5. 完了ログ出力（リセット対象ユーザー数、合計リセットポイント数）
```

### 重要な判定ロジック

- **daily_limit 判定**: `SELECT COUNT(*) FROM point_transactions WHERE user_id = ? AND scope_type = ? AND scope_id = ? AND action_type = ? AND earned_on = CURDATE()` が `point_rules.daily_limit` 未満であること
- **二重付与防止**: `SELECT COUNT(*) FROM point_transactions WHERE user_id = ? AND scope_type = ? AND scope_id = ? AND reference_type = ? AND reference_id = ?` が 0 であること
- **バッジ重複獲得防止**: `user_badges` の `uq_ub_badge_user_period` ユニーク制約で DB レベルで保証。`INSERT IGNORE` でアプリ層の例外を回避
- **ランキング非公開ユーザーの扱い**: ランキング API レスポンス構築時に `gamification_user_settings.show_in_ranking = FALSE` のユーザーを除外。順位は全ユーザーで算出（非公開ユーザーも含む）し、表示だけ除外する（順位の飛びが発生する）
- **SUPPORTER の扱い**: ポイント付与対象だがランキングには表示しない（`ranking_snapshots` に含めない）。バッジは獲得可能

## 6. セキュリティ考慮事項

- **認可チェック**: `GamificationService` の入り口でスコープ（`teamId` / `orgId`）と `currentUser` の所属を必ず検証する。ADMIN 専用エンドポイントは `GAMIFICATION_MANAGE` パーミッションを追加チェック
- **レートリミット**: `POST /points/adjust` および `POST /points/reset` に `Bucket4j` で 1分間に 10 回の制限を適用（ADMIN による誤操作・連打防止）
- **ポイント改ざん防止**: ポイント付与は `ApplicationEvent` リスナー経由のみ。直接 INSERT する API は `ADMIN_ADJUST` のみで、操作ログ（`audit_logs`）に記録
- **ADMIN_ADJUST の監査**: `POST /points/adjust` は必ず `description`（理由）を必須とし、`audit_logs` に操作者・対象者・ポイント増減値・理由を記録
- **バッジアイコンアップロード**: Pre-signed URL フロー準拠。MIME タイプ `image/png`, `image/webp` のみ許可。最大 500KB。ファイルキーは UUID ベース（推測不可能）

## 8. 未解決事項

設計判断はすべて解決済み。

- ~~ポイント残高を非正規化カラムで持つか、トランザクションの SUM で都度算出するか~~ → **SUM で都度算出する方式を採用**。理由: ポイント残高のリアルタイム正確性を保証でき、リセットや調整時の整合性維持が容易。ランキングは Redis Sorted Set でキャッシュするため、DB の SUM クエリは個人のポイント履歴画面でのみ使用し、パフォーマンスへの影響は軽微
- ~~バッジ判定をリアルタイム（イベント駆動）にするか、日次バッチにするか~~ → **日次バッチを採用**。理由: バッジ条件は期間集計（月間出欠率、累計件数等）を伴うため、リアルタイム判定はクエリコストが高い。日次バッチで一括評価し、翌日までに反映されれば UX として十分
- ~~ランキングの同率処理をどうするか~~ → **同一順位方式を採用**（例: 1位, 1位, 3位）。理由: スポーツの順位方式として直感的。表示上のタイブレークはユーザー ID 昇順
- ~~ポイントのインフレ防止策をどうするか~~ → **ADMIN によるポイントルール調整 + オプション年次リセットで対応**。自動調整（デフレーション）は導入しない。理由: 自動調整はメンバーの混乱を招く。ADMIN が組織の状況に応じて手動で調整するのが適切
- ~~SUPPORTER のランキング表示をどうするか~~ → **ランキングには表示しない**。理由: SUPPORTER は補助的な役割のため、正規メンバーのモチベーション指標には含めない。ポイント・バッジは閲覧可能
