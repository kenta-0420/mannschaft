# F09.2: プロモーション配信（ターゲット通知・クーポン）

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 9
> **最終更新**: 2026-03-13

---

## 1. 概要

ADMIN や事業者（整骨院・美容院・飲食店等）が、郵便番号・ロール・チーム属性などの条件でセグメントを定義し、メンバーやサポーターにプッシュ通知型のプロモーション（クーポン・販促情報）を配信する機能。通知基盤（F04.3）の `notification_type = PROMOTION` として既存チャネル（IN_APP / Push / LINE）で配信し、クーポンの作成・配布・利用追跡・有効期限管理を一元的に行う。プラットフォームの収益化手段として、配信回数に応じた従量課金モデルを採用する。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | プロモーション課金設定の管理、全プロモーションの監査・停止、課金状況一覧 |
| ADMIN | プロモーション・クーポンの作成・配信・効果測定、セグメント管理 |
| DEPUTY_ADMIN | `SEND_PROMOTION` 権限を持つ場合: プロモーション作成・配信（大規模配信 1000人以上は ADMIN 承認が必要） |
| MEMBER | 受信したプロモーション閲覧、クーポン利用 |
| SUPPORTER | 受信したプロモーション閲覧、クーポン利用 |
| GUEST | プロモーション受信対象外 |

### 対象レベル
- [x] 組織 (Organization) — 組織配下の全チームメンバーへの配信
- [x] チーム (Team) — チームメンバー・サポーターへの配信
- [ ] 個人 (Personal) — 対象外（個人間のプロモーションは想定しない）

### テーブル定義

#### `promotions`

プロモーション（販促キャンペーン）の定義。1プロモーションにつき1回の配信（1プロモーション = 1配信）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION') | NO | — | 配信スコープ |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ対象の ID |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users（作成者） |
| `title` | VARCHAR(200) | NO | — | プロモーションタイトル |
| `body` | TEXT | NO | — | 本文（Markdown サポート） |
| `image_url` | VARCHAR(500) | YES | NULL | アイキャッチ画像の S3 オブジェクトキー（API レスポンス時に Pre-signed URL を動的生成） |
| `coupon_id` | BIGINT UNSIGNED | YES | NULL | FK → coupons（紐付けクーポン、nullable） |
| `status` | ENUM('DRAFT', 'PENDING_APPROVAL', 'SCHEDULED', 'PUBLISHED', 'COMPLETED', 'CANCELLED') | NO | 'DRAFT' | 配信ステータス |
| `approved_by` | BIGINT UNSIGNED | YES | NULL | FK → users（承認した ADMIN。DEPUTY_ADMIN 作成時のみ使用） |
| `approved_at` | DATETIME | YES | NULL | 承認日時 |
| `scheduled_at` | DATETIME | YES | NULL | 予約配信日時（SCHEDULED 時のみ使用） |
| `published_at` | DATETIME | YES | NULL | 実際の配信日時 |
| `expires_at` | DATETIME | YES | NULL | プロモーション有効期限（期限後は通知一覧で「期限切れ」表示） |
| `target_count` | INT UNSIGNED | YES | NULL | 配信時のセグメント対象ユーザー数（配信実行時にスナップショット） |
| `delivered_count` | INT UNSIGNED | NO | 0 | 実際の配信完了数 |
| `opened_count` | INT UNSIGNED | NO | 0 | 開封数（denormalize、配信完了後にアトミック更新） |
| `skipped_count` | INT UNSIGNED | NO | 0 | 配信スキップ数（日次受信上限等で除外されたユーザー数） |
| `failed_count` | INT UNSIGNED | NO | 0 | 配信失敗数（リトライ上限超過でFAILEDとなったユーザー数） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_promo_scope (scope_type, scope_id, status)   -- スコープ別プロモーション一覧
INDEX idx_promo_status_scheduled (status, scheduled_at)  -- 予約配信バッチのスキャン用
INDEX idx_promo_created_by (created_by)                  -- 作成者別一覧
```

**制約・備考**
- 論理削除: `deleted_at DATETIME nullable`（SoftDeletableEntity 適用）
- `status` 遷移: DRAFT → SCHEDULED → PUBLISHED → COMPLETED / DRAFT → PUBLISHED → COMPLETED / DRAFT → PENDING_APPROVAL → DRAFT（差し戻し） or SCHEDULED/PUBLISHED（承認後）/ いずれの状態からも CANCELLED へ遷移可
- **承認ワークフロー**: DEPUTY_ADMIN が作成したプロモーションで `target_count >= 1000`（大規模配信）の場合、`publish` 時に自動で `PENDING_APPROVAL` に遷移。ADMIN が承認すると SCHEDULED or PUBLISHED へ進行。ADMIN 自身が作成した場合は承認不要で直接配信可能
- `COMPLETED` は `expires_at` 到達時にバッチで自動設定、または ADMIN が手動設定
- `target_count` / `delivered_count` / `opened_count` / `skipped_count` / `failed_count` はダッシュボードの効果測定に使用
- `scope_id` は `teams.id` または `organizations.id` へのポリモーフィック参照（ENUM で分岐するため FK 制約は定義しない）
- **スコープ削除時**: チーム/組織が削除された場合、配下の `promotions`（DRAFT / SCHEDULED）は自動 CANCELLED に遷移。PUBLISHED / COMPLETED の集計データは保持（課金データの整合性維持のため物理削除しない）。`coupons` は `is_active = false` に自動設定。処理は ApplicationEvent（`TeamDeletedEvent` / `OrganizationDeletedEvent`）で非同期実行

#### `promotion_segments`

プロモーションのターゲットセグメント条件。1プロモーションに複数のセグメント条件を設定可能（AND 結合）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `promotion_id` | BIGINT UNSIGNED | NO | — | FK → promotions（ON DELETE CASCADE） |
| `segment_type` | ENUM('POSTAL_CODE', 'ROLE', 'TEAM', 'GROUP', 'AGE_RANGE', 'GENDER') | NO | — | セグメント条件種別 |
| `segment_value` | VARCHAR(500) | NO | — | 条件値（JSON 配列形式で複数値を格納） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_ps_promo_type (promotion_id, segment_type) -- 同一プロモーション内で segment_type は1つのみ（値は JSON 配列で複数格納）
INDEX idx_ps_promotion (promotion_id)                     -- プロモーション別セグメント一覧
```

**制約・備考**
- `segment_value` の形式（`segment_type` ごと）:
  - `POSTAL_CODE`: `["100-0001", "100-0002", "150-*"]` — 前方一致ワイルドカード対応（`150-*` = 150-xxxx 全域）
  - `ROLE`: `["MEMBER", "SUPPORTER"]` — 対象ロール
  - `TEAM`: `["123", "456"]` — チーム ID（組織スコープ時に特定チームに絞り込む場合）
  - `GROUP`: `["789"]` — グループ ID（グループ階層内の特定グループに絞り込む場合。TEAM/ORGANIZATION 両スコープで使用可）
  - `AGE_RANGE`: `{"min": 20, "max": 40}` — 年齢範囲（`users.date_of_birth` から配信実行時点で動的算出。未入力ユーザーは対象外）
  - `GENDER`: `["MALE", "FEMALE", "OTHER", "PREFER_NOT_TO_SAY"]` — 性別（`users.gender` を参照。未入力ユーザーは対象外）
- 同一 `promotion_id` 内の複数セグメント条件は **AND 結合** で評価
- 同一 `segment_type` 内の複数値は **OR 結合** で評価
  - 例: `POSTAL_CODE: ["100-*", "150-*"]` AND `ROLE: ["MEMBER"]` → 「郵便番号が100-xxxxまたは150-xxxxのMEMBER」
- `POSTAL_CODE` は `users` テーブルの `postal_code`（VARCHAR(10) nullable、Phase 1 で追加・任意入力）を参照。住所未登録ユーザーは郵便番号セグメントの対象外
- `AGE_RANGE` は `users.date_of_birth`（DATE nullable、Phase 1 で追加・任意入力）から配信実行時点で動的算出。未入力ユーザーは年齢セグメントの対象外
- `GENDER` は `users.gender`（ENUM: MALE/FEMALE/OTHER/PREFER_NOT_TO_SAY、nullable、Phase 1 で追加・任意入力）を参照。未入力ユーザーは性別セグメントの対象外
- **属性未入力ユーザーの扱い**: セグメント条件に該当する属性が未入力のユーザーはそのセグメントの対象外となる。プレビュー API のレスポンスに「属性未登録: N名（対象外）」を表示し、ADMIN に可視化する
- **スコープ別の有効セグメント種別**:
  - `scope_type = TEAM` の場合: `TEAM` セグメントは無効（既にチームに絞り込み済みのため）。設定しようとした場合は 400 エラー
  - `scope_type = ORGANIZATION` の場合: 全セグメント種別が使用可能。`TEAM` セグメントで組織内の特定チームに絞り込める

#### `promotion_deliveries`

ユーザーごとの配信実績。F04.3 の `notifications` テーブルとは別に、プロモーション固有の効果測定データを管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `promotion_id` | BIGINT UNSIGNED | NO | — | FK → promotions |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（配信先ユーザー） |
| `notification_id` | BIGINT UNSIGNED | YES | NULL | FK → notifications（F04.3 の通知レコード; ON DELETE SET NULL — notifications は既読90日後に物理削除されるため） |
| `delivery_status` | ENUM('SUCCESS', 'FAILED', 'RETRYING') | NO | 'SUCCESS' | 配信ステータス |
| `retry_count` | TINYINT UNSIGNED | NO | 0 | リトライ回数（最大3回） |
| `delivered_at` | DATETIME | NO | CURRENT_TIMESTAMP | 配信日時 |
| `opened_at` | DATETIME | YES | NULL | 開封日時 |
| `clicked_at` | DATETIME | YES | NULL | 詳細リンク・クーポンクリック日時 |
| `channels_sent` | JSON | YES | NULL | 配信チャネル記録（例: `["IN_APP", "PUSH"]`） |

**インデックス**
```sql
UNIQUE KEY uq_pd_promo_user (promotion_id, user_id)   -- 同一ユーザーへの重複配信防止
INDEX idx_pd_user (user_id, delivered_at)               -- ユーザー別配信履歴
INDEX idx_pd_delivered (delivered_at)                    -- 180日バッチ削除用
INDEX idx_pd_retry (delivery_status, retry_count)        -- リトライ対象レコード検索用
```

**制約・備考**
- 180日経過後にバッチで物理削除（`promotions` の集計カラムおよび `promotion_delivery_summaries` に永続化済みのため生データは不要）
- `notification_id` で F04.3 の通知レコードと紐付け。通知の既読・開封は `notifications.is_read` で管理し、`opened_at` は通知開封イベント時に本テーブルも同期更新
- **配信失敗リトライ**: F04.3 への配信リクエストが失敗した場合、`delivery_status = RETRYING` で記録し、最大3回リトライ（exponential backoff: 30秒 → 2分 → 10分）。全リトライ失敗時は `delivery_status = FAILED` に遷移し、`promotions.failed_count` をアトミック更新。Spring `@Scheduled(fixedRate = 30000)` の配信リトライバッチが `delivery_status = RETRYING` のレコードを処理

#### `coupons`

クーポンの定義。プロモーションに紐付けて配布、またはスタンドアロンで配布可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION') | NO | — | 発行スコープ |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ対象の ID |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users（作成者） |
| `code` | VARCHAR(20) COLLATE utf8mb4_general_ci | NO | — | クーポンコード（英数字、大文字正規化して保存） |
| `title` | VARCHAR(200) | NO | — | クーポン名（例: 「初回施術20%OFF」） |
| `description` | TEXT | YES | NULL | クーポン説明 |
| `discount_type` | ENUM('PERCENTAGE', 'FIXED_AMOUNT', 'FREE_ITEM') | NO | — | 割引種別 |
| `discount_value` | DECIMAL(10,2) | NO | — | 割引値（PERCENTAGE: 10.00 = 10%、FIXED_AMOUNT: 500.00 = 500円） |
| `min_purchase_amount` | DECIMAL(10,2) | YES | NULL | 最低利用金額（null = 条件なし） |
| `max_discount_amount` | DECIMAL(10,2) | YES | NULL | 割引上限額（PERCENTAGE 時の上限。null = 上限なし） |
| `max_total_uses` | INT UNSIGNED | YES | NULL | 全体の利用上限数（null = 無制限） |
| `max_uses_per_user` | INT UNSIGNED | NO | 1 | ユーザーあたりの利用上限数 |
| `total_used_count` | INT UNSIGNED | NO | 0 | 全体の利用済み数（denormalize） |
| `valid_from` | DATETIME | NO | — | 利用可能開始日時 |
| `valid_until` | DATETIME | YES | NULL | 利用可能終了日時（null = 無期限） |
| `is_active` | BOOLEAN | NO | TRUE | 有効/無効フラグ（手動無効化用） |
| `is_combinable` | BOOLEAN | NO | TRUE | 他クーポンとの併用可否（FALSE = 同一スコープ内で他クーポンと同時利用不可） |
| `reminder_days_before` | INT UNSIGNED | YES | NULL | 有効期限 N 日前にリマインド通知（null = リマインドなし。`valid_until` 設定時のみ有効） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除（SoftDeletableEntity 適用） |

**インデックス**
```sql
UNIQUE KEY uq_coupon_code (code)                         -- クーポンコード一意（論理削除時は code を '{code}_deleted_{id}' に書き換え）
INDEX idx_coupon_scope (scope_type, scope_id, is_active) -- スコープ別有効クーポン一覧
INDEX idx_coupon_valid (valid_from, valid_until)          -- 有効期間フィルタ
```

**制約・備考**
- `code` は英数字 4〜20 文字。ADMIN が手動入力、または自動生成（`SecureRandom` + Base36、8桁）を選択可能。**大文字正規化**: INSERT 時に `UPPER()` 変換して保存。検索・手入力時も `UPPER()` で比較するため、ユーザーが `spring2026` と入力しても `SPRING2026` にマッチする。COLLATION は `utf8mb4_general_ci`（case-insensitive）
- `discount_type = FREE_ITEM` の場合、`discount_value` は無視（0.00 を格納）し、`description` にフリーアイテムの内容を記載
- `max_total_uses` 到達時は自動的に `is_active = false` に設定（Service 層で制御）
- `discount_type = PERCENTAGE` の場合、`discount_value` は 0.01〜100.00 の範囲でバリデーション。100超は 400 エラー
- `valid_until = NULL`（無期限クーポン）の場合、`is_active` での手動無効化に依存する。棚卸し忘れ防止のため、ADMIN ダッシュボードに「有効期限なしの有効クーポン一覧」ウィジェットを表示する
- **併用ルール**: `is_combinable = FALSE` のクーポンは、同一スコープ内で他のクーポンと同時に利用できない。`redeem` 時に「同一ユーザーが同日・同スコープで他の `is_combinable = FALSE` クーポンを利用済みか」をバリデーション。デフォルトは `TRUE`（併用可）で、ADMIN がクーポン作成時に設定
- **期限切れリマインダー**: `reminder_days_before` が設定されている場合、日次バッチ（毎日 AM 9:00 JST）で `valid_until - reminder_days_before <= NOW() AND valid_until > NOW()` のクーポンを持つユーザーに通知を送信。`notification_type = COUPON_EXPIRY_REMINDER`（F04.3 に追加）。リマインドは1クーポンにつき1回のみ送信（`coupon_distributions` に `reminder_sent_at DATETIME nullable` を追加して重複防止）
- 論理削除: `deleted_at DATETIME nullable`（SoftDeletableEntity 適用）。削除時に `code` を `{code}_deleted_{id}` にリネームし、UNIQUE 制約を解放する（同一コードでの再作成を可能にする）

#### `coupon_distributions`

ユーザーへのクーポン配布記録。プロモーション配信時に対象ユーザーへ自動配布、または ADMIN が手動配布。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `coupon_id` | BIGINT UNSIGNED | NO | — | FK → coupons |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users |
| `promotion_id` | BIGINT UNSIGNED | YES | NULL | FK → promotions（プロモーション経由の場合） |
| `distributed_at` | DATETIME | NO | CURRENT_TIMESTAMP | 配布日時 |
| `used_count` | INT UNSIGNED | NO | 0 | このユーザーの利用回数 |
| `reminder_sent_at` | DATETIME | YES | NULL | リマインド通知送信日時（重複送信防止用） |

**インデックス**
```sql
UNIQUE KEY uq_cd_coupon_user (coupon_id, user_id)   -- 同一ユーザーへの重複配布防止
INDEX idx_cd_user (user_id)                          -- ユーザー別クーポン一覧
INDEX idx_cd_promotion (promotion_id)                -- プロモーション別配布一覧
```

**制約・備考**
- `used_count` は `coupon_redemptions` INSERT 時にアトミック更新（`UPDATE coupon_distributions SET used_count = used_count + 1 WHERE id = :id`）。`coupons.total_used_count` も同一トランザクション内で更新
- 日次バッチで `used_count` と `coupon_redemptions` の実数を照合し、ドリフト検出時にログ警告を出力（reconciliation）

#### `promotion_billing_records`

プロモーション配信の課金記録。配信実行ごとに1レコードを生成。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `promotion_id` | BIGINT UNSIGNED | NO | — | FK → promotions |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION') | NO | — | 課金先スコープ（スナップショット） |
| `scope_id` | BIGINT UNSIGNED | NO | — | 課金先スコープ ID |
| `delivered_count` | INT UNSIGNED | NO | — | 課金対象の配信数 |
| `unit_price` | DECIMAL(10,2) | NO | — | 1通あたり単価（配信時点のスナップショット） |
| `subtotal` | DECIMAL(10,2) | NO | — | 税抜額（`delivered_count × unit_price`） |
| `tax_amount` | DECIMAL(10,2) | NO | — | 税額 |
| `total_amount` | DECIMAL(10,2) | NO | — | 税込額 |
| `billing_month` | DATE | NO | — | 課金月（月初日） |
| `is_free_tier` | BOOLEAN | NO | FALSE | 無料枠内の配信か（TRUE = 無料枠、FALSE = 課金対象） |
| `status` | ENUM('PENDING', 'INVOICED', 'PAID') | NO | 'PENDING' | 課金ステータス |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_pbr_promotion (promotion_id)                   -- 1プロモーション = 1課金レコード
INDEX idx_pbr_scope (scope_type, scope_id, billing_month)   -- スコープ別月次課金一覧
INDEX idx_pbr_month_status (billing_month, status)           -- 月次請求バッチ用
INDEX idx_pbr_free_tier (scope_type, scope_id, is_free_tier, billing_month)  -- 無料枠利用状況分析用
```

**制約・備考**
- 月次で `PENDING` → `INVOICED` に一括更新し、`subscription_invoices` と合算して請求書を発行
- 単価は SYSTEM_ADMIN が管理する設定テーブル（後述の外部設定）から取得し、配信時点の値をスナップショットとして記録

#### `saved_segment_presets`

よく使うセグメント条件の組み合わせを名前付きで保存・再利用するためのプリセット。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION') | NO | — | プリセットのスコープ |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ対象の ID |
| `name` | VARCHAR(100) | NO | — | プリセット名（例: 「渋谷エリアのMEMBER」） |
| `conditions` | JSON | NO | — | セグメント条件（`[{"segment_type": "POSTAL_CODE", "segment_value": ["150-*"]}, ...]`） |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users（作成者） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
INDEX idx_ssp_scope (scope_type, scope_id, deleted_at)   -- スコープ別プリセット一覧
```

**制約・備考**
- プリセットはスコープ内の ADMIN+ が共有利用可能（作成者以外も使用可）
- プロモーション作成時に「プリセットから読み込み」ボタンで条件を一括設定。プリセット適用後に個別条件を追加・変更可能（プリセットのコピーであり参照ではない）
- 1スコープあたり最大50件まで（超過時は古いプリセットを削除促進）

#### `promotion_delivery_summaries`

配信実績の日次集約テーブル。`promotion_deliveries` は180日で物理削除されるため、長期トレンド分析用に集約データを永続保持する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `promotion_id` | BIGINT UNSIGNED | NO | — | FK → promotions |
| `summary_date` | DATE | NO | — | 集約対象日 |
| `delivered_count` | INT UNSIGNED | NO | 0 | 当日の配信数 |
| `opened_count` | INT UNSIGNED | NO | 0 | 当日の開封数 |
| `clicked_count` | INT UNSIGNED | NO | 0 | 当日のクリック数 |
| `failed_count` | INT UNSIGNED | NO | 0 | 当日の配信失敗数 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_pds_promo_date (promotion_id, summary_date)  -- 同一プロモーション・同一日の重複防止
INDEX idx_pds_date (summary_date)                            -- 日付範囲検索用
```

**制約・備考**
- 日次バッチ（毎日 AM 2:00 JST）で前日分の `promotion_deliveries` を集約して INSERT/UPDATE（`ON DUPLICATE KEY UPDATE`）
- `promotion_deliveries` の180日物理削除後も本テーブルのデータは永続保持し、年間トレンド分析に使用
- stats API で180日以前のデータを参照する場合、本テーブルにフォールバック

#### `POST /api/v1/coupons/{id}/redeem`

店舗スタッフ（ADMIN）が、メンバーが提示したクーポンの利用処理を行う。提示方法は以下の2通り:
- **メイン: QR コード方式** — `GET /coupons/my` のクーポン詳細画面に QR コードを動的生成して表示。ADMIN がスキャンすると本 API が自動実行される。QR ペイロードは `{"cd_id": <distribution_id>, "coupon_id": <id>, "user_id": <id>, "exp": <unix_timestamp>, "sig": "<hmac_sha256_hex>"}` を Base64URL エンコード。`exp` は QR 表示時刻 + 5分（300秒）の UNIX タイムスタンプで、期限切れの場合は画面リロードで再生成。HMAC-SHA256 署名の対象は `cd_id|coupon_id|user_id|exp` の連結文字列で、改ざんとリプレイの両方を防止する（`member_cards` QR 会員証と同一の署名方式を共用）
- **フォールバック: 手入力方式** — ADMIN の利用処理画面でクーポンコード + ユーザー ID を手入力

**リクエストボディ**
```json
{
  "user_id": 789,
  "note": "整体60分コースに適用"
}
```

**レスポンス（200 OK）**
```json
{
  "data": {
    "redemption_id": 1,
    "coupon_id": 456,
```

## 5. ビジネスロジック

### 主要フロー

#### プロモーション配信フロー
```
1. ADMIN/DEPUTY_ADMIN がプロモーションを作成（DRAFT）
2. セグメント条件を設定（郵便番号・ロール等）。保存済みプリセットからの読み込みも可能
3. プレビューで対象ユーザー数・見積コストを確認
4. 配信実行（即時 or 予約）
   a. DEPUTY_ADMIN 作成 && target_count >= 1000 の場合:
      → status = PENDING_APPROVAL、ADMIN に承認リクエスト通知を送信
      → ADMIN が approve → SCHEDULED or PUBLISHED へ遷移
      → ADMIN が reject → DRAFT に差し戻し（作成者に通知）
   b. 予約の場合: status = SCHEDULED、Spring @Scheduled バッチ（1分間隔、fixedRate = 60000）が scheduled_at <= NOW() のレコードを取得して配信実行
   c. 即時の場合: status = PUBLISHED、非同期で配信開始
5. セグメント条件で対象ユーザーリストを生成
   a. users テーブルから postal_code / date_of_birth / gender 等の属性でフィルタリング
   b. notification_preferences / notification_type_preferences で opt-out ユーザーを除外
   c. archived_at IS NOT NULL のユーザーを除外
   d. GUEST ロールのユーザーを除外
   e. ユーザー日次受信上限（1日3通）に達しているユーザーを除外（Redis カウンター promo:daily:{userId}:{date} で判定）
   f. 属性未入力ユーザーは該当セグメントの対象外（対象外数をプレビューで表示）
6. F04.3 通知基盤に notification_type = PROMOTION として配信リクエスト
   a. IN_APP: 全対象ユーザーに notifications レコードを INSERT
   b. PUSH: priority に応じて Push 通知を送信
   c. LINE: Phase 9 の LINE 連携で配信
   d. 配信失敗時: promotion_deliveries に delivery_status = RETRYING で記録、最大3回リトライ（exponential backoff: 30秒→2分→10分）。全リトライ失敗で FAILED に遷移
7. promotion_deliveries に配信ユーザーごとの配信記録を INSERT（スキップユーザーは INSERT しない。配信失敗ユーザーは delivery_status = RETRYING/FAILED で INSERT）
8. coupon_id が紐付いている場合、配信成功ユーザーの coupon_distributions にクーポン配布記録を INSERT
9. promotion_billing_records に課金レコードを INSERT（is_free_tier = 無料枠内なら TRUE）
10. promotions.delivered_count / skipped_count / failed_count をアトミック更新
11. 日次バッチ（AM 2:00 JST）で promotion_delivery_summaries に前日分を集約 INSERT
```

#### クーポン利用フロー
```
1. メンバーが店舗でクーポンを提示
   - メイン: アプリのクーポン詳細画面で QR コードを表示 → ADMIN がスキャン → redeem API 自動実行
   - フォールバック: ADMIN がクーポンコード + ユーザーID を手入力 → redeem API 実行
2. ADMIN/スタッフが POST /coupons/{id}/redeem を実行
3. バリデーション:
   a. クーポンが有効期間内か
   b. is_active = true か
   c. total_used_count < max_total_uses か
   d. ユーザーの used_count < max_uses_per_user か
   e. ユーザーに coupon_distributions レコードが存在するか
   f. is_combinable = false の場合: 同日・同スコープで他の is_combinable = false クーポンを利用済みでないか
4. coupon_redemptions にレコードを INSERT
5. coupon_distributions.used_count を +1 アトミック更新
6. coupons.total_used_count を +1 アトミック更新
7. total_used_count == max_total_uses の場合、is_active = false に自動設定
8. ユーザーに IN_APP 通知を送信（notification_type = COUPON_REDEEMED）:
   「〇〇クーポンが利用されました（利用場所: △△整骨院、日時: 2026-04-15 14:30）」
   — 不正利用の即時検知にも有用（身に覚えのない利用 → 問い合わせ可能）
```

### 重要な判定ロジック

#### セグメント評価
- **AND 結合**: 異なる `segment_type` 間は AND。`POSTAL_CODE: ["150-*"]` AND `ROLE: ["MEMBER"]` = 郵便番号150-xxxxのMEMBERのみ
- **OR 結合**: 同一 `segment_type` 内の値は OR。`POSTAL_CODE: ["150-*", "151-*"]` = 150-xxxxまたは151-xxxx
- **前方一致**: `POSTAL_CODE` のワイルドカード `*` は SQL の `LIKE 'xxx%'` に変換
- **動的計算**: `AGE_RANGE` は配信実行時点の `users.date_of_birth` から動的算出
- **セグメント未設定**: スコープ内の全対象ユーザー（MEMBER + SUPPORTER）に配信

#### 配信制限
- **月次配信上限（ADMIN 側）**: チーム/組織ごとに月間配信数上限を設定可能（SYSTEM_ADMIN が管理）。デフォルト: 月5回（うち5回は無料枠）。上限到達時は `409 Conflict` を返す
- **日次受信上限（ユーザー側）**: 1ユーザーあたり1日3通まで（SYSTEM_ADMIN が設定変更可能）。超過分は配信スキップし、`promotions.skipped_count` にカウント（`promotion_deliveries` にはスキップ分の行を INSERT しない。翌日への自動繰り越しは行わない — 鮮度が落ちたプロモーションの遅延配信は効果が低いため）。Redis カウンター `promo:daily:{userId}:{date}`（TTL 48h）で軽量管理。ADMIN のプレビュー時に「対象 N名のうち M名は本日の受信上限に達しているため配信されません」と表示
- **1回あたりの配信上限**: 10,000ユーザーまで。超過する場合はバッチ分割して非同期配信
- **配信間隔**: 同一プロモーションの再配信は不可（1プロモーション = 1配信）
- **配信時間帯**: 8:00〜21:00 JST のみ。時間外の即時配信は翌日 8:00 に自動スケジュール
- **予約配信バッチ**: Spring `@Scheduled(fixedRate = 60000)` で1分間隔ポーリング。`status = SCHEDULED AND scheduled_at <= NOW()` のレコードを取得して配信実行（F04.3 のスヌーズバッチと同一の設計方針。最大59秒の遅延を許容）
- **有効期限バッチ**: 日次バッチ（毎日 AM 1:00 JST）で `status = PUBLISHED AND expires_at < NOW()` のレコードを `COMPLETED` に自動遷移。`COMPLETED` 後は stats API での閲覧のみ可能

#### 課金判定
- **デフォルト単価**: 1通1円（税抜）。SYSTEM_ADMIN が `promotion_billing_settings` で変更可能
- **月間無料枠**: チーム/組織ごとに月5回まで無料（5回の配信アクションが無料。対象ユーザー数は問わない）。6回目以降から課金対象
- **課金算出**: 配信実行時に `unit_price × delivered_count` で課金額を算出。無料枠内の配信は `promotion_billing_records` に `is_free_tier = TRUE` + `total_amount = 0` で記録（配信実績は残す）
- **将来の段階料金**: 月間配信数 10,000通以上で 0.8円/通、50,000通以上で 0.5円/通 など、SYSTEM_ADMIN が段階テーブルを設定できるよう将来拡張を想定（Phase 9 初期はフラット単価のみ）
- 月次バッチで `PENDING` → `INVOICED` に一括更新し、`subscription_invoices` と合算
- **月間無料枠の消費判定**: `SELECT COUNT(*) FROM promotion_billing_records WHERE scope_type = :type AND scope_id = :id AND billing_month = :month` で当月の配信回数を取得。5回未満なら `is_free_tier = TRUE` で記録

#### セグメント評価キャッシュ戦略

大規模組織（数千人）でプレビューを繰り返すケースを高速化するため、セグメント評価結果を Redis にキャッシュする。

- **キャッシュキー**: `segment:preview:{sha256(promotion_id + conditions_json)}`
- **TTL**: 5分（ユーザー属性の変更頻度を考慮した許容範囲）
- **キャッシュ対象**: プレビュー API のレスポンス全体（`total_target_count`、`excluded` 内訳、`estimated_cost`）
- **キャッシュ破棄**: セグメント条件の追加・更新・削除時に当該プロモーションの全プレビューキャッシュを削除（`segment:preview:{promotion_id}:*` のパターン削除）
- **配信実行時**: キャッシュは使用せず、常に最新のユーザーデータで再評価する（配信精度を優先）

#### クーポン期限切れリマインダーバッチ

- **実行タイミング**: 毎日 AM 9:00 JST（`@Scheduled(cron = "0 0 9 * * *")`)
- **対象条件**: `coupons.reminder_days_before IS NOT NULL AND coupons.valid_until IS NOT NULL AND coupons.is_active = TRUE AND coupons.valid_until - INTERVAL reminder_days_before DAY <= NOW() AND coupons.valid_until > NOW()`
- **送信対象**: 上記クーポンの `coupon_distributions` で `reminder_sent_at IS NULL AND used_count < max_uses_per_user` のユーザー
- **通知**: `notification_type = COUPON_EXPIRY_REMINDER`（F04.3 に追加）。内容: 「〇〇クーポンの有効期限が△日後に迫っています」
- **重複防止**: 送信完了後に `coupon_distributions.reminder_sent_at = NOW()` を更新

#### プロモーションカレンダービュー

ADMIN ダッシュボードに月間カレンダー形式で SCHEDULED / PUBLISHED / COMPLETED のプロモーションを可視化する。API 追加は不要で、既存の `GET /api/v1/promotions` を `scheduled_at` / `published_at` / `expires_at` の月間レンジでフィルタして取得する。フロントエンドでカレンダー UI にマッピング。配信の重複・空白期間が一目でわかり、ADMIN の配信計画立案を支援する。

### 広告機能（`ads` / `ad_campaigns`）との関係

プロモーション配信と広告機能はビジネスドメインが異なるため、**別系統として維持**する。

| 項目 | 広告（`ads` / `ad_campaigns`） | プロモーション（`promotions`） |
|------|-------------------------------|------------------------------|
| 管理者 | SYSTEM_ADMIN（プラットフォーム広告） | ADMIN / 事業者（自チーム販促） |
| 課金モデル | CPM / CPC（将来） | 配信数従量課金 |
| 対象 | フリープラン全ユーザー | スコープ内 MEMBER / SUPPORTER |
| 審査 | SYSTEM_ADMIN 承認必須 | ADMIN が自律配信 |
| ターゲティング | `ad_targeting_rules`（REGION / ORG_CATEGORY 等） | `promotion_segments`（POSTAL_CODE / ROLE 等） |

**将来のリファクタリング方針**: `ad_targeting_rules.rule_type` と `promotion_segments.segment_type` は概念的に重複する部分がある（REGION ≒ POSTAL_CODE、USER_ROLE ≒ ROLE）。将来的に共通の `SegmentEvaluator` サービスクラスを抽出し、`AdTargetingService` と `PromotionSegmentService` の両方から呼び出す設計にすることで実装の重複を防ぐ。ただしテーブル統合は行わない（ドメインの独立性を維持）

## 6. セキュリティ考慮事項

- **認可チェック**: PromotionService / CouponService の入り口で `scope_type` + `scope_id` と `currentUser` の所属・ロールを必ず検証する。他チーム/組織のプロモーション・クーポンへのアクセスは 403 で拒否
- **レートリミット**: 配信 API（`POST /promotions/{id}/publish`）は `Bucket4j` で1時間に5回まで（チーム/組織単位）。セグメントプレビュー API は1分間に10回まで。クーポン利用 API（`POST /coupons/{id}/redeem`）は1分間に20回まで（ユーザー単位）— クーポンコード総当たり防止
- **入力サニタイズ**: `title`・`body`・`description` は XSS 対策として HTML タグを除去（Markdown のみ許可）。`image_url` は S3 ドメインのみ許可するバリデーション
- **クーポンコード推測防止**: `code` は最低4文字で、連番禁止。自動生成時は `SecureRandom` + Base36 で8桁のランダム文字列を生成
- **QR コード改ざん防止**: クーポン提示用 QR ペイロードに HMAC-SHA256 署名を付与。署名キーはサーバーサイド環境変数で管理。`member_cards`（QR 会員証）と同一の署名方式を共用し、署名検証は共通ユーティリティ（`QrSignatureVerifier`）で一元管理
- **配信対象の個人情報保護**: セグメント条件のプレビューではユーザー数のみ返却し、個人情報（名前・住所等）は返さない。配信実績の閲覧もプロモーション作成者のスコープ内に限定
- **郵便番号の取り扱い**: 郵便番号はプロフィール情報（`users` テーブル）から参照。ユーザーが住所を非公開設定にしている場合でも、セグメント対象としては使用する（配信のフィルタリング目的であり、閲覧はできない）
- **監査ログ**: プロモーション作成・配信・キャンセル、クーポン作成・利用のすべてを `audit_logs` に記録。SYSTEM_ADMIN による強制停止は `PROMOTION_FORCE_SUSPENDED` イベントとして `reason` 付きで記録
