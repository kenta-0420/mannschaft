## 1. 概要

組織・チーム内の知識を階層構造（親ページ → 子ページ）で管理・共有する内部 Wiki 機能。CMS/ブログ（F06.1）が外部向け記事の作成・配信に最適化されているのに対し、ナレッジベースは内部のマニュアル・規約・FAQ・引き継ぎ資料など、メンバーが繰り返し参照する「生きたドキュメント」を体系的に蓄積・維持するために設計されている。

**ユースケース**:
- マンション管理組合: 管理規約・細則・修繕履歴・FAQ
- 企業: 業務マニュアル・社内規程・オンボーディング資料
- スポーツチーム: 戦術ノート・練習メニュー集・大会ルール
- 学校: 教材・カリキュラム・保護者向けガイド

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全組織・チームのナレッジベースを参照・強制削除 |
| ADMIN | ページの作成・編集・削除・移動、テンプレート管理、アクセス制御設定、全ページの閲覧 |
| DEPUTY_ADMIN | `MANAGE_KNOWLEDGE_BASE` 権限を持つ場合: ページの作成・編集・削除・移動。アクセス制御設定は不可 |
| MEMBER | `access_level` が `ALL_MEMBERS` のページの閲覧。`WRITE_KNOWLEDGE_BASE` 権限を持つ場合: ページの作成・自分が作成したページの編集 |
| SUPPORTER | 対象外（内部向け機能のためアクセス不可） |
| GUEST | 対象外（内部向け機能のためアクセス不可） |

### 対象レベル

### テーブル定義

#### `kb_pages`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別。値: `TEAM`, `ORGANIZATION` |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID（scope_type = TEAM → teams.id、ORGANIZATION → organizations.id） |
| `parent_id` | BIGINT UNSIGNED | YES | NULL | FK → kb_pages（自己参照。NULL = ルートページ）。ON DELETE RESTRICT（論理削除はアプリ層で連鎖処理） |
| `path` | VARCHAR(500) | NO | — | マテリアライズドパス（例: `/1/5/12`）。祖先検索を高速化 |
| `depth` | TINYINT UNSIGNED | NO | 0 | 階層の深さ（0 = ルート、1 = 第1子、2 = 第2子...）。UI 上は3階層推奨だがシステム上限は10 |
| `title` | VARCHAR(200) | NO | — | ページタイトル |
| `slug` | VARCHAR(200) | NO | — | URL スラッグ（スコープ内で一意。`[a-z0-9-]` のみ許可） |
| `body` | MEDIUMTEXT | NO | — | ページ本文（Markdown 形式。F06.1 と同一の marked.js + DOMPurify でレンダリング） |
| `icon` | VARCHAR(10) | YES | NULL | ページアイコン（絵文字1文字。例: 📖。NULL = デフォルトアイコン） |
| `access_level` | VARCHAR(20) | NO | 'ALL_MEMBERS' | 閲覧権限。値: `ALL_MEMBERS`, `ADMIN_ONLY`, `CUSTOM` |
| `status` | VARCHAR(15) | NO | 'DRAFT' | ページステータス。値: `DRAFT`, `PUBLISHED`, `ARCHIVED` |
| `sort_order` | INT | NO | 0 | 同一 parent_id 内での表示順 |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（SET NULL on delete。作成者） |
| `last_edited_by` | BIGINT UNSIGNED | YES | NULL | FK → users（SET NULL on delete。最終編集者） |
| `published_at` | DATETIME | YES | NULL | 初回公開日時 |
| `view_count` | INT UNSIGNED | NO | 0 | 閲覧数（denormalize。参考値） |
| `version` | BIGINT | NO | 0 | 楽観的ロック用バージョン番号 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
-- スコープ + 親ページ内の一覧取得（ツリー構築用）
INDEX idx_kp_scope_parent (scope_type, scope_id, parent_id, sort_order)
-- スコープ内スラッグ一意性（論理削除対応: MySQL の NULL != NULL 特性を利用）
UNIQUE KEY uq_kp_slug (scope_type, scope_id, slug, deleted_at)
-- マテリアライズドパスによる子孫検索（`path LIKE '/1/5/%'` で高速前方一致）
INDEX idx_kp_path (scope_type, scope_id, path)
-- 作成者別ページ一覧
INDEX idx_kp_created_by (created_by)
-- FULLTEXT インデックス（F04.6 グローバル検索連携用。N-gram パーサーで日本語対応）
FULLTEXT INDEX ft_kp_title_body (title, body) WITH PARSER ngram
-- 最終更新順一覧（最近編集されたページ）
INDEX idx_kp_scope_updated (scope_type, scope_id, updated_at DESC)
```

**制約・備考**
- `scope_type` の CHECK 制約:
  ```sql
  CONSTRAINT chk_kp_scope_type
    CHECK (scope_type IN ('TEAM', 'ORGANIZATION'))
  ```
- `access_level` の CHECK 制約:
  ```sql
  CONSTRAINT chk_kp_access_level
    CHECK (access_level IN ('ALL_MEMBERS', 'ADMIN_ONLY', 'CUSTOM'))
  ```
- `status` の CHECK 制約:
  ```sql
  CONSTRAINT chk_kp_status
    CHECK (status IN ('DRAFT', 'PUBLISHED', 'ARCHIVED'))
  ```
- `status` のライフサイクル:
  - `DRAFT` → `PUBLISHED`（公開。ADMIN / DEPUTY_ADMIN が実行）
  - `PUBLISHED` → `ARCHIVED`（アーカイブ。一覧から非表示だがデータ保持）
  - `ARCHIVED` → `PUBLISHED`（再公開。アーカイブ解除）
  - `DRAFT` は直接 `ARCHIVED` にできない（未公開ページのアーカイブは不要）
- `depth` の上限チェック: アプリケーション層で `depth <= 10` をバリデーション。UI 上は3階層を推奨するが、システムとしては10階層まで許容
- `path` の構成: `/{id}/{id}/...` 形式。ルートページの path は `/{self.id}`。子ページの path は `{parent.path}/{self.id}`。子孫検索は `WHERE path LIKE CONCAT(:parentPath, '/%')` で実行
- `slug` 自動生成: 未指定時は `title` から生成。日本語タイトルは nanoid（12文字英数字）をベースに生成。重複時はサフィックス `-2`, `-3` を自動付与
- `body` サイズ制限: 最大 100,000 文字（マニュアル等の長文を想定し、ブログの50,000文字より緩和）。MEDIUMTEXT を使用
- 論理削除: `deleted_at DATETIME nullable`（SoftDeletableEntity 適用）。子ページがある場合、親ページの削除時に子ページも連鎖的に論理削除
- 楽観的ロック: `version` カラムで同時編集を検知。更新リクエストに `version` を含め、不一致は 409 Conflict
- `CUSTOM` アクセスレベル: 将来的にページ単位のアクセス制御リスト（ACL）テーブルを追加して、特定ユーザー/ロールのみ閲覧可能にする拡張ポイント。Phase 11 では `CUSTOM` を設定した場合 ADMIN_ONLY と同等に扱う
- `view_count`: ページ閲覧時にインクリメント。同一ユーザーの連続アクセスは5分以内を重複カウント除外（Valkey で管理。キー: `mannschaft:kb:view:{pageId}:{userId}`、TTL 5分）

#### `kb_page_revisions`

ページの版管理。編集保存のたびに編集前の全文スナップショットを作成する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `kb_page_id` | BIGINT UNSIGNED | NO | — | FK → kb_pages（ON DELETE CASCADE） |
| `revision_number` | INT UNSIGNED | NO | — | 版番号（1始まり、kb_page_id 内で連番） |
| `title` | VARCHAR(200) | NO | — | その時点のタイトル |
| `body` | MEDIUMTEXT | NO | — | その時点の本文（全文スナップショット。差分ではない） |
| `editor_id` | BIGINT UNSIGNED | YES | NULL | FK → users（SET NULL on delete。編集者） |
| `change_summary` | VARCHAR(200) | YES | NULL | 変更概要（手動入力。NULL = 自動保存） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_kpr_page_revision (kb_page_id, revision_number)  -- ページ×版番号で一意
INDEX idx_kpr_page_date (kb_page_id, created_at DESC)           -- 版履歴一覧（新しい順）
```

**制約・備考**
- ページ削除時にカスケード削除
- **保存タイミング**: PUBLISHED ページを再編集して保存した時点で、編集前の状態を自動保存
- **上限30版**: ブログ記事（10版）より多く保持。マニュアル等は長期にわたり頻繁に更新されるため。31版目の作成時に最も古い版を物理削除（FIFO）
- **復元操作**: リビジョンの `title` / `body` を現在のページにコピーする（新しい版として扱う）

|---------|---|------|-----------|------|

| `kb_page_id` | BIGINT UNSIGNED | NO | — | FK → kb_pages（ON DELETE CASCADE） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_kpf_page_user (kb_page_id, user_id)    -- 同一ページ×ユーザーの重複防止
```

#### `kb_templates`

ページ作成用のテンプレートプリセット。ADMIN が定型テンプレートを管理し、メンバーがページ作成時にテンプレートを選択して初期内容を自動入力する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別。値: `TEAM`, `ORGANIZATION`, `SYSTEM`（SYSTEM = 公式テンプレート） |
| `scope_id` | BIGINT UNSIGNED | YES | NULL | スコープ ID（SYSTEM の場合は NULL） |
| `name` | VARCHAR(100) | NO | — | テンプレート名（例: 新メンバー向けガイド、管理規約、練習メニュー集） |
| `description` | VARCHAR(500) | YES | NULL | テンプレートの説明 |
| `body` | MEDIUMTEXT | NO | — | テンプレート本文（Markdown 形式。プレースホルダー可） |
| `icon` | VARCHAR(10) | YES | NULL | テンプレートアイコン（絵文字1文字） |
| `sort_order` | INT | NO | 0 | 表示順 |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（SET NULL on delete） |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
INDEX idx_kt_scope (scope_type, scope_id, sort_order)               -- スコープ別テンプレート一覧
UNIQUE KEY uq_kt_name (scope_type, scope_id, name, deleted_at)      -- スコープ内テンプレート名一意性
```

**制約・備考**
- `scope_type` の CHECK 制約:
  ```sql
  CONSTRAINT chk_kt_scope_type
    CHECK (scope_type IN ('TEAM', 'ORGANIZATION', 'SYSTEM'))
  ```
- SYSTEM テンプレート: SYSTEM_ADMIN のみ作成・編集・削除可能。全チーム/組織から利用可能
- チーム/組織テンプレート: ADMIN が作成・編集・削除。1スコープあたり上限20件
- 論理削除: 削除済みテンプレートで作成されたページには影響しない（作成時にコンテンツがコピーされるため）
- **公式テンプレート（初期データ）**:
  - 新メンバー向けガイド: チームの基本情報・ルール・連絡先の雛形
  - 管理規約: マンション管理組合向けの規約テンプレート
  - 練習メニュー集: スポーツチーム向けの練習メニュー記録テンプレート
  - FAQ: よくある質問と回答のテンプレート
  - 議事録: 会議の議事録テンプレート
  - 引き継ぎ資料: 担当者交代時の引き継ぎ資料テンプレート

#### `kb_image_uploads`

ナレッジベースページに挿入された画像の管理テーブル。F06.1 の `blog_image_uploads` と同一の孤立画像クリーンアップパターンを使用する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `kb_page_id` | BIGINT UNSIGNED | YES | NULL | FK → kb_pages（SET NULL on delete。NULL = 未紐付け / アップロード中） |
| `uploader_id` | BIGINT UNSIGNED | YES | NULL | FK → users（SET NULL on delete） |
| `s3_key` | VARCHAR(500) | NO | — | S3 オブジェクトキー |
| `file_size` | INT UNSIGNED | NO | — | ファイルサイズ（bytes） |
| `content_type` | VARCHAR(50) | NO | — | MIME タイプ（image/jpeg, image/png, image/webp 等） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_kiu_page (kb_page_id)                          -- ページ別画像一覧
INDEX idx_kiu_orphan (kb_page_id, created_at)            -- 孤立画像クリーンアップ用
UNIQUE KEY uq_kiu_s3_key (s3_key)                        -- S3 キー一意性
```

**制約・備考**
- F06.1 `blog_image_uploads` と同一パターン
- 画像アップロード時に `kb_page_id = NULL` で INSERT。ページ保存時に `body` 内の S3 URL を解析し、該当レコードの `kb_page_id` を更新
- **孤立画像クリーンアップ**: `kb_page_id IS NULL AND created_at < NOW() - INTERVAL 72 HOUR` のレコードを日次バッチで S3 削除 + 物理削除
- 1ページあたりの画像数上限: 50枚（マニュアル等は画像が多いためブログより緩和）
- 許可する Content-Type: `image/jpeg`, `image/png`, `image/webp`, `image/gif`
- 1画像あたりのファイルサイズ上限: 10MB

## 5. ビジネスロジック

### ページ作成フロー

```
1. ユーザーが POST /knowledge-base/pages を実行
2. バリデーション:
   a. ユーザーがスコープ（チーム/組織）に所属しているか
   b. ロール/権限チェック（ADMIN / DEPUTY_ADMIN + MANAGE_KNOWLEDGE_BASE / MEMBER + WRITE_KNOWLEDGE_BASE）
   c. ナレッジベースモジュールが有効化されているか（module_team_selections チェック）
   d. parent_id が指定されている場合、同一スコープ内に存在するか
   e. depth チェック: parent の depth + 1 <= 10
   f. title / body のバリデーション（body <= 100,000文字）
   g. slug の一意性チェック（スコープ内 + deleted_at IS NULL）
3. kb_pages に INSERT（status = DRAFT）
4. path を設定: parent_id が NULL → "/{self.id}"、あり → "{parent.path}/{self.id}"
5. template_id が指定されており body が空の場合、テンプレートの body をコピー
6. レスポンス返却（201 Created）
```

### ページ更新フロー

```
1. ユーザーが PATCH /knowledge-base/pages/{pageId} を実行
2. バリデーション:
   a. ページが存在し、論理削除されていないか
   b. ロール/権限チェック（MEMBER は自分が作成したページのみ）
   c. access_level 変更は ADMIN のみ
   d. version の楽観的ロックチェック
3. 現在のページが PUBLISHED の場合:
   a. 現在の title / body を kb_page_revisions に保存（自動リビジョン）
   b. revision_number はそのページの最大版番号 + 1
   c. リビジョン数が30を超えた場合、最古のリビジョンを物理削除
4. kb_pages を UPDATE（title, body, slug, icon, access_level, version++）
5. last_edited_by を更新
6. body 内の S3 URL を解析し、kb_image_uploads の kb_page_id を更新
   - body に含まれなくなった画像は kb_page_id = NULL にリセット（72時間後にクリーンアップ対象）
7. レスポンス返却（200 OK）
```

### ページ移動フロー

```
1. ユーザーが PATCH /knowledge-base/pages/{pageId}/move を実行
2. バリデーション:
   a. ロール/権限チェック
   b. version の楽観的ロックチェック
   c. new_parent_id が同一スコープ内か
   d. 循環参照チェック: new_parent_id が対象ページの子孫でないか
     → 対象ページの path を取得し、new_parent_id の path が対象の path で始まるか確認
   e. 移動後の depth チェック: new_parent_id の depth + 1 + 対象サブツリーの最大深さ <= 10
3. 対象ページ + 全子孫の path と depth を再計算:
   a. 対象ページ: path = "{new_parent.path}/{self.id}"、depth = new_parent.depth + 1
   b. 子孫: path の旧親パスプレフィックスを新親パスプレフィックスに置換
4. sort_order を更新
5. version++ で UPDATE
6. レスポンス返却
```

### リビジョン復元フロー

```
1. ユーザーが POST /knowledge-base/pages/{pageId}/revisions/{revisionId}/restore を実行
2. バリデーション:
   a. ロール/権限チェック（ADMIN / DEPUTY_ADMIN + MANAGE_KNOWLEDGE_BASE）
   b. リビジョンが対象ページに属しているか
3. 現在の title / body を新しいリビジョンとして kb_page_revisions に保存
4. リビジョンの title / body を現在のページにコピー
5. last_edited_by を復元操作者に更新
6. version++
7. レスポンス返却
```

### ページ削除フロー（連鎖論理削除）

```
1. ユーザーが DELETE /knowledge-base/pages/{pageId} を実行
2. バリデーション: ロール/権限チェック
3. 対象ページの path を取得
4. path LIKE '{対象の path}/%' で全子孫を SELECT
5. 対象ページ + 全子孫の deleted_at = NOW() を一括 UPDATE
6. ピン留め (kb_page_pins) を物理削除（対象ページ + 子孫）
7. レスポンスに children_deleted（削除された子孫数）を含めて返却
```

### F04.6 グローバル検索連携

```
■ 検索対象テーブル: kb_pages
■ 検索対象カラム: title, body
■ FULLTEXT インデックス: ft_kp_title_body (title, body) WITH PARSER ngram
■ search_content_type: 'KNOWLEDGE_BASE'
■ 認可フィルタ:
  - (kp.scope_type = 'TEAM' AND kp.scope_id IN (:userTeamIds))
    OR (kp.scope_type = 'ORGANIZATION' AND kp.scope_id IN (:userOrgIds))
  - AND kp.status = 'PUBLISHED'
  - AND kp.deleted_at IS NULL
  - AND kp.access_level = 'ALL_MEMBERS'
    （ADMIN_ONLY / CUSTOM ページはグローバル検索結果に含めない。スコープ内検索でのみヒット）
■ レスポンスの url: /teams/{scopeId}/knowledge-base/pages/{id}
■ actions: ["BOOKMARK"]（お気に入り登録）
```

### 閲覧カウント

```
1. GET /pages/{pageId} が呼ばれるたびに実行
2. Valkey チェック: mannschaft:kb:view:{pageId}:{userId} が存在するか
   a. 存在する → カウントしない（5分以内の再閲覧）
   b. 存在しない → Valkey SET（TTL 5分）+ view_count を INCREMENT
3. view_count は参考値。DB 上のカウントは直接 UPDATE view_count = view_count + 1 でアトミック更新
```

### ApplicationEvent 連携

| イベント | 発行タイミング | 購読先 |
|---------|--------------|-------|
| `KbPagePublishedEvent` | ページが DRAFT → PUBLISHED に遷移した時 | F04.3 通知サービス（スコープ内 MEMBER に「新しいナレッジベースページが公開されました」）、F04.6 グローバル検索（検索インデックス登録） |
| `KbPageUpdatedEvent` | PUBLISHED ページが更新された時 | F04.6 グローバル検索（検索インデックス更新） |
| `KbPageDeletedEvent` | ページが論理削除された時 | F04.6 グローバル検索（検索インデックスから除外） |
| `KbPageArchivedEvent` | ページが PUBLISHED → ARCHIVED に遷移した時 | F04.6 グローバル検索（検索インデックスから除外） |

### 重要な判定ロジック

- **access_level チェック**:
  - `ALL_MEMBERS`: MEMBER 以上の全員が閲覧可能
  - `ADMIN_ONLY`: ADMIN / DEPUTY_ADMIN（`MANAGE_KNOWLEDGE_BASE` 権限あり）のみ閲覧可能
  - `CUSTOM`: Phase 11 では `ADMIN_ONLY` と同等に扱う。将来の ACL テーブル拡張で個別ユーザー/ロール制御を追加

- **MEMBER の編集権限**:
  - `WRITE_KNOWLEDGE_BASE` パーミッション付与時のみページ作成可能
  - 自分が `created_by` であるページのみ編集可能
  - `access_level` の変更・ページの公開・ピン留め操作は不可
  - DRAFT の自分のページのみ削除可能

## 6. セキュリティ考慮事項

- **認可チェック**: KnowledgeBaseService の入り口で `scope_type` + `scope_id` と `currentUser` の所属を必ず検証する。チーム/組織のメンバーでないユーザーはアクセス不可
- **access_level の強制**: ページ取得・検索の全クエリに `access_level` フィルタを付加。MEMBER がアクセスできないページが検索結果やツリーに含まれないようにする
- **XSS 防止**: Markdown レンダリングはフロントエンド側で `marked.js` + `DOMPurify` を使用（F06.1 と同一基盤）。バックエンドは Markdown テキストをそのまま保存
- **パストラバーサル防止**: `path` カラムはバックエンドが自動生成。ユーザー入力は受け付けない
- **slug インジェクション防止**: `slug` は `[a-z0-9-]` のみ許可。アプリケーション層で正規表現バリデーション
- **レートリミット**: 画像アップロード API は1ユーザーあたり1分間に20回。ページ作成/更新は1ユーザーあたり1分間に30回
- **FULLTEXT インジェクション防止**: 検索キーワードの特殊文字（`+`, `-`, `*`, `~`, `"`, `(`, `)`）をエスケープ処理（F04.6 と同一方式）
- **論理削除後のアクセス防止**: 論理削除済みページは `@SQLRestriction("deleted_at IS NULL")` で自動除外。管理者が復元する場合のみカスタムクエリで取得
- **モジュール有効化チェック**: 全 API の入り口でナレッジベースモジュール（選択式モジュール #15）が対象チーム/組織で有効化されているか検証。無効の場合は 404

**マイグレーション上の注意点**
- `kb_pages` の自己参照 FK（`parent_id → kb_pages.id`）は同一テーブル内のため、テーブル作成と同時に定義可能
- `kb_page_revisions` は `kb_pages` に依存するため V11.010 より後（V11.011）

- V11.015 の公式テンプレート: `scope_type = 'SYSTEM'`, `scope_id = NULL` で INSERT。テンプレートの `body` は Markdown テキスト
- V11.017: `module_definitions` に `{ module_key: 'KNOWLEDGE_BASE', display_name: 'ナレッジベース', ... }` を INSERT。`module_level_availability` に組織・チームの2レコードを INSERT（個人は対象外）
