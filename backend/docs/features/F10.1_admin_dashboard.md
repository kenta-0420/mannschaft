# F10.1: 管理者ダッシュボード・通報モデレーション

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 10
> **最終更新**: 2026-03-17
> **モジュール種別**: デフォルト機能 #1（ダッシュボード）の管理者向け拡張 + 通報・モデレーション（デフォルト機能 #10）
> **関連ドキュメント**: F10.3 監査ログ（監査証跡の閲覧）、F01.1 認証（ロール・パーミッション基盤）、F04.3 プッシュ通知（通報通知）

---

## 1. 概要

チーム/組織の ADMIN 向け管理画面と、プラットフォーム全体を管理する SYSTEM_ADMIN 向けシステム管理画面を提供する。全管理機能を `/admin`（チーム/組織管理）と `/system-admin`（プラットフォーム管理）に集約し、ユーザー管理・権限設定・モジュール制御・課金管理・通報対応等を一元的に操作できる。

また、不適切コンテンツの通報・レビュー・対応（削除・警告・凍結）のモデレーションワークフローを提供し、プラットフォームの健全性を維持する。

### 本ドキュメントのスコープ

| 機能 | 対象 | 説明 |
|------|------|------|
| 管理者ダッシュボード（AA） | ADMIN / DEPUTY_ADMIN | チーム/組織レベルの管理画面。権限グループ・モジュール ON/OFF・MEMBER 権限調整 |
| システム管理者ダッシュボード（AB） | SYSTEM_ADMIN | プラットフォーム全体の管理。価格設定・テンプレート管理・ユーザーBAN・エラーレポート |
| 通報・モデレーション | 全ロール（通報）/ ADMIN・SYSTEM_ADMIN（対応） | 不適切コンテンツの通報 → レビュー → 段階的対応 |

**ダッシュボードウィジェット（個人/チーム/組織）** は Phase 2 で基盤実装済み。本ドキュメントでは Phase 10 で追加する**管理画面機能**に焦点を当てる。

### テーブル定義

#### `content_reports`

コンテンツ通報。ポリモーフィック参照で任意のコンテンツを通報対象にできる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `target_type` | VARCHAR(30) | NO | — | 通報対象の種別（`TIMELINE_POST` / `CHAT_MESSAGE` / `BULLETIN_THREAD` / `BULLETIN_COMMENT` / `MATCHING_REQUEST` / `PROPERTY_LISTING` / `USER_PROFILE`） |
| `target_id` | BIGINT UNSIGNED | NO | — | 通報対象の ID |
| `reported_by` | BIGINT UNSIGNED | NO | — | FK → users。通報者 |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `reason` | VARCHAR(30) | NO | — | 通報理由（`SPAM`: スパム / `HARASSMENT`: ハラスメント / `INAPPROPRIATE`: 不適切 / `MISINFORMATION`: 誤情報 / `COPYRIGHT`: 著作権侵害 / `OTHER`: その他） |
| `target_user_id` | BIGINT UNSIGNED | YES | NULL | FK → users。通報対象コンテンツの作成者（WARNING / FREEZE の対象。`USER_PROFILE` 通報の場合は対象ユーザー自身） |
| `content_snapshot` | JSON | YES | NULL | 通報時点のコンテンツスナップショット（通報後の編集・削除に備えた証拠保全。タイトル・本文・画像URL等を保存） |
| `description` | TEXT | YES | NULL | 通報の詳細説明（任意。最大1,000文字） |
| `status` | VARCHAR(20) | NO | 'PENDING' | ステータス（`PENDING`: 未対応 / `REVIEWING`: 確認中 / `ESCALATED`: エスカレーション中 / `RESOLVED`: 対応済み / `DISMISSED`: 却下） |
| `reviewed_by` | BIGINT UNSIGNED | YES | NULL | FK → users。レビューした管理者 |
| `reviewed_at` | DATETIME | YES | NULL | レビュー日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_cr_scope (scope_type, scope_id, status, created_at DESC)  -- スコープ別の通報一覧
INDEX idx_cr_target (target_type, target_id)                         -- 対象別の通報件数
INDEX idx_cr_status (status, created_at DESC)                        -- ステータス別一覧（SYSTEM_ADMIN 用）
INDEX idx_cr_reported_by (reported_by)                               -- 通報者別一覧
UNIQUE KEY uq_cr_target_reporter (target_type, target_id, reported_by)  -- 同一ユーザーによる同一コンテンツの重複通報防止
INDEX idx_cr_target_user (target_user_id, status)                    -- 対象ユーザー別の通報（エスカレーション閾値チェック用）
```

**制約・備考**
- ポリモーフィック参照（FK 制約なし、アプリ層で検証）
- 同一ユーザーが同一コンテンツを複数回通報不可（UNIQUE KEY `uq_cr_target_reporter` で DB 制約）
- `PENDING` → `REVIEWING`（ADMIN がレビュー開始）→ `RESOLVED`（対応実施）/ `DISMISSED`（問題なしとして却下）/ `ESCALATED`（SYSTEM_ADMIN にエスカレーション）
- `DISMISSED` → `REVIEWING`（誤却下の差し戻し。`PATCH /reopen`）
- `ESCALATED` → `RESOLVED` / `DISMISSED`（SYSTEM_ADMIN が対応）
- `RESOLVED` への遷移時に `report_actions` にアクションを記録
- SYSTEM_ADMIN は全スコープの通報を閲覧可能。ADMIN はスコープ内の通報のみ
- `target_user_id`: 通報作成時にバックエンドが対象コンテンツの作成者を自動取得して設定。`USER_PROFILE` の場合は `target_id` = `target_user_id`
- `content_snapshot`: 通報作成時にバックエンドが対象コンテンツの現在の状態を JSON で保存。通報後にコンテンツが編集・削除されても ADMIN が元の内容を確認可能。**サイズ制限: 最大 64KB**。超過する場合は本文を先頭 10,000 文字に切り詰め、画像 URL は最大 20 件まで保持
- `scope_type` / `scope_id`: 通報作成時にバックエンドが対象コンテンツのスコープを自動導出（例: タイムライン投稿 → 投稿の team_id / organization_id）。リクエストボディには含めない
- グルーピング解決: `RESOLVED` / `DISMISSED` 遷移時に、同一 `target_type + target_id` を持つ全通報のステータスを一括更新

#### `report_actions`

通報への対応アクション履歴。1件の通報に対して複数のアクションを記録可能（警告→削除等の段階的対応）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `report_id` | BIGINT UNSIGNED | NO | — | FK → content_reports。ON DELETE CASCADE |
| `action_type` | VARCHAR(20) | NO | — | アクション種別（`WARNING`: 投稿者への警告 / `CONTENT_DELETE`: コンテンツ削除 / `USER_FREEZE`: ユーザー凍結 / `DISMISS`: 問題なし却下 / `ESCALATE`: SYSTEM_ADMIN へエスカレーション / `REOPEN`: 誤却下の差し戻し） |
| `action_by` | BIGINT UNSIGNED | NO | — | FK → users。アクション実行者 |
| `note` | TEXT | YES | NULL | 管理者メモ（対応理由等） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_ra_report (report_id, created_at)  -- 通報別のアクション履歴
```

**制約・備考**
- `CONTENT_DELETE` 実行時: 対象コンテンツの論理削除を同一トランザクションで実行
- `USER_FREEZE` 実行時: `users.is_frozen = TRUE` を設定。凍結中ユーザーはログイン不可
- `WARNING` 実行時: 対象ユーザーにプッシュ通知 + メール送信（「コミュニティガイドラインに違反するコンテンツが報告されました」）
- 監査ログに `REPORT_ACTION_TAKEN` イベントを記録

#### `feature_flags`

フィーチャーフラグ。新機能のコードデプロイ後に、ダッシュボードから ON/OFF を制御する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `flag_key` | VARCHAR(100) | NO | — | フラグキー（例: `FEATURE_PARKING_MODULE`）。UNIQUE |
| `is_enabled` | BOOLEAN | NO | FALSE | 有効/無効 |
| `description` | VARCHAR(500) | YES | NULL | フラグの説明（どの機能を制御するか） |
| `updated_by` | BIGINT UNSIGNED | YES | NULL | FK → users。最後に変更した SYSTEM_ADMIN |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_ff_key (flag_key)  -- フラグキーの一意制約
```

**制約・備考**
- フラグの追加は Flyway マイグレーション or `POST /system-admin/feature-flags` で実施
- フラグの値は Valkey にキャッシュ（`mannschaft:feature:{key}`、TTL なし）。PUT 時に即座にキャッシュ更新
- API 層は `FeatureFlagService.isEnabled(key)` で判定。無効なフラグキーは `false` を返却
- フラグの物理削除は不可（非活性化のみ。誤って削除するとコード側の参照が壊れるため）

#### `maintenance_schedules`

メンテナンス予定。事前告知 + 自動切替を管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `title` | VARCHAR(200) | NO | — | メンテナンスタイトル（例: 「定期メンテナンス v2.5.0」） |
| `message` | TEXT | YES | NULL | ユーザー向けメッセージ |
| `mode` | VARCHAR(20) | NO | 'MAINTENANCE' | 切替先モード（`READ_ONLY` / `MAINTENANCE`） |
| `starts_at` | DATETIME | NO | — | メンテナンス開始日時 |
| `ends_at` | DATETIME | NO | — | メンテナンス終了予定日時 |
| `status` | VARCHAR(20) | NO | 'SCHEDULED' | ステータス（`SCHEDULED`: 予定 / `ACTIVE`: 実行中 / `COMPLETED`: 完了 / `CANCELLED`: キャンセル） |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。作成した SYSTEM_ADMIN |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_ms_status (status, starts_at)  -- 予定・実行中のメンテナンス検索
```

**制約・備考**
- `starts_at` < `ends_at`（Service 層バリデーション）
- `SCHEDULED` → `ACTIVE`（開始時刻到達で自動遷移、バッチ1分間隔）→ `COMPLETED`（終了時刻到達で自動遷移 + メンテナンスモード → NORMAL に復帰）
- `SCHEDULED` → `CANCELLED`（手動キャンセル）
- 事前通知: 登録時 + 24時間前 + 1時間前にプッシュ通知 + フロントエンドにバナー表示
- メンテナンスモードの状態は Valkey（`mannschaft:maintenance:mode`）で管理。Spring Filter で全リクエストをインターセプト
- SYSTEM_ADMIN の `/system-admin/**` API はメンテナンスモード中も常に利用可能

#### `platform_announcements`

運営からのお知らせ。SYSTEM_ADMIN がプラットフォーム全ユーザーに一括通知する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `title` | VARCHAR(200) | NO | — | お知らせタイトル |
| `body` | TEXT | NO | — | 本文（Markdown 対応） |
| `priority` | VARCHAR(10) | NO | 'NORMAL' | 優先度（`URGENT`: 緊急 / `NORMAL`: 通常 / `LOW`: 低優先度） |
| `target_scope` | VARCHAR(20) | NO | 'ALL' | 配信対象（`ALL`: 全ユーザー / `ADMINS_ONLY`: ADMIN 以上のみ） |
| `is_pinned` | BOOLEAN | NO | FALSE | ダッシュボードへの固定表示。TRUE = 全ユーザーのダッシュボード上部に常時表示 |
| `published_at` | DATETIME | YES | NULL | 公開日時。NULL = 下書き |
| `expires_at` | DATETIME | YES | NULL | 表示期限。到達後はダッシュボードから非表示（アーカイブ参照は可能） |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。作成した SYSTEM_ADMIN |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_pa_published (published_at DESC, expires_at)     -- 公開中のお知らせ一覧
INDEX idx_pa_priority (priority, published_at DESC)         -- 緊急度別一覧
```

**制約・備考**
- `published_at IS NOT NULL` のレコードのみユーザーに表示。NULL = 下書き状態
- `URGENT` 優先度: 公開時に全ユーザーにプッシュ通知。ダッシュボード上部に赤色バナーで固定表示
- `NORMAL` 優先度: ダッシュボードのお知らせウィジェットに表示。プッシュ通知なし
- `LOW` 優先度: お知らせウィジェットに表示。プッシュ通知なし。ダッシュボードの折りたたみセクションに格納
- `is_pinned = TRUE`: `expires_at` に関わらずダッシュボード上部に常時表示（SYSTEM_ADMIN が手動で解除するまで）
- `target_scope = 'ADMINS_ONLY'`: 運用上の連絡事項（料金改定予告・API 仕様変更等）を ADMIN にのみ配信
- 全ユーザーのダッシュボード API（`GET /dashboard`）のレスポンスに `platform_announcements` 配列を含める

#### `feedback_submissions`

目安箱。組織・チーム・プラットフォームの3レベルでアイデア・クレーム・要望を受付。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION` / `PLATFORM`） |
| `scope_id` | BIGINT UNSIGNED | YES | NULL | TEAM/ORG の ID。`PLATFORM` の場合は NULL |
| `category` | VARCHAR(20) | NO | — | カテゴリ（`IDEA`: アイデア / `COMPLAINT`: クレーム / `BUG_REPORT`: 不具合報告 / `QUESTION`: 質問 / `OTHER`: その他） |
| `title` | VARCHAR(200) | NO | — | タイトル |
| `body` | TEXT | NO | — | 本文（最大5,000文字） |
| `is_anonymous` | BOOLEAN | NO | FALSE | 匿名投稿フラグ。TRUE = 投稿者を ADMIN にも非公開（DB には記録） |
| `submitted_by` | BIGINT UNSIGNED | NO | — | FK → users。投稿者（匿名でも DB には記録。悪質投稿の追跡用） |
| `status` | VARCHAR(20) | NO | 'OPEN' | ステータス（`OPEN` / `IN_PROGRESS` / `RESOLVED` / `CLOSED`） |
| `admin_response` | TEXT | YES | NULL | ADMIN / SYSTEM_ADMIN の回答（公開用） |
| `responded_by` | BIGINT UNSIGNED | YES | NULL | FK → users。回答した管理者 |
| `responded_at` | DATETIME | YES | NULL | 回答日時 |
| `is_public_response` | BOOLEAN | NO | FALSE | 回答を全メンバーに公開するか（FAQ 的活用） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_fs_scope (scope_type, scope_id, status, created_at DESC)  -- スコープ別一覧
INDEX idx_fs_submitted_by (submitted_by)                              -- 投稿者別一覧
INDEX idx_fs_category (scope_type, scope_id, category)                -- カテゴリ別一覧
```

**制約・備考**
- `PLATFORM` スコープ: SYSTEM_ADMIN 宛。機能改善リクエスト・プラットフォームへのクレーム等
- `TEAM` / `ORGANIZATION` スコープ: チーム/組織の ADMIN 宛。組織運営改善の提案等
- `is_anonymous = TRUE`: API レスポンスで `submitted_by` を返却しない。SYSTEM_ADMIN が DB 直接アクセスでのみ確認可能（悪質投稿の追跡用に保持）
- `is_public_response = TRUE`: 回答を全メンバーに公開。FAQ として蓄積され、GET /feedback で閲覧可能
- ステータス遷移: `OPEN` → `IN_PROGRESS`（対応開始）→ `RESOLVED`（回答完了）/ `CLOSED`（対応不要として終了）

#### `batch_job_logs`

バッチジョブ実行ログ。全 `@Scheduled` バッチの実行状況を記録する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `job_name` | VARCHAR(100) | NO | — | ジョブ名（例: `LeaseExpiryNotificationJob`） |
| `status` | VARCHAR(20) | NO | — | ステータス（`RUNNING` / `SUCCESS` / `FAILED`） |
| `started_at` | DATETIME | NO | — | 実行開始日時 |
| `finished_at` | DATETIME | YES | NULL | 実行完了日時 |
| `processed_count` | INT | NO | 0 | 処理件数 |
| `error_message` | TEXT | YES | NULL | エラー時のメッセージ・スタックトレース |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_bjl_job (job_name, started_at DESC)  -- ジョブ別の実行履歴
INDEX idx_bjl_status (status, started_at DESC)  -- 失敗ジョブの検索
```

**制約・備考**
- 共通の `BatchJobLogger` ユーティリティクラスで INSERT/UPDATE を統一。各バッチが個別に実装しない
- `RUNNING` が30分以上続く場合: バッチでタイムアウト検知 → `FAILED` に遷移 + SYSTEM_ADMIN アラート
- `FAILED` 発生時: SYSTEM_ADMIN にプッシュ通知 + Webhook（Slack）
- 保持期間: 90日。90日超のレコードは物理削除（月次バッチ）

#### `notification_delivery_stats`

通知配信統計。チャネル別の日次配信結果を集計する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `date` | DATE | NO | — | 集計日 |
| `channel` | VARCHAR(20) | NO | — | チャネル（`PUSH` / `EMAIL` / `LINE` / `IN_APP`） |
| `sent_count` | INT UNSIGNED | NO | 0 | 送信数 |
| `delivered_count` | INT UNSIGNED | NO | 0 | 到達数 |
| `failed_count` | INT UNSIGNED | NO | 0 | 失敗数 |
| `bounce_count` | INT UNSIGNED | NO | 0 | バウンス数（EMAIL のみ） |

**インデックス**
```sql
UNIQUE KEY uq_nds_date_channel (date, channel)  -- 日次 × チャネルの一意制約
```

**制約・備考**
- 日次バッチ（毎日 AM 2:00）で前日分を集計して INSERT
- 到達率 = `delivered_count / sent_count * 100`。95% を下回った場合に SYSTEM_ADMIN アラート
- 保持期間: 1年。1年超のレコードは物理削除

#### `feedback_votes`

目安箱への賛同票。他ユーザーの投稿に「賛同」を表明して ADMIN の優先度判断を支援する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `feedback_id` | BIGINT UNSIGNED | NO | — | FK → feedback_submissions。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。賛同したユーザー |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_fv_feedback_user (feedback_id, user_id)  -- 同一ユーザーの重複投票防止
INDEX idx_fv_feedback (feedback_id)                      -- 投稿別の投票数集計
```

**制約・備考**
- 1ユーザーにつき同一投稿への賛同は1回のみ（UNIQUE KEY で制約）
- `is_anonymous = TRUE` の投稿にも賛同可能（投稿者は匿名だが、投稿内容は閲覧可能なため）
- 賛同の取り消し: `DELETE /feedback/{id}/vote` で物理削除
- `GET /admin/feedback` のレスポンスに `vote_count` を含める（ADMIN の優先度判断材料）
- `GET /feedback` の公開回答一覧にも `vote_count` を含める

## 5. ビジネスロジック

### 通報・モデレーションワークフロー

```
1. ユーザーが不適切コンテンツを発見 → POST /reports で通報
   a. 通報理由を選択（SPAM / HARASSMENT / INAPPROPRIATE / MISINFORMATION / COPYRIGHT / OTHER）
   b. 詳細説明を任意で記入
   c. スコープの ADMIN にプッシュ通知

2. ADMIN が通報を確認
   a. GET /admin/reports で未対応の通報一覧を確認
   b. 同一コンテンツへの複数通報はグルーピング表示（report_count）
   c. PATCH /admin/reports/{id}/review でレビュー開始（PENDING → REVIEWING）

3. ADMIN が対応を決定
   a. 対象コンテンツを確認（通報詳細にスナップショットを含む）
   b. 段階的対応:
      - WARNING: 投稿者に警告通知。コンテンツはそのまま
      - CONTENT_DELETE: コンテンツを論理削除
      - USER_FREEZE: ユーザーを凍結（ログイン不可）。SYSTEM_ADMIN のみ実行可能
      - DISMISS: 問題なしとして却下
   c. PATCH /admin/reports/{id}/resolve で対応実施
   d. 通報者に「対応しました」通知
```

### 重要な判定ロジック

- **通報のグルーピング**: 同一 `target_type + target_id` への複数通報は UI 上でグルーピング表示。ADMIN は最初の通報を起点に対応し、関連通報は一括で `RESOLVED` に遷移
- **USER_FREEZE の権限制限**: ユーザー凍結は SYSTEM_ADMIN のみ実行可能。ADMIN / DEPUTY_ADMIN はスコープ内のコンテンツ削除・警告まで
- **DEPUTY_ADMIN のコンテンツ削除**: 権限グループで `CONTENT_DELETE` パーミッションが明示的に ON の場合のみ。デフォルト OFF
- **エスカレーション判断**: 同一ユーザー（`target_user_id`）に対する `PENDING` + `REVIEWING` 状態の通報が N 件以上蓄積した場合、SYSTEM_ADMIN に自動通知（常習的な違反の可能性）。N のデフォルト値は 3。SYSTEM_ADMIN が `PUT /system-admin/moderation-settings` で変更可能。`RESOLVED` / `DISMISSED` の通報はカウント対象外（対応済みの件で繰り返しエスカレーションしないため）
- **ADMIN の自己降格防止**: `PATCH /admin/users/{id}/role` でスコープ内の最後の ADMIN が自分自身を MEMBER / SUPPORTER に降格することを禁止（409 エラー）。管理者不在によるロック状態を防止
- **通報者の匿名性**: 通報者の情報は対象ユーザーには開示しない。ADMIN / SYSTEM_ADMIN のみ閲覧可能
- **凍結ユーザーの即座無効化**: `USER_FREEZE` 実行時、Valkey のトークン無効化リストに追加し、既存セッションを即座に無効化
- **価格変更の適用タイミング**: モジュール価格・パッケージ価格・組織数課金の変更は翌請求サイクルから適用。現在の請求期間中は旧価格が維持される
- **モジュールレベル別適用**: SYSTEM_ADMIN が各モジュールの組織/チーム/個人レベルでの利用可否を ON/OFF で制御。OFF にしたレベルでは API アクセスも拒否
- **一括対応**: `POST /admin/reports/bulk-resolve` で最大50件を同一 `action_type` で一括処理。各通報のステータスが `PENDING` / `REVIEWING` でなければスキップ（エラーにならず skipped_count に加算）。**`CONTENT_DELETE` の一括処理時の注意**: 対象通報が複数の `target_type`（タイムライン投稿・チャットメッセージ・掲示板スレッド等）にまたがる場合、各 `target_type` の論理削除処理はそれぞれの Service に委譲する。全件を単一トランザクションで処理し、1件でも削除失敗した場合は全体をロールバック
- **違反履歴の参照範囲**: ADMIN はスコープ内の通報のみ、SYSTEM_ADMIN は全スコープの通報を含む違反履歴を返却
- **通報アーカイブ**: 対応済み（`RESOLVED` / `DISMISSED`）の通報は6ヶ月後にアーカイブテーブルに移行。統計 API は必要に応じてアーカイブテーブルも参照
- **フィーチャーフラグ**: Valkey キャッシュで高速参照。PUT 時に即座にキャッシュ更新。フラグの物理削除は不可（コード側の参照保護）
- **メンテナンスモード**: Valkey で状態管理。Spring Filter で全リクエストをインターセプト。SYSTEM_ADMIN の `/system-admin/**` は常にアクセス可能
- **メンテナンス予告**: 登録時 + 24時間前 + 1時間前 にリマインド通知。フロントエンドに予告バナーを開始24時間前から表示
- **運営お知らせ**: `URGENT` は公開時にプッシュ通知。`is_pinned` はダッシュボード上部に常時固定。`target_scope = 'ADMINS_ONLY'` で ADMIN 限定配信
- **虚偽通報対策**: 却下率 70%+ かつ 通報5件+ で「要注意」フラグ。直近30日で DISMISSED 5件以上で自動レート制限（1日1件）。ADMIN が手動で通報権限を最大90日停止可能
- **コンテンツ復元**: `CONTENT_DELETE` 後72時間以内のみ復元可能。ステータスを `REVIEWING` に差し戻し
- **デプロイ後異常検知**: 5分間のエラーレポート件数が閾値（デフォルト10件）を超えた場合に SYSTEM_ADMIN に自動アラート + Webhook
- **データエクスポート**: 非同期ジョブ。30日に1回制限。ZIP（JSON）を S3 に保存、Pre-signed URL で24時間ダウンロード可能
- **SYSTEM_ADMIN 最低2人体制**: SYSTEM_ADMIN が1名のみの場合にダッシュボードに警告表示。重要操作時に外部 Webhook（Slack）に通知
- **目安箱の匿名保護**: `is_anonymous = TRUE` の投稿は API レスポンスで `submitted_by` を返却しない。DB には記録（悪質投稿の追跡用。SYSTEM_ADMIN の DB 直接アクセスでのみ確認可能）
- **目安箱の公開回答**: `is_public_response = TRUE` の回答は全メンバーが閲覧可能。FAQ として蓄積
- **バッチジョブのタイムアウト**: `RUNNING` が30分以上続くジョブは自動で `FAILED` 判定
- **ヘルスチェック**: `/api/v1/health` は認証不要（ALB ヘルスチェック用）。`/system-admin/health` はコンポーネント別の詳細情報を返却（SYSTEM_ADMIN のみ）
- **縮退運転**: 外部サービス障害時は該当機能の API のみ 503 を返却し、他機能は正常動作を維持。Resilience4j Circuit Breaker で自動検知・復旧
- **通知到達率アラート**: チャネル別の到達率が 95% を下回った場合に SYSTEM_ADMIN に自動通知
- **定期レポート**: 週次（月曜 AM 8:00）+ 月次（1日 AM 8:00）で主要指標を SYSTEM_ADMIN にメール自動送信
- **通報の優先度スコアリング**: `GET /admin/reports` に `sort=PRIORITY` オプション。通報件数・違反履歴・理由の重み・通報者の信頼度から 0-100 のスコアを算出（詳細は「通報優先度スコアリング」セクション参照）
- **通報対応テンプレート**: スコープ内の ADMIN が定型文テンプレートを作成。`resolve` / `dismiss` 時に `template_id` を指定すると `note` の初期値として使用。1スコープあたり最大50件
- **通報の却下時通知**: `DISMISSED` 遷移時にも通報者にプッシュ通知を送信（「確認の結果、ガイドライン違反には該当しませんでした」）。通報品質の向上を促す
- **誤却下の差し戻し**: `PATCH /admin/reports/{id}/reopen` で `DISMISSED` → `REVIEWING` に差し戻し可能。同一 `target_type + target_id` の全通報を一括で `REVIEWING` に遷移。`report_actions` に `REOPEN` アクションを記録
- **エスカレーション**: `POST /admin/reports/{id}/escalate` で ADMIN が SYSTEM_ADMIN にエスカレーション。理由を記入し、通報の status を `ESCALATED` に遷移。SYSTEM_ADMIN 全員にプッシュ通知。自動エスカレーション（閾値 N 件）とは別に、ADMIN の判断による手動エスカレーションも可能
- **通報の平均対応時間**: `GET /admin/reports/stats` のレスポンスに `avg_resolution_hours` / `median_resolution_hours` を含む。`created_at` → `reviewed_at` の差分（RESOLVED 通報のみ対象）
- **目安箱の賛同機能**: `POST /feedback/{id}/vote` で他ユーザーの投稿に賛同。1ユーザー1投稿1票。`GET /admin/feedback` のレスポンスに `vote_count` を含め、ADMIN の優先度判断を支援。匿名投稿にも賛同可能（投稿内容は閲覧可能なため）
- **お知らせの予約公開**: `platform_announcements.published_at` に未来の日時を設定可能。バッチ（1分間隔）で `published_at <= NOW()` かつ未処理のレコードを自動公開。URGENT の場合はプッシュ通知も自動発火
- **バッチジョブの手動リトライ**: `POST /system-admin/batch-jobs/{jobName}/retry` で FAILED ジョブを即座に再実行。同名ジョブが RUNNING 中は 409。1時間に3回までのレートリミット
- **content_snapshot のサイズ制限**: JSON カラムの最大サイズは 64KB。超過する場合は本文を先頭 10,000 文字に切り詰め、画像 URL は最大 20 件まで保持
- **ADMIN 通知ダイジェスト**: F04.3（プッシュ通知）で設計済み。`notification_type_preferences` テーブルの `DAILY_DIGEST` タイプで opt-in 制御。毎日 AM 8:00 に前日分の通知をサマリー化して配信

## 6. セキュリティ考慮事項

- **認可チェック**: `/admin/**` は ADMIN / DEPUTY_ADMIN（権限グループ依存）のみ。`/system-admin/**` は SYSTEM_ADMIN のみ。全エンドポイントでロール + スコープメンバーシップを検証
- **SYSTEM_ADMIN 権限の厳格管理**: SYSTEM_ADMIN の作成は DB 直接操作 or 初期シードのみ。API からの昇格は不可
- **通報者の匿名保護**: 通報者の `user_id` / `display_name` は対象ユーザーの API レスポンスに含めない。ADMIN 管理画面でのみ表示
- **コンテンツ削除の証跡保全**: 論理削除のみ（物理削除しない）。削除されたコンテンツは監査ログ + `report_actions` で追跡可能
- **凍結の即時性**: `USER_FREEZE` 時に Valkey のトークン無効化リストに追加。次回 API アクセス時に 401 を返却
- **レートリミット**: `Bucket4j` で以下の制限を適用
  - 通報 API（`POST /reports`）: 1分間5件（通報スパム防止）
  - 一括対応 API（`POST /admin/reports/bulk-resolve`）: 1分間3件
  - フィーチャーフラグ切替: 1分間10件
  - データエクスポート: 30日に1件/ユーザー
  - 運営お知らせ（URGENT）: 1時間に3件（プッシュ通知スパム防止）
  - 目安箱投稿: 1日5件/ユーザー（投稿スパム防止）
  - 目安箱賛同: 1分間30件/ユーザー
  - バッチジョブ手動リトライ: 同一ジョブにつき1時間3件
  - 管理者 API（`/admin/**`）: 1分間60件
  - システム管理者 API（`/system-admin/**`）: 1分間60件
- **監査ログ**: 以下のイベントを記録
  - `REPORT_CREATED`: 通報作成
  - `REPORT_ACTION_TAKEN`: 通報への対応（action_type を含む）
  - `USER_FROZEN`: ユーザー凍結
  - `USER_UNFROZEN`: ユーザー凍結解除
  - `PERMISSION_GROUP_CREATED`: 権限グループ作成
  - `PERMISSION_GROUP_ASSIGNED`: 権限グループ割り当て
  - `MODULE_PRICE_CHANGED`: モジュール価格変更
  - `PACKAGE_CREATED`: パッケージ作成
  - `DISCOUNT_CAMPAIGN_CREATED`: 割引キャンペーン作成
  - `ROLE_CHANGED`: ユーザーロール変更
  - `REPORTS_BULK_RESOLVED`: 通報の一括対応（件数を含む）
  - `FEATURE_FLAG_CHANGED`: フィーチャーフラグ ON/OFF（key + 新旧値）
  - `MAINTENANCE_MODE_CHANGED`: メンテナンスモード切替（mode + 実行者）
  - `MAINTENANCE_SCHEDULED`: メンテナンス予定登録
  - `ANNOUNCEMENT_PUBLISHED`: 運営お知らせ公開
  - `CONTENT_RESTORED`: 誤削除コンテンツの復元
  - `REPORTING_RESTRICTED`: ユーザーの通報権限停止
  - `DATA_EXPORT_REQUESTED`: データエクスポートリクエスト
  - `FEEDBACK_SUBMITTED`: 目安箱投稿
  - `FEEDBACK_RESPONDED`: 目安箱への回答
  - `BATCH_JOB_FAILED`: バッチジョブ失敗（job_name + error_message）
  - `BATCH_JOB_MANUAL_RETRY`: バッチジョブ手動リトライ（job_name + 実行者）
  - `REPORT_ESCALATED`: 通報のエスカレーション（report_id + エスカレーション理由 + エスカレーション元 ADMIN）
  - `REPORT_REOPENED`: 誤却下の差し戻し（report_id + 差し戻し理由）
  - `EXTERNAL_SERVICE_DOWN`: 外部サービス障害検知（service_name）
  - `EXTERNAL_SERVICE_RECOVERED`: 外部サービス復旧
