# F09.5: 共用施設予約管理

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 9
> **最終更新**: 2026-03-15
> **関連ドキュメント**: F09.3 駐車場区画管理（来場者予約パターン流用）、F09.1 住民台帳（共用施設の居室連携）、F04.3 プッシュ通知（予約承認・却下・リマインド通知）、F08.2 決済・アクセス制御（Stripe 施設利用料決済）、F03.4 予約管理（サービス予約との差異整理）

---

## 1. 概要

マンション・スポーツ施設・学校・コミュニティセンター等において、共用施設（ゲストルーム、集会室/会議室、パーティールーム、BBQエリア、多目的ホール、キッズルーム、フィットネスルーム、ランドリールーム等）の予約・利用管理を一元化する機能。管理者（ADMIN / DEPUTY_ADMIN）が施設マスタ・利用ルール・時間帯別料金・備品を登録し、メンバーが30分単位のタイムスロットで予約を申請する。承認制フロー（PENDING_APPROVAL → CONFIRMED）により管理者が利用目的・ルール遵守を事前確認し、自動承認オプション（`auto_approve`）でセルフ管理施設にも対応する。施設ごとの利用ルール（最大利用時間、月あたり予約上限、連続予約制限）、時間帯別従量課金（平日/休日/夜間で異なるレート）、備品貸出（プロジェクター、食器、BBQグリル等の在庫数量管理）、Stripe による利用料決済（F08.2 と同パターンの DIRECT/STRIPE 選択制）、予約間の清掃・準備バッファ時間、定員管理を提供する。翌日バッチで NO_SHOW を自動検出し、24時間キャンセルポリシーで直前キャンセルの濫用を防止する。F09.3 駐車場の来場者予約で確立した承認制・30分スロット・キャンセルポリシー・NO_SHOW検出パターンを踏襲しつつ、施設固有の利用ルール・従量課金・備品管理を追加した設計。

### テーブル定義

#### `shared_facilities`

共用施設のマスタ情報。施設名・種別・定員・所在階・基本料金・清掃バッファ等を管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `name` | VARCHAR(100) | NO | — | 施設名（例: 'ゲストルーム A', '集会室', 'BBQエリア'） |
| `facility_type` | ENUM('GUEST_ROOM', 'MEETING_ROOM', 'PARTY_ROOM', 'BBQ_AREA', 'MULTIPURPOSE_HALL', 'KIDS_ROOM', 'FITNESS_ROOM', 'LAUNDRY_ROOM', 'STUDY_ROOM', 'MUSIC_ROOM', 'ROOFTOP', 'OTHER') | NO | — | 施設種別 |
| `facility_type_label` | VARCHAR(50) | YES | NULL | 種別の補足ラベル（`facility_type = OTHER` 時の自由記述。例: 'ペットサロン', 'ゴルフシミュレーター室'） |
| `capacity` | SMALLINT UNSIGNED | NO | — | 定員（最大収容人数） |
| `floor` | VARCHAR(10) | YES | NULL | 所在階・エリア（例: '1F', 'B1', '屋上', '中庭'）。表示・フィルタ用 |
| `location_detail` | VARCHAR(200) | YES | NULL | 詳細な場所説明（例: 'エントランス棟2階 エレベーター右手', '南棟屋上'） |
| `description` | TEXT | YES | NULL | 施設説明（Markdown 対応。設備・利用上の注意等） |
| `image_urls` | JSON | YES | NULL | 施設画像URL配列（最大5枚。S3 Pre-signed URL。例: `["https://...jpg", "https://...jpg"]`） |
| `hourly_rate` | DECIMAL(10,0) | YES | NULL | 基本料金（円/30分スロット）。NULL = 無料。時間帯別料金（`facility_time_rates`）が設定されている場合はそちらが優先 |
| `cleaning_buffer_minutes` | SMALLINT UNSIGNED | NO | 0 | 予約間の清掃・準備バッファ（分）。0 = バッファなし。例: 30 の場合、10:00-12:00 の予約後は 12:30 まで次の予約不可 |
| `is_active` | BOOLEAN | NO | TRUE | 施設の有効/無効（FALSE = メンテナンス等で一時停止。予約不可だが閲覧可能） |
| `display_order` | SMALLINT UNSIGNED | NO | 0 | 表示順（昇順ソート） |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users（登録者）。ON DELETE RESTRICT |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
UNIQUE KEY uq_sf_scope_name (scope_type, scope_id, name, deleted_at)       -- 同一スコープ内の施設名重複防止（論理削除考慮）
INDEX idx_sf_scope_active (scope_type, scope_id, is_active, display_order)  -- スコープ別の施設一覧
INDEX idx_sf_scope_type (scope_type, scope_id, facility_type, is_active)    -- 種別絞り込み
```

**制約・備考**
- `scope_type` + `scope_id` はポリモーフィック参照。FK 制約なし（アプリ層で検証）
- `name` は同一スコープ内で一意。論理削除後は同一名で再作成可能（`deleted_at` を UNIQUE KEY に含めることで対応）
- `facility_type_label`: `facility_type = OTHER` 以外でも補足情報として使用可能（例: `MEETING_ROOM` + `'防音仕様'`）
- `hourly_rate` と `facility_time_rates` の優先順位: `facility_time_rates` にレコードが存在する場合はそちらの `rate_per_slot` を適用。存在しない場合は `hourly_rate` をフォールバックとして使用。両方 NULL の場合は無料
- `cleaning_buffer_minutes` は30分単位でなくてもよい（15分等も設定可能）。予約の空き判定時に `time_to + cleaning_buffer_minutes` を次の予約可能開始時刻として計算
- `image_urls` は JSON 配列で管理。S3 Pre-signed URL を格納し、フロントエンドで直接表示。最大5枚はService層でバリデーション
- `is_active = FALSE` の施設は一覧に表示されるが「利用停止中」のラベルを付与。新規予約は不可。既存の CONFIRMED 予約は有効のまま維持（ADMIN が個別にキャンセル判断）
- `capacity` は予約時の参考情報として表示。予約数による自動制限は行わない（同一時間帯に1予約のみ = 施設の排他利用が基本）
- 論理削除: 予約履歴の参照先として必要。物理削除不可
- F09.1 住民台帳の `dwelling_units.unit_type = 'COMMON'` は共用スペースの居室管理（部屋番号・面積等）。本テーブルは予約管理に特化した施設情報（利用ルール・料金・備品等）を管理。両テーブルは独立して運用し、必要に応じて `location_detail` で居室番号を参照

#### `facility_usage_rules`

施設ごとの利用ルール。予約可能な時間帯・利用制限を管理する。1施設に1レコード。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `facility_id` | BIGINT UNSIGNED | NO | — | FK → shared_facilities。ON DELETE CASCADE |
| `max_hours_per_booking` | DECIMAL(3,1) | NO | 4.0 | 1予約あたりの最大利用時間（時間単位。例: 4.0 = 4時間 = 8スロット） |
| `min_hours_per_booking` | DECIMAL(3,1) | NO | 0.5 | 1予約あたりの最小利用時間（時間単位。0.5 = 30分 = 1スロット） |
| `max_bookings_per_month_per_user` | SMALLINT UNSIGNED | NO | 4 | 1ユーザーあたりの月間予約上限（0 = 無制限） |
| `max_consecutive_slots` | SMALLINT UNSIGNED | NO | 8 | 連続予約可能な最大スロット数（8 = 4時間分。0 = 無制限） |
| `min_advance_hours` | SMALLINT UNSIGNED | NO | 1 | 最低何時間前に予約が必要か（0 = 当日直前でも可） |
| `max_advance_days` | SMALLINT UNSIGNED | NO | 30 | 最大何日先まで予約可能か |
| `available_time_from` | TIME | NO | '09:00:00' | 利用可能開始時刻 |
| `available_time_to` | TIME | NO | '22:00:00' | 利用可能終了時刻 |
| `available_days_of_week` | VARCHAR(20) | NO | '0,1,2,3,4,5,6' | 利用可能曜日（カンマ区切り。0=日〜6=土。例: '1,2,3,4,5' = 平日のみ） |
| `blackout_dates` | JSON | YES | NULL | 利用不可日リスト（JSON配列。例: `["2026-12-31", "2027-01-01"]`。年末年始・定期点検日等） |
| `notes` | VARCHAR(500) | YES | NULL | 利用ルールの補足説明（フロントエンドで表示） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_fur_facility (facility_id)  -- 1施設1ルール
```

**制約・備考**
- 1施設1ルール（UNIQUE KEY で保証）。ルール未作成時はデフォルト値（max_hours_per_booking=4, max_bookings_per_month_per_user=4, max_advance_days=30, 09:00-22:00, 全曜日）をService層で適用
- `max_hours_per_booking` と `max_consecutive_slots` の関係: `max_consecutive_slots = max_hours_per_booking * 2`（30分スロット換算）が推奨だが、独立して設定可能。予約時は両方の制約を満たす必要がある
- `available_days_of_week`: カンマ区切りの曜日番号。Java の `DayOfWeek` ではなく `Calendar.DAY_OF_WEEK` 互換（0=日曜）。Service 層で `Set<Integer>` にパースして検証
- `blackout_dates`: JSON 配列で管理。年次の繰り返し（毎年1/1等）は未対応（毎年手動更新が必要）。将来的に `blackout_patterns`（例: `"every_year:01-01"`）の拡張を検討
- `min_advance_hours = 0` の場合: 現在時刻以降のスロットであれば即時予約可能
- `max_bookings_per_month_per_user = 0` の場合: 月間上限なし（無制限）
- バリデーション: `available_time_from < available_time_to`、`min_hours_per_booking <= max_hours_per_booking`、`max_advance_days >= 1`

#### `facility_time_rates`

施設の時間帯別料金設定。曜日種別・時間帯ごとに30分スロット単価を管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `facility_id` | BIGINT UNSIGNED | NO | — | FK → shared_facilities。ON DELETE CASCADE |
| `day_type` | ENUM('WEEKDAY', 'WEEKEND', 'HOLIDAY') | NO | — | 曜日種別（平日 / 土日 / 祝日） |
| `time_from` | TIME | NO | — | 適用開始時刻（30分単位） |
| `time_to` | TIME | NO | — | 適用終了時刻（30分単位） |
| `rate_per_slot` | DECIMAL(10,0) | NO | — | 30分スロット単価（円） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_ftr_facility (facility_id, day_type, time_from)                  -- 施設の料金検索
UNIQUE KEY uq_ftr_facility_day_time (facility_id, day_type, time_from)     -- 同一施設・曜日種別・時刻の重複防止
```

**制約・備考**
- 1施設に対して複数の料金レコードを登録可能（曜日種別 × 時間帯の組み合わせ）
- `day_type` の判定: `HOLIDAY` は祝日マスタ（将来的に `holidays` テーブルを参照。Phase 9 時点ではアプリケーション定数で管理）。`WEEKEND` は土日。`WEEKDAY` は月〜金（祝日でない場合）
- 判定優先順位: `HOLIDAY` > `WEEKEND` > `WEEKDAY`。祝日が土曜の場合は `HOLIDAY` レートを適用
- `time_from` / `time_to` は30分単位で指定（例: 09:00, 09:30, 10:00, ...）
- 料金設定例（集会室の場合）:
  - WEEKDAY / 09:00-18:00: 1,000円/スロット
  - WEEKDAY / 18:00-22:00: 1,500円/スロット（夜間割増）
  - WEEKEND / 09:00-22:00: 1,500円/スロット
  - HOLIDAY / 09:00-22:00: 2,000円/スロット
- 未設定の時間帯は `shared_facilities.hourly_rate` をフォールバック。それも NULL なら無料
- 予約の合計料金計算: 予約の各30分スロットについて該当する `rate_per_slot` を合算。例: 平日 17:00-19:00（4スロット）= 17:00-18:00 の2スロット × 1,000円 + 18:00-19:00 の2スロット × 1,500円 = 5,000円

#### `facility_equipment`

施設に設置・貸出可能な備品。在庫数量を管理し、予約時に備品を選択・数量指定できる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `facility_id` | BIGINT UNSIGNED | NO | — | FK → shared_facilities。ON DELETE CASCADE |
| `name` | VARCHAR(100) | NO | — | 備品名（例: 'プロジェクター', '折りたたみ椅子', 'BBQグリル', '食器セット（5人分）'） |
| `description` | VARCHAR(500) | YES | NULL | 備品説明（使用上の注意等） |
| `total_quantity` | SMALLINT UNSIGNED | NO | 1 | 総在庫数 |
| `price_per_use` | DECIMAL(10,0) | YES | NULL | 1個あたりの利用料金（円/回）。NULL = 無料 |
| `is_available` | BOOLEAN | NO | TRUE | 貸出可能か（FALSE = 故障中・メンテナンス等） |
| `display_order` | SMALLINT UNSIGNED | NO | 0 | 表示順 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_fe_facility (facility_id, is_available, display_order)           -- 施設の備品一覧
UNIQUE KEY uq_fe_facility_name (facility_id, name, deleted_at)             -- 同一施設内の備品名重複防止
```

**制約・備考**
- `total_quantity`: 施設が保有する備品の総数。予約時に指定可能な数量は「総数 − 同一時間帯の他予約で確保済み数量」で動的に算出
- 在庫確認クエリ: 予約の時間帯（`time_from` 〜 `time_to + cleaning_buffer_minutes`）と重複する PENDING_APPROVAL / CONFIRMED / CHECKED_IN 予約で確保されている数量を合算し、`total_quantity` から差し引く
- `price_per_use` は「1個あたり・1予約あたり」の料金。時間に比例しない（プロジェクター1台を2時間使っても4時間使っても同一料金）
- `is_available = FALSE` の備品は一覧に表示されるが「利用不可」のラベルを付与。新規予約での選択不可
- 論理削除: 予約履歴の参照先として必要。既存予約で使用中の備品は削除不可（409 エラー）

#### `facility_bookings`

施設予約の本体。F09.3 の `parking_visitor_reservations` と同様の承認制フロー・30分スロット・ステータス遷移を採用。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `facility_id` | BIGINT UNSIGNED | NO | — | FK → shared_facilities。ON DELETE RESTRICT |
| `booked_by` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE。予約者 |
| `booking_date` | DATE | NO | — | 利用日 |
| `time_from` | TIME | NO | — | 利用開始時刻（30分単位） |
| `time_to` | TIME | NO | — | 利用終了時刻（30分単位） |
| `slot_count` | SMALLINT UNSIGNED | NO | — | 利用スロット数（非正規化。`(time_to - time_from) / 30分`） |
| `purpose` | VARCHAR(500) | YES | NULL | 利用目的（例: '自治会総会', '誕生日パーティー', 'ヨガ教室'） |
| `attendee_count` | SMALLINT UNSIGNED | YES | NULL | 予定参加人数（任意。定員超過は警告のみ、ブロックしない） |
| `usage_fee` | DECIMAL(10,0) | NO | 0 | 施設利用料（円）。時間帯別料金の合算。0 = 無料 |
| `equipment_fee` | DECIMAL(10,0) | NO | 0 | 備品利用料（円）。備品の `price_per_use * quantity` の合算。0 = 備品なし or 無料 |
| `total_fee` | DECIMAL(10,0) | NO | 0 | 合計利用料（`usage_fee + equipment_fee`）。0 = 無料 |
| `status` | ENUM('PENDING_APPROVAL', 'CONFIRMED', 'CHECKED_IN', 'COMPLETED', 'CANCELLED', 'REJECTED', 'NO_SHOW') | NO | 'PENDING_APPROVAL' | 予約ステータス |
| `admin_comment` | VARCHAR(500) | YES | NULL | ADMIN のコメント（承認/却下理由。任意） |
| `approved_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL。承認した管理者 |
| `approved_at` | DATETIME | YES | NULL | 承認日時 |
| `checked_in_at` | DATETIME | YES | NULL | チェックイン日時 |
| `completed_at` | DATETIME | YES | NULL | 利用完了日時 |
| `cancelled_at` | DATETIME | YES | NULL | キャンセル日時 |
| `cancelled_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL。キャンセル実行者 |
| `cancellation_reason` | VARCHAR(500) | YES | NULL | キャンセル理由 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_fb_facility_date (facility_id, booking_date, time_from, time_to)  -- 施設別の予約一覧・重複チェック
INDEX idx_fb_user (booked_by, booking_date)                                  -- ユーザーの予約一覧
INDEX idx_fb_date_status (booking_date, status)                              -- 日別の予約管理・NO_SHOW検出バッチ
INDEX idx_fb_scope_status (facility_id, status, booking_date DESC)           -- 管理者の予約管理画面
```

**制約・備考**
- `time_from` / `time_to` は **30分単位** で指定（例: 09:00, 09:30, 10:00, ...）。30分刻みでない値は 400 エラー（Service 層で `minutes % 30 == 0` を検証）
- 最小予約時間: `facility_usage_rules.min_hours_per_booking`（デフォルト: 30分 = 1スロット）
- 時間帯の重複チェック: 同一 `facility_id` + `booking_date` で、予約時間帯 + 清掃バッファ（`cleaning_buffer_minutes`）が重複する **PENDING_APPROVAL / CONFIRMED / CHECKED_IN** の予約が存在する場合は 409（仮予約でも枠を確保する）
- 重複チェック SQL:
  ```sql
  SELECT COUNT(*) FROM facility_bookings fb
  JOIN shared_facilities sf ON sf.id = fb.facility_id
  WHERE fb.facility_id = :facilityId
    AND fb.booking_date = :bookingDate
    AND fb.status IN ('PENDING_APPROVAL', 'CONFIRMED', 'CHECKED_IN')
    AND fb.time_from < :timeTo
    AND ADDTIME(fb.time_to, SEC_TO_TIME(sf.cleaning_buffer_minutes * 60)) > :timeFrom
    AND fb.id != :excludeBookingId
  ```
- `status` 遷移: `PENDING_APPROVAL`（MEMBER が申請）→ `CONFIRMED`（ADMIN が承認）/ `REJECTED`（ADMIN が却下）/ `CANCELLED`（申請者がキャンセル）。`CONFIRMED` → `CHECKED_IN`（利用開始。予約者本人 or ADMIN が操作）/ `CANCELLED`（キャンセル）/ `NO_SHOW`（翌日バッチ）。`CHECKED_IN` → `COMPLETED`（利用終了。予約者本人 or ADMIN が操作）
- `slot_count` の非正規化: レスポンス生成・統計集計で便利なため保持。`INSERT` 時に `(time_to - time_from) / 30分` で自動計算
- `usage_fee` の計算: 予約の各30分スロットについて `facility_time_rates` を参照し合算。Service 層の `FacilityFeeCalculator` が担当
- `attendee_count` が `shared_facilities.capacity` を超過する場合: レスポンスに `"warning": "定員（{capacity}名）を超過しています"` を含めるが予約はブロックしない（利用者の自己責任）
- NO_SHOW 検出: 翌日0時のバッチで `booking_date < TODAY AND status = 'CONFIRMED'` を `NO_SHOW` に自動更新。`PENDING_APPROVAL` のまま期日を過ぎた予約は `CANCELLED` に自動更新
- `facility_id` ON DELETE RESTRICT: 施設削除時に予約履歴が存在する場合はエラー。論理削除を使用
- `booked_by` ON DELETE CASCADE: ユーザー退会時に予約もカスケード削除

#### `facility_booking_equipment`

予約ごとの備品予約。中間テーブルとして予約と備品を紐付け、予約時の数量を管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `booking_id` | BIGINT UNSIGNED | NO | — | FK → facility_bookings。ON DELETE CASCADE |
| `equipment_id` | BIGINT UNSIGNED | NO | — | FK → facility_equipment。ON DELETE RESTRICT |
| `quantity` | SMALLINT UNSIGNED | NO | 1 | 予約数量 |
| `unit_price` | DECIMAL(10,0) | NO | 0 | 予約時点の1個あたり料金（`facility_equipment.price_per_use` のスナップショット。料金変更の遡及防止） |
| `subtotal` | DECIMAL(10,0) | NO | 0 | 小計（`unit_price * quantity`） |

**インデックス**
```sql
UNIQUE KEY uq_fbe_booking_equip (booking_id, equipment_id)  -- 同一予約内で同一備品の重複防止
INDEX idx_fbe_equipment (equipment_id)                        -- 備品の使用状況検索
```

**制約・備考**
- `quantity` は予約時に `facility_equipment.total_quantity` − 同一時間帯の確保済み数量 の範囲内であることをService層でバリデーション
- `unit_price` は予約作成時の `facility_equipment.price_per_use` をスナップショット保存。料金変更後も既存予約の金額は変わらない
- `equipment_id` ON DELETE RESTRICT: 予約で使用中の備品は削除不可
- 予約キャンセル時: `facility_booking_equipment` レコードは残置（履歴保持）。在庫は予約の `status` が CANCELLED/REJECTED/NO_SHOW に遷移した時点で自動的に解放される（在庫計算が PENDING_APPROVAL/CONFIRMED/CHECKED_IN のみを対象とするため）

#### `facility_booking_payments`

施設利用料の決済記録。F09.3 の `parking_sublease_payments` と同パターンで DIRECT/STRIPE の選択制。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `booking_id` | BIGINT UNSIGNED | NO | — | FK → facility_bookings。ON DELETE RESTRICT |
| `payer_user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT。支払い者（= 予約者） |
| `payment_method` | ENUM('DIRECT', 'STRIPE') | NO | 'DIRECT' | 決済方法（DIRECT=フロント/現金等で直接支払い / STRIPE=Stripe 経由のオンライン決済） |
| `amount` | DECIMAL(10,0) | NO | — | 支払い金額（= `facility_bookings.total_fee`） |
| `stripe_fee` | DECIMAL(10,0) | YES | NULL | Stripe 決済手数料（円）。STRIPE の場合のみ。`amount * 3.6% + 40` |
| `platform_fee` | DECIMAL(10,0) | YES | NULL | Mannschaft プラットフォーム手数料（円）。STRIPE の場合のみ |
| `platform_fee_rate` | DECIMAL(5,4) | YES | NULL | 適用された手数料率（例: 0.0500 = 5%）。請求時点のレートを記録 |
| `net_amount` | DECIMAL(10,0) | YES | NULL | 施設管理者の受取金額（STRIPE の場合: `amount - stripe_fee - platform_fee`） |
| `stripe_payment_intent_id` | VARCHAR(100) | YES | NULL | Stripe PaymentIntent ID |
| `status` | ENUM('PENDING', 'SUCCEEDED', 'FAILED', 'REFUNDED') | NO | 'PENDING' | 決済ステータス |
| `failed_reason` | VARCHAR(500) | YES | NULL | 失敗理由（FAILED 時） |
| `paid_at` | DATETIME | YES | NULL | 支払い完了日時 |
| `refunded_at` | DATETIME | YES | NULL | 返金日時（REFUNDED 時） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_fbp_booking (booking_id)                       -- 1予約1決済レコード
INDEX idx_fbp_payer (payer_user_id, status)                   -- ユーザーの決済一覧
INDEX idx_fbp_status (status, created_at)                     -- FAILED の再試行対象検索
```

**制約・備考**
- `total_fee > 0` の予約にのみ決済レコードを作成。`total_fee = 0`（無料施設）の場合はレコードなし
- `payment_method` の決定: `facility_settings.allow_stripe_payment = TRUE` の場合、予約者が DIRECT / STRIPE を選択可能。`FALSE` の場合は常に DIRECT
- STRIPE 決済フロー:
  1. 予約承認時（CONFIRMED 遷移時）に Stripe PaymentIntent を作成
  2. フロントエンドで Stripe Elements による決済UI表示
  3. 決済成功で `status = SUCCEEDED`、`paid_at` を設定
  4. 決済失敗で `status = FAILED`、`failed_reason` を記録。24時間以内にリトライ可能。リトライ失敗で予約自動キャンセル
- DIRECT の場合: `status` は `PENDING` のまま保持。利用当日にフロントで支払い確認後、ADMIN が手動で `SUCCEEDED` に更新
- キャンセル時の返金: STRIPE + SUCCEEDED の予約がキャンセルされた場合、Stripe Refund API で全額返金 → `status = REFUNDED`、`refunded_at` を設定
- `platform_fee_rate` は F08.2 の `payment_settings.platform_fee_rate` から取得。請求時点のレートを記録して変更の遡及を防止
- Stripe Destination Charge パターン: F09.3 サブリースと同一アーキテクチャ。組織/チームの Stripe Connect アカウントに施設利用料を送金し、`application_fee_amount` でプラットフォーム手数料を徴収

**処理の流れ**
1. **施設の有効性チェック**: `is_active = TRUE` を検証（FALSE なら 422）
2. **利用ルールのバリデーション**:
   - `time_from` / `time_to` が30分単位であること
   - `time_from < time_to` であること
   - 利用時間が `min_hours_per_booking` 〜 `max_hours_per_booking` の範囲内
   - スロット数が `max_consecutive_slots` 以内
   - `booking_date` が `min_advance_hours` 〜 `max_advance_days` の範囲内
   - `booking_date` の曜日が `available_days_of_week` に含まれる
   - `booking_date` が `blackout_dates` に含まれない
   - `time_from >= available_time_from` かつ `time_to <= available_time_to`
3. **月間予約上限チェック**: 当月の `booked_by` = currentUser の予約数（CANCELLED/REJECTED/NO_SHOW 除く）が `max_bookings_per_month_per_user` 以内
4. **同日予約上限チェック**: `booking_date` の全施設合計予約数が `facility_settings.max_bookings_per_day_per_user` 以内
5. **NO_SHOW ペナルティチェック**: `no_show_penalty_enabled = TRUE` の場合、直近30日の NO_SHOW 回数を検証
6. **悲観ロック取得**: `SELECT ... FOR UPDATE` で `shared_facilities` を行ロック
7. **時間帯重複チェック**: 清掃バッファを含む時間帯で既存予約（PENDING_APPROVAL/CONFIRMED/CHECKED_IN）との重複を検証
8. **備品在庫チェック**: 各備品について、同一時間帯の確保済み数量 + リクエスト数量 が `total_quantity` 以内を検証
9. **料金計算**: `FacilityFeeCalculator` で `usage_fee` + `equipment_fee` を算出
10. **予約レコード作成**: `facility_bookings` + `facility_booking_equipment` を INSERT
11. **自動承認チェック**: `facility_settings.requires_approval = FALSE` の場合、即座に `CONFIRMED` に遷移
12. **通知送信**: ADMIN に `FACILITY_BOOKING_REQUESTED` 通知（承認必要時）
13. **定員超過警告**: `attendee_count > capacity` の場合、`warnings` に含める

**遷移ルール**
- `PENDING_APPROVAL` → `CONFIRMED`: ADMIN が承認。`requires_approval = FALSE` の場合、予約作成時に自動遷移
- `PENDING_APPROVAL` → `REJECTED`: ADMIN が却下。枠 + 備品在庫を解放
- `PENDING_APPROVAL` → `CANCELLED`: 予約者本人 or ADMIN。いつでもキャンセル可能
- `CONFIRMED` → `CHECKED_IN`: 予約者本人 or ADMIN。利用開始操作
- `CONFIRMED` → `CANCELLED`: キャンセルポリシーに従う（後述）
- `CONFIRMED` → `NO_SHOW`: 翌日バッチで自動検出
- `CHECKED_IN` → `COMPLETED`: 予約者本人 or ADMIN。利用終了操作
- 終端ステータス（`COMPLETED` / `CANCELLED` / `REJECTED` / `NO_SHOW`）からの遷移は不可

### 利用料金計算ロジック

`FacilityFeeCalculator` が予約の各30分スロットについて適用料金を決定し、合計を算出する。

```java
@Component
public class FacilityFeeCalculator {

    private final HolidayResolver holidayResolver;

    /**
     * 施設利用料を計算する
     * @param facility      施設情報
     * @param rates         時間帯別料金リスト
     * @param bookingDate   利用日
     * @param timeFrom      利用開始時刻
     * @param timeTo        利用終了時刻
     * @return 料金明細（スロット別料金 + 合計）
     */
    public FeeBreakdown calculateUsageFee(
            SharedFacility facility,
            List<FacilityTimeRate> rates,
            LocalDate bookingDate,
            LocalTime timeFrom,
            LocalTime timeTo) {

        // 1. 曜日種別の判定（優先順位: HOLIDAY > WEEKEND > WEEKDAY）
        DayType dayType = resolveDayType(bookingDate);

        // 2. 該当 dayType の料金レコードを抽出
        List<FacilityTimeRate> applicableRates = rates.stream()
            .filter(r -> r.getDayType() == dayType)
            .sorted(Comparator.comparing(FacilityTimeRate::getTimeFrom))
            .toList();

        // 3. 30分スロットごとに料金を決定
        List<SlotFee> slotFees = new ArrayList<>();
        LocalTime cursor = timeFrom;
        while (cursor.isBefore(timeTo)) {
            LocalTime slotEnd = cursor.plusMinutes(30);
            BigDecimal rate = findRate(applicableRates, cursor, facility.getHourlyRate());
            slotFees.add(new SlotFee(cursor, slotEnd, rate, dayType));
            cursor = slotEnd;
        }

        // 4. 合計を算出
        BigDecimal usageSubtotal = slotFees.stream()
            .map(SlotFee::rate)
            .reduce(BigDecimal.ZERO, BigDecimal::add);

        return new FeeBreakdown(slotFees, usageSubtotal);
    }

    /**
     * スロットに適用する料金を検索
     * 1. facility_time_rates で該当する時間帯があればその rate_per_slot
     * 2. なければ shared_facilities.hourly_rate（フォールバック）
     * 3. 両方 NULL なら 0（無料）
     */
    private BigDecimal findRate(
            List<FacilityTimeRate> rates,
            LocalTime slotStart,
            BigDecimal fallbackRate) {

        return rates.stream()
            .filter(r -> !slotStart.isBefore(r.getTimeFrom())
                      && slotStart.isBefore(r.getTimeTo()))
            .findFirst()
            .map(FacilityTimeRate::getRatePerSlot)
            .orElse(fallbackRate != null ? fallbackRate : BigDecimal.ZERO);
    }

    /**
     * 曜日種別の判定
     */
    private DayType resolveDayType(LocalDate date) {
        if (holidayResolver.isHoliday(date)) return DayType.HOLIDAY;
        DayOfWeek dow = date.getDayOfWeek();
        if (dow == DayOfWeek.SATURDAY || dow == DayOfWeek.SUNDAY) return DayType.WEEKEND;
        return DayType.WEEKDAY;
    }
}
```

**料金計算の具体例**

```
■ 例1: 平日 10:00-14:00（集会室。WEEKDAY 09:00-18:00 = 1,000円/スロット）
  10:00-10:30 = 1,000円
  10:30-11:00 = 1,000円
  11:00-11:30 = 1,000円
  11:30-12:00 = 1,000円
  12:00-12:30 = 1,000円
  12:30-13:00 = 1,000円
  13:00-13:30 = 1,000円
  13:30-14:00 = 1,000円
  合計: 8 × 1,000 = 8,000円

■ 例2: 平日 17:00-20:00（昼間→夜間をまたぐ。WEEKDAY 09:00-18:00=1,000円, 18:00-22:00=1,500円）
  17:00-17:30 = 1,000円（WEEKDAY_DAYTIME）
  17:30-18:00 = 1,000円（WEEKDAY_DAYTIME）
  18:00-18:30 = 1,500円（WEEKDAY_EVENING）
  18:30-19:00 = 1,500円（WEEKDAY_EVENING）
  19:00-19:30 = 1,500円（WEEKDAY_EVENING）
  19:30-20:00 = 1,500円（WEEKDAY_EVENING）
  合計: 2 × 1,000 + 4 × 1,500 = 8,000円

■ 例3: 祝日 09:00-12:00（HOLIDAY 09:00-22:00=2,000円/スロット）
  6 × 2,000 = 12,000円

■ 例4: 無料施設（hourly_rate=NULL、facility_time_rates レコードなし）
```

```java
/**
 * 備品利用料を計算する
 * @param equipmentRequests 備品予約リクエスト（equipment_id + quantity）
 * @return 備品料金合計
 */
public BigDecimal calculateEquipmentFee(List<EquipmentRequest> equipmentRequests) {
    BigDecimal total = BigDecimal.ZERO;
    for (EquipmentRequest req : equipmentRequests) {
        FacilityEquipment equip = equipmentRepository.findById(req.equipmentId())
            .orElseThrow(() -> new NotFoundException("EQUIPMENT_NOT_FOUND"));

        BigDecimal unitPrice = equip.getPricePerUse() != null
            ? equip.getPricePerUse() : BigDecimal.ZERO;
        BigDecimal subtotal = unitPrice.multiply(BigDecimal.valueOf(req.quantity()));
        total = total.add(subtotal);
    }
    return total;
}
```

### 予約作成時の悲観ロック

同一施設・同一時間帯への同時予約を排他制御するため、予約作成時に `shared_facilities` に対して `SELECT ... FOR UPDATE` を実行する。

```java
@Transactional
public FacilityBooking createBooking(Long teamId, CreateBookingRequest request) {
    // 1. 施設を悲観ロックで取得
    SharedFacility facility = facilityRepository
        .findByIdForUpdate(request.getFacilityId())
        .orElseThrow(() -> new NotFoundException("FACILITY_NOT_FOUND"));

    // 2. 施設の有効性チェック
    if (!facility.getIsActive()) {
        throw new UnprocessableEntityException("FACILITY_INACTIVE",
            "この施設は現在利用停止中です");
    }

    // 3. 利用ルールのバリデーション
    FacilityUsageRules rules = rulesRepository
        .findByFacilityId(facility.getId())
        .orElse(FacilityUsageRules.defaults());
    validateBookingRules(rules, request);

    // 4. 重複チェック
    long overlapCount = bookingRepository.countOverlapping(
        facility.getId(),
        request.getBookingDate(),
        request.getTimeFrom(),
        request.getTimeTo(),
        facility.getCleaningBufferMinutes(),
        null  // excludeBookingId
    );
    if (overlapCount > 0) {
        throw new ConflictException("TIME_SLOT_CONFLICT",
            "指定された時間帯は既に予約されています");
    }

    // 5. 備品在庫チェック（悲観ロック）
    if (request.getEquipment() != null && !request.getEquipment().isEmpty()) {
        validateEquipmentAvailability(
            facility, request.getBookingDate(),
            request.getTimeFrom(), request.getTimeTo(),
            request.getEquipment()
        );
    }

    // 6. 料金計算
    List<FacilityTimeRate> rates = rateRepository
        .findByFacilityId(facility.getId());
    FeeBreakdown usageFee = feeCalculator.calculateUsageFee(
        facility, rates,
        request.getBookingDate(),
        request.getTimeFrom(), request.getTimeTo()
    );
    BigDecimal equipmentFee = feeCalculator
        .calculateEquipmentFee(request.getEquipment());
    BigDecimal totalFee = usageFee.total().add(equipmentFee);

    // 7. 予約レコード作成
    FacilityBooking booking = FacilityBooking.builder()
        .facilityId(facility.getId())
        .bookedBy(currentUser.getId())
        .bookingDate(request.getBookingDate())
        .timeFrom(request.getTimeFrom())
        .timeTo(request.getTimeTo())
        .slotCount(calculateSlotCount(request.getTimeFrom(), request.getTimeTo()))
        .purpose(request.getPurpose())
        .attendeeCount(request.getAttendeeCount())
        .usageFee(usageFee.total())
        .equipmentFee(equipmentFee)
        .totalFee(totalFee)
        .status(resolveInitialStatus(teamId))
        .build();
    bookingRepository.save(booking);

    // 8. 備品予約レコード作成
    saveBookingEquipment(booking, request.getEquipment());

    // 9. 自動承認チェック
    FacilitySettings settings = settingsRepository
        .findByScopeTypeAndScopeId("TEAM", teamId)
        .orElse(FacilitySettings.defaults());
    if (!settings.getRequiresApproval()) {
        booking.confirm(currentUser.getId());
    }

    // 10. 通知送信
    if (booking.getStatus() == BookingStatus.PENDING_APPROVAL) {
        notificationService.notifyAdmins(teamId,
            NotificationType.FACILITY_BOOKING_REQUESTED, booking);
    }

    return booking;
}
```

### NO_SHOW 検出バッチ

毎日0時に実行するバッチで、前日以前の未処理予約を自動処理する。

```java
@Scheduled(cron = "0 0 0 * * *")
@Transactional
public void detectNoShows() {
    LocalDate today = LocalDate.now();

    // 1. CONFIRMED のまま予約日を過ぎた予約 → NO_SHOW
    int noShowCount = bookingRepository.updateStatusByDateAndCurrentStatus(
        today,                          // booking_date < today
        BookingStatus.CONFIRMED,        // 現在のステータス
        BookingStatus.NO_SHOW           // 更新先ステータス
    );

    // 2. PENDING_APPROVAL のまま予約日を過ぎた予約 → CANCELLED（自動失効）
    int cancelledCount = bookingRepository.updateStatusByDateAndCurrentStatus(
        today,
        BookingStatus.PENDING_APPROVAL,
        BookingStatus.CANCELLED
    );

    log.info("NO_SHOW batch completed: {} no-shows, {} auto-cancelled",
        noShowCount, cancelledCount);
}
```

```java
/**
 * キャンセルポリシーの判定
 */
public void validateCancellation(FacilityBooking booking, Long currentUserId, boolean isAdmin) {
    switch (booking.getStatus()) {
        case PENDING_APPROVAL:
            // 予約者本人 or ADMIN: いつでもキャンセル可能
            if (!isAdmin && !booking.getBookedBy().equals(currentUserId)) {
                throw new ForbiddenException("他人の予約はキャンセルできません");
            }
            break;

        case CONFIRMED:
            if (isAdmin) break;  // ADMIN はいつでも可能

            if (!booking.getBookedBy().equals(currentUserId)) {
                throw new ForbiddenException("他人の予約はキャンセルできません");
            }

            // 期限チェック
            FacilitySettings settings = getSettings(booking);
            LocalDateTime bookingStart = LocalDateTime.of(
                booking.getBookingDate(), booking.getTimeFrom());
            LocalDateTime deadline = bookingStart
                .minusHours(settings.getCancellationDeadlineHours());

            if (LocalDateTime.now().isAfter(deadline)) {
                throw new ForbiddenException(String.format(
                    "利用開始の%d時間前を過ぎたため、キャンセルできません。管理者にお問い合わせください。",
                    settings.getCancellationDeadlineHours()));
            }
            break;

        case CHECKED_IN:
            if (!isAdmin) {
                throw new ForbiddenException("利用中のキャンセルは管理者のみ実行可能です");
            }
            break;

        default:
            throw new ConflictException("INVALID_STATUS",
                "この予約はキャンセルできません（ステータス: " + booking.getStatus() + "）");
    }
}
```

### Stripe 決済フロー

```
1. MEMBER が予約申請（payment_method = STRIPE）
   └─ status = PENDING_APPROVAL
   └─ 決済はまだ発生しない

2. ADMIN が承認（CONFIRMED 遷移時）
   ├─ Stripe PaymentIntent を作成
   │   └─ amount = total_fee
   │   └─ application_fee_amount = platform_fee（Mannschaft 手数料）
   │   └─ transfer_data.destination = 組織/チームの Connect アカウント
   ├─ facility_booking_payments に PENDING レコード INSERT
   └─ フロントエンドに client_secret を返却

3. フロントエンドで Stripe Elements 決済UI表示
   ├─ ユーザーがカード情報を入力
   └─ PaymentIntent を confirm

4. Stripe Webhook: payment_intent.succeeded
   ├─ facility_booking_payments.status → SUCCEEDED
   ├─ paid_at を設定
   └─ 手数料内訳を計算して記録:
       ├─ stripe_fee = amount × 3.6% + ¥40
       ├─ platform_fee = amount × platform_fee_rate
       └─ net_amount = amount - stripe_fee - platform_fee

5. Stripe Webhook: payment_intent.payment_failed
   ├─ facility_booking_payments.status → FAILED
   ├─ failed_reason を記録
   └─ 24時間以内にリトライ可能。リトライ失敗で予約自動キャンセル

6. キャンセル時（STRIPE + SUCCEEDED）
   ├─ Stripe Refund API で全額返金
   ├─ facility_booking_payments.status → REFUNDED
   └─ refunded_at を設定
```

**手数料計算例**
```
施設利用料: 10,000円（= total_fee）
  ├─ Stripe 決済手数料:     400円（10,000 × 3.6% + ¥40 = 400円）→ Stripe が徴収
  ├─ プラットフォーム手数料:   500円（10,000 × 5% = 500円）→ Mannschaft に入金
  └─ 施設管理者の受取:       9,100円（10,000 - 400 - 500 = 9,100円）→ Connect アカウント経由
```

## 6. セキュリティ考慮事項

- **認可チェック**: 施設管理操作（CRUD・ルール設定・料金設定・備品管理・予約の承認/却下）は ADMIN / DEPUTY_ADMIN（`MANAGE_FACILITIES`）限定。Service 層入り口でスコープメンバーシップ + ロール + パーミッションを検証
- **スコープ分離**: 全 API で `teamId` パスパラメータとリクエストユーザーのスコープメンバーシップを検証。他チーム/組織の施設情報へのアクセスを防止
- **予約者本人チェック**: 予約のチェックイン・完了・キャンセルは予約者本人 or ADMIN のみ。他人の予約への操作は 403 エラー
- **プライバシー保護（予約情報）**: MEMBER は自分の予約のみ閲覧可能。空き状況の `booking` 情報では他人の予約者名を `"予約済み"` にマスキング。ADMIN は全予約者名が表示
- **レート制限**: Bucket4j による API レート制限
  - 予約作成 API: 1ユーザーあたり1分間に10回
  - 施設登録 API: 1ユーザーあたり1分間に20回
  - 一括登録 API: 1ユーザーあたり1分間に5回
- **一括登録のサイズ制限**: `bulk-create` は最大50件。DoS 防止
- **悲観ロック**: 予約作成時に `shared_facilities` を `FOR UPDATE` でロックし、同一時間帯への同時予約を排他制御。デッドロック防止のため常に `facility_id` 昇順でロック取得
- **備品在庫の整合性**: 備品在庫チェックと予約作成をトランザクション内で実行し、在庫のオーバーブッキングを防止
- **Stripe 決済セキュリティ**: F08.2 と同一のセキュリティ要件を適用。PaymentIntent の `client_secret` はフロントエンドにのみ返却。Webhook の署名検証。カード情報は Stripe 側で管理（PCI DSS 準拠）
- **料金改ざん防止**: 料金は常にサーバー側で計算。フロントエンドから送信された金額は使用しない。`facility_booking_equipment.unit_price` は予約時点のスナップショット
- **監査ログ**: 以下の操作を `audit_logs`（F10.3）に記録:
  - `FACILITY_CREATED` / `FACILITY_UPDATED` / `FACILITY_DELETED`（施設管理）
  - `FACILITY_BOOKING_CREATED` / `FACILITY_BOOKING_APPROVED` / `FACILITY_BOOKING_REJECTED`（予約管理）
  - `FACILITY_BOOKING_CANCELLED`（キャンセル。キャンセル理由含む）
  - `FACILITY_PAYMENT_CONFIRMED`（DIRECT 決済の手動確認）
  - `FACILITY_SETTINGS_UPDATED`（設定変更）
