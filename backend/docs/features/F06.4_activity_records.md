## 1. 概要

チーム/組織の活動内容を構造化データとして記録・蓄積する機能。試合結果・練習記録・イベント報告・会議議事録など、業種に応じた記録項目をカスタムフィールドで自由に定義し、統計・集計機能で活動傾向を可視化する。

CMS のブログ・お知らせ（F06.1）が「エディトリアルコンテンツ（自由記述の記事）」であるのに対し、活動記録は「構造化データ（定義済みフィールドに値を記録）」として管理する。

**主要ユースケース**:
- スポーツチーム: 試合結果（スコア・対戦相手・会場）、練習記録（メニュー・参加者・時間）
- 学校・PTA: 授業記録、行事報告、PTA 活動報告
- マンション管理組合: 理事会議事録、定期点検記録、修繕工事記録
- 整骨院・美容室: 日報・業務記録（施術件数・売上等の集計用）
- コミュニティ: イベント報告、ボランティア活動記録

### 本ドキュメントのスコープ

| 機能 | 説明 |
|------|------|
| 活動テンプレート | チームごとにカスタムフィールドを定義した記録テンプレート |
| 活動記録 CRUD | テンプレートに基づいた活動記録の作成・編集・削除 |
| 参加者管理 | 活動への参加メンバーの記録（メンバー番号での選択対応） |
| 統計・集計 | 期間別・テンプレート別・フィールド別の集計・グラフ表示 |
| 公開設定 | 公開記録は SSR で SEO 対応。メンバー限定記録は SPA |

**スコープ外**:
- ブログ・お知らせ（自由記述の記事） → F06.1
- パフォーマンスデータの個人統計ダッシュボード → F07.2
- サービス履歴（施術記録・来店記録） → F07.1
- カルテ（問診票・身体チャート） → F07.4

### テーブル定義

#### `activity_templates`

活動記録のテンプレート。チームごとにカスタムフィールドを定義した記録フォーマット。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `name` | VARCHAR(100) | NO | — | テンプレート名（例: 「試合結果」「練習記録」「理事会議事録」） |
| `description` | VARCHAR(500) | YES | NULL | テンプレートの説明 |
| `icon` | VARCHAR(30) | YES | NULL | アイコン（emoji or アイコン名。例: `⚽`, `📋`） |
| `color` | VARCHAR(7) | YES | NULL | テーマカラー（hex。例: `#3B82F6`） |
| `is_participant_required` | BOOLEAN | NO | TRUE | 参加者の記録が必須か |
| `default_visibility` | VARCHAR(20) | NO | 'MEMBERS_ONLY' | デフォルトの公開設定（`PUBLIC` / `MEMBERS_ONLY`） |
| `sort_order` | INT | NO | 0 | 表示順 |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。作成した ADMIN |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_at_scope (scope_type, scope_id, sort_order)  -- スコープ別のテンプレート一覧
```

**制約・備考**
- 1スコープあたり最大 **20 テンプレート**
- テンプレートを論理削除しても、既存の活動記録は閲覧可能（テンプレート名はスナップショットされない。JOIN で取得）
- `default_visibility = 'PUBLIC'`: SSR でインデックスされる。ゲストも閲覧可能
- テンプレートごとにカスタムフィールドを `activity_template_fields` で定義

#### `activity_template_fields`

テンプレートのカスタムフィールド定義。チームが記録したい項目を自由に追加。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `template_id` | BIGINT UNSIGNED | NO | — | FK → activity_templates。ON DELETE CASCADE |
| `field_key` | VARCHAR(50) | NO | — | フィールドキー（英数字・アンダースコア。例: `score`, `opponent`, `location`） |
| `field_label` | VARCHAR(100) | NO | — | 表示ラベル（例: 「スコア」「対戦相手」「会場」） |
| `field_type` | VARCHAR(20) | NO | — | フィールド型（`TEXT` / `NUMBER` / `DATE` / `SELECT` / `CHECKBOX` / `TEXTAREA`） |
| `is_required` | BOOLEAN | NO | FALSE | 必須フィールドか |
| `options_json` | JSON | YES | NULL | `SELECT` 型の選択肢（例: `["勝ち", "負け", "引き分け"]`） |
| `placeholder` | VARCHAR(200) | YES | NULL | プレースホルダーテキスト |
| `unit` | VARCHAR(20) | YES | NULL | 単位（例: 「点」「km」「時間」。`NUMBER` 型の場合に表示） |
| `is_aggregatable` | BOOLEAN | NO | FALSE | 統計集計の対象か（`NUMBER` 型のみ TRUE 可能。合計・平均・最大・最小の集計対象） |
| `sort_order` | INT | NO | 0 | 表示順 |

**インデックス**
```sql
INDEX idx_atf_template (template_id, sort_order)                -- テンプレート別のフィールド一覧
UNIQUE KEY uq_atf_key (template_id, field_key)                   -- テンプレート内でフィールドキー一意
```

**制約・備考**
- 1テンプレートあたり最大 **15 フィールド**
- `field_type` ごとの保存形式:
  - `TEXT`: 文字列（最大 200 文字）
  - `NUMBER`: 数値（DECIMAL(10,2)）
  - `DATE`: 日付（YYYY-MM-DD）
  - `SELECT`: 選択肢の値（`options_json` から選択）
  - `CHECKBOX`: 真偽値（true/false）
  - `TEXTAREA`: 長文（最大 2,000 文字）
- `is_aggregatable = TRUE` は `NUMBER` 型のみ。統計 API で合計・平均・最大・最小を算出
- フィールド定義の変更ルール:
  - **追加**: 許可。既存記録には新フィールドの値がない → 表示時に「未記録」と表示
  - **削除**: 許可。既存記録に旧フィールドの値が残る → 表示時に「（削除済みフィールド）」として値を表示（データは消さない）
  - **`field_type` 変更**: 禁止（既存 JSON の値が新しい型と不整合になるため。型を変えたい場合は旧フィールド削除 + 新フィールド追加）
  - **`field_key` 変更**: 禁止（JSON のキーと一致しなくなるため）
  - **`field_label` 変更**: 許可（表示名のみの変更で既存データに影響なし）
  - **`is_required` 変更**: 許可（新規記録のみ影響。既存記録は再バリデーション対象外）
  - **`options_json` 変更**: 許可（新規記録のみ新しい選択肢を適用。既存記録の値が新選択肢に含まれなくてもエラーにしない）

#### `activity_results`

活動記録の実データ。テンプレートに基づいて記録される。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `template_id` | BIGINT UNSIGNED | NO | — | FK → activity_templates。使用テンプレート |
| `title` | VARCHAR(200) | NO | — | 活動タイトル（例: 「第15節 vs ○○チーム」「2026年4月度 練習記録」） |
| `activity_date` | DATE | NO | — | 活動日 |
| `activity_time_start` | TIME | YES | NULL | 開始時刻（任意） |
| `activity_time_end` | TIME | YES | NULL | 終了時刻（任意） |
| `description` | TEXT | YES | NULL | 活動の詳細説明（Markdown 対応。任意） |
| `field_values` | JSON | NO | '{}' | カスタムフィールドの値（`{ "score": "3-1", "opponent": "○○チーム", "location": "○○スタジアム" }`。フィールドなし or 全て任意で未入力の場合は `{}`） |
| `attachments` | JSON | YES | NULL | 添付ファイル ID 配列（`{ "file_ids": [101, 102, 103] }`。F05.5 の `shared_files` を参照。最大10件） |
| `visibility` | VARCHAR(20) | NO | 'MEMBERS_ONLY' | 公開設定（`PUBLIC` / `MEMBERS_ONLY`） |
| `schedule_id` | BIGINT UNSIGNED | YES | NULL | FK → schedules。スケジュールから自動作成した場合 |
| `created_by` | BIGINT UNSIGNED | YES | — | FK → users（ON DELETE SET NULL）。作成者。退会時は NULL に遷移 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_ar_scope (scope_type, scope_id, activity_date DESC)           -- スコープ別の活動記録一覧
INDEX idx_ar_template (template_id, activity_date DESC)                  -- テンプレート別の記録一覧
INDEX idx_ar_schedule (schedule_id)                                      -- スケジュールからの逆引き
INDEX idx_ar_visibility (scope_type, scope_id, visibility, activity_date DESC)  -- 公開記録の一覧（SSR 用）
```

**制約・備考**
- `field_values`: `activity_template_fields` で定義されたフィールドの値を JSON で保存。キーは `field_key`、値は `field_type` に応じた形式
- `field_values` のバリデーション: Service 層で `activity_template_fields` の定義に基づき型チェック・必須チェックを実行
- `visibility = 'PUBLIC'`: ゲストも閲覧可能。SSR でレンダリングし SEO 対象
- `schedule_id`: F03.1 のスケジュール完了時に「活動記録を作成しますか？」の導線で紐付け。イベントのタイトル・日付を初期値に使用
- アクセス解析: `page_view_logs` に `target_type = 'ACTIVITY_RECORD'`, `target_id = activity_results.id` で記録
- `attachments`: F05.5 の `shared_files.id` を参照する JSON 配列。1活動記録あたり最大10ファイル。F05.5 のストレージ容量管理がそのまま適用。表示時に `shared_files` から URL・サムネイルを IN 句で取得
- `description`（Markdown）: Markdown テキストをそのまま保存（HTML 変換はフロントエンドで実施）。サニタイズは DOMPurify で SSR 時含めてフロントエンドで実行。最大 10,000 文字
- `created_by` の ON DELETE: `SET NULL`。作成者がアーカイブ/退会しても活動記録は残る。`created_by IS NULL` の場合は「退会済みユーザー」と表示

#### `activity_participants`

活動への参加者。活動記録と users の多対多リレーション。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `activity_result_id` | BIGINT UNSIGNED | NO | — | FK → activity_results。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE）。参加者。退会時は参加記録も削除 |
| `role_label` | VARCHAR(50) | YES | NULL | 活動内での役割（例: 「GK」「司会」「記録係」） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_ap_activity_user (activity_result_id, user_id)  -- 同一活動の重複参加防止
INDEX idx_ap_user (user_id, created_at DESC)                    -- ユーザー別の参加履歴
```

**制約・備考**
- 参加者はスコープ内のメンバーであること（Service 層バリデーション）
- `role_label`: 任意。テンプレートの `is_participant_required = TRUE` の場合、最低1名の参加者が必須
- F06.2 のメンバー番号（背番号・社員番号等）でも参加者を検索・選択可能

| `user_id` | BIGINT UNSIGNED | YES | — | FK → users（ON DELETE SET NULL）。投稿者 |

| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_ac_activity (activity_result_id, created_at)  -- 活動記録別のコメント一覧
```

#### `system_activity_template_presets`

プラットフォーム標準テンプレート。SYSTEM_ADMIN がカテゴリ別の推奨テンプレートを管理し、チーム作成時のオンボーディングで提供。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `category` | VARCHAR(30) | NO | — | カテゴリ（`SPORTS` / `SCHOOL` / `COMMUNITY` / `BUSINESS` / `MANAGEMENT_ASSOCIATION`） |
| `name` | VARCHAR(100) | NO | — | プリセット名（例: 「試合結果（スポーツ）」「理事会議事録（管理組合）」） |
| `description` | VARCHAR(500) | YES | NULL | プリセットの説明 |
| `icon` | VARCHAR(30) | YES | NULL | アイコン |
| `color` | VARCHAR(7) | YES | NULL | テーマカラー |
| `is_participant_required` | BOOLEAN | NO | TRUE | 参加者必須設定 |
| `default_visibility` | VARCHAR(20) | NO | 'MEMBERS_ONLY' | デフォルト公開設定 |
| `fields_json` | JSON | NO | — | フィールド定義の JSON 配列（`activity_template_fields` と同形式） |
| `is_active` | BOOLEAN | NO | TRUE | 有効/無効 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_satp_category (category, is_active)  -- カテゴリ別のプリセット一覧
```

**制約・備考**
- `fields_json`: `[{"field_key": "opponent", "field_label": "対戦相手", "field_type": "TEXT", ...}]` 形式。`POST /activity-templates/import-preset` で `activity_template_fields` に展開
- `category` でチーム種別（テンプレートの `template_type`）に応じたフィルタリングが可能
- `system_activity_template_preset_fields` テーブルは設けない（`fields_json` で十分。プリセットのフィールドは DB に正規化する必要がない）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `field_type` 変更の試行 / `field_key` のリネーム試行（既存キーが別のキーに変わった場合） / フィールド数超過 |
| 403 | 権限不足 |
| 404 | テンプレートが存在しない |

## 5. ビジネスロジック

### 活動記録の作成フロー

```
1. ADMIN / MEMBER がテンプレートを選択
   a. テンプレート一覧から記録フォーマットを選択
   b. カスタムフィールドの入力フォームが動的に生成

2. フィールド値の入力
   a. field_type に応じた入力 UI を表示
      - TEXT: テキスト入力
      - NUMBER: 数値入力（unit 表示付き）
      - DATE: デートピッカー
      - SELECT: ドロップダウン（options_json から選択肢生成）
      - CHECKBOX: チェックボックス
      - TEXTAREA: テキストエリア
   b. is_required = TRUE のフィールドはバリデーション必須

3. 参加者の選択
   a. スコープ内のメンバー一覧から選択（チェックボックス）
   b. メンバー番号（F06.2 の背番号等）での検索・フィルタ対応
   c. 前回の参加者をデフォルト選択（同一テンプレートの直近記録から引き継ぎ）

4. 保存
   a. field_values を JSON として activity_results に INSERT
   b. participant_user_ids を activity_participants に INSERT
   c. visibility = 'PUBLIC' の場合: SSR キャッシュを無効化
```

### スケジュールからの活動記録作成

```
1. F03.1 のスケジュール完了時（status = 'COMPLETED'）
   a. 「活動記録を作成しますか？」のプロンプトを表示
   b. 「はい」→ 活動記録作成画面に遷移
      - title: schedules.title をプリフィル
      - activity_date: schedules.starts_at の DATE 部分をプリフィル
      - activity_time_start / end: schedules.starts_at / ends_at の TIME 部分をプリフィル
      - participant_user_ids: schedule_attendances で status = 'ATTENDING' のユーザーをプリフィル
      - schedule_id: 該当スケジュールの ID
   c. 「いいえ」→ スキップ
```

### カスタムフィールドの集計ロジック

```
1. GET /activities/stats/fields で is_aggregatable = TRUE の NUMBER フィールドを集計
2. 集計対象: activity_results.field_values の JSON から該当 field_key の値を抽出
3. SQL:
   SELECT
     COUNT(*) as count,
     SUM(CAST(JSON_EXTRACT(field_values, CONCAT('$.', :fieldKey)) AS DECIMAL(10,2))) as sum,
     AVG(CAST(JSON_EXTRACT(field_values, CONCAT('$.', :fieldKey)) AS DECIMAL(10,2))) as avg,
     MAX(CAST(JSON_EXTRACT(field_values, CONCAT('$.', :fieldKey)) AS DECIMAL(10,2))) as max,
     MIN(CAST(JSON_EXTRACT(field_values, CONCAT('$.', :fieldKey)) AS DECIMAL(10,2))) as min
   FROM activity_results
   WHERE template_id = :templateId
     AND scope_type = :scopeType AND scope_id = :scopeId
     AND deleted_at IS NULL
     AND activity_date BETWEEN :dateFrom AND :dateTo
4. MySQL 8.0 の JSON_EXTRACT で JSON 内の値を直接集計
5. セキュリティ: :fieldKey は CONCAT でJSON パスに組み込まれるため、
   field_key のバリデーション（/^[a-zA-Z0-9_]+$/）を Service 層 + DB の
   UNIQUE KEY uq_atf_key で二重に保証。SQL インジェクションリスクを排除
6. パフォーマンス: template_id + activity_date のインデックスで絞り込み後に JSON_EXTRACT
   → 大量データ（1万件超）の場合はマテリアライズドビュー or 日次バッチ集計を検討
```

### 活動記録の複製フロー

```
1. ユーザーが活動記録詳細画面で「この記録をコピー」ボタンをクリック
2. POST /activities/{id}/duplicate を呼び出し
3. バックエンド:
   a. 元の記録の field_values / visibility / description をコピー
   b. 元の activity_participants を全員コピー
   c. file_ids はコピーしない（添付ファイルは元の記録と共有すると
      削除時に影響するため。必要なら手動で再添付）
   d. activity_date を今日の日付でプリフィル（リクエストで上書き可能）
   e. title は元のタイトルをそのままコピー（リクエストで上書き可能）
4. フロントエンド: 複製結果の編集画面を開く（即座に保存ではなく編集→保存の2ステップ）
```

### タイムラインへの自動投稿

```
1. POST /activities で post_to_timeline = true の場合
2. 活動記録の INSERT 完了後に ApplicationEvent を発行
3. TimelineAutoPostListener が非同期で受信:
   a. タイムライン投稿のテキストを自動生成:
      「📋 活動記録: {title}
       {field_values_summary の必須フィールドを箇条書き}
       👥 参加者 {participant_count} 名」
   b. POST /timeline（内部呼び出し）でタイムライン投稿を作成
   c. 投稿に活動記録へのリンクを含める
   d. 投稿の visibility は活動記録の visibility に合わせる
4. 失敗時: 活動記録の作成はロールバックしない（タイムライン投稿は副作用）
5. 注意: 編集時（PUT /activities）には自動投稿しない（初回作成時のみ）
```

### コメントフロー

```
1. メンバーが活動記録詳細画面でコメントを投稿
2. POST /activities/{id}/comments で activity_comments に INSERT
3. 活動記録の created_by（作成者）にプッシュ通知:
   「{commenter_name} が活動記録「{title}」にコメントしました」
4. コメントの body は Markdown 対応（description と同一のサニタイズ方針）
5. コメント一覧は created_at ASC でソート（時系列順）
```

### プリセットテンプレートのインポートフロー

```
1. チーム作成直後のオンボーディング画面 or テンプレート管理画面で
   「おすすめテンプレートをインポート」ボタンを表示
2. GET /system-admin/activity-template-presets?category={チームのカテゴリ}
   でカテゴリに合ったプリセット一覧を取得
3. ユーザーがプリセットを選択 → POST /activity-templates/import-preset で
   スコープにインポート
4. インポート後のテンプレートは通常のテンプレートと同一に扱える
   （フィールド追加・削除・名前変更等すべて可能）
5. 1プリセットを複数チームにインポート可能（各チームに独立したコピーが作成される）
```

### JSON 集計のパフォーマンス戦略

```
1. Phase 6 初期実装: JSON_EXTRACT のまま
   a. 大半のチームは年間 100-500 件。JSON_EXTRACT で十分
   b. stats API にクエリタイムアウト（3秒）を設定
   c. タイムアウト時: 部分結果を返却 + warnings に "集計がタイムアウトしました" を含める
   d. レスポンスに query_time_ms を含めて監視指標にする

2. Phase 11 負荷テストで 1万件超のケースを検証
   a. 超過した場合の対策: activity_field_aggregates テーブル（日次バッチ集計）
      - template_id × field_key × date → sum / avg / max / min / count
      - 日次バッチ（AM 3:00）で前日分を集計して UPSERT
      - stats API はバッチ集計テーブルから読み取り（JSON_EXTRACT を回避）
   b. このテーブルは Phase 6 では作成しない（YAGNI 原則）
```

### Markdown サニタイズ方針

```
1. 保存: activity_results.description には Markdown テキストをそのまま保存
   - バックエンドでは HTML 変換しない
   - 最大 10,000 文字のバリデーションのみ

2. 表示（フロントエンド）:
   a. marked.js で Markdown → HTML に変換
   b. DOMPurify で sanitize（SSR 時は isomorphic-dompurify を使用）
   c. 許可タグ（ホワイトリスト）:
      p, h1-h6, ul, ol, li, a, strong, em, code, pre, blockquote,
      img, br, hr, table, thead, tbody, tr, th, td
   d. 禁止: script, iframe, form, input, style, on* イベントハンドラ
   e. a タグの href: http / https のみ許可。javascript: スキーム禁止
   f. img タグの src: https のみ許可（http 禁止。mixed content 防止）

3. 本方針の適用範囲:
   - activity_results.description（本機能）
   - blog_posts.body（F06.1）
   - bulletin_threads.body / bulletin_replies.body（F05.1）
   - 他の Markdown 対応フィールド全般
   → FRONTEND_CODING_CONVENTION.md に共通方針として追記すべき
```

### 重要な判定ロジック

- **Cursor ページネーション**: `cursor` は `{activity_date}_{id}` の複合キー。SQL: `WHERE (activity_date, id) < (:cursorDate, :cursorId) ORDER BY activity_date DESC, id DESC`。activity_date 順のソートと ID 順の作成が異なるケース（「昨日の活動を今日登録」等）で重複・スキップを防止
- **カスタムフィールドのバリデーション**: `POST /activities` 時に `activity_template_fields` の定義に基づき、`field_values` の各フィールドの型・必須・選択肢を検証。未定義の `field_key` は無視（将来のフィールド削除に対応）
- **テンプレート削除の影響**: 論理削除された `activity_templates` のフィールド定義は `activity_results` の `field_values` を表示する際に引き続き参照可能（JOIN で取得）。テンプレート削除後も既存記録の閲覧・編集は可能
- **JSON フィールドの検索**: `field_values` の JSON 内容での検索は `JSON_EXTRACT` + `WHERE` で可能だが、大量データではパフォーマンスに注意。全文検索が必要な場合は `description` カラムの FULLTEXT インデックスを活用
- **参加者のプリフィル**: 同一テンプレートの直近の活動記録から `activity_participants` を取得し、デフォルト選択として提示。レギュラーメンバーの入力手間を削減
- **SSR と公開記録**: `visibility = 'PUBLIC'` の記録は `/public/teams/{teamId}/activities` で認証不要のアクセスを提供。Nuxt 3 の SSR で `useFetch` を使用し SEO 対象に。LCP < 2.5s を維持
- **MEMBER の作成権限**: MEMBER は活動記録を作成可能（ただし自分が `created_by` の記録のみ編集・削除可能）。ADMIN / DEPUTY_ADMIN は全記録の編集・削除が可能
- **アクセス解析連携**: `page_view_logs` に `target_type = 'ACTIVITY_RECORD'` で自動記録。ダッシュボードの `TEAM_PAGE_VIEWS` ウィジェットに反映
- **テンプレートフィールド変更の安全性**: `field_type` / `field_key` の変更は禁止（既存 JSON データとの不整合防止）。`field_label` / `is_required` / `options_json` の変更は許可（新規記録のみ影響）。削除済みフィールドの値は「（削除済みフィールド）」として表示継続
- **添付ファイル**: `attachments.file_ids` で F05.5 の `shared_files` を参照。最大10ファイル。F05.5 のストレージ容量管理がそのまま適用
- **CSV エクスポート**: `template_id` 指定時はカスタムフィールドを個別カラムに展開。BOM 付き UTF-8。最大 5,000 件
- **JSON 集計パフォーマンス**: Phase 6 は JSON_EXTRACT で対応（クエリタイムアウト 3秒）。1万件超は Phase 11 で日次サマリーテーブルを検討（YAGNI 原則）
- **Markdown サニタイズ**: description は Markdown テキストで保存。HTML 変換 + サニタイズはフロントエンド（marked.js + DOMPurify）。バックエンドは最大文字数バリデーションのみ
- **活動記録の複製**: `POST /activities/{id}/duplicate` で前回の `field_values` / `participant_user_ids` / `visibility` / `description` をコピー。`file_ids` はコピーしない（共有ファイルの削除連鎖を防止）。複製結果は編集画面で確認後に保存
- **タイムライン自動投稿**: `post_to_timeline = true` で活動記録作成時に F04.1 タイムラインにサマリーを自動投稿。ApplicationEvent + 非同期 Listener で実装。PUT（編集）時は自動投稿しない
- **コメント機能**: `activity_comments` テーブルで活動記録への補足情報・フィードバックを管理。投稿者本人のみ編集可能、削除は投稿者 or ADMIN。作成者にプッシュ通知
- **テンプレート共有**: `POST /activity-templates/{id}/duplicate` で組織内の別チームにテンプレートをコピー。コピー元/先の両方の ADMIN 権限が必要
- **プリセットテンプレート**: `system_activity_template_presets` で SYSTEM_ADMIN がカテゴリ別の標準テンプレートを管理。`POST /activity-templates/import-preset` でチームにインポート。インポート後は通常のテンプレートとして自由に編集可能

## 6. セキュリティ考慮事項

- **認可チェック**: テンプレート管理は ADMIN / DEPUTY_ADMIN（`ACTIVITY_MANAGE`）。活動記録の作成は MEMBER 以上。編集・削除は作成者 or ADMIN。公開記録の閲覧は認証不要
- **スコープメンバーシップ検証**: 活動記録の作成・閲覧はスコープメンバーであること。非メンバーは 403（公開記録を除く）
- **参加者の検証**: `participant_user_ids` に含まれるユーザーがスコープメンバーであること
- **JSON Injection 防止**: `field_values` の JSON 入力は Service 層で構造を検証し、予期しないキーや深いネストを拒否
- **XSS 対策**: `description`（Markdown）はバックエンドでは Markdown テキストのまま保存（HTML 変換しない）。フロントエンドで marked.js → DOMPurify のパイプラインでサニタイズ。`field_values` の TEXT / TEXTAREA は API レスポンス時に HTML エスケープ
- **添付ファイルの検証**: `file_ids` に含まれる `shared_files` がスコープ内のファイルであること。他チームのファイルを添付不可（403）
- **レートリミット**: `Bucket4j` で以下の制限を適用
  - 活動記録作成: 1分間10件/ユーザー
  - テンプレート作成: 1分間5件/ユーザー
  - 統計 API: 1分間30件/ユーザー
  - CSV エクスポート: 1分間3件/ユーザー
  - コメント投稿: 1分間10件/ユーザー
  - テンプレート複製: 1分間5件/ユーザー
  - プリセットインポート: 1分間5件/ユーザー
  - 公開 API: 1分間60件/IP（認証不要のため IP ベース）
- **監査ログ**: 以下のイベントを記録
  - `ACTIVITY_TEMPLATE_CREATED`: テンプレート作成
  - `ACTIVITY_TEMPLATE_DELETED`: テンプレート削除
  - `ACTIVITY_CREATED`: 活動記録作成
  - `ACTIVITY_UPDATED`: 活動記録更新
  - `ACTIVITY_DELETED`: 活動記録削除
  - `ACTIVITY_EXPORTED`: CSV エクスポート（scope, template, 件数, 期間）
  - `ACTIVITY_DUPLICATED`: 活動記録の複製（source_id, new_id）
  - `ACTIVITY_TEMPLATE_DUPLICATED`: テンプレートのスコープ間コピー（source_id, target_scope）
  - `ACTIVITY_TEMPLATE_PRESET_IMPORTED`: プリセットのインポート（preset_id, target_scope）
