## 1. 概要

プラットフォーム上の不適切コンテンツに対する通報・レビュー・段階的対応のモデレーションワークフローを提供する。全ユーザーが通報可能で、ADMIN（スコープ内）/ SYSTEM_ADMIN（プラットフォーム全体）がレビュー・対応を行う。

本ドキュメントは F10.1 で定義された `content_reports` / `report_actions` テーブルと基本 API を基盤として、以下の**運用ワークフロー・エスカレーション・自動化・ユーザー保護**に特化する:

- **段階的対応**: 警告 → コンテンツ削除 → ユーザー凍結の3段階
- **自動エスカレーション**: 同一ユーザーへの通報が閾値を超えた場合に SYSTEM_ADMIN へ自動エスカレーション
- **証拠保全**: 通報時点のコンテンツスナップショット（JSON）を自動保存
- **匿名通報**: 通報者の身元は通報対象者には非公開（ADMIN / SYSTEM_ADMIN のみ閲覧可）
- **違反累積管理**: ユーザーごとの違反回数を追跡し、累積に応じた自動制裁
- **アピール（異議申立て）**: 凍結・削除されたユーザーが異議を申し立てるフロー
- **WARNING 再レビュー**: 警告を受けたユーザーが ADMIN に再レビューを依頼し、棄却された場合のみ SYSTEM_ADMIN への異議申立てに昇格する2段階フロー
- **ヤバいやつ制度**: 違反累積が閾値を超えたユーザーの実名を所属スコープ内に公開する抑止力。登録時に本制度の存在を明示し、不適切行為を強力に抑止する
- **ユーザー向け透明性**: 自分の違反履歴・通報ステータスの閲覧。WARNING 時のガイドライン条項参照。コンテンツ自主修正による WARNING 取下げフロー
- **管理者効率化**: 対応テンプレート（定型文）、レビュー中の一時非表示、通報キューの自動優先度付け、ADMIN 間の内部メモ
- **運用自動化**: 月次モデレーションレポート自動生成、通報のバンドル表示（同一ユーザー・同一理由の自動グルーピング）、設定変更履歴の自動記録

> **DB テーブル定義・基本 API 仕様**: F10.1 管理者ダッシュボードに記載。本ドキュメントでは追加テーブル・追加 API・ビジネスロジックを定義する。

| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `user_violations` | ユーザーごとの違反累積記録 | なし |
| `moderation_appeals` | ユーザーからの異議申立て（凍結・削除用） | なし |
| `warning_re_reviews` | WARNING に対する再レビュー依頼（ADMIN → SYSTEM_ADMIN 2段階） | なし |
| `yabai_unflag_requests` | ヤバいやつ認定の解除申請 | なし |
| `moderation_settings` | モデレーション設定（エスカレーション閾値等。SYSTEM_ADMIN 管理） | なし |

| `moderation_settings_history` | モデレーション設定の変更履歴 | なし |

### テーブル定義

#### `user_violations`

ユーザーごとの違反履歴。`report_actions` で `WARNING` / `CONTENT_DELETE` / `USER_FREEZE` が実行された際に自動記録。累積数に応じた自動制裁の判定に使用。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE） |
| `report_id` | BIGINT UNSIGNED | NO | — | FK → content_reports（トリガーとなった通報） |
| `action_id` | BIGINT UNSIGNED | NO | — | FK → report_actions（実行されたアクション） |
| `violation_type` | VARCHAR(20) | NO | — | `WARNING` / `CONTENT_DELETE` / `TEMPORARY_FREEZE` / `USER_FREEZE` |
| `reason` | VARCHAR(30) | NO | — | 通報理由（content_reports.reason のコピー） |
| `expires_at` | DATETIME | YES | NULL | 違反記録の有効期限（NULL = 永久。WARNING は6ヶ月、CONTENT_DELETE は12ヶ月で自動失効） |
| `is_active` | BOOLEAN | NO | TRUE | 有効/失効。`expires_at` 経過後にバッチで `false` に更新 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_uv_user (user_id, is_active, created_at DESC)    -- ユーザー別の有効な違反履歴
INDEX idx_uv_type (violation_type, is_active)                -- 種別別の有効な違反数
```

**制約・備考**
- `report_actions` の INSERT トリガー（アプリ層）で自動 INSERT。`DISMISS` アクションでは記録しない
- 有効な違反の累積数で自動制裁を判定（`moderation_settings` の閾値を参照）:
  - 有効な WARNING 3回 → 自動で次回通報を SYSTEM_ADMIN にエスカレーション
  - 有効な CONTENT_DELETE 3回 → SYSTEM_ADMIN への自動エスカレーション + 一時凍結の推奨通知
- `expires_at`: 軽微な違反が永久に累積しないための時効。WARNING = 6ヶ月、CONTENT_DELETE = 12ヶ月、USER_FREEZE = 永久（NULL）
- 日次バッチで `expires_at < NOW()` かつ `is_active = true` のレコードを `is_active = false` に更新

#### `moderation_appeals`

凍結・コンテンツ削除されたユーザーからの異議申立て。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（申立て者） |
| `report_id` | BIGINT UNSIGNED | NO | — | FK → content_reports（異議対象の通報） |
| `action_id` | BIGINT UNSIGNED | NO | — | FK → report_actions（異議対象のアクション） |
| `appeal_token` | VARCHAR(64) | NO | — | メールリンク認証用トークンの SHA-256 ハッシュ（平文はメールで送付。DB にはハッシュのみ保存） |
| `appeal_token_expires_at` | DATETIME | NO | — | トークンの有効期限（= アクション実行日 + appeal_window_days） |
| `appeal_reason` | TEXT | NO | — | 異議の理由（最大2,000文字） |
| `status` | VARCHAR(20) | NO | 'INVITED' | `INVITED`（仮作成。USER_FREEZE/TEMPORARY_FREEZE 実行時に自動生成）/ `PENDING`（ユーザーが申立て理由を送信）/ `ACCEPTED` / `REJECTED` |
| `reviewed_by` | BIGINT UNSIGNED | YES | NULL | FK → users（レビューした SYSTEM_ADMIN） |
| `review_note` | TEXT | YES | NULL | レビューメモ |
| `reviewed_at` | DATETIME | YES | NULL | |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_ma_user_action (user_id, action_id)           -- 同一アクションへの重複申立て防止
UNIQUE KEY uq_ma_token (appeal_token)                        -- トークン検索（凍結中ユーザーの認証）
INDEX idx_ma_status (status, created_at DESC)                 -- ステータス別一覧
```

**制約・備考**
- 1つのアクションに対して1回のみ異議申立て可能（UNIQUE KEY）
- `ACCEPTED` の場合: 対応を取消（コンテンツの論理削除解除 / 凍結解除）+ `user_violations` の該当レコードを `is_active = false` に更新
- `REJECTED` の場合: 現状維持。再申立て不可
- **凍結中ユーザーの認証（メールリンク方式）**: USER_FREEZE / TEMPORARY_FREEZE 実行時にバックエンドが以下を実行: (1) `moderation_appeals` に仮レコード INSERT（`status = INVITED`、`appeal_reason = NULL`、`appeal_token` 生成済み）。(2) メールで異議申立てリンク（`https://mannschaft.app/appeal?token={appeal_token}`）を送信。(3) ユーザーがリンクからアクセス → `appeal_token` で認証 → `appeal_reason` を入力して `PATCH /appeals/{id}/submit` → `status = PENDING` に遷移。トークンは `appeal_token_expires_at` まで有効で、期間内は何度でもアクセス可能（GET /appeals/{id} + PATCH /appeals/{id}/submit のみ許可）
- SYSTEM_ADMIN のみがレビュー可能（ADMIN レベルの判断を上位でオーバーライドする設計）

#### `warning_re_reviews`

WARNING に対する再レビュー依頼。ユーザーがまず ADMIN に再レビューを依頼し、ADMIN が棄却（UPHELD）した場合のみ SYSTEM_ADMIN への異議申立て（ESCALATED）に昇格する2段階フロー。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（依頼者） |
| `report_id` | BIGINT UNSIGNED | NO | — | FK → content_reports（対象の通報） |
| `action_id` | BIGINT UNSIGNED | NO | — | FK → report_actions（対象の WARNING アクション） |
| `reason` | TEXT | NO | — | 再レビューの理由（最大2,000文字） |
| `status` | VARCHAR(20) | NO | 'PENDING' | `PENDING`（ADMIN レビュー待ち）/ `OVERTURNED`（ADMIN が取消し）/ `UPHELD`（ADMIN が維持）/ `ESCALATED`（SYSTEM_ADMIN に昇格）/ `APPEAL_ACCEPTED`（SYSTEM_ADMIN が承認）/ `APPEAL_REJECTED`（SYSTEM_ADMIN が棄却） |
| `admin_reviewed_by` | BIGINT UNSIGNED | YES | NULL | FK → users（レビューした ADMIN） |
| `admin_review_note` | TEXT | YES | NULL | ADMIN のレビューメモ |
| `admin_reviewed_at` | DATETIME | YES | NULL | |
| `escalation_reason` | TEXT | YES | NULL | SYSTEM_ADMIN への昇格理由（最大2,000文字。UPHELD 後にユーザーが記入） |
| `system_admin_reviewed_by` | BIGINT UNSIGNED | YES | NULL | FK → users（レビューした SYSTEM_ADMIN） |
| `system_admin_review_note` | TEXT | YES | NULL | SYSTEM_ADMIN のレビューメモ |
| `system_admin_reviewed_at` | DATETIME | YES | NULL | |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_wrr_user_action (user_id, action_id)     -- 同一 WARNING への重複依頼防止
INDEX idx_wrr_status (status, created_at DESC)           -- ステータス別一覧
INDEX idx_wrr_report (report_id)                         -- 通報別の再レビュー依頼
```

**制約・備考**
- 対象は `report_actions.action_type = WARNING` のみ。CONTENT_DELETE / FREEZE は `moderation_appeals` を使用
- 1つの WARNING アクションに対して1回のみ再レビュー依頼可能（UNIQUE KEY）
- 依頼可能期間: WARNING 実行から `appeal_window_days`（デフォルト30日）以内
- **ADMIN 再レビューで OVERTURNED の場合**: `user_violations` の該当レコードを `is_active = false` に更新 + ヤバいやつ再判定
- **ADMIN 再レビューで UPHELD の場合**: ユーザーに通知「再レビューの結果、WARNING は維持されました。この判断に異議がある場合は、上位管理者に申し立てることができます。」+ 14日以内に昇格可能
- **ESCALATED（SYSTEM_ADMIN 昇格）**: SYSTEM_ADMIN がレビュー → APPEAL_ACCEPTED なら `user_violations` 無効化 + 元の ADMIN に通知、APPEAL_REJECTED なら現状維持
- 利益相反チェック: 元の WARNING を実行した ADMIN は再レビュー不可（`report_actions.action_by = admin_reviewed_by` → 403。同スコープの別 ADMIN がレビュー）

#### `yabai_unflag_requests`

ヤバいやつ認定の解除申請。ユーザーが認定後一定期間（`yabai_unflag_eligible_months`）経過 + 新規違反ゼロの条件を満たした場合に申請可能。SYSTEM_ADMIN がレビュー。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（申請者） |
| `reason` | TEXT | NO | — | 解除申請の理由・反省文（最大2,000文字） |
| `status` | VARCHAR(20) | NO | 'PENDING' | `PENDING` / `ACCEPTED` / `REJECTED` |
| `reviewed_by` | BIGINT UNSIGNED | YES | NULL | FK → users（レビューした SYSTEM_ADMIN） |
| `review_note` | TEXT | YES | NULL | レビューメモ |
| `reviewed_at` | DATETIME | YES | NULL | |
| `next_eligible_at` | DATETIME | YES | NULL | REJECTED 時に設定。次回申請可能日（= `reviewed_at` + 3ヶ月） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_yur_user (user_id, created_at DESC)    -- ユーザー別の申請履歴
INDEX idx_yur_status (status, created_at DESC)    -- ステータス別一覧
```

**制約・備考**
- 申請条件（全てを満たす必要あり）:
  1. `users.yabai_flagged_at IS NOT NULL`（認定中であること）
  2. `yabai_flagged_at` から `yabai_unflag_eligible_months` ヶ月以上経過
  3. `yabai_unflag_clean_required = true` の場合: `yabai_flagged_at` 以降の `user_violations`（`is_active = true`）が0件
  4. 直前の解除申請（現在の `yabai_flagged_at` 以降のもの）が REJECTED の場合: `next_eligible_at < NOW()` であること
- `ACCEPTED` の場合: `users.yabai_flagged_at = NULL` に更新 + ユーザーにメール通知「実名公開が解除されました」
- `REJECTED` の場合: `next_eligible_at = reviewed_at + 3ヶ月` を設定。ユーザーにメール通知「解除申請が却下されました。{next_eligible_at} 以降に再申請可能です」
- PENDING の申請が既に存在する場合、新規申請は不可（409）
- SYSTEM_ADMIN 手動解除 → 再認定のケース: 新しい `yabai_flagged_at` 以降のレコードのみ参照するため、古い REJECTED の `next_eligible_at` は影響しない

#### `moderation_action_templates`

対応時の定型文テンプレート。ADMIN が WARNING / CONTENT_DELETE / DISMISS 等を実行する際に1クリックで `note` を入力できる。SYSTEM_ADMIN が管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `action_type` | VARCHAR(20) | NO | — | 対象アクション（`WARNING` / `CONTENT_DELETE` / `DISMISS` / `USER_FREEZE`） |

### リクエスト／レスポンス仕様

#### `DELETE /api/v1/reports/{id}`

**認可**: 通報者本人

誤って作成した通報を取消す。PENDING 状態のみ取消し可能（`CANCELLED` に遷移）。

**レスポンス（200 OK）**: 通報詳細（`status: "CANCELLED"`）

**補足**
- `status = PENDING` のみ。REVIEWING 以降は ADMIN が着手済みのため取消不可
- 論理取消し: `status = CANCELLED` に遷移（物理削除ではない）。`daily_report_limit` のカウント対象に含まれるため、取消し→再通報の繰り返しによるスパムを防止
- 同一コンテンツへの再通報: CANCELLED の通報は UNIQUE KEY `uq_cr_target_reporter` の制約に残るため、一度取消したコンテンツへの再通報は不可。誤通報で取り消した場合は ADMIN に直接連絡する運用
- `content_reports.status` の enum に `CANCELLED` を追加（F10.1 への追加要請）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | 通報者本人でない |
| 404 | 通報が存在しない |
| 409 | status が PENDING でない |

#### `PATCH /api/v1/system-admin/appeals/{id}/review`

**認可**: SYSTEM_ADMIN

異議申立てをレビューし、承認または却下する。

**リクエストボディ**
```json
{
  "decision": "ACCEPTED",
  "review_note": "再確認の結果、ガイドライン違反に該当しないと判断。コンテンツを復元します。"
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `decision` | String | Yes | `ACCEPTED` / `REJECTED` |
| `review_note` | String | No | レビューメモ（最大2,000文字） |

**レスポンス（200 OK）**: 異議申立ての詳細（`status` 更新済み）

**副作用（ACCEPTED の場合）**:
1. `CONTENT_DELETE` への異議: 対象コンテンツの `deleted_at = NULL`（論理削除解除）
2. `TEMPORARY_FREEZE` への異議: `users.is_frozen = FALSE` + `freeze_until = NULL`（凍結解除）+ Valkey トークン無効化を解除
3. `USER_FREEZE` への異議: `users.is_frozen = FALSE`（凍結解除）+ Valkey トークン無効化を解除
4. 対応する `user_violations` の `is_active = false` に更新
5. ヤバいやつ判定を再実行: 有効な違反数 < `yabai_violation_threshold` → `users.yabai_flagged_at = NULL`（自動解除）+ メール通知 + 監査ログ `YABAI_UNFLAGGED(method=APPEAL)`
6. 申立てユーザーにプッシュ通知（「異議申立てが承認されました」）+ メール（凍結解除の場合はメールのみ到達するため）
7. 元のアクション実行者（ADMIN）に通知（「異議申立てが承認され、対応が取り消されました」）
8. 監査ログ: `APPEAL_REVIEWED`（decision, appeal_id, original_action_id）

**副作用（REJECTED の場合）**:
1. 申立てユーザーにプッシュ通知 + メール（「異議申立てが却下されました」）
2. 監査ログに記録

## 5. ビジネスロジック

### 主要フロー

#### 通報 → レビュー → 対応フロー
```
1. ユーザーが POST /reports で通報（F10.1 で定義）
2. バックエンドが以下を実行:
   a. content_reports に INSERT（status = PENDING）
   b. content_snapshot に対象コンテンツの JSON を保存（証拠保全）
   c. scope_type / scope_id / target_user_id を自動導出
   d. スコープの ADMIN にプッシュ通知
3. 自動エスカレーション判定:
   a. 同一 target_type + target_id の通報件数 ≥ escalation_report_threshold
      → SYSTEM_ADMIN にプッシュ通知（「エスカレーション: {target_type} に {count}件の通報」）
   b. target_user_id の有効な WARNING 累積 ≥ escalation_user_warning_threshold
      → SYSTEM_ADMIN にプッシュ通知（「常習違反者: {user_name} に {count}件の有効な違反」）
4. ADMIN がレビュー開始（PATCH /reports/{id}/review → REVIEWING）
   - 利益相反チェック: target_user_id = ログインユーザー ID の場合 → 403
     （他の ADMIN がいなければ自動的に SYSTEM_ADMIN にエスカレーション）
5. ADMIN が対応を決定:
   a. DISMISS: 問題なし → report_actions に記録。user_violations には記録しない
   b. WARNING: 警告 → report_actions + user_violations に記録。対象ユーザーにプッシュ通知 + メール
      - 通知に guideline_section（違反条項）を含める。ADMIN がテンプレート選択 or 手入力
      - 通知文末に「コンテンツを修正して改善報告を送ることもできます（{self_correct_window_days}日以内）」を追記
   c. CONTENT_DELETE: 論理削除 + 記録。対象ユーザーにプッシュ通知 + メール（「コンテンツが削除されました」）+ 異議申立てメール（appeal_token 付き）
   d. TEMPORARY_FREEZE（SYSTEM_ADMIN のみ）: 期限付き凍結（freeze_until 指定）+ Valkey セッション切断 + 記録 + 異議申立てメール送信
   e. USER_FREEZE（SYSTEM_ADMIN のみ）: 無期限凍結 + Valkey セッション切断 + 記録 + 異議申立てメール送信
6. 同一コンテンツの全通報を一括ステータス更新
7. 通報者に対応結果を通知（アクション種別は非開示。「対応しました」/「問題なしと判断しました」のみ）
```

#### 違反累積による自動エスカレーション
```
1. report_actions に WARNING / CONTENT_DELETE が INSERT された時点で:
   a. user_violations に INSERT（expires_at を設定値に基づいて計算）
   b. 対象ユーザーの有効な違反数を集計:
      - 有効な WARNING 数 ≥ escalation_user_warning_threshold → SYSTEM_ADMIN に通知
      - 有効な CONTENT_DELETE 数 ≥ auto_freeze_content_delete_count
        → SYSTEM_ADMIN に「自動凍結推奨」通知（自動凍結は実行しない。判断は人間が行う）
2. SYSTEM_ADMIN がエスカレーション通知を確認:
   a. ユーザーの違反履歴を GET /system-admin/users/{id}/violations で確認
   b. 必要に応じて USER_FREEZE を実行（PATCH /admin/reports/{id}/resolve）
```

#### 異議申立てフロー
```
1. USER_FREEZE / TEMPORARY_FREEZE / CONTENT_DELETE 実行時:
   a. moderation_appeals に仮レコード INSERT（status = INVITED、appeal_token 生成）
   b. 対象ユーザーにメール送信:
      - 凍結の場合: 「アカウントが凍結されました。異議申立てはこちら: {link}」
      - コンテンツ削除の場合: 「コンテンツが削除されました。異議申立てはこちら: {link}」
2. ユーザーがメールリンクからアクセス:
   a. GET /appeals/{id}（appeal_token 認証）で対象アクションの詳細を確認
   b. PATCH /appeals/{id}/submit で申立て理由を送信 → status = PENDING
3. SYSTEM_ADMIN にプッシュ通知
4. SYSTEM_ADMIN が GET /system-admin/appeals で一覧確認
5. SYSTEM_ADMIN が PATCH /system-admin/appeals/{id}/review でレビュー:
   a. ACCEPTED: 対応取消（コンテンツ復元 / 凍結解除 + Valkey セッション再有効化）+ user_violations 無効化
      + ヤバいやつ判定を再実行（違反数が閾値未満になれば自動解除）
   b. REJECTED: 現状維持。再申立て不可
6. 申立てユーザーにメール + プッシュ通知で結果通知
   （凍結中ユーザーにはメールのみ。プッシュ通知はログイン不可のため届かない）
```

#### WARNING 再レビューフロー
```
1. WARNING を受けたユーザーが POST /warnings/{actionId}/re-review で再レビューを依頼
   - JWT 認証（WARNING 対象ユーザーは凍結されていないため通常ログイン可能）
   - appeal_window_days（デフォルト30日）以内のみ
2. スコープの ADMIN にプッシュ通知
3. ADMIN が PATCH /admin/warning-re-reviews/{id}/review でレビュー:
   - 利益相反チェック: 元の WARNING を実行した ADMIN はレビュー不可（別 ADMIN が対応）
   - スコープ内に別の ADMIN がいない場合 → 自動的に ESCALATED に昇格（手順5へ）
   a. OVERTURNED（取消し）: user_violations の該当レコードを is_active = false + ヤバいやつ再判定
   b. UPHELD（維持）: ユーザーに通知「14日以内に上位管理者に申し立て可能」
4. UPHELD の場合、ユーザーが14日以内に PATCH /warnings/re-reviews/{id}/escalate で SYSTEM_ADMIN に昇格
   - escalation_reason を記入（ADMIN 判断に納得できない理由）
5. SYSTEM_ADMIN が PATCH /system-admin/warning-re-reviews/{id}/review で最終判定:
   a. APPEAL_ACCEPTED: user_violations 無効化 + ヤバいやつ再判定 + 元の ADMIN に通知
   b. APPEAL_REJECTED: 現状維持。再依頼不可
```

#### ヤバいやつ認定フロー
```
1. report_actions に WARNING / CONTENT_DELETE / TEMPORARY_FREEZE / USER_FREEZE が INSERT された時点で:
   a. user_violations に INSERT（既存フロー）
   b. 対象ユーザーの有効な違反数（is_active = true）を集計
   c. 有効な違反数 ≥ yabai_violation_threshold の場合:
      - users.yabai_flagged_at = NOW() を設定（未設定の場合のみ）
      - 対象ユーザーにメール通知:
        「コミュニティガイドライン違反の累積により、あなたの実名がスコープ内のメンバーに
         公開されます。詳細は利用規約をご確認ください。」
      - 監査ログ: YABAI_FLAGGED を記録
2. 実名公開の表示ルール:
   a. ユーザーが1つ以上のチーム/組織に所属 → 所属スコープのメンバーに実名を表示
      - 表示箇所: プロフィール画面・投稿者名の横に ⚠️ アイコン + ツールチップ
        「このユーザーはコミュニティガイドライン違反の履歴があります（実名: {real_name}）」
   b. ユーザーがどのスコープにも未所属 → プラットフォーム全体に実名を公開
3. 解除の3段階:
   a. 【自動解除】有効な違反数が yabai_violation_threshold 未満に減少した場合（時効失効 or 異議申立てACCEPTED）
      → users.yabai_flagged_at = NULL + メール通知 + 監査ログ YABAI_UNFLAGGED(method=AUTO)
   b. 【解除申請】認定後 yabai_unflag_eligible_months（デフォルト3ヶ月）経過 + 新規違反ゼロの場合:
      → ユーザーが POST /yabai/unflag-request で申請（反省文を記載）
      → SYSTEM_ADMIN がレビュー → ACCEPTED なら解除、REJECTED なら3ヶ月後に再申請可能
   c. 【SYSTEM_ADMIN 手動解除】誤認定・特殊事情用の即時解除
      → PATCH /system-admin/users/{id}/yabai/unflag で即座に解除
```

```
1. POST /reports のバリデーション（既存のレートリミットに追加）:

   b. 件数 ≥ daily_report_limit（デフォルト: 20）→ 429 Too Many Requests
      エラー: 「1日の通報件数上限に達しました。翌日に再度お試しください。」

      「ユーザー {display_name} が本日 {count}件の通報を行いました（通報スパムの可能性）」
2. 通報スパム常習者の検出:
   a. 3日連続で daily_report_alert_threshold を超過 → SYSTEM_ADMIN に通知
```

#### コンテンツ自主修正フロー

```
1. WARNING を受けたユーザーが対象コンテンツを編集（既存の編集 API を使用）
2. PATCH /warnings/{actionId}/self-correct で修正完了を通知

3. スコープの ADMIN にプッシュ通知（「自主修正の確認依頼」）
4. ADMIN が修正後のコンテンツを確認:
```

#### 違反時効バッチ
```
1. Spring @Scheduled バッチが日次で実行（例: 03:00 JST）
2. 一括 DB 更新:
   UPDATE user_violations SET is_active = false
   WHERE expires_at < NOW() AND is_active = true
   → 更新された行数をログに記録
3. 失効対象ユーザーの ID リストを取得（STEP 2 で更新された行の DISTINCT user_id）
4. 各ユーザーについてヤバいやつ判定を再実行（ユーザー単位で try-catch）:
   a. 有効な違反数 < yabai_violation_threshold → users.yabai_flagged_at = NULL（解除）
   b. 解除時にメール通知:「実名公開が解除されました」
   c. 監査ログ: YABAI_UNFLAGGED を記録
   d. 失敗時: エラーログ出力 + batch_job_logs に失敗ユーザー ID を記録。次回バッチで再試行
5. 集計結果をアプリケーションログに記録（失効件数・ヤバいやつ解除件数・失敗件数）
6. 失敗件数 > 0 の場合: SYSTEM_ADMIN にプッシュ通知（「違反時効バッチで {count}件の副作用処理に失敗」）
```

#### 期限付き凍結解除バッチ
```
1. Spring @Scheduled バッチが日次で実行（違反時効バッチと同時実行可）
2. 一括 DB 更新:
   UPDATE users SET is_frozen = false, freeze_until = NULL
   WHERE is_frozen = true AND freeze_until IS NOT NULL AND freeze_until < NOW()
   → 更新された行数をログに記録
3. 解除対象ユーザーの ID リストを取得（STEP 2 で更新された行）
4. 各ユーザーについて副作用を実行（ユーザー単位で try-catch）:
   a. Valkey のトークン無効化を解除
   b. ユーザーにメール通知:「アカウントの一時凍結が解除されました。ログインできます。」
   c. 失敗時: エラーログ出力 + batch_job_logs に記録。Valkey 解除のみ失敗の場合は次回ログイン時に自動復旧
5. 集計結果をアプリケーションログに記録（解除件数・通知失敗件数）
```

### 重要な判定ロジック

- **通報の取消し**: 通報者本人が PENDING 状態の通報を `CANCELLED` に遷移可能（論理取消し）。REVIEWING 以降は ADMIN が着手済みのため取消不可。CANCELLED 後も `uq_cr_target_reporter` の制約で同一コンテンツへの再通報は不可（取消し→再通報スパムの防止）。`daily_report_limit` のカウント対象に CANCELLED も含む
- **匿名通報**: 通報者（`reported_by`）の身元は、通報対象者には一切公開しない。ADMIN / SYSTEM_ADMIN の管理画面でのみ閲覧可能。WARNING 通知にも通報者情報を含めない
- **自動エスカレーション**: 閾値は `moderation_settings` で SYSTEM_ADMIN が調整可能。閾値を超えた時点でプッシュ通知を送信するのみで、自動的なアクション（凍結等）は実行しない。最終判断は常に人間が行う
- **グルーピング解決**: 同一コンテンツ（`target_type + target_id`）に複数の通報がある場合、1件の通報で `RESOLVED` / `DISMISSED` を実行すると、関連する全通報のステータスが一括更新される
- **利益相反の防止**: ADMIN が自分のコンテンツに対する通報をレビューすることを禁止（`target_user_id = reviewed_by` の場合は 403）。スコープ内に他の ADMIN がいなければ SYSTEM_ADMIN に自動エスカレーション
- **USER_FREEZE の排他性**: `USER_FREEZE`（無期限凍結）は SYSTEM_ADMIN のみ実行可能。ADMIN が凍結が必要と判断した場合は通報を SYSTEM_ADMIN にエスカレーション
- **期限付き凍結（TEMPORARY_FREEZE）**: `action_type = TEMPORARY_FREEZE` は SYSTEM_ADMIN が実行。`freeze_until` を指定し、期間経過後に日次バッチで自動解除。軽微な違反には一時凍結（デフォルト7日）、重大な違反には無期限凍結（USER_FREEZE）と使い分ける
- **違反の時効**: WARNING は6ヶ月、CONTENT_DELETE は12ヶ月で自動失効。USER_FREEZE は永久記録。時効により軽微な違反が永久に累積することを防止
- **異議申立ての期限**: アクション実行から `appeal_window_days`（デフォルト30日）以内のみ。期限後は異議申立て不可
- **凍結中ユーザーの制限**: `is_frozen = true` のユーザーはログイン不可。異議申立て API（`/api/v1/appeals`）のみ `appeal_token` によるメールリンク認証でアクセス可能。USER_FREEZE 実行時にメールでリンクを送付し、`appeal_window_days` 以内は何度でもアクセス可能
- **削除済みコンテンツの通報**: 論理削除済み（`deleted_at IS NOT NULL`）のコンテンツも通報可能。`content_snapshot` は論理削除済みでもデータが DB に残っているため取得可能。通報一覧に「投稿者が既に削除済み」フラグを表示し、ADMIN の判断材料とする。物理削除済みの場合は通報不可（400）
- **コンテンツ復元**: 異議申立て ACCEPTED 時、`CONTENT_DELETE` で論理削除されたコンテンツの `deleted_at` を NULL に戻す。ただし物理削除済み（S3 ファイル等）の場合は復元不可のため、`review_note` にその旨を記載。F10.1 の `PATCH /admin/reports/{id}/restore-content`（72時間以内の ADMIN 手動復元）で既に復元済みの場合は、異議申立て ACCEPTED 時にコンテンツ復元をスキップ（user_violations の無効化のみ実行）
- **WARNING 再レビュー（2段階異議）**: WARNING は軽微なため `moderation_appeals` ではなく `warning_re_reviews` で対応。(1) まず ADMIN に再レビューを依頼（軽量フロー。元の WARNING 実行者以外の ADMIN がレビュー）。(2) ADMIN が UPHELD（維持）した場合のみ、14日以内に SYSTEM_ADMIN に昇格可能。これにより SYSTEM_ADMIN の負荷を軽減しつつ、ユーザーの不服申立て権を保障する
- **DEPUTY_ADMIN の制限**: `MANAGE_CONTENT` 権限を持つ DEPUTY_ADMIN はスコープ全体（所属チーム/組織）の通報レビュー + WARNING のみ。CONTENT_DELETE は ADMIN、USER_FREEZE / TEMPORARY_FREEZE は SYSTEM_ADMIN の権限。target_type 単位の細分化した権限制御は Phase 10 では対象外
- **通報者への結果通知**: 具体的なアクション種別（WARNING / CONTENT_DELETE / USER_FREEZE）は通報者に非開示。通知文面は2パターンのみ: RESOLVED →「ご報告いただいたコンテンツに対応しました。ご協力ありがとうございます。」 DISMISSED →「ご報告いただいたコンテンツを確認しましたが、現時点ではガイドライン違反に該当しないと判断しました。」 理由: 具体的な対応を開示すると通報がハラスメントの道具として悪用されるリスクがある
- **ヤバいやつ制度（実名公開制裁）**: 有効な違反累積が `yabai_violation_threshold`（デフォルト3件）以上のユーザーは「ヤバいやつ」認定。登録時の実名が所属スコープ内のメンバーに公開される。どのスコープにも未所属の場合はプラットフォーム全体に公開。表示はプロフィール・投稿者名に ⚠️ アイコン + ツールチップで実名を表示。違反が時効で失効し閾値を下回れば自動解除。**抑止力として機能**: ユーザーは「違反を重ねると実名が晒される」ことを登録時に承知しているため、不適切行為の強力な抑止になる
- **ヤバいやつ解除の3段階**: (1) 時効自動解除（違反失効で閾値未満に。最短6ヶ月）。(2) 解除申請（認定後3ヶ月 + 新規違反ゼロで申請可能。SYSTEM_ADMIN レビュー。REJECTED なら3ヶ月後に再申請可）。(3) SYSTEM_ADMIN 手動解除（即時。誤認定・特殊事情用）。解除申請は善行期間の証明として機能し、ユーザーが主体的に改心を示す仕組み
- **ヤバいやつ制度の登録時告知**: ユーザー登録フロー（F01.1）の利用規約同意画面に以下の注意書きを明示表示: 「コミュニティガイドラインに繰り返し違反した場合、登録された実名が所属チーム・組織のメンバーに公開されることがあります（ヤバいやつ制度）。詳細は利用規約をご確認ください。」チェックボックスでの同意を必須とする
- **通報スパム対策**: ユーザーごとの日次通報上限（`daily_report_limit`、デフォルト20件）。1日10件以上で ADMIN に通知、3日連続で SYSTEM_ADMIN にエスカレーション。正当な通報で1日20件を超えることは稀であり、通報スパムによる ADMIN リソースの浪費を防止
- **自分の違反履歴閲覧**: ユーザーは `GET /users/me/violations` で自分の違反履歴を確認可能。ヤバいやつ認定までの残り件数（`remaining_until_yabai`）を表示し、自制を促す。`report_id` / `action_id` / 通報者情報は非公開
- **通報ステータス追跡**: 通報者は `GET /users/me/reports` で自分の通報ステータスを確認可能。`target_id` / 対象コンテンツ詳細 / `action_type` は非公開（追跡防止）。`status` のみ表示
- **ガイドライン条項参照**: WARNING / CONTENT_DELETE 実行時に ADMIN が `guideline_section`（違反条項名）を指定。通知に含めることで、ユーザーが何を改善すべきかを明確にし、同一違反の繰り返しを減らす
- **コンテンツ自主修正**: WARNING を受けたユーザーが対象コンテンツを修正し、ADMIN に確認を依頼するフロー。修正が適切なら WARNING を取り下げ。`self_correct_window_days`（デフォルト7日）以内。`warning_re_reviews`（再レビュー依頼）とは排他
- **一時非表示（CONTENT_HIDE）**: REVIEWING 中のコンテンツを可逆的に非表示にする機能。CONTENT_DELETE（論理削除）より軽い措置。DISMISS / RESOLVED 時に自動復帰。`content_hide_on_review = true` で REVIEWING 遷移時に自動非表示化も可能
- **対応テンプレート**: `moderation_action_templates` で理由別 × アクション別の定型文を管理。ADMIN が1クリックで `note` を入力でき、対応速度と品質のばらつきを改善。`{user_name}` / `{content_type}` 等のプレースホルダ対応
- **ADMIN 間の内部メモ**: `report_internal_notes` で REVIEWING 中の通報に内部メモを追加可能。ADMIN 同士の判断相談用。通報対象者・通報者には一切非公開。編集・削除不可（改ざん防止）
- **通報キューの自動優先度付け**: `report_priority_weights` で理由別のスコアを設定。通報一覧を priority スコア × 通報件数でソートし、HARASSMENT 等の深刻な通報を先に処理できる
- **通報バンドル表示**: 同一 `target_user_id` + 同一 `reason` + `report_bundle_window_hours`（デフォルト24時間）以内の通報を自動グルーピング。ADMIN の通報一覧で折りたたみ表示し、重複レビュー作業を削減
- **月次モデレーションレポート**: 月初にバッチで前月の統計を自動集計（通報件数・対応時間・ADMIN 別処理件数等）。S3 に保存し ADMIN にメール配信。トレンド把握と負荷分散の参考に
- **設定変更履歴**: `moderation_settings_history` で閾値変更の old_value / new_value を自動記録。問題発生時のロールバック判断に使用。`GET /system-admin/moderation/settings/history` で閲覧可能

## 6. セキュリティ考慮事項

- **通報者の匿名性**: `reported_by` は API レスポンスで ADMIN / SYSTEM_ADMIN にのみ返却。通報対象者向けの通知には通報者情報を含めない
- **凍結中ユーザーの API アクセス制御**: Spring Security で `is_frozen` チェックのホワイトリスト（`/api/v1/appeals/**`）を設定。それ以外の全 API は 403 で拒否
- **appeal_token のセキュア保存**: DB には `SHA-256` ハッシュ値のみ保存（パスワードリセットトークンと同様のパターン）。メール送信時に平文トークンを送付し、認証時はハッシュ化して照合。DB 漏洩時のトークン悪用を防止
- **content_snapshot のサニタイズ**: JSON に保存するコンテンツは HTML タグを除去（XSS 防止）。画像 URL は Pre-signed URL ではなく S3 キーで保存（URL 期限切れ対策）
- **レートリミット**: 通報作成は Bucket4j で 1分間に5回（通報スパム防止）。異議申立ては 1日に3回。モデレーション API は 1分間に30回
- **監査ログ**: 全アクション（`REPORT_ACTION_TAKEN`）、異議申立て（`APPEAL_CREATED` / `APPEAL_REVIEWED`）、エスカレーション（`REPORT_ESCALATED`）を `audit_logs` に記録
- **権限の段階性**: DEPUTY_ADMIN(WARNING のみ) → ADMIN(WARNING + CONTENT_DELETE) → SYSTEM_ADMIN(全アクション + USER_FREEZE) の3段階。上位権限のアクションを下位ロールが実行しようとした場合は 403
- **ヤバいやつ制度のプライバシー保護**: 実名公開は `yabai_flagged_at IS NOT NULL` のユーザーのみ。公開範囲はスコープ内に限定（全体公開はスコープ未所属の場合のみ）。実名以外の個人情報（メールアドレス・電話番号等）は公開しない。制度の法的根拠は利用規約への事前同意（登録時のチェックボックス）で担保
- **通報スパムの検出と対策**: レートリミット（1分5回）に加えて日次上限（20件/日）を適用。日次10件超過で ADMIN 通知、3日連続超過で SYSTEM_ADMIN エスカレーション
