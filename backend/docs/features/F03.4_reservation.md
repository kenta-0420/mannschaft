# F03.4: 予約管理

> **ステータス**: 🟡 設計中

> **最終更新**: 2026-03-14
> **関連ドキュメント**: F03.5 シフト管理（スタッフ稼働連携）、F04.3 プッシュ通知（リマインド）、F07.1 サービス履歴（来店記録連携）、F07.4 カルテ（次回推奨メモ連携）、F01.3 テンプレート・モジュール（選択式モジュール #3）

## 1. 概要

整骨院・美容室・飲食店・ジム・マンション共用施設等のサービス業チームが、時間枠（スロット）を定義し、メンバーやサポーターが予約を申し込む機能。チームは最大5本の予約ライン（席・ベッド・施術台等）を定義し、ユーザーは空きラインを選んで予約する。ADMIN はスタッフ別・日付別の予約枠を30分単位で作成・管理し、メンバーは空き枠を確認して予約・キャンセルを行う。予約時間が近づいたらプッシュ通知でリマインドし、来店後はサービス履歴（F07.1）やカルテ（F07.4）と連携する。

**役割分離**: チームアカウント = 枠管理（提供側）、個人アカウント = 予約のみ（利用側）。

### テーブル定義

#### `reservation_lines`

チームが保有する予約ライン（席・ベッド・施術台・テーブル等）。1チームあたり最大5ラインまで定義可能。各ラインは独立して予約可能で、ユーザーが予約時にどのラインを使うかを選択する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE |
| `name` | VARCHAR(50) | NO | — | ライン名（例: 「ベッドA」「席1」「テーブル2」） |
| `description` | VARCHAR(200) | YES | NULL | ライン説明（例: 「窓際の席」「個室」） |
| `display_order` | INT UNSIGNED | NO | 1 | 表示順（1〜5） |
| `is_active` | BOOLEAN | NO | TRUE | 有効/無効フラグ（一時的にラインを閉鎖する場合に FALSE） |
| `default_staff_user_id` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL（このラインのデフォルト担当スタッフ。NULL = 未指定） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_rl_team (team_id, display_order)           -- チーム別ライン一覧（表示順）
INDEX idx_rl_team_active (team_id, is_active)        -- 有効ライン検索
```

**制約・備考**
- 1チームあたり最大5ラインまで。6ライン目の作成は 400 エラー
- `display_order` はチーム内で 1〜5 の範囲。Service 層で一意性を保証（DB UNIQUE 制約は不要。論理削除との組み合わせが複雑になるため）
- `is_active = FALSE` のラインは新規予約の対象外。既存予約は維持される
- `default_staff_user_id`: 予約枠作成時に `staff_user_id` を省略した場合、このラインのデフォルトスタッフが適用される（省略可能な便利機能）
- ライン削除: 未来の予約が存在する場合は論理削除不可（409 エラー）。先に予約をキャンセルする運用
- **ライン名の例**: 整骨院 →「ベッド1」「ベッド2」「ベッド3」、飲食店 →「テーブルA」「テーブルB」「カウンター」、美容室 →「席1」「席2」、マンション共用施設 →「会議室」「多目的ホール」

#### `reservation_slots`

チーム ADMIN が作成する予約枠。1枠 = 特定日時の特定スタッフ（または指名なし共通枠）の予約可能枠。予約時間は **30分単位** で設定。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE |
| `staff_user_id` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL（担当スタッフ。NULL = 指名なし共通枠） |
| `title` | VARCHAR(200) | YES | NULL | 枠タイトル（例: 「整体60分コース」「ランチ予約」。NULL の場合はスタッフ名 + 時間帯で自動表示） |
| `slot_date` | DATE | NO | — | 予約枠の日付 |
| `start_time` | TIME | NO | — | 開始時刻（30分単位: 00分 or 30分） |
| `end_time` | TIME | NO | — | 終了時刻（30分単位: 00分 or 30分） |
| `booked_count` | INT UNSIGNED | NO | 0 | 現在の予約確定数（denormalize。チームの有効ライン数と比較して FULL 判定） |
| `slot_status` | ENUM('AVAILABLE', 'FULL', 'CLOSED') | NO | 'AVAILABLE' | 枠ステータス |
| `recurrence_rule` | JSON | YES | NULL | 繰り返しルール（後述）。NULL = 単発枠 |
| `parent_slot_id` | BIGINT UNSIGNED | YES | NULL | FK → reservation_slots（繰り返し親枠。自己参照）。ON DELETE RESTRICT |
| `is_exception` | BOOLEAN | NO | FALSE | 繰り返し展開後に個別編集された枠か |
| `price` | DECIMAL(10,2) | YES | NULL | 予約料金（表示用。決済は F08 会費・決済モジュール連携。NULL = 無料 or 料金表示なし） |
| `closed_reason` | VARCHAR(20) | YES | NULL | CLOSE 原因（`MANUAL` / `BUSINESS_HOURS` / `BLOCKED`）。NULL = 非CLOSED |
| `note` | TEXT | YES | NULL | ADMIN 向け枠メモ（予約者には非公開） |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_rs_team_date (team_id, slot_date, start_time)            -- チーム別日付順の枠一覧
INDEX idx_rs_staff_date (staff_user_id, slot_date, start_time)     -- スタッフ別の枠一覧
INDEX idx_rs_status (team_id, slot_status, slot_date)              -- 空き枠検索
INDEX idx_rs_parent (parent_slot_id)                                -- 繰り返し親枠からの子枠検索
UNIQUE KEY uq_rs_parent_date (parent_slot_id, slot_date, start_time) -- 繰り返し展開時の重複防止
```

**制約・備考**
- 論理削除: `deleted_at DATETIME nullable`（SoftDeletableEntity 適用）
- **30分単位バリデーション**: `start_time` と `end_time` の分は `00` または `30` のみ許可。それ以外は 400 エラー（例: 10:00, 10:30, 11:00 は OK、10:15 は NG）。最小枠は30分（`end_time - start_time >= 30分`）
- `start_time < end_time` のバリデーション（Service 層で検証。日をまたぐ枠は非対応）
- `slot_status` の自動遷移:
  - `booked_count` がチームの有効ライン数（`reservation_lines` で `is_active = TRUE` の件数）に達したら `FULL` に自動更新
  - キャンセルで `booked_count` が有効ライン数未満に戻ったら `AVAILABLE` に自動復帰
  - ADMIN が手動で `CLOSED` に設定可能（`closed_reason = 'MANUAL'`）。`CLOSED` の枠は新規予約不可
  - 営業時間変更による自動 CLOSE: `closed_reason = 'BUSINESS_HOURS'`
  - ブロック時間帯設定による自動 CLOSE: `closed_reason = 'BLOCKED'`
  - REOPEN / ブロック解除 / 営業時間復帰時: `closed_reason` を NULL に戻す。復帰対象は `closed_reason` の値で判別し、他の原因で CLOSE された枠は復帰しない
  - ライン無効化時: 有効ライン数が減少した結果 `booked_count >= 有効ライン数` になった場合は `FULL` に自動更新
- `staff_user_id` は `team_memberships` で当該チームに所属する MEMBER/ADMIN/DEPUTY_ADMIN を参照。退職等でチーム脱退した場合、既存予約枠は保持するが新規作成は不可（Service 層で所属チェック）
- `slot_date` は当日以降のみ作成可能。過去日の枠作成は 400 エラー
- **同一スタッフの枠重複チェック**: 同一 `staff_user_id` × `slot_date` で時間が重複する枠の作成を禁止（Service 層で検証）。共通枠（`staff_user_id = NULL`）同士の重複は許可（異なるメニューの並行予約を想定）
- **`booked_count` の並行更新**: 予約申込時は `UPDATE reservation_slots SET booked_count = booked_count + 1, slot_status = IF(booked_count + 1 >= :activeLineCount, 'FULL', slot_status) WHERE id = :slotId AND slot_status = 'AVAILABLE' AND deleted_at IS NULL` の1 SQL で実行。`affected rows = 0` の場合は 409 エラー（他ユーザーが先に予約済み or CLOSED）。キャンセル時は `booked_count = booked_count - 1` + `FULL → AVAILABLE` 復帰を同様にアトミック更新。`SELECT FOR UPDATE` によるペシミスティックロックは不要（単一行のアトミック更新で十分）

#### 繰り返しルール（`recurrence_rule`）

F03.1 スケジュールの繰り返しルールと同一フォーマット。

```json
{
  "type": "WEEKLY",
  "interval": 1,
  "days_of_week": ["MON", "WED", "FRI"],
  "end_type": "DATE",
  "end_date": "2026-06-30"
}
```

| フィールド | 型 | 説明 |
|-----------|---|------|
| `type` | String | `DAILY` / `WEEKLY` / `MONTHLY` |
| `interval` | Integer | 繰り返し間隔（1 = 毎週、2 = 隔週） |
| `days_of_week` | String[] | `WEEKLY` 時の対象曜日 |
| `end_type` | String | `DATE`（日付指定）/ `COUNT`（回数指定） |
| `end_date` | String | `end_type = DATE` の場合の終了日 |
| `end_count` | Integer | `end_type = COUNT` の場合の回数 |

- 繰り返し展開は最大90日先まで。日次バッチ（AM 0:00 JST）で7日分を先行展開
- 展開された子枠は `parent_slot_id` で親枠を参照。`is_exception = FALSE` の状態で生成
- 個別枠の変更は `is_exception = TRUE` に設定。親枠の変更は `is_exception = FALSE` の子枠に伝播
- 親枠を削除した場合: `ON DELETE RESTRICT` のため、未予約の子枠を先に削除する必要がある。予約が入っている子枠がある場合は 409 エラー
- **営業時間・定休日との連携**:
  - 繰り返し枠作成時: `recurrence_rule.days_of_week` に定休日（`reservation_business_hours.is_open = FALSE`）が含まれている場合は 400 エラー（「{曜日}は定休日です」）
  - 繰り返し展開バッチ: 子枠生成時に営業時間と `reservation_blocked_times` を再チェックし、定休日やブロック時間帯に該当する日の子枠は生成しない（スキップ）
  - 営業時間の後変更（定休日追加等）: 既存の繰り返し親枠の `recurrence_rule` は変更しない。未展開の子枠がバッチで生成される際に自動スキップされる。既に展開済みの子枠は営業時間変更時の自動 CLOSE ロジック（`closed_reason = 'BUSINESS_HOURS'`）で対応

#### `reservations`

ユーザーの予約レコード。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `reservation_slot_id` | BIGINT UNSIGNED | NO | — | FK → reservation_slots。ON DELETE RESTRICT |
| `line_id` | BIGINT UNSIGNED | NO | — | FK → reservation_lines。ON DELETE RESTRICT（予約ライン。ユーザーが選択） |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE（denormalize: 検索高速化用） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT（予約者） |
| `status` | ENUM('PENDING', 'CONFIRMED', 'CANCELLED', 'COMPLETED', 'NO_SHOW') | NO | — | 予約ステータス |
| `booked_at` | DATETIME | NO | CURRENT_TIMESTAMP | 予約申込日時 |
| `confirmed_at` | DATETIME | YES | NULL | 確定日時（自動確定 or ADMIN 承認時に設定） |
| `cancelled_at` | DATETIME | YES | NULL | キャンセル日時 |
| `cancel_reason` | VARCHAR(500) | YES | NULL | キャンセル理由 |
| `cancelled_by` | ENUM('USER', 'ADMIN', 'SYSTEM') | YES | NULL | キャンセル実行者種別 |
| `completed_at` | DATETIME | YES | NULL | 完了日時（来店確認後） |
| `user_note` | VARCHAR(500) | YES | NULL | 予約者からのメモ（希望・症状等） |
| `admin_note` | VARCHAR(500) | YES | NULL | ADMIN 向け内部メモ（予約者には非公開） |
| `service_record_id` | BIGINT UNSIGNED | YES | NULL | FK → service_records（来店後にサービス履歴と紐付け） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除（SoftDeletableEntity 適用） |
| `active_unique_key` | TINYINT | — | GENERATED | `GENERATED ALWAYS AS (IF(deleted_at IS NULL, 1, NULL)) VIRTUAL`。パーシャルユニーク用 |

**インデックス**
```sql
UNIQUE KEY uq_rv_slot_line_active (reservation_slot_id, line_id, active_unique_key)  -- 有効予約の重複防止（生成カラム方式）
INDEX idx_rv_user_status (user_id, status, booked_at DESC)             -- ユーザー別予約一覧
INDEX idx_rv_team_date (team_id, status, booked_at DESC)               -- チーム別予約一覧
INDEX idx_rv_slot (reservation_slot_id, status)                        -- 枠別予約一覧
INDEX idx_rv_line (line_id, status)                                    -- ライン別予約一覧
INDEX idx_rv_upcoming (user_id, status, reservation_slot_id)           -- 直近予約検索
INDEX idx_rv_service_record (service_record_id)                        -- サービス履歴連携
```

**制約・備考**
- 論理削除: `deleted_at DATETIME nullable`（SoftDeletableEntity 適用）
- `uq_rv_slot_line_active`: 同一枠・同一ラインに対する有効予約の重複を防止。MySQL では NULL 同士は UNIQUE 比較で distinct 扱い（= 重複とみなされない）のため、`deleted_at` を直接 UNIQUE に含める方式では機能しない。代わりに生成カラム `active_unique_key`（有効時 = 1、削除時 = NULL）を使用することで、有効な予約のみで一意性を保証し、論理削除済みの行は制約をすり抜ける
- 同一ユーザーが同一枠の**異なるライン**を予約することは可能（例: 飲食店で2テーブル予約）。`max_active_reservations` 設定で全体の予約数上限を制御
- `status` のライフサイクル:
  - **自動確定モード**（デフォルト）: 予約申込時に即座に `CONFIRMED`（PENDING をスキップ）
  - **手動承認モード**: `PENDING` → ADMIN が承認 → `CONFIRMED` / ADMIN が拒否 → `CANCELLED`
  - `CONFIRMED` → 来店 → `COMPLETED` / 来店なし → `NO_SHOW`
  - `PENDING` / `CONFIRMED` → ユーザーまたは ADMIN がキャンセル → `CANCELLED`
- **承認モード設定**: チーム単位で設定（`reservation_settings` は `teams` テーブルの拡張カラムではなく、`team_settings` JSON カラムまたは専用設定テーブルで管理。詳細は Section 5 の予約管理設定を参照）
- `service_record_id` は来店完了後に ADMIN がサービス履歴を作成した際にリンク（NULL の場合はサービス履歴未作成）
- **キャンセルポリシー**: チーム設定で `cancel_deadline_hours`（予約開始 N 時間前までキャンセル可能）を設定可能。デフォルト: 24時間前。期限後のユーザーキャンセルは 400 エラー（ADMIN はいつでもキャンセル可能）

#### `reservation_business_hours`

チームの曜日別予約受付時間（営業時間）。予約枠の作成・予約申込時にこの時間帯内であることをバリデーションする。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE |
| `day_of_week` | VARCHAR(3) | NO | — | 曜日（`MON` / `TUE` / `WED` / `THU` / `FRI` / `SAT` / `SUN`） |
| `is_open` | BOOLEAN | NO | TRUE | 営業日か（FALSE = 定休日。予約枠作成・予約申込不可） |
| `open_time` | TIME | YES | NULL | 受付開始時刻（`is_open = TRUE` の場合は必須。30分単位） |
| `close_time` | TIME | YES | NULL | 受付終了時刻（`is_open = TRUE` の場合は必須。30分単位） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_rbh_team_day (team_id, day_of_week)   -- チーム×曜日で一意
```

**制約・備考**
- チーム作成時（予約モジュール有効化時）にデフォルト7レコードを自動 INSERT:
  - 月〜金: `is_open = TRUE`, `open_time = '09:00'`, `close_time = '18:00'`
  - 土・日: `is_open = FALSE`, `open_time = NULL`, `close_time = NULL`
- `is_open = FALSE` の曜日は `open_time` / `close_time` を NULL にする（Service 層で検証）
- `open_time < close_time` のバリデーション（日をまたぐ営業時間は非対応）
- `open_time` / `close_time` は30分単位（`:00` or `:30`）
- 予約枠作成時: 枠の `start_time` 〜 `end_time` が当該曜日の `open_time` 〜 `close_time` 内に収まっていない場合は 400 エラー
- 予約申込時: 枠自体が営業時間内に作成されているため、通常はバリデーション不要。ただし営業時間を後から変更した場合に既存枠が範囲外になる可能性があるため、枠のステータスで制御（後述）

#### `reservation_blocked_times`

ADMIN が手動で設定する予約不可時間帯。臨時休業・特別休暇・メンテナンス・年末年始等に使用。営業時間の例外（オーバーライド）として機能する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE |
| `blocked_date` | DATE | NO | — | 予約不可日 |
| `start_time` | TIME | YES | NULL | 不可開始時刻（NULL = 終日不可） |
| `end_time` | TIME | YES | NULL | 不可終了時刻（NULL = 終日不可） |
| `reason` | VARCHAR(200) | YES | NULL | 理由（例: 「臨時休業」「スタッフ研修」「年末年始休暇」） |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_rbt_team_date (team_id, blocked_date)       -- チーム別日付検索
INDEX idx_rbt_date_range (team_id, blocked_date, start_time, end_time)  -- 時間帯重複チェック
```

**制約・備考**
- `start_time` と `end_time` が両方 NULL: 終日予約不可（全日ブロック）
- `start_time` と `end_time` が両方指定: その時間帯のみ予約不可（部分ブロック）
- 片方だけ NULL は不可（Service 層で検証: 両方 NULL or 両方指定）
- `start_time` / `end_time` は30分単位（`:00` or `:30`）
- `blocked_date` は当日以降のみ作成可能。過去日の作成は 400 エラー
- 予約枠作成時: ブロック時間帯と重複する枠の作成は 400 エラー
- 既存予約枠への影響: ブロック設定後に該当時間帯の既存枠は `CLOSED` に自動更新。ブロック解除時は `AVAILABLE` に復帰（予約が入っていれば `booked_count` に基づき判定）
- 同一日に複数のブロックを設定可能（例: 12:00-13:00 と 17:00-18:00 をそれぞれブロック）

#### `reservation_reminders`

予約のリマインド通知管理。予約確定時に自動生成される。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `reservation_id` | BIGINT UNSIGNED | NO | — | FK → reservations。ON DELETE CASCADE |
| `remind_at` | DATETIME | NO | — | リマインド送信予定日時 |
| `notification_id` | BIGINT UNSIGNED | YES | NULL | FK → notifications（送信後に紐付け） |
| `status` | ENUM('PENDING', 'SENT', 'CANCELLED') | NO | 'PENDING' | リマインドステータス |
| `sent_at` | DATETIME | YES | NULL | 実際の送信日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_rr_remind (status, remind_at)              -- リマインドバッチのスキャン用
INDEX idx_rr_reservation (reservation_id)             -- 予約別リマインド一覧
```

**制約・備考**
- 予約確定時（`CONFIRMED`）にデフォルトで2件のリマインドを自動生成:
  - 1日前（前日 AM 9:00 JST）
  - 1時間前（開始1時間前）
- チーム設定でリマインドタイミングをカスタマイズ可能（`remind_before_hours: [24, 1]` のように配列で複数指定）
- 予約キャンセル時: 未送信（`PENDING`）のリマインドを `CANCELLED` に一括更新
- 予約変更時: 既存リマインドを `CANCELLED` → 新しい日時で再生成
- `notification_type = RESERVATION_REMINDER`（F04.3 に追加）

2. 新枠の `booked_count` を +1 アトミック更新。有効ライン数に達した場合は `FULL` に遷移
3. `reservations` の `reservation_slot_id`、`line_id` を更新。`status` は変更なし（CONFIRMED のまま）
4. 旧リマインドを `CANCELLED` → 新しい日時でリマインドを再生成
5. 予約者に `RESERVATION_RESCHEDULED` 通知を送信（旧→新の日時を含む）
6. 担当スタッフにも変更通知を送信

※ キャンセル期限は**旧枠**の `start_time` 基準で判定（旧枠の直前に変更されるのを防ぐため）。
※ 手動承認モードの場合、`PENDING` 予約のリスケジュールは再承認不要（PENDING のまま移動）。

## 5. ビジネスロジック

### 主要フロー

#### 予約枠作成フロー
```
1. ADMIN が予約枠を作成（単発 or 繰り返し）
   ※ チームに reservation_lines が1件以上存在していることが前提。0件の場合は「先にラインを作成してください」で 400 エラー
2. バリデーション:
   a. staff_user_id がチームに所属しているか
   b. start_time < end_time か
   c. start_time / end_time が30分単位か（分が 00 or 30）
   d. slot_date が今日以降か
   e. 同一スタッフの同日同時間帯に重複枠がないか
   f. slot_date の曜日が営業日（reservation_business_hours.is_open = TRUE）か
   g. start_time >= open_time かつ end_time <= close_time（営業時間内か）
   h. slot_date + start_time 〜 end_time が reservation_blocked_times に該当しないか
3. 繰り返しの場合:
   a. recurrence_rule に基づいて最大90日先までの子枠を一括生成
   b. 各子枠に parent_slot_id を設定
   c. 繰り返し展開バッチ（日次 AM 0:00 JST）で7日分を先行展開
4. reservation_slots に INSERT
```

#### 予約申込フロー
```
1. メンバーが空き枠を選択し、利用するラインを選んで予約申込
2. バリデーション:
   a. 予約枠が AVAILABLE か（CLOSED でないか）
   b. 指定 line_id がチームの有効ライン（is_active = TRUE）か
   c. 指定ラインがこの枠で未予約か（uq_rv_slot_line_active で保証）
   d. 予約枠の日時が未来か
   e. allow_same_day_booking = FALSE の場合、slot_date が明日以降か
   f. min_advance_hours > 0 の場合、予約枠開始時刻まで min_advance_hours 時間以上あるか
   g. ユーザーの同時有効予約数が max_active_reservations 以下か
3. reservations に INSERT（line_id を含む）
   a. 自動確定モード: status = CONFIRMED、confirmed_at = NOW()
   b. 手動承認モード: status = PENDING
4. reservation_slots.booked_count を +1 アトミック更新
   a. booked_count がチームの有効ライン数に達した場合、slot_status = FULL に自動更新
5. 自動確定モードの場合:
   a. reservation_reminders にデフォルトリマインドを INSERT
   b. ユーザーに予約確定通知を送信（notification_type = RESERVATION_CONFIRMED）
   c. 担当スタッフにも予約通知を送信
6. 手動承認モードの場合:
   a. ADMIN に承認リクエスト通知を送信（notification_type = RESERVATION_PENDING）
   b. ADMIN が承認 → CONFIRMED に遷移、リマインド生成、ユーザーに確定通知
   c. ADMIN が拒否 → CANCELLED に遷移、booked_count -1、ユーザーに拒否通知
```

#### 予約キャンセルフロー
```
1. ユーザーまたは ADMIN がキャンセル実行
2. バリデーション:
   a. status が PENDING or CONFIRMED か
   b. ユーザーの場合: キャンセル期限（cancel_deadline_hours）を超過していないか
3. reservations.status = CANCELLED、cancelled_at/cancel_reason/cancelled_by を更新
4. reservation_slots.booked_count を -1 アトミック更新
   a. slot_status が FULL だった場合、AVAILABLE に復帰
5. 未送信の reservation_reminders.status を CANCELLED に一括更新
6. 通知:
   a. ユーザーキャンセルの場合: 担当スタッフ・ADMIN に通知
   b. ADMIN キャンセルの場合: ユーザーに通知（cancel_reason 付き）
```

#### 来店完了フロー
```
1. ADMIN が来店完了処理を実行
2. reservations.status = COMPLETED、completed_at = NOW()
3. service_record_id がリクエストに含まれている場合、紐付けを更新
4. F07.1 サービス履歴が未作成の場合、ダッシュボードに「サービス履歴を追加しますか？」のプロンプトを表示
```

#### ノーショー処理フロー
```
1. 自動検出: ノーショー自動検出バッチ（毎時）が予約枠終了から auto_no_show_hours 時間経過した
   CONFIRMED 予約を自動で NO_SHOW に更新
   手動: ADMIN が来店なしを確認し、即座に NO_SHOW を記録（バッチ待ちの前に記録可能）
2. reservations.status = NO_SHOW
3. ユーザーに NO_SHOW 通知は送信しない（プライバシー配慮。ADMIN ダッシュボードでのみ表示）
4. stats API の no_show_rate に反映
```

### 重要な判定ロジック

#### 予約管理設定

チーム単位の設定。`teams` テーブルの `reservation_settings` JSON カラム、または `team_feature_settings` テーブルで管理。

| 設定キー | 型 | デフォルト | 説明 |
|---------|---|-----------|------|
| `approval_mode` | String | `AUTO` | `AUTO`（自動確定）/ `MANUAL`（手動承認） |
| `pending_expiry_hours` | Integer | 48 | 手動承認モード時、PENDING 予約が未承認のまま N 時間経過で自動キャンセル（0 = 無効） |
| `cancel_deadline_hours` | Integer | 24 | ユーザーキャンセル可能な期限（予約開始 N 時間前まで） |
| `remind_before_hours` | Integer[] | `[24, 1]` | リマインド送信タイミング（予約開始 N 時間前。複数指定可） |
| `max_active_reservations` | Integer | 5 | 1ユーザーあたりの同時有効予約数上限 |
| `allow_same_day_booking` | Boolean | TRUE | 当日予約を許可するか |
| `min_advance_hours` | Integer | 0 | 予約可能な最短時間（予約開始時刻の N 時間前まで。0 = 制限なし） |
| `default_slot_duration_minutes` | Integer | 30 | 枠作成時のデフォルト時間（分。30分単位） |
| `auto_no_show_hours` | Integer | 2 | 予約枠終了時刻から N 時間後に自動で NO_SHOW にする（0 = 無効、ADMIN 手動のみ） |
| `no_show_threshold` | Integer | 3 | 直近90日間の NO_SHOW 回数がこの値以上のユーザーに予約時に警告表示（0 = 無効） |

#### ノーショー警告の挙動

`no_show_threshold` を超えたユーザーが予約申込する際の警告は**ソフト警告**（予約自体はブロックしない）。

**ユーザー側（フロントエンド）**:
- 予約申込確認画面に「直近の予約で無断キャンセルが複数回記録されています。当日のご来店をお願いいたします。」と警告メッセージを表示
- 予約申込 API 自体は 201 で成功する（ブロックしない）
- ユーザーには具体的な NO_SHOW 回数は表示しない（プライバシー配慮）

**ADMIN 側**:
- 予約一覧・詳細に「このユーザーは直近90日間で N 回ノーショーです」と表示
- ADMIN が手動承認モードに切り替える判断材料にする
- 統計 API の `no_show_rate` でチーム全体のノーショー率を把握

**判定タイミング**: `POST /reservations` の処理中に `reservations` テーブルから `user_id` の直近90日間 `status = NO_SHOW` 件数を COUNT。閾値超過の場合、レスポンスに `no_show_warning: true` を付与（フロントエンドが警告表示に利用）。

**ハード制限にしない理由**: 正当なユーザーが体調不良等のやむを得ない事情で NO_SHOW になるケースがあり、自動ブロックは過剰制裁になる。ADMIN が状況を見て手動承認モードに切り替えるのが適切。

#### リマインドバッチ

- **実行タイミング**: Spring `@Scheduled(fixedRate = 60000)` で1分間隔ポーリング
- **対象**: `reservation_reminders.status = PENDING AND remind_at <= NOW()`
- **処理**: F04.3 通知基盤に `notification_type = RESERVATION_REMINDER` として送信。送信後に `status = SENT`、`sent_at = NOW()`、`notification_id` を更新
- **内容**: 「明日 10:00〜11:00 に〇〇整骨院の予約があります（田中スタッフ / 整体60分コース）」

#### ノーショー自動検出バッチ

- **実行タイミング**: 毎時実行（`@Scheduled(cron = "0 0 * * * *")`）
- **対象**: `reservations.status = CONFIRMED` かつ `reservation_slots.slot_date + end_time` から `auto_no_show_hours` 時間以上経過したレコード
- **処理**: 対象の `reservations.status` を `NO_SHOW` に更新。`completed_at` は設定しない
- **設定**: チーム設定 `auto_no_show_hours`（デフォルト: 2、0 = 無効）で調整可能
- **注意**: ADMIN が先に `COMPLETED` をマークした予約はバッチ対象外（`status = CONFIRMED` のみ対象）

#### PENDING 予約自動期限切れバッチ

- **実行タイミング**: 毎時実行（`@Scheduled(cron = "0 0 * * * *")`）— ノーショーバッチと同じタイミングで実行可能
- **対象**: `reservations.status = PENDING` かつ `booked_at` から `pending_expiry_hours` 時間以上経過したレコード（チーム設定の `pending_expiry_hours` を参照。0 = 無効）
- **処理**: 対象の `reservations.status` を `CANCELLED` に更新。`cancelled_by = 'SYSTEM'`、`cancel_reason = '承認期限切れによる自動キャンセル'`
- **副作用**:
  - `reservation_slots.booked_count` を -1 アトミック更新
  - 予約者に `RESERVATION_EXPIRED` 通知を送信（「予約リクエストが承認期限切れにより自動キャンセルされました。再度予約をお試しください。」）
  - ADMIN に「N件の予約リクエストが期限切れになりました」と通知（承認漏れへの気づきを促す）

#### 繰り返し枠展開バッチ

- **実行タイミング**: 日次 AM 0:00 JST（`@Scheduled(cron = "0 0 0 * * *")`）
- **処理**: `parent_slot_id IS NULL AND recurrence_rule IS NOT NULL` の親枠について、未展開の7日先までの子枠を生成
- **重複防止**: `UNIQUE KEY uq_rs_parent_date` で保証
- **例外枠**: `is_exception = TRUE` の既存子枠は上書きしない

## 6. セキュリティ考慮事項

- **認可チェック**: ReservationSlotService / ReservationService の入り口で `teamId` と `currentUser` の所属・ロールを必ず検証。他チームの予約枠・予約へのアクセスは 403 で拒否
- **予約者本人チェック**: 予約詳細・更新・キャンセルは `reservation.user_id == currentUser.id` または ADMIN+ であることを検証
- **レートリミット**: 予約申込 API（`POST /reservations`）は `Bucket4j` で1ユーザー1分間に5回まで — 予約枠の買い占め防止。空き枠検索 API は1分間に30回まで
- **入力サニタイズ**: `user_note`・`admin_note`・`cancel_reason`・`reservation_blocked_times.reason` は XSS 対策として HTML タグを除去
- **ノーショーデータの保護**: ノーショー回数はユーザー本人には直接表示しない（ADMIN ダッシュボードでのみ参照可能）。ユーザーには「予約時に警告」として間接的にフィードバック
- **監査ログ**: 予約枠の作成・変更・削除、予約の申込・承認・拒否・キャンセル・完了・ノーショー、営業時間変更、ブロック時間帯の追加・削除のすべてを `audit_logs` に記録
- **個人情報**: `user_note`（症状等のセンシティブ情報を含む可能性あり）はスコープ内 ADMIN+ のみアクセス可能。API レスポンスでも権限に応じてフィルタ
