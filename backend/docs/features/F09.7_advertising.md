> **ステータス**: 🟢 設計完了

## 1. 概要

フリープラン利用中のチーム/組織のページにプラットフォーム広告を自動表示し、プラットフォーム収益を確保する機能。有料プラン加入済みのチーム/組織では広告を完全非表示にし、広告非表示を有料プランへのアップグレードインセンティブとして機能させる。

**Phase 9（初期実装）**: Amazon アソシエイト・楽天アフィリエイトの静的バナー表示。SYSTEM_ADMIN が管理ダッシュボード（F10.1）からプロバイダ・タグ ID・配置位置・バナークリエイティブ・有効期間を設定し、フロントエンドが SSR 時に描画する。

**将来拡張（Phase 10+）**: 組織アカウントが広告主として出稿し、ターゲティング配信（組織カテゴリ・地域・ユーザーロール）+ CPM/CPC 課金で本格的な広告プラットフォームに拡張する。将来拡張の DB スキーマ・API は本ドキュメントに設計方針のみ記載し、実装は FUTURE_CONSIDERATIONS.md に委ねる。

**Phase 1〜8 は `FEATURE_V9_ENABLED=false` により UI・API を完全非表示。DB テーブルは Phase 1 で先行作成。**

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `affiliate_configs` | アフィリエイト設定（Amazon アソシエイト等のタグ・配置管理） | あり（`deleted_at`） |

### テーブル定義

#### `affiliate_configs`

SYSTEM_ADMIN が管理するアフィリエイト設定。Amazon アソシエイト・楽天アフィリエイトに対応。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `provider` | VARCHAR(20) | NO | — | アフィリエイトプロバイダ（`AMAZON` / `RAKUTEN`） |
| `tag_id` | VARCHAR(100) | NO | — | アフィリエイトタグ ID（Amazon: 'mannschaft-22'、楽天: アフィリエイト ID） |
| `placement` | VARCHAR(30) | NO | — | 表示配置。`SIDEBAR_RIGHT`（右サイドバー）/ `BANNER_FOOTER`（フッターバナー）/ `BANNER_HEADER`（ヘッダーバナー。将来追加可能） |
| `description` | VARCHAR(200) | YES | NULL | 管理者メモ（配置の用途説明等） |
| `banner_image_url` | VARCHAR(500) | YES | NULL | バナー画像 URL（NULL: FE がプロバイダ別デフォルト画像を使用） |
| `banner_width` | SMALLINT UNSIGNED | YES | NULL | バナー幅 px（NULL: FE がプロバイダ別デフォルトサイズを使用） |
| `banner_height` | SMALLINT UNSIGNED | YES | NULL | バナー高さ px |
| `alt_text` | VARCHAR(200) | YES | NULL | バナー代替テキスト（アクセシビリティ・SEO。NULL: プロバイダ名をデフォルト表示） |
| `is_active` | BOOLEAN | NO | TRUE | 有効/無効（手動トグル。スケジューリングとは独立） |
| `active_from` | DATETIME | YES | NULL | 有効開始日時（NULL: 即時有効） |
| `active_until` | DATETIME | YES | NULL | 有効終了日時（NULL: 無期限） |
| `display_priority` | SMALLINT UNSIGNED | NO | 0 | 同一配置に複数設定がある場合の優先度（昇順。最小値が優先表示） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_affiliate_configs_provider_placement (provider, placement)                  -- プロバイダ×配置の検索用（一意性はアプリ層で保証）
INDEX idx_affiliate_configs_placement_active_priority (placement, is_active, display_priority)  -- 配置別の優先度順取得
```

**制約・備考**
- `provider` は VARCHAR で ENUM ではない（新プロバイダ追加時に ALTER TABLE 不要）
- `placement` も VARCHAR（新しい配置位置の追加を容易にする）
- 同一 `placement` に異なる `provider` を設定可能（例: サイドバーに Amazon、フッターに楽天）。同一 `placement` × `provider` の組み合わせはアプリ層で一意性を保証（INSERT/UPDATE 前に `SELECT ... WHERE provider = ? AND placement = ? AND deleted_at IS NULL` でチェック）。MySQL の UNIQUE KEY に `deleted_at`（NULL許容）を含めると NULL 同士が等しくない扱いとなり制約が機能しないため、DB レベルの一意制約は使用しない
- **設定上限**: 全体で最大20件（アプリ層で検証）。プロバイダ × 配置の組み合わせは現実的に10件未満だが、将来の配置追加を見越して余裕を持たせる
- `tag_id`: プロバイダごとにフォーマットが異なるため汎用 VARCHAR。Amazon は「xxxx-22」形式、楽天はアフィリエイト ID（数値文字列）
- `banner_image_url` / `banner_width` / `banner_height` / `alt_text`: 全て NULL 許容。NULL の場合 FE がプロバイダ別デフォルト（Amazon 公式バナー / 楽天公式バナー）を描画。SYSTEM_ADMIN が管理画面から任意のバナーに差し替え可能（FE デプロイ不要）
- `active_from` / `active_until`: スケジューリング用。`is_active` とは独立して評価。広告が表示されるには「`is_active = TRUE` AND スケジュール期間内」の両方を満たす必要がある。バッチジョブ不要（`/ads/active` のクエリ条件で制御）
- 論理削除: 設定変更の履歴追跡用。`deleted_at` は `NOW(6)`（マイクロ秒精度）で設定

### リクエスト／レスポンス仕様

#### `GET /api/v1/ads/active`

**認可**: 認証不要（公開 API。SSR 時にサーバーサイドから呼び出す）

表示すべき広告設定を取得する。フリープラン判定はフロントエンド側で実行。

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `placement` | String | No | 配置フィルタ（`SIDEBAR_RIGHT` / `BANNER_FOOTER` / 将来追加の配置も対応）。未指定: 全配置 |

**レスポンス（200 OK）**
```json
{
  "data": [
    {
      "provider": "AMAZON",
      "affiliate_url": "https://www.amazon.co.jp/?tag=mannschaft-22",
      "placement": "SIDEBAR_RIGHT",
      "display_priority": 0,
      "banner_image_url": null,
      "banner_width": null,
      "banner_height": null,
      "alt_text": null
    },
    {
      "provider": "RAKUTEN",
      "affiliate_url": "https://hb.afl.rakuten.co.jp/hgc/XXXXXXXX/?pc=https%3A%2F%2Fwww.rakuten.co.jp%2F",
      "placement": "BANNER_FOOTER",
      "display_priority": 0,
      "banner_image_url": "https://example.com/rakuten-banner-728x90.png",
      "banner_width": 728,
      "banner_height": 90,
      "alt_text": "楽天市場でお買い物"
    }
  ]
}
```

**補足**
- `is_active = TRUE` かつ `deleted_at IS NULL` かつスケジュール期間内（`active_from <= NOW()` AND `active_until IS NULL OR active_until >= NOW()`）の設定のみ返却
- `placement` ごとに `display_priority ASC, id ASC`（同一優先度は登録順）でソート
- **内部 DB の `id` は返却しない**: 認証不要の公開 API で AUTO_INCREMENT id を露出すると、レコード数の推測が可能になる。FE は `placement + provider` の組み合わせで一意識別する。Phase 10+ のインプレッション/クリック計測 API では UUID ベースの公開キーを別途導入する
- `affiliate_url`: バックエンドが `tag_id` + プロバイダ固有のベース URL から完成済みアフィリエイトリンクを構築して返却。`tag_id` フィールド自体は返却しない（管理情報と公開情報の分離が目的）
- `banner_image_url` / `banner_width` / `banner_height` / `alt_text`: NULL の場合 FE がプロバイダ別デフォルトバナーを描画。値がある場合はそのまま使用。SYSTEM_ADMIN がバナーを差し替えても FE デプロイ不要
- プロバイダ別の URL 構築ルール:
  - `AMAZON`: `https://www.amazon.co.jp/?tag={tag_id}`
  - `RAKUTEN`: `https://hb.afl.rakuten.co.jp/hgc/{tag_id}/?pc=https%3A%2F%2Fwww.rakuten.co.jp%2F`（`tag_id` = 楽天アフィリエイト ID）
  - 将来追加のプロバイダも同様にバックエンドで URL を構築（FE はリンクをそのまま使用）
- **キャッシュ**: Valkey TTL 10分（キー: `mannschaft:cache:ads:active:{placement}`、placement 未指定時は `mannschaft:cache:ads:active`）。設定変更時にキャッシュ無効化
- **フリープラン判定**: フロントエンド（Nuxt SSR）がチーム/組織のプラン情報を確認し、フリープランの場合のみ広告コンポーネントを描画。バックエンド API は判定しない（`/ads/active` は常に設定を返す）。FE 判定の理由: 広告非表示はユーザーに有利な操作であり改ざんによるセキュリティ被害がない、認証不要にすることで SSR キャッシュ効率を最大化できる、Phase 10+ のターゲティング広告移行時に BE 判定に切り替え予定

## 5. ビジネスロジック

### tag_id バリデーション

プロバイダ別にアフィリエイトタグの形式をバリデーションする（SYSTEM_ADMIN の入力ミス防止。不正な tag_id が設定されると全ユーザーのアフィリエイト収益がゼロになるため影響大）。

```
■ プロバイダ別バリデーションルール:
  - AMAZON: 正規表現 ^[a-zA-Z0-9]+-[0-9]{2}$ （例: mannschaft-22）
  - RAKUTEN: 正規表現 ^[a-f0-9]{8}$ （楽天アフィリエイト ID。8桁16進数）
  - 将来プロバイダ追加時: AffiliateProviderValidator Strategy パターンで拡張

■ 実装方針:
  - POST / PUT 時にプロバイダに応じたバリデーションを実行
  - 不正な場合は 400 Bad Request + fieldErrors（field: "tag_id"）

■ スケジュールバリデーション:
  - active_from / active_until が両方指定された場合: active_from < active_until であること
  - active_until が過去日時の場合: 警告ログを出力するがエラーにはしない（履歴として保持）
```

### 広告表示判定フロー

```
1. フロントエンド（Nuxt SSR）がページレンダリング時に実行:
   a. チーム/組織のプラン情報を Pinia ストアから取得
   b. フリープランの場合のみ広告コンポーネントを描画対象にする
   c. `GET /ads/active` を呼び出し（Valkey キャッシュで高速応答）
   d. placement に応じた位置に広告バナーを挿入

2. SSR 描画:
   a. バックエンドから返却された `affiliate_url` をそのまま HTML に埋め込み
   b. `banner_image_url` が返却されている場合はその画像を表示。NULL の場合は FE がプロバイダ別デフォルトバナーを描画:
      - AMAZON: Amazon 公式バナーコード（PA-API または静的バナー）
      - RAKUTEN: 楽天公式バナーコード
   c. `banner_width` / `banner_height` が指定されている場合は明示的にサイズ指定（CLS 防止）
   d. `alt_text` が NULL の場合はプロバイダ名をデフォルト表示（例: 「Amazon」「楽天市場」）
   e. FE 側で tag_id を直接扱うことはない（URL 構築はバックエンド責務）

3. アップグレード CTA（Call to Action）:
   a. 広告バナーの下部に小さなテキストリンクを常時表示: 「広告なしで使う →」
   b. F08.2 のプランアップグレード画面に直接遷移
   c. フリープランユーザーが広告を見るたびに有料プランの存在を自然に認知させる
   d. 実装: FE のみ（BE 変更なし。広告コンポーネント内に静的リンクを配置）

4. SPA ナビゲーション時:
   a. Pinia ストアのキャッシュから広告設定を使用（API 再呼び出し不要）
   b. プラン変更時（有料 → フリー or フリー → 有料）に広告コンポーネントの表示/非表示を即時切り替え
```

### 広告非表示のトリガー

```
以下のいずれかに該当する場合、広告を非表示:
  - チーム/組織ページ: 当該チーム/組織が有料プラン加入中（subscription_status != 'FREE'）
  - 個人ダッシュボード: ユーザーが所属するチーム/組織のいずれかが有料プラン加入中
    → FE の Pinia ストアで所属先のプラン情報を確認（追加 API 不要）
    → 有料プラン加入済みユーザーに広告を表示するのは UX を損なうため、所属先の最上位プランに準拠
  - FEATURE_V9_ENABLED = false（Phase 1〜8）
  - affiliate_configs に is_active = TRUE かつスケジュール期間内の設定が0件
```

### キャッシュ戦略

```
■ /ads/active のキャッシュ:
  - Valkey キー: mannschaft:cache:ads:active (placement パラメータなし) / mannschaft:cache:ads:active:{placement}
  - TTL: 10分
  - 無効化タイミング: affiliate_configs の CREATE / UPDATE / DELETE / TOGGLE / BULK_TOGGLE 時に全キャッシュを削除
  - キャッシュミス時: DB から SELECT → Valkey にセット → レスポンス返却

■ フロントエンド側:
  - Pinia ストア `adsStore` に広告設定をキャッシュ（TTL: 10分。Valkey TTL と統一し、設定変更の反映遅延を最小化）
  - SSR 初回レンダリング時に API 呼び出し → ストアに保存
  - SPA ナビゲーション時はストアから取得（API 呼び出しなし）

■ グレースフルデグラデーション:
  - Valkey 接続エラー時: DB から直接取得（Valkey 復旧まで毎回 DB アクセス）
  - DB も応答しない場合: 空配列を返却（広告なし表示）。ページ描画はブロックしない
  - FE 側: /ads/active の API エラー時は広告コンポーネントを非表示（ページ本体の描画に影響させない）
  - 監視: Valkey 接続エラーを ERROR ログ出力 + アラート通知
```

## 6. エラーコード定義

| コード | メッセージ | 用途 |
|--------|-----------|------|
| `AFF_001` | 指定されたアフィリエイト設定が見つかりません | GET/PUT/PATCH/DELETE で id 不在時（404） |
| `AFF_002` | 同一プロバイダ×配置の設定が既に存在します | POST/PUT で provider × placement 重複時（409） |
| `AFF_003` | アフィリエイト設定の上限（20件）に達しています | POST で上限超過時（400） |
| `AFF_004` | タグ ID のフォーマットが不正です | POST/PUT で tag_id バリデーション失敗時（400） |
| `AFF_005` | 有効期間の指定が不正です | active_from >= active_until の場合（400） |

## 7. セキュリティ考慮事項

- **SYSTEM_ADMIN 限定**: アフィリエイト設定の CRUD は SYSTEM_ADMIN のみ。一般ユーザーが広告設定を変更・閲覧する API はない（`/ads/active` は設定値のみ返却し、管理情報は含まない）
- **`tag_id` の情報分離**: `/ads/active` では `tag_id` フィールド自体は返却せず、構築済み `affiliate_url` のみを返す。アフィリエイト URL にはタグが含まれるため完全な秘匿ではないが、管理画面情報（`description` 等）と公開情報の分離が目的。SYSTEM_ADMIN API でのみ管理情報を返す
- **XSS 防止**: バナー HTML はフロントエンドのテンプレートで生成。バックエンドが `affiliate_url` を構築する際に `tag_id` を URL エンコードしてからパラメータに埋め込む。`banner_image_url` はバックエンドが URL 形式バリデーション（`https://` で始まること）を実施し、FE は `<img>` の `src` にのみ使用。FE は返却された URL をそのまま `<a>` タグの `href` / `<img>` の `src` に設定し、プロバイダ公式バナーコード以外の任意 HTML は挿入しない
- **広告ブロッカー対策**: CSS クラス名は保守性を維持しつつ、広告専用であることを直接示す名前（`ad-`, `affiliate-`, `banner-` 等）は避ける。汎用的な命名（`promo-card`, `sponsor-section` 等）を使用する。完全なブロッカー回避は目指さない（Amazon URL 自体が検出対象のため効果が限定的）が、最も基本的なフィルタルールには引っかからないようにする。ブロッカー検出時は代替メッセージ（「広告収入でサービスを維持しています」）を表示する
- **レート制限**: `/ads/active` は認証不要の公開 API のため、IP ベースのレート制限を適用（Nginx: 30 req/min per IP）
- **監査ログ**: `AFFILIATE_CONFIG_CREATED` / `AFFILIATE_CONFIG_UPDATED` / `AFFILIATE_CONFIG_DELETED` / `AFFILIATE_CONFIG_TOGGLED` / `AFFILIATE_CONFIG_BULK_TOGGLED` を記録

## 10. 将来拡張: ターゲティング広告プラットフォーム（Phase 10+）

Phase 9 では Amazon アソシエイトの静的バナーのみ。以下は Phase 10+ で実装予定の拡張設計方針。詳細は FUTURE_CONSIDERATIONS.md に記載済み。

- 広告主（組織 ADMIN）: 広告出稿申請・キャンペーン統計閲覧
