# F05.3: 電子印鑑

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 5
> **最終更新**: 2026-03-13

---

## 1. 概要

ユーザーの登録姓・名から印鑑風の SVG 画像を3種類（姓のみ・フルネーム・名のみ）自動生成し、回覧板・議決権行使・カルテ（問診票・同意書）など複数機能で横断利用できる電子押印基盤を提供する。チーム・組織ごとにどの印鑑をデフォルトで使用するか設定でき、同姓メンバーが多い環境でも個人を識別しやすい。押印のたびに日時・対象ドキュメントの証跡ログを記録し、改ざん検知用のハッシュを付与することで、簡易的な真正性を担保する。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全ユーザーの印鑑情報の参照・強制再生成・一括再生成バッチ実行 |
| ADMIN | チーム/組織内メンバーの印鑑参照、押印ログ閲覧、一括検証 |
| DEPUTY_ADMIN | `MANAGE_CONTENT` 権限を持つ場合: 押印ログ閲覧 |
| MEMBER | 自分の印鑑の参照・再生成・スコープ別デフォルト設定、押印操作 |
| SUPPORTER | 自分の印鑑の参照・再生成・スコープ別デフォルト設定、押印操作（回覧板のみ。議決権・カルテは対象外） |
| GUEST | 対象外 |

### 対象レベル
- [x] 組織 (Organization)
- [x] チーム (Team)
- [x] 個人 (Personal)

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `electronic_seals` | ユーザーごとの印鑑 SVG データ（3バリアント） | あり |
| `seal_scope_defaults` | チーム/組織ごとの使用印鑑デフォルト設定 | なし |
| `seal_stamp_logs` | 押印履歴（いつ・誰が・どの印鑑で・どのドキュメントに押印したか） | なし |

### テーブル定義

#### `electronic_seals`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT（論理削除で対応するため物理削除不可） |
| `variant` | ENUM('LAST_NAME', 'FULL_NAME', 'FIRST_NAME') | NO | — | 印鑑バリアント |
| `display_text` | VARCHAR(50) | NO | — | 印鑑に表示するテキスト（例: 「田中」「田中太郎」「太郎」） |
| `svg_data` | TEXT | NO | — | 印鑑風 SVG データ |
| `seal_hash` | VARCHAR(64) | NO | — | SVG データの SHA-256 ハッシュ（改ざん検知用） |
| `generation_version` | INT UNSIGNED | NO | 1 | SVG 生成アルゴリズムのバージョン（フォント・テンプレート変更時にインクリメント） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
UNIQUE KEY uq_electronic_seals_user_variant (user_id, variant)
```

**制約・備考**
- 1ユーザーにつき3件（バリアント別 UNIQUE 制約）。論理削除後も既存レコードを復元する方式のため、削除→再INSERT は行わない（Service 層で徹底）
- `generation_version` は SVG 生成アルゴリズム（フォント・レイアウト・テンプレート）の変更時にインクリメント。一括再生成バッチで「旧バージョンのみ再生成」のフィルタリングに使用
- ユーザー登録時に3バリアントすべてを自動生成（ApplicationEvent `UserCreatedEvent` を購読）
- `display_text` はバリアントに応じて自動設定: `LAST_NAME` → `users.last_name`、`FULL_NAME` → `users.last_name` + `users.first_name`、`FIRST_NAME` → `users.first_name`
- `seal_hash` は SVG データから算出。押印時に照合することで、押印後の印鑑すり替えを検知可能
- 論理削除（SoftDeletableEntity）: ユーザー退会時に全3件の `deleted_at` をセット。印鑑取得 API は 404 を返すが、verify API では `SEAL_NOT_FOUND` として応答し過去の押印検証は継続可能

#### `seal_scope_defaults`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users |
| `scope_type` | ENUM('DEFAULT', 'TEAM', 'ORGANIZATION') | NO | — | スコープ種別 |
| `scope_id` | BIGINT UNSIGNED | YES | NULL | FK → teams または organizations（`DEFAULT` の場合は NULL） |
| `seal_id` | BIGINT UNSIGNED | NO | — | FK → electronic_seals（使用する印鑑） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_seal_scope_defaults_user_scope (user_id, scope_type, scope_id)
INDEX idx_seal_scope_defaults_seal (seal_id)
```

**制約・備考**
- `scope_type = 'DEFAULT'` の場合、`scope_id` は NULL（ユーザーのグローバルデフォルト）
- `scope_type = 'TEAM'` の場合、`scope_id` は `teams.id`
- `scope_type = 'ORGANIZATION'` の場合、`scope_id` は `organizations.id`
- ユーザー登録時に `DEFAULT` 行を自動作成（`LAST_NAME` バリアントの seal_id を初期値に設定）
- `seal_id` が指す `electronic_seals` が論理削除された場合、押印時にフォールバック（後述）

#### `seal_stamp_logs`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（押印者）。ON DELETE RESTRICT（退会時も証跡を保持するため削除不可） |
| `seal_id` | BIGINT UNSIGNED | NO | — | FK → electronic_seals（使用した印鑑）。ON DELETE RESTRICT |
| `seal_hash` | VARCHAR(64) | NO | — | 押印時点の印鑑ハッシュ（事後検証用スナップショット） |
| `target_type` | VARCHAR(50) | NO | — | 対象ドキュメント種別（例: `CIRCULATION`, `VOTE`, `MEDICAL_CONSENT`） |
| `target_id` | BIGINT UNSIGNED | NO | — | 対象ドキュメントの PK |
| `ip_address` | VARCHAR(45) | NO | — | 押印時の IP アドレス |
| `user_agent` | VARCHAR(500) | YES | NULL | 押印時の User-Agent |
| `stamped_at` | DATETIME | NO | CURRENT_TIMESTAMP | 押印日時 |
| `revoked_at` | DATETIME | YES | NULL | 取消日時（NULL = 有効） |
| `revoked_by` | ENUM('SELF', 'ADMIN') | YES | NULL | 取消実行者種別（NULL = 未取消） |
| `revoked_reason` | VARCHAR(200) | YES | NULL | 取消理由 |

**インデックス**
```sql
INDEX idx_seal_stamp_logs_user (user_id, stamped_at DESC)        -- ユーザー別の押印履歴
INDEX idx_seal_stamp_logs_target (target_type, target_id)         -- ドキュメント別の押印一覧
INDEX idx_seal_stamp_logs_seal (seal_id)                          -- 印鑑別の使用履歴
```

**制約・備考**
- 同一ユーザー × 同一ドキュメントへの重複押印は**アプリケーション層で制御**（ドキュメント種別ごとに重複可否ポリシーが異なるため DB 制約ではなくサービス層で判定）
- `seal_id` でどのバリアントの印鑑で押印したかを記録。`seal_hash` は押印時点のスナップショット（再生成後も過去の押印記録と照合可能）
- 原則 INSERT のみ — 監査証跡として DELETE 不可。`revoked_at` / `revoked_by` / `revoked_reason` の UPDATE のみ例外的に許可（取消操作）
- `stamped_at` が事実上の作成日時（`created_at` 相当）。イミュータブルテーブルのため `updated_at` も不要
- 取消された押印も物理削除しない（取消の事実自体が監査証跡）
- **再押印**: 取消後に同一ドキュメントへの再押印は可能（重複チェックは `revoked_at IS NULL` の有効押印のみが対象）
- **データ増大**: 押印は回覧板・議決権・カルテの確認操作でのみ発生し高頻度ではないため、当面パーティション/アーカイブは不要。将来のアーカイブ戦略:
  - **パーティショニング基準**: `stamped_at` の年月単位 RANGE パーティション（100万件超で導入検討）
  - **アーカイブ対象条件**: `stamped_at` が2年以上前 AND `revoked_at IS NOT NULL`（取消済みの古い押印を優先アーカイブ）
  - **有効押印の保持**: `revoked_at IS NULL` の有効押印は年数に関わらずアーカイブ対象外（検証の継続性を維持）

### ER図（テキスト形式）
```
users (1) ──── (3) electronic_seals
users (1) ──── (N) seal_scope_defaults
users (1) ──── (N) seal_stamp_logs

electronic_seals (1) ──── (N) seal_stamp_logs
electronic_seals (1) ──── (N) seal_scope_defaults

seal_stamp_logs.target_type + target_id ──→ circulation_documents / proxy_votes / chart_records 等（ポリモーフィック参照）
```

#### `POST /api/v1/seal/stamps/{stampId}/revoke`

押印を取消する。レコードは削除せず `revoked_at` に日時をセットする（監査証跡の不変性を維持）。取消可能かどうかは連携先機能のステータスに依存する。

押印者本人による通常取消に加え、ADMIN による強制取消にも対応する（退職済みメンバーの押印が業務フローをブロックするケース等）。

**リクエストボディ**
```json
{
  "reason": "誤って押印したため"
}
```

- `reason`: 必須・1〜200文字

**レスポンス（200 OK）**
```json
{
  "data": {
    "stamp_id": 42,
    "revoked_at": "2026-03-12T15:00:00+09:00",
    "revoked_reason": "誤って押印したため",
    "revoked_by": "SELF"
  }
}
```

- `revoked_by`: `SELF`（本人取消）または `ADMIN`（管理者強制取消）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `reason` が未入力または200文字超 |
| 403 | 押印者本人でも対象ドキュメントの ADMIN でもない |
| 404 | 指定の押印記録が存在しない |
| 409 | 既に取消済み |
| 422 | 連携先機能のステータスにより取消不可（例: 回覧板で次の人が押印済み、議決権で投票締切済み） |

#### `GET /api/v1/seal/stamps/{stampId}/verify`

押印記録の真正性を検証する。`is_valid` はハッシュ整合性（押印時点の印鑑が改ざんされていないか）のみを判定し、`is_revoked` は運用上の取消状態を独立して表現する。

**レスポンス（200 OK）**
```json
{
  "data": {
    "stamp_id": 42,
    "user_id": 1,
    "seal_variant": "FULL_NAME",
    "target_type": "CIRCULATION",
    "target_id": 15,
    "stamped_at": "2026-03-10T09:15:00+09:00",
    "seal_hash_at_stamp": "a1b2c3d4...",
    "current_seal_hash": "a1b2c3d4...",
    "is_valid": true,
    "is_revoked": false,
    "invalid_reason": null
  }
}
```

**フィールド定義**:
- `is_valid` — **ハッシュ整合性のみ**を判定。押印時点のハッシュと現在の印鑑ハッシュが一致すれば `true`。取消状態には影響されない
- `is_revoked` — 押印が運用上取消されているかどうか。`is_valid` とは独立した軸

**組み合わせパターン**:
| `is_valid` | `is_revoked` | 意味 |
|-----------|-------------|------|
| `true` | `false` | 有効な押印（正常） |
| `true` | `true` | 取消済みだが印鑑の真正性は確認済み |
| `false` | `false` | 印鑑が再生成/削除されている（要調査） |
| `false` | `true` | 取消済み かつ 印鑑の整合性も崩れている |

`is_valid` が `false` の場合、`invalid_reason` に理由を返す（複数該当時は上から優先）:
1. `SEAL_NOT_FOUND` — 印鑑が削除されている（ユーザー退会等）
2. `SEAL_REGENERATED` — 押印後に印鑑が再生成されている

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | 押印者本人でも対象ドキュメントの ADMIN でもない |
| 404 | 指定の押印記録が存在しない |

## 5. ビジネスロジック

### 印鑑 SVG 生成フロー

```
1. ユーザー登録完了 → UserCreatedEvent 発行
2. SealService が購読 → 姓（last_name）・名（first_name）を取得
3. 姓名のバリデーション（空文字・スペースのみの場合はデフォルト印鑑「印」を生成）
4. 3バリアントそれぞれについて:
   a. display_text を決定:
      - LAST_NAME: last_name（例: 田中）
      - FULL_NAME: last_name + first_name（例: 田中太郎）
      - FIRST_NAME: first_name（例: 太郎）
   b. 文字数に応じてレイアウトを決定:
      - 1文字: 中央配置（大）
      - 2文字: 縦2文字配置
      - 3文字: 縦3文字配置（やや縮小）
      - 4文字以上: 上2文字+下2文字の4文字グリッド（5文字以上は先頭4文字）
   c. 印鑑風 SVG を生成（丸枠 + 篆書風フォント配置）
   d. SHA-256 ハッシュを算出
   e. generation_version に現在のアルゴリズムバージョンをセット
5. electronic_seals テーブルへ 3件 INSERT
6. seal_scope_defaults へ DEFAULT 行を INSERT（LAST_NAME バリアントの seal_id を初期値に設定）
```

### 姓名変更時の自動再生成

```
1. ユーザーが姓または名を変更 → UserProfileUpdatedEvent 発行
2. SealService が購読 → electronic_seals の display_text と現在の姓名を比較
3. 不一致のバリアントがある場合:
   a. 該当バリアントの SVG を再生成
   b. electronic_seals を UPDATE（display_text, svg_data, seal_hash）
   c. ※ 過去の seal_stamp_logs.seal_hash は変更しない（押印時点のスナップショットを保持）
```

### ユーザー退会・復元時の処理

```
■ 退会時（UserDeletedEvent）
1. electronic_seals 全3件の deleted_at をセット（論理削除）
2. seal_scope_defaults はそのまま保持（復元時にデフォルト設定を復元するため）
3. seal_stamp_logs はそのまま保持（監査証跡）
4. 印鑑取得 API は 404、verify API は SEAL_NOT_FOUND で応答

■ 復元時（UserRestoredEvent）
1. electronic_seals 全3件の deleted_at をクリア（復元）
2. 姓名変更チェック → 不一致のバリアントがあれば SVG を再生成
3. 復元後は通常通り印鑑取得・押印が可能
```

### 押印履歴の target_title 解決

押印履歴 API のレスポンスに `target_title`（対象ドキュメントの件名）を含める。ポリモーフィック参照のため、`target_type` に応じて連携先のリポジトリから件名を取得する:

```
target_type → 解決先テーブル → 取得カラム
CIRCULATION → circulation_documents → title
VOTE → proxy_votes → title
MEDICAL_CONSENT → chart_records → title
```

- IN 句でバッチ取得し N+1 を回避
- 対象ドキュメントが削除済みの場合は `target_title = null` を返す（履歴から辿れないことをフロントエンドで表示）

### 印鑑生成失敗のリカバリ

```
1. UserCreatedEvent での自動生成が失敗（フォントロードエラー等）
2. Spring Retry で最大3回リトライ
3. 3回失敗 → Dead Letter として記録（DB に seal 未作成のまま）
4. ユーザーのダッシュボードに「印鑑未生成」警告を表示
5. ユーザーが手動で POST /seals/regenerate を実行、または
   SYSTEM_ADMIN が一覧から確認して対処
```

### スコープ別デフォルト解決フロー

```
押印時にどの印鑑を使用するか:
1. 連携先機能からスコープ情報を取得（team_id or organization_id）
2. seal_scope_defaults を検索:
   a. scope_type = 'TEAM' AND scope_id = {team_id} → あればその seal_id を使用
   b. scope_type = 'ORGANIZATION' AND scope_id = {org_id} → あればその seal_id を使用
   c. scope_type = 'DEFAULT' → グローバルデフォルトの seal_id を使用
3. 解決した seal_id の electronic_seals が論理削除されている場合:
   → LAST_NAME バリアントにフォールバック（それも削除済みなら押印不可エラー）
```

### 連携先 API の押印エンドポイント規約

押印は電子印鑑の独立 API ではなく、**連携先機能の API 内で `SealStampService.stamp()` を内部呼び出し**する方式を採用する。これにより、押印と業務フロー（ステータス更新・通知等）が1リクエスト・1トランザクションで完結する。

連携先が押印エンドポイントを提供する際は、以下の命名規約に従う:

**パス規約**: `POST /{resource}/{id}/stamp`

| 連携先機能 | エンドポイント | `target_type` | 内部処理 |
|-----------|--------------|---------------|---------|
| 回覧板 | `POST /circulation/{id}/stamp` | `CIRCULATION` | 押印 + 次の受取人へ通知（順番回覧の場合） + 全員押印完了時にステータス更新 |
| 議決権行使 | `POST /proxy-votes/{id}/cast` | `VOTE` | 投票（賛成/反対/棄権）+ 押印 + 集計更新 ※命名例外（下記注参照） |
| カルテ（問診票・同意書） | `POST /charts/{id}/stamp` | `MEDICAL_CONSENT` | 押印 + 同意済みステータス更新 |

**リクエスト規約**: 各エンドポイントは任意パラメータ `seal_id` を受け付ける。指定時はスコープ別デフォルト解決をスキップし、指定した印鑑で押印する（自分の印鑑以外は 400 エラー）

**レスポンス規約**: 各エンドポイントのレスポンスに `stamp` オブジェクトを含める

```json
{
  "data": {
    "...連携先固有のフィールド...",
    "stamp": {
      "stamp_id": 42,
      "seal_variant": "FULL_NAME",
      "stamped_at": "2026-03-12T14:00:00+09:00"
    }
  }
}
```

**サービス層の呼び出し規約**: 各連携先の Service クラスで以下のパターンを統一する

```java
// 連携先サービス内（例: CirculationService）
@Transactional
public CirculationStampResult stamp(Long documentId, Long userId, Long sealIdOverride, HttpServletRequest request) {
    // 1. 連携先固有の前提条件チェック（ステータス・順番等）
    CirculationDocument doc = circulationRepository.findByIdOrThrow(documentId);
    doc.validateStampable(userId);

    // 2. SealStampService に委譲（共通ロジック）
    //    sealIdOverride が指定されていればスコープ別デフォルト解決をスキップ
    SealStampResult stampResult = sealStampService.stamp(
        userId, SealTargetType.CIRCULATION, documentId, doc.getTeamId(), sealIdOverride, request
    );

    // 3. 連携先固有の後続処理（ステータス更新・通知等）
    doc.markAsStamped(userId);
    applicationEventPublisher.publishEvent(new CirculationStampedEvent(doc, userId));

    return new CirculationStampResult(doc, stampResult);
}
```

> **命名例外: 議決権行使** — 議決権の `POST /proxy-votes/{id}/cast` は「意思表示（投票）」が主目的であり、押印は証跡としての付随操作。`/stamp` にすると投票の意味が消えるため `/cast` を維持する。内部実装で `SealStampService.stamp()` を呼び出し、レスポンスに `stamp` オブジェクトを含める点は他の連携先と同一。

### 押印フロー（連携先機能から呼び出し）

```
1. 連携先 Controller が POST /{resource}/{id}/stamp を受け付け
2. 連携先 Service が固有の前提条件チェック（押印可能なステータスか等）
3. SealStampService.stamp(userId, targetType, targetId, scopeId, sealIdOverride, request) を呼び出し:
   a. sealIdOverride が指定されていればそのまま使用（本人の印鑑か検証）、未指定ならスコープ別デフォルト解決 → 使用する seal_id を決定
   b. 決定した electronic_seals 行を `SELECT ... FOR UPDATE` で取得（排他ロック。同時リクエストによる二重押印を防止）
   c. 重複チェック（ロック取得後に seal_stamp_logs を検索）:
      - CIRCULATION: 同一 circulation_document への重複押印は不可（revoked_at IS NULL の有効押印が対象）
      - VOTE: 同一 proxy_vote への重複押印は不可（同上）
      - MEDICAL_CONSENT: 同一 chart_record への重複押印は不可（同上）
   d. seal_stamp_logs へ INSERT（seal_id + seal_hash スナップショット + IP + User-Agent）
   e. stamp_id を返却
4. 連携先 Service が後続処理（ステータス更新・通知・監査ログ）
5. 1トランザクションとしてコミット
```

### 押印取消フロー

```
1. ユーザーが revoke API を呼び出し（stampId + reason）
2. seal_stamp_logs から対象レコードを取得
3. 認可チェック: 押印者本人 または 対象ドキュメントの ADMIN であることを確認
   - ADMIN の場合: `revoked_reason` に「【管理者取消】」プレフィックスを自動付与
4. 取消済みチェック: revoked_at が既にセットされていれば 409 Conflict
5. 取消可否チェック: 連携先機能に問い合わせ（SealRevocationPolicy インターフェース）:
   - CIRCULATION: 次の受取人が未押印であること（順番回覧の場合）
   - VOTE: 投票締切前であること
   - MEDICAL_CONSENT: 施術が未開始であること
   - ※ 未登録の target_type はデフォルトで取消拒否（安全側に倒す）
6. seal_stamp_logs の revoked_at, revoked_reason を UPDATE
7. 連携先機能へ取消を通知（ApplicationEvent: SealStampRevokedEvent）
   - ADMIN 強制取消の場合、押印者本人にも通知を送信
8. 監査ログ記録（SEAL_STAMP_REVOKED / SEAL_STAMP_FORCE_REVOKED）
```

### 重要な判定ロジック
- **押印権限**: 押印操作は本人のみ（他者の印鑑を代理押印することは不可）
- **押印時のバリアント選択**: スコープ別デフォルト（TEAM → ORGANIZATION → DEFAULT）で自動選択。リクエストに `seal_id` が指定されていればデフォルト解決をスキップし指定印鑑を使用（本人の印鑑であることを検証）
- **取消権限**: 押印者本人 または 対象ドキュメントの ADMIN。ADMIN 強制取消は `revoked_by = 'ADMIN'` で記録し、通常取消と監査ログ上で区別する。押印者本人にも強制取消の通知を送信
- **再押印**: 取消後に同一ドキュメントへの再押印は可能。重複チェックは `revoked_at IS NULL` の有効押印のみが対象
- **再生成制限**: 手動再生成は1時間に3回まで（Bucket4j でレートリミット）。頻繁な再生成による過去押印の無効化を抑止
- **再生成影響通知**: regenerate レスポンスに `affected_stamp_count` を含め、フロントエンドで影響範囲を警告表示
- **退会時**: ユーザー退会（論理削除）時に electronic_seals 全3件を論理削除。seal_stamp_logs も保持（監査証跡）。印鑑取得 API は 404 を返すが、verify API は `SEAL_NOT_FOUND` で応答

### 通知トリガー一覧

押印に関連する通知は連携先機能の Event 経由で発行されるが、印鑑基盤側からも以下の通知を発行する:

| トリガー | 通知先 | 条件 |
|---------|-------|------|
| 印鑑自動再生成完了 | 本人 | 姓名変更による自動再生成時 |
| 印鑑生成失敗 | 本人 | Spring Retry 3回失敗後（ダッシュボード警告 + プッシュ通知） |
| ADMIN 強制取消 | 押印者本人 | ADMIN が他者の押印を強制取消した時 |
| 一括再生成バッチ完了 | SYSTEM_ADMIN | バッチ完了時（成功/失敗件数を含む） |

## 6. セキュリティ考慮事項

- **認可チェック**: 印鑑参照・再生成は本人 or ADMIN（同一チーム/組織）or SYSTEM_ADMIN のみ
- **レートリミット**: 再生成 API に `Bucket4j` で 1時間3回の制限を適用
- **SVG サニタイズ**: サーバーサイドで生成するため XSS リスクは低いが、SVG 内に `<script>` タグが混入しないことをバリデーション
- **証跡の不変性**: seal_stamp_logs は原則 INSERT のみ。DELETE は API・サービス層ともに提供しない。`revoked_at` / `revoked_by` / `revoked_reason` の UPDATE のみ取消操作として例外的に許可
- **ADMIN 強制取消の監査**: `revoked_by = 'ADMIN'` で通常取消と区別。監査ログは `SEAL_STAMP_FORCE_REVOKED` として記録し、押印者本人にも通知を送信
- **IP アドレス記録**: 押印時の IP を記録し、不正押印の調査に利用可能
- **一括再生成のアクセス制御**: `POST /admin/seals/regenerate-all` は SYSTEM_ADMIN のみ。同時実行防止（409）あり
- **一括検証のレート制限**: `GET /seal/stamps/verify-batch` に `Bucket4j` で 1分間10回の制限を適用。stamps レスポンスは最大100件に制限
