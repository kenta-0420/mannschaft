# F05.5: ファイル共有

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 5
> **最終更新**: 2026-03-14
> **関連ドキュメント**: F01.3 テンプレート・モジュール（デフォルト機能 #8）、.claudecode.md §26 Pre-signed URL フロー

---

## 1. 概要

チーム・組織内のドキュメント・資料を共有するストレージ機能。階層フォルダ管理、アクセス権限制御（チーム内/グループ内/個人指定）、バージョン管理を備え、Pre-signed URL による S3 直接アップロード/ダウンロードで高速なファイル転送を実現する。チームごとのストレージ使用量をリアルタイムに追跡し、無料枠を超過した場合はストレージプランへのアップグレードを促す。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全チーム/組織のファイル参照。ストレージプラン管理 |
| ADMIN | フォルダ・ファイルの CRUD・権限設定・ストレージプラン変更 |
| DEPUTY_ADMIN | `MANAGE_FILES` 権限を持つ場合: フォルダ・ファイルの CRUD・権限設定 |
| MEMBER | 権限が付与されたフォルダ・ファイルの閲覧・アップロード・ダウンロード |
| SUPPORTER | 権限が付与されたフォルダ・ファイルの閲覧・ダウンロードのみ（アップロード不可） |
| GUEST | 対象外 |

### 対象レベル
- [x] 組織 (Organization) — 組織共有フォルダ
- [x] チーム (Team) — チーム共有フォルダ
- [x] 個人 (Personal) — 個人用ストレージ（チームのストレージ枠を消費）

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `shared_folders` | フォルダ階層管理 | あり |
| `shared_files` | ファイルメタデータ | あり |
| `shared_file_versions` | ファイルのバージョン履歴 | なし |
| `file_permissions` | フォルダ・ファイルのアクセス権限 | なし |

### テーブル定義

#### `shared_folders`

フォルダの階層構造。隣接リスト方式（`parent_id`）で管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（TEAM / ORGANIZATION / PERSONAL） |
| `team_id` | BIGINT UNSIGNED | YES | NULL | FK → teams。TEAM / PERSONAL の場合に設定 |
| `organization_id` | BIGINT UNSIGNED | YES | NULL | FK → organizations。ORGANIZATION の場合に設定 |
| `user_id` | BIGINT UNSIGNED | YES | NULL | FK → users。PERSONAL の場合に設定 |
| `parent_id` | BIGINT UNSIGNED | YES | NULL | FK → shared_folders（自己参照）。NULL = ルートフォルダ |
| `name` | VARCHAR(255) | NO | — | フォルダ名 |
| `description` | VARCHAR(500) | YES | NULL | フォルダの説明 |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `version` | BIGINT | NO | 0 | 楽観的ロックバージョン（@Version） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_shared_folders_team (team_id, parent_id, name)               -- チーム別のフォルダ一覧
INDEX idx_shared_folders_org (organization_id, parent_id, name)        -- 組織別のフォルダ一覧
INDEX idx_shared_folders_user (user_id, parent_id, name)               -- 個人別のフォルダ一覧
INDEX idx_shared_folders_parent (parent_id)                             -- 子フォルダ取得
```

**制約・備考**
- `scope_type` のスコープ制約:
  - `TEAM`: `team_id` NOT NULL, `organization_id` NULL, `user_id` NULL
  - `ORGANIZATION`: `team_id` NULL, `organization_id` NOT NULL, `user_id` NULL
  - `PERSONAL`: `team_id` NOT NULL, `organization_id` NULL, `user_id` NOT NULL（個人用だがチームのストレージ枠を消費）
- フォルダ名は同一親フォルダ内で一意（Service 層バリデーション。大文字/小文字を区別しない）
- フォルダ階層の最大深度: 10階層（Service 層で検証。深すぎる階層はナビゲーション UX を損なうため）
- 論理削除時: 配下のファイル・サブフォルダはそのまま保持（フォルダ復元時に復活）。ストレージ使用量は論理削除時に減算しない（完全削除バッチで減算）
- ルートフォルダ（`parent_id = NULL`）はスコープごとに自動作成。ユーザーが明示的に作成するのはサブフォルダ以降

#### `shared_files`

ファイルのメタデータ。実ファイルは S3 に保存し、`file_key` で参照。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `folder_id` | BIGINT UNSIGNED | NO | — | FK → shared_folders。ON DELETE CASCADE |
| `name` | VARCHAR(255) | NO | — | ファイル名（拡張子含む） |
| `file_key` | VARCHAR(500) | NO | — | S3 オブジェクトキー（最新バージョン） |
| `file_size` | BIGINT UNSIGNED | NO | — | ファイルサイズ（バイト。最新バージョン） |
| `content_type` | VARCHAR(100) | NO | — | MIME タイプ |
| `description` | VARCHAR(500) | YES | NULL | ファイルの説明 |
| `current_version` | INT UNSIGNED | NO | 1 | 現在のバージョン番号 |
| `uploaded_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `download_count` | INT UNSIGNED | NO | 0 | ダウンロード数（denormalize） |
| `version` | BIGINT | NO | 0 | 楽観的ロックバージョン（@Version） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_shared_files_folder (folder_id, name)                -- フォルダ内のファイル一覧
INDEX idx_shared_files_uploader (uploaded_by, created_at DESC)  -- アップロード者別
FULLTEXT INDEX ft_shared_files_name (name)                      -- ファイル名検索
```

**制約・備考**
- `file_key`: S3 オブジェクトキー。フォーマット: `files/{scopeType}/{scopeId}/{uuid}.{ext}`（§26 準拠）
- `file_size`: 最新バージョンのサイズ。バージョン更新時に上書き。ストレージ使用量の計算に使用
- ファイルサイズ上限: **1ファイル 100MB**。それ以上の大容量ファイルは FUTURE_CONSIDERATIONS.md のチャンクアップロードで対応
- ファイル名は同一フォルダ内で一意（Service 層バリデーション）。重複時は自動リネーム（`report.pdf` → `report (1).pdf`）
- `download_count`: ダウンロード URL 発行時にアトミック更新

#### `shared_file_versions`

ファイルのバージョン履歴。同一ファイルの過去バージョンを保持し、ロールバックを可能にする。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `file_id` | BIGINT UNSIGNED | NO | — | FK → shared_files。ON DELETE CASCADE |
| `version_number` | INT UNSIGNED | NO | — | バージョン番号（1始まり） |
| `file_key` | VARCHAR(500) | NO | — | S3 オブジェクトキー（このバージョンの実体） |
| `file_size` | BIGINT UNSIGNED | NO | — | ファイルサイズ（バイト） |
| `content_type` | VARCHAR(100) | NO | — | MIME タイプ |
| `uploaded_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `comment` | VARCHAR(500) | YES | NULL | バージョンコメント（例: 「第2稿修正」） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_file_versions (file_id, version_number)           -- 同一ファイルのバージョン番号一意
INDEX idx_file_versions_file (file_id, version_number DESC)     -- バージョン一覧（新しい順）
```

**制約・備考**
- バージョン保持上限: 1ファイルあたり最大20バージョン。超過時は最古バージョンを自動削除（S3 オブジェクトも削除。ストレージ使用量を減算）
- 新バージョンアップロード時: `shared_files.file_key` / `file_size` / `content_type` / `current_version` を更新し、旧バージョンの情報を `shared_file_versions` に INSERT
- 全バージョンのファイルサイズ合計がストレージ使用量に加算される

#### `file_permissions`

フォルダ・ファイル単位のアクセス権限。ポリモーフィック方式で権限付与先（ユーザー/ロール/グループ）を指定。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `target_type` | VARCHAR(20) | NO | — | 権限対象種別（FOLDER / FILE） |
| `target_id` | BIGINT UNSIGNED | NO | — | 対象の shared_folders.id または shared_files.id |
| `grantee_type` | VARCHAR(20) | NO | — | 権限付与先種別（USER / ROLE / GROUP） |
| `grantee_id` | BIGINT UNSIGNED | NO | — | 付与先の users.id / roles.id / group_hierarchy.id |
| `permission` | VARCHAR(20) | NO | — | 権限レベル（VIEW / DOWNLOAD / UPLOAD / MANAGE） |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_file_permissions (target_type, target_id, grantee_type, grantee_id)  -- 同一権限の重複防止
INDEX idx_file_permissions_target (target_type, target_id)                           -- 対象別の権限一覧
INDEX idx_file_permissions_grantee (grantee_type, grantee_id)                       -- 付与先別の権限一覧
```

**制約・備考**
- `permission` の階層（上位は下位を包含）:
  - `VIEW`: ファイル名・メタデータの閲覧のみ
  - `DOWNLOAD`: VIEW + ファイルのダウンロード
  - `UPLOAD`: DOWNLOAD + ファイルのアップロード・バージョン更新
  - `MANAGE`: UPLOAD + ファイル/フォルダの削除・移動・権限変更
- フォルダの権限は配下のファイル・サブフォルダに継承される（Service 層で再帰的に評価）。ファイル個別に権限を設定した場合はフォルダ権限より優先
- `grantee_type = 'ROLE'` の場合: そのロールを持つ全ユーザーに適用（例: `MEMBER` ロール → チーム全メンバー）
- ADMIN / DEPUTY_ADMIN（MANAGE_FILES）は暗黙的に全フォルダ・ファイルへの MANAGE 権限を持つ
- 個人フォルダ（`scope_type = 'PERSONAL'`）は所有者に暗黙的 MANAGE 権限。他ユーザーへの共有は `file_permissions` で明示設定

### ER図（テキスト形式）
```
teams (1) ──── (N) shared_folders（team_id; TEAM / PERSONAL）
organizations (1) ──── (N) shared_folders（organization_id; ORGANIZATION）
shared_folders (1) ──── (N) shared_folders（parent_id; 階層構造）
shared_folders (1) ──── (N) shared_files
shared_files (1) ──── (N) shared_file_versions
file_permissions (N) ──── shared_folders / shared_files（ポリモーフィック）

users (1) ──── (N) shared_folders（user_id; PERSONAL）
users (1) ──── (N) shared_files（uploaded_by）
users (1) ──── (N) shared_file_versions（uploaded_by）
```

| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `sort` | String | — | ソート（`name_asc` / `name_desc` / `updated_at_desc` / `size_desc`。デフォルト: `name_asc`） |

**レスポンス（200 OK）**
```json
{
  "data": {
    "folder": {
      "id": 1,
      "name": "プロジェクト資料",
      "description": "新メニュー開発用",
      "parent_id": null,
      "scope_type": "TEAM",
      "breadcrumb": [
        { "id": null, "name": "ルート" },
        { "id": 1, "name": "プロジェクト資料" }
      ],
      "permissions": { "can_upload": true, "can_manage": true },
      "version": 1,
      "created_by": { "id": 10, "display_name": "田中太郎" },
      "created_at": "2026-03-01T09:00:00+09:00"
    },
    "subfolders": [
      { "id": 2, "name": "議事録", "file_count": 5, "updated_at": "2026-03-14T10:00:00+09:00" }

        "id": 101,
        "name": "企画書_v2.pdf",

```

- `breadcrumb`: ルートからの階層パス（フロントエンドのパンくずリスト用）

**エラーレスポンス**
| ステータス | 条件 |

| 403 | フォルダへのアクセス権限がない |

#### `POST /api/v1/files/upload-url`

ファイルアップロード用の Pre-signed URL を発行する（§26 準拠）。

**リクエストボディ**
```json
{
  "folder_id": 1,
  "file_name": "企画書.pdf",
  "content_type": "application/pdf",
  "file_size": 2048000
}
```

- `folder_id`: 必須。アップロード先フォルダ
- `file_size`: 最大 100MB（104,857,600 バイト）
- ストレージ枠チェック: チームの `storage_used_bytes + file_size` が上限を超える場合は 409 エラー

**レスポンス（200 OK）**
```json
{
  "data": {
    "upload_url": "https://s3...",
    "file_key": "files/team/5/abc123.pdf",
    "expires_in": 600
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | content_type が許可されていない / file_size が上限超過 |
| 403 | フォルダへの UPLOAD 権限がない / SUPPORTER はアップロード不可 |
| 404 | フォルダが存在しない |
| 409 | ストレージ枠不足（プランアップグレードを促すメッセージを含む） |

---

#### `POST /api/v1/files`

S3 アップロード完了後にファイルを DB に登録する。

**リクエストボディ**
```json
{
  "folder_id": 1,
  "name": "企画書.pdf",
  "file_key": "files/team/5/abc123.pdf",
  "file_size": 2048000,
  "content_type": "application/pdf",
  "description": "新メニュー企画書 第1稿"
}
```

- `file_key`: S3 上に実在することを `HeadObject` で検証
- 同一フォルダ内にファイル名が重複する場合は自動リネーム

**レスポンス（201 Created）**: ファイル詳細を返却

**副作用**:
1. チームの `storage_used_bytes` をアトミックに加算
2. `shared_file_versions` にバージョン1を INSERT

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | バリデーションエラー / file_key が S3 に存在しない |
| 403 | フォルダへの UPLOAD 権限がない |
| 404 | フォルダが存在しない |
| 409 | ストレージ枠不足 |

#### `POST /api/v1/files/{id}/versions`

既存ファイルの新バージョンをアップロードする。

**リクエストボディ**
```json
{
  "file_key": "files/team/5/def456.pdf",
  "file_size": 2560000,
  "content_type": "application/pdf",
  "comment": "第2稿: 価格表を更新",
  "version": 1
}
```

- `version`: 必須。楽観的ロック（同時に別ユーザーがバージョン更新するのを防止）
- バージョン上限（20件）超過時: 最古バージョンを自動削除 + S3 オブジェクト削除 + ストレージ使用量減算

**レスポンス（201 Created）**: 更新後のファイル詳細を返却

**副作用**:
1. `shared_files` の `file_key` / `file_size` / `content_type` / `current_version` を更新
2. 旧バージョン情報を `shared_file_versions` に INSERT
3. ストレージ使用量: 新バージョン分を加算（旧バージョンも保持するため減算なし。上限超過で自動削除時に減算）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | バリデーションエラー / file_key が S3 に存在しない |
| 403 | ファイルへの UPLOAD 権限がない |
| 404 | ファイルが存在しない |
| 409 | 楽観的ロック競合 / ストレージ枠不足 |

#### `PUT /api/v1/files/{id}/permissions`

ファイルまたはフォルダのアクセス権限を設定する。既存の権限を全件置換する方式。

**リクエストボディ**
```json
{
  "permissions": [
    { "grantee_type": "ROLE", "grantee_id": 4, "permission": "DOWNLOAD" },
    { "grantee_type": "USER", "grantee_id": 11, "permission": "UPLOAD" },
    { "grantee_type": "GROUP", "grantee_id": 2, "permission": "VIEW" }
  ]
}
```

- 空配列を送信すると全権限を削除（親フォルダの権限のみが適用される）

**レスポンス（200 OK）**: 設定後の権限一覧を返却

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | MANAGE 権限がない |
| 404 | ファイル/フォルダが存在しない |

## 5. ビジネスロジック

### ファイルプレビュー

```
1. content_type に基づくインラインプレビュー（フロントエンド実装）:
   a. application/pdf: <iframe src="{pre-signed download URL}"> で表示
   b. image/*: <img src="{pre-signed download URL}" loading="lazy"> で表示
      CSS: max-width: 100%; max-height: 600px; object-fit: contain
   c. その他: ダウンロードボタンのみ表示（プレビュー不可）

2. プレビュー用 URL:
   a. GET /files/{id}/download-url と同一の Pre-signed URL を使用
   b. Content-Disposition: inline（プレビュー用）vs attachment（ダウンロード用）の区別は
      クエリパラメータ ?disposition=inline で制御
```

### ゴミ箱

```
1. ゴミ箱一覧:
   a. GET /files?deleted=true / GET /files/folders?deleted=true で取得
   b. レスポンスに remaining_days（完全削除までの残日数 = 30 - (NOW() - deleted_at).days）を付与
   c. スコープ別にフィルタ可能（team_id / organization_id）

2. 復元:
   a. POST /files/{id}/restore: deleted_at = NULL に更新
   b. POST /files/folders/{id}/restore: 配下を再帰的に復元
   c. 元のフォルダが削除済みの場合: ルートフォルダ直下に復元（フロントエンドで通知）
   d. ストレージ使用量: 論理削除時に減算していないため復元時の加算も不要

3. 即時完全削除:
   a. DELETE /files/{id}?permanent=true（ADMIN のみ）
   b. S3 オブジェクト削除 + DB 物理削除 + storage_used_bytes 減算（全バージョン含む）
   c. 監査ログ: FILE_PERMANENTLY_DELETED を記録

4. 自動完全削除:
   a. 既存の完全削除バッチ（日次 04:00 JST）で処理
   b. deleted_at < NOW() - INTERVAL 30 DAY のレコードを対象
```

### ファイルアップロードフロー

```
1. フロントエンド: POST /files/upload-url でPre-signed URL を取得
   a. バックエンド: content_type ホワイトリスト検証
   b. バックエンド: 拡張子と content_type の整合性チェック（偽装検知。例: .pdf なのに application/x-executable は拒否）
   c. バックエンド: file_size 上限チェック（100MB）
   c. バックエンド: ストレージ枠チェック（used_bytes + file_size <= plan.included_bytes or 従量課金）
   d. バックエンド: file_key をUUIDベースで生成（files/{scopeType}/{scopeId}/{uuid}.{ext}）

2. フロントエンド: S3 に直接アップロード（PUT {uploadUrl}）

3. フロントエンド: POST /files でDB登録
   a. バックエンド: file_key の S3 存在確認（HeadObject）
   b. バックエンド: shared_files INSERT + shared_file_versions INSERT（v1）
   c. バックエンド: storage_used_bytes をアトミック加算
   d. ApplicationEvent: FileUploadedEvent（通知連携用）
```

### ストレージ使用量管理

```
1. 使用量の追跡:
   a. チームごとの storage_used_bytes を teams テーブルの denormalize カラムで管理
   b. 加算タイミング: ファイル登録（POST /files）、新バージョンアップロード（POST /versions）
   c. 減算タイミング: ファイル完全削除（バッチ）、バージョン自動削除
   ※ 論理削除時は減算しない（復元の可能性があるため）

2. 完全削除バッチ:
   a. @Scheduled で日次実行（深夜 04:00 JST）
   b. 条件: deleted_at < NOW() - INTERVAL 30 DAY（論理削除から30日経過）
   c. S3 オブジェクト削除 + DB 物理削除 + storage_used_bytes 減算
   d. 全バージョンの S3 オブジェクトも削除

3. ストレージ枠超過時の挙動:
   a. アップロード API で 409 エラー（「ストレージ容量が不足しています」）
   b. レスポンスに現在の使用量・上限・推奨プランを含める
   c. 使用量が80%を超えた時点で管理者にプッシュ通知
     （ApplicationEvent: StorageWarningEvent。チームの storage_used_bytes を定期チェック）
```

### 権限評価フロー

```
1. リクエスト時の権限チェック:
   a. ADMIN / DEPUTY_ADMIN（MANAGE_FILES）→ 暗黙的 MANAGE 権限（チェック終了）
   b. 対象ファイル/フォルダの file_permissions を検索
   c. ファイル個別の権限があればそれを適用
   d. なければ親フォルダ → 祖父フォルダ → ... → ルートフォルダの順で再帰的に検索
   e. grantee_type の評価順序:
      - USER: 完全一致
      - GROUP: ユーザーが所属するグループに一致
      - ROLE: ユーザーのロールに一致
   f. 複数マッチした場合は最も高い permission を採用

2. 権限の継承:
   a. フォルダに設定した権限は配下の全ファイル・サブフォルダに自動継承
   b. サブフォルダ/ファイル個別の設定で上書き可能
   c. 上書きは「追加」ではなく「置換」（親の権限を完全にオーバーライド）
```

### フォルダ操作

```
1. フォルダ移動（PUT /folders/{id}）:
   a. 移動先フォルダへの MANAGE 権限を検証
   b. 循環参照チェック: 移動先が自分自身の配下でないことを検証
   c. スコープ変更不可: TEAM → ORGANIZATION 等のスコープ間移動は禁止

2. フォルダ削除（DELETE /folders/{id}）:
   a. 配下の全ファイル・サブフォルダを再帰的に論理削除
   b. 30日後にバッチで完全削除
   c. ルートフォルダは削除不可

3. パンくずリスト:
   a. parent_id を再帰的に辿ってルートまでのパスを構築
   b. MySQL 8.0 の WITH RECURSIVE で1クエリで取得
```

### 重要な判定ロジック

- **ファイル管理権限**: ADMIN or DEPUTY_ADMIN（`MANAGE_FILES` 権限）は全フォルダ・ファイルへの暗黙的 MANAGE 権限
- **アップロード権限**: UPLOAD 以上の権限 + SUPPORTER 以外
- **ダウンロード権限**: DOWNLOAD 以上の権限
- **個人フォルダ**: 所有者に暗黙的 MANAGE 権限。他ユーザーへの共有は明示設定
- **ファイル名重複時**: 自動リネーム（`(N)` サフィックス追加）
- **フォルダ深度上限**: 10階層（Service 層バリデーション）
- **バージョン上限**: 20件/ファイル（最古バージョンを自動削除）
- **ストレージ使用量**: 論理削除時は減算しない、完全削除時に減算

## 6. セキュリティ考慮事項

- **認可チェック**: 全 API エンドポイントで `file_permissions` を評価。権限のないファイル/フォルダへのアクセスは 403
- **Pre-signed URL の安全性**: URL の有効期限（アップロード 10分、ダウンロード 5分）を厳守。`file_key` は UUID ベースで推測不可能
- **IDOR 防止**: `PUT /files/{id}` / `DELETE /files/{id}` は権限チェック必須。他チームのファイル ID を URL 直打ちしてもアクセスは 403
- **ファイル内容の検証**: `content_type` のホワイトリスト検証（`image/jpeg`, `image/png`, `image/webp`, `application/pdf`, `application/vnd.openxmlformats-officedocument.*`, `text/plain`, `text/csv`, `application/zip`）。実行可能ファイル（`.exe`, `.sh`, `.bat` 等）は拒否。拡張子と `content_type` の整合性チェックで偽装ファイルを検知
- **監査ログ**: 以下のイベントを監査ログに記録
  - `FILE_UPLOADED`: ファイルアップロード
  - `FILE_DOWNLOADED`: ファイルダウンロード（Pre-signed URL 発行時に記録）
  - `FILE_DELETED`: ファイル削除
  - `FILE_PERMISSION_CHANGED`: 権限変更
  - `FOLDER_DELETED`: フォルダ削除（配下のファイル数を記録）
  - `FILE_PERMANENTLY_DELETED`: ADMIN による即時完全削除
  - `FILE_RESTORED`: ゴミ箱からの復元
- **レートリミット**: `Bucket4j` で以下の制限を適用
  - アップロード URL 発行（`POST /upload-url`）: 1分間20件
  - ダウンロード URL 発行（`GET /download-url`）: 1分間60件
- **ストレージ使用量の整合性**: 日次バッチで `storage_used_bytes` と実際の `SUM(file_size)` を照合し、不整合があれば補正 + WARN ログ出力
