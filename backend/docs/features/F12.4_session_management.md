# F12.4: セッション管理画面

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 12
> **最終更新**: 2026-04-02
> **関連ドキュメント**: F01.1 認証（refresh_tokens テーブル、JWT/セッション設計）、F04.3 プッシュ通知、F12.3 GDPR / 個人情報管理

---

## 1. 概要

「見知らぬデバイスからのログイン」を自分で検知・無効化できるセキュリティ機能を提供する。ユーザーは自身のアクティブセッション一覧を閲覧し、不審なセッションを個別に無効化したり、現在のセッション以外を一括ログアウトすることができる。

**解決する課題**:
- アカウントが不正利用されていても、ユーザーが気付く手段がない
- 共有端末でログアウトし忘れた場合に、リモートからセッションを切断できない
- `getSessions()` で `rememberMe` と `isCurrent` がハードコードされており、正確なセッション情報が返却されていない

**既存実装との関係**: F01.1 で `refresh_tokens` テーブルと基本 API（GET/DELETE）は実装済み。本機能では User-Agent パース・現セッション判定・デバイス名カスタマイズ・一括ログアウト（現セッション除外）の拡充と、既存バグ（`rememberMe`/`isCurrent` ハードコード）の修正を行う。加えて、新規デバイスからのログイン時にプッシュ通知で即座にユーザーへ警告する仕組みを F04.3 基盤上に構築する。

## 3. DB設計

### テーブル一覧

新規テーブルは作成しない。既存の `refresh_tokens` テーブルに `device_name` カラムを追加する。

| テーブル名 | 役割 | 変更種別 |
|-----------|------|---------|
| `refresh_tokens` | リフレッシュトークン（= セッション） | ALTER: `device_name` カラム追加 |

### カラム追加

#### `refresh_tokens` への追加カラム

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `device_name` | VARCHAR(100) | YES | NULL | ユーザーが設定可能なデバイス表示名。未設定時はバックエンドが `user_agent` から自動生成（例: "Chrome on Windows 11"） |

**制約・備考**
- `device_name` は User-Agent パースによる自動命名と、ユーザーによる手動リネームの2つの設定経路を持つ
- 自動命名は Refresh Token 新規発行時（ログイン）に User-Agent から生成する。トークンローテーション（`/auth/refresh`）時は旧トークンの `device_name` を新レコードにコピーする。ユーザーが手動リネーム済みの場合も上書きしない
- 手動リネーム時のバリデーション: 1〜100文字、制御文字禁止
- 既存レコード（`device_name IS NULL`）は API レスポンス生成時に `user_agent` からその場でパースして返却する（バッチ更新は不要）

### ER図（テキスト形式）

```
users (1) ──── (N) refresh_tokens
                    │ user_id → users.id
                    │
                    ├── セッション一覧 = revokedAt IS NULL AND expiresAt > NOW()
                    └── 現セッション判定 = リクエスト中の Refresh Token ハッシュと照合
```

### リクエスト／レスポンス仕様

#### `GET /api/v1/auth/sessions`

```json
{

    {

      "deviceName": "Chrome on Windows 11",
      "deviceType": "DESKTOP",
      "ipAddress": "203.0.113.1",

      "rememberMe": true,
      "createdAt": "2026-03-20T09:00:00",

      "expiresAt": "2026-04-19T09:00:00",
      "isCurrent": true
    },
    {
      "id": 38,
      "deviceName": "Safari on iPhone",
      "deviceType": "MOBILE",
      "ipAddress": "198.51.100.5",
      "userAgent": "Mozilla/5.0 (iPhone; CPU iPhone OS 17_4 ...",
      "rememberMe": false,
      "createdAt": "2026-03-28T18:45:00",
      "lastUsedAt": "2026-04-01T12:10:00",
      "expiresAt": "2026-04-04T18:45:00",
      "isCurrent": false
    }
  ]
}
```

**`isCurrent` の判定方法**:

2つの経路で現セッションを特定する（Web / モバイル両対応）:

1. **Cookie 経路（Web）**: リクエストの Cookie（`refresh_token`）から Refresh Token を取得し、SHA-256 ハッシュを計算。一覧内の各セッションの `token_hash` と照合し、一致するレコードを `isCurrent: true` とする
2. **クエリパラメータ経路（モバイル / 共通）**: `?currentSessionId={id}` が指定されている場合、該当 ID のセッション（かつ `user_id` が一致するもの）を `isCurrent: true` とする。`currentSessionId` はログインレスポンス（`TokenResponse`）に `sessionId` フィールドとして追加し、クライアントがローカルに保持する
3. 両方とも取得できない場合は全セッションの `isCurrent` を `false` とする
4. Cookie とクエリパラメータの両方がある場合は Cookie を優先する（Cookie は偽装が困難なため）

**`deviceName` の自動生成ルール**:
- `device_name` カラムが NULL の場合、`user_agent` 文字列をパースして生成
- パース規則: `{ブラウザ名} on {OS名}` 形式
  - 例: `"Chrome on Windows 11"`, `"Safari on iOS 17"`, `"Firefox on macOS"`, `"Samsung Internet on Android 14"`
  - ブラウザ・OS が特定できない場合: `"Unknown Browser"` / `"Unknown Device"`
- パースには外部ライブラリ（[ua-parser](https://github.com/ua-parser/uap-java)）の使用を推奨

**`lastUsedAt` の null ハンドリング**:
- `lastUsedAt` はトークンローテーション時に更新される。新規発行直後のトークンは `null`
- `null` の場合はレスポンスで `null` を返却する。フロントエンドは `null` 時に「まだ使用されていません」と表示する
- ソート時は `null` を最古として扱う（`COALESCE(lastUsedAt, createdAt)` 相当）

**ソート順**: `isCurrent=true` を先頭、それ以降は `lastUsedAt` 降順（null は最後尾）

#### `DELETE /api/v1/auth/sessions/{id}`

個別セッションを無効化する。

**レスポンス / エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 204 | 無効化成功 |
| 404 | セッションが存在しない、または自分のセッションでない、または既に無効化済み |
| 409 | 現在のセッションを無効化しようとした場合（自分自身のログアウトには `POST /api/v1/auth/logout` を使用すること） |

**処理フロー**:
1. `refresh_tokens.id` で検索し、`user_id` がリクエストユーザーと一致するか検証
2. `revoked_at IS NOT NULL` または `expires_at < NOW()` の場合は `404` を返す（既に無効化・期限切れ）
3. 現セッション判定: Cookie の Refresh Token ハッシュ照合、または `currentSessionId` パラメータ照合で一致する場合は `409 Conflict` を返す
4. `revoked_at` を設定（`entity.revoke()`）
5. `LogoutEvent(userId, 1, LogoutType.SESSION)` を発行

---

#### `DELETE /api/v1/auth/sessions?keepCurrent=true`

現セッション以外の全セッションを一括ログアウトする。

**クエリパラメータ**
| パラメータ | 型 | 必須 | デフォルト | 説明 |
|-----------|---|------|-----------|------|
| `keepCurrent` | Boolean | いいえ | `false` | `false`: 全セッション無効化（既存の `logoutAllDevices` と同等の後方互換動作）。`true`: 現セッションを保持して他を全無効化 |

**レスポンス（204 No Content）**

**処理フロー**（`keepCurrent=true` の場合）:
1. 現セッションを特定する: Cookie の Refresh Token ハッシュ照合、または `currentSessionId` パラメータを使用。**いずれの方法でも現セッションを特定できない場合は `keepCurrent=true` を無視して全セッション無効化にフォールバックする**（現セッションを特定できないため保持不可能）
2. ユーザーの全アクティブセッション（`revoked_at IS NULL AND expires_at > NOW()`）を取得
3. 特定した現セッションを除外
4. 残りの全セッションに `revoked_at` を設定
5. Valkey に `user_invalidated_at:{userId}` を設定（TTL 900秒）— 他デバイスの Access Token を即時無効化
6. `LogoutEvent(userId, revokedCount, LogoutType.ALL_SESSIONS)` を発行

## 5. ビジネスロジック

### 5.1 セッション一覧取得フロー

```
1. ユーザーが「セッション管理」画面を開く
2. フロントエンドが GET /api/v1/auth/sessions を呼び出す
3. バックエンドが refresh_tokens から未失効・未期限切れのレコードを取得
4. リクエストの Refresh Token ハッシュと照合し isCurrent を判定
5. device_name が NULL のレコードは user_agent をパースして deviceName を生成
6. isCurrent=true を先頭にして lastUsedAt 降順でソートし返却
```

### 5.2 既存実装の修正点

| 修正箇所 | 現状 | 修正後 |
|---------|------|--------|
| `AuthService.getSessions()` | `rememberMe` を `false` にハードコード | `token.getRememberMe()` を使用 |
| `AuthService.getSessions()` | `isCurrent` を `false` にハードコード | Refresh Token ハッシュ照合で判定 |
| `AuthService.logoutAllDevices()` | 現セッションも含めて全無効化 | `keepCurrent` パラメータで挙動を分岐 |

### 5.3 User-Agent パース

User-Agent 文字列からデバイス情報を抽出するユーティリティクラスを `com.mannschaft.app.auth.util.UserAgentParser` に作成する。

**ライブラリ選定: [uap-java](https://github.com/ua-parser/uap-java) を採用**

| 選択肢 | 判定 | 理由 |
|--------|------|------|
| **uap-java** | ✅ 採用 | BrowserScope 由来のコミュニティ共有 `regexes.yaml` を使用。新ブラウザ・OS は YAML 更新のみで対応可能。依存は snakeyaml のみで軽量（JAR 約200KB）。GitHub/Netflix/Wikipedia で採用実績あり |
| 自前正規表現 | ❌ 不採用 | User-Agent は数千パターン以上あり自前で網羅は非現実的。新ブラウザ・OS リリースのたびにメンテナンスコストが発生する |

```kotlin
// build.gradle.kts
implementation("com.github.ua-parser:uap-java:1.6.1")
```

```java
public class UserAgentParser {

    private static final Parser UA_PARSER = new Parser();

    /**
     * パース結果キャッシュ。同一 User-Agent の繰り返しパースを防ぐ。
     * キー: User-Agent 文字列、値: パース結果。TTL は1時間、最大1000エントリ。
     * セッション一覧取得のたびに正規表現マッチングが走るのを防止する（UserLocaleCache と同パターン）。
     */
    private static final Map<String, ParseResult> CACHE = new ConcurrentHashMap<>();

    /** パース結果を保持するレコード。 */
    public record ParseResult(String deviceName, DeviceType deviceType) {}

    /**
     * User-Agent 文字列からデバイス表示名とデバイス種別を生成する。
     * 例: "Mozilla/5.0 (Windows NT 10.0; Win64; x64)...Chrome/120" → ("Chrome on Windows 10", DESKTOP)
     *
     * @param userAgent User-Agent 文字列（null許容）
     * @return パース結果。パース不可の場合は ("Unknown Device", UNKNOWN)
     */
    public static ParseResult parse(String userAgent) {
        if (userAgent == null || userAgent.isBlank()) {
            return new ParseResult("Unknown Device", DeviceType.UNKNOWN);
        }
        return CACHE.computeIfAbsent(userAgent, ua -> {
            Client client = UA_PARSER.parse(ua);
            String browser = client.userAgent.family;
            String os = client.os.family;
            String osVersion = client.os.major;

            // デバイス名生成
            String deviceName;
            if ("Other".equals(browser) && "Other".equals(os)) {
                deviceName = "Unknown Device";
            } else {
                String osDisplay = "Other".equals(os) ? "Unknown OS"
                        : (osVersion != null ? os + " " + osVersion : os);
                String browserDisplay = "Other".equals(browser) ? "Unknown Browser" : browser;
                deviceName = browserDisplay + " on " + osDisplay;
            }

            // デバイス種別判定（uap-java の client.device.family を使用）
            DeviceType deviceType = classifyDeviceType(client.device.family, os);
            return new ParseResult(deviceName, deviceType);
        });
    }

    private static DeviceType classifyDeviceType(String deviceFamily, String osFamily) {
        if (deviceFamily != null) {
            String lower = deviceFamily.toLowerCase();
            if (lower.contains("iphone") || lower.contains("android") || lower.contains("mobile")) {
                return DeviceType.MOBILE;
            }
            if (lower.contains("ipad") || lower.contains("tablet")) {
                return DeviceType.TABLET;
            }
        }
        if (osFamily != null) {
            String lower = osFamily.toLowerCase();
            if (lower.contains("ios") || lower.contains("android")) {
                return DeviceType.MOBILE;
            }
        }
        return DeviceType.DESKTOP;  // Windows, macOS, Linux 等はデスクトップとして扱う
    }
}
```

**`DeviceType` 列挙型**:
```java
public enum DeviceType {
    DESKTOP,   // PC（Windows, macOS, Linux）
    MOBILE,    // スマートフォン（iPhone, Android）
    TABLET,    // タブレット（iPad, Android Tablet）
    UNKNOWN    // 判定不能
}
```

フロントエンドは `deviceType` に応じてアイコンを出し分ける: 💻 DESKTOP / 📱 MOBILE / 📟 TABLET / ❓ UNKNOWN

**パース結果キャッシュの保守**:
- `ConcurrentHashMap` を使用。エントリ数が1000を超えた場合は全クリアする（LRU は過剰）
- キャッシュクリアは `@Scheduled(fixedRate = 3600000)` で1時間ごとに実行

**パース優先順位**:
1. `device_name` カラムに値あり → そのまま使用（ユーザー手動命名）。`deviceType` はキャッシュまたは `user_agent` から都度判定
2. `device_name` が NULL → `UserAgentParser.parse(userAgent).deviceName()` で自動生成
3. `user_agent` も NULL → `"Unknown Device"` / `DeviceType.UNKNOWN` を返却

### 5.4 現セッション判定の引数渡し

`getSessions()` メソッドのシグネチャを変更し、現セッション特定情報を引数で渡す。

```java
// 変更前
public ApiResponse<List<SessionResponse>> getSessions(Long userId)

// 変更後
public ApiResponse<List<SessionResponse>> getSessions(Long userId, String currentTokenHash, Long currentSessionId)
```

**Controller 側の処理**:
1. Cookie（`refresh_token`）から Refresh Token を取得し、SHA-256 ハッシュを `currentTokenHash` として渡す（Web クライアント）
2. クエリパラメータ `currentSessionId` をそのまま渡す（モバイルクライアント）
3. 両方 `null` の場合は全セッションの `isCurrent` を `false` とする
4. `currentTokenHash` が非 null の場合はハッシュ照合を優先し、`currentSessionId` は無視する

**TokenResponse への `sessionId` フィールド追加**:
```java
// ログイン・トークンリフレッシュのレスポンスに追加
public class TokenResponse {
    private final String accessToken;
    private final Long sessionId;  // 追加: refresh_tokens.id（モバイルが保持し GET /sessions 時に送信）
    // ...
}
```

モバイル SDK はログインレスポンスの `sessionId` をセキュアストレージに保持し、`GET /api/v1/auth/sessions?currentSessionId={id}` で送信する。

### 5.5 新規デバイスログイン検知通知（F04.3 連携）

新しいデバイスからのログイン時にプッシュ通知でユーザーへ警告する。既存の `NotificationDispatchService`（F04.3）を活用する。

**判定ロジック**:
1. ログイン成功時（Refresh Token 発行後）に、同一ユーザーの直近30日間の `refresh_tokens` レコードを検索
2. 今回の `ip_address` と `device_fingerprint`（User-Agent の SHA-256）の組み合わせが過去に存在しない場合、「新規デバイス」と判定
3. 初回ログイン（`refresh_tokens` に過去レコードが0件、つまり今回発行分のみ）の場合は通知しない（新規アカウント作成直後の不要通知を防止）

**必要なリポジトリクエリ**（`RefreshTokenRepository` に追加）:
```java
/**
 * 指定ユーザーの直近N日以内に同一IP+デバイスフィンガープリントのトークンが存在するか確認する。
 * 新規デバイスログイン検知に使用。
 */
boolean existsByUserIdAndIpAddressAndDeviceFingerprintAndCreatedAtAfter(
        Long userId, String ipAddress, String deviceFingerprint, LocalDateTime since);

/**
 * 指定ユーザーのトークン総数を返す（初回ログイン判定用）。
 */
long countByUserId(Long userId);
```

**通知仕様**:
- **通知種別**: `NEW_DEVICE_LOGIN`（通知基盤の種別に追加。なお `AuditEventType.NEW_DEVICE_LOGIN` は監査ログ用として既に定義済みであり、本通知と併用する）
- **通知タイトル**: `新しいデバイスからのログイン`
- **通知本文**: `{deviceName} からログインがありました。心当たりがない場合はセッション管理画面を確認してください。`（IP アドレスはロック画面プレビューでの漏洩を防ぐため通知本文に含めない。詳細はセッション管理画面で確認する）
- **遷移先**: セッション管理画面（`/account/sessions`）
- **送信先**: ログインしたユーザー自身（今回ログインしたデバイス以外の登録済みプッシュサブスクリプション宛）
- **即時アクションボタン（Web Push Actions）**: 通知に「心当たりがない」ボタンを追加する。タップすると `DELETE /api/v1/auth/sessions?keepCurrent=true` を実行し、現セッション以外を即座に全無効化する。Service Worker の `notificationclick` イベントで処理する
  ```json
  {
    "actions": [
      { "action": "logout_others", "title": "心当たりがない" }
    ]
  }
  ```

**通知テンプレートの外部化**:
- 通知タイトル・本文はコードにハードコードせず、`messages_{locale}.properties` に外部化する
  ```properties
  # messages.properties / messages_{locale}.properties
  notification.new_device_login.title=新しいデバイスからのログイン
  notification.new_device_login.body={0} からログインがありました。心当たりがない場合はセッション管理画面を確認してください。
  notification.new_device_login.action.logout_others=心当たりがない
  ```
- F11.3 の多言語基盤に乗るため、ユーザーの `locale` に応じた言語で通知が送信される
- 文言変更時にデプロイ不要（プロパティファイルの変更のみ）

**処理フロー**:
```
ログイン成功
    ↓
AuthService.login() 内で Refresh Token を発行
    ↓
NewDeviceDetectionService.checkAndNotify(userId, ipAddress, deviceFingerprint, deviceName) を @Async で呼び出し
    ※ ログインレスポンスの遅延を防ぐため非同期実行。locale は引数で明示渡し（§F11.3 4.8 準拠）
    ↓
直近30日の refresh_tokens から同一 ip_address + device_fingerprint の有無を検索
    ↓
[存在しない AND 過去レコード1件以上]
    → AuditEventType.NEW_DEVICE_LOGIN で監査ログ記録
    → NotificationDispatchService で通知送信（1時間3件制限チェック後）
[存在する OR 初回ログイン]
    → 何もしない
```

**スコープ外（将来拡張）**:
- 地理的距離に基づく異常検知（GeoIP 依存のため本フェーズでは行わない）
- メール通知（プッシュ通知のみ。メールは Phase 13 以降で検討）
- ログインブロック機能（検知と通知のみ。ブロックはしない）
- 信頼済みデバイス登録: `ip_address + device_fingerprint` の組み合わせを「信頼済み」として登録し、常用デバイスからの通知を抑止する機能。専用テーブル（`trusted_devices`）の設計が必要なため本フェーズでは行わない

### 5.6 パスワード変更時の自動一括ログアウト

パスワードが変更された場合、現セッション以外の全セッションを自動で無効化する。「パスワードが漏洩したかも」と思ってパスワードを変更したユーザーが、わざわざセッション管理画面に移動して手動で全ログアウトする必要がなくなる。

**実装方針**:
- 既存の `AuthService.changePassword()` の成功後に、`keepCurrent=true` の一括ログアウトロジックを呼び出す
- `LogoutEvent(userId, revokedCount, LogoutType.PASSWORD_CHANGE)` を発行する（`LogoutType` に `PASSWORD_CHANGE` を追加）
- Valkey に `user_invalidated_at:{userId}` を設定し、他デバイスの Access Token を即時無効化する

```
パスワード変更成功
    ↓
AuthService.changePassword() 内で DB 更新完了
    ↓
logoutOtherSessions(userId, currentTokenHash) を呼び出し
    ※ §4 DELETE /sessions?keepCurrent=true と同一ロジックを Service メソッドとして共有
    ↓
他デバイスの Refresh Token を全て revoke + Valkey 設定
    ↓
LogoutEvent(userId, revokedCount, LogoutType.PASSWORD_CHANGE) を発行
```

### 5.7 アクティブセッション上限

1ユーザーあたりのアクティブセッション数に上限を設け、超過時は最も古いセッションを自動 revoke する。セッション数が無制限に増加することによる一覧取得の遅延と DB のゴミ蓄積を防ぐ。

**仕様**:
- **上限**: 10セッション（`mannschaft.auth.max-active-sessions=10` で設定変更可能）
- **判定タイミング**: ログイン時（Refresh Token 新規発行直前）
- **超過時の処理**: `revoked_at IS NULL AND expires_at > NOW()` のセッションを `last_used_at ASC`（最も長く使われていないもの）の順で revoke し、上限 - 1 以下にしてから新規発行する
- **ユーザーへの通知**: 自動 revoke 時はサイレント（通知しない）。セッション管理画面を開いた際に自然と消えている

**必要なリポジトリクエリ**（`RefreshTokenRepository` に追加）:
```java
long countByUserIdAndRevokedAtIsNullAndExpiresAtAfter(Long userId, LocalDateTime now);
```

## 6. セキュリティ考慮事項

- **所有者チェック必須**: 全エンドポイントで `refresh_tokens.user_id == currentUserId` を必ず検証する。ID の連番推測による他ユーザーセッションの操作を防ぐ
- **現セッション保護**: `DELETE /api/v1/auth/sessions/{id}` で現セッションの無効化を禁止する（409 Conflict）。ユーザーが誤って自分のセッションを切断するのを防止
- **レートリミット**: `DELETE /api/v1/auth/sessions`（一括）は Bucket4j で 1分間に5回、`DELETE /api/v1/auth/sessions/{id}`（個別）は 1分間に10回の制限を適用（連番推測による総当たり無効化を抑止）
- **IP アドレスの表示**: IPv6 のフルアドレスは長大なため、表示時に短縮形（`::1` 等）への変換はフロントエンド側で行う。バックエンドは取得時のまま返却する
- **Refresh Token のログ出力禁止**: ハッシュ値であっても `token_hash` をログに出力しない。セッション ID（`refresh_tokens.id`）のみ記録する
- **Valkey 即時無効化**: 一括ログアウト時に `user_invalidated_at` を Valkey に設定し、他デバイスの Access Token（最大15分有効）を即時無効化する。個別ログアウトではこの設定は不要（対象の Refresh Token のみ失効させれば、Access Token は最大15分で自然失効する）
- **PATCH のXSS対策**: `deviceName` はユーザー入力文字列のため、レスポンス返却前に HTML エスケープは行わない（JSON レスポンスのため不要）が、フロントエンドで `v-text` / `textContent` を使用し `v-html` / `innerHTML` を禁止すること
- **セッション列挙攻撃の防止**: `DELETE /api/v1/auth/sessions/{id}` で他ユーザーのセッションIDを指定した場合は `404`（`403` ではなく）を返す。セッションの存在自体を漏洩させない
- **新規デバイス通知の悪用防止**: 攻撃者が大量のログイン試行で通知を乱発する可能性がある。通知送信は `NEW_DEVICE_LOGIN` 種別で1ユーザーあたり1時間に最大3件に制限する（超過分は抑止し、監査ログにのみ記録）

### 8.2 UI 仕様

- 現セッション（`isCurrent=true`）は先頭に固定表示し、緑のインジケーターで強調する
- 現セッションには「ログアウト」ボタンを表示しない（通常のログアウトフローを使用）
- 「全てログアウト」ボタンは確認ダイアログを挟む: 「現在のセッション以外の全デバイスからログアウトします。よろしいですか？」
- 個別「ログアウト」ボタンも確認ダイアログを挟む: 「{deviceName} からログアウトします。よろしいですか？」
- 「最終アクティブ」は相対時間で表示（5分前、1時間前、昨日、3日前 等）
- 「有効期限」は `expiresAt` から計算した残日数 + 日付で表示（例: 「あと17日（2026-04-19）」）。残り3日以下はオレンジ、残り1日以下は赤で警告色表示
- デバイスアイコンは `deviceType` に応じて出し分ける: 💻 `DESKTOP` / 📱 `MOBILE` / 📟 `TABLET` / ❓ `UNKNOWN`
- セッション一覧は手動リロードボタンを設置（自動ポーリングは行わない）
- `userAgent`（生文字列）はデフォルト非表示。各セッションカードに「詳細」トグルを設け、展開時に表示する
- 「全てログアウト」ボタンは `DELETE /api/v1/auth/sessions?keepCurrent=true` を送信する（`keepCurrent` のデフォルトは `false`（後方互換）のため、フロントから明示的に `true` を指定すること）
