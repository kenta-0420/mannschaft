# F11.12: インシデント・メンテナンス管理

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 11
> **最終更新**: 2026-03-19
> **モジュール種別**: 選択式モジュール #20（組織・チーム対象）
> **関連ドキュメント**: F11.1 ワークフロー・承認エンジン（修繕費用の承認連携）、F11.2 予算・会計（修繕費用の支出計上）、F04.3 プッシュ通知（ステータス変更・SLA 超過通知）

---

## 1. 概要

物理的な問題（設備不具合・備品破損・漏水・騒音苦情等）の報告から対応完了までのライフサイクルを一元管理する機能。F10.2 のモデレーションが「コンテンツの報告」を扱うのに対し、本機能は「物理的な問題の報告 → 受付 → 担当者アサイン → 対応 → 完了確認 → クローズ」のフローを提供する。カテゴリごとに SLA（対応期限）を設定し、期限超過時にはエスカレーション通知を自動送信する。修繕費用が発生する場合は F11.1 承認ワークフローを経由し、承認後に F11.2 予算から支出を自動計上する。定期メンテナンス（月次点検・年次検査等）を CRON 式で定義し、日次バッチでインシデントを自動生成する機能も提供する。

**主要ユースケース**:
- マンション管理組合: 漏水報告 → 管理会社に通知 → 業者手配 → 修繕完了 → 住民に報告
- スポーツチーム: 備品破損報告 → 購入承認 → 予算から支出
- 企業: 設備不具合報告 → 総務部が対応 → 完了報告
- 学校: 施設破損報告 → 事務局が対応 → 報告者に完了通知

|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |

| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `name` | VARCHAR(100) | NO | — | カテゴリ名（例: 「設備故障」「漏水」「騒音」「清掃」「セキュリティ」「その他」） |
| `description` | VARCHAR(500) | YES | NULL | カテゴリの説明 |
| `icon` | VARCHAR(30) | YES | NULL | アイコン（emoji or アイコン名） |
| `color` | VARCHAR(7) | YES | NULL | テーマカラー（hex。例: `#EF4444`） |
| `sla_hours` | INT | NO | 168 | SLA 対応期限（時間単位。デフォルト: 168 = 7日） |
| `is_active` | BOOLEAN | NO | TRUE | カテゴリが有効か（無効化すると新規報告で選択不可） |
| `sort_order` | INT | NO | 0 | 表示順 |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。作成した ADMIN。ON DELETE RESTRICT |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_ic_scope (scope_type, scope_id, is_active, sort_order)  -- スコープ別の有効カテゴリ一覧
UNIQUE KEY uq_ic_scope_name (scope_type, scope_id, name)          -- スコープ内でカテゴリ名一意
```

**制約・備考**
- `scope_type` + `scope_id` はポリモーフィック参照。FK 制約なし（アプリ層で検証）
- 1スコープあたり最大 **20 カテゴリ**（論理削除されていないもの）
- カテゴリを論理削除しても、既存インシデントのカテゴリ表示は「（削除済みカテゴリ）」として値を表示
- `is_active = FALSE`: 既存のインシデントには影響しないが、新規報告時にはこのカテゴリを選択不可
- `sla_hours` はカテゴリごとのデフォルト値。インシデント作成時に `incidents.sla_deadline` として個別に算出される

#### `incidents`

インシデント報告レコード。MEMBER 以上が作成し、ADMIN / DEPUTY_ADMIN がアサイン・対応管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `category_id` | BIGINT UNSIGNED | NO | — | FK → incident_categories。ON DELETE RESTRICT |
| `incident_type` | VARCHAR(20) | NO | 'REPORT' | 種別（`REPORT` = 通常報告 / `MAINTENANCE` = 定期メンテナンス自動生成） |
| `title` | VARCHAR(200) | NO | — | インシデントタイトル（例: 「3階廊下の照明が点灯しない」） |
| `description` | TEXT | NO | — | 詳細説明（最大 3,000 文字） |
| `location` | VARCHAR(200) | YES | NULL | 場所（例: 「3階 廊下」「A棟 エントランス」） |
| `priority` | VARCHAR(20) | NO | 'MEDIUM' | 緊急度（`CRITICAL` / `HIGH` / `MEDIUM` / `LOW`） |
| `status` | VARCHAR(20) | NO | 'REPORTED' | ステータス（`REPORTED` / `ACKNOWLEDGED` / `IN_PROGRESS` / `RESOLVED` / `CONFIRMED` / `CLOSED` / `REOPENED`） |
| `sla_deadline` | DATETIME | NO | — | SLA 期限（報告日時 + category.sla_hours。priority による上書きあり） |
| `is_sla_breached` | BOOLEAN | NO | FALSE | SLA 超過フラグ |
| `reported_by` | BIGINT UNSIGNED | NO | — | FK → users（報告者）。ON DELETE RESTRICT |
| `reported_at` | DATETIME | NO | CURRENT_TIMESTAMP | 報告日時 |
| `acknowledged_at` | DATETIME | YES | NULL | 受付日時 |
| `resolved_at` | DATETIME | YES | NULL | 対応完了日時 |
| `closed_at` | DATETIME | YES | NULL | クローズ日時 |
| `resolution_summary` | TEXT | YES | NULL | 対応結果の概要（最大 2,000 文字。RESOLVED 時に担当者が記入） |
| `estimated_cost` | DECIMAL(12,0) | YES | NULL | 見積り費用（日本円） |
| `actual_cost` | DECIMAL(12,0) | YES | NULL | 実績費用（日本円） |
| `workflow_request_id` | BIGINT UNSIGNED | YES | NULL | FK → workflow_requests。修繕費用の承認申請 ID（F11.1 連携） |
| `maintenance_schedule_id` | BIGINT UNSIGNED | YES | NULL | FK → maintenance_schedules。定期メンテナンスからの自動生成元 |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_inc_scope (scope_type, scope_id, status, reported_at DESC)       -- スコープ別のインシデント一覧
INDEX idx_inc_reported_by (reported_by, status, reported_at DESC)           -- 報告者別の一覧
INDEX idx_inc_category (category_id, status, reported_at DESC)             -- カテゴリ別の一覧
INDEX idx_inc_priority_status (scope_type, scope_id, priority, status)     -- 緊急度×ステータスでの絞り込み
INDEX idx_inc_sla (is_sla_breached, sla_deadline)                          -- SLA 超過チェックバッチ用
INDEX idx_inc_workflow (workflow_request_id)                                -- ワークフロー連携からの逆引き
INDEX idx_inc_maintenance (maintenance_schedule_id)                         -- メンテナンススケジュールからの逆引き
```

**制約・備考**
- `status` 遷移:
  - `REPORTED` → `ACKNOWLEDGED`（ADMIN / DEPUTY_ADMIN が受付）
  - `ACKNOWLEDGED` → `IN_PROGRESS`（担当者がアサインされ対応開始）
  - `IN_PROGRESS` → `RESOLVED`（担当者が対応完了を報告。`resolution_summary` 必須）
  - `RESOLVED` → `CONFIRMED`（報告者が「修理完了を確認」と判断）
  - `RESOLVED` → `REOPENED`（報告者が「まだ直っていない」と判断。`REOPENED` は `ACKNOWLEDGED` と同等の扱いで再対応フローに入る）
  - `CONFIRMED` → `CLOSED`（ADMIN が最終クローズ。または `CONFIRMED` から 7 日経過で自動クローズ）
  - `REOPENED` → `IN_PROGRESS`（再アサイン後に対応再開）
  - ADMIN は任意のステータスから `CLOSED` への強制クローズが可能
- `priority` による SLA 上書き: 報告時にカテゴリの `sla_hours` を基準とするが、`priority` によって以下のルールで上書き:
  - `CRITICAL`: min(category.sla_hours, 4) 時間
  - `HIGH`: min(category.sla_hours, 24) 時間
  - `MEDIUM`: category.sla_hours そのまま
  - `LOW`: max(category.sla_hours, 168) 時間（最低 7 日）
- `is_sla_breached`: `@Scheduled` バッチ（15 分間隔）で `sla_deadline < NOW() AND status NOT IN ('RESOLVED', 'CONFIRMED', 'CLOSED')` を検出し TRUE に更新
- `incident_type = 'MAINTENANCE'`: `maintenance_schedules` からの自動生成。`maintenance_schedule_id` で生成元を追跡
- `workflow_request_id`: 修繕費用が発生する場合、F11.1 の external API で承認申請を作成し、その `workflow_requests.id` を保存。`source_type = 'INCIDENT'`, `source_id = incidents.id`
- 論理削除: `REPORTED` / `CLOSED` の場合のみ削除可能。処理中のインシデントは削除不可（証跡保持）
- `reported_by` ON DELETE RESTRICT: 報告者の退会時、処理中のインシデントがある場合はエラー

#### `incident_assignments`

担当者アサイン。内部ユーザーまたは外部業者を指定。1 インシデントに複数アサイン可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `incident_id` | BIGINT UNSIGNED | NO | — | FK → incidents。ON DELETE CASCADE |
| `assignee_type` | VARCHAR(20) | NO | — | 担当者種別（`USER` = 内部ユーザー / `EXTERNAL` = 外部業者） |
| `user_id` | BIGINT UNSIGNED | YES | NULL | FK → users。`assignee_type = 'USER'` の場合。ON DELETE SET NULL |
| `external_name` | VARCHAR(100) | YES | NULL | 外部業者名（`assignee_type = 'EXTERNAL'` の場合） |
| `external_contact` | VARCHAR(200) | YES | NULL | 外部業者の連絡先（電話番号・メール等） |
| `external_note` | VARCHAR(500) | YES | NULL | 外部業者への依頼メモ |
| `assigned_by` | BIGINT UNSIGNED | NO | — | FK → users（アサイン実行者）。ON DELETE RESTRICT |
| `assigned_at` | DATETIME | NO | CURRENT_TIMESTAMP | アサイン日時 |
| `is_primary` | BOOLEAN | NO | FALSE | 主担当フラグ（1 インシデントにつき 1 人のみ TRUE） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_iasn_incident (incident_id, is_primary)              -- インシデント別アサイン一覧
INDEX idx_iasn_user (user_id, assigned_at DESC)                -- ユーザー別のアサイン一覧
INDEX idx_iasn_assigned_by (assigned_by)                       -- アサイン実行者からの逆引き
```

**制約・備考**
- `assignee_type = 'USER'`: `user_id` 必須、`external_name` / `external_contact` は NULL
- `assignee_type = 'EXTERNAL'`: `external_name` 必須、`user_id` は NULL
- アプリ層で XOR チェック: `(user_id IS NOT NULL AND assignee_type = 'USER') OR (external_name IS NOT NULL AND assignee_type = 'EXTERNAL')`
- `is_primary = TRUE` は 1 インシデントにつき最大 1 レコード。アプリ層で制御
- `user_id` ON DELETE SET NULL: アサイン先ユーザーが退会した場合、アサイン記録は残るが user_id は NULL に。再アサインが必要
- 外部業者のアサインは通知対象外（内部ユーザーのみ通知）

#### `maintenance_schedules`

定期メンテナンススケジュール定義。CRON 式でスケジュールを定義し、日次バッチでインシデントを自動生成する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `category_id` | BIGINT UNSIGNED | NO | — | FK → incident_categories。ON DELETE RESTRICT |
| `title` | VARCHAR(200) | NO | — | メンテナンスタイトル（例: 「月次消防設備点検」「年次エレベーター検査」） |
| `description` | TEXT | YES | NULL | 作業内容の説明（最大 3,000 文字） |
| `location` | VARCHAR(200) | YES | NULL | 作業場所 |
| `priority` | VARCHAR(20) | NO | 'MEDIUM' | デフォルト緊急度 |
| `cron_expression` | VARCHAR(100) | NO | — | CRON 式（例: `0 0 1 * *` = 毎月1日、`0 0 1 4 *` = 毎年4月1日） |
| `next_execution_date` | DATE | NO | — | 次回実行予定日（バッチが参照。実行後に次回日を再計算して更新） |
| `default_assignee_type` | VARCHAR(20) | YES | NULL | デフォルト担当者種別（`USER` / `EXTERNAL`）。NULL = 未指定 |
| `default_assignee_user_id` | BIGINT UNSIGNED | YES | NULL | デフォルト担当者 user_id |
| `default_external_name` | VARCHAR(100) | YES | NULL | デフォルト外部業者名 |
| `default_external_contact` | VARCHAR(200) | YES | NULL | デフォルト外部業者連絡先 |
| `is_active` | BOOLEAN | NO | TRUE | スケジュールが有効か |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。作成した ADMIN。ON DELETE RESTRICT |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_ms_scope (scope_type, scope_id, is_active)                 -- スコープ別の有効スケジュール一覧
INDEX idx_ms_next_exec (is_active, next_execution_date)              -- 日次バッチでの実行対象検索
INDEX idx_ms_category (category_id)                                  -- カテゴリ別スケジュール
```

**制約・備考**
- `cron_expression`: Spring の `CronExpression` クラスでパース可能な形式。バリデーション時にパース成功を検証
- `next_execution_date`: スケジュール作成時に CRON 式から算出。バッチ実行後に次回日を再計算して更新
- 日次バッチ（06:00 JST）で `next_execution_date <= TODAY AND is_active = TRUE` のスケジュールを取得し、インシデント（`incident_type = 'MAINTENANCE'`）を自動生成
- `default_assignee_*`: 自動生成されたインシデントに自動アサインする担当者。NULL の場合はアサインなし（ADMIN が手動アサイン）
- 1スコープあたり最大 **50 スケジュール**（論理削除されていないもの）
- 論理削除しても、既に生成済みのインシデントには影響しない

## 5. ビジネスロジック

### インシデント報告フロー

```
1. MEMBER 以上が報告画面でカテゴリ・タイトル・説明・場所・緊急度・写真を入力
2. Service 層で以下を実行:
   a. カテゴリの存在・有効性チェック
   b. 添付写真の S3 存在チェック（HeadObject）
   c. SLA 期限を算出: reported_at + priority に応じた sla_hours
   d. incidents に INSERT（status = 'REPORTED'）
   e. incident_attachments に INSERT（写真がある場合）
   f. incident_status_histories に INSERT（from_status = NULL, to_status = 'REPORTED'）
   g. ApplicationEvent (IncidentReportedEvent) を発行
3. 通知サービスがイベントを受信:
   a. スコープ内の ADMIN + DEPUTY_ADMIN（INCIDENT_MANAGE 権限あり）にプッシュ通知
   b. 「新しいインシデント報告があります: {title}」
   c. priority = 'CRITICAL' の場合は URGENT 通知
```

### 担当者アサインフロー

```
1. ADMIN / DEPUTY_ADMIN がインシデントに担当者をアサイン
2. Service 層で以下を実行:
   a. is_primary の排他チェック（既に主担当がいる場合は is_primary = FALSE に強制変更するか確認）
   b. incident_assignments に INSERT
   c. インシデントのステータスが ACKNOWLEDGED でない場合:
      - REPORTED → ACKNOWLEDGED に自動遷移（受付とアサインを同時実行）
      - incident_status_histories に記録
   d. ApplicationEvent (IncidentAssignedEvent) を発行
3. 通知サービスがイベントを受信:
   a. アサインされた内部ユーザーにプッシュ通知（「インシデントが割り当てられました: {title}」）
   b. 報告者にステータス変更通知（「報告が受け付けられました」）
```

### ステータス遷移フロー

```
■ REPORTED → ACKNOWLEDGED（受付）
  - ADMIN / DEPUTY_ADMIN が実行
  - acknowledged_at をセット

■ ACKNOWLEDGED / REOPENED → IN_PROGRESS（対応開始）
  - アサインされた担当者または ADMIN が実行
  - 担当者が1人もアサインされていない場合はエラー

■ IN_PROGRESS → RESOLVED（対応完了）
  - アサインされた担当者または ADMIN が実行
  - resolution_summary 必須
  - resolved_at をセット
  - 報告者に通知:「インシデントが解決されました。確認をお願いします」

■ RESOLVED → CONFIRMED（報告者確認）
  - 報告者本人のみが実行
  - 確認日時を記録

■ RESOLVED → REOPENED（再オープン）
  - 報告者本人のみが実行
  - reopen のコメント必須（理由の記録）
  - 担当者 + ADMIN に通知:「インシデントが再オープンされました: {reopen_comment}」

■ CONFIRMED → CLOSED（クローズ）
  - ADMIN が実行。または CONFIRMED から 7 日経過で自動クローズ（日次バッチ）
  - closed_at をセット

■ 強制クローズ（ADMIN のみ）
  - 任意のステータスから CLOSED に遷移可能
  - 強制クローズ理由のコメント必須
  - 監査ログに INCIDENT_FORCE_CLOSED として記録
  - 報告者 + 全担当者に通知
```

### SLA 超過チェックバッチ

```
1. @Scheduled バッチ（15 分間隔）で実行
2. 対象: is_sla_breached = FALSE AND sla_deadline < NOW()
         AND status NOT IN ('RESOLVED', 'CONFIRMED', 'CLOSED')
3. 該当インシデントの is_sla_breached を TRUE に更新
4. ApplicationEvent (IncidentSlaBreachedEvent) を発行
5. 通知サービスがイベントを受信:
   a. スコープ内の ADMIN にエスカレーション通知（URGENT 優先度）
   b. アサイン済み担当者にリマインド通知
   c. 「SLA 期限を超過しました: {title}（緊急度: {priority}）」
6. 監査ログに INCIDENT_SLA_BREACHED として記録
```

### 自動クローズバッチ

```
1. 日次バッチ（01:00 JST）で実行
2. 対象: status = 'CONFIRMED' AND (confirmed 遷移日時から 7 日経過)
   - incident_status_histories から CONFIRMED への最新遷移日時を参照
3. status を CLOSED に更新、closed_at をセット
4. incident_status_histories に記録（changed_by = SYSTEM ユーザー）
5. 報告者に通知:「インシデントが自動クローズされました」
```

### 定期メンテナンスバッチ

```
1. 日次バッチ（06:00 JST）で実行
2. 対象: maintenance_schedules WHERE is_active = TRUE
         AND next_execution_date <= TODAY
         AND deleted_at IS NULL
3. 該当スケジュールごとに:
   a. incidents に INSERT:
      - incident_type = 'MAINTENANCE'
      - title = maintenance_schedules.title
      - description = maintenance_schedules.description
      - location = maintenance_schedules.location
      - priority = maintenance_schedules.priority
      - category_id = maintenance_schedules.category_id
      - status = 'REPORTED'
      - maintenance_schedule_id = maintenance_schedules.id
      - reported_by = maintenance_schedules.created_by（バッチ実行者としてスケジュール作成者を使用）
   b. default_assignee_type が設定されている場合:
      - incident_assignments に INSERT（自動アサイン）
      - status を ACKNOWLEDGED に自動遷移
   c. next_execution_date を CRON 式から再計算して更新
   d. ApplicationEvent (MaintenanceIncidentCreatedEvent) を発行
4. 通知: ADMIN + アサイン担当者に「定期メンテナンスが生成されました: {title}」
```

### 修繕費用承認連携フロー（F11.1 / F11.2）

```
1. ADMIN / DEPUTY_ADMIN がインシデント画面で「費用承認を申請」を実行
2. Service 層:
   a. F11.1 の external API を内部呼び出し:
      POST /api/v1/workflows/templates/{templateId}/requests/external
      - source_type: 'INCIDENT'
      - source_id: incidents.id
      - auto_submit: true
      - field_values: { "amount": estimated_cost, "incident_title": title, "description": ... }
   b. 返却された workflow_requests.id を incidents.workflow_request_id に保存
   c. incidents.estimated_cost を更新

3. 承認完了時（F11.1 → 本機能へのイベント連携）:
   @TransactionalEventListener(phase = AFTER_COMMIT)
   @Async("event-pool")
   public void onWorkflowApproved(WorkflowApprovedEvent event) {
       if ("INCIDENT".equals(event.getSourceType())) {
           incidentExpenseService.onExpenseApproved(event.getSourceId());
       }
   }

4. incidentExpenseService.onExpenseApproved():
   a. incidents.actual_cost を更新（承認された金額）
   b. ApplicationEvent (IncidentExpenseApprovedEvent) を発行

5. F11.2 予算連携:
   @TransactionalEventListener(phase = AFTER_COMMIT)
   @Async("event-pool")
   public void onIncidentExpenseApproved(IncidentExpenseApprovedEvent event) {
       // budget_transactions に支出レコードを自動計上
       // source_type = 'INCIDENT', source_id = incidents.id
       budgetTransactionService.recordExpense(event);
   }

6. 却下時: WorkflowRejectedEvent を受信 → workflow_request_id を NULL にリセット
   → 報告者 + ADMIN に「費用承認が却下されました」通知
```

### 重要な判定ロジック

- **ステータス変更権限**: 各遷移で実行可能なロールを厳密にチェック。ADMIN は全遷移可能（強制クローズ含む）。担当者は `IN_PROGRESS` → `RESOLVED` のみ。報告者は `RESOLVED` → `CONFIRMED` / `REOPENED` のみ
- **インシデントの可視性**: ADMIN / DEPUTY_ADMIN は全インシデントを閲覧可能。MEMBER は自分が報告したインシデント + 自分にアサインされたインシデントのみ
- **内部コメントの可視性**: `is_internal = TRUE` のコメントは ADMIN / DEPUTY_ADMIN のみ閲覧可能。報告者・一般担当者には非表示
- **SLA 期限の再計算**: 緊急度（priority）を変更した場合、SLA 期限を再計算する。ただし既に `is_sla_breached = TRUE` の場合はリセットしない
- **定期メンテナンスの重複防止**: 同一 `maintenance_schedule_id` で同日に既にインシデントが存在する場合はスキップ（冪等性保証）

### 通知トリガー一覧

| トリガー | 通知先 | 優先度 | 条件 |
|---------|-------|--------|------|
| インシデント報告 | スコープ内 ADMIN + DEPUTY_ADMIN（INCIDENT_MANAGE） | NORMAL（CRITICAL 時は URGENT） | インシデント作成 |
| 担当者アサイン | アサインされた内部ユーザー | NORMAL | アサイン追加 |
| 受付（ACKNOWLEDGED） | 報告者 | NORMAL | ステータス遷移 |
| 対応完了（RESOLVED） | 報告者 | NORMAL | ステータス遷移 |
| 再オープン（REOPENED） | 全担当者 + ADMIN | HIGH | 報告者が再オープン |
| クローズ（CLOSED） | 報告者 | LOW | ステータス遷移 |
| 強制クローズ | 報告者 + 全担当者 | HIGH | ADMIN 強制実行 |
| SLA 超過 | ADMIN + アサイン済み担当者 | URGENT | バッチ検出 |
| コメント投稿 | 報告者 + 全内部担当者（投稿者除く） | NORMAL | コメント追加 |
| 定期メンテナンス生成 | ADMIN + アサイン担当者 | NORMAL | バッチ生成 |
| 費用承認完了 | 報告者 + ADMIN | NORMAL | F11.1 からのイベント |
| 費用承認却下 | 報告者 + ADMIN | HIGH | F11.1 からのイベント |

## 6. セキュリティ考慮事項

- **認可チェック**: IncidentService の入り口で `scope_type` / `scope_id` と `currentUser` の所属を必ず検証する。インシデントの閲覧は報告者本人・アサイン担当者・ADMIN / DEPUTY_ADMIN に限定
- **レートリミット**: インシデント報告 API に `Bucket4j` で 1 時間に 10 回の制限を適用（大量報告によるスパム防止）
- **楽観的ロック**: `incidents`, `incident_categories`, `maintenance_schedules` に `version` カラムを適用。ステータス変更操作時に `version` の一致を検証し、同時操作による不整合を防止
- **添付ファイル**: Pre-signed URL 方式でアップロード。`contentType` ホワイトリスト検証（`image/jpeg`, `image/png`, `image/webp` のみ）+ `fileSize` 上限チェック（5MB）
- **ステータス遷移の不正防止**: 各ステータス遷移の実行可能ロールと前提ステータスをサービス層で厳密にチェック。不正な遷移は 409 Conflict で拒否
- **内部コメントのアクセス制御**: `is_internal = TRUE` のコメントは API レスポンスで ADMIN / DEPUTY_ADMIN 以外にフィルタ。SQL クエリレベルでも分離
- **SLA バッチの冪等性**: 同一インシデントに対する SLA 超過通知は `is_sla_breached` フラグで 1 回のみ送信。重複通知を防止
- **強制クローズの監査**: ADMIN がインシデントを強制クローズした場合、監査ログに `INCIDENT_FORCE_CLOSED` として記録し、報告者 + 全担当者に通知

**マイグレーション上の注意点**
