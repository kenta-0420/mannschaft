# F09.1: 住民台帳・物件情報

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 9
> **最終更新**: 2026-03-15
> **モジュール種別**: 選択式モジュール #9
> **関連ドキュメント**: F09.3 駐車場区画管理（譲渡掲示の棲み分け）、F04.3 プッシュ通知（入退居・掲示・契約満了通知）

---

## 1. 概要

マンション管理組合・集合住宅・社宅等で、居室（部屋番号）と区分所有者・賃借人を紐付けて管理する機能。所有者/賃借人の区別、入退居日の記録、共有持分比率の管理を行う。台帳全体を ADMIN/DEPUTY_ADMIN のみ閲覧可に設定でき、個人は自室・自身の情報のみ閲覧可能（プライバシー保護）。居住者間の物件売買・賃貸希望を掲示する物件掲示板を併設し、コミュニティ内での住み替え・転貸の情報共有を支援する。駐車場区画の譲渡希望は駐車場区画管理モジュール（F09.3）で掲示するため、本機能の対象外とする。

### テンプレート推奨

| テンプレート | 住民台帳 | 備考 |
|-------------|---------|------|
| マンション | ○（自動有効化） | 主要ターゲット |
| その他全テンプレート | — | 手動で有効化可能 |

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全組織・チームの台帳参照 |
| ADMIN | 居室 CRUD・居住者 CRUD・台帳プライバシー設定・物件掲示板管理・居住者確認（verify） |
| DEPUTY_ADMIN | `MANAGE_RESIDENTS` 権限を持つ場合: ADMIN と同等 |
| MEMBER | 自室・自身の情報のみ閲覧・物件掲示板の閲覧/投稿（自室の物件のみ） |
| SUPPORTER | 物件掲示板の閲覧のみ（台帳アクセス不可） |
| GUEST | 対象外 |

### 対象レベル
- [x] 組織 (Organization) — 管理組合全体の台帳
- [x] チーム (Team) — 棟別・建物別の台帳
- [x] 個人 (Personal) — 自室・自身の情報のみ閲覧

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `dwelling_units` | 居室マスター（部屋番号・間取り・面積等） | あり |
| `resident_registry` | 居住者情報（区分所有者/賃借人・入退居日・共有持分） | あり |
| `property_listings` | 物件掲示板（売買/賃貸希望の掲示） | あり |

### テーブル定義

#### `dwelling_units`

居室マスター。1建物内の各部屋を管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `team_id` | BIGINT UNSIGNED | YES | NULL | FK → teams。TEAM の場合に設定 |
| `organization_id` | BIGINT UNSIGNED | YES | NULL | FK → organizations。ORGANIZATION の場合に設定 |
| `unit_number` | VARCHAR(50) | NO | — | 部屋番号（例: 「101」「202A」「3F-5」） |
| `floor` | SMALLINT | YES | NULL | 階数 |
| `area_sqm` | DECIMAL(8,2) | YES | NULL | 専有面積（㎡） |
| `layout` | VARCHAR(20) | YES | NULL | 間取り（例: 「3LDK」「1R」「2DK」） |
| `unit_type` | VARCHAR(20) | NO | 'STANDARD' | 居室種別（`STANDARD`: 住戸 / `COMMERCIAL`: 店舗 / `OFFICE`: 事務所 / `COMMON`: 共用施設） |
| `notes` | TEXT | YES | NULL | 備考（管理者用メモ） |
| `resident_count` | SMALLINT | NO | 0 | 現入居者数（denormalize。`move_out_date IS NULL` の居住者数） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
UNIQUE KEY uq_du_team_unit (team_id, unit_number)            -- TEAM スコープ内の部屋番号重複防止
UNIQUE KEY uq_du_org_unit (organization_id, unit_number)     -- ORGANIZATION スコープ内の部屋番号重複防止
INDEX idx_du_scope (scope_type, team_id, organization_id, unit_type)  -- スコープ別居室一覧
```

**制約・備考**
- `scope_type` のスコープ制約: TEAM → `team_id` NOT NULL、ORGANIZATION → `organization_id` NOT NULL（XOR 制約）
- `unit_number` はスコープ内でユニーク。UNIQUE KEY を `team_id` / `organization_id` 別に2本定義（単一の UNIQUE KEY に nullable カラムを含むと MySQL で重複防止が効かないため）
- `unit_type = 'COMMON'`: エントランス・集会室等の共用施設。居住者は登録不可
- `resident_count`: 居住者の追加/退去時にアトミック更新（+1/-1）
- 論理削除: 居住者が紐付いている居室は削除不可（先に全居住者の退去処理が必要）

---

#### `resident_registry`

居住者情報。区分所有者/賃借人の区別・入退居日・連絡先を管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `dwelling_unit_id` | BIGINT UNSIGNED | NO | — | FK → dwelling_units。ON DELETE RESTRICT |
| `user_id` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL。プラットフォーム登録済みユーザーとの紐付け。未登録居住者は NULL |
| `resident_type` | VARCHAR(20) | NO | — | 居住者種別（`OWNER`: 区分所有者 / `RENTER`: 賃借人） |
| `resident_name` | VARCHAR(100) | NO | — | 居住者氏名（`user_id` が設定されている場合でも管理台帳用に独立して保持） |
| `resident_phone` | VARCHAR(255) | YES | NULL | 連絡先電話番号（AES-256 暗号化 + Base64 エンコードして保存。平文は最大20桁だが暗号化後は44文字以上） |
| `resident_email` | VARCHAR(255) | YES | NULL | 連絡先メールアドレス |
| `move_in_date` | DATE | NO | — | 入居日 |
| `move_out_date` | DATE | YES | NULL | 退居日。NULL = 現入居者 |
| `is_primary` | BOOLEAN | NO | FALSE | 主たる居住者フラグ（通知・連絡の宛先）。1居室に複数居住者がいる場合、1人を主居住者に指定。Service 層で自動判定: 居室の最初の居住者は自動的に TRUE |
| `ownership_share` | DECIMAL(5,2) | YES | NULL | 共有持分比率（%）。`OWNER` の場合に使用（例: 50.00 = 50%）。単独所有の場合は NULL or 100.00 |
| `lease_end_date` | DATE | YES | NULL | 賃貸借契約満了日。`RENTER` の場合に使用 |
| `is_public` | BOOLEAN | NO | FALSE | 個人への公開フラグ。`FALSE` = ADMIN/DEPUTY_ADMIN のみ閲覧可。`TRUE` = 本人が自分の情報を閲覧可能 |
| `verified_at` | DATETIME | YES | NULL | ADMIN による確認日時 |
| `verified_by` | BIGINT UNSIGNED | YES | NULL | FK → users。確認した管理者 |
| `last_lease_notified_days` | INT | YES | NULL | 最後に契約満了通知を送信した残日数ポイント（90 or 30）。同一ポイントの重複送信を防止 |
| `notes` | TEXT | YES | NULL | 備考（管理者用メモ。例: 「長期海外出張中」） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_rr_unit (dwelling_unit_id, move_out_date)         -- 居室別の現入居者（move_out_date IS NULL）
INDEX idx_rr_user (user_id)                                  -- ユーザー別の居住情報
INDEX idx_rr_type (dwelling_unit_id, resident_type)          -- 居室別の所有者/賃借人一覧
```

**制約・備考**
- `dwelling_unit_id` ON DELETE RESTRICT: 居住者がいる居室は削除不可
- `user_id`: プラットフォームに登録済みのユーザーと紐付け。高齢者等の未登録居住者は NULL のまま `resident_name` のみで管理
- `resident_phone`: 個人情報のため AES-256 で暗号化して保存（`users.postal_code` 等と同じ方式）
- `move_out_date` >= `move_in_date`（Service 層バリデーション）
- `lease_end_date` >= `move_in_date`（Service 層バリデーション）
- `ownership_share`: `OWNER` の場合に 0.01〜100.00 の範囲。同一居室の全 OWNER の合計が 100.00 を超えないことを Service 層で検証
- `is_public` のデフォルト `FALSE`: プライバシー保護を優先。ADMIN が明示的に `TRUE` に設定しない限り、居住者本人にも情報は非公開
- `is_primary`: 1居室あたり `is_primary = TRUE` は最大1名（Service 層バリデーション）
- 論理削除: `move_out_date IS NOT NULL`（退去済み）の居住者のみ論理削除可能。現入居者を論理削除すると `resident_count` との不整合が発生するため、先に退去処理（PATCH /move-out）を行う必要がある。退去済みでも台帳履歴として保持可能（論理削除しなければ past_residents に表示される）

#### `property_listings`

物件掲示板。居住者間の売買希望・賃貸希望を掲示する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `dwelling_unit_id` | BIGINT UNSIGNED | NO | — | FK → dwelling_units。ON DELETE CASCADE |
| `posted_by` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT |
| `listing_type` | VARCHAR(20) | NO | — | 掲示種別（`SALE`: 売買希望 / `RENT`: 賃貸希望） |
| `title` | VARCHAR(200) | NO | — | 掲示タイトル（例: 「101号室 売却希望」） |
| `description` | TEXT | YES | NULL | 詳細説明（間取り・リフォーム状況・希望条件等） |
| `asking_price` | DECIMAL(15,2) | YES | NULL | 売却希望価格（円）。`SALE` の場合に使用 |
| `monthly_rent` | DECIMAL(10,2) | YES | NULL | 希望月額賃料（円）。`RENT` の場合に使用 |
| `contact_method` | VARCHAR(20) | NO | 'IN_APP' | 連絡方法（`IN_APP`: アプリ内メッセージ / `EMAIL`: メール / `PHONE`: 電話） |
| `status` | VARCHAR(20) | NO | 'ACTIVE' | ステータス（`ACTIVE`: 掲示中 / `CLOSED`: 成約済み / `WITHDRAWN`: 取り下げ） |
| `expires_at` | DATE | YES | NULL | 掲示期限。到達時に自動 `WITHDRAWN` |
| `photo_keys` | JSON | YES | NULL | 写真の S3 キー配列（最大5枚。例: `["listings/1/photo1.webp", ...]`）。Pre-signed URL で配信 |
| `view_count` | INT UNSIGNED | NO | 0 | 閲覧数（denormalize） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_pl_scope (dwelling_unit_id, status, created_at DESC)  -- 居室別掲示一覧
INDEX idx_pl_type (listing_type, status, created_at DESC)        -- 種別別の掲示一覧
INDEX idx_pl_posted_by (posted_by, created_at DESC)              -- 投稿者別一覧
INDEX idx_pl_expires (status, expires_at)                         -- 期限切れバッチ用
```

**制約・備考**
- `posted_by` ON DELETE RESTRICT: 投稿者退会時は先に掲示の取り下げ or 引継ぎが必要
- `SALE` の場合: `asking_price` が必須（Service 層バリデーション）。`monthly_rent` は NULL
- `RENT` の場合: `monthly_rent` が必須。`asking_price` は NULL
- 投稿権限: 該当居室の現入居者（`move_out_date IS NULL`）のみ。ADMIN は任意の居室に投稿可能
- 公開範囲: スコープ内のメンバーのみ閲覧可能（外部公開なし）
- 駐車場区画の譲渡希望は `parking_listings`（F09.3）で管理。本テーブルには含まない
- ステータス遷移: `ACTIVE` → `CLOSED`（成約）/ `WITHDRAWN`（取り下げ）。`CLOSED` / `WITHDRAWN` からの復帰は不可（再投稿で対応）
- `view_count`: `GET /property-listings/{id}` の詳細取得時にアトミック更新（+1）。一覧取得ではカウントしない。同一ユーザーの複数回閲覧もカウント（厳密なユニーク管理はコスト対効果が合わないためシンプルに総閲覧数）
- 論理削除: `ACTIVE` の掲示は論理削除不可（先に `WITHDRAWN` に遷移してから削除）。`CLOSED` / `WITHDRAWN` は論理削除可能

### ER図（テキスト形式）
```
teams (1) ──── (N) dwelling_units（team_id; TEAM）
organizations (1) ──── (N) dwelling_units（organization_id; ORGANIZATION）
dwelling_units (1) ──── (N) resident_registry
dwelling_units (1) ──── (N) property_listings

users (1) ──── (N) resident_registry（user_id; nullable）
users (1) ──── (N) property_listings（posted_by）
```

### リクエスト／レスポンス仕様

#### `POST /api/v1/dwelling-units`

**認可**: ADMIN / DEPUTY_ADMIN（MANAGE_RESIDENTS）

居室を作成する。

**リクエストボディ**
```json
{
  "scope_type": "ORGANIZATION",
  "organization_id": 1,
  "unit_number": "101",
  "floor": 1,
  "area_sqm": 72.50,
  "layout": "3LDK",
  "unit_type": "STANDARD",
  "notes": "角部屋・南向き"
}
```

- `unit_number`: 必須。スコープ内でユニーク
- `floor` / `area_sqm` / `layout`: 任意
- `unit_type`: 任意（デフォルト: `STANDARD`）

**レスポンス（201 Created）**: 居室詳細を返却

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | バリデーションエラー / scope_type とスコープ ID の不整合 |
| 403 | 権限不足 |
| 409 | 同一スコープ内に同じ `unit_number` が存在 |

#### `GET /api/v1/dwelling-units`

**認可**: ADMIN / DEPUTY_ADMIN（MANAGE_RESIDENTS）

スコープ別の居室一覧をオフセットベースのページネーションで取得する。

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `scope_type` | String | Yes | `TEAM` / `ORGANIZATION` |
| `scope_id` | Long | Yes | 対象スコープの ID |
| `floor` | Integer | No | 階数フィルタ |
| `unit_type` | String | No | 居室種別フィルタ |
| `has_residents` | Boolean | No | `true` = 入居中の居室のみ / `false` = 空室のみ |
| `keyword` | String | No | 部屋番号の部分一致検索 |
| `page` | Integer | No | ページ番号（デフォルト: 0） |
| `size` | Integer | No | 取得件数（デフォルト: 50、最大: 100） |
| `sort` | String | No | ソート（`unit_number` / `floor`。デフォルト: `unit_number`） |

```json
{
  "data": [
    {

      "unit_number": "101",

        "resident_name": "山田太郎",

  ],
  "meta": {
    "page": 0,
    "size": 50,
    "total_elements": 120,
    "total_pages": 3
  }
}
```

- `primary_resident`: 主居住者のサマリー。居住者がいない場合は `null`

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | ADMIN / DEPUTY_ADMIN（MANAGE_RESIDENTS）ではない |

## 5. ビジネスロジック

### 居室・居住者の管理フロー

```
1. ADMIN が居室マスターを一括 or 個別登録
   a. 部屋番号・階数・面積・間取り・種別を入力
   b. CSV 一括インポート（POST /dwelling-units/import）で初期導入をサポート（最大500件）

2. ADMIN が各居室に居住者を登録
   a. 居住者種別（OWNER / RENTER）を選択
   b. プラットフォーム登録済みユーザーなら user_id を紐付け
   c. 未登録の高齢者等は氏名・連絡先のみ手動入力
   d. 共有持分比率を設定（区分所有者の場合）
   e. is_public を設定（本人への公開可否）

3. 居住者の退去処理
   a. ADMIN が PATCH /move-out で退去日を記録
   b. resident_count を自動更新
   c. 主居住者が退去した場合は次の居住者に自動移譲

4. 居住者の確認（verify）
   a. ADMIN が本人確認書類等を確認後、PATCH /verify で確認印を付与
   b. verified_at / verified_by が記録される
   c. UI 上で「確認済み」バッジが表示される
```

### 物件掲示板フロー

```
1. 現入居者が POST /property-listings で掲示を投稿
   a. 1居室につき ACTIVE な掲示は1件のみ
   b. 掲示期限（expires_at）を任意で設定
   c. 写真を最大5枚添付可能（S3 にアップロード、photo_keys に格納）

2. スコープ内メンバーが掲示を閲覧
   a. 閲覧時に view_count をアトミック更新
   b. 連絡方法に応じてコンタクト（IN_APP / EMAIL / PHONE）

3. 成約・取り下げ
   a. PATCH /close で成約処理（ACTIVE → CLOSED）
   b. PATCH /withdraw で取り下げ（ACTIVE → WITHDRAWN）
   c. CLOSED / WITHDRAWN は復帰不可（再投稿で対応）

4. 期限切れ自動取り下げ
   a. @Scheduled バッチ（1日1回）で expires_at 到達の ACTIVE 掲示を WITHDRAWN に遷移
```

### 賃貸借契約満了通知バッチ

```
1. @Scheduled で日次実行（毎朝 9:00）
2. 対象: resident_registry の move_out_date IS NULL AND lease_end_date IS NOT NULL
3. 通知タイミング:
   a. lease_end_date まで残り 90 日以内: 初回通知
   b. lease_end_date まで残り 30 日以内: リマインド通知
4. 通知先: スコープの ADMIN / DEPUTY_ADMIN（MANAGE_RESIDENTS）
5. 通知内容: 「賃貸借契約が○○日後に満了します: {unit_number} {resident_name}（満了日: {lease_end_date}）」
6. 重複防止: `resident_registry.last_lease_notified_days` で制御
   a. 残日数が 90 日以内かつ last_lease_notified_days IS NULL → 通知送信、last_lease_notified_days = 90 に更新
   b. 残日数が 30 日以内かつ last_lease_notified_days = 90 → 通知送信、last_lease_notified_days = 30 に更新
   c. last_lease_notified_days = 30 → 送信済みのためスキップ
   d. lease_end_date が更新（契約更新）された場合: last_lease_notified_days を NULL にリセット（Service 層で PUT /residents/{id} 時に実装）
```

### プライバシー管理

```
1. 台帳のデフォルトは非公開（is_public = FALSE）
   - ADMIN / DEPUTY_ADMIN のみが全居住者情報を閲覧可能
   - 個人は自室・自身の情報すら閲覧不可（明示的に is_public = TRUE にするまで）

2. ADMIN が is_public = TRUE に設定した居住者のみ:
   - 本人が GET /dwelling-units/my で自分の情報を閲覧可能
   - 他居住者の情報は見えない（本人の情報のみ）

3. 物件掲示板は is_public とは独立
   - スコープ内メンバー全員が閲覧可能（SUPPORTER 含む）
   - 投稿はメンバー以上かつ現入居者のみ
```

### 重要な判定ロジック

- **居室削除の制約**: `resident_count > 0` の居室は論理削除不可。先に全居住者の退去処理が必要
- **共有持分の整合性**: 同一居室の `OWNER` の `ownership_share` 合計が 100.00 を超えないことを検証
- **主居住者の唯一性**: 1居室あたり `is_primary = TRUE` は最大1名。自動判定: 最初の居住者は自動 TRUE、2人目以降はデフォルト FALSE。明示的に TRUE を指定した場合は既存の主居住者を FALSE に自動変更（排他制御）
- **物件掲示の排他**: 1居室あたり `ACTIVE` な `property_listings` は最大1件
- **退去時の連鎖処理**: 主居住者退去 → `is_primary` 自動移譲 + 掲示の自動取り下げ
- **議決権行使（F08.3）との関係**: `eligible_count`（議決権総数）と居住者台帳は**独立管理**。`eligible_count` = スコープ内 MEMBER 以上のユーザー数（プラットフォーム上の権限）、居住者台帳 = 物理的な居住実態（不動産の権利関係）であり、概念が異なる。台帳未登録の区分所有者、台帳登録済みだが MEMBER 未登録の高齢者、MEMBER だが区分所有者でない管理会社スタッフ等のケースで不一致が生じるため、統合すると柔軟性が失われる
- **CSV インポート**: 居室マスター（`dwelling_units`）のみ対象。居住者情報は個人情報のため CSV インポート対象外（ADMIN が手動登録）
- **ユーザー紐付け**: `resident_registry.user_id` は ADMIN の手動紐付けのみ。メールアドレス一致での自動紐付けは行わない（共有メール・typo による誤紐付け → 個人情報漏洩リスク）
- **物件掲示の写真**: `photo_keys` JSON で最大5枚。別テーブルではなく JSON カラムを採用（メタデータの個別管理不要・最大5枚と少量・掲示の CRUD と一体管理）

## 6. セキュリティ考慮事項

- **認可チェック**: 全 API でスコープメンバーシップとロールを検証。台帳の CRUD は ADMIN / DEPUTY_ADMIN（MANAGE_RESIDENTS）に限定
- **個人情報保護**: `resident_phone` は AES-256 で暗号化して保存。API レスポンスでは ADMIN と本人にのみ返却
- **プライバシーバイデフォルト**: `is_public = FALSE` がデフォルト。居住者情報は明示的に公開設定しない限り、本人にも非公開
- **IDOR 防止**: `GET /dwelling-units/my` は `user_id = currentUser` に限定。他人の居住者情報を ID 推測でアクセスできない
- **写真の安全性**: 物件掲示の写真は Content-Type + マジックバイトの二重チェック。S3 に保存し CloudFront 経由の Pre-signed URL（有効期限1時間）で配信
- **CSV インポートのバリデーション**: ファイルサイズ上限 1MB。500行超過は拒否。各行の unit_number にスクリプトインジェクション文字列が含まれていないことを検証
- **レートリミット**: `Bucket4j` で以下の制限を適用
  - 居室・居住者 CRUD: 1分間20件
  - CSV インポート: 1分間1件
  - 物件掲示投稿: 1分間5件
  - 統計 API: 1分間10件
- **監査ログ**: 以下のイベントを記録
  - `DWELLING_UNIT_CREATED`: 居室作成
  - `DWELLING_UNIT_DELETED`: 居室削除（論理削除）
  - `DWELLING_UNITS_IMPORTED`: CSV 一括インポート（件数を含む）
  - `RESIDENT_ADDED`: 居住者追加
  - `RESIDENT_UPDATED`: 居住者情報更新（user_id 紐付け変更を含む）
  - `RESIDENT_MOVED_OUT`: 退去処理
  - `RESIDENT_VERIFIED`: 居住者確認
  - `LISTING_POSTED`: 物件掲示投稿
  - `LISTING_CLOSED`: 成約処理

- [x] ~~賃貸借契約満了通知~~ → **導入する**。日次バッチ（毎朝9時）で `lease_end_date` の90日前・30日前に ADMIN に通知。既存の通知基盤（F04.3）を利用。契約更新の見逃しは管理上の重大リスクであり、低コストで高効果
- [x] ~~議決権行使（F08.3）との統合深度~~ → **独立管理のまま。統合しない**。`eligible_count`（プラットフォーム上の MEMBER 数）と居住者台帳（物理的居住実態）は異なる概念。台帳未登録の区分所有者、MEMBER 未登録の高齢者、MEMBER だが非所有者の管理会社スタッフ等で不一致が生じる。統合すると柔軟性が失われるため、ADMIN がセッション作成時に手動確認する現行設計を維持
- [x] ~~物件掲示の写真添付~~ → **Phase 9 スコープに含める**。`property_listings.photo_keys` JSON カラムで S3 キー配列を保持（最大5枚）。別テーブルではなく JSON を採用（メタデータ個別管理不要・少量・掲示の CRUD と一体管理）。S3 + CloudFront Pre-signed URL（有効期限1時間）で配信。POST/PUT を multipart/form-data に変更
