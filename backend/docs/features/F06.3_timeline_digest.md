# F06.3: タイムラインダイジェスト（自動要約ブログ生成）

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 6
> **最終更新**: 2026-03-18
> **関連ドキュメント**: F04.1 タイムライン（投稿データソース）、F06.1 CMS・ブログ（ブログ記事出力先）、F04.3 プッシュ通知（生成完了通知）
> **モジュール種別**: 選択式モジュール（チーム/組織レベル）

---

## 1. 概要

タイムラインに投稿されたコンテンツを AI が要約し、ブログ記事の下書きとして自動生成する機能。日次ダイジェスト（「今日のチームまとめ」）や任意期間の振り返り記事をワンクリックで作成できる。

**ユースケース**:
- **日次ダイジェスト**: 「今日チームで何があったか」をブログ記事として蓄積。新メンバーのキャッチアップや活動記録に
- **週次/月次レポート**: 期間を指定してプロジェクトの振り返り記事を自動生成
- **イベント振り返り**: イベント前後の投稿をまとめて活動報告記事に
- **組織広報**: 組織タイムラインの投稿を要約し、外部向けブログとして公開

**設計方針**:
- **AI 生成は下書きのみ**: 自動公開は行わない。必ず人間がレビュー・編集してから公開する
- **コスト制御**: AI API 呼び出しにレートリミット + 月次上限を設定
- **プライバシー**: 要約対象はスコープ内の投稿のみ。通報対象・論理削除済み・返信（reply）は除外
- **スコープ一致**: チームのタイムライン → チームのブログ、組織 → 組織のブログのみ生成可能
- **AI 不使用オプション**: `TEMPLATE` スタイルで統計ベースのダイジェストを生成可能（AI API コストゼロ。エンゲージメント上位投稿の引用 + 統計情報をテンプレートに流し込み）
- **課金モデル**: TEMPLATE スタイルは無料（AI 不使用）。AI スタイル（SUMMARY / NARRATIVE / HIGHLIGHTS）はモジュール月額に含む（月30回まで）。超過時は TEMPLATE のみ利用可能（従量課金は発生しない）
- **段階的リリース**: AI スタイルは `feature_flags`（F10.1）の `FEATURE_DIGEST_AI` フラグで制御。初期リリース時は `false`（TEMPLATE のみ利用可能）。運営の収益が安定した段階で SYSTEM_ADMIN がダッシュボードから `true` に切替えて AI 機能を解放

> **AI API**: Claude API（`claude-haiku-4-5` をデフォルト。コスト効率重視）。AI 設定は `application.yml`（`mannschaft.digest.*`）で管理し、API キーは環境変数 or AWS Secrets Manager で保持。将来的に OpenAI 等への切替も可能な抽象化設計（`DigestAiProvider` インターフェース）。
>
> ```yaml
> mannschaft.digest:
>   ai-provider: claude             # claude / openai（将来拡張）
>   ai-model: claude-haiku-4-5
>   ai-temperature: 0.3             # 低めで事実重視
>   ai-max-tokens: 2000
>   monthly-limit-per-scope: 30     # スコープあたり月次生成上限
>   cost-alert-threshold-jpy: 1000   # 月次 AI コストがこの閾値を超えたら SYSTEM_ADMIN にアラート
>   defaults:                        # config 未作成スコープでの手動生成時のフォールバック
>     min-posts-threshold: 3
>     max-posts-per-digest: 100
>     content-max-chars: 500
> ```

|--------|------|

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `timeline_digest_configs` | ダイジェスト自動生成の設定（スケジュール・スタイル・対象条件） | あり |
| `timeline_digests` | 生成されたダイジェストの履歴・ステータス管理 | なし |

### テーブル定義

#### `timeline_digest_configs`

ダイジェスト自動生成の設定。チーム/組織ごとに1つの有効な設定を持てる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | `TEAM` / `ORGANIZATION` |
| `scope_id` | BIGINT UNSIGNED | NO | — | FK → teams / organizations |
| `schedule_type` | VARCHAR(20) | NO | 'MANUAL' | `MANUAL`（手動のみ）/ `DAILY`（毎日）/ `WEEKLY`（毎週指定曜日）/ `MONTHLY`（毎月1日） |
| `schedule_time` | TIME | YES | '09:00:00' | 自動生成の実行時刻（JST） |
| `schedule_day_of_week` | TINYINT | YES | 1 | WEEKLY 時の曜日（0=日曜〜6=土曜。デフォルト: 1=月曜。WEEKLY 以外では無視） |
| `last_executed_at` | DATETIME | YES | NULL | 最後に自動生成を実行した日時（重複実行防止用。バッチ開始時に更新） |
| `digest_style` | VARCHAR(30) | NO | 'SUMMARY' | `SUMMARY`（箇条書き要約）/ `NARRATIVE`（読み物風記事）/ `HIGHLIGHTS`（ハイライトピックアップ）/ `TEMPLATE`（AI 不使用。統計ベーステンプレート） |
| `auto_publish` | BOOLEAN | NO | FALSE | 自動スケジュール生成時に GENERATED → 即ブログ DRAFT 作成まで自動実行するか（手動 publish 不要になる。F06.1 の公開フローは別途必要） |
| `style_presets` | JSON | YES | NULL | 名前付きスタイルプリセット（例: `[{"name":"カジュアル週報","style":"NARRATIVE","prompt":"フランクなトーンで"}]`） |
| `include_reactions` | BOOLEAN | NO | TRUE | リアクション数の多い投稿をハイライトするか |
| `include_polls` | BOOLEAN | NO | TRUE | 投票結果を要約に含めるか |
| `include_diff_from_previous` | BOOLEAN | NO | FALSE | 前回ダイジェストとの差分ハイライトを含めるか（AI スタイルのみ有効） |
| `min_posts_threshold` | INTEGER | NO | 3 | 対象期間の投稿がこの件数未満の場合は生成をスキップ |
| `max_posts_per_digest` | INTEGER | NO | 100 | 1回のダイジェストで要約対象にする最大投稿数（コスト制御） |
| `timezone` | VARCHAR(50) | NO | 'Asia/Tokyo' | スケジュール実行・期間計算のタイムゾーン（IANA 形式） |
| `content_max_chars` | INTEGER | NO | 500 | AI に送信する各投稿 content の最大文字数（コストと精度のトレードオフ） |
| `language` | VARCHAR(10) | NO | 'ja' | 生成言語（`ja` / `en`） |
| `custom_prompt_suffix` | TEXT | YES | NULL | AI プロンプトに追加するカスタム指示（最大500文字。例: 「技術用語は避けてください」） |
| `auto_tag_ids` | JSON | YES | NULL | 生成されたブログ記事に自動付与するタグ ID 配列（例: `[1, 5]`） |
| `is_enabled` | BOOLEAN | NO | TRUE | 有効/無効 |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
-- NOTE: スコープごとに有効な設定は1つの制約はアプリ層で保証
-- （MySQL の UNIQUE KEY は deleted_at=NULL の複数行を許容するため DB 制約では不十分）
INDEX idx_tdc_scope (scope_type, scope_id, deleted_at)       -- スコープ検索用
INDEX idx_tdc_schedule (is_enabled, schedule_type, schedule_time)  -- バッチ対象の検索
```

**制約・備考**
- スコープごとに有効な設定は1つのみ。MySQL の UNIQUE KEY は `deleted_at = NULL` の複数行を区別できないため、**アプリ層で `WHERE deleted_at IS NULL AND scope_type = ? AND scope_id = ?` の存在チェック**を行い、既存の有効設定がある場合は 409 を返す
- `schedule_type = MANUAL` の場合、`schedule_time` / `schedule_day_of_week` / `last_executed_at` は無視（手動トリガーのみ）
- `schedule_day_of_week`: `WEEKLY` の場合のみ有効（0=日曜〜6=土曜）。`DAILY` / `MONTHLY` では無視
- `last_executed_at`: バッチが実行対象を判定する際に「`last_executed_at` の日付 < 今日の日付」で未実行を確認。生成開始前に即座に更新して重複実行を防止
- `custom_prompt_suffix` は ADMIN が自由記述可能だが、AI に送信される前にサニタイズ（プロンプトインジェクション対策として禁止ワードフィルタを適用）
- `max_posts_per_digest` 超過時は `reaction_count + reply_count` の降順で上位を選択（エンゲージメントの高い投稿を優先）
- `timezone`: 自動スケジュールの期間計算（「前日 00:00:00」等）はこのタイムゾーンで実行。サーバーの TZ 設定に依存しない。手動生成の `period_start` / `period_end` は ISO 8601 でタイムゾーン付きを受け付ける（例: `2026-03-17T00:00:00+09:00`）
- `content_max_chars`: 各投稿の `content` をこの文字数で切り詰めて AI に送信。デフォルト500。技術チーム（長文議論が多い）は1000、カジュアルなチーム（短文中心）は300等にチューニング可能。`GET /system-admin/timeline-digest/usage` の `ai_input_tokens` でコスト影響を監視
- `auto_tag_ids`: publish 時に F06.1 の `blog_tags` テーブルで `deleted_at IS NULL` の有効なタグのみフィルタ。削除済みタグ ID は無視する（エラーにしない）

#### `timeline_digests`

生成されたダイジェストの履歴。AI の生成結果とブログ記事への紐付けを管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `config_id` | BIGINT UNSIGNED | YES | NULL | FK → timeline_digest_configs（手動生成の場合は NULL） |
| `scope_type` | VARCHAR(20) | NO | — | `TEAM` / `ORGANIZATION` |
| `scope_id` | BIGINT UNSIGNED | NO | — | FK → teams / organizations |
| `period_start` | DATETIME | NO | — | 要約対象の開始日時 |
| `period_end` | DATETIME | NO | — | 要約対象の終了日時 |
| `post_count` | INTEGER | YES | NULL | 要約対象に含まれた投稿数（GENERATING 時は NULL。全フィルタ適用後の実数をステップ4a完了時に設定） |
| `digest_style` | VARCHAR(30) | NO | — | 生成時のスタイル |
| `generated_title` | VARCHAR(200) | YES | NULL | AI が生成したタイトル（GENERATING / FAILED 時は NULL） |
| `generated_body` | TEXT | YES | NULL | AI が生成した本文（Markdown。GENERATING / FAILED 時は NULL） |
| `generated_excerpt` | VARCHAR(500) | YES | NULL | AI が生成した抜粋（GENERATING / FAILED 時は NULL） |
| `ai_model` | VARCHAR(50) | YES | NULL | 使用した AI モデル（例: `claude-haiku-4-5`。GENERATING 時は NULL、生成完了時に設定） |
| `ai_input_tokens` | INTEGER | YES | NULL | AI API 入力トークン数（生成完了時に設定） |
| `ai_output_tokens` | INTEGER | YES | NULL | AI API 出力トークン数（生成完了時に設定） |
| `status` | VARCHAR(20) | NO | 'GENERATING' | `GENERATING`（AI 処理中。INSERT 時の初期値）/ `GENERATED`（生成完了・未公開）/ `PUBLISHED`（ブログ記事として公開済み）/ `DISCARDED`（破棄）/ `FAILED`（生成失敗） |
| `blog_post_id` | BIGINT UNSIGNED | YES | NULL | FK → blog_posts（ブログ記事として公開した場合） |
| `generating_timeout_at` | DATETIME | YES | NULL | GENERATING ステータスのタイムアウト期限（= `created_at + 5分`。超過時にバッチで FAILED に遷移） |
| `source_post_ids` | JSON | YES | NULL | 要約に含めた投稿 ID のリスト（例: `[101,102,105]`。publish 時の stale 検知用） |
| `error_message` | TEXT | YES | NULL | FAILED 時のエラー詳細 |
| `triggered_by` | BIGINT UNSIGNED | YES | NULL | FK → users（手動トリガーの場合。自動の場合は NULL） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_td_scope (scope_type, scope_id, created_at DESC)  -- スコープ別のダイジェスト履歴
INDEX idx_td_status (status)                                  -- ステータス別検索
INDEX idx_td_period (scope_type, scope_id, period_start, period_end)  -- 期間重複チェック
```

**制約・備考**
- **重複防止**: 同一スコープ・同一期間（`period_start` AND `period_end` が完全一致）のダイジェストが `GENERATED` / `PUBLISHED` で存在する場合は生成をスキップ。期間が部分的に重複するダイジェスト（例: 日次と週次）は異なる目的のため共存可能
- **GENERATING タイムアウト**: `generating_timeout_at`（= `created_at + 5分`）を超過した `GENERATING` レコードは、1時間ごとの自動スケジュールバッチの冒頭で `FAILED`（`error_message = 'Generation timed out'`）に自動遷移。ADMIN に `DIGEST_FAILED` 通知を送信。最大検知遅延は1時間
- `GENERATED` 状態のダイジェストは30日後に自動 `DISCARDED`（未使用の下書きを自動クリーンアップ）
- `ai_input_tokens` / `ai_output_tokens`: コスト追跡用。月次集計で利用量を監視
- `blog_post_id`: ブログ記事として公開した場合に紐付け。ブログ側で論理削除された場合は `ApplicationEvent` で `status = GENERATED` に差し戻し + `blog_post_id = NULL` にクリア（再 publish 可能にする）

### リクエスト／レスポンス仕様

#### `POST /api/v1/timeline-digest/generate`

**認可**: ADMIN / DEPUTY_ADMIN（`WRITE_BLOG` 権限）

指定した期間のタイムライン投稿を AI で要約し、ダイジェストを生成する。非同期処理。

**リクエストボディ**
```json
{
  "scope_type": "TEAM",
  "scope_id": 5,
  "period_start": "2026-03-17T00:00:00+09:00",
  "period_end": "2026-03-17T23:59:59+09:00",
  "digest_style": "NARRATIVE",
  "custom_prompt_suffix": "カジュアルなトーンでお願いします"
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `scope_type` | String | Yes | `TEAM` / `ORGANIZATION` |
| `scope_id` | Long | Yes | スコープ ID |
| `period_start` | DateTime | Yes | 要約対象の開始日時 |
| `period_end` | DateTime | Yes | 要約対象の終了日時 |
| `digest_style` | String | No | `SUMMARY` / `NARRATIVE` / `HIGHLIGHTS` / `TEMPLATE`（デフォルト: `SUMMARY`） |
| `custom_prompt_suffix` | String | No | AI への追加指示（最大500文字。`TEMPLATE` 時は無視） |
| `preset_name` | String | No | `style_presets` に保存済みのプリセット名を指定。指定時は `digest_style` / `custom_prompt_suffix` をプリセット値で上書き |

**レスポンス（202 Accepted）**
```json
{
  "data": {
    "id": 1,
    "status": "GENERATING",
    "estimated_post_count": 15
  },
  "ai_quota": { "enabled": true, "used": 11, "limit": 30, "remaining": 19 }
}
```

**補足**
- `ai_quota`: AI スタイルの場合、今回の生成で消費した後の残量を返す。TEMPLATE の場合は `ai_quota` フィールドを省略
- AI スタイル（SUMMARY / NARRATIVE / HIGHLIGHTS）: 非同期処理。`GENERATING` 状態で即座にレスポンス（202）し、AI 生成完了後に `GENERATED` に遷移
- `TEMPLATE` スタイル: 同期処理。統計集計 + テンプレート適用のみで AI 不使用。即座に `GENERATED` 状態で返却（201）
- 生成完了時にプッシュ通知（「ダイジェストの生成が完了しました」）
- `period_end - period_start` は最大31日（月次上限）
- 同一スコープ・同一期間のダイジェストが既に GENERATED / PUBLISHED の場合は 409

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | 期間不正（start > end / 31日超過）/ custom_prompt_suffix が500文字超過 |
| 403 | 権限不足 / スコープへのアクセス権なし / F06.3 モジュールが無効 / AI スタイル指定時に `FEATURE_DIGEST_AI` が無効 |
| 409 | 同一期間のダイジェストが既に存在 / AI スタイルの月次生成上限に到達（TEMPLATE は対象外）/ 同一スコープで GENERATING 中のダイジェストが存在 |
| 422 | 対象期間の投稿が `min_posts_threshold` 未満 |

#### `POST /api/v1/timeline-digest/{id}/publish`

**認可**: ADMIN

ダイジェストをブログ記事の下書きとして保存する。必要に応じてタイトル・本文を編集してから公開。

**リクエストボディ**
```json
{
  "title": "2026年3月17日のチームダイジェスト",
  "body": "（AI 生成の本文を編集したもの）",
  "tag_ids": [1, 5],
  "visibility": "MEMBERS_ONLY"
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `title` | String | No | タイトル（未指定時は AI 生成タイトルを使用） |
| `body` | String | No | 本文（未指定時は AI 生成本文を使用） |
| `tag_ids` | Long[] | No | タグ ID 配列（未指定時は `auto_tag_ids` を使用） |
| `visibility` | String | No | `MEMBERS_ONLY` / `SUPPORTERS_AND_ABOVE` / `PUBLIC`（デフォルト: `MEMBERS_ONLY`） |

**レスポンス（201 Created）**
```json
{
  "data": {
    "digest_id": 1,
    "blog_post_id": 42,
    "blog_post_slug": "team-digest-2026-03-17-42",
    "status": "DRAFT"
  },
  "stale_warning": {
    "edited_since_generation": 2,
    "deleted_since_generation": 1
  }
}
```

**補足**
- `stale_warning`: publish 時に `source_post_ids` の投稿を検査し、ダイジェスト生成後（`timeline_digests.created_at`）に編集・削除された件数を返す。publish 自体はブロックしない（ダイジェストは生成時点のスナップショットとして運用）。`edited + deleted = 0` の場合は `stale_warning` フィールド自体を省略
- F06.1 の `POST /api/v1/blog/posts` 相当の処理を内部で実行。`status = DRAFT` で作成
- `blog_posts.post_type = BLOG`、`blog_posts.excerpt` = AI 生成の `generated_excerpt`、`blog_posts.is_ai_generated = TRUE`
- `timeline_digests.status = PUBLISHED`、`blog_post_id` を紐付け
- ブログ記事の本文末尾に自動付記: 「*この記事は {period_start} 〜 {period_end} のタイムライン投稿を基に AI が生成し、管理者が編集・確認したものです。*」
- `is_ai_generated = TRUE` により、フロントエンドで記事ヘッダーに「AI 生成」バッジを表示（本文の付記は ADMIN が編集可能だが、メタデータバッジは改ざん不可）
- `blog_posts.slug` は `{scope}-digest-{date}-{digest_id}`（例: `team-digest-2026-03-17-42`）で自動生成。F06.1 の slug ユニーク制約に違反した場合はサフィックスに digest_id を付与してリトライ
- 監査ログ: `DIGEST_PUBLISHED`

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | ADMIN でない |
| 404 | ダイジェストが存在しない |
| 409 | status が GENERATED でない（既に公開済み / 破棄済み） |

## 5. ビジネスロジック

### 主要フロー

#### 手動ダイジェスト生成フロー
```
1. ADMIN が POST /timeline-digest/generate で生成をリクエスト
   - スコープに timeline_digest_configs が存在する場合: config の設定値を使用
   - config が存在しない場合: application.yml の mannschaft.digest.defaults をフォールバック
2. バックエンドが以下をバリデーション:
   a. AI スタイルの場合: `FeatureFlagService.isEnabled("FEATURE_DIGEST_AI")` が `true` であること
      - false の場合: 403 を返す。エラーメッセージ: 「AI ダイジェスト機能は現在利用できません。TEMPLATE スタイルをご利用ください。」
   b. AI スタイルの場合: 月次 AI 生成上限（digest_monthly_limit）に達していないか（カウント対象: AI スタイルの GENERATED + PUBLISHED + GENERATING のみ。TEMPLATE は対象外）
      - 上限到達時: 409 を返す。エラーメッセージ: 「今月の AI 生成枠（{limit}回）を使い切りました。TEMPLATE スタイルは引き続き利用可能です。」
      - TEMPLATE スタイルの場合: 月次上限チェックをスキップ（AI コストゼロのため制限不要）
   c. 同一スコープで GENERATING 中のダイジェストが存在しないか（並行制御）
   d. 同一スコープ・同一期間のダイジェストが GENERATED / PUBLISHED で存在しないか
   e. 対象期間の投稿数 ≥ min_posts_threshold
3. timeline_digests に INSERT（status = GENERATING）
4. 非同期ジョブをキューに投入:
   a. timeline_posts から対象投稿を取得:
      - scope_type / scope_id が一致
      - created_at が period_start 〜 period_end 内
      - status = PUBLISHED（DRAFT / SCHEDULED は除外）
      - deleted_at IS NULL（論理削除済みは除外）
      - parent_id IS NULL（返信は除外。トップレベル投稿のみ）
      - exclude_from_digest = FALSE（オプトアウトされていない投稿のみ）
      - 通報対象（content_reports.status = REVIEWING / RESOLVED）の投稿は除外
      - 単純リポスト（repost_of_id IS NOT NULL かつ content が空）は除外
      - 引用リポスト（repost_of_id IS NOT NULL かつ content あり）は引用者の content のみ対象（元投稿の内容は送信しない。スコープ外コンテンツの漏洩防止）
   b. max_posts_per_digest 超過時: reaction_count + reply_count 降順で上位を選択
   c. 投稿データを AI プロンプトに整形:
      - 各投稿: content（先頭 content_max_chars 文字。デフォルト500）、attachment_count、reaction_count、reply_count、posted_by（display_name）
      - 添付のみ投稿（content が空 or 20文字以下かつ attachment_count > 0）: AI には個別内容を送らず、
        「他に {media_only_count}件の画像/動画投稿がありました」としてカウント情報のみ渡す
      - include_reactions = true: リアクション上位3投稿をハイライト指示
      - include_polls = true: 投票結果のサマリを含める
   d. AI API を呼び出し（Claude API）:
      - システムプロンプト: スタイル別テンプレート（SUMMARY / NARRATIVE / HIGHLIGHTS）
      - ユーザープロンプト: 整形した投稿データ + custom_prompt_suffix
      - 出力: タイトル + 本文（Markdown）+ 抜粋（120文字以内）
   e. timeline_digests を更新（status = GENERATED、生成結果 + source_post_ids を保存）
5. ADMIN にプッシュ通知（「ダイジェストの生成が完了しました」）
6. ADMIN が GET /timeline-digest/{id} で結果を確認・プレビュー
7. ADMIN が POST /timeline-digest/{id}/publish でブログ下書きとして保存
   → F06.1 の blog_posts に DRAFT 状態で INSERT
   → ADMIN が F06.1 のフローで編集・レビュー・公開
```

#### 自動ダイジェスト生成フロー（スケジュール）
```
1. Spring @Scheduled バッチが1時間ごとに実行
2. timeline_digest_configs から以下の条件に合致する設定を取得:
   a. is_enabled = true
   b. deleted_at IS NULL
   c. schedule_type ≠ MANUAL
   d. スコープの F06.3 モジュールが有効（`module_selections` テーブルで確認。無効ならスキップ）
   e. last_executed_at の日付 < 今日の日付（未実行）
   f. 現在時刻が schedule_time の実行タイミングに該当:
      - DAILY: 毎日 schedule_time
      - WEEKLY: 毎週 schedule_day_of_week 曜日の schedule_time
      - MONTHLY: 毎月1日 schedule_time
3. 各設定について:
   a. 対象期間を自動計算:
      - DAILY: 前日 00:00:00 〜 23:59:59
      - WEEKLY: 前週の schedule_day_of_week 曜日 00:00:00 〜 その6日後 23:59:59
      - MONTHLY: 前月1日 00:00:00 〜 末日 23:59:59
   b. last_executed_at = NOW() を即座に更新（重複実行防止）
   c. 同一期間のダイジェストが既に存在する場合はスキップ
   d. 対象投稿数 < min_posts_threshold の場合はスキップ（通知なし）
   e. 手動生成フローの手順4と同じ AI 生成処理を実行
4. 生成完了後: スコープの**現在の ADMIN 全員**にプッシュ通知（`created_by` 個人ではなくロールベース。ADMIN 異動に影響されない）
   「{scope_name} の{period}ダイジェストが生成されました。確認してブログに公開できます。」
5. 30日間 GENERATED 状態のまま放置されたダイジェストは日次バッチで DISCARDED に自動遷移
```

#### TEMPLATE スタイル生成フロー（AI 不使用）
```
1. digest_style = TEMPLATE の場合、AI API を呼び出さずに以下のデータを集計:
   a. 投稿件数（total_posts）
   b. ユニーク投稿者数（unique_authors）
   c. リアクション総数（total_reactions）
   d. 返信総数（total_replies）
   e. エンゲージメント上位3投稿（reaction_count + reply_count 降順。content の先頭100文字 + 投稿者名）
   f. 投票結果サマリ（include_polls = true の場合）
   g. 添付投稿件数（media_only_count）
2. 以下の Mustache テンプレートに流し込み:
   title: 「{scope_name} ダイジェスト（{period_start_date} 〜 {period_end_date}）」
   body:
     ## 期間の概要
     - 投稿数: {total_posts}件（{unique_authors}名が参加）
     - リアクション: {total_reactions}件 / 返信: {total_replies}件

     ## 注目の投稿
     {#top_posts}
     ### {rank}. {author_name} さんの投稿（👍{reaction_count} 💬{reply_count}）
     > {content_preview}
     {/top_posts}

     {#poll_summary}
     ## 投票結果
     {poll_details}
     {/poll_summary}
   excerpt: 「{total_posts}件の投稿から注目トピックをピックアップ」
3. 生成は同期処理（AI 不使用のため即座に完了）
   - レスポンスは 201 Created（202 ではない）で GENERATED 状態のダイジェストを返す
   - 月次上限カウントには含まれるが、AI トークンコストは 0
   - ai_model / ai_input_tokens / ai_output_tokens は NULL のまま
```

#### 差分ハイライト（前回ダイジェストとの比較）
```
1. include_diff_from_previous = true かつ AI スタイル（SUMMARY / NARRATIVE / HIGHLIGHTS）の場合:
   a. 同一スコープ・同一 schedule_type の直前の PUBLISHED ダイジェストを取得
   b. 直前ダイジェストの generated_body をシステムプロンプトに追加:
      「前回の要約: {previous_body}。今回の要約では、前回から新しく登場したトピックに★を付け、
       前回から継続しているトピックにはその旨を記載してください。」
   c. TEMPLATE スタイルでは無効（AI 不使用のため差分判定不可）
2. 直前ダイジェストが存在しない場合（初回生成）: 差分なしで通常生成
```

#### 自動 publish フロー（auto_publish = true）
```
1. 自動スケジュール生成フローの手順 e 完了後:
   a. config.auto_publish = true の場合:
      - 手動生成フローの手順7と同じ publish 処理を自動実行
      - blog_posts に DRAFT 状態で INSERT（自動公開はしない。F06.1 の公開フローは別途必要）
      - timeline_digests.status = PUBLISHED に遷移
   b. auto_publish = false の場合: 通常通り GENERATED で停止。ADMIN に通知
2. auto_publish 時の通知文面:
   「{scope_name} のダイジェストが自動生成され、ブログ下書きとして保存されました。確認して公開してください。」
```

#### WebSocket リアルタイムステータス通知
```
1. timeline_digests のステータス遷移時に WebSocket（STOMP）でイベントを配信:
   - トピック: /topic/digest/{scope_type}/{scope_id}
   - ペイロード: { "digest_id": 1, "status": "GENERATED", "generated_title": "..." }
2. フロントエンドが subscribe → GENERATING 中はスピナー表示
   → GENERATED 受信で自動的にプレビュー画面に遷移
   → FAILED 受信でエラーメッセージ表示
3. 購読権限: ADMIN / DEPUTY_ADMIN（WRITE_BLOG 権限）のみ
```

#### AI レスポンスキャッシュ
```
1. AI API 呼び出し前に、入力データ（投稿一覧 + スタイル + custom_prompt_suffix）の SHA-256 ハッシュを計算
2. Redis キャッシュキー: mannschaft:digest:ai_cache:{hash}
3. キャッシュヒット（TTL: 1時間）: AI API を呼び出さず、キャッシュされた結果を使用
   - regenerate で同じスタイル + 同じ期間を指定した場合に有効（コスト削減 + 高速化）
   - ai_input_tokens / ai_output_tokens は 0 を記録（キャッシュヒットの識別用）
4. キャッシュミス: 通常通り AI API を呼び出し、結果をキャッシュに保存
5. custom_prompt_suffix が異なればハッシュが異なるため、キャッシュミスになる（意図通り）
```

#### GENERATING タイムアウト検知（1時間バッチに同居）
```
1. 自動スケジュールバッチ（1時間ごと実行）の冒頭で、タイムアウト検知を実行:
   UPDATE timeline_digests SET status = 'FAILED', error_message = 'Generation timed out'
   WHERE status = 'GENERATING' AND generating_timeout_at < NOW()
2. 対象スコープの現在の ADMIN 全員に DIGEST_FAILED 通知を送信（ユーザー単位で try-catch）
3. → その後、通常のスケジュール処理（手順2以降）を続行
```

#### ダイジェストクリーンアップバッチ
```
1. Spring @Scheduled バッチが日次で実行（例: 04:00 JST）
2. 未使用 GENERATED クリーンアップ:
   UPDATE timeline_digests SET status = 'DISCARDED'
   WHERE status = 'GENERATED' AND created_at < NOW() - INTERVAL 30 DAY
3. DISCARDED 物理削除（ストレージ圧迫防止）:
   DELETE FROM timeline_digests
   WHERE status = 'DISCARDED' AND created_at < NOW() - INTERVAL 90 DAY
   → generated_body（TEXT 型）の長期蓄積を防止
4. 更新件数・削除件数をアプリケーションログに記録
```

### 重要な判定ロジック

- **投稿のフィルタリング**: 論理削除済み・DRAFT/SCHEDULED・返信・通報対象（REVIEWING/RESOLVED）の投稿は要約対象外。プライバシーとコンテンツ品質を確保
- **コスト制御**: `digest_monthly_limit`（デフォルト: スコープあたり月30回。`application.yml` の `mannschaft.digest.monthly-limit-per-scope` で定義。SYSTEM_ADMIN が `PUT /system-admin/settings` で変更可能）で生成回数を制限。`max_posts_per_digest`（デフォルト: 100投稿）で1回あたりの入力量を制限。SYSTEM_ADMIN が `GET /system-admin/timeline-digest/usage` でコストを監視
- **重複防止**: 同一スコープ・同一期間（`period_start` AND `period_end` の**完全一致**）のダイジェストが `GENERATED` / `PUBLISHED` で存在する場合は生成不可（409）。期間の部分重複（例: 3/17 日次ダイジェスト + 3/15〜3/20 週次ダイジェスト）は異なる目的のため共存を許可。再生成は `POST /{id}/regenerate` で元を破棄してから
- **下書きのみ**: AI 生成結果は必ず `GENERATED` 状態。ブログへの公開は ADMIN が `POST /{id}/publish` で明示的に実行し、F06.1 のブログワークフロー（DRAFT → PUBLISHED）に乗せる
- **プロンプトインジェクション対策**: `custom_prompt_suffix` はサニタイズ（`system` / `assistant` / `ignore previous` 等のキーワードを除去）。AI 出力も HTML タグを除去してから保存
- **スコープ一致**: チームタイムライン → チームブログ、組織タイムライン → 組織ブログのみ。クロススコープ生成は不可
- **PUBLIC タイムラインの除外**: PUBLIC スコープの投稿は機密性が低いが、異なるチーム/組織のメンバーが混在するため要約対象外。チーム/組織スコープのみ対象
- **AI API 障害時**: `FAILED` 状態で記録し、`error_message` にエラー詳細を保存。ADMIN に通知し、手動で `POST /{id}/regenerate` で再試行可能
- **GENERATING タイムアウト**: `generating_timeout_at`（= `created_at + 5分`）を超過した `GENERATING` レコードは、1時間ごとの自動スケジュールバッチの冒頭で `FAILED` に自動遷移（最大検知遅延1時間）。AI API のレイテンシは通常数秒〜30秒だが、ネットワーク障害やキュー詰まりに5分の猶予を確保
- **投稿者オプトアウト**: `timeline_posts.exclude_from_digest = TRUE` の投稿は AI 要約対象外。投稿者がプライバシーを制御する手段を提供。デフォルトは `FALSE`（全投稿が対象）
- **リポストの扱い**: 単純リポスト（自身の content なし）は除外。引用リポスト（自身の content あり）は引用者の content のみ対象とし、元投稿の内容は AI に送信しない（スコープ外コンテンツの漏洩防止）
- **添付のみ投稿**: content が空 or 20文字以下かつ attachment_count > 0 の投稿は、個別内容ではなくカウント情報（「他に X 件の画像/動画投稿がありました」）として AI に渡す。画像/動画を AI に送信するとコスト増 + プライバシーリスクが高いため
- **AI 出力の構造化**: Claude API の tool_use（Structured Output）を使用し、`title` / `body` / `excerpt` を型安全に取得。自由形式出力のパース失敗リスクを排除
- **AI 生成コンテンツの明示**: ブログ記事に `is_ai_generated = TRUE` メタデータを設定。フロントエンドで「AI 生成」バッジを表示（本文の付記は ADMIN が編集可能だが、メタデータバッジは改ざん不可）。EU AI Act 等の AI コンテンツ表示義務にも対応
- **WEEKLY 曜日の柔軟性**: `schedule_day_of_week`（0=日曜〜6=土曜）で WEEKLY の実行曜日を設定可能。金曜（週の振り返り）や日曜（翌週の準備）など組織に合わせた運用が可能
- **段階的リリース（フィーチャーフラグ）**: AI スタイル（SUMMARY / NARRATIVE / HIGHLIGHTS）は `feature_flags` テーブル（F10.1）の `FEATURE_DIGEST_AI` フラグで制御。`false`（初期値）の場合: TEMPLATE スタイルのみ利用可能。AI スタイルを指定した generate / regenerate は 403。フロントエンドは `ai_quota.enabled = false` を見て AI 関連 UI を非表示に。`true` に切替え時: SYSTEM_ADMIN がダッシュボードからワンクリックで AI 機能を解放。Redis キャッシュ即時更新（F10.1 の既存仕組み）で全スコープに即座に反映。切替え後の差し戻し（`true` → `false`）も可能（GENERATING 中のジョブがあれば完了を待って停止）
- **課金モデル（TEMPLATE 無料 + AI 枠付き）**: モジュール月額サブスクリプションに AI 生成枠（月30回/スコープ）を含む。TEMPLATE スタイルは AI 不使用のため無制限。AI 枠超過時は TEMPLATE のみ利用可能で従量課金は発生しない。理由: (1) 既存の定額サブスクリプション課金モデルと一貫性を保つ。(2) Haiku の月30回コスト（約¥90〜150）はモジュール月額で十分吸収可能。(3) 従量課金の決済基盤を新規開発する必要がない。(4) TEMPLATE があるので枠切れでも「要約は作れる」状態を維持できる
- **月次 AI 上限のカウント対象**: AI スタイル（SUMMARY / NARRATIVE / HIGHLIGHTS）の `GENERATED` + `PUBLISHED` + `GENERATING` のみカウント。TEMPLATE スタイル・`FAILED`・`DISCARDED` は除外。理由: TEMPLATE は AI コストゼロなので制限不要。FAILED はユーザーが有効な結果を得られなかった試行のため不公平
- **同一スコープ並行生成の制限**: 同一スコープで `GENERATING` 状態のダイジェストが存在する場合、`POST /generate` は 409 で拒否。エラーメッセージ: 「前回の生成が処理中です。完了後に再度お試しください。」理由: 並行 AI API 呼び出しによるコスト急増を防止。生成は数秒〜30秒で完了するため UX への影響は軽微
- **ブログ記事削除時のステータス差し戻し**: F06.1 側でダイジェスト由来のブログ記事が論理削除された場合、`ApplicationEvent` で `timeline_digests` の `status` を `GENERATED` に差し戻し + `blog_post_id = NULL` にクリア。ダイジェストの内容は有効なので再 publish 可能。完全に不要なら ADMIN が手動で DISCARD
- **タイムゾーン管理**: `timeline_digest_configs.timezone`（IANA 形式。デフォルト: `Asia/Tokyo`）で自動スケジュールの期間計算を制御。手動生成の `period_start` / `period_end` は ISO 8601 でタイムゾーン付きを受け付ける。サーバーの TZ 設定に依存しない堅牢な設計
- **投稿 content の切り詰め文字数**: `content_max_chars`（デフォルト: 500）でスコープごとにチューニング可能。技術チーム（長文議論が多い）は1000、カジュアルなチーム（短文中心）は300と使い分け。`ai_input_tokens` でコスト影響を監視しながら調整
- **slug 衝突防止**: publish 時の `blog_posts.slug` は `{scope}-digest-{date}-{digest_id}`（例: `team-digest-2026-03-17-42`）で自動生成。digest_id を含めることで同日の再 publish でも一意性を保証
- **publish 時の stale 検知**: `source_post_ids` に記録された投稿の `updated_at` / `deleted_at` を検査し、ダイジェスト生成後に変更があった件数を `stale_warning` としてレスポンスに含める。publish 自体はブロックしない（スナップショット運用）。ADMIN が stale_warning を見て再生成するか判断する
- **モジュール無効化時のスケジュール停止**: 自動スケジュールバッチはスコープの F06.3 モジュール有効状態を `module_selections` テーブルで確認。無効ならスキップ。`is_enabled` はユーザーによる ON/OFF、モジュール状態はプラットフォームレベルの ON/OFF で二重チェック。手動生成 API（`POST /generate`）もモジュール無効時は 403
- **自動スケジュール通知のロールベース送信**: 自動ダイジェストの生成完了・失敗通知はスコープの**現在の ADMIN 全員**に送信。`created_by`（設定を作成した個人）ではなくロールベース。ADMIN の異動・退職に影響されない。`created_by` は「誰が設定したか」の記録用途のみ
- **TEMPLATE スタイル（AI 不使用）**: 統計データ（投稿数・ユニーク投稿者数・リアクション総数・エンゲージメント上位投稿）を Mustache テンプレートに流し込むだけの同期処理。AI API コストゼロ。月次上限カウントには含まれるが `ai_input_tokens` / `ai_output_tokens` は NULL。コスト意識の高いチームや「とりあえず数字だけまとめたい」ユースケースに最適
- **差分ハイライト**: `include_diff_from_previous = true` の場合、前回の PUBLISHED ダイジェストの `generated_body` をコンテキストとして AI に渡し、新規トピックと継続トピックを区別。DAILY 連続利用で「昨日と何が変わったか」が一目でわかる。TEMPLATE スタイルでは無効（AI 不使用のため）
- **自動 publish（auto_publish）**: 自動スケジュール生成完了後に自動で `POST /{id}/publish` 相当の処理を実行し、ブログ DRAFT を作成。あくまで DRAFT なので F06.1 の公開フローは別途必要。信頼が蓄積されたチーム向けのオプション
- **プレビュー編集（PATCH /{id}）**: GENERATED 状態のダイジェストの `generated_title` / `generated_body` / `generated_excerpt` をインライン編集可能。publish 時にリクエストボディで上書きする必要がなくなり、プレビュー→微修正→publish の UX がスムーズに
- **スタイルプリセット**: `style_presets` JSON カラムで `digest_style` + `custom_prompt_suffix` の組み合わせに名前を付けて保存。手動生成時に `preset_name` を指定するだけで設定を再利用可能（毎回 `custom_prompt_suffix` を手入力する手間を省略）
- **WebSocket ステータス通知**: GENERATING → GENERATED / FAILED の遷移を `/topic/digest/{scope_type}/{scope_id}` で即座に配信。フロントエンドがポーリング不要でスピナー→結果表示に自動遷移
- **AI レスポンスキャッシュ**: 入力データの SHA-256 ハッシュをキーに Redis キャッシュ（TTL: 1時間）。regenerate で同一入力の場合に AI 呼び出しをスキップしてコスト削減 + 高速化
- **コストアラート**: 月次 AI コストが `cost-alert-threshold-jpy`（デフォルト: ¥1,000）を超過した場合、SYSTEM_ADMIN にアラート通知。usage API での手動確認に加えた自動監視
- **DISCARDED 物理削除**: クリーンアップバッチで DISCARDED から90日経過したレコードを物理削除。`generated_body`（TEXT 型）の長期蓄積によるストレージ圧迫を防止
- **auto_tag_ids のバリデーション**: publish 時に F06.1 の `blog_tags` テーブルで `deleted_at IS NULL` の有効なタグのみフィルタ。削除済みタグ ID は無視（エラーにしない）。理由: タグ管理は F06.1 の責務であり、ダイジェスト側でタグの有効性を常時保証するのは過剰
- **config 未作成スコープの手動生成**: `timeline_digest_configs` が存在しないスコープでも手動生成（`POST /generate`）は可能。`min_posts_threshold` / `max_posts_per_digest` / `content_max_chars` は `application.yml` の `mannschaft.digest.defaults` をフォールバックとして使用。config 作成を強制しないことで、スケジュール不要なスコープでもワンショット生成が可能
- **非同期処理の完了通知**: generate レスポンスに待ち時間の推定値は含めない（AI レイテンシ・キュー状況で大幅に変動するため不正確）。代わりに生成完了時のプッシュ通知で ADMIN に知らせる。フロントエンドは WebSocket でステータス変更をリアルタイム受信し、スピナー → 結果表示に遷移
- **generated_title / generated_body の NULL 許容**: `GENERATING`（AI 処理中）/ `FAILED`（生成失敗）時は NULL。`GENERATED` / `PUBLISHED` への遷移時にアプリ層で NOT NULL をバリデーション。DB の NOT NULL 制約でダミー値を強制するとデータの意味が曖昧になるため

## 6. セキュリティ考慮事項

- **AI API キーの管理**: API キーは環境変数 or AWS Secrets Manager で管理。`timeline_digest_configs` やDBには保存しない
- **プロンプトインジェクション対策**: `custom_prompt_suffix` を AI に渡す前にサニタイズ。禁止パターン（`ignore previous instructions` / `system:` / `<|` 等）を除去
- **AI 出力のサニタイズ**: 生成された `generated_body` から HTML タグ・スクリプトを除去（Markdown のみ許可）
- **投稿データの最小送信**: AI に送信する投稿データは `content` の先頭 `content_max_chars` 文字（デフォルト500）+ メタデータのみ。画像・添付ファイルの URL は送信しない
- **display_name の外部送信**: AI に `display_name` を送信するため、実名を `display_name` に設定しているユーザーの個人情報が外部 AI サービスに渡る。利用規約（F01.1）に「AI 要約機能ではタイムライン投稿と表示名が外部 AI サービスに送信されます」と明記。オプトアウト（`exclude_from_digest`）で個別制御可能
- **Markdown XSS 対策**: AI 出力のサニタイズに加えて、Markdown 内の危険なリンク（`[text](javascript:...)` / `[text](data:...)` 等）を除去。ブログ公開時のフロントエンド Markdown レンダラでも `allowedSchemes: ['http', 'https', 'mailto']` でスキーム制限
- **アクセス制御**: ダイジェスト生成・閲覧は ADMIN / DEPUTY_ADMIN（`WRITE_BLOG` 権限）に限定。一般ユーザーは生成されたブログ記事のみ閲覧
- **コスト DoS 対策**: 月次生成上限（`digest_monthly_limit`）+ 1分間のレートリミット（Bucket4j: 1分3回）+ 同一スコープ並行生成制限で AI API の過剰呼び出しを防止。月次 AI コストが `cost-alert-threshold-jpy`（デフォルト: ¥1,000）を超過した場合、SYSTEM_ADMIN にアラート通知
- **監査ログ**: `DIGEST_GENERATED` / `DIGEST_PUBLISHED` / `DIGEST_REGENERATED` / `DIGEST_DISCARDED` / `DIGEST_FAILED` / `DIGEST_CONFIG_UPDATED` を `audit_logs` に記録。AI 利用状況の追跡と不正利用（大量生成→破棄の繰り返し等）の検知に使用
- **AI キャッシュのセキュリティ**: Redis キャッシュに保存される AI レスポンスはスコープ内の投稿内容を含む。Redis の認証・暗号化（TLS）を有効にし、キャッシュキーにスコープ情報を含めてクロススコープ参照を防止
