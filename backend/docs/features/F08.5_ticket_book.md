# F08.5: 回数券（プリペイドチケット）

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 8
> **最終更新**: 2026-03-18
> **関連ドキュメント**: F08.2 決済・アクセスコントロール（Stripe 基盤・`stripe_customers`）、F03.4 予約（予約との連携）、F07.1 サービス記録/カルテ（来店記録との連携）
> **モジュール種別**: 選択式モジュール（チームレベル）

---

## 1. 概要

整骨院・美容院・ジム・スクール等のサービス業チームが、回数券（プリペイドチケット）を販売・管理する機能。顧客（MEMBER / SUPPORTER）がチケットを購入し、来店ごとに1枚ずつ消化する。

**ユースケース**:
- **整骨院**: 「骨盤矯正10回券 ¥25,000」を販売。来店ごとにスタッフが消化
- **美容院**: 「カット5回券 ¥18,000」。予約と連動して自動消化
- **ジム**: 「パーソナルトレーニング8回券 ¥40,000」。有効期限3ヶ月
- **スクール**: 「レッスン4回券 ¥12,000」。月をまたいで利用可能

### 決済方式: Stripe + 現地決済のハイブリッド

| 方式 | フロー | 用途 |
|------|--------|------|
| **Stripe 決済** | 顧客がアプリ内で購入 → Stripe Checkout Session → Webhook で支払い確認 → チケット発行 | オンライン事前購入、カード決済 |
| **現地決済（手動記録）** | 来店時に現金/カード/電子マネーで支払い → スタッフが管理画面で手動記録 → チケット発行 | 店頭での対面販売、現金払い |

> **Stripe 基盤**: F08.2 の `stripe_customers` / `payment_items` / Checkout Session フローを再利用。回数券専用の `ticket_products` で商品定義し、購入時に `payment_items` → Checkout Session の既存フローに乗せる。

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `ticket_products` | 回数券の商品定義（名前・回数・価格・有効期間） | あり |
| `ticket_books` | 発行済み回数券（誰が何を購入したか） | なし |
| `ticket_consumptions` | チケット消化履歴（いつ・誰が・何回目） | なし |
| `ticket_payments` | 回数券の決済記録（Stripe / 現地決済） | なし |

### テーブル定義

#### `ticket_products`

回数券の商品カタログ。チームが販売する回数券の種類を定義。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams |
| `name` | VARCHAR(200) | NO | — | 商品名（例: 「骨盤矯正10回券」） |
| `description` | TEXT | YES | NULL | 商品説明（最大1,000文字） |
| `total_tickets` | INTEGER | NO | — | 回数（例: 10） |
| `price` | INTEGER | NO | — | 税込価格（円。例: 25000） |
| `tax_rate` | DECIMAL(5,2) | NO | 10.00 | 消費税率（%。内税方式） |
| `validity_days` | INTEGER | YES | NULL | 有効期間（購入日から何日間。NULL = 無期限） |
| `stripe_product_id` | VARCHAR(100) | YES | NULL | Stripe Product ID（Stripe 販売時に使用） |
| `stripe_price_id` | VARCHAR(100) | YES | NULL | Stripe Price ID |
| `image_url` | VARCHAR(500) | YES | NULL | 商品画像（S3 Pre-signed URL。Stripe Product の `images` にも同期） |
| `is_online_purchasable` | BOOLEAN | NO | TRUE | アプリ内 Stripe 購入を許可するか |
| `is_active` | BOOLEAN | NO | TRUE | 販売中かどうか |
| `sort_order` | INTEGER | NO | 0 | 表示順 |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_tp_team (team_id, deleted_at, is_active, sort_order)  -- チーム別の販売中商品一覧
```

**制約・備考**
- `price` は税込（内税方式）。税抜価格は `price / (1 + tax_rate / 100)` で算出
- `stripe_product_id` / `stripe_price_id`: 商品作成時に Stripe API で Product + Price を同時作成。`is_online_purchasable = false` の場合は作成しない（現地決済のみの商品）
- `is_active = false` に変更しても既存の発行済みチケットには影響しない（販売停止のみ。利用は継続可能）
- **Stripe Price の不変性**: Stripe の Price オブジェクトは immutable。`price` を変更した場合、新しい Stripe Price を作成し `stripe_price_id` を更新。旧 Price は Stripe 側で `active = false` に設定。既存の PENDING Checkout Session は旧価格で決済される（旧 Price は Checkout 完了まで有効）
- 1チームあたり最大30商品
- **同一商品の複数冊購入**: 制限なし（1人が同じ10回券を複数冊購入可能）。回数券は実質プリペイド残高であり、多く買うほど店側の売上が先行するためビジネス的にプラス。「家族用に2冊」「プレゼント用」も自然なユースケース。チケットは `user_id` に紐付くため他人への譲渡は ADMIN が手動で `user_id` を変更する運用

#### `ticket_books`

発行済み回数券。1レコード = 1冊の回数券。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams |
| `product_id` | BIGINT UNSIGNED | NO | — | FK → ticket_products |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（購入者/利用者） |
| `total_tickets` | INTEGER | NO | — | 総回数（購入時の `ticket_products.total_tickets` をコピー） |
| `used_tickets` | INTEGER | NO | 0 | 消化済み回数（デノーマライズ。`ticket_consumptions` の件数と一致） |
| `remaining_tickets` | INTEGER GENERATED ALWAYS AS (`total_tickets` - `used_tickets`) STORED | — | — | 残り回数（生成カラム） |
| `status` | VARCHAR(20) | NO | 'PENDING' | `PENDING`（決済待ち。デフォルト値。手動発行時は明示的に `ACTIVE` を設定）/ `ACTIVE`（利用可能）/ `EXPIRED`（期限切れ）/ `EXHAUSTED`（使い切り）/ `CANCELLED`（キャンセル・返金） |
| `purchased_at` | DATETIME | YES | NULL | 購入日時（ACTIVE に遷移した日時。PENDING 時は NULL） |
| `expires_at` | DATETIME | YES | NULL | 有効期限（`purchased_at + validity_days`。NULL = 無期限） |
| `payment_id` | BIGINT UNSIGNED | YES | NULL | FK → ticket_payments（決済情報への参照） |
| `issued_by` | BIGINT UNSIGNED | YES | NULL | FK → users（手動発行の場合のスタッフ。Stripe 購入は NULL） |
| `note` | VARCHAR(500) | YES | NULL | 管理メモ（例: 「初回キャンペーン適用」） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_tb_user (user_id, team_id, status)              -- ユーザーの有効チケット一覧
INDEX idx_tb_team (team_id, status, created_at DESC)       -- チームのチケット発行一覧
INDEX idx_tb_expires (status, expires_at)                   -- 期限切れバッチ用
INDEX idx_tb_pending_cleanup (status, created_at)           -- PENDING クリーンアップバッチ用（status=PENDING, created_at < 2h ago）
```

**制約・備考**
- `remaining_tickets`: MySQL の `GENERATED ALWAYS AS` で自動算出（`total_tickets - used_tickets`）。アプリ層での不整合を防止
- `status` 遷移: `PENDING`（Stripe 決済待ち）→ `ACTIVE`（決済完了 or 手動発行）→ `EXHAUSTED`（残0）/ `EXPIRED`（期限切れ）/ `CANCELLED`（返金）
- `expires_at`: `validity_days` が設定されている商品の場合、`purchased_at + validity_days` で自動計算。期限切れバッチで `EXPIRED` に遷移
- 手動発行（現地決済）: ADMIN が `POST /ticket-books/issue` で即座に `ACTIVE` 状態で作成
- Stripe 購入: `PENDING` → Webhook `checkout.session.completed` → `ACTIVE`

#### `ticket_consumptions`

チケット消化の個別履歴。来店ごとに1レコード。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `book_id` | BIGINT UNSIGNED | NO | — | FK → ticket_books |
| `consumed_by` | BIGINT UNSIGNED | NO | — | FK → users（消化を記録したスタッフ） |
| `reservation_id` | BIGINT UNSIGNED | YES | NULL | FK → reservations（予約と連携した場合） |
| `service_record_id` | BIGINT UNSIGNED | YES | NULL | FK → service_records（カルテと連携した場合） |
| `consumed_at` | DATETIME | NO | CURRENT_TIMESTAMP | 消化日時 |
| `note` | VARCHAR(500) | YES | NULL | 消化メモ（例: 「腰の痛み改善。次回は肩中心で」） |
| `is_voided` | BOOLEAN | NO | FALSE | 取消済み（誤消化の取消。`used_tickets` をデクリメント） |
| `voided_at` | DATETIME | YES | NULL | 取消日時 |
| `voided_by` | BIGINT UNSIGNED | YES | NULL | FK → users（取消したスタッフ） |

**インデックス**
```sql
INDEX idx_tc_book (book_id, is_voided, consumed_at)  -- チケットの消化履歴（取消除外）
```

**制約・備考**
- 消化時: `ticket_books.used_tickets` をインクリメント（同一トランザクション）。`remaining_tickets` が0になったら `status = EXHAUSTED` に遷移
- 取消（`is_voided = true`）: `ticket_books.used_tickets` をデクリメント。`EXHAUSTED` → `ACTIVE` に復帰
- `reservation_id`: F03.4 の予約を `COMPLETED` にする際、チケットの自動消化オプションを提供
- `service_record_id`: F07.1 のカルテ作成時にチケット消化を紐付け

---

#### `ticket_payments`

回数券の決済記録。Stripe 決済と現地決済の両方を統一管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（購入者） |
| `product_id` | BIGINT UNSIGNED | NO | — | FK → ticket_products |
| `payment_method` | VARCHAR(20) | NO | — | `STRIPE`（オンライン決済）/ `CASH`（現金）/ `CARD_ON_SITE`（店頭カード）/ `E_MONEY`（電子マネー）/ `OTHER`（その他） |
| `amount` | INTEGER | NO | — | 支払い金額（円） |
| `stripe_checkout_session_id` | VARCHAR(200) | YES | NULL | Stripe Checkout Session ID（`STRIPE` の場合） |
| `stripe_payment_intent_id` | VARCHAR(200) | YES | NULL | Stripe PaymentIntent ID（Webhook で取得） |
| `stripe_refund_id` | VARCHAR(200) | YES | NULL | Stripe Refund ID（返金時） |
| `status` | VARCHAR(20) | NO | 'PENDING' | `PENDING`（決済待ち）/ `PAID`（支払い完了）/ `REFUNDED`（返金済み）/ `PARTIALLY_REFUNDED`（部分返金）/ `CANCELLED`（キャンセル） |
| `refund_amount` | INTEGER | YES | NULL | 返金額（部分返金対応） |
| `recorded_by` | BIGINT UNSIGNED | YES | NULL | FK → users（現地決済を記録したスタッフ。STRIPE の場合は NULL） |
| `paid_at` | DATETIME | YES | NULL | 支払い完了日時 |
| `webhook_received_at` | DATETIME | YES | NULL | Stripe Webhook 到達日時（STRIPE の場合のみ。未到達の追跡・障害調査用） |
| `note` | VARCHAR(500) | YES | NULL | 決済メモ |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_tpay_team (team_id, status, created_at DESC)      -- チームの決済一覧
INDEX idx_tpay_user (user_id, status)                        -- ユーザーの決済一覧
UNIQUE KEY uq_tpay_session (stripe_checkout_session_id)      -- Stripe Session の一意性
INDEX idx_tpay_intent (stripe_payment_intent_id)             -- Webhook からの検索
```

**制約・備考**
- Stripe 決済: `POST /ticket-products/{id}/checkout` → Checkout Session 作成 → `PENDING` → Webhook `checkout.session.completed` → `PAID` → `ticket_books.status = ACTIVE`
- 現地決済: `POST /ticket-books/issue` で `ticket_payments`（`status = PAID`）+ `ticket_books`（`status = ACTIVE`）を同一トランザクションで作成
- 返金: Stripe は `Stripe Refund API`、現地決済は手動で `REFUNDED` / `PARTIALLY_REFUNDED` に更新。残チケットの扱いは返金時に決定
- `PARTIALLY_REFUNDED`: 未消化分のみ返金する場合（例: 10回券で7回消化済み → 3回分を返金）

### リクエスト／レスポンス仕様

#### `POST /api/v1/teams/{teamId}/ticket-products`

**認可**: ADMIN

回数券商品を作成する。`is_online_purchasable = true` の場合、Stripe API で Product + Price を同時作成。

**リクエストボディ**
```json
{
  "name": "骨盤矯正10回券",
  "description": "通常1回 ¥3,500 のところ、10回券で ¥25,000（約29%OFF）",
  "total_tickets": 10,
  "price": 25000,
  "tax_rate": 10.00,
  "validity_days": 90,
  "is_online_purchasable": true,
  "image_url": "https://s3.amazonaws.com/mannschaft/tickets/seikotsu-10.jpg"
}
```

**レスポンス（201 Created）**
```json
{
  "data": {
    "id": 1,
    "name": "骨盤矯正10回券",
    "total_tickets": 10,
    "price": 25000,
    "price_excluding_tax": 22727,
    "tax_rate": 10.00,
    "validity_days": 90,
    "is_online_purchasable": true,
    "stripe_product_id": "prod_xxx",
    "stripe_price_id": "price_xxx",
    "is_active": true
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | name が空 / total_tickets ≤ 0 / price ≤ 0 / validity_days ≤ 0 |
| 403 | ADMIN でない |
| 409 | チームの商品数が30上限に到達 |
| 502 | Stripe API エラー（Product/Price 作成失敗） |

---

#### `PUT /api/v1/teams/{teamId}/ticket-products/{id}`

**認可**: ADMIN

商品情報を更新する。

**更新可能フィールド**: `name` / `description` / `validity_days` / `is_online_purchasable` / `is_active` / `sort_order` / `price` / `tax_rate`

**補足**
- `price` または `tax_rate` を変更した場合: 新しい Stripe Price を作成し `stripe_price_id` を差し替え（Stripe Price は immutable のため）
- `total_tickets` は**変更不可**（既発行チケットの `total_tickets` とのコピー整合性が壊れるため。回数を変えたい場合は新商品を作成し旧商品を `is_active = false` にする）
- 既存の ACTIVE / PENDING チケットには影響しない（購入時の値がコピー済み）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `total_tickets` の変更を試みた / バリデーションエラー |
| 403 | ADMIN でない |
| 404 | 商品が存在しない |
| 502 | Stripe Price 再作成失敗 |

---

**レスポンス（200 OK）**

|-----------|------|
| 403 | GUEST / 商品が `is_online_purchasable = false` |

#### `POST /api/v1/teams/{teamId}/ticket-books/{id}/refund`

**認可**: ADMIN

返金処理。Stripe 決済は Stripe Refund API、現地決済は手動で記録。

**リクエストボディ**
```json
{
  "refund_type": "PARTIAL",
  "refund_amount": 7500,
  "note": "未消化3回分を返金"
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `refund_type` | String | Yes | `FULL`（全額返金）/ `PARTIAL`（部分返金） |
| `refund_amount` | Integer | PARTIAL 時必須 | 返金額（`FULL` の場合は `ticket_payments.amount - COALESCE(refund_amount, 0)` で自動計算。既に部分返金済みの場合は残額が FULL 返金額になる） |
| `adjusted_remaining` | Integer | No | 返金後の残回数（ADMIN が明示指定。0 = 全チケット無効化。未指定 = 残回数変更なし） |
| `note` | String | No | 返金理由（最大500文字） |

**レスポンス（200 OK）**: チケット詳細（`status: "CANCELLED"` or `"ACTIVE"`（部分返金で残あり））

**副作用**
- `FULL` 返金: `ticket_books.status = CANCELLED`。`ticket_payments.status = REFUNDED`
- `PARTIAL` 返金: `ticket_payments.status = PARTIALLY_REFUNDED`、`refund_amount` を記録。`adjusted_remaining` の指定に応じてチケット残数を調整:
  - `adjusted_remaining = 0` → `used_tickets = total_tickets`、`status = CANCELLED`
  - `adjusted_remaining = N`（1以上）→ `used_tickets = total_tickets - N`、`status = ACTIVE`（残回数が調整される）
  - `adjusted_remaining` 未指定 → 残回数変更なし（金銭的な返金のみ。チケットはそのまま使える）
- Stripe 決済の場合: `stripe.refunds.create({ payment_intent: xxx, amount: refund_amount })` を実行。`stripe_refund_id` を記録
- 現地決済の場合: 返金はオフラインで実施。DB 上のステータスのみ更新
- 購入者にメール + プッシュ通知
- 監査ログ: `TICKET_REFUNDED`

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `refund_amount` が支払い額を超過 / PARTIAL で `refund_amount` 未指定 / `adjusted_remaining` が `total_tickets` を超過 or 負数 |
| 403 | ADMIN でない |
| 409 | 既に `CANCELLED` / `REFUNDED` |
| 502 | Stripe Refund API エラー |

## 5. ビジネスロジック

### 主要フロー

#### Stripe 購入フロー
```
1. 顧客が POST /ticket-products/{id}/checkout でチケット購入をリクエスト
2. バックエンド:
   a. stripe_customers テーブルでユーザーの Stripe Customer を取得 or 作成（F08.2 既存フロー）
   b. Stripe Checkout Session を作成（mode=payment, line_items=[{price: stripe_price_id, quantity: 1}]）
   c. ticket_payments に INSERT（status = PENDING, stripe_checkout_session_id）
   d. ticket_books に INSERT（status = PENDING）
   e. checkout_url を返却
3. 顧客が Stripe Checkout 画面で支払い
4. Stripe Webhook: checkout.session.completed
   a. metadata から ticket_payment_id / ticket_book_id を取得
   b. ticket_payments.status = PAID, stripe_payment_intent_id を記録
   c. ticket_books.status = ACTIVE, purchased_at = NOW(), expires_at を算出
   d. 購入者にプッシュ通知
5. Stripe Webhook: checkout.session.expired（30分以内に支払い完了しなかった場合）
   a. ticket_payments.status = CANCELLED
   b. ticket_books.status = CANCELLED
```

#### 現地決済フロー
```
1. 来店した顧客が現金/カード/電子マネーで支払い（オフライン）
2. スタッフが POST /ticket-books/issue で手動発行
   - payment_method, amount を記録
   - ticket_payments（PAID）+ ticket_books（ACTIVE）を同一トランザクションで作成
3. 購入者にプッシュ通知
```

#### 予約連動チケット消化フロー
```
1. 顧客が F03.4 の予約フローで来店予約
2. 来店後、スタッフが予約を COMPLETED に変更（F03.4 PATCH /reservations/{id}/complete）
3. フロントエンドが「チケットを消化しますか？」ダイアログを表示:
   a. 顧客の ACTIVE なチケット一覧を表示（GET /my-tickets フィルタ）
   b. スタッフが対象チケットを選択
   c. POST /ticket-books/{id}/consume（reservation_id を指定）
4. カルテ作成時も同様に service_record_id を指定して消化可能
```

#### QR コード消化フロー
```
1. 顧客がスマホで GET /my-tickets/{bookId}/qr を実行 → QR コードを画面表示
2. スタッフがスキャナ（カメラ）で QR コードを読み取り
3. フロントエンドが qr_payload を POST /ticket-books/consume-by-qr に送信
4. バックエンド:
   a. qr_payload から token を抽出
   b. Redis で mannschaft:ticket_qr:{token} → book_id を取得（不存在 or 期限切れ → 401）
   c. book_id のチケットに対して POST /consume と同じ消化処理を実行
   d. Redis からトークンを即座に削除（二重スキャン防止）
5. 結果をスタッフ画面に表示（「✓ 骨盤矯正10回券 残り9回」）
```

#### チケット期限切れバッチ
```
1. Spring @Scheduled バッチが日次で実行（00:30 JST）
2. ticket_books から以下を検索:
   - status = 'ACTIVE'
   - expires_at IS NOT NULL
   - expires_at < NOW()
3. 一括 UPDATE: status = 'EXPIRED'
4. 期限切れユーザーにプッシュ通知:
   「⚠️ {product_name} の有効期限が切れました（残り{remaining_tickets}回未使用）」
5. 期限切れ事前通知（30日前 / 7日前 / 3日前 / 1日前 の4回）:
   - DATEDIFF(expires_at, NOW()) IN (30, 7, 3, 1) のレコードを検索
   - 購入者にプッシュ通知:「🎫 {product_name} の有効期限があと{days}日です（残り{remaining}回）」
   - 重複防止: 日数の完全一致で判定するため、バッチが同日に複数回実行されても同じ通知は1回のみ（notification_delivery で冪等性保証）
   - **延長購入提案**: 30日前通知のタイミングで、対象商品が `is_active = true` の場合、通知に「同じ回数券を追加購入する」リンクを含める（checkout URL を生成せず、商品ページへのディープリンクのみ）
```

### 重要な判定ロジック

- **消化の排他制御**: `ticket_books.used_tickets` のインクリメントは `SELECT ... FOR UPDATE` で排他ロック。同時消化による二重カウントを防止
- **部分返金の計算**: `PARTIAL` 返金時のデフォルト提示額 = `price / total_tickets * remaining_tickets`（1回あたり単価 × 残回数）。ただし ADMIN が任意額を指定可能（キャンペーン割引等の考慮）
- **Stripe Webhook のべき等性**: `stripe_checkout_session_id` の UNIQUE KEY で同一 Webhook の二重処理を防止。既に `PAID` のレコードに対する再処理はスキップ
- **商品価格変更の影響**: `ticket_products.price` を変更しても、既発行の `ticket_books` には影響しない（購入時の `ticket_payments.amount` で記録済み）。新規購入のみ新価格が適用
- **無期限チケットの扱い**: `validity_days = NULL` → `expires_at = NULL`。期限切れバッチの対象外。ただし商品が論理削除されても利用継続可能
- **Stripe Price 変更時の同期**: `ticket_products.price` 変更時に新 Stripe Price を作成し `stripe_price_id` を差し替え。旧 Price は `active = false`。既存 PENDING Checkout は旧価格で完了する（Stripe 側で旧 Price が Session に紐付き済みのため）
- **重複 Checkout 防止**: 同一ユーザー × 同一商品で `PENDING` の `ticket_books` が存在する場合、既存 Checkout Session の URL を再返却。新規 Session は作成しない
- **有効期限延長**: `PATCH /extend` で `expires_at` を未来に変更可能。`EXPIRED` → `ACTIVE` への復帰も可能。延長回数に制限なし（ADMIN の裁量。監査ログで追跡可能）
- **消化取消（void）**: 誤消化を `is_voided = true` で論理的に取消。`used_tickets` をデクリメントし `remaining_tickets` が復元。`EXHAUSTED` → `ACTIVE` への復帰も可能。**72時間制限**: `consumed_at` から72時間超過の消化は取消不可（月次会計の遡及修正防止）。補填が必要な場合は `POST /issue`（`amount = 0`）で新規発行
- **部分返金とチケット残数**: 部分返金時に `adjusted_remaining` で返金後の残回数を ADMIN が明示指定。金額とチケット数の関係は ADMIN の裁量（理由が多様なため自動計算は不適切）。未指定なら残回数変更なし（金銭返金のみ）
- **消化日時のバックデート**: `consumed_at` を任意指定可能（未指定時は `NOW()`）。過去72時間以内のみ許可、未来は 400 で拒否。「昨日来店したが消化忘れ」の対応。F07.1 カルテの `service_date` と一致させることでデータ整合性を確保
- **商品画像**: `ticket_products.image_url` で商品カタログにビジュアルを追加。Stripe Product の `images` にも同期。Checkout 画面にも反映される
- **領収書発行**: Stripe 決済は `receipt_url` にリダイレクト。現地決済はサーバーで PDF を生成（商品名・金額・支払い方法・日時・チーム情報を記載）。`PENDING` 状態では発行不可
- **QR コード消化**: `GET /my-tickets/{bookId}/qr` でワンタイムトークン付き QR を生成（Redis TTL 5分）。スタッフがスキャン → `POST /consume-by-qr` でトークン検証 + 消化。トークンは使用後即座に削除（二重スキャン防止）。F02.1 QR会員証の表示コンポーネントを再利用
- **残回数ダッシュボードウィジェット**: `GET /my-tickets/widget` で ACTIVE チケットの残数 + 期限 + 緊急度（NORMAL/WARNING/CRITICAL）を返却。フロントエンドで色分け表示（緑→黄→赤）
- **一括消化**: `POST /bulk-consume` で1回の来店で複数チケットを同時消化（最大5件）。同一トランザクション。`reservation_id` / `service_record_id` / `consumed_at` は全件に共通適用
- **顧客チケットサマリ**: `GET /users/{userId}/ticket-summary` で ADMIN がカルテ画面から顧客の全チケット残数を横断参照。消化対象のチケット選択をスムーズに
- **売上レポートエクスポート**: `GET /stats/export?format=csv&period=2026-03` で月次売上レポートを CSV/PDF ダウンロード。確定申告・会計ソフト連携用
- **期限切れ前の延長購入提案**: 30日前通知に「同じ回数券を追加購入する」ディープリンクを含める（商品が `is_active = true` の場合のみ）
- **Webhook 到達追跡**: `ticket_payments.webhook_received_at` で Stripe Webhook の到達を記録。PENDING クリーンアップ時の原因調査に使用
- **消化ペース分析**: `avg_days_to_exhaust`（購入→使い切りの平均日数）+ `expiry_rate`（期限切れ率）で `validity_days` の最適値を判断。stats API で商品別に表示
- **退会/脱退時の ACTIVE チケットブロック**: ユーザーに `status = ACTIVE` の `ticket_books` が存在する場合、アカウント退会（F01.1）/ チーム脱退（F01.2）を 409 で拒否。エラーメッセージ: 「有効な回数券が残っています。利用完了 or 返金後に退会してください。」理由: チケットは代金支払い済みの資産であり、一方的に無効化するのは顧客保護の観点から問題。返金フローが既に存在するため、ADMIN が返金 → CANCELLED → 退会可能の導線が自然
- **Stripe Dashboard 直接返金の同期**: `charge.refunded` Webhook 受信時: (1) `refund.amount == charge.amount` → FULL 返金: `ticket_payments.status = REFUNDED` + `ticket_books.status = CANCELLED`。(2) `refund.amount < charge.amount` → PARTIAL 返金: `ticket_payments.status = PARTIALLY_REFUNDED` + `refund_amount` を記録。`ticket_books.status` は変更しない（残回数の調整は ADMIN に委ねる）。ADMIN にプッシュ通知「Stripe Dashboard から返金が実行されました。チケット残数を確認してください」。監査ログに `TICKET_REFUNDED(source=STRIPE_DASHBOARD)` を記録
- **Stripe 決済失敗時のクリーンアップ**: `checkout.session.expired` Webhook で `PENDING` のまま放置されたレコードを `CANCELLED` に遷移。日次バッチでも `PENDING` かつ `created_at < NOW() - INTERVAL 2 HOUR` のレコードをクリーンアップ（Webhook 漏れ対策）

## 6. セキュリティ考慮事項

- **Stripe Webhook 署名検証**: `stripe-signature` ヘッダーで HMAC 検証。Webhook エンドポイントは `WebhookSignatureVerificationFilter` で保護
- **決済金額の改ざん防止**: Checkout Session の金額は `stripe_price_id`（サーバー側で管理）で決定。クライアントから金額を指定するパラメータは存在しない
- **消化権限の限定**: チケット消化は ADMIN / DEPUTY_ADMIN（`MANAGE_PAYMENTS` 権限）のみ。顧客（MEMBER）は自分のチケットを自分で消化できない（不正消化防止）
- **返金権限**: ADMIN のみ。Stripe 返金は Stripe Dashboard からも可能だが、DB 側のステータス更新は `charge.refunded` Webhook で自動同期
- **個人情報保護**: チケット購入・消化履歴は本人 + ADMIN のみ閲覧可能。他の MEMBER からは見えない
- **監査ログ**: `TICKET_BOOK_ISSUED` / `TICKET_CONSUMED` / `TICKET_VOIDED` / `TICKET_REFUNDED` / `TICKET_EXTENDED` / `TICKET_EXPIRED` を `audit_logs` に記録
