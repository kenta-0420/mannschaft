# F04.3: プッシュ通知

> **ステータス**: 🟡 設計中
> **実装フェーズ**: Phase 4
> **最終更新**: 2026-03-13

---

## 1. 概要

アプリ内で発生するイベント（スケジュール更新、出欠督促、お知らせ配信、チャットメンション等）をユーザーに適切なタイミング・チャネルで届ける通知基盤。WebSocket によるリアルタイムポップアップ、ブラウザ Push（PWA Service Worker）、将来の LINE 連携（Phase 9）を統一的に管理し、重要度に応じた通知チャネルの自動選択と、ユーザーごとの受信設定（組織・チーム単位の ON/OFF）を提供する。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全通知の閲覧（監査用）、システム通知の一斉配信 |
| ADMIN | チーム/組織内の通知配信（お知らせ・督促等）、通知設定の閲覧 |
| DEPUTY_ADMIN | `SEND_NOTIFICATION` 権限を持つ場合: お知らせ・督促通知の配信 |
| MEMBER | 自分宛て通知の閲覧・既読・受信設定管理 |
| SUPPORTER | 自分宛て通知の閲覧・既読・受信設定管理 |
| GUEST | 自分宛て通知の閲覧・既読のみ（受信設定は変更不可・全受信） |

### 対象レベル
- [x] 組織 (Organization) — 組織スコープの通知配信・受信設定
- [x] チーム (Team) — チームスコープの通知配信・受信設定
- [x] 個人 (Personal) — 個人宛て通知（DM、リマインド等）

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `notifications` | ユーザー宛て通知レコード | なし（物理削除: 既読90日後バッチ） |
| `notification_preferences` | ユーザーの組織/チーム単位の通知受信設定（スコープ別） | なし |
| `notification_type_preferences` | ユーザーの通知種別ごとの受信設定（種別別） | なし |
| `push_subscriptions` | PWA Push 通知の subscription 情報（デバイス単位） | なし（物理削除: stale 30日バッチ） |

### テーブル定義

#### `notifications`

ユーザーに配信される個別通知レコード。1イベントにつき対象ユーザー数分のレコードを生成する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（通知先ユーザー; ON DELETE CASCADE） |
| `notification_type` | VARCHAR(50) | NO | — | 通知種別（後述の ENUM 相当値） |
| `priority` | ENUM('LOW', 'NORMAL', 'HIGH', 'URGENT') | NO | 'NORMAL' | 通知重要度（チャネル自動選択に使用） |
| `title` | VARCHAR(200) | NO | — | 通知タイトル（表示用） |
| `body` | VARCHAR(1000) | YES | NULL | 通知本文（省略可） |
| `source_type` | VARCHAR(50) | NO | — | トリガー元のエンティティ種別（例: SCHEDULE, CHAT_MESSAGE, BLOG_POST, SYSTEM） |
| `source_id` | BIGINT UNSIGNED | YES | NULL | トリガー元のエンティティ ID（SYSTEM_NOTICE 等 source なしの場合は NULL） |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION', 'PERSONAL', 'SYSTEM') | NO | — | 通知のスコープ |
| `scope_id` | BIGINT UNSIGNED | YES | NULL | スコープ ID（PERSONAL/SYSTEM の場合は NULL） |
| `action_url` | VARCHAR(500) | YES | NULL | ディープリンク先の相対パス（例: `/teams/1/schedules/10`） |
| `actor_id` | BIGINT UNSIGNED | YES | NULL | FK → users（操作実行者; SET NULL on delete）。自分自身の操作通知を除外する判定に使用 |
| `is_read` | BOOLEAN | NO | FALSE | 既読フラグ |
| `read_at` | DATETIME | YES | NULL | 既読日時 |
| `channels_sent` | JSON | YES | NULL | 実際に送信されたチャネルリスト（例: `["IN_APP", "PUSH"]`） |
| `snoozed_until` | DATETIME | YES | NULL | スヌーズ解除予定日時（設定中は通知一覧の未読として非表示） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_noti_user_read_created (user_id, is_read, created_at DESC)  -- 未読通知一覧（メイン）
INDEX idx_noti_user_created (user_id, created_at DESC)                 -- 全通知一覧
INDEX idx_noti_source (source_type, source_id)                         -- 特定エンティティの通知検索
INDEX idx_noti_read_readat (is_read, read_at)                          -- 既読90日バッチ削除用
INDEX idx_noti_scope (scope_type, scope_id, created_at DESC)           -- スコープ別通知一覧
INDEX idx_noti_snoozed (snoozed_until)                                -- スヌーズ解除バッチ用（NULL 以外のみ対象）
```

**制約・備考**
- 論理削除なし。既読通知は `is_read = true` かつ `read_at` が **90日以上前** のレコードをバッチで物理削除（大量蓄積防止）
- 未読通知は物理削除しない（ユーザーが確認するまで保持）
- `notification_type` は VARCHAR で管理し、アプリ層の Java enum で定義（後述）。DB の ENUM にしない理由: 通知種別は機能追加に伴い頻繁に増加するため、ALTER TABLE を回避
- `channels_sent` は配信後に記録。障害調査・再送判定に使用
- `actor_id = user_id` の通知は生成しない（自分自身の操作は通知対象外）
- `source_type` + `source_id` のポリモーフィック設計により、任意のエンティティからの通知生成に対応
- `updated_at` カラムは不要（通知は作成後に `is_read` フラグの更新のみで、変更履歴を追う必要がないため軽量化）

---

#### `notification_preferences`

ユーザーが組織/チーム単位で通知受信を ON/OFF する設定。opt-out 方式（レコードなし = 受信する）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE） |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION') | NO | — | 設定対象スコープ |
| `scope_id` | BIGINT UNSIGNED | NO | — | チーム or 組織の ID |
| `is_enabled` | BOOLEAN | NO | TRUE | 通知受信の有効/無効 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_np_user_scope (user_id, scope_type, scope_id)   -- ユーザー×スコープで一意
INDEX idx_np_scope (scope_type, scope_id)                       -- スコープ別の設定一覧取得
```

**制約・備考**
- **opt-out 方式**: レコードが存在しない場合はデフォルトで通知を受信する。ユーザーが明示的に OFF にした場合のみレコードを INSERT（`is_enabled = false`）
- `is_enabled = true` のレコードも存在しうる（一度 OFF にした後に ON に戻した場合）
- `scope_type` に `GROUP` を追加する拡張を将来的に検討（グループ単位の通知制御）
- GUEST ロールはこのテーブルに書き込み不可（常に全受信）
- 通知配信時に `notification_preferences` を参照し、`is_enabled = false` のスコープへの通知はスキップ

---

#### `notification_type_preferences`

ユーザーが通知種別ごとに受信を ON/OFF する設定。スコープ横断で適用される。opt-out 方式（レコードなし = 受信する）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE） |
| `notification_type` | VARCHAR(50) | NO | — | 通知種別（Java enum `NotificationType` の値） |
| `is_enabled` | BOOLEAN | NO | TRUE | 通知受信の有効/無効 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_ntp_user_type (user_id, notification_type)   -- ユーザー×通知種別で一意
```

**制約・備考**
- **opt-out 方式**: `notification_preferences`（スコープ別）と同様、レコードなし = 受信する
- スコープ別設定（`notification_preferences`）と種別別設定（`notification_type_preferences`）は AND 条件で評価。いずれか一方でも `is_enabled = false` なら通知を生成しない
- `priority = URGENT` の通知種別（`SAFETY_CHECK` 等）は種別別 OFF を無視して強制配信。ただし設定画面では URGENT 種別のトグルをグレーアウトし「この通知は無効化できません」と表示
- GUEST ロールはこのテーブルに書き込み不可（常に全受信）
- **ユースケース**: ADMIN が `ATTENDANCE_RESPONDED`（出欠回答通知）を OFF にして、出欠結果は管理画面で確認する運用を選択可能

**種別別 OFF が想定される主な通知種別**:

| notification_type | 想定ユーザー | 理由 |
|-------------------|------------|------|
| `ATTENDANCE_RESPONDED` | ADMIN | 大人数チームで回答のたびに通知が来ると煩雑。管理画面で一括確認すれば十分 |
| `TIMELINE_REPLY` | 全ユーザー | 返信通知が多すぎる場合にミュート |
| `MEMBER_JOINED` | ADMIN | メンバー追加が頻繁なチームで通知過多を防止 |
| `BLOG_PUBLISHED` | 全ユーザー | 投稿頻度が高いチームで通知を間引きたい場合 |

#### `push_subscriptions`

PWA Push 通知の subscription 情報。ブラウザ/デバイス単位で管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE） |
| `endpoint` | VARCHAR(2000) | NO | — | Push サービスの endpoint URL |
| `p256dh_key` | VARCHAR(500) | NO | — | 暗号化用公開鍵 |
| `auth_key` | VARCHAR(500) | NO | — | 認証シークレット |
| `user_agent` | VARCHAR(500) | YES | NULL | 登録時のブラウザ UA（デバイス識別用） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `last_used_at` | DATETIME | YES | NULL | 最後に Push 送信に成功した日時 |

**インデックス**
```sql
INDEX idx_ps_user (user_id)                   -- ユーザーの subscription 一覧
UNIQUE KEY uq_ps_endpoint (endpoint(500))     -- endpoint は一意（prefix index）
INDEX idx_ps_last_used (last_used_at)         -- stale subscription バッチ削除用
```

**制約・備考**
- 1ユーザーが複数デバイスを持つ場合、デバイスごとに subscription レコードが存在する。**1ユーザーあたり最大10件**（超過時は `last_used_at` が最も古い subscription を自動削除して新規登録）
- `last_used_at` が 30日以上前の subscription は日次バッチで自動削除（stale subscription の清掃）
- `endpoint` の UNIQUE 制約は prefix index（MySQL の VARCHAR(2000) に UNIQUE KEY を直接付与できないため先頭500文字でユニーク化）
- Push 送信で 410 Gone が返された場合はそのレコードを即時物理削除

### ER図（テキスト形式）
```
users (1) ──── (N) notifications [user_id]
users (1) ──── (N) notifications [actor_id]
users (1) ──── (N) notification_preferences
users (1) ──── (N) notification_type_preferences
users (1) ──── (N) push_subscriptions

※ notifications.source_type + source_id はポリモーフィック参照
  → schedules, chat_messages, blog_posts, timeline_posts, surveys 等
```

#### `PATCH /api/v1/notifications/{id}/read`

**認可**: 認証済み全ユーザー（自分の通知のみ）

**リクエストボディ**: なし

**レスポンス（200 OK）**
```json
{
  "data": {
```

#### `PATCH /api/v1/notifications/{id}/unread`

**認可**: 認証済み全ユーザー（自分の通知のみ）

既読通知を未読に戻す。「あとで対応したい」通知を見失わないための機能。

**リクエストボディ**: なし

**レスポンス（200 OK）**
```json
{
  "data": {
    "id": 500,
    "is_read": false,
    "unread_count": 12
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 401 | 未認証 |
| 403 | 他ユーザーの通知 |
| 404 | 通知が存在しない |
| 409 | 既に未読状態 |

**副作用**: Redis カウンター `unread:{userId}` を +1。`read_at` を NULL にリセット

---

#### `PATCH /api/v1/notifications/{id}/snooze`

**認可**: 認証済み全ユーザー（自分の通知のみ）

通知をスヌーズし、指定時間後に再度未読として浮上させる。出欠回答や予約確認など「今は対応できないが忘れたくない」通知に有用。

**リクエストボディ**
```json
{
  "duration": "1h"
}
```

| duration | 意味 |
|----------|------|
| `30m` | 30分後 |
| `1h` | 1時間後 |
| `3h` | 3時間後 |
| `tomorrow` | 翌日 AM 9:00 JST |

**レスポンス（200 OK）**
```json
{
  "data": {
    "id": 500,
    "snoozed_until": "2026-03-12T16:00:00",
    "unread_count": 11
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `duration` が不正な値 |
| 401 | 未認証 |
| 403 | 他ユーザーの通知 |
| 404 | 通知が存在しない |

**処理フロー**:
1. `notifications.snoozed_until` に解除予定日時を SET
2. `notifications.is_read = true`, `read_at = NOW()` に UPDATE（スヌーズ中は一覧から非表示にするため）
3. Redis カウンター `unread:{userId}` を -1
4. スヌーズ解除は後述のバッチで処理

---

#### `PATCH /api/v1/notifications/read-all`

**認可**: 認証済み全ユーザー

自分の未読通知を全件既読にする。

**リクエストボディ**
```json
{
  "scope_type": "TEAM",
  "scope_id": 1
}
```
※ ボディを省略した場合は全スコープの未読を一括既読。`scope_type` + `scope_id` 指定時はそのスコープのみ。

**レスポンス（200 OK）**
```json
{
  "data": {
    "marked_count": 5,
    "unread_count": 7
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `scope_type` が無効な値 / `scope_type` なしで `scope_id` のみ指定 |
| 401 | 未認証 |

**副作用**: Redis カウンター `unread:{userId}` を更新（一括 UPDATE した件数分を減算）

### WebSocket 仕様

#### 接続

```
WS /ws
Authorization: Bearer <JWT>
```

Spring Security の `HandshakeInterceptor` で JWT を検証。STOMP `CONNECT` フレームでも JWT を付与する。

**接続維持・再接続**:
- STOMP heartbeat: サーバー→クライアント 10秒、クライアント→サーバー 10秒（`config.setTaskScheduler()` で設定）
- クライアント側は heartbeat 途絶を検知したら指数バックオフ（1s → 2s → 4s → ... 最大30s）で自動再接続
- 再接続時は最新の JWT で再認証。JWT 期限切れの場合はリフレッシュトークンで更新後に再接続。再接続成功後に `GET /notifications/unread-count` と `GET /notifications?is_read=false&limit=20` を呼び出して切断中の未配信通知をリカバリする
- JWT 有効期限切れを検知した場合（サーバー側の定期チェック、30秒間隔）、STOMP `ERROR` フレームを送信しセッションを切断。クライアントは再認証フローを開始

#### 購読トピック

| トピック | 説明 |
|---------|------|
| `/user/queue/notifications` | 個人宛てリアルタイム通知（STOMP user destination）。`NOTIFICATION`, `UNREAD_COUNT_UPDATE`, `SAFETY_CHECK_ALERT` の全メッセージ種別をこの単一トピックで配信 |

Spring の `@SendToUser` / `SimpMessagingTemplate.convertAndSendToUser()` を使用。ユーザーの WebSocket セッションにのみ配信される。

#### フロントエンド通知トースト仕様

WebSocket で `NOTIFICATION` メッセージを受信した際のフロントエンド表示動作。

```
表示ルール:
1. 画面右上にトースト通知をポップアップ（5秒で自動消去）
2. トーストにはタイトル + アクターアバター + 「表示」ボタンを含む
3. 「表示」ボタンクリック → router.push(action_url) でアプリ内遷移
4. トーストをスワイプ or ×ボタンで手動消去可能（既読にはしない）
5. 複数通知が短時間に到着した場合はスタック表示（最大3件）
   4件目以降は「他 N 件の通知」としてバッジに集約
6. priority = URGENT の場合はトーストではなく全画面モーダル
   （SAFETY_CHECK_ALERT は専用モーダル、それ以外の URGENT は確認ダイアログ）
7. ブラウザタブが非アクティブの場合はトーストを表示せず、
   タブ復帰時に未読カウントバッジのみ更新
```

**実装**: Pinia `notificationStore` の `toastQueue` 配列で管理。Nuxt の `<Teleport>` でルートレベルにレンダリング。

#### 緊急安否確認メッセージ（SAFETY_CHECK_ALERT）

`SAFETY_CHECK` 通知は通常の `NOTIFICATION` メッセージとは**別に**、画面制御用の専用メッセージ `SAFETY_CHECK_ALERT` を push する。フロントエンドはこのメッセージ種別を検知してブロッキングモーダルを表示する。

```json
{
  "type": "SAFETY_CHECK_ALERT",
  "payload": {
    "safety_check_id": 1,
    "scope_type": "TEAM",
    "scope_id": 1,
    "scope_name": "FCバルセロナ",
    "message": "安否を報告してください",
    "created_at": "2026-03-12T14:30:00"
  }
}
```

**通知基盤と安否確認機能の責務分離**:
- **通知基盤の責務**: `notifications` テーブルへの INSERT + 全チャネル強制配信（`priority = URGENT`）。通知一覧に「安否確認のお知らせ」として残る
- **安否確認機能の責務**: `SAFETY_CHECK_ALERT` WebSocket メッセージによるモーダル制御、回答状態の管理（安否確認専用テーブルで管理）、未回答チェック
- 分離理由: 通知の「既読」と安否確認の「回答済み」は別概念。通知を既読にしたが安否は未回答、というケースが発生しうるため、安否確認の状態管理を通知テーブルに依存させない
- ログイン時の未回答チェックは `GET /api/v1/safety-checks/pending` で安否確認専用テーブルを参照（通知テーブルの `is_read` は使わない）

**複数スコープからの安否確認一括回答**:

地震等の災害時、ユーザーが複数のチーム・組織に所属している場合、各スコープから安否確認が同時に届く。1件ずつ回答するのは緊急時に非現実的なため、**一括回答 UI** を提供する。

```
フロントエンドのフロー:
1. SAFETY_CHECK_ALERT を受信するたびに pending リストに追加
   （Pinia safetyCheckStore.pendingChecks に蓄積）
2. モーダルは最初の1件受信時にのみ表示（重複ポップアップしない）
3. モーダル内に未回答の安否確認を一覧表示:
   ┌──────────────────────────────────────────────┐
   │  【安否確認】                                   │
   │                                                │
   │  以下の所属先から安否確認が届いています:           │
   │  ☑ FCバルセロナ（チーム）  14:30                 │
   │  ☑ 東京スポーツ連盟（組織）14:31                 │
   │  ☑ 野球チーム（チーム）    14:32                 │
   │                                                │
   │  一言メッセージ（任意）:                          │
   │  ○ 無事です。特に被害はありません                 │
   │  ○ 自宅にいます。問題ありません                   │
   │  ○ 避難中です。後ほど詳細を連絡します             │
   │  ○ 軽微な被害がありますが無事です                 │
   │  ○ 現在移動中です。安全を確認しています           │
   │  ○ 自由入力 → [____________________________]    │
   │                                                │
   │  □ GPS位置情報を送信する                         │
   │                                                │
   │  [全件まとめて安全を報告する]                     │
   └──────────────────────────────────────────────┘
   ※ メッセージはプリセット例文から選択 or 自由入力。未選択（空欄）でも回答可能
4. 「全件まとめて安全を報告する」→ POST /api/v1/safety-checks/bulk-respond
5. 個別にチェックを外して一部だけ回答することも可能
6. 回答後に新たな SAFETY_CHECK_ALERT が届いた場合はモーダルを再表示
```

```
バックエンドの一括回答 API:
POST /api/v1/safety-checks/bulk-respond
{
  "safety_check_ids": [1, 2, 3],
  "status": "SAFE",
  "message": "自宅にいます。問題ありません",
  "message_source": "PRESET",
  "send_gps": false
}

message_source:
  "PRESET"    — プリセット例文から選択（message にプリセット文面をそのまま格納）
  "CUSTOM"    — 自由入力（message にユーザー入力テキストを格納）
  null        — メッセージなし（message も null）

→ 各安否確認に対して回答レコードを一括 INSERT
→ 対応する SAFETY_CHECK 通知を一括既読にする（notifications.is_read = true）
→ 各スコープの管理者側にリアルタイムで回答が反映（WebSocket）
```

**安否確認メッセージのプリセット例文**:

運営が初期データとして用意する定型文5件。緊急時にユーザーが考える手間を省き、ワンタップで回答できるようにする。

| # | プリセット例文 |
|---|--------------|
| 1 | 無事です。特に被害はありません |
| 2 | 自宅にいます。問題ありません |
| 3 | 避難中です。後ほど詳細を連絡します |
| 4 | 軽微な被害がありますが無事です |
| 5 | 現在移動中です。安全を確認しています |

- プリセット例文は `safety_check_message_presets` テーブル（`id`, `body`, `sort_order`, `is_active`, `created_at`）で管理。SYSTEM_ADMIN がシステム管理画面から追加・編集・無効化可能
- フロントエンドは `GET /api/v1/safety-checks/message-presets` でプリセット一覧を取得し、モーダル内にラジオボタンとして表示
- プリセットを選択せず自由入力欄に記入した場合は `message_source = "CUSTOM"` となる
- メッセージ未入力（空欄）でも回答は可能（`message = null`, `message_source = null`）。緊急時はメッセージなしでも「安全」を報告できることを最優先
- 自由入力は最大200文字（VARCHAR(200)）。プリセット例文も同じ上限に収まる長さで運営が管理

**設計判断**: 一括回答 API は安否確認機能（Phase 3）の責務であり、通知基盤は通知の一括既読処理のみを担う。通知基盤側では `SafetyCheckBulkRespondedEvent` を受信して該当通知を既読にする連携を行う。

### 通知チャネルと重要度

| priority | IN_APP（WebSocket） | PUSH（PWA） | LINE（Phase 9） |
|----------|:-------------------:|:-----------:|:---------------:|
| LOW | ○ | - | - |
| NORMAL | ○ | ○ | - |
| HIGH | ○ | ○ | ○ |
| URGENT | ○ | ○（音・バイブ付き） | ○ |

- **IN_APP**: WebSocket 接続中のユーザーにリアルタイム配信。未接続の場合はログイン時に未読として表示
- **PUSH**: PWA の Service Worker 経由でブラウザ Push 通知を送信。Web Push API（VAPID 認証）を使用
- **LINE**: Phase 9 で LINE Messaging API 連携後に有効化。LINE アカウント連携済みユーザーのみ

### 通知テンプレート（多言語対応）

通知の文面は **Java リソースバンドル**（Spring `MessageSource`）で管理する。

**テンプレート定義**:
```properties
# messages_ja.properties
notification.SCHEDULE_CREATED.title=新しいスケジュールが作成されました
notification.SCHEDULE_CREATED.body={0}が「{1}」を作成しました
notification.ATTENDANCE_REMINDER.title=出欠回答の締切が近づいています
notification.ATTENDANCE_REMINDER.body={0} — 残り{1}
notification.SAFETY_CHECK.title=【緊急】安否確認
notification.SAFETY_CHECK.body={0}から安否確認が届いています。安否を報告してください
```

```properties
# messages_en.properties
notification.SCHEDULE_CREATED.title=New schedule created
notification.SCHEDULE_CREATED.body={0} created "{1}"
notification.ATTENDANCE_REMINDER.title=Attendance deadline approaching
notification.ATTENDANCE_REMINDER.body={0} — {1} remaining
notification.SAFETY_CHECK.title=[URGENT] Safety Check
notification.SAFETY_CHECK.body=Safety check from {0}. Please report your status
```

各通知種別のプレースホルダー引数は `NotificationType` enum 内の Javadoc で定義する:
- `SCHEDULE_CREATED.body`: `{0}` = actorName, `{1}` = scheduleTitle
- `ATTENDANCE_REMINDER.body`: `{0}` = scheduleTitle, `{1}` = remainingTime
- `SAFETY_CHECK.body`: `{0}` = scopeName

**解決ロジック**:
- `NotificationMessageResolver` が `users.locale`（`ja` / `en`、デフォルト `ja`）に基づき `MessageSource.getMessage()` で文面を取得
- プレースホルダーは `java.text.MessageFormat` の `{0}`, `{1}` 形式（Spring `MessageSource` 標準）
- `notifications` テーブルには**解決済みの文面**（ユーザーのロケールで展開した結果）を保存。理由: テンプレートが後から変更されても過去の通知が遡及的に変わらないようにする（送信時点の文面が正）

**設計判断**:
- README に「Spring MessageSource + フロントエンド i18n プラグイン」と明記されており、既存アーキテクチャと一貫
- 初期対応は日本語/英語の2言語のみであり、DB テーブル管理はオーバースペック
- 将来的に SYSTEM_ADMIN がテンプレートを管理画面から編集したい場合は、`notification_templates` テーブルを追加し `MessageSource` を DB ソースに差し替える拡張が可能（現時点では YAGNI）

### 通知チャネルアーキテクチャ（Strategy パターン）

通知の配信チャネルは `NotificationChannel` インターフェース + Spring `@Component` 自動登録方式で拡張可能に設計する。

```java
public interface NotificationChannel {
    /** このチャネルのキー（IN_APP, PUSH, LINE） */
    String getChannelKey();

    /** この priority 以上で配信対象になるか */
    NotificationPriority getMinPriority();

    /** チャネルが現在利用可能か（LINE未連携ならfalse等） */
    boolean isAvailable(Long userId);

    /** 通知を送信する */
    void send(NotificationMessage message, Long userId);
}
```

**Phase 4 時点の実装クラス**:

| クラス | channelKey | minPriority | 説明 |
|--------|-----------|-------------|------|
| `InAppChannel` | IN_APP | LOW | WebSocket で push。全 priority で配信 |
| `WebPushChannel` | PUSH | NORMAL | PWA Service Worker 経由。LOW 以外で配信 |

**Phase 9 で追加**:

| クラス | channelKey | minPriority | 説明 |
|--------|-----------|-------------|------|
| `LineChannel` | LINE | HIGH | LINE Messaging API。`@ConditionalOnProperty(name = "mannschaft.line.enabled")` で制御 |

**ディスパッチャ**:
```java
@Service
@RequiredArgsConstructor
public class NotificationDispatcher {
    private final List<NotificationChannel> channels; // Spring が自動注入

    public List<String> dispatch(NotificationMessage message, Long userId) {
        List<String> sentChannels = new ArrayList<>();
        for (NotificationChannel ch : channels) {
            if (message.getPriority().isAtLeast(ch.getMinPriority())
                    && ch.isAvailable(userId)) {
                try {
                    ch.send(message, userId);
                    sentChannels.add(ch.getChannelKey());
                } catch (Exception e) {
                    log.warn("Channel {} failed for user {}: {}",
                             ch.getChannelKey(), userId, e.getMessage());
                    // 個別チャネルの失敗は他チャネルに影響させない
                }
            }
        }
        return sentChannels;
    }
}
```

**設計判断**:
- Spring DI コンテナが `List<NotificationChannel>` を自動収集するため、新チャネル追加時に `NotificationDispatcher` を変更しない（Open-Closed 原則）
- `@ConditionalOnProperty` で LINE チャネルの有効/無効をプロパティ制御。Phase 4 時点では LINE の `@Component` が存在しなくても問題なく、Phase 9 で追加するだけで有効になる
- `isAvailable(userId)` でユーザー単位の利用可否を判定（LINE 未連携、Push subscription なし等）。チャネルごとに判定ロジックが閉じる
- `dispatch()` の戻り値（実際に送信されたチャネルキーのリスト）をそのまま `channels_sent` に記録

### 主要フロー

#### 通知生成・配信フロー
```
1. 各機能の Service が ApplicationEvent を発行
   （例: ScheduleCreatedEvent, ChatMentionEvent）
2. NotificationEventListener がイベントを受信
3. 通知対象ユーザーリストを算出:
   a. スコープ（チーム/組織）のメンバー一覧を取得
   b. actor_id（操作実行者）を除外
   c. notification_preferences（スコープ別）で is_enabled = false のユーザーを除外
      ※ priority = URGENT の場合は c, d をスキップ（opt-out 無視で強制配信）
   d. notification_type_preferences（種別別）で is_enabled = false のユーザーを除外
   e. archived_at が非 NULL のユーザーを除外
   f. 各ステップの除外理由を DEBUG ログに出力（配信トラブルシューティング用）:
      DEBUG NotificationEventListener - user={id} excluded: self_actor
      DEBUG NotificationEventListener - user={id} excluded: scope_preference OFF (team_id={x})
      DEBUG NotificationEventListener - user={id} excluded: type_preference OFF ({notificationType})
      DEBUG NotificationEventListener - user={id} excluded: archived
      DEBUG NotificationEventListener - user={id} excluded: dedup (key={redisKey})
4. notifications テーブルに対象ユーザー分のレコードを一括 INSERT（batch insert）
5. Redis カウンター unread:{userId} を対象ユーザー分 +1（パイプラインで一括実行）
6. NotificationDispatcher.dispatch() を呼び出し:
   a. 登録済み NotificationChannel の中から priority と isAvailable で配信対象を選定
   b. 各チャネルの send() を実行（InAppChannel → WebPushChannel → 将来 LineChannel）
   c. SAFETY_CHECK の場合は追加で SAFETY_CHECK_ALERT WebSocket メッセージを push
7. dispatch() の戻り値（配信されたチャネルキーのリスト）を channels_sent にバッチ UPDATE
   ※ ステップ 4 の INSERT 時に channels_sent = NULL で挿入し、全ユーザーへの dispatch 完了後に
     CASE 式で一括更新する:
     UPDATE notifications SET channels_sent = CASE id
       WHEN :id1 THEN :json1 WHEN :id2 THEN :json2 ... END
     WHERE id IN (:id1, :id2, ...)
     （ユーザーごとに送信成功チャネルが異なるため、通知 ID ごとに異なる JSON を設定。
       1件ずつの UPDATE（N 回クエリ）は回避し、1クエリで完了させる）
```

#### 大規模通知のスループット戦略

対象ユーザー数 N に応じた段階的処理方式。

| 区分 | 対象ユーザー数 | 処理方式 | 所要時間目安 |
|------|-------------|---------|------------|
| 通常 | N ≤ 500 | 同期処理: batch INSERT 500件 + Redis INCR パイプライン | ~100ms |
| 大規模 | 500 < N ≤ 5,000 | `@Async` 非同期: 500件チャンクに分割して逐次処理 | ~1s |
| 超大規模 | N > 5,000 | Redis Stream にジョブ投入、ワーカーが500件/チャンクで処理 | 10,000件で ~5s |

```
超大規模（N > 5,000）の処理フロー:
1. NotificationEventListener が対象ユーザーリストを算出
2. 先頭 500件は即時処理（同期 batch INSERT + Redis パイプライン + WebSocket push）
3. 残りを Redis Stream notification:dispatch:queue にジョブとして投入
   XADD notification:dispatch:queue * job_id {uuid} chunk_offset 500 total {N}
4. NotificationDispatchWorker（@Scheduled, 100ms間隔でポーリング）がチャンクを消費
5. 進捗は Redis key notification:job:{jobId}:progress で追跡可能
6. 全チャンク完了後にジョブキーを自動削除（TTL: 1時間）
```

**設計判断**:
- 大多数のチームは数十〜数百人であり、通常ケース（N ≤ 500）の単純さを最優先
- MySQL batch INSERT は 500件/バッチで約20ms、Redis INCR パイプラインは500件で約5ms
- `@Async` + チャンク処理で 5,000件は約1秒で完了するため、キューイングのオーバーヘッドに見合わない。5,000超でのみ Redis Stream を使用
- **Phase 4 の実装範囲**: 区分「通常」（N ≤ 500 の同期処理）のみ。「大規模」「超大規模」は実際にそのスケールの組織が登場した時点で追加実装（通知テーブルの INSERT ロジック自体は同じで、呼び出し方の違いのみ）

#### 既読処理フロー
```
1. ユーザーが通知をクリック or 一括既読
2. notifications.is_read = true, read_at = NOW() に UPDATE
   （一括既読の場合は WHERE user_id = :userId AND is_read = false で一括 UPDATE）
   ※ 既読済み通知への再既読リクエストは冪等（200 を返し、DECR しない）
3. Redis カウンター unread:{userId} を減算（個別: -1, 一括: affected rows 分を DECRBY）
   ※ DECR 結果が 0 未満にならないよう Lua スクリプトで MAX(0, current - delta) を保証
4. WebSocket で UNREAD_COUNT_UPDATE メッセージを push
```

#### 既読通知のバッチ削除フロー
```
1. 日次バッチ（Spring @Scheduled, 毎日 AM 3:00 JST）
2. DELETE FROM notifications
   WHERE is_read = true AND read_at < NOW() - INTERVAL 90 DAY
   LIMIT 10000
3. 10000 件未満になるまでループ（大量削除時のテーブルロック防止）
4. 削除件数をログ出力（監査ログには記録しない）
```

#### Redis 未読カウンターの整合性保証
```
1. 通知作成時: INCR unread:{userId}（パイプラインで一括）
2. 個別既読時: DECR unread:{userId}（0 未満にならないよう MAX(0, DECR) で保護）
3. 一括既読時: 一括 UPDATE の affected rows 分を DECRBY
4. 整合性回復（日次バッチ、AM 4:00 JST）:
   a. SELECT user_id, COUNT(*) FROM notifications
      WHERE is_read = false GROUP BY user_id
   b. 各ユーザーの Redis カウンターを DB の実カウントで SET（上書き）
   c. DB とのズレが 5 件以上あったユーザーを WARN ログ出力
```

#### スヌーズ解除バッチ
```
1. 定期バッチ（Spring @Scheduled, 1分間隔）
2. SELECT * FROM notifications
   WHERE snoozed_until IS NOT NULL AND snoozed_until <= NOW()
3. 対象レコードごとに:
   a. is_read = false, read_at = NULL, snoozed_until = NULL に UPDATE
   b. Redis カウンター unread:{userId} を +1
   c. WebSocket で NOTIFICATION メッセージを再 push（通知の再浮上）
   d. Push 通知も再送（priority が NORMAL 以上の場合）
4. 処理件数をログ出力
```

**設計判断**: 1分間隔のポーリングで十分な精度。ユーザーが「30分後」を指定した場合、最大1分の誤差は許容範囲。Spring `@Scheduled(fixedRate = 60000)` で実装。

#### Stale Push Subscription 清掃バッチ
```
1. 日次バッチ（Spring @Scheduled, 毎日 AM 3:30 JST）
2. DELETE FROM push_subscriptions
   WHERE last_used_at < NOW() - INTERVAL 30 DAY
      OR (last_used_at IS NULL AND created_at < NOW() - INTERVAL 30 DAY)
3. 削除件数をログ出力
```

#### 日次ダイジェスト通知バッチ
```
1. 日次バッチ（Spring @Scheduled, 毎日 AM 8:00 JST）
2. notification_type_preferences で DAILY_DIGEST を is_enabled = true にしているユーザーを取得
3. 各ユーザーについて、前日（AM 0:00 〜 AM 23:59 JST）の通知を集計:
   a. 通知種別ごとの件数をカウント
   b. 「昨日の活動: 出欠回答 15件、新メンバー 2名、ブログ投稿 3件」形式の本文を生成
4. DAILY_DIGEST 通知として notifications テーブルに INSERT
   source_type = SYSTEM, source_id = NULL, scope_type = SYSTEM, priority = LOW
5. 集計対象が 0 件のユーザーにはダイジェストを送信しない
```

**設計判断**:
- `DAILY_DIGEST` はデフォルト OFF（opt-in）。ほとんどのユーザーは個別通知で十分であり、ダイジェストは ADMIN 等の通知量が多いユーザー向け
- ダイジェスト本文は `MessageSource` のテンプレート `notification.DAILY_DIGEST.body` で多言語化
- ダイジェスト通知自体が重複防止（dedup）の対象になるよう `source_type = SYSTEM` + 日付ベースの `source_id`（例: `20260312`）を使用

#### PWA Push 通知フロー
```
1. ユーザーがブラウザで通知許可を付与（Notification.requestPermission()）
2. Service Worker が PushManager.subscribe() で Push Subscription を取得
3. フロントエンドが POST /api/v1/push-subscriptions に subscription を送信
4. バックエンドが push_subscriptions テーブルに保存（VAPID public key で暗号化済み）
5. 通知配信時:
   a. WebPushService が対象ユーザーの push_subscriptions を取得
   b. Web Push Protocol で各 subscription endpoint に暗号化ペイロードを POST
   c. レスポンスコード別の処理:
      - 201 Created: 送信成功。push_subscriptions.last_used_at を更新
      - 410 Gone: subscription 無効。push_subscriptions から即時物理削除
      - 429 Too Many Requests: Retry-After ヘッダーに従って遅延リトライ
      - 5xx / タイムアウト: 指数バックオフで最大3回リトライ（1s → 4s → 16s）
      - リトライ上限超過: channels_sent に PUSH を含めず、WARN ログ出力
   d. @Async で非同期実行（メイン通知フローをブロックしない）
```

#### Push 通知ペイロード仕様

Push 通知は常に**固定軽量ペイロード**（タイトル + 短縮本文 + action_url）のみ送信する。ブラウザの Push ペイロード上限（4KB）に対して常に 1KB 以下に収まる設計とし、サイズ超過時のフォールバック分岐を不要にする。

```json
{
  "title": "出欠回答の締切が近づいています",
  "body": "3月15日 練習試合 vs チームA — 残り2時間",
  "tag": "ATTENDANCE_REMINDER:SCHEDULE:10",
  "data": {
    "notification_id": 501,
    "action_url": "/teams/1/schedules/10",
    "priority": "HIGH"
  }
}
```

- `tag`: Service Worker が `showNotification()` で使用するグルーピングキー。同一 tag の通知は上書き表示される。生成ルールは「重要な判定ロジック > 通知のグルーピング」を参照

| フィールド | 上限 | 超過時の処理 |
|-----------|------|------------|
| `title` | 100文字 | 末尾 `…` で切り詰め |
| `body` | 200文字 | 末尾 `…` で切り詰め |
| `data.action_url` | 500文字 | 通常100文字以下のため超過しない |

**設計判断**:
- Push 通知は「気づき」を与えるチャネルであり、詳細はアプリ内で確認するもの。フルコンテンツを Push に詰め込む必要はない
- 動的にペイロードサイズを計測して分岐するロジックは不要な複雑さを生む
- Service Worker の `notificationclick` イベントで `action_url` を使ってアプリ内に遷移させる
- `priority = URGENT` の場合、Service Worker 側で音・バイブレーションパターンを適用（`{ vibrate: [200, 100, 200] }`）

### 重要な判定ロジック

- **自己通知の抑止**: `actor_id = user_id` となる通知は生成しない。自分がスケジュールを作成した場合、自分には通知しない
- **2層 opt-out の適用**: 通知生成時に `notification_preferences`（スコープ別）と `notification_type_preferences`（種別別）の両方を AND 条件で評価。いずれか一方でも `is_enabled = false` なら通知を生成しない（DB INSERT 自体をスキップ）
- **種別別 opt-out のユースケース**: ADMIN が `ATTENDANCE_RESPONDED` を OFF にしておけば、出欠回答のたびに通知が来ることを防げる。結果は管理画面の出欠一覧で確認する運用を選択可能
- **GUEST の制約**: GUEST はアプリ内通知の閲覧・既読のみ可能。通知受信設定の変更は不可（`notification_preferences` / `notification_type_preferences` への書き込みを 403 で拒否）
- **archived ユーザーの除外**: `users.archived_at IS NOT NULL` のユーザーは通知対象から除外。Push subscription も archived 時に自動削除しない（復帰時にそのまま使えるため）
- **URGENT 通知の特別扱い**: `priority = URGENT`（安否確認等）の通知はスコープ別・種別別の両方の設定に関係なく強制配信。opt-out で除外しない
- **バッチ INSERT の最適化**: 500件ずつの batch insert + Redis パイプラインで実行。N > 500 は @Async、N > 5,000 は Redis Stream キューイング（詳細は「大規模通知のスループット戦略」参照）
- **WebSocket 未接続ユーザー**: IN_APP チャネルの通知は WebSocket 未接続でも `notifications` テーブルに必ず INSERT される。ユーザーがログイン後に通知一覧で確認できる
- **重複通知の防止**: Redis キー `noti:dedup:{sourceType}:{sourceId}:{userId}:{notificationType}` を SET（TTL: 5分）。キーが存在する場合は通知生成をスキップ。`source_id` が NULL の場合（`SYSTEM_NOTICE` 等）はキーを `noti:dedup:{sourceType}:0:{userId}:{notificationType}` とする（`0` はプレースホルダー）。DB UNIQUE 制約ではなく Redis で制御する理由: (1) 通知テーブルにカラム追加不要 (2) TTL で自動期限切れ (3) 高速チェック。リマインダーの再送は `notification_type` のサフィックスを変える（例: `ATTENDANCE_REMINDER_2ND`）か、5分以上の間隔を空ける
- **通知のグルーピング（Phase 4 以降で検討）**: 同一 `source_type` + `source_id` + `notification_type` の通知が短時間に大量生成される場合（出欠回答10件等）、フロントエンド（Pinia `notificationStore`）が表示上グルーピングする（「田中太郎 他9名が出欠を回答しました」）。バックエンドは1通知1レコードのまま変更しない。Push 通知は Service Worker の `tag` パラメータで同一 tag を上書き表示。tag キーは通知種別によって異なる: `source_id` ベース（例: `tag: "ATTENDANCE_RESPONDED:SCHEDULE:10"`）と `scope` ベース（例: `tag: "MEMBER_JOINED:TEAM:1"`）。`source_id` が NULL の種別（`SYSTEM_NOTICE` 等）は `scope_type:scope_id` を使用
- **チーム脱退時の通知処理**: `TeamMemberRemovedEvent` を受信し、(1) 該当スコープの未読通知を一括既読化（`is_read = true`）、(2) Redis `unread:{userId}` を該当分減算、(3) 該当スコープの `notification_preferences` レコードを物理削除。通知レコード自体は物理削除しない（90日バッチで自然消滅）。フロントエンドは脱退済みスコープの通知に「このチームから脱退しました」ラベルを表示し `action_url` のクリックを無効化
- **チーム/組織削除時の清掃**: `TeamDeletedEvent` / `OrganizationDeletedEvent` を受信し、該当 `scope_id` の `notification_preferences` レコードを一括物理削除（FK 制約なしのポリモーフィック設計のため、アプリ層でのイベント駆動清掃が必要）
- **おやすみモード**: Phase 4 では実装しない。ブラウザ/OS のシステムレベルおやすみモードが既に存在するため、アプリ側での二重制御は見送り。Phase 9（LINE 連携）で時間帯制限の需要が高まった時点で `notification_preferences` に `quiet_hours_start` / `quiet_hours_end` カラムを追加して対応

## 5.5. 可観測性（Observability）

### Micrometer メトリクス

通知基盤の健全性を Grafana 等で可視化するためのメトリクス定義。Spring Boot Actuator + Micrometer で自動収集し、Prometheus エンドポイント `/actuator/prometheus` で公開する。

| メトリクス名 | 種類 | タグ | 説明 |
|-------------|------|------|------|
| `notification.created.count` | Counter | `type`, `priority` | 通知生成数 |
| `notification.dispatch.count` | Counter | `channel`, `status` (success/failure) | チャネル別配信数 |
| `notification.dispatch.duration` | Timer | `channel` | チャネル別配信所要時間 |
| `notification.push.error.count` | Counter | `error_code` (410/429/5xx/timeout) | Push 送信エラー数 |
| `notification.push.retry.count` | Counter | `attempt` (1/2/3) | Push リトライ回数 |
| `notification.snooze.active.gauge` | Gauge | — | 現在スヌーズ中の通知数 |
| `notification.redis.drift` | Counter | — | 日次整合性バッチで検出した Redis/DB のズレ件数 |
| `notification.batch.deleted.count` | Counter | — | 既読90日バッチで削除した通知数 |

**アラート推奨閾値**:
- `notification.dispatch.count{status=failure}` が 5分間で 50件超 → Slack アラート
- `notification.push.error.count{error_code=5xx}` が 5分間で 100件超 → PagerDuty
- `notification.redis.drift` が 1日で 100件超 → 調査必要

### 配信判定デバッグログ

通知が届かない問い合わせ対応のため、配信フローの各除外ステップで DEBUG レベルのログを出力する（詳細は「通知生成・配信フロー > ステップ 3f」参照）。

本番環境では DEBUG ログはデフォルト OFF（`logging.level.com.mannschaft.app.notification=INFO`）。調査時に動的に ON に切り替える:
```

POST /actuator/loggers/com.mannschaft.app.notification
{"configuredLevel": "DEBUG"}
```

## 6. セキュリティ考慮事項

- **認可チェック**: 全通知 API は自分の通知のみ操作可能。`user_id = currentUser.id` をサーバーサイドで必ず検証（パスパラメータの通知 ID からユーザー ID を逆引き）
- **WebSocket 認証**: ハンドシェイク時の JWT 検証 + STOMP CONNECT フレームでの JWT 検証の二重チェック。JWT の有効期限切れ時は WebSocket セッションを強制切断
- **Push Subscription の保護**: `push_subscriptions` のデータは他ユーザーに公開しない。endpoint URL が漏洩すると第三者から Push 通知を送信される可能性があるため、API レスポンスでは endpoint の完全な値を返さない
- **レートリミット**: 通知一覧 API は Bucket4j で 1分間に120回。既読 API は1分間に300回（一括既読は60回）。Push Subscription 登録は1分間に10回。通知設定変更 API（`notification-preferences`, `notification-type-preferences`）は1分間に30回
- **通知本文の XSS 対策**: `title` / `body` はサーバーサイドで HTML エスケープ済みの状態で保存。フロントエンドでも `v-text`（`v-html` は使わない）でレンダリング
- **action_url のバリデーション**: `action_url` はアプリ内相対パス（`/` 始まり）のみ許可。外部 URL（`http://`, `//` 始まり等）を拒否し、オープンリダイレクト攻撃を防止。フロントエンド側でも `router.push()` 前にパスをバリデーション
- **VAPID 鍵管理**: VAPID の秘密鍵は環境変数（`VAPID_PRIVATE_KEY`）で管理。Git リポジトリにコミットしない
- **WebSocket メッセージサイズ制限**: STOMP メッセージの最大サイズを 64KB に制限（`configureWebSocketTransport()` で設定）。異常に大きなメッセージによるメモリ枯渇を防止

### 外部設定（application.yml）

通知基盤の各種閾値・TTL をプロパティで外部化し、本番の負荷状況に応じてチューニング可能にする。

```yaml
mannschaft:
  notification:
    # バッチ処理
    batch-size: 500              # batch INSERT のチャンクサイズ
    async-threshold: 500         # @Async に切り替える対象ユーザー数閾値
    stream-threshold: 5000       # Redis Stream に切り替える対象ユーザー数閾値
    # 重複防止
    dedup-ttl-seconds: 300       # 重複通知防止 Redis キーの TTL（秒）
    # Push
    max-push-subscriptions: 10   # 1ユーザーあたりの Push Subscription 上限
    push-retry-max-attempts: 3   # Push 送信リトライ回数上限
    stale-subscription-days: 30  # Stale subscription 自動削除までの日数
    # 保持期間
    read-retention-days: 90      # 既読通知の物理削除までの日数
    # スヌーズ
    snooze-check-interval-ms: 60000  # スヌーズ解除チェック間隔（ミリ秒）
    # ダイジェスト
    digest-send-hour: 8          # 日次ダイジェスト送信時刻（JST、0-23）
  vapid:
    public-key: ${VAPID_PUBLIC_KEY}
    private-key: ${VAPID_PRIVATE_KEY}
```

**設計判断**: ハードコードだと本番での閾値調整のたびにデプロイが必要になる。`@ConfigurationProperties(prefix = "mannschaft.notification")` で型安全にバインドし、テスト時のオーバーライドも容易にする。
