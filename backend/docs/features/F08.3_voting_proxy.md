# F08.3: 議決権行使・委任状

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 8
> **最終更新**: 2026-03-15
> **モジュール種別**: 選択式モジュール #8
> **関連ドキュメント**: F05.3 電子印鑑（委任状署名連携）、F05.4 アンケート・投票（汎用投票との棲み分け）、F04.3 プッシュ通知

---

## 1. 概要

総会・定期会議の議案に対して電子で賛否を投票（出席/賛成/反対/棄権）し、出席できない場合は委任状を電子提出する機能。NPO・協同組合・マンション管理組合・株主総会など「総会を持つ組織」で汎用利用可能。投票期限・定足数の設定、リアルタイム集計・結果表示を備え、電子印鑑（F05.3）と連携して提出の証跡を記録する。無記名/記名投票の切り替えに対応し、無記名時は個人の投票内容を ADMIN にも非公開とする。**無記名セッションでは委任を禁止する**（委任状の delegation_id から投票内容が逆引き可能になり、無記名の原則が崩壊するため）。

### 決議モード

セッション作成時に決議モードを選択する。

| モード | 名称 | 用途 | 投票タイミング |
|--------|------|------|--------------|
| `MEETING` | 総会決議 | 対面/オンライン総会の当日に決議 | 総会当日に ADMIN が議案ごとに投票を開始。討論を踏まえてリアルタイム投票。委任状は事前収集 |
| `WRITTEN` | 書面決議 | 総会を開催せず書面で決議 | 投票期間（数日〜数週間）に全員が非同期で投票。委任も投票も同時に受付 |

```
MEETING（総会決議）:
  事前: 委任状を収集（OPEN 中は委任のみ受付。投票は不可）
  当日: ADMIN が議案ごとに投票開始 → 出席者がリアルタイム投票 → 集計
        または全議案一括で投票開始 → 一括投票 → 集計
  集計: 出席者の投票 + 委任票を加算して可決/否決を判定

WRITTEN（書面決議）:
  投票期間中に全員が非同期で投票（賛成/反対/棄権）
  委任状も同時に受付
  締切後に集計して可決/否決を判定
```

### F05.4 アンケート・投票との棲み分け

| 観点 | F05.4 アンケート・投票 | F08.3 議決権行使・委任状 |
|------|----------------------|------------------------|
| 目的 | 意見収集・満足度調査 | 組織の意思決定（法的拘束力のある決議） |
| 質問形式 | 自由（単一選択/複数選択/自由記述/評価スケール） | 固定（賛成/反対/棄権 + 出席確認） |
| 委任 | なし | あり（代理人指定 or 白紙委任。無記名セッションでは不可） |
| 署名 | 不要 | 電子印鑑連携 |
| 定足数 | なし（任意の回答率表示のみ） | あり（法的要件に基づく定足数チェック） |
| フェーズ | Phase 5 | Phase 8 |

### テーブル定義

#### `proxy_vote_sessions`

投票セッション。総会1回 = 1セッション。複数の議案を含む。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（TEAM / ORGANIZATION） |
| `team_id` | BIGINT UNSIGNED | YES | NULL | FK → teams。TEAM の場合に設定 |
| `organization_id` | BIGINT UNSIGNED | YES | NULL | FK → organizations。ORGANIZATION の場合に設定 |
| `title` | VARCHAR(200) | NO | — | セッション名（例: 「2026年度 第1回定期総会」） |
| `description` | TEXT | YES | NULL | セッション概要・議事要旨 |
| `resolution_mode` | VARCHAR(20) | NO | — | 決議モード（`MEETING`: 総会決議 / `WRITTEN`: 書面決議） |
| `status` | VARCHAR(20) | NO | 'DRAFT' | ステータス（DRAFT / OPEN / CLOSED / FINALIZED） |
| `meeting_date` | DATE | YES | NULL | 総会開催日。`MEETING` モードの場合に設定 |
| `voting_start_at` | DATETIME | YES | NULL | 投票受付開始日時。`WRITTEN` = 投票期間の開始 / `MEETING` = 委任受付の開始 |
| `voting_end_at` | DATETIME | YES | NULL | 投票受付締切日時。`WRITTEN` = 投票期間の終了 / `MEETING` = 委任受付の締切（総会前日等） |
| `is_anonymous` | BOOLEAN | NO | FALSE | 無記名投票フラグ。TRUE = 個人の投票内容は ADMIN にも非公開 |
| `quorum_type` | VARCHAR(20) | NO | 'MAJORITY' | 定足数算出方式（MAJORITY: 過半数 / TWO_THIRDS: 2/3 / CUSTOM: カスタム） |
| `quorum_threshold` | DECIMAL(5,2) | YES | NULL | カスタム定足数の閾値（%）。`quorum_type = 'CUSTOM'` の場合に使用 |
| `eligible_count` | INT UNSIGNED | NO | — | 議決権総数（投票権を持つメンバー総数。セッション作成時にスナップショット） |
| `is_auto_accept_delegation` | BOOLEAN | NO | FALSE | 委任状の自動承認。TRUE = 提出即受理（小規模チーム向け）。FALSE = ADMIN 承認フロー |
| `remind_before_hours` | JSON | YES | NULL | 自動リマインドスケジュール（配列。例: `[72, 24]` = 3日前・1日前に未投票者へ通知）。`WRITTEN` モードのみ有効。NULL = デフォルト `[24]` |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT（議事録 PDF に作成者名が必要。退会前にセッション完了 or 引継ぎを要求） |
| `version` | BIGINT | NO | 0 | 楽観的ロックバージョン（@Version） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_pvs_scope (scope_type, team_id, organization_id, status)  -- スコープ別セッション一覧
INDEX idx_pvs_status (status, voting_end_at)                         -- ステータス別・締切順
```

**制約・備考**
- `scope_type` のスコープ制約: TEAM → `team_id` NOT NULL、ORGANIZATION → `organization_id` NOT NULL
- ステータスのライフサイクル:
  - 共通: `DRAFT` → `OPEN` → `CLOSED` → `FINALIZED`
  - `MEETING` モード: `OPEN` = 委任受付中（投票は不可）。総会当日に ADMIN が議案ごとに投票を制御。全議案 `VOTED` 後に `CLOSED` → `FINALIZED`
  - `WRITTEN` モード: `OPEN` = 投票 + 委任を同時受付。`voting_end_at` 到達で自動 `CLOSED` → ADMIN が手動で `FINALIZED`
- `meeting_date`: `MEETING` モードでは必須（Service 層バリデーション）。`WRITTEN` モードでは NULL
- `eligible_count`: セッション作成時にスコープ内の MEMBER 以上のユーザー数をスナップショット。投票後のメンバー増減に影響されない
- `is_anonymous = TRUE` の場合: `proxy_votes` にはレコードを INSERT するが、API レスポンスでは投票者の個人情報を一切返却しない。集計結果のみ公開。ADMIN でも個人の投票内容を確認不可。**委任（`POST /delegate`）は禁止**（`delegation_id` → `delegator_id` の結合で投票内容が逆引きされ、無記名の原則が崩壊するため）
- `quorum_type`:
  - `MAJORITY`: 議決権総数の過半数（50%超）が出席（投票 + 委任）で定足数充足
  - `TWO_THIRDS`: 2/3 以上
  - `CUSTOM`: `quorum_threshold` で任意の割合を指定（Service 層で 0.01〜100.00 の範囲バリデーション）

#### `proxy_vote_motions`

議案。1セッションに複数の議案を紐付ける。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `session_id` | BIGINT UNSIGNED | NO | — | FK → proxy_vote_sessions。ON DELETE CASCADE |
| `motion_number` | INT UNSIGNED | NO | — | 議案番号（表示順。1始まり） |
| `title` | VARCHAR(200) | NO | — | 議案タイトル（例: 「第1号議案: 2025年度決算報告の承認」） |
| `description` | TEXT | YES | NULL | 議案の詳細説明 |
| `voting_status` | VARCHAR(20) | NO | 'PENDING' | 議案別投票状態（`PENDING` / `VOTING` / `VOTED`）。`MEETING` モードで使用。`WRITTEN` モードではセッションの OPEN/CLOSED に連動 |
| `vote_deadline_at` | DATETIME | YES | NULL | 投票自動終了日時。`MEETING` モードで `start-vote` 時に `duration_seconds` から算出。到達時にバッチで自動 `end-vote` |
| `required_approval` | VARCHAR(20) | NO | 'MAJORITY' | 可決要件（MAJORITY: 過半数 / TWO_THIRDS: 2/3 / UNANIMOUS: 全員一致） |
| `result` | VARCHAR(20) | YES | NULL | 採決結果（APPROVED / REJECTED）。`MEETING`: `VOTED` 遷移時に自動判定 / `WRITTEN`: `FINALIZED` 遷移時に一括判定 |
| `approve_count` | INT UNSIGNED | NO | 0 | 賛成票数（denormalize） |
| `reject_count` | INT UNSIGNED | NO | 0 | 反対票数（denormalize） |
| `abstain_count` | INT UNSIGNED | NO | 0 | 棄権票数（denormalize） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_pvm_session (session_id, motion_number)  -- セッション内の議案一覧（motion_number 順）
```

**制約・備考**
- 1セッションあたり最大30議案（Service 層バリデーション）
- `required_approval`: 議案ごとに可決要件を設定可能（例: 予算は過半数、定款変更は 2/3 等）
- `voting_status` のライフサイクル（`MEETING` モード）: `PENDING`（未開始）→ `VOTING`（投票受付中）→ `VOTED`（投票完了・集計確定）。ADMIN が議案ごとに遷移を制御
- `voting_status`（`WRITTEN` モード）: セッション OPEN 時に全議案を `VOTING` に一括変更、CLOSED 時に `VOTED` に一括変更
- `result`: `MEETING` モード = 各議案の `VOTED` 遷移時に委任票を加算して自動判定。`WRITTEN` モード = `FINALIZED` 遷移時に一括判定
- `approve_count` / `reject_count` / `abstain_count`: 投票時にアトミック更新。委任票は委任先の投票に含算

#### `proxy_vote_attachments`

セッション・議案共通の添付ファイル。ポリモーフィック参照で対象を切り替える。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `target_type` | VARCHAR(20) | NO | — | 添付先種別（`SESSION` / `MOTION`） |
| `target_id` | BIGINT UNSIGNED | NO | — | `SESSION` → proxy_vote_sessions.id / `MOTION` → proxy_vote_motions.id |
| `file_key` | VARCHAR(500) | NO | — | S3 オブジェクトキー |
| `original_filename` | VARCHAR(255) | NO | — | アップロード時の元ファイル名 |
| `file_size` | INT | NO | — | ファイルサイズ（バイト） |
| `mime_type` | VARCHAR(100) | NO | — | MIMEタイプ |
| `attachment_type` | VARCHAR(20) | NO | 'DOCUMENT' | 種別（`MINUTES`: 議事録 / `DOCUMENT`: 配布資料 / `OTHER`: その他）。`MINUTES` は `SESSION` のみ |
| `sort_order` | SMALLINT | NO | 0 | 表示順 |
| `uploaded_by` | BIGINT UNSIGNED | NO | — | FK → users。アップロード者 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_pva_target (target_type, target_id, sort_order)  -- 対象別添付ファイル一覧
```

**制約・備考**
- ポリモーフィック参照（FK 制約なし、アプリ層で検証）。F05.2 `circulation_documents.scope_type + scope_id` と同一パターン
- `SESSION` 添付:
  - 議事録（`MINUTES`）はセッションのどのステータスでもアップロード可能（FINALIZED 後に正式議事録を添付するユースケース）
  - 配布資料（`DOCUMENT`）は DRAFT / OPEN 中のみ
  - ファイルサイズ上限: 1ファイル50MB（録音等）、1セッション最大10ファイル
  - 対応形式: PDF, DOCX, JPEG, PNG, WebP, MP3, MP4, M4A
- `MOTION` 添付:
  - DRAFT / OPEN 中にアップロード可能（委任者の判断材料・総会当日の補足資料）
  - ファイルサイズ上限: 1ファイル20MB、1議案最大5ファイル
  - 対応形式: PDF, JPEG, PNG, WebP
- S3 配信: CloudFront 経由の Pre-signed URL（有効期限: 1時間）
- Service/Repository を1組で管理でき、添付まわりのコード重複を排除

```sql
INDEX idx_pvmc_motion (motion_id, created_at)  -- 議案別コメント一覧
```

**制約・備考**

#### `proxy_votes`

個人の投票回答。1ユーザー × 1議案 = 1レコード。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `motion_id` | BIGINT UNSIGNED | NO | — | FK → proxy_vote_motions。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。投票権の所有者（本人投票 = 本人 / 代理投票 = 委任者）。UNIQUE KEY で1議案1人につき1票を保証 |
| `vote_type` | VARCHAR(20) | NO | — | 投票内容（APPROVE / REJECT / ABSTAIN） |
| `is_proxy_vote` | BOOLEAN | NO | FALSE | 委任による代理投票か |
| `delegation_id` | BIGINT UNSIGNED | YES | NULL | FK → proxy_delegations。代理投票の場合に紐付け |
| `voted_at` | DATETIME | NO | CURRENT_TIMESTAMP | 投票日時 |

**インデックス**
```sql
UNIQUE KEY uq_pv_motion_user (motion_id, user_id)       -- 同一ユーザーの二重投票防止
INDEX idx_pv_motion (motion_id, vote_type)               -- 議案別の集計
INDEX idx_pv_user (user_id)                               -- ユーザー別の投票履歴
```

**制約・備考**
- 投票は1議案1ユーザーにつき1回。変更する場合は DELETE → INSERT（Service 層で実装。`OPEN` ステータスの間のみ変更可能）
- `is_proxy_vote = TRUE`: 委任を受けた代理人が委任者の代わりに投票した場合。`user_id` = 委任者（delegator_id）、`delegation_id` で委任状と紐付け。代理人の user_id は `proxy_delegations.delegate_id` から参照可能
- 白紙委任の場合: 委任を受けた代理人が全議案を一括投票する。代理人が投票しない議案は「棄権」扱い
- `is_anonymous = TRUE` のセッション: レコードは INSERT するが、API レスポンスでは `user_id` を返却しない

#### `proxy_delegations`

委任状。出席できないメンバーが代理人を指定、または白紙委任を行う。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `session_id` | BIGINT UNSIGNED | NO | — | FK → proxy_vote_sessions。ON DELETE CASCADE |
| `delegator_id` | BIGINT UNSIGNED | NO | — | FK → users。委任者（投票権を持つが出席しないメンバー） |
| `delegate_id` | BIGINT UNSIGNED | YES | NULL | FK → users。代理人。NULL = 白紙委任（議長に一任） |
| `is_blank` | BOOLEAN | NO | FALSE | 白紙委任フラグ（TRUE = 議長に一任。`delegate_id` は NULL） |
| `electronic_seal_id` | BIGINT UNSIGNED | YES | NULL | FK → electronic_seals。委任状提出の電子印鑑証跡 |
| `reason` | VARCHAR(500) | YES | NULL | 委任理由（任意） |
| `status` | VARCHAR(20) | NO | 'SUBMITTED' | ステータス（SUBMITTED / ACCEPTED / REJECTED / CANCELLED） |
| `reviewed_by` | BIGINT UNSIGNED | YES | NULL | FK → users。承認/却下した管理者 |
| `reviewed_at` | DATETIME | YES | NULL | 承認/却下日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_pd_session_delegator (session_id, delegator_id)  -- 1セッション1委任者1件
INDEX idx_pd_session (session_id, status)                       -- セッション別の委任状一覧
INDEX idx_pd_delegate (delegate_id)                             -- 代理人が受けた委任一覧
```

**制約・備考**
- 委任者は MEMBER 以上で投票権を持つユーザーのみ
- 代理人は同一スコープ（チーム/組織）のメンバーであること。自分自身への委任は不可
- 白紙委任（`is_blank = TRUE`）: 議長（= セッション作成者 or ADMIN）が全議案を一括投票
- 委任状の承認フロー: `SUBMITTED` → ADMIN が `ACCEPTED` or `REJECTED`。`ACCEPTED` 後に代理人は投票可能
- 委任者が自分で投票した場合（心変わりで出席）: 委任状は自動的に `CANCELLED` に遷移し、委任は無効化
- `electronic_seal_id`: 電子印鑑で署名することで委任の正当性を担保。署名なしでも委任は可能だが、「署名あり」のマークが UI に表示される

### ER図（テキスト形式）
```
teams (1) ──── (N) proxy_vote_sessions（team_id; TEAM）
organizations (1) ──── (N) proxy_vote_sessions（organization_id; ORGANIZATION）

proxy_vote_sessions (1) ──── (N) proxy_vote_attachments（target_type=SESSION）
proxy_vote_motions (1) ──── (N) proxy_vote_attachments（target_type=MOTION）
proxy_vote_motions (1) ──── (N) proxy_vote_motion_comments
proxy_vote_motions (1) ──── (N) proxy_votes

users (1) ──── (N) proxy_vote_sessions（created_by）
users (1) ──── (N) proxy_votes（user_id）
users (1) ──── (N) proxy_vote_motion_comments（user_id）
users (1) ──── (N) proxy_delegations（delegator_id / delegate_id）

```

#### `POST /api/v1/proxy-votes/{id}/cast`

全議案に対して一括投票する。

**リクエストボディ**
```json
{
  "votes": [
    { "motion_id": 1, "vote_type": "APPROVE" },
    { "motion_id": 2, "vote_type": "REJECT" },
    { "motion_id": 3, "vote_type": "ABSTAIN" }
  ]
}
```

- **`WRITTEN` モード**: 全議案への投票が必須（一部のみ投票は不可。投票しない議案は `ABSTAIN` を指定）
- **`MEETING` モード**: `voting_status = 'VOTING'` の議案のみ投票可能。議案ごとの個別投票も、複数議案の一括投票も可。`votes` 配列には `VOTING` 状態の議案のみ含めること
- 委任を受けている代理人の場合: バックエンドが `proxy_delegations` から `delegate_id = currentUser AND status = 'ACCEPTED'` の委任を自動取得し、各委任者の代理投票レコードを `is_proxy_vote = TRUE` + `delegation_id` 紐付けで一括生成。代理人は自分の投票内容と同じ内容が全委任者に適用される（議案ごとの個別指定は不可）

**レスポンス（201 Created）**
```json
{
  "data": {
    "session_id": 1,
    "voted_motions": 3,
    "voted_at": "2026-04-05T14:30:00+09:00"
  }
}
```

**副作用**:
1. `proxy_vote_motions` の `approve_count` / `reject_count` / `abstain_count` をアトミック更新
2. 委任者が自分で投票した場合: 既存の委任状を `CANCELLED` に自動遷移
3. ApplicationEvent: `ProxyVoteCastEvent`（管理者にリアルタイム集計更新を通知）
4. 投票者本人に確認通知（「投票が記録されました」+ 投票内容サマリー。`is_anonymous = TRUE` のセッションでも本人のみに送信）
5. 代理投票がある場合: 各委任者に通知（「○○さんがあなたの代理として投票しました」）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | バリデーションエラー / `WRITTEN`: 全議案への投票が揃っていない / `MEETING`: `VOTING` 状態でない議案が含まれている |
| 403 | 投票権がない（SUPPORTER / GUEST）/ セッションが OPEN でない / `MEETING`: 投票対象の議案が `VOTING` 状態でない |
| 404 | セッション/議案が存在しない |
| 409 | 既に投票済み（変更は PUT /cast を使用） |

#### `PATCH /api/v1/proxy-votes/motions/{motionId}/start-vote`

**認可**: ADMIN / DEPUTY_ADMIN（MANAGE_VOTES）。`MEETING` モードのみ

議案の投票を開始する（PENDING → VOTING）。

**リクエストボディ**
```json
{
  "duration_seconds": 300

```

- `duration_seconds`: 任意。投票制限時間（秒）。指定時は `vote_deadline_at = NOW() + duration_seconds` を算出し、タイマー満了で自動 `end-vote`。未指定 = ADMIN が手動で `end-vote`

**レスポンス（200 OK）**: 議案詳細を返却（`voting_status: "VOTING"`, `vote_deadline_at` 含む）

**副作用**:
1. 出席メンバーにプッシュ通知（「第○号議案の投票が開始されました」）
2. WebSocket: `MOTION_VOTE_STARTED` イベントを配信（フロントエンドで投票 UI + カウントダウンタイマーを即表示）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `WRITTEN` モードのセッション |
| 403 | 権限不足 |
| 409 | セッションが OPEN でない / 議案が PENDING でない |

#### `PATCH /api/v1/proxy-votes/motions/{motionId}/end-vote`

**認可**: ADMIN / DEPUTY_ADMIN（MANAGE_VOTES）。`MEETING` モードのみ

議案の投票を終了する（VOTING → VOTED）。委任票を加算して result を自動判定する。

**リクエストボディ**: なし

**レスポンス（200 OK）**
```json
{

    "voting_status": "VOTED",
```

**副作用**:
1. 未投票の出席者は棄権扱い（`ABSTAIN` で自動 INSERT）。出席者 = 投票権を持つメンバーのうち、委任状を提出していない（`proxy_delegations` に `ACCEPTED` レコードがない）メンバー

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `WRITTEN` モードのセッション |
| 403 | 権限不足 |
| 409 | 議案が VOTING でない |

```json
}
```

| ステータス | 条件 |
|-----------|------|
| 403 | 権限不足 |
| 409 | CLOSED 以外のステータス |

## 5. ビジネスロジック

### 投票セッションのライフサイクル

#### 共通フロー

```
1. ADMIN がセッション作成（DRAFT）
   a. resolution_mode を選択（MEETING or WRITTEN）
   b. 議案の追加・編集・削除が可能
   c. eligible_count をスコープ内メンバー数から自動算出してスナップショット

2. 受付開始（DRAFT → OPEN）
   a. PATCH /open で遷移。議案が1件以上存在することを検証
   b. 全メンバーにプッシュ通知（ApplicationEvent: VoteSessionOpenedEvent）
   c. voting_start_at が未来日時の場合はバッチで自動遷移（5分間隔）
   d. OPEN 後は議案の追加・削除不可（タイトル・説明の軽微な修正のみ可能）
```

#### WRITTEN モード（書面決議）

```
3w. 投票・委任の受付（OPEN 中）
    a. メンバーは全議案に一括投票（POST /cast）
    b. 出席できないメンバーは委任状を提出（POST /delegate）
    c. ADMIN が委任状を承認/却下
    d. 代理人が委任者の分を投票
    e. 投票変更は OPEN 中のみ可能（PUT /cast で上書き）
    f. OPEN 遷移時に全議案を voting_status = VOTING に一括変更

4w. 投票締切（OPEN → CLOSED）
    a. PATCH /close で手動遷移、または voting_end_at 到達時にバッチで自動遷移
    b. 全議案を voting_status = VOTED に一括変更
    c. CLOSED 後は投票・委任の追加/変更不可

5w. 結果確定（CLOSED → FINALIZED）
    a. PATCH /finalize で遷移
    b. 各議案の result を一括自動判定:
       - approve_count > reject_count + abstain_count → APPROVED（MAJORITY）
       - TWO_THIRDS: approve_count >= total_votes * 2/3
       - UNANIMOUS: reject_count == 0 AND abstain_count == 0
    c. 未投票の委任（代理人が投票しなかった白紙委任）は棄権扱い
    d. 監査ログ: VOTE_SESSION_FINALIZED を記録
```

#### MEETING モード（総会決議）

```
3m. 委任状の事前収集（OPEN 中）
    a. 出席できないメンバーが委任状を提出（POST /delegate）
    b. ADMIN が委任状を承認/却下
    c. この期間中は投票（POST /cast）は不可。投票は総会当日に行う

4m. 総会当日の議案別投票
    a. ADMIN が議案ごとに投票を開始:
       パターン A: 議案別投票
         - PATCH /motions/{id}/start-vote で1議案ずつ PENDING → VOTING
         - 出席者が投票（POST /cast。VOTING 状態の議案のみ）
         - 討論を踏まえて投票。WebSocket でリアルタイム集計表示
         - PATCH /motions/{id}/end-vote で VOTING → VOTED（result 自動判定）
         - 次の議案へ進む
       パターン B: 一括投票
         - PATCH /proxy-votes/{id}/start-all-votes で全議案を一括 VOTING
         - 出席者が全議案に一括投票（POST /cast）
         - ADMIN が各議案を end-vote で VOTED に遷移
    b. 委任票は end-vote 時に自動加算（代理人の投票内容を委任者に適用）
    c. 未投票の出席者は end-vote 時に棄権扱い

5m. 全議案完了後（OPEN → CLOSED → FINALIZED）
    a. 全議案が VOTED になった時点で CLOSED に遷移可能
    b. PATCH /finalize で FINALIZED。result は各議案の end-vote 時に確定済み
    c. 議事録を添付（POST /proxy-votes/{id}/attachments）
    d. 監査ログ: VOTE_SESSION_FINALIZED を記録
```

### 定足数チェック

```
1. 定足数 = 投票権を行使した人数（自己投票 + 委任）
   - 自己投票: proxy_votes の DISTINCT user_id（is_proxy_vote = FALSE）
   - 委任: proxy_delegations の status = 'ACCEPTED' かつ CANCELLED でない件数
   - 合計が quorum_required 以上であれば定足数充足

2. 定足数算出:
   - MAJORITY: CEIL(eligible_count / 2) + 1（過半数）
   - TWO_THIRDS: CEIL(eligible_count * 2 / 3)
   - CUSTOM: CEIL(eligible_count * quorum_threshold / 100)

3. 定足数未充足の場合:
   - FINALIZE 遷移時に警告を表示（強制的に FINALIZED にはできるが、結果に「定足数未達」ラベルを付与）
   - 議案の result は判定するが、「参考決議（定足数未達）」として記録
```

### 委任投票フロー

```
1. 委任者がPOST /delegate で委任状を提出
   ※ is_anonymous = TRUE のセッションでは 400 エラーで拒否（無記名×委任の排他制約）
   ※ MEETING モード: OPEN 中（総会前）に事前提出。WRITTEN モード: OPEN 中に投票と並行して提出可能
2. 承認フロー分岐:
   a. is_auto_accept_delegation = TRUE（自動承認）:
      status = 'ACCEPTED' で即座に INSERT。代理人にプッシュ通知
   b. is_auto_accept_delegation = FALSE（手動承認）:
      status = 'SUBMITTED' で INSERT。ADMIN がPATCH /delegations/{id}/review で承認（ACCEPTED）
3. ACCEPTED 後: 代理人にプッシュ通知（「○○さんからの委任を受けました」）
4. 代理人がPOST /cast で投票:
   a. 自分の投票 + 委任者の代理投票を同時に処理
   b. 代理投票は is_proxy_vote = TRUE、delegation_id で紐付け
   c. 複数人の委任を受けている場合、全員分を一括投票
5. 委任者が心変わりで自分で投票した場合:
   a. 委任状は自動的に CANCELLED に遷移
   b. 代理人が既に代理投票済みの場合: 代理投票を DELETE し、本人の投票を INSERT
   c. proxy_vote_motions のカウントをアトミック補正
```

### 自動遷移バッチ

```
1. @Scheduled で5分間隔で実行
2. DRAFT → OPEN 自動遷移（両モード共通）:
   - 条件: status = 'DRAFT' AND voting_start_at <= NOW() AND voting_start_at IS NOT NULL
3. OPEN → CLOSED 自動遷移（WRITTEN モードのみ）:
   - 条件: status = 'OPEN' AND resolution_mode = 'WRITTEN' AND voting_end_at <= NOW() AND voting_end_at IS NOT NULL
   - MEETING モードはバッチ対象外（投票は総会当日に ADMIN が制御。voting_end_at は委任受付締切であり、セッション自体のクローズとは独立）
4. MEETING モード委任受付自動締切:
   - 条件: status = 'OPEN' AND resolution_mode = 'MEETING' AND voting_end_at <= NOW() AND voting_end_at IS NOT NULL
   - 委任の新規受付を停止（POST /delegate を 409 で拒否）。ただしセッションは OPEN のまま維持（総会投票のため）
5. MEETING モード投票タイマー自動終了:
   - 条件: voting_status = 'VOTING' AND vote_deadline_at <= NOW() AND vote_deadline_at IS NOT NULL
   - 該当議案の end-vote を自動実行（未投票者は棄権扱い・委任票加算・result 判定）
   - WebSocket: MOTION_VOTE_ENDED を配信（フロントエンドでカウントダウン終了 → 結果表示に遷移）
6. バッチエラー時: ERROR ログ出力 + 次回実行で再試行
```

### 自動リマインド通知（WRITTEN モード）

```
1. @Scheduled バッチが1時間間隔で実行
2. 対象: resolution_mode = 'WRITTEN' AND status = 'OPEN' AND voting_end_at IS NOT NULL
3. 各セッションの remind_before_hours（デフォルト: [24]）を参照:
   - voting_end_at - NOW() が配列内のいずれかの時間以下になった時点で、未投票・未委任のメンバーに通知
   - 例: remind_before_hours = [72, 24] → 締切72時間前と24時間前に通知
4. 通知内容: 「[議決権行使] 投票期限が近づいています: {title}（期限: {voting_end_at}）」
5. 各リマインドポイントにつき1回のみ送信（重複防止）
   - proxy_vote_sessions に last_remind_hour INT nullable カラムを追加し、最後に通知した時間ポイントを記録
```

### 議事録 PDF エクスポート

```
1. GET /proxy-votes/{id}/minutes-pdf（FINALIZED セッションのみ。ADMIN / DEPUTY_ADMIN）
2. PDF 内容:
   a. セッション情報（タイトル・日時・定足数状況）
   b. 各議案: タイトル・説明・投票結果（賛成/反対/棄権の票数 + 可決/否決）
   c. 定足数充足状況（充足 or 未達 + 「参考決議」ラベル）
   d. 出席者リスト: 投票者 + 委任者（委任先を明記）
   e. is_anonymous = TRUE の場合: 投票者リストは「投票者○名」のみ表示（個人名なし）
3. 生成方式: Thymeleaf テンプレート → Flying Saucer で PDF 変換（F07.4 カルテ PDF と同一アーキテクチャ）
4. タイムアウト: 10秒（写真なしのため短め）
```

### 重要な判定ロジック

- **投票権**: MEMBER 以上（SUPPORTER / GUEST は閲覧のみ）
- **議案編集**: DRAFT 中のみ追加/削除可能。OPEN 中はタイトル・説明の軽微修正のみ
- **投票可能タイミング**: `WRITTEN` = OPEN 中に全議案投票可能。`MEETING` = 各議案が `VOTING` 状態の間のみ投票可能（ADMIN が開始/終了を制御）
- **投票変更**: `WRITTEN` = OPEN 中のみ。`MEETING` = 各議案が `VOTING` 状態の間のみ
- **MEETING モードの投票禁止**: OPEN 中でも議案が `PENDING` / `VOTED` の場合は投票不可。ADMIN が `start-vote` するまで待機
- **委任と投票の排他**: 同一セッションで投票と委任は同時に行えない。投票した場合は委任が自動取消、委任中に自分で投票した場合も委任が自動取消
- **無記名投票**: `is_anonymous = TRUE` の場合、API レスポンスで個人の投票内容は一切返却しない。DB にはレコードが存在するが、監査目的でのみ保持。**委任は禁止**（`delegation_id` → `proxy_delegations.delegator_id` の結合で投票内容が逆引き可能になり、無記名の原則が崩壊するため。法的にも無記名投票と委任を同時に要求するケースは NPO 法・区分所有法等で実務上ほぼない）
- **結果確定後の変更禁止**: FINALIZED 後はセッション・議案・投票の全データが変更不可（議事録としての証跡保全）。ただし添付ファイル（`MINUTES` 種別）とコメントは FINALIZED 後も追加可能
- **投票タイマー**: `MEETING` モードの `start-vote` 時に `duration_seconds` を指定すると、`vote_deadline_at` を算出してバッチで自動 `end-vote`。タイマーなしの場合は ADMIN が手動で `end-vote`
- **投票確認通知**: 投票者本人に投票内容のサマリーを通知。代理投票時は委任者にも「代理投票が完了しました」を通知
- **議案コメント**: OPEN 中のみ投稿可能。MEMBER 以上。通知は F05.2 のスレッド参加者パターン（作成者 + 過去にコメント済みユーザー）
- **セッション複製**: 議案構成・設定のみコピー。投票データ・委任状・添付・`eligible_count` はコピーしない

## 6. セキュリティ考慮事項

- **認可チェック**: 全 API エンドポイントでスコープメンバーシップとロールを検証
- **無記名投票の保護**: `is_anonymous = TRUE` のセッションでは、DB の `proxy_votes.user_id` は監査用にのみ保持。API レスポンス・管理画面・CSV エクスポート等のいかなる経路でも個人の投票内容を返却しない。SQL 直接アクセス以外では確認不可能な設計。**委任（`POST /delegate`）を禁止**することで、`delegation_id` → `delegator_id` 結合による投票内容の逆引きリスクを根本排除
- **投票の改竄防止**: FINALIZED 後は全データの UPDATE / DELETE をアプリ層で厳密にブロック。DB 側には追加の制約を設けない（運用上の柔軟性を維持）
- **電子印鑑連携**: 委任状の `electronic_seal_id` で提出の正当性を担保。署名なしでも委任は有効だが、法的紛争時のエビデンスとしては署名ありが優位
- **監査ログ**: 以下のイベントを監査ログに記録
  - `VOTE_SESSION_CREATED`: セッション作成
  - `VOTE_SESSION_OPENED`: 投票受付開始
  - `VOTE_SESSION_CLOSED`: 投票締切
  - `VOTE_SESSION_FINALIZED`: 結果確定（各議案の result を含む）
  - `VOTE_CAST`: 投票（無記名セッションでは user_id を記録しない）
  - `DELEGATION_SUBMITTED`: 委任状提出
  - `DELEGATION_REVIEWED`: 委任状承認/却下
- **レートリミット**: `Bucket4j` で以下の制限を適用
  - 投票 API（`POST /cast`）: 1分間10件
  - 委任状提出（`POST /delegate`）: 1分間5件
  - リマインド送信（`POST /remind`）: 1セッションあたり1時間に1回
  - コメント投稿（`POST /comments`）: 1分間20件
  - セッション複製（`POST /clone`）: 1分間3件
  - CSV エクスポート（`GET /results/csv`）: 1分間5件
- **IDOR 防止**: `PUT /cast` は投票者本人チェック必須。`DELETE /delegate` は委任者本人のみ。`DELETE /comments/{id}` は投稿者本人 or ADMIN
- **CSV エクスポートの情報漏洩防止**: `is_anonymous = TRUE` のセッションでは集計行のみを出力。個人の投票データは含めない
