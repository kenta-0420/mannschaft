## 1. 概要

「思いついたことを最小ステップで投げ込む」ためのメモ機能。題名のみで投稿でき、後でじっくり見返して必要なものだけ TODO に昇格させる**二段階フロー**を提供する。

### 設計思想
- **入力摩擦ゼロ**: 題名以外すべて任意。考える前に書ける。
- **整理は後回しでよい**: カテゴリ分けや優先度設定を強制しない。
- **必要なものだけ TODO へ昇格**: メモ→TODO への一方通行の導線で、思いつきを行動に変える。
- **個人専用**: チーム共有はスコープ外。共有したくなったメモは TODO に昇格してから割り当てる。

ADHD 等の特性により「思いついた瞬間に記録しないと忘れる」「整理を強制されると入力しなくなる」ユーザーを主要ターゲットとする。汎用ユーザーにも「日常のひらめきメモ」として有用。

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `quick_memos` | メモ本体（題名・内容・状態） | あり（`deleted_at`） |
| `quick_memo_attachments` | 画像添付 | なし（メモ削除時 CASCADE） |
| `tags` | **汎用タグマスタ**（ポイっとメモ・TODO 共通。スコープ単位で管理） | なし |
| `quick_memo_tag_links` | メモとタグの中間テーブル | なし |
| `todo_tag_links` | **TODO とタグの中間テーブル**（F02.3 拡張） | なし |
| `user_quick_memo_settings` | ユーザー単位のリマインドデフォルト設定 | なし |
| `user_voice_input_consents` | **音声入力の同意履歴**（NH4 対応・GDPR 同意証跡） | なし |
| `pending_uploads` | **Presigned URL 発行履歴**（NH6 対応・重複/容量制御） | なし（confirm 後 or 期限切れで物理削除） |
| `withdraw_jobs` | **退会 SAGA 進捗管理**（NC2 対応） | なし |

> **設計上の重要決定**: タグはポイっとメモ専用ではなく**汎用テーブル `tags` として実装**し、TODO 機能（F02.3）にもタグ機能を同時に追加する。これにより:
> - メモ→TODO 昇格時にタグをそのまま引き継げる
> - ユーザーは「アイデア」「買い物」「仕事」などのタグを一度作れば、メモにも TODO にも使える
> - チーム TODO ではチーム共有タグ、個人スコープでは個人タグ、と自然にスコープ分離できる

### テーブル定義

#### `quick_memos`

メモ本体。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `title` | VARCHAR(200) | NO | — | 題名（必須。空文字なら API 層で `無題メモ_YYYYMMDD-HHMM` を自動生成） |
| `body` | TEXT | YES | NULL | 内容（任意。Markdown は保存するが**プレビューはしない**＝書き殴り前提） |
| `status` | VARCHAR(20) | NO | 'UNSORTED' | `UNSORTED`（未整理）/ `ARCHIVED`（アーカイブ済み）/ `CONVERTED`（TODOに昇格済み） |
| `converted_to_todo_id` | BIGINT UNSIGNED | YES | NULL | FK → todos。ON DELETE SET NULL。昇格先の TODO ID |
| `converted_at` | DATETIME | YES | NULL | TODO 昇格日時 |
| `reminder_uses_default` | BOOLEAN | NO | TRUE | TRUE = ユーザーのデフォルト設定を使用 / FALSE = メモ個別の設定を使用 |
| `reminder_1_scheduled_at` | DATETIME | YES | NULL | 第1リマインドの送信予定日時（計算済み。NULL ならこの枠は使わない） |
| `reminder_1_sent_at` | DATETIME | YES | NULL | 第1リマインドの実送信日時（NULL なら未送信） |
| `reminder_2_scheduled_at` | DATETIME | YES | NULL | 第2リマインドの送信予定日時 |
| `reminder_2_sent_at` | DATETIME | YES | NULL | 第2リマインドの実送信日時 |
| `reminder_3_scheduled_at` | DATETIME | YES | NULL | 第3リマインドの送信予定日時 |
| `reminder_3_sent_at` | DATETIME | YES | NULL | 第3リマインドの実送信日時 |
| `user_timezone_at_creation` | VARCHAR(50) | NO | 'Asia/Tokyo' | 作成時のユーザータイムゾーン（NM3-3rd 対応・将来のグローバル展開時に既存メモのスケジュール再計算で使用） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_qm_user_status (user_id, status, created_at DESC)         -- ユーザー別の一覧（新しい順）
INDEX idx_qm_user_deleted (user_id, deleted_at DESC)                -- ゴミ箱一覧（H7 対応）
INDEX idx_qm_reminder_1 (status, reminder_1_scheduled_at, reminder_1_sent_at)  -- 第1リマインド対象抽出
INDEX idx_qm_reminder_2 (status, reminder_2_scheduled_at, reminder_2_sent_at)  -- 第2リマインド対象抽出
INDEX idx_qm_reminder_3 (status, reminder_3_scheduled_at, reminder_3_sent_at)  -- 第3リマインド対象抽出
INDEX idx_qm_converted_todo (converted_to_todo_id)                  -- 昇格先 TODO からの逆引き
```

**インデックス補足（H6 対応）**
- リマインド系インデックスはカラム順 `(status, scheduled_at, sent_at)` で範囲スキャン後に `sent_at IS NULL` を絞り込む
- `deleted_at IS NULL` 条件はインデックスに含めない。理由: `status='UNSORTED'` の論理削除済みメモは事実上発生しない（削除時に DELETE→`deleted_at` がセットされ、status は遷移しない）が、念のため WHERE 句では明示する

**制約・備考**
- `user_id` ON DELETE CASCADE: ユーザー退会時にメモも削除（個人データのため）
- **`title` 自動補完（NL8 / L1-3rd 対応 i18n）**: API 層で空文字 / 空白のみの場合、`{無題メモ_prefix}_YYYYMMDD-HHMM` を自動セット。プレフィックス文字列は **リクエストヘッダの `Accept-Language` の最初の言語タグ（プライマリ）から決定**:
  - パース処理: `Accept-Language: ja-JP,ja;q=0.9,en;q=0.8` → `Locale.lookup()` で最も優先度の高い対応言語を選択
  - サポートする言語タグ（プロジェクトの 6 言語と一致）:
    - `ja`, `ja-JP` → `無題メモ`
    - `en`, `en-US`, `en-GB` → `Untitled Memo`
    - `zh`, `zh-CN`, `zh-Hans`, `zh-Hant`, `zh-TW` → `无标题备忘录`（簡体字に統一）
    - `ko`, `ko-KR` → `제목 없는 메모`
    - `es`, `es-ES`, `es-MX` → `Memo sin título`
    - `de`, `de-DE`, `de-AT` → `Memo ohne Titel`
  - マッチしない言語は **デフォルトで `ja`**（プロジェクトのデフォルトロケール）
  - サーバー側マップは `QuickMemoUntitledPrefixResolver` クラスに集約
  - 空タイトルでも保存できるが、データ層では NOT NULL を維持
- **`status` 遷移**:
  - `UNSORTED` → `ARCHIVED`（手動アーカイブ）
  - `UNSORTED` → `CONVERTED`（TODO 昇格時に自動）
  - `ARCHIVED` → `UNSORTED`（復元）
  - `CONVERTED` → 戻れない（TODO 削除しても元メモは復活しない。`converted_to_todo_id` だけ NULL になる）
- **TODO 昇格時の挙動**: `quick_memos.title` → `todos.title`、`quick_memos.body` → `todos.description` にコピー。元メモは `status = 'CONVERTED'` に更新し、削除はしない（履歴として残す）
- **未整理メモ上限**: 1ユーザーあたり `UNSORTED` ステータスのメモは最大 **500 件**（アプリ層で検証。`ARCHIVED` / `CONVERTED` は上限に含めない）。**ADHD 配慮の文言（L3 対応）**: 上限到達時は「整理してください」ではなく以下のメッセージを返す
  ```
  「未整理メモが多くなっています 🌱
   いくつかを TODO に昇格 / アーカイブすると、新しいメモを保存できます」
   [未整理メモを開く]  [一括アーカイブ]
  ```
  450 件超えたら一覧上部に黄色バナーで早めに気づかせる
- **物理削除のタイミング**: `deleted_at IS NOT NULL AND deleted_at < NOW() - INTERVAL 90 DAY` を日次バッチで物理削除（誤削除復元の猶予 90 日）。物理削除バッチの詳細は §11 データ保護を参照
- `converted_to_todo_id` ON DELETE SET NULL（H4 対応）: F02.3 の TODO は **論理削除のみ**（physical delete は退会・F12.3 GDPR 削除リクエスト時のみ）であることを前提とする。F02.3 の TODO 論理削除時はメモ側は変化しない（昇格履歴を保持）。F02.3 で物理削除が発生する場合のみ FK 制約により `converted_to_todo_id` が NULL に戻る
- **物理削除と S3 添付の整合（C8 対応）**: メモの物理削除は専用バッチ `QuickMemoPhysicalDeleteBatchService` で実施し、必ず以下の順序で行う:
  1. 対象メモに紐付く `quick_memo_attachments.s3_key` を SELECT
  2. `S3ObjectDeleteEvent(s3Keys)` を発行（既存パターン）
  3. メモを物理削除（FK CASCADE で `quick_memo_attachments` も削除）
  これにより S3 オブジェクトが孤児化しない
- **リマインドのスケジュール解決**:
  - メモ作成時 / リマインド設定変更時に `reminder_1_scheduled_at` 〜 `reminder_3_scheduled_at` をバックエンドで計算して保存
  - **タイムゾーン規約（NM3 対応）**: 全計算は **`Asia/Tokyo` (UTC+9) 固定**。Mannschaft は日本国内向けサービスのため、ユーザー個別のタイムゾーン設定は持たない。バッチ cron も同タイムゾーンで実行
  - 計算疑似コード:
    ```java
    String tz = memo.getUserTimezoneAtCreation();  // 通常 "Asia/Tokyo"
    ZonedDateTime created = createdAt.atZone(ZoneId.of(tz));
    LocalDate baseDate = created.toLocalDate();
    LocalDateTime scheduled = baseDate.plusDays(dayOffset).atTime(reminderTime);  // reminderTime: LocalTime
    Instant scheduledInstant = scheduled.atZone(ZoneId.of(tz)).toInstant();
    // DB には UTC で保存（DATETIME 列）
    ```
  - **将来のグローバル展開対応（M1-3rd 先制策）**: `quick_memos.user_timezone_at_creation` カラムを最初から保持。将来 `users.timezone` を追加してもメモ作成時のタイムゾーンが固定で残るため、既存メモの再計算が不要
  - `reminder_uses_default = TRUE` の場合: ユーザーの `user_quick_memo_settings` の `default_offset_N_days` / `default_time_N` を使用
  - `reminder_uses_default = FALSE` の場合: メモごとに API 呼び出し時に渡された `reminders` 配列を使用
  - 不要な枠は `scheduled_at = NULL`（最大3枠だが、ユーザーが「2回でいい」と設定すれば3枠目は NULL）
- **リマインド送信ロジック**:
  1. バッチが起動するたびに以下のクエリで対象を抽出（3枠を UNION）
     ```sql
     -- 第1枠
     SELECT id, user_id, 1 AS slot FROM quick_memos
     WHERE status = 'UNSORTED' AND deleted_at IS NULL
       AND reminder_1_sent_at IS NULL
       AND reminder_1_scheduled_at IS NOT NULL
       AND reminder_1_scheduled_at <= NOW()
     UNION ALL
     -- 第2枠・第3枠も同様
     ```
     **タイトルは SELECT に含めない（H2 / プライバシー対応）**。バッチは件数のみ集約し、通知本文にメモタイトルを混入させない
  2. 該当行ごとにプッシュ通知を送信し、対応する `reminder_N_sent_at = NOW()` を更新
  3. ユーザーが `ARCHIVED` / `CONVERTED` / 削除した時点で自動的に対象から外れる（`status = 'UNSORTED'` 条件）
  4. **集約ルール（M8 対応）**: 「同一バッチ実行内で抽出された同一ユーザーの全レコード」を 1 通に集約する。バッチ間隔は 30 分なので、同一ユーザーへの通知は最大でも 30 分に 1 回。集約通知本文は固定文言「**未整理のメモが N 件あります**」のみ（タイトル混入禁止）
  5. **送信失敗時のリトライ**: `NotificationService` 側の責務に委譲。`reminder_N_sent_at` は通知送信成功時のみ更新。失敗時は次回バッチで再試行

#### `quick_memo_attachments`

メモへの画像添付。1メモあたり最大 5 枚。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `memo_id` | BIGINT UNSIGNED | NO | — | FK → quick_memos。ON DELETE CASCADE |
| `s3_key` | VARCHAR(500) | NO | — | S3 オブジェクトキー。**user_id を含めない**（C5 対応）。形式: `quick-memo/{yyyymm}/{uuidv4}.{ext}` 例: `quick-memo/202604/a1b2c3d4-e5f6-4710-8b80-0123456789ab.jpg` |
| `original_filename` | VARCHAR(255) | YES | NULL | アップロード時のファイル名。NULL バイト・制御文字を除去してから保存 |
| `content_type` | VARCHAR(50) | NO | — | サーバー側で**マジックバイト検証後**に確定した MIME タイプ。クライアント送信値は信用しない |
| `file_size_bytes` | INT UNSIGNED | NO | — | ファイルサイズ（バイト）。S3 から取得した実サイズを保存 |
| `width_px` | SMALLINT UNSIGNED | YES | NULL | 画像幅（ImageConverter で取得） |
| `height_px` | SMALLINT UNSIGNED | YES | NULL | 画像高さ |
| `sort_order` | TINYINT UNSIGNED | NO | 0 | 表示順（0 〜 4） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_qma_memo_order (memo_id, sort_order)
```

**制約・備考**
- 1メモあたり最大 **5 枚**（アプリ層で検証）
- 1ファイルあたり最大 **10 MB**（Presigned URL 発行時に Content-Length 制限）
- **Content-Length 強制の実装方法（NC4 / NHH6 対応）**: Presigned URL は署名済みなら有効期限内で任意サイズの PUT が成功してしまうため、**confirm 時のサーバー側 HEAD 検証を真の防御層**とする。AWS SDK v2 の `presignPutObject` の `contentLength` パラメータは PUT リクエスト時の `Content-Length` ヘッダ値を署名スコープに含めないため、これだけでは強制できない。
  
  **二段構え（confirm 時の HEAD が必須）**:
  1. **発行時の事前チェック**: `pending_uploads` テーブルに `declared_size_bytes` を保存。10MB 超なら presign API を 400 で拒否
     ```java
     if (req.getDeclaredSize() > MAX_FILE_SIZE_BYTES) {
         throw new BusinessException(QM_ATTACHMENT_TOO_LARGE);
     }
     ```
  2. **発行時の S3 制約付与（ベストエフォート）**: `presignPutObject` で `contentType` を固定。これは S3 側のリクエスト署名検証に影響する場合があり、ない場合でも次の confirm 検証で確実に防げる
  3. **confirm 時のサーバー側 HEAD 検証（必須・真の防御）**:
     ```java
     HeadObjectResponse head = s3Client.headObject(b -> b
         .bucket(bucket)
         .key(s3Key)
     );
     if (head.contentLength() > MAX_FILE_SIZE_BYTES) {
         s3Client.deleteObject(b -> b.bucket(bucket).key(s3Key));
         pendingUploadRepo.deleteByS3Key(s3Key);
         throw new BusinessException(QM_ATTACHMENT_TOO_LARGE);
     }
     // 宣言サイズと実サイズの大幅な乖離も拒否（10% 以上のずれは異常）
     if (Math.abs(head.contentLength() - pendingUpload.getDeclaredSizeBytes()) > pendingUpload.getDeclaredSizeBytes() * 0.1) {
         s3Client.deleteObject(...);
         throw new BusinessException(QM_ATTACHMENT_SIZE_MISMATCH);
     }
     ```
- このフローにより、署名付き URL を悪用しても confirm を経ずには `quick_memo_attachments` に登録されないため、孤児ファイルは 5 分後に `PendingUploadCleanupBatchService` で削除される
- **許可 MIME**: `image/jpeg`, `image/png`, `image/webp`, `image/gif` のみ。`image/svg+xml` は **明示的に拒否**（C6 対応 / SVG XSS 防止）
- **マジックバイト検証（C6 必須）**: `confirm` API で以下を実施
  1. S3 から AWS SDK で先頭バイトを取得
  2. ファイルシグネチャ照合
     - JPEG: `FF D8 FF`
     - PNG: `89 50 4E 47 0D 0A 1A 0A`
     - WebP: `52 49 46 46 ?? ?? ?? ?? 57 45 42 50`
     - GIF: `47 49 46 38 (37|39) 61`
  3. シグネチャ不一致なら S3 オブジェクトを削除しエラー（400 Bad Request）
  4. `ImageIO.read()` で読み込めることを確認（破損・decompression bomb 検出）
  5. 検証済み MIME タイプを `content_type` に保存（クライアント送信値は使用しない）
- **S3 キーの構造（C5 対応）**: `quick-memo/{yyyymm}/{uuidv4}.{ext}` で user_id を含めない。アクセス制御は IAM + アプリ層で二重化
- **EXIF データの扱い**: アップロード時に `ImageConverter` で EXIF を**全削除**してから保存（位置情報・端末情報の漏洩防止）
- **S3 アップロード**: 既存の `S3StorageService` を再利用。Presigned URL 方式（有効期限 5 分、Content-Length / Content-Type 制約付き）
- **メモ削除時の S3 削除**: メモ論理削除時点では S3 オブジェクトは保持（90 日復元猶予のため）。物理削除バッチで `S3ObjectDeleteEvent` を発行（C8 / M7 対応・§11 参照）
- **画像のリサイズ**: アップロード時に長辺 1920px に縮小（既存 `ImageConverter` を使用）。サムネイルは CDN 側オンザフライリサイズで対応（既存パターン踏襲）
- **配信時の Content-Disposition**: `inline; filename="*.jpg"` で配信。HTML として解釈されないよう `X-Content-Type-Options: nosniff` ヘッダを必須とする
- **配信時のキャッシュ制御（NM6 / M5-3rd 対応）**:
  - `Cache-Control: private, max-age=86400`（24 時間。private は CDN 共有キャッシュ禁止 — 個人画像のため）
  - `ETag`: S3 から自動付与。**画像のリサイズ・更新時は新しい `s3_key` を発行**（同一キーを上書きしない）し、フロントエンドのキャッシュとの整合性を維持
  - 既存 `CachePolicy.resolve(s3Key)` を再利用（プロジェクト規約踏襲）
- **CORS（NM6 対応）**:
  - 画像配信ドメインは `frontend.mannschaft.app` のみ許可（既存 CORS 設定踏襲）
  - Presigned URL の S3 ドメインは `Access-Control-Allow-Origin` を S3 バケット側の CORS ポリシーで制御

#### `user_quick_memo_settings`

ユーザー単位のリマインドデフォルト設定。3枠まで自由なオフセット日数 + 時刻を指定可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `user_id` | BIGINT UNSIGNED | NO | — | PK / FK → users。ON DELETE CASCADE |
| `reminder_enabled` | BOOLEAN | NO | FALSE | リマインド機能の ON/OFF。**初期は OFF**（ユーザーが明示的に有効化） |
| `default_offset_1_days` | SMALLINT UNSIGNED | YES | 3 | 第1リマインド: メモ作成から N 日後（1〜90 日。NULL ならこの枠を使わない） |
| `default_time_1` | TIME | YES | '09:00:00' | 第1リマインドの送信時刻 |
| `default_offset_2_days` | SMALLINT UNSIGNED | YES | 5 | 第2リマインド: N 日後 |
| `default_time_2` | TIME | YES | '09:00:00' | 第2リマインドの送信時刻 |
| `default_offset_3_days` | SMALLINT UNSIGNED | YES | 10 | 第3リマインド: N 日後 |
| `default_time_3` | TIME | YES | '09:00:00' | 第3リマインドの送信時刻 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**制約・備考**
- **初期化タイミング（H5 対応）**: 以下の 2 段構えで NPE を防ぐ:
  1. **ユーザー登録時に自動 INSERT**: F01.1 認証機能のサインアップ完了フックで `UserQuickMemoSettings.createDefault(userId)` を呼び出し、`reminder_enabled=FALSE` + 上記デフォルト値で行を作成
  2. **メモ作成 API のフォールバック**: 既存ユーザー対策として `findById(userId).orElseGet(() -> createDefault(userId))` で取得時に存在しなければ自動生成
- **デフォルト値のセマンティクス**: ユーザーが特に設定しなくても「3日後・5日後・10日後の朝9時」という汎用的なリマインドプランが入る
- **枠の使い方**: 各 `default_offset_N_days` を NULL にすればその枠は使用しない。例: 第1枠だけ 7 日後、第2枠 NULL、第3枠 NULL → 7日後に1回だけリマインド
- **オフセット順序の制約（M9 対応）**: アプリ層で以下を検証:
  - 非 NULL 値のみを抽出し、昇順かつ重複なし
  - パターン例:
    - `[3, 5, 10]` → OK
    - `[3, NULL, 10]` → OK（第2枠を使わない）
    - `[NULL, 5, 10]` → OK
    - `[5, 3, NULL]` → NG（順序逆）
    - `[3, 3, 5]` → NG（重複）
  - 実装疑似コード:
    ```java
    List<Integer> nonNull = Stream.of(off1, off2, off3)
        .filter(Objects::nonNull).collect(toList());
    for (int i = 1; i < nonNull.size(); i++) {
        if (nonNull.get(i) <= nonNull.get(i-1))
            throw new ValidationException("Reminder offsets must be strictly ascending");
    }
    ```
- **デフォルト変更時の挙動**: 設定変更時、フロントエンドが**確認ポップアップ**を表示し、ユーザーに以下から選択させる:
  - **`NEW_ONLY`「今後のメモのみ適用」** (デフォルト選択) — 既存メモは現状維持
  - **`UNSENT`「未送信リマインドにも反映」** — 既存の `UNSORTED` メモのうち、まだ送信されていない枠 (`reminder_N_sent_at IS NULL`) のスケジュールを新設定で再計算
  - **`ALL`「すべてのメモに反映」** — 送信済み枠も含めて全リマインドを再計算
- **再計算時の安全策（C7 対応）**:
  - 再計算後 `scheduled_at < NOW()` の枠は **送信対象としない**（過去日時の即時送信を防止）
  - `apply_to=ALL` でも `reminder_N_sent_at` は **クリアしない**（既送信のリマインドが再送信されるのを防止）
  - 結果として `ALL` モードでも「未来日時に再計算された未送信枠のみ」が新スケジュールで届く
- **トランザクション分離レベル（H3 対応）**: 設定更新と既存メモ再計算は同一トランザクションで `@Transactional(isolation = SERIALIZABLE)` を使用。設定変更中に新規メモ作成が走った場合のレース条件を防ぐ
- **バッチとの並行性対策（NH5 / NHH5 対応）**: SERIALIZABLE は単一トランザクション内のみ保護するため、別プロセスのリマインドバッチが古い `scheduled_at` を読んで二重送信するリスクが残る。以下で対処:
  1. `apply_to=UNSENT` / `ALL` の再計算時、対象行に対して `SELECT ... FOR UPDATE` で行ロックを取ってから `reminder_N_scheduled_at` を更新
  2. 同時にバッチ側も対象行を `SELECT ... FOR UPDATE SKIP LOCKED` で取得し、ロック競合中の行は次回バッチに送る
  3. 結果として「再計算中の行はバッチが触れない」「バッチ送信中の行は再計算されない」という相互排他が成立
  4. 補足: `apply_to=ALL` でも既送信枠 (`reminder_N_sent_at IS NOT NULL`) はクリアしないため、二重送信は構造的に発生しない
- **Deadlock リトライ戦略（NHH5 対応）**: SERIALIZABLE + `FOR UPDATE` 併用時に Deadlock が発生し得るため、Spring Retry で自動リトライ:
  ```java
  @Retryable(
      retryFor = { DeadlockLoserDataAccessException.class, CannotAcquireLockException.class },
      maxAttempts = 3,
      backoff = @Backoff(delay = 100, multiplier = 2, maxDelay = 1000)
  )
  @Transactional(isolation = Isolation.SERIALIZABLE)
  public void updateSettings(UpdateSettingsRequest req) { ... }
  ```
  - 100ms → 200ms → 400ms の指数バックオフで最大 3 回リトライ
  - すべてのリトライが失敗した場合は 503 Service Unavailable + `Retry-After: 5` を返却
  - 同パターンを `convertToTodo` / `deleteMemo` / `updateMemo` にも適用
- API 側では `PUT /api/v1/quick-memos/settings` に `apply_to` クエリ (`NEW_ONLY` / `UNSENT` / `ALL`) を持たせる
- リマインドバッチは `quick_memos.reminder_*_scheduled_at` を直接見るため、本テーブルはあくまでメモ作成時のテンプレート役

#### `pending_uploads`（NH6 / NCC1 対応）

Presigned URL の発行履歴と容量・重複制御。confirm 完了 or 期限切れで物理削除される短命テーブル。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `memo_id` | BIGINT UNSIGNED | NO | — | FK → quick_memos。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `s3_key` | VARCHAR(500) | NO | — | 発行された S3 オブジェクトキー |
| `declared_size_bytes` | INT UNSIGNED | NO | — | クライアント宣言サイズ（≤ 10485760） |
| `content_type` | VARCHAR(50) | NO | — | クライアント宣言 MIME（許可リストのみ） |
| `presigned_url_expires_at` | DATETIME | NO | — | 発行 URL の有効期限（発行から 5 分後） |

| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql

INDEX idx_pu_memo_pending (memo_id, confirmed_at)                  -- 同一メモの未 confirm 検索
INDEX idx_pu_user_created (user_id, created_at)                    -- 1時間100MB集計用
INDEX idx_pu_expires (presigned_url_expires_at, confirmed_at)      -- 孤児クリーンアップ
```

**制約・備考**
- **「1 メモあたり同時有効な Presigned URL は 1 個」原則（NH6 確定）**: 新規 presign 発行時、`memo_id` の `confirmed_at IS NULL AND presigned_url_expires_at > NOW()` の行があれば、その S3 オブジェクトを削除して `pending_uploads` 行も削除してから新規発行。実装上はトランザクション内で SELECT FOR UPDATE する
- **「1 ユーザーあたり 1 時間で発行できる総容量は 100MB まで」（NH6 確定）**: presign 発行時に `SELECT SUM(declared_size_bytes) FROM pending_uploads WHERE user_id = :uid AND created_at > NOW() - INTERVAL 1 HOUR` を計算し、合計 + 新規宣言サイズが 100MB 超なら 429 エラー。Redis を併用してもよいが、DB 集計でも十分（個人スコープ）
- **孤児クリーンアップバッチ** `PendingUploadCleanupBatchService`: 5 分ごとに `confirmed_at IS NULL AND presigned_url_expires_at < NOW()` を抽出し、S3 オブジェクトを削除して行を削除
- 「ブラックリスト方式」は採用しない（pending_uploads テーブル方式に確定）
- メモ削除時 / 退会時に CASCADE で削除される

**インデックス**
```sql

```

**制約・備考**
- フロントエンドの localStorage は **UI 最適化用のキャッシュ**でしかなく、認可判定にはサーバーの本テーブルを必ず参照（NHH1）
- **`version` の改ざん防止（M2-3rd 対応）**: バックエンドに `CURRENT_VOICE_POLICY_VERSION` 定数（初期値 1）を保持し、`POST` / `GET active` のリクエストで `version > CURRENT_VOICE_POLICY_VERSION` が来たら 400 エラーで拒否。これによりクライアントが「未来のバージョン」を捏造して既存同意を有効化することを防ぐ
- 同意 API: `POST /api/v1/me/voice-input-consents`（version 指定）。冪等性: 同一 user_id × version の有効同意があれば既存を返す
- アクティブ確認 API: `GET /api/v1/me/voice-input-consents/active?version=1` → `{ hasConsent: true/false, version: 1, consented_at: ... }`
- 撤回 API: `DELETE /api/v1/me/voice-input-consents/active`（`UPDATE SET revoked_at = NOW()`）
- 音声入力ボタンが押された時、フロントエンドはまずサーバーに `GET /api/v1/me/voice-input-consents/active` を呼んで `revoked_at IS NULL AND version = 現在のポリシー版` の同意があるか確認
- 同意がなければダイアログを表示し、「同意」押下後にサーバー POST → 成功してから音声認識を開始
- localStorage は 24h キャッシュ程度として併用（毎回サーバーに問い合わせない最適化）
- F12.3 退会時に CASCADE で削除される（個人データのため）
- ポリシー改訂時は `version` をインクリメントし、既存同意は自動的に無効化（バージョン不一致）

### 共通認可規約（C1 / NHH2 対応・必須）

**全エンドポイントに適用される事前チェック（必須順序）**:

1. **退会状態チェック（NHH2 必須）**:
   ```java
   // SecurityFilter または共通インターセプタで全リクエスト処理前に実行
   public static void checkNotWithdrawn() {
       User user = getCurrentUser();
       if (user.getDeletedAt() != null) {
           throw new AuthorizationException("USER_WITHDRAWN", HttpStatus.GONE);  // 410
       }
       // 退会 SAGA 進行中もブロック
       if (withdrawJobRepo.existsByUserIdAndStatus(user.getId(), "IN_PROGRESS")) {
           throw new AuthorizationException("WITHDRAWAL_IN_PROGRESS", HttpStatus.CONFLICT);  // 409
       }
   }
   ```
   - 全 `/quick-memos/*` および `/me/tags/*` `/me/voice-input-consents/*` の **書き込み系 API** で必須
   - GET 系も「退会した本人の閲覧」は禁止（個人データへのアクセスを停止）
   - これにより SAGA Step 4（quick_memos 物理削除中）に新規メモ作成が走るタイミング問題を構造的に排除
   
2. **所有権検証**（C1）:
- すべての `/quick-memos/{id}` 系エンドポイント（GET / PUT / DELETE / PATCH / POST 派生）で以下を必須:
  ```java
  // Service 層の冒頭で必ず実行
  QuickMemoEntity memo = repository.findByIdAndDeletedAtIsNull(memoId)
      .orElseThrow(() -> new ResourceNotFoundException(QM_NOT_FOUND));
  if (!memo.getUserId().equals(SecurityUtils.getCurrentUserId())) {
      throw new AuthorizationException(QM_FORBIDDEN);  // 403 Forbidden
  }
  ```
- `attachments/{attachmentId}` 系では追加で `attachment.memo_id == memo.id` を検証（パス改ざん防止）
- 削除済みメモ（`deleted_at IS NOT NULL`）への通常アクセスは **404 Not Found** を返却（存在自体を隠蔽）。ゴミ箱専用エンドポイント (`/trash`, `/{id}/undelete`) のみが削除済みを参照可能
- **SYSTEM_ADMIN ロールでも他人の個人メモは取得不可**（個人情報の最小権限原則）。本機能は完全に「自分のリソースのみ」
- 認可漏れ防止のため `@PreAuthorize("@quickMemoAccessGuard.canAccess(#memoId, authentication)")` のような **アスペクト型ガード** を全エンドポイントに適用することを推奨

```java
@Transactional
public void updateMemo(Long memoId, UpdateMemoRequest req) {
    // 1. 所有権検証（C1）
    QuickMemoEntity memo = repo.findByIdAndUserId(memoId, currentUserId())
        .orElseThrow(() -> new ResourceNotFoundException(QM_NOT_FOUND));

    // 2. リクエストの全 tag_id について再検証（C3）
    for (Long tagId : req.getTagIds()) {
        TagEntity tag = tagRepo.findById(tagId).orElseThrow(...);
        if (!tag.isPersonal() || !tag.getScopeId().equals(currentUserId())) {
            throw new AuthorizationException(TAG_FORBIDDEN);
        }
    }

    // 3. 既存タグと新タグの差分計算
    Set<Long> oldTagIds = tagLinkRepo.findTagIdsByMemoId(memoId);
    Set<Long> newTagIds = new HashSet<>(req.getTagIds());

    Set<Long> toRemove = Sets.difference(oldTagIds, newTagIds);
    Set<Long> toAdd    = Sets.difference(newTagIds, oldTagIds);
    // 不変タグ（intersection）は触らない

    // 4. 差分を原子的に反映
    for (Long tagId : toRemove) {
        tagLinkRepo.deleteByMemoIdAndTagId(memoId, tagId);
        tagRepo.decrementUsageCount(tagId);  // 個別 UPDATE
    }
    for (Long tagId : toAdd) {
        tagLinkRepo.insert(memoId, tagId);
        tagRepo.incrementUsageCount(tagId);  // 個別 UPDATE
    }

    // 5. メモ本体の他フィールド更新
    memo.update(req);
}
```

**バリデーション**: メモ作成と同様（題名・本文・tag_ids・reminders すべて）

**処理フロー（C3 / NHH3 対応・SERIALIZABLE トランザクション必須）**

```java
@Transactional(isolation = Isolation.SERIALIZABLE)
public ConvertResponse convertToTodo(Long memoId, ConvertToTodoRequest req) {
    // 1. checkNotWithdrawn() — 共通認可ガード（NHH2）
    
    // 2. メモを SELECT FOR UPDATE で取得（NHH3: 削除との競合排除）
    QuickMemoEntity memo = memoRepository
        .findByIdAndUserIdForUpdate(memoId, currentUserId())
        .orElseThrow(() -> new ResourceNotFoundException(QM_NOT_FOUND));
    
    // 3. 状態チェック
    if (memo.getDeletedAt() != null) {
        throw new ResourceNotFoundException(QM_NOT_FOUND);  // 削除済みメモは昇格不可
    }
    if ("CONVERTED".equals(memo.getStatus())) {
        throw new BusinessException(QM_ALREADY_CONVERTED);  // 409
    }
    
    // 4. 紐付くタグ ID を取得（メモ行ロック中なので削除トランザクションに先んじる）
    List<Long> tagIds = tagLinkRepo.findTagIdsByMemoId(memoId);
    
    // 5. 各タグを再検証（C3）
    List<Long> validTagIds = new ArrayList<>();
    for (Long tagId : tagIds) {
        TagEntity tag = tagRepo.findById(tagId).orElse(null);
        if (tag == null) continue;
        if (!tag.isPersonal() || !tag.getScopeId().equals(currentUserId())) {
            auditLogger.warn("TAG_SCOPE_VIOLATION", Map.of("tag_id", tagId, "memo_id", memoId));
            continue;  // スキップ
        }
        validTagIds.add(tagId);
    }
    
    // 6. todos テーブルに INSERT
    TodoEntity todo = todoService.createFromMemo(memo, req);
    
    // 7. todo_tag_links に検証済みタグを INSERT
    for (Long tagId : validTagIds) {
        todoTagLinkRepo.insert(todo.getId(), tagId);
    }
    
    // 8. 検証通過した各タグについて usage_count を個別に原子インクリメント（NC5）
    for (Long tagId : validTagIds) {
        tagRepo.incrementUsageCount(tagId);
        // ※ WHERE id IN (...) の一括更新は禁止（行ロック粗化と並行リクエスト時のカウント漏れ防止）
    }
    
    // 9. メモを CONVERTED に更新
    memo.setStatus("CONVERTED");
    memo.setConvertedToTodoId(todo.getId());
    memo.setConvertedAt(Instant.now());
    memoRepository.save(memo);
    
    return new ConvertResponse(memoId, todo.getId(), "CONVERTED");
}
```

**競合制御の鍵（NHH3）**:
- メモを `SELECT FOR UPDATE` で取得することで、同時並行する削除トランザクションは memo 行のロック解放まで待機
- 削除側も同様に `findByIdForUpdate` を使うことで、削除→昇格 / 昇格→削除のどちらが先でも整合する
- usage_count のインクリメント・デクリメントが二重実行されない

| `page` | int | 1 | |

**LIKE エスケープ仕様（H1 / NM5 必須対応）**:
- `q` パラメータの LIKE クエリでは `%`, `_`, `\` をエスケープしてから WHERE 句に渡す
- 共通 Util `LikeEscapeUtils.escape(String)` を `backend/src/main/java/com/mannschaft/app/common/util/LikeEscapeUtils.java` に追加し、検索系すべてで再利用

**実装例（必須）**:
```java
// common/util/LikeEscapeUtils.java
public final class LikeEscapeUtils {
    private LikeEscapeUtils() {}

    public static String escape(String input) {
        if (input == null) return null;
        return input
            .replace("\\", "\\\\")
            .replace("%",  "\\%")
            .replace("_",  "\\_");
    }
}

// QuickMemoRepository.java （Spring Data JPA）
@Query(value = "SELECT m FROM QuickMemoEntity m " +
               "WHERE m.userId = :userId " +
               "AND m.deletedAt IS NULL " +
               "AND (m.title LIKE :pattern ESCAPE '\\\\' " +
               "  OR m.body  LIKE :pattern ESCAPE '\\\\')")
List<QuickMemoEntity> searchByKeyword(
    @Param("userId") Long userId,
    @Param("pattern") String pattern   // 呼び出し側で "%" + LikeEscapeUtils.escape(q) + "%"
);
```

**注意点**:
- `ESCAPE '\\\\'` の `\\\\` は Java 文字列リテラルとして `\\`、SQL に渡る時点で `\` 1 文字
- バインド変数 `:pattern` 自体は Prepared Statement なので SQL Injection リスクなし
- ESCAPE 句は SQL リテラルなのでバインド変数化できない。固定文字列 `'\\\\'` をクエリに直接書く
- ユーザーが意図しないワイルドカード解釈を防ぎ、LIKE による情報漏洩リスクを排除

**レスポンス（200 OK）**: 各メモの基本情報 + ページング情報。削除済みは含まれない。`unsorted_count` フィールドで現在の未整理件数を含める（NL3 対応）

**実装要件（NH4 / NHH1 対応・サーバー側永続化必須）**:
- **同意は `user_voice_input_consents` テーブルに保存**（§3 参照）。**localStorage は信頼しない**（NHH1 重要修正）
- 音声入力ボタン押下時のフロー（必ずこの順序）:
  1. **必ずサーバーに `GET /api/v1/me/voice-input-consents/active?version={CURRENT_POLICY_VERSION}` を呼ぶ**
  2. レスポンスが `{ hasConsent: true, version: <現在版> }` の場合のみ音声認識を開始
  3. それ以外は同意ダイアログを表示し、ユーザーが「同意」を押したら `POST /api/v1/me/voice-input-consents { version: <現在版> }` を実行
  4. POST 成功後に音声認識を開始
- **localStorage はあくまで UX 補助のキャッシュ**: ダイアログを表示するか否かの初期判定材料としてのみ使用。**音声認識の起動可否はサーバーレスポンスにのみ依存**する。これにより F12 で localStorage を改ざんされても同意なしで起動することを防止
- **`version` の検証（NHH1 補強）**: サーバーは `CURRENT_POLICY_VERSION` の定数を持ち、リクエストの `version` がそれを超えていれば 400 エラー。クライアントから任意の version を指定して既存同意を有効化することを防ぐ
- POST API は冪等性あり: 同一 user_id × version で `revoked_at IS NULL` の行が既にあれば新規 INSERT せず既存を返す
- ポリシー改訂時は `CURRENT_POLICY_VERSION` をインクリメントし、全ユーザーに再同意を要求（過去の同意は version 不一致で無効扱い）
- 同意撤回 UI: 設定ページに「音声入力の同意を取り消す」リンク → `DELETE /api/v1/me/voice-input-consents/active`（`UPDATE` で `revoked_at = NOW()`）
- 録音中は「🎤 録音中... 話し終わったら自動停止します」のオーバーレイを表示（誤録音防止）
- 録音音声は **Mannschaft サーバーには送信しない**（テキスト変換結果のみメモ本文に追記）
- F12.3 GDPR ドキュメント・プライバシーポリシーに「Web Speech API 利用時の音声送信」を明記
- 同意証跡（user_agent, ip_address）は **GDPR 第 7 条**「同意の証明責任」に対応するための記録

### バッチ

#### 1. リマインドバッチ `QuickMemoReminderBatchService`

`@Scheduled(cron = "0 */30 * * * *")` で **30 分ごと** に起動。  
リマインド時刻は分単位指定不可（00 / 30 分のみ）。最大 30 分の遅延が許容される設計。

各起動時の処理:
1. `reminder_N_scheduled_at <= NOW() AND reminder_N_sent_at IS NULL AND status = 'UNSORTED' AND deleted_at IS NULL` を 3 枠分 UNION ALL で抽出（タイトルは SELECT しない）
2. ユーザー単位に集約（`Map<userId, count>`）
3. F04.3 `NotificationService` で固定文言「未整理のメモが N 件あります」を送信（タイトル混入禁止）
4. 送信成功した行の `reminder_N_sent_at = NOW()` を更新（個別の UPDATE で原子的に）
5. 大量メモ抱え込みユーザーの DoS 対策として `LIMIT 10000` を 1 回の起動あたりの上限とし、超過分は次回バッチで処理

#### 2. 物理削除バッチ `QuickMemoPhysicalDeleteBatchService`（C8 / M7 対応）

`@Scheduled(cron = "0 0 3 * * *")` で **毎日深夜 3 時** に起動。

処理フロー（必ずこの順序・NH3 対応）:
1. `deleted_at IS NOT NULL AND deleted_at < NOW() - INTERVAL 90 DAY` のメモを抽出（**LIMIT 50000**・NM7 対応）
2. 各メモに紐付く `quick_memo_attachments.s3_key` を SELECT して集める
3. **S3 オブジェクトを同期削除**（NH3 必須）
   - `S3StorageService.deleteObjects(s3Keys)` を**直接同期呼び出し**（イベント発行ではない）
   - もしくは `S3ObjectDeleteEventListener` を使う場合は `@TransactionalEventListener(phase = AFTER_COMMIT)` ではなく **`@EventListener` で同期処理**にする
   - S3 削除が失敗した s3_key はリトライキューに積み、当該メモは今回の物理削除対象から除外（次回バッチで再試行）
4. S3 削除成功を確認した後に **メモを物理削除**（`quick_memo_attachments` / `quick_memo_tag_links` は FK CASCADE で同時削除）
5. リンク削除に伴う `tags.usage_count` のデクリメントは、**削除直前に同期実行**（NM1 / NHH4 対応）。後続バッチ任せにしない:
   ```java
   // 重複を集計して、各タグごとにまとめてデクリメント
   Map<Long, Integer> decrementByTag = tagLinkRepo
       .findByMemoIdIn(memoIds)
       .stream()
       .collect(Collectors.groupingBy(TagLink::getTagId, Collectors.summingInt(l -> 1)));
   
   for (Map.Entry<Long, Integer> e : decrementByTag.entrySet()) {
       tagRepo.decrementUsageCountBy(e.getKey(), e.getValue());
       // SQL: UPDATE tags SET usage_count = usage_count - :delta WHERE id = :tagId
   }
   ```
   **重要（NHH4 対応）**: 複数のメモが同じタグを参照していた場合、個別 UPDATE を `delete count` 回繰り返すと正しく集計できないため、**先にメモ ID 集合に対して `tag_id ごとの参照件数` を集計**してから一度の UPDATE でまとめてデクリメントする
6. 整合性監視バッチ (`TagUsageCountReconcileBatchService`) は保険として動作（通常は不整合を検出しない想定）

#### 3. usage_count 整合性監視バッチ `TagUsageCountReconcileBatchService`（M2 / NC3 対応）

`@Scheduled(cron = "0 30 3 * * *")` で **毎日深夜 3:30** に起動。物理削除バッチの後に走らせる。

**重要（NC3 対応）**: マルチテナント環境での誤更新を防ぐため、**全テナント無条件 UPDATE は禁止**。以下の方式で限定的に再計算する:

**フェーズ 1: 不整合タグの検出**
```sql
SELECT t.id, t.scope_type, t.scope_id, t.usage_count,
       COALESCE(qm.cnt, 0) + COALESCE(td.cnt, 0) AS actual
FROM tags t
LEFT JOIN (SELECT tag_id, COUNT(*) cnt FROM quick_memo_tag_links GROUP BY tag_id) qm ON qm.tag_id = t.id
LEFT JOIN (SELECT tag_id, COUNT(*) cnt FROM todo_tag_links GROUP BY tag_id) td ON td.tag_id = t.id
WHERE t.usage_count <> COALESCE(qm.cnt, 0) + COALESCE(td.cnt, 0)
LIMIT 10000;  -- 1 回の起動で扱う最大件数
```

**フェーズ 2: 検出された tag_id を ID 単位で個別 UPDATE**
```java
for (TagMismatch m : mismatches) {
    tagRepository.setUsageCount(m.getId(), m.getActual());
    auditLogger.warn("USAGE_COUNT_RECONCILED", 
        Map.of("tag_id", m.getId(), "old", m.getCurrent(), "new", m.getActual()));
}
```

**安全策**:
- 1 回の起動で最大 10000 件まで（DoS 防止）
- 各 UPDATE は WHERE id = ? の単一行更新（テナント越境を構造的に防止）
- 検出件数が 100 件超なら ERROR ログ + 運用アラート（異常検知）
- 通常は **削除時の同期デクリメント（NM1）** で整合性が保たれているため、本バッチは保険として動作する

## 8. セキュリティ・認可仕様

### 8.1 認可マトリクス

| 操作 | 対象 | 認可ルール |
|------|------|-----------|
| メモ作成・取得・更新・削除 | `quick_memos` | `memo.user_id == currentUserId()`（C1） |
| 添付の追加・削除・取得 | `quick_memo_attachments` | `attachment.memo.user_id == currentUserId()` かつ `attachment.memo_id == pathMemoId` |
| 個人タグ CRUD | `tags (PERSONAL)` | `tag.scope_id == currentUserId()`（C2） |
| チームタグ CRUD | `tags (TEAM)` | チーム所属。作成・更新・削除は ADMIN / DEPUTY_ADMIN(MANAGE_TAG) |
| 組織タグ CRUD | `tags (ORGANIZATION)` | 組織所属。作成・更新・削除は ORGANIZATION_ADMIN |
| メモ→TODO 昇格 | `quick_memos` + `todos` + `todo_tag_links` | メモ所有 + 各タグ所有検証（C3） |
| リマインド設定 | `user_quick_memo_settings` | `setting.user_id == currentUserId()` |

### 8.2 IDOR 防止策（C1 / C2 / C3 対応）

1. **URL パスにスコープを埋め込む**: 個人タグは `/me/tags`、チームタグは `/teams/{teamId}/tags`。`scope_id` をクエリで受け付けない
2. **Service 層先頭で所有権検証**: すべての `*ById` メソッドで `if (resource.userId != currentUserId()) throw 403`
3. **AOP ガード**: `@PreAuthorize("@quickMemoAccessGuard.canAccess(#memoId, authentication)")` を Controller メソッドに付与
4. **タグ付与時の二重検証**: メモ更新 / 昇格時に `tag_ids` 各要素のスコープを毎回再検証
5. **削除済みリソース**: 通常 GET では `404 Not Found`（存在隠蔽）。ゴミ箱専用 API のみが見える

### 8.3 入力サニタイゼーション

| フィールド | 処理 |
|----------|------|
| `title` | 前後トリム / NULL バイト除去 / 制御文字除去（タブ・改行は許可） / 200 文字制限 |
| `body` | NULL バイト除去 / 10000 文字制限 / Markdown 保存はそのまま、表示時に DOMPurify |
| `tag.name` | 前後トリム / NULL バイト除去 / 制御文字・改行除去 / 30 文字制限 |
| `tag.color` | 正規表現 `^#[0-9A-Fa-f]{6}$` で厳格検証（H9） |
| `original_filename` | NULL バイト・パス区切り (`/`, `\`) 除去 / 255 文字制限 |
| `q`（検索） | LIKE エスケープ（H1 / `LikeEscapeUtils.escape`） / 100 文字制限 |
| `day_offset` | `@Min(1) @Max(90)` 整数のみ |
| `time` | `^([0-1][0-9]|2[0-3]):(00|30)$` |

### 8.4 ファイルアップロードのセキュリティ（C6）

| 対策 | 実装方法 |
|------|---------|
| MIME タイプ偽装防止 | サーバー側マジックバイト検証 |
| SVG XSS 防止 | `image/svg+xml` 拒否リスト |
| Decompression bomb 対策 | `ImageIO.read()` 失敗で破棄 / サイズ上限 10MB / ピクセル上限 **4000×4000**（NL5 対応：10MB JPEG の現実的最大に整合） |
| EXIF 漏洩防止 | アップロード時に EXIF を全削除 |
| パストラバーサル防止 | S3 キーは UUID 生成（クライアント値を使わない） |
| 画像配信時の XSS 防止 | `Content-Disposition: inline` + `X-Content-Type-Options: nosniff` |
| user_id 漏洩防止 | S3 キーから user_id を除外（C5） |

## 9. データ保護・GDPR 対応

### 9.1 ライフサイクルとライフタイム

| データ | 状態 | 保持期間 |
|------|------|---------|
| アクティブメモ | `UNSORTED` / `ARCHIVED` / `CONVERTED` | ユーザー削除リクエストまで |
| 論理削除メモ | `deleted_at IS NOT NULL` | 90 日（誤削除復元の猶予） |
| 物理削除済みメモ | DB から完全消去 | — |
| S3 上の添付ファイル | アクティブ + 論理削除中 | メモの物理削除と同期して S3 から削除 |
| 監査ログ | 操作の事実のみ記録 | 1 年（既存規約踏襲）。退会時はユーザー ID を匿名化 |

### 9.2 ユーザー退会時のカスケード処理（M3 / NC2 対応）

退会フローは F09.5 / F12.3 と整合させる。`tags.created_by` が `ON DELETE RESTRICT` のため、**事前にタグ処理を完了させてから users DELETE を実行**する。これを怠ると FK 違反で退会フローがハングし、ユーザーが詰む。

#### 9.2.1 退会前準備（必須・順序固定）

退会実行の **前** に以下を順番に実行する。すべて 1 つの SAGA トランザクションで処理し、いずれかが失敗したらロールバック:

```
Step 1: PERSONAL タグの処理
  → DELETE FROM tags WHERE created_by = :userId AND scope_type = 'PERSONAL'
  → CASCADE で quick_memo_tag_links / todo_tag_links も削除
  → 個人タグは退会ユーザー専用なので無条件削除

Step 2: TEAM タグの処理（チーム単位でループ）
  → SELECT DISTINCT scope_id FROM tags WHERE created_by = :userId AND scope_type = 'TEAM'
  → 各 team_id について以下を実施:
     a. 同チームの ADMIN を 1 人選定（最も古い ADMIN を優先）
     b. ADMIN が存在する場合: UPDATE tags SET created_by = :adminId WHERE created_by = :userId AND scope_id = :teamId AND scope_type = 'TEAM'
     c. ADMIN が存在しない場合（解散済みチーム）: DELETE FROM tags WHERE ...

Step 3: ORGANIZATION タグの処理（同様）
  → 移譲先がいなければ 退会ブロック（人間オペレーター対応が必要）
  → エラーコード: WITHDRAW_BLOCKED_NO_ORG_ADMIN

Step 4: quick_memos の物理削除（30日論理削除を経た場合は猶予期間後）
  → §11 物理削除バッチと同じ S3 削除順序を適用

Step 5: user_quick_memo_settings の物理削除

Step 6: 監査ログの匿名化
  → UPDATE audit_logs SET user_id = NULL WHERE user_id = :userId

Step 7: users 物理削除
  → これで FK 制約に引っかからない
```

#### 9.2.2 通常退会（30 日猶予あり）

1. `users.deleted_at = NOW()` で論理削除（30 日間は復元可）
2. 30 日経過後、F12.3 GDPR バッチが §9.2.1 のフローを実行
3. 各 Step の進捗・失敗を `withdraw_jobs` テーブルに記録（`status: PENDING / IN_PROGRESS / COMPLETED / FAILED`）
4. Step 3 で「移譲先 ADMIN なし」エラーが発生した場合、`status = BLOCKED_MANUAL` として運用通知

#### 9.2.3 GDPR「忘れられる権利」リクエスト（即時削除）

1. ユーザーが F12.3 経由で削除リクエスト
2. 30 日猶予なしで §9.2.1 のフローを即座に実行（GDPR 第 17 条）
3. 削除完了の証跡を別系統の監査ログに残す（ユーザー名は記録しない、user_id のみ）

#### 9.2.4 監査ログ要件

- `WITHDRAW_TAG_TRANSFERRED`: tag_id, from_user_id, to_user_id, scope_type, scope_id
- `WITHDRAW_TAG_DELETED`: tag_id, scope_type, scope_id
- `WITHDRAW_COMPLETED`: user_id (匿名化前の最後の記録)
- `WITHDRAW_BLOCKED_NO_ORG_ADMIN`: user_id, organization_id（運用通知トリガー）

### 9.3 個人データのエクスポート（GDPR 第 15 条）

F12.3 のデータエクスポート機能で以下を含める:
- メモ題名・内容・タグ・添付（S3 から取得）
- リマインド設定
- 論理削除中のメモ（ゴミ箱内のもの）
- 形式: JSON + 添付ファイルを ZIP 圧縮

### 9.4 ログ出力ポリシー

| 種類 | メモタイトル / 本文の扱い |
|------|----------------------|
| アクセスログ | パスのみ。`{id}` も含めない（`/quick-memos/:id` でマスク） |
| アプリケーションログ | エラー時にメモ ID は記録、タイトル・本文は記録禁止 |
| 監査ログ | 操作種別（CREATE / UPDATE / DELETE / CONVERT）と ID のみ。タイトル・本文は記録しない |
| プッシュ通知 | 固定文言「未整理のメモが N 件あります」。タイトル混入禁止（H2） |

### 9.5 410 Gone の使い分け（M5 対応）

- 通常 GET で削除済みメモにアクセス: **404 Not Found**（存在隠蔽）
- ゴミ箱経由 API で 90 日経過後の物理削除済みにアクセス: **410 Gone**（「以前は存在したが完全削除済み」を明示）

## 11. レート制限（H11 対応）

DoS / API 乱用を防ぐため、ユーザー単位のレート制限を設ける。実装は Resilience4j または bucket4j を利用。

| エンドポイント | 制限 | 超過時 |
|--------------|------|-------|
| `POST /quick-memos` | 60 req / min / user | 429 + `Retry-After` ヘッダ |
| `PUT /quick-memos/{id}` | 60 req / min / user | 同上 |
| `DELETE /quick-memos/{id}` | 60 req / min / user | 同上 |
| `POST /quick-memos/{id}/attachments/presign` | 10 req / min / user | 同上 |
| `POST /quick-memos/{id}/attachments/confirm` | 10 req / min / user | 同上 |

**Presigned URL 多重発行による容量攻撃対策（NH6 対応）**:
- レート制限「10 req/min」に加え、以下の追加制約を設ける:
  1. **1 メモあたり同時有効な Presigned URL は最大 1 個**: 新規 presign を発行するとき、同じメモに対する有効な未 confirm の URL があれば、それを **無効化**してから新規発行する（`pending_uploads` テーブルを追加して URL ID を管理、または既存 URL をブラックリストに追加）
  2. **1 ユーザーあたり 1 時間で発行できる総容量は 100MB まで**（`declared_size` の累計を Redis などで集計）
  3. confirm されないまま 5 分経過した URL に紐付く `s3_key` は、別バッチで S3 から削除（孤児ファイル防止）
- これにより「10 個の Presigned URL を同時並行で 100MB ずつ PUT」のような攻撃が成立しない
| `POST /quick-memos/{id}/convert-to-todo` | 30 req / min / user | 同上 |
| `PUT /me/tags/{id}` | 60 req / min / user | 同上 |
| `GET` 系全般 | 600 req / min / user | 同上 |

レスポンスヘッダ:
- `X-RateLimit-Limit`: 上限
- `X-RateLimit-Remaining`: 残数
- `X-RateLimit-Reset`: リセット時刻 (Unix epoch)
- `Retry-After`: 超過時のみ。秒数
