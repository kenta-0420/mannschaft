# F12.2: フィーチャーフラグ（スコープ対応拡張）

> **ステータス**: 🟢 設計確定
> **実装フェーズ**: Phase 12
> **最終更新**: 2026-04-02

## 1. 概要

Phase 10 で導入したグローバルフィーチャーフラグ（`feature_flags` テーブル）を拡張し、**組織単位・ユーザー単位のスコープ切り替え**を実現する。
これにより、カナリアリリース（特定組織に先行公開）、A/B テスト（ユーザー単位で機能分岐）、障害時の機能切り離しをデプロイなしで行えるようになる。
バックエンドでは `@FeatureFlag` アノテーションによる宣言的ガード、フロントエンドでは `useFeatureFlag` コンポーザブルによるリアクティブ参照を提供する。

### 既存実装（F10.1 で導入済み）

| 要素 | 状態 |
|------|------|
| `feature_flags` テーブル | ✅ V10.003 で作成済み |
| `FeatureFlagEntity` | ✅ admin パッケージに実装済み |
| `FeatureFlagService.isEnabled()` | ✅ Valkey キャッシュ付き |
| `SystemAdminFeatureFlagController` | ✅ CRUD API 実装済み |

### F12.2 で新規追加するもの

- `feature_flag_overrides` テーブル（スコープ別オーバーライド）
- `@FeatureFlag` アノテーション + AOP アスペクト
- `GET /api/v1/feature-flags/me` エンドポイント（ログインユーザー向け解決済みフラグ一覧）
- フロントエンド `useFeatureFlag` コンポーザブル

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | フラグの CRUD、全スコープのオーバーライド管理 |
| ADMIN | 自組織のオーバーライド閲覧（読み取り専用） |
| DEPUTY_ADMIN | − |
| MEMBER | 自分に適用されたフラグの参照（`/me` エンドポイント） |
| SUPPORTER | − |
| GUEST | − |

### 対象レベル
- [x] グローバル (GLOBAL) — 既存 `feature_flags.is_enabled` がデフォルト値
- [x] 組織 (ORGANIZATION) — 組織単位でオーバーライド
- [x] 個人 (USER) — ユーザー単位でオーバーライド

### フラグ解決優先順位

```
USER override > ORGANIZATION override > GLOBAL default
```

ユーザーに USER レベルのオーバーライドがあればそれを使用。なければ所属組織のオーバーライドを探索。いずれもなければ `feature_flags.is_enabled`（GLOBAL デフォルト）にフォールバック。

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `feature_flags` | フラグ定義（GLOBAL デフォルト値）| なし（既存） |
| `feature_flag_overrides` | スコープ別オーバーライド | なし |

### テーブル定義

#### `feature_flag_overrides`（新規）

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `flag_id` | BIGINT UNSIGNED | NO | − | FK → feature_flags.id |
| `scope` | ENUM('ORGANIZATION','USER') | NO | − | オーバーライドスコープ |
| `target_id` | BIGINT UNSIGNED | NO | − | scope=ORGANIZATION → organization_id, scope=USER → user_id |
| `is_enabled` | BOOLEAN | NO | FALSE | オーバーライド値 |
| `updated_by` | BIGINT UNSIGNED | YES | NULL | 更新者 user_id |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_override_scope_target (flag_id, scope, target_id)  -- 同一フラグ×スコープ×対象で一意
INDEX idx_override_target (scope, target_id)                      -- ユーザー/組織別の逆引き
```

**制約・備考**
- `flag_id` は `feature_flags.id` への外部キー（CASCADE DELETE）
- フラグ削除時にオーバーライドも連動削除される
- 論理削除なし（オーバーライド不要になったら物理削除）

### ER図（テキスト形式）
```
feature_flags (1) ──── (N) feature_flag_overrides
                                  │
                      scope=ORGANIZATION → organizations.id
                      scope=USER         → users.id
```

#### `GET /api/v1/feature-flags/me`

**レスポンス（200 OK）**
```json
{
  "data": {
    "new_dashboard": true,
    "webhook.test": false,
    "beta_chat_v2": true
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | scope/targetId のバリデーションエラー |
| 403 | 権限不足 |
| 404 | フラグキーが存在しない |
| 409 | 同一スコープ×対象のオーバーライドが既に存在（PUT で upsert するため通常は発生しない） |

## 5. ビジネスロジック

### フラグ解決フロー

```
1. FeatureFlagResolver.resolve(flagKey, userId, organizationId) を呼び出す
2. キャッシュキー "featureFlagResolved:{flagKey}:{userId}" を参照
3. キャッシュミス時:
   a. feature_flag_overrides から scope=USER, target_id=userId のレコードを検索
   b. 見つかればその is_enabled を返却
   c. なければ scope=ORGANIZATION, target_id=organizationId を検索
   d. 見つかればその is_enabled を返却
   e. なければ feature_flags.is_enabled（GLOBAL デフォルト）を返却
4. 結果をキャッシュに格納（TTL: 1分）
```

### `@FeatureFlag` アノテーション

```java
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
public @interface FeatureFlag {
    String value();  // フラグキー（例: "webhook.test"）
}
```

AOP アスペクトで対象メソッドの実行前にフラグを評価し、無効な場合は `FeatureDisabledException`（HTTP 403 + エラーコード `FEATURE_DISABLED`）をスローする。
ユーザー情報は `SecurityUtils.getCurrentUserId()` + `SecurityUtils.getCurrentOrganizationId()` から取得。

### `/me` エンドポイントのバルク解決

```
1. feature_flags から全フラグを取得
2. feature_flag_overrides から scope=USER, target_id=currentUserId のオーバーライドを一括取得
3. feature_flag_overrides から scope=ORGANIZATION, target_id=currentOrgId のオーバーライドを一括取得
4. フラグごとに USER > ORG > GLOBAL の優先順位で解決
5. Map<String, Boolean> として返却
```

N+1 問題を避けるため、個別フラグの解決ではなくバルク取得＋メモリ上でマージする。

## 6. セキュリティ考慮事項

- **認可チェック**: オーバーライド管理 API は SYSTEM_ADMIN 専用。`/me` は認証済みユーザーなら誰でもアクセス可能
- **情報漏洩防止**: `/me` は自分のフラグのみ返却。他ユーザー・他組織のオーバーライドは参照不可
- **レートリミット**: `/me` は `Bucket4j` で 1分間に60回の制限（フロントエンドのポーリング対策）
- **フラグキーの命名規則**: `[a-z][a-z0-9_.]{1,98}[a-z0-9]`（英小文字・数字・ドット・アンダースコア、最大100文字）

## 7. キャッシュ戦略

| キャッシュ名 | キー | TTL | 無効化タイミング |
|------------|------|-----|---------------|
| `featureFlags` | `{flagKey}` | 30分（既存） | グローバル値更新時 `@CacheEvict` |
| `featureFlagResolved` | `{flagKey}:{userId}` | **1分** | オーバーライド作成/更新/削除時 |
| `featureFlagMe` | `{userId}` | **1分** | オーバーライド変更時 |

### TTL を1分にする理由

フィーチャーフラグの切り替えは即座に反映されることが期待される（障害時の機能切り離し等）。30分では長すぎるため、スコープ解決済みキャッシュは1分 TTL とする。

### カスタム TTL の実装

`RedisConfig` のデフォルト 30分 TTL に加え、`featureFlagResolved` と `featureFlagMe` のみ 1分 TTL を適用する:

```java
@Bean
public RedisCacheManagerBuilderCustomizer featureFlagCacheCustomizer() {
    return builder -> builder
        .withCacheConfiguration("featureFlagResolved",
            RedisCacheConfiguration.defaultCacheConfig().entryTtl(Duration.ofMinutes(1)))
        .withCacheConfiguration("featureFlagMe",
            RedisCacheConfiguration.defaultCacheConfig().entryTtl(Duration.ofMinutes(1)));
}
```

## 9. フロントエンド実装構成

### `useFeatureFlag` コンポーザブル

```typescript
// composables/useFeatureFlag.ts
export function useFeatureFlag(flagKey: string): Ref<boolean> {
  const flags = useFeatureFlags() // 内部で /api/v1/feature-flags/me を呼び出し
  return computed(() => flags.value[flagKey] ?? false)
}
```

### フラグストア

```typescript
// stores/featureFlags.ts
export const useFeatureFlagStore = defineStore('featureFlags', () => {
  const flags = ref<Record<string, boolean>>({})

  async function fetch() {
    const { data } = await $fetch('/api/v1/feature-flags/me')
    flags.value = data
  }

  return { flags, fetch }
})
```

- アプリ初期化時（`app.vue` の `onMounted` またはミドルウェア）で `fetch()` を1回呼び出す
- 以降はストア内の値を参照（サーバーへの再問い合わせなし）
- 管理画面でフラグ変更時、必要に応じて `fetch()` を再実行

### テンプレートでの使用例

```vue
<template>
  <NewDashboard v-if="showNewDashboard" />
  <LegacyDashboard v-else />
</template>

<script setup>
const showNewDashboard = useFeatureFlag('new_dashboard')
</script>
```

## 11. 解決済み事項

- [x] ~~フラグの「パーセンテージロールアウト」（ユーザーの N% に有効化）は Phase 12 スコープに含めるか？~~ → **含めない**。現時点のユースケース（カナリアリリース・障害時切り離し）は GLOBAL/ORG/USER の3段階で十分。ハッシュ振り分け・分析基盤（F10.4）への依存が大きく、Phase 12 の法的・セキュリティ系タスク（F12.3/F12.4）を優先すべき。FUTURE_CONSIDERATIONS.md に「F10.4 実装後に A/B テスト基盤と合わせて検討」として記録する。
- [x] ~~オーバーライド変更時の監査ログ連携方式~~ → **ApplicationEvent で疎結合にする**。F10.3 の実装時期が未確定のため、フラグ側は `FeatureFlagChangedEvent` を発行するだけにする。F10.3 実装時に `@EventListener` を追加すれば連携完了。既存の ApplicationEvent パターン（F04.2 等）と統一。
- [x] ~~`/me` のレスポンスにフラグの `description` を含めるか~~ → **含めない**。`/me` はフロントエンドの ON/OFF 判定用で description は不要。レスポンスサイズ増加によるキャッシュ効率低下を避ける。description は管理画面の既存 API（`GET /system-admin/feature-flags`）で取得可能。将来ユーザー向け表示が必要になった場合は description とは別に `label` フィールドを検討する。
