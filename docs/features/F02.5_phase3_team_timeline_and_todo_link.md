## 1. 背景・動機

F02.5 本体は「ADHD 傾向のあるユーザー向けの個人生産性ツール」として、**完全に個人スコープ専用**で設計・実装された。しかし実運用フェーズに入ったことで、以下の業務運用上の不足が明らかになった。

1. **管理職からの可視性ゼロ** — メンバーが日々行動メモを取っていても `PERSONAL` スコープのため、上長・チームリーダーは部下の「何を、どれくらい、どの TODO に対して進めたか」を一切把握できない。
2. **仕事／私事の混在** — 現行 `action_memos` にはカテゴリ軸がなく、「家族との外食」と「クライアント対応」が同じタイムラインに混在する。
3. **実績時間・進捗率の記録不可** — TODO 側には `progress_rate` / `progress_manual` があるが、行動メモ側から実績ベースで進捗を更新する導線がない。
4. **TODO との片方向連携** — メモに `related_todo_id` は存在するが、メモ側で「完了した」と記録しても TODO 側の `status` は `OPEN` のまま放置される。

本 Phase 3 拡張は、F02.5 の「入力摩擦ゼロ」哲学を壊さずに、上記4点を解決することを目的とする。

- 個人利用はこれまで通り `PERSONAL` のドライモードを維持（既定）
- 仕事利用者は「WORK カテゴリ + チーム指定」で明示的にオプトインすることで、所属チームのタイムラインに投稿可能
- メモ1件単位で「実績時間・進捗率・TODO完了フラグ」を任意付加でき、これらは TODO 側に同期反映される
- 既存 API はすべて非破壊。追加するフィールドは全て nullable／デフォルト値付き。

## 4. 機能詳細

### 4.1 仕事／プライベート区分（category フィールド）

**目的**: メモ1件に「これは仕事の記録か、私事か、その他か」の属性をユーザーが明示的に与える。タグとは別軸として扱う。

#### データモデル

```sql
ALTER TABLE action_memos
  ADD COLUMN category VARCHAR(16) NOT NULL DEFAULT 'PRIVATE'
    COMMENT 'メモのカテゴリ。WORK = 仕事 / PRIVATE = 私事 / OTHER = その他'
  AFTER content;
```

Java 側:

```java
public enum ActionMemoCategory {
    WORK,
    PRIVATE,
    OTHER
}
```

#### タグとの併用方針

- `category` は必ず1つ（NOT NULL）。`tags` は0〜10個（既存）。
- UI では `category` を固定セグメント（3択トグル）、`tags` を自由サジェスト検索で明確に分ける。
- サーバー側ではタグ検索 API に `category` フィルタを追加。
- 既存メモは全件 `PRIVATE` として初期化。

#### デフォルト値の個人化

```sql
ALTER TABLE user_action_memo_settings
  ADD COLUMN default_category VARCHAR(16) NOT NULL DEFAULT 'PRIVATE'
    COMMENT 'メモ作成時のデフォルトカテゴリ';
```

POST `/api/v1/action-memos` で `category` 省略時は `default_category` が適用される。

### 4.2 実績時間・進捗率

```sql
ALTER TABLE action_memos
  ADD COLUMN duration_minutes INT UNSIGNED NULL DEFAULT NULL
    COMMENT '実績時間（分）。NULL = 未入力'
  AFTER category,
  ADD COLUMN progress_rate DECIMAL(5,2) NULL DEFAULT NULL
    COMMENT '記録時点の進捗率（0.00〜100.00）。NULL = 未入力'
  AFTER duration_minutes;
```

#### バリデーション

| 項目 | 範囲 | 理由 |
|------|------|------|
| `duration_minutes` | 0 〜 1440（1日=24h） | 1メモに1日超の実績は入れない |
| `progress_rate` | 0.00 〜 100.00 | TODO 側と同レンジ |
| `progress_rate` 指定時の `related_todo_id` | 必須 | 紐付け先がない進捗率は意味を持たない |
| `duration_minutes` 指定時の `related_todo_id` | 任意 | 「資料作成30分」のように TODO 無しでも時間記録は有効 |

#### TODO への伝播

`progress_rate` を指定して POST または PATCH された瞬間、同一トランザクション内で以下を実行:

1. `related_todo_id` の TODO を取得（所有者検証）
2. `TodoService.setProgressRate(todoId, newRate)` を呼ぶ（既存 API。内部で `TodoProgressService.setManualProgressRate` が `progress_manual = true` を付与する）
3. TODO の `progress_rate` が上書きされ、`progress_manual = true` がセットされる

### 4.3 チームタイムライン投稿

#### 4.3.1 投稿方式（2系統）

| 方式 | API | タイミング | 用途 |
|------|-----|-----------|------|
| **個別即時投稿** | `POST /api/v1/action-memos/{id}/publish-to-team` | メモ1件ごと | 「今の進捗をすぐに共有したい」 |
| **日次まとめ投稿** | `POST /api/v1/action-memos/publish-daily-to-team` | 終業時 | 「1日分を WORK だけ抜粋してチーム投稿」 |

既存 `POST /api/v1/action-memos/publish-daily` は無変更。

#### 4.3.2 `POST /api/v1/action-memos/{id}/publish-to-team`

**リクエスト**:
```json
{
  "team_id": 42,
  "extra_comment": "午前のレビュー対応、完了しました"
}
```

`team_id` 省略時は `user_action_memo_settings.default_post_team_id` を使用。どちらも NULL の場合は 400 `team_id_required`。

**処理**:
1. メモ所有者検証
2. カテゴリ検証（`memo.category == WORK`）
3. チーム所属検証
4. `timeline_posts` に INSERT（`scope_type = TEAM`, `scope_id = team_id`）
5. `action_memos.posted_team_id` = `team_id`、`timeline_post_id` = 新投稿 ID を更新

**本文フォーマット**:
```
[HH:MM] {content}
⏱️ {duration_minutes}分 / 📊 進捗 {progress_rate}%
🔗 関連TODO: {todo.title}

---
{extra_comment}
```

#### 4.3.3 `POST /api/v1/action-memos/publish-daily-to-team`

対象メモは `category = WORK` かつ `posted_team_id` が NULL のメモ（重複投稿防止）。

#### 4.3.4 所属チーム数に応じた UI フロー

| 所属チーム数 | UI 挙動 |
|-------------|---------|
| **0件** | `category = WORK` を選択しても投稿先チーム欄は「所属チームがありません」の非活性プレースホルダ。保存はできるが `publish-to-team` ボタンは非表示（UI 層）。保存されたメモは個人ログには残る |
| **1件** | そのチームが `default_post_team_id` として自動設定される（設定画面未訪問でもバックエンドが遅延 UPSERT）。投稿先ドロップダウンは1件のみ選択済みで表示 |
| **2件以上** | 初回利用時に設定画面で `default_post_team_id` を明示選択。未選択のまま `category = WORK` に切り替えた瞬間、ドロップダウンが強調表示され選択を促す |

追加ルール:

- デフォルトチームから脱退した場合、設定の `default_post_team_id` は NULL に自動リセット（`UserTeamMembershipRemovedEventHandler`）
- 自動リセット発生時はフロントの設定画面に再訪したタイミングで「以前設定していた投稿先チームから脱退したため、デフォルトをリセットしました」の通知バナーを1度だけ表示（クエリパラメータ `?default_team_reset=1` で制御）

#### 4.3.5 データモデル（`posted_team_id`）

```sql
ALTER TABLE action_memos
  ADD COLUMN posted_team_id BIGINT UNSIGNED NULL DEFAULT NULL
    COMMENT 'チームタイムラインに投稿済みの場合のチームID。NULL = 未投稿'
  AFTER timeline_post_id;

ALTER TABLE action_memos
  ADD CONSTRAINT fk_am_posted_team
    FOREIGN KEY (posted_team_id) REFERENCES teams(id)
    ON DELETE SET NULL;
```

### 4.4 TODO連動（`completes_todo`）

```sql
ALTER TABLE action_memos
  ADD COLUMN completes_todo BOOLEAN NOT NULL DEFAULT FALSE
    COMMENT 'related_todo_id の TODO を完了扱いにするフラグ'
  AFTER related_todo_id;
```

#### 動作仕様

`POST /api/v1/action-memos` または `PATCH /api/v1/action-memos/{id}` で `completes_todo = true` かつ `related_todo_id` が非 NULL の場合:

1. TODO 所有者検証（PERSONAL スコープの TODO のみ許可）
2. TODO の現在の `status` が `COMPLETED` でない場合、`TodoService.changeStatus(todoId, new TodoStatusChangeRequest("COMPLETED"), currentUserId)` を同一トランザクション内で同期呼び出し（既存 API 流用）
3. 失敗時は全体ロールバック
4. `progress_rate` 指定があれば同時に反映（`setProgressRate` を別途呼ぶ）

#### メモ削除時の TODO 復元ポリシー

**判断**: メモを論理削除しても TODO の `status` は復元しない。ユーザーが意図的に戻したい場合は TODO 側で明示的に操作する。

#### 監査ログ

メモ起因の TODO 遷移は `AUDIT_LOG_TODO_STATUS_CHANGED` に `source = "ACTION_MEMO"` / `source_id = memo_id` として記録。

- 既存 `TodoStatusChangedEvent` は `source` フィールドを持たないため、監査ログ書き込みは `ActionMemoService` 側で `TodoService.changeStatus` 呼び出し後に直接 `AuditLogService.record(...)` を実行する（event の signature は変更しない）。
- これにより、通常の TODO 画面からの完了操作（source = "USER"）とメモ起因（source = "ACTION_MEMO"）を後から区別可能にする。

#### 同期 vs 非同期

**同期実行**を採用。トランザクション境界が明確、UI 即時反映、WebSocket リアルタイム配信が可能。

## 8. 権限・セキュリティ

### 8.1 認可ルール

| 操作 | 許可条件 |
|------|---------|
| メモ作成（どのカテゴリでも） | 本人のみ |
| `category = WORK` でのメモ作成 | 本人。追加制約なし |
| `publish-to-team` | (1) メモ所有者であること、(2) `category = WORK`、(3) `team_id` のチームメンバーであること |
| `publish-daily-to-team` | 上記 (1)(3) に加え、当日に WORK メモが1件以上存在すること |
| `completes_todo = true` でのメモ作成 | 本人のメモ + `related_todo_id` の TODO も本人所有（PERSONAL） |

### 8.2 `MANAGE_TIMELINE` 権限の扱い

不要。通常投稿扱い。不適切投稿の削除は F04.1 の既存削除フローを使用。

### 8.3 IDOR 対策

すべて 404 を返却（403 ではなく）。

### 8.4 プライバシー漏洩防止

- `category = PRIVATE` のメモはチーム投稿 API 経由でも絶対にチームタイムラインに流出しない（API 層で 400）

## 9. バリデーション

### 9.1 バックエンド

| 項目 | ルール | エラー時 |
|------|--------|---------|
| `category` | ENUM(`WORK`,`PRIVATE`,`OTHER`) | 400 `invalid_category` |
| `duration_minutes` | NULL or 0-1440 | 400 `invalid_duration` |
| `progress_rate` | NULL or 0.00-100.00 | 400 `invalid_progress_rate` |
| `progress_rate` 指定時 `related_todo_id` | 必須 | 400 `progress_requires_todo` |
| `completes_todo = true` 時 `related_todo_id` | 必須 | 400 `completes_requires_todo` |
| `publish-to-team` の `team_id` | 所属チームのみ | 404 `team_not_found` |
| `publish-to-team` のメモ `category` | WORK であること | 400 `only_work_can_be_posted` |
| 既に投稿済みのメモ | `posted_team_id != NULL` の場合 | 409 `already_posted` |
| `default_post_team_id` | 所属チーム or NULL | 400 `invalid_default_team` |
| `publish-to-team` の `team_id` / `default_post_team_id` 両方 NULL | どちらか必須 | 400 `team_id_required` |
| `publish-daily-to-team` 当日 WORK メモ0件 | 投稿対象が存在しない | 400 `no_work_memo_today` |

### 9.2 レートリミット

- `publish-to-team`: 10回/分（既存 `publish-daily` の5回/分と同じ系統の制限）
- Bucket4j で IP + userId 複合キー制御

## 13. 決着済み設計判断（旧「未解決事項」の決着）

精査を経て、以下の判断で確定した。

| 項目 | 判断 | 理由 |
|------|------|------|
| チーム TODO（`scope_type = TEAM`）の `completes_todo` 対応 | **Phase 3 非対応（Phase 4 以降）** | 完了権限・承認フロー・通知パターンが別設計になる。Phase 3 は個人 TODO のみ |
| 組織（ORGANIZATION）スコープへの投稿 | **Phase 3 非対応（Phase 4 以降）** | 複数チーム横断のユースケース調査が別途必要 |
| チーム投稿済みメモの本文編集時の挙動 | **投稿内容は固定。メモ編集はチーム投稿に反映しない** | F02.5 本体 `publish-daily` と同じ設計で一貫性維持。再投稿が必要ならメモ再作成 |
| 管理職向けの横断ビュー | **Phase 3 非対応（Phase 4 以降）** | 権限モデル・可視化要件の別途設計が必要 |
| リマインド通知 | **Phase 3 非対応** | F04.3 通知拡張と合わせて別フェーズで実装 |
| メモ経由 TODO 完了の監査可視化 UI | **Phase 3 非対応** | 監査ログは記録するが UI は Phase 4 以降 |
| `publish-to-team` レートリミット | **10回/分に確定**（§ 9.2） | 既存 `publish-daily` の5回/分と近い系統で運用実績に基づく |
| WebSocket 配信 | **既存 F04.1 機構に自動で乗る** | 追加配線不要。§10.2 で統合テストによる実測確認を必須化 |
