> **ステータス**: 🟢 設計完了

> **最終更新**: 2026-03-18
> **モジュール種別**: デフォルト機能 #28（領収書発行）
> **関連ドキュメント**: F08.2 支払い管理（member_payments 連携）、F05.3 電子印鑑（領収書への押印）、F03.1 スケジュール・出欠管理（但し書き自動取得）、F01.2 組織・チーム・メンバー・ロール（住所情報）

## 1. 概要

チーム/組織の ADMIN が、会費・イベント参加費等の支払いに対して**電子領収書**を発行する機能。日本のインボイス制度（適格請求書等保存方式）に対応し、登録番号・発行者情報・税率別内訳を含む法的要件を満たす領収書を PDF で生成する。

**主要ユースケース**:
- マンション管理組合: 管理費・修繕積立金の領収書発行
- スポーツチーム: 月会費・年会費・大会参加費の領収書発行
- 学校・PTA: 会議参加費・イベント費用の領収書発行
- 整骨院・美容室: 施術費用の領収書発行（サービス履歴連携）

**インボイス制度対応**:
- 適格請求書発行事業者の登録番号（T + 13桁）をチーム/組織に登録
- 登録済みの場合: 適格請求書（インボイス）形式で発行（税率別内訳・登録番号記載）
- 未登録の場合: 通常の領収書形式で発行（登録番号なし。「適格請求書発行事業者以外の者からの領収書」と明記）

**但し書きの自動取得**:
- `member_payments` が `schedules`（イベント）に紐付いている場合、スケジュール名を但し書きの初期値に自動設定
- 例: 「第12回 定例理事会 会議費として」「2026年4月度 月会費として」
- ADMIN が手動で編集可能

### 本ドキュメントのスコープ

| 機能 | 説明 |
|------|------|
| 発行者設定 | チーム/組織のインボイス登録番号・発行者名・住所・押印設定 |
| 領収書発行 | 支払い実績からの自動発行 / 手動発行。但し書き自動取得 |
| PDF 生成 | インボイス対応の領収書 PDF 生成・ダウンロード |
| 電子印鑑連携 | F05.3 の電子印鑑を領収書に押印 |
| 領収書一覧・検索 | 発行済み領収書の検索・再ダウンロード |

**スコープ外**:
- Stripe の自動発行レシート（`member_payments.stripe_receipt_url`）→ F08.2 で管理。本機能はそれとは別の「組織が発行する正式な領収書」
- 請求書（Invoice）発行 → 将来検討（FUTURE_CONSIDERATIONS 向き）
- 確定申告連携 → スコープ外

**既存テーブル（他フェーズで作成済み）の参照**:

### テーブル定義

#### `receipt_issuer_settings`

領収書発行者の設定。チーム/組織ごとに1レコード。インボイス制度の登録情報を含む。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `issuer_name` | VARCHAR(200) | NO | — | 発行者名（例: 「○○マンション管理組合」） |
| `postal_code` | VARCHAR(10) | YES | NULL | 発行者の郵便番号 |
| `address` | VARCHAR(500) | YES | NULL | 発行者の住所 |
| `phone` | VARCHAR(20) | YES | NULL | 発行者の電話番号 |
| `is_qualified_invoicer` | BOOLEAN | NO | FALSE | 適格請求書発行事業者か |
| `invoice_registration_number` | VARCHAR(14) | YES | NULL | 登録番号（T + 13桁。`is_qualified_invoicer = TRUE` の場合は必須） |
| `default_seal_user_id` | BIGINT UNSIGNED | YES | NULL | FK → users。デフォルト押印者（領収書に押印する ADMIN のユーザー ID） |
| `default_seal_variant` | VARCHAR(20) | YES | NULL | デフォルト印鑑バリアント（`LAST_NAME` / `FULL_NAME` / `FIRST_NAME`） |
| `receipt_note_template` | TEXT | YES | NULL | 但し書きのテンプレート（例: 「{item_name}として」。{} はプレースホルダー） |
| `logo_storage_key` | VARCHAR(500) | YES | NULL | S3 上のロゴ画像キー（PDF ヘッダー左上に配置。最大 200x60px。PNG/JPEG） |
| `custom_footer` | TEXT | YES | NULL | PDF カスタムフッターテキスト（例: 「お振込先: ○○銀行 ○○支店 普通 1234567」） |
| `next_receipt_number` | INT UNSIGNED | NO | 1 | 次の領収書番号（スコープ内で連番管理） |
| `receipt_number_prefix` | VARCHAR(20) | YES | NULL | 領収書番号のプレフィックス（例: 「RC-2026-」） |
| `fiscal_year_start_month` | INT | NO | 4 | 会計年度の開始月（1-12。日本の多くの組織は4月始まり） |
| `auto_reset_number` | BOOLEAN | NO | TRUE | 年度切替時に `next_receipt_number` を自動リセットするか |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_ris_scope (scope_type, scope_id)  -- スコープごとに1設定
```

**制約・備考**
- `invoice_registration_number`: `T` + 13桁の数字（例: `T1234567890123`）。バリデーション: `/^T\d{13}$/`。`is_qualified_invoicer = TRUE` の場合は NOT NULL 制約をアプリ層で適用
- `default_seal_user_id`: 領収書に押印する ADMIN のユーザー ID。NULL の場合は押印なしで発行可能
- `next_receipt_number`: `SELECT ... FOR UPDATE` で排他制御し、発行時にインクリメント
- `fiscal_year_start_month` + `auto_reset_number`: 年度切替の自動リセット設定。日次バッチ（AM 0:00）で `auto_reset_number = TRUE` かつ今日が `fiscal_year_start_month` の1日の場合、`next_receipt_number` を 1 にリセットし、`receipt_number_prefix` の年部分を自動更新（例: `RC-2025-` → `RC-2026-`）。`auto_reset_number = FALSE` の場合は ADMIN が手動管理
- `receipt_note_template`: プレースホルダー `{item_name}`（支払い項目名）、`{schedule_name}`（イベント名）、`{period}`（対象期間）が使用可能

**インデックス**
```sql
UNIQUE KEY uq_r_receipt_number (scope_type, scope_id, receipt_number)  -- スコープ内で領収書番号一意
INDEX idx_r_scope (scope_type, scope_id, issued_at DESC)                -- スコープ別の発行一覧
INDEX idx_r_recipient (recipient_user_id, issued_at DESC)               -- 受領者別の一覧（マイページ用）
INDEX idx_r_payment (member_payment_id)                                  -- 支払い実績からの逆引き
INDEX idx_r_schedule (schedule_id)                                       -- イベントからの逆引き
FULLTEXT INDEX ft_r_recipient (recipient_name)                             -- 受領者名の部分一致検索（電子帳簿保存法の取引先検索要件）
```

**制約・備考**
- 発行者情報（`issuer_name` / `issuer_address` / `invoice_registration_number`）は発行時点のスナップショットを保存。発行後に `receipt_issuer_settings` が変更されても既発行の領収書に影響しない
- 受領者情報（`recipient_name` / `recipient_address`）も発行時点のスナップショット。同上
- `member_payment_id`: 支払い実績に紐付く場合に設定。1つの `member_payments` に対して複数の領収書を発行可能（分割領収書）
- `voided_at`: 無効化された領収書は PDF に「無効」の赤スタンプをオーバーレイ。再ダウンロード時に無効化表示付きの PDF を返却
- `seal_stamp_log_id`: F05.3 の `seal_stamp_logs` に `target_type = 'RECEIPT'` で記録。NULL の場合は押印なし
- `schedule_id`: 但し書き自動取得元。イベントの `title` + 日付を但し書きの初期値に使用
- PDF 保持期間: **7年**（電子帳簿保存法の保存義務期間）。S3 の Lifecycle Policy で管理

#### `receipt_line_items`

領収書の税率別明細行。1枚の領収書に複数税率（標準10% + 軽減8%等）が混在する場合に使用。明細行がない場合は `receipts` テーブルの `tax_rate` / `tax_amount` / `amount_excl_tax` を単一税率として使用。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `receipt_id` | BIGINT UNSIGNED | NO | — | FK → receipts。ON DELETE CASCADE |
| `description` | VARCHAR(200) | NO | — | 明細の説明（例: 「会議費」「飲食代」） |
| `amount` | DECIMAL(10,0) | NO | — | 税込金額 |
| `tax_rate` | DECIMAL(4,2) | NO | — | 適用税率（%）。10.00 or 8.00 |
| `tax_amount` | DECIMAL(10,0) | NO | — | 税額 |
| `amount_excl_tax` | DECIMAL(10,0) | NO | — | 税抜金額 |
| `sort_order` | INT | NO | 0 | 表示順（昇順） |

**インデックス**
```sql
INDEX idx_rli_receipt (receipt_id, sort_order)  -- 領収書別の明細一覧
```

**制約・備考**
- 明細行がある場合: `receipts.amount` = `SUM(receipt_line_items.amount)`、`receipts.tax_amount` = `SUM(receipt_line_items.tax_amount)` であることを Service 層でバリデーション
- 明細行がない場合（単一税率）: `receipts` テーブルの `tax_rate` / `tax_amount` / `amount_excl_tax` をそのまま使用。`receipt_line_items` は0件
- 1領収書あたり最大10明細行
- インボイス制度では税率ごとの区分記載が義務。複数税率がある場合は明細行で税率別合計を記載

| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |

| `line_items_json` | JSON | YES | NULL | 複数税率の明細行定義（`receipt_line_items` と同形式の JSON 配列。NULL = 単一税率） |
| `payment_method_label` | VARCHAR(50) | YES | NULL | 支払い方法の表示名 |
| `seal_stamp` | BOOLEAN | NO | TRUE | 電子印鑑を押印するか |

| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_rp_scope (scope_type, scope_id)  -- スコープ別のプリセット一覧
```

**制約・備考**
- 1スコープあたり最大 **30 プリセット**
- `description` にはテンプレートプレースホルダー（`{period}` 等）を使用可能。発行時に実値に置換
- `line_items_json`: `[{"description": "会議費", "amount": 3300, "tax_rate": 10.00}, ...]` 形式。発行時に `receipt_line_items` に展開
- プリセットの使用: `POST /admin/receipts` のリクエストに `preset_id` を指定 → プリセットの値をデフォルト値として適用（ADMIN が個別に上書き可能）

#### `receipt_queue`

発行待ちキュー。Stripe 決済完了時に自動追加され、ADMIN の承認を待つ。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `member_payment_id` | BIGINT UNSIGNED | NO | — | FK → member_payments。決済完了した支払い |
| `recipient_user_id` | BIGINT UNSIGNED | NO | — | FK → users。受領者 |
| `suggested_description` | VARCHAR(500) | YES | NULL | 自動生成された但し書き候補 |
| `suggested_amount` | DECIMAL(10,0) | NO | — | 支払い額（`member_payments.amount` から取得） |
| `preset_id` | BIGINT UNSIGNED | YES | NULL | FK → receipt_presets。マッチしたプリセット |
| `status` | VARCHAR(20) | NO | 'PENDING' | ステータス（`PENDING`: 承認待ち / `APPROVED`: 承認済み（発行処理中） / `SKIPPED`: スキップ） |
| `processed_receipt_id` | BIGINT UNSIGNED | YES | NULL | FK → receipts。承認後に発行された領収書 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_rq_scope_status (scope_type, scope_id, status, created_at DESC)  -- スコープ別の未処理キュー
UNIQUE KEY uq_rq_payment (member_payment_id)                                -- 同一支払いの重複キュー防止
```

**制約・備考**
- Stripe Webhook（`checkout.session.completed`）で `member_payments` が `PAID` になった時点で、`ActivityFeedEventListener` と同様のパターンで `ReceiptQueueEventListener` が非同期で INSERT
- `preset_id`: `payment_items` と `receipt_presets` の紐付けがある場合に自動設定。マッチしない場合は NULL（ADMIN が手動で但し書き・金額を入力）
- `SKIPPED`: 領収書が不要な支払い（例: 寄付金）の場合に ADMIN がスキップ
- `APPROVED` → `receipts` への INSERT + PDF 生成は `POST /admin/receipts` と同一フローを使用
- キューの保持期間: 90日。90日超の `PENDING` レコードは月次バッチで `SKIPPED` に自動遷移 + ADMIN アラート

### ER図（テキスト形式）
```
receipt_issuer_settings (1) ──── (N) receipts（scope_type + scope_id 経由）
receipts (1) ──── (N) receipt_line_items（receipt_id）
member_payments (1) ──── (N) receipts（member_payment_id）
member_payments (1) ──── (1) receipt_queue（member_payment_id）
receipt_queue (1) ──── (0..1) receipts（processed_receipt_id）
receipt_presets (1) ──── (N) receipt_queue（preset_id）
users (1) ──── (N) receipts（recipient_user_id）
users (1) ──── (N) receipts（issued_by）
seal_stamp_logs (1) ──── (1) receipts（seal_stamp_log_id）
schedules (1) ──── (N) receipts（schedule_id）
```

#### `POST /api/v1/admin/receipts`

**認可**: ADMIN / DEPUTY_ADMIN（`RECEIPT_MANAGE` パーミッション）

領収書を発行する。

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `scope_type` | String | Yes | `TEAM` / `ORGANIZATION` |
| `scope_id` | Long | Yes | スコープ ID |

**リクエストボディ**
```json
{
  "preset_id": null,
  "status": "ISSUED",
  "member_payment_id": 123,
  "recipient_user_id": 50,
  "recipient_name": "田中太郎",
  "recipient_postal_code": "150-0002",
  "recipient_address": "東京都渋谷区渋谷2-2-2",
  "description": "第12回 定例理事会 会議費として",
  "amount": 5500,
  "tax_rate": 10.00,
  "line_items": [
    { "description": "会議費", "amount": 3300, "tax_rate": 10.00 },
    { "description": "飲食代", "amount": 2200, "tax_rate": 8.00 }
  ],
  "payment_method_label": "銀行振込",
  "payment_date": "2026-04-01",
  "seal_stamp": true,
  "schedule_id": 201,
  "email_delivery": {
    "enabled": true,
    "email": null
  }
}
```

- `preset_id`: プリセット ID（任意）。指定時はプリセットの値をデフォルト適用。リクエストで個別指定したフィールドはプリセットより優先
- `status`: `ISSUED`（デフォルト。即時発行）/ `DRAFT`（下書き。DEPUTY_ADMIN は DRAFT のみ指定可能）。DRAFT の場合は採番・PDF 生成・通知を行わない
- `member_payment_id`: 支払い実績に紐付ける場合に指定。NULL の場合は手動発行
- `recipient_user_id`: プラットフォームユーザーの場合に指定。NULL の場合は外部受領者（`recipient_name` 等を手動入力）
- `recipient_name`: 省略時は `recipient_user_id` の `display_name` or `member_payment_id` の支払者名を自動設定。外部受領者の場合は必須
- `recipient_postal_code` / `recipient_address`: 省略時は `users` テーブルから自動取得。外部受領者の場合は手動入力（省略可）
- `description`: 省略時は `receipt_note_template` + `payment_items.name` or `schedules.title` から自動生成
- `amount`: 税込合計金額。`tax_amount` / `amount_excl_tax` はバックエンドが `tax_rate` から算出
- `line_items`: 複数税率の場合に指定（任意）。指定時は `amount` = `SUM(line_items[].amount)` であること。省略時は単一税率（`tax_rate` から算出）。各明細の `tax_amount` / `amount_excl_tax` はバックエンドが算出
- `seal_stamp`: `true` の場合、`receipt_issuer_settings.default_seal_user_id` の電子印鑑を押印
- `schedule_id`: 但し書き自動取得元のイベント。リクエストに `description` が未指定の場合にのみ参照
- `email_delivery`: 領収書 PDF のメール送信設定（任意）。`enabled = true` の場合、PDF 生成完了後にメール送信。`email` を省略すると `recipient_user_id` の `users.email` を使用。外部受領者の場合は `email` を明示指定。`enabled = false`（デフォルト）の場合はメール送信なし

**レスポンス（201 Created）**
```json
{
  "data": {
    "id": 1,
    "receipt_number": "RC-2026-00001",
    "recipient_name": "田中太郎",
    "description": "第12回 定例理事会 会議費として",
    "amount": 5500,
    "line_items": [
      { "description": "会議費", "amount": 3300, "tax_rate": 10.00, "tax_amount": 300, "amount_excl_tax": 3000 },
      { "description": "飲食代", "amount": 2200, "tax_rate": 8.00, "tax_amount": 163, "amount_excl_tax": 2037 }
    ],
    "tax_summary": {
      "10%": { "amount_excl_tax": 3000, "tax_amount": 300 },
      "8%": { "amount_excl_tax": 2037, "tax_amount": 163 }
    },
    "is_qualified_invoice": true,
    "invoice_registration_number": "T1234567890123",
    "payment_date": "2026-04-01",
    "issued_at": "2026-04-02T10:00:00+09:00",
    "seal_stamped": true,
    "pdf_status": "GENERATING",
    "pdf_download_url": "/api/v1/admin/receipts/1/pdf"
  },
  "warnings": []
}
```

**副作用**:
1. `receipt_issuer_settings.next_receipt_number` をインクリメント（`SELECT ... FOR UPDATE` で排他制御）
2. `seal_stamp = true` の場合: `seal_stamp_logs` に `target_type = 'RECEIPT'`, `target_id = receipts.id` で記録
3. PDF を非同期生成して S3 にアップロード（`pdf_storage_key` を設定）。`GET /admin/receipts/{id}/pdf` は `pdf_storage_key` が NULL の場合は同期的に PDF を生成して返却（フォールバック）。生成済みの場合は S3 の Pre-signed URL にリダイレクト
4. 受領者（`recipient_user_id`）にプッシュ通知（「領収書が発行されました」）
5. `email_delivery.enabled = true` の場合: PDF 生成完了後にメール送信（非同期）
6. 監査ログ: `RECEIPT_ISSUED` を記録

| ステータス | 条件 |
|-----------|------|
| 400 | バリデーションエラー |
| 403 | 権限不足 |
| 404 | `member_payment_id` / `schedule_id` が存在しない |
| 409 | 発行者設定が未登録（先に `PUT /admin/receipt-settings` が必要） |

| ステータス | 条件 |
|-----------|------|
| 403 | ADMIN 権限なし |
| 404 | 領収書が存在しない |
| 409 | 既に無効化済み |

## 5. ビジネスロジック

### 領収書発行フロー

```
1. ADMIN が領収書発行画面にアクセス
   a. 発行者設定が未登録の場合 → 「先に発行者設定を行ってください」案内
   b. 支払い実績一覧（member_payments）から対象を選択 or 手動入力

2. 但し書きの自動生成
   a. member_payment_id が指定された場合:
      i.  payment_items.name を取得 → 「{item_name}として」
      ii. member_payments に紐付く schedule_id があれば schedules.title を優先
          → 「{schedule_title}（{schedule_date}）{item_name}として」
   b. schedule_id が直接指定された場合:
      → 「{schedule_title}（{schedule_date}）会議費として」
   c. どちらもない場合:
      → receipt_note_template のテンプレートを適用
   d. ADMIN が手動で自由に編集可能（自動生成はあくまで初期値）

3. 受領者情報の自動取得
   a. member_payment_id から支払者の user_id を取得
   b. users テーブルから display_name / postal_code / address を取得
   c. 住所が未登録の場合: 宛名のみで発行（住所なし。法的には有効）

4. 領収書番号の採番
   a. receipt_issuer_settings の next_receipt_number を SELECT ... FOR UPDATE で排他取得
   b. receipt_number = receipt_number_prefix + zero-padded next_receipt_number
      例: "RC-2026-" + "00001" → "RC-2026-00001"
   c. next_receipt_number を +1 して UPDATE

5. 電子印鑑の押印（seal_stamp = true の場合）
   a. default_seal_user_id の electronic_seals から指定 variant の印鑑 SVG を取得
   b. seal_stamp_logs に INSERT（target_type = 'RECEIPT', target_id = receipts.id）
   c. seal_stamp_log_id を receipts に設定

6. PDF 生成（非同期）
   a. 領収書データ + 電子印鑑 SVG から PDF を生成
   b. S3 にアップロード → pdf_storage_key を設定
   c. Pre-signed URL（24時間有効）でダウンロード提供
```

### イベント出欠からの一括発行フロー

```
1. ADMIN がイベント詳細画面で「参加費の領収書を一括発行」ボタンをクリック
2. バックエンド:
   a. schedule_attendances で status = 'ATTENDING' のユーザーを取得
   b. member_payments で schedule_id に紐付く PAID の支払いを取得
   c. 出席 + 支払い済みのユーザーに対して POST /admin/receipts/bulk を呼び出し
3. 但し書き: schedules.title から自動生成
   例: 「第12回 定例理事会 会議費として」
4. 受領者情報: 各ユーザーの users テーブルから自動取得
5. 結果: 「5件の領収書を発行しました」トースト通知
```

### 領収書番号の年度自動リセット

```
1. 日次バッチ（AM 0:00）
   a. receipt_issuer_settings の auto_reset_number = TRUE を全件取得
   b. 今日が fiscal_year_start_month の1日か判定
   c. 該当する場合:
      - next_receipt_number を 1 にリセット
      - receipt_number_prefix の年部分を自動更新
        例: "RC-2025-" → "RC-2026-"（現在の年を使用）
        ※ プレフィックスに年が含まれない場合は更新しない（正規表現で年4桁を検出）
      - 監査ログ: RECEIPT_NUMBER_RESET を記録

2. 手動リセット（auto_reset_number = FALSE の場合）
   a. ADMIN が PUT /admin/receipt-settings で next_receipt_number を手動変更
   b. 任意のタイミングで変更可能
```

### 重複発行の警告

```
1. POST /admin/receipts のバリデーション時
   a. member_payment_id が指定されている場合、同一 member_payment_id で
      既に発行済み（voided_at IS NULL）の receipts を検索

   c. 発行はブロックしない（分割領収書のユースケースがあるため）

2. GET /admin/receipts/preview のプレビュー時
   a. 同一の重複チェックを実行し、warnings を返却
   b. フロントエンドで「既に発行済みの領収書があります。続行しますか？」確認ダイアログを表示
   c. ADMIN が確認して発行ボタンを押した場合のみ POST /admin/receipts を呼び出し

### PDF 生成の技術選定

```
1. ライブラリ: OpenPDF 2.x（iText 5 の LGPL フォーク）
   a. 商用利用可能（LGPL。iText 7 の AGPL 制約を回避）
   b. 日本語フォント: IPAex明朝（領収書の正式感）を PDF に埋め込み
   c. フォントファイル: src/main/resources/fonts/ipaexm.ttf に配置

   a. Apache Batik で SVG → PNG に変換（解像度: 300dpi）

   c. 印影サイズ: 24mm × 24mm（一般的な認印サイズ）

   a. 領収書発行時に非同期（@Async）で生成
   b. 生成完了後に receipts.pdf_storage_key を UPDATE
   c. 生成失敗時: WARN ログ + pdf_storage_key = NULL のまま

   a. Thymeleaf → HTML → PDF 変換は品質が安定しない

### 電子帳簿保存法への対応

```
1. 改ざん防止措置
   a. S3 Object Lock（Compliance モード、7年）で PDF の改ざん・削除を防止
   b. receipts テーブルの UPDATE は voided_at / voided_by / voided_reason のみ許可
      description / amount / tax_rate 等のデータ項目は UPDATE 禁止（Service 層で制約）
   c. 無効化（void）は新しい状態の追加であり、元データの改ざんではない

2. 検索機能（電子帳簿保存法の3要件）
   a. 日付検索: GET /admin/receipts の issued_from / issued_to パラメータ
   b. 金額検索: GET /admin/receipts の amount_min / amount_max パラメータ
   c. 取引先検索: GET /admin/receipts の recipient_name パラメータ（部分一致）

3. ディスプレイ・プリンタ出力
   a. GET /admin/receipts/{id}/pdf で PDF ダウンロード → ブラウザ表示・印刷

   d. Object Lock: Compliance モード、Retention 7年
```

### 承認制ワークフロー（DRAFT → ISSUED）

```
1. DEPUTY_ADMIN が POST /admin/receipts で status=DRAFT を指定して下書き作成
   a. receipt_number は採番しない（NULL）
   b. PDF は生成しない
   c. プッシュ通知・メール送信は行わない

2. ADMIN が GET /admin/receipts?status=DRAFT で下書き一覧を確認
3. ADMIN が PATCH /admin/receipts/{id}/approve で承認
   a. status を DRAFT → ISSUED に遷移
   b. receipt_number を採番
   c. PDF を非同期生成
   d. 受領者に通知（プッシュ / メール）
   e. 監査ログ: RECEIPT_APPROVED を記録

4. ADMIN が直接 POST /admin/receipts で status を指定しない場合
   → デフォルト status=ISSUED で即時発行（従来の動作と同一。承認フローを使わない場合）

5. DEPUTY_ADMIN は DRAFT の作成のみ可能。ISSUED への直接発行は ADMIN のみ
   （RECEIPT_MANAGE パーミッションがあっても DEPUTY_ADMIN は DRAFT 止まり）
```

### 重要な判定ロジック

- **発行者情報のスナップショット**: 領収書の発行者名・住所・登録番号は発行時点の値を `receipts` テーブルに複製。後から `receipt_issuer_settings` を変更しても既発行の領収書は不変（法的要件）
- **受領者情報のスナップショット**: 同上。ユーザーが住所を変更しても既発行の領収書は不変
- **領収書番号の連番保証**: `SELECT ... FOR UPDATE` による排他制御。同時発行時の番号重複を防止。一括発行（bulk）では必要件数分を一度にインクリメント（例: 5件なら `next_receipt_number += 5`）し、即座にロック解放。その後、各領収書の INSERT は採番済みの番号を使用（ロック保持時間を最小化）
- **無効化の不可逆性**: `voided_at` が設定された領収書は再有効化不可。新しい領収書を発行して対応
- **1支払い複数領収書**: `member_payment_id` に対して複数の領収書を発行可能（分割領収書。例: 会費と雑費を分けて領収）。合計金額が元の支払い額を超えるバリデーションは行わない（実務上、端数処理等で合わない場合がある）
- **手動発行**: `member_payment_id = NULL` で支払い実績に紐付かない領収書を発行可能。任意の金額・但し書きを設定
- **電子印鑑の連動**: `seal_stamp_logs.target_type = 'RECEIPT'` で F05.3 の印鑑検証（`GET /seal/stamps/{id}/verify`）が使用可能。領収書の無効化時は印鑑も自動取り消し
- **PDF 保持期間**: 7年（電子帳簿保存法）。S3 Lifecycle Policy で `Glacier Deep Archive` に移行後、7年経過で削除
- **但し書き候補の生成**: 直近30日以内の `attendance_required = TRUE` のイベント + スコープ内の `payment_items` から候補を自動生成。ADMIN が選択 or 手動入力
- **税額の計算**: `tax_amount = FLOOR(amount * tax_rate / (100 + tax_rate))`（税込→税抜の逆算。1円未満切り捨て）。`amount_excl_tax = amount - tax_amount`
- **メール送信**: `email_delivery.enabled = true` の場合、PDF 生成完了後に非同期送信。送信失敗は領収書発行の成否に影響しない。一括発行（bulk）ではメール送信を含めない（スパム判定回避）。一括発行後は `POST /admin/receipts/{id}/send-email` で個別送信
- **CSV エクスポート**: BOM 付き UTF-8 で出力（Excel の文字化け防止）。複数税率の領収書は明細行ごとに1行。最大 10,000件
- **PDF ロゴ**: `receipt_issuer_settings.logo_storage_key` に画像キーを保持。アップロード時に 200x60px にリサイズ。ロゴなしの場合はテキストタイトルのみ
- **PDF フッター**: `receipt_issuer_settings.custom_footer` のプレーンテキスト（最大500文字）を発行者情報の下に配置
- **論理削除の不使用**: `receipts` テーブルに `deleted_at` は設けない。領収書は法的文書のため削除不可。取り消しは `voided_at` で管理し、元データは永続保持
- **発行プリセット**: スコープ内で最大30件。但し書きテンプレート・金額・税率・明細行を保存。`POST /admin/receipts` に `preset_id` 指定でデフォルト値適用（個別上書き可能）
- **発行キュー**: Stripe 決済完了時に `ReceiptQueueEventListener` が自動投入。ADMIN がキュー一覧を確認 → 承認で発行。90日超の PENDING は自動 SKIPPED + アラート
- **承認制ワークフロー**: DEPUTY_ADMIN は `status=DRAFT` で下書き作成のみ。ADMIN が `PATCH /approve` で ISSUED に遷移（採番・PDF生成・通知はこの時点で実行）。ADMIN が直接発行する場合はデフォルト `status=ISSUED` で従来通り
- **一括無効化**: `POST /admin/receipts/bulk-void` で最大50件。ADMIN のみ。各領収書に対して個別 void と同等の処理（PDF 再生成・印鑑取り消し・通知・監査ログ）
- **ZIP 一括ダウンロード**: 非同期生成（S3 に ZIP アップロード。24時間保持で自動削除）。最大500件。`GET /download-zip/{jobId}` で進捗確認 + ダウンロード
- **年間サマリー**: `GET /my/receipts/annual-summary` で年間合計・件数・税率別合計を返却。無効化済み領収書は合計から除外
- **F08.2 連携**: `GET /members/{id}/payments` のレスポンスに `receipt_ids` 配列を追加（F08.2 側の変更。Phase 8 統合テスト時に対応）
- **無効化の権限制限**: `POST /admin/receipts/{id}/void` は ADMIN のみ実行可能。DEPUTY_ADMIN は `RECEIPT_MANAGE` パーミッションがあっても不可。法的文書の取り消しは最高権限者に制限（電子帳簿保存法の改ざん防止の観点）
- **再発行フロー**: `POST /admin/receipts/{id}/reissue` で無効化済み領収書のデータを流用してプレビュー取得 → `POST /admin/receipts` で正式発行。元の領収書が未無効化の場合は 409 で拒否（無効化せずに再発行はできない）。再発行元の `receipt_number` はレスポンスに `reissue_source_receipt_number` として含め、トレーサビリティを確保

## 6. セキュリティ考慮事項

- **認可チェック**: `/admin/receipts/**` は ADMIN + DEPUTY_ADMIN（`RECEIPT_MANAGE`）のみ。`/my/receipts/**` は自分宛の領収書のみ。他人の領収書 PDF へのアクセスは 403
- **PDF アクセス制御**: Pre-signed URL（24時間有効）で S3 から直接ダウンロード。URL 推測によるアクセスを防止するため、`pdf_storage_key` にランダム UUID を含める
- **インボイス登録番号の検証**: 形式チェック（`/^T\d{13}$/`）のみ。国税庁 API との連携による実在性検証は行わない（API が不安定なため。ADMIN の自己申告を信頼）
- **レートリミット**: `Bucket4j` で以下の制限を適用
  - 領収書発行（`POST /admin/receipts`）: 1分間20件/ユーザー
  - 一括発行（`POST /admin/receipts/bulk`）: 1分間5件/ユーザー
  - PDF ダウンロード: 1分間30件/ユーザー
  - メール送信（`POST /admin/receipts/{id}/send-email`）: 1時間30件/ユーザー（メールスパム防止）
  - CSV エクスポート: 1分間3件/ユーザー
  - ロゴアップロード: 1分間5件/ユーザー
  - 一括無効化（`POST /admin/receipts/bulk-void`）: 1分間3件/ユーザー
  - キュー一括承認（`POST /admin/receipt-queue/bulk-approve`）: 1分間5件/ユーザー
  - ZIP ダウンロード生成: 1時間10件/ユーザー
- **監査ログ**: 以下のイベントを記録
  - `RECEIPT_ISSUED`: 領収書発行（receipt_number, amount, recipient）
  - `RECEIPT_VOIDED`: 領収書無効化（receipt_number, reason）
  - `RECEIPT_SETTINGS_UPDATED`: 発行者設定変更
  - `RECEIPT_BULK_ISSUED`: 一括発行（件数）
  - `RECEIPT_PDF_DOWNLOADED`: PDF ダウンロード（receipt_number, downloader）
  - `RECEIPT_NUMBER_RESET`: 領収書番号の年度リセット（scope, old_prefix → new_prefix）
  - `RECEIPT_EMAIL_SENT`: 領収書メール送信（receipt_number, recipient_email）
  - `RECEIPT_CSV_EXPORTED`: CSV エクスポート（scope, 件数, 期間）
  - `RECEIPT_LOGO_UPLOADED`: ロゴ画像アップロード
  - `RECEIPT_REISSUED`: 無効化後の再発行（new_receipt_number, source_receipt_number）
  - `RECEIPT_APPROVED`: 下書き承認（receipt_number, approver）
  - `RECEIPT_BULK_VOIDED`: 一括無効化（件数, reason）
  - `RECEIPT_QUEUE_APPROVED`: キューアイテム承認（receipt_number, queue_id）
  - `RECEIPT_QUEUE_BULK_APPROVED`: キュー一括承認（件数）
  - `RECEIPT_ZIP_GENERATED`: ZIP 一括ダウンロード生成（件数, 期間）
- **データ改ざん防止**: `receipts` テーブルの `description` / `amount` / `tax_rate` / `recipient_name` 等のデータ項目は発行後の UPDATE 禁止（Service 層で制約）。修正が必要な場合は無効化 + 再発行で対応（電子帳簿保存法の改ざん防止要件）
- **S3 Object Lock**: 領収書 PDF 専用バケット `mannschaft-receipts` に Compliance モード（Retention 7年）を設定。アップロード後の上書き・削除を物理的に防止
