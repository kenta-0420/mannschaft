## 1. 概要

新メンバーがチーム・組織に加入した際に「何をすればいいか」を順序立てて案内するガイド付きチェックリスト機能。プロフィール設定・アプリの使い方・チームルールの理解・必要書類の提出を一連のフローで完了させることで、新メンバーの定着率向上と ADMIN の個別対応コストを削減する。ADMIN がテンプレートとしてチェックリストの手順を定義し、メンバー加入時に自動でオンボーディングが開始される。各ステップの自動完了検知（フォーム提出・ナレッジベース閲覧・プロフィール完了）により、手動チェックの手間を最小化する。

**主要ユースケース**:
- スポーツチーム: プロフィール写真設定 → 入団申込書提出（F05.7）→ チームルール確認（F06.5）→ 保険加入手続き → 練習スケジュール確認
- マンション管理組合: 居住者情報登録 → 管理規約確認（F06.5）→ ゴミ出しルール確認 → 駐車場申請（F05.7）→ 防災訓練日程確認
- 企業: 社員情報入力 → 就業規則確認 → 情報セキュリティ研修 → 各種届出提出 → 業務マニュアル確認
- 整骨院: スタッフプロフィール設定 → 資格情報登録（F07.5）→ 施術マニュアル確認 → シフト希望提出

### テーブル定義

#### `onboarding_templates`

オンボーディングテンプレート。ADMIN がスコープ（チーム/組織）ごとに作成する。スコープ内でアクティブなテンプレートは1つのみ。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `name` | VARCHAR(100) | NO | — | テンプレート名（例: 「新入部員オンボーディング」「新入居者ガイド」） |
| `description` | VARCHAR(500) | YES | NULL | テンプレートの説明（オンボーディング開始時にメンバーに表示） |
| `welcome_message` | VARCHAR(1000) | YES | NULL | ウェルカムメッセージ（オンボーディング開始画面に表示） |
| `status` | VARCHAR(20) | NO | 'DRAFT' | ステータス（`DRAFT` / `ACTIVE` / `ARCHIVED`） |
| `is_order_enforced` | BOOLEAN | NO | FALSE | ステップの順序を強制するか（TRUE = 前のステップ完了まで次に進めない） |
| `deadline_days` | SMALLINT | YES | NULL | 加入日からの完了期限（日数）。NULL = 期限なし |
| `reminder_days_before` | SMALLINT | YES | 3 | 期限の何日前にリマインダーを送信するか。NULL = リマインダーなし |
| `is_admin_notified_on_complete` | BOOLEAN | NO | TRUE | 全ステップ完了時に ADMIN へ通知するか |
| `is_timeline_posted_on_complete` | BOOLEAN | NO | FALSE | 全ステップ完了時にタイムラインに投稿するか |
| `preset_id` | BIGINT UNSIGNED | YES | NULL | FK → system_onboarding_presets。プリセットから作成した場合のみ |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_ot_scope (scope_type, scope_id, status)                   -- スコープ別のテンプレート一覧
INDEX idx_ot_created_by (created_by)                                 -- 作成者別一覧
```

**制約・備考**
- `scope_type` + `scope_id` はポリモーフィック参照。FK 制約なし（アプリ層で検証）
- `scope_type` の CHECK 制約:
  ```sql
  CONSTRAINT chk_ot_scope_type
    CHECK (scope_type IN ('TEAM', 'ORGANIZATION'))
  ```
- `status` の CHECK 制約:
  ```sql
  CONSTRAINT chk_ot_status
    CHECK (status IN ('DRAFT', 'ACTIVE', 'ARCHIVED'))
  ```
- 1スコープあたり `ACTIVE` なテンプレートは最大 **1つ**。新テンプレートを ACTIVE にすると、既存の ACTIVE テンプレートは自動的に ARCHIVED に遷移。アプリ層で制御
- 1スコープあたり最大 **10 テンプレート**（`status` に関わらず論理削除されていないもの）
- `status` のライフサイクル: `DRAFT` → `ACTIVE` → `ARCHIVED`
  - `DRAFT`: 下書き。ステップの追加・変更が可能。オンボーディング自動開始の対象外
  - `ACTIVE`: 有効。新メンバー加入時に自動開始。**ステップの追加・変更・削除は不可**（進行中のオンボーディングとの整合性を保つため）
  - `ARCHIVED`: アーカイブ済み。新規開始の対象外。既存の進行中オンボーディングは完了まで継続
- テンプレートを論理削除しても、既存の `onboarding_progresses` は閲覧可能
- `deadline_days`: 0 を指定すると「当日中」を意味する。最大 365 日
- `reminder_days_before`: `deadline_days` が NULL の場合は無視される

#### `onboarding_template_steps`

テンプレートのステップ定義。ADMIN がオンボーディングの各手順を設定する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `template_id` | BIGINT UNSIGNED | NO | — | FK → onboarding_templates。ON DELETE CASCADE |
| `title` | VARCHAR(100) | NO | — | ステップタイトル（例: 「プロフィール写真を設定する」） |
| `description` | VARCHAR(1000) | YES | NULL | ステップの説明・手順案内 |
| `step_type` | VARCHAR(20) | NO | — | ステップ種別（後述） |
| `reference_id` | BIGINT UNSIGNED | YES | NULL | 外部参照 ID（`FORM` → form_templates.id、`KNOWLEDGE_BASE` → kb_pages.id） |
| `reference_url` | VARCHAR(500) | YES | NULL | `URL` 型の場合のリンク先 URL |
| `deadline_offset_days` | SMALLINT | YES | NULL | 加入日からこのステップの期限（日数）。NULL = テンプレートの `deadline_days` に従う |
| `sort_order` | INT | NO | 0 | 表示順（ドラッグ＆ドロップで並び替え） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_ots_template (template_id, sort_order)    -- テンプレート別のステップ一覧
```

**制約・備考**
- 1テンプレートあたり最大 **20 ステップ**
- `step_type` の種類:
  - `MANUAL`: 手動チェック。メンバーが自分でチェックを入れる（自由記述の手順案内。例: 「保険に加入してください」）
  - `URL`: URL リンク。メンバーがリンク先を確認後にチェックを入れる。`reference_url` が必須
  - `FORM`: フォーム提出（F05.7 連携）。`reference_id` に `form_templates.id` を設定。対象フォームを提出すると自動完了
  - `KNOWLEDGE_BASE`: ナレッジベース閲覧（F06.5 連携）。`reference_id` に `kb_pages.id` を設定。対象ページを閲覧すると自動完了
  - `PROFILE_COMPLETION`: プロフィール完了。プロフィールの必須フィールド（display_name, avatar_url）が全て入力済みになると自動完了
- `step_type` の CHECK 制約:
  ```sql
  CONSTRAINT chk_ots_step_type
    CHECK (step_type IN ('MANUAL', 'URL', 'FORM', 'KNOWLEDGE_BASE', 'PROFILE_COMPLETION'))
  ```
- `reference_id` の必須条件:
  - `FORM`: `reference_id` NOT NULL かつ `form_templates` に存在すること（アプリ層で検証）
  - `KNOWLEDGE_BASE`: `reference_id` NOT NULL かつ `kb_pages` に存在すること（アプリ層で検証）
  - `MANUAL` / `URL` / `PROFILE_COMPLETION`: `reference_id` は NULL
- `reference_url` の必須条件: `URL` 型の場合のみ NOT NULL（アプリ層で検証）
- ステップ定義の変更ルール:
  - テンプレートが `DRAFT` の場合: 追加・削除・変更すべて可能
  - テンプレートが `ACTIVE` / `ARCHIVED` の場合: 変更不可
- `deadline_offset_days`: ステップ個別の期限。テンプレートの `deadline_days` よりも細かい粒度で期限を設定可能。例: 加入3日以内にプロフィール設定、7日以内にフォーム提出

#### `onboarding_progresses`

メンバーごとのオンボーディング進捗レコード。メンバーがスコープに加入した時点で自動 INSERT。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `template_id` | BIGINT UNSIGNED | NO | — | FK → onboarding_templates。ON DELETE RESTRICT |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（対象メンバー）。ON DELETE CASCADE |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `status` | VARCHAR(20) | NO | 'IN_PROGRESS' | 進捗ステータス（`IN_PROGRESS` / `COMPLETED` / `SKIPPED`） |
| `total_steps` | SMALLINT | NO | — | テンプレートの総ステップ数（開始時にスナップショット） |
| `completed_steps` | SMALLINT | NO | 0 | 完了済みステップ数（denormalize） |
| `deadline_at` | DATETIME | YES | NULL | 完了期限（加入日時 + `deadline_days`。NULL = 期限なし） |
| `started_at` | DATETIME | NO | CURRENT_TIMESTAMP | オンボーディング開始日時（= 加入日時） |
| `completed_at` | DATETIME | YES | NULL | 全ステップ完了日時 |
| `last_reminded_at` | DATETIME | YES | NULL | 最後にリマインダーを送信した日時（重複送信防止用） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_op_template_user (template_id, user_id)                                -- テンプレート×ユーザー一意
INDEX idx_op_scope (scope_type, scope_id, status)                                     -- スコープ別の進捗一覧
INDEX idx_op_user (user_id, status)                                                    -- ユーザー別の進捗一覧
INDEX idx_op_deadline (status, deadline_at)                                             -- リマインダーバッチ用
INDEX idx_op_scope_completed (scope_type, scope_id, completed_at DESC)                 -- 完了者一覧
```

**制約・備考**
- `scope_type` の CHECK 制約:
  ```sql
  CONSTRAINT chk_op_scope_type
    CHECK (scope_type IN ('TEAM', 'ORGANIZATION'))
  ```
- `status` の CHECK 制約:
  ```sql
  CONSTRAINT chk_op_status
    CHECK (status IN ('IN_PROGRESS', 'COMPLETED', 'SKIPPED'))
  ```
- `status` 遷移:
  - `IN_PROGRESS` → `COMPLETED`: 全ステップ完了時に自動遷移
  - `IN_PROGRESS` → `SKIPPED`: ADMIN が手動でスキップ（オンボーディング免除）
- `total_steps`: テンプレートの ACTIVE 時点のステップ数をスナップショット。テンプレートが後で変更されても進捗の完了率計算には影響しない
- `completed_steps`: `onboarding_step_completions` の COUNT と同期。ステップ完了時にインクリメント。完了率 = `completed_steps / total_steps`
- 同一ユーザーが同一テンプレートで複数の進捗を持つことはない（UNIQUE 制約）
- `user_id` ON DELETE CASCADE: ユーザー退会時に進捗レコードも削除
- `template_id` ON DELETE RESTRICT: テンプレートの論理削除時に物理削除はされない。進捗が残っている場合はテンプレートを物理削除できない
- リマインダー送信の重複防止: `last_reminded_at` が同日（JST）であればスキップ

#### `onboarding_step_completions`

メンバーごとのステップ完了記録。各ステップの完了/未完了を管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `progress_id` | BIGINT UNSIGNED | NO | — | FK → onboarding_progresses。ON DELETE CASCADE |
| `step_id` | BIGINT UNSIGNED | NO | — | FK → onboarding_template_steps。ON DELETE RESTRICT |
| `completion_type` | VARCHAR(20) | NO | — | 完了方法（`MANUAL` / `AUTO_FORM` / `AUTO_KB_VIEW` / `AUTO_PROFILE` / `ADMIN_OVERRIDE`） |
| `completed_at` | DATETIME | NO | CURRENT_TIMESTAMP | 完了日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_osc_progress_step (progress_id, step_id)      -- 進捗×ステップ一意
INDEX idx_osc_step (step_id)                                  -- ステップ別の完了状況
```

**制約・備考**
- `completion_type` の CHECK 制約:
  ```sql
  CONSTRAINT chk_osc_completion_type
    CHECK (completion_type IN ('MANUAL', 'AUTO_FORM', 'AUTO_KB_VIEW', 'AUTO_PROFILE', 'ADMIN_OVERRIDE'))
  ```
- `completion_type` の意味:
  - `MANUAL`: メンバーが手動でチェック（`MANUAL` / `URL` 型ステップ）
  - `AUTO_FORM`: `FormSubmittedEvent` による自動完了（`FORM` 型ステップ）
  - `AUTO_KB_VIEW`: `PageViewEvent` による自動完了（`KNOWLEDGE_BASE` 型ステップ）
  - `AUTO_PROFILE`: `ProfileUpdatedEvent` による自動完了（`PROFILE_COMPLETION` 型ステップ）
  - `ADMIN_OVERRIDE`: ADMIN が手動でステップを完了にマーク
- 完了レコードが存在しない = 未完了。レコードの INSERT = 完了を意味する
- `progress_id` ON DELETE CASCADE: 進捗削除時にステップ完了記録も削除
- `step_id` ON DELETE RESTRICT: ステップ定義の物理削除は不可（完了記録が参照しているため）

#### `system_onboarding_presets`

SYSTEM_ADMIN が管理する標準テンプレートプリセット。「スポーツチーム新入部員向け」「マンション新入居者向け」等を提供する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `name` | VARCHAR(100) | NO | — | プリセット名（例: 「スポーツチーム新入部員向け」） |
| `description` | VARCHAR(500) | YES | NULL | プリセットの説明 |
| `category` | VARCHAR(30) | NO | — | カテゴリ（`SPORTS` / `RESIDENTIAL` / `BUSINESS` / `MEDICAL` / `EDUCATION` / `OTHER`） |
| `welcome_message` | VARCHAR(1000) | YES | NULL | ウェルカムメッセージ |
| `is_order_enforced` | BOOLEAN | NO | FALSE | ステップ順序の強制 |
| `deadline_days` | SMALLINT | YES | NULL | 完了期限（日数） |
| `steps_json` | JSON | NO | — | ステップ定義の JSON 配列（複製時に `onboarding_template_steps` に展開） |
| `is_active` | BOOLEAN | NO | TRUE | 有効か（FALSE = カタログに表示しない） |
| `sort_order` | INT | NO | 0 | 表示順 |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users（SYSTEM_ADMIN）。ON DELETE RESTRICT |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_sop_category (category, is_active, sort_order)    -- カテゴリ別のプリセット一覧
```

**制約・備考**
- `category` の CHECK 制約:
  ```sql
  CONSTRAINT chk_sop_category
    CHECK (category IN ('SPORTS', 'RESIDENTIAL', 'BUSINESS', 'MEDICAL', 'EDUCATION', 'OTHER'))
  ```
- `steps_json` の形式:
```json
[
  {
    "title": "プロフィール写真を設定する",
    "description": "メンバーにあなたの顔を覚えてもらいましょう！",
    "step_type": "PROFILE_COMPLETION",
    "sort_order": 1
  },
  {
    "title": "チームルールを確認する",
    "description": "活動に参加する前に必ず目を通してください",
    "step_type": "KNOWLEDGE_BASE",
    "sort_order": 2
  },
  {
    "title": "入団申込書を提出する",
    "description": "必要事項を記入して提出してください",
    "step_type": "FORM",
    "sort_order": 3
  }
]
```
- プリセットからテンプレートを作成する場合: `steps_json` を解析して `onboarding_template_steps` にレコードを INSERT。`step_type` が `FORM` / `KNOWLEDGE_BASE` の場合、`reference_id` はテンプレート作成後に ADMIN が個別に設定する（プリセットにはスコープ固有の参照先を含めない）
- 作成後はテンプレートとプリセットは独立（プリセット変更はテンプレートに影響しない）

### ER図（テキスト形式）
```
onboarding_templates (1) ──── (N) onboarding_template_steps
onboarding_templates (1) ──── (N) onboarding_progresses

onboarding_progresses (1) ──── (N) onboarding_step_completions

onboarding_template_steps.reference_id ──→ form_templates (step_type = FORM)
onboarding_template_steps.reference_id ──→ kb_pages (step_type = KNOWLEDGE_BASE)
```

## 5. ビジネスロジック

### オンボーディング自動開始フロー

```
1. メンバーがチーム/組織に加入（F01.2）
2. F01.2 が MemberJoinedEvent を発行
3. OnboardingAutoStartListener が MemberJoinedEvent を購読
   (@TransactionalEventListener(phase = AFTER_COMMIT), @Async("event-pool"))
4. リスナーが以下を実行:
   a. scope_type + scope_id で ACTIVE なテンプレートを検索
   b. テンプレートが存在しない場合: 処理終了（ログ出力のみ）
   c. テンプレートが存在する場合:
      i.   onboarding_progresses に INSERT
           - total_steps: テンプレートのステップ数
           - deadline_at: 現在日時 + deadline_days（NULL の場合は NULL）
           - started_at: 現在日時
      ii.  メンバーにプッシュ通知（F04.3）:
           「{scope_name} へようこそ！オンボーディングが開始されました」
      iii. ApplicationEvent (OnboardingStartedEvent) を発行
```

### テンプレート有効化フロー

```
1. ADMIN が POST /onboarding/templates/{id}/activate を呼び出し
2. Service 層で以下を実行:
   a. テンプレートの status が DRAFT であることを検証
   b. ステップが1つ以上存在することを検証
   c. FORM 型ステップの reference_id が有効な form_templates.id であることを検証
   d. KNOWLEDGE_BASE 型ステップの reference_id が有効な kb_pages.id であることを検証
   e. 同一スコープ内の既存 ACTIVE テンプレートを ARCHIVED に更新
   f. テンプレートの status を ACTIVE に更新
   g. ApplicationEvent (OnboardingTemplateActivatedEvent) を発行
```

### プリセットからのテンプレート作成フロー

```
1. ADMIN がプリセットカタログから選択
2. POST /onboarding/templates に preset_id を含めてリクエスト
3. Service 層で以下を実行:
   a. system_onboarding_presets から steps_json と設定を取得
   b. onboarding_templates にレコードを INSERT（welcome_message, is_order_enforced, deadline_days を複製）
   c. steps_json を解析して onboarding_template_steps にレコードを INSERT
      - FORM / KNOWLEDGE_BASE 型の reference_id は NULL で作成（ADMIN が後で設定）
   d. onboarding_templates.preset_id に参照元を記録
4. 作成後のテンプレートはプリセットから独立（ADMIN が自由に編集可能）
```

### 自動完了検知フロー

#### フォーム提出（F05.7 連携）
```
1. メンバーがフォームを提出 → F05.7 が FormSubmissionCompletedEvent を発行
2. OnboardingFormCompletionListener が購読
   (@TransactionalEventListener(phase = AFTER_COMMIT), @Async("event-pool"))
3. リスナーが以下を実行:
   a. 提出者の user_id で IN_PROGRESS な onboarding_progresses を検索
   b. 各進捗のテンプレートから step_type = 'FORM' かつ reference_id = 提出元 form_templates.id のステップを検索
   c. マッチするステップが存在し、まだ完了していない場合:
      i.   onboarding_step_completions に INSERT (completion_type = 'AUTO_FORM')
      ii.  onboarding_progresses.completed_steps をインクリメント
      iii. 全ステップ完了チェック → 完了なら完了処理を実行
```

#### ナレッジベース閲覧（F06.5 連携）
```
1. メンバーがナレッジベースページを閲覧 → PageViewEvent を発行
2. OnboardingKbViewCompletionListener が購読
   (@TransactionalEventListener(phase = AFTER_COMMIT), @Async("event-pool"))
3. リスナーが以下を実行:
   a. 閲覧者の user_id で IN_PROGRESS な onboarding_progresses を検索
   b. 各進捗のテンプレートから step_type = 'KNOWLEDGE_BASE' かつ reference_id = 閲覧した kb_pages.id のステップを検索
   c. マッチするステップが存在し、まだ完了していない場合:
      i.   onboarding_step_completions に INSERT (completion_type = 'AUTO_KB_VIEW')
      ii.  onboarding_progresses.completed_steps をインクリメント
      iii. 全ステップ完了チェック → 完了なら完了処理を実行
```

#### プロフィール完了
```
1. メンバーがプロフィールを更新 → F01.2 が ProfileUpdatedEvent を発行
2. OnboardingProfileCompletionListener が購読
   (@TransactionalEventListener(phase = AFTER_COMMIT), @Async("event-pool"))
3. リスナーが以下を実行:
   a. 更新者の user_id で IN_PROGRESS な onboarding_progresses を検索
   b. 各進捗のテンプレートから step_type = 'PROFILE_COMPLETION' のステップを検索
   c. マッチするステップが存在し、まだ完了していない場合:
      i.   users テーブルから display_name, avatar_url を取得
      ii.  両フィールドが非 NULL かつ空文字でないことを確認
      iii. 条件を満たす場合:
           - onboarding_step_completions に INSERT (completion_type = 'AUTO_PROFILE')
           - onboarding_progresses.completed_steps をインクリメント
           - 全ステップ完了チェック → 完了なら完了処理を実行
```

### 全ステップ完了処理

```
1. completed_steps == total_steps であることを確認
2. onboarding_progresses.status を COMPLETED に更新
3. onboarding_progresses.completed_at をセット
4. is_admin_notified_on_complete = TRUE の場合:
   a. スコープ内の ADMIN にプッシュ通知（F04.3）:
      「{user_name} がオンボーディングを完了しました」
5. is_timeline_posted_on_complete = TRUE の場合:
   a. タイムラインに自動投稿（F04.1 連携）:
      「{user_name} がオンボーディングを完了しました 🎉」
6. ApplicationEvent (OnboardingCompletedEvent) を発行
```

### リマインダーバッチ

```
1. 日次バッチ（09:00 JST）で実行（@Scheduled）
2. onboarding_progresses.status = 'IN_PROGRESS' かつ deadline_at IS NOT NULL を取得
3. 各進捗について:
   a. deadline_at - reminder_days_before（テンプレートの設定）の日付が今日以降かどうかを判定
   b. リマインダー対象の場合:
      i.   last_reminded_at が今日（JST）であればスキップ（重複送信防止）
      ii.  対象メンバーにプッシュ通知（F04.3）:
           「{template_name} の完了期限が近づいています（期限: {deadline_at}）。未完了のステップを確認してください」
      iii. last_reminded_at を更新
4. 期限超過（deadline_at < 現在日時）の進捗について:
   a. ADMIN にプッシュ通知:
      「{user_name} のオンボーディングが期限を超過しています」
   b. 期限超過通知は1メンバーにつき1回のみ（last_reminded_at で管理）
```

### ステップ順序制御ロジック

```
1. is_order_enforced = TRUE の場合:
   a. メンバーがステップ N を完了しようとした時:
      i.   ステップ 1 〜 N-1 が全て完了済みかチェック
      ii.  未完了のステップが存在する場合: 409 Conflict エラー
           （「先に前のステップを完了してください」）
   b. 自動完了検知の場合:
      i.   同じチェックを実施
      ii.  前のステップが未完了の場合: 自動完了をスキップ（ログ出力）
           次回の前ステップ完了時に再チェックは行わない（メンバーが手動で完了する必要がある）
2. is_order_enforced = FALSE の場合:
   a. 任意の順序でステップを完了可能
```

### 重要な判定ロジック

- **テンプレートの有効化可否**: `DRAFT` かつステップが1つ以上存在すること。`FORM` 型ステップの `reference_id` が有効な `form_templates.id`（`PUBLISHED` 状態）であること。`KNOWLEDGE_BASE` 型ステップの `reference_id` が有効な `kb_pages.id`（`PUBLISHED` 状態）であること
- **進捗の可視性**: ADMIN は全メンバーの進捗を閲覧可能。MEMBER は自分の進捗のみ
- **リマインダーの送信条件**: `IN_PROGRESS` かつ `deadline_at IS NOT NULL` かつ `deadline_at - reminder_days_before <= 現在日時 < deadline_at`
- **自動完了と is_order_enforced の相互作用**: `is_order_enforced = TRUE` の場合でも自動完了検知のイベントは常に購読する。ただし前のステップが未完了の場合は完了処理をスキップする
- **テンプレート削除**: `DRAFT` テンプレートのみ論理削除可能。`ACTIVE` テンプレートは先にアーカイブしてからのみ削除可能。`ARCHIVED` テンプレートは進行中の進捗がなければ削除可能
- **進捗リセット**: `onboarding_step_completions` を全て物理削除し、`completed_steps` を 0 に戻し、`status` を `IN_PROGRESS` に戻す。`completed_at` を NULL に戻す

### 通知トリガー一覧

| トリガー | 通知先 | 条件 |
|---------|-------|------|
| オンボーディング自動開始 | 対象メンバー | MemberJoinedEvent 受信時かつ ACTIVE テンプレートあり |
| ステップ完了（自動検知） | 対象メンバー | AUTO_FORM / AUTO_KB_VIEW / AUTO_PROFILE で自動完了時（「{step_title} が完了しました」） |
| 全ステップ完了 | ADMIN | is_admin_notified_on_complete = TRUE 時 |
| リマインダー（自動） | 未完了メンバー | 期限 N 日前（日次バッチ） |
| リマインダー（手動） | 未完了メンバー | ADMIN が手動送信 |
| 期限超過 | ADMIN | 期限超過の進捗あり（日次バッチ） |

## 6. セキュリティ考慮事項

- **認可チェック**: OnboardingTemplateService / OnboardingProgressService の入り口で `scope_type` / `scope_id` と `currentUser` の所属を必ず検証する。進捗の閲覧は対象メンバー本人・ADMIN に限定
- **レートリミット**: 手動リマインダー送信 API に `Bucket4j` で 1スコープあたり1日3回の制限を適用（スパム防止）
- **楽観的ロック**: `onboarding_templates` に `version` カラムを適用。更新時に `version` の一致を検証
- **イベントリスナーの冪等性**: 自動完了検知のリスナーは、`onboarding_step_completions` の UNIQUE 制約（`progress_id`, `step_id`）で重複 INSERT を防止。`DataIntegrityViolationException` をキャッチして無視する
- **reference_id の有効性検証**: テンプレート有効化時に `FORM` / `KNOWLEDGE_BASE` 型ステップの `reference_id` が有効なレコードを指しているかを検証。参照先が後から削除された場合、該当ステップは ADMIN が手動で完了マークするか、テンプレートを再作成する必要がある
- **スコープ間のデータ分離**: 異なるスコープの進捗やテンプレートへのアクセスを防止するため、全クエリに `scope_type` + `scope_id` 条件を含める
