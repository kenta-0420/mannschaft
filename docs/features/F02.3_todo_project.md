## 1. 概要

個人・チーム・組織のタスク管理と、目標（プロジェクト）に対する進捗可視化を統合する機能。TODO はチェックボックス付きのタスクリストで、期限・優先度・担当者の管理を行う。プロジェクトは「文化祭準備」「結婚式」「夏休みの宿題」等の目標を定義し、紐付く TODO の完了率から進捗を自動算出してダッシュボードのゲージ・プログレスバーで可視化する。

**TODO 単体での利用**も**プロジェクトに紐付けた利用**も可能。プロジェクトを使わずに TODO だけ運用するケース（日常のタスク管理）と、プロジェクトに TODO を紐付けてゴールまでの進捗を追跡するケース（イベント準備等）の両方をサポート。

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全組織・チームの TODO・プロジェクトの参照 |
| ADMIN | TODO・プロジェクトの CRUD・担当者割り当て・全メンバーの TODO 閲覧 |
| DEPUTY_ADMIN | `MANAGE_TODO` 権限を持つ場合: TODO・プロジェクトの CRUD・担当者割り当て |
| MEMBER | 自分に割り当てられた TODO の閲覧・ステータス変更・コメント。自分の TODO 作成。チーム/組織の公開プロジェクト閲覧 |
| SUPPORTER | チーム/組織の公開プロジェクトの閲覧（TODO は対象外） |
| GUEST | 対象外 |

### 対象レベル
- [x] 組織 (Organization) — 組織横断のプロジェクト・TODO
- [x] チーム (Team) — チーム内のプロジェクト・TODO
- [x] 個人 (Personal) — 個人のプロジェクト・TODO

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `projects` | プロジェクト（目標）定義。ゴール・期限・進捗率を管理 | あり（`deleted_at`） |

### テーブル定義

#### `projects`

目標・プロジェクトの定義。文化祭準備、結婚式、子供の夏休み宿題等。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | `PERSONAL` / `TEAM` / `ORGANIZATION` |
| `scope_id` | BIGINT UNSIGNED | NO | — | users.id / teams.id / organizations.id |
| `title` | VARCHAR(200) | NO | — | プロジェクト名（例: '文化祭2026 準備', '結婚式 2026年秋', '夏休みの宿題'） |
| `description` | TEXT | YES | NULL | プロジェクトの説明（Markdown 対応） |
| `emoji` | VARCHAR(10) | YES | NULL | アイコン絵文字（ダッシュボード表示用。例: '🎪', '💒', '📚'） |
| `color` | VARCHAR(7) | YES | NULL | テーマカラー（HEX。例: '#FF6B6B'）。プログレスバーの色 |
| `due_date` | DATE | YES | NULL | 目標期限日（NULL = 期限なし） |
| `status` | VARCHAR(20) | NO | 'ACTIVE' | ステータス。`ACTIVE`（進行中）/ `COMPLETED`（完了）/ `ARCHIVED`（アーカイブ） |
| `progress_rate` | DECIMAL(5,2) | NO | 0.00 | 進捗率（%。0.00〜100.00。TODO の完了率から自動算出。非正規化） |
| `total_todos` | SMALLINT UNSIGNED | NO | 0 | 紐付く TODO の総数（非正規化） |
| `completed_todos` | SMALLINT UNSIGNED | NO | 0 | 完了済み TODO の数（非正規化） |
| `visibility` | VARCHAR(20) | NO | 'MEMBERS_ONLY' | 公開範囲。`PRIVATE`（作成者のみ。個人プロジェクト用）/ `MEMBERS_ONLY`（メンバーのみ）/ `PUBLIC`（SUPPORTER も閲覧可） |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT |
| `completed_at` | DATETIME | YES | NULL | 完了日時（`progress_rate = 100` or ADMIN が手動完了） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
UNIQUE KEY uq_proj_scope_title (scope_type, scope_id, title, deleted_at)  -- 同一スコープ内のプロジェクト名重複防止
INDEX idx_proj_scope_status (scope_type, scope_id, status, due_date)      -- スコープ別のプロジェクト一覧
INDEX idx_proj_due_date (due_date, status)                                 -- 期限順の一覧
INDEX idx_proj_created_by (created_by)                                     -- 作成者別一覧
```

**制約・備考**
- `progress_rate` は `completed_todos / total_todos * 100` で自動算出。TODO の INSERT / UPDATE / DELETE 時にアトミック更新（`UPDATE projects SET completed_todos = completed_todos ± 1, progress_rate = ... WHERE id = :projectId`）
- `total_todos = 0` の場合: `progress_rate = 0`（0除算防止）
- **自動完了**: `progress_rate` が 100.00 に到達した場合、`status` は自動で `COMPLETED` に遷移しない（ADMIN が手動で完了にする or そのまま ACTIVE で運用）。理由: 追加 TODO が後から発生する可能性がある
- **期限までの残り日数**: API レスポンスで `days_remaining = due_date - today` を動的算出（DB カラムは不要）
- `scope_type = 'PERSONAL'` の場合: `scope_id = users.id`。個人のプロジェクト（宿題・資格勉強等）
- `emoji` / `color` はダッシュボードウィジェットでの視覚的な識別に使用。設定は任意
- **プロジェクト上限**: 1スコープあたり ACTIVE プロジェクトは最大 **20件**（アプリ層で検証。COMPLETED / ARCHIVED は上限に含まない）
- `created_by` ON DELETE RESTRICT: 作成者の退会時は先に `created_by` を他の ADMIN に移譲する必要がある（F09.5 と同パターン）
- **ARCHIVED 状態**: ADMIN が手動で `ACTIVE` → `ARCHIVED` に変更。完了したプロジェクトを一覧から非表示にする用途。`status` の更新時に `updated_at` が自動セットされるため、アーカイブ日時は `updated_at` で判別可能
- **visibility の制約**: `PRIVATE` は `scope_type = 'PERSONAL'` のみ許可。チーム/組織スコープでは `MEMBERS_ONLY` / `PUBLIC` のみ選択可能（チーム/組織のプロジェクトは本質的に「メンバーと共有する目標」であり、ADMIN だけが見える目標はチーム運営として意味がないため。バリデーション: `scope_type != 'PERSONAL' && visibility == 'PRIVATE'` → 400 エラー）

#### `project_milestones`

マイルストーン（中間目標）。プロジェクトを段階的に分割する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `project_id` | BIGINT UNSIGNED | NO | — | FK → projects。ON DELETE CASCADE |
| `title` | VARCHAR(200) | NO | — | マイルストーン名（例: '企画書完成', '会場予約', '招待状発送'） |
| `due_date` | DATE | YES | NULL | 期限日 |
| `sort_order` | SMALLINT UNSIGNED | NO | 0 | 表示順 |
| `is_completed` | BOOLEAN | NO | FALSE | 完了フラグ |
| `completed_at` | DATETIME | YES | NULL | 完了日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_pm_project_order (project_id, sort_order)     -- プロジェクト内のマイルストーン一覧
UNIQUE KEY uq_pm_project_title (project_id, title)  -- 同一プロジェクト内のマイルストーン名重複防止
```

**制約・備考**
- マイルストーンは TODO の上位グルーピング。TODO に `milestone_id` を設定してマイルストーンに紐付ける（任意。紐付けなしの TODO も可）
- マイルストーンの進捗率: 紐付く TODO の完了率で動的算出（DB に非正規化カラムは持たない。レスポンスで計算して返却）
- 1プロジェクトあたりのマイルストーン上限: 20件（アプリ層で検証）
- `is_completed` は手動切り替え。紐付く TODO が全完了しても自動完了しない（マイルストーンの「完了」は ADMIN の判断）

#### `todos`

TODO タスク本体。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | `PERSONAL` / `TEAM` / `ORGANIZATION` |
| `scope_id` | BIGINT UNSIGNED | NO | — | users.id / teams.id / organizations.id |
| `project_id` | BIGINT UNSIGNED | YES | NULL | FK → projects。ON DELETE SET NULL。紐付くプロジェクト（NULL = プロジェクトなし。単独 TODO） |
| `milestone_id` | BIGINT UNSIGNED | YES | NULL | FK → project_milestones。ON DELETE SET NULL。紐付くマイルストーン（NULL = マイルストーンなし） |
| `title` | VARCHAR(300) | NO | — | TODO タイトル（例: '会場の下見', '招待状のデザイン作成', '数学ドリル P.10-20'） |
| `description` | TEXT | YES | NULL | 詳細説明 |
| `status` | VARCHAR(20) | NO | 'OPEN' | ステータス。`OPEN`（未着手）/ `IN_PROGRESS`（進行中）/ `COMPLETED`（完了） |
| `priority` | VARCHAR(10) | NO | 'MEDIUM' | 優先度。`LOW` / `MEDIUM` / `HIGH` / `URGENT` |
| `due_date` | DATE | YES | NULL | 期限日（NULL = 期限なし） |
| `due_time` | TIME | YES | NULL | 期限時刻（NULL = 当日中。日付 + 時刻で細かい期限設定が可能） |
| `completed_at` | DATETIME | YES | NULL | 完了日時 |
| `completed_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL。完了操作した人 |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT |
| `sort_order` | INT | NO | 0 | 表示順（同一プロジェクト/マイルストーン内） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_todo_scope_status (scope_type, scope_id, status, priority, due_date)  -- スコープ別の TODO 一覧
INDEX idx_todo_project (project_id, status, sort_order)                          -- プロジェクト内の TODO 一覧
INDEX idx_todo_milestone (milestone_id, status, sort_order)                      -- マイルストーン内の TODO 一覧
INDEX idx_todo_due_date (due_date, status)                                       -- 期限順の一覧・期限切れ検出
INDEX idx_todo_created_by (created_by)                                           -- 作成者別一覧
```

**制約・備考**
- `project_id = NULL` の TODO はプロジェクトに属さない単独タスク（日常の買い物リスト等）
- **スコープ整合性**: `project_id` が非 NULL の場合、`todos.scope_type` / `scope_id` が `projects.scope_type` / `scope_id` と一致することを検証（異なるスコープのプロジェクトに TODO を紐付けることは不可）
- `milestone_id` は `project_id` が非 NULL の場合のみ設定可能（アプリ層で検証。`milestone_id` のマイルストーンが同一 `project_id` に属することも検証）
- **MEMBER の TODO 作成**: MEMBER は自分が担当する TODO を作成可能（`created_by` = 自分、`todo_assignees` に自分を自動追加）。他メンバーへの割り当ては ADMIN / DEPUTY_ADMIN のみ。個人スコープでは常に作成可能
- `created_by` ON DELETE RESTRICT: 退会前に作成 TODO の `created_by` を ADMIN に移譲（F09.5 と同パターン）
- **project_id の変更（プロジェクト間移動）**: `PUT /todos/{id}` で `project_id` を変更可能。以下の制約を適用:
  1. 変更先の project が同一 `scope_type` / `scope_id` に属することを検証（異なるスコープのプロジェクトへの移動は不可）
  2. 旧プロジェクトの `progress_rate` を再計算（TODO が1件減る）
  3. 新プロジェクトの `progress_rate` を再計算（TODO が1件増える。`COMPLETED` の TODO なら `completed_todos` も +1）
  4. `milestone_id` は NULL にリセット（移動先プロジェクトのマイルストーンとの紐付けは手動で再設定）
  5. `project_id` を NULL に変更（プロジェクトから切り離して単独 TODO にする）も許可
  6. `project_id` が NULL → 非 NULL に変更（単独 TODO を後からプロジェクトに紐付け）も許可
- **完了時のプロジェクト進捗更新**: `status` → `COMPLETED` に遷移した時、`project_id` が非 NULL なら `projects.completed_todos` をアトミック +1 → `progress_rate` を再計算。逆に `COMPLETED` → `OPEN`/`IN_PROGRESS` に戻した場合は -1
- **論理削除時のプロジェクト進捗更新**: TODO が論理削除された場合、`projects.total_todos` を -1（`COMPLETED` の TODO なら `completed_todos` も -1）→ `progress_rate` を再計算
- **期限リマインダー**: F04.3 のプッシュ通知と連携。`due_date` の前日・当日にリマインダーを送信（バッチで毎朝チェック）
- `priority` はソートとフィルタに使用。ダッシュボードウィジェットでは `URGENT` / `HIGH` を赤色で強調
- `due_time` は任意。「13時までに提出」のような細かい期限に対応。`due_date` のみ設定の場合は「当日中」扱い

#### `todo_assignees`

TODO の担当者。1つの TODO に複数人を割り当て可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `todo_id` | BIGINT UNSIGNED | NO | — | FK → todos。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `assigned_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL。割り当てた人 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_ta_todo_user (todo_id, user_id)  -- 同一 TODO への重複割り当て防止
INDEX idx_ta_user (user_id)                      -- ユーザーに割り当てられた TODO 一覧
```

**制約・備考**
- 個人 TODO（`scope_type = 'PERSONAL'`）の場合: 担当者は作成者本人のみ（アプリ層で検証）
- チーム/組織 TODO: ADMIN / DEPUTY_ADMIN が任意のメンバーを割り当て可能。MEMBER は自分に割り当てられた TODO のみ閲覧
- 担当者なしの TODO も許可（全員で確認する共有タスク的な使い方）

## 4. API設計

### 組織レベルの API パス

個人は `/api/v1/todos/...`、チームは `/api/v1/teams/{teamId}/todos/...`、組織は `/api/v1/organizations/{orgId}/todos/...` で同一構造の API を提供する。

```json
      "total_todos": 20,
      "completed_todos": 8
    }
  }
}
```

**補足**
- `project_progress`: `project_id` が非 NULL の場合のみ返却。更新後のプロジェクト進捗率を含む（ダッシュボードウィジェットの楽観更新に使用）

**処理の流れ**
1. ステータス遷移の検証（`OPEN` ↔ `IN_PROGRESS` ↔ `COMPLETED` は自由遷移可能）
2. `COMPLETED` に遷移: `completed_at` = NOW()、`completed_by` = currentUser
3. `COMPLETED` から戻す: `completed_at` = NULL、`completed_by` = NULL
4. **プロジェクト進捗の更新**（`project_id` が非 NULL の場合）:
   - `COMPLETED` に遷移: `projects.completed_todos` を +1
   - `COMPLETED` から戻す: `projects.completed_todos` を -1
   - `progress_rate = completed_todos / total_todos * 100` を再計算
5. ダッシュボードウィジェット向けに `TodoStatusChangedEvent` を発行（WebSocket でリアルタイム更新）

```java
/**
 * プロジェクトの進捗率を再計算する
 * TODO の INSERT / UPDATE(status) / DELETE 時にアトミック実行
 */
public void recalculateProjectProgress(Long projectId) {
    if (projectId == null) return;

    // total_todos / completed_todos / progress_rate を単一の原子的 UPDATE で再計算
    // （2回の UPDATE に分けると、間に別のTODO完了が割り込んで不整合になるリスクがある）
    projectRepository.recalculateProgress(projectId);
}

// SQL（単一の原子的 UPDATE）:
// UPDATE projects SET
//   total_todos = (SELECT COUNT(*) FROM todos WHERE project_id = :id AND deleted_at IS NULL),
//   completed_todos = (SELECT COUNT(*) FROM todos WHERE project_id = :id AND status = 'COMPLETED' AND deleted_at IS NULL),
//   progress_rate = IFNULL(
//     (SELECT COUNT(*) FROM todos WHERE project_id = :id AND status = 'COMPLETED' AND deleted_at IS NULL)
//     * 100.0
//     / NULLIF((SELECT COUNT(*) FROM todos WHERE project_id = :id AND deleted_at IS NULL), 0),
//     0
//   ),
//   updated_at = NOW()
```

### 期限リマインダーバッチ

```java
@Scheduled(cron = "0 0 8 * * *")  // 毎朝8時
public void sendTodoReminders() {
    LocalDate today = LocalDate.now();
    LocalDate tomorrow = today.plusDays(1);

    // 明日期限の未完了 TODO
    List<Todo> dueTomorrow = todoRepository
        .findByDueDateAndStatusNot(tomorrow, TodoStatus.COMPLETED);
    for (Todo todo : dueTomorrow) {
        notifyAssignees(todo, NotificationType.TODO_DUE_TOMORROW);
    }

    // 期限超過の未完了 TODO（初回のみ通知。F04.3 の notification_logs で送信済みチェック）
    List<Todo> overdue = todoRepository
        .findByDueDateBeforeAndStatusNot(today, TodoStatus.COMPLETED);
    for (Todo todo : overdue) {
        // notification_logs に TODO_OVERDUE × todo_id の送信記録がなければ通知
        if (!notificationLogService.hasBeenSent(
                NotificationType.TODO_OVERDUE, "TODO", todo.getId())) {
            notifyAssignees(todo, NotificationType.TODO_OVERDUE);
        }
    }

    // プロジェクトの期限が1週間以内
    List<Project> projectsDueSoon = projectRepository
        .findByDueDateBetweenAndStatus(today, today.plusDays(7), ProjectStatus.ACTIVE);
    for (Project project : projectsDueSoon) {
        notifyProjectCreator(project, NotificationType.PROJECT_DUE_SOON);
    }
}
```

## 6. セキュリティ考慮事項

- **認可チェック**: プロジェクト・TODO の CRUD は ADMIN / DEPUTY_ADMIN（`MANAGE_TODO`）。MEMBER は自分に割り当てられた TODO のステータス変更・コメントのみ
- **スコープ分離**: 全 API で `teamId` / `orgId` パスパラメータとスコープメンバーシップを検証
- **MEMBER の TODO 閲覧制限**: チーム/組織 TODO の一覧は自分に割り当てられたもの + 担当者なしの TODO のみ。他人に割り当てられた TODO は非表示（ADMIN は全件閲覧可能）
- **個人プロジェクトの PRIVATE 設定**: `visibility = 'PRIVATE'` のプロジェクトは作成者本人のみ閲覧・操作可能。ADMIN も閲覧不可
- **コメントの XSS 対策**: Markdown レンダリング時に許可タグのホワイトリストで制限
- **コメント編集/削除権限**: 編集は作成者本人のみ。削除は本人 or ADMIN。他人のコメントの編集は不可
- **IDOR 防止**: `todo_id` / `project_id` / `milestone_id` のパスパラメータが当該スコープに所属することを検証。他スコープの TODO への操作を防止
- **一括操作のサイズ制限**: `bulk-status` は最大50件。DoS 防止
- **レート制限**: TODO 作成 API は1ユーザーあたり1分間に30回。コメント作成は1分間に10回。一括ステータス変更は1分間に5回
- **監査ログ**: `PROJECT_CREATED` / `PROJECT_COMPLETED` / `TODO_STATUS_CHANGED` / `TODO_ASSIGNED` 等を記録

## 付録: 組織委員会機能（F04.10）との連携

F04.10 で新設される組織委員会（Committee）は、プロジェクト・TODO のスコープとして `scope_type = 'COMMITTEE'` を受け入れる。

### スコープ仕様

- `projects.scope_type = 'COMMITTEE'` / `scope_id = committee_id`
- 委員会スコープのプロジェクトは **委員会メンバーのみ可視**（現役: `committee_members.left_at IS NULL`）
- 非メンバーからは存在自体が見えない（404 を返す）

### 権限マトリックス（委員会スコープ）

| 操作 | CHAIR | VICE_CHAIR | SECRETARY | MEMBER |
|---|---|---|---|---|
| プロジェクト作成 | ✅ | ✅ | ✅ | ✅ |
| プロジェクト設定変更 | ✅ | ✅ | × | 作成者のみ |
| TODO 作成・編集 | ✅ | ✅ | ✅ | ✅ |
| TODO 削除 | ✅ | ✅ | 作成者のみ | 作成者のみ |

### 実装上の注意

- `ScopeResolver.resolveCommittee(scopeId, currentUser)` を既存認可フローに追加
- `GET /projects` 等の横断一覧 API は、明示的に `scope_type=COMMITTEE` を指定しない限り委員会スコープのプロジェクトを返さない
- 委員会 ARCHIVED 状態では新規プロジェクト作成不可（既存 TODO 編集は可）

- V9.083: `projects.scope_type` コメント更新（`'COMMITTEE'` 値を含む旨を追記）
