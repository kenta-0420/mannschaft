# F14.1: 代理入力・非デジタル住民対応

> **ステータス**: 🟢 設計確定
> **実装フェーズ**: Phase 12（基盤）／ Phase 13（拡張）
> **最終更新**: 2026-04-29

---

## 1. 概要

スマートフォン・PC を所持しない／使えない住民（特にマンション管理組合の高齢居住者を想定）が、Mannschaft の各種入力系機能（出欠回答・アンケート・申請・既読確認等）から取り残されないようにする。理事・管理員・委員が紙申込書や対面聞き取りで集めた回答を、本人アカウントに紐付けて代理入力できる仕組みを提供する。

代理入力は **本人同意・職務分掌・監査ログ** を必須とし、既存の集計指標（既読率・回答率・出席率）が代理入力で偽装されないよう、`is_proxy_input` フラグで集計分離する。

### 既存機能との責務分離（重要）

| 既存機能 | 責務 | 本機能との関係 |
|---|---|---|
| `proxy_delegations`（F08.3） | 議決権の **委任**（特定の投票セッションで A が B に票を委ねる） | 概念別物。本書では `proxy_input_consents` と命名分離 |
| F03.12 ケア対象者見守り（`user_care_links`） | 高齢者・要介護者の見守り者紐付け | 連携。care_link がある場合は同意書承認の事前条件として活用可 |
| F10.3 監査ログ（`audit_logs`） | 全システム横断の人間可読操作ログ | `proxy_input_records` は集計分離用の最小列のみ保持し、人間可読ログは `audit_logs` に一本化（重複保持しない） |
| F11.1 オフライン PWA | オンライン住民が一時的にオフラインになる場合のキューイング | 別概念。本機能は「永続的に非デジタル」住民が対象 |
| F12.3 GDPR | データエクスポート・削除 | 連携。代理入力履歴も本人エクスポートに含める |

## 2. スコープ

### 対象ロール

既存 6 ロール（V2.014 で固定: SYSTEM_ADMIN/ADMIN/DEPUTY_ADMIN/MEMBER/SUPPORTER/GUEST）を拡張せず、`permissions` テーブルに新権限 **`PROXY_INPUT_EXECUTE`** **`PROXY_CONSENT_APPROVE`** を追加し、ロールに付与する方式とする。

| ロール | 付与権限 | 操作可能な範囲 |
|--------|---------|--------------|
| SYSTEM_ADMIN | 両方 | 全組合の代理入力ログ閲覧・同意書承認・代理入力 |
| ADMIN（理事長相当） | `PROXY_CONSENT_APPROVE` | 自組合内の同意書承認（**自身が代理者となる同意書は別 ADMIN/SYSTEM_ADMIN による承認必須＝職務分掌**） |
| DEPUTY_ADMIN（管理員・副理事相当） | `PROXY_INPUT_EXECUTE` | 承認済み同意書に基づく代理入力実行 |
| MEMBER（オフライン住民含む） | なし | 代理入力対象として登録される側 |
| SUPPORTER（家族・隣人など外部支援者） | `PROXY_INPUT_EXECUTE`（条件付） | 同意書で明示された対象機能のみ。**同居確認 or care_link 経由 or 第三者立会**いずれかが必須 |
| GUEST | なし | 不可 |

### 対象レベル

- [x] 組織 (Organization) — 管理組合単位での同意書管理
- [x] チーム (Team) — 委員会・班単位での代理者任命
- [ ] 個人 (Personal) — 個人スコープでは不要

### 対象機能（代理入力可）

| 機能 | feature_scope | 集計分離 | 備考 |
|---|---|---|---|
| アンケート（F05.4） | `SURVEY` | ◎ 必須 | `survey_responses.is_proxy_input` で分離 |
| 出欠回答（F03.1 / F03.8） | `SCHEDULE_ATTENDANCE` | ◎ 必須 | 定足数判定は本人入力分のみ |
| シフト希望（F03.5） | `SHIFT_REQUEST` | ◎ 必須 | 本人意思要件強化のため `consent_method=PAPER_SIGNED` 必須 |
| お知らせ既読（F02.6 / F04.9） | `ANNOUNCEMENT_READ` | ◎ 必須 | **「代理確認済」別フラグ**（既読率には含めない） |
| 駐車場申請（F09.3） | `PARKING_APPLICATION` | ◎ 必須 | 申請責任は本人＋代理者連帯記録 |
| 回覧（F05.2） | `CIRCULAR` | ◎ 必須 | `circulation_acknowledgements` に列追加 |

### 対象外（代理入力禁止）

- **チャット・メンション（F04.2）** — 発言主体性が揺らぐため不可
- **無記名投票（F08.3 で `anonymous=true`）** — 投票内容が代理者に漏れるため絶対禁止（Service 層で 403 拒否）
- **記名投票** — 既存 `proxy_delegations`（議決委任）を使う。本機能対象外
- **本人認証を伴う重要操作**（パスワード変更、退会申請、決済、本人プロフィール編集）

## 3. DB設計

### テーブル一覧

| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `proxy_input_consents` | 代理入力の本人同意書（投票委任 `proxy_delegations` と語を分離） | あり |

|---|---|---|

| `users.status` enum | 値 `DECEASED` `RELOCATED` を追加 | ライフイベント時の同意書自動失効トリガ |
| `permissions` | レコード追加: `PROXY_INPUT_EXECUTE`, `PROXY_CONSENT_APPROVE` | 権限定義 |
| `survey_responses` | `is_proxy_input TINYINT(1) NOT NULL DEFAULT 0`, `proxy_input_record_id BIGINT UNSIGNED NULL` | 集計分離 |
| `schedule_attendance_responses` | 同上 | 集計分離 |
| `shift_requests` | 同上 | 集計分離 |

| `parking_applications` | `is_proxy_input`, `proxy_input_record_id` 同上 | 申請責任の所在 |

### テーブル定義

#### `proxy_input_consents`

代理入力に対する本人の同意書。紙の同意書をスキャンして S3 に保管（直接URLでなく key のみ保持）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | — *(AUTO_INCREMENT)* | PK |
| `subject_user_id` | BIGINT UNSIGNED | NO | | 代理される本人 FK→users.id |
| `proxy_user_id` | BIGINT UNSIGNED | NO | | 代理者 FK→users.id |
| `organization_id` | BIGINT UNSIGNED | NO | | 同意の有効範囲（組合） |
| `consent_method` | VARCHAR(32) | NO | | `PAPER_SIGNED` / `WITNESSED_ORAL` / `DIGITAL_SIGNATURE` / `GUARDIAN_BY_COURT` |
| `scanned_document_s3_key` | VARCHAR(512) | YES | NULL | 同意書スキャン PDF の S3 オブジェクトキー（presigned URL は閲覧時に都度発行・5分TTL） |
| `guardian_certificate_s3_key` | VARCHAR(512) | YES | NULL | `GUARDIAN_BY_COURT` 時の後見登記事項証明書 |
| `witness_user_id` | BIGINT UNSIGNED | YES | NULL | `WITNESSED_ORAL` 時の立会人（ADMIN 必須） |
| `effective_from` | DATE | NO | | 有効開始日 |
| `effective_until` | DATE | NO | | 有効期限（最長1年・更新要） |
| `revoked_at` | DATETIME | YES | NULL | 撤回日時 |
| `revoke_method` | VARCHAR(32) | YES | NULL | `API_BY_SUBJECT` / `PAPER_BY_SUBJECT` / `AUTO_BY_LIFE_EVENT` / `AUTO_BY_TENURE_END` |
| `revoke_witnessed_by_user_id` | BIGINT UNSIGNED | YES | NULL | 紙撤回届の立会 ADMIN |
| `revoke_reason` | VARCHAR(255) | YES | NULL | 撤回理由 |
| `approved_by_user_id` | BIGINT UNSIGNED | YES | NULL | 承認した管理者（**subject 側組合の別 ADMIN または SYSTEM_ADMIN**。proxy_user_id 自身による自己承認は禁止） |
| `approved_at` | DATETIME | YES | NULL | 承認日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
INDEX idx_subject (subject_user_id, effective_until)
INDEX idx_proxy   (proxy_user_id, effective_until)
INDEX idx_org     (organization_id, effective_until)
UNIQUE KEY uq_active_consent (subject_user_id, proxy_user_id, organization_id, effective_from, deleted_at)
```

**制約・備考**
- CHECK 制約 (Service 層で担保): `approved_by_user_id != proxy_user_id`（自己承認禁止）
- CHECK: `consent_method=GUARDIAN_BY_COURT` の場合 `guardian_certificate_s3_key NOT NULL`
- CHECK: `consent_method=WITNESSED_ORAL` の場合 `witness_user_id NOT NULL`
- 論理削除: `SoftDeletableEntity` 適用

#### `proxy_input_records`

代理入力 1 件ごとに必ず 1 レコード。集計分離専用の最小列構成。人間可読ログは `audit_logs` に同期記録。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | — *(AUTO_INCREMENT)* | PK |
| `proxy_input_consent_id` | BIGINT UNSIGNED | NO | | 根拠となる同意書 |
| `subject_user_id` | BIGINT UNSIGNED | NO | | 本人（冗長保持／集計用） |
| `proxy_user_id` | BIGINT UNSIGNED | NO | | 代理者 |
| `feature_scope` | VARCHAR(64) | NO | | 操作対象機能 |
| `target_entity_type` | VARCHAR(64) | NO | | `SURVEY_RESPONSE` 等 |
| `target_entity_id` | BIGINT UNSIGNED | NO | | 操作対象レコード ID |
| `input_source` | VARCHAR(32) | NO | | `PAPER_FORM` / `PHONE_INTERVIEW` / `IN_PERSON` |
| `original_storage_location` | VARCHAR(255) | NO | | 紙原本の保管場所（必須・紛争時証跡） |

**インデックス**
```sql
INDEX idx_subject_feature (subject_user_id, feature_scope, created_at)
INDEX idx_proxy_created   (proxy_user_id, created_at)
INDEX idx_consent         (proxy_input_consent_id)
UNIQUE KEY uq_idempotent  (proxy_input_consent_id, target_entity_type, target_entity_id)  -- 冪等性担保（紙運用での二重登録防止）
```

**制約・備考**
- 物理削除は **「議決関連 5 年 + 関連議決後 5 年」のいずれか長い方**を経過した後、`ProxyInputRecordRetentionJob` でまとめて削除（区分所有法・標準管理規約と整合）
- Service 層ガード: `effective_from ≤ created_at ≤ effective_until` かつ `revoked_at IS NULL` かつ `feature_scope ∈ proxy_input_consent_scopes`

## 4. API設計

### エンドポイント一覧

| メソッド | パス | 必要権限 | 説明 |
|---------|-----|----------|------|
| POST | `/api/v1/organizations/{orgId}/proxy-input-consents` | DEPUTY_ADMIN+ | 同意書登録（S3 アップロード→key 提出） |
| GET | `/api/v1/organizations/{orgId}/proxy-input-consents` | ADMIN+ | 同意書一覧 |
| GET | `/api/v1/proxy-input-consents/active` | DEPUTY_ADMIN+ | 自身が代理者として保有する有効同意書一覧（`ProxyInputDeskView` 起動時） |
| PATCH | `/api/v1/proxy-input-consents/{id}/approve` | `PROXY_CONSENT_APPROVE` 権限保有者で proxy_user_id ≠ 自分 | ADMIN による承認（自己承認禁止） |
| PATCH | `/api/v1/proxy-input-consents/{id}/revoke` | 本人 or ADMIN（紙撤回届代行時） | 撤回 |
| GET | `/api/v1/proxy-input-records` | ADMIN+ or 本人 | 監査用ログ閲覧 |
| POST | `/api/v1/proxy-input-consents/{id}/scan-upload-url` | DEPUTY_ADMIN+ | S3 presigned PUT URL 発行（5分TTL） |
| GET | `/api/v1/proxy-input-consents/{id}/scan-download-url` | ADMIN+ or 本人 | S3 presigned GET URL 発行（5分TTL） |

### 既存 API への代理入力ヘッダ

代理入力可能な既存 API には、リクエストヘッダで代理入力モードを指定：

```
X-Proxy-For-User-Id: 12345
X-Proxy-Consent-Id: 678
X-Proxy-Input-Source: PAPER_FORM
X-Proxy-Original-Storage: "理事会金庫 No.3 — 2026年4月"
```

`ProxyInputContextFilter` がヘッダを **DB 再検証** したうえで、リクエストスコープに代理入力情報を保持する。各 Service 層は `ProxyInputContext.isProxy()` を参照し `proxy_input_records` 作成 → 業務テーブルに `is_proxy_input=true` で書き込み → `audit_logs` 同期記録（同一トランザクション）。

#### `POST /api/v1/organizations/{orgId}/proxy-input-consents`

**リクエストボディ**
```json
{
  "subjectUserId": 12345,
  "proxyUserId": 67890,
  "consentMethod": "PAPER_SIGNED",
  "scannedDocumentS3Key": "consents/2026/04/29/abc-def.pdf",
  "guardianCertificateS3Key": null,
  "witnessUserId": null,
  "effectiveFrom": "2026-04-30",
  "effectiveUntil": "2027-04-29",
  "scopes": ["SURVEY", "SCHEDULE_ATTENDANCE", "ANNOUNCEMENT_READ"]
}
```

**レスポンス（201 Created）** — `status: PENDING_APPROVAL`、別 ADMIN による承認待ち。

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `effectiveUntil - effectiveFrom > 365 日` ／ scopes 空 ／ method ごとの必須 key 欠落 |
| 403 | リクエスト者に権限なし／自己承認試行 |
| 404 | subject/proxy ユーザーが組織に所属していない |
| 409 | 同一組み合わせの有効同意書が既に存在 |

## 5. ビジネスロジック

### 主要フロー — 紙アンケート回答の代理入力

```
1. 管理員が住民から紙アンケート用紙と同意書を受領
2. 管理員がフロント「同意書登録」画面で:
   a. POST /scan-upload-url で presigned PUT URL を取得
   b. ブラウザから直接 S3 にアップロード（バックエンド経由しない）
   c. POST /proxy-input-consents で S3 key を提出
3. 別 ADMIN（subject 側組合の理事長等）が PATCH /approve（自己承認は 403）
4. 管理員が「代理入力デスク」(`ProxyInputDeskView`) を起動
   - 対象住民・同意書 ID をピン留め
   - 紙束を順次処理（住民切替時のみヘッダ更新）
5. POST /surveys/{id}/responses にヘッダ付きで送信
6. ProxyInputContextFilter がヘッダを DB 再検証
7. SurveyResponseService が単一トランザクションで:
   a. proxy_input_records 作成（uq_idempotent で二重登録防止）
   b. survey_responses を is_proxy_input=true で保存
   c. audit_logs に PROXY_INPUT_EXECUTED を記録（proxy_input_records.audit_log_id にセット）
8. 集計サービスは Specification パターン or DB ビュー経由でのみ集計（直接 SELECT は ArchUnit テストで禁止）
```

### バッチジョブ（ShedLock 利用）

| ジョブ名 | 周期 | 処理 |
|---|---|---|
| `ProxyConsentExpiryJob` | 日次 02:00 | `effective_until < TODAY` の同意書を `revoke_method=AUTO_BY_TENURE_END` で失効 |
| `ProxyConsentLifeEventJob` | `users.status` 変更時イベント駆動 | `FROZEN/ARCHIVED/DECEASED/RELOCATED` で当該ユーザの全同意書を `AUTO_BY_LIFE_EVENT` で失効 |
| `ProxyMonthlySummaryBatchJob` | 月次 1日 03:00 | 各組合の代理入力対象住民ごとに月次サマリ PDF を生成（F12.1 PDF 基盤利用）→ 管理員が印刷・配布 |
| `ProxyInputRecordRetentionJob` | 月次 04:00 | 「議決関連 5 年 + 関連議決後 5 年」経過済みレコードを物理削除 |

### 重要な判定ロジック

- **代理入力可否**: `X-Proxy-For-User-Id` ヘッダがある場合のみ代理モード。なしは通常入力
- **同意書の有効性**: `effective_from ≤ NOW() ≤ effective_until` AND `revoked_at IS NULL` AND `approved_at IS NOT NULL`
- **無記名投票での代理拒否**: 投票セッションが `anonymous=true` の場合、代理ヘッダ付きリクエストは 403
- **既読の代理確認**: `announcement_reads.is_proxy_confirmed=true` は **既読率の分子に含めない**。「周知済み件数」として別 KPI で集計
- **SUPPORTER による代理入力**: 以下のいずれかが必要
  - `user_care_links` で subject との見守り関係が登録済み
  - `consent_method=WITNESSED_ORAL` で `witness_user_id` が ADMIN

### 集計 KPI の扱い

| KPI | 代理入力分の扱い | 実装方式 |
|---|---|---|
| アンケート回答率 | 本人入力のみで集計＋「代理含む合計」併記 | `v_survey_responses_by_person` ビュー＋集計サービス（Specification 強制） |
| 出席率（定足数） | **本人入力のみで定足数判定** | `AttendanceQuorumCalculator` 改修・代理は補助情報 |
| 既読率 | 本人既読のみ。代理確認済は「周知済み件数」として別 KPI | `v_announcement_reads_by_person` ビュー |
| シフト充足率 | 本人・代理を合算（実運用上、希望は希望） | 既存ロジック流用＋ source 列で内訳表示 |
| 駐車場申請件数 | 合算（代理者は連帯記録） | 既存ロジック＋ note に proxy_user_id |

### 集計画面 UX（フロント実装ガイド）

「回答率」表示は **本人 60% / 代理 +15% / 合計 75%** の三段併記をデフォルトとする。理事会向けダッシュボード（F02系）にこのモックを取り込む。

### Specification パターンによる集計汚染防止

`SurveyResponseRepository` 等への直接 `findAll` `count` 呼び出しは ArchUnit テスト（`@ArchTest`）で禁止し、必ず以下経由とする：

```java
public class SurveyResponseSpecs {
    public static Specification<SurveyResponseEntity> byPersonOnly() {
        return (root, query, cb) -> cb.isFalse(root.get("isProxyInput"));
    }
    public static Specification<SurveyResponseEntity> includingProxy() { ... }
}
```

DB ビュー `v_survey_responses_by_person` も併設し、BI 接続時の事故を防ぐ。

## 6. セキュリティ考慮事項

- **認可チェック**: `ProxyInputContextFilter` が全代理入力リクエストで以下を検証
  - リクエスト者 == `proxy_input_consents.proxy_user_id`
  - 同意書の有効期限・スコープ・承認状態
  - 対象機能が代理入力許可リスト内
- **職務分掌**:
  - `proxy_input_consents.approved_by_user_id != proxy_user_id` を Service 層で必須検証
  - 同一 ADMIN による自己承認は 403
- **SUPPORTER 悪用対策**:
  - 同居確認 (`user_care_links`) または ADMIN 立会 (`witness_user_id`) のいずれか必須
  - 月次サマリで本人（または家族の別人）に通知
- **scanned_document の保護**:
  - S3 オブジェクトは直接公開せず、サーバ発行の **presigned URL（5分 TTL）** でのみ閲覧
  - SSE-KMS 暗号化必須
  - バックエンドは `scanned_document_s3_key` のみ保持（永続 URL は持たない）
- **監査ログ**: `audit_logs.AuditEventType` に以下を追加
  - `PROXY_CONSENT_CREATED` / `PROXY_CONSENT_APPROVED` / `PROXY_CONSENT_REVOKED`
  - `PROXY_INPUT_EXECUTED`（metadata: feature_scope, target_entity_type, target_entity_id, original_storage_location）
- **本人通知**:
  - オンライン住民: メール／プッシュで個別通知（F04.3 通知基盤）
  - オフライン住民: 月次紙サマリ（`ProxyMonthlySummaryBatchJob`）で代替
- **個人情報**: 既存 `users` の AES-256-GCM 暗号化方針を踏襲。`proxy_input_records` の `original_storage_location` は暗号化対象外（PII でないため）
- **GDPR エクスポート連携（F12.3）**:
  - 本人エクスポートに `proxy_input_consents`（自身が subject のもの）と `proxy_input_records`（同）を含める
  - エクスポート JSON では `proxy_user_id` を仮名化（`PROXY_USER_001` 等のローカル ID）
- **GDPR 撤回**:
  - オンライン: PATCH `/revoke` を本人が実行 (`revoke_method=API_BY_SUBJECT`)
  - オフライン: 紙の撤回届を ADMIN が代行入力 (`revoke_method=PAPER_BY_SUBJECT`、`revoke_witnessed_by_user_id` 必須)
- **レートリミット**: `Bucket4j` で
  - 代理者 1 名: 30 回/分
  - 組合単位: 100 回/分（複数管理員の同時並行を制御）
- **F11.1 PWA との競合**: PWA オフラインキューに `X-Proxy-*` ヘッダを保持し再送可能とする。ただし送信時に同意書の有効性を再検証してから enqueue する

## 10. テスト方針

- **ユニット**: `ProxyInputContextFilter` の同意書検証ロジック・自己承認禁止・無記名投票拒否
- **統合**: 代理入力 → `proxy_input_records` + `audit_logs` + 業務テーブルが単一トランザクションで整合
- **ArchUnit**: 業務 Repository への直接 `findAll`/`count` を禁止するルール（Specification 経由を強制）
- **E2E**: `ProxyInputDeskView` で住民切替→紙束処理→監査画面で確認、撤回→以降の代理入力が 403
- **負荷**: 組合単位 100 回/分のレートリミット動作確認
