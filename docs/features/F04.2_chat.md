# F04.2: チャット

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 4
> **最終更新**: 2026-03-18
> **関連ドキュメント**: F04.1 タイムライン、F04.3 プッシュ通知、F01.3 テンプレート・モジュール（デフォルト機能）

---

## 1. 概要

チーム・組織内のリアルタイムコミュニケーションを実現するチャット機能。Slack/Teams スタイルの階層化スレッド（特定メッセージへの返信）と絵文字リアクションを備え、情報の埋没を防止する。WebSocket (STOMP) + Valkey メッセージブローカーでリアルタイム配信し、チーム内チャンネル・組織横断チャンネル・DM（1対1）をサポートする。メッセージの全文検索（MySQL FULLTEXT）、ファイル添付（Pre-signed URL）、メンション（@ユーザー名）に対応する。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全チャンネルの参照・管理（モデレーション）|
| ADMIN | チーム/組織内チャンネルの作成・管理・メッセージ削除。DM の利用 |
| DEPUTY_ADMIN | `MANAGE_CHAT` 権限を持つ場合: チャンネル管理・メッセージ削除 |
| MEMBER | チャンネルへの参加・メッセージ送受信・リアクション。DM の利用 |
| SUPPORTER | 参加済みチャンネルの閲覧のみ（投稿不可）|
| GUEST | 対象外 |

### 対象レベル
- [x] 組織 (Organization) — 組織横断チャンネル
- [x] チーム (Team) — チーム内チャンネル
- [x] 個人 (Personal) — DM（1対1）

### テーブル定義

#### `chat_channels`

チャンネルまたは DM のマスター。1チャンネル = 1レコード。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `channel_type` | VARCHAR(20) | NO | — | チャンネル種別（TEAM / ORGANIZATION / CROSS_TEAM / DIRECT） |
| `team_id` | BIGINT UNSIGNED | YES | NULL | FK → teams。TEAM チャンネルの場合に設定 |
| `organization_id` | BIGINT UNSIGNED | YES | NULL | FK → organizations。ORGANIZATION チャンネルの場合に設定 |
| `name` | VARCHAR(100) | YES | NULL | チャンネル名。DM の場合は NULL |
| `icon_key` | VARCHAR(500) | YES | NULL | チャンネルアイコンの **Cloudflare R2 オブジェクトキー**。Presigned URL フローでアップロード（カラム名は歴史的経緯で `_key` を維持） |
| `description` | VARCHAR(500) | YES | NULL | チャンネルの説明 |
| `is_private` | BOOLEAN | NO | FALSE | TRUE = 招待制チャンネル。FALSE = スコープ内の全メンバーが参加可能 |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `last_message_at` | DATETIME | YES | NULL | 最新メッセージ日時（denormalize。チャンネル一覧のソート用） |
| `last_message_preview` | VARCHAR(100) | YES | NULL | 最新メッセージ冒頭100字（denormalize。チャンネル一覧のプレビュー用） |
| `source_type` | VARCHAR(30) | YES | NULL | 自動作成元の機能種別（`'MATCHING'`=マッチング成立 / `'RESERVATION'`=予約成立 等）。手動作成チャンネルは NULL |
| `source_id` | BIGINT UNSIGNED | YES | NULL | 自動作成元のレコードID（`source_type` とペアで使用。例: `source_type='MATCHING'` の場合は `match_proposals.id`）|
| `is_archived` | BOOLEAN | NO | FALSE | アーカイブ済みフラグ。TRUE = 新規投稿不可・閲覧のみ |
| `version` | BIGINT | NO | 0 | 楽観的ロックバージョン（@Version） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_chat_channels_team (team_id, is_archived, last_message_at DESC)          -- チーム別チャンネル一覧（WHERE deleted_at IS NULL を Repository デフォルト条件に含む）
INDEX idx_chat_channels_org (organization_id, is_archived, last_message_at DESC)   -- 組織別チャンネル一覧
INDEX idx_chat_channels_type (channel_type, last_message_at DESC)                  -- 種別別の一覧
UNIQUE KEY uq_chat_channels_source (source_type, source_id)                       -- 自動作成元ごとの重複作成防止（両方 NULL の場合は制約対象外）
```

**制約・備考**
- `channel_type` のスコープ制約:
  - `TEAM`: `team_id` NOT NULL, `organization_id` NULL
  - `ORGANIZATION`: `team_id` NULL, `organization_id` NOT NULL
  - `CROSS_TEAM`: `team_id` NULL, `organization_id` NULL（ADMIN が作成し、メンバーを明示的に招待。同一プラットフォーム内の任意のユーザーを招待可能。招待は OWNER / チャンネル ADMIN のみ）
  - `DIRECT`: `team_id` NULL, `organization_id` NULL, `name` NULL（2人固定、メンバーテーブルで管理）
- `source_type` / `source_id`: 他機能からの自動作成チャンネルを識別するためのポリモーフィック参照。`source_type = 'MATCHING'` + `source_id = proposalId`（F08.1 マッチング成立時の自動作成）、`source_type = 'RESERVATION'` + `source_id = reservationId`（F03.4 予約成立時）等。UNIQUE KEY により同一ソースからの重複作成を防止。MySQL では `source_type` / `source_id` が両方 NULL の場合は UNIQUE 制約の対象外（NULL != NULL）
- **`GET /chat/channels/{id}` の `source_data` 拡張**: `source_type` が非 NULL のチャンネルでは、レスポンスに `source_data` オブジェクトを含めて返却する。チャット画面上部に表示するコンテキスト情報カードのデータソース:
  - `source_type = 'RESERVATION'`: `source_data` に予約サマリー（`reservation_id`, `slot_date`, `start_time`, `end_time`, `title`, `staff_name`, `status`）を含める。`reservations` テーブルから `source_id` で JOIN して取得
  - `source_type = 'MATCHING'`: `source_data` にマッチング提案サマリーを含める（F08.1 側で定義）
  - フロントエンドは `source_data` が存在する場合、チャット画面上部に予約カード / マッチングカードを固定表示
- **SUPPORTER の予約チャット投稿許可**: `source_type = 'RESERVATION'` の DIRECT チャンネルに限り、SUPPORTER にもメッセージ送信を許可する。認可チェックで `source_type` を参照してホワイトリスト判定（F03.4 予約チャット連携の要件）
- DM は同一2人の組み合わせで1チャンネルのみ（Service 層で重複チェック）
- `last_message_at` / `last_message_preview`: メッセージ送信時にアトミック更新。チャンネル一覧 API で `chat_messages` への JOIN を排除。メッセージ削除時は直前の未削除メッセージから再計算する（`SELECT body FROM chat_messages WHERE channel_id = ? AND deleted_at IS NULL ORDER BY created_at DESC LIMIT 1`）。システムメッセージ（`is_system = TRUE`）はプレビューに反映しない（「田中太郎さんが参加しました」等は表示価値が低いため）
- `member_count` はレスポンス用に `chat_channel_members` の `COUNT` で算出（チャンネル数が限定的なため denormalize 不要。パフォーマンス課題が出た場合にカラム追加を検討）
- `is_archived`: ADMIN がチャンネルをアーカイブ可能。アーカイブ後も過去メッセージは閲覧可能

#### `chat_messages`

メッセージ本体。スレッド返信は `parent_id` で階層化。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `channel_id` | BIGINT UNSIGNED | NO | — | FK → chat_channels。ON DELETE CASCADE |
| `sender_id` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `parent_id` | BIGINT UNSIGNED | YES | NULL | FK → chat_messages（自己参照）。NULL = トップレベルメッセージ、非 NULL = スレッド返信 |
| `body` | TEXT | NO | — | メッセージ本文（最大10,000文字） |
| `forwarded_from_id` | BIGINT UNSIGNED | YES | NULL | 転送元メッセージ ID（NULL = 通常メッセージ）。元メッセージへの参照リンクを表示 |
| `is_edited` | BOOLEAN | NO | FALSE | 編集済みフラグ |
| `is_system` | BOOLEAN | NO | FALSE | システムメッセージフラグ（メンバー参加・退出等） |
| `scheduled_at` | DATETIME | YES | NULL | 予約送信日時（NULL = 即時送信）。未来日時を指定すると送信待ちキューに入る |
| `reply_count` | INT UNSIGNED | NO | 0 | スレッド返信数（denormalize） |
| `reaction_count` | INT UNSIGNED | NO | 0 | リアクション総数（denormalize） |
| `is_pinned` | BOOLEAN | NO | FALSE | ピン留めフラグ。チャンネル OWNER / ADMIN / チーム ADMIN がピン留め可能 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時。DB 側は `body` を保持したまま、API レスポンスでのみ `body = null`, `sender = null` として返却し「このメッセージは削除されました」と表示 |

**インデックス**
```sql
INDEX idx_chat_messages_channel (channel_id, created_at DESC)     -- チャンネル内のメッセージ一覧（カーソルページネーション）
INDEX idx_chat_messages_parent (parent_id, created_at ASC)        -- スレッド内の返信一覧
INDEX idx_chat_messages_sender (sender_id, created_at DESC)       -- ユーザー別の投稿検索
INDEX idx_chat_messages_pinned (channel_id, is_pinned, created_at DESC)  -- ピン留めメッセージ一覧
FULLTEXT INDEX ft_chat_messages_body (body)                       -- メッセージ全文検索
```

**制約・備考**
- `parent_id` は1階層のみ（スレッド返信への返信は不可。Slack と同じ仕様）。parent_id が非 NULL のメッセージに対する返信は、同じ parent_id を設定する
- `is_edited`: 本文を編集した場合に TRUE にセット。フロントエンドで「(編集済み)」ラベルを表示。**編集は送信後24時間以内のみ可能**（Service 層で `created_at + 24h > NOW()` を検証。超過時は 409 エラー）。過去の発言改ざんによるスレッド文脈の破壊を防止
- `is_system`: メンバー参加・退出・チャンネル名変更等のシステムメッセージ。`sender_id = NULL`
- `reply_count` / `reaction_count`: denormalize。スレッド返信追加/削除・リアクション追加/削除時にアトミック更新
- FULLTEXT: MySQL の `ngram` パーサーを使用（日本語対応）。`MATCH(body) AGAINST('+検索語' IN BOOLEAN MODE)`
- `is_pinned`: 1チャンネルあたり最大50件。超過した場合は 400 エラー（古いピンを解除してから追加）。チャンネル OWNER / ADMIN / チーム ADMIN のみピン留め可能
- `forwarded_from_id`: 転送メッセージの元 ID。FK 制約は付けない（元メッセージが削除されても転送メッセージは残す）。API レスポンスでは元メッセージの sender / body / channel 名をスナップショットとして返却
- `scheduled_at`: 予約送信。`status = 'SCHEDULED'` としてDB に INSERT し、バッチ（1分間隔）が `scheduled_at <= NOW()` のメッセージを通常送信フローで処理。送信前であれば編集・取り消し可能

#### `chat_channel_members`

チャンネルメンバーシップと未読管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `channel_id` | BIGINT UNSIGNED | NO | — | FK → chat_channels。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `role` | VARCHAR(20) | NO | 'MEMBER' | チャンネル内ロール（OWNER / ADMIN / MEMBER） |
| `unread_count` | INT UNSIGNED | NO | 0 | 未読メッセージ数（denormalize） |
| `last_read_at` | DATETIME | YES | NULL | 最終既読日時。この日時以降のメッセージが未読 |
| `is_muted` | BOOLEAN | NO | FALSE | ミュート設定。TRUE = このチャンネルの通知を抑制 |
| `is_pinned` | BOOLEAN | NO | FALSE | ピン留め。TRUE = チャンネル一覧で上位固定表示 |
| `category` | VARCHAR(50) | YES | NULL | 個人用カテゴリ名（例: 「プロジェクト」「雑談」）。サイドバーのセクション分けに使用 |
| `joined_at` | DATETIME | NO | CURRENT_TIMESTAMP | 参加日時 |

**インデックス**
```sql
UNIQUE KEY uq_chat_members_channel_user (channel_id, user_id)  -- 同一チャンネルへの重複参加防止
INDEX idx_chat_members_user (user_id, is_pinned DESC, last_read_at DESC)  -- ユーザーのチャンネル一覧
```

**制約・備考**
- `unread_count`: メッセージ送信時に送信者以外の全メンバーの `unread_count` を +1（バッチ UPDATE）。既読 API 呼び出し時に 0 リセット + `last_read_at` 更新
- `role`: `OWNER` = チャンネル作成者（1名のみ）。`ADMIN` = チャンネル管理権限（複数可）。`MEMBER` = 一般参加者
- `is_muted`: ミュート中もメッセージは受信するが、プッシュ通知は送信しない
- DM チャンネルの場合: 2レコード固定。退出不可（チャンネル削除で対応）

#### `chat_message_reactions`

メッセージへの絵文字リアクション。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `message_id` | BIGINT UNSIGNED | NO | — | FK → chat_messages。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `emoji` | VARCHAR(50) | NO | — | 絵文字（Unicode 絵文字 or カスタム絵文字コード） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_chat_reactions_user_emoji (message_id, user_id, emoji)  -- 同一ユーザーの同一絵文字重複防止
INDEX idx_chat_reactions_message (message_id)                          -- メッセージ別のリアクション一覧
```

**制約・備考**
- 1ユーザーが1メッセージに同一絵文字を複数回付与することは不可（UNIQUE 制約）
- 1ユーザーが1メッセージに異なる絵文字を複数付与することは可能（最大20種類、Service 層バリデーション）
- リアクション追加/削除時に `chat_messages.reaction_count` をアトミック更新

#### `chat_message_attachments`

メッセージの添付ファイル。Presigned URL フロー（§26）で **Cloudflare R2** にアップロード後、メッセージ送信時に紐付ける。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `message_id` | BIGINT UNSIGNED | NO | — | FK → chat_messages。ON DELETE CASCADE |
| `file_key` | VARCHAR(500) | NO | — | R2 オブジェクトキー（カラム名は歴史的経緯で維持） |
| `file_name` | VARCHAR(255) | NO | — | 元のファイル名 |
| `file_size` | BIGINT UNSIGNED | NO | — | ファイルサイズ（バイト） |
| `content_type` | VARCHAR(100) | NO | — | MIME タイプ |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_chat_attachments_message (message_id)  -- メッセージ別の添付一覧
```

**制約・備考**
- 1メッセージあたり最大10件の添付ファイル
- **ファイルサイズ上限**:
  - 静止画・ドキュメント: 1ファイル **10MB**
  - 動画: 1ファイル **50MB**（チャットでは長尺動画は F04.1 タイムライン・F05.5 ファイル共有・F06.2 ギャラリーへ誘導）
- **許可する MIME タイプ**: `image/jpeg`, `image/png`, `image/gif`, `image/webp`, `application/pdf`, `text/plain`, `application/zip`, および **動画**: `video/mp4`（チームの用途に応じて拡張可能）
- **動画添付の方針**: チャットでも短い動画添付を許可するが、長尺動画や視聴数が想定される動画はタイムライン（F04.1 VIDEO_FILE）やギャラリー（F06.2）に誘導する。チャット動画は Cloudflare Workers でのサムネイル生成を行わず、ブラウザの `<video>` タグがプレイヤー側でフレーム抽出する（軽量実装）
- **クォータ計上**: 全添付ファイル（画像・ドキュメント・動画）は `file_size` 分がチャンネルの scope_type（TEAM / ORGANIZATION）または送信者の個人クォータ（DIRECT / GROUP_DM）に計上される。詳細は横断設計 `docs/cross-cutting/storage_quota.md` を参照

#### `chat_message_bookmarks`

メッセージの個人用ブックマーク。ピン留め（チャンネル全体に公開）と異なり、個人が「後で読む」「参考にしたい」メッセージを保存する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `message_id` | BIGINT UNSIGNED | NO | — | FK → chat_messages。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `note` | VARCHAR(200) | YES | NULL | ブックマークメモ（例: 「要対応」「参考資料」） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_chat_bookmarks_user_message (user_id, message_id)  -- 同一メッセージへの重複ブックマーク防止
INDEX idx_chat_bookmarks_user (user_id, created_at DESC)          -- ブックマーク一覧（新しい順）
```

**制約・備考**
- 他ユーザーには非公開（本人のみ参照可能）
- ブックマーク数の上限なし（ユーザーごとに自由に管理）
- 元メッセージが論理削除されても、ブックマークは保持（ただし API レスポンスではメッセージ本文を「削除されたメッセージ」と表示）


#### `POST /api/v1/chat/channels/{id}/messages`

メッセージを送信する。WebSocket でリアルタイム配信も行う。

**リクエストボディ**
```json
{
  "body": "明日の朝礼は10時からに変更です。@佐藤花子 確認お願いします。",
  "parent_id": null,
  "attachment_keys": ["chat/TEAM/5/channel-10/abc123.pdf"]
}
```

- `body`: 必須・1〜10,000文字
- `parent_id`: 任意。スレッド返信の場合は親メッセージ ID。親メッセージは同一チャンネル内であること
- `also_to_channel`: 任意（デフォルト: false）。`parent_id` 指定時に `true` にすると、スレッド返信に加えてチャンネルのトップレベルにも同一メッセージを表示（Slack の「チャンネルにも投稿」機能）
- `scheduled_at`: 任意。未来日時を指定すると予約送信。指定日時に自動送信される。送信前は編集・取り消し可能
- `attachment_keys`: 任意。事前に `upload-url` API で取得した R2 オブジェクトキー。最大10件。R2 上に実在することを HeadObject で検証

**レスポンス（201 Created）**: メッセージ単体を返却

**副作用**:
1. `chat_channels.last_message_at` / `last_message_preview` をアトミック更新
2. 送信者以外の全チャンネルメンバーの `chat_channel_members.unread_count` を +1
3. WebSocket で全チャンネルメンバーにリアルタイム配信（STOMP destination: `/topic/chat.channel.{channelId}`）
4. メンション解析: `@ユーザー名` を検出し `mentions` テーブルに INSERT + メンション先ユーザーにプッシュ通知
5. ミュート中のメンバーにはプッシュ通知を送信しない（WebSocket 配信は行う）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | バリデーションエラー / parent_id が同一チャンネル内でない |
| 403 | チャンネルに参加していない / SUPPORTER は投稿不可 / アーカイブ済み |
| 404 | チャンネル/親メッセージが存在しない |

#### `POST /api/v1/chat/conversations`

会話を開始する。参加者数に応じてバックエンドが自動的にチャンネル種別を振り分ける。

- **2人**（自分 + 相手1人）→ **Kabine**（`ChannelType.DM`）: 既存 DM が存在する場合は既存チャンネルを返却
- **3人以上**（自分 + 相手2人以上）→ **Zimmer**（`ChannelType.GROUP_DM`）: 常に新規作成

**リクエストボディ**
```json
{
  "user_ids": [11, 12]
}
```

- `user_ids`: 必須。会話相手のユーザー ID リスト（自分自身は含めない）。1〜9名。

**レスポンス**

| 人数 | ステータス | 説明 |
|------|-----------|------|
| 2人（DM既存） | 200 OK | 既存 Kabine チャンネルを返却 |
| 2人（DM新規） | 201 Created | 新規 Kabine チャンネルを返却 |
| 3人以上 | 201 Created | 新規 Zimmer チャンネルを返却 |

レスポンスボディはチャンネル詳細（`ChannelResponse`）と同形式。

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `user_ids` が空 / 自分自身が含まれる / 10人以上 |
| 403 | 相手ユーザーにブロックされている / DM 受信制限に抵触 |
| 404 | `user_ids` に存在しないユーザー ID が含まれる |

**振り分けロジック（バックエンド実装）**
```
user_ids.size == 1 → DM (Kabine)
  └─ 既存 DM チャンネル検索 → 存在すれば 200 返却、なければ新規作成して 201 返却
  └─ ブロック・DM受信制限チェック（validateDmCreation）

user_ids.size >= 2 → GROUP_DM (Zimmer)
  └─ 常に新規作成して 201 返却
  └─ 全参加者分のブロックチェック（相互）
```

#### `POST /api/v1/chat/channels/{id}/convert-to-group`

DM チャンネルをグループチャンネルに拡張する。新しいメンバーを追加しつつ、既存の DM を残すか統合するかを選択できる。

**リクエストボディ**
```json
{
  "additional_user_ids": [13, 14],
  "name": "プロジェクトチーム",
  "keep_original": true,
  "share_history": false
}
```

- `additional_user_ids`: 必須。追加するメンバーの user_id（1名以上）
- `name`: 必須。新しいグループチャンネル名（1〜100文字）
- `keep_original`: 必須。
  - `true`: 元の DM は残したまま、新しいグループチャンネルを作成
  - `false`: 元の DM をアーカイブし、新しいグループチャンネルに統合
- `share_history`: 必須。
  - `true`: 元の DM のメッセージ履歴を新メンバーにも閲覧可能にする（新チャンネルの `chat_channel_members.joined_at` を DM の最初のメッセージ日時に設定し、`last_read_at` より前のメッセージも表示）
  - `false`: 新メンバーには新チャンネル作成後のメッセージのみ表示（`joined_at` 以降のメッセージのみ閲覧可能）

**レスポンス（201 Created）**: 新しいグループチャンネルの詳細を返却

**処理フロー**:
1. 新しい CROSS_TEAM チャンネルを作成（`channel_type = 'CROSS_TEAM'`, `is_private = TRUE`）
2. 元の DM メンバー + 追加メンバーを `chat_channel_members` に INSERT
3. `keep_original = false` の場合: 元の DM チャンネルを `is_archived = TRUE` に更新
4. `share_history = true` の場合: 元 DM のメッセージを新チャンネルにコピー INSERT（`forwarded_from_id` は設定せず、元メッセージとして扱う）。大量メッセージのコピーは非同期ジョブで実行し、完了後に通知
5. システムメッセージ: 「○○さんが DM をグループに拡張しました。△△さん、□□さんが参加しました」

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | DM チャンネル以外に対して実行 / additional_user_ids が空 |
| 403 | DM のメンバーではない |
| 404 | チャンネル/ユーザーが存在しない |

#### `POST /api/v1/chat/channels/{id}/invite-to-zimmer`

Kabine（DM）を維持したまま、新しい Zimmer（GROUP_DM）を作成してユーザーを招待する。

**ユースケース**: A と B が Kabine でチャットしている状態で C を招待したい場合。元の Kabine は残したまま、A・B・C 全員が参加する新 Zimmer を作成する。

**リクエストボディ**
```json
{
  "user_ids": [15],
  "share_history": true
}
```

- `user_ids`: 必須。新たに招待するユーザー ID リスト（1〜8名）
- `share_history`:
  - `true`: Kabine のトップレベルメッセージを `forwarded_from_id` 付きで新 Zimmer にコピーする（スレッド返信は除外）
  - `false`: Zimmer 作成時点からのメッセージのみ表示

**レスポンス（201 Created）**: 新しく作成された Zimmer のチャンネル詳細を返却

**処理フロー**:
1. `channel_id` のチャンネルが `ChannelType.DM`（Kabine）であることを確認
2. 呼び出しユーザーがその Kabine のメンバーであることを確認
3. `ChannelType.GROUP_DM`（Zimmer）の新チャンネルを作成
4. Kabine の既存メンバー全員を Zimmer に追加（呼び出しユーザーは OWNER、その他は MEMBER）
5. `user_ids` に指定された新メンバーを Zimmer に MEMBER として追加
6. `share_history = true` の場合: Kabine のトップレベルメッセージ（`parent_id IS NULL`）を古い順で Zimmer にコピー（`forwarded_from_id` に元メッセージの ID をセット）
7. 元の Kabine はアーカイブ・削除せずそのまま維持

**convert-to-group との違い**
| | `convert-to-group` | `invite-to-zimmer` |
|---|---|---|
| 元 Kabine | GROUP_DM に変換される（元チャンネルが消える） | そのまま残る |
| 用途 | 2人の会話を後から拡張したいとき | 既存の 1:1 を維持しつつグループを新設したいとき |

**エラーレスポンス**
| ステータス | エラーコード | 条件 |
|-----------|------------|------|
| 400 | CHAT_016 | `channel_type` が DM 以外 |
| 403 | CHAT_005 | 呼び出しユーザーが Kabine のメンバーではない |
| 404 | CHAT_001 | チャンネルが存在しない |

### WebSocket 仕様

#### 接続

```
WS /ws/chat
Authorization: Bearer <JWT>
```

- STOMP プロトコルで接続
- `HandshakeInterceptor` で JWT を検証
- 接続成功時に Valkey にセッション登録（`mannschaft:ws:session:{sessionId}`）
- 切断時に Valkey からセッション削除
- **SUBSCRIBE 認可**: `/topic/chat.channel.{channelId}` への SUBSCRIBE 時に `ChannelInterceptor` でチャンネルメンバーシップを検証。非メンバーの SUBSCRIBE はエラーフレームで拒否（有効な JWT を持っていても他人のチャンネルを盗聴できないようにする）

#### STOMP Destination

| Destination | 説明 |
|------------|------|
| `/topic/chat.channel.{channelId}` | チャンネルのメッセージ配信（SUBSCRIBE） |
| `/topic/chat.user.{userId}` | ユーザー個別通知（未読数更新、チャンネル更新等）|
| `/app/chat.send` | メッセージ送信（SEND。REST API と同一のバリデーション・副作用を実行。フォーマット: `{ "channel_id": 1, "body": "...", "parent_id": null }`。添付ファイルは REST API のみ対応） |
| `/app/chat.typing` | タイピングインジケーター送信 |

## 5. ビジネスロジック

### メッセージ送信フロー

```
1. REST API（POST /channels/{id}/messages）またはWebSocket（/app/chat.send）でメッセージ受信
2. バリデーション:
   a. チャンネル存在確認 + メンバーシップ確認
   b. チャンネルがアーカイブ済みでないこと
   c. SUPPORTER は投稿不可
   d. メッセージ本文のサニタイズ（XSS 防止）
3. chat_messages INSERT
4. 添付ファイルの紐付け: attachment_keys の R2 存在確認（HeadObject）→ chat_message_attachments INSERT → StorageQuotaService.recordUpload() で使用量加算
5. denormalize 更新（1トランザクション内）:
   a. chat_channels.last_message_at / last_message_preview を更新
   b. chat_channel_members.unread_count を送信者以外で +1（バッチ UPDATE）
   c. parent_id が指定されている場合: 親メッセージの reply_count を +1
6. メンション解析:
   a. body 内の @ユーザー名 を正規表現で抽出
   b. mentions テーブルに INSERT（target_type = 'CHAT_MESSAGE', target_id = message.id）
   c. メンション先ユーザーにプッシュ通知（ApplicationEvent: ChatMentionEvent）
   d. 特殊メンション（ADMIN / チャンネル OWNER のみ使用可能。一般 MEMBER が使用した場合は通常テキストとして扱う）:
      - @channel: チャンネル全メンバーにプッシュ通知（ミュート設定も無視）
      - @here: 現在 WebSocket 接続中（オンライン）のメンバーのみにプッシュ通知。
        オンライン判定: Valkey の `mannschaft:ws:session:*` を SCAN し、接続中の userId を収集
7. 通知フィルタリング:
   a. notification_preferences テーブルを参照し、該当チーム/組織の通知を OFF にしているユーザーにはプッシュ通知を送信しない（WebSocket 配信は行う。通知設定はプッシュ通知のみに影響）
   b. is_muted = TRUE のメンバーにはプッシュ通知を送信しない（@channel を除く）
   c. user_blocks: ブロック中のユーザーからのメンション通知は抑制
8. WebSocket 配信:
   a. /topic/chat.channel.{channelId} に MESSAGE_CREATED フレームを送信
   b. Valkey Pub/Sub で全アプリインスタンスに伝播（マルチインスタンス対応）
9. プッシュ通知:
   a. WebSocket 未接続のメンバーにプッシュ通知（ApplicationEvent: ChatMessageEvent）
   b. 通知フィルタリング（ステップ7）の結果に基づき送信先を決定
```

### 未読管理

```
1. メッセージ送信時:
   a. 送信者以外の全メンバーの unread_count を +1（バッチ UPDATE）
   b. バッチ UPDATE: UPDATE chat_channel_members SET unread_count = unread_count + 1
      WHERE channel_id = {id} AND user_id != {senderId}

2. 既読マーク（POST /channels/{id}/read）:
   a. unread_count を 0 にリセット
   b. last_read_at を現在日時に更新
   c. WebSocket で /topic/chat.user.{userId} にカウントリセットを通知（ヘッダーバッジ更新用）

3. フロントエンド:
   a. チャンネルを開いた時点で自動的に既読マーク API を呼び出す
   b. バックグラウンドのチャンネルは未読のまま保持
   c. ブラウザの Visibility API でフォーカスを検知し、フォーカス復帰時に既読処理
```

### パブリックチャンネルへの自動参加

```
1. TEAM チャンネル（is_private = FALSE）:
   a. チームに新メンバーが参加した際に自動的に全パブリックチャンネルへ追加
     （ApplicationEvent: TeamMemberJoinedEvent を購読）
   b. システムメッセージ:「田中太郎さんが参加しました」を自動投稿

2. ORGANIZATION チャンネル（is_private = FALSE）:
   a. 組織に新チームが参加した際に、そのチームの ADMIN を自動的に全パブリック組織チャンネルへ追加

3. プライベートチャンネル: 自動参加なし。OWNER / ADMIN が明示的に招待
```

### チャンネルアーカイブ

```
1. ADMIN がチャンネルをアーカイブ（PATCH /channels/{id}/archive）
2. is_archived = TRUE に更新
3. アーカイブ後:
   a. 新規メッセージの投稿不可（403 エラー）
   b. 既存メッセージの閲覧は可能
   c. リアクションの追加・削除は不可
   d. チャンネル一覧ではデフォルト非表示（is_archived=true でフィルタすると表示）
4. アーカイブ解除: 同じ API で is_archived = FALSE に戻す
```

### WebSocket リコネクション戦略

```
1. フロントエンド側の自動再接続（指数バックオフ）:
   a. 初回1秒 → 2秒 → 4秒 → 8秒 → 最大30秒の間隔で再接続を試行
   b. 5回連続失敗後: UI に「接続が切れています。再接続を試みています…」を表示
   c. ブラウザの Visibility API と連携:
      - タブがバックグラウンド: リコネクション間隔を60秒に延長（サーバーリソース節約）
      - フォアグラウンド復帰: 即座にリコネクト

2. 再接続成功時のメッセージキャッチアップ:
   a. 各チャンネルの最後に受信した messageId を IndexedDB に保持
   b. 再接続後: GET /channels/{id}/messages?cursor={lastMessageId}&direction=after で
      切断中のメッセージを REST API で取得
   c. WebSocket が受信するリアルタイムメッセージと REST 応答の重複を messageId で排除

3. JWT 期限切れ時の処理:
   a. WebSocket 接続で 401 を受信 → リフレッシュトークンで JWT を再取得
   b. リフレッシュ成功 → 新しい JWT で再接続
   c. リフレッシュ失敗 → ログイン画面に遷移

4. バックエンド側:
   a. JWT 期限切れ検知時に WebSocket セッションを DISCONNECT フレームで切断
   b. Valkey の WebSocket セッション（mannschaft:ws:session:{sessionId}）にも TTL を設定し、
      異常切断時にゴミセッションが残らないようにする（TTL: JWT の有効期限と同一）
```

### メッセージのピン留め

```
1. PATCH /chat/messages/{id}/pin でトグル（is_pinned = !is_pinned）
2. ピン留め時のバリデーション:
   a. チャンネル OWNER / ADMIN / チーム ADMIN であること
   b. 1チャンネルあたりのピン留め上限（50件）チェック:
      SELECT COUNT(*) FROM chat_messages WHERE channel_id = {channelId} AND is_pinned = TRUE
      50件以上の場合は 400 エラー（「古いピンを解除してから追加してください」）
3. WebSocket で MESSAGE_PINNED / MESSAGE_UNPINNED イベントを配信
4. フロントエンド:
   a. チャンネルヘッダーにピン留めアイコン + 件数バッジを表示
   b. クリックでピン留めメッセージ一覧をサイドパネル表示
   c. 各メッセージのコンテキストメニューに「ピン留め」オプション（権限保持者のみ）
```

### 予約送信バッチ

```
1. @Scheduled で1分間隔で実行
2. 条件: scheduled_at <= NOW() AND deleted_at IS NULL
3. 対象メッセージを通常送信フローで処理（denormalize更新・WebSocket配信・通知等の全副作用を実行）
4. 送信完了後: scheduled_at カラムは保持（送信済みの識別用。is_system = FALSE で通常メッセージとして表示）
5. フロントエンド:
   a. 予約送信メッセージは送信前に「予約: 3/15 09:00 に送信」ラベルを表示
   b. 送信前のメッセージは本人のみ閲覧可能。編集（PUT /messages/{id}）・取り消し（DELETE）可能
   c. 送信後は通常メッセージとして表示（予約だったことは表示しない）
```

### 未読位置マーカー（フロントエンド）

```
1. チャンネルを開いた際に last_read_at を基準に「ここから未読です」のラインを描画
2. 実装:
   a. GET /channels/{id}/messages で取得したメッセージ配列を走査
   b. created_at > last_read_at の最初のメッセージの直前に区切り線を表示
   c. 区切り線には「N件の新着メッセージ」のラベルを付与
3. 既読マーク（POST /read）送信後も、区切り線はページ遷移まで保持（再表示時は消える）
```

### WebSocket / REST の統一アーキテクチャ

```
1. メッセージ送信は REST API（POST /channels/{id}/messages）と
   WebSocket（/app/chat.send）の2経路を提供するが、ビジネスロジックは1箇所に集約:
   a. ChatMessageService.sendMessage() を共通の Service メソッドとして実装
   b. REST Controller はリクエストを DTO に変換して Service を呼び出す
   c. WebSocket の @MessageMapping ハンドラも同一の Service を呼び出す
   d. バリデーション・denormalize 更新・メンション解析・通知は全て Service 内で実行
2. 利点: ロジックの二重実装を防止し、一方だけバグが入るリスクを排除
3. WebSocket 送信は REST のフォールバック（WebSocket 接続が不安定な場合に REST にフォールバック）
```

### 大規模チャンネルの未読カウント最適化

```
1. 現行方式: メッセージ送信時に chat_channel_members の全メンバーを UPDATE（unread_count + 1）
   - 小〜中規模チャンネル（〜100名）: 問題なし
   - 大規模チャンネル（500名超）: 1メッセージあたり499行の UPDATE が発生しレイテンシに影響

2. 最適化方式（メンバー200名超のチャンネルに自動適用）:
   a. Valkey INCR で未読カウントを管理（key: mannschaft:chat:unread:{channelId}:{userId}）
   b. DB の unread_count は非同期バッチ（5分間隔）で Valkey の値と同期
   c. GET /channels 一覧取得時は Valkey を優先参照（Valkey に値がなければ DB にフォールバック）
   d. 既読マーク時: Valkey DEL + DB unread_count = 0 を同一処理で実行

3. 閾値（200名）は application.yml で設定可能:
   mannschaft.chat.unread-redis-threshold: 200
```

### 重要な判定ロジック

- **チャンネル作成権限**: TEAM チャンネルはチームの MEMBER 以上。ORGANIZATION / CROSS_TEAM チャンネルは ADMIN のみ
- **メッセージ削除権限**: 投稿者本人 or チーム/組織 ADMIN or チャンネル OWNER/ADMIN
- **DM の重複防止**: `POST /channels/dm` で既存 DM を検索。`chat_channel_members` で `channel_type = 'DIRECT'` かつ `user_id IN (自分, partner_id)` のチャンネルを検索。存在すれば既存を返却
- **スレッドの階層制限**: `parent_id` を指定する場合、その親メッセージの `parent_id` が NULL であること（2階層目まで）。parent_id が非 NULL のメッセージへの返信は、同一の parent_id を使用する
- **チャンネルメンバーシップの可視性**: パブリックチャンネルは非メンバーでも存在を確認可能（参加を促す UI 用）。プライベートチャンネルは非メンバーには完全非公開
- **SUPPORTER の制約**: チャンネルには参加可能だが投稿・リアクション不可（閲覧のみ）
- **unread_count とメッセージ削除**: メッセージ削除時に `unread_count` の減算は行わない。既読マーク時に 0 リセットされるため実害は小さく、「そのメッセージを未読だったメンバー」の特定が複雑なため費用対効果に見合わない
- **メッセージ編集期限**: 送信後24時間以内のみ。ADMIN によるモデレーション削除は時間制限なし
- **DM のブロック**: プラットフォーム共通の `user_blocks` テーブルで管理。ブロック中のユーザーからの DM 作成は 403 で拒否（詳細: db_design_details.md「ユーザーブロック」参照）
- **OWNER 移譲**: OWNER のみ実行可能。移譲後は元 OWNER が ADMIN に降格。OWNER を移譲してから退出する運用
- **ピン留め権限**: チャンネル OWNER / ADMIN / チーム ADMIN のみ。一般 MEMBER は不可
- **ピン留め上限**: 1チャンネルあたり50件。超過時は 400 エラー

## 6. セキュリティ考慮事項

- **認可チェック**: 全 API エンドポイントでチャンネルメンバーシップを検証。プライベートチャンネルは非メンバーのアクセスを完全遮断
- **WebSocket 認証・認可**: 接続ハンドシェイク時に JWT を検証。無効なトークンでの接続を拒否。トークン期限切れ時は WebSocket セッションを切断。SUBSCRIBE 時に `ChannelInterceptor` でチャンネルメンバーシップを検証し、非メンバーの盗聴を防止
- **メッセージのサニタイズ**: XSS 防止のためメッセージ本文の HTML エスケープを徹底。リンクは自動検出してクリック可能にするが、`javascript:` スキーム等は除去
- **IDOR 防止**: `PUT /chat/messages/{id}` は投稿者本人チェック必須。`DELETE /chat/channels/{id}/members/{userId}` は本人退出 or チャンネル OWNER/ADMIN のアクション
- **監査ログ**: 以下のイベントを監査ログに記録
  - `CHAT_CHANNEL_DELETED`: チャンネル削除
  - `CHAT_MESSAGE_DELETED_BY_ADMIN`: ADMIN によるメッセージ削除（投稿者本人以外の削除）
  - `CHAT_MEMBER_REMOVED`: メンバーの強制退出
- **レートリミット**: `Bucket4j` で以下の制限を適用
  - メッセージ送信 API（`POST /messages`）: 1分間60件
  - リアクション追加 API（`POST /reactions`）: 1分間30件
  - WebSocket メッセージ送信: 1秒間5件
- **添付ファイル**: アップロード前に MIME タイプをホワイトリスト検証（`image/*`, `application/pdf`, `text/plain`, `application/zip`, `video/mp4`）。ファイルサイズ上限は静止画・ドキュメント10MB、動画50MB。R2 Presigned URL の有効期限 10分。配信はチャンネルメンバーシップを Cloudflare Workers で検証した上で、Workers ルート経由の署名付き URL or R2 ストリーミングで行う
