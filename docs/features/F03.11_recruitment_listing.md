# F03.11: 募集型予約（クラス・個人参加・対戦相手募集・審判募集）

> **ステータス**: 🟡 設計レビュー中
> **実装フェーズ**: Phase 3
> **最終更新**: 2026-04-08
> **バージョン**: v0.3.2（マスター御回答10項目を確定 — 設計書フリーズ）
> **関連ドキュメント**: F03.4 予約管理（ライン連動・楽観的ロックパターン）、F03.8 イベント管理、F04.3 プッシュ通知（新着配信基盤）、F04.9 確認通知システム（確認ボタンUI流用）、F02.2 マイダッシュボード（ウィジェット追加）、F05.1 掲示板（カテゴリ階層パターン）、`Supporter` / `Follow` 機能（配信先）

## 1. 概要

スポーツチーム・施設・教室などが「日時・人数・カテゴリ」を指定した **募集枠** を作成し、フォロワーやサポーター、または不特定多数のユーザーが個人またはチーム単位で参加申込できる機能。LaBOLA予約のような **「フォロー先の新着募集が自動でマイページに流れる」** 体験を、Mannschaftの既存サポーター/フォロー基盤の上に構築する。

### 想定ユースケース

| カテゴリ | ユースケース例 |
|---|---|
| **個人参加クラス** | フットサル個人参加・ヨガクラス・水泳教室・ダンスレッスン・ボルダリング体験会 |
| **チーム参加** | 草サッカー対戦相手募集・バスケット練習試合相手募集・少年野球の練習試合 |
| **役職募集** | 審判募集（公式戦の審判員）・コーチ募集・スタッフ募集 |
| **イベント** | 体験会・交流会・親睦会・見学会 |

### 既存予約 (F03.4) との違い

| 観点 | F03.4 予約管理 | **F03.11 募集型予約（本機能）** |
|---|---|---|
| 1枠あたり | 1ユーザーが時間枠を独占 | 複数ユーザー/チームが同枠に参加 |
| 主体 | 整骨院・美容室等の施設 | スポーツチーム・教室・施設 |
| 通知 | 確定者のみリマインド | **フォロワー/サポーターへ新着配信** |
| 場所 | 施設のライン（席・ベッド） | 自由入力 + 既存予約ライン任意連動 |
| 決済 | チームで管理 | 募集ごとにON/OFF切替可 |
| 公開範囲 | スコープ内のみ | PUBLIC / SCOPE_ONLY / SUPPORTERS_ONLY |
| キャンセル待ち | なし | あり（自動昇格） |
| 最小定員 | なし | あり（未達時に自動キャンセル） |

### 役割分離

- **施設/チーム側**: 募集枠の作成・テンプレート管理・参加者管理・自動配信スコープ設定
- **個人側**: 募集の発見・申込・キャンセル・自分の参加予定確認・自動キャンセル通知への応答

### タイムゾーン方針

- 全ての DATETIME カラムは **UTC で保存**
- API のリクエスト/レスポンスは **ISO 8601 形式（タイムゾーン情報を含む）**
- 表示時はクライアント側の `Asia/Tokyo` （日本ユーザーのデフォルト）に変換
- バッチ処理の比較は `NOW()` (= UTC) で実施

**インデックス**
```sql

```

**CHECK 制約（DB レベルで保証 — MySQL 8.0.16+ で有効）**
```sql
CHECK (min_capacity <= capacity)
CHECK (application_deadline < start_at)         -- 厳密に「開始前」
CHECK (auto_cancel_at <= application_deadline)
CHECK (start_at < end_at)
CHECK (waitlist_max <= 1000)                    -- DoS対策の上限
CHECK (NOT (payment_enabled = TRUE AND price IS NULL))
```

**重要**: MySQL 8.0.16 未満では CHECK 制約は無視されるため、Service 層でも同等のバリデーションを実施する（防御的二重検証）。

**重要な仕様**
- `confirmed_count` と `capacity` は **同じ単位**（INDIVIDUAL=人、TEAM=チーム）
- チーム参加で「1チーム10人」のような選手数管理は本テーブルでは行わない（チームIDで参加し、参加メンバー詳細はチームメンバー管理に委ねる）
- `participation_type` は作成後変更不可（参加者既存のため）
- **編集時の制約**: 既に `confirmed_count > 0` の状態で `capacity < confirmed_count` への変更は 400 エラー（`RECRUITMENT_206`）
- **`participant_count_cache` の更新責任**: **Service 層で管理** することを v0.2.1 で確定。トリガーは使用しない。理由: (1) InnoDB トリガーのデッドロックリスク回避、(2) アプリ層で更新箇所を集約しやすい、(3) テスト容易性。RecruitmentParticipantService の `applyConfirmed()` / `applyCancel()` / `promoteFromWaitlist()` 等のメソッド内で必ず本カラムも同一トランザクションで更新する

| `active_subject_key` | VARCHAR(50) | NO | (Generated, Stored) | 生成カラム: `CONCAT(participant_type, '-', COALESCE(user_id, team_id), '-', COALESCE(deleted_at_epoch, 'active'))` |

**生成カラム定義**
```sql
-- MySQL 8.0 互換: UNIX_TIMESTAMP() は非決定性（タイムゾーン依存）のため使わず
-- DATE_FORMAT を使用。VARCHAR(100) で BIGINT 19桁 × 2 + フォーマット文字列を許容
active_subject_key VARCHAR(100) GENERATED ALWAYS AS (
  CONCAT(
    participant_type, '-',
    COALESCE(CAST(user_id AS CHAR), CAST(team_id AS CHAR)), '-',
    COALESCE(DATE_FORMAT(deleted_at, '%Y%m%d%H%i%s'), 'active')
  )
) STORED
```

**生成カラム設計の根拠**:
- `UNIX_TIMESTAMP()` は非決定性関数（タイムゾーン依存）のため STORED 生成カラムでは禁止
- `DATE_FORMAT(deleted_at, '%Y%m%d%H%i%s')` は決定性で安全
- 削除→再作成シナリオで `deleted_at` 値が異なれば `active_subject_key` も異なるため、UNIQUE 制約に抵触しない
- VARCHAR(100) は `INDIVIDUAL-9223372036854775807-20261231235959` (44文字) × 余裕を確保

**UNIQUE / INDEX**
```sql
-- 同じ参加者が同じ募集に複数同時申込できない（NULL問題を回避）
UNIQUE KEY uk_rp_listing_subject (listing_id, active_subject_key)
INDEX idx_rp_listing_status (listing_id, status, applied_at)
INDEX idx_rp_user_status (user_id, status, applied_at)
INDEX idx_rp_team_status (team_id, status, applied_at)
INDEX idx_rp_waitlist (listing_id, status, waitlist_position)
```

**CHECK 制約**
```sql
CHECK (
  (participant_type = 'USER' AND user_id IS NOT NULL AND team_id IS NULL) OR
  (participant_type = 'TEAM' AND team_id IS NOT NULL AND user_id IS NULL)
)
```

**バリデーション（Service層）**
- `participation_type=INDIVIDUAL` の listing には `participant_type=USER` のみ
- `participation_type=TEAM` の listing には `participant_type=TEAM` のみ

### 3.9 F04.9 `confirmable_notifications` 拡張

V3.124 マイグレーションで `confirmable_notifications.source_type` に `RECRUITMENT_LISTING` を追加。新規テーブル `recruitment_announcements` は **作成しない**。

```sql
  MODIFY COLUMN source_type ENUM('EMERGENCY_CLOSURE','RECRUITMENT_LISTING') NOT NULL;
```

確認通知の利用パターン:

- `source_id` = `recruitment_listings.id`

- `i18n_message_key` で多言語対応（例: `recruitment.notification.auto_cancelled`）
- `expires_at` を created_at + 72時間で自動セット
- 既存の確認済みボタンUIをそのまま流用

**🔒 デプロイ順序の制約（重要）**

F04.9 の既存実装が `source_type='EMERGENCY_CLOSURE'` だけを前提としているため、`RECRUITMENT_LISTING` 拡張のデプロイは以下の順序を **厳守**:

```
ステップ1: F04.9 の Service / Batch / UI を更新し、未知の source_type を安全に無視する防御コードを追加 → デプロイ
ステップ2: V3.124 マイグレーション実行（ENUM 拡張）
ステップ3: F03.11 Phase 3 の本番リリース（confirmable_notifications への INSERT 開始）
```

ステップ1 を飛ばすと、既存 F04.9 バッチが `RECRUITMENT_LISTING` レコードを処理できず例外を吐く可能性。**Phase 3 の実装着手前に F04.9 の実装責任者と調整必須。**

### 3.11 `recruitment_cancellation_policy_tiers`

キャンセル料の段階定義。1ポリシーあたり最大4段階。

**カラム命名の重要原則**: 「`applies_at_or_before_hours`」は **「開催の何時間前以内ならこの段階」** という意味で、値が **小さいほど開催に近い**。段階は「広い時間範囲（無料境界）から狭い時間範囲（直前）へ」の順で設定する。

| カラム | 型 | NULL | デフォルト | 説明 |
|---|---|---|---|---|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `policy_id` | BIGINT UNSIGNED | NO | — | FK → recruitment_cancellation_policies ON DELETE CASCADE |
| `tier_order` | TINYINT UNSIGNED | NO | — | 段階番号（1〜4。値が大きいほど開催直前・高料金） |
| `applies_at_or_before_hours` | INT UNSIGNED | NO | — | この段階が適用される境界（開催の N 時間前以内）。値が**小さいほど開催直前** |
| `fee_type` | ENUM('PERCENTAGE','FIXED') | NO | — | 料金タイプ |
| `fee_value` | INT UNSIGNED | NO | — | 値（PERCENTAGE: 1〜100、FIXED: 円） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | |

**CHECK 制約**
```sql
CHECK (tier_order BETWEEN 1 AND 4)
CHECK ((fee_type = 'PERCENTAGE' AND fee_value BETWEEN 1 AND 100) OR (fee_type = 'FIXED' AND fee_value > 0))
```

**UNIQUE / インデックス**
```sql
UNIQUE KEY uk_rcpt_policy_order (policy_id, tier_order, deleted_at)
INDEX idx_rcpt_policy_lookup (policy_id, applies_at_or_before_hours)  -- 計算用
```

**バリデーション（Service 層で必須）**
- `tier_order` が大きいほど `applies_at_or_before_hours` は **小さい**（=より直前）
- 全 tier の `applies_at_or_before_hours` が `policy.free_until_hours_before` より小さいこと（無料境界より直前のみ段階定義可能）
- 4段階を超える追加は 400 + `RECRUITMENT_303`

**段階設定の例（フットサル個参の場合）**

`free_until_hours_before = 168` の前提で:

| tier_order | applies_at_or_before_hours | fee_type | fee_value | 適用範囲（解釈） |
|---|---|---|---|---|
| (無料) | — (free_until=168) | — | — | 168時間以上前のキャンセル |
| 1 | 168 | PERCENTAGE | 30 | 168時間以内（=7日以内・3日以上前） |
| 2 | 72 | PERCENTAGE | 50 | 72時間以内（=3日以内・1日以上前） |
| 3 | 24 | PERCENTAGE | 80 | 24時間以内（=1日以内・2時間以上前） |
| 4 | 2 | PERCENTAGE | 100 | 2時間以内（=開催直前） |

**料金計算アルゴリズム（5.9 と整合）**:
1. `hoursBefore = (start_at - now) / 3600`
2. `hoursBefore >= free_until_hours_before` → 無料
3. それ以外: 全 tier から `applies_at_or_before_hours >= hoursBefore` を満たすもののうち **最大の `tier_order`** を選択（=最も直前の段階）

**重要な仕様**
- **永続保持**: ペナルティが解除されても履歴は残す（マスター指定）
- **ペナルティ判定の対象**: `confirmed = TRUE AND (dispute_resolution IS NULL OR dispute_resolution != 'REVOKED')` のレコードのみ

```sql
  AND recorded_at <= NOW() - INTERVAL 24 HOUR
```

```sql
-- v0.3.1: 「同一ユーザー × 同一スコープに同時1件」をDB制約で強制
-- MySQL 8.0.13+ の関数インデックスを利用
UNIQUE KEY uk_rup_active_one (
  user_id, 
  scope_type, 
  (CASE WHEN scope_id IS NULL THEN 0 ELSE scope_id END),
  (CASE WHEN lifted_at IS NULL THEN 1 ELSE 0 END)
)
-- ※ 解釈: 同じ user × scope_type × scope_id × アクティブフラグ の組み合わせは1件のみ
-- アクティブ(lifted_at IS NULL)が複数になることを禁止
```

**ペナルティ判定の基本クエリ**

```sql
-- ユーザーが特定スコープへの申込を許可されているか
SELECT COUNT(*) = 0 AS allowed
FROM recruitment_user_penalties
WHERE user_id = ?
  AND lifted_at IS NULL
  AND expires_at > NOW()
  AND (
    (scope_type = 'GLOBAL') OR
    (scope_type = ? AND scope_id = ?)
  )
```

**重要な仕様**
- **永続保持**: `lifted_at IS NOT NULL` のレコードも履歴として残す
- **同一ユーザー × 同一スコープに同時1件のみ**: DB UNIQUE 制約で強制（上記）+ Service層は **PESSIMISTIC_WRITE で SELECT FOR UPDATE** してから INSERT
- ペナルティ中の同スコープへの申込は 403 + `RECRUITMENT_300`
- TOCTOU 対策: §5.11 のコード例参照

## 4. ステータス遷移

### 4.1 `recruitment_listings.status`

```
DRAFT ──[publish]──▶ OPEN ──[満員到達]──▶ FULL
   │                  │  ▲                │
   │                  │  │  [キャンセル発生で空きあり]
   │                  │  └────────────────┘
   │                  │
   │                  ├──[application_deadline 到達]──▶ CLOSED ──[end_at 到達]──▶ COMPLETED
   │                  │                                    │
   │                  │                                    └──▶ COMPLETED
   │                  │
   │                  ├──[管理者がキャンセル]──▶ CANCELLED
   │                  │
   │                  └──[最小定員未達 + auto_cancel_at到達]──▶ AUTO_CANCELLED
   │
   └──[作成者が破棄]──▶ (論理削除)
```

| 状態 | 説明 | 申込可 | 編集可 | 閲覧可 |
|---|---|---|---|---|
| `DRAFT` | 下書き。作成者・ADMIN のみ閲覧可 | ✗ | ✓ | 作成者・ADMIN のみ |
| `OPEN` | 募集中 | ✓ | ✓（一部制限） | visibility に従う |
| `FULL` | 満員 | キャンセル待ちのみ | ✓（一部制限） | visibility に従う |
| `CLOSED` | 締切後・開催待ち | ✗ | ✗（場所/時刻のみ） | visibility に従う |
| `CANCELLED` | 主催者キャンセル | ✗ | ✗ | visibility に従う |
| `AUTO_CANCELLED` | 自動キャンセル | ✗ | ✗ | visibility に従う |
| `COMPLETED` | 開催終了 | ✗ | ✗ | visibility に従う |

**FULL → OPEN 自動復帰**: `CONFIRMED` 参加者がキャンセルし `confirmed_count < capacity` となった瞬間、`status` を `FULL` から `OPEN` に自動遷移する（5.3 参照）。

### 4.2 `recruitment_participants.status`

```
APPLIED ─┬─[空きあり]──▶ CONFIRMED ─┬─[出席チェック]──▶ ATTENDED
         │                          │
         │                          ├─[欠席]──▶ NO_SHOW
         │                          │
         │                          └─[本人キャンセル]──▶ CANCELLED
         │
         └─[満員]──▶ WAITLISTED ─┬─[他者キャンセル → 自動昇格]──▶ CONFIRMED
```

## 5. ビジネスルール

### 5.1 募集作成

1. ADMIN または `MANAGE_RECRUITMENTS` 権限を持つ DEPUTY_ADMIN のみ
2. 必須項目: `category_id`, `title`, `start_at`, `end_at`, `application_deadline`, `auto_cancel_at`, `capacity`, `min_capacity`, `participation_type`, `visibility`, 配信対象 (`distribution_targets` 1件以上)
3. CHECK 制約 (3.4) を全て満たすこと
4. **`visibility` と `distribution_targets` の整合性チェック**（3.7 参照）
5. テンプレートから複製する場合は `template_id` を指定し、デフォルト値を **スナップショットコピー**
6. `reservation_line_id` を指定した場合、衝突チェック（5.6）
7. 作成直後は `DRAFT`。`POST .../publish` で `OPEN` に遷移
8. `OPEN` 遷移時に新着配信通知を非同期キューイング

### 5.2 参加申込

1. ログイン必須
2. 公開範囲チェック
3. `application_deadline > NOW()` を確認
4. **🔒 ペナルティチェック（v0.3 で追加）**: ユーザーが該当スコープに対して有効なペナルティを保持していないこと
   ```sql
   SELECT 1 FROM recruitment_user_penalties
   WHERE user_id = :userId
     AND lifted_at IS NULL
     AND expires_at > NOW()
     AND (
       scope_type = 'GLOBAL' OR
       (scope_type = :listingScopeType AND scope_id = :listingScopeId)
     )
   LIMIT 1
   ```
   → 1件でも該当 → 403 + `RECRUITMENT_300`（解除予定時刻もエラーレスポンスに含める）

5. **🔒 未払いキャンセル料チェック（v0.3.1 で追加）**: ユーザーが過去の未払いキャンセル料を保持していないこと
   ```sql
   SELECT 1 FROM recruitment_cancellation_records
   WHERE user_id = :userId
     AND payment_status IN ('PENDING','FAILED')
   LIMIT 1
   ```
   → 1件でも該当 → 402 + `RECRUITMENT_301`（未払い決済リンクをレスポンスに含める）
   - 例外: `THIS_SCOPE_ONLY` 設定で別スコープの未払いがある場合の挙動はマスター確認事項（§20）
5. 1ユーザー(or 1チーム)は同じ募集に1回のみ申込可（生成カラム UNIQUE で保証）
6. `participation_type=TEAM` の場合、申込者がそのチームの ADMIN/DEPUTY_ADMIN/MEMBER であること
7. **楽観的ロック**: 
   ```sql
   UPDATE recruitment_listings
   SET confirmed_count = confirmed_count + 1,
       status = CASE WHEN confirmed_count + 1 = capacity THEN 'FULL' ELSE status END
   WHERE id = ? AND status IN ('OPEN') AND confirmed_count < capacity
   ```
   affected rows = 0 → 満員 or 不正状態。キャンセル待ちフローへ
7. 確定時:
   - `recruitment_participants.status = CONFIRMED`
   - `recruitment_participant_history` に記録
   - `recruitment_reminders` に開催 24h 前リマインドを登録
8. キャンセル待ち時（**楽観的ロックで上限を保証**）:
   ```sql
   -- waitlist_count を原子的にインクリメント。上限超過なら affected rows = 0
   UPDATE recruitment_listings
   SET waitlist_count = waitlist_count + 1,
       next_waitlist_position = next_waitlist_position + 1
   WHERE id = ? AND waitlist_count < waitlist_max
   ```
   - affected rows = 0 → `RECRUITMENT_106`（上限超過）
   - 続けて `recruitment_participants.status = WAITLISTED` で INSERT
   - **`waitlist_position` の採番**: 上記 UPDATE で得られた `next_waitlist_position` の値を使用（O(1)・MAX クエリ不要）
9. **レート制限**: 1ユーザーあたり申込API は 1分10回まで（連投防止）

**注**: `recruitment_listings` に `next_waitlist_position INT UNSIGNED DEFAULT 1` カラムを追加し、O(1)で順位採番できるようにする。3.4 のテーブル定義に追加要。

### 5.3 キャンセル待ち / 自動昇格 / FULL→OPEN 復帰

**設計方針**: 全ての操作を **PESSIMISTIC_WRITE 行ロック** で原子化。トランザクション分離レベルは `READ_COMMITTED`（Spring Boot のデフォルト）。

1. `CONFIRMED` 参加者がキャンセル → 以下を**単一トランザクション内**で実行:
   ```java
   @Transactional
   public void cancelByParticipant(Long participantId) {
     // (1) 参加者をキャンセル
     RecruitmentParticipant participant = participantRepo.findByIdForUpdate(participantId);
     participant.cancel(currentUser());
     historyRepo.save(participant.toHistoryRecord());
     
     // (2) 募集行を PESSIMISTIC_WRITE でロック
     RecruitmentListing listing = listingRepo.findByIdForUpdate(participant.getListingId());
     
     // (3) confirmed_count 減算 + FULL→OPEN 自動復帰 + cache 更新
     listing.decrementConfirmed();  // status を内部で同時更新
     // → エンティティ内で: if(status==FULL && confirmed_count<capacity) status=OPEN
     listing.decrementParticipantCountCache();
     
     // (4) 同じトランザクション内でキャンセル待ち昇格を試行
     promoteFromWaitlistIfPossible(listing);  // 5.3.5へ
   }
   ```

2. **キャンセル待ち昇格処理 (`promoteFromWaitlistIfPossible`)**: 既に PESSIMISTIC_WRITE で行ロック済みの listing を引き継ぐ
   ```java
   private void promoteFromWaitlistIfPossible(RecruitmentListing listing) {
     if (listing.getConfirmedCount() >= listing.getCapacity()) return;
     
     // 待機中の最古を取得（同一トランザクション内）
     Optional<RecruitmentParticipant> next = participantRepo
       .findFirstByListingIdAndStatusOrderByWaitlistPosition(listing.getId(), WAITLISTED);
     if (next.isEmpty()) return;
     
     // 昇格 + count 更新
     RecruitmentParticipant promoted = next.get();
     promoted.promoteToConfirmed();
     historyRepo.save(promoted.toHistoryRecord(AUTO_PROMOTE));
     listing.incrementConfirmed();  // 必要なら status を OPEN→FULL へ自動遷移
     listing.decrementWaitlistCount();
     
     // リマインド + 通知配信
     reminderRepo.save(buildReminder(promoted));
     notificationService.publish(WAITLIST_PROMOTED, promoted);
   }
   ```

3. **重要な保証**: `listing` を `findByIdForUpdate`（PESSIMISTIC_WRITE）でロック取得しているため、ステップ (1)〜(4) の間に別のトランザクションが `confirmed_count` を変更することはない。**FULL→OPEN 復帰漏れの race condition は発生しない**。

4. **キャンセル待ち順位の再採番**: O(n) 操作を避けるため、**即時の再採番は行わず**、`waitlist_position` を「採番済み数値」として保持。表示時にユーザーの相対順位を `ROW_NUMBER() OVER (ORDER BY waitlist_position) WHERE status='WAITLISTED'` で計算。`recruitment-waitlist-defrag-batch`（1時間ごと、PESSIMISTIC_WRITE）で「採番値の最大値が `waitlist_count * 3` を超えた場合」のみ再採番を実施。

### 5.4 最小定員未達 / 自動キャンセル

1. バッチ処理（5分間隔）で対象を **1件ずつ** ロック取得（**`PESSIMISTIC_WRITE`**）:
   ```java
   @Lock(LockModeType.PESSIMISTIC_WRITE)
   @Query("""
     SELECT r FROM RecruitmentListing r
     WHERE r.status = 'OPEN' AND r.autoCancelAt <= :now
     ORDER BY r.autoCancelAt ASC
     """)
   List<RecruitmentListing> lockAutoCancelCandidates(LocalDateTime now, Pageable pageable);
   ```
   ※ 1回のバッチで処理する募集数を Pageable で制限（例: 50件/バッチ）
2. `confirmed_count < min_capacity` の場合:
   - `status = AUTO_CANCELLED`
   - **参加者の CANCELLED 遷移は 100件チャンクで実行**:
     ```java
     // 大量参加者のロック保持時間を抑える
     while (true) {
       List<RecruitmentParticipant> chunk = participantRepo
         .findByListingIdAndStatusIn(listingId, List.of(CONFIRMED, WAITLISTED), PageRequest.of(0, 100));
       if (chunk.isEmpty()) break;
       chunk.forEach(p -> {
         p.cancelBySystem();
         historyRepo.save(p.toHistoryRecord(AUTO_CANCEL));
         confirmableNotificationService.create(  // F04.9 連携
           p.getUserId(),
           "RECRUITMENT_LISTING",
           listingId,
           "recruitment.notification.auto_cancelled",
           NOW().plusHours(72)
         );
       });
       participantRepo.saveAll(chunk);
     }
     ```
   - プッシュ通知 `RECRUITMENT_AUTO_CANCELLED` 配信
3. `confirmed_count >= min_capacity` の場合: 何もしない（開催確定）

**レースコンディション対策**: 
- `PESSIMISTIC_WRITE` で募集行をロックし、同時刻に新規申込が走っても整合性を保つ
- バッチの **同時実行防止**: ShedLock または Quartz の `disallowConcurrentExecution=true` を採用（11章参照）
- バッチ間競合防止: 5.4 の auto-cancel と 5.5（status-close）が同じ募集を扱う場合、`status = 'OPEN'` 条件で互いに排他

### 5.5 確認済みボタンフロー（F04.9 連携）

1. 自動キャンセル等の重大通知が発生 → F04.9 `confirmable_notifications` に INSERT
2. 個人ダッシュボードに F04.9 提供のバナー UI が自動表示
3. 「確認済みにする」ボタン押下 → F04.9 既存 API で `acknowledged_at` 記録
4. 主催者管理画面では「未確認の参加者一覧」を F04.9 API 経由で取得
5. **未確認 24時間経過 → 再通知**（F04.9 既存バッチを利用）
6. **未確認 72時間経過 → expires_at により無効化**（F04.9 既存バッチ）

### 5.6 予約ライン (F03.4) 衝突チェック

`reservation_line_id` を指定した募集の作成・編集時に以下をチェック:

```sql
-- 既存予約 (F03.4) との衝突
SELECT 1 FROM reservations
WHERE reservation_line_id = ?
  AND status IN ('PENDING','CONFIRMED','COMPLETED')   -- CANCELLED/NO_SHOW は除外
  AND start_at < ?     -- 募集の end_at
  AND end_at > ?       -- 募集の start_at
  AND deleted_at IS NULL
LIMIT 1

-- 既存募集との衝突
SELECT 1 FROM recruitment_listings
WHERE reservation_line_id = ?
  AND status IN ('OPEN','FULL','CLOSED','COMPLETED')  -- CANCELLED 系は除外
  AND start_at < ?
  AND end_at > ?
  AND deleted_at IS NULL
  AND id <> ?  -- 編集時は自分自身を除外
LIMIT 1
```

衝突あり → **409 + `RECRUITMENT_009`**

### 5.7 編集時の制約（PATCH API）

- `participation_type` は変更不可（既存参加者がいるため）
- `capacity` を変更する場合: `capacity >= confirmed_count` を強制（違反時 400 + `RECRUITMENT_206`）
- `min_capacity` を変更する場合: `min_capacity <= capacity` を強制
- `start_at` / `end_at` / `application_deadline` / `auto_cancel_at` 変更時は CHECK 制約全てを再評価
- 既存 `confirmed_count > 0` の状態で `start_at` を変更した場合、参加者全員に `RECRUITMENT_UPDATED` 通知を配信
- `visibility` を狭める変更（PUBLIC → SUPPORTERS_ONLY 等）は許可するが、`distribution_targets` の整合性を再チェック

### 5.8 チーム参加時のメンバー脱退

- 申込チームから申込者(`applied_by`)が脱退しても、参加レコードは保持
- ただし、**チーム自体が論理削除された場合**は参加レコードも `CANCELLED` に自動遷移し、確定数を減算

### 5.9 キャンセル料計算ロジック

**前提**: `recruitment_listings.cancellation_policy_id IS NOT NULL` かつ `payment_enabled = TRUE` のとき有効。`payment_enabled = FALSE` の場合、ポリシーが設定されていてもキャンセル料は発生しない（NO_SHOW ペナルティのみ適用）。

**計算アルゴリズム（v0.3.1 で命名・論理を全面修正）**:
```
1. now = キャンセル発生時刻（精度: 秒単位）
2. hoursBefore = (start_at - now) を小数時間で算出（例: 2.5時間前）
3. hoursBefore >= policy.free_until_hours_before → 完全無料、fee_amount = 0
4. それ以外:
   -- 該当する段階を取得（直前ほど tier_order が大きい）
   matchingTier = SELECT * FROM recruitment_cancellation_policy_tiers
                  WHERE policy_id = ?
                    AND applies_at_or_before_hours >= hoursBefore   -- 境界以内
                    AND deleted_at IS NULL
                  ORDER BY tier_order DESC LIMIT 1   -- 最も直前（高料金）の段階
   
   if matchingTier IS NULL → tier 設定漏れ、fee_amount = 0（フォールバック）
   else if matchingTier.fee_type = 'PERCENTAGE' → fee_amount = CEILING(listing.price * fee_value / 100)
   else (FIXED) → fee_amount = matchingTier.fee_value
5. recruitment_cancellation_records に記録（INSERT only、後で UPDATE 禁止）:
   - applied_tier_id = matchingTier?.id
   - fee_amount = 計算結果
   - payment_status = (fee_amount > 0) ? 'PENDING' : 'NOT_REQUIRED'
```

**境界の厳密性（v0.3.1 修正）**:
- `hoursBefore >= free_until_hours_before` （**等号含む**）→ 無料
- 境界値ちょうど（例: 168時間前ぴったり）にキャンセルした場合 → 無料側に倒れる（ユーザー有利）

**ロジックの説明（マスター向け）**:
- 段階は「`applies_at_or_before_hours` 以内のキャンセルに適用」される
- 例: tier1 (`168h`) / tier2 (`72h`) / tier3 (`24h`) / tier4 (`2h`)
  - キャンセルが100時間前 → tier1 と tier2 が条件を満たす（100 ≤ 168, 100 ≤ 72?）→ いや 100 > 72 なので tier2 は不該当 → tier1 のみ → 30%
  - キャンセルが50時間前 → tier1, tier2 が条件を満たす（50 ≤ 168, 50 ≤ 72）→ tier_order 大きい方 → tier2 → 50%
  - キャンセルが10時間前 → tier1, tier2, tier3 → tier3 → 80%
  - キャンセルが1時間前 → tier1, tier2, tier3, tier4 → tier4 → 100%

**決済フロー**:
1. キャンセル成立時に `recruitment_cancellation_records` を `payment_status='PENDING'` で作成
2. 既存決済システム（F03.4 と同じ pay-link）に引き渡し → `payment_id` を記録
3. 決済成功 → `payment_status='PAID'`
4. 決済失敗 → `payment_status='FAILED'` + 管理者通知 + ユーザーへリトライ案内
5. 管理者判断で免除可能 → `payment_status='WAIVED'` + 監査ログ

**「キャンセル料が発生する」ユーザー体験**:
- キャンセル前に **必ず確認モーダル** を表示
- モーダル内容: 「現在のキャンセル料: ¥{fee_amount}（policy 内の段階表示）」
- ユーザーが同意した上でキャンセル実行
- 確認モーダルを経由しない API 呼び出しは 400 + `RECRUITMENT_304`（フロント・バック双方で `acknowledged_fee=true` フラグを必須化）

**ポリシーが NULL の場合**:
- `cancellation_policy_id IS NULL` → 完全無料キャンセル
- ただし NO_SHOW ペナルティは別途適用される

### 5.10 無断キャンセル（NO_SHOW）判定

**判定方法は2つ**:

**方法A: 管理者が手動マーク（推奨デフォルト）**
- 開催終了後、主催者が参加者一覧画面で「出席」「欠席」「無断欠席」を手動チェック
- `recruitment_participants.status` を `NO_SHOW` に変更
- `recruitment_no_show_records` に `reason='ADMIN_MARKED'` で INSERT
- `recruitment_cancellation_records` も `cancel_source='SYSTEM_NO_SHOW'`, `fee_amount=0` で記録（料金請求は別途、ポリシー適用外）

**方法B: 自動検出（オプション機能、デフォルトOFF）**
- 開催終了から **24時間後** に `recruitment-no-show-detect-batch` が実行
- 条件: `status='CONFIRMED'` のまま、`status_changed_at < end_at` （管理者がチェックしていない）
- `recruitment_penalty_settings.is_enabled = TRUE` かつ `auto_no_show_detection = TRUE`（カラム追加）の場合のみ自動マーク
- `reason='AUTO_DETECTED'` で INSERT
- 自動検出は誤判定リスクが高いため、必ず **管理者確認画面** を経由（即座に NO_SHOW 確定はしない）

**異議申立フロー**:
- ユーザーは NO_SHOW 記録に対して 7日以内に異議申立可能
- `recruitment_no_show_records.disputed = TRUE` に更新
- 主催者が確認 → `dispute_resolution = 'UPHELD'`（維持） or `'REVOKED'`（取り消し）
- `REVOKED` の場合、ペナルティ判定の対象外にする（5.11 のクエリで除外）

### 5.11 ペナルティ適用フロー

**トリガー**: `recruitment_no_show_records` に新規 INSERT された直後（同期処理）

**判定アルゴリズム**:
```sql
-- (1) 該当ユーザー × 該当スコープのペナルティ設定を取得
SELECT * FROM recruitment_penalty_settings
WHERE scope_type = ? AND scope_id = ? AND is_enabled = TRUE

-- (2) 過去 threshold_period_days 以内の NO_SHOW 回数をカウント
--     ただし dispute_resolution='REVOKED' は除外
SELECT COUNT(*) FROM recruitment_no_show_records nsr
JOIN recruitment_listings rl ON nsr.listing_id = rl.id
WHERE nsr.user_id = ?
  AND nsr.recorded_at >= NOW() - INTERVAL :threshold_period_days DAY
  AND (nsr.dispute_resolution IS NULL OR nsr.dispute_resolution != 'REVOKED')
  AND (
    -- apply_scope に応じてフィルタ
    (:apply_scope = 'THIS_SCOPE_ONLY' AND rl.scope_type = ? AND rl.scope_id = ?) OR
    (:apply_scope = 'ALL_SCOPES')
  )

-- (3) count >= threshold_count → ペナルティ発動
```

**ペナルティ発動処理（v0.3.1: TOCTOU 対策で PESSIMISTIC_WRITE 採用）**:
```java
@Transactional
public void applyPenaltyIfNeeded(Long userId, ScopeType scopeType, Long scopeId, int noShowCount) {
  PenaltySetting setting = settingsRepo.findByScope(scopeType, scopeId);
  if (!setting.isEnabled() || noShowCount < setting.getThresholdCount()) return;
  
  ScopeType effectiveScopeType = setting.getApplyScope() == ALL_SCOPES ? GLOBAL : scopeType;
  Long effectiveScopeId = setting.getApplyScope() == ALL_SCOPES ? null : scopeId;
  
  // ① PESSIMISTIC_WRITE で行ロック取得（既存ペナルティ検索 = 排他制御）
  Optional<UserPenalty> existing = penaltyRepo
    .findActiveByUserAndScopeForUpdate(userId, effectiveScopeType, effectiveScopeId);
  
  if (existing.isPresent()) {
    // ② 既存ペナルティの「期限延長」処理（v0.3.1: 単純スキップではなく延長）
    UserPenalty current = existing.get();
    LocalDateTime newExpiresAt = current.getExpiresAt().plusDays(setting.getPenaltyDurationDays());
    current.extendExpiresAt(newExpiresAt, noShowCount);
    notificationService.publish(RECRUITMENT_PENALTY_EXTENDED, userId, current);
    auditLog.record("RECRUITMENT_PENALTY_EXTENDED", userId, scopeType, scopeId, noShowCount);
    return;
  }
  
  // ③ 新規ペナルティ作成（DB UNIQUE 制約と Service 排他制御の二重保証）
  UserPenalty penalty = UserPenalty.builder()
    .userId(userId)
    .scopeType(effectiveScopeType)
    .scopeId(effectiveScopeId)
    .triggeredBySettingId(setting.getId())
    .triggeredNoShowCount(noShowCount)
    .startedAt(now())
    .expiresAt(now().plusDays(setting.getPenaltyDurationDays()))
    .build();
  penaltyRepo.save(penalty);
  
  // ④ F04.9 確認通知 + 通常通知
  confirmableNotificationService.create(
    userId, "RECRUITMENT_PENALTY", penalty.getId(),
    "recruitment.notification.penalty_applied",
    now().plusHours(72)
  );
  notificationService.publish(RECRUITMENT_PENALTY_APPLIED, userId, penalty);
  auditLog.record("RECRUITMENT_PENALTY_APPLIED", userId, scopeType, scopeId, noShowCount);
}
```

**ペナルティ重複防御の三重構造（v0.3.1）**:
1. **DB UNIQUE 制約** (`uk_rup_active_one`): 物理的に同時2件を防止
2. **PESSIMISTIC_WRITE FOR UPDATE**: トランザクション内の TOCTOU 問題回避
3. **Service 層分岐**: 既存ペナルティがあれば「延長」処理（新規作成と区別）

**期限延長 vs 新規発動の判定**:
- 既存ペナルティが **存在しない** → 新規作成
- 既存ペナルティが **存在する** → `expires_at` を `+ penalty_duration_days` 延長 + `triggered_no_show_count` 累積更新 + `RECRUITMENT_PENALTY_EXTENDED` 通知

### 5.12 ペナルティ解除

**自動解除**:
- `recruitment-penalty-lift-batch` （1日1回 03:00 JST）が実行
- 対象: `lifted_at IS NULL AND expires_at <= NOW()`
- 処理: `lifted_at = NOW()`, `lift_reason = 'AUTO_EXPIRED'`
- 通知: `RECRUITMENT_PENALTY_LIFTED` を該当ユーザーに配信

**手動解除（管理者）— v0.3.2 マスター御回答を反映**:
- 権限: **ペナルティを発動した該当チーム/組織の ADMIN**（および `MANAGE_RECRUITMENT_PENALTIES` 権限を持つ DEPUTY_ADMIN）+ SYSTEM_ADMIN
  - `apply_scope='THIS_SCOPE_ONLY'` の場合: 該当 scope の ADMIN/権限保有 DEPUTY のみ
  - `apply_scope='ALL_SCOPES'`（GLOBAL）の場合: 元発動 scope の ADMIN または SYSTEM_ADMIN
- 条件: `lifted_at IS NULL`
- 必須項目:
  - `notes` 必須（**最低150文字**、解除理由の詳細）
  - `lift_reason = 'ADMIN_MANUAL'`
- レート制限: 同一管理者×同一ユーザーへの解除は **24時間に1回まで**
- 処理: `lifted_at = NOW()`, `lifted_by = currentUser`
- 監査ログ: 全項目を `audit_logs` に記録（before_state / after_state + notes 全文）
- ユーザーへの通知: `RECRUITMENT_PENALTY_LIFTED` + 解除した管理者の表示名 + `notes` の管理者コメント表示
- **権力濫用対策**: 同じ管理者が同じユーザーに対してペナルティ発動・解除を3回以上繰り返した場合、SYSTEM_ADMIN にアラート通知（監査検知ルール）

**異議申立による解除**:
- 元になった NO_SHOW 記録の `dispute_resolution = 'REVOKED'` に変更
- 同期的に `recruitment-penalty-recompute` を実行（後述）
- 閾値未満に戻れば `lift_reason = 'DISPUTE_REVOKED'` で自動解除

**ペナルティ再計算（v0.3.1 で追加）**:
- 専用ロジック `RecruitmentPenaltyService.recompute(userId, scopeType, scopeId)` を実装
- 実行タイミング:
  - 異議申立 REVOKED 時（同期）
  - `recruitment-penalty-recompute-batch`（1日1回、全アクティブペナルティ対象、闕値割れチェック）
- アルゴリズム:
  ```
  1. 該当ユーザーのアクティブペナルティを取得
  2. 該当スコープの penalty_settings.threshold_period_days で NO_SHOW 件数を再カウント（REVOKED 除外）
  3. count < threshold_count なら lifted_at = NOW(), lift_reason = 'DISPUTE_REVOKED' or 'AUTO_RECOMPUTED'
  4. 通知配信
  ```

**履歴の保持**:
- 解除されたペナルティレコードは **永続削除しない**（ユーザーマイページに「過去のペナルティ履歴」として表示可能）
- データ保持期間: マスター・法務確認事項（§20 オープン課題）

## 7. テンプレート機能

「同じ募集を毎週やる」ユースケースに対応。**スナップショット方式**で運用。

### 操作フロー
1. ADMIN がテンプレート作成
2. 新規募集作成画面で「テンプレートから作成」 → テンプレート選択
3. デフォルト値が自動入力された状態で募集作成画面が開く
4. 開催日時のみ調整して保存
5. **作成された募集はテンプレートから完全に独立**（テンプレート編集の影響を受けない）

### スナップショット方式の理由
- テンプレート編集後に既存募集の値が勝手に変わると主催者が混乱する
- `recruitment_listings.template_id` は履歴参照のみ（テンプレートが論理削除されても募集は維持）

```sql

WHERE team_id = ? AND status = 'ACTIVE'
  AND role IN ('ADMIN','DEPUTY_ADMIN','MEMBER')

-- SUPPORTERS: サポーター承認済み

WHERE scope_type = ? AND scope_id = ? AND status = 'APPROVED'

WHERE follower_type = 'USER' AND followed_type = ? AND followed_id = ?
```

複数の `distribution_targets` が設定された場合、UNION DISTINCT でユーザー集合をマージし、各ユーザーの `notification_type_preferences` で opt-out チェックを行ってから配信。

## 10. 通知種別

F04.3 通知システムに以下を追加。**優先度ごとの動作**を明確化:

| 種別 | トリガー | 配信先 | 優先度 | 動作 |
|---|---|---|---|---|
| `RECRUITMENT_PUBLISHED` | 募集が `OPEN` に遷移 | distribution_targets に従う | NORMAL | アプリ内通知 + プッシュ |
| `RECRUITMENT_CONFIRMED` | 申込が `CONFIRMED` に | 申込者本人 | NORMAL | アプリ内通知 + プッシュ |
| `RECRUITMENT_WAITLISTED` | 申込が `WAITLISTED` に | 申込者本人 | LOW | アプリ内通知のみ |
| `RECRUITMENT_WAITLIST_PROMOTED` | 自動昇格で `CONFIRMED` に | 対象参加者 | HIGH | アプリ内通知 + プッシュ + メール |
| `RECRUITMENT_REMINDER` | 開催24時間前 | 確定参加者 | NORMAL | アプリ内通知 + プッシュ |
| `RECRUITMENT_AUTO_CANCELLED` | 最小定員未達による自動キャンセル | 全参加者 | URGENT | F04.9 確認通知 + プッシュ + メール |
| `RECRUITMENT_CANCELLED` | 主催者による手動キャンセル | 全参加者 | URGENT | F04.9 確認通知 + プッシュ + メール |
| `RECRUITMENT_UPDATED` | 場所/時間等の重要変更 | 全確定参加者 | HIGH | アプリ内通知 + プッシュ |
| `RECRUITMENT_FULL` | 募集が満員に | 主催者 | LOW | アプリ内通知のみ |
| `RECRUITMENT_DEADLINE_APPROACHING` | 締切24時間前 | 主催者 | LOW | アプリ内通知のみ |
| `RECRUITMENT_CANCELLATION_FEE_NOTICE` | キャンセル料が発生（PENDING） | キャンセル者 | HIGH | プッシュ + メール（決済リンク含む） |
| `RECRUITMENT_PENALTY_APPLIED` | 無断キャンセルペナルティ発動 | 該当ユーザー | URGENT | F04.9 確認通知 + プッシュ + メール |
| `RECRUITMENT_PENALTY_LIFTED` | ペナルティ自動 or 手動解除 | 該当ユーザー | NORMAL | アプリ内通知 + プッシュ |
| `RECRUITMENT_NO_SHOW_RECORDED` | NO_SHOW として記録された | 該当ユーザー | HIGH | アプリ内通知 + プッシュ + 異議申立リンク |
| `RECRUITMENT_NO_SHOW_DISPUTE_RAISED` | 異議申立を受領 | 主催者 | NORMAL | アプリ内通知 |

**優先度別の動作仕様**:
- `URGENT`: F04.9 確認通知バナー（赤色）+ プッシュ通知 + メール送信。確認ボタンによる acknowledge 必須
- `HIGH`: 黄色バナー + プッシュ通知 + メール（オプション）
- `NORMAL`: 通常通知（青色）+ プッシュ通知
- `LOW`: アプリ内通知のみ（プッシュ・メールなし）

| `recruitment-no-show-confirm-batch` | 1時間 | NO_SHOW 仮マークから 24h 経過したものを `confirmed=TRUE` に確定 | OPTIMISTIC | ShedLock |
| `recruitment-penalty-lift-batch` | 1日（03:00 JST） | `expires_at <= NOW()` のペナルティを `AUTO_EXPIRED` で解除 | OPTIMISTIC | ShedLock |
| `recruitment-penalty-recompute-batch` | 1日（04:00 JST） | 全アクティブペナルティの闕値再判定（REVOKED 反映） | PESSIMISTIC_WRITE | ShedLock |
| `recruitment-payment-retry-batch` | 1時間 | `recruitment_cancellation_records.payment_status = 'FAILED'` のリトライ（最大3回） | OPTIMISTIC | ShedLock |

F04.9 既存バッチで対応する処理（`recruitment_announcements` 不要）:
- 確認通知の再通知（24h 経過）
- 確認通知の自動失効（72h 経過）

### 11.1 バッチ同時実行防止（重要）

複数Pod環境/重複起動でも安全に動作させるため、全バッチに **ShedLock** を適用:

```java
@Scheduled(fixedDelay = 5 * 60 * 1000)
@SchedulerLock(name = "recruitment-auto-cancel-batch", lockAtLeastFor = "PT4M", lockAtMostFor = "PT15M")
public void autoCancelBatch() { ... }
```

### 11.2 バッチ間の競合防止

`auto-cancel-batch` と `status-close-batch` が同じ募集を扱う場合の競合を防ぐ:
- 両バッチとも `status = 'OPEN'` を条件にしているため、片方が遷移させればもう片方は対象外
- ただし、`PESSIMISTIC_WRITE` を取らない `status-close-batch` は楽観的ロックで実装し、`@Version` カラムによる衝突検知で再試行

```java
// status-close-batch は OPTIMISTIC + Retry
@Retryable(retryFor = OptimisticLockingFailureException.class, maxAttempts = 3)
public void closeExpiredListings() { ... }
```

## 14. セキュリティ・バリデーション

### 14.1 認可
- 既存の `@PreAuthorize` パターン使用
- DRAFT 募集の閲覧は `created_by = currentUser OR currentUserHasRole('ADMIN')` を強制
- 公開範囲チェック: `visibility = SUPPORTERS_ONLY` の閲覧時は `SupporterApplicationEntity.status = APPROVED` を確認
- `visibility` と `distribution_targets` の整合性を Service 層で必ず検証（3.7 表）

### 14.2 楽観的ロック・整合性・トランザクション分離

**`confirmed_count` の更新ルール（必須）**:
- 通常の参加申込（5.2）→ **楽観的ロック**: `UPDATE ... WHERE confirmed_count < capacity` で原子的に処理。affected rows = 0 → 409 + `RECRUITMENT_005`
- キャンセル + 自動昇格（5.3）→ **悲観的ロック**: `findByIdForUpdate()` で行ロック取得後にトランザクション内で全操作
- 自動キャンセルバッチ（5.4）→ **悲観的ロック**: PESSIMISTIC_WRITE で並行更新を抑止
- **混在禁止**: 同一トランザクション内では楽観的・悲観的のいずれか一方のみを使用

**トランザクション分離レベル**:
- Spring Boot のデフォルト `READ_COMMITTED` を採用
- ファントムリードのリスクは PESSIMISTIC_WRITE で防止
- `application.yml` に明示:
  ```yaml
  spring:
    jpa:
      properties:
        hibernate:
          connection:
            isolation: 2  # READ_COMMITTED
  ```

### 14.3 N+1対策
- `participant_count_cache` カラムで一覧表示時のカウントクエリを排除
- 詳細取得時は `JOIN FETCH` で `category`, `subcategory`, `team`, `created_by` を一括取得
- 全体検索は denormalize した `participant_count_cache` を使用

### 14.4 画像URL ホワイトリスト

許可ドメイン（`application.yml` で管理）:
```yaml
mannschaft:
  recruitment:
    allowed-image-domains:
      - "cdn.mannschaft.app"
      - "mannschaft-prod.s3.ap-northeast-1.amazonaws.com"
      - "mannschaft-dev.s3.ap-northeast-1.amazonaws.com"
```

ホワイトリスト外URL → 400 + `RECRUITMENT_205`

### 14.5 レート制限
- 全体検索 API: 60 req/min/IP
- 申込 API: 10 req/min/user（連投・スクリプト対策）
- 新規募集作成 API: 30 req/hour/user（DoS 対策）

### 14.6 CSRF 対策
- 全 POST/PATCH/DELETE API は Spring Security の CSRF 保護を有効化
- Cookie ベースの CSRF トークンを使用（既存パターン踏襲）

### 14.7 SQLインジェクション対策
- 全クエリは Spring Data JPA / `@Query` のパラメタライズ使用
- 検索キーワードは LIKE エスケープ（`%` `_` をエスケープ）

### 14.8 監査ログ
- 募集の作成・編集・キャンセル・削除は既存の audit log 機構（`audit_logs` テーブル）に記録
- `entity_type='RECRUITMENT_LISTING'`, `action='CREATE|UPDATE|CANCEL|ARCHIVE'`
- 参加者ステータス遷移は `recruitment_participant_history` で別途管理

### 14.9 個人情報・プライバシー
- 参加者一覧の表示範囲:
  - **管理者**: 氏名・連絡先含む全情報
  - **同じ募集の他参加者**: 表示名のみ（連絡先非公開）
  - **未参加者**: 「現在 X 名参加中」のカウントのみ
- **`applied_by` の表示範囲**: 申込者個人情報のため、他参加者には公開しない。管理者のみ閲覧可
- GDPR: ユーザー削除時、本テーブルの `created_by` / `applied_by` / `cancelled_by` は SET NULL（参加履歴は保持、本人特定情報のみ削除）
- **チーム参加の場合の補足**: `team_id IS NOT NULL` の参加レコードでも `applied_by` は GDPR 削除時に SET NULL となる。チーム管理者の監査ニーズには `recruitment_participant_history` の `changed_by` を参照（こちらも同様に SET NULL）

### 14.10 NO_SHOW 記録・ペナルティの改竄防止
- `recruitment_no_show_records` / `recruitment_user_penalties` / `recruitment_cancellation_records` への **直接 UPDATE は禁止**（PUT/PATCH API を提供しない）
- 例外操作のみ:
  - NO_SHOW: 異議申立フローのみ更新可（7日以内、`disputed` フィールドのみ）
  - ペナルティ: `lift` API のみ（`lifted_at` / `lifted_by` / `lift_reason` のみ更新）
- 全ての変更は `audit_logs` に記録（`entity_type`, `action`, `before_state`, `after_state`）
- DB レベルの整合性制約: `lifted_at IS NULL OR lifted_at >= started_at` を Service 層で強制

### 14.11 キャンセル料の改竄防止
- クライアントから送信される `fee_amount_at_request` は **参考値のみ**。サーバ側で必ず再計算
- 計算結果の乖離 → 409 + 最新計算結果を返す（クライアント表示の更新を強制）
- 決済システムへの引き渡し時、サーバ計算値のみを使用
- `recruitment_cancellation_records.fee_amount` は INSERT 後の UPDATE を禁止（`payment_status` 関連カラムのみ更新可）

### 14.12 ペナルティ判定の DoS 対策
- 申込時のペナルティチェッククエリは、`idx_rup_user_active` インデックスで O(log n)
- ペナルティ件数の上限: 1ユーザーあたり同時に最大10件のアクティブペナルティ（過剰な発動を防止）
- ペナルティ自動発動のレート制限: 1ユーザーあたり1日1回まで（重複発動防止）

### 14.13 自動 NO_SHOW 検出の安全策
- デフォルト OFF（管理者が明示的に有効化）
- 開催終了から **24時間** の猶予（管理者がチェックする時間を確保）
- 自動検出後も「管理者確認画面」を経由（即座にペナルティ発動しない）
- 異議申立の機会を必ず提供（72時間以内に通知到達 + 7日以内に申立可能）

## 17. 用語集

| 用語 | 説明 |
|---|---|
| **募集枠 (Listing)** | 主催者が作成する1つの募集レコード |
| **参加者 (Participant)** | 募集枠に申込んだ個人 or チーム |
| **参加形式 (Participation Type)** | `INDIVIDUAL`（個人） or `TEAM`（チーム） |
| **スコープ (Scope)** | `TEAM` or `ORGANIZATION`。募集の主体 |
| **公開範囲 (Visibility)** | `PUBLIC` / `SCOPE_ONLY` / `SUPPORTERS_ONLY` |
| **配信スコープ (Distribution Target)** | 新着募集の通知配信先 4種類 |
| **キャンセル待ち (Waitlist)** | 定員到達後の追加申込。空き発生時に自動昇格 |
| **自動昇格 (Auto Promote)** | キャンセル発生時の待機者の自動 CONFIRMED 化 |
| **自動キャンセル (Auto Cancel)** | 最小定員未達時の自動キャンセル |
| **確認済みボタン** | 自動キャンセル等の重大通知を本人が確認したことを記録するUI（F04.9 提供） |
| **テンプレート (Template)** | 同じ募集を再利用するための保存済み雛形（スナップショット方式） |
| **サブカテゴリ (Subcategory)** | 固定大カテゴリの下にチーム/組織が任意で追加する細分類 |
| **マイダッシュボード** | 個人ユーザーのトップページ。本書では「個人ダッシュボード」の表記に統一 |
| **キャンセルポリシー (Cancellation Policy)** | 募集ごとに設定可能なキャンセル料の段階的な定義（無料境界＋最大4段階） |
| **無料境界 (Free Until)** | キャンセルが完全無料となる開催何時間前か（`free_until_hours_before`） |
| **段階 (Tier)** | キャンセル料の段階。1ポリシーあたり最大4段階。`PERCENTAGE` or `FIXED` で料金指定 |
| **NO_SHOW（無断キャンセル）** | 申込後にキャンセルもせず開催当日に欠席すること。管理者マーク or 自動検出 |
| **ペナルティ (Penalty)** | NO_SHOW を一定回数累積したユーザーに対する一定期間の申込制限 |
| **発動条件 (Threshold)** | ペナルティ発動の条件（期間内回数）。例: 半年で3回 |
| **適用範囲 (Apply Scope)** | ペナルティが効く範囲。`THIS_SCOPE_ONLY`（チーム内のみ） or `ALL_SCOPES`（全Mannschaft） |
| **異議申立 (Dispute)** | NO_SHOW 記録に対するユーザーからの取り消し要求（7日以内） |
| **手動解除 / 自動解除** | ペナルティの解除方法。手動=管理者、自動=`expires_at` 到達時のバッチ |

## 17.5 マスター御回答（v0.3.2 で確定）

v0.3.1 で挙げた10項目について、マスターの御回答により以下のとおり確定。Phase 5 着手前に追加の確認は不要。

| # | 論点 | 確定方針 |
|---|---|---|
| 1 | **ALL_SCOPES ペナルティの妥当性** | 設計上は対応可能。チームが `apply_scope` を選択。デフォルトは `THIS_SCOPE_ONLY` |
| 2 | **NO_SHOW 自動検出のデフォルト** | OFF（チーム単位で ON 設定可） |
| 3 | **異議申立期限** | 30日（`recruitment_penalty_settings.dispute_allowed_days` で設定変更可、デフォルト30） |
| 4 | **未払いキャンセル料の申込ブロック範囲** | 全スコープ（売上回収優先） |
| 5 | **ペナルティ・NO_SHOW 履歴の保持期間** | 永続保持（GDPR削除要求時は SET NULL で対応）。法務最終確認は別途 |
| 6 | **手動解除権限** | **🆕 該当チーム/組織の ADMIN + `MANAGE_RECRUITMENT_PENALTIES` 権限保有 DEPUTY_ADMIN + SYSTEM_ADMIN**（マスター御指示で変更） |
| 7 | **チーム参加時の NO_SHOW 単位** | 個人単位（メンバーごとに記録） |
| 8 | **キャンセル料の決済タイミング** | 既存決済システム（F03.4 と同じ pay-link）に従う。即時決済が原則 |
| 9 | **ペナルティ事前警告** | 有（あと1回で発動状態のとき `RECRUITMENT_PENALTY_WARNING` 通知配信） |
| 10 | **`participant_count_cache` トリガー or Service** | Service層管理（v0.2.1 で確定済み） |

**ペナルティ事前警告の追加実装**:
- NO_SHOW 確定（`confirmed=TRUE`）時に `noShowCount = thresholdCount - 1` の場合、`RECRUITMENT_PENALTY_WARNING` を該当ユーザーに配信
- 通知種別 (§10) に追加: `RECRUITMENT_PENALTY_WARNING` / 優先度 HIGH / アプリ内通知 + プッシュ
- 文言例: 「あと1回の無断キャンセルで30日間の予約制限が発動します」
