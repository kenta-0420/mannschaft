> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 8
> **最終更新**: 2026-03-21
> **モジュール種別**: 選択式モジュール #14（大会・リーグ管理）
> **関連ドキュメント**: F08.1 マッチング・対外交流（対戦相手募集との連携）、F06.4 活動記録（試合結果の活動記録自動作成）、F07.2 パフォーマンス管理（個人成績の統計連携）、F03.1 スケジュール・出欠管理（試合日程のカレンダー連携）、F04.3 プッシュ通知（結果通知・昇降格通知）

## 1. 概要

組織内のリーグ戦・トーナメント・大会を管理する機能。ディビジョン（部・リーグ）を自由に作成し、参加チームの対戦成績を記録、順位表・個人ランキングを自動計算する。シーズン終了後の昇格・降格もADMINが操作可能。

競技ごとの勝点配分・タイブレークルール・記録項目はテンプレートで定義し、SYSTEM_ADMINが主要競技のプリセットを提供しつつ、ADMINが複製・カスタマイズまたはゼロから自作できる。

**主要ユースケース**:
- 都道府県サッカー協会: 1部〜4部リーグの運営、勝点順位表、得点ランキング、シーズン後の昇降格
- バスケットボールリーグ: 勝率ベースの順位、個人スタッツ（得点・リバウンド・アシスト）
- 地域バレーボール協会: セット率・得点率ベースの順位
- 将棋・囲碁クラブ: 勝敗のみのリーグ戦、段位別グループ
- eスポーツコミュニティ: トーナメント形式の大会運営
- 社内フットサル大会: グループステージ + 決勝トーナメント

### 本ドキュメントのスコープ

| 機能 | 説明 |
|------|------|
| テンプレート管理 | 競技別の勝点ルール・タイブレーク・個人成績項目の定義 |
| 大会・リーグ CRUD | 大会の作成・編集・開催・終了 |
| ディビジョン管理 | 部/リーグの階層作成・参加チーム登録 |
| 対戦カード・スコア記録 | 節(matchday)ごとの対戦カード作成・結果入力 |
| 順位表自動計算 | 勝点・タイブレークルールに基づく順位の自動算出 |
| 個人ランキング | 得点王・アシスト王等の個人成績集計 |
| 昇格・降格管理 | シーズン終了後のチーム入れ替え操作 |
| PDF出力 | 順位表・トーナメント表のPDFエクスポート |

**スコープ外**:
- 対戦相手の募集・応募 → F08.1 マッチング
- 個人パフォーマンスの長期追跡・目標管理 → F07.2 パフォーマンス管理
- 試合の詳細な活動記録（練習メニュー等） → F06.4 活動記録

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | プリセットテンプレートの管理。全組織の大会データ参照 |
| ADMIN | 大会・リーグの CRUD、テンプレート管理、対戦カード作成・結果入力、昇降格操作、PDF出力 |
| DEPUTY_ADMIN | `MANAGE_TOURNAMENT` 権限を持つ場合: 対戦カード作成・結果入力、順位表・ランキング閲覧（※ `MANAGE_TOURNAMENT` は F01.2 の `permissions` テーブルに Flyway で追加が必要） |
| MEMBER | 順位表・ランキング・対戦結果の閲覧。自チームの試合詳細閲覧 |
| SUPPORTER | 公開設定の大会の順位表・結果閲覧のみ |
| GUEST | 公開設定の大会の順位表・結果閲覧のみ（SSR ページ） |

### 対象レベル
- [x] 組織 (Organization) — 大会・リーグの主催。ディビジョン・参加チーム管理
- [x] チーム (Team) — 参加チームとして大会に登録される側。自チームの成績閲覧
- [ ] 個人 (Personal) — 対象外（個人戦は将来検討）

> **設計判断**: 大会は「組織が主催し、組織傘下のチームが参加する」モデル。大分県サッカー協会（組織）が1部〜4部リーグ（ディビジョン）を作成し、各チーム（チーム）が所属する構造。組織に属さないチーム同士の大会は、先に組織（協会・連盟等）を作成してからチームを招待する運用。

### テーブル定義

#### `system_tournament_presets`

SYSTEM_ADMIN が管理するプラットフォーム標準の競技別テンプレート。ADMINが `tournament_templates` に複製して利用。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `name` | VARCHAR(100) | NO | — | プリセット名（例: サッカー、バスケットボール、バレーボール） |
| `sport_category` | VARCHAR(50) | NO | — | 競技カテゴリ（例: FOOTBALL, BASKETBALL, VOLLEYBALL, GENERAL）。検索・分類用 |
| `description` | VARCHAR(500) | YES | NULL | 説明 |
| `icon` | VARCHAR(30) | YES | NULL | アイコン（emoji or アイコン名） |
| `supported_formats` | JSON | NO | — | 対応する大会形式の配列（例: `["LEAGUE","KNOCKOUT","GROUP_KNOCKOUT"]`） |
| `win_points` | TINYINT UNSIGNED | NO | 3 | 勝利時の勝点 |
| `draw_points` | TINYINT UNSIGNED | NO | 1 | 引分時の勝点（トーナメント形式では未使用） |
| `loss_points` | TINYINT UNSIGNED | NO | 0 | 敗北時の勝点 |
| `has_draw` | BOOLEAN | NO | TRUE | 引分けありか（バスケ等は FALSE → 延長で決着） |
| `has_sets` | BOOLEAN | NO | FALSE | セット制か（バレー、テニス等） |
| `sets_to_win` | TINYINT UNSIGNED | YES | NULL | 勝利に必要なセット数（`has_sets = TRUE` 時。例: バレー3、テニス2 or 3） |
| `has_extra_time` | BOOLEAN | NO | FALSE | 延長戦ありか |
| `has_penalties` | BOOLEAN | NO | FALSE | PK戦/SO戦ありか |
| `score_unit_label` | VARCHAR(20) | NO | '点' | スコアの単位ラベル（点、ゴール、ポイント、セット等） |
| `bonus_point_rules` | JSON | YES | NULL | 条件付き勝点加算ルール（NULL = 固定勝点のみ） |
| `sort_order` | INT | NO | 0 | 表示順 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
INDEX idx_stp_category (sport_category, sort_order)   -- カテゴリ別一覧取得
```

> **プリセット名の一意性**: MySQL 8.0 では `UNIQUE (name, deleted_at)` は `deleted_at = NULL` 同士を区別するため、論理削除と併用できない。Service 層で `SELECT ... WHERE name = ? AND deleted_at IS NULL` の存在チェックにより一意性を保証する。

#### `system_tournament_preset_tiebreakers`

プリセットのタイブレーク（順位決定）ルール。優先順位順に適用。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `preset_id` | BIGINT UNSIGNED | NO | — | FK → system_tournament_presets。ON DELETE CASCADE |
| `priority` | TINYINT UNSIGNED | NO | — | 優先順位（1が最優先） |
| `criteria` | ENUM('POINTS', 'SCORE_DIFFERENCE', 'SCORE_FOR', 'HEAD_TO_HEAD_POINTS', 'HEAD_TO_HEAD_SCORE_DIFFERENCE', 'WINS', 'SET_RATIO', 'POINT_RATIO', 'LOSSES', 'DRAWS') | NO | — | 順位決定基準 |
| `direction` | ENUM('DESC', 'ASC') | NO | 'DESC' | ソート方向（DESC = 多い方が上位） |

**インデックス**
```sql
UNIQUE INDEX uq_stp_tb (preset_id, priority)           -- 同一プリセット内で優先順位ユニーク
```

**制約・備考**
- サッカーの典型例: 1. POINTS DESC → 2. SCORE_DIFFERENCE DESC → 3. SCORE_FOR DESC → 4. HEAD_TO_HEAD_POINTS DESC
- バスケの典型例: 1. WINS DESC → 2. HEAD_TO_HEAD_POINTS DESC
- ENUM値 `SET_RATIO` `POINT_RATIO` はバレー等セット制競技用

**制約・備考**
- `start_date <= end_date` は Service 層でバリデーション
- `format = KNOCKOUT` の場合、`draw_points` は未使用（延長/PK戦で決着）
- `status` 遷移: DRAFT → OPEN → IN_PROGRESS → COMPLETED → ARCHIVED / CANCELLED。`ARCHIVED` は一覧のデフォルト表示から除外（フィルタで表示可能）
- ルール値（`win_points` 等）はテンプレートから初期値をコピーするが、大会単位で上書き可能
- `bonus_point_rules` JSON スキーマ:
  - **キー形式**: `{条件}` — 条件名は英大文字+アンダースコア。Service層で対応するEvaluatorを呼び出す
  - **値**: 整数。正 = 加算、負 = 減算。セットスコア系は基本勝点を**上書き**（`win_points` の代わりに適用）
  - **セットスコア系条件**（`has_sets = true` 時に評価）:
    - `3_SET_WIN` / `4_SET_WIN` / `5_SET_WIN`: 勝利時のセット数に応じた勝点（上書き）
    - `3_SET_LOSS` / `4_SET_LOSS` / `5_SET_LOSS`: 敗北時のセット数に応じた勝点（上書き）
  - **ボーナス加算系条件**（基本勝点に加算）:
    - `4_TRIES_BONUS`: 4トライ以上のボーナス（ラグビー）。トライ数は `tournament_match_player_stats` の `stat_key = 'tries'` を集計
    - `LOSS_WITHIN_7`: 7点差以内の敗北ボーナス（ラグビー）。`score_difference` で判定
  - 例（バレー）: `{"3_SET_WIN": 3, "4_SET_WIN": 3, "5_SET_WIN": 2, "5_SET_LOSS": 1}`
  - 例（ラグビー）: `{"4_TRIES_BONUS": 1, "LOSS_WITHIN_7": 1}`
  - ADMINがカスタムテンプレートで独自条件キーを定義した場合、Service層に対応Evaluatorがなければ無視（ログ出力のみ）

## 5. 主要ビジネスロジック

### 5.1 対戦カード自動生成

#### リーグ戦（LEAGUE）
- **総当たり方式**: `league_round_type` に応じてカード数を決定
  - `SINGLE`: N*(N-1)/2 カード（片道総当たり）
  - `DOUBLE`: N*(N-1) カード（ホーム&アウェイ2回戦）
- 節への振り分けはラウンドロビンアルゴリズム（参加チーム数が奇数の場合は BYE を自動追加）

#### トーナメント（KNOCKOUT）
- **シード順考慮**: `seed` 値に基づいてトーナメントブラケットを自動生成
- 参加チーム数が 2^n でない場合、上位シードに1回戦 BYE を割り当て
- `next_match_id` + `next_match_slot` でツリー構造を構築

#### グループステージ + 決勝（GROUP_KNOCKOUT）
- グループステージ: 各グループ内で総当たり
- 決勝トーナメント: グループステージ完了後に ADMIN が手動で決勝トーナメントのディビジョンにチームを配置、またはグループ順位から自動配置

### 5.2 順位表計算

1. 該当ディビジョンの全 `COMPLETED` 試合を集計
2. 各チームの wins/draws/losses/score_for/score_against を算出（没収試合 `FORFEIT_*` は勝敗に含め、スコアは大会規定に従い ADMIN が入力）
3. 基本勝点 = wins * `win_points` + draws * `draw_points` + losses * `loss_points`
4. `bonus_point_rules` が非NULLの場合、各試合のセットスコア・点差等の条件に基づきボーナス/ペナルティポイントを加算。例:
   - バレー 3-2勝利 → `5_SET_WIN` = 2点（通常の3点ではなく2点）、3-2敗北 → `5_SET_LOSS` = 1点
   - ラグビー 4トライ以上 → `4_TRIES_BONUS` = +1点、7点差以内の負け → `LOSS_WITHIN_7` = +1点
5. 大会の `tournament_tiebreakers` の優先順位に従ってソート
6. `HEAD_TO_HEAD_*` 系タイブレーク: 同勝点チーム間の直接対決成績のみで小グループ内順位を再計算
7. `tournament_standings` テーブルに UPSERT
8. `promotion_zone` を `promotion_slots` / `relegation_slots` から判定

**トリガー**: 試合スコアの INSERT/UPDATE 時に `StandingsRecalculationEvent` を発火 → 非同期で再計算。一括スコア入力（`/scores/batch`）時はイベントを1回だけ発火（N回ではなく）。

**冪等性**: 再計算ロジックは毎回全 `COMPLETED` 試合からゼロ計算して UPSERT する冪等方式。イベント二重発火時も結果が同一になることを保証。手動再計算 API（`POST /.../{divId}/standings/recalculate`）も同一ロジックを使用し、ADMINのトラブルリカバリ手段として提供。

**Valkey キャッシュ戦略**:
- `mannschaft:cache:standings:{divisionId}` — TTL 1時間。スコア更新・再計算完了時に即時 invalidate
- `mannschaft:cache:rankings:{tournamentId}:{statKey}` — TTL 1時間。個人成績更新時に即時 invalidate
- `mannschaft:cache:matrix:{divisionId}` — TTL 1時間。スコア更新時に即時 invalidate
- 公開API・埋め込みAPIはキャッシュヒット時にDBクエリを発行しない

**監査ログ**: スコア更新時は `audit_logs`（Phase 10, F10.3）に `TOURNAMENT_MATCH_SCORE_UPDATED` イベントを記録（変更前後の値を保持）。`status = COMPLETED` の試合スコアを修正する場合は以下の制約を適用:
- ADMIN のみ実行可（DEPUTY_ADMIN は不可）
- `notes`（修正理由）の入力を必須化
- 順位表・個人ランキングの全再計算をトリガー

### 5.3 個人ランキング計算

1. 大会の全 `COMPLETED` 試合から `tournament_match_player_stats` を集計
2. 大会の `tournament_stat_defs` で `is_ranking_target = TRUE` の項目ごとに集計
3. `aggregation_type`（SUM/AVG/MAX/MIN）に従って集計値を算出
4. `tournament_individual_rankings` テーブルに UPSERT

**トリガー**: 個人成績の INSERT/UPDATE 時に `RankingsRecalculationEvent` を発火

### 5.4 昇降格処理

1. ADMIN が `/promotions/preview` で昇降格候補を確認
   - 各ディビジョンの `promotion_slots` / `relegation_slots` に基づき自動判定
   - 入替戦枠（`playoff_promotion_slots`）がある場合はその旨も表示
2. ADMIN がプレビュー内容を確認・調整して `/promotions` で確定実行
3. `tournament_promotion_records` に記録
4. 次シーズンの大会作成時、前シーズンの昇降格結果を参照して自動でディビジョン振り分けを提案

### 5.5 シーズン継続作成

`POST /organizations/{orgId}/tournaments/continue/{previousTournamentId}` で前シーズンから新シーズンを作成する処理:

1. 前シーズン大会の `status = COMPLETED` or `ARCHIVED` を検証
2. ディビジョン構成（名前・レベル・昇降格枠）をコピー
3. ルール値（`win_points` 等）・タイブレーク・成績項目をコピー
4. `tournament_promotion_records` を参照し、昇降格を反映したチーム配置を**提案**（DRAFT状態で返却）
5. ADMINが提案内容を確認・微調整して確定（新規チーム追加・脱退チーム除外等）
6. `previous_tournament_id` に前シーズンの大会IDを設定（シーズン系譜の追跡）

### 5.6 PDF出力

- **順位表PDF**: ディビジョン名、順位・チーム名・試合数・勝敗・得失点・勝点のテーブル。昇格/降格ゾーンを色分け
- **トーナメント表PDF**: ブラケット図。スコア付き。未決定のカードは空欄
- **個人ランキングPDF**: 順位・選手名・所属チーム・集計値のテーブル
- **対戦マトリクスPDF**: 縦軸×横軸チーム名、交差点にスコアの表形式

生成方式は既存の PDF 基盤（OpenPDF / Flying Saucer）を使用。日本語フォント対応済み。

> **パフォーマンス方針**: Phase 8 初期リリースでは同期 GET で提供。大半のユースケース（10〜20チーム）では問題ないと想定。大規模大会（40チーム以上）でタイムアウトが発生した場合、既存のデータエクスポート基盤（`POST /export` → 202 + `jobId` → `GET /export/status/{jobId}` → `GET /export/download/{jobId}`）への非同期化を検討。

### 5.7 WebSocket リアルタイム更新

既存の WebSocket 基盤（STOMP over SockJS）を利用し、順位表・トーナメント表のリアルタイム更新を提供。

| トピック | 発火タイミング | ペイロード |
|---------|--------------|-----------|
| `/topic/tournaments/{tId}/standings/{divId}` | 順位表再計算完了時 | 更新後の順位表全体（`StandingsResponse`） |
| `/topic/tournaments/{tId}/bracket` | トーナメント試合結果確定時 | 更新された試合カード（`MatchResponse`） |
| `/topic/tournaments/{tId}/rankings/{statKey}` | 個人ランキング再計算完了時 | 上位N件のランキング（`RankingResponse`） |

DB・テーブル追加なし。`StandingsRecalculationEvent` / `RankingsRecalculationEvent` のリスナーに WebSocket 送信処理を追加するだけ。

### 5.9 ディビジョン別ターゲティング（他機能との横断連携）

ディビジョン所属（`tournament_participants` のチーム×ディビジョン紐付け）を、他機能の**配信ターゲット**として横断的に利用する。

**ユースケース例**:
- 「1部リーグ所属チームのみに審判派遣の案内を掲示板に投稿したい」
- 「2部リーグのチームだけに昇格プレーオフの日程変更をプッシュ通知したい」
- 「3部・4部の全チームにスポンサー募集のダイレクトメールを送りたい」

**実装方針**:
- 既存の通知・掲示板・DM等の配信対象選択UIに「大会ディビジョンで絞り込み」フィルタを追加
- ADMINが配信時に「大会 → ディビジョン」を選択すると、`tournament_participants` から対象チームを解決 → チームのメンバー一覧を取得 → 配信対象に設定
- API側は汎用のターゲティングパラメータとして `target_type = 'TOURNAMENT_DIVISION'` + `target_id = {divisionId}` を受け付ける

| 連携先 | ターゲティング方式 |
|--------|------------------|
| F04.3 プッシュ通知 | `notification_targets` に `target_type = 'TOURNAMENT_DIVISION'` を追加。該当ディビジョンの全参加チームメンバーに配信 |
| F05.1 掲示板 | 投稿の `visibility_scope` に `TOURNAMENT_DIVISION` を追加。該当ディビジョン所属チームのメンバーのみ閲覧可 |
| F09.6 ダイレクトメール | 配信対象フィルタに「大会ディビジョン」を追加。宛先をディビジョン単位で一括選択 |
| F04.1 タイムライン | フィード配信のスコープに `TOURNAMENT_DIVISION` を追加。ディビジョン限定の投稿が可能 |

> **DB変更**: 既存の各連携先テーブルの `target_type` ENUM に `TOURNAMENT_DIVISION` 値を追加するだけ。新テーブル不要。`tournament_participants` がチーム解決の源泉となる。`status = ACTIVE` のチームのみ配信対象（WITHDRAWN / DISQUALIFIED は除外）。
