## 1. 概要

組織・チーム内でカスタム設問のアンケートを作成し、メンバーから回答を収集・集計する機能。単一選択・複数選択・自由記述・評価スケール等の設問タイプをサポートし、匿名/記名を切り替え可能。回答期限を設定してリマインド通知を送信し、締切後は集計結果をグラフ（円グラフ・棒グラフ）で可視化する。

**他機能との棲み分け**:
| 機能 | 用途 | Phase |
|------|------|-------|
| **F05.4 アンケート・投票（本機能）** | 独立した汎用アンケート。カスタム設問・集計グラフ・匿名回答 | Phase 5 |
| F04.1 タイムライン投票 | 投稿内の軽量投票（2〜4択の簡易 YES/NO 型） | Phase 4 |
| F03.1 イベントアンケート | スケジュール出欠確認時の追加質問（出欠に付随） | Phase 3 |
| 議決権行使・委任状 | 総会議案への正式投票・委任状管理 | Phase 8 |

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `surveys` | アンケートマスター | あり（`deleted_at`） |
| `survey_questions` | 設問定義（順序付き） | なし |
| `survey_options` | 選択肢定義（選択式設問用） | なし |
| `survey_responses` | 回答レコード（1ユーザー1設問1レコード） | なし |
| `survey_targets` | 対象者指定配信の配信先（TARGETED モード時） | なし |
| `survey_result_viewers` | 結果閲覧権限を持つメンバー（作成者が指定） | なし |

### テーブル定義

#### `surveys`

アンケートのマスター情報。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | FK → teams.id or organizations.id（scope_type に応じた参照先） |
| `title` | VARCHAR(200) | NO | — | アンケートタイトル |
| `description` | TEXT | YES | NULL | 説明文 |
| `status` | VARCHAR(20) | NO | 'DRAFT' | ステータス（`DRAFT` / `PUBLISHED` / `CLOSED`） |
| `is_anonymous` | BOOLEAN | NO | FALSE | 匿名回答モード。TRUE = 回答者を非公開（ADMIN にも個別回答者は見えない） |
| `allow_multiple_submissions` | BOOLEAN | NO | FALSE | 回答修正の許可。TRUE = 締切前なら何度でも回答を上書き可能 |
| `results_visibility` | VARCHAR(20) | NO | 'AFTER_CLOSE' | 結果公開タイミング（`ALWAYS` / `AFTER_RESPONSE` / `AFTER_CLOSE` / `ADMIN_ONLY`） |
| `distribution_mode` | VARCHAR(20) | NO | 'ALL' | 配信モード（`ALL` = スコープ全員 / `TARGETED` = 対象者指定） |
| `auto_post_to_timeline` | BOOLEAN | NO | TRUE | 公開時・締切時にタイムラインへ自動投稿するか |
| `series_id` | VARCHAR(50) | YES | NULL | シリーズ識別子（定期実施アンケートの紐付け用。例: `satisfaction_2026_q1`）。同一 `series_id` のアンケートを時系列比較可能 |
| `remind_before_hours` | JSON | YES | NULL | リマインドスケジュール（配列。例: `[72, 24]` = 3日前・1日前）。NULL の場合はデフォルト `[24]` |
| `manual_remind_count` | INT UNSIGNED | NO | 0 | 手動リマインド送信回数（最大3回まで） |
| `starts_at` | DATETIME | YES | NULL | 回答受付開始日時（NULL = 公開と同時に受付開始） |
| `expires_at` | DATETIME | YES | NULL | 回答期限（NULL = 無期限。期限到達で自動 CLOSED） |
| `response_count` | INT UNSIGNED | NO | 0 | 回答者数（denormalize。ユニークユーザー数） |
| `target_count` | INT UNSIGNED | YES | NULL | 対象者数（公開時のスコープメンバー数をスナップショット。回答率算出用） |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `published_at` | DATETIME | YES | NULL | 公開日時 |
| `closed_at` | DATETIME | YES | NULL | 締切日時（手動 or 自動） |
| `version` | BIGINT | NO | 0 | 楽観的ロックバージョン（@Version） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_sv_scope (scope_type, scope_id, status, created_at DESC)  -- スコープ別アンケート一覧
INDEX idx_sv_status (status, expires_at)                             -- 自動締切バッチ用
INDEX idx_sv_created_by (created_by, created_at DESC)                -- 作成者別一覧
```

**制約・備考**
- `status` のライフサイクル: `DRAFT` → `PUBLISHED` → `CLOSED`
  - `DRAFT`: 下書き。設問の追加・変更が可能。回答不可
  - `PUBLISHED`: 公開中。回答受付中。**設問の追加・変更・削除は不可**（回答の整合性を保つため）
  - `CLOSED`: 締切済み。新規回答不可。結果閲覧のみ
- `is_anonymous = TRUE` の場合: `survey_responses.user_id` は記録するが、**API レスポンスでは個別の回答者情報を一切返さない**（ADMIN にも非公開）。集計結果のみ参照可能。DB 上は `user_id` を保持する理由: 重複回答防止 + 監査対応（不正調査時に SYSTEM_ADMIN がDBレベルで確認可能）
- `allow_multiple_submissions = TRUE`: ユーザーが回答を修正できる。既存回答を DELETE → 再 INSERT する方式（更新ではなく置換）。`response_count` は変わらない（ユニークユーザー数のため）
- `results_visibility`:
  - `ALWAYS`: 回答前でも結果を閲覧可能（リアルタイム集計）
  - `AFTER_RESPONSE`: 自分が回答した後に結果を閲覧可能
  - `AFTER_CLOSE`: 締切後のみ結果を閲覧可能
  - `ADMIN_ONLY`: ADMIN のみ結果を閲覧可能（メンバーには非公開）
- `target_count`: 公開時点のスコープメンバー数をスナップショットする。「回答率 = response_count / target_count × 100」の算出に使用。後からメンバーが増減しても変わらない
- `scope_type + scope_id`: ポリモーフィック FK。Service 層で `scope_type = 'TEAM'` なら `teams` テーブル、`scope_type = 'ORGANIZATION'` なら `organizations` テーブルの存在を検証
- `PUBLISHED` → `DRAFT` への戻しは不可（回答が0件でも不可。誤公開の場合は削除して再作成）

#### `survey_questions`

アンケートの設問定義。1アンケートに複数の設問を持つ。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `survey_id` | BIGINT UNSIGNED | NO | — | FK → surveys。ON DELETE CASCADE |
| `question_type` | VARCHAR(20) | NO | — | 設問タイプ（`SINGLE_CHOICE` / `MULTIPLE_CHOICE` / `FREE_TEXT` / `RATING_SCALE`） |
| `question_text` | VARCHAR(500) | NO | — | 設問文 |
| `is_required` | BOOLEAN | NO | TRUE | 必須回答か |
| `display_order` | INT UNSIGNED | NO | — | 表示順（1始まり） |
| `max_selections` | INT UNSIGNED | YES | NULL | `MULTIPLE_CHOICE` の場合の最大選択数（NULL = 無制限） |
| `scale_min` | INT | YES | NULL | `RATING_SCALE` の最小値（例: 1） |
| `scale_max` | INT | YES | NULL | `RATING_SCALE` の最大値（例: 5） |
| `scale_min_label` | VARCHAR(50) | YES | NULL | 最小値のラベル（例: 「非常に不満」） |
| `scale_max_label` | VARCHAR(50) | YES | NULL | 最大値のラベル（例: 「非常に満足」） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_sq_survey (survey_id, display_order)  -- アンケート別設問一覧（表示順）
```

**制約・備考**
- 1アンケートあたり最大30設問（Service 層バリデーション）
- `question_type` ごとの追加フィールド使用ルール:
  - `SINGLE_CHOICE`: `survey_options` に選択肢を定義（2〜20個）。`max_selections` 不使用
  - `MULTIPLE_CHOICE`: `survey_options` に選択肢を定義（2〜20個）。`max_selections` で上限指定可能
  - `FREE_TEXT`: `survey_options` 不要。回答は `survey_responses.answer_text` に格納
  - `RATING_SCALE`: `survey_options` 不要。`scale_min` / `scale_max` を使用。回答は `survey_responses.answer_number` に格納
- アンケートが `PUBLISHED` 後は設問の追加・変更・削除・並び替え不可
- `display_order`: Service 層で survey_id 内の連番を保証

#### `survey_options`

選択式設問の選択肢。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `question_id` | BIGINT UNSIGNED | NO | — | FK → survey_questions。ON DELETE CASCADE |
| `option_text` | VARCHAR(200) | NO | — | 選択肢テキスト |
| `display_order` | INT UNSIGNED | NO | — | 表示順（1始まり） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_so_question (question_id, display_order)  -- 設問別選択肢一覧（表示順）
```

**制約・備考**
- 1設問あたり2〜20個の選択肢（Service 層バリデーション）
- `FREE_TEXT` / `RATING_SCALE` タイプの設問には選択肢不要（0件）
- アンケートが `PUBLISHED` 後は選択肢の追加・変更・削除不可

#### `survey_responses`

ユーザーの回答レコード。1ユーザー × 1設問 = 1レコード（選択式は選択肢ごとに1レコード）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `survey_id` | BIGINT UNSIGNED | NO | — | FK → surveys。ON DELETE CASCADE（denormalize: 検索高速化） |
| `question_id` | BIGINT UNSIGNED | NO | — | FK → survey_questions。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT（回答者削除時に response_count との不整合を防ぐ。ユーザー退会時は先にアンケート回答を処理する運用） |
| `option_id` | BIGINT UNSIGNED | YES | NULL | FK → survey_options。ON DELETE CASCADE。選択式の場合に設定 |
| `answer_text` | TEXT | YES | NULL | 自由記述の回答テキスト（`FREE_TEXT` 時に使用） |
| `answer_number` | INT | YES | NULL | 数値回答（`RATING_SCALE` 時に使用） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_sr_user_question_option (survey_id, question_id, user_id, option_id)  -- 同一ユーザーの重複回答防止
INDEX idx_sr_survey_user (survey_id, user_id)                                        -- ユーザー別回答確認
INDEX idx_sr_question_option (question_id, option_id)                                -- 設問別集計
INDEX idx_sr_survey_question (survey_id, question_id, created_at DESC)               -- 設問別回答一覧
```

**制約・備考**
- **選択式の回答方式**:
  - `SINGLE_CHOICE`: 1設問1ユーザーにつき `option_id` が異なる1レコードのみ。`answer_text` = NULL, `answer_number` = NULL
  - `MULTIPLE_CHOICE`: 1設問1ユーザーにつき選択した `option_id` ごとに1レコード。`max_selections` を超える場合は 400 エラー
  - `FREE_TEXT`: `option_id` = NULL。`answer_text` に回答テキスト（最大2,000文字）
  - `RATING_SCALE`: `option_id` = NULL。`answer_number` にスケール値（`scale_min` 〜 `scale_max` の範囲内。範囲外は 400 エラー）
- `uq_sr_user_question_option`: `option_id` が NULL の場合（FREE_TEXT / RATING_SCALE）は `(survey_id, question_id, user_id, NULL)` で1レコードのみ（MySQL では NULL 同士は UNIQUE をすり抜けるため、Service 層で重複チェック）
- `allow_multiple_submissions = TRUE` の場合: 既存回答を全 DELETE → 新回答を INSERT（アトミック）。回答修正のたびにレコードが差し替わる
- 匿名アンケート（`is_anonymous = TRUE`）: `user_id` はDB に記録するが、API レスポンスの回答一覧では `user_id` / `user_name` を返さない
- **`response_count` のアトミック更新**: 初回回答時は `UPDATE surveys SET response_count = response_count + 1 WHERE id = :surveyId AND deleted_at IS NULL` の1 SQL で実行。回答修正時（DELETE → INSERT）は `response_count` を変更しない。ユーザー退会等で回答が削除された場合は `response_count` の再集計バッチ（`SELECT COUNT(DISTINCT user_id) FROM survey_responses WHERE survey_id = :id`）で補正

#### `survey_targets`

対象者指定配信（`distribution_mode = 'TARGETED'`）の配信先ユーザー。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `survey_id` | BIGINT UNSIGNED | NO | — | FK → surveys。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `notified_at` | DATETIME | YES | NULL | 通知送信日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_st_survey_user (survey_id, user_id)  -- 同一ユーザーへの重複指定防止
INDEX idx_st_user (user_id, survey_id)               -- ユーザー別の対象アンケート検索
```

**制約・備考**
- `distribution_mode = 'ALL'` の場合: `survey_targets` レコードなし（スコープ全員が対象）
- `distribution_mode = 'TARGETED'` の場合: 公開前に対象者を設定（1〜500名）
- 対象者は `scope_type + scope_id` のスコープに所属するメンバーであること（Service 層バリデーション）
- 公開時の `target_count` は `survey_targets` のレコード数をスナップショット
- `TARGETED` モードでは、`GET /surveys` の一覧に対象者以外はアンケートが表示されない
- 通知は対象者のみに送信される（スコープ全員には送られない）

#### `survey_result_viewers`

結果閲覧権限を持つメンバー。ADMIN でなくても、作成者がここに登録したユーザーは集計結果・個別回答（非匿名時）・CSV エクスポート・回答者一覧を閲覧できる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `survey_id` | BIGINT UNSIGNED | NO | — | FK → surveys。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `can_view_individual` | BOOLEAN | NO | FALSE | 個別回答の閲覧権限。TRUE = 非匿名時に `GET /responses/{userId}` が利用可能。FALSE = 集計結果のみ閲覧可能 |
| `granted_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL（権限を付与した人） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_srv_survey_user (survey_id, user_id)  -- 同一ユーザーへの重複付与防止
INDEX idx_srv_user (user_id)                          -- ユーザー別の閲覧権限検索
```

**制約・備考**
- 作成者がアンケート作成時（`POST /surveys`）または更新時（`PUT /surveys/{id}`）に `result_viewer_ids` で指定
- PUBLISHED 後も閲覧者の追加・削除は可能（設問凍結とは独立。結果を見る人が増えても回答データに影響しない）
- `can_view_individual = TRUE` でも匿名アンケート（`is_anonymous = TRUE`）の場合は個別回答 API が 403 を返す（匿名性が最優先）
- 作成者自身は `survey_result_viewers` に登録不要（`created_by` として常にフルアクセスを持つ）
- ADMIN+ も `survey_result_viewers` に登録不要（ロールベースのアクセス権で常にフルアクセス）
- 閲覧者に登録されたユーザーには `results_visibility` の制約が適用されない（`AFTER_CLOSE` であっても PUBLISHED 中に結果を閲覧可能）
- 最大20名まで指定可能（Service 層バリデーション）

## 4. API設計

### エンドポイント一覧
| メソッド | パス | 認証 | 説明 |
|---------|-----|------|------|
| GET | `/api/v1/surveys` | MEMBER+ | アンケート一覧（スコープ別） |
| POST | `/api/v1/surveys` | MEMBER+ | アンケート作成（下書き） |
| GET | `/api/v1/surveys/{id}` | MEMBER+ | アンケート詳細（設問・選択肢含む） |

| DELETE | `/api/v1/surveys/{id}` | 作成者 / ADMIN+ | アンケート削除（論理削除） |

| POST | `/api/v1/surveys/{id}/generate-blog-draft` | ADMIN+ | 結果からブログ記事の下書きを自動生成 |
| POST | `/api/v1/surveys/{id}/duplicate` | MEMBER+ | アンケートを DRAFT としてコピー |
| PATCH | `/api/v1/surveys/{id}/extend` | 作成者 / ADMIN+ | 回答期限の延長 |
| POST | `/api/v1/surveys/{id}/remind` | 作成者 / ADMIN+ | 未回答者への手動リマインド送信 |
| GET | `/api/v1/surveys/{id}/responses/{userId}` | ADMIN+ / 作成者 / 閲覧者(個別) | 特定ユーザーの個別回答詳細（非匿名時のみ） |
| GET | `/api/v1/surveys/series/{seriesId}/comparison` | ADMIN+ | シリーズ内アンケートの時系列比較 |

※ `ADMIN+` = ADMIN または `MANAGE_SURVEYS` 権限を持つ DEPUTY_ADMIN
※ `MEMBER+` = MEMBER 以上（SUPPORTER 含む）
※ `作成者` = `surveys.created_by == currentUser.id`

#### `POST /api/v1/surveys/{id}/responses`

アンケートに回答する。全設問の回答を一括送信。

**リクエストボディ**
```json
{
  "answers": [
```

| フィールド | 型 | 必須 | バリデーション |
|-----------|---|------|-------------|
| `answers` | Array | Yes | 必須設問が全て含まれていること |
| `answers[].question_id` | Long | Yes | 当該アンケートの設問であること |
| `answers[].option_ids` | Long[] | 条件付き | `SINGLE_CHOICE`（1個）/ `MULTIPLE_CHOICE`（1個以上）の場合に必須 |
| `answers[].answer_text` | String | 条件付き | `FREE_TEXT` の場合に必須。最大2,000文字 |
| `answers[].answer_number` | Integer | 条件付き | `RATING_SCALE` の場合に必須。`scale_min` 〜 `scale_max` の範囲内 |

**レスポンス（201 Created）**
```json
{
  "data": {
    "survey_id": 1,
    "responded_at": "2026-03-15T10:30:00+09:00",
    "is_first_response": true,
    "message": "回答を受け付けました。ありがとうございます。"
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | 必須設問の回答漏れ / option_id が不正 / スケール範囲外 / `SINGLE_CHOICE` で複数選択 / `max_selections` 超過 |
| 401 | 未認証 |
| 403 | スコープ非所属 / GUEST ロール / アンケートが `PUBLISHED` 以外 / 回答受付期間外（`starts_at` 〜 `expires_at`） |
| 404 | アンケート不存在 |
| 409 | 既に回答済み（`allow_multiple_submissions = FALSE` の場合） |

**処理フロー**:
1. `allow_multiple_submissions = TRUE` かつ既回答の場合: 既存回答を全 DELETE → 新回答を INSERT
2. `allow_multiple_submissions = FALSE` かつ既回答の場合: 409 エラー
3. 初回回答の場合: `surveys.response_count` を +1 アトミック更新
4. 回答修正（再送信）の場合: `response_count` は変更しない（ユニークユーザー数のため）

#### `POST /api/v1/surveys/{id}/generate-blog-draft`

アンケート結果からブログ記事の下書きを自動生成する。テンプレート方式で集計データを Markdown に差し込み、`DRAFT` 状態のブログ記事として作成する。ADMIN+ のみ。`CLOSED` アンケート（最終結果）、または `PUBLISHED` + `results_visibility = ALWAYS`（中間結果）で実行可能。

**リクエストボディ**
```json

}
```

| フィールド | 型 | 必須 | バリデーション |
|-----------|---|------|-------------|
| `blog_scope_type` | String | No | ブログの公開先スコープ。デフォルト: アンケートと同じスコープ |
| `blog_scope_id` | Long | No | ブログの公開先スコープ ID |

```json
{

    "blog_post_id": 100,

    "message": "ブログ記事の下書きを作成しました。編集画面で内容を確認・修正してから公開してください。"

}
```

**自動生成される下書きの構成**:
1. **ヘッダー**: アンケートタイトル + 「結果報告」
2. **概要セクション**: 実施期間、回答率（response_count / target_count）
3. **設問別セクション**:
   - `SINGLE_CHOICE` / `MULTIPLE_CHOICE`: 選択肢ごとの得票数・パーセンテージ（テキストベースの横棒グラフ付き）

**生成テンプレート（サーバーサイド Thymeleaf / String.format）**:
- 集計データは `GET /surveys/{id}/results` と同じロジックで取得
- Markdown 形式で生成し、F06.1 ブログの `POST /blog-posts` を内部呼び出しして `DRAFT` 状態で保存
- 外部 AI API は使用しない（Phase 5 はテンプレート方式。将来 AI 生成は FUTURE_CONSIDERATIONS に記載）
- 同一アンケートから複数回呼び出し可能（毎回新しい DRAFT ブログ記事を生成）。1アンケートあたり最大5回まで（超過時は 400 エラー「既に5件の下書きが生成されています」）。既存の下書きを確認してから再生成する運用を促す

|-----------|------|
| 400 | アンケートが `CLOSED` でなく、かつ `PUBLISHED` + `results_visibility = ALWAYS` でもない |
| 401 | 未認証 |
| 403 | ADMIN 以外 |
| 404 | アンケート不存在 |

## 5. ビジネスロジック

### 主要フロー

#### アンケート作成→公開フロー
```
1. MEMBER+ がアンケートを作成（POST /surveys）→ DRAFT 状態
2. 設問・選択肢を調整（PUT /surveys/{id}）※ DRAFT 中のみ
3. 準備完了後に公開（PATCH /surveys/{id}/publish）
   a. status = PUBLISHED、published_at = NOW()
   b. target_count のスナップショット:
      - ALL モード: スコープのメンバー数
      - TARGETED モード: survey_targets のレコード数
   c. 通知送信:
      - ALL モード: スコープ内全メンバーに SURVEY_CREATED 通知
      - TARGETED モード: survey_targets の対象者のみに通知
   d. expires_at が設定されている場合、remind_before_hours に基づくリマインド通知をスケジュール
   e. auto_post_to_timeline = TRUE の場合: タイムラインにアンケート告知投稿を自動生成（後述）
4. メンバーが回答（POST /surveys/{id}/responses）
   a. 必須設問の回答チェック
   b. survey_responses に INSERT
   c. 初回回答時: response_count を +1 アトミック更新
5. 期限到達 or ADMIN が手動締切（PATCH /surveys/{id}/close）
   a. status = CLOSED、closed_at = NOW()
   b. results_visibility = AFTER_CLOSE の場合: SURVEY_RESULTS_AVAILABLE 通知
   c. auto_post_to_timeline = TRUE の場合: タイムラインに結果サマリー投稿を自動生成（後述）
```

#### 自動締切バッチ
```
- 実行タイミング: 毎時実行（@Scheduled(cron = "0 0 * * * *")）
- 対象: status = PUBLISHED かつ expires_at <= NOW()
- 処理: status を CLOSED に更新、closed_at = expires_at
- 通知: results_visibility = AFTER_CLOSE の場合に SURVEY_RESULTS_AVAILABLE 通知を送信
- タイムライン: auto_post_to_timeline = TRUE かつ results_visibility != ADMIN_ONLY の場合、SurveyClosedEvent を発行 → タイムラインに結果サマリー投稿を自動生成
```

#### リマインド通知バッチ
```
- 実行タイミング: 毎時実行（@Scheduled(cron = "0 0 * * * *")）
- 対象: status = PUBLISHED のアンケートのうち、remind_before_hours の各タイミングに
  該当するもの（例: remind_before_hours = [72, 24] → 期限72時間前と24時間前に発火）
- 処理: 未回答メンバーに SURVEY_RESPONSE_REMINDER 通知を送信
  - distribution_mode = ALL: スコープ内全未回答メンバー
  - distribution_mode = TARGETED: survey_targets 内の未回答者
- 注意: 既に回答済みのメンバーにはリマインドを送らない
- 重複防止: 各タイミングで1回のみ送信（送信済みフラグは survey_id + remind_hour の
  組み合わせで管理。専用テーブルは不要、remind_before_hours JSON から送信済み要素を
  除外する方式で十分）
```

### タイムライン・ブログ連携

#### 1. アンケート公開時 → タイムライン自動投稿

`auto_post_to_timeline = TRUE` のアンケートが公開された際に、タイムライン（F04.1）にアンケート告知投稿を自動生成する。通知だけだと見逃されやすいが、タイムラインに流れることで回答率が向上する。

**実装**:
- `SurveyPublishedEvent`（ApplicationEvent）を `TimelinePostService` がリスナーで購読
- タイムライン投稿を自動 INSERT:
  - `scope_type` / `scope_id`: アンケートと同じスコープ
  - `posted_as_type = 'SYSTEM'`（システム自動投稿）
  - `body`: テンプレート生成（下記参照）
  - `ref_type = 'SURVEY'`、`ref_id = survey.id`（リンクカード用）

**投稿テンプレート**:
```
📋 アンケートのお知らせ

「{title}」が公開されました。
{description（先頭100文字）}

回答期限: {expires_at（yyyy/MM/dd HH:mm）}
設問数: {question_count}問

▶ 回答する
```

- 「▶ 回答する」はフロントエンドがリンクカード（`ref_type = 'SURVEY'`）として描画し、タップでアンケート回答画面に遷移
- `expires_at` が NULL の場合は「回答期限: なし」と表示

#### 2. アンケート締切時 → 結果サマリーをタイムライン投稿

`auto_post_to_timeline = TRUE` のアンケートが締切られた際に、タイムラインに結果サマリー投稿を自動生成する。

**実装**:
- `SurveyClosedEvent`（ApplicationEvent）を `TimelinePostService` がリスナーで購読
- `results_visibility` が `ADMIN_ONLY` の場合は投稿しない（結果非公開のため）

**投稿テンプレート**:
```
📊 アンケート結果

「{title}」の結果が出ました。
回答率: {response_rate}%（{response_count}名 / {target_count}名）

{上位3件の設問サマリー}
  Q1. {question_text} → 1位「{top_option}」({percentage}%)
  Q2. {question_text} → 平均 {average} / {scale_max}
  ...

▶ 詳しい結果を見る
```

- 選択式設問: 最多得票の選択肢とパーセンテージを表示
- 評価スケール設問: 平均値を表示
- 自由回答設問: サマリーには含めない（プライバシー配慮）
- 4設問目以降は省略（「...他 N 問」）

#### 3. ブログ記事の下書き自動生成

`POST /surveys/{id}/generate-blog-draft` で ADMIN が手動トリガー。テンプレート方式で集計データを Markdown に差し込み、F06.1 ブログの `DRAFT` 記事として生成する。

**生成テンプレート構成**:
```markdown
# 「{title}」結果報告

## アンケート概要
| 項目 | 内容 |
|------|------|
| 実施期間 | {published_at} 〜 {closed_at} |
| 回答率 | {response_rate}%（{response_count}名 / {target_count}名） |
| 匿名 | {is_anonymous ? "はい" : "いいえ"} |

## 設問別結果

### Q1. {question_text}
| 選択肢 | 得票数 | 割合 |
|--------|--------|------|
| {option_text} | {count}名 | {percentage}% ████████░░ |
| ... | ... | ... |

### Q2. {question_text}（評価スケール）
**平均: {average} / {scale_max}** ★★★★☆

| 評価 | 回答数 | 割合 |
|------|--------|------|
| 5 - {scale_max_label} | {count}名 | {percentage}% |
| ... | ... | ... |

### Q3. {question_text}（自由回答）
回答件数: {total_responses}件

> 「{answer_text_1}」
> 「{answer_text_2}」
> ...

---
※ このレポートはアンケート結果から自動生成されました。
  内容を確認・編集してから公開してください。
```

**処理フロー**:
1. `GET /surveys/{id}/results` と同じ集計ロジックで結果データを取得
2. Thymeleaf テンプレートエンジン（または `String.format`）で Markdown を生成
3. F06.1 の `BlogPostService.create()` を内部呼び出し:
   - `title`: 「{survey.title}」結果報告
   - `body`: 生成された Markdown
   - `status = 'DRAFT'`
   - `scope_type` / `scope_id`: リクエストで指定（デフォルトはアンケートと同スコープ）
   - `created_by`: API 呼び出し元の ADMIN ユーザー
4. 生成された `blog_post_id` をレスポンスで返す
5. `status = PUBLISHED`（中間結果）の場合: フッターに「※ この結果は {生成日時} 時点の中間集計です。最終結果は締切後に確定します。」を自動追加

#### 4. タイムライン/ブログへのアンケートカード埋め込み

タイムライン投稿やブログ記事にアンケートへのリンクカードを埋め込む機能。ユーザーが「回答する」ボタンを直接タップしてアンケート画面に遷移できる。

**実装**:
- F04.1 タイムラインの `ref_type` ENUM に `'SURVEY'` を追加（Flyway ALTER TABLE）
- F06.1 ブログの投稿本文内に `[survey:{surveyId}]` のショートコードを記述すると、フロントエンドがリンクカードとしてレンダリング

**リンクカードの表示内容**:
```
┌──────────────────────────────┐
│ 📋 新メニューについてのアンケート   │
│ 回答率: 72% ████████░░░░       │
│ 期限: 2026/04/15               │
│              [回答する →]       │
└──────────────────────────────┘
```

- ステータスに応じた表示:
  - `PUBLISHED`: 「回答する →」ボタン（未回答時）/ 「回答済み ✓」（回答済み時）
  - `CLOSED`: 「結果を見る →」ボタン（`results_visibility` の条件を満たす場合）
  - `DRAFT`: カード非表示（未公開のため）

**フロントエンド取得**: カード表示に必要なデータは `GET /surveys/{id}` のレスポンス（`title`, `status`, `response_rate`, `expires_at`, `has_responded`）から取得。既に既存 API でカバーされており、追加 API は不要。

### フロントエンド表示仕様

#### 回答進捗バー（一覧画面）

`GET /surveys` の一覧画面で、各アンケートの回答率をプログレスバーで視覚的に表示する。

| 回答率 | 色 | 表示 |
|--------|-----|------|
| 0〜49% | 赤（`#EF4444`） | 「回答率低」警告アイコン付き |
| 50〜79% | 黄（`#F59E0B`） | 通常表示 |
| 80〜100% | 緑（`#22C55E`） | 「ほぼ完了」チェックアイコン付き |

- ADMIN 向け: プログレスバーの横に「{response_count} / {target_count}」を数値表示
- MEMBER 向け: プログレスバーのみ（具体的な人数は非表示）
- `target_count = NULL` の場合: プログレスバー非表示、`response_count` のみ表示

### 重要な判定ロジック

- **結果閲覧権限の判定**: 以下の優先順で判定（上位条件に該当すれば `results_visibility` を無視して閲覧可能）:
  1. **作成者**（`created_by == currentUser.id`）: 常にフルアクセス
  2. **ADMIN+**: 常にフルアクセス
  3. **`survey_result_viewers` 登録者**: `results_visibility` に関わらず閲覧可能。`can_view_individual = TRUE` なら個別回答も閲覧可能（ただし匿名時は集計のみ）
  4. **一般メンバー**: `results_visibility` に従う:
     - `ALWAYS`: 誰でも閲覧可能
     - `AFTER_RESPONSE`: `has_responded = true` のユーザーのみ
     - `AFTER_CLOSE`: `status = 'CLOSED'` の場合のみ全員
     - `ADMIN_ONLY`: 上記 1〜3 のみ（一般メンバーには非公開）
- **回答修正**: `allow_multiple_submissions = TRUE` の場合のみ。全回答を DELETE → INSERT で差し替え（部分修正は不可。全設問を再送信）
- **匿名アンケートの制約**: DB には `user_id` を記録（重複防止 + 監査用）。API レスポンスでは回答者情報を一切返さない。CSV エクスポートでも「匿名」と表示。ただし `GET /respondents` では「誰が回答したか」の yes/no は確認可能（回答内容は非公開）
- **PUBLISHED 後の設問凍結**: 回答の整合性を保つため、公開後は設問・選択肢の追加・変更・削除を一切禁止。誤りを発見した場合はアンケートを削除して再作成する運用
- **回答受付期間**: `starts_at` 〜 `expires_at` の間のみ回答可能。`starts_at = NULL` の場合は `published_at` が開始日時。`expires_at = NULL` の場合は手動締切まで無期限

### F04.3 通知基盤との連携

| 通知タイプ | トリガー | 送信先 |
|-----------|---------|--------|
| `SURVEY_CREATED` | アンケート公開時 | スコープ内全メンバー |
| `SURVEY_RESPONSE_REMINDER` | 回答期限24時間前（日次バッチ） | 未回答メンバー |
| `SURVEY_RESULTS_AVAILABLE` | アンケート締切時（`results_visibility = AFTER_CLOSE` の場合） | スコープ内全メンバー |

**F04.3 側への変更**:
- `NotificationType` Java enum に上記3タイプを追加
- `notification_type_preferences` のデフォルトで全タイプ `true`（opt-out 方式）

## 6. セキュリティ考慮事項

- **認可チェック**: SurveyService の入り口で `scope_type + scope_id` と `currentUser` の所属・ロールを検証。他スコープのアンケートへのアクセスは 403 で拒否
- **作成者チェック**: 更新・削除・公開・締切は `created_by == currentUser.id` または ADMIN+ であること
- **匿名回答の保護**: `is_anonymous = TRUE` のアンケートでは API レスポンスに個別回答者情報を含めない。回答者→回答内容の紐付けは API 経由では不可能。SYSTEM_ADMIN がDB レベルで監査する場合のみ確認可能
- **匿名 + リアルタイム結果のプライバシー制限**: `is_anonymous = TRUE` かつ `results_visibility = ALWAYS` の場合、回答者数が少ないと「自分が回答した直後の結果変化」から回答内容が推測される可能性がある。フロントエンドで「回答者数が5名未満の間は集計結果を表示しません」のガードを実装（`response_count < 5` の場合は結果を伏せる。API 側は返すがフロントで制御。閾値は将来調整可能）
- **入力サニタイズ**: `title`・`description`・`question_text`・`option_text`・`answer_text` は XSS 対策として HTML タグを除去
- **レートリミット**: 回答 API（`POST /responses`）は `Bucket4j` で1ユーザー1分間に10回まで（連打防止）
- **監査ログ**: アンケートの作成・公開・締切・削除を `audit_logs` に記録。回答自体は監査ログに記録しない（匿名性の担保）
- **個別回答閲覧の保護**: `GET /surveys/{id}/responses/{userId}` は非匿名アンケートでのみ利用可能。匿名アンケートに対する呼び出しは 403 で拒否。アクセスは監査ログに記録
- **CSV エクスポートのセキュリティ**: ADMIN+ / 作成者 / `survey_result_viewers` 登録者が利用可能。エクスポートアクションは監査ログに記録（個人情報含むデータの持ち出し追跡）
- **回答データ保持ポリシー**: `CLOSED` から12ヶ月経過したアンケートの `survey_responses` は物理削除可能（`surveys` マスターと集計結果キャッシュは保持）。Phase 10 のアーカイブバッチで対応。匿名アンケートの回答データは特にプライバシーリスクが高いため、保持期間を6ヶ月に短縮することを推奨

## 7.1 お知らせウィジェット連携（F02.6）

アンケートは、作成画面で「お知らせウィジェットに表示する」チェックを入れることで、ダッシュボード上部の横断お知らせウィジェット（F02.6）に表示できる。

### 自動お知らせ化

- `surveys` に `post_announcement_on_publish BOOLEAN NOT NULL DEFAULT FALSE` を追加（Flyway V13.021）
- ユーザーが作成画面でチェックを入れた場合、`status = PUBLISHED` へ遷移した時点で **`SurveyService` 内で自動的に** `AnnouncementFeedService.createFromSource()` を呼び出し、`announcement_feeds` に INSERT
- `expires_at` は `surveys.expires_at` をコピー（回答期限と同時にお知らせも自動失効）。`expires_at` が NULL のアンケートはお知らせも期限なし

### 既存の `auto_post_to_timeline` との棲み分け

| 設定 | 効果 |
|---|---|
| `auto_post_to_timeline = TRUE` | 公開時/締切時に**タイムライン投稿**を自動生成（F05.4 既存仕様） |
| `post_announcement_on_publish = TRUE`（新） | 公開時に**お知らせウィジェット**に登録 |

両方 TRUE にすることも可能（タイムライン流速下でも告知を両立させたい場合）。UI 上は別々のチェックボックスとして表示。

### 優先度のマッピング

アンケートには優先度カラムがないため `announcement_feeds.priority = NORMAL` 固定。ただし `expires_at` が 24 時間以内のアンケートはフロントエンドで「締切間近」バッジを付与（UI 演出のみ、DB の priority は NORMAL のまま）。

### 連動削除

- アンケートが `CLOSED` に遷移した時、`SurveyClosedEvent` → `AnnouncementSyncEventListener` が `announcement_feeds` を物理削除（締切後のアンケートはお知らせの意味を失うため）
- アンケートの論理削除時、`SurveyDeletedEvent` → `announcement_feeds.source_deleted_at` をセット

### UI 配置

`SurveyForm.vue` の `auto_post_to_timeline` チェックの下に `AnnouncementToggle.vue` を配置。

### Source Meta 表示

- `response_count` / `target_count`（回答率バー表示用）
- `expires_at`（締切日時）

ウィジェットカードに「📊 春合宿の日程アンケート（8/20 回答・締切 4/25）」のように表示。
