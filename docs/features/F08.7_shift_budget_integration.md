## 1. 概要

### 1.1 背景・目的

TODO 一覧 #38 として記録された運用要望は次のとおり:

> シフトと連動した予算管理機能。追記の予算を決め、時給などを考慮してシフト作成できるといい。TODO は予算を決めて TODO を決められる？

文面は具体性に乏しいが、ヒアリングで意図を分解すると次の三層になる:

1. **「予算を決めてシフトを組む」** — 月次の人件費予算が先にあり、そこから「平均時給」「1 スロット時間」「必要枠数」を逆算したい。シフト確定後に予算超過していたら気づきたい
2. **「シフト確定 → 予算消化」** — シフトが公開された段階で、F08.6 の費目（人件費）から自動的に予算を引き当て、消化率をリアルタイムに見たい
3. **「TODO/プロジェクトに予算をつける」** — 「文化祭 30 万円まで」「夏祭りスタッフ人件費 5 万円まで」のように、TODO（プロジェクト）の単位でも予算枠を持ちたい

これら三層は既存機能（F03.5 シフト・F08.6 予算・F02.3 TODO）の境界をまたぐため、各設計書に追補するのではなく **F08.7 として独立した連携層** を立て、3 機能の間に位置する「予算割当 → 消化 → 紐付け → 警告」という一貫したフローを定義する。

### 1.2 解決する課題

1. **予算策定とシフト編成の分断** — 現状 F08.6 で年間予算を立てても、それがシフト編成画面に伝わらず、管理者は別画面・別エクセルで突合している
2. **人件費の事後発覚** — 月末に F08.6 の集計を見て初めて予算超過が判明する。事前警告がない
3. **TODO ベースの予算管理ができない** — 「このイベントに使える予算」を TODO/プロジェクト単位で見える化する手段がない

### 1.3 非目標

本機能は次を **扱わない**:

- **給与計算そのもの** — 源泉徴収・社会保険料・所得税・年末調整等は対象外。あくまで「予算管理目的の概算」に留める
- **税務処理・確定申告書類の生成** — F08.6 の年次報告書（既存）に委ねる
- **タイムカード/打刻による実労働時間集計** — F03.1 出欠管理の領域。本機能はシフト枠（計画値）を基に予算消化を計算する。実績差分は将来拡張（Phase 10 以降）
- **複数通貨の同時運用** — Phase 9 では単一通貨（既存 F08.6 と同様、日本円固定）。**ただし Phase 10 拡張ポイントとして** `shift_budget_allocations` / `shift_budget_consumptions` / `todo_budget_links` の各テーブルに `currency CHAR(3) NOT NULL DEFAULT 'JPY'` を予め配置済（§5.2 / §5.3 / §5.4）。Phase 10 では (1) per-organization の主通貨設定、(2) 為替レート変換テーブル、(3) 集計時の通貨統一ロジックを追加することで多通貨切替可能化を達成する設計
- **AI による最適シフト提案** — 予算制約を満たすシフトの自動生成は本機能のスコープ外。既存 F03.5 の自動割当に「予算上限」を **入力パラメータ** として渡すだけに留める

## 2. スコープ

### 2.1 含むもの

- 予算→必要シフト枠数の **逆算計算 API**（数式提供）
- シフトスロット ↔ 予算費目の **割当テーブル**（`shift_budget_allocations`）
- シフト確定時の **消化記録**（`shift_budget_consumptions`、計画値ベース）
- TODO/プロジェクト → 予算割当の **紐付テーブル**（`todo_budget_links`）
- リアルタイム消化率の **集計 API**
- 80% / 100% / 超過の **閾値警告**（`budget_threshold_alerts`、F04.3 通知連携）
- 月跨ぎの **按分ルール**（深夜跨ぎスロットを含む）
- シフトキャンセル時の **逆仕訳**
- 監査ログ（F10.3）への記録

### 2.2 含まないもの

- 既存 `budget_allocations`（年度×費目で 1 レコード）の置換 — F08.7 は **シフト期間（月単位）** で割り当てる別概念。年度予算配分の上に乗る「月割り内訳」として位置づける
- 既存 `budget_transactions` への自動仕訳作成 — 消化記録は F08.7 専用テーブルに保持し、F08.6 トランザクション帳へは「月次締め確定時」にまとめて反映する（v1.0 時点では別画面で確認、自動連携は Phase 9-δ 末尾で評価）
- 給与明細・賞与・退職金管理

### 2.3 既存機能との境界線

| 機能 | 担当範囲 | F08.7 から見た境界 |
|------|----------|--------------------|
| F03.5 シフト管理 | スロット定義、希望収集、自動割当、確定、公開 | 確定/公開された `shift_slots.assigned_user_ids` を入力として読み取る |
| F08.6 予算会計 | 年度・費目・配分・収支トランザクション・月次/年次報告書 | `budget_categories.id` と `budget_fiscal_years.id` を参照する。逆向きに F08.6 のトランザクション帳に書き戻すのは月次締めバッチのみ |

## 3. ユースケース

### UC-1: 予算上限から必要シフト枠数を逆算する

**アクター**: ADMIN / DEPUTY_ADMIN（`MANAGE_SHIFTS` + `BUDGET_VIEW`）
**事前条件**: 当月の人件費予算が `budget_allocations` に登録済み（例: 30 万円）。チームメンバーの時給が `shift_hourly_rates` に登録済み

**フロー**:
1. ADMIN がシフト作成画面を開き、対象月（例: 2026 年 6 月）を選択
2. 「予算逆算」ボタンを押下
3. システムは `POST /api/v1/shift-budget/calc/required-slots` に当月の `budget_amount` と `slot_hours`（既定: 4 時間）を送信
4. サーバは `shift_hourly_rates` の現行平均時給を計算し、必要枠数を返却:
   `必要枠数 = floor(予算額 / (平均時給 × 1スロット時間))`
5. UI に「予算 30 万円 / 平均時給 1,200 円 / 4 時間スロット → 最大 62 枠」と表示
6. ADMIN はこの上限を参考にシフト編成を進める

**事後条件**: 計算結果はステートレス（DB 書き込みなし）。割当を確定したい場合は UC-2 に進む。

### UC-2: シフト確定時に人件費見込を予算費目から消化する

**アクター**: ADMIN / DEPUTY_ADMIN
**事前条件**: 当月の `shift_budget_allocations` レコードが作成済み（費目=人件費、期間=2026-06-01〜2026-06-30、配分額=30 万円）

**フロー**:
1. シフトスケジュールを `ADJUSTING → PUBLISHED` に遷移（F03.5 §5）
2. システムは公開対象スケジュール内の全 `shift_slots.assigned_user_ids` を走査し、各 (slot, user) について `shift_hourly_rates` から適用時給を取得
3. `shift_budget_consumptions` に `status = PLANNED` で 1 行ずつ INSERT（`hourly_rate_snapshot`、`hours`、`amount` を保存）
4. 同時に `shift_budget_allocations` の集計値を更新（非正規化キャッシュ `consumed_amount`）
5. 公開イベント（F03.5 既存）を発火し、UI ダッシュボードに反映
6. 消化率が閾値を超えていれば UC-5 を発火

**事後条件**: 消化記録は `PLANNED`。月次締めで `CONFIRMED` に昇格する。

### UC-3: TODO（プロジェクト・タスク）に予算枠を紐付け、進捗とともに消化を可視化する

**アクター**: ADMIN / DEPUTY_ADMIN（`MANAGE_TODO` + `BUDGET_VIEW`）
**事前条件**: プロジェクト「文化祭 2026」が登録済み。当該プロジェクトに紐付く予算割当（人件費 5 万円、消耗品費 3 万円）を立てたい

**フロー**:
1. プロジェクト編集画面で「予算」タブを開く
2. `POST /api/v1/todo-budget/links` で `project_id` と `allocation_id`、`link_amount=50000` を送信
3. プロジェクト詳細画面に「予算 5 万円 / 消化 1.2 万円 / 残 3.8 万円（24%）」と表示
4. プロジェクトに紐付く F03.5 シフトの消化が進むと、紐付分が自動的に集計される
5. プロジェクト完了時には残予算と実消化額を比較したサマリーが報告書に出る

**事後条件**: TODO 単位の紐付けも可能だが、UI 主導線はプロジェクト単位。

### UC-4: 月跨ぎ・締め処理時の予算精算

**アクター**: ADMIN（経理担当）
**事前条件**: 月末を迎え、当月の `shift_budget_consumptions` がすべて `PLANNED` 状態

**フロー**:
1. 月次締めバッチ（手動起動 or cron）が起動
2. 当月の `PLANNED` レコードを `CONFIRMED` に昇格
3. 同時に F08.6 `budget_transactions` に「シフト人件費 月次集計」として 1 件にまとめて INSERT（`source_type = 'SHIFT_BUDGET_MONTHLY'`、`source_id = allocation_id`）
4. 翌月の `shift_budget_allocations` を自動コピー（金額・費目を引き継ぎ。ADMIN が後から微調整可）

**事後条件**: 当月の予算は確定。以降キャンセル等の修正は **取消仕訳** で記録する（F08.6 のパターンに準拠）。

### UC-5: 予算超過警告と承認フロー連携

**アクター**: ADMIN（受信者）、予算管理者（承認者）
**事前条件**: 消化率が 80% を初めて超えた

**フロー**:
1. UC-2 内のチェックで閾値超過を検知
2. `budget_threshold_alerts` に 1 件 INSERT（`threshold_percent=80`、`triggered_at=now()`）
3. F04.3 プッシュ通知で ADMIN/予算管理者に通知（i18n キー `shiftBudget.alert.warn80`）
4. 100% 到達時は別レコード（`threshold_percent=100`）と承認ワークフロー起動（F05.6）
5. 100% 超過してもシフト確定操作はブロックされない（運用判断を妨げない）。ただし監査ログに `BUDGET_OVER_LIMIT_PROCEED` が記録され、承認待ちフラグが立つ
6. 予算管理者が `POST /api/v1/shift-budget/alerts/{id}/acknowledge` で確認応答すると `acknowledged_at` がセットされる

**事後条件**: 同一 allocation × 同一閾値で重複通知しないよう、`UNIQUE (allocation_id, threshold_percent)` 制約を貼る。

## 4. 機能要件

### 4.1 予算→シフト枠の逆算機能

**数式**:

```
必要枠数 = floor(予算額 / (平均時給 × 1スロット時間))
```

- `予算額`: `shift_budget_allocations.allocated_amount`（円）
- `平均時給`: 後述のパラメータで決定
- `1スロット時間`: 既定 4 時間（API パラメータで上書き可能、最小 0.25h、最大 24h）

**平均時給算出パラメータ** — リクエストボディで以下のいずれかを指定:

| `mode` | 説明 | 計算式 |
|--------|------|--------|
| `MEMBER_AVG`（既定） | 対象チームの全アクティブメンバーの現行時給を単純平均 | `SUM(hourly_rate) / COUNT(user_id)` |
| `POSITION_AVG` | ポジション（役割）別に平均時給を出し、必要人数で重み付け | Σ(position_avg × required_count_in_position) ÷ Σ(required_count) |

**`POSITION_AVG` モードの詳細リクエスト形式（v1.1 追記）**:

```json
{
  "team_id": 12,
  "budget_amount": 300000,
  "slot_hours": 4.0,
  "rate_mode": "POSITION_AVG",
  "position_required_counts": [
    { "position_id": 1, "required_count": 5 },
    { "position_id": 2, "required_count": 3 },
    { "position_id": 3, "required_count": 2 }
  ]
}
```

**サーバー側ロジック**:

1. `position_required_counts` の各エントリについて、当該 `position_id` を持つ `team_id` 内 **アクティブメンバー** の `shift_hourly_rates` から **ポジション別平均時給** を自動取得
2. **加重平均** を計算: `weighted_avg = Σ(position_avg × required_count) ÷ Σ(required_count)`
3. その加重平均を `avg_hourly_rate` として通常の必要枠数計算に投入
4. レスポンスに `position_breakdown` 配列を含める（各ポジションの平均時給と寄与度を可視化）

**「アクティブメンバー」の定義（v1.2 確定）**:

隠れ-1 指摘を受け、`POSITION_AVG` モードで集計対象とするメンバーの定義を確定する:

- **アクティブメンバー** = 「当該 team に所属し、退職していない、かつ当該 position_id にアサインされているメンバー」
- 具体的には以下の条件を **AND** で満たすこと:
  - `team_members.team_id = :teamId`
  - `team_members.left_at IS NULL`（退職メンバー除外）
  - `team_members.position_id = :positionId`（ポジション未定者除外）
  - `shift_hourly_rates.deleted_at IS NULL`（時給設定が論理削除されていない）

**確定 SQL**:

```sql
SELECT AVG(h.hourly_rate) AS avg_rate, COUNT(h.user_id) AS member_count
FROM shift_hourly_rates h
JOIN team_members tm ON h.user_id = tm.user_id
WHERE tm.team_id = :teamId
  AND tm.left_at IS NULL          -- v1.2 確定: 退職メンバー除外
  AND tm.position_id = :positionId -- v1.2 確定: ポジション未定者除外
  AND h.deleted_at IS NULL;
```

**境界ケースの扱い**:

| ケース | 挙動 |
|--------|------|
| 当該 position に該当メンバー 0 人 | そのポジションをスキップ + `warning: POSITION_NO_RATE_DATA` |
| 当該 position に該当メンバーは居るが全員時給未設定（`shift_hourly_rates` レコードなし） | `avg_rate = NULL` → そのポジションをスキップ + `warning: POSITION_NO_RATE_DATA` |
| 当該 position に該当メンバーは居るが一部のみ時給設定済 | 設定済メンバーのみで AVG 計算（未設定者は除外） |
| `team_members.left_at IS NOT NULL`（退職済）のメンバーが時給を持っている | **集計から除外**（AVG に含めない）。理由: アクティブな勤務者の時給で予算計画を立てるため |
| `team_members.position_id IS NULL`（ポジション未定）のメンバー | **集計から除外**。理由: position_required_counts でポジション別に明示的に指定されたメンバーのみで加重平均を取る方針のため |

**バリデーション**:
- `position_required_counts` が空配列 → 400 + `error: EMPTY_POSITION_LIST`
- 重複する position_id → 400 + `error: DUPLICATE_POSITION_ID`
- `required_count <= 0` → 400 + `error: INVALID_REQUIRED_COUNT`
- 当該 position に時給設定済メンバーが 0 人 → そのポジションをスキップ + `warning: POSITION_NO_RATE_DATA` を含む

| `EXPLICIT` | 呼び出し側が `avg_hourly_rate` を直接指定 | リクエストボディの値をそのまま使用 |

**境界**:
- `平均時給 = 0` または `予算額 = 0` の場合 → 必要枠数 0 を返却し、`warning: "AVG_RATE_ZERO"` または `"BUDGET_ZERO"` を併記
- 平均時給未設定メンバーが過半数を占める場合 → `warning: "INSUFFICIENT_RATE_DATA"` を返却（計算は未設定者を除外して継続）
- スロット時間 ≤ 0 → 400 エラー

### 4.2 シフト→予算消化リンク

公開トリガー:
- `shift_schedules.status` が `ADJUSTING → PUBLISHED` に遷移したタイミング
- F03.5 既存の公開イベント（`ShiftPublishedEvent`）に対する `@EventListener` で受け取る

書き込みルール:
- 1 つの `shift_slots` に対し、`assigned_user_ids` の **配列要素ごと** に `shift_budget_consumptions` の 1 行を作成
- `hourly_rate_snapshot` には公開時点で適用される時給を保存（後で時給が変わっても消化額は変動しない）
- `hours` は `end_time - start_time`。深夜跨ぎ（end < start）の場合は + 24 時間

按分ルール（同一スロット複数アサイン時）:
- 各メンバーに **同一の勤務時間** を割当（按分しない）。理由: F03.5 のスロットは「全員が同時に同じ時間働く」前提。ペア勤務 = 2 人分の人件費が発生するのが正

### 4.3 TODO→予算紐付（粒度: プロジェクト単位を主、TODO 単位はオプション）

紐付テーブル `todo_budget_links` の制約:
- `project_id` と `todo_id` は **排他**（CHECK 制約 `chk_tbl_target_xor`）
- 同一 (project_id, allocation_id) は重複禁止
- 同一 (todo_id, allocation_id) は重複禁止
- `link_amount` と `link_percentage` も排他（どちらか一方、または両方 NULL = 全額紐付）

**プロジェクト単位の予算割当戦略 — v1.1 で確定**:

シフト→プロジェクトへの予算集計には **2 経路** を併用する:

1. **直接経路（Direct）**: シフトスケジュール自身がプロジェクトに紐付いている場合
   - `shift_schedules.linked_project_id` を Phase 9-γ で V3.130 にて追加（NULLABLE、ON DELETE SET NULL、**1:1 関係**。M:N にはしない — 1 つのスケジュールは多くて 1 つのプロジェクトに属する単純化を採用）
   - その shift_schedules に紐付く全ての shift_budget_consumptions を、当該プロジェクトの消化として SUM
   - **割当側**: `shift_budget_allocations.project_id`（NULLABLE、§5.2 参照）に対象プロジェクトを設定して「プロジェクト専用割当」とすることもできる

2. **TODO 経由（Via TODO）**: シフト/プロジェクト自体は紐付いていないが、TODO 単位で予算を引きたいケース
   - `todo_budget_links` で TODO（または親プロジェクト）と allocation を紐付
   - 紐付額（`link_amount` または `link_percentage`）を上限として、当該 allocation の消化額のうち按分された分をプロジェクトへ加算

**重複防止ルール**:

直接経路と TODO 経由の二重計上を防ぐため、以下のルールを Service 層で強制する:

- **直接経路が存在する shift_schedules** に紐付く consumption は、`todo_budget_links` 経由の集計から **除外** する（`shift_schedules.linked_project_id IS NOT NULL` を WHERE で検出）
- **`shift_budget_allocations.project_id` が NOT NULL の割当** は「プロジェクト専用割当」として扱い、当該 allocation の消化は当該プロジェクト以外の `todo_budget_links` から参照不可（API 層で 409）
- それ以外の通常の割当は両形式（月×費目×team 単独 / 月×費目×team×project）を **両方許容** する設計

**集計 SQL の概要**（プロジェクト P の消化額）:

```sql
-- 経路1: 直接経路（shift_schedules.linked_project_id = P）
SELECT COALESCE(SUM(c.amount), 0) AS direct_amount
FROM shift_budget_consumptions c
JOIN shift_schedules s ON c.shift_id = s.id
WHERE s.linked_project_id = :projectId
  AND c.status IN ('PLANNED', 'CONFIRMED');

-- 経路2: TODO 経由（todo_budget_links 経由、直接経路と重複する分は除外）
SELECT COALESCE(SUM(
  CASE
    WHEN tbl.link_amount IS NOT NULL THEN LEAST(c.amount, tbl.link_amount)
    WHEN tbl.link_percentage IS NOT NULL THEN c.amount * tbl.link_percentage / 100
    ELSE c.amount
  END
), 0) AS via_todo_amount
FROM todo_budget_links tbl
JOIN shift_budget_consumptions c ON c.allocation_id = tbl.allocation_id
LEFT JOIN shift_schedules s ON c.shift_id = s.id
WHERE tbl.project_id = :projectId
  AND c.status IN ('PLANNED', 'CONFIRMED')
  AND (s.linked_project_id IS NULL OR s.linked_project_id != :projectId);  -- 直接経路と重複する分を除外

-- 合計
total_amount = direct_amount + via_todo_amount;
```

**集計の流れ**（テーブル形式）:

| 経路 | データソース | 紐付方法 | 重複時の優先度 |
|------|--------------|----------|----------------|
| 直接 | `shift_schedules.linked_project_id` → `shift_budget_consumptions` | スケジュール単位 | 最優先（TODO 経由から除外） |
| TODO経由・上限金額 | `todo_budget_links.link_amount` | 紐付分のみ、`LEAST(消化, 上限)` | 直接経路が無い場合のみ |
| TODO経由・割合 | `todo_budget_links.link_percentage` | 消化額 × 割合 | 直接経路が無い場合のみ |
| TODO経由・全額 | `todo_budget_links` (両 NULL) | 当該 allocation の全消化 | 直接経路が無い場合のみ |

**v1.1 における決着**:

- F03.5 `shift_schedules.linked_project_id` の追加は **Phase 9-γ で V3.130 にて確定** （F03.5 修正ブランチで実施。詳細 §5.7 / §12.1）
- `shift_budget_allocations.project_id` を **NULLABLE で追加**（§5.2 確定）。両形式（月×費目×team 主形式、project_id 付加形式）を許容
- 割当 — シフト — プロジェクトの関係は **1:1**（M:N は採用しない）。複数プロジェクトに分配したい場合は `todo_budget_links` の `link_percentage` で表現する

### 4.6 月跨ぎ・締め処理

**UNIQUE 制約に `deleted_at` を含める設計意図（v1.2 追記）**:

`uq_sba_scope_category_period` に `deleted_at` を含めているのは、**論理削除済みレコードと新規レコードの並存を許可する** ためである。具体的には:

- `deleted_at IS NULL`（生存レコード）と `deleted_at = '2026-04-30 12:34:56'`（論理削除済み）は、UNIQUE 評価では別のキーとして扱われる
- 「同一スコープ（org × team × project × category × period）の割当を削除 → 同一条件で新規作成」というシナリオを許可する（経理ミスの差し替え運用に対応）
- 単なる `(organization_id, team_id, project_id, budget_category_id, period_start, period_end)` だけの UNIQUE 制約だと、論理削除済みレコードが残っている限り再作成できないという**運用上の手詰まり**が発生するため、`deleted_at` を含めて回避する

**運用上の注意とアプリ層での重複チェック強化**:

DB 制約だけでは「論理削除済みが複数ぶら下がる」「生存が誤って 2 件以上できる」という稀な競合（同時 INSERT）を完全には防げないため、Service 層で以下を追加実施する:

1. **作成・更新時の生存重複チェック**: `INSERT` / `UPDATE` の前に `SELECT ... FOR UPDATE` で **`deleted_at IS NULL` のロックされた行を先取り** し、同一スコープの生存レコードが既に存在していれば 409 Conflict + `error: ALLOCATION_ALREADY_EXISTS` を返却
2. **論理削除→再作成 の監査ログ強化**: 同一スコープで「直前 24 時間以内に論理削除されたレコード」がある場合、新規作成イベントの監査ログ `SHIFT_BUDGET_ALLOCATION_CREATED` に `replaces_allocation_id` を含めて履歴の連鎖を追跡可能にする
3. **論理削除済みレコードの累積上限**: 同一スコープで論理削除済みが 10 件を超えた場合、警告ログを出力（運用者が「正しく経理整理できていない」サインとして検知）。10 件超過時の API 動作は阻害しない（あくまで運用警告）

これらは §11 のエッジケース「同一スコープで論理削除→再作成」シナリオでも参照される。

**制約**:
```sql
CONSTRAINT chk_sba_amount CHECK (allocated_amount >= 0)
CONSTRAINT chk_sba_period CHECK (period_start <= period_end)
CONSTRAINT chk_sba_consumed CHECK (consumed_amount >= 0 AND confirmed_amount >= 0 AND confirmed_amount <= consumed_amount)
```

**HAS_CONSUMPTIONS 制約の詳細運用ルール（v1.1 追記）**:

割当の論理削除（DELETE）時、以下のロジックを Service 層で強制する:

| 残存 consumption の状態 | 削除可否 | 返却ステータス |
|--------------------------|----------|----------------|
| `PLANNED` が 1 件以上残存 | **不可** | 409 + `error: HAS_CONSUMPTIONS_PLANNED` |
| `CONFIRMED` が 1 件以上残存 | **不可** | 409 + `error: HAS_CONSUMPTIONS_CONFIRMED`（経理影響大、月次締め前提） |
| `CANCELLED` のみ存在 | **可能** | 200。論理削除を実行（監査証跡として CANCELLED レコードは残置） |
| consumption 0 件 | **可能** | 200 |

**年度跨ぎ運用**:

- 年度コピー時（UC-4 の翌月コピーロジックを年度版に拡張）は **新しい allocation_id で新規 INSERT** する。allocation_id は変更されるため、過去の consumption は古い allocation_id を引き続き参照する（監査証跡を完全に保持）
- 旧年度 allocation は論理削除しない（経理参照用）。Phase 11 でアーカイブテーブル分離を検討
- **多年度集計**（複数年度をまたぐプロジェクト等）は `todo_budget_links` で複数の allocation を紐付けることで表現。Service 層で年度単位 / プロジェクト全期間の 2 軸サマリ API を提供（Phase 9-δ で `/shift-budget/projects/{id}/multi-year-summary` を追加）

```sql
-- (1) budget_configs 拡張
ALTER TABLE budget_configs
  ADD COLUMN over_limit_workflow_id BIGINT UNSIGNED NULL,
  ADD COLUMN shift_budget_enabled BOOLEAN NULL DEFAULT NULL,
  ADD CONSTRAINT fk_bc_workflow FOREIGN KEY (over_limit_workflow_id) REFERENCES workflow_definitions(id) ON DELETE SET NULL;

-- (2) BUDGET_ADMIN 自動付与（既存の MANAGE_SHIFTS + BUDGET_VIEW 保有者全員に対して）
INSERT INTO role_assignments (user_id, organization_id, permission_code, granted_by, granted_at, source)
SELECT DISTINCT
  ra1.user_id,
  ra1.organization_id,
  'BUDGET_ADMIN' AS permission_code,
  NULL AS granted_by,        -- システム自動付与
  NOW() AS granted_at,
  'V11.034_AUTO_GRANT' AS source
FROM role_assignments ra1
JOIN role_assignments ra2
  ON ra1.user_id = ra2.user_id
  AND ra1.organization_id = ra2.organization_id
WHERE ra1.permission_code = 'MANAGE_SHIFTS'
  AND ra2.permission_code = 'BUDGET_VIEW'
  AND NOT EXISTS (
    SELECT 1 FROM role_assignments ra3
    WHERE ra3.user_id = ra1.user_id
      AND ra3.organization_id = ra1.organization_id
      AND ra3.permission_code = 'BUDGET_ADMIN'

{

  "allocated_amount": 300000,
  "consumed_amount": 245000,
  "confirmed_amount": 200000,
  "planned_amount": 45000,
  "remaining_amount": 55000,
  "consumption_rate": 0.8167,
  "status": "WARN",
  "flags": ["BY_USER_HIDDEN"],
  "alerts": [
    { "threshold_percent": 80, "triggered_at": "2026-06-25T14:30:00Z", "acknowledged_at": null }
  ],
  "by_user": []
}
```

**形状確定ルール（v1.2）**:
- `by_user` は権限の有無に関わらず必ず **配列**（`null` ではない）。権限なし時は **空配列 `[]`**
- `flags` は権限の有無に関わらず必ず **配列**（`null` ではない）。権限あり時は空配列 `[]`、権限なし時は **`["BY_USER_HIDDEN"]`** を必ず含む
- 「`by_user=[]` かつ `flags=[]`」の組み合わせは「権限はあるが消化記録ゼロ」を意味する（権限なし時は必ず `BY_USER_HIDDEN` が `flags` に入る）
- フロントは `flags.includes("BY_USER_HIDDEN")` のみで権限判定し、`null` チェックは不要

`status` 判定:
- `< 0.80` → `OK`
- `0.80 ≤ rate < 1.00` → `WARN`
- `1.00 ≤ rate < 1.20` → `EXCEEDED`
- `≥ 1.20` → `SEVERE_EXCEEDED`

- 全エンドポイントは `X-Organization-Id` ヘッダで組織スコープを強制

- 集計 API（`/consumption-summary`）の `by_user` 配列の権限なし時挙動 **— v1.2 で確定**:

  - 詳細な DTO 形状は §9.3 / §6.2.3 を参照
- 全 DELETE は論理削除 or status 遷移、物理削除 API は提供しない

### 7.2 シフト作成画面の予算インジケーター（既存 F03.5 画面への追加）

**配置 — 案A（サイドバー配置）を採用と確定（v1.1）**:

検討した代替案:

- 案B（不採用）: ヘッダ右上のミニバー — 既存 F03.5 ヘッダが密集しておりタブレットで折りたたまれる懸念
- 案C（不採用）: フッター固定 — モバイルで操作領域を圧迫する

**採用理由**:
- サイドバーは F03.5 既存のフィルタ・統計エリアと同じレイアウト規約に従い、保守性が高い
- カード形式なら詳細リンク（ダッシュボード遷移）や複数割当の縦積み表示が自然
- タブレット時はサイドバー折りたたみと連動するため、予算カードも同時に隠れて整合する

**追加要素**:

- 予算カードのヘッダに「当月予算: 30 万円 / 使用見込: 24.5 万円（81%）」のミニバー
- 81% 超で黄バー、100% 超で赤バー、120% 超で赤+点滅アイコン
- カード内「予算逆算」ボタン押下で UC-1 のモーダル表示
- 複数割当が存在する場合はカード内で縦積み表示（最大 3 件、超過分は「他 N 件」リンクでダッシュボード遷移）

**「最大3件カード表示」とタブレットレスポンシブの関係（v1.2 明確化）**:

m-i 指摘を受け、画面サイズごとの予算カード挙動を以下に確定する。**「最大 3 件カード表示」は §7.4 ダッシュボードの『4 カラムグリッド』とは別物**（前者はシフト作成画面サイドバー、後者は専用ダッシュボード画面）であることを明記:

| 画面幅 | サイドバー状態 | 予算カードの挙動 |
|--------|----------------|------------------|
| **PC（≥1280px）** | 常時展開 | 予算カードを最大 3 件まで縦積み表示。4 件目以降は「他 N 件」リンク表示 |
| **タブレット（768〜1279px、サイドバー折りたたみ時）** | 折りたたみ | サイドバー左端に **アイコン + バッジ（消化率 %）のみ表示**。アイコンタップで全画面モーダルが開き、その中で最大 3 件のカードを縦表示 |
| **タブレット（768〜1279px、サイドバー展開時）** | 展開 | PC と同じく最大 3 件カード表示 |
| **モバイル（<768px）** | 強制折りたたみ | フローティングバッジ（右下に丸型 FAB ライク）に消化率 % のみ表示。タップでモーダル展開（最大 3 件） |

**§7.4 ダッシュボード「4 カラムグリッド」との関係**:
- §7.4 はあくまで **専用画面 `/dashboard/shift-budget`** のレイアウト
- §7.2 シフト作成画面のサイドバーカードは **シフト編成中の補助情報**として最大 3 件
- 両者は「同じ予算情報を別文脈で出すウィジェット」だが、件数上限・表示形式は異なる（混同しない）

**表示条件（v1.1 確定）**:

予算カードは **以下の AND 条件をすべて満たすときに限り表示** する:

1. グローバル feature flag: `feature.shiftBudget.enabled = true`
2. 組織別 feature flag: `budget_configs.shift_budget_enabled = TRUE`
3. 当月かつ当該 team_id（または組織全体）の `shift_budget_allocations` レコードが 1 件以上存在
4. ログインユーザーが `MANAGE_SHIFTS` 権限を保持

**MEMBER（一般メンバー）への非表示を確定**:
- MEMBER は MANAGE_SHIFTS 権限を持たないため、上記条件 4 で必ず非表示
- 自分自身の時給確認は別途 `VIEW_OWN_HOURLY_RATE`（§8）の専用 API/画面で対応
- 予算情報がメンバー一覧画面の片隅に漏れることを防ぐ（時給漏洩防止 §9.3 と整合）

## 8. 権限・ロール

### 8.1 新権限の導入（v1.2 でクリーンカット方式に確定）

検分指摘 M-5（一次）および M-iii（二次）を受け、既存権限の組み合わせ（`MANAGE_SHIFTS` + `BUDGET_VIEW`）で全運用を表現する方式は (1) 煩雑、(2) 権限設定取り違えのリスク、(3) Service 層の OR 評価ロジックがコードベース内に分散して保守性低下、という三つの問題があるため、**v1.2 で以下の方針に統一**:

| 権限名 | 役割 | 含意 |
|--------|------|------|
| **`BUDGET_ADMIN`** | F08.7 専用の管理権限。**v1.2 でクリーンカット方式に統一** — マイグレーション V11.034 で従来 `MANAGE_SHIFTS` + `BUDGET_VIEW` 保有者に自動付与し、移行後は **コード側で BUDGET_ADMIN のみで判定**（OR 後方互換ロジックは廃止） | 時給情報を含む全データへのアクセス、予算割当の作成・編集・削除、警告の承認応答、月次締めバッチ起動を一元的に表現 |
| **`VIEW_OWN_HOURLY_RATE`** | 自分自身の時給情報のみを参照する権限 | デフォルトで全 MEMBER に付与。組織管理者が剥奪することも可能（時給非表示運用の組織向け） |

**v1.2 で確定したクリーンカット方式の理由**:

v1.1 までは「`BUDGET_ADMIN` または `MANAGE_SHIFTS` + `BUDGET_VIEW` の OR 評価」という後方互換ロジックを Service 層に残していたが、以下の理由で **廃止** する:

1. **マイグレーション V11.034 で自動付与済**: 既存ユーザーは Phase 9-α 投入時点で全員 `BUDGET_ADMIN` を保持するため、OR 後方互換ロジックは実質的に不要
2. **新規組織・新規ユーザーには `BUDGET_ADMIN` 単独で運用**: マイグレーション以降に作成される組織・ユーザーは `BUDGET_ADMIN` を直接付与する設計のため、OR ロジックを残すと「新規ユーザーに `MANAGE_SHIFTS` + `BUDGET_VIEW` を付与すれば動いてしまう」という混乱を招く
3. **コード可読性・テスト容易性**: 権限チェックが単一条件 (`hasPermission("BUDGET_ADMIN")`) になり、テストケースが半減
4. **将来追加権限との整合**: F08.x シリーズで今後追加される権限も同様に「単独権限」を導入すれば、組み合わせ評価のコードベース汚染を防げる

**移行ルール（v1.2 で正式版）**:

- マイグレーション V11.034 で **(a) `budget_configs` 拡張** と **(b) `role_assignments` 一括 INSERT による BUDGET_ADMIN 自動付与** を一括実施（責務統合、§5.7 / §m-iii 参照）
- Service 層・Controller 層の権限チェックは `hasPermission("BUDGET_ADMIN")` のみで判定。`MANAGE_SHIFTS` + `BUDGET_VIEW` の OR ロジックは **コードベースに残さない**
- §9.3 の `resolveView()` 等のサンプルコードも v1.2 で BUDGET_ADMIN 単独判定に書き換え済み
- マイグレーション後の検証として、§14.1 ユニットテストに「移行マイグレーション動作テスト」を追加（v1.2）

- `MANAGE_SHIFTS` / `MANAGE_TODO` は F03.5 / F02.3 で既定義（変更なし）
- **新規追加権限**: `BUDGET_ADMIN`、`VIEW_OWN_HOURLY_RATE` の 2 種
- **v1.2 で確定**: 新規組織・新規ユーザーには **必ず `BUDGET_ADMIN` を直接付与** する運用とし、`MANAGE_SHIFTS` + `BUDGET_VIEW` の組み合わせで F08.7 を運用してはならない（OR 後方互換ロジックが廃止されているため動作しない）

### 8.4 ワークフロー承認者の指定方式

F05.6 の予算超過承認ワークフローについて:

- **承認者は組織が `budget_configs.over_limit_workflow_id` で指定するワークフローテンプレートに従う**
- ワークフローテンプレート側でロール条件（例: `BUDGET_ADMIN` 保有者の中から指定 N 名）を定義する
- F08.7 設計書では承認者ロジックを直接記述せず、F05.6 の責務に委ねる
- 組織が `over_limit_workflow_id` を未設定の場合、警告通知のみ送信（承認ワークフローはスキップ）+ 監査ログに `WORKFLOW_NOT_CONFIGURED` を記録

### 8.5 `budget_configs` テーブルへの新カラム追加（既存 F08.6）

- `over_limit_workflow_id BIGINT UNSIGNED NULL FK workflow_definitions(id) ON DELETE SET NULL` — 100% 超過時に起動するワークフロー定義
- `shift_budget_enabled BOOLEAN NULL DEFAULT NULL` — 組織別フィーチャーフラグの **三値論理**（v1.1 確定。詳細 §13）:
  - `NULL`: 組織未設定 → グローバル既定値 `feature.shiftBudget.enabled` を継承
  - `FALSE`: 明示的にオプトアウト
  - `TRUE`: 明示的に有効化

### 9.3 時給情報の漏洩防止

**実装方式 — Jackson `@JsonView` 採用に確定（v1.1）**:

`@JsonView` を採用する理由:
- DTO クラスを 1 つに保てる（権限ごとの DTO 分離より保守性が高い）
- Spring MVC の `@JsonView` アノテーションで Controller メソッド単位に切替可能
- フィールド単位のマスキング粒度を細かく制御できる

**View クラス定義例**:

```java
// org.mannschaft.shiftbudget.view.BudgetView
public final class BudgetView {
    /** 一般ユーザー向け（金額・時給を全てマスク） */
    public interface Public {}
    /** BUDGET_VIEW 保有者向け（金額は見えるが個人別時給は見えない） */
    public interface BudgetViewer extends Public {}
    /** BUDGET_ADMIN 保有者向け（個人別時給まで全公開）— v1.2 クリーンカットで MANAGE_SHIFTS+BUDGET_VIEW の OR 廃止 */
    public interface BudgetAdmin extends BudgetViewer {}
}
```

**DTO 例**:

```java
public class ConsumptionSummaryDto {
    @JsonView(BudgetView.Public.class)
    private Long allocationId;

    @JsonView(BudgetView.BudgetViewer.class)
    private BigDecimal allocatedAmount;

    @JsonView(BudgetView.BudgetViewer.class)
    private BigDecimal consumedAmount;

    @JsonView(BudgetView.BudgetAdmin.class)
    private List<UserConsumptionDto> byUser;  // 個人別時給は ADMIN のみ

    @JsonView(BudgetView.Public.class)
    private String status;  // BY_USER_HIDDEN を含めうる
}
```

**Controller での切替（権限チェック関数のサンプル）**:

```java
@GetMapping("/allocations/{id}/consumption-summary")
public MappingJacksonValue getSummary(@PathVariable Long id, @AuthenticationPrincipal CurrentUser user) {
    ConsumptionSummaryDto dto = service.getSummary(id, user.getOrganizationId());
    MappingJacksonValue wrapper = new MappingJacksonValue(dto);
    wrapper.setSerializationView(resolveView(user));
    return wrapper;
}

/**
 * ユーザー権限から最も詳細な View を解決する。
 * 上位 View ほど多くのフィールドを公開する（継承関係: Public ⊂ BudgetViewer ⊂ BudgetAdmin）。
 *
 * v1.2 クリーンカット方式: BUDGET_ADMIN 単独判定。MANAGE_SHIFTS+BUDGET_VIEW の OR 後方互換ロジックは廃止。
 * 既存ユーザーへの BUDGET_ADMIN 自動付与は V11.034 マイグレーションで完了済。
 */
private Class<?> resolveView(CurrentUser user) {
    if (user.hasPermission("BUDGET_ADMIN")) {
        return BudgetView.BudgetAdmin.class;
    }
    if (user.hasPermission("BUDGET_VIEW")) {
        return BudgetView.BudgetViewer.class;
    }
    return BudgetView.Public.class;
}
```

**集計 API の `by_user` 配列の権限なし時挙動 — v1.2 確定（DTO 形状を一意に固定）**:

権限の有無に関わらず、レスポンスの DTO 形状を**完全に同一**に保つことで、フロント側の `null` チェックや「フィールド存在判定」を不要にする。具体ルール:

| ユーザー権限 | `byUser` フィールド | `status` フィールド | フロントの扱い |
|--------------|--------------------|--------------------|---------------|
| `BUDGET_ADMIN` または `MANAGE_SHIFTS`+`BUDGET_VIEW` | 実データ配列（例: `[{userId:5, amount:80000, hours:66.67}]`） | `OK` / `WARN` / `EXCEEDED` / `SEVERE_EXCEEDED` | そのまま表示 |
| `BUDGET_VIEW` のみ（個人別不可視） | **空配列 `[]`**（必ず空配列、`null` ではない） | 上記ステータスに加えて `BY_USER_HIDDEN` フラグを併設 | バナー「権限がないため個人別内訳は非表示」を表示 |
| `BUDGET_VIEW` も無し | 上位 View に到達しないため `byUser` フィールド自体が**シリアライズ除外**されるが、Service 層で `byUser = Collections.emptyList()` を必ず初期化してから View 解決に渡す（DTO 内部状態を `null` のまま渡さない） | `BY_USER_HIDDEN` を付加 | 同上 |

**設計上の確定事項**:

- **`null` は使わない**: フロント側は `byUser` を必ず配列として扱える（`byUser?.length` のような optional chaining は不要）
- **`status` は配列ではなく文字列カンマ区切り**: 既存仕様通り `OK` / `WARN` / `EXCEEDED` / `SEVERE_EXCEEDED` に加え、ハイブリッドで `BY_USER_HIDDEN` を含める場合は別フィールド `flags: ["BY_USER_HIDDEN"]` を追加（v1.2 で `flags: string[]` を DTO に新設）
- **「空配列」と「ステータスフラグ」は必ず併設**: 権限なし時に `byUser=[]` のみで `flags` に `BY_USER_HIDDEN` が無い、という状態は禁止（フロントが「単に消化記録ゼロ」と誤認するため）

**v1.2 で確定した DTO 形状**:

```java
public class ConsumptionSummaryDto {
    @JsonView(BudgetView.Public.class)
    private Long allocationId;

    @JsonView(BudgetView.BudgetViewer.class)
    private BigDecimal allocatedAmount;

    @JsonView(BudgetView.BudgetViewer.class)
    private BigDecimal consumedAmount;

    @JsonView(BudgetView.BudgetAdmin.class)
    private List<UserConsumptionDto> byUser = Collections.emptyList();  // v1.2: 必ず空配列で初期化、null 不可

    @JsonView(BudgetView.Public.class)
    private String status;  // OK / WARN / EXCEEDED / SEVERE_EXCEEDED

    @JsonView(BudgetView.Public.class)
    private List<String> flags = new ArrayList<>();  // v1.2 追加: BY_USER_HIDDEN 等
}
```

- 自分自身の時給だけは `VIEW_OWN_HOURLY_RATE` 権限（v1.1 新規、§8）で別エンドポイント `/shift-budget/me/hourly-rate` から取得可能

**その他**:

- ログ出力（`logger.info`）に時給数値を出力しない（DTO の `toString()` でマスキング）
- アクセスログには `hourly_rate_snapshot` を含めない（リクエスト/レスポンスダンプから除外設定）

### 9.4 改竄防止

- `shift_budget_consumptions` は **物理削除禁止**。CANCEL のみ
- `status = CONFIRMED` 後は `amount` / `hours` / `hourly_rate_snapshot` の UPDATE を Service 層で禁止（IllegalStateException）
- 月次締め後の割当 UPDATE は ADMIN のみ + 監査ログ必須

### 9.5 多テナント分離

- 全クエリに `WHERE organization_id = ?` を強制（Hibernate `@Filter` または `BaseRepository` の override）
- 検索時、`X-Organization-Id` ヘッダと `current_user` の所属組織が一致しない場合は 403
- TODO 紐付時、`projects.scope_id` または `todos` 経由のチーム/組織が `allocation.organization_id` と一致することを検証

## 11. エッジケース・例外処理

| ケース | 設計上の決着 |
|--------|--------------|
| **月跨ぎシフト（深夜跨ぎ）** | `slot_date` が属する月に全額計上。22:00-翌03:00 のスロットは開始日の月に 5 時間まとめて計上 |
| **月跨ぎシフト（複数日連続）** | F03.5 の現状は 1 スロット = 同日内のため、複数日連続スロットは存在しない。将来追加された場合は本書 §15 ではなくこのケースに追加 |
| **予算ゼロ円の割当** | 作成自体は許可。逆算 API は `required_slots = 0` + `warning: BUDGET_ZERO`。消化発生時は即 100% 超過 → 警告発火 |
| **時給未設定メンバーがアサインされた場合** | 消化記録は INSERT する。`hourly_rate_snapshot = 0`、`amount = 0`、`status = PLANNED`。集計時に `warning: USER_HAS_NO_RATE` を含めて UI に「時給未設定: N 名」と表示。給与概算は 0 円扱い（F03.5 の既存挙動と整合） |
| **多通貨混在** | Phase 9 では `currency = 'JPY'` 固定。CHECK 制約は付けず、将来拡張余地として `currency` カラムだけ用意。同一割当内で異なる通貨が混在する設計は本書スコープ外 |
| **シフトキャンセル時の消化レコード逆仕訳** | `shift_schedules` の論理削除や `shift_slots` の `assigned_user_ids` 配列要素削除を `ShiftCancelledEvent` で受信し、対応する `shift_budget_consumptions` を `status = CANCELLED` + `cancel_reason = 'USER_UNASSIGNED'` に遷移。`allocation.consumed_amount` をアトミックに減算 |
| **予算超過してもシフト確定可能** | UC-5 の通り、シフト公開操作はブロックしない。警告 + 監査ログ + 承認ワークフロー起動のみ。運用判断（緊急対応・特例承認）を妨げない |
| **削除されたメンバーの過去消化レコード** | `shift_budget_consumptions.user_id` の FK は `ON DELETE RESTRICT`。ユーザーが論理削除（F00 既存）された場合は `users.deleted_at` で判別。集計時に「退会者: N 円」として括る |
| **同一スロットに複数メンバーがアサイン** | §4.2 の通り按分しない。各メンバー分を 1 行ずつ INSERT。理由: 全員が同時間働く前提のため |
| **同一 (slot, user) に複数回 INSERT を試行（公開を取り消して再公開）** | UNIQUE `(slot_id, user_id, status)` で衝突 → 409 ではなく Service 層で「既存 PLANNED を CANCELLED に遷移してから新規 INSERT」する更新ロジック（v1.1: 擬似コード §11.1 参照） |
| **月次締め後の遅延キャンセル** | CONFIRMED 状態の消化は UPDATE 不可。代わりに `cancel_reason = 'POST_CLOSE_REVERSE'` の **新規 CANCELLED 行** を INSERT し、`allocation.confirmed_amount` から差し引く（取消仕訳パターン）。F08.6 `budget_transactions` にも対応する取消仕訳を起こす |
| **`allocation` 削除を試行（消化レコードあり）** | API で 409 Conflict + `error: HAS_CONSUMPTIONS`。消化を全て CANCEL してから再試行を促す |
| **TODO/プロジェクトが論理削除された** | `todo_budget_links.project_id` / `todo_id` は `ON DELETE CASCADE`。物理削除は通常起きない（F02.3 は論理削除）が、復元時の再紐付は別途運用 |
| **割当の `period_end` が過去日になる UPDATE** | 許可（過去予算の修正用）。ただし監査ログ必須 + 承認応答時に再評価しない |
| **フィーチャーフラグ OFF 時の API 呼出** | 全 F08.7 API は 503 Service Unavailable + `error: FEATURE_DISABLED` を返却。既存データ（過去に有効化していた場合）は閲覧のみ可能 |
| **「1スロット = 当日中」の限界（v1.1 明記）** | F03.5 / F08.7 の前提として **1 つの shift_slot は同一日内で完結** する。月跨ぎは深夜跨ぎ（22:00-翌03:00 など）に限り許可し、§4.6 の按分ルール（開始日属する月へ全額計上）で処理する。**複数日連続の連泊・夜勤連続スロットは F03.5 のスキーマで表現できないため、本書のスコープ外**。Phase 10 以降で F03.5 拡張時に再検討 |
| **同一スコープで論理削除→再作成（v1.2 追加）** | UNIQUE `uq_sba_scope_category_period` に `deleted_at` を含めているため許可される（§5.2 設計意図参照）。具体フロー: (1) 経理ミスにより 2026 年 6 月人件費割当を一旦論理削除（`deleted_at` セット）、(2) 翌日同一スコープ（org × team × project × category × period）で新規割当を作成、(3) 監査ログ `SHIFT_BUDGET_ALLOCATION_CREATED` に `replaces_allocation_id`（旧レコード ID）を含めて履歴連鎖を保持。Service 層は新規作成前に `SELECT ... FOR UPDATE` で生存レコード重複を排除する。論理削除済みレコードが 10 件を超えた場合は運用警告ログを出力（§5.2 運用注意参照） |

### 11.1 同一 (slot, user) 再 INSERT のサービス層擬似コード（v1.1 追記）

```java
// org.mannschaft.shiftbudget.service.ConsumptionService
@Transactional
public void recordConsumption(ShiftPublishedEvent event) {
    for (ShiftSlot slot : event.getSlots()) {
        for (Long userId : slot.getAssignedUserIds()) {
            BigDecimal hourlyRate = hourlyRateRepo.findCurrentRate(userId, slot.getOrgId())
                .orElse(BigDecimal.ZERO);
            BigDecimal hours = calcHours(slot.getStartTime(), slot.getEndTime());

            // (1) 同一 (slot_id, user_id) の PLANNED を CANCELLED に遷移
            consumptionRepo.findBySlotAndUserAndStatus(slot.getId(), userId, "PLANNED")
                .ifPresent(existing -> {
                    existing.setStatus("CANCELLED");
                    existing.setCancelledAt(LocalDateTime.now());
                    existing.setCancelReason("RE_PUBLISHED");
                    consumptionRepo.save(existing);
                    // allocation.consumed_amount から差し引き
                    allocationRepo.decrementConsumed(existing.getAllocationId(), existing.getAmount());
                });

            // (2) CONFIRMED が既に存在する場合は再 INSERT を拒否（月次締め後の再公開は不可）
            if (consumptionRepo.existsBySlotAndUserAndStatus(slot.getId(), userId, "CONFIRMED")) {
                throw new IllegalStateException(
                    "Cannot re-record consumption: CONFIRMED record exists for slot=" + slot.getId() + ", user=" + userId);
            }

            // (3) 新規 PLANNED を INSERT
            ShiftBudgetConsumption fresh = new ShiftBudgetConsumption();
            fresh.setAllocationId(resolveAllocationId(slot, event.getMonth()));
            fresh.setShiftId(event.getShiftScheduleId());
            fresh.setSlotId(slot.getId());
            fresh.setUserId(userId);
            fresh.setHourlyRateSnapshot(hourlyRate);
            fresh.setHours(hours);
            fresh.setAmount(hourlyRate.multiply(hours));
            fresh.setStatus("PLANNED");
            fresh.setRecordedAt(LocalDateTime.now());
            consumptionRepo.save(fresh);

            // (4) allocation.consumed_amount に加算
            allocationRepo.incrementConsumed(fresh.getAllocationId(), fresh.getAmount());

            // (5) 閾値判定 → 必要なら警告発火（重複防止 §4.5 の UNIQUE 制約に依拠）
            thresholdAlertService.evaluateAndTrigger(fresh.getAllocationId());
        }
    }
}
```

**設計上のポイント**:
- (1) で UNIQUE 制約衝突を未然に防ぐ（CANCELLED 遷移後は status が異なるため新規 INSERT が通る）
- (2) で月次締め後の再公開を物理的に禁止（経理整合性を守る）
- (4) で `consumed_amount` の二重加算を防ぐため、(1) で減算してから (4) で加算する順序を厳守
- 全体を `@Transactional` で囲み、途中失敗時に整合性を保つ

**Phase 10-α 詳細（F05.6 ワークフロー本格起動）**:

Phase 9-δ では `tryStartWorkflow` が「設定済の場合は監査ログ出力 + TODO 残置」のみだった。Phase 10-α で本格起動を実装:

- `budget_configs.over_limit_workflow_id` が設定済 → `WorkflowRequestService.createRequest()` + `submitRequest()` を呼出し、`workflow_requests.id` を `budget_threshold_alerts.workflow_request_id` に書戻
- 未設定 → 既存通り `WORKFLOW_NOT_CONFIGURED` 監査ログ + 警告通知のみ（挙動維持）
- 起動例外 → try-catch で握りつぶし（main の警告発火トランザクションを巻き戻さない、9-δ AFTER_COMMIT パターン踏襲）+ ERROR ログ + `WORKFLOW_START_FAILED` 監査ログ
- 起動申請者 (`workflow_requests.requested_by`) はシステム自動起動のため `null`（カラム NULL 許容で対応済）
- `sourceType = "BUDGET_THRESHOLD_ALERT"` / `sourceId = alert.id` でワークフロー側から警告レコードを逆引き可能
- DDL: V11.037 で `budget_threshold_alerts.workflow_request_id` への FK (`workflow_requests.id`, `ON DELETE SET NULL`) を追加（V11.033 でカラムは既存、Phase 9-δ では FK 保留だった）

**Phase 10-β 詳細（通知失敗リトライ + 月次締め部分失敗リカバリ）**:

9-δ 第2段で AFTER_COMMIT hook の swallow パターンが残しがちだったサイレント失敗を、専用テーブル + リトライバッチ + 管理 API で根治する。

- **新規 DDL（V11.038）**: `shift_budget_failed_events`（`event_type` / `source_id` / `payload`(JSON) / `error_message` / `retry_count` / `status` / `last_retried_at`）。
  - INDEX: `idx_sbfe_org_status (organization_id, status)`、`idx_sbfe_pending (status, last_retried_at)`
  - CHECK: `status IN ('PENDING','RETRYING','SUCCEEDED','EXHAUSTED','MANUAL_RESOLVED')`、`event_type IN ('CONSUMPTION_RECORD','CONSUMPTION_CANCEL','THRESHOLD_ALERT','WORKFLOW_START','NOTIFICATION_SEND')`
- **失敗イベント記録の入口**: 既存 hook の swallow 箇所に `failedEventService.recordFailure(...)` を追加。`recordFailure` 自体は `REQUIRES_NEW` で動き、メインの hook 例外伝播から独立する
  - `ShiftBudgetConsumptionRecordListener` の致命的失敗 + 閾値判定失敗
  - `ShiftBudgetConsumptionCancelListener` の致命的失敗 + 閾値再評価失敗
  - `ThresholdAlertEvaluationService.tryStartWorkflow` の WORKFLOW_START_FAILED
  - `ThresholdAlertEvaluationService.sendNotifications` の NotificationHelper.notifyAll 外周失敗
- **リトライバッチ**（`ShiftBudgetRetryBatchJob`）: `@Scheduled(cron = "0 */15 * * * ?", zone = "Asia/Tokyo")` で 15 分毎に PENDING/RETRYING を再実行、`retry_count < 3` まで継続、3 回失敗で EXHAUSTED に固定。ShedLock で複数インスタンス重複実行を防止
  - `feature.shift-budget.retry-batch-enabled` プロパティ（default false）で OFF/ON
  - 1 バッチ最大 100 件 + 直近 10 分以内に試行済の行はスキップ（バックオフ）
  - `CONSUMPTION_RECORD` / `CONSUMPTION_CANCEL` は誤再実行で重複 INSERT を起こすリスクが高いため、自動再実行は対象外（即 EXHAUSTED 化 → 運用者が DB 直接補正 + 管理 API でマーク）
  - `THRESHOLD_ALERT` / `WORKFLOW_START` は `ThresholdAlertEvaluationService.evaluateAndTrigger` を呼び直し、UNIQUE 制約で冪等
  - `NOTIFICATION_SEND` は `NotificationHelper.notifyAll` を payload から再構築して呼び直し
- **管理 API**: `/api/v1/shift-budget/failed-events` (GET=一覧 BUDGET_VIEW、POST /{id}/retry=個別再実行 BUDGET_ADMIN、POST /{id}/resolve=手動補正済マーク BUDGET_ADMIN)。`X-Organization-Id` ヘッダで多テナント分離強制、別組織 ID は IDOR 対策で 404
- **月次締め部分失敗リカバリ**: `MonthlyShiftBudgetCloseService.closeAll(YearMonth)` を新規追加（既存 `close` / `closeFromBatch` は不変）。全組織横断で締め、組織単位で try-catch + failed_events 記録 + 続行。戻り値 `AllOrgsCloseResult` で `processedOrganizationIds` / `failedOrganizationIds` / `alreadyClosedOrganizationIds` を返却。`MonthlyShiftBudgetCloseBatchJob` を `closeAll` 経路に切替、API #11 レスポンス DTO に同フィールドを追加（単一組織 close でも互換のためフィールド自体は存在）
- **監査ログ**: `FAILED_EVENT_RETRIED` (個別再実行時)、`FAILED_EVENT_RESOLVED` (手動補正済マーク時)
- **運用 Runbook**: `docs/operations/f087_monthly_close_recovery.md` を新規作成（月次締め失敗時の復旧手順）

|---|---|---|
| `false` | （任意） | **OFF**（強制無効。グローバル不許可は組織でも上書き不可） |
| `true` | `NULL`（未設定） | **ON**（グローバル既定値を継承） |
| `true` | `FALSE`（明示的オプトアウト） | **OFF**（組織が明示的に無効化） |
| `true` | `TRUE`（明示的有効化） | **ON** |

**カラム定義**:
- `budget_configs.shift_budget_enabled BOOLEAN DEFAULT NULL`（NOT NULL ではない）
- 組織が API でフラグを切替時、`null` / `true` / `false` の 3 値を送信可能
- フロントの管理画面では「グローバル設定を継承」「明示的に有効」「明示的に無効」のラジオ 3 択

**判定ロジック**:

```java
public boolean isFeatureEnabled(Long organizationId) {
    if (!globalProperties.shiftBudgetEnabled()) {
        return false;  // グローバルが OFF なら全組織 OFF
    }
    Boolean orgFlag = budgetConfigsRepo.findByOrgId(organizationId)
        .map(BudgetConfig::getShiftBudgetEnabled)
        .orElse(null);
    if (orgFlag == null) {
        return true;  // 組織未設定 → グローバル既定値（ON）を継承
    }
    return orgFlag;  // 明示値（TRUE/FALSE）に従う
}
```

ロールバック方針:
- フィーチャーフラグを OFF にしても既存データは保持。再 ON で復活
- DDL ロールバック用の `Vundo_FUTURE_F087__*.sql` も同時に用意（実装フェーズで作成）
