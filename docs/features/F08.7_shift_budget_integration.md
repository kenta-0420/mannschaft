## 1. 概要

### 1.1 背景・目的

TODO 一覧 #38 として記録された運用要望は次のとおり:

> シフトと連動した予算管理機能。追記の予算を決め、時給などを考慮してシフト作成できるといい。TODO は予算を決めて TODO を決められる？

文面は具体性に乏しいが、ヒアリングで意図を分解すると次の三層になる:

1. **「予算を決めてシフトを組む」** — 月次の人件費予算が先にあり、そこから「平均時給」「1 スロット時間」「必要枠数」を逆算したい。シフト確定後に予算超過していたら気づきたい
2. **「シフト確定 → 予算消化」** — シフトが公開された段階で、F08.6 の費目（人件費）から自動的に予算を引き当て、消化率をリアルタイムに見たい
3. **「TODO/プロジェクトに予算をつける」** — 「文化祭 30 万円まで」「夏祭りスタッフ人件費 5 万円まで」のように、TODO（プロジェクト）の単位でも予算枠を持ちたい

これら三層は既存機能（F03.5 シフト・F08.6 予算・F02.3 TODO）の境界をまたぐため、各設計書に追補するのではなく **F08.7 として独立した連携層** を立て、3 機能の間に位置する「予算割当 → 消化 → 紐付け → 警告」という一貫したフローを定義する。

### 1.2 解決する課題

1. **予算策定とシフト編成の分断** — 現状 F08.6 で年間予算を立てても、それがシフト編成画面に伝わらず、管理者は別画面・別エクセルで突合している
2. **人件費の事後発覚** — 月末に F08.6 の集計を見て初めて予算超過が判明する。事前警告がない
3. **TODO ベースの予算管理ができない** — 「このイベントに使える予算」を TODO/プロジェクト単位で見える化する手段がない

### 1.3 非目標

本機能は次を **扱わない**:

- **給与計算そのもの** — 源泉徴収・社会保険料・所得税・年末調整等は対象外。あくまで「予算管理目的の概算」に留める
- **税務処理・確定申告書類の生成** — F08.6 の年次報告書（既存）に委ねる
- **タイムカード/打刻による実労働時間集計** — F03.1 出欠管理の領域。本機能はシフト枠（計画値）を基に予算消化を計算する。実績差分は将来拡張（Phase 10 以降）
- **複数通貨の同時運用** — Phase 9 では単一通貨（既存 F08.6 と同様、日本円固定）。多通貨対応は将来拡張
- **AI による最適シフト提案** — 予算制約を満たすシフトの自動生成は本機能のスコープ外。既存 F03.5 の自動割当に「予算上限」を **入力パラメータ** として渡すだけに留める

## 2. スコープ

### 2.1 含むもの

- 予算→必要シフト枠数の **逆算計算 API**（数式提供）
- シフトスロット ↔ 予算費目の **割当テーブル**（`shift_budget_allocations`）
- シフト確定時の **消化記録**（`shift_budget_consumptions`、計画値ベース）
- TODO/プロジェクト → 予算割当の **紐付テーブル**（`todo_budget_links`）
- リアルタイム消化率の **集計 API**
- 80% / 100% / 超過の **閾値警告**（`budget_threshold_alerts`、F04.3 通知連携）
- 月跨ぎの **按分ルール**（深夜跨ぎスロットを含む）
- シフトキャンセル時の **逆仕訳**
- 監査ログ（F10.3）への記録

### 2.2 含まないもの

- 既存 `budget_allocations`（年度×費目で 1 レコード）の置換 — F08.7 は **シフト期間（月単位）** で割り当てる別概念。年度予算配分の上に乗る「月割り内訳」として位置づける
- 既存 `budget_transactions` への自動仕訳作成 — 消化記録は F08.7 専用テーブルに保持し、F08.6 トランザクション帳へは「月次締め確定時」にまとめて反映する（v1.0 時点では別画面で確認、自動連携は Phase 9-δ 末尾で評価）
- 給与明細・賞与・退職金管理

### 2.3 既存機能との境界線

| 機能 | 担当範囲 | F08.7 から見た境界 |
|------|----------|--------------------|
| F03.5 シフト管理 | スロット定義、希望収集、自動割当、確定、公開 | 確定/公開された `shift_slots.assigned_user_ids` を入力として読み取る |
| F08.6 予算会計 | 年度・費目・配分・収支トランザクション・月次/年次報告書 | `budget_categories.id` と `budget_fiscal_years.id` を参照する。逆向きに F08.6 のトランザクション帳に書き戻すのは月次締めバッチのみ |
| F02.3 TODO/プロジェクト | TODO・プロジェクト・進捗率・マイルストーン | `projects.id` または `todos.id` を参照する。TODO 側の進捗率算出には影響しない |
| F02.5 シフト→TODO 連動 | シフト公開時の Todo 自動生成 | 連動 TODO に対しても予算紐付けを許容する（特別扱いしない） |
| F05.6 ワークフロー承認 | 承認ルート・閾値・通知 | 予算超過警告の段階に応じて承認リクエストを起動する |

## 3. ユースケース

### UC-1: 予算上限から必要シフト枠数を逆算する

**アクター**: ADMIN / DEPUTY_ADMIN（`MANAGE_SHIFTS` + `BUDGET_VIEW`）
**事前条件**: 当月の人件費予算が `budget_allocations` に登録済み（例: 30 万円）。チームメンバーの時給が `shift_hourly_rates` に登録済み

**フロー**:
1. ADMIN がシフト作成画面を開き、対象月（例: 2026 年 6 月）を選択
2. 「予算逆算」ボタンを押下
3. システムは `POST /api/v1/shift-budget/calc/required-slots` に当月の `budget_amount` と `slot_hours`（既定: 4 時間）を送信
4. サーバは `shift_hourly_rates` の現行平均時給を計算し、必要枠数を返却:
   `必要枠数 = floor(予算額 / (平均時給 × 1スロット時間))`
5. UI に「予算 30 万円 / 平均時給 1,200 円 / 4 時間スロット → 最大 62 枠」と表示
6. ADMIN はこの上限を参考にシフト編成を進める

**事後条件**: 計算結果はステートレス（DB 書き込みなし）。割当を確定したい場合は UC-2 に進む。

### UC-2: シフト確定時に人件費見込を予算費目から消化する

**アクター**: ADMIN / DEPUTY_ADMIN
**事前条件**: 当月の `shift_budget_allocations` レコードが作成済み（費目=人件費、期間=2026-06-01〜2026-06-30、配分額=30 万円）

**フロー**:
1. シフトスケジュールを `ADJUSTING → PUBLISHED` に遷移（F03.5 §5）
2. システムは公開対象スケジュール内の全 `shift_slots.assigned_user_ids` を走査し、各 (slot, user) について `shift_hourly_rates` から適用時給を取得
3. `shift_budget_consumptions` に `status = PLANNED` で 1 行ずつ INSERT（`hourly_rate_snapshot`、`hours`、`amount` を保存）
4. 同時に `shift_budget_allocations` の集計値を更新（非正規化キャッシュ `consumed_amount`）
5. 公開イベント（F03.5 既存）を発火し、UI ダッシュボードに反映
6. 消化率が閾値を超えていれば UC-5 を発火

**事後条件**: 消化記録は `PLANNED`。月次締めで `CONFIRMED` に昇格する。

### UC-3: TODO（プロジェクト・タスク）に予算枠を紐付け、進捗とともに消化を可視化する

**アクター**: ADMIN / DEPUTY_ADMIN（`MANAGE_TODO` + `BUDGET_VIEW`）
**事前条件**: プロジェクト「文化祭 2026」が登録済み。当該プロジェクトに紐付く予算割当（人件費 5 万円、消耗品費 3 万円）を立てたい

**フロー**:
1. プロジェクト編集画面で「予算」タブを開く
2. `POST /api/v1/todo-budget/links` で `project_id` と `allocation_id`、`link_amount=50000` を送信
3. プロジェクト詳細画面に「予算 5 万円 / 消化 1.2 万円 / 残 3.8 万円（24%）」と表示
4. プロジェクトに紐付く F03.5 シフトの消化が進むと、紐付分が自動的に集計される
5. プロジェクト完了時には残予算と実消化額を比較したサマリーが報告書に出る

**事後条件**: TODO 単位の紐付けも可能だが、UI 主導線はプロジェクト単位。

### UC-4: 月跨ぎ・締め処理時の予算精算

**アクター**: ADMIN（経理担当）
**事前条件**: 月末を迎え、当月の `shift_budget_consumptions` がすべて `PLANNED` 状態

**フロー**:
1. 月次締めバッチ（手動起動 or cron）が起動
2. 当月の `PLANNED` レコードを `CONFIRMED` に昇格
3. 同時に F08.6 `budget_transactions` に「シフト人件費 月次集計」として 1 件にまとめて INSERT（`source_type = 'SHIFT_BUDGET_MONTHLY'`、`source_id = allocation_id`）
4. 翌月の `shift_budget_allocations` を自動コピー（金額・費目を引き継ぎ。ADMIN が後から微調整可）

**事後条件**: 当月の予算は確定。以降キャンセル等の修正は **取消仕訳** で記録する（F08.6 のパターンに準拠）。

### UC-5: 予算超過警告と承認フロー連携

**アクター**: ADMIN（受信者）、予算管理者（承認者）
**事前条件**: 消化率が 80% を初めて超えた

**フロー**:
1. UC-2 内のチェックで閾値超過を検知
2. `budget_threshold_alerts` に 1 件 INSERT（`threshold_percent=80`、`triggered_at=now()`）
3. F04.3 プッシュ通知で ADMIN/予算管理者に通知（i18n キー `shiftBudget.alert.warn80`）
4. 100% 到達時は別レコード（`threshold_percent=100`）と承認ワークフロー起動（F05.6）
5. 100% 超過してもシフト確定操作はブロックされない（運用判断を妨げない）。ただし監査ログに `BUDGET_OVER_LIMIT_PROCEED` が記録され、承認待ちフラグが立つ
6. 予算管理者が `POST /api/v1/shift-budget/alerts/{id}/acknowledge` で確認応答すると `acknowledged_at` がセットされる

**事後条件**: 同一 allocation × 同一閾値で重複通知しないよう、`UNIQUE (allocation_id, threshold_percent)` 制約を貼る。

## 4. 機能要件

### 4.1 予算→シフト枠の逆算機能

**数式**:

```
必要枠数 = floor(予算額 / (平均時給 × 1スロット時間))
```

- `予算額`: `shift_budget_allocations.allocated_amount`（円）
- `平均時給`: 後述のパラメータで決定
- `1スロット時間`: 既定 4 時間（API パラメータで上書き可能、最小 0.25h、最大 24h）

**平均時給算出パラメータ** — リクエストボディで以下のいずれかを指定:

| `mode` | 説明 | 計算式 |
|--------|------|--------|
| `MEMBER_AVG`（既定） | 対象チームの全アクティブメンバーの現行時給を単純平均 | `SUM(hourly_rate) / COUNT(user_id)` |
| `POSITION_AVG` | ポジション（役割）別に平均時給を出し、必要人数で重み付け | Σ(position_avg × required_count_in_position) ÷ Σ(required_count) |
| `EXPLICIT` | 呼び出し側が `avg_hourly_rate` を直接指定 | リクエストボディの値をそのまま使用 |

**境界**:
- `平均時給 = 0` または `予算額 = 0` の場合 → 必要枠数 0 を返却し、`warning: "AVG_RATE_ZERO"` または `"BUDGET_ZERO"` を併記
- 平均時給未設定メンバーが過半数を占める場合 → `warning: "INSUFFICIENT_RATE_DATA"` を返却（計算は未設定者を除外して継続）
- スロット時間 ≤ 0 → 400 エラー

### 4.2 シフト→予算消化リンク

公開トリガー:
- `shift_schedules.status` が `ADJUSTING → PUBLISHED` に遷移したタイミング
- F03.5 既存の公開イベント（`ShiftPublishedEvent`）に対する `@EventListener` で受け取る

書き込みルール:
- 1 つの `shift_slots` に対し、`assigned_user_ids` の **配列要素ごと** に `shift_budget_consumptions` の 1 行を作成
- `hourly_rate_snapshot` には公開時点で適用される時給を保存（後で時給が変わっても消化額は変動しない）
- `hours` は `end_time - start_time`。深夜跨ぎ（end < start）の場合は + 24 時間

按分ルール（同一スロット複数アサイン時）:
- 各メンバーに **同一の勤務時間** を割当（按分しない）。理由: F03.5 のスロットは「全員が同時に同じ時間働く」前提。ペア勤務 = 2 人分の人件費が発生するのが正

### 4.3 TODO→予算紐付（粒度: プロジェクト単位を主、TODO 単位はオプション）

紐付テーブル `todo_budget_links` の制約:
- `project_id` と `todo_id` は **排他**（CHECK 制約 `chk_tbl_target_xor`）
- 同一 (project_id, allocation_id) は重複禁止
- 同一 (todo_id, allocation_id) は重複禁止
- `link_amount` と `link_percentage` も排他（どちらか一方、または両方 NULL = 全額紐付）

**集計の流れ**:
1. プロジェクトに紐付く F03.5 シフト（`shift_schedules.linked_project_id` が将来的に追加される、または F02.5 経由）の消化額を SUM
2. プロジェクト直下の `todo_budget_links` で指定された割当の消化分も加算
3. `link_amount` 指定時は「上限金額」、`link_percentage` 指定時は「割合按分」、未指定時は「紐付分すべて」

**v1.0 時点での簡略化**:
- F03.5 `shift_schedules` には現状 `linked_project_id` カラムが無い。Phase 9-γ 着手時に追加する FK（`linked_project_id BIGINT UNSIGNED NULL FK projects(id) ON DELETE SET NULL`）を本設計書 §12 で明示する

### 4.4 リアルタイム消化率表示

集計 API:
- `GET /api/v1/shift-budget/allocations/{id}/consumption-summary`
- レスポンスに `allocated_amount`、`consumed_amount`、`consumption_rate`（0.0000〜1.0000+）、`remaining_amount`、`status`（`OK` / `WARN` / `EXCEEDED`）

非正規化:
- `shift_budget_allocations` に `consumed_amount`（DECIMAL(12,0)）を持ち、消化記録 INSERT 時にアトミック更新
- `consumption_rate` は SELECT 時に計算（DB カラム不要）
- 月次締めで `consumed_amount` を `confirmed_amount` に固定（PLANNED→CONFIRMED 昇格時に再計算）

### 4.5 閾値超過時の警告

閾値: 80%、100%、120%（超過マーカー）

通知:
- F04.3 プッシュ通知で対象ロール（ADMIN + 予算管理者）に送信
- 通知本文の i18n キー: `shiftBudget.alert.warn80`、`shiftBudget.alert.exceeded`、`shiftBudget.alert.severeExceeded120`

承認連携:
- 100% 到達時のみ F05.6 ワークフローを自動起動（ワークフロー定義 ID は組織設定 `budget_configs.over_limit_workflow_id` を参照）
- 80% / 120% は通知のみ

冪等性:
- `UNIQUE KEY uq_bta_allocation_threshold (allocation_id, threshold_percent)` で重複通知を防止

### 4.6 月跨ぎ・締め処理

**月跨ぎシフトの按分**:
- 22:00-06:00 のような深夜跨ぎスロットは、`slot_date` ベースで「開始日が属する月」に全額計上する（簡易ルール）
- 月末日 23:00-翌日 03:00 のスロットは、当月 1 時間 + 翌月 3 時間の按分ではなく、当月に 4 時間まとめて計上
- 理由: スロット単位の整合性を優先。Phase 10 で精緻化予定（要件次第）

**締め処理**:
- バッチ `monthlyShiftBudgetCloseJob`（手動起動 or `cron(0 2 1 * ?)`、毎月 1 日 02:00 JST）
- 当月分 `PLANNED` を `CONFIRMED` に一括昇格
- F08.6 `budget_transactions` に月次集計を 1 件 INSERT（`description = '2026年6月 シフト人件費（自動集計）'`）
- 監査ログ `SHIFT_BUDGET_MONTHLY_CLOSED` を記録

## 5. DB 設計

Flyway バージョンは Phase 9 着手時に確定するため、本書では `V_FUTURE_F087__*` の placeholder で記す。

### 5.1 テーブル一覧

| テーブル名 | 役割 | 論理削除 |
|------------|------|----------|
| `shift_budget_allocations` | シフト期間（月）×費目に予算を割当 | あり |
| `shift_budget_consumptions` | シフト確定時の消化記録（計画値→確定値） | なし（`status = CANCELLED` で論理キャンセル） |
| `todo_budget_links` | TODO/プロジェクトへの予算紐付 | なし |
| `budget_threshold_alerts` | 閾値超過警告通知履歴 | なし |

### 5.2 `shift_budget_allocations`

シフト人件費を月単位で予算費目に割り当てるテーブル。F08.6 の `budget_allocations`（年度×費目で 1 行）の細分化として位置づける。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|----------|----|------|------------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `organization_id` | BIGINT UNSIGNED | NO | — | FK → organizations。多テナント分離。ON DELETE CASCADE |
| `team_id` | BIGINT UNSIGNED | YES | NULL | FK → teams。チーム単位で割当する場合のみ。NULL = 組織全体。ON DELETE CASCADE |
| `fiscal_year_id` | BIGINT UNSIGNED | NO | — | FK → budget_fiscal_years。ON DELETE RESTRICT |
| `budget_category_id` | BIGINT UNSIGNED | NO | — | FK → budget_categories。通常は「人件費」配下の費目を指定。ON DELETE RESTRICT |
| `period_start` | DATE | NO | — | 適用開始日（通常は月初） |
| `period_end` | DATE | NO | — | 適用終了日（通常は月末） |
| `allocated_amount` | DECIMAL(12,0) | NO | — | 割当額（円） |
| `consumed_amount` | DECIMAL(12,0) | NO | 0 | 消化額キャッシュ（PLANNED + CONFIRMED の合計） |
| `confirmed_amount` | DECIMAL(12,0) | NO | 0 | 確定済み消化額（CONFIRMED のみ） |
| `currency` | CHAR(3) | NO | 'JPY' | ISO 4217。Phase 9 では JPY 固定 |
| `note` | VARCHAR(500) | YES | NULL | 備考 |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT |
| `version` | BIGINT | NO | 0 | 楽観的ロック |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

### 5.3 `shift_budget_consumptions`

シフト公開時に作成される消化記録。1 レコード = 1 (slot, user) ペア。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|----------|----|------|------------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `allocation_id` | BIGINT UNSIGNED | NO | — | FK → shift_budget_allocations。ON DELETE RESTRICT |
| `shift_id` | BIGINT UNSIGNED | NO | — | FK → shift_schedules。ON DELETE RESTRICT |
| `slot_id` | BIGINT UNSIGNED | NO | — | FK → shift_slots。ON DELETE RESTRICT |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT |
| `hourly_rate_snapshot` | DECIMAL(10,2) | NO | — | 公開時点の適用時給（後の時給改定の影響を受けない） |
| `hours` | DECIMAL(5,2) | NO | — | 勤務時間（深夜跨ぎは + 24h で計算済み） |
| `amount` | DECIMAL(12,2) | NO | — | `hourly_rate_snapshot * hours` |
| `status` | VARCHAR(20) | NO | 'PLANNED' | `PLANNED` / `CONFIRMED` / `CANCELLED` |
| `recorded_at` | DATETIME | NO | CURRENT_TIMESTAMP | 公開イベント発火時刻 |
| `confirmed_at` | DATETIME | YES | NULL | CONFIRMED 昇格時刻（月次締め） |
| `cancelled_at` | DATETIME | YES | NULL | CANCELLED 遷移時刻 |
| `cancel_reason` | VARCHAR(100) | YES | NULL | キャンセル理由（`SHIFT_DELETED`, `SLOT_REMOVED`, `USER_UNASSIGNED`, `MANUAL_REVERSE`） |

**インデックス**:
```sql
INDEX idx_sbc_allocation_status (allocation_id, status)
INDEX idx_sbc_slot (slot_id)
INDEX idx_sbc_user_recorded (user_id, recorded_at)
INDEX idx_sbc_shift (shift_id, status)
UNIQUE KEY uq_sbc_slot_user_status (slot_id, user_id, status)  -- 同一スロット×同一ユーザー×同一状態は重複禁止（CANCELLED は status 違いで再記録可）
```

**制約**:
```sql
CONSTRAINT chk_sbc_amount CHECK (amount >= 0 AND hours >= 0)
CONSTRAINT chk_sbc_status CHECK (status IN ('PLANNED', 'CONFIRMED', 'CANCELLED'))
```

**運用ルール**:
- **物理削除禁止**。誤って書いた場合も `status = CANCELLED` に遷移させる（監査証跡）
- `status = CONFIRMED` 昇格後は `amount` / `hours` / `hourly_rate_snapshot` の UPDATE 不可（アプリ層で禁止、CHECK では表現困難）
- キャンセル時は逆仕訳ではなく status 遷移で表現。`shift_budget_allocations.consumed_amount` から差し引く

## 9. セキュリティ

### 9.1 監査ログ（F10.3）

以下のイベントを `audit_logs` に記録:

| イベント名 | 発火タイミング |
|------------|----------------|
| `SHIFT_BUDGET_ALLOCATION_CREATED` | 割当新規作成時 |
| `SHIFT_BUDGET_ALLOCATION_UPDATED` | 割当更新時（`amount` の変更を `before/after` で記録） |
| `SHIFT_BUDGET_ALLOCATION_DELETED` | 論理削除時 |
| `SHIFT_BUDGET_CONSUMPTION_RECORDED` | 公開時の消化記録一括 INSERT 後（slot_count を記録） |
| `SHIFT_BUDGET_CONSUMPTION_CANCELLED` | 個別消化の CANCELLED 遷移時 |
| `SHIFT_BUDGET_MONTHLY_CLOSED` | 月次締めバッチ完了時 |
| `BUDGET_THRESHOLD_ALERT_TRIGGERED` | 警告発火時 |
| `BUDGET_THRESHOLD_ALERT_ACKNOWLEDGED` | 承認応答時 |
| `BUDGET_OVER_LIMIT_PROCEED` | 100% 超過状態でシフト公開を強行した時 |
| `TODO_BUDGET_LINK_CREATED` / `..._DELETED` | 紐付の作成・削除時 |

### 9.2 楽観的ロック

- `shift_budget_allocations` に `version` カラム必須
- PUT エンドポイントは `If-Match` ヘッダまたはリクエストボディの `version` で検証、不一致時は 409 Conflict

### 9.3 時給情報の漏洩防止

- API レスポンスのシリアライズで `hourly_rate_snapshot` および `amount` フィールドは Jackson の `@JsonView` または DTO 分離で **MANAGE_SHIFTS + BUDGET_VIEW を持たないユーザーには `null` で返却**
- 集計 API（消化サマリの `by_user`）も同条件で自動マスキング
- ログ出力（`logger.info`）に時給数値を出力しない（DTO の `toString()` でマスキング）

### 9.4 改竄防止

- `shift_budget_consumptions` は **物理削除禁止**。CANCEL のみ
- `status = CONFIRMED` 後は `amount` / `hours` / `hourly_rate_snapshot` の UPDATE を Service 層で禁止（IllegalStateException）
- 月次締め後の割当 UPDATE は ADMIN のみ + 監査ログ必須

### 9.5 多テナント分離

- 全クエリに `WHERE organization_id = ?` を強制（Hibernate `@Filter` または `BaseRepository` の override）
- 検索時、`X-Organization-Id` ヘッダと `current_user` の所属組織が一致しない場合は 403
- TODO 紐付時、`projects.scope_id` または `todos` 経由のチーム/組織が `allocation.organization_id` と一致することを検証

## 11. エッジケース・例外処理

| ケース | 設計上の決着 |
|--------|--------------|
| **月跨ぎシフト（深夜跨ぎ）** | `slot_date` が属する月に全額計上。22:00-翌03:00 のスロットは開始日の月に 5 時間まとめて計上 |
| **月跨ぎシフト（複数日連続）** | F03.5 の現状は 1 スロット = 同日内のため、複数日連続スロットは存在しない。将来追加された場合は本書 §15 ではなくこのケースに追加 |
| **予算ゼロ円の割当** | 作成自体は許可。逆算 API は `required_slots = 0` + `warning: BUDGET_ZERO`。消化発生時は即 100% 超過 → 警告発火 |
| **時給未設定メンバーがアサインされた場合** | 消化記録は INSERT する。`hourly_rate_snapshot = 0`、`amount = 0`、`status = PLANNED`。集計時に `warning: USER_HAS_NO_RATE` を含めて UI に「時給未設定: N 名」と表示。給与概算は 0 円扱い（F03.5 の既存挙動と整合） |
| **多通貨混在** | Phase 9 では `currency = 'JPY'` 固定。CHECK 制約は付けず、将来拡張余地として `currency` カラムだけ用意。同一割当内で異なる通貨が混在する設計は本書スコープ外 |
| **シフトキャンセル時の消化レコード逆仕訳** | `shift_schedules` の論理削除や `shift_slots` の `assigned_user_ids` 配列要素削除を `ShiftCancelledEvent` で受信し、対応する `shift_budget_consumptions` を `status = CANCELLED` + `cancel_reason = 'USER_UNASSIGNED'` に遷移。`allocation.consumed_amount` をアトミックに減算 |
| **予算超過してもシフト確定可能** | UC-5 の通り、シフト公開操作はブロックしない。警告 + 監査ログ + 承認ワークフロー起動のみ。運用判断（緊急対応・特例承認）を妨げない |
| **削除されたメンバーの過去消化レコード** | `shift_budget_consumptions.user_id` の FK は `ON DELETE RESTRICT`。ユーザーが論理削除（F00 既存）された場合は `users.deleted_at` で判別。集計時に「退会者: N 円」として括る |
| **同一スロットに複数メンバーがアサイン** | §4.2 の通り按分しない。各メンバー分を 1 行ずつ INSERT。理由: 全員が同時間働く前提のため |
| **同一 (slot, user) に複数回 INSERT を試行（公開を取り消して再公開）** | UNIQUE `(slot_id, user_id, status)` で衝突 → 409 ではなく Service 層で「既存 PLANNED を CANCELLED に遷移してから新規 INSERT」する更新ロジック |
| **月次締め後の遅延キャンセル** | CONFIRMED 状態の消化は UPDATE 不可。代わりに `cancel_reason = 'POST_CLOSE_REVERSE'` の **新規 CANCELLED 行** を INSERT し、`allocation.confirmed_amount` から差し引く（取消仕訳パターン）。F08.6 `budget_transactions` にも対応する取消仕訳を起こす |
| **`allocation` 削除を試行（消化レコードあり）** | API で 409 Conflict + `error: HAS_CONSUMPTIONS`。消化を全て CANCEL してから再試行を促す |
| **TODO/プロジェクトが論理削除された** | `todo_budget_links.project_id` / `todo_id` は `ON DELETE CASCADE`。物理削除は通常起きない（F02.3 は論理削除）が、復元時の再紐付は別途運用 |
| **割当の `period_end` が過去日になる UPDATE** | 許可（過去予算の修正用）。ただし監査ログ必須 + 承認応答時に再評価しない |
| **フィーチャーフラグ OFF 時の API 呼出** | 全 F08.7 API は 503 Service Unavailable + `error: FEATURE_DISABLED` を返却。既存データ（過去に有効化していた場合）は閲覧のみ可能 |

## 12. 既存機能との連携

### 12.1 F03.5 シフト管理

接続点:
- 入力: `shift_schedules.id`、`shift_slots.id`、`shift_slots.assigned_user_ids`、`shift_hourly_rates`
- イベント連携: `ShiftPublishedEvent` / `ShiftCancelledEvent` / `ShiftSlotAssignmentChangedEvent` を購読
- 追加カラム（要 Phase 9-γ）: `shift_schedules.linked_project_id BIGINT UNSIGNED NULL FK projects(id) ON DELETE SET NULL`（プロジェクト単位の集計を可能にする）

### 12.2 F08.6 予算会計

接続点:
- 入力: `budget_fiscal_years.id`、`budget_categories.id`
- 月次締め時に `budget_transactions` へ「シフト人件費 月次集計」の 1 行を INSERT（`source_type = 'SHIFT_BUDGET_MONTHLY'`、`source_id = allocation_id`）
- `budget_configs` を拡張（§8 参照）

### 12.3 F02.3 TODO/プロジェクト

接続点:
- 入力: `projects.id`、`todos.id`
- TODO 進捗率（`projects.progress_rate`）には影響しない（独立指標）

### 12.4 F05.6 ワークフロー承認

接続点:
- 100% 超過時に `workflow_requests` を起動。ワークフロー定義 ID は `budget_configs.over_limit_workflow_id`
- 承認結果を `budget_threshold_alerts.workflow_request_id` 経由で参照

### 12.5 F02.5 シフト→TODO 連動

接続点:
- F02.5 が自動生成する Todo は通常の TODO と同じ扱い。`todo_budget_links` で予算紐付可能
- F02.5 側で生成された Todo の `project_id` が設定されていれば、プロジェクト経由の紐付も継承される（紐付方法は ADMIN の運用判断）

### 12.6 F04.3 プッシュ通知

- 警告通知のチャネル: `BUDGET_THRESHOLD_WARN`、`BUDGET_THRESHOLD_EXCEEDED`、`BUDGET_THRESHOLD_SEVERE`
- 受信ロール: 当該組織の ADMIN + `BUDGET_MANAGE` 保有 DEPUTY_ADMIN

## 14. テスト方針

### 14.1 ユニットテスト

| 対象 | テストケース |
|------|--------------|
| 逆算ロジック | `budget=300000, rate=1200, hours=4 → 62`、`budget=0 → 0 + warning`、`rate=0 → 0 + warning`、`hours=0 → 400` |
| 平均時給計算 | `MEMBER_AVG`（全員設定済 / 一部未設定 / 全員未設定）、`POSITION_AVG`（重み付け検証）、`EXPLICIT`（パススルー） |
| 消化額計算 | 通常スロット、深夜跨ぎ、複数アサイン |
| ステータス遷移 | PLANNED → CONFIRMED、PLANNED → CANCELLED、CONFIRMED → CANCELLED（取消仕訳） |
| 閾値判定 | 79.99%（OK）、80.00%（WARN）、99.99%（WARN）、100.00%（EXCEEDED）、120.00%（SEVERE） |
| 紐付バリデーション | project_id と todo_id の排他、link_amount と link_percentage の排他、percentage 範囲外 |

### 14.2 統合テスト

- 予算→シフト→消化の一連フロー（割当作成 → シフト公開 → 消化サマリ取得）
- 月次締めバッチ（PLANNED → CONFIRMED → F08.6 トランザクション生成）
- シフトキャンセル → 逆仕訳 → consumed_amount 減算
- 警告発火 → F04.3 通知 → 承認応答

### 14.3 E2E テスト（Playwright）

- ADMIN がシフト作成画面で逆算 → 枠数表示 → シフト確定 → ダッシュボードで消化率確認
- 80% 警告発火 → バナー表示 → 確認応答
- 100% 超過 → 承認ワークフロー起動 → 承認後にバッジ消失
- TODO 編集画面で予算紐付 → プロジェクト詳細に消化額表示

### 14.4 多テナント分離テスト

- 組織 A の ADMIN が組織 B の `allocation_id` を URL 直叩き → 403
- 組織 A のシフト公開イベントが組織 B の allocation を消化しない（`organization_id` 一致チェック）

### 14.5 セキュリティテスト

- 一般 MEMBER のレスポンスに他人の `hourly_rate_snapshot` が含まれないこと
- 監査ログに必須イベントが全て記録されること
- フィーチャーフラグ OFF 時に API が 503 を返すこと
