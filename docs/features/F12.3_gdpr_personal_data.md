# F12.3 GDPR / 個人情報管理 設計書

> **作成日**: 2026-04-02
> **ステータス**: 🟢 設計確定
> **実装フェーズ**: Phase 12（最優先）

## 1. 概要

GDPR（EU一般データ保護規則）および日本の個人情報保護法への対応として、ユーザー自身がデータを管理できる機能を実装する。

### 対応する法的義務

| 権利 | GDPR 条項 | 対応機能 |
|---|---|---|
| データポータビリティ権 | 第20条 | データエクスポート |
| 消去権（忘れられる権利） | 第17条 | アカウント削除 + 物理削除バッチ |
| アクセス権 | 第15条 | データエクスポート |
| 同意撤回権 | 第7条 | アカウント削除申請 |

### 既存実装との関係

| 機能 | 既存実装 | 本設計で追加・拡張する範囲 |
|---|---|---|
| 退会（論理削除） | `UserService.requestWithdrawal()` | そのまま活用 |
| 退会取り消し | `UserService.cancelWithdrawal()` | そのまま活用（30日猶予） |
| RefreshToken全失効 | `UserService.revokeAllRefreshTokens()` | そのまま活用 |
| Stripeサブスクリプション解約 | 部分実装（Webhook処理のみ） | **退会時自動キャンセルを追加** |
| データエクスポート | なし | **新規実装** |
| 物理削除バッチ | なし | **新規実装** |
| データ保持ポリシー管理 | なし | **新規実装** |

## 3. データエクスポート

### 3.1 API 仕様

```
POST /api/v1/account/data-export
  Body: { "categories": ["account", "payments", "chat_messages"] }  // 省略時は全カテゴリ
  → パスワードユーザー: 再認証後、非同期でエクスポートジョブを開始
  → OAuthユーザー: OTPをメール送信し 202 Accepted を返す

POST /api/v1/account/data-export/verify-otp
  → OAuthユーザー専用。OTP照合成功後にエクスポートジョブを開始

GET  /api/v1/account/data-export/status
  → 最新のエクスポートジョブの状態・進捗を返す

GET  /api/v1/account/data-export/download
  → 完了済みZIPファイルをダウンロード（署名付きURL、24時間有効）

GET  /api/v1/account/deletion-preview
  → 退会時に削除/匿名化されるデータの件数サマリーを返す（読み取り専用・再認証不要）
```

### 3.2 エクスポート対象データ

| カテゴリキー | カテゴリ名 | データソース | ファイル名 | 形式 |
|---|---|---|---|---|
| `account` | アカウント情報 | `users` | `account.json` | JSON |
| `oauth` | OAuth連携 | `oauth_accounts` | `oauth_accounts.json` | JSON |
| `memberships` | 所属チーム・組織 | `team_org_memberships` | `memberships.json` | JSON |
| `profiles` | メンバープロフィール | `member_profiles` | `profiles.json` | JSON |
| `payments` | 決済履歴 | `member_payments`, `stripe_customers` | `payments.json` | JSON |
| `charts` | カルテ（共有済み） | `chart_records` (isSharedToCustomer=true) | `charts.json` | JSON |
| `files` | ファイル共有 | `shared_files` (自分がオーナー) | `files.json`（メタデータ+署名付きURL） | JSON |
| `chat_messages` | チャット | `chat_messages` (自分の発言) | `chat_messages.json` | JSON |
| `timeline` | タイムライン投稿 | `timeline_posts` (自分の投稿) | `timeline_posts.json` | JSON |
| `action_memos` | 行動メモ | `action_memos`, `action_memo_tags`, `action_memo_tag_links`, `user_action_memo_settings` (F02.5) | `action_memos.json` | JSON |
| `audit_logs` | 監査ログ | `audit_logs` (自分のアクション) | `audit_logs.json` | JSON |
| `notifications` | 通知履歴 | `notifications` | `notifications.json` | JSON |

> **カテゴリ選択**: `POST /data-export` の `categories` パラメータにカテゴリキーの配列を指定することで、必要なデータのみエクスポートできる。省略時は全カテゴリが対象。GDPR第15条のアクセス権は特定カテゴリへの請求も認めている。

> **注意**: 暗号化フィールド（氏名・電話番号・郵便番号）は復号してエクスポートする。エクスポートZIPは一時的にS3に保存し、24時間後に自動削除する。

> **注意**: ファイル共有（`shared_files`）の実ファイルはZIPに含めない。`files.json` にメタデータと24時間有効の署名付きURLを記載する。GDPR第20条のポータビリティ権は「構造化された機械可読な形式」を求めており、メタデータ+URLで要件を満たす。ZIP全体のサイズ上限は **500MB** とし、超過した場合はメタデータのみにフォールバックする。

### 3.3 エクスポート処理フロー

```
ユーザー → POST /api/v1/account/data-export
              │
              ├─ レートリミットチェック（1日1回）
              ├─ パスワード再認証
              ├─ DataExportRequestedEvent 発行
              │
              └─ @Async("job-pool") でバックグラウンド処理開始
                   │
                   ├─ PersonalDataCollector がテーブル横断でデータ収集
                   │   ├─ account.json（EncryptionService で復号）
                   │   ├─ memberships.json
                   │   ├─ payments.json
                   │   ├─ ... （各カテゴリ）
                   │   └─ files.json（メタデータ+署名付きURL）
                   │
                   ├─ ZIP圧縮（ランダム12文字パスワード付き）
                   ├─ パスワードのbcryptハッシュを data_exports.zip_password_hash に保存
                   ├─ S3にアップロード（s3Key: gdpr-exports/{userId}/{timestamp}.zip）
                   ├─ data_exports テーブルにステータス更新（COMPLETED）
                   ├─ 完了通知メール送信（ダウンロードリンク + ZIPパスワード平文を記載）
                   │
                   └─ ※失敗時: status=FAILED, error_message記録, 失敗通知メール送信
                      自動リトライはしない。ユーザーに再リクエストを促す。
                      FAILEDレコードは24時間後にリセット扱いとし再リクエスト可能。
```

### 3.4 PersonalDataCollector

```java
/**
 * 個人データ収集オーケストレータ。
 * 各リポジトリからユーザーに紐づくデータを収集し、エクスポート用のMap構造に変換する。
 * categories が指定された場合は該当カテゴリのみ収集する。
 */
@Service
@RequiredArgsConstructor
public class PersonalDataCollector {

    private final UserRepository userRepository;
    private final OAuthAccountRepository oAuthAccountRepository;
    private final EncryptionService encryptionService;
    // ... 各リポジトリ

    /** カテゴリキー → 収集メソッドのマッピング */
    private static final Map<String, String> CATEGORY_FILES = Map.ofEntries(
        Map.entry("account", "account.json"),
        Map.entry("oauth", "oauth_accounts.json"),
        Map.entry("memberships", "memberships.json"),
        Map.entry("profiles", "profiles.json"),
        Map.entry("payments", "payments.json"),
        Map.entry("charts", "charts.json"),
        Map.entry("chat_messages", "chat_messages.json"),
        Map.entry("timeline", "timeline_posts.json"),
        Map.entry("action_memos", "action_memos.json"),
        Map.entry("audit_logs", "audit_logs.json"),
        Map.entry("notifications", "notifications.json")
    );

    /**
     * 指定ユーザーの個人データをカテゴリ別に収集する。
     *
     * @param userId     対象ユーザーID
     * @param categories 収集対象カテゴリキー（nullまたは空の場合は全カテゴリ）
     * @return カテゴリ名 → JSON文字列のマップ
     */
    public Map<String, String> collect(Long userId, Set<String> categories) {
        Set<String> targets = (categories == null || categories.isEmpty())
                ? CATEGORY_FILES.keySet()
                : categories;

        Map<String, String> data = new LinkedHashMap<>();
        for (String category : targets) {
            String fileName = CATEGORY_FILES.get(category);
            if (fileName != null) {
                data.put(fileName, collectByCategory(userId, category));
            }
        }
        return data;
    }
}
```

### 3.5 @PersonalData アノテーションと起動時網羅性チェック

新機能で個人データを持つテーブルを追加した際に `PersonalDataCollector` への追加を忘れると GDPR 違反になる。これを防ぐため、アノテーション + 起動時検証で網羅性をシステム的に担保する。

```java
/**
 * 個人データを保持するエンティティに付与する。
 * 起動時に PersonalDataCoverageValidator が全 @PersonalData 付きエンティティを列挙し、
 * PersonalDataCollector でカバーされていないものがあれば警告を出す。
 */
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface PersonalData {
    /** PersonalDataCollector のカテゴリキー */
    String category();
}

// 使用例
@PersonalData(category = "charts")
@Entity
@Table(name = "chart_records")
public class ChartRecordEntity extends BaseEntity { ... }

@PersonalData(category = "payments")
@Entity
@Table(name = "member_payments")
public class MemberPaymentEntity extends BaseEntity { ... }
```

```java
/**
 * 起動時にエクスポート網羅性を検証する。
 * @PersonalData 付きエンティティのカテゴリが PersonalDataCollector.CATEGORY_FILES に
 * すべて登録されていることを確認す��。未登録があれば起動時に ERROR ログを出力。
 */
@Slf4j
@Component
@RequiredArgsConstructor
public class PersonalDataCoverageValidator implements ApplicationRunner {

    private final PersonalDataCollector collector;

    @Override
    public void run(ApplicationArguments args) {
        Set<String> registered = collector.getCategoryKeys();
        Set<String> annotated = new ClassPathScanningCandidateComponentProvider(false)
                // @PersonalData 付きクラスをスキャン
                // → 各クラスの category() を収集
                ...;

        Set<String> missing = new HashSet<>(annotated);
        missing.removeAll(registered);
        if (!missing.isEmpty()) {
            log.error("【GDPR網羅性チェック失敗】以下のカテゴリが PersonalDataCollector に未登録: {}", missing);
            // 本番ではここで起動失敗にすることも検討
        }
    }
}
```

> **運用ルール**: 個人データを持つ新規エンティティには必ず `@PersonalData(category = "xxx")` を付与すること。CI のコードレビューチェックリストにも追加を推奨。

## 4. アカウント物理削除バッチ

### 4.1 概要

退会（論理削除）から30日経過したユーザーのデータを物理削除する。既存の `AuthCleanupBatchService` と同じパターン（`@Scheduled` + `@SchedulerLock`）で実装する。

### 4.2 削除順序

FK制約を考慮し、参照先テーブルから順に削除する。

```
Phase 1: トークン・セッション系・エクスポートデータ
  ├─ refresh_tokens
  ├─ email_verification_tokens
  ├─ email_change_tokens
  ├─ password_reset_tokens
  ├─ mfa_recovery_tokens
  ├─ two_factor_auth
  ├─ webauthn_credentials
  ├─ otp_challenges
  └─ data_exports → S3のZIPファイルも削除

Phase 2: 活動データ — 明示的な匿名化・削除
  ├─ audit_logs → 匿名化（user_id=NULL, ip_address=NULL）※法的保存義務 ※FK: ON DELETE SET NULL 設定済み
  ├─ chat_messages (sender_id) → FK: ON DELETE SET NULL 設定済み（自動NULL化）
  ├─ chart_records (customer_user_id) → NULL化（匿名化） ※FK: ON DELETE RESTRICT のため明示UPDATE必須
  └─ shared_files (created_by) → S3オブジェクトを先に削除 ※FK: ON DELETE SET NULL 設定済み

  ※以下は ON DELETE CASCADE 設定済みのため Phase 5 の users 物理DELETE で自動連鎖削除:
  ├─ notifications (user_id) → CASCADE
  ├─ timeline_posts (user_id) → CASCADE（子投稿 parent_id も CASCADE で連鎖削除）
  └─ timeline_post_reactions (user_id) → CASCADE

Phase 3: メンバーシップ・プロフィール・ロール
  ├─ user_roles → 物理削除（FK: ON DELETE RESTRICT のため明示DELETE必須）
  │   ※ granted_by カラムもFK参照 → 先に granted_by=NULL に更新してからDELETE
  ├─ member_profiles (user_id) → FK: ON DELETE SET NULL 設定済み（自動NULL化）
  └─ team_org_memberships → invited_by, responded_by を NULL化（FK: ON DELETE RESTRICT のため明示UPDATE必須）
    ※メンバーシップレコード自体は組織の記録として残す

Phase 4: 決済・外部連携
  ├─ member_payments → user_id をセンチネルユーザー（id=0）に差し替え ※FK: ON DELETE RESTRICT ※帳簿保存義務（7年）
  ├─ stripe_customers → Stripe API で Customer 削除後、レコード物理削除
  └─ oauth_accounts

Phase 5: ユーザー本体
  └─ users（物理DELETE）→ CASCADE設定のテーブルが自動連鎖削除
```

> **注意（カルテ保存義務）**: カルテ（`chart_records`）はスタッフが作成した医療記録のため、顧客としてのユーザー削除時も `customer_user_id` を NULL 化（匿名化）するのみで記録自体は残す。`customer_user_id` の FK制約は `ON DELETE RESTRICT` のため、Phase 5 の users 物理DELETE前に Phase 2 で明示的に UPDATE SET NULL する必要がある。なお `staff_user_id` は `ON DELETE SET NULL` 設定済みのため自動対応。

> **注意（監査ログの匿名化）**: `audit_logs` は物理削除ではなく匿名化（`user_id = NULL`、IPアドレス等の個人識別子を削除）とする。FK制約は `ON DELETE SET NULL` 設定済みのため、NULL化に問題はない。

> **注意（決済履歴の匿名化）**: `member_payments.user_id` は `NOT NULL` 制約があるため NULL 化できない。代わりに **センチネルユーザー（id=0, 表示名="退会済みユーザー"）** に `user_id` を差し替えることで、FK制約・NOT NULL制約を壊さずに匿名化する。センチネルユーザーは `V12.004__seed_deleted_sentinel_user.sql` で投入する。チャットの投稿者表示等、他の箇所でも退会ユーザーの表示に統一的に使える。

> **注意（team_org_memberships）**: `invited_by` と `responded_by` のFK制約は `ON DELETE` 指定なし（デフォルト: RESTRICT）のため、Phase 3 で明示的に NULL化する。メンバーシップレコード自体はチーム・組織の運営記録として残す。

> **注意（CASCADE自動削除）**: `notifications`, `timeline_posts`, `timeline_post_reactions` は `ON DELETE CASCADE` 設定済みのため、Phase 5 の users 物理DELETE時に自動連鎖削除される。バッチ内での明示削除は不要だが、大量データの場合はCASCADE削除がロングトランザクションになるリスクがあるため、件数が多い場合は事前にバッチ削除することを推奨。

> **法的根拠**: GDPR第17条3項(b)「法的義務の履行に必要な場合」の例外規定、および日本の電子帳簿保存法（決済記録7年保存義務）に対応。`user_id` をセンチネルに差し替えれば個人との紐付けは切れるため GDPR 準拠となる。

### 4.3 バッチ実装

> **注意（@SQLRestriction との競合）**: `UserEntity` には `@SQLRestriction("deleted_at IS NULL")` が付与されており、通常のJPAクエリでは論理削除済みユーザーを取得できない。物理削除バッチでは **ネイティブSQL クエリ** で `@SQLRestriction` をバイパスする必要がある。

```java
/**
 * 退会済みユーザーの物理削除バッチ。
 * 毎日 AM4:00（JST）に実行し、論理削除から30日経過したユーザーを完全消去する。
 * dry-run モードでは実際の削除を行わず、対象件数のみログ出力する。
 */
@Slf4j
@Service
@RequiredArgsConstructor
public class AccountPurgeService {

    private static final int RETENTION_DAYS = 30;
    private static final int BATCH_SIZE = 100;

    /** dry-run モード（application.yml で gdpr.purge.dry-run=true に設定） */
    @Value("${gdpr.purge.dry-run:false}")
    private boolean dryRun;

    @Scheduled(cron = "0 0 4 * * *", zone = "Asia/Tokyo")
    @SchedulerLock(name = "accountPurgeBatch", lockAtMostFor = "PT30M", lockAtLeastFor = "PT1M")
    public void purgeExpiredAccounts() {
        LocalDateTime cutoff = LocalDateTime.now().minusDays(RETENTION_DAYS);
        // @SQLRestriction をバイパスするためネイティブSQLで取得
        List<UserEntity> targets = userRepository
                .findPurgeTargets(cutoff, PageRequest.of(0, BATCH_SIZE));

        int successCount = 0;
        int failedCount = 0;

        for (UserEntity user : targets) {
            try {
                if (dryRun) {
                    logDryRun(user);
                } else {
                    purgeUser(user);
                    log.info("ユーザー物理削除完了: userId={}", user.getId());
                }
                successCount++;
            } catch (Exception e) {
                log.error("ユーザー物理削除失敗: userId={}", user.getId(), e);
                failedCount++;
                // 1件の失敗で全体を止めない
            }
        }

        log.info("物理削除バッチ完了{}: 対象={}件, 成功={}件, 失敗={}件",
                dryRun ? "（DRY-RUN）" : "", targets.size(), successCount, failedCount);

        // 失敗があれば運用チームに通知（既存の通知基盤 F04.3 を活用）
        if (failedCount > 0) {
            notifyPurgeFailure(failedCount, targets.size());
        }
    }

    @Transactional
    void purgeUser(UserEntity user) {
        Long userId = user.getId();
        // Phase 1〜5 の順序で削除（本文参照）
        // ...
    }

    private void logDryRun(UserEntity user) {
        // 各テーブルの対象件数のみカウントしてログ出力
        log.info("[DRY-RUN] userId={}: 削除予定データを集計中...", user.getId());
        // refresh_tokens: X件, chat_messages: X件, ... のように件数を出力
    }

    private void notifyPurgeFailure(int failedCount, int totalCount) {
        // 既存の通知基盤で Slack/メール通知
        // 「物理削除バッチ: {failedCount}/{totalCount}件失敗。手動確認が必要です」
        log.warn("物理削除バッチ失敗通知: {}件失敗", failedCount);
    }
}
```

リポジトリのネイティブクエリ定義:

```java
public interface UserRepository extends JpaRepository<UserEntity, Long> {
    // ... 既存メソッド

    /**
     * 物理削除対象ユーザーを取得する。
     * @SQLRestriction("deleted_at IS NULL") をバイパスするためネイティブSQLを使用。
     * Pageable で件数制限（MySQL の LIMIT パラメータバインド制約を回避）。
     */
    @Query(value = """
        SELECT * FROM users
        WHERE deleted_at < :cutoff
          AND purged_at IS NULL
          AND id != 0
        ORDER BY deleted_at ASC
        """, nativeQuery = true)
    List<UserEntity> findPurgeTargets(
            @Param("cutoff") LocalDateTime cutoff,
            Pageable pageable);
}
```

呼び出し側:

```java
List<UserEntity> targets = userRepository
        .findPurgeTargets(cutoff, PageRequest.of(0, BATCH_SIZE));
```

## 5. 退会フロー拡張

### 5.1 退会前チェック: 唯一の管理者の移譲

退会リクエスト時、対象ユーザーがプラットフォームレベルの **唯一の SYSTEM_ADMIN** である場合は退会をブロックし、先に管理者権限の移譲を要求する。

> **注意（ロール管理の仕組み）**: ロールは `roles` テーブル（固定6種）+ `user_roles` テーブル（ユーザーへの割当）で管理される。`user_roles.scope_key` が `'platform'`（team_id も organization_id も NULL）のレコードがプラットフォームレベルのロール割当。SYSTEM_ADMIN のチェックは `user_roles` JOIN `roles` で `roles.name = 'SYSTEM_ADMIN'` AND `scope_key = 'platform'` の件数で判定する。

```
UserService.requestWithdrawal()
  ├─ 唯一のSYSTEM_ADMINチェック（新規）
  │   ├─ user_roles JOIN roles で name='SYSTEM_ADMIN' AND scope_key='platform' の件数を取得
  │   ├─ 件数が1かつ対象ユーザーのみ → ブロック（GDPR_006 エラー）
  │
  ├─ 既存処理（論理削除、Token失効）
  └─ WithdrawalRequestedEvent 発行（既存）
```

> **根拠**: GDPR第17条3項(e)も「他者の権利行使に必要な場合」は消去権の例外を認めている。システム管理者不在によるサービス運用不能を防ぐのは正当な理由である。

### 5.2 退会時 Stripe 自動キャンセル

既存の `UserService.requestWithdrawal()` が発行する `WithdrawalRequestedEvent` に新規リスナーを追加する。

> **注意**: 現時点で `WithdrawalRequestedEvent` の専用リスナーは未実装。本機能で初めてリスナーを作成する。非同期処理は既存の `@Async("event-pool") + @TransactionalEventListener(phase = AFTER_COMMIT)` パターンに従う。

```
UserService.requestWithdrawal()
  ├─ 既存処理（論理削除、Token失効）
  └─ WithdrawalRequestedEvent 発行（既存）
        │
        └─ WithdrawalStripeHandler（新規リスナー）
              ├─ StripeCustomerEntity を検索
              ├─ TeamSubscriptionEntity で ACTIVE なサブスクリプションを検索
              ├─ Stripe API: subscription.cancel()（即時キャンセル、期間終了まで有効）
              ├─ TeamSubscriptionEntity の status を CANCELLED に更新
              └─ log.info("退会時Stripeサブスク自動キャンセル: userId={}, subId={}")
```

-- id=0 の固定ユーザーを投入。通常の自動採番（AUTO_INCREMENT）は1始まりのため衝突しない。

## 8. セキュリティ要件

| 要件 | 対策 |
|---|---|
| エクスポート前の本人確認 | パスワード再認証必須（OAuth専用ユーザーはメールOTP、下記参照） |
| エクスポートZIPの保護 | ランダム12文字英数記号パスワード付きZIP。パスワードはメールで別送。DBにはbcryptハッシュのみ保存（平文非保存） |
| 一時ファイルの残留防止 | S3 Lifecycle Policy で24時間後に自動削除 |
| エクスポートサイズ上限 | ZIP全体500MB上限。超過時はメタデータのみにフォールバック |
| レートリミット | エクスポート: 1日1回、退会: 1分3回（既存） |
| ログ監査 | エクスポート・削除操作はすべて `audit_logs` に記録 |
| 暗号化データの復号 | エクスポート時のみ `EncryptionService` で復号。転送はHTTPS |

### 8.1 OAuth専用ユーザーのメールOTP認証

パスワードを持たない OAuth 専用ユーザー（Google/LINE/Apple ログインのみ）がエクスポートや退会を行う場合、メールOTPで本人確認する。

**実装方針**: 既存の `email_verification_tokens` は `purpose` カラムがなく用途区別ができないため、OTP専用テーブル `otp_challenges` を新設する。試行回数・ロック管理もテーブル内で完結させる。

| 項目 | 仕様 |
|---|---|
| OTPコード形式 | 6桁数字（`SecureRandom` で生成） |
| 有効期限 | 10分間 |
| 試行制限 | 3回失敗でロックアウト（30分間） |
| 保存方法 | `otp_challenges` テーブルに `code_hash`（SHA-256）で保存 |
| 送信先 | `users.email`（OAuth連携時に取得済み） |

```
POST /api/v1/account/data-export
  ├─ OAuth専用ユーザー判定（password_hash IS NULL）
  ├─ 6桁OTPを生成 → otp_challenges に保存
  ├─ OTPをメール送信
  └─ 202 Accepted（OTP入力待ち）

POST /api/v1/account/data-export/verify-otp
  ├─ OTP照合（SHA-256ハッシュ比較）
  ├─ 3回失敗 → GDPR_007 エラー（30分ロックアウト）
  └─ 成功 → エクスポートジョブ開始
```

## 9. エクスポート関連バッチ

### 9.1 PROCESSING 状態のリカバリ

`@Async("job-pool")` はインメモリのスレッドプールで実行するため、エクスポート処理中にサーバーが再起動すると `status=PROCESSING` のまま永久に完了しない。ユーザーは `GDPR_002`（処理中）エラーで再リクエストもできなくなる。

これを防ぐため、1時間以上 `PROCESSING` のままのレコードを `FAILED` にリセットするリカバリ処理を実行する。

```java
/**
 * 異常終了したエクスポートジョブをリカバリする。
 * 毎時00分に実行。1時間以上PROCESSINGのままのジョブをFAILEDにリセットし、
 * ユーザーが再リクエスト可能な状態に戻す。
 */
@Scheduled(cron = "0 0 * * * *", zone = "Asia/Tokyo")
@SchedulerLock(name = "exportRecoveryBatch", lockAtMostFor = "PT5M", lockAtLeastFor = "PT1M")
@Transactional
public void recoverStuckExports() {
    LocalDateTime stuckThreshold = LocalDateTime.now().minusHours(1);
    int recovered = dataExportRepository
            .resetStuckProcessing(stuckThreshold, "サーバー再起動により処理が中断されました");
    if (recovered > 0) {
        log.warn("エクスポートジョブリカバリ: {}件をFAILEDにリセット", recovered);
    }
}
```

### 9.2 エクスポートZIPクリーンアップ

```java
/**
 * 期限切れのエクスポートZIPをS3から削除する。
 * 毎日 AM5:00（JST）に実行。
 */
@Scheduled(cron = "0 0 5 * * *", zone = "Asia/Tokyo")
@SchedulerLock(name = "exportCleanupBatch", lockAtMostFor = "PT10M", lockAtLeastFor = "PT1M")
public void cleanupExpiredExports() {
    List<DataExportEntity> expired = dataExportRepository
            .findByExpiresAtBeforeAndS3KeyIsNotNull(LocalDateTime.now());
    for (DataExportEntity export : expired) {
        storageService.delete(export.getS3Key());
        export.clearS3Key();
        dataExportRepository.save(export);
    }
    log.info("エクスポートZIPクリーンアップ完了: {}件削除", expired.size());
}
```
