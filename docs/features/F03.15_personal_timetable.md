## 1. 概要

ユーザーが **自分専用の時間割** を作成・管理できる機能。大学生のように授業を自分で組む必要がある層、塾や習い事を複数掛け持ちしている層、複数の所属チーム時間割を統合して見たい層に向けた個人スコープのモジュール。

本モジュールは F03.9（チーム時間割）と次の3点で連動する：

1. **コマのチームリンク**: 個人時間割のコマに「このコマは XX 講義チームの時間割と同期」を設定すると、リンク先チームの臨時変更（休講・教室変更等）が自動で個人スケジュールに反映される
2. **統一メモ機能**: チーム時間割のコマにも個人時間割のコマにも、ユーザー個人のメモ（予習・復習・持参物・自由メモ・カスタム項目・添付ファイル）を残せる。**個人メモは本人以外には見えない**
3. **ダッシュボード統合**: 個人ダッシュボードの「今日の時間割」ウィジェットで、所属チーム時間割と個人時間割を1画面に統合表示する

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| 全認証ユーザー | 自分の個人時間割の CRUD・自分のメモの CRUD |
| 家族チームのメンバー（MEMBER+） | 共有された他人の個人時間割の **コマ表示のみ** 閲覧（メモ・添付は不可） |
| SYSTEM_ADMIN | 監査目的でのメタデータ参照のみ。本文・メモ・添付の閲覧は不可 |

### 対象レベル
- [x] 個人 (Personal) — 本機能のメインスコープ
- [x] チーム時間割のコマ（既存 F03.9）— 個人メモ機能のみ統合的に対応
- [ ] 組織 (Organization)
- [ ] チーム (Team)（時間割そのものは F03.9 管轄）

### パーミッション
本モジュールは原則として **個人スコープのみ** で動作するため、新規パーミッション追加は不要。リソース所有権チェック（`personal_timetables.user_id == currentUser.id`）で認可判定する。

ただし以下2点はサーバ側で追加の検証を要する:
- **チームリンク登録時**: `linked_team_id` を指定する場合、`currentUser` がそのチームの MEMBER 以上であることを確認（離脱後はリンクを自動切断）
- **家族共有先指定時**: 共有先 `team_id` が `team_type = 'FAMILY'` かつ `currentUser` がその家族チームの MEMBER 以上であることを確認

**インデックス**
```sql

INDEX idx_pt_user_effective (user_id, effective_from, effective_until)
```

**制約**
- `effective_from <= effective_until`（非 NULL 時）の CHECK
- 同一ユーザー × name の重複は許容（学期や用途が違うケースを想定）
- ステータス遷移は F03.9 の `timetables` と同じルール（DRAFT⇄ARCHIVED⇄ACTIVE）。同一ユーザー×期間重複の ACTIVE 化時は既存 ACTIVE を自動 ARCHIVED
- **DRAFT/ARCHIVED は家族共有先からも非表示**。家族閲覧 API は `status = 'ACTIVE'` のみ返す
- **`visibility = 'FAMILY_SHARED'` かつ `share_targets` が0件** の場合、PATCH レスポンスに警告フラグ `share_target_empty: true` を含めて返す（UI で「公開先未設定」を表示）
- **削除時のカスケード**: `personal_timetables` の物理削除は行わず論理削除のみ。論理削除時、関連 `personal_timetable_slots` は残置するが、API は子リソースを返さない。週次 `PersonalTimetableHardDeleteBatch` で `deleted_at < NOW() - 90日` の本体と関連子レコード（slots/periods/share_targets/関連メモ）を物理削除

**インデックス**
```sql
UNIQUE KEY uq_pts_day_period_week (personal_timetable_id, day_of_week, period_number, week_pattern)
INDEX idx_pts_linked_slot (linked_slot_id)
INDEX idx_pts_linked_team (linked_team_id)
```

**制約・備考**
- `linked_team_id` を指定する場合は `linked_timetable_id` も必須（曖昧性回避）。`linked_slot_id` は任意（指定しない場合は曜日×時限でマッチング）
- リンク先チームから currentUser が脱退した場合、`PersonalTimetableSlotMembershipRevokeListener` がコマの `linked_team_id` / `linked_timetable_id` / `linked_slot_id` を NULL クリア（コマ自体は残し、メモも保持）
- `EVERY` と `A`/`B` の同時登録は禁止（F03.9 と同様、アプリ層で検証）
- F03.9 の `is_break = TRUE` 時限と同様、break 枠への割当は不可
- **コマ編集は親 `personal_timetables.status = 'DRAFT'` 時のみ許可**。ACTIVE/ARCHIVED 時の PUT は 409 エラー（恒久変更は revert-to-draft → 編集 → activate のフロー）
- **リンク先コマの整合性検証**: `linked_slot_id` 指定時、リンク先 `timetable_slots` の `day_of_week` × `period_number` が個人コマのそれと **一致する場合のみ受付**。不一致は 409 `link_position_mismatch`
- **リンク先 timetable の status 検証**: `linked_timetable_id` の status は ACTIVE であること。DRAFT/ARCHIVED へのリンク登録は 409
- **`week_pattern_enabled` を TRUE → FALSE に変更する場合**、既存の `week_pattern IN ('A','B')` のコマがあれば 409 `week_pattern_in_use`。ユーザーに「A/B 週コマを削除してから無効化」を促す
- **`week_pattern_base_date` の検証**: `effective_from <= week_pattern_base_date` で、かつ `effective_until` がある場合は `<= effective_until`

#### `timetable_slot_user_notes`

ユーザー個人のメモ。チーム時間割のコマ（`timetable_slots`）と個人時間割のコマ（`personal_timetable_slots`）の **両方** にぶら下がる統一テーブル。`slot_kind` で識別する。

| `slot_kind` | ENUM('TEAM','PERSONAL') | NO | — | 紐付くスロットの種別 |
| `slot_id` | BIGINT UNSIGNED | NO | — | TEAM の場合 timetable_slots.id、PERSONAL の場合 personal_timetable_slots.id（FK は貼らない。論理参照） |
| `preparation` | TEXT | YES | NULL | 予習メモ（最大2,000字、Markdown） |
| `review` | TEXT | YES | NULL | 復習メモ（最大2,000字、Markdown） |
| `items_to_bring` | TEXT | YES | NULL | 持参物メモ（最大2,000字、Markdown） |
| `free_memo` | TEXT | YES | NULL | 自由メモ（最大10,000字、Markdown） |
| `custom_fields` | JSON | YES | NULL | カスタムフィールド値。形式: `[{"field_id": 1, "value": "..."}]` |
| `target_date` | DATE | YES | NULL | 特定日付に限定したメモ（NULL = コマ全般のメモ。例: "次回4/15は教科書 p.30 まで") |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
UNIQUE KEY uq_tsun_user_slot_date (user_id, slot_kind, slot_id, target_date)
INDEX idx_tsun_user_slot (user_id, slot_kind, slot_id)
INDEX idx_tsun_user_date (user_id, target_date)
```

**制約・備考**
- **DB レベルの FK は貼らない**（slot_kind による多態参照のため）。代わりに以下の整合性メンテを実施:
  - `TimetableSlotDeletedEvent` / `PersonalTimetableSlotDeletedEvent` を発行
  - `UserNoteCleanupListener` が対応する `timetable_slot_user_notes` を論理削除
  - 週次バッチ `OrphanedUserNoteSweepBatch` で取り残し（slot_kind/slot_id 不整合）を物理削除（実行時刻: 毎週日曜 03:00 JST、SLO: バッチ完走 30分以内）
- `target_date` が NULL のレコードは「常設メモ」、非 NULL は「日付限定メモ」。両方の存在を許容するため UNIQUE は `target_date` を含む
- **個人メモは本人のみ閲覧**: 認可は `user_id == currentUser.id` のみ。家族共有でも閲覧不可。レスポンス DTO 側でフィールド除外を強制
- **チーム共通の周知メモ**は本テーブルではなく `timetable_slots.notes`（既存 F03.9）を編集する。MANAGE_TIMETABLE 権限保持者がそちらを更新すると `TimetableSlotNoteUpdatedEvent` を発行し、F04.3 経由で当該チームのメンバーに通知
- **楽観排他**: メモ PUT 時、リクエストに `if_unmodified_since` ヘッダで前回 `updated_at` を送信させ、サーバ側で照合。不一致は 412 `precondition_failed` で返す（同時編集の上書き事故防止）
- **Markdown XSS 対策**: クライアント側で `DOMPurify` 相当のサニタイザでレンダリング。サーバ側は保存時に `<script>` / `javascript:` / `data:text/html` の検出 → 422 `unsafe_markdown` で拒否
- **カスタムフィールド削除後の値**: `custom_fields` JSON 内の該当 field_id 値はサーバ側で残置するが、レスポンス時に `is_orphaned: true` フラグを付与して UI で灰色表示

| `deleted_at` | DATETIME | YES | NULL | 論理削除（R2 オブジェクトは週次バッチで物理削除） |

**インデックス**
```sql
INDEX idx_tsuna_note (note_id)
INDEX idx_tsuna_user (user_id)
```

**制約**
- 1メモあたり最大5件
- ファイル形式は `image/jpeg`, `image/png`, `image/webp`, `image/heic`, `application/pdf` のみ。プリサイン URL 生成時に Content-Type 制約 + サーバ側で magic byte 検証
- 配信は **5分 TTL のプリサイン URL** で都度生成（直リンク禁止）
- 物理削除は `R2OrphanCleanupBatch` が `deleted_at < NOW() - 30日` を対象に実行

## 5. ビジネスロジック

### 5.1 個人時間割作成〜公開フロー
```
1. ユーザーが /me/personal-timetables を POST（DRAFT で作成）
   - init_period_template = 'UNIV_90MIN' を指定すると periods が自動投入される
2. ユーザーが時限を編集 (PUT /periods)
3. ユーザーがコマを編集 (PUT /slots)
   - 必要に応じてチームリンクを設定 (POST /slots/{id}/link)
4. activate で ACTIVE 化
   - 同期間の既存 ACTIVE 時間割は自動 ARCHIVED
5. ダッシュボードに反映
```

### 5.2 チームリンクと休講自動反映フロー
```
【契機】
- F03.9 の TimetableChangeCreatedEvent（REPLACE / CANCEL / ADD / DAY_OFF）

【処理】PersonalTimetableLinkSyncListener
1. event.timetable_id を linked_timetable_id に持つ personal_timetable_slots を全取得
2. 各スロットについて、slot.user の personal_timetable_settings.auto_reflect_class_changes_to_calendar を確認
3. ON かつ slot.auto_sync_changes = TRUE かつ 親 personal_timetable.status = 'ACTIVE' の場合のみ進む
4. 個人スケジュール（schedules テーブル, user_id = slot.user_id）に以下を INSERT/UPDATE:
   - CANCEL / DAY_OFF: title="[休講] {subject_name}", start_at = 該当時限の開始時刻, end_at = 終了時刻, color="#999999"
   - REPLACE: title="[変更] {original} → {new}", room/teacher の変更があれば description に反映
   - ADD: title="[補講] {subject_name}", 時間は change.target_date + period_number から算出
5. external_ref（schedules.external_ref TEXT, JSON）として {source: 'F03.15', timetable_change_id, personal_timetable_slot_id} を保持し idempotent 化:
   - 同一 external_ref のレコードが存在する場合は UPDATE（タイトル・時刻のみ）
   - 重複 INSERT は発生させない（UNIQUE INDEX を schedules に追加: idx_sch_extref_uq (md5(external_ref))）
6. ユーザーに F04.3 経由でプッシュ通知（タイトル「ドイツ語Ⅰ が休講になりました（5/10）」）
7. **DAY_OFF と個別 CANCEL の同日重複時**: F03.9 の優先度ルール（DAY_OFF 優先）に従い、DAY_OFF レコードのみ schedule に反映。個別変更は無視

【取消フロー】
- timetable_changes が削除された場合は TimetableChangeDeletedEvent 経由で
  external_ref で紐付く schedules レコードを削除し、取消通知を送る

【auto_reflect 設定を OFF に変更した場合】
- 既存反映済みレコードは残置（過去分の自動削除はしない）
- 以後の新規 timetable_changes は反映しない
- ユーザーが過去分の掃除をしたい場合は schedules 画面で個別削除
```

### 5.3 チーム時間割コマの共通メモ更新通知
```
【契機】
- F03.9 の TimetableSlotNoteUpdatedEvent（既存 timetable_slots.notes が更新された時に発行する。F03.9 側に Event 発行を追加）

【処理】TeamSlotNoteNotifyListener
1. 該当 slot を持つチームの全 MEMBER を取得
2. 各メンバーの personal_timetable_settings.notify_team_slot_note_updates を確認
3. ON のメンバーへ F04.3 経由でプッシュ通知
   - タイトル: 「{subject_name} の共通メモが更新されました」
   - 本文: 更新後の notes の冒頭100文字
4. **過剰通知の抑制**: 同一 slot に対する連続更新は **5分間のデバウンス**（最後の更新から5分経過後に1度だけ通知）。デバウンス対象は Valkey の Sorted Set で管理
5. notes が空文字または NULL に変更された場合は通知しない（削除＝重要度低）
```

### 5.4 共有先離脱・チーム削除時のクリーンアップ
```
【契機A】TeamMemberLeftEvent（ユーザーがチームから離脱）
- TeamMemberLeftListener が以下を実施:
  - linked_team_id = leftTeam.id を持つ自分の personal_timetable_slots の link 情報を NULL クリア（コマとメモは保持）
  - 共有先になっていた場合、personal_timetable_share_targets から削除（共有相手側からは見えなくなる）

【契機B】FamilyTeamDissolvedEvent（家族チーム解散）
- ON DELETE CASCADE で personal_timetable_share_targets が自動削除される
- visibility=FAMILY_SHARED の personal_timetables は visibility は維持するが、share_targets が空になるため誰にも見えなくなる
```

### 5.5 ダッシュボード「今日の時間割」
- データ取得: `GET /me/dashboard/timetable-today`
- マージロジック:
  1. 個人の所属チームで時間割モジュール ON のチーム全部の今日のコマを取得（既存 F03.9）
  2. 個人時間割のうち status=ACTIVE かつ today が effective 範囲内のものの今日のコマを取得
  3. 開始時刻でソートして1配列にマージ
  4. 同一 slot_kind/slot_id にユーザーメモがあれば user_note_id を付与（ダッシュボードからメモを開ける）
  5. 個人時間割のコマと同時刻にチーム時間割のコマがあっても **両方表示**（学校＋塾の重なりを可視化）

### 5.6 メモの target_date と常設メモの関係
- 同一 slot に対して常設メモ（target_date=NULL）と日付限定メモ（target_date=2026-05-10）が共存可能
- 表示優先度: 日付限定 > 常設。日付限定がある日はそちらを表示し、「常設メモ表示」リンクで切替可能
- メモ取得 API は target_date 指定で日付限定のみ、未指定で常設のみ、`?include_default=true` で両方返す
- **「来週の準備物」集約ビュー**: `GET /api/v1/me/timetable-slot-notes/upcoming?from=YYYY-MM-DD&to=YYYY-MM-DD` で期間内の `items_to_bring` を集約取得（ダッシュボード「今週の持ち物」用）

### 5.7 添付ファイル管理
1. **アップロード**:
   - クライアント → POST `/attachments/presign` → サーバが UUID を発行し R2 への PUT 用プリサイン URL を返す（5分 TTL、Content-Type/Length 制約付き）
   - クライアント → R2 へ直接 PUT
   - クライアント → POST `/attachments/confirm` でメタデータ確定。サーバが以下を実施:
     - R2 HEAD で実体確認（サイズが 5MB 以下、Content-Type が許可リスト内）
     - 先頭 16 バイトの magic byte 検証（Content-Type と実体形式の一致）
     - 不一致なら R2 オブジェクトを即削除して 422 を返す
2. **confirm の冪等性**: 同一 R2 オブジェクトキーに対する2回目以降の confirm は既存 attachment レコードを返却（重複 INSERT しない）
3. **ダウンロード**: 都度プリサイン URL を発行（5分 TTL）。`Cache-Control: private, no-store` を強制。直リンクキャッシュを禁止
4. **削除**: 論理削除 → 30日後に R2OrphanCleanupBatch が物理削除（誤削除のリカバリ余地、毎日 04:00 JST 実行）
5. **クォータ**: F13 ストレージ機構の `personal_timetable_notes` カテゴリで管理。1ユーザー累計 100MB を超える場合、新規 presign 発行で 429 `quota_exceeded`

### 5.8 個人時間割の duplicate
- duplicate 時にコピーされるもの: 時限定義（periods）、コマ（slots）、設定（visibility/week_pattern_*）
- コピーされないもの: share_targets（共有先）、user_notes（メモ）、attachments（添付）
- リクエストで `effective_from` / `effective_until` / `name` / `term_label` を上書き可能
- `linked_team_id` 等は **コピー対象**。ただし currentUser が当該チームから離脱済みの場合は NULL クリアして複製
- 複製先は status = DRAFT で作成され、明示的に activate するまで反映されない

### 5.9 履修登録ヘルパー（大学生向け）
- `POST /api/v1/me/personal-timetables/{id}/slots/import-from-team` でリンク先候補チームの ACTIVE 時間割からコマを一括取り込み
- リクエスト: `{ "team_id": 56, "select_slot_ids": [314, 315, 316] }`
- 取り込み時、空き時限のみ INSERT。重複する曜日×時限は 409 で衝突一覧を返す
- 取り込んだコマには `linked_team_id` / `linked_timetable_id` / `linked_slot_id` を自動セット
- subject_name / teacher_name / room_name / course_code / credits はリンク先から初期コピー（後から個人で上書き可）

### 5.10 ステータス遷移と編集可否のサマリ
| 状態 | コマ編集 | リンク変更 | 共有先変更 | メモ編集 | 家族から閲覧 |
|---|:---:|:---:|:---:|:---:|:---:|
| DRAFT | ○ | ○ | ○ | ○（本人のみ） | × |
| ACTIVE | × (revert必要) | ○ | ○ | ○（本人のみ） | ○（条件付き） |
| ARCHIVED | × | × | × | ○（本人のみ、参照用） | × |

## 6. セキュリティ考慮事項

### 6.1 認可
- 全 `/me/...` エンドポイントは `@PreAuthorize("isAuthenticated()")` + Service 層で `resource.user_id == currentUser.id` の二重チェック
- `/families/{teamId}/members/{userId}/...` は以下を全て満たす場合のみ 200:
  - `teamId` の `team_type = 'FAMILY'`
  - `currentUser` が `teamId` の MEMBER 以上
  - `userId` も同じ `teamId` の MEMBER
  - 対象の `personal_timetables.visibility = 'FAMILY_SHARED'` かつ `personal_timetable_share_targets` に `teamId` が含まれる
  - レスポンスにメモ・添付・カスタムフィールドを **絶対に含めない**（DTO レイヤーで除外を強制）

### 6.2 メモの本人専有
- `timetable_slot_user_notes` を返す全 API は `user_id == currentUser.id` を強制
- 家族閲覧用 DTO には `user_note_id` / `has_attachments` フィールド自体を含めない

### 6.3 添付ファイル
- R2 のオブジェクトキーに `user/{user_id}/...` プレフィックスを必須化し、ダウンロード用プリサイン発行時にキーのプレフィックスと currentUser を照合
- プリサイン URL TTL は 5分。再生成には認証必須
- アップロード時に Content-Type を MIME 制約 + サーバ側 magic byte 検証（image/* と application/pdf のみ）
- 1ユーザーあたりの累計ストレージサイズに F13 のクォータ機構を流用（既存）

### 6.4 チームリンク権限
- `linked_team_id` の登録/更新時に `currentUser` がそのチームの MEMBER 以上であることを Service 層で検証
- 離脱後はリスナーが自動でリンク解除（コマとメモは保持）し、自動カレンダー反映も停止

### 6.5 共有先指定
- `team_id` が `team_type = 'FAMILY'` であることを Service 層で検証（DB トリガは使わない）
- 自分が所属していない家族チームは指定不可（403）

### 6.6 IDOR 対策
- 全 API パスの `{id}` は currentUser のリソースかどうかを Service 層で必ず検証
- メモのアップサート時、`slot_id` の所有権を検証:
  - `slot_kind=PERSONAL`: 該当 personal_timetable_slot.personal_timetable.user_id == currentUser
  - `slot_kind=TEAM`: 該当 timetable_slot のチームに currentUser が MEMBER 以上で所属

### 6.7 レートリミット
- プリサイン URL 発行: 1分5回/ユーザー
- メモ PUT: 1分30回/ユーザー（紙媒体からの一括入力を許容しつつ濫用を抑制）
- カスタムフィールド作成: 1分3回/ユーザー（誤連打抑制）

### 6.8 監査ログ
- 共有先の追加/削除、visibility 変更、添付ファイル削除、カスタムフィールド削除は監査ログに記録（既存 F11 監査基盤の `audit_logs` テーブルへ INSERT）

### 6.9 家族閲覧 API のエラー区別
- 対象 user の personal_timetable が存在しない: 404
- 対象が存在するが visibility != FAMILY_SHARED または currentUser の家族チームが share_targets に含まれない: 404 を返す（403 にすると共有有無を漏らすため）

### 6.10 R2 オブジェクトキーの安全性
- 形式: `user/{user_id}/timetable-notes/{uuid_v4}.{ext}` で固定。クライアント指定不可
- ext は MIME から導出（jpg/png/webp/heic/pdf）
- ダウンロード用プリサイン発行時、キー先頭が `user/{currentUser.id}/` で始まることを再検証

### 6.11 リンク先 timetable_slots 参照時のフォールバック
- 週間ビュー API がリンク先 slot を参照する際、currentUser がリンク先チームから既に脱退していた場合（リスナー未走行 / レース）:
  - 当該コマは個人時間割の自前情報のみで表示
  - レスポンスに `link_revoked: true` フラグを付け、UI に「このコマのチームリンクは無効になりました（再リンクしてください）」を表示

## 10. 大学生向けの最適化ポイント

設計レベルで意識した大学生向けの考慮:

| 配慮点 | 対応 |
|---|---|
| 講義名が長い | `subject_name VARCHAR(200)` |
| 教室名が「工学部 第3校舎 2階 3-201」のように長い | `room_name VARCHAR(200)` |
| 履修番号で管理 | `course_code VARCHAR(50)` 追加 |
| 単位数の管理 | `credits DECIMAL(3,1)` 追加（必要なユーザーだけ入力） |
| 90分/100分授業 | `personal_timetable_periods` を個人ごとに自由定義 |
| 隔週講義 | A/B 週パターン（既存 F03.9 と同設計） |
| 集中講義 | `effective_from` / `effective_until` を講義期間に絞った別個人時間割を作成 |
| 学期切替（前期→後期） | duplicate API + term_label で簡単切替 |
| 前期 14週 / 後期 14週などの日程 | `effective_until` で表現 |
| 講義チーム加入で時間割同期 | `linked_team_id` + 自動カレンダー連動 |
| 長期休暇中の集中講義 | 別個人時間割として作成、終わったら ARCHIVED |
| 履修登録の一括取り込み | `POST /slots/import-from-team`（§5.9） |
| 課題・レポート締切の管理 | F02.3 Todo と連携（メモから「Todo にする」アクション。Phase 5 以降で UI 検討） |

> 単位数集計・GPA 管理は本機能のスコープ外とする（将来 F12 系で別モジュール化）。`credits` は表示用のメタデータに留める。

## 14. i18n / テスト / 運用

### 14.1 i18n
- 6言語（ja/en/zh/ko/es/de）のロケールファイルに以下のキー名空間を追加:
  - `timetable.personal.*` — 個人時間割 UI
  - `timetable.notes.*` — メモ UI
  - `timetable.share.*` — 家族共有 UI
- フィールドラベル「予習」「復習」「持参物」は ja ロケール固定文言。en/zh/ko/es/de それぞれの翻訳を必須
- 標準4項目の表示 ON/OFF はユーザー設定でサーバ保持（`personal_timetable_settings` に `visible_default_fields JSON DEFAULT '["preparation","review","items_to_bring","free_memo"]'` を追加）。デバイス間で同期させるため `localStorage` ではなくサーバ保持

### 14.2 テスト
- ユニット: Service 層の認可・バリデーション・自動連携ロジック（TEST_CONVENTION 準拠）
- 統合: Controller 経由の E2E（Testcontainers MySQL）
- E2E (Playwright):
  - 個人時間割の作成→コマ登録→ACTIVE 化→ダッシュボード反映
  - チームリンク → 休講登録 → カレンダー自動反映
  - 家族共有設定 → 家族アカウントから閲覧（メモが見えないことを検証）
  - メモ添付ファイルのアップロード→ダウンロード→削除

### 14.3 OpenAPI
- swagger.json に以下のタグを追加: `personal-timetable`, `timetable-notes`, `family-personal-timetable`
- フロントの型定義は `frontend/app/types/personal-timetable.ts`, `frontend/app/types/timetable-note.ts` を新設

### 14.4 監視
- 自動カレンダー反映の失敗率: Grafana ダッシュボードに追加。失敗率 1% 超でアラート
- 添付ファイル uploads/min, downloads/min を Prometheus メトリクスに追加
