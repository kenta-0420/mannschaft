# F04.9: 確認通知（Confirmable Notification）

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 4+
> **最終更新**: 2026-04-06
> **関連ドキュメント**: F04.3 プッシュ通知（配信基盤）、F03.4 予約管理（臨時休業通知）、F03.6 緊急安否確認（エッジケース参考）、F12.3 GDPR対応

## 1. 概要

「送ったら終わり」ではなく、**受信者が確認したことを送信者が追跡できる通知**を提供する機能。整骨院の臨時休業のような緊急ケースでは、メール＋アプリ内通知を二重で送り、患者が「確認しました」を押したかどうかを院長がリアルタイムに把握できる。確認していない受信者には自動でリマインドが再送される。

**ユースケース例**:
- **臨時休業通知**（F03.4 連携）: 先生の体調不良で当日休み → 予約済み患者全員に通知 → 確認追跡 → 未確認者にリマインド → それでも未確認なら院長にアラート
- **練習場所変更連絡**: チームの練習場所が急遽変更 → メンバー全員に通知 → 確認追跡
- **重要なお知らせ**: 組織からの規約変更通知 → 全メンバーに確認を要求
- **イベント参加確認**: 特定メンバーへの出欠確認リマインド

**設計思想**: 臨時休業通知で実証されたパターン（送信→確認追跡→2段階リマインド）を汎用モジュールとして抽出し、予約以外のあらゆるスコープ（チーム・組織）で利用可能にする。

**F04.3 通知基盤との責務分担**:
- **confirmable 系テーブル**（本機能）: 確認追跡（誰が確認したか、リマインドの送信状況、期限管理）
- **notifications テーブル**（F04.3）: 配信チャネル管理・未読管理・通知一覧表示
- 二重レコード構造: 1回の確認通知送信で (1) confirmable_notification_recipients に確認追跡レコード + (2) notifications に配信レコード の両方が生成される
- 確認 → 通知既読の**一方向連携**: 受信者が確認ボタンを押すと、対応する notifications レコードも既読にする。逆方向（通知を既読にしても確認済みにはならない）は行わない
- 連携方式: `ConfirmableNotificationConfirmedEvent`（ApplicationEvent）を F04.3 の `NotificationEventListener` が受信して notifications を既読化する疎結合設計

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `confirmable_notification_settings` | スコープ別デフォルト設定（リマインド時間等） | なし |
| `confirmable_notifications` | 確認通知本体（送信履歴・ステータス管理） | なし |
| `confirmable_notification_recipients` | 受信者別の確認追跡レコード | なし |
| `confirmable_notification_templates` | 確認通知テンプレート（件名・本文・重要度のプリセット） | あり（`deleted_at`） |

### テーブル定義

#### `confirmable_notification_settings`

チームまたは組織単位の確認通知デフォルト設定。予約システム利用時は設定必須（初回利用時にデフォルト値で自動作成）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id（ポリモーフィック） |
| `recipient_reminder_minutes` | INT | NO | 180 | 受信者リマインドの事前分数（デフォルト3時間 = 180分） |
| `sender_reminder_minutes` | INT | NO | 120 | 送信者アラートの事前分数（デフォルト2時間 = 120分） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**制約**
```sql
UNIQUE KEY uk_cns_scope (scope_type, scope_id)
```

**バリデーションルール**:
- DB側の許容範囲: 0〜1440（0 = 無効化）
- API側の許容範囲: 0〜1440（DB と統一）
- `recipientReminderMinutes` > `senderReminderMinutes`（両方 > 0 の場合）。受信者リマインドの方が先に発火しないと、送信者アラートの前に確認チャンスがない
- 片方のみ 0 の場合の挙動: `senderReminderMinutes > 0` かつ `recipientReminderMinutes = 0` は**許可するが警告表示**（「受信者へのリマインドが無効のため、送信者アラート時点で未確認の可能性があります」）。逆も同様

**scope_type / scope_id のポリモーフィック設計**:
- FK 制約は設けない。アプリ層で teams / organizations テーブルの存在を検証する
- 理由: scope_type により参照先テーブルが動的に変わるため、単一 FK では表現できない

#### `confirmable_notifications`

確認が必要な通知の本体。臨時休業通知・一般確認通知の両方をこのテーブルで管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | `TEAM` / `ORGANIZATION`（ポリモーフィック、FK制約なし） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `notification_type` | VARCHAR(50) | NO | — | 通知種別（`EMERGENCY_CLOSURE` / `GENERAL` 等） |
| `priority` | VARCHAR(20) | NO | 'NORMAL' | 許可値: `NORMAL` / `HIGH` / `URGENT` |
| `status` | VARCHAR(20) | NO | 'ACTIVE' | 通知ステータス: `ACTIVE` / `COMPLETED` / `EXPIRED` / `CANCELLED` |
| `subject` | VARCHAR(200) | NO | — | 件名 |
| `message_body` | TEXT | NO | — | 本文（メール送信にも使用） |
| `source_type` | VARCHAR(50) | YES | NULL | 元リソースの種別（`EMERGENCY_CLOSURE` / `SCHEDULE` 等。汎用は NULL） |
| `source_id` | BIGINT UNSIGNED | YES | NULL | 元リソースのID |
| `recipient_reminder_minutes` | INT | YES | NULL | 受信者リマインド時間の個別指定。NULL = settings のデフォルトを使用 |
| `sender_reminder_minutes` | INT | YES | NULL | 送信者アラート時間の個別指定。NULL = settings のデフォルトを使用 |
| `send_email` | BOOLEAN | NO | TRUE | メールも送信するか |
| `sent_count` | INT | NO | 0 | 送信人数（スナップショット: ユーザー退会後も変動しない） |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users.id ON DELETE SET NULL（送信者） |
| `cancelled_at` | DATETIME | YES | NULL | 取消日時 |
| `cancelled_by` | BIGINT UNSIGNED | YES | NULL | FK → users.id ON DELETE SET NULL（取消者） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_cn_scope (scope_type, scope_id, created_at DESC)
INDEX idx_cn_type (notification_type)
INDEX idx_cn_status (status)
INDEX idx_cn_status_created (status, created_at DESC)   -- ACTIVE 通知のバッチ絞り込み用
```

**備考**
- `notification_type` は拡張可能な文字列。初期値は `EMERGENCY_CLOSURE`（臨時休業）と `GENERAL`（汎用）。サーバー側で自動決定: 汎用APIは常に `GENERAL`、臨時休業は `EMERGENCY_CLOSURE`。将来の拡張のためリクエストパラメータとしては受け付けない
- `priority` は VARCHAR(20) で管理。許可値リスト: `NORMAL` / `HIGH` / `URGENT`。`LOW` は確認を要求する通知の性質（確認を要求する以上、低優先度は矛盾）に適さないため、DB レベルでもバリデーションで拒否する
- `status` カラムによりバッチ処理の効率化を実現: リマインドバッチはまず `status = 'ACTIVE'` の通知IDで recipients を絞る（全テーブルスキャン回避）
- `source_type` / `source_id` は逆参照用。臨時休業なら `source_type = 'EMERGENCY_CLOSURE'`, `source_id` は `reservation_blocked_times.id` 等の既存テーブルを参照する（旧 `emergency_closures` テーブルは confirmable 系テーブルに統合済み）。汎用なら NULL
- `recipient_reminder_minutes` / `sender_reminder_minutes` が NULL の場合、`confirmable_notification_settings` のスコープ別デフォルトにフォールバック
- `send_email` が FALSE の場合、アプリ内通知（WebSocket + PWA Push）のみ送信
- `created_by`: NULL 許容。`ON DELETE SET NULL` により送信者ユーザーが削除されてもレコードは保持される
- `cancelled_at` / `cancelled_by`: 取消操作時にセットされる。`status` を `CANCELLED` に変更する際に同時に記録
- `sent_count` はスナップショット: 送信時点の受信者数を記録し、以降のユーザー退会等で変動しない

#### `confirmable_notification_recipients`

受信者ごとの確認状態を管理するレコード。確認通知1件につき受信者の数だけレコードが生成される。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `confirmable_notification_id` | BIGINT UNSIGNED | NO | — | FK → confirmable_notifications.id ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users.id ON DELETE CASCADE |
| `deadline_at` | DATETIME | NO | — | 期限（予約時刻 or 回答期限）。リマインドの起点時刻として使用 |
| `confirmed_at` | DATETIME | YES | NULL | 受信者が確認した日時。NULL = 未確認 |
| `recipient_reminder_sent_at` | DATETIME | YES | NULL | 受信者へのリマインド送信日時。NULL = 未送信 |
| `sender_reminder_sent_at` | DATETIME | YES | NULL | 送信者へのアラート送信日時。NULL = 未送信 |
| `resolved_recipient_reminder_minutes` | INT | NO | — | INSERT時に解決済みのリマインド分数を保存（非正規化） |
| `resolved_sender_reminder_minutes` | INT | NO | — | INSERT時に解決済みのアラート分数を保存（非正規化） |
| `confirm_token` | VARCHAR(36) | YES | NULL | メールワンクリック確認用トークン（UUID v4）。UNIQUE |
| `excluded_at` | DATETIME | YES | NULL | チーム脱退等で除外された日時。NULL = 有効 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**制約・インデックス**
```sql
UNIQUE KEY uk_cnr_notification_user (confirmable_notification_id, user_id)

-- STEP 1: 受信者リマインドバッチ用
INDEX idx_cnr_recipient_remind (
    confirmable_notification_id,
    confirmed_at,
    excluded_at,
    recipient_reminder_sent_at,
    deadline_at
)

-- STEP 2: 送信者アラートバッチ用
INDEX idx_cnr_sender_remind (
    confirmable_notification_id,
    confirmed_at,
    excluded_at,
    sender_reminder_sent_at,
    deadline_at
)

-- ユーザー別未確認一覧用
INDEX idx_cnr_user_unconfirmed (user_id, confirmed_at, excluded_at, deadline_at)

-- メールワンクリック確認トークン用
INDEX idx_cnr_token (confirm_token)
```

**備考**
- `deadline_at` は臨時休業の場合は予約スロットの開始日時、汎用の場合は送信者が指定する回答期限
- 確認済み（`confirmed_at IS NOT NULL`）のレコードはリマインドバッチの対象外
- 1通知あたりの受信者上限は 500人（大規模配信は F04.3 のチャンク配信を利用）
- `resolved_recipient_reminder_minutes` / `resolved_sender_reminder_minutes`: INSERT 時にリマインド時間の解決ロジック（通知個別指定 → スコープデフォルト → システムデフォルト）を実行し、解決済みの値を保存する。バッチ処理時に settings テーブルへの JOIN が不要になる
- `excluded_at`: チーム脱退時に未確認の recipients にセットする。バッチ処理でスキップされるが、レコード自体は残す（監査目的）
- `user_id` ON DELETE CASCADE: ユーザー退会時に recipients レコードも削除。`sent_count` はスナップショットとして confirmable_notifications に保持されるため、集計値のずれは許容
- メール送信結果は本テーブルには保存しない。F04.3 の `notifications.channels_sent` に委ねる
- `confirm_token`: メール本文に含めるワンクリック確認用トークン（UUID v4）。送信時に生成し、`GET /api/v1/confirmable-notifications/confirm-by-token?token={uuid}` でログイン不要の確認を実現する。有効期限は `deadline_at + 24時間`。UNIQUE 制約あり

#### `confirmable_notification_templates`

確認通知の送信テンプレート。よく使う件名・本文・重要度をプリセットとして保存し、送信フォームで選択して自動入力できる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id（ポリモーフィック） |
| `name` | VARCHAR(100) | NO | — | テンプレート名（管理用表示名） |
| `subject` | VARCHAR(200) | NO | — | 件名テンプレート |
| `message_body_template` | TEXT | NO | — | 本文テンプレート |
| `default_priority` | VARCHAR(20) | NO | 'NORMAL' | デフォルト重要度（`NORMAL` / `HIGH` / `URGENT`） |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users.id ON DELETE SET NULL（作成者） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時。NULL = 有効 |

**制約・インデックス**
```sql
UNIQUE KEY uk_cnt_scope_name (scope_type, scope_id, name, deleted_at)
    -- 同一スコープ内でテンプレート名の重複防止（削除済み除く）

INDEX idx_cnt_scope (scope_type, scope_id, deleted_at)
```

**備考**
- 論理削除（`deleted_at`）: テンプレートを削除しても、過去にそのテンプレートを使用して送信した通知の参照整合性を保つ
- `scope_type` / `scope_id` のポリモーフィック設計は `confirmable_notification_settings` と同様（FK制約なし、アプリ層でバリデーション）
- `default_priority` は `NORMAL` / `HIGH` / `URGENT` のみ許可（`LOW` は確認通知の性質上不適切）

## 5. ビジネスロジック

### 5.1 リマインド時間の解決

リマインドの事前時間は以下の優先順で解決する。

```
1. confirmable_notifications.recipient_reminder_minutes（通知ごとの個別指定）
2. confirmable_notification_settings（スコープ別デフォルト）
3. システムデフォルト: 受信者 180分 / 送信者 120分
```

値が `0` の場合はそのリマインドを無効化する。

**解決済み値の非正規化**: recipients レコード INSERT 時に上記ロジックで解決した値を `resolved_recipient_reminder_minutes` / `resolved_sender_reminder_minutes` に保存する。バッチ処理時に settings テーブルへの JOIN が不要になり、設定変更が既存の送信済み通知に遡及的に影響しない。

### 5.2 送信フロー

```
ADMIN が確認通知を送信
  │
  ├─ @Transactional 開始
  │     ├─ confirmable_notifications レコード INSERT (status=ACTIVE)
  │     ├─ recipientScope="ALL_MEMBERS" の場合、スコープ内全メンバーを解決
  │     ├─ 受信者リストを batch INSERT → confirmable_notification_recipients
  │     │     ├─ resolved_*_reminder_minutes にリマインド解決済み値を保存
  │     │     └─ confirm_token に UUID v4 を生成・保存（メールワンクリック確認用）
  │     └─ sent_count をセット
  │
  ├─ @TransactionalEventListener(phase = AFTER_COMMIT)
  │     ├─ 受信者ごとに:
  │     │     ├─ アプリ内通知送信（NotificationHelper.notify, sourceType=CONFIRMABLE）
  │     │     │     └─ 送信者自身が受信者に含まれる場合: アプリ内通知スキップ
  │     │     └─ send_email=true なら EmailService でメール送信
  │     │           ├─ メール本文末尾にワンクリック確認リンクを挿入:
  │     │           │   `{baseUrl}/confirm?token={confirm_token}`
  │     │           └─ 送信者自身が受信者に含まれる場合: メールは送信する
  │     └─ メール送信失敗は DB ロールバックしない（非同期のため）
  │
  ├─ 監査ログ記録（audit_logs: action=CONFIRMABLE_NOTIFICATION_SENT）
  │
  └─ 送信完了レスポンス返却
```

**500人のトランザクション設計**: recipients の batch INSERT は DB トランザクション内で行い、メール送信は `@TransactionalEventListener(phase = AFTER_COMMIT)` で DB コミット後に非同期実行する。これにより、メール送信の失敗が DB のロールバックを引き起こさない。

**ドライラン（テスト送信）**: `dryRun = true` の場合、上記フロー全体をスキップし、`recipientUserIds` の解決（ALL_MEMBERS 含む）とバリデーションのみ実行する。DB INSERT・メール送信・通知送信は一切行わない。レスポンスは `{ "data": { "resolvedRecipientCount": 25 } }` を返す。

### 5.3 確認フロー

```
受信者がアプリ内通知の「確認しました」ボタンを押下
  │  （またはメール内ワンクリック確認リンク経由）
  │
  ├─ POST /confirmable-notifications/{id}/confirm
  │  （またはGET /confirmable-notifications/confirm-by-token?token={uuid}）
  │
  ├─ confirmed_at = NOW() に更新
  │
  ├─ ConfirmableNotificationConfirmedEvent を発行
  │     ├─ F04.3 の NotificationEventListener が受信
  │     │     → 対応する notifications レコードを既読化
  │     └─ WebSocket で確認状況をリアルタイム push（→ 5.13 参照）
  │
  ├─ 全受信者が確認済みの場合: status を COMPLETED に更新
  │
  └─ 以降のリマインドバッチでスキップされる
```

### 5.4 2段階リマインドバッチ

1分間隔のバッチジョブ（`@Scheduled(fixedDelay = 60_000)` + `@SchedulerLock`）が以下を実行:

```
毎分実行
  │
  ├─ STEP 0: ACTIVE 通知の絞り込み
  │     SELECT id FROM confirmable_notifications WHERE status = 'ACTIVE'
  │     → 以降の STEP はこの通知IDリストで recipients を絞る
  │
  ├─ STEP 1: 受信者リマインド
  │     条件: confirmable_notification_id IN (ACTIVE通知IDリスト)
  │            AND confirmed_at IS NULL
  │            AND excluded_at IS NULL
  │            AND recipient_reminder_sent_at IS NULL
  │            AND NOW() >= deadline_at - INTERVAL resolved_recipient_reminder_minutes MINUTE
  │            AND deadline_at > NOW()
  │     処理: send_email フラグに従い再通知（アプリ内 + 条件付きメール）
  │           → recipient_reminder_sent_at = NOW()
  │
  └─ STEP 2: 送信者アラート
        条件: confirmable_notification_id IN (ACTIVE通知IDリスト)
               AND confirmed_at IS NULL
               AND excluded_at IS NULL
               AND sender_reminder_sent_at IS NULL
               AND NOW() >= deadline_at - INTERVAL resolved_sender_reminder_minutes MINUTE
               AND deadline_at > NOW()
        処理: 送信者に「○○さんが未確認です」通知（アプリ内 + メール）
              → sender_reminder_sent_at = NOW()
              ※ 送信者が脱退済みの場合: スコープの現ADMIN全員に転送
```

**リマインドバッチのクエリ条件の解説**: `NOW() >= deadline_at - INTERVAL {分数} MINUTE AND deadline_at > NOW()` は「deadline まで残り{分数}分以内 かつ deadline 自体はまだ到来していない」を意味する。旧条件 `deadline_at BETWEEN NOW() AND NOW() + {分数}` は時制が逆転しており誤りだった。

**リマインドのメール送信**: 元通知の `send_email` フラグに従う。`send_email = false` の通知のリマインドではメールを送信しない。

**受信者リマインドの通知内容**:
- `notificationType = 元の通知と同じ型`（EMERGENCY_CLOSURE なら EMERGENCY_CLOSURE）
- `priority = URGENT`
- `title = "【再送】{元の件名}"`
- `sourceType = CONFIRMABLE` / `sourceId = confirmable_notification.id`
- フロントエンドで「確認しました」ボタンが表示される
- 確認通知は opt-out 無視で**強制配信**する（F04.3 の `priority = URGENT` の挙動に準拠）

**送信者アラートの通知内容**:
- `notificationType = CONFIRMABLE_UNCONFIRMED_ALERT`
- `priority = URGENT`
- `title = "【要確認】{受信者名}さんが未確認です"`
- メールも同時送信（send_email フラグに従う）
- 送信者が脱退済みの場合: スコープの現 ADMIN 全員にアラートを転送

### 5.5 F03.4 臨時休業との連携

`EmergencyClosureService.sendClosure()` は以下の処理を維持する:
1. 予約固有ロジック: スロット検索、予約自動キャンセル
2. `confirmable_notifications` + `confirmable_notification_recipients` レコード作成を `ConfirmableNotificationService` に委譲
3. `source_type = 'EMERGENCY_CLOSURE'`, `source_id = reservation_blocked_times.id` で逆参照可能（旧 `emergency_closures` テーブルは統合済み）
4. `deadline_at` には予約スロットの開始日時を設定

既存の臨時休業専用バッチ（`EmergencyClosureReminderBatchService`）は汎用バッチ（`ConfirmableNotificationReminderBatchService`）に統合・置換する。

### 5.6 予約システム利用時の設定必須化

チームが予約モジュール（F03.4）を有効化する際:
1. `confirmable_notification_settings` レコードがなければデフォルト値（受信者180分 / 送信者120分）で自動作成
2. 管理者は `reservation-settings` 画面からリマインド時間を変更可能
3. 設定値は臨時休業通知と汎用確認通知の両方に適用される

#### Micrometer メトリクス

| メトリクス名 | 型 | タグ | 説明 |
|-------------|---|------|------|
| `confirmable.notification.sent.total` | Counter | `scope_type`, `notification_type` | 送信された確認通知の総数 |
| `confirmable.notification.confirmed.total` | Counter | `scope_type`, `notification_type` | 確認された件数 |
| `confirmable.notification.reminder.sent.total` | Counter | `step`(`recipient`/`sender`), `notification_type` | リマインド送信回数 |
| `confirmable.notification.batch.duration` | Timer | `step`(`recipient_remind`/`sender_remind`) | バッチ各ステップの実行時間 |
| `confirmable.notification.batch.processed` | Gauge | — | 直近バッチ実行で処理したレコード数 |
| `confirmable.notification.email.failure.total` | Counter | `notification_type` | メール送信失敗回数 |

#### アラート推奨閾値

| 条件 | 閾値 | アラートレベル |
|------|------|-------------|
| バッチ実行時間 | > 30秒 | WARNING |
| メール送信失敗率 | > 5%（直近1時間） | CRITICAL |
| ACTIVE 通知で全員未確認かつ deadline 超過 | — | WARNING（日次レポート） |

#### 障害リカバリ手順

1. **バッチが長時間停止した場合**: `@SchedulerLock` の lockAtLeastFor を超過すると次回実行でリカバリ。リマインド条件は `*_sent_at IS NULL` のため、停止期間中に条件を満たしたレコードは次回バッチで一括処理される
2. **メール送信が大量失敗した場合**: `confirmable.notification.email.failure.total` のアラートで検知。メール送信結果は `notifications.channels_sent` に記録されるため、失敗した通知を `channels_sent` で検索し手動再送可能
3. **DB コネクション枯渇時**: バッチは `@SchedulerLock` により単一インスタンスで実行。コネクションプール枯渇時は次回バッチまで自動リトライ

### 5.9 期限切れ処理

#### 自動ステータス遷移バッチ

日次バッチ（毎日 AM 3:00 JST）で、全受信者の `deadline_at` が過去かつ `status = 'ACTIVE'` の通知を自動的に `EXPIRED` に遷移させる。

```
@Scheduled(cron = "0 0 3 * * *", zone = "Asia/Tokyo")
@SchedulerLock(name = "confirmableNotificationExpiryBatch", lockAtLeastFor = "5m", lockAtMostFor = "30m")

処理:
  1. SELECT id FROM confirmable_notifications WHERE status = 'ACTIVE'
  2. 各通知について: 全受信者の deadline_at < NOW() かどうかチェック
     （受信者ごとに deadline_at が異なる場合、MAX(deadline_at) < NOW() で判定）
  3. 条件を満たす通知の status を 'EXPIRED' に更新
```

**注意**: EXPIRED への遷移後も、受信者は引き続き確認操作が可能（遅延確認を許可する設計方針のため）。

#### メンバー脱退・退会時の処理

| イベント | 処理 |
|---------|------|
| チーム脱退 | 未確認の recipients に `excluded_at = NOW()` をセット。以降のリマインドバッチでスキップ |
| ユーザー退会 | `user_id` ON DELETE CASCADE で recipients レコード削除。`sent_count` はスナップショット値を維持 |
| 送信者脱退 | 送信者アラートをスコープの現 ADMIN 全員に転送 |

#### データ保持期間

- **保持期間**: 1年
- **1年経過後**: `confirmable_notification_recipients` を物理削除 + `confirmable_notifications.message_body` を NULL にクリア
- **日次バッチ**: 毎日 AM 4:00 JST に実行

#### GDPR 対応

- `confirmable_notification_recipients.user_id`, `confirmable_notifications.created_by` に `@PersonalData` アノテーション付与
- F12.3 `PersonalDataCollector` に登録必須（ユーザーデータエクスポート・削除要求に対応）

### 5.10 F04.3 通知基盤との連携

#### 二重レコード構造

```
確認通知送信時:
  (1) confirmable_notification_recipients  ← 確認追跡（本機能）
  (2) notifications                         ← 配信・未読管理（F04.3）

確認操作時:
  受信者が確認ボタン押下
    → confirmable_notification_recipients.confirmed_at = NOW()
    → ConfirmableNotificationConfirmedEvent 発行
    → notifications.is_read = true（一方向連携）
```

#### WebSocket 連携

- 確認通知の配信は F04.3 の `NOTIFICATION` メッセージを使用する（独自メッセージ種別は追加しない）
- PWA Push の tag は `CONFIRMABLE:{confirmable_notification_id}` を使用。リマインド送信時に同じ tag を使うことで、ブラウザ上の Push 通知が上書きされる（通知の重複を防止）

#### 強制配信

確認通知は opt-out 設定を無視して強制配信する。F04.3 の通知配信フローにおいて:
- `notification_preferences`（スコープ別 OFF）をスキップ
- `notification_type_preferences`（種別別 OFF）をスキップ
- これは `priority = URGENT` の標準挙動に準拠する

#### メール送信結果の管理

メール送信結果は confirmable 系テーブルには保存しない。F04.3 の `notifications.channels_sent` に記録される。障害調査時は `notifications` テーブルの `source_type = 'CONFIRMABLE'` で検索する。

#### メール本文のサニタイズ

メール形式はプレーンテキスト（HTML 埋め込み不可）。本文は `StringEscapeUtils.escapeHtml4` でサニタイズしてから HTML メールテンプレートにはめ込む。

### 5.14 遅延確認の挙動

deadline 経過後の確認は許可する（Section 5.3 で既に規定済み）。

**集計への影響**:
- `confirmedCount` / `totalRecipientCount` には遅延確認も含まれる（確認は確認）
- 期限内確認率を別途算出する場合は `confirmed_at <= deadline_at` で条件を絞る（現時点では API レスポンスには含めない）

**UI表示**:
- **確認状況一覧**: 期限後に確認した受信者には「確認済み（遅延）」ラベルを表示。判定条件: `confirmedAt > deadlineAt`
- **受信者通知カード**: deadline 超過後は「期限切れ -- それでもご確認をお願いします」メッセージを表示し、ボタンは引き続き有効

## 6. セキュリティ考慮事項

- **送信権限**: ADMIN / DEPUTY_ADMIN のみ確認通知を送信可能。MEMBER は受信・確認のみ
- **スコープ検証**: 送信時に `recipientUserIds` が全員スコープ内（チームメンバー or 組織メンバー）であることを検証。`scope_type` / `scope_id` はポリモーフィック設計（FK制約なし、アプリ層でバリデーション）
- **確認API**: 認証済みユーザーが自分宛のレコードのみ確認可能（他人の確認は不可）。受信者レコードが存在しない場合は 200 を返して no-op（ID列挙攻撃防止）
- **確認状況閲覧**: ADMIN+ のみ全受信者の確認状況を閲覧可能
- **recipients レスポンスのプライバシー**: `userEmail` はデフォルト非表示。`?includeEmail=true` かつ ADMIN のみ取得可
- **メール本文**: XSS 防止のため、メール本文は `StringEscapeUtils.escapeHtml4` で sanitize してから HTML メールテンプレートにはめ込む。プレーンテキスト部分への HTML 埋め込みは不可
- **URGENT 通知 / 強制配信**: F04.3 の仕様に従い、確認通知は受信設定 opt-out を無視して強制配信
- **レート制限**: 送信 5回/分、confirm 60回/分、一覧 120回/分
- **監査ログ連携**: 送信（`CONFIRMABLE_NOTIFICATION_SENT`）と取消（`CONFIRMABLE_NOTIFICATION_CANCELLED`）は `audit_logs` テーブルに記録する
- **ワンクリック確認トークン**: トークンは UUID v4 で生成され推測不可能。有効期限は `deadline_at + 24時間`。HTTPS 必須（トークンが平文でURLに含まれるため、HTTP 環境では無効化する）。トークンは1回使用後も再利用可能（冪等設計: `confirmed_at` の更新は初回のみ）。トークンのブルートフォース攻撃対策として、不正トークンの連続試行時は IP ベースのレート制限を適用する
