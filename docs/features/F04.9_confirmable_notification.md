# F04.9: 確認通知システム

> **ステータス**: 🟡 設計完了・実装待ち
> **実装フェーズ**: Phase 4（F04.3 プッシュ通知の拡張）
> **最終更新**: 2026-04-11

---

## 1. 概要

「読んだだけでは終わらない」通知——確認 (confirm) を受信者に求め、未確認者への自動リマインドと送信者への集計ビューを提供する汎用確認通知基盤。

既存の F04.3 プッシュ通知基盤（`notifications` テーブル）に乗せる **二重レコード構造** を採用する。確認追跡ロジックは `confirmable_*` テーブル群が担い、アプリ内通知・メール配信は既存の `NotificationDispatchService` に委譲することで、重複実装を避けつつ確認状態の管理を独立して行う。

### 主なユースケース

| ユースケース | 優先度 |
|------------|------|
| 臨時休業・施設クローズの一括通知 → 受信者の確認取得 | 高 |
| 緊急連絡（大会中止・天候中止等）の確認追跡 | 高 |
| 重要規約・ガイドライン変更の同意取得 | 中 |
| 保護者向け同意書（遠征・合宿参加）の電子確認 | 中 |

## 3. 前段実装について（V3.065 まで）

F03.4 予約管理の設計時点では「臨時休業の一括通知」を `reservation_blocked_times` で賄う設計だった。しかし以下の理由から、予約管理に閉じない **汎用確認通知モジュール** として独立実装することに変更した。

- 予約管理以外のユースケース（緊急連絡・同意書等）にも同じ確認追跡が必要
- `reservation_blocked_times` は「時間帯のブロック」の管理であり、通知・確認追跡のロジックを持たせるには責務が不適切
- 「確認したかどうか」の追跡・リマインド・集計は独立したテーブル設計が望ましい

このため V3.066〜V3.069 相当の `emergency_closure_*` テーブルは **作成しない**。すべて本フィーチャー（V13.005〜）の `confirmable_*` テーブルとして実装する。

## 4. アーキテクチャ方針

### 二重レコード構造

```
確認通知送信
    │
    ├─→ confirmable_notifications (確認追跡の親レコード)
    │       └─→ confirmable_notification_recipients (受信者ごとの確認状態)
    │
    └─→ notifications (既存テーブル / F04.3)  ← 1受信者につき1レコード生成
            └─→ NotificationDispatchService (WebSocket + PWA Push)
```

- `confirmable_*` テーブルは **確認の追跡・リマインド・集計** に特化する
- `notifications` テーブルへのレコード生成は `ApplicationEvent` 経由で疎結合に行う
- 受信者が通知を「既読」にしても「確認」にはならない（2つの操作は独立）

### opt-out 無視（強制配信）

確認通知は `notification_preferences` / `notification_type_preferences` の opt-out 設定を **無視** して強制配信する。重要連絡の到達保証が目的のため。

### ワンクリック確認

各受信者に `confirm_token`（UUID）を発行し、認証不要で確認できる URL（`/confirm/{token}`）を提供する。メールからワンクリックで確認可能にする。

### 送信者=受信者

送信者が自分のスコープメンバーに通知を送る際、自分も受信対象に含まれる場合はアプリ内通知をスキップし、メール送信のみ行う。

### リマインド時間の解決順序（3段フォールバック）

```
通知個別設定（confirmable_notifications.first_reminder_minutes）
    ↓ NULL の場合
スコープ設定（confirmable_notification_settings）
    ↓ NULL の場合
システムデフォルト（first: 180分 / second: 120分）
```

解決した分数値は `confirmable_notification_recipients.resolved_*_reminder_minutes` に **非正規化** して保存する（設定変更後もリマインドタイミングが変わらないよう）。

### 500人トランザクション対応

受信者 500 人規模を想定し、`confirmable_notification_recipients` は batch INSERT を使用する。メール送信は `@TransactionalEventListener` の非同期処理に委譲してトランザクション外で実行する。

### テーブル定義

#### `confirmable_notification_settings`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|

| `scope_id` | BIGINT UNSIGNED | NO | — | 対象スコープの ID |
| `default_first_reminder_minutes` | INT | YES | NULL | 未確認者へのリマインド1回目の分数。NULL = システムデフォルト(180) |
| `default_second_reminder_minutes` | INT | YES | NULL | リマインド2回目の分数。NULL = システムデフォルト(120) |
| `sender_alert_threshold_percent` | INT | NO | 80 | 確認率がこのパーセント未満の場合、送信者にアラートを送る閾値 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_cns_scope (scope_type, scope_id)   -- スコープ単位で一意
```

#### `confirmable_notifications`

確認通知の親レコード。1回の送信につき1レコード。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION') | NO | — | 送信先スコープ種別 |
| `scope_id` | BIGINT UNSIGNED | NO | — | 送信先スコープの ID |
| `title` | VARCHAR(200) | NO | — | 通知タイトル（例: 「臨時休業のお知らせ」） |
| `body` | TEXT | YES | NULL | 本文 |
| `priority` | ENUM('NORMAL', 'HIGH', 'URGENT') | NO | 'NORMAL' | 優先度（LOW は使用しない） |
| `status` | ENUM('ACTIVE', 'COMPLETED', 'EXPIRED', 'CANCELLED') | NO | 'ACTIVE' | 通知ステータス |
| `deadline_at` | DATETIME | YES | NULL | 確認期限日時。NULL = 期限なし |
| `first_reminder_minutes` | INT | YES | NULL | リマインド1回目の分数（NULL = スコープ設定 or システムデフォルト） |
| `second_reminder_minutes` | INT | YES | NULL | リマインド2回目の分数（NULL = スコープ設定 or システムデフォルト） |
| `action_url` | VARCHAR(500) | YES | NULL | ディープリンク先（例: `/teams/1/schedules/10`） |
| `cancelled_at` | DATETIME | YES | NULL | キャンセル日時 |
| `cancelled_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）キャンセル操作者 |
| `completed_at` | DATETIME | YES | NULL | 全員確認完了日時（全受信者が確認した際にセット） |
| `expired_at` | DATETIME | YES | NULL | 期限切れとしてバッチが EXPIRED にした日時 |
| `total_recipient_count` | INT | NO | 0 | 送信時点の受信者数（スナップショット） |
| `template_id` | BIGINT UNSIGNED | YES | NULL | FK → confirmable_notification_templates（ON DELETE SET NULL）使用テンプレート |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）送信した ADMIN |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_cn_scope (scope_type, scope_id, created_at DESC)   -- スコープ別の通知一覧
INDEX idx_cn_status (status, created_at DESC)                 -- ACTIVE 通知の抽出用
INDEX idx_cn_deadline (status, deadline_at)                   -- 期限切れバッチ用
INDEX idx_cn_created_by (created_by)                          -- 送信者の発信履歴
```

**ステータス遷移**
```
ACTIVE
  ├─→ COMPLETED  (全受信者が確認した)
  ├─→ EXPIRED    (deadline_at を過ぎてもACTIVEの場合、バッチが日次 AM3:00 に更新)
  └─→ CANCELLED  (ADMIN が手動でキャンセル)
```

#### `confirmable_notification_recipients`

受信者ごとの確認状態。送信時に batch INSERT で生成する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `confirmable_notification_id` | BIGINT UNSIGNED | NO | — | FK → confirmable_notifications（ON DELETE CASCADE） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE） |
| `confirm_token` | VARCHAR(36) | NO | — | UUID。認証不要の確認 URL に使用（`/confirm/{token}`） |
| `is_confirmed` | BOOLEAN | NO | FALSE | 確認済みフラグ |
| `confirmed_at` | DATETIME | YES | NULL | 確認日時 |
| `confirmed_via` | ENUM('APP', 'TOKEN', 'BULK') | YES | NULL | 確認方法。APP=アプリ内、TOKEN=トークンURL、BULK=管理者一括 |
| `resolved_first_reminder_minutes` | INT | YES | NULL | 解決済みリマインド1回目の分数（非正規化）。NULL = リマインドなし |
| `resolved_second_reminder_minutes` | INT | YES | NULL | 解決済みリマインド2回目の分数（非正規化）。NULL = 2回目リマインドなし |
| `first_reminder_sent_at` | DATETIME | YES | NULL | リマインド1回目の送信日時 |
| `second_reminder_sent_at` | DATETIME | YES | NULL | リマインド2回目の送信日時 |
| `excluded_at` | DATETIME | YES | NULL | 除外日時（ADMIN が個別受信者を対象外にした場合） |
| `excluded_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）除外した ADMIN |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_cnr_notification_user (confirmable_notification_id, user_id)   -- 重複防止
UNIQUE KEY uq_cnr_token (confirm_token)                                        -- トークン検索
INDEX idx_cnr_user_unconfirmed (user_id, is_confirmed, created_at DESC)       -- 保留中一覧用
INDEX idx_cnr_reminder (confirmable_notification_id, is_confirmed, first_reminder_sent_at, second_reminder_sent_at)   -- リマインドバッチ用
INDEX idx_cnr_excluded (excluded_at)                                           -- 除外済みフィルタ用
```

**制約・備考**
- `excluded_at IS NOT NULL` の受信者はリマインド対象から除外する
- `confirm_token` は送信時に `UUID.randomUUID().toString()` で生成する
- `resolved_*_reminder_minutes` は送信時に解決して保存する（後から設定変更しても既送信のリマインドタイミングは変わらない）

| `scope_type` | String | No | `TEAM` / `ORGANIZATION` でフィルタ |
| `scope_id` | Long | No | スコープ ID でフィルタ |

## 8. バッチ設計

### ConfirmableNotificationReminderBatchService

**実行間隔**: 1分ごと（`@Scheduled(fixedDelay = 60_000)`）

**処理フロー**:
```
STEP 0: status = 'ACTIVE' の confirmable_notifications を取得（INDEX idx_cn_status 使用）
STEP 1: 各通知の受信者リスト（is_confirmed = false AND excluded_at IS NULL）を取得
    - resolved_first_reminder_minutes が経過しており first_reminder_sent_at IS NULL → リマインド1回目送信
    - resolved_second_reminder_minutes がさらに経過しており second_reminder_sent_at IS NULL → リマインド2回目送信
STEP 2: 送信者アラート確認
    - 確認率が sender_alert_threshold_percent を下回る場合、created_by ユーザーに未確認アラート通知
    - アラートは1通知につき1回のみ送信（DB フラグまたは Valkey で管理）
```

**リマインド時間計算**:
```
リマインド1回目送信条件:
    created_at + resolved_first_reminder_minutes <= NOW()
    AND first_reminder_sent_at IS NULL

リマインド2回目送信条件:
    first_reminder_sent_at + resolved_second_reminder_minutes <= NOW()
    AND second_reminder_sent_at IS NULL
```

---

### ConfirmableNotificationExpiryBatchService

**実行間隔**: 日次 AM 3:00（`@Scheduled(cron = "0 0 3 * * *")`）

**処理フロー**:
```
1. status = 'ACTIVE' AND deadline_at < NOW() の通知を取得
2. status を 'EXPIRED' に更新、expired_at = NOW() をセット
3. 未確認受信者が残っている場合、送信者 (created_by) に期限切れ通知を送信
```
