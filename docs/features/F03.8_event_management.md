## 1. 概要

大規模イベント（大会・発表会・総会・文化祭・交流会）のライフサイクル全体を管理する機能。F03.1 のスケジュールが「日程 + 出欠」に特化しているのに対し、イベント管理は「企画 → 参加登録 → チケット発行 → 受付チェックイン → 事後処理」の一連のフローを提供する。`events` テーブルは `schedules` テーブルの拡張情報として機能し、FK（`schedule_id`）で紐付ける。

**主要機能**:
- **イベント作成**: スケジュール（F03.1）を拡張し、参加登録フォーム・チケット・受付機能を追加
- **外部参加者の登録**: ゲスト招待 URL を発行。メンバー以外も参加登録可能
- **チケット発行**: QR コード付きチケット（F02.1 連携）。有料イベントは F08.2 で事前決済
- **受付チェックイン**: QR スキャンで参加確認。チェックイン率のリアルタイム表示
- **事後アンケート**: イベント終了後に参加者へアンケートを自動送信（F05.4 連携）
- **イベントページ**: 公開イベントページ（SSR。SEO 対応）。概要・タイムテーブル・アクセス情報を掲載
- **企画承認**: F05.6 ワークフローで企画書の承認を取得

**ユースケース**:
- スポーツチーム: 大会運営（チーム登録 → 受付 → 試合進行 → 結果記録）
- 学校: 文化祭・発表会（企画承認 → チケット配布 → 当日受付 → 事後アンケート）
- コミュニティ: 交流イベント（参加登録 → 決済 → QR 受付 → フォトギャラリー連携）
- マンション: 総会（出席登録 → 委任状 → 受付 → 議事録連携）

**インデックス**
```sql
INDEX idx_ev_scope (scope_type, scope_id, status, created_at DESC)   -- スコープ別イベント一覧
UNIQUE KEY uq_ev_schedule (schedule_id)                               -- スケジュールとの 1:1 対応
UNIQUE KEY uq_ev_slug (slug)                                          -- スラッグの一意性（公開 URL 用）
INDEX idx_ev_status (status, registration_ends_at)                    -- 登録締切バッチ用
INDEX idx_ev_post_survey (post_survey_id)                              -- アンケート連携の逆引き
INDEX idx_ev_workflow (workflow_request_id)                            -- ワークフロー連携の逆引き
INDEX idx_ev_public (is_public, status, created_at DESC)              -- 公開イベント一覧（SSR）
```

**制約・備考**
- `scope_type` + `scope_id`: ポリモーフィック FK。Service 層で `scope_type = 'TEAM'` なら `teams`、`scope_type = 'ORGANIZATION'` なら `organizations` の存在を検証
- `schedule_id`: `schedules` テーブルとの 1:1 対応。イベント作成時にスケジュールも同時に作成する（トランザクション内）。スケジュールの `title`・`start_at`・`end_at`・`location` はイベントの基本情報と同期する
- `slug`: URL セーフな英数字・ハイフンのみ（`[a-z0-9-]{3,100}`）。自動生成はイベント名のローマ字化 + ランダム6桁。手動上書き可能
- `status` のライフサイクル:
  - `DRAFT` → `PUBLISHED`（公開。参加登録は `registration_starts_at` まで不可）
  - `PUBLISHED` → `REGISTRATION_OPEN`（参加登録受付中。`registration_starts_at` 到来で自動遷移、または手動遷移）
  - `REGISTRATION_OPEN` → `REGISTRATION_CLOSED`（参加登録締切。`registration_ends_at` 到来で自動遷移、定員到達で自動遷移、または手動遷移）
  - `REGISTRATION_CLOSED` → `IN_PROGRESS`（イベント開催中。`schedules.start_at` 到来で自動遷移、または手動遷移）
  - `IN_PROGRESS` → `COMPLETED`（終了。`schedules.end_at` 到来で自動遷移、または手動遷移）
  - `DRAFT` / `PUBLISHED` / `REGISTRATION_OPEN` / `REGISTRATION_CLOSED` → `CANCELLED`（中止。手動のみ）
  - `COMPLETED` / `CANCELLED` からの遷移は不可
- `is_public = TRUE`: `/events/{slug}` の公開ページ（SSR）が有効になる。OGP メタタグを出力し、SNS シェア時にリッチプレビューを表示
- `is_approval_required = TRUE`: 参加登録時に `event_registrations.status = 'PENDING'` で作成。ADMIN が承認すると `APPROVED` に遷移
- `max_capacity`: `registration_count >= max_capacity` で自動的に `REGISTRATION_CLOSED` に遷移
- `post_survey_id`: イベント `COMPLETED` 遷移の翌日 09:00 JST に `SurveyDistributionEvent` を発行し、F05.4 のアンケート配信フローに委譲する
- `workflow_request_id`: F05.6 で企画承認を取得する場合に設定。`workflow_requests.source_type = 'EVENT'`、`source_id = events.id`
- 論理削除: `DRAFT` / `CANCELLED` のイベントのみ削除可能。`PUBLISHED` 以降は証跡保持のため削除不可

#### `event_ticket_types`

チケット種別定義。1イベントに複数のチケット種別（一般・VIP・学生・招待等）を設定可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `event_id` | BIGINT UNSIGNED | NO | — | FK → events。ON DELETE CASCADE |
| `name` | VARCHAR(100) | NO | — | 種別名（例: 「一般」「VIP」「学生」「招待」） |
| `description` | VARCHAR(500) | YES | NULL | 種別の説明 |
| `price` | DECIMAL(12,0) | NO | 0 | チケット価格（0 = 無料） |
| `currency` | CHAR(3) | NO | 'JPY' | 通貨コード（ISO 4217） |
| `max_quantity` | INT UNSIGNED | YES | NULL | 種別ごとの上限枚数（NULL = 無制限。イベント全体の `max_capacity` 内） |
| `issued_count` | INT UNSIGNED | NO | 0 | 発行済み枚数（denormalize） |
| `min_registration_role` | VARCHAR(30) | NO | 'MEMBER+' | この種別を選択可能な最小権限。イベント全体の `min_registration_role` より狭い範囲に制限可能 |
| `is_active` | BOOLEAN | NO | TRUE | 販売中か（FALSE = 新規購入不可。既存チケットは有効） |
| `sort_order` | INT | NO | 0 | 表示順 |
| `payment_item_id` | BIGINT UNSIGNED | YES | NULL | FK → payment_items（F08.2）。ON DELETE SET NULL。有料チケットの場合の Stripe 決済連携。NULL = 無料チケット |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_ett_event (event_id, is_active, sort_order)   -- イベント別チケット種別一覧
INDEX idx_ett_payment (payment_item_id)                  -- 決済連携の逆引き
```

**制約・備考**
- 1イベントあたり最大 **10 種別**（アプリ層でバリデーション）
- `price > 0` の場合は `payment_item_id` が必須。イベント作成時に `PaymentItemService` で `payment_items` レコードを自動生成する
- `max_quantity` を超える発行は不可。`issued_count >= max_quantity` で自動的に `is_active = FALSE` に遷移
- チケット種別の削除は `issued_count = 0` の場合のみ物理削除可能。発行済みがある場合は `is_active = FALSE` で無効化
- `min_registration_role`: イベント全体の `min_registration_role` より緩い設定は不可（アプリ層でバリデーション）

#### `event_registrations`

参加登録レコード。メンバーと外部ゲストの両方を同一テーブルで管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `event_id` | BIGINT UNSIGNED | NO | — | FK → events。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL（メンバー登録の場合。NULL = 外部ゲスト） |
| `guest_name` | VARCHAR(100) | YES | NULL | 外部ゲストの氏名（`user_id IS NULL` の場合は必須） |
| `guest_email` | VARCHAR(254) | YES | NULL | 外部ゲストのメールアドレス（チケット送信・連絡用） |
| `guest_phone` | VARCHAR(20) | YES | NULL | 外部ゲストの電話番号（任意） |
| `ticket_type_id` | BIGINT UNSIGNED | NO | — | FK → event_ticket_types。ON DELETE RESTRICT。選択したチケット種別 |
| `status` | VARCHAR(20) | NO | 'PENDING' | 登録ステータス（`PENDING` / `APPROVED` / `REJECTED` / `CANCELLED` / `WAITLISTED`） |
| `quantity` | INT UNSIGNED | NO | 1 | チケット枚数（グループ登録対応。1〜10 枚） |
| `note` | VARCHAR(500) | YES | NULL | 備考（参加者が入力） |
| `approved_by` | BIGINT UNSIGNED | YES | NULL | FK → users。承認者（ON DELETE SET NULL） |
| `approved_at` | DATETIME | YES | NULL | 承認日時 |
| `cancelled_at` | DATETIME | YES | NULL | キャンセル日時 |
| `cancel_reason` | VARCHAR(500) | YES | NULL | キャンセル理由 |
| `invite_token_id` | BIGINT UNSIGNED | YES | NULL | FK → event_guest_invite_tokens。ON DELETE SET NULL。外部ゲストの場合の招待トークン |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_er_event_status (event_id, status, created_at DESC)       -- イベント別参加者一覧
INDEX idx_er_user (user_id, event_id)                                -- ユーザー別参加イベント一覧
INDEX idx_er_ticket_type (ticket_type_id, status)                    -- チケット種別別集計
INDEX idx_er_guest_email (guest_email, event_id)                     -- ゲストメールで重複確認
UNIQUE KEY uq_er_user_event (user_id, event_id)                      -- メンバーの重複登録防止（user_id NOT NULL の場合のみ有効）
```

**制約・備考**
- `user_id IS NULL` の場合は外部ゲスト。`guest_name` + `guest_email` が必須（アプリ層でバリデーション）
- `user_id IS NOT NULL` の場合はメンバー登録。`guest_name` / `guest_email` / `guest_phone` は NULL
- `uq_er_user_event`: MySQL では `user_id` が NULL の場合は UNIQUE 制約の対象外となるため、外部ゲストの重複チェックは `guest_email` + `event_id` でアプリ層で行う
- `status` 遷移:
  - `PENDING` → `APPROVED`（ADMIN 承認、または `is_approval_required = FALSE` の場合は自動承認）
  - `PENDING` → `REJECTED`（ADMIN 拒否）
  - `PENDING` / `APPROVED` → `CANCELLED`（参加者本人またはADMINがキャンセル）
  - `PENDING` → `WAITLISTED`（定員超過。キャンセル発生時に FIFO で自動 APPROVED）
  - `WAITLISTED` → `APPROVED`（キャンセルによる空き発生時）
  - `WAITLISTED` → `CANCELLED`（待機中にキャンセル）
- `quantity`: 1 登録で複数枚のチケットを取得可能（グループ登録）。`event_tickets` が `quantity` 分生成される
- `APPROVED` 時にチケット自動発行（`event_tickets` に INSERT + `event_ticket_types.issued_count` を INCREMENT）
- 有料チケットの場合: `PENDING` 状態で Stripe Checkout セッションを開始し、決済成功の webhook 受信後に `APPROVED` に遷移（`is_approval_required = FALSE` の場合）

#### `event_tickets`

発行済みチケット。1参加登録につき `quantity` 枚が生成される。QR コードで受付チェックインに使用。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `registration_id` | BIGINT UNSIGNED | NO | — | FK → event_registrations。ON DELETE CASCADE |
| `event_id` | BIGINT UNSIGNED | NO | — | FK → events。ON DELETE CASCADE（冗長だが高速検索用） |
| `ticket_type_id` | BIGINT UNSIGNED | NO | — | FK → event_ticket_types。ON DELETE RESTRICT |
| `qr_token` | CHAR(36) | NO | — | QR コードのペイロード（UUID v4）。F02.1 QR スキャン基盤で読取 |
| `ticket_number` | VARCHAR(30) | NO | — | チケット番号（表示用。形式: `{event_slug}-{連番4桁}`） |
| `status` | VARCHAR(20) | NO | 'VALID' | ステータス（`VALID` / `USED` / `CANCELLED` / `EXPIRED`） |
| `used_at` | DATETIME | YES | NULL | チェックイン日時（`VALID` → `USED` 遷移時に設定） |
| `cancelled_at` | DATETIME | YES | NULL | キャンセル日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_et_qr_token (qr_token)                                 -- QR トークンの一意性
UNIQUE KEY uq_et_ticket_number (ticket_number)                        -- チケット番号の一意性
INDEX idx_et_event_status (event_id, status)                          -- イベント別チケット一覧
INDEX idx_et_registration (registration_id)                           -- 登録別チケット一覧
```

**制約・備考**
- `qr_token`: UUID v4 で生成。QR コードには `https://{domain}/events/checkin/{qr_token}` をエンコード。F02.1 のスキャン認証とは独立した専用エンドポイントを使用（会員証 QR とイベントチケット QR の混同を防止）
- `ticket_number`: イベント単位で連番。形式: `{slug}-{4桁連番}`（例: `summer-fest-2026-0001`）
- `status` 遷移:
  - `VALID` → `USED`（チェックイン完了）
  - `VALID` → `CANCELLED`（参加登録キャンセルに連動、またはADMIN手動）
  - `VALID` → `EXPIRED`（イベント `COMPLETED` 後に未使用チケットをバッチで自動失効）
  - `USED` / `CANCELLED` / `EXPIRED` からの遷移は不可
- チケット表示: フロントエンドで `qr_token` から QR コードを動的生成（画像は保存しない。F02.1 と同方式）
- 1チケット = 1回のチェックイン。USED 後の再チェックインは不可

#### `event_checkins`

受付チェックイン履歴。QR スキャンによる参加確認の証跡。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `event_id` | BIGINT UNSIGNED | NO | — | FK → events。ON DELETE CASCADE（高速集計用） |
| `ticket_id` | BIGINT UNSIGNED | NO | — | FK → event_tickets。ON DELETE CASCADE |
| `checked_in_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL（スタッフスキャン時のスキャン者。セルフチェックイン時は NULL） |
| `checkin_type` | VARCHAR(20) | NO | 'STAFF_SCAN' | チェックイン方式（`STAFF_SCAN` / `SELF`） |
| `checked_in_at` | DATETIME | NO | CURRENT_TIMESTAMP | チェックイン日時 |
| `note` | VARCHAR(300) | YES | NULL | 備考（スタッフメモ） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_ec_ticket (ticket_id)                                    -- 1チケット1チェックイン
INDEX idx_ec_event (event_id, checked_in_at DESC)                     -- イベント別チェックイン履歴
INDEX idx_ec_staff (checked_in_by, checked_in_at DESC)                -- スタッフ別チェックイン履歴
```

**制約・備考**
- 1チケットにつき1回のみチェックイン可能（`uq_ec_ticket`）。USED チケットの再スキャンは 409 で拒否
- チェックイン時に `event_tickets.status = 'USED'`・`used_at = NOW()` を同時更新 + `events.checkin_count` を INCREMENT
- チェックイン履歴は物理削除しない（監査証跡として保持）
- `checkin_type = 'STAFF_SCAN'`: ADMIN / DEPUTY_ADMIN（`CHECKIN_EVENTS` 権限）がスタッフ端末からスキャン
- `checkin_type = 'SELF'`: 会場に設置された QR ポップ（`checkin_locations` F02.1）とは別方式。参加者がチケットの QR を自分のスマホで見せる方式

外部ゲスト向けの招待トークン。メンバー以外がイベントに参加登録するための URL を発行する。

| カラム名 | 型 | NULL | デフォルト | 説明 |

| `label` | VARCHAR(100) | YES | NULL | 招待用途のラベル（例: 「一般公開用」「関係者限定」） |
| `max_uses` | INT UNSIGNED | YES | NULL | 最大利用回数（NULL = 無制限） |
| `used_count` | INT UNSIGNED | NO | 0 | 利用回数 |
| `expires_at` | DATETIME | YES | NULL | 有効期限（NULL = イベント終了まで有効） |

**制約・備考**

- `expires_at` 超過、`is_active = FALSE`、イベント `status = REGISTRATION_CLOSED` 以降は 403

## 5. ビジネスロジック

### 主要フロー

#### イベント作成フロー
```
1. ADMIN が POST /events で作成リクエストを送信
2. EventService がバリデーション実行
   - モジュール有効チェック（選択式モジュール #16）
   - 権限チェック（MANAGE_EVENTS）
   - 日時整合性チェック（start_at < end_at、registration_ends_at < start_at 等）
3. トランザクション内で以下を実行:
   a. schedules テーブルに INSERT（event_type = 'EVENT'、attendance_required = FALSE）
   b. events テーブルに INSERT（schedule_id = 上記の PK）
   c. event_ticket_types に INSERT（リクエストの ticket_types 配列分）
   d. event_timetable_items に INSERT（リクエストの timetable 配列分）
   e. 有料チケットがある場合: PaymentItemService で payment_items を自動生成
4. EventCreatedEvent を発行
5. レスポンスを返却（201 Created）
```

#### 参加登録フロー（メンバー）
```
1. メンバーが POST /events/{id}/registrations で登録リクエスト送信
2. EventRegistrationService がバリデーション実行
   - イベント status チェック（REGISTRATION_OPEN のみ受付）
   - 権限チェック（min_registration_role 以上）
   - 重複チェック（user_id + event_id のユニーク制約）
   - 定員チェック（registration_count + quantity <= max_capacity）
   - チケット種別の在庫チェック（issued_count + quantity <= max_quantity）
3. トランザクション内で以下を実行:
   a. event_registrations に INSERT
   b. [無料 + 承認不要] status = APPROVED → チケット自動発行
      - event_tickets に quantity 分 INSERT（qr_token = UUID v4）
      - event_ticket_types.issued_count を INCREMENT
      - events.registration_count を INCREMENT
   c. [有料] Stripe Checkout セッション作成 → status = PENDING のまま待機
   d. [承認必要] status = PENDING → ADMIN へ通知
   e. [定員超過] status = WAITLISTED → キャンセル待ちリストに追加
4. EventRegistrationCreatedEvent を発行（通知サービスが購読）
```

#### 有料チケット決済フロー
```
1. メンバーが参加登録 → status = PENDING で作成
2. フロントエンドが Stripe Checkout にリダイレクト
3. 決済成功 → Stripe Webhook（checkout.session.completed）受信
4. PaymentWebhookService が処理:
   a. member_payments に PAID レコードを INSERT
   b. event_registrations.status を APPROVED に更新
   c. チケット自動発行（event_tickets INSERT）
   d. events.registration_count を INCREMENT
5. 決済失敗 / 期限切れ → event_registrations.status = CANCELLED
```

#### 受付チェックインフロー
```
1. スタッフが QR スキャン → POST /events/checkin に qr_token を送信
2. EventCheckinService がバリデーション実行
   - qr_token → event_tickets を検索
   - ticket.status = 'VALID' チェック（USED → 409、CANCELLED/EXPIRED → 409）
   - イベント status チェック（IN_PROGRESS または REGISTRATION_CLOSED のみ受付）
3. トランザクション内で以下を実行:
   a. event_tickets.status = 'USED'、used_at = NOW()
   b. event_checkins に INSERT
   c. events.checkin_count を INCREMENT
4. EventCheckinEvent を発行（リアルタイムダッシュボード更新用）
5. チェックイン結果を返却（参加者情報・チェックイン率含む）
```

#### 事後アンケート自動配信フロー
```
1. イベント status が COMPLETED に遷移
2. 翌日 09:00 JST に EventPostSurveyJob（日次バッチ）が実行
3. events.post_survey_id が設定されているイベントを検索
4. SurveyDistributionEvent を発行
   - 対象者: event_registrations（status = APPROVED）のうち user_id IS NOT NULL のメンバー
   - 外部ゲストには guest_email にメール送信（アンケート回答 URL 付き）
5. F05.4 のアンケートサービスが購読して配信処理を実行
```

#### イベントステータス自動遷移（日次/分次バッチ）
```
1. EventStatusTransitionJob が定期実行（5分間隔）
2. 以下の条件でステータスを自動遷移:
   a. PUBLISHED → REGISTRATION_OPEN:
      - registration_starts_at IS NULL または registration_starts_at <= NOW()
   b. REGISTRATION_OPEN → REGISTRATION_CLOSED:
      - registration_ends_at IS NOT NULL かつ registration_ends_at <= NOW()
      - または registration_count >= max_capacity（定員到達）
   c. REGISTRATION_CLOSED → IN_PROGRESS:
      - schedules.start_at <= NOW()
   d. IN_PROGRESS → COMPLETED:
      - schedules.end_at IS NOT NULL かつ schedules.end_at <= NOW()
3. 各遷移で ApplicationEvent を発行（通知送信等のトリガー）
```

### 重要な判定ロジック

- **参加登録権限の判定**: `events.min_registration_role` と `event_ticket_types.min_registration_role` の両方を満たす必要がある。チケット種別のほうが厳しい場合、そのチケット種別は表示されない
- **定員管理**: `events.registration_count` は `APPROVED` ステータスの登録のみカウント。PENDING / WAITLISTED は含まない。キャンセル時に DECREMENT し、WAITLISTED の先頭を自動 APPROVE
- **チケット在庫管理**: `event_ticket_types.issued_count` は発行済み（`VALID` + `USED`）チケット数。`CANCELLED` は含まない。キャンセル時に DECREMENT
- **外部ゲストの重複判定**: `guest_email` + `event_id` の組み合わせでアプリ層チェック。同一メールでの重複登録は 409
- **セルフチェックイン**: チケット QR を直接スキャンするため認証不要。`qr_token` の UUID 自体がセキュリティトークンとして機能（推測不可能）

## 6. セキュリティ考慮事項

- **認可チェック**: `EventService` の入り口で `scope_type` + `scope_id` と `currentUser` の所属 + ロールを必ず検証する
- **モジュール有効チェック**: 全 API の入り口で選択式モジュール #16 がスコープ内で有効化されているか検証する
- **QR トークンの安全性**: `event_tickets.qr_token` は UUID v4（128bit ランダム）で推測不可能。ブルートフォース対策としてチェックイン API にレートリミット（`Bucket4j` で 1分間に 60 回 / IP）を適用
- **外部ゲスト招待トークン**: UUID v4 + 有効期限 + 利用回数上限で保護。期限切れ・上限到達で即座に失効
- **公開イベントページ**: `is_public = TRUE` の場合のみ SSR 公開。XSS 対策として `summary`・`venue_access_info` の Markdown レンダリング時にサニタイズを必須とする
- **有料チケットの不正防止**: Stripe Webhook の署名検証を必須とし、Webhook 以外の経路でのステータス変更は不可（ADMIN 手動を除く）
- **レートリミット**: 参加登録 API に `Bucket4j` で 1分間に 10 回 / ユーザーの制限を適用（Bot による大量登録防止）
- **個人情報保護**: 外部ゲストの `guest_email`・`guest_phone` は ADMIN のみ参照可能。API レスポンスでは MEMBER 以下にはマスク処理（`y***@example.com`）

## 10. Phase 2 — 出欠確認・事前アンケート選択

### 10.1 概要

イベント作成時に「参加方式」と「アンケート設定」を選択できるようにする。

**参加方式（attendance_mode）**:

| モード | 説明 | 使用テーブル |
|--------|------|-------------|
| `NONE` | 自由参加。登録不要 | — |
| `RSVP` | 出欠確認（ATTENDING/NOT_ATTENDING/MAYBE/UNDECIDED） | `event_rsvp_responses`（新規） |
| `REGISTRATION` | 参加登録（チケット・承認フロー）★既存デフォルト | `event_registrations`（既存） |

**アンケート設定**（参加方式と独立して任意選択）:

| フィールド | 既存/新規 | 説明 |
|-----------|---------|------|
| `pre_survey_id` | 新規 | 事前アンケート（参加者に回答させる） |
| `post_survey_id` | 既存 | 事後アンケート（終了後に自動配信） |

### 10.2 DB 設計

#### events テーブル追加カラム（V13.025）

```sql
ALTER TABLE events
  ADD COLUMN attendance_mode VARCHAR(20) NOT NULL DEFAULT 'REGISTRATION'
    COMMENT 'NONE=自由参加 / RSVP=出欠確認 / REGISTRATION=参加登録',
  ADD COLUMN pre_survey_id BIGINT UNSIGNED NULL
    COMMENT '事前アンケートFK (F05.4連携)',
  ADD CONSTRAINT fk_events_pre_survey
    FOREIGN KEY (pre_survey_id) REFERENCES surveys(id) ON DELETE SET NULL;
```

#### event_rsvp_responses テーブル（V13.026・新規）

```sql
CREATE TABLE event_rsvp_responses (
  id           BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  event_id     BIGINT UNSIGNED NOT NULL,
  user_id      BIGINT UNSIGNED NOT NULL,
  response     VARCHAR(20)     NOT NULL DEFAULT 'UNDECIDED'
               COMMENT 'ATTENDING / NOT_ATTENDING / MAYBE / UNDECIDED',
  comment      VARCHAR(500),
  responded_at DATETIME,
  created_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uq_rsvp_event_user (event_id, user_id),
  CONSTRAINT fk_rsvp_event FOREIGN KEY (event_id) REFERENCES events(id) ON DELETE CASCADE,
  CONSTRAINT fk_rsvp_user  FOREIGN KEY (user_id)  REFERENCES users(id)  ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**RSVP レコードの自動生成**: `attendance_mode=RSVP` のイベントで `openRegistration` 呼び出し時、スコープ（チーム/組織）のメンバー全員の UNDECIDED レコードを自動生成する。

### 10.4 ビジネスロジック

**RSVP フロー**:
1. イベント作成時に `attendanceMode=RSVP` を指定
2. `openRegistration` API 呼び出し時にメンバー全員の UNDECIDED レコードを自動生成
3. 各メンバーが `POST .../rsvp-responses` で回答（初回）/ `PUT .../rsvp-responses/me` で更新
4. 管理者が `/summary` で集計を確認

**事前アンケートフロー**:
1. 別途 F05.4 で作成した survey の `id` を `preSurveyId` に設定してイベント作成
2. `openRegistration` 時に survey の配信対象（surveyTargets）にメンバーを追加
3. メンバーはアンケートページで回答

**バリデーション**:
- `attendance_mode=RSVP` でかつ `max_capacity` が設定されている場合: 警告を返す（RsvpはキャパシのSoft limit扱い）
- `submitRsvp` で既存レコードがある場合: 409 `already_rsvped`（`updateRsvp` を使うよう促す）
- `response` の値: `ATTENDING / NOT_ATTENDING / MAYBE / UNDECIDED` 以外は 400
