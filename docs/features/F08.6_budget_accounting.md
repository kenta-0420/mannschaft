# F08.6: 予算・会計管理

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 8
> **最終更新**: 2026-03-19
> **モジュール種別**: 選択式モジュール #14（組織・チーム対象）
> **関連ドキュメント**: F05.6 ワークフロー・承認エンジン（支出承認連携）、F08.2 会費徴収（収入自動計上）、F05.5 ファイル共有（レシート画像参照）、F04.3 プッシュ通知（予算超過アラート）
>

## 1. 概要

組織・チームの年度予算策定から日々の収支記録、予算消化率のリアルタイム可視化、月次・年次報告書の自動生成までを一元管理する会計機能。費目（カテゴリ）ごとに予算枠を設定し、収入・支出を記録することで、会計担当者の負担を軽減しつつ透明性の高い収支管理を実現する。一定額以上の支出は F05.6 承認エンジンとの連携により承認ワークフローを自動起動し、不正支出を抑止する。F08.2 の会費入金を収入として自動計上する連携により、手動入力の重複を排除する。

**主要ユースケース**:
- スポーツチーム: 遠征費・用具費・大会参加費の予算管理、保護者への収支報告
- PTA: 活動費の予算策定、支出承認、年度末報告書の自動生成
- マンション管理組合: 修繕積立金・管理費の収支管理、総会資料の自動生成
- コミュニティ: イベント予算、スポンサー収入の管理

### テーブル定義

#### `budget_fiscal_years`

会計年度の定義。組織・チームごとに年度の開始月・終了月を自由に設定可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `name` | VARCHAR(100) | NO | — | 年度名（例: 「2026年度」「2026-2027」） |
| `start_date` | DATE | NO | — | 年度開始日（例: 2026-04-01） |
| `end_date` | DATE | NO | — | 年度終了日（例: 2027-03-31） |
| `status` | VARCHAR(20) | NO | 'OPEN' | 年度ステータス（`OPEN` / `CLOSED`） |
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users。作成した ADMIN。ON DELETE RESTRICT |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_bfy_scope (scope_type, scope_id, status)           -- スコープ別の年度一覧
UNIQUE KEY uq_bfy_scope_name (scope_type, scope_id, name)    -- スコープ内で年度名一意
INDEX idx_bfy_dates (scope_type, scope_id, start_date, end_date)  -- 日付範囲での年度検索
```

**制約・備考**
- `scope_type` + `scope_id` はポリモーフィック参照。FK 制約なし（アプリ層で検証）
- `start_date` < `end_date` を CHECK 制約で保証:
  ```sql
  CONSTRAINT chk_bfy_date_range CHECK (start_date < end_date)
  ```
- 同一スコープ内で年度期間の重複はアプリ層で検証（重複不可）
- `status = 'OPEN'`: トランザクション記録可能。`CLOSED`: 新規トランザクション記録不可（読み取り専用）。再度 `OPEN` に戻すことは ADMIN のみ可能
- 1スコープあたり `OPEN` な年度は同時に最大 **2つ**（年度跨ぎの移行期間を考慮）

#### `budget_categories`

費目（カテゴリ）マスタ。年度に紐づく。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `fiscal_year_id` | BIGINT UNSIGNED | NO | — | FK → budget_fiscal_years。ON DELETE RESTRICT |
| `parent_id` | BIGINT UNSIGNED | YES | NULL | FK → budget_categories（自己参照）。NULL = 大分類。ON DELETE RESTRICT |
| `name` | VARCHAR(100) | NO | — | 費目名（例: 「交通費」「備品費」「会費収入」） |
| `category_type` | VARCHAR(20) | NO | — | 費目種別（`INCOME` / `EXPENSE`） |
| `description` | VARCHAR(500) | YES | NULL | 費目の説明 |
| `sort_order` | INT | NO | 0 | 表示順 |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_bc_fiscal_year (fiscal_year_id, category_type, sort_order)  -- 年度別の費目一覧
INDEX idx_bc_parent (parent_id)                                        -- 子費目取得
UNIQUE KEY uq_bc_name (fiscal_year_id, parent_id, name)                -- 同一親内で費目名一意（parent_id NULL の場合はルートレベルで一意）
```

**制約・備考**
- 費目は 2 階層まで（大分類 → 小分類）。`parent_id` の親がさらに `parent_id` を持つ場合はアプリ層でエラー
- 1年度あたり最大 **50 費目**（大分類+小分類合計）
- 論理削除した費目に紐づくトランザクションは表示時に「（削除済み費目）」として値を表示
- 年度を跨いで費目をコピーする機能を提供（前年度の費目構成を新年度に一括コピー）

#### `budget_allocations`

費目ごとの予算配分額。年度内の各費目にいくらの予算を割り当てるかを定義。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `fiscal_year_id` | BIGINT UNSIGNED | NO | — | FK → budget_fiscal_years。ON DELETE RESTRICT |
| `category_id` | BIGINT UNSIGNED | NO | — | FK → budget_categories。ON DELETE RESTRICT |
| `amount` | DECIMAL(12,0) | NO | — | 予算配分額（日本円。0以上） |
| `note` | VARCHAR(500) | YES | NULL | 備考（予算根拠等） |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_ba_fiscal_category (fiscal_year_id, category_id)  -- 年度×費目で一意
```

**制約・備考**
- 1費目に対し1年度内で1レコード。配分額の変更は UPDATE（楽観的ロックで同時編集防止）
- `amount >= 0` を CHECK 制約で保証:
  ```sql
  CONSTRAINT chk_ba_amount CHECK (amount >= 0)
  ```
- 収入費目にも予算（見込額）を設定可能。収入の予算は「収入見込」として報告書に反映
- 子費目に配分がある場合、親費目の配分額は子費目の合計と一致する必要はない（親=大枠の目安、子=詳細配分として運用可能）

#### `budget_transactions`

収入・支出の仕訳帳。全トランザクションを記録する。物理削除不可（監査証跡保持）。修正は「取消仕訳」パターンで行う。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `fiscal_year_id` | BIGINT UNSIGNED | NO | — | FK → budget_fiscal_years。ON DELETE RESTRICT |
| `category_id` | BIGINT UNSIGNED | NO | — | FK → budget_categories。ON DELETE RESTRICT |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `transaction_type` | VARCHAR(20) | NO | — | トランザクション種別（`INCOME` / `EXPENSE`） |
| `amount` | DECIMAL(12,0) | NO | — | 金額（正数=通常仕訳、負数=取消仕訳） |
| `transaction_date` | DATE | NO | — | 取引日 |
| `title` | VARCHAR(200) | NO | — | 摘要（例: 「4月分 遠征バス代」「2026年度 年会費 田中太郎」） |
| `description` | TEXT | YES | NULL | 詳細説明（最大 2,000 文字） |
| `payment_method` | VARCHAR(30) | YES | NULL | 支払方法（`CASH` / `BANK_TRANSFER` / `CREDIT_CARD` / `STRIPE` / `OTHER`） |
| `reference_number` | VARCHAR(100) | YES | NULL | 参照番号（領収書番号、振込明細番号等） |
| `counterparty` | VARCHAR(200) | YES | NULL | 取引先名（支払先・入金元） |
| `is_auto_recorded` | BOOLEAN | NO | FALSE | 自動計上フラグ（F08.2 連携等で自動 INSERT された場合 TRUE） |
| `source_type` | VARCHAR(50) | YES | NULL | 自動計上元種別（例: `MEMBER_PAYMENT`） |
| `source_id` | BIGINT UNSIGNED | YES | NULL | 自動計上元の PK（例: member_payments.id） |
| `reversal_of_id` | BIGINT UNSIGNED | YES | NULL | FK → budget_transactions（自己参照）。取消対象トランザクション ID。取消仕訳の場合に設定。ON DELETE RESTRICT |
| `workflow_request_id` | BIGINT UNSIGNED | YES | NULL | FK → workflow_requests。承認ワークフローと紐付け。ON DELETE SET NULL |
| `approval_status` | VARCHAR(20) | NO | 'APPROVED' | 承認ステータス（`PENDING_APPROVAL` / `APPROVED` / `REJECTED`） |
| `recorded_by` | BIGINT UNSIGNED | NO | — | FK → users。記録者。ON DELETE RESTRICT |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時。物理削除は不可 |

**インデックス**
```sql
INDEX idx_bt_fiscal_year (fiscal_year_id, transaction_type, transaction_date DESC)  -- 年度別の仕訳一覧
INDEX idx_bt_scope (scope_type, scope_id, transaction_date DESC)                     -- スコープ別の仕訳一覧
INDEX idx_bt_category (category_id, transaction_date DESC)                           -- 費目別の仕訳一覧
INDEX idx_bt_source (source_type, source_id)                                         -- 自動計上元からの逆引き
INDEX idx_bt_workflow (workflow_request_id)                                           -- 承認ワークフローからの逆引き
INDEX idx_bt_reversal (reversal_of_id)                                               -- 取消対象からの逆引き
INDEX idx_bt_approval (approval_status, created_at DESC)                             -- 承認待ち一覧
```

**制約・備考**
- **物理削除不可**: `deleted_at` による論理削除のみ。物理削除バッチの対象外とする。税法に基づき **7 年間** 保持（.claudecode.md §8 準拠）
- **取消仕訳パターン**: トランザクションの修正は既存レコードを編集せず、以下の手順で行う:
  1. 元トランザクションと同額のマイナス金額で新規レコード INSERT（`reversal_of_id` に元 ID を設定）
  2. 正しい金額で新規レコード INSERT
  3. 元トランザクションの `deleted_at` は NULL のまま維持（帳簿上の証跡を保持）
- `transaction_type` と `category_type` の整合性: `INCOME` トランザクションは `INCOME` 費目にのみ記録可能。`EXPENSE` も同様。アプリ層でバリデーション
- `approval_status = 'PENDING_APPROVAL'`: 承認閾値以上の支出で承認フロー起動中。予算消化率の計算には含めない。`APPROVED` に遷移後に予算消化率に反映。`REJECTED`: 却下された支出。予算消化率に含めない
- `is_auto_recorded = TRUE`: F08.2 会費入金の自動計上や、将来の外部連携で自動 INSERT されたトランザクション。手動編集不可（取消仕訳のみ可能）
- `amount` の符号規則: 通常仕訳は正数。取消仕訳はマイナス。収入・支出の区別は `transaction_type` で行う

#### `budget_reports`

月次・年次報告書のメタデータ。PDF ファイルは S3 に保存。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `fiscal_year_id` | BIGINT UNSIGNED | NO | — | FK → budget_fiscal_years。ON DELETE RESTRICT |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `report_type` | VARCHAR(20) | NO | — | 報告書種別（`MONTHLY` / `ANNUAL`） |
| `period_start` | DATE | NO | — | 対象期間開始日 |
| `period_end` | DATE | NO | — | 対象期間終了日 |
| `file_key` | VARCHAR(500) | YES | NULL | S3 オブジェクトキー（PDF）。生成完了前は NULL |
| `file_size` | INT UNSIGNED | YES | NULL | ファイルサイズ（バイト） |
| `status` | VARCHAR(20) | NO | 'GENERATING' | 生成ステータス（`GENERATING` / `COMPLETED` / `FAILED`） |
| `generated_by` | BIGINT UNSIGNED | NO | — | FK → users。生成を要求した ADMIN。ON DELETE RESTRICT |
| `generated_at` | DATETIME | YES | NULL | 生成完了日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_br_fiscal_year (fiscal_year_id, report_type, period_start DESC)  -- 年度別の報告書一覧
INDEX idx_br_scope (scope_type, scope_id, report_type, period_start DESC)  -- スコープ別の報告書一覧
```

**制約・備考**
- PDF 生成は非同期（`@Async("job-pool")`）。生成完了後に `file_key` / `file_size` / `status = 'COMPLETED'` を更新
- PDF 生成ライブラリ: OpenPDF（Java）でサーバーサイド生成
- 生成済み PDF は S3 に保存し、Pre-signed URL（有効期限 5 分）でダウンロード提供
- 同一期間の報告書を再生成した場合、既存レコードを UPDATE（旧 PDF は S3 から削除）
- S3 キー形式: `reports/budget/{scopeType}/{scopeId}/{fiscalYearId}/{reportType}_{periodStart}_{periodEnd}.pdf`
- 報告書の内容: 収支サマリー、費目別収支一覧、予算消化率、月次推移グラフデータ、トランザクション明細

#### `budget_configs`

会計設定。スコープごとに1レコード。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|

| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `approval_threshold` | DECIMAL(12,0) | YES | NULL | 承認閾値（円）。この金額以上の支出は F05.6 承認ワークフローに自動連携。NULL = 承認不要 |
| `workflow_template_id` | BIGINT UNSIGNED | YES | NULL | FK → workflow_templates。支出承認に使用するテンプレート。ON DELETE SET NULL |
| `auto_record_payments` | BOOLEAN | NO | TRUE | F08.2 会費入金を収入として自動計上するか |
| `default_income_category_id` | BIGINT UNSIGNED | YES | NULL | FK → budget_categories。自動計上時のデフォルト収入費目。ON DELETE SET NULL |
| `budget_warning_threshold` | SMALLINT | NO | 80 | 予算消化率警告閾値（%）。この値を超えた費目に警告表示 |
| `budget_critical_threshold` | SMALLINT | NO | 95 | 予算消化率危険閾値（%）。この値を超えた費目に危険アラート |
| `version` | BIGINT | NO | 0 | 楽観的ロック用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_bconf_scope (scope_type, scope_id)  -- スコープごとに1レコード
```

**制約・備考**
- `approval_threshold` が設定されている場合、`workflow_template_id` も必須（アプリ層でバリデーション）
- `budget_warning_threshold` < `budget_critical_threshold` をアプリ層で検証
- `default_income_category_id`: F08.2 自動計上時に使用する費目。現在 `OPEN` の年度の `INCOME` 費目であること。NULL の場合は自動計上時にエラーログを出力し、手動記録を促す通知を ADMIN に送信
- モジュール有効化時に初期レコードを自動 INSERT（デフォルト値で作成）

### ER図（テキスト形式）
```
budget_fiscal_years (1) ──── (N) budget_categories
budget_fiscal_years (1) ──── (N) budget_allocations
budget_fiscal_years (1) ──── (N) budget_transactions
budget_fiscal_years (1) ──── (N) budget_reports

budget_categories (1) ──── (N) budget_allocations
budget_categories (1) ──── (N) budget_transactions
budget_categories (N) ──── (1) budget_categories [parent_id, 自己参照]

budget_transactions (1) ──── (N) budget_transaction_attachments
budget_transactions (N) ──── (1) budget_transactions [reversal_of_id, 取消仕訳参照]

budget_configs (1) ──── (1) workflow_templates [workflow_template_id]
budget_configs (1) ──── (1) budget_categories [default_income_category_id]

users (1) ──── (N) budget_fiscal_years [created_by]
users (1) ──── (N) budget_transactions [recorded_by]
users (1) ──── (N) budget_reports [generated_by]

budget_transactions.workflow_request_id ──→ workflow_requests (FK)
budget_transactions.source_type + source_id ──→ member_payments 等 (ポリモーフィック参照)
```

## 5. ビジネスロジック

### トランザクション記録フロー（支出の場合）

```
1. ユーザーが支出トランザクション記録 API を呼び出し
2. Service 層で以下を実行:
   a. 年度の status = 'OPEN' を検証
   b. transaction_date が年度期間内であることを検証
   c. category_id が fiscal_year_id に属し、category_type = 'EXPENSE' であることを検証
   d. budget_configs.approval_threshold をチェック:
      ■ approval_threshold IS NULL、または amount < approval_threshold:
        → approval_status = 'APPROVED' で INSERT
      ■ amount >= approval_threshold:
        → approval_status = 'PENDING_APPROVAL' で INSERT
        → F05.6 承認エンジンの external API を呼び出し:
          POST /workflows/templates/{workflow_template_id}/requests/external
          source_type = 'BUDGET_EXPENSE'
          source_id = budget_transactions.id
          auto_submit = true
          field_values = { "amount": 15000, "category": "交通費", "title": "4月 遠征バス代" }
        → workflow_request_id を budget_transactions に UPDATE
   e. ApplicationEvent (BudgetTransactionCreatedEvent) を発行
3. 通知サービスがイベントを受信:
   a. 承認待ちの場合: ADMIN に通知（「新しい支出承認依頼があります」）
   b. 予算消化率が warning_threshold を超えた場合: ADMIN に警告通知
   c. 予算消化率が critical_threshold を超えた場合: ADMIN に緊急アラート
```

### 承認ワークフロー完了時のフロー

```
1. F05.6 ワークフローエンジンが WorkflowApprovedEvent / WorkflowRejectedEvent を発行
2. BudgetWorkflowListener が購読:
   @TransactionalEventListener(phase = AFTER_COMMIT)
   @Async("event-pool")
   public void onWorkflowApproved(WorkflowApprovedEvent event) {
       if ("BUDGET_EXPENSE".equals(event.getSourceType())) {
           budgetTransactionService.approveTransaction(event.getSourceId());
       }
   }
3. 承認の場合:
   a. budget_transactions.approval_status = 'APPROVED' に更新
   b. 予算消化率を再計算し、閾値チェック
   c. 記録者に承認完了通知を送信
4. 却下の場合:
   a. budget_transactions.approval_status = 'REJECTED' に更新
   b. 記録者に却下通知を送信（却下理由を含む）
```

### F08.2 会費入金の自動計上フロー

```
1. F08.2 で会費の入金が完了（member_payments.status = 'PAID' に遷移）
2. F08.2 が PaymentCompletedEvent を発行:
   PaymentCompletedEvent {
       memberPaymentId: 123,
       paymentItemId: 5,
       userId: 42,
       amount: 20000,
       scopeType: "TEAM",
       scopeId: 7
   }
3. BudgetPaymentListener が購読:
   @TransactionalEventListener(phase = AFTER_COMMIT)
   @Async("event-pool")
   public void onPaymentCompleted(PaymentCompletedEvent event) {
       budgetTransactionService.autoRecordPaymentIncome(event);
   }
4. autoRecordPaymentIncome の処理:
   a. budget_configs を取得し、auto_record_payments = TRUE を確認
   b. default_income_category_id を取得
   c. 現在 OPEN の年度を特定（transaction_date = 入金日が含まれる年度）
   d. budget_transactions に INSERT:
      - transaction_type = 'INCOME'
      - category_id = default_income_category_id
      - amount = event.amount
      - title = "会費入金: {ユーザー名} {支払い項目名}"
      - is_auto_recorded = TRUE
      - source_type = 'MEMBER_PAYMENT'
      - source_id = event.memberPaymentId
      - approval_status = 'APPROVED'（自動計上は承認不要）
   e. 重複チェック: 同一 source_type + source_id のトランザクションが既に存在する場合はスキップ（冪等性保証）
5. 設定エラー時の処理:
   a. auto_record_payments = FALSE → スキップ
   b. default_income_category_id = NULL → エラーログ出力 + ADMIN に通知
   c. OPEN な年度が見つからない → エラーログ出力 + ADMIN に通知
```

### 取消仕訳フロー

```
1. ADMIN が取消仕訳 API を呼び出し
2. Service 層で以下を実行:
   a. 元トランザクションが存在し、既に取消されていないことを確認
   b. 年度の status = 'OPEN' を検証
   c. 新規トランザクションを INSERT:
      - amount = -(元トランザクションの amount)
      - transaction_date = 本日
      - title = "【取消】" + 元トランザクションの title
      - description = リクエストの reason
      - reversal_of_id = 元トランザクション ID
      - approval_status = 'APPROVED'（取消仕訳は承認不要）
   d. ApplicationEvent (BudgetTransactionReversedEvent) を発行
3. 元トランザクションは変更しない（証跡保持）
```

### 報告書生成フロー

```
1. ADMIN が報告書生成 API を呼び出し
2. Controller が budget_reports レコードを status = 'GENERATING' で INSERT
3. 非同期ジョブ（@Async("job-pool")）を起動:
   a. 対象期間のトランザクションを集計:
      - 費目別の収入合計・支出合計
      - 予算配分額と消化率
      - 月次推移データ
   b. OpenPDF で PDF を生成:
      - 表紙（組織/チーム名、対象期間、生成日）
      - 収支サマリーテーブル
      - 費目別収支一覧
      - 予算消化率グラフ（棒グラフ）
      - トランザクション明細一覧
   c. S3 にアップロード
   d. budget_reports を UPDATE:
      - file_key = S3 キー
      - file_size = ファイルサイズ
      - status = 'COMPLETED'
      - generated_at = 現在日時
   e. 生成失敗時: status = 'FAILED'。エラーログ出力
4. 生成完了後、要求者に通知（「報告書の生成が完了しました」）
```

### CSV インポートフロー

```
1. ADMIN が CSV インポート（プレビュー）API を呼び出し
2. Service 層で以下を実行:
   a. CSV ファイルをパース（UTF-8 + BOM 対応）
   b. 各行を検証:
      - 取引日: DATE 形式、年度期間内
      - 種別: INCOME / EXPENSE
      - 費目: 名前で budget_categories を検索（完全一致）
      - 金額: 正の整数
      - 摘要: 必須、200文字以内
   c. 検証結果をレスポンスに含める（プレビュー）
   d. import_token を生成し、Valkey に一時保存（TTL: 15分）:
      キー: mannschaft:budget:import:{import_token}
      値: パース済みのトランザクションデータ（JSON）
3. ユーザーがプレビューを確認し、インポート確定 API を呼び出し
4. Service 層で以下を実行:
   a. Valkey から import_token でデータを取得
   b. 有効な行のみ budget_transactions に一括 INSERT
   c. Valkey キーを削除
   d. インポート結果をレスポンスに返す
```

### 予算消化率の計算ロジック

```
■ 費目別消化率:
  consumed = SUM(budget_transactions.amount)
             WHERE category_id = ? AND transaction_type = 'EXPENSE'
             AND approval_status = 'APPROVED' AND deleted_at IS NULL
  budget = budget_allocations.amount WHERE category_id = ?
  consumption_rate = (consumed / budget) * 100

■ 警告レベル判定:
  - consumption_rate < budget_warning_threshold → NORMAL
  - budget_warning_threshold <= consumption_rate < budget_critical_threshold → WARNING
  - consumption_rate >= budget_critical_threshold → CRITICAL
  - budget = 0 の場合: consumed > 0 なら CRITICAL、consumed = 0 なら NORMAL

■ 予算配分が未設定の費目: 消化率は計算せず「予算未設定」と表示
```

### 重要な判定ロジック

- **年度の CLOSED 判定**: CLOSED 年度への新規トランザクション記録・取消仕訳は不可。ADMIN のみ再開可能
- **費目のカテゴリ種別整合性**: `INCOME` トランザクションは `INCOME` 費目にのみ、`EXPENSE` は `EXPENSE` 費目にのみ記録可能
- **自動計上トランザクションの編集制限**: `is_auto_recorded = TRUE` のトランザクションは直接編集不可。修正が必要な場合は取消仕訳を作成後、正しい金額で新規記録
- **承認閾値の適用**: `INCOME` トランザクションには承認フローを適用しない（`EXPENSE` のみ）
- **取消仕訳の連鎖禁止**: 取消仕訳（`reversal_of_id IS NOT NULL`）に対する取消仕訳は作成不可
- **年度跨ぎ**: `transaction_date` は `budget_fiscal_years.start_date` 〜 `end_date` の範囲内であること。範囲外の場合は 400 エラー
- **MEMBER の操作範囲**: MEMBER は自分が記録したトランザクションの閲覧のみ。全トランザクションの閲覧は ADMIN / DEPUTY_ADMIN（`BUDGET_VIEW` 以上）

### 通知トリガー一覧

| トリガー | 通知先 | 条件 |
|---------|-------|------|
| 予算消化率 WARNING 到達 | スコープの ADMIN 全員 | 費目の消化率が budget_warning_threshold を超過 |
| 予算消化率 CRITICAL 到達 | スコープの ADMIN 全員 | 費目の消化率が budget_critical_threshold を超過 |
| 支出承認待ち | 承認者（F05.6 経由） | 承認閾値以上の支出記録 |
| 支出承認完了 | トランザクション記録者 | 承認ワークフロー完了 |
| 支出却下 | トランザクション記録者 | 承認ワークフローで却下 |
| 報告書生成完了 | 生成要求者 | 非同期の報告書生成が完了 |
| F08.2 自動計上エラー | スコープの ADMIN 全員 | 設定不備で自動計上に失敗 |

## 6. セキュリティ考慮事項

- **認可チェック**: BudgetService の入り口で `scope_type` / `scope_id` と `currentUser` の所属・ロールを必ず検証する。MEMBER はスコープ所属かつ自分が記録したトランザクションのみ閲覧可能
- **レートリミット**: トランザクション記録 API に `Bucket4j` で 1分間に30回の制限を適用（大量の不正記録防止）
- **楽観的ロック**: `budget_fiscal_years`, `budget_categories`, `budget_allocations`, `budget_transactions`, `budget_configs` に `version` カラムを適用。同時編集時の不整合を防止
- **監査証跡**: `budget_transactions` は物理削除不可。全ての修正は取消仕訳パターンで記録し、操作履歴を完全に保持。`audit_logs` にも操作を記録（`BUDGET_TRANSACTION_CREATED`, `BUDGET_TRANSACTION_REVERSED`, `BUDGET_FISCAL_YEAR_CLOSED` 等）
- **CSV インポートの検証**: アップロードされた CSV のサイズ上限（5MB）、行数上限（1,000行）、文字エンコーディング検証（UTF-8 のみ受付）を実施。パース結果は Valkey に一時保存し、15分で失効
- **添付ファイル**: Pre-signed URL 方式でアップロード。`contentType` ホワイトリスト検証（JPEG, PNG, WebP, PDF のみ）+ `fileSize` 上限チェック（10MB/ファイル）
- **金額の整合性**: `DECIMAL(12,0)` で日本円の整数値のみを扱い、浮動小数点の丸め誤差を排除
- **年度の再開制限**: CLOSED → OPEN の再開は `audit_logs` に記録し、不正な年度操作を追跡可能にする
