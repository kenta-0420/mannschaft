# F09.8.1: コルクボード・ピン止め＋ダッシュボード統合

> **ステータス**: 🟢 設計完了
> **バージョン**: v1.0
> **最終更新**: 2026-05-03
> **モジュール種別**: 既存機能の拡張（F09.8 コルクボード）
> **関連機能**: F09.8（コルクボード本体）、F02.2（マイダッシュボード）、F02.2.1（ウィジェットのロール別可視性）、F01.3（モジュール）
> **対象スコープ**: 個人（PERSONAL）コルクボードのみ

---

## 1. 概要

### 目的

コルクボードに「ピン止め」フラグを導入し、**個人ダッシュボードのウィジェットとして「マイコルクボード」を新設**することで、

- ユーザーが日常的に頻繁に参照する重要カード（チームの招待状、保留中の URL、メモ等）をダッシュボードから一目で見つけられる
- カードのクリックひとつで参照先（チーム・組織・投稿・イベント・ドキュメント・URL）へジャンプできる
- ボード深層に埋もれて見失う事態を防ぐ

ことを目的とする。

### ユーザーストーリー

| # | ストーリー |
|---|---|
| US-1 | 個人として、お気に入りのカードに「ピン止め」マークを付けて、後で見つけやすくしたい |
| US-2 | 個人として、ダッシュボードを開いたときに、ピン止めしたカードがウィジェットに並んでいて欲しい |
| US-3 | 個人として、ウィジェット内のカードをクリックしたら、参照先（チーム/組織/投稿/イベント/ドキュメント等）へすぐジャンプしたい |
| US-4 | 個人として、ウィジェット見出しから専用ページ `/my/corkboard` へ遷移し、全ピン止めカードのメモを編集・並べ替え・追加削除したい |
| US-5 | 個人として、参照先の閲覧権限を失った場合（チームから外された等）でも、ダッシュボードが壊れず「閲覧権限なし」として安全に表示されて欲しい |

### 成功指標

- ピン止めカードを持つアクティブユーザーの割合 ≧ 30%（リリース後 1 ヶ月）
- 「マイコルクボード」ウィジェット経由での参照先ジャンプ数 / 個人ダッシュボード全 PV ≧ 5%
- ピン止め操作のエラー率 ≦ 0.5%

## 3. DB 変更

### 3.1 マイグレーションファイル

- **ファイル名**: `backend/src/main/resources/db/migration/V9.095__add_is_pinned_to_corkboard_cards.sql`
- **採番根拠**: 既存最新は `V9.094__create_blog_post_reactions.sql`。F09.8.1 は次番号 `V9.095` を使用。

### 3.2 DDL（最終版）

```sql
-- F09.8.1: コルクボード・ピン止めカラム追加
ALTER TABLE corkboard_cards
    ADD COLUMN is_pinned BOOLEAN NOT NULL DEFAULT FALSE AFTER is_archived,
    ADD COLUMN pinned_at DATETIME NULL AFTER is_pinned;

-- ピン止めカード絞り込み専用インデックス
-- ダッシュボードウィジェットの GET /users/me/corkboards/pinned-cards で使用。
-- 「個人ボードに属するカードのうち is_pinned = true かつ未削除・非アーカイブ」を絞る複合キー。
-- 実 SQL では corkboard_cards JOIN corkboards ON corkboards.scope_type = 'PERSONAL' AND owner_id = ?
-- で個人ボードに限定するため、corkboard_id を先頭に置く。
CREATE INDEX idx_cc_pinned ON corkboard_cards (corkboard_id, is_pinned, is_archived, deleted_at);

-- ピン止め日時降順ソート専用（ダッシュボード用 ORDER BY pinned_at DESC）
CREATE INDEX idx_cc_pinned_at ON corkboard_cards (is_pinned, pinned_at DESC);
```

### 3.3 カラム仕様

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `is_pinned` | BOOLEAN | NO | FALSE | ピン止めフラグ。`true` のカードがダッシュボードウィジェットに表示される |
| `pinned_at` | DATETIME | YES | NULL | ピン止め日時。`is_pinned = true` の場合は必須、`false` に戻されたら NULL に戻す。ウィジェット内ソートに使用 |

### 3.4 既存データへの影響

- 既存カード（V9.094 までで作成済み）はすべて `is_pinned = FALSE`、`pinned_at = NULL` で開始
- ALTER TABLE 中に長期ロックを避けるため、`ALGORITHM=INSTANT`（MySQL 8.0.29+）が自動採用される（カラム追加 + インデックス追加は ONLINE 可能）
- 既存の F09.8 機能には一切破壊的影響なし（新カラム追加のみ・既存 SELECT は影響を受けない）

### 3.5 互換性検証

- `corkboard_cards` テーブルを参照する既存 Repository（`CorkboardCardRepository`）の `findAll`・`findByCorkboardId` 等は変更不要。Hibernate が自動的に新カラムを認識する
- フロントエンド既存画面（`/corkboard` index）は `is_pinned` を意識せずに動作可能

## 4. API 設計

### 4.1 エンドポイント一覧

| メソッド | パス | 認証 | 説明 |
|---------|-----|------|------|
| PATCH | `/api/v1/corkboards/{boardId}/cards/{cardId}/pin` | 必要 | カードのピン止め状態を更新（個人ボード所有者のみ） |
| GET | `/api/v1/users/me/corkboards/pinned-cards` | 必要 | 自分の個人ボード横断のピン止めカード一覧（参照先解決込み） |

> 個人ボード一覧 `GET /api/v1/users/me/corkboards`、個人ボード詳細 `GET /api/v1/users/me/corkboards/{id}` は既存（`MyCorkboardController`）。F09.8.1 では新規追加せず、既存 API のレスポンスにのみ `is_pinned` / `pinned_at` フィールドを追加する。

### 4.2 `PATCH /api/v1/corkboards/{boardId}/cards/{cardId}/pin`

**用途**: カードのピン止め状態をトグル（true ↔ false）。

**認可**:
- 対象カードが属する `corkboards` の `scope_type = PERSONAL` かつ `owner_id = 認証ユーザー` であることを必須検証
- `scope_type = TEAM` / `ORGANIZATION` のボード上のカードに対するピン止めは v1.0 では受け付けない（403 `CORKBOARD_011`）

**リクエスト**

```http
PATCH /api/v1/corkboards/12/cards/345/pin
Content-Type: application/json

{
  "is_pinned": true
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `is_pinned` | Boolean | Yes | `true`: ピン止めする / `false`: ピン止め解除 |

**レスポンス（200 OK）**

```json
{
  "data": {
    "id": 345,
    "is_pinned": true,
    "pinned_at": "2026-05-03T14:23:00"
  }
}
```

`is_pinned = false` 時は `pinned_at: null` を返却。

**エラー**

| ステータス | コード | 条件 |
|-----------|--------|------|
| 400 | `CORKBOARD_002` | カードが見つからない（論理削除済み含む） |
| 400 | `CORKBOARD_012` | アーカイブ済みカードはピン止めできない |
| 403 | `CORKBOARD_011` | 個人ボード以外のカード、または所有者ではない |
| 404 | `CORKBOARD_001` | ボードが見つからない |
| 409 | `CORKBOARD_013` | ピン止めカード上限超過（後述） |

**ピン止め上限**:
- **1 ユーザーあたり 50 枚**（個人ダッシュボードの可読性確保のため）
- 上限到達時に 51 枚目を `is_pinned = true` にしようとすると 409 `CORKBOARD_013` を返却
- フロントエンドは「ピン止め枠が満杯です。先に他のカードのピン止めを解除してください」と i18n メッセージで案内

- `items[].reference.is_accessible`: 参照先に対する**閲覧権限の二重チェック結果**。false の場合フロントは「閲覧権限なし」表示にし、`navigate_to` を `null` にする
- `items[].reference.is_deleted`: 参照先が論理削除済みなら true。`content_snapshot` の値で表示
- `items[].reference.navigate_to`: フロントが `<NuxtLink :to="navigate_to">` で使う相対パス（URL カードのみ絶対 URL）
- `next_cursor`: `null` ならこれ以上のページなし

**N+1 防止戦略**:
- バックエンドは 1 クエリで全ピン止めカードを取得 → `referenceType` でグループ化 → 各 type の Repository に `findAllByIdIn(ids)` で一括取得 → 各参照先の閲覧権限を `AccessControlService` で**バッチ判定**（後述 §5.2）
- 1 リクエストあたり SQL 数は最大: 1（カード一覧）+ N（参照タイプ数、最大 5）+ 1（ボード名取得）+ 1（権限チェック用 JOIN）= **8 クエリ以内**

**参照先解決対応 type**

| `reference.type` | バックエンド参照先 | `navigate_to` の生成例 |
|-------------|-------------|----------------------|
| `TIMELINE_POST` | `timeline_posts` | `/timeline/posts/{id}` |
| `BULLETIN_THREAD` | `bulletin_threads` | `/bulletin/threads/{id}` |
| `BLOG_POST` | `blog_posts` | `/blog/{id}` |
| `CHAT_MESSAGE` | `chat_messages` | `/chat/channels/{channelId}?messageId={id}` |
| `FILE` | `shared_files`（または相当） | `/files/{id}` |
| `TEAM` | `teams` | `/teams/{id}` |
| `ORGANIZATION` | `organizations` | `/organizations/{id}` |
| `EVENT` | `schedules` | `/schedules/{id}` |
| `DOCUMENT` | `circulation_documents` | `/documents/{id}` |
| `URL` | （外部）`url` カラム | `url` の値そのもの |
| （null / MEMO / SECTION_HEADER） | — | null（ナビゲーション無効。専用ページでメモ閲覧のみ） |

> 既存 `corkboard_cards.reference_type` には `TEAM` / `ORGANIZATION` / `EVENT` / `DOCUMENT` がまだ採用されていない可能性がある。F09.8.1 では「対応する参照タイプはバックエンドの `ReferenceTypeResolver`（仮）で抽象化し、未対応 type は `is_accessible = false` + `navigate_to = null` でフォールバック」とする。新タイプ追加時は本表に追記。

## 5. 権限・セキュリティ

### 5.1 ピン止め操作の認可

| 操作 | 認可ルール |
|------|-----------|
| `PATCH .../pin` | カードが属するボードの `scope_type = PERSONAL` AND `owner_id = currentUserId` |
| `GET .../pinned-cards` | 認証済みユーザー本人。`owner_id = currentUserId` のボード配下のみ抽出 |

実装は `MyCorkboardController` / 新設 `MyPinnedCardsController` 内で `SecurityUtils.getCurrentUserId()` と JOIN で強制。Service 層で「ボード ID を入力にせず常に `owner_id` 条件付きクエリで取得する」ことで IDOR を構造的に排除。

### 5.2 参照先の閲覧権限二重チェック

ピン止めしたあとに参照先の権限を失う事態（チームから外された / 投稿が visibility 変更された等）を考慮し、**ウィジェット表示時点で都度参照先の閲覧権限を再判定**する。

実装パターン:

```java
// 擬似コード
List<PinnedCardRow> rows = pinnedCardRepository.findByOwner(userId, limit, cursor);

// type ごとに ID をグルーピング
Map<String, List<Long>> idsByType = rows.stream()
    .filter(r -> r.referenceType() != null)
    .collect(Collectors.groupingBy(
        PinnedCardRow::referenceType,
        Collectors.mapping(PinnedCardRow::referenceId, Collectors.toList())));

// 各 type ごとに「閲覧可能な ID 集合」を一括取得
Map<String, Set<Long>> accessibleIdsByType = new HashMap<>();
for (var e : idsByType.entrySet()) {
    Set<Long> accessible = accessControlService.filterAccessible(
        userId, e.getKey(), e.getValue());
    accessibleIdsByType.put(e.getKey(), accessible);
}

// 各カードの reference.is_accessible を判定
return rows.stream()
    .map(r -> toResponse(r, accessibleIdsByType, ...))
    .toList();
```

`AccessControlService.filterAccessible(userId, refType, ids)` は `refType` ごとに既存のアクセス制御ロジックへディスパッチする（`TIMELINE_POST` → `TimelinePostAccessChecker`、`TEAM` → `TeamMembershipChecker` など）。各 Checker は既存実装を再利用し、F09.8.1 では新設しない。

### 5.3 IDOR 防止策

- ピン止め API のパス変数 `boardId` / `cardId` は受け取るが、Service では **必ず `WHERE corkboards.owner_id = :userId AND cards.id = :cardId AND cards.corkboard_id = :boardId AND cards.deleted_at IS NULL`** の複合条件でロード。所有者でない場合は `findOptional` が空になり 403 を投げる
- `pinned-cards` API は `boardId` を受け取らない。所有者のボード横断で固定取得する
- E2E テストで「他人の cardId を URL に直接入力 → 403」「他人の boardId 配下のカード ID で 403」のケースを必須検証（§10.3）

### 5.4 権限喪失時のフォールバック

- ウィジェット側: `reference.is_accessible = false` のカードは「（閲覧権限なし）」グレーアウト表示。クリック時はトーストで「このカードの参照先にアクセスできません」と案内し、ナビゲーションは行わない
- 専用ページ側: 同上の表示 + 「このカードのピン止めを外す」ボタンを併記（ユーザーが自発的にクリーンアップ可能）
- DB 側: `is_pinned` フラグは自動的にはオフにしない（ユーザーが権限を取り戻せば復活するため）

### 5.5 監査ログ

ピン止めの ON/OFF は **audit_logs に記録しない**。理由:
- 個人ボード内の単独操作であり、他者影響なし
- 操作頻度が高いため監査ログのノイズ源になる
- 既存 F09.8 の他のカード操作（位置更新等）も audit_logs 対象外

## 8. イベント・通知

### 8.1 WebSocket 同期方針

- 個人コルクボードは**単独ユーザー**しか操作しないため、`PATCH .../pin` の WebSocket ブロードキャストは **不要**
- ただし「同一ユーザーが複数タブを開いている」ケースを考慮し、フロントエンドはダッシュボードを開く度に都度 `GET /pinned-cards` を実行する（F02.2 既存パターン）。リアルタイム同期は v1.0 では諦める（操作頻度低・他人影響なしのため許容）
- v2.0 でユーザーごとの「自分宛 WS チャネル」を持つ場合は、その経路で `PINNED_CARDS_UPDATED` イベントを配信可能（**§12 見送り事項**）

### 8.2 通知

- ピン止めはユーザーの個人操作。誰にも通知しない
- 「参照先が削除された場合」もダッシュボード上で「（参照元は削除されました）」表示でユーザーに気付かせるのみで、Push 通知 / メール通知は送らない（既存 F09.8 のデッドリファレンス検知バッチに準じる）

## 9. マイグレーション・後方互換性

### 9.1 既存個人コルクボードへの影響

- 既存の個人ボード・カードは V9.095 適用後すべて `is_pinned = FALSE` で開始
- 既存のカード CRUD API（`POST /cards` 等）は `is_pinned` を**受け付けない**（カード作成時にピン止め指定不可。作成後に `PATCH .../pin` で別途設定）
- `GET /api/v1/users/me/corkboards/{id}` のレスポンスに `is_pinned`、`pinned_at` フィールドが新規追加。既存フロントエンドはこれを無視するだけで動作継続可能（破壊的変更なし）

### 9.2 ロールバック手順

万一 V9.095 をロールバックする場合:

```sql
ALTER TABLE corkboard_cards
    DROP INDEX idx_cc_pinned,
    DROP INDEX idx_cc_pinned_at,
    DROP COLUMN pinned_at,
    DROP COLUMN is_pinned;
```

を `V9.095_R__rollback.sql` として用意（手動実行用、Flyway では自動実行されない）。

### 9.3 データ移行

既存データの一括変換は不要（全レコード `FALSE` で良い）。

### 9.4 個人スコープ以外の既存 `corkboard_cards` レコードへの影響

V9.095 は `corkboard_cards` テーブル全体にカラムを追加するため、TEAM / ORGANIZATION スコープのカードにも `is_pinned` カラムが付与される。**ただし v1.0 では API がそれらに対する pin 操作を 403 で拒否**するため、実体としては常に FALSE のまま。

## 11. 設計判断ログ

| # | 判断 | 理由 |
|---|------|------|
| D-1 | 個人スコープのみ対応（チーム/組織は v2.0 以降） | 共有ボードでのピン止めは「誰のピン止めか」のスコーピングが複雑（ユーザー個別 vs ADMIN 設定）。v1.0 は最小スコープで価値検証 |
| D-2 | `is_pinned` + `pinned_at` を分離 | ソート順を `pinned_at DESC` に固定するため。`is_pinned = false` で `pinned_at = NULL` に戻すことで再ピン止め時に最新化 |
| D-3 | ピン止め上限 50 枚 | ダッシュボードウィジェットの可読性確保 + DB スキャン量制限。Twitter のブックマーク無制限と異なり、ダッシュボード起点であることを重視 |
| D-4 | 参照先解決をバックエンドで一括 | フロント側で type ごとに API を叩くと N+1 になり、参照タイプ追加時に毎回フロント改修が必要。バックエンドの `ReferenceAccessResolver` で抽象化 |
| D-5 | 権限喪失カードを自動で unpin しない | ユーザー意図的なクリーンアップを尊重。権限を取り戻したら復活する余地を残す |
| D-6 | WebSocket 同期不要 | 単独ユーザー操作・他人影響なし・操作頻度高くない（マイクロ秒単位の同期ニーズなし）。サーバー資源節約 |
| D-7 | カード作成時に `is_pinned` を受け付けない | 「作成 → ピン止め」を 2 操作に分離することで、誤ピンを防ぎ・既存 `POST /cards` の DTO を変更しない（後方互換維持） |
| D-8 | 参照タイプの拡張は `ReferenceTypeResolver` で吸収 | 新規追加 type（v1.0 で `TEAM` / `ORGANIZATION` 等）は Resolver にケースを追加するだけで、API スキーマは固定 |
| D-9 | `idx_cc_pinned` を `(corkboard_id, is_pinned, is_archived, deleted_at)` 順で設計 | 個人ボード横断クエリは `WHERE corkboard_id IN (...) AND is_pinned = TRUE AND is_archived = FALSE AND deleted_at IS NULL` が支配的。先頭に `corkboard_id` を置きセレクティビティを高める |
| D-10 | パス設計を `/api/v1/users/me/corkboards/pinned-cards` に統一 | 既存 `MyCorkboardController` のベース URL `/api/v1/users/me/corkboards` に揃えることで API 探索性を上げる。指示文書の `/api/v1/corkboards/personal/pinned-cards` 案より既存規約に整合 |
| D-11 | エラーコード prefix は `CORKBOARD_` | 既存 `CorkboardErrorCode` enum の規約（`CORKBOARD_001`〜`CORKBOARD_009`）に従う。指示文書の `CORK_xxx` 案より既存に整合 |

## 12. 見送り事項（Out of Scope / Future Work）

| 項目 | 見送り理由 / 検討タイミング |
|------|---------------------------|
| 共有ボード（TEAM / ORGANIZATION）でのピン止め | スコーピング設計が複雑。v2.0 で「ユーザー個別ピン止め」テーブルを別途追加して実現候補 |
| ピン止め並び順のドラッグ＆ドロップ | v1.0 はピン止め日時降順固定。需要が出れば `pinned_sort_order` カラム追加で対応 |
| ピン止めへの通知（誰かにピン止めされたよ） | 共有ボード対応とセットで検討 |
| マイコルクボード専用ページからの「新規ピン」直接作成 UI | 既存の「カード作成 → ピン止め」フローで代用可。UI 重複を避けるため見送り |
| WebSocket での `PINNED_CARDS_UPDATED` 配信 | 単独ユーザー単複数タブ同期のため必要だが、優先度低。v2.0 で「自分宛 WS チャネル」基盤と合わせて実装 |
| カードにタグ付け / カテゴリ分類してピン止めをグルーピング | スコープ膨張のため見送り。現状 `corkboard_id` での絞り込みで十分 |
| ダッシュボード以外（ヘッダー / 検索）からのピン止め一覧 | 必要性が出てから検討 |
| ピン止めカードの自動アーカイブ除外 | 既存 `auto_archive_at` バッチが pinned カードもアーカイブする設計で問題ないか検討要。v1.0 は既存挙動を維持し、要望が出てから「pin されたカードは auto-archive スキップ」を実装 |

### 13.1 実装済 MVP の補足（Phase 3 完了時点）

- §5.2 の `AccessControlService.filterAccessible(userId, type, ids)` 相当は、`AccessControlDispatcher` クラスに集約して新設した。type 別 Checker（`TimelinePostAccessChecker` 等）が現行のコードベースに統一実装として存在しないため、**Phase 3 では「対応 type は閲覧可、未対応 type は閲覧不可」フォールバックを採用**（参照先の論理削除判定も同様にフォールバック）
- 将来 type 別 Checker を整備する際は、`AccessControlDispatcher#filterAccessible` の switch 文に「実 Checker 呼び出し」を追加するのみで API スキーマ・Service 層の改修不要
- 採用理由: §10.4 の SQL 数 8 以内の構造を維持しつつ、参照先解決のフレームワーク（type 別バッチ判定 + N+1 防止）を構築するのが Phase 3 の本質。Checker 実装そのものは別軍議扱い（`TimelineAccessChecker` 等の整備は本機能のスコープ外）
