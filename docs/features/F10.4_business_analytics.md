# F10.4: 経営分析ダッシュボード

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 10
> **最終更新**: 2026-03-28
> **モジュール種別**: デフォルト機能（SYSTEM_ADMIN 専用）
> **関連ドキュメント**: F10.1 管理者ダッシュボード（`/system-admin/dashboard` の基盤統計）、F08.2 会費・決済（決済データソース）、F08.6 予算・会計管理、F09.7 広告、F09.11 広告主ダッシュボード（広告主向けの分析・請求。本機能はプラットフォーム運営者向け）

---

## 1. 概要

SYSTEM_ADMIN 向けの経営分析ダッシュボードを提供する。プラットフォーム全体の収益構造・成長指標・解約動向をリアルタイムに可視化し、データに基づいた経営判断を支援する。

F10.1 の `GET /api/v1/system-admin/dashboard` が返す基礎統計（MRR・アクティブサブスク数・滞納件数）を土台に、時系列推移・セグメント分析・ファネル分析・予測値を追加する。

### 本ドキュメントのスコープ

| 機能 | 説明 |
|------|------|
| 収益サマリ | MRR / ARR / ARPU / LTV / NRR / Quick Ratio の算出と時系列推移 |
| 課金ファネル | 無料登録 → モジュール有効化 → 課金開始 → 継続のコンバージョン率 |
| セグメント別分析 | 組織種別・チームテンプレート・規模・地域別の収益・ARPU 比較 |
| 解約分析（チャーン） | 月次チャーンレート・解約理由・コホート分析 |
| モジュール収益貢献度 | 各選択式モジュールの有効化数・収益・解約率 |
| 広告収益分析 | 広告インプレッション・クリック・収益の推移（Phase 9 広告機能連携） |
| 収益予測 | 過去データに基づく 3/6/12 ヶ月の収益トレンド予測 |
| アラート閾値通知 | KPI 異常値の自動検知と SYSTEM_ADMIN へのプッシュ通知 |
| データバックフィル | 過去データの再集計（バッチロジック修正時） |
| 月次レポート自動生成 | 主要 KPI スナップショットの自動保存とメール配信 |

### F10.1 との関係

- F10.1 の `billing_stats`（MRR・active_subscriptions・overdue_count）はリアルタイム集計のスナップショット
- 本機能は **日次集計テーブル**に蓄積した時系列データを元に、推移・トレンド・セグメント比較を提供する
- F10.1 の週次/月次レポート（`/reports/weekly`, `/reports/monthly`）のデータソースとして以下のテーブルを参照:
  - 週次レポート: `analytics_daily_revenue`（直近7日）、`analytics_daily_users`（直近7日）
  - 月次レポート: `analytics_daily_revenue`（当月）、`analytics_daily_users`（当月）、`analytics_monthly_cohorts`（当月分）
- **数値整合性**: F10.1 のリアルタイム MRR と F10.4 の日次 MRR はバッチ実行タイミングにより差異が生じうる。ダッシュボード上の MRR はリアルタイム値（F10.1）を表示し、トレンドグラフは日次集計値（F10.4）を使用する

### テーブル定義

#### `analytics_daily_revenue`

日次の収益を収益源別に集計。バッチジョブが毎日深夜に INSERT する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `date` | DATE | NO | — | 集計対象日 |
| `revenue_source` | ENUM('MODULE_SUBSCRIPTION', 'ONE_TIME_PAYMENT', 'STORAGE_ADDON', 'ORG_COUNT_BILLING', 'ADVERTISING', 'AFFILIATE') | NO | — | 収益源 |
| `gross_revenue` | DECIMAL(12,2) | NO | 0 | 総収益（税込） |
| `refund_amount` | DECIMAL(12,2) | NO | 0 | 返金額 |
| `net_revenue` | DECIMAL(12,2) | NO | 0 | 純収益（gross - refund） |
| `transaction_count` | INT UNSIGNED | NO | 0 | 取引件数 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | レコード作成日時 |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | レコード更新日時 |

**UNIQUE KEY**: `(date, revenue_source)`

**備考**: UNIQUE KEY のプレフィックスにより `date` 単独での範囲検索も INDEX 利用可能。

## 4. API設計

### 共通エラーレスポンス

全エンドポイント共通で以下のエラーを返す。

| ステータス | 条件 | レスポンス |
|-----------|------|-----------|
| 400 | 日付形式が不正 / `from > to` / 日付範囲超過 / ENUM パラメータが不正値 | `ErrorResponse` (error + fieldErrors) |
| 401 | 未認証（JWT なし or 期限切れ） | `ErrorResponse` |
| 403 | SYSTEM_ADMIN 以外のロール | `ErrorResponse` |

### 入力バリデーション共通ルール

| ルール | 内容 |
|-------|------|
| 日付範囲の最大幅 | `granularity=DAILY`: 最大 90 日、`WEEKLY`: 最大 52 週（364日）、`MONTHLY`: 最大 36 ヶ月。超過時は 400 |
| `from > to` | 400 Bad Request を返す |
| 未来日 | `date` / `to` に未来日を指定した場合、最新の集計済み日付に切り詰めて処理（エラーにはしない） |
| ENUM 不正値 | `granularity`, `segment_by`, `metric`, `sort_by`, `type` に定義外の値を渡した場合は 400 |
| `forecast.months` | 3 / 6 / 12 のいずれかのみ許可。それ以外は 400 |

#### `GET /api/v1/system-admin/analytics/revenue/summary`

**認可**: SYSTEM_ADMIN

**クエリパラメータ**:
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `date` | DATE | NO | 基準日（デフォルト: 最新の集計済み日付。バッチ未完了の場合は前日） |

**レスポンス（200 OK）**
```json
{
  "data": {
    "date": "2026-03-28",
    "mrr": 450000,
    "mrr_growth_rate": 5.2,
    "arr": 5400000,
    "arpu": 529,
    "arpu_paying_only": 1607,
    "ltv": 19284,
    "paying_users": 280,
    "total_active_users": 8500,
    "paying_ratio": 3.29,
    "nrr": 108.5,
    "quick_ratio": 4.2,
    "payback_months": 12.3,
    "comparison": {
      "mrr_prev_month": 427800,
      "mrr_change_pct": 5.19,
      "mrr_yoy_change_pct": 42.3,
      "arpu_prev_month": 503,
      "arpu_change_pct": 5.17,
      "arpu_yoy_change_pct": 35.8,
      "nrr_prev_month": 106.2,
      "quick_ratio_prev_month": 3.8
    },
    "stripe_reconciliation": {
      "last_reconciled_date": "2026-03-27",
      "status": "MATCHED",
      "discrepancy_amount": 0,
      "discrepancy_pct": 0.0
    }
  }
}
```

`stripe_reconciliation.status` の値: `MATCHED`（乖離なし）/ `DISCREPANCY`（乖離あり・要確認）/ `PENDING`（照合未実行）。詳細は §9.1 参照。

**計算式**:
- `MRR` = 当月の定期課金収益合計（analytics_daily_revenue の MODULE_SUBSCRIPTION + STORAGE_ADDON + ORG_COUNT_BILLING を月合算）
- `ARR` = MRR × 12
- `ARPU` = 当月純収益合計 / アクティブユーザー数（`active_users = 0` の場合は `null` を返す）
- `ARPU（課金ユーザーのみ）` = 当月純収益合計 / 課金ユーザー数（`paying_users = 0` の場合は `null` を返す）
- `LTV` = ARPU / 月次チャーンレート（チャーンレート = 0 の場合は `null` を返す。解約者ゼロ＝LTV 無限大のため算出不可）
- `NRR`（Net Revenue Retention）= (月初 MRR + Expansion MRR - Churned MRR) / 月初 MRR × 100。100% 超 = 既存顧客だけで成長。月初 MRR = 0 の場合は `null`
- `Quick Ratio` = (新規 MRR + 復活 MRR + Expansion MRR) / Churned MRR。Churned MRR = 0 の場合は `null`（解約ゼロ＝理想状態）。4.0 以上が健全の目安
- `Payback Period`（月） = 当月広告・マーケティング費用合計 / 新規顧客1人あたり月間平均収益。新規ユーザーがゼロの場合は `null`。マーケティング費用は `analytics_daily_revenue` の `ONE_TIME_PAYMENT`（広告出稿費）から推定、正確な CAC は外部データ（広告プラットフォーム費用）が必要なため概算値
- `mrr_yoy_change_pct` = (当月 MRR - 前年同月 MRR) / 前年同月 MRR × 100。前年同月データが未蓄積の場合は `null`

## 5. ビジネスロジック

### 日次集計バッチ

```
1. @Scheduled(cron="0 0 2 * * *", zone="Asia/Tokyo") で毎日 AM 2:00 (JST) に実行
2. 排他制御: ShedLock (@SchedulerLock) で同時実行を防止。ロックは最大30分保持
3. 対象: 前日分（date = yesterday (JST)）
4. 処理順序:
   a. analytics_daily_revenue: member_payments (status=PAID, paid_at=yesterday) + member_payments (status=REFUNDED, refunded_at=yesterday) を revenue_source 別に集計
   b. analytics_daily_users: users テーブルから新規登録数、ログイン履歴から DAU、member_subscriptions から課金/解約/復帰数を集計
   c. analytics_daily_modules: module_subscriptions のスナップショットを集計
   d. analytics_daily_ads: ad_impressions / ad_clicks を集計（Phase 9 テーブルが存在する場合のみ）
   e. analytics_funnel_snapshots: 各ステージの累計ユーザー数をスナップショット
5. 冪等性: UNIQUE KEY (date, ...) により INSERT ... ON DUPLICATE KEY UPDATE で上書き可能
6. エラー時:
   a. batch_job_logs に ERROR 記録
   b. SYSTEM_ADMIN にプッシュ通知（F04.3 連携）で即時通知
   c. 手動リトライ: F10.1 の POST /api/v1/system-admin/batch-jobs/{jobName}/retry で再実行可能
   d. 次回定時実行でも自動再試行
```

### 月次コホートバッチ

```
1. @Scheduled(cron="0 0 3 1 * *", zone="Asia/Tokyo") で毎月1日 AM 3:00 (JST) に実行
2. 排他制御: ShedLock (@SchedulerLock) で同時実行を防止。ロックは最大60分保持
3. 処理:
   a. 全コホート（過去24ヶ月分）について、前月末時点の残存数を再計算
   b. analytics_monthly_cohorts に INSERT ... ON DUPLICATE KEY UPDATE
4. コホートサイズ: その月に users.created_at (JST 基準) で登録したユーザー数
5. 残存判定: member_subscriptions.status = 'ACTIVE' または users.last_login_at が当月内
6. 残存課金: member_subscriptions.status = 'ACTIVE' かつ member_payments に当月の PAID レコードあり
7. エラー時: 日次集計バッチと同様（batch_job_logs + SYSTEM_ADMIN 通知 + 手動リトライ対応）
```

### 収益予測ロジック

```
1. 直近6ヶ月の月次 net_revenue を取得
2. データ点が6以上:
   a. 単純線形回帰で傾き (slope) と切片 (intercept) を算出
   b. 各予測月の projected_mrr = intercept + slope × month_index
   c. confidence_low / high = projected_mrr ± 1.96 × 残差の標準偏差
3. データ点が3〜5（サービス初期）:
   a. 直近3ヶ月の平均成長率を算出
   b. projected_mrr = current_mrr × (1 + growth_rate) ^ month_offset
   c. confidence_low / high = ±15%（固定幅）
4. データ点が2以下:
   a. 予測不可。forecast 配列を空で返す
   b. assumptions.method = "INSUFFICIENT_DATA"
5. projected_users: 同様のロジックで total_users を予測
```

### セグメント集計ロジック

```
1. ORG_TYPE: organizations.org_type (NONPROFIT/FORPROFIT) で GROUP BY → 各組織に紐づく teams → member_payments を集計
2. TEAM_TEMPLATE: teams.template (SPORTS/SCHOOL/CLINIC 等) で GROUP BY → member_payments を集計
3. ORG_SIZE: organizations ごとにメンバー数をカウント → SMALL/MEDIUM/LARGE/ENTERPRISE に分類 → 集計
4. REGION: organizations.prefecture で GROUP BY → 集計（最大47件）
5. ARPU: セグメント内の net_revenue / active_users（active_users = 0 の場合は null）
6. churn_rate: セグメント内の当月解約 / 月初課金ユーザー数
```

### アラート閾値評価ロジック

日次集計バッチの完了直後に実行される。

```
1. 日次集計バッチの全 Step 完了後にイベント発火（ApplicationEvent）
2. AnalyticsAlertEvaluator が有効な全ルール（enabled=true, deleted_at IS NULL）を取得
3. 各ルールについて:
   a. metric に対応する最新値を取得（analytics_daily_* テーブルから）
   b. comparison_period に基づく比較値を取得:
      - PREV_DAY: 前日の同メトリック
      - PREV_WEEK: 7日前の同メトリック
      - PREV_MONTH: 前月同日（存在しない場合は前月末日）
      - AVG_30D: 過去30日の平均値
   c. condition に基づき判定:
      - ABOVE: metric_value > threshold
      - BELOW: metric_value < threshold
      - CHANGE_ABOVE: change_pct > threshold
      - CHANGE_BELOW: change_pct < threshold
   d. 条件を満たした場合:
      - analytics_alert_history に INSERT
      - consecutive_triggers チェック: 直近 N 回の評価結果が連続で条件を満たしているか確認
      - cooldown チェック: 前回の notified=true レコードから cooldown_hours が経過しているか
      - 両方を満たす場合のみ通知を実行:
        - PUSH: F04.3 プッシュ通知で SYSTEM_ADMIN に送信
        - EMAIL: F09.6 SES 経由でメール送信
        - analytics_alert_history.notified = true に更新
4. エラー時: 個別ルールの評価失敗は他ルールに影響しない（各ルール独立で try-catch）
```

### データバックフィルロジック

```
1. POST /backfill で非同期ジョブとして起動（@Async）
2. 排他制御: ShedLock で同時に1ジョブのみ実行可能
3. from〜to の日付範囲を1日ずつループ:
   a. targets に応じて日次集計バッチと同じロジックを実行
   b. INSERT ... ON DUPLICATE KEY UPDATE で既存データを安全に上書き
   c. 進捗を batch_job_logs に記録（10日ごとに中間ログ）
4. targets に COHORTS が含まれる場合:
   a. from〜to に含まれる全月について月次コホートバッチを再実行
5. 完了時:
   a. batch_job_logs に COMPLETED 記録
   b. SYSTEM_ADMIN にプッシュ通知（「バックフィル完了: {from}〜{to}, {targets}」）
6. エラー時:
   a. 個別日のエラーはスキップして次の日に進む（部分成功を許容）
   b. スキップした日を batch_job_logs に WARNING 記録
   c. 全日失敗した場合のみ FAILED として通知
```

### 月次 KPI スナップショットバッチ

```
1. @Scheduled(cron="0 0 4 1 * *", zone="Asia/Tokyo") で毎月1日 AM 4:00 (JST) に実行
   （月次コホートバッチ AM 3:00 の後）
2. 排他制御: ShedLock で同時実行を防止
3. 対象: 前月分
4. 処理:
   a. analytics_daily_revenue から前月の net_revenue を合算
   b. analytics_daily_users から前月末日の active_users / paying_users / total_users を取得
   c. analytics_daily_users から前月の new_users / churned_users を合算
   d. 前月の MRR / ARR / ARPU / LTV / NRR / Quick Ratio / Payback Period を算出
      （revenue/summary API と同一ロジック）
   e. analytics_monthly_snapshots に INSERT ... ON DUPLICATE KEY UPDATE
5. 自動メール送信:
   a. スナップショット保存後、SYSTEM_ADMIN 全員のメールアドレスを取得
   b. F09.6 SES 経由で月次レポートメールを自動送信
   c. report_sent = true, report_sent_at = NOW() に更新
6. エラー時: batch_job_logs に ERROR + SYSTEM_ADMIN 通知
```

サービス開始時に以下のアラートルールを初期データとして投入する（Flyway or アプリ起動時シード）:

## 6. セキュリティ考慮事項

- **認可チェック**: 全 20 エンドポイントは SYSTEM_ADMIN 専用。`SecurityFilterChain` で `/system-admin/**` パスに対し `hasRole('SYSTEM_ADMIN')` を適用。ロールが不足する場合は 403 を返却
- **SYSTEM_ADMIN 以外のアクセス遮断**: ADMIN / DEPUTY_ADMIN を含む全ロールからアクセス不可。経営データ（MRR・LTV・NRR・チャーンレート・収益予測）はプラットフォーム運営者のみが閲覧すべき機密情報
- **レートリミット**: `Bucket4j` で以下の制限を適用
  - 分析 API（`/system-admin/analytics/**`）: 1分間30件（重い集計クエリの連打防止）
  - CSV エクスポート（`/system-admin/analytics/export`）: 1時間10件（大量データ抽出の負荷抑制）
  - バックフィル（`/system-admin/analytics/backfill`）: 1時間1件（重い再集計処理の連打防止）
  - レポート手動送信（`.../send-report`）: 1時間5件（メール送信スパム防止）
- **CSV エクスポートのセキュリティ**: エクスポートデータには収益額・ユーザー数等の経営機密が含まれるため、以下を適用
  - エクスポート実行時に監査ログ（`DATA_EXPORT_REQUESTED`）を記録（実行者・エクスポート種別・期間を含む）
  - レスポンスヘッダーに `Cache-Control: no-store` を設定（ブラウザキャッシュ防止）
- **監査ログ**: 以下のイベントを `audit_logs` に記録
  - `ANALYTICS_EXPORT_REQUESTED`: CSV エクスポート実行（type・from・to を含む）
  - `ANALYTICS_FORECAST_VIEWED`: 収益予測の閲覧（months パラメータを含む）
  - `ANALYTICS_ALERT_CREATED`: アラートルール作成（metric・condition・threshold を含む）
  - `ANALYTICS_ALERT_UPDATED`: アラートルール更新（変更前後の値を含む）
  - `ANALYTICS_ALERT_DELETED`: アラートルール削除
  - `ANALYTICS_BACKFILL_REQUESTED`: バックフィル実行（from・to・targets・実行者を含む）
  - `ANALYTICS_REPORT_SENT`: 月次レポート送信（month・recipients を含む）
- **バックフィルの安全性**: バックフィルは `INSERT ... ON DUPLICATE KEY UPDATE` で冪等に実行。手動トリガーのため、実行者と対象期間を監査ログに必ず記録
- **バッチジョブの保護**: 日次集計・月次コホート・月次スナップショット・照合バッチは `@Scheduled` による内部実行のみ。バックフィルのみ API から手動トリガー可能（SYSTEM_ADMIN 限定 + 同時実行制限1件）
- **Stripe API キーの保護**: 照合バッチで使用する Stripe API キーは環境変数 or AWS Secrets Manager で管理。DB・ログへの出力禁止
- **集計データの完全性**: `analytics_*` テーブルの UNIQUE KEY 制約により、同一日・同一キーの重複 INSERT を防止。バッチの冪等性を保証
- **メールレポートの情報漏洩対策**: 月次レポートメールには経営機密（MRR・チャーンレート等）が含まれるため、受信者は SYSTEM_ADMIN のメールアドレスに限定。TLS 暗号化（SES のデフォルト）で送信

## 8. パフォーマンス考慮

- 全 API は**事前集計テーブル**を参照するため、リアルタイム集計による負荷を回避
- 日次バッチは AM 2:00〜3:00 のオフピーク時間帯に実行
- `date` カラムに INDEX があるため、期間指定クエリは高速
- `granularity=WEEKLY/MONTHLY` の場合はアプリ層で日次データを GROUP BY 集約（最大365行の集約なので問題なし）
- コホート分析は最大24ヶ月 × 24ヶ月 = 576行の小テーブルのため、全件取得でも問題なし

### API レスポンスキャッシュ戦略

集計データは日次バッチ完了後にのみ更新されるため、HTTP キャッシュを活用して同一条件のリクエスト負荷を削減する。

| エンドポイント種別 | キャッシュヘッダー | 理由 |
|------------------|------------------|------|
| 閲覧系（revenue/*, users/*, churn, cohorts, funnel, modules/*, segments, ads, forecast, snapshots） | `Cache-Control: private, max-age=300`（5分）、`ETag` ヘッダー付与 | 同一 SYSTEM_ADMIN が短時間にリロードする場面を最適化。`private` で CDN キャッシュを防止 |
| アラート一覧 / 履歴 | `Cache-Control: private, max-age=60`（1分） | ルール変更や新規発火の反映を比較的早く |
| エクスポート | `Cache-Control: no-store` | 経営機密。キャッシュ厳禁 |
| 書き込み系（POST / PUT / DELETE） | キャッシュなし | — |

**ETag 生成**: 集計テーブルの `MAX(updated_at)` をハッシュして `ETag` を生成。クライアントが `If-None-Match` ヘッダーを送信した場合、一致すれば `304 Not Modified` を返す。

**バッチ完了時のキャッシュ無効化**: 日次バッチ完了時に内部キャッシュ（Spring Cache）を全エビクトする。ETag の値も更新されるため、次回リクエストで最新データが返る。

### 9.1 Stripe ダッシュボードとの数値整合性検証

#### 方針

Stripe を「決済の正（Source of Truth）」とし、本システムの集計値との乖離を自動検知する。

#### 日次照合バッチ

```
1. @Scheduled(cron="0 0 4 * * *", zone="Asia/Tokyo") で毎日 AM 4:00 (JST) に実行（日次集計バッチ AM 2:00 の後）
   排他制御: ShedLock で同時実行を防止
2. Stripe Balance Transactions API で前日分の決済データを取得:
   GET /v1/balance_transactions?created[gte]={yesterday_start}&created[lt]={today_start}&type=charge
   GET /v1/balance_transactions?created[gte]={yesterday_start}&created[lt]={today_start}&type=refund
3. 照合ロジック:
   a. Stripe 側の charge 合計額 vs analytics_daily_revenue の gross_revenue 合計
   b. Stripe 側の refund 合計額 vs analytics_daily_revenue の refund_amount 合計
   c. Stripe 側の取引件数 vs analytics_daily_revenue の transaction_count 合計
4. 許容誤差: 乖離率 ±0.1% または 絶対額 ±100円 のいずれか大きい方（少額取引が多い場合の丸め誤差を吸収）
5. 乖離検知時:
   a. batch_job_logs に WARNING レコードを記録（乖離額・乖離率を含む）
   b. SYSTEM_ADMIN にプッシュ通知（F04.3 連携）
   c. 自動修正は行わない（手動リコンシリエーションで対応）
6. 冪等性: 同日の再実行は前回結果を上書き
```

#### 手動リコンシリエーション

F08.2 で設計済みの `POST /api/v1/admin/stripe/reconcile/{paymentId}`（SYSTEM_ADMIN 専用）を利用する。

- 個別の member_payments について Stripe Checkout Session / PaymentIntent の状態を取得し、DB と同期
- 照合バッチで乖離が検知された場合、乖離の原因となった個別決済を特定 → 手動リコンシリエーションで修正

#### 経営分析ダッシュボードでの表示

```json
// GET /api/v1/system-admin/analytics/revenue/summary のレスポンスに追加
{
  "data": {
    // ... 既存フィールド ...
    "stripe_reconciliation": {
      "last_reconciled_date": "2026-03-27",
      "status": "MATCHED",
      "discrepancy_amount": 0,
      "discrepancy_pct": 0.0
    }
  }
}
```

`status` の値:
- `MATCHED`: 乖離なし（許容誤差内）
- `DISCREPANCY`: 乖離あり（要確認）
- `PENDING`: 照合未実行（サービス開始直後等）

### 9.2 広告収益の集計ロジック確定

#### データソース（F09.7 設計済み）

| テーブル | 主要カラム | 保持期間 |
|---------|-----------|---------|
| `ad_impressions` | ad_id, campaign_id, team_id, user_id(NULL可), occurred_at | 90日（生ログ） |
| `ad_clicks` | ad_id, campaign_id, team_id, user_id(NULL可), occurred_at | 90日（生ログ） |
| `ad_campaigns` | advertiser_organization_id, daily_impression_limit, status | 永続 |

#### 集計ロジック（日次バッチ analytics_daily_ads）

```
1. 日次集計バッチ（AM 2:00）の Step d で実行
2. 前提条件: ad_impressions / ad_clicks テーブルが存在する場合のみ実行
   - Phase 9 未実装時はスキップ（テーブル存在チェックで判定）
3. 集計:
   a. impressions = SELECT COUNT(*) FROM ad_impressions WHERE DATE(occurred_at) = yesterday
   b. clicks = SELECT COUNT(*) FROM ad_clicks WHERE DATE(occurred_at) = yesterday
   c. ctr = clicks / impressions（impressions = 0 の場合は 0）
4. 収益計算（収益モデルに応じて分岐）:
   a. アフィリエイト（CPS）: 外部 API（Amazon Associates 等）から前日分の成果報酬を取得
      - 取得不可の場合: ad_revenue = 0、batch_job_logs に WARNING
   b. CPM 広告: SUM(campaign.cpm_rate × impressions / 1000) を campaign 別に計算
   c. CPC 広告: SUM(campaign.cpc_rate × clicks) を campaign 別に計算
5. ecpm = ad_revenue × 1000 / impressions（impressions = 0 の場合は 0）
6. analytics_daily_revenue にも revenue_source = 'ADVERTISING' として INSERT
```

#### analytics_daily_revenue との連携

`analytics_daily_ads` は広告特化の詳細指標（CTR・eCPM 等）を保持し、`analytics_daily_revenue` の `ADVERTISING` 行は収益額のみを保持する。両テーブルの `ad_revenue` / `net_revenue` は一致する。

**ゼロ除算の保証**: `ctr` / `ecpm` はバッチ側で `impressions = 0` チェック済みの値を DB に格納する。API は DB 値をそのまま返し、再計算しない。

### 9.3 収益予測モデルの決定

#### 採用: 線形回帰（Phase 10 初期リリース）

| 項目 | 決定内容 |
|------|---------|
| **初期モデル** | 単純線形回帰（OLS） |
| **理由** | 実装が単純、Java 標準ライブラリ（Apache Commons Math `SimpleRegression`）で完結、解釈が容易 |
| **データ要件** | 最低6ヶ月の月次データ（それ未満は成長率ベースにフォールバック） |
| **信頼区間** | 95%信頼区間（±1.96σ） |

#### 季節性への対応方針

```
Phase 10 初期:
  - 線形回帰のみ。季節性は考慮しない
  - assumptions.method = "LINEAR_REGRESSION" をレスポンスに明示

将来拡張（12ヶ月以上のデータ蓄積後）:
  - 12ヶ月周期の季節性指数を算出（同月比の平均乖離率）
  - 線形回帰の予測値に季節性指数を乗算する「乗法型季節調整」を追加
  - assumptions.method = "LINEAR_REGRESSION_SEASONAL"
  - Apache Commons Math の DescriptiveStatistics で月別平均を算出可能（外部ライブラリ追加不要）
```

#### 不採用とした代替案

| モデル | 不採用理由 |
|-------|-----------|
| ARIMA / SARIMA | Python (statsmodels) 前提、Java 実装は複雑でメンテコスト高 |
| Prophet | Python ライブラリ、Spring Boot に組み込むには Python サイドカーが必要 |
| 指数平滑法 | 線形回帰とほぼ同精度だが実装が若干複雑、初期段階ではオーバースペック |

#### 予測精度の表示

ダッシュボード上に「予測精度に関する注意」を常時表示する:
- データ蓄積期間が6ヶ月未満: 「⚠ データ不足のため簡易予測です。精度は限定的です」
- データ蓄積期間が6〜11ヶ月: 「予測は直近のトレンドに基づいています。季節変動は考慮されていません」
- データ蓄積期間が12ヶ月以上: （将来）季節調整モデルへの自動切替通知

### 9.4 グラフライブラリの確定

#### 採用: Chart.js（PrimeVue Charts / vue-chartjs ラッパー）

プロジェクト全体で既に決定済み（F02.2 ダッシュボード、F07.2 パフォーマンス、F07.4 カルテで同一ライブラリを使用）。

| 項目 | 内容 |
|------|------|
| **ライブラリ** | Chart.js 4.x（vue-chartjs ラッパー経由） |
| **PrimeVue 統合** | PrimeVue Charts コンポーネント（`<Chart>` タグ）で利用 |
| **Tree-shaking** | 必要な要素のみ登録（BarElement, LineElement, ArcElement, PointElement, Filler, Tooltip, Legend） |
| **ダークモード** | Tailwind CSS のカラー変数と連動 |
| **アニメーション** | duration: 1000ms, easing: 'easeOutQuart' |

#### 経営分析ダッシュボードで使用するグラフ種別

| 画面 | グラフ種別 | Chart.js タイプ |
|------|-----------|----------------|
| 収益推移 | 折れ線グラフ（面積塗り） | `line` (fill: true) |
| 収益源別内訳 | ドーナツチャート | `doughnut` |
| ユーザー動態 | 積み上げ棒グラフ | `bar` (stacked: true) |
| チャーンレート | 折れ線グラフ（二軸） | `line` (dual y-axis) |
| コホート分析 | ヒートマップ風テーブル | CSS グリッド + 色分け（Chart.js 不使用） |
| ファネル | 横棒グラフ（漏斗型） | `bar` (horizontal, 幅で遷移率を表現) |
| モジュールランキング | 横棒グラフ | `bar` (horizontal, indexAxis: 'y') |
| 収益予測 | 折れ線 + 信頼区間帯 | `line` (実績: solid, 予測: dashed, 信頼区間: fill between) |
| KPI サマリカード | 数値 + 前月比/前年同月比の矢印 | PrimeVue Card + CSS（Chart.js 不使用） |
| アラート履歴 | タイムラインリスト | PrimeVue Timeline コンポーネント（Chart.js 不使用） |
| 月次スナップショット比較 | 折れ線（複数 KPI オーバーレイ） | `line` (multi-axis, NRR/Quick Ratio を右軸) |

#### コホート分析のヒートマップ表現

Chart.js にはネイティブのヒートマップがないため、CSS グリッドで実装する:
- 行: コホート月、列: 経過月数
- セルの背景色: 残存率に応じた Tailwind カラーグラデーション（green-500 → yellow-500 → red-500）
- セル内テキスト: 残存率（%）
- PrimeVue の `DataTable` コンポーネントでセル背景をカスタマイズすることで実現可能
