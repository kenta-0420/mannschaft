> **ステータス**: 🟢 設計完了

> **関連ドキュメント**: スケジュール基盤 → `F03.1_schedule_shared.md` / CMS/ブログ → `F06.1_cms_blog.md` / 個人スケジュール → `F03.2_schedule_personal.md`

## 1. 概要

カレンダー予定（schedules）に写真・動画を添付する機能を提供する。試合写真・練習記録・遠征の記録など、スケジュールに紐付いたメディアを Cloudflare R2 に保存し、事後確認・活動記録・経費証憑として活用できる。ブログ投稿（F06.1）や経費精算（F13.x 予定）へのメディア転用も設計に組み込む。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全スケジュールのメディアを参照・強制削除 |
| ADMIN | スケジュールへのメディアアップロード・削除・一覧参照 |
| DEPUTY_ADMIN | `MANAGE_SCHEDULES` 権限を持つ場合: アップロード・削除・一覧参照 |
| MEMBER | スケジュール作成者または `MANAGE_SCHEDULES` 権限を持つ場合: アップロード・自分がアップロードしたメディアの削除・自分のメディアの `caption` / `taken_at` / `is_expense_receipt` 更新。`is_cover` の変更は ADMIN/DEPUTY_ADMIN のみ。閲覧は `schedule.min_view_role` に準じる |
| SUPPORTER | スケジュールを閲覧できる場合、添付メディアも閲覧可能（アップロード・削除は不可）|
| GUEST | `min_view_role = 'ANYONE'` のスケジュールのメディアのみ閲覧可 |

### 対象レベル
- [x] 組織 (Organization)
- [x] チーム (Team)
- [x] 個人 (Personal)（schedulesと同じスコープを引き継ぐ）

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `schedule_media_uploads` | スケジュール添付メディア（写真・動画）の管理。R2 キー・メタ情報・経費証憑フラグ等を保持 | なし（物理削除 + R2 削除）|

### テーブル定義

#### `schedule_media_uploads`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `schedule_id` | BIGINT UNSIGNED | YES | NULL | FK → schedules（SET NULL on delete。NULL = 未紐付け / アップロード中）|
| `uploader_id` | BIGINT UNSIGNED | YES | NULL | FK → users（SET NULL on delete）|
| `media_type` | ENUM('IMAGE', 'VIDEO') | NO | 'IMAGE' | メディア種別 |
| `r2_key` | VARCHAR(500) | NO | — | R2 オブジェクトキー（`schedules/{scheduleId}/{uuid}.{ext}` 形式。単一バケット `mannschaft-storage` 内） |
| `thumbnail_r2_key` | VARCHAR(500) | YES | NULL | 動画サムネイル R2 オブジェクトキー（VIDEO のみ）|
| `file_name` | VARCHAR(255) | NO | — | 元のファイル名（アップロード時の表示名）|
| `file_size` | BIGINT UNSIGNED | NO | — | ファイルサイズ（bytes）|
| `content_type` | VARCHAR(50) | NO | — | MIME タイプ（`image/jpeg`, `image/png`, `image/webp`, `video/mp4` 等）|
| `duration_seconds` | INT UNSIGNED | YES | NULL | 動画再生時間（秒。VIDEO のみ）|
| `caption` | VARCHAR(500) | YES | NULL | 写真キャプション・説明文（最大500文字）|
| `taken_at` | DATETIME | YES | NULL | 撮影日時（Exif から取得または手動入力）|
| `is_cover` | BOOLEAN | NO | FALSE | スケジュールのカバー写真フラグ（schedule_media_uploads 内で1枚のみ TRUE 可）|
| `is_expense_receipt` | BOOLEAN | NO | FALSE | 経費証憑フラグ（交通費精算等の証憑として指定。F13.x 経費精算機能連携用）|
| `processing_status` | ENUM('PENDING', 'PROCESSING', 'READY', 'FAILED') | NO | 'READY' | IMAGE は即時 READY。VIDEO は PENDING → PROCESSING → READY に遷移 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_smu_schedule  (schedule_id)                    -- スケジュール別メディア一覧
INDEX idx_smu_orphan    (schedule_id, created_at)        -- 孤立メディアクリーンアップ用
UNIQUE KEY uq_smu_r2_key (r2_key)                        -- R2 キー一意性
INDEX idx_smu_uploader  (uploader_id)                    -- アップローダー別一覧
INDEX idx_smu_cover     (schedule_id, is_cover)          -- カバー写真高速取得
INDEX idx_smu_expense   (schedule_id, is_expense_receipt) -- 経費証憑フィルタ用
```

**制約・備考**
- `is_cover = TRUE` は schedule_id ごとに1件のみ許容（アプリ層で制御。UPDATE で切り替え時に旧カバーを FALSE に）
- `is_expense_receipt = TRUE` のメディアは F13.x 経費精算機能から参照される（削除時は経費申請が紐付いている場合は 409 を返す）
- 孤立メディアクリーンアップ: `schedule_id IS NULL AND created_at < NOW() - INTERVAL 72 HOUR` を日次バッチで削除
- `taken_at` は EXIF が取得できない場合は NULL 許容（フロントエンドで任意手動入力）
- 動画対応フォーマット: MP4 (H.264+AAC)推奨、WebM(VP9)、MOV(QuickTime)
- 1スケジュールあたりの上限: 画像 **50枚**、動画 **5本**（ブログより緩い。活動記録の性質上）
- 個々ファイルサイズ上限: 画像 **50MB**、動画 **1GB**（UXガード。容量課金はスコープの容量枠に従う）
- クォータ計上: `StorageQuotaService` 経由でスコープに計上（ブログと共通ロジック）
- 論理削除なし（物理削除 + R2 削除の組み合わせ）

```
schedules (1) ──── (N) schedule_media_uploads (R2 画像・動画管理)
schedules (N) ──── (1) users (uploader)
```

## 4. API設計

### エンドポイント一覧
| メソッド | パス | 認証 | 説明 |
|---------|-----|------|------|
| POST | `/api/v1/schedules/{scheduleId}/media/upload-url` | 必要 | アップロード用署名付きURLを発行 |
| GET | `/api/v1/schedules/{scheduleId}/media` | 必要 | メディア一覧取得 |
| PATCH | `/api/v1/schedules/{scheduleId}/media/{mediaId}` | 必要 | キャプション・撮影日時・カバー設定・経費証憑フラグ更新 |
| DELETE | `/api/v1/schedules/{scheduleId}/media/{mediaId}` | 必要 | メディア削除（R2 + DB） |

### リクエスト／レスポンス仕様

#### `POST /api/v1/schedules/{scheduleId}/media/upload-url`

**リクエストボディ**
```json
{
  "mediaType": "IMAGE",
  "contentType": "image/jpeg",
  "fileSize": 2048000,
  "fileName": "practice_photo.jpg"
}
```

> `scheduleId` はパスパラメータから取得するため、ボディには含めない。アップロード時にパスの `{scheduleId}` を用いて直接 `schedule_id` を紐付ける（ブログと異なり、本文解析による後付け紐付けは使用しない）。

**レスポンス（200 OK）**
```json
{
  "data": {
    "mediaId": 123,
    "uploadUrl": "https://...(Presigned PUT URL, 600秒有効)",
    "r2Key": "schedules/456/uuid.jpg"
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | mediaType / contentType 不正、fileSize = 0、画像50枚超・動画5本超 |
| 403 | アップロード権限なし |
| 404 | scheduleId 不存在 |

#### `PATCH /api/v1/schedules/{scheduleId}/media/{mediaId}`

リクエストボディ（全フィールド任意）:
```json
{
  "caption": "練習後の集合写真",
  "takenAt": "2026-04-17T18:00:00",
  "isCover": true,
  "isExpenseReceipt": false
}
```

**レスポンス（200 OK）**: 更新後のメディアオブジェクト（GET 一覧の items 内の1件と同形式）

**権限マトリクス**
| フィールド | MEMBER（自分のメディア） | MEMBER（他人のメディア） | ADMIN/DEPUTY_ADMIN |
|-----------|------------------------|------------------------|-------------------|
| `caption` / `taken_at` | ✅ | ❌ | ✅ |
| `is_expense_receipt` | ✅（true への設定のみ。false への変更は不可） | ❌ | ✅ |
| `is_cover` | ❌ | ❌ | ✅ |

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | 権限なし（他人のメディアに対して MEMBER が変更しようとした場合、または MEMBER が制限フィールドを変更しようとした場合） |
| 404 | mediaId が scheduleId に紐付かない |
| 409 | `isExpenseReceipt = false` にしようとしたが経費申請が紐付いている（F13.x 実装前は常に許可）|

## 5. ビジネスロジック

### アップロードフロー

> スケジュールはブログと異なり「本文」を持たないため、URL解析による後付け紐付けは行わない。**アップロード時点で scheduleId を指定し、直接紐付け**する。

```
1. POST /upload-url に scheduleId + mediaType + contentType + fileSize + fileName を送信
2. バックエンドが schedule_media_uploads に INSERT（schedule_id = 指定値, processing_status = READY/PENDING）
3. IMAGE（100MB以下）: Presigned PUT URL を返却。クライアントが R2 へ直接 PUT
   VIDEO または 100MB超: Multipart Upload 開始。F05.5 Multipart Upload フロー（POST /api/v1/files/multipart/start, target_prefix = "schedules/{id}/"）を利用してアップロード後、返却された file_key を schedule_media_uploads に登録
4. 孤立メディアクリーンアップ（日次 02:30）: schedule_id IS NULL AND created_at < NOW() - 72H を削除
   ※ スケジュール削除時（ON DELETE SET NULL）に schedule_id が NULL になったレコードも72時間後に対象となる
```

### カバー写真切り替えフロー

```
1. PATCH /media/{mediaId} で isCover = true をリクエスト
2. 同一 schedule_id の既存 is_cover = TRUE を FALSE に UPDATE（@Transactional）
3. 対象レコードを is_cover = TRUE に UPDATE
4. schedule の cover_image_url を更新（denormalize）
```

### 経費証憑フラグとF13.x連携

```
1. PATCH /media/{mediaId} で isExpenseReceipt = true に設定
2. F13.x 経費申請時に GET /schedules/{id}/media?expenseReceiptOnly=true で証憑一覧を取得
3. 経費申請に紐付いた後は isExpenseReceipt = false への変更・削除を禁止（409 返却）
```

### ブログ投稿への転用フロー（将来連携 F06.1）

```
1. スケジュール詳細画面から「活動記録をブログに投稿」ボタン
2. POST /api/v1/blog-posts/from-schedule/{scheduleId} を呼び出し
3. schedule_media_uploads の is_cover = TRUE の写真を blog_posts.cover_image_url にセット
4. schedule.result_summary を blog_posts.body の初期テキストに流用
5. ユーザーが編集して公開（自動公開はしない）
```

## 6. セキュリティ考慮事項

- **認可チェック**: `ScheduleMediaService` の入り口で `scheduleId` とリクエストユーザーのスコープ所属を検証。スケジュールが属するチーム/組織への所属がない場合は 403
- **署名付きURL（Presigned URL）**: 有効期限 600秒。URLにスコープキー（`schedules/{scheduleId}/`）を含め、バケットポリシーで他スコープへのアクセスを制限
- **ファイルサイズ検証**: `fileSize` パラメータとR2の実際のサイズを不一致チェック（Workers 側でポストアップロード検証）
- **MIMEタイプ検証**: `contentType` を許可リスト（jpeg, png, webp, gif, mp4, webm, mov）で検証。svg は XSS リスクのため禁止
- **レートリミット**: アップロードURL発行は `Bucket4j` で1分間に20リクエストまで
- **経費証憑アクセス制御**: `isExpenseReceipt = true` への設定は MEMBER（自分のメディアに限る）も可。`isExpenseReceipt = false` への変更（証憑解除）は ADMIN/DEPUTY_ADMIN のみ許可（F13.x 実装前は例外的に ADMIN も含め全権限で許可）。経費申請に紐付いた後の証憑解除は 409 を返す（詳細は §4 PATCH 権限マトリクス参照）
- **孤立メディアの漏洩防止**: `schedule_id = NULL` のメディアは GET /media 一覧には含めない。URLを知っていてもR2側ではWorkers経由のアクセスのみ許可

```sql
-- V13.015__create_schedule_media_uploads.sql
CREATE TABLE schedule_media_uploads (
    id                  BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    schedule_id         BIGINT UNSIGNED NULL     COMMENT 'FK → schedules (SET NULL on delete)',
    uploader_id         BIGINT UNSIGNED NULL     COMMENT 'FK → users (SET NULL on delete)',
    media_type          ENUM('IMAGE', 'VIDEO') NOT NULL DEFAULT 'IMAGE',
    r2_key              VARCHAR(500)    NOT NULL COMMENT 'R2 オブジェクトキー (schedules/{id}/{uuid}.ext)',
    thumbnail_r2_key    VARCHAR(500)    NULL,
    file_name           VARCHAR(255)    NOT NULL,
    file_size           BIGINT UNSIGNED NOT NULL,
    content_type        VARCHAR(50)     NOT NULL,
    duration_seconds    INT UNSIGNED    NULL,
    caption             VARCHAR(500)    NULL,
    taken_at            DATETIME        NULL,
    is_cover            BOOLEAN         NOT NULL DEFAULT FALSE,
    is_expense_receipt  BOOLEAN         NOT NULL DEFAULT FALSE,
    processing_status   ENUM('PENDING', 'PROCESSING', 'READY', 'FAILED') NOT NULL DEFAULT 'READY',
    created_at          DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    INDEX  idx_smu_schedule  (schedule_id),
    INDEX  idx_smu_orphan    (schedule_id, created_at),
    UNIQUE KEY uq_smu_r2_key (r2_key),
    INDEX  idx_smu_uploader  (uploader_id),
    INDEX  idx_smu_cover     (schedule_id, is_cover),
    INDEX  idx_smu_expense   (schedule_id, is_expense_receipt),
    CONSTRAINT fk_smu_schedule   FOREIGN KEY (schedule_id)  REFERENCES schedules(id) ON DELETE SET NULL,
    CONSTRAINT fk_smu_uploader   FOREIGN KEY (uploader_id)  REFERENCES users(id)     ON DELETE SET NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

**マイグレーション上の注意点**
- `schedules` テーブルの作成（V3.007）より後に実行される
- `users` テーブルの作成（V1.xxx）より後に実行される
- V13.014 と V13.015 は独立しているが、V13.014 を先に実行する（schedules テーブルの構造変更後にメディアテーブルを作成する順序が自然）
