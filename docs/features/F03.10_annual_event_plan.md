# F03.10: 年間行事計画（スケジュール機能拡張）

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 11
> **最終更新**: 2026-04-10
> **関連ドキュメント**: F03.1 スケジュール・出欠管理（拡張元）、F03.9 時間割管理（学期連携）

---

## 1. 概要

既存のスケジュール機能（F03.1）を拡張し、年度単位の行事カレンダー表示と前年度トレース（行事一括コピー）を提供する。学校の入学式・運動会・定期テスト等の年間行事は毎年ほぼ同じ構成で日付だけ変わるため、前年度の行事を一括コピーして日付を調整するだけで翌年度の行事計画を素早く作成できる。行事カテゴリによる色分け表示で、年間を俯瞰しやすくする。

**本ドキュメントは F03.1 への追加仕様を定義する。** 既存のスケジュール CRUD・出欠管理・繰り返し等の仕様は F03.1 を参照。

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全チーム・組織の行事カテゴリ・年間計画参照 |
| ADMIN | 行事カテゴリの CRUD。前年度トレース（コピー）の実行。年間ビューの閲覧 |
| DEPUTY_ADMIN | `MANAGE_SCHEDULES` 権限を持つ場合: ADMIN と同等 |
| MEMBER | 年間ビューの閲覧。行事カテゴリ別フィルタ |
| SUPPORTER | `min_view_role` に基づき閲覧可能なスケジュールのみ年間ビューに表示 |
| GUEST | `min_view_role = 'ANYONE'` のスケジュールのみ |

### 対象レベル
- [x] 組織 (Organization) — 組織全体の年間行事計画
- [x] チーム (Team) — チーム固有の年間行事計画
- [ ] 個人 (Personal)

## 3. DB設計

### 追加テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `schedule_event_categories` | 行事カテゴリマスター（色分け用） | なし |
| `schedule_annual_copy_logs` | 前年度トレースの実行ログ（監査用） | なし |

### 既存テーブルの変更
| テーブル名 | 変更内容 |
|-----------|---------|
| `schedules` | `event_category_id`, `source_schedule_id`, `academic_year` カラム追加 |

### テーブル定義

#### `schedule_event_categories`

行事をカテゴリ別に色分け表示するためのマスターテーブル。チームまたは組織ごとに定義。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | YES | NULL | FK → teams（チーム固有カテゴリ。NULL = 組織レベル） |
| `organization_id` | BIGINT UNSIGNED | YES | NULL | FK → organizations（組織レベルカテゴリ。NULL = チーム固有） |
| `name` | VARCHAR(100) | NO | — | カテゴリ名（例: "式典", "テスト・試験", "スポーツ行事", "文化行事", "保護者会", "休業日"） |
| `color` | VARCHAR(7) | NO | '#3B82F6' | 表示色（HEXカラーコード） |
| `icon` | VARCHAR(50) | YES | NULL | アイコン識別子（フロントエンド用。例: "ceremony", "exam", "sports"） |
| `is_day_off_category` | BOOLEAN | NO | FALSE | 休業日系カテゴリフラグ。TRUE の場合、このカテゴリのスケジュール作成時に時間割の DAY_OFF 自動連携が発動する |
| `sort_order` | TINYINT UNSIGNED | NO | 0 | 表示順 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_sec_team (team_id)
INDEX idx_sec_org (organization_id)
```

**制約・備考**
- `team_id` と `organization_id` は XOR
  ```sql
  CONSTRAINT chk_sec_scope CHECK (
    (team_id IS NOT NULL AND organization_id IS NULL) OR
    (team_id IS NULL AND organization_id IS NOT NULL)
  )
  ```
- **同一スコープ × 名前の重複防止**: MySQL の UNIQUE KEY は NULL カラムを含む場合に重複を防げないため（`NULL != NULL`）、アプリ層で `team_id/organization_id + name` の一意性を INSERT/UPDATE 前に検証する
- 1スコープあたり最大30カテゴリ（アプリ層でバリデーション）
- チームは自チームのカテゴリに加え、親組織のカテゴリも利用可能（組織カテゴリはチーム間で共有される共通ラベル）
- スケジュールに紐づくカテゴリのスコープ整合性はアプリ層で検証: チームスコープのスケジュールは自チームのカテゴリまたは親組織のカテゴリのみ利用可能

**テンプレート別カテゴリプリセット**

チーム/組織のテンプレート種別に応じて、モジュール有効化時にデフォルトカテゴリを自動作成する。ADMIN が後から自由に編集・削除・追加可能。

| テンプレート | 自動作成されるカテゴリ |
|-------------|---------------------|
| 学校 | 式典 (`#EF4444`), テスト・試験 (`#F59E0B`), スポーツ行事 (`#10B981`), 文化行事 (`#8B5CF6`), 保護者会 (`#EC4899`), 休業日 (`#6B7280`), 遠足・校外学習 (`#06B6D4`) |
| スポーツ | 大会 (`#EF4444`), 合宿 (`#F59E0B`), 練習試合 (`#10B981`), オフ日 (`#6B7280`) |
| 会社 | 全社行事 (`#3B82F6`), 研修 (`#8B5CF6`), 休業日 (`#6B7280`), 株主総会 (`#EF4444`) |
| マンション | 総会 (`#EF4444`), 点検・工事 (`#F59E0B`), 清掃 (`#10B981`), お祭り・イベント (`#8B5CF6`) |
| その他 | プリセットなし（空の状態で開始） |

> プリセットの自動作成は `ScheduleEventCategoryService.initializePresets(teamId, templateType)` で実装。ApplicationEvent（`ModuleEnabledEvent`）をリッスンし、年間行事計画の初回有効化時に実行する

#### `schedules` テーブルへの追加カラム

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `event_category_id` | BIGINT UNSIGNED | YES | NULL | FK → schedule_event_categories（ON DELETE SET NULL）。行事カテゴリ。NULL = カテゴリ未設定 |
| `source_schedule_id` | BIGINT UNSIGNED | YES | NULL | 前年度トレースでコピー元となったスケジュールの ID（追跡用。FK なし＝参照整合性は強制しない） |
| `academic_year` | SMALLINT | YES | NULL | 年度（例: 2026）。年度ビューでの絞り込みに使用。NULL = 年度に紐づかないスケジュール |

**インデックス（追加分）**
```sql
INDEX idx_sch_category (event_category_id)
INDEX idx_sch_academic_year (team_id, academic_year, start_at)        -- チーム×年度別の行事取得
INDEX idx_sch_org_academic_year (organization_id, academic_year, start_at)  -- 組織×年度別の行事取得
```

**制約・備考**
- `event_category_id` は任意。未設定のスケジュールはデフォルト色で表示
- `source_schedule_id` はソフトリファレンス（FK制約なし）。コピー元が削除されても影響しない
- `academic_year` の整合性: アプリ層で `start_at` の日付が当該 `academic_year` の範囲内（デフォルト: 4月1日〜翌年3月31日）に含まれることを検証する。範囲外の場合は 400 エラー
- 既存の `event_type` ENUM（PRACTICE / MATCH / EVENT 等）とは独立。`event_type` はシステム定義の分類、`event_category_id` はユーザー定義のラベル

### F03.1 既存 API への変更

本拡張により、F03.1 の以下の API リクエスト/レスポンスに変更が生じる:

#### `POST /api/v1/teams/{id}/schedules` / `POST /api/v1/organizations/{id}/schedules`（リクエストボディ追加フィールド）
```json
{
  "event_category_id": 1,
  "academic_year": 2026
}
```
- 両フィールドとも任意（nullable）。既存リクエストとの後方互換性あり

#### `PATCH /api/v1/teams/{id}/schedules/{scheduleId}` / `PATCH /api/v1/organizations/{id}/schedules/{scheduleId}`（更新可能フィールド追加）
- `event_category_id`: カテゴリの変更・解除（null 指定で解除）
- `academic_year`: 年度の変更・解除

#### レスポンス（一覧・詳細共通で追加されるフィールド）
```json
{
  "event_category": { "id": 1, "name": "式典", "color": "#EF4444", "icon": "ceremony" },
  "academic_year": 2026,
  "source_schedule_id": null
}
```
- `event_category`: カテゴリ未設定時は `null`
- 一覧 API（`GET /teams/{id}/schedules`）のクエリパラメータに `category_id` と `academic_year` フィルタを追加

---

#### `schedule_annual_copy_logs`

前年度トレースの実行ログ。コピー結果の確認と監査に使用。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | YES | NULL | FK → teams |
| `organization_id` | BIGINT UNSIGNED | YES | NULL | FK → organizations |
| `source_academic_year` | SMALLINT | NO | — | コピー元年度（例: 2025） |
| `target_academic_year` | SMALLINT | NO | — | コピー先年度（例: 2026） |
| `total_copied` | INT UNSIGNED | NO | 0 | コピーされた行事数 |
| `total_skipped` | INT UNSIGNED | NO | 0 | スキップされた行事数（重複・除外等） |
| `date_shift_mode` | ENUM('EXACT_DAYS', 'SAME_WEEKDAY') | NO | 'SAME_WEEKDAY' | 日付調整モード |
| `executed_by` | BIGINT UNSIGNED | YES | NULL | FK → users（SET NULL on delete） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_sacl_team (team_id, target_academic_year)
INDEX idx_sacl_org (organization_id, target_academic_year)
```

**制約・備考**
- `team_id` と `organization_id` は XOR（スコープ制約は `schedule_event_categories` と同様）
- コピー実行後のログは削除不可（監査目的）
- `date_shift_days` カラムは設けない: `SAME_WEEKDAY` モードではイベントごとにシフト日数が異なるため、ログレベルの単一値では表現できない。個別のコピー先日付は `schedules.source_schedule_id` + `start_at` の差分で追跡可能

---

### ER図（テキスト形式）
```
teams / organizations (1) ──── (N) schedule_event_categories
schedule_event_categories (1) ──── (N) schedules    ※ event_category_id（任意）
schedules (copy元) ・・・・ (copy先) schedules        ※ source_schedule_id（ソフト参照）
teams / organizations (1) ──── (N) schedule_annual_copy_logs
timetable_terms (F03.9) ・・・・ schedules            ※ 学期別行事グループ（JOIN 連携。FK なし）
```

## 5. ビジネスロジック

### 主要フロー

#### 年間行事計画の閲覧フロー
```
1. MEMBER が年間ビュー API を呼び出し（academic_year 指定）
2. バックエンドが schedules を academic_year + scope で絞り込み
3. event_category_id を JOIN してカテゴリ情報を付与
4. F03.9 の timetable_terms が存在する場合、学期情報を付与
5. 月別にグループ化してレスポンス
6. フロントエンドがカテゴリ色で色分け表示
```

#### 前年度トレースフロー
```
1. ADMIN が preview-copy を呼び出し（source_year → target_year）
2. バックエンドが source_year の全行事を取得
3. date_shift_mode に基づき各行事の候補日（start_at + end_at）を算出
4. コピー先の既存行事との重複（conflict）を検出
5. ADMIN がプレビュー結果を確認・日付を微調整・除外を選択
6. ADMIN が copy を実行
7. バックエンドが一括で schedules を INSERT（source_schedule_id と academic_year を記録）
8. schedule_annual_copy_logs にログ記録
```

### 日付調整ロジック

#### `SAME_WEEKDAY` モード（デフォルト）
同じ月の同じ「第N曜日」に配置する。

```
例: 2025-10-11 (第2土曜日) → 2026-10-10 (第2土曜日)
例: 2025-04-07 (第1月曜日) → 2026-04-06 (第1月曜日)
```

**算出ロジック:**
1. コピー元の `start_at` から「第N（何番目の）X曜日」を算出
2. コピー先年度の同じ月で「第N X曜日」に該当する日を特定
3. 該当日が存在しない場合（例: 第5月曜が月内にない）、その月の最終X曜日にフォールバック
4. `end_at` が非 NULL の場合: `start_at` のシフト日数（`suggested_start_at - source_start_at`）と同じ日数で `end_at` もシフトする（イベントの期間を保持）

#### `EXACT_DAYS` モード
単純に365日（閏年は366日）シフトする。曜日はずれる可能性がある。

```
例: 2025-04-07 (月) → 2026-04-07 (火)  ※曜日がずれる
```

`end_at` も同じ日数でシフトする。

### 重要な判定ロジック
- **年度の範囲**: デフォルトは4月1日〜翌年3月31日。将来的にチーム/組織設定で年度開始月をカスタマイズ可能にする（未解決事項参照）
- **コピー対象のフィルタ**: `deleted_at IS NULL` かつ `status != 'CANCELLED'` の行事のみ。繰り返しスケジュールの子（`parent_schedule_id IS NOT NULL`）は除外し、親のみコピーする
- **コピー時の属性引き継ぎ**: `title`, `description`, `location`, `all_day`, `event_type`, `event_category_id`, `visibility`, `min_view_role`, `min_response_role`, `attendance_required`, `comment_option` をコピー。`status` は `SCHEDULED`、`attendance_deadline` と `recurrence_rule` は NULL にリセット。`academic_year` は `target_year` を設定。`source_schedule_id` にコピー元の ID を記録
- **子テーブルのコピー方針**: コピー時に以下の子テーブルデータは**引き継がない**:
  - `event_surveys`（出欠アンケート設問）: 設問内容が年度固有の場合が多く、コピー先で改めて設定すべき
  - `schedule_attendance_reminders`（リマインダー）: 日時指定のため前年度の値は無意味
  - `schedule_cross_refs`（クロスチーム招待）: 対戦相手・交流先は年度ごとに異なるため引き継ぎ不可
- **重複判定**: 同一 scope で `title` が完全一致かつ `start_at` の日付が一致する行事がある場合に conflict として検出
- **ソース変更時の挙動（preview → copy 間の競合）**: copy API 実行時に `source_schedule_id` が存在しない場合（プレビュー後に削除された等）は、当該アイテムを自動スキップし `total_skipped` に加算する。エラーにはしない。ソースの `title` / `description` 等がプレビュー後に変更された場合は、copy 実行時点の最新値を使用する
- **`items` の部分送信**: copy API の `items` 配列は部分送信可能。プレビュー結果のうちコピーしたいアイテムのみ（`include: true`）を送信すればよい。`items` に含まれないソーススケジュールはスキップ扱いとなる。`include: false` のアイテムを含めて送信しても可（明示的な除外として処理）

### スケジュール → 時間割の自動連携

`all_day = true` かつ `event_category` が「休業日」系カテゴリのスケジュールが作成された場合、時間割モジュール（F03.9）が有効なチームでは ACTIVE な時間割に自動で DAY_OFF を登録する。詳細なフローは F03.9 §5「スケジュール ↔ 時間割の自動連携フロー」を参照。

**「休業日」系カテゴリの判定方法**: `schedule_event_categories` に `is_day_off_category BOOLEAN DEFAULT FALSE` カラムを追加。ADMIN がカテゴリ作成・編集時にこのフラグを設定する。テンプレート別プリセットでは「休業日」「オフ日」カテゴリに自動で `TRUE` を設定する。

## 6. セキュリティ考慮事項

- **認可チェック**: 年間ビューは既存のスケジュール `min_view_role` フィルタを継承。SUPPORTER / GUEST には閲覧権限のある行事のみ表示
- **一括コピーの制限**: 1回のコピーで最大200件まで（アプリ層でバリデーション）。大規模組織での意図しない大量作成を防止
- **レートリミット**: copy API は `Bucket4j` で 1時間に5回の制限（大量実行防止）
- **カテゴリスコープ検証**: スケジュールの `event_category_id` が、そのスケジュールの所属チーム/組織のカテゴリまたは親組織のカテゴリであることをアプリ層で検証
