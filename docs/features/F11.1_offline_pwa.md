## 1. 概要

通信環境が不安定な場面（スポーツ会場・アウトドアイベント・地下施設・移動中の電車内）でも主要機能を利用可能にする PWA 対応。Service Worker による静的アセットと API レスポンスのキャッシュ、IndexedDB を使ったオフライン下書き・未送信データのキューイング、復帰検知による自動同期を提供する。

**ユースケース**:
- スポーツチーム: 試合会場で活動記録を作成、帰路の移動中に自動同期
- アウトドアイベント: 山間部で出欠回答を入力、電波復帰時に自動送信
- 地下施設: 地下の会議室でチャットメッセージを入力、エレベーター内で同期
- 災害時: 安否確認の回答をオフラインで作成、通信復旧後に即時送信

本機能は**フロントエンド中心**の設計であり、バックエンドの DB テーブルは最小限（`offline_sync_conflicts` のみ新規追加。`push_subscriptions` は F04.3 で作成済み）とする。

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `offline_sync_conflicts` | オフライン同期時のコンフリクト記録（ユーザーの手動解決待ち） | なし（解決後90日で物理削除） |

**既存テーブル（他フェーズで作成済み）の参照**:
- `push_subscriptions`（Phase 4、F04.3）: PWA Push 通知の subscription 情報。本機能では Service Worker のプッシュイベントハンドリングで参照
- `users`（Phase 1）: コンフリクト記録のユーザー紐付け

### テーブル定義

#### `offline_sync_conflicts`

オフライン中にユーザーが作成・更新したデータを同期する際、サーバー側で楽観的ロック（`version` 不一致）により 409 Conflict が返された場合のコンフリクト記録。ユーザーが手動でマージ/破棄を判断するまで保持する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE）。コンフリクトを起こしたユーザー |
| `resource_type` | VARCHAR(50) | NO | — | 対象リソース種別（`ACTIVITY_RECORD` / `CHAT_MESSAGE` / `ATTENDANCE_RESPONSE` 等） |
| `resource_id` | BIGINT UNSIGNED | NO | — | 対象リソースの ID |
| `client_data` | JSON | NO | — | クライアント側のデータ（オフラインで作成した内容） |
| `server_data` | JSON | NO | — | サーバー側の最新データ（コンフリクト発生時点） |
| `client_version` | BIGINT | NO | — | クライアントが保持していた version 値 |
| `server_version` | BIGINT | NO | — | サーバー側の現在の version 値 |
| `resolution` | VARCHAR(20) | YES | NULL | 解決方法（`CLIENT_WIN` / `SERVER_WIN` / `MANUAL_MERGE` / `DISCARDED`）。未解決時は NULL |
| `resolved_at` | DATETIME | YES | NULL | 解決日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | コンフリクト発生日時 |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | 更新日時 |

**インデックス**
```sql
INDEX idx_osc_user_resolution (user_id, resolution, created_at DESC)  -- ユーザーの未解決コンフリクト一覧
INDEX idx_osc_resource (resource_type, resource_id)                    -- 特定リソースのコンフリクト検索
INDEX idx_osc_resolved_at (resolved_at)                                -- 解決済み90日バッチ削除用
```

**制約・備考**
- 論理削除なし。`resolved_at` が 90日以上前のレコードを日次バッチで物理削除
- `client_data` / `server_data` は JSON カラムで柔軟にリソース種別ごとのデータを保持。最大サイズは 64KB を想定
- `resolution = NULL` のレコードがユーザーの未解決コンフリクト。フロントエンドの同期結果画面で表示する
- 同一 `resource_type` + `resource_id` + `user_id` で未解決のコンフリクトが既に存在する場合は、新しいコンフリクトで上書き（UPSERT）する

### ER図（テキスト形式）
```
users (1) ──── (N) offline_sync_conflicts [user_id]
users (1) ──── (N) push_subscriptions [user_id]  ※ F04.3 で作成済み

※ offline_sync_conflicts.resource_type + resource_id はポリモーフィック参照
  → activity_results, chat_messages, schedule_attendances 等
```

#### `PATCH /api/v1/sync/conflicts/{id}/resolve`

コンフリクトの解決方法を指定する。

**リクエストボディ**
```json
{
  "resolution": "CLIENT_WIN",
  "merged_data": null
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `resolution` | String | YES | 解決方法: `CLIENT_WIN`（クライアントデータで上書き）/ `SERVER_WIN`（サーバーデータを採用）/ `MANUAL_MERGE`（手動マージ） |
| `merged_data` | JSON | `MANUAL_MERGE` 時のみ必須 | 手動マージしたデータ。`CLIENT_WIN` / `SERVER_WIN` 時は null |

**レスポンス（200 OK）**
```json
{
  "data": {
    "id": 42,
    "resolution": "CLIENT_WIN",
    "resolved_at": "2026-03-15T15:00:00"
  }
}
```

- `CLIENT_WIN` の場合: サーバーの対象リソースをクライアントデータで強制上書き（version をインクリメント）
- `SERVER_WIN` の場合: コンフリクトレコードの resolution を更新するのみ（サーバーデータは変更なし）
- `MANUAL_MERGE` の場合: `merged_data` でサーバーの対象リソースを更新

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `MANUAL_MERGE` 指定時に `merged_data` が未指定 |
| 403 | 他ユーザーのコンフリクトを操作しようとした |
| 404 | コンフリクトが存在しない |
| 409 | 解決中にさらに新しいバージョンが作られた（再度コンフリクト） |

#### 5.2 Service Worker 戦略（Workbox ベース）

Service Worker は Workbox ライブラリを使用し、`@vite-pwa/nuxt` が自動生成する。カスタムロジック（オフラインキュー・Push 通知ハンドリング）は `public/sw-custom.js` に記述し、Workbox が inject する。

**Service Worker ライフサイクル**:
```
1. install イベント: 静的アセットをプリキャッシュ（precache manifest）
2. activate イベント: 古いキャッシュの削除。clients.claim() で即時制御取得
3. fetch イベント: キャッシュ戦略に従ってリクエストをインターセプト
4. push イベント: バックグラウンド Push 通知の表示（F04.3 連携）
5. sync イベント: Background Sync API でオフラインキューの送信
6. notificationclick イベント: 通知タップ時のディープリンク遷移
```

#### 5.3 キャッシュ戦略

| 対象 | 戦略 | TTL | 最大エントリ | 説明 |
|------|------|-----|------------|------|
| 静的アセット（JS/CSS/フォント） | Cache First（プリキャッシュ） | SW 更新時に破棄 | — | ビルドハッシュ付きファイル。SW インストール時に一括キャッシュ |
| API データ（GET リクエスト） | Stale-While-Revalidate | 24時間 | 200件 | キャッシュを即表示 + バックグラウンドで最新データを取得して更新 |
| 画像（S3 / CDN） | Cache First with expiration | 7日 | 300件 | 一度取得した画像はキャッシュ優先。期限切れ後にネットワーク取得 |
| HTML（ナビゲーション） | Network First | 1時間 | 50件 | オンライン時はネットワーク優先。オフライン時はキャッシュフォールバック |
| 認証 API（POST /auth/*） | Network Only | — | — | キャッシュしない。認証トークンの鮮度を保証 |

**Workbox ランタイムキャッシュ設定**:
```ts
// nuxt.config.ts > pwa.workbox.runtimeCaching
runtimeCaching: [
  {
    // API データ: Stale-While-Revalidate
    urlPattern: /^https?:\/\/.*\/api\/v1\/.*$/,
    handler: 'StaleWhileRevalidate',
    method: 'GET',
    options: {
      cacheName: 'api-cache',
      expiration: { maxEntries: 200, maxAgeSeconds: 86400 },
      cacheableResponse: { statuses: [0, 200] },
    },
  },
  {
    // 画像: Cache First with expiration
    urlPattern: /\.(?:png|jpg|jpeg|svg|gif|webp)$/,
    handler: 'CacheFirst',
    options: {
      cacheName: 'image-cache',
      expiration: { maxEntries: 300, maxAgeSeconds: 604800 },
      cacheableResponse: { statuses: [0, 200] },
    },
  },
  {
    // フォント: Cache First（長期キャッシュ）
    urlPattern: /\.(?:woff|woff2|ttf|eot)$/,
    handler: 'CacheFirst',
    options: {
      cacheName: 'font-cache',
      expiration: { maxEntries: 20, maxAgeSeconds: 2592000 },
    },
  },
],
```

**キャッシュサイズ制限**: 全キャッシュの合計上限は **50MB**。超過時は LRU（Least Recently Used）で古いエントリから自動削除する。Workbox の `expiration` プラグインが自動管理する。

#### 5.4 IndexedDB スキーマ（Dexie.js）

オフラインキューとキャッシュメタデータを IndexedDB で管理する。Dexie.js をラッパーとして使用し、型安全な操作を提供する。

```ts
// composables/useOfflineDb.ts
import Dexie, { type Table } from 'dexie'

export interface OfflineQueueItem {
  id?: number                    // auto-increment PK
  clientId: string               // ユニークID（timestamp + random）
  method: 'POST' | 'PATCH' | 'PUT' | 'DELETE'
  path: string                   // API パス（例: /api/v1/teams/1/activity-records）
  body: Record<string, unknown>  // リクエストボディ
  version: number | null         // 楽観的ロック用 version（新規作成時は null）
  status: 'PENDING' | 'SYNCING' | 'SUCCESS' | 'FAILED' | 'CONFLICT'
  retryCount: number             // リトライ回数（最大3回）
  errorMessage: string | null    // 失敗時のエラーメッセージ
  createdAt: string              // ISO 8601 文字列（JST）
  syncedAt: string | null        // 同期完了日時
}

export interface CachedApiResponse {
  id?: number
  url: string                    // キャッシュしたAPIのURL
  data: Record<string, unknown>  // レスポンスデータ
  cachedAt: string               // キャッシュ日時
  expiresAt: string              // 有効期限
}

export interface OfflineDraft {
  id?: number
  draftType: 'ACTIVITY_RECORD' | 'CHAT_MESSAGE' | 'ATTENDANCE_RESPONSE'
  scopeType: 'TEAM' | 'ORGANIZATION' | 'PERSONAL'
  scopeId: number
  data: Record<string, unknown>  // 下書きデータ
  createdAt: string
  updatedAt: string
}

class MannschaftOfflineDb extends Dexie {
  offlineQueue!: Table<OfflineQueueItem>
  cachedResponses!: Table<CachedApiResponse>
  offlineDrafts!: Table<OfflineDraft>

  constructor() {
    super('mannschaft-offline')
    this.version(1).stores({
      offlineQueue: '++id, clientId, status, createdAt',
      cachedResponses: '++id, url, expiresAt',
      offlineDrafts: '++id, draftType, [scopeType+scopeId], updatedAt',
    })
  }
}

export const offlineDb = new MannschaftOfflineDb()
```

**IndexedDB テーブル一覧**:

| テーブル | 用途 | 最大件数 |
|---------|------|---------|
| `offlineQueue` | 未送信リクエストのキュー（FIFO） | 500件 |
| `cachedResponses` | API レスポンスのローカルキャッシュ（SWR 補助） | 1,000件 |
| `offlineDrafts` | 作成途中の下書き（未送信とは別管理） | 100件 |

#### 5.5 オフラインキュー管理

```ts
// composables/useOfflineSync.ts
export const useOfflineSync = () => {
  const isOnline = useOnline() // @vueuse/core

  // オフラインキューにアイテムを追加
  const enqueue = async (item: Omit<OfflineQueueItem, 'id' | 'status' | 'retryCount' | 'errorMessage' | 'syncedAt'>) => {
    await offlineDb.offlineQueue.add({
      ...item,
      status: 'PENDING',
      retryCount: 0,
      errorMessage: null,
      syncedAt: null,
    })
  }

  // オンライン復帰時の同期処理
  const syncAll = async () => {
    const pendingItems = await offlineDb.offlineQueue
      .where('status')
      .anyOf(['PENDING', 'FAILED'])
      .sortBy('createdAt') // FIFO: 作成順に送信

    if (pendingItems.length === 0) return

    // 50件ずつバッチ送信
    const batches = chunk(pendingItems, 50)
    for (const batch of batches) {
      await syncBatch(batch)
    }
  }

  return { enqueue, syncAll, isOnline }
}
```

#### 5.6 オフライン同期フロー

```
1. ユーザーがオフライン状態で操作（活動記録作成・チャット送信・出欠回答）
2. フロントエンドがネットワーク状態を検知（navigator.onLine + @vueuse/core useOnline）
3. オフライン時: IndexedDB の offlineQueue テーブルにリクエストを保存
   - clientId を生成（タイムスタンプ + ランダム文字列）
   - status = 'PENDING' で保存
   - UI にはオフラインバッジ（雲マーク + 斜線）を表示
4. ユーザーに「オフラインで保存しました。オンラインに戻ると自動送信されます」トースト通知
5. オンライン復帰を検知（navigator.onLine イベント + Fetch ベースのヘルスチェック）
6. Background Sync API が利用可能な場合:
   a. Service Worker の sync イベントで自動実行
   b. タグ: 'mannschaft-offline-sync'
7. Background Sync API が利用不可の場合:
   a. online イベントハンドラーで composable の syncAll() を呼び出し
8. POST /api/v1/sync に未送信アイテムを一括送信（FIFO 順、50件ずつ）
9. 各アイテムの結果を処理:
   a. SUCCESS: offlineQueue から削除。UI に成功トースト
   b. CONFLICT: offlineQueue.status = 'CONFLICT' に更新。コンフリクト解決画面を表示
   c. FAILED (400/403/404): 3回リトライ後に status = 'FAILED'。ユーザーに通知
10. 全アイテムの処理結果サマリーをトースト通知
    「3件中2件を同期しました。1件のコンフリクトがあります」
```

#### 5.7 コンフリクト解決フロー

```
1. POST /api/v1/sync のレスポンスで status = 'CONFLICT' のアイテムを検知
2. バックエンドが offline_sync_conflicts テーブルにレコードを作成
   - client_data: クライアントが送信したデータ
   - server_data: サーバー側の最新データ（409 Conflict 発生時に取得）
3. フロントエンドがコンフリクト通知を表示（トースト + ベルアイコンバッジ）
4. ユーザーがコンフリクト解決画面を開く（GET /api/v1/sync/conflicts/{id}）
5. 画面に以下を表示:
   a. 左パネル: クライアントデータ（オフラインで作成した内容）
   b. 右パネル: サーバーデータ（他ユーザーが更新した内容）
   c. 差分ハイライト（変更箇所を色分け）
6. ユーザーが解決方法を選択:
   a.「自分のデータを採用」→ PATCH /sync/conflicts/{id}/resolve { resolution: 'CLIENT_WIN' }
   b.「サーバーのデータを採用」→ PATCH /sync/conflicts/{id}/resolve { resolution: 'SERVER_WIN' }
   c.「手動でマージ」→ マージエディタで編集 → PATCH /sync/conflicts/{id}/resolve { resolution: 'MANUAL_MERGE', merged_data: {...} }
   d.「破棄する」→ DELETE /sync/conflicts/{id}
7. 解決後、offlineQueue から該当アイテムを削除
```

**コンフリクト解決のデフォルト戦略**:
- **チャットメッセージ（新規作成）**: コンフリクトは発生しない（POST は冪等でないため、常に新規レコード作成）
- **出欠回答（更新）**: 楽観的ロック（`version`）で 409 Conflict。ユーザーに手動マージを促す
- **活動記録（更新）**: 楽観的ロック（`version`）で 409 Conflict。ユーザーに手動マージを促す
- **活動記録（新規作成）**: コンフリクトは発生しない（POST は常に新規レコード作成）

### バックエンド設計

#### 5.11 同期エンドポイントのビジネスロジック

```
1. POST /api/v1/sync を受信
2. items 配列を FIFO 順（createdAt 昇順）で逐次処理
3. 各アイテムについて:
   a. path + method を解析し、対応する Service メソッドを呼び出し
   b. version が指定されている場合（更新系）:
      - サーバー側の version と比較
      - 一致 → 通常の更新処理を実行
      - 不一致 → 409 Conflict
        i. offline_sync_conflicts テーブルにレコードを INSERT
        ii. client_data にクライアント送信データを保存
        iii. server_data にサーバー側の最新データを保存
        iv. レスポンスの results に status = 'CONFLICT' + conflict_id を返却
   c. version が null の場合（新規作成）:
      - 通常の POST 処理を実行
      - 成功 → results に status = 'SUCCESS' + resource_id を返却
   d. バリデーションエラー等 → results に status = 'FAILED' + message を返却
4. 全アイテムの処理結果を集約して 200 OK で返却
```

#### 5.12 コンフリクト解決のバックエンドロジック

```
1. PATCH /api/v1/sync/conflicts/{id}/resolve を受信
2. offline_sync_conflicts から対象レコードを取得
3. user_id = currentUser.id を検証（他ユーザーのコンフリクトは操作不可）
4. resolution に応じた処理:
   a. CLIENT_WIN:
      - resource_type に対応する Service の強制更新メソッドを呼び出し
      - version を無視してクライアントデータで上書き
      - offline_sync_conflicts の resolution = 'CLIENT_WIN', resolved_at を更新
   b. SERVER_WIN:
      - サーバーデータは変更なし
      - offline_sync_conflicts の resolution = 'SERVER_WIN', resolved_at を更新
   c. MANUAL_MERGE:
      - merged_data のバリデーション
      - resource_type に対応する Service の強制更新メソッドを呼び出し
      - offline_sync_conflicts の resolution = 'MANUAL_MERGE', resolved_at を更新
5. 更新中に再度 version 不一致が発生した場合は 409 Conflict を返却
```

#### 5.13 コンフリクト記録の清掃バッチ

```
1. 日次バッチ（Spring @Scheduled, 毎日 AM 4:00 JST）
2. DELETE FROM offline_sync_conflicts
   WHERE resolved_at IS NOT NULL
     AND resolved_at < NOW() - INTERVAL 90 DAY
3. 削除件数をログ出力
```

## 6. セキュリティ考慮事項

- **認可チェック**: `POST /api/v1/sync` の各アイテムは、対応する元 API の認可ロジックを必ず経由する。同期 API だからといって権限チェックを省略しない
- **IndexedDB のデータ保護**: IndexedDB に保存するデータは平文。端末の紛失・盗難リスクを考慮し、機密性の高いデータ（パスワード、トークン等）は IndexedDB に保存しない。JWT は従来通り localStorage に保存（`FRONTEND_CODING_CONVENTION.md` 7 準拠）
- **Service Worker のスコープ制限**: Service Worker のスコープはアプリのルート（`/`）に限定。外部ドメインのリクエストはインターセプトしない
- **キャッシュ内容の機密性**: API キャッシュには認証済みユーザーのデータが含まれる。ログアウト時に全キャッシュ（Cache Storage + IndexedDB）をクリアする
- **同期 API のレートリミット**: `POST /api/v1/sync` は Bucket4j で 1分間に10回（悪意ある大量同期リクエストを防止）
- **コンフリクト API のレートリミット**: コンフリクト関連 API は1分間に60回
- **items 数の上限**: 1回の同期リクエストで送信可能な items 数は最大50件。超過時は 413 を返却
- **VAPID 鍵管理**: Push 通知の VAPID 秘密鍵は環境変数（`MANNSCHAFT_VAPID_PRIVATE_KEY`）で管理（F04.3 と共通）
- **XSS 対策**: offlineQueue に保存するユーザー入力データは、サーバー送信時にバックエンドでサニタイズ。フロントエンドでのキャッシュ表示時は `v-text` を使用

**マイグレーション上の注意点**

**V11.060__create_offline_sync_conflicts_table.sql**:
```sql
-- オフライン同期時のコンフリクト記録を管理するテーブル
CREATE TABLE offline_sync_conflicts (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    user_id BIGINT UNSIGNED NOT NULL,
    resource_type VARCHAR(50) NOT NULL,
    resource_id BIGINT UNSIGNED NOT NULL,
    client_data JSON NOT NULL,
    server_data JSON NOT NULL,
    client_version BIGINT NOT NULL,
    server_version BIGINT NOT NULL,
    resolution VARCHAR(20) DEFAULT NULL,
    resolved_at DATETIME DEFAULT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    PRIMARY KEY (id),

    CONSTRAINT fk_offline_sync_conflicts_user
        FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,

    INDEX idx_osc_user_resolution (user_id, resolution, created_at DESC),
    INDEX idx_osc_resource (resource_type, resource_id),
    INDEX idx_osc_resolved_at (resolved_at),

    CONSTRAINT chk_osc_resolution
        CHECK (resolution IN ('CLIENT_WIN', 'SERVER_WIN', 'MANUAL_MERGE', 'DISCARDED') OR resolution IS NULL)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

全件解決済み。

- [x] ~~キャッシュサイズの上限をいくつにするか~~ → **50MB**。Workbox の expiration プラグインで LRU 自動削除。モバイルブラウザのストレージ制限（概ね端末容量の50%が上限）を考慮し、余裕を持った設定
- [x] ~~Background Sync API 非対応ブラウザ（Safari 16 以前）でのフォールバック~~ → `navigator.onLine` の `online` イベントハンドラーで `syncAll()` を呼び出すフォールバックを実装。Background Sync API 対応ブラウザでは Service Worker の `sync` イベントを優先使用
- [x] ~~オフラインキューの送信順序（FIFO vs 優先度ベース）~~ → **FIFO（createdAt 昇順）**を採用。理由: 操作の因果関係を保持するため（例: 活動記録作成 → その記録へのコメント）。優先度ベースは因果関係が崩れるリスクがある
- [x] ~~コンフリクト解決のデフォルト戦略~~ → 新規作成（POST）はコンフリクトなし。更新（PATCH/PUT）は楽観的ロック（`version`）で 409 Conflict → ユーザーに手動解決を促す。Last Write Wins（LWW）は重要データの消失リスクがあるため採用しない
- [x] ~~IndexedDB のデータ暗号化要否~~ → **暗号化しない**。理由: (1) IndexedDB は同一オリジンのみアクセス可能 (2) 機密データ（パスワード・トークン）は IndexedDB に保存しない (3) 暗号化のオーバーヘッドがオフライン UX を損なう。端末紛失リスクは OS レベルの暗号化（BitLocker / FileVault）で対応
- [x] ~~Service Worker の更新戦略~~ → `registerType: 'autoUpdate'` を採用。新しい SW が利用可能になると自動的にアクティベートする。`skipWaiting()` + `clients.claim()` で即時適用。ユーザーへのリロード促進はトースト通知で行う
- [x] ~~オフラインキューの最大件数~~ → **500件**。超過時は最も古い FAILED アイテムから削除。PENDING アイテムは削除しない（ユーザーデータの消失防止）
- [x] ~~Push 通知と Service Worker の統合方法~~ → F04.3 で定義済みの `push_subscriptions` テーブルと VAPID 鍵管理をそのまま使用。Service Worker 内の push イベントハンドラーは `public/sw-custom.js` に記述し、Workbox と共存
