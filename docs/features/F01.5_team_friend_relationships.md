## 1. 概要

チーム同士がチーム単位で相互にフォローし合うことで成立する「フレンドチーム」関係と、その関係から派生するコンテンツ・通知の二段階配信モデルを提供する中核機能。既存の `follows` テーブル（F04.4）に `TEAM` タイプの follower/followed を追加し、相互フォロー成立時に自動的にフレンドチームとして認定する。チーム内でのノイズ制御のため、フレンドチームから発せられた通知やタイムライン投稿は **直接一般メンバーには届かず**、受信チームの ADMIN / DEPUTY_ADMIN だけが閲覧できる「管理者フィード」に集約される。管理者が「転送」判断を行った場合のみ、自チームの MEMBER や SUPPORTER へリブロードキャストされる（回覧板・掲示板モデル）。さらにフレンドチームを目的別に整理する「フレンドフォルダ」を各チーム視点で管理でき、非対称なラベル付け・通知配信対象の絞り込みが可能。

### 主なユースケース

- 姉妹チーム・系列校・地域同業ネットワーク同士が公式に「提携」を可視化する
- 同一エリアのスポーツチーム同士が練習試合・合同イベントを効率的に呼びかける（F08.1 マッチングと異なり、継続的な関係前提）
- 傘下クラブや親子団体がゆるやかな情報交換ネットワークを構築する
- チーム ADMIN が大量のフレンド通知を「掲示する価値があるか」事前審査し、下位メンバーへの情報過多を防ぐ
- フレンドチーム同士で BLOG 記事を優先公開範囲として指定する（オプション）

### 既存機能との関係

| 機能 | 関係性 |
|------|-------|
| F01.2 組織・チーム・メンバー・ロール管理 | フレンドチーム関係は **チーム同士** の関係であり、`user_roles` とは独立。ただし操作可否は ADMIN / DEPUTY_ADMIN のロールと `MANAGE_FRIEND_TEAMS` 権限に準拠（`role_permissions.is_default` の扱いは F01.2 の標準に従う） |
| F04.1 タイムライン | `scope_type` に `FRIEND_TEAM`（管理者フィード専用）および `FRIEND_FORWARD`（転送投稿）を追加する |
| F04.3 プッシュ通知 | `scope_type` に `FRIEND_TEAM` / `FRIEND_FOLDER` を追加。通知チャネル選択・受信設定の仕組みは既存のままを流用 |
| F04.4 ソーシャルプロフィール・フォロー | `follows` テーブルを拡張（`follower_type` / `followed_type` に `TEAM` を正式追加）。相互フォロー成立の検知ロジックはこちらで新設 |
| F06.1 CMS・ブログ | 記事 `visibility` に `FRIEND_TEAMS_ONLY` を追加する（オプション・Phase 4 でのみ定義、実稼働は Phase 6 以降） |
| F08.1 マッチング | 将来拡張として「フレンドチーム優先マッチング」オプションを検討（本フェーズでは対象外） |

> **段階リリース**: 本機能は F04.4 パブリックタイムライン解放と同一タイミングでリリース可能。`platform_settings.friend_team_enabled = false` の間、フレンド関連の全 API は 403 を返却し、DB のみ先行整備する構成を取る。

## 2. 用語定義

| 用語 | 定義 |
|------|-----|
| **フレンドチーム** | チーム A とチーム B が相互に `follows` 関係を結んだ状態。`team_friends` テーブルの1レコードに対応 |
| **相互フォロー** | `follows` に `(TEAM, A, TEAM, B)` と `(TEAM, B, TEAM, A)` の2行が同時に存在する状態 |
| **フレンドフォルダ** | 自チーム視点でフレンドチーム群を分類するラベル。例: 「系列校」「練習試合候補」「姉妹団体」。**非対称**（チーム A のフォルダにチーム B をどう入れるかは、チーム B のフォルダ構成と無関係） |
| **管理者フィード** | フレンドチームから配信された通知・投稿を受信する、ADMIN / DEPUTY_ADMIN 限定のタイムラインバケット。`timeline_posts.scope_type = 'FRIEND_TEAM'` に相当 |
| **二段階配信** | フレンドチーム発のコンテンツが (1) 受信チーム管理者への通知配信 → (2) 管理者判断による自チーム内リブロードキャスト、の2ステップで届く仕組み |
| **転送（Forward）** | 管理者が管理者フィードの投稿を自チーム内タイムラインに再配信する操作。転送元フレンドチーム名が出典として明示される |
| **転送対象** | 転送時に選べる配信範囲。Phase 1 は `MEMBER` のみ。`MEMBER_AND_SUPPORTER` は Phase 3 で申請・審査メカニズム付きで実装 |
| **フレンド成立通知** | 相互フォローが成立した瞬間に両チームの ADMIN / DEPUTY_ADMIN に送られるシステム通知 |
| **フォロー解除の連鎖** | 片方がフォロー解除した瞬間に相互状態が崩れるため、`team_friends` レコードを自動削除し、フレンド解除通知を両チーム管理者に送る |

## 2.5 詳細ユースケース

### ユースケース1: 地域スポーツチーム間の練習試合募集

**登場人物**: チーム A（自チーム／大阪市東成区のサッカーチーム）、チーム B（京都市東山区のサッカーチーム）

```
[1] チーム A の ADMIN がチーム B のプロフィールを閲覧
       ↓
[2] POST /api/v1/teams/{A}/friends/follow { target_team_id: B, comment: "よろしく" }
       → 片方向フォロー成立（is_mutual = false）
       → B チーム管理者へ FRIEND_FOLLOW_REQUEST 通知
       ↓
[3] B チームの ADMIN が「フォロー返し」を実行
       POST /api/v1/teams/{B}/friends/follow { target_team_id: A }
       → 相互フォロー成立（is_mutual = true）
       → team_friends に INSERT（team_a_id = min(A,B), team_b_id = max(A,B)）
       → 両チーム管理者へ FRIEND_ESTABLISHED 通知
       ↓
[4] チーム A が練習試合募集を投稿（通常の TEAM スコープ投稿 + share_with_friends = TRUE マーク）
       → timeline_posts INSERT
       → フレンド配信: チーム B の管理者フィードにクエリ時結合で表示
       → B の ADMIN へ FRIEND_FEED_POST_RECEIVED 通知（集約配信）
       ↓
[5] B の ADMIN が内容を確認 → 「転送」判断
       POST /api/v1/teams/{B}/friend-feed/{postId}/forward { target: "MEMBER", comment: "Aチームから練習試合のお誘い" }
       → B チーム内に scope_type='FRIEND_FORWARD' の投稿が生成される
       → B の MEMBER へ FRIEND_FORWARD_RECEIVED 通知
```

### ユースケース2: 系列校同士の情報共有

**登場人物**: 本校・分校の2つのチームが「系列校」フォルダで相互分類

- 本校 ADMIN が `/api/v1/teams/{本校}/friend-folders` で「系列校」フォルダを作成
- フレンド成立済みの分校を `POST /api/v1/teams/{本校}/friend-folders/{folderId}/members` でフォルダに追加
- 定期的な学校行事案内を `POST /api/v1/teams/{本校}/friend-notifications/send` で `target_type: FOLDER` でフォルダ一斉通知
- 分校 ADMIN は管理者フィードで確認し、自校の保護者（MEMBER）へ転送

### ユースケース3: 地域ネットワーク（複数チーム）の合同イベント告知

**登場人物**: 地域連合（5チーム）の合同イベント

- 主催チームが自チームの「地域連合」フォルダに4チームを登録
- `POST /api/v1/teams/{主催}/friend-notifications/send` で `target_type: FOLDER` を指定
- 1コールで4チームの管理者全員に通知が配信される（非同期バッチ・202 Accepted）
- 各受信チーム管理者が自チーム状況に応じて転送判断

## 3. スコープ・権限

### 対象ロール

| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全フレンド関係の参照・強制解除（モデレーション）。フレンド関連テーブルの監査閲覧 |
| ADMIN | 自チームとしての他チームフォロー/解除、フレンドフォルダ CRUD、管理者フィード閲覧、転送実行/取消、フレンド向け通知送信 |
| DEPUTY_ADMIN | `MANAGE_FRIEND_TEAMS` 権限を持つ場合: ADMIN と同等（フレンド関係の追加/解除・フォルダ管理・転送・通知送信） |
| MEMBER | 自チームのフレンドチーム一覧の閲覧のみ（公開設定に準拠）。転送されたフレンド発投稿は通常タイムライン上で閲覧可 |
| SUPPORTER | 自チームのフレンドチーム一覧の閲覧のみ（`is_public = TRUE` のフレンドのみ）。Phase 3 以降で転送対象が `MEMBER_AND_SUPPORTER` の場合のみ転送投稿を閲覧可 |
| GUEST | 対象外 |

> **新規パーミッション**: `MANAGE_FRIEND_TEAMS`（TEAM scope）を `permissions` テーブルに追加する。F01.2 の権限管理メカニズムに従い、`role_permissions` には以下の通り登録する:
>
> | ロール | `is_default` | 意味 |
> |-------|--------------|------|
> | ADMIN | `TRUE` | デフォルト付与。チーム作成時に自動で ADMIN へ付与される |
> | DEPUTY_ADMIN | `FALSE` | **天井のみ**。エントリは存在するが実付与はされない。ADMIN が `permission_groups` 経由で明示的に付与することで有効化される |
> | MEMBER / SUPPORTER | エントリなし | 天井が存在しないため、どう頑張っても絶対に付与不可（安全側設計） |
>
> 権限付与時には `audit_logs` に `PERMISSION_GRANTED_MANAGE_FRIEND_TEAMS`、剥奪時には `PERMISSION_REVOKED_MANAGE_FRIEND_TEAMS` を必ず記録する。

### 対象レベル

- [x] チーム (Team) — フレンドチーム関係はチーム間の関係であるため、唯一のスコープ
- [ ] 組織 (Organization) — 本フェーズでは対象外（組織間連携は将来拡張）
- [ ] 個人 (Personal) — 対象外

### 通知対象範囲ルール

**【確定仕様】二段階配信により、フレンドチーム発のコンテンツ・通知は以下のルールに従う:**

| 段階 | 受信者 | 通知チャネル |
|------|------|--------------|
| 第1段階: 受信チーム管理者 | ADMIN / `MANAGE_FRIEND_TEAMS` 権限を持つ DEPUTY_ADMIN | アプリ内通知 + オプションで Push |
| 第2段階: 転送判断後 | 管理者が選択した範囲（Phase 1 = MEMBER のみ / Phase 3 以降 = MEMBER + SUPPORTER） | 既存タイムライン通知フローに準拠 |

**【Phase 1 の重要な制約】** GDPR / 個人情報保護の観点から、SUPPORTER（保護者・サポーター）への転送配信は Phase 1 では**一切実施しない**。Phase 1 の `target` パラメータは `MEMBER` 固定とし、API 層で `MEMBER_AND_SUPPORTER` を指定した場合は 400 Bad Request を返す。Phase 3 では:

- フレンドチーム側の明示的な同意（「うちの投稿がそちらの SUPPORTER にも見えてよい」同意）
- 申請・審査メカニズム（`friend_content_forward_approval_requests` 新設予定）
- SYSTEM_ADMIN のレビュー導線

これらを整備してから `MEMBER_AND_SUPPORTER` を解禁する。

### フレンド関係の公開範囲

- フレンド関係のデフォルトは **非公開**（`team_friends.is_public = FALSE`）
- 公開設定は ADMIN のみ変更可能（両チーム双方の ADMIN が承認した場合のみ `TRUE` に切り替わる）
- 非公開フレンドはパブリック API（ログイン不要のプロフィール閲覧等）から除外される
- 公開フレンドのみがチームプロフィールページや検索結果に表示される

## 4. DB設計

### テーブル一覧

| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `team_friends` | フレンドチーム関係キャッシュ（相互フォロー成立時に生成。`follows` から自動導出） | なし（物理削除） |
| `team_friend_folders` | 自チーム視点のフレンドフォルダ | あり |
| `team_friend_folder_members` | フレンドフォルダとフレンドチームの多対多中間テーブル | なし |
| `friend_content_forwards` | フレンドコンテンツ転送履歴（冪等性・監査ログ） | `is_revoked` フラグによる論理取消（物理削除は CASCADE 時のみ） |

### 既存テーブルへの波及変更

| テーブル | 変更内容 |
|---------|--------|
| `follows` | `follower_type` / `followed_type` が受理する値に `TEAM` を正式追加（ドキュメント上の拡張。DB は VARCHAR のため DDL 変更なし） |
| `timeline_posts` | `scope_type` に `FRIEND_TEAM`（管理者フィード専用）・`FRIEND_FORWARD`（転送投稿）を受理させる（アプリ層 enum 拡張。VARCHAR のため DDL 変更なし） |
| `notifications` | `scope_type` に `FRIEND_TEAM` / `FRIEND_FOLDER` を追加（同上） |
| `blog_posts` | `visibility` に `FRIEND_TEAMS_ONLY` を追加（オプション・Phase 6 以降） |
| `permissions` | シードデータに `MANAGE_FRIEND_TEAMS` を追加 |
| `role_permissions` | ADMIN = `is_default TRUE` / DEPUTY_ADMIN = `is_default FALSE` を追加 |

### テーブル定義

#### `team_friends`

相互フォロー成立時に生成されるキャッシュテーブル。`follows` テーブルの `(TEAM, A) ↔ (TEAM, B)` 双方向エントリから自動的に導出される。検索・一覧表示の高速化のために正規化せずに保持する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_a_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。関係の片側（常に `team_a_id < team_b_id` となるよう正規化） |
| `team_b_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。関係のもう片側 |
| `established_at` | DATETIME | NO | CURRENT_TIMESTAMP | 相互フォロー成立日時（フレンドシップ成立日） |
| `a_follow_id` | BIGINT UNSIGNED | NO | — | FK → follows（A→B のフォローレコード ID） |
| `b_follow_id` | BIGINT UNSIGNED | NO | — | FK → follows（B→A のフォローレコード ID） |
| `is_public` | BOOLEAN | NO | FALSE | フレンド関係を公開するか。`TRUE` の場合のみパブリック API から閲覧可能。両チームの ADMIN 承認により切り替え |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス・制約**
```sql
UNIQUE KEY uq_tf_pair (team_a_id, team_b_id)     -- 同一ペアの重複登録防止（競合状態における最終防衛線）
INDEX idx_tf_team_a (team_a_id)                   -- 自チーム側からの逆引き
INDEX idx_tf_team_b (team_b_id)                   -- 相手チーム側からの逆引き
INDEX idx_tf_established (established_at DESC)     -- 成立日時順
INDEX idx_tf_is_public (is_public, team_a_id)      -- パブリック API 向け公開フレンド絞り込み

CONSTRAINT chk_tf_id_order CHECK (team_a_id < team_b_id)   -- DB レベルでの ID 順序強制（MySQL 8.0+ 必須）
```

**制約・備考**
- `team_a_id < team_b_id` を **MySQL 8.0 の CHECK 制約で DB レベルでも強制** する（MySQL 8.0 以降が必須前提）。アプリ層でも Service のバリデーションで事前に正規化するが、二重安全ネットとして CHECK 制約を置く。
- `a_follow_id` / `b_follow_id` は監査・整合性チェック用に両方向の `follows.id` を保持する。`follows` から該当レコードが削除されたらトリガー不要で Service 層が `team_friends` を削除し、`follows` 側には FK 制約を付けない（ポリモーフィック参照のため）
- 論理削除なし。フレンド解除時は物理削除
- `is_public` のデフォルトは `FALSE`（プライバシー安全側）。公開切替は ADMIN 操作でのみ可能
- `updated_at` は将来的なメタデータ（承認ステータス等）拡張を想定して保持

#### `team_friend_folders`

**自チーム視点のみ**の非対称フォルダ。A チームが B チームを「系列校」フォルダに入れていても、B チームの A に対する分類はまったく別の構造となる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。このフォルダを所有するチーム（= 自チーム） |
| `name` | VARCHAR(50) | NO | — | フォルダ名（例: '系列校', '姉妹団体', '練習試合候補') |
| `color` | VARCHAR(7) | NO | '#6B7280' | 表示色（HEX。UI バッジ色として使用） |
| `sort_order` | INT | NO | 0 | 並び替え順（小さいほど上に表示） |
| `description` | VARCHAR(300) | YES | NULL | フォルダの説明（任意） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
UNIQUE KEY uq_tff_team_name (team_id, name, deleted_at)   -- 論理削除対応のチーム内フォルダ名一意性
INDEX idx_tff_team_sort (team_id, sort_order)              -- 自チームのフォルダ一覧（並び替え順）
```

**制約・備考**
- `UNIQUE (team_id, name, deleted_at)` は MySQL の `NULL != NULL` 特性を利用した論理削除対応パターン（F06.1 と同方針）
- フォルダ上限: 1チームあたり **最大20フォルダ**（アプリ層で検証。超過時は 409）
- フォルダ削除時は、`team_friend_folder_members` の関連レコードを CASCADE で削除（フレンド関係自体は維持される）
- **自動転送カラム（`auto_forward_enabled` / `auto_forward_target`）は Phase 1 では作成しない。** 管理者の事前確認なしに配信される機能はリスクが高いため、Phase 3 で別マイグレーション（V9.086 予定）として追加予定。Phase 1 時点の DDL にはこれらのカラムを含めない。

#### `team_friend_folder_members`

フレンドフォルダとフレンドチームの多対多中間テーブル。**1つのフレンドチームは複数フォルダに所属可能**（例: 「系列校」かつ「練習試合候補」）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `folder_id` | BIGINT UNSIGNED | NO | — | FK → team_friend_folders。ON DELETE CASCADE |
| `team_friend_id` | BIGINT UNSIGNED | NO | — | FK → team_friends。ON DELETE CASCADE |
| `added_by` | BIGINT UNSIGNED | YES | NULL | FK → users（追加実行者; SET NULL on delete） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_tffm_folder_friend (folder_id, team_friend_id)   -- 同一フレンドを同じフォルダに二重登録防止
INDEX idx_tffm_folder (folder_id)
INDEX idx_tffm_friend (team_friend_id)
```

**制約・備考**
- フォルダ所有チームとフレンド関係の整合性（フォルダの `team_id` が `team_friends.team_a_id` または `team_b_id` のどちらかと一致すること）をアプリ層で検証
- フレンド解除時（`team_friends` が削除される時）は CASCADE により自動で関連レコードも削除される

#### `friend_content_forwards`

管理者が「転送」を実行した履歴。冪等性担保（二重転送防止）と監査ログとして使用する。**冪等性問題の設計修正**として、ソフトデリートに `revoked_at` を使う旧案から `is_revoked` フラグに切り替え、UNIQUE 制約を `(source_post_id, forwarding_team_id, is_revoked)` で設計する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `source_post_id` | BIGINT UNSIGNED | NO | — | FK → timeline_posts（転送元投稿。ON DELETE CASCADE） |
| `source_team_id` | BIGINT UNSIGNED | NO | — | FK → teams（転送元チーム = フレンドチーム側）。ON DELETE CASCADE |
| `forwarding_team_id` | BIGINT UNSIGNED | NO | — | FK → teams（転送実行チーム = 自チーム）。ON DELETE CASCADE |
| `forwarded_post_id` | BIGINT UNSIGNED | NO | — | FK → timeline_posts（転送で生成された自チーム内投稿。ON DELETE CASCADE） |
| `target` | VARCHAR(30) | NO | 'MEMBER' | 配信範囲。Phase 1 は 'MEMBER' 固定。Phase 3 で 'MEMBER_AND_SUPPORTER' 解禁予定 |
| `forwarded_by` | BIGINT UNSIGNED | NO | — | FK → users（転送操作を行った ADMIN / DEPUTY_ADMIN）。ON DELETE RESTRICT |
| `comment` | VARCHAR(500) | YES | NULL | 転送時に管理者が付与したコメント（任意） |
| `is_revoked` | BOOLEAN | NO | FALSE | 転送取消フラグ。`TRUE` = 取消済み / `FALSE` = アクティブ |
| `revoked_at` | DATETIME | YES | NULL | 取消日時（`is_revoked = TRUE` の場合のみ値を持つ） |
| `revoked_by` | BIGINT UNSIGNED | YES | NULL | FK → users（取消実行者。SET NULL on delete） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス・FK**
```sql
UNIQUE KEY uq_fcf_active (source_post_id, forwarding_team_id, is_revoked)
    -- アクティブな転送は (is_revoked=FALSE) で1つだけ許可。取消済み (is_revoked=TRUE) は複数並立を許容しないため、
    -- 実装上は取消後に新規転送したら「旧レコードの is_revoked=TRUE を維持したまま新レコード INSERT」とする。
    -- 厳密に「(source_post_id, forwarding_team_id) あたりアクティブ1つ」が保証される。
INDEX idx_fcf_forwarding_team (forwarding_team_id, created_at DESC)   -- 自チームの転送履歴
INDEX idx_fcf_source_team (source_team_id, created_at DESC)           -- フレンド側の被転送履歴（監査）
INDEX idx_fcf_forwarded_post (forwarded_post_id)                       -- 転送投稿からの逆引き

CONSTRAINT fk_fcf_source_post FOREIGN KEY (source_post_id)
    REFERENCES timeline_posts(id) ON DELETE CASCADE,
CONSTRAINT fk_fcf_forwarded_post FOREIGN KEY (forwarded_post_id)
    REFERENCES timeline_posts(id) ON DELETE CASCADE,
CONSTRAINT fk_fcf_source_team FOREIGN KEY (source_team_id)
    REFERENCES teams(id) ON DELETE CASCADE,
CONSTRAINT fk_fcf_forwarding_team FOREIGN KEY (forwarding_team_id)
    REFERENCES teams(id) ON DELETE CASCADE,
CONSTRAINT fk_fcf_forwarded_by FOREIGN KEY (forwarded_by)
    REFERENCES users(id) ON DELETE RESTRICT,
CONSTRAINT fk_fcf_revoked_by FOREIGN KEY (revoked_by)
    REFERENCES users(id) ON DELETE SET NULL
```

**制約・備考**
- 冪等性: 同じ `source_post_id` を同じ `forwarding_team_id` で同時に2回転送しようとすると、アクティブレコード（`is_revoked=FALSE`）が既に存在するため UNIQUE 制約違反 → 409 Conflict
- 取消実装: `is_revoked = TRUE` にフラグを立てる。同時に `revoked_at = NOW()` / `revoked_by = user_id` を記録
- 取消後の再転送: 旧レコードを保持したまま新レコードを `is_revoked = FALSE` で INSERT できる
- 取消時は `forwarded_post_id` の投稿を論理削除する（`timeline_posts.deleted_at` 設定）。ただし履歴は残す
- **転送元投稿の連鎖削除**: 転送元 `timeline_posts` が削除された場合、FK の `ON DELETE CASCADE` により `friend_content_forwards` レコードが自動削除される。加えて Service 層のフックで `forwarded_post_id` 側の `timeline_posts` も自動で論理削除し、受信チームの MEMBER には「転送元が削除されました」プレースホルダを表示する（§6 参照）

```
teams (1) ──── (N) team_friends [team_a_id]
teams (1) ──── (N) team_friends [team_b_id]
follows (1) ──── (1) team_friends [a_follow_id]
follows (1) ──── (1) team_friends [b_follow_id]

teams (1) ──── (N) team_friend_folders [team_id]

```

## 5. API設計

### 共通エラーハンドリング方針

すべてのエンドポイントは以下の方針に従う:

- **認証失敗**: 401 Unauthorized
- **権限不足**: 403 Forbidden（404 ではない。ただしチーム自体の存在を隠蔽する必要がある場合は SYSTEM_ADMIN 判断で 404 も可）
- **IDOR 対策**: パスパラメータの `{id}` / `{folderId}` / `{teamFriendId}` 等が認証ユーザーの所属チームに属さない・自チームのフォルダでない場合は 403 Forbidden を返す（リソース存在情報を漏らさない）
- **クエリパラメータの validation**: `cursor` / `limit` 等は型不一致・範囲外（`limit > 100` 等）の場合 400 Bad Request
- **プラットフォーム無効化時**: `platform_settings.friend_team_enabled = FALSE` の間、全フレンド関連エンドポイントは 403 Forbidden + `FEATURE_DISABLED` エラーコード

### リクエスト／レスポンス仕様

#### `POST /api/v1/teams/{id}/friends/follow`

**認可**: `{id}` チームの ADMIN、または `MANAGE_FRIEND_TEAMS` 権限を持つ DEPUTY_ADMIN。それ以外は 403 Forbidden。

指定した他チームをフォローする（`follows` テーブルに `(TEAM, {id}, TEAM, target_team_id)` を INSERT）。相手チームが既に自チームをフォロー済みだった場合、自動的にフレンド関係が成立する。

**リクエストボディ**
```json
{
  "target_team_id": 42,
  "comment": "練習試合でお世話になっているチームです"
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `target_team_id` | Long | Yes | フォロー先チーム ID |
| `comment` | String | No | フォロー時に相手チーム管理者に表示される挨拶メッセージ（最大300文字。フレンド成立通知に含まれる） |

**レスポンス（201 Created）**
```json
{
  "data": {
    "follow_id": 1001,
    "follower_team_id": 7,
    "followed_team_id": 42,
    "is_mutual": true,
    "team_friend": {
      "id": 55,
      "team_a_id": 7,
      "team_b_id": 42,
      "established_at": "2026-04-15T10:00:00",
      "is_public": false
    },
    "created_at": "2026-04-15T10:00:00"
  }
}
```

**補足**
- `is_mutual = true` の場合、相互フォロー成立として同時に `team_friends` にレコードが INSERT される
- `is_mutual = false` の場合、相手チームのフォロー待ち状態。`team_friends` は生成されない
- 相互フォロー成立時、両チームの ADMIN / DEPUTY_ADMIN（`MANAGE_FRIEND_TEAMS` 保持者）に `FRIEND_ESTABLISHED` 通知を送信
- `is_public` デフォルトは `false`。公開設定は PATCH `/visibility` で変更

**レスポンス（202 Accepted）** — NOWAIT 競合時
```json
{
  "data": {
    "accepted": true,
    "retry_after_seconds": 5,
    "detail": "相互フォロー処理が競合しました。バックグラウンドで再試行します"
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `target_team_id` 未指定 / 自チームへのフォロー / validation 違反 |
| 401 | 未認証 |
| 403 | ADMIN / DEPUTY_ADMIN（`MANAGE_FRIEND_TEAMS` 保持）でない / `platform_settings.friend_team_enabled = false` |
| 404 | `target_team_id` のチームが存在しない（論理削除含む） |
| 409 | 既にフォロー済み |
| 429 | フォローレート制限超過（§9 参照） |

#### `DELETE /api/v1/teams/{id}/friends/follow/{targetTeamId}`

**認可**: `{id}` チームの ADMIN、または `MANAGE_FRIEND_TEAMS` 権限を持つ DEPUTY_ADMIN。それ以外は 403 Forbidden。

指定した他チームへのフォローを解除する。相互フォロー状態だった場合は **自動的にフレンド関係も解除** される。

**リクエストボディ（オプション）**
```json
{
  "past_forward_handling": "KEEP"
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `past_forward_handling` | String | No | 過去転送投稿の扱い。`KEEP`（デフォルト・保持）/ `SOFT_DELETE`（論理削除）/ `ARCHIVE`（アーカイブセクションに移動）|

**レスポンス（204 No Content）**

**補足**
- フレンド解除時の処理:
  1. `follows` から A → B のレコードを DELETE
  2. `team_friends` から該当ペアのレコードを DELETE（物理削除）
  3. `team_friend_folder_members` の関連レコードを CASCADE で削除
  4. `friend_content_forwards` の扱いは `past_forward_handling` に従う（§6 参照）
  5. 両チーム管理者に `FRIEND_DISSOLVED` 通知を送信
  6. Valkey キャッシュ無効化（`friend:list:{team_a_id}`, `friend:list:{team_b_id}`）
- B → A のフォロー（相手側）は自動削除しない。相手が自発的に `DELETE` を実行するまで片方向フォロー状態が残る

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 401 | 未認証 |
| 403 | 権限不足 / IDOR（`{id}` が自チームでない） |
| 404 | フォロー関係が存在しない |
| 429 | フォロー解除レート制限超過（§9 参照） |

|-----------|---|------|------|

| `comment` | String | No | 管理者コメント（最大500文字。転送投稿の冒頭に表示される） |

**レスポンス（201 Created）**

```json
{

  }
}
```

**補足**
- バックエンド処理:
  1. `friend_content_forwards` に INSERT（UNIQUE 制約違反なら 409）
  2. 転送用の `timeline_posts` を生成:
     - `scope_type = 'FRIEND_FORWARD'`
     - `scope_id = forwarding_team_id`
     - `content` = 管理者コメント + 元投稿内容の抜粋（元投稿 ID を `source_post_id_ref` として JSON メタに保存）

| 403 | 権限不足 / IDOR |
| 404 | `postId` が存在しない |
| 409 | 既に転送済み（冪等性違反） |

## 6. ビジネスロジック

### 主要フロー

#### 相互フォロー検知・フレンド成立（競合対策込み）

```
1. POST /api/v1/teams/{id}/friends/follow を呼び出し
2. バックエンドが以下を検証:
   a. 認証ユーザーが {id} チームの ADMIN または MANAGE_FRIEND_TEAMS 保持 DEPUTY_ADMIN
   b. target_team_id が存在し、論理削除されていない
   c. 自己フォロー（{id} == target_team_id）でない
   d. 重複フォローでない（UNIQUE 制約で保証）
   e. platform_settings.friend_team_enabled = TRUE
3. トランザクション開始（分離レベル REPEATABLE READ）
4. follows に INSERT（follower_type='TEAM', follower_id={id}, followed_type='TEAM', followed_id=target_team_id）
5. 対称性チェック（SELECT FOR UPDATE NOWAIT による排他制御）:
   SELECT 1 FROM follows
   WHERE follower_type='TEAM' AND follower_id=target_team_id
     AND followed_type='TEAM' AND followed_id={id}
   FOR UPDATE NOWAIT
   → NOWAIT タイムアウト発生時は:
     a. トランザクションをロールバック
     b. レスポンス 202 Accepted + retry_after_seconds を返却
     c. バックグラウンドジョブキュー（Valkey Stream）に再試行タスクを投入
     d. 数秒後にワーカーが再実行
6. 対称性成立時:
   a. team_a_id = min(id, target) / team_b_id = max(id, target) に正規化
   b. team_friends に INSERT（CHECK chk_tf_id_order + UNIQUE uq_tf_pair で最終的な重複防止）
      → 二重成立が起きた場合（非常にまれなレース）は UNIQUE 違反で例外 → ロールバック
   c. 両チームの ADMIN / MANAGE_FRIEND_TEAMS 保持 DEPUTY_ADMIN に FRIEND_ESTABLISHED 通知を生成
   d. Valkey キャッシュ無効化（friend:list:{team_a_id}, friend:list:{team_b_id}）
7. コミット
8. レスポンスに is_mutual フラグを含めて返却
```

**根拠（なぜこの競合対策が必要か）**: A → B のフォローと B → A のフォローがほぼ同時に実行された場合、単純な SELECT だけでは「両方とも相手側レコードを見つけて `team_friends` INSERT を 2 回試みる」ことが起こり得る。`FOR UPDATE NOWAIT` で対称レコードにロックをかけ、UNIQUE 制約を最終防衛線とし、NOWAIT で待たされる代わりに 202 で即座に応答しバックグラウンド再試行することで、UX を損なわずに整合性を保つ。

#### フォロー解除・フレンド解除

```
1. DELETE /api/v1/teams/{id}/friends/follow/{targetTeamId} を呼び出し
2. バックエンドが以下を検証:
   a. 認証・権限（上記と同じ）
   b. follows に該当レコードが存在
3. トランザクション開始:
   a. follows から DELETE（follower_id={id}, followed_id=targetTeamId の行）
   b. team_friends から DELETE（team_a_id/team_b_id が {id,targetTeamId} ペアの行）
      → CASCADE で team_friend_folder_members も削除される
   c. friend_content_forwards の扱い（リクエストの past_forward_handling に従う）:
      - KEEP（デフォルト）: 保持。ただし以降の閲覧時は「フレンド解除済み」注釈付き
      - SOFT_DELETE: 全 forwarded_post_id を論理削除（deleted_at 設定）
      - ARCHIVE: forwarded_post_id の scope_type を 'FRIEND_ARCHIVE' に変更し、通常タイムラインから除外
        (別 API `/api/v1/teams/{id}/friend-archive` で閲覧可能)
   d. 両チームに FRIEND_DISSOLVED 通知を生成
   e. Valkey キャッシュ無効化
4. コミット
5. 204 を返却
```

> **UI**: ADMIN がフレンド解除ボタンを押すと、モーダルで「過去転送投稿をどうするか」選択（KEEP / SOFT_DELETE / ARCHIVE）を提示する。

> **注意**: B → A のフォローは自動削除しない。相手側の意思を尊重し、片方向フォロー状態が残る。

#### 二段階配信: フレンドチーム発コンテンツの受信

```
[ フレンドチーム A のタイムライン投稿発生時 ]
1. timeline_posts に scope_type='TEAM', scope_id=A_id で INSERT
2. 投稿配信フローの拡張として:
   a. team_friends を検索: A_id を含むペア全てを取得
   b. 各フレンドチーム B に対し、以下を実行:
      i.   timeline_posts に「シャドーエントリ」を INSERT しない（= 物理コピーを作らない設計を採用）
      ii.  代わりに friend_feed_items という仮想ビュー、またはクエリ時結合で実現
         → 実装選択肢（推奨: クエリ時結合）
         A: 物理コピー案: scope_type='FRIEND_TEAM', scope_id=B_id として別レコード
         B: クエリ時結合案: GET friend-feed 実行時に team_friends + timeline_posts を JOIN
         → Phase 1 は B 案（ストレージ節約・整合性自動維持）、Phase 3 で性能要件に応じて A 案に移行検討
   c. 通知生成: B チームの ADMIN / MANAGE_FRIEND_TEAMS 保持 DEPUTY_ADMIN に対し
      notifications INSERT（scope_type='FRIEND_TEAM', scope_id=B_id, source_type='TIMELINE_POST', source_id={原投稿ID}）
   d. Push 通知チャネル選択は F04.3 の既定ルールに従う
3. B チーム内の MEMBER や SUPPORTER には**何も送らない**（ノイズ制御の要諦）
```

#### 二段階配信: 管理者による転送実行

```
1. ADMIN が GET /api/v1/teams/B/friend-feed で受信投稿を確認
2. 転送判断: POST /api/v1/teams/B/friend-feed/{postId}/forward
   body: { target: 'MEMBER', comment }   ※ Phase 1 は MEMBER 固定
3. バックエンドが以下を実行:
   a. friend_content_forwards に INSERT（冪等性チェック: source_post_id × forwarding_team_id × is_revoked=FALSE で重複確認）
   b. 新たな timeline_posts を生成:
      - scope_type = 'FRIEND_FORWARD'
      - scope_id = forwarding_team_id (= B_id)
      - content = comment + 元投稿の抜粋
      - metadata = { source_post_id, source_team_id, source_team_name, target }
   c. friend_content_forwards.forwarded_post_id に新投稿 ID を紐付け
   d. 通知配信:
      - target='MEMBER': B チームの MEMBER 全員に FRIEND_FORWARD_RECEIVED 通知
      (Phase 3 以降) target='MEMBER_AND_SUPPORTER': MEMBER + SUPPORTER
   e. Valkey Sorted Set（timeline:FRIEND_FORWARD:{B_id}）に ZADD
   f. WebSocket 配信（B チーム内メンバー向け）
   g. audit_logs に FRIEND_CONTENT_FORWARDED を記録
      metadata: { source_post_id, source_team_id, forwarding_team_id, target, comment, admin_user_id }
4. 201 Created を返却
```

#### 二段階配信: 転送取消

```
1. DELETE /api/v1/teams/B/friend-feed/{postId}/forward/{forwardId}
2. バックエンドが以下を実行:
   a. friend_content_forwards.is_revoked = TRUE, revoked_at = NOW(), revoked_by = ユーザーID
   b. 対応する timeline_posts（FRIEND_FORWARD 種別）を論理削除
   c. 既送信通知は削除しない（「取り消されました」と表示）
   d. Valkey から ZREM
   e. audit_logs に FRIEND_CONTENT_FORWARD_REVOKED を記録
3. 204 を返却
```

#### 転送元投稿削除時の CASCADE 連鎖

```
[ フレンドチーム A が自チーム投稿を削除した場合 ]
1. timeline_posts.deleted_at 設定（論理削除）
2. Service 層のフックで以下を実行:
   a. friend_content_forwards WHERE source_post_id = {削除された投稿ID} を全検索
   b. 各レコードの forwarded_post_id の timeline_posts を論理削除
   c. 受信チーム MEMBER の該当タイムライン表示箇所には「転送元が削除されました」プレースホルダ表示
3. 物理削除（hard delete）が発生した場合は:
   a. FK の ON DELETE CASCADE により friend_content_forwards レコードが自動削除される
   b. forwarded_post_id → timeline_posts も CASCADE で削除される
   c. 連鎖削除のため、受信チーム MEMBER のタイムラインから転送投稿が消える
```

### フレンド解除後の過去転送投稿の扱い（3モード）

| モード | 動作 | 想定ユースケース |
|-------|-----|----------------|
| `KEEP`（デフォルト） | 過去転送投稿は保持し、通常タイムラインで閲覧可。「（元フレンドチームからの転送）」注釈を付ける | 履歴価値・監査目的でそのまま残したい |
| `SOFT_DELETE` | 過去転送投稿を論理削除し、タイムラインから非表示 | 関係解消時にフレンド発情報を全消去したい |
| `ARCHIVE` | `scope_type` を `FRIEND_ARCHIVE` に変更し、通常タイムラインから除外。別エンドポイント `/api/v1/teams/{id}/friend-archive` で閲覧可能 | 残しつつ目立たせたくない |

ADMIN がフレンド解除時にモーダルで選択する（UI 側で強制的に選択ステップを挟む）。

### 重要な判定ロジック

- **フレンド関係のキャッシュ**: Valkey `friend:list:{teamId}` に Set 型でフレンド相手のチーム ID 群を保持。TTL 1時間。フォロー/フォロー解除時に両チーム分を INVALIDATE
- **管理者フィードの取得クエリ**: `team_friends` テーブルで自チームの相手チーム ID 一覧を取得後、`timeline_posts WHERE scope_type='TEAM' AND scope_id IN (相手チーム ID 群) AND deleted_at IS NULL AND share_with_friends = TRUE` を UNION で抽出（Phase 1 = クエリ時結合）
- **通知送信時の対象ユーザー解決**: `MANAGE_FRIEND_TEAMS` 権限保持者を `user_roles` + `user_permission_groups` + `permission_group_permissions` の JOIN で解決
- **転送対象範囲の判定**: Phase 1 は `MEMBER` 固定。Phase 3 以降は `friend_content_forwards.target` を `timeline_posts` 表示フィルタで利用。GET /timeline のレスポンス絞り込み時に `scope_type='FRIEND_FORWARD'` の投稿について、閲覧者のロールが target の許可範囲内かチェック
- **パブリック API でのフレンド露出**: `team_friends.is_public = TRUE` のレコードのみを返却。それ以外は除外

### スケーラビリティ

| 観点 | 対策 |
|------|-----|
| フレンド関係検索の高頻度化 | Valkey `friend:list:{teamId}` によるキャッシュ。TTL 1時間 + 書き込み時 INVALIDATE |
| 管理者フィードのフィード生成コスト | Phase 1 はクエリ時結合（コピー不要・整合性自動維持）。Phase 3 で1000 フレンド超えの大規模チームに備え物理コピー方式へ移行可能な設計 |
| 通知の大量発行 | 逆方向通知（フレンドフォルダ全員宛て）は非同期バッチ配信（Valkey Stream or DB テーブルキュー）。同期API は 202 Accepted を返却し `queued_teams_count` のみ応答 |
| 整合性チェック | 日次バッチで `follows` の対称性と `team_friends` レコードの整合を突合。齟齬があれば自動修復 or アラート（§9.5 参照） |
| レート制限 | §9 参照 |

   - 配信範囲: ラジオボタン `MEMBER`（Phase 1 はこれ固定）。Phase 3 以降に `MEMBER_AND_SUPPORTER` 追加
   - 管理者コメント: テキストエリア（最大500文字）
3. 「転送する」ボタン押下 → API 呼び出し
4. 転送後はカードが「転送済み」バッジ付きに更新

### フレンド解除モーダル

1. 「フレンドを解除」ボタン押下
2. モーダルで以下を選択:
   - 過去転送投稿の扱い: `KEEP` / `SOFT_DELETE` / `ARCHIVE`
   - 相手チームに理由を伝える（オプション・最大300文字）
3. 「解除する」確認ボタン押下 → API 呼び出し

### 通常メンバーの閲覧体験

- 自チームのタイムラインに転送投稿が混ざる（`scope_type='FRIEND_FORWARD'`）
- 転送投稿には「📢 FC 東山（フレンドチーム）からの情報」のような出典ラベルを表示
- 直接「いいね」や返信は不可（読み取り専用）。反応したい場合は ADMIN 経由で連絡

## 7.5 エッジケース対応

### E1: 転送元投稿削除時の整合性

- **事象**: フレンドチーム A が自チーム投稿を削除した際、B チームで既に転送された投稿がある
- **対応**:
  - 論理削除（`timeline_posts.deleted_at` 設定）時: Service 層フックで `friend_content_forwards.source_post_id = {削除 ID}` を検索し、`forwarded_post_id` 側の `timeline_posts` も論理削除
  - 物理削除時: FK `ON DELETE CASCADE` により `friend_content_forwards` と `forwarded_post_id` が自動削除
  - 受信側表示: 「転送元が削除されました」プレースホルダ

### E2: フレンド関係成立後のチーム論理削除

- **事象**: フレンドチーム B が `teams.deleted_at` 設定で論理削除された
- **対応**:
  - `team_friends.team_a_id/b_id` FK `ON DELETE CASCADE` は物理削除時のみ発動するので、論理削除時は Service 層フックで対応
  - B の論理削除時に `team_friends` から該当ペアを DELETE
  - `team_friend_folder_members` は CASCADE で自動削除
  - 両チーム ADMIN に `FRIEND_DISSOLVED`（理由: `TEAM_DELETED`）通知

### E3: ADMIN の MANAGE_FRIEND_TEAMS 権限喪失

- **事象**: DEPUTY_ADMIN から `MANAGE_FRIEND_TEAMS` 権限が剥奪された
- **対応**:
  - 以降のフレンド関連 API 呼び出しは 403
  - `audit_logs` に `PERMISSION_REVOKED_MANAGE_FRIEND_TEAMS` 記録
  - 既存の転送履歴 (`friend_content_forwards.forwarded_by`) は残す（監査価値）

### E4: 大量フォローからの復帰

- **事象**: レート制限超過でフォロー操作が 429 になり、ユーザーがその後操作を継続したい
- **対応**:
  - レスポンスヘッダ `Retry-After` でリセット時刻を返却
  - UI で「○分後に再試行できます」表示
  - アンチスパム自動ブロック対象となった場合は、SYSTEM_ADMIN への申告導線 `/api/v1/support/team-friend-unblock-request` を提供

### E5: フレンドフォルダ削除時の処理

- **事象**: ADMIN がフォルダを削除
- **対応**:
  - `team_friend_folders.deleted_at` 設定（論理削除）
  - `team_friend_folder_members` は FK CASCADE で物理削除
  - フレンド関係（`team_friends`）自体は影響なし
  - 日次バッチで `deleted_at` から 90 日経過したレコードを物理削除

### E6: フレンド通知のレート制限超過時

- **事象**: `POST /friend-notifications/send` で1日100回制限を超えた
- **対応**:
  - 429 Too Many Requests 返却
  - `Retry-After` ヘッダで翌日リセット時刻
  - バースト対策として1分10回制限も課す（§9 参照）
  - ADMIN への警告通知: 「本日の送信上限に達しました」

## 8. 通知・プッシュ通知設計

### 新規通知種別（`NotificationType` enum 追加）

| 種別 | トリガー | 受信者 | デフォルト Push |
|------|---------|--------|-----|
| `FRIEND_FOLLOW_REQUEST` | 片方向フォローされた（まだ相互未成立） | フォローされた側の ADMIN / MANAGE_FRIEND_TEAMS 保持 DEPUTY_ADMIN | NORMAL |

- `notification_type_preferences` で個別種別の ON/OFF が可能（既存メカニズムを流用）
- `notification_preferences` の `scope_type` に `FRIEND_TEAM` / `FRIEND_FOLDER` を追加し、特定フレンドやフォルダ単位の受信制御を可能にする（Phase 3）
- `FRIEND_NOTIFICATION_RECEIVED` は `priority = HIGH` の場合は種別 OFF を無視して配信（安全側設計）

### 集約配信（rate-limiting digest）

- `FRIEND_FEED_POST_RECEIVED` は1時間に1回のまとめ通知方式がデフォルト（UX 配慮）
- 例: 「過去1時間で3件のフレンド投稿が届きました。管理者フィードをご確認ください」
- `platform_settings.friend_feed_digest_window_minutes` でウィンドウ幅調整可能（初期値60、β検証後に調整）

## 9. セキュリティ・プライバシー

### チーム非公開コンテンツのフレンド可視性

- フレンドチームの TEAM スコープ投稿は、相手チーム内コンテンツであり**外部非公開が前提**
- 管理者フィードで受信する時点でも、元投稿の `visibility` 属性を尊重する
  - `visibility = PRIVATE` の投稿は**フレンド配信の対象外**
  - `visibility = MEMBERS_ONLY` の投稿も Phase 1 ではフレンド配信対象外（安全側設計）
  - フレンド配信対象となるのは、明示的に `share_with_friends = TRUE` マークされた投稿、または将来拡張の `visibility = FRIENDS_INCLUSIVE` 投稿のみ（Phase 3 で詳細仕様確定）
- Phase 1 のリリース時点では: 「フレンド配信用に明示的に投稿された通知」に限定。既存の TEAM 投稿は自動的にフレンド連携されない

### 転送時のプライバシー確認

- **Phase 1 では `target = MEMBER` のみ**。SUPPORTER（保護者等の外部支援者）への波及は禁止
- 転送 API 実行時、元投稿に個人を特定できるフィールド（実名・添付画像内の顔）が含まれる場合は、UI で警告を表示（Phase 3 で AI 判定を検討）
- 転送投稿のメタデータに「転送元フレンドチーム名」を必ず含める。匿名化しない（透明性担保）
- 転送元チーム側からは、自分の投稿がどのフレンドチームに転送されたか履歴を**Phase 1 から**確認可能（`friend_content_forwards.source_team_id` からの逆引き API `GET /api/v1/teams/{id}/friend-forward-exports`。透明性重視）

### 悪意ある大量フォローからの保護（レート制限強化）

| 操作 | 制限 |
|-----|------|
| フォロー実行 | 1チームあたり 1時間 10回 |
| フォロー解除 | 1チームあたり 1時間 5回 |
| フォロー → 解除 → 再フォローの反復 | 24時間以内に同一ペアで3回繰り返すと一時ブロック（24時間）。自動で `team_follow_temp_blocks` テーブルに記録 |
| 管理者フィード取得 | 1チームあたり 1分 10回 |
| 通知送信（`/friend-notifications/send`） | 1リクエストあたり最大受信者 50人 / 1日 1,000通知 / 1分 10通知 |
| フォルダ操作（CRUD） | 制限なし（軽量操作のため） |

- アンチスパム: 24時間以内に同じチームからのフォロー→解除を3回繰り返した場合、そのチームを**フォロー拒否リスト**に自動追加（F04.5 モデレーション連携）
- SYSTEM_ADMIN は `DELETE /api/v1/admin/team-friends/{teamFriendId}` で強制解除可能（監査ログ `FRIEND_FORCE_DISSOLVED` 記録）
- 新規設立から3日以内のチームはフォロー送信不可（スパム対策）

### 日次バッチでの整合性チェック SQL

```sql
-- 1. follows の対称性と team_friends の整合性チェック
-- orphan な team_friends（片方向 follows のみ存在）を検出
SELECT tf.id, tf.team_a_id, tf.team_b_id
FROM team_friends tf
LEFT JOIN follows fa ON fa.follower_type='TEAM' AND fa.follower_id=tf.team_a_id
    AND fa.followed_type='TEAM' AND fa.followed_id=tf.team_b_id
LEFT JOIN follows fb ON fb.follower_type='TEAM' AND fb.follower_id=tf.team_b_id
    AND fb.followed_type='TEAM' AND fb.followed_id=tf.team_a_id
WHERE fa.id IS NULL OR fb.id IS NULL;

-- 2. 相互フォロー済みなのに team_friends が存在しないペアを検出
SELECT f1.follower_id AS team_a, f1.followed_id AS team_b
FROM follows f1
JOIN follows f2 ON f2.follower_type='TEAM' AND f2.follower_id=f1.followed_id
    AND f2.followed_type='TEAM' AND f2.followed_id=f1.follower_id
WHERE f1.follower_type='TEAM' AND f1.followed_type='TEAM'
  AND f1.follower_id < f1.followed_id
  AND NOT EXISTS (
    SELECT 1 FROM team_friends tf
    WHERE tf.team_a_id = f1.follower_id AND tf.team_b_id = f1.followed_id
  );

-- 3. 転送レコードで forwarded_post_id が存在しない行を検出
SELECT fcf.id FROM friend_content_forwards fcf
LEFT JOIN timeline_posts tp ON tp.id = fcf.forwarded_post_id
WHERE tp.id IS NULL AND fcf.is_revoked = FALSE;
```

## 10.5 テスト戦略

### ユニットテスト対象

| 対象クラス | 主要メソッド | 重点検証項目 |
|----------|-----------|-----------|
| `TeamFriendsService` | `follow()` | 権限チェック、自己フォロー禁止、重複フォロー検出、NOWAIT リトライ、相互成立時の `team_friends` INSERT |
| `TeamFriendsService` | `unfollow()` | 片方向解除時の `team_friends` 削除、`past_forward_handling` モード分岐 |
| `TeamFriendsService` | `detectMutual()` | `team_a_id < team_b_id` 正規化、UNIQUE 制約違反のリトライ |
| `FriendContentForwardService` | `forward()` | 冪等性（同一ペアの二重転送禁止）、`target` バリデーション（Phase 1 は MEMBER 固定）、監査ログ記録 |
| `FriendContentForwardService` | `revoke()` | `is_revoked = TRUE` 遷移、`forwarded_post_id` 論理削除、再転送可能性 |
| `FriendContentForwardService` | `handleSourcePostDeletion()` | CASCADE 連鎖、受信側プレースホルダ表示 |
| `TeamFriendFolderService` | `create()` / `update()` / `delete()` | フォルダ上限20件、名前一意性、論理削除 |
| `FriendNotificationService` | `sendToFolder()` / `sendToTeams()` | レート制限、フレンド関係存在チェック、非同期キュー投入 |

### 統合テスト対象

- **フレンド成立フロー**: POST /follow → 対称側 follow → team_friends INSERT → 両チーム通知 → Valkey cache invalidation
- **二段階配信フロー**: フレンド A 投稿 → B 管理者フィード反映 → 転送 → B の MEMBER タイムライン表示
- **フレンド解除の3モード**: KEEP / SOFT_DELETE / ARCHIVE それぞれで期待されるタイムライン状態を確認
- **転送元削除連鎖**: フレンド A の元投稿削除 → B の転送投稿が論理削除 → プレースホルダ表示
- **大規模フレンド**: 1チーム 100 / 500 / 1000 フレンド規模での管理者フィード取得性能
- **権限テスト**: MEMBER / SUPPORTER / DEPUTY_ADMIN（権限なし） / DEPUTY_ADMIN（MANAGE_FRIEND_TEAMS 保持）でのアクセス可否

### 並行・競合テスト

- **相互フォロー同時実行**: A → B と B → A がミリ秒差で到達した場合の UNIQUE 制約動作と NOWAIT リトライ
- **同時転送**: 同一投稿への転送を 2 ユーザーが同時実行 → 1 つだけ成功、もう 1 つは 409
- **フレンド解除中の転送**: トランザクション分離レベルでの挙動確認

### 性能テスト（負荷試験）

| シナリオ | 目標 |
|---------|-----|
| フレンド 10 件 / 管理者フィード取得 | p95 < 500ms |
| フレンド 100 件 / 管理者フィード取得 | p95 < 1秒 |
| フレンド 1000 件 / 管理者フィード取得 | p95 < 3秒（Phase 3 物理コピー方式移行の閾値判定） |

| 相互フォロー同時 100 ペア | デッドロック 0、成立率 100% |

### セキュリティテスト

- IDOR: `{folderId}` に他チームの ID を指定 → 403
- 権限昇格: MEMBER が `/friend-feed` にアクセス → 403
- Phase 1 制約: `target = MEMBER_AND_SUPPORTER` 指定 → 400

## 13. セキュリティ考慮事項

- **所有権チェック**: フォロー/フォロー解除 API は `{id}` チームの ADMIN / MANAGE_FRIEND_TEAMS 保持 DEPUTY_ADMIN であることを必ず検証。IDOR 時は 403 Forbidden
- **フレンドチームのみ閲覧可**: 管理者フィード・フレンド通知・転送 API は **フレンド関係が成立している場合のみ** 機能する（`team_friends` 存在チェック必須）
- **Phase 1 での SUPPORTER 除外**: 転送 API は `target = MEMBER` のみ受理。`MEMBER_AND_SUPPORTER` 指定は 400
- **GDPR 相当の同意**: フレンド関係成立時に両チームの ADMIN が「自チームの投稿・通知がフレンドチームに共有される可能性がある」ことを理解している状態で成立させる（規約同意フロー）
- **凍結されたチームはフォロー不可**: `teams.deleted_at IS NOT NULL` または `archived_at IS NOT NULL` のチームはフォロー/フォロー対象から除外
- **通報連携**: `content_reports.target_type` に `TEAM_FRIEND_RELATION` を追加し、不適切なフレンド関係を通報可能とする（Phase 3）
- **レート制限の多段設計**: §9 参照

## 14. 懸念事項・残課題（すべて解決済み）

### 設計上の懸念（解決方針確定）

- [x] **大規模環境でのパフォーマンス**: 1チームあたり1000フレンドを超える運用が想定される組織（地域連合・大規模教育法人等）で、管理者フィードのクエリ時結合がボトルネックにならないか
    → **解決**: Phase 1 はクエリ時結合方式（JOIN）、Phase 3 で負荷試験に基づき物理コピー方式へ移行可能な設計。§9.5 のメトリクス `friend_feed_delivery_latency_p95` で監視し閾値超過時に Phase 3 移行判断
- [x] **片方向フォロー状態の UX**: 相手が気づかずに承認しない状態が長期化した場合のフォロー解除フローを明示すべきか
    → **解決**: 30日間返事がない場合、フォロー元の ADMIN に「相手からの返事がありません。解除しますか?」リマインダー通知を送る。自動タイムアウト削除はしない（相手の意思尊重）
- [x] **フォルダの同期**: A チームが B をフォルダ X に入れた後、フレンド解除→再成立した場合にフォルダ情報を復元するか
    → **解決**: 現設計では失われる。過去フォルダ所属履歴を `friend_folder_history` テーブル（Phase 3 追加）に残し、再成立時に「前回は○○フォルダに所属していました。復元しますか?」と提案する UI で対応
- [x] **SUPPORTER の視認範囲拡大**: 転送 `MEMBER_AND_SUPPORTER` 時にサポーターが他団体情報を見られる状態は、プライバシー観点で慎重に扱う
    → **解決**: Phase 1 では `MEMBER` 固定で SUPPORTER 配信しない。Phase 3 で申請・審査メカニズム＋利用規約明文化後に解禁

### 将来拡張への布石

- [x] **非対称な「親子関係」・「パートナーシップ」**: 現設計は対等な相互フォローだが、将来的に「上位組織→下位組織」の階層関係（例: 本部→支部）を別機能として追加する可能性。`team_friend_relation_type` ENUM カラムの追加で対応可能な設計
- [x] **組織間フレンド（Org-to-Org）**: 本機能の拡張として、`follows` の `TEAM` に加えて `ORGANIZATION` 値を受け入れれば組織単位フレンドも実現可能。現設計の影響なく拡張可能
- [x] **フレンド限定イベント・マッチング**: F08.1 マッチングとの統合で、`match_requests.visibility` に `FRIEND_TEAMS_ONLY` を追加

### 旧・未解決事項（すべて解決済み）

- [x] **自動転送機能（`auto_forward_enabled = TRUE`）の具体的な動作仕様**
    → **解決**: Phase 1 では実装しない（カラム自体を作成しない）。Phase 3 で V9.086 マイグレーションにて `auto_forward_enabled` / `auto_forward_target` カラムを追加し、管理者の事前同意 + 配信前プレビュー通知を組み合わせた段階的な導入を行う
- [x] **管理者フィードの集約配信ウィンドウのチューニング**
    → **解決**: 初期値 60分（`platform_settings.friend_feed_digest_window_minutes = 60`）でリリース。β検証後に UX フィードバックを元に 30〜120分の範囲で調整。A/B テスト導線を Phase 3 で整備
- [x] **転送元チーム側に「うちの投稿が転送されました」という逆方向可視性**
    → **解決**: 透明性重視で **Phase 1 から実装**。`GET /api/v1/teams/{id}/friend-forward-exports` エンドポイントで自チーム投稿の転送履歴を参照可能（`friend_content_forwards.source_team_id = {id}` で逆引き）
- [x] **フレンド成立時の「挨拶コメント」の保存先**
    → **解決**: Phase 1 では通知メッセージへ埋め込みのみ（非永続）。Phase 3 で `team_friends.establishment_comment` VARCHAR(300) NULL カラムを追加してメタデータとして永続化（V9.087 マイグレーション予定）
