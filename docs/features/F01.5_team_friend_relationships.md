> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 4（F04.4 フォロー基盤と同時期にリリース可能）
> **最終更新**: 2026-04-15
> **関連ドキュメント**: F01.2 組織・チーム・メンバー・ロール管理 / F04.1 タイムライン / F04.3 プッシュ通知 / F04.4 ソーシャルプロフィール・フォロー / F06.1 CMS・ブログ / F08.1 マッチング

## 1. 概要

チーム同士がチーム単位で相互にフォローし合うことで成立する「フレンドチーム」関係と、その関係から派生するコンテンツ・通知の二段階配信モデルを提供する中核機能。既存の `follows` テーブル（F04.4）に `TEAM` タイプの follower/followed を追加し、相互フォロー成立時に自動的にフレンドチームとして認定する。チーム内でのノイズ制御のため、フレンドチームから発せられた通知やタイムライン投稿は **直接一般メンバーには届かず**、受信チームの ADMIN / DEPUTY_ADMIN だけが閲覧できる「管理者フィード」に集約される。管理者が「転送」判断を行った場合のみ、自チームの MEMBER や SUPPORTER へリブロードキャストされる（回覧板・掲示板モデル）。さらにフレンドチームを目的別に整理する「フレンドフォルダ」を各チーム視点で管理でき、非対称なラベル付け・通知配信対象の絞り込みが可能。

### 主なユースケース

- 姉妹チーム・系列校・地域同業ネットワーク同士が公式に「提携」を可視化する
- 同一エリアのスポーツチーム同士が練習試合・合同イベントを効率的に呼びかける（F08.1 マッチングと異なり、継続的な関係前提）
- 傘下クラブや親子団体がゆるやかな情報交換ネットワークを構築する
- チーム ADMIN が大量のフレンド通知を「掲示する価値があるか」事前審査し、下位メンバーへの情報過多を防ぐ
- フレンドチーム同士で BLOG 記事を優先公開範囲として指定する（オプション）

| 機能 | 関係性 |

## 2. 用語定義

| 用語 | 定義 |
|------|-----|
| **フレンドチーム** | チーム A とチーム B が相互に `follows` 関係を結んだ状態。`team_friends` テーブルの1レコードに対応 |
| **相互フォロー** | `follows` に `(TEAM, A, TEAM, B)` と `(TEAM, B, TEAM, A)` の2行が同時に存在する状態 |
| **フレンドフォルダ** | 自チーム視点でフレンドチーム群を分類するラベル。例: 「系列校」「練習試合候補」「姉妹団体」。**非対称**（チーム A のフォルダにチーム B をどう入れるかは、チーム B のフォルダ構成と無関係） |
| **管理者フィード** | フレンドチームから配信された通知・投稿を受信する、ADMIN / DEPUTY_ADMIN 限定のタイムラインバケット。`timeline_posts.scope_type = 'FRIEND_TEAM'` に相当 |
| **二段階配信** | フレンドチーム発のコンテンツが (1) 受信チーム管理者への通知配信 → (2) 管理者判断による自チーム内リブロードキャスト、の2ステップで届く仕組み |
| **転送（Forward）** | 管理者が管理者フィードの投稿を自チーム内タイムラインに再配信する操作。転送元フレンドチーム名が出典として明示される |
| **転送対象** | 転送時に選べる配信範囲。`MEMBER`（メンバーまで）/`MEMBER_AND_SUPPORTER`（メンバー+サポーター）の2種 |
| **フレンド成立通知** | 相互フォローが成立した瞬間に両チームの ADMIN / DEPUTY_ADMIN に送られるシステム通知 |
| **フォロー解除の連鎖** | 片方がフォロー解除した瞬間に相互状態が崩れるため、`team_friends` レコードを自動削除し、フレンド解除通知を両チーム管理者に送る |

## 3. スコープ・権限

### 対象ロール

| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全フレンド関係の参照・強制解除（モデレーション）。フレンド関連テーブルの監査閲覧 |
| ADMIN | 自チームとしての他チームフォロー/解除、フレンドフォルダ CRUD、管理者フィード閲覧、転送実行/取消、フレンド向け通知送信 |
| DEPUTY_ADMIN | `MANAGE_FRIEND_TEAMS` 権限を持つ場合: ADMIN と同等（フレンド関係の追加/解除・フォルダ管理・転送・通知送信） |
| MEMBER | 自チームのフレンドチーム一覧の閲覧のみ（公開設定に準拠）。転送されたフレンド発投稿は通常タイムライン上で閲覧可 |
| SUPPORTER | 自チームのフレンドチーム一覧の閲覧のみ。転送対象が `MEMBER_AND_SUPPORTER` の場合のみ転送投稿を閲覧可 |
| GUEST | 対象外 |

> **新規パーミッション**: `MANAGE_FRIEND_TEAMS`（TEAM scope）を `permissions` に追加。ADMIN は `is_default = TRUE`、DEPUTY_ADMIN は `is_default = FALSE`（天井のみ、ADMIN が権限グループ経由で付与）。MEMBER 以下には天井エントリなし（F01.2 §permissions / role_permissions の方針を踏襲）。

### 対象レベル

- [x] チーム (Team) — フレンドチーム関係はチーム間の関係であるため、唯一のスコープ
- [ ] 組織 (Organization) — 本フェーズでは対象外（組織間連携は将来拡張）
- [ ] 個人 (Personal) — 対象外

### 通知対象範囲ルール

**【確定仕様】二段階配信により、フレンドチーム発のコンテンツ・通知は以下のルールに従う:**

| 段階 | 受信者 | 通知チャネル |
|------|------|--------------|
| 第1段階: 受信チーム管理者 | ADMIN / `MANAGE_FRIEND_TEAMS` 権限を持つ DEPUTY_ADMIN | アプリ内通知 + オプションで Push |
| 第2段階: 転送判断後 | 管理者が選択した範囲（MEMBER のみ / MEMBER + SUPPORTER） | 既存タイムライン通知フローに準拠 |

SUPPORTER は第1段階の通知対象外（管理者ではないため）、かつ転送時に `MEMBER_AND_SUPPORTER` が選択された場合のみ転送投稿を目にする。

## 4. DB設計

### テーブル一覧

| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `team_friends` | フレンドチーム関係キャッシュ（相互フォロー成立時に生成。`follows` から自動導出） | なし（物理削除） |
| `team_friend_folders` | 自チーム視点のフレンドフォルダ | あり |
| `team_friend_folder_members` | フレンドフォルダとフレンドチームの多対多中間テーブル | なし |
| `friend_content_forwards` | フレンドコンテンツ転送履歴（冪等性・監査ログ） | なし（物理削除: 転送元投稿削除時に CASCADE） |

### 既存テーブルへの波及変更

| テーブル | 変更内容 |
|---------|--------|
| `follows` | `follower_type` / `followed_type` が受理する値に `TEAM` を正式追加（ドキュメント上の拡張。DB は VARCHAR のため DDL 変更なし） |
| `timeline_posts` | `scope_type` に `FRIEND_TEAM`（管理者フィード専用）・`FRIEND_FORWARD`（転送投稿）を受理させる（アプリ層 enum 拡張。VARCHAR のため DDL 変更なし） |
| `notifications` | `scope_type` に `FRIEND_TEAM` / `FRIEND_FOLDER` を追加（同上） |
| `blog_posts` | `visibility` に `FRIEND_TEAMS_ONLY` を追加（オプション・Phase 6 以降） |
| `permissions` | シードデータに `MANAGE_FRIEND_TEAMS` を追加 |
| `role_permissions` | ADMIN ✓ / DEPUTY_ADMIN △ を追加 |

### テーブル定義

#### `team_friends`

相互フォロー成立時に生成されるキャッシュテーブル。`follows` テーブルの `(TEAM, A) ↔ (TEAM, B)` 双方向エントリから自動的に導出される。検索・一覧表示の高速化のために正規化せずに保持する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_a_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。関係の片側（常に `team_a_id < team_b_id` となるよう正規化） |
| `team_b_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。関係のもう片側 |
| `established_at` | DATETIME | NO | CURRENT_TIMESTAMP | 相互フォロー成立日時（フレンドシップ成立日） |
| `a_follow_id` | BIGINT UNSIGNED | NO | — | FK → follows（A→B のフォローレコード ID） |
| `b_follow_id` | BIGINT UNSIGNED | NO | — | FK → follows（B→A のフォローレコード ID） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_tf_pair (team_a_id, team_b_id)     -- 同一ペアの重複登録防止
INDEX idx_tf_team_a (team_a_id)                   -- 自チーム側からの逆引き
INDEX idx_tf_team_b (team_b_id)                   -- 相手チーム側からの逆引き
INDEX idx_tf_established (established_at DESC)     -- 成立日時順
```

**制約・備考**
- `team_a_id < team_b_id` をアプリ層で強制する（CHECK 制約は MySQL 8.0 で利用可能だが、保守性のため Service 層のバリデーションとする）。これにより重複レコード判定と UNIQUE 制約の設計が単純化される
- `a_follow_id` / `b_follow_id` は監査・整合性チェック用に両方向の `follows.id` を保持する。`follows` から該当レコードが削除されたらトリガー不要で Service 層が `team_friends` を削除し、`follows` 側には FK 制約を付けない（ポリモーフィック参照のため）
- 論理削除なし。フレンド解除時は物理削除
- `updated_at` は将来的なメタデータ（承認ステータス等）拡張を想定して保持

#### `team_friend_folders`

**自チーム視点のみ**の非対称フォルダ。A チームが B チームを「系列校」フォルダに入れていても、B チームの A に対する分類はまったく別の構造となる。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams。ON DELETE CASCADE。このフォルダを所有するチーム（= 自チーム） |
| `name` | VARCHAR(50) | NO | — | フォルダ名（例: '系列校', '姉妹団体', '練習試合候補') |
| `color` | VARCHAR(7) | NO | '#6B7280' | 表示色（HEX。UI バッジ色として使用） |
| `sort_order` | INT | NO | 0 | 並び替え順（小さいほど上に表示） |
| `description` | VARCHAR(300) | YES | NULL | フォルダの説明（任意） |
| `auto_forward_enabled` | BOOLEAN | NO | FALSE | TRUE の場合、このフォルダ内のフレンドチーム発コンテンツを**自動で転送**（ただし初期運用では FALSE 推奨） |
| `auto_forward_target` | VARCHAR(30) | YES | NULL | 自動転送時の配信範囲。'MEMBER' または 'MEMBER_AND_SUPPORTER'。`auto_forward_enabled = FALSE` の場合 NULL |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
UNIQUE KEY uq_tff_team_name (team_id, name, deleted_at)   -- 論理削除対応のチーム内フォルダ名一意性
INDEX idx_tff_team_sort (team_id, sort_order)              -- 自チームのフォルダ一覧（並び替え順）
```

**制約・備考**
- `UNIQUE (team_id, name, deleted_at)` は MySQL の `NULL != NULL` 特性を利用した論理削除対応パターン（F06.1 と同方針）
- `auto_forward_enabled = TRUE` の場合、`auto_forward_target` は必須（CHECK 制約相当をアプリ層で検証）
- フォルダ上限: 1チームあたり **最大20フォルダ**（アプリ層で検証。超過時は 409）
- フォルダ削除時は、`team_friend_folder_members` の関連レコードを CASCADE で削除（フレンド関係自体は維持される）

#### `team_friend_folder_members`

フレンドフォルダとフレンドチームの多対多中間テーブル。**1つのフレンドチームは複数フォルダに所属可能**（例: 「系列校」かつ「練習試合候補」）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `folder_id` | BIGINT UNSIGNED | NO | — | FK → team_friend_folders。ON DELETE CASCADE |
| `team_friend_id` | BIGINT UNSIGNED | NO | — | FK → team_friends。ON DELETE CASCADE |
| `added_by` | BIGINT UNSIGNED | YES | NULL | FK → users（追加実行者; SET NULL on delete） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_tffm_folder_friend (folder_id, team_friend_id)   -- 同一フレンドを同じフォルダに二重登録防止
INDEX idx_tffm_folder (folder_id)
INDEX idx_tffm_friend (team_friend_id)
```

**制約・備考**
- フォルダ所有チームとフレンド関係の整合性（フォルダの `team_id` が `team_friends.team_a_id` または `team_b_id` のどちらかと一致すること）をアプリ層で検証
- フレンド解除時（`team_friends` が削除される時）は CASCADE により自動で関連レコードも削除される

#### `friend_content_forwards`

管理者が「転送」を実行した履歴。冪等性担保（二重転送防止）と監査ログとして使用する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `source_post_id` | BIGINT UNSIGNED | NO | — | FK → timeline_posts（転送元投稿。ON DELETE CASCADE） |
| `source_team_id` | BIGINT UNSIGNED | NO | — | FK → teams（転送元チーム = フレンドチーム側）。ON DELETE CASCADE |
| `forwarding_team_id` | BIGINT UNSIGNED | NO | — | FK → teams（転送実行チーム = 自チーム）。ON DELETE CASCADE |
| `forwarded_post_id` | BIGINT UNSIGNED | NO | — | FK → timeline_posts（転送で生成された自チーム内投稿。ON DELETE CASCADE） |
| `target` | VARCHAR(30) | NO | 'MEMBER' | 配信範囲。'MEMBER' / 'MEMBER_AND_SUPPORTER' |
| `forwarded_by` | BIGINT UNSIGNED | NO | — | FK → users（転送操作を行った ADMIN / DEPUTY_ADMIN）。ON DELETE RESTRICT |
| `comment` | VARCHAR(500) | YES | NULL | 転送時に管理者が付与したコメント（任意） |
| `revoked_at` | DATETIME | YES | NULL | 転送取消日時（NULL = 転送中） |
| `revoked_by` | BIGINT UNSIGNED | YES | NULL | FK → users（取消実行者。SET NULL on delete） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_fcf_source_forwarding (source_post_id, forwarding_team_id, revoked_at)
    -- 同一転送元×同一転送先の重複防止（revoked_at 込みで論理削除対応）
INDEX idx_fcf_forwarding_team (forwarding_team_id, created_at DESC)   -- 自チームの転送履歴
INDEX idx_fcf_source_team (source_team_id, created_at DESC)           -- フレンド側の被転送履歴（監査）
INDEX idx_fcf_forwarded_post (forwarded_post_id)                       -- 転送投稿からの逆引き
```

**制約・備考**
- 冪等性: 同じ `source_post_id` を同じ `forwarding_team_id` で2回転送しようとすると 409 Conflict（UNIQUE 制約違反）
- 取消実装: 物理削除ではなく `revoked_at` を設定するソフトデリート。取消後に再度転送可能（UNIQUE 制約が `revoked_at` を含むため）
- 取消時は `forwarded_post_id` の投稿を論理削除する（`timeline_posts.deleted_at` 設定）。ただし履歴は残す
- 1つの転送元投稿が削除された場合: 関連転送投稿を連鎖削除（CASCADE）。読み手側では「転送元が削除されました」のプレースホルダ表示

### ER図（テキスト形式）

```
teams (1) ──── (N) team_friends [team_a_id]
teams (1) ──── (N) team_friends [team_b_id]
follows (1) ──── (1) team_friends [a_follow_id]
follows (1) ──── (1) team_friends [b_follow_id]

teams (1) ──── (N) team_friend_folders [team_id]
team_friend_folders (1) ──── (N) team_friend_folder_members [folder_id]
team_friends (1) ──── (N) team_friend_folder_members [team_friend_id]

timeline_posts (1) ──── (N) friend_content_forwards [source_post_id]
timeline_posts (1) ──── (1) friend_content_forwards [forwarded_post_id]
teams (1) ──── (N) friend_content_forwards [source_team_id]
teams (1) ──── (N) friend_content_forwards [forwarding_team_id]
users (1) ──── (N) friend_content_forwards [forwarded_by / revoked_by]
```

## 5. API設計

### エンドポイント一覧

| メソッド | パス | 認証 | 説明 |
|---------|-----|------|------|
| POST | `/api/v1/teams/{id}/friends/follow` | 必要 | 他チームをフォロー（TEAM → TEAM の follows を INSERT） |
| DELETE | `/api/v1/teams/{id}/friends/follow/{targetTeamId}` | 必要 | 他チームのフォローを解除 |
| GET | `/api/v1/teams/{id}/friends` | 必要 | フレンドチーム一覧取得 |
| GET | `/api/v1/teams/{id}/friends/pending` | 必要 | 自チームが片方向フォロー中で未成立の相手一覧 |
| GET | `/api/v1/teams/{id}/friend-folders` | 必要 | フレンドフォルダ一覧 |
| POST | `/api/v1/teams/{id}/friend-folders` | 必要 | フォルダ作成 |
| PUT | `/api/v1/teams/{id}/friend-folders/{folderId}` | 必要 | フォルダ更新 |

#### `POST /api/v1/teams/{id}/friends/follow`

**認可**: `{id}` チームの ADMIN、または `MANAGE_FRIEND_TEAMS` 権限を持つ DEPUTY_ADMIN

指定した他チームをフォローする（`follows` テーブルに `(TEAM, {id}, TEAM, target_team_id)` を INSERT）。相手チームが既に自チームをフォロー済みだった場合、自動的にフレンド関係が成立する。

**リクエストボディ**
```json

  "target_team_id": 42,
  "comment": "練習試合でお世話になっているチームです"
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `target_team_id` | Long | Yes | フォロー先チーム ID |
| `comment` | String | No | フォロー時に相手チーム管理者に表示される挨拶メッセージ（最大300文字。フレンド成立通知に含まれる） |

**レスポンス（201 Created）**
```json
{
  "data": {
    "follow_id": 1001,
    "follower_team_id": 7,
    "followed_team_id": 42,
    "is_mutual": true,
    "team_friend": {
      "id": 55,
      "team_a_id": 7,
      "team_b_id": 42,
      "established_at": "2026-04-15T10:00:00"
    },
    "created_at": "2026-04-15T10:00:00"
  }
}
```

**補足**
- `is_mutual = true` の場合、相互フォロー成立として同時に `team_friends` にレコードが INSERT される
- `is_mutual = false` の場合、相手チームのフォロー待ち状態。`team_friends` は生成されない
- 相互フォロー成立時、両チームの ADMIN / DEPUTY_ADMIN（`MANAGE_FRIEND_TEAMS` 保持者）に `FRIEND_ESTABLISHED` 通知を送信

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `target_team_id` 未指定 / 自チームへのフォロー |
| 403 | ADMIN / DEPUTY_ADMIN（`MANAGE_FRIEND_TEAMS` 保持）でない / `platform_settings.friend_team_enabled = false` |
| 404 | `target_team_id` のチームが存在しない（論理削除含む） |
| 409 | 既にフォロー済み |
| 429 | フォロー レート制限（1チーム1時間に30回まで） |

#### `DELETE /api/v1/teams/{id}/friends/follow/{targetTeamId}`

**認可**: `{id}` チームの ADMIN、または `MANAGE_FRIEND_TEAMS` 権限を持つ DEPUTY_ADMIN

指定した他チームへのフォローを解除する。相互フォロー状態だった場合は **自動的にフレンド関係も解除** される。

**レスポンス（204 No Content）**

**補足**
- フレンド解除時の処理:
  1. `follows` から A → B のレコードを DELETE
  2. `team_friends` から該当ペアのレコードを DELETE（物理削除）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | ADMIN / DEPUTY_ADMIN（`MANAGE_FRIEND_TEAMS` 保持）でない |
| 404 | フォロー関係が存在しない |

## 6. ビジネスロジック

### 主要フロー

#### 相互フォロー検知・フレンド成立

```
1. POST /api/v1/teams/{id}/friends/follow を呼び出し
2. バックエンドが以下を検証:
   a. 認証ユーザーが {id} チームの ADMIN または MANAGE_FRIEND_TEAMS 保持 DEPUTY_ADMIN
   b. target_team_id が存在し、論理削除されていない
   c. 自己フォロー（{id} == target_team_id）でない
   d. 重複フォローでない（UNIQUE 制約で保証）
   e. platform_settings.friend_team_enabled = TRUE
3. follows に INSERT（follower_type='TEAM', follower_id={id}, followed_type='TEAM', followed_id=target_team_id）
4. 対称性チェック:
   SELECT 1 FROM follows
   WHERE follower_type='TEAM' AND follower_id=target_team_id
     AND followed_type='TEAM' AND followed_id={id}
5. 対称性成立時:
   a. team_friends に INSERT（team_a_id = min(id, target), team_b_id = max, a_follow_id, b_follow_id）
   b. 両チームの ADMIN / MANAGE_FRIEND_TEAMS 保持 DEPUTY_ADMIN に FRIEND_ESTABLISHED 通知を生成
   c. Valkey キャッシュ無効化（friend:list:{team_a_id}, friend:list:{team_b_id}）
6. レスポンスに is_mutual フラグを含めて返却
```

#### フォロー解除・フレンド解除

```
1. DELETE /api/v1/teams/{id}/friends/follow/{targetTeamId} を呼び出し
2. バックエンドが以下を検証:
   a. 認証・権限（上記と同じ）
   b. follows に該当レコードが存在
3. トランザクション開始:
   a. follows から DELETE（follower_id={id}, followed_id=targetTeamId の行）
   b. team_friends から DELETE（team_a_id/team_b_id が {id,targetTeamId} ペアの行）
      → CASCADE で team_friend_folder_members も削除される
   c. friend_content_forwards は保持（履歴価値）。ただし以降の閲覧時は「フレンド解除済み」注釈付き
   d. 両チームに FRIEND_DISSOLVED 通知を生成
   e. Valkey キャッシュ無効化
4. コミット
5. 204 を返却
```

> **注意**: B → A のフォローは自動削除しない。相手側の意思を尊重し、片方向フォロー状態が残る。

#### 二段階配信: フレンドチーム発コンテンツの受信

```
[ フレンドチーム A のタイムライン投稿発生時 ]
1. timeline_posts に scope_type='TEAM', scope_id=A_id で INSERT
2. 投稿配信フローの拡張として:
   a. team_friends を検索: A_id を含むペア全てを取得
   b. 各フレンドチーム B に対し、以下を実行:
      i.   timeline_posts に「シャドーエントリ」を INSERT しない（= 物理コピーを作らない設計を採用）
      ii.  代わりに friend_feed_items という仮想ビュー、またはクエリ時結合で実現
         → 実装選択肢（推奨: クエリ時結合）
         A: 物理コピー案: scope_type='FRIEND_TEAM', scope_id=B_id として別レコード
         B: クエリ時結合案: GET friend-feed 実行時に team_friends + timeline_posts を JOIN
         → Phase 1 は B 案（ストレージ節約・整合性自動維持）、Phase 3 で性能要件に応じて A 案に移行検討
   c. 通知生成: B チームの ADMIN / MANAGE_FRIEND_TEAMS 保持 DEPUTY_ADMIN に対し
      notifications INSERT（scope_type='FRIEND_TEAM', scope_id=B_id, source_type='TIMELINE_POST', source_id={原投稿ID}）
   d. Push 通知チャネル選択は F04.3 の既定ルールに従う
3. B チーム内の MEMBER や SUPPORTER には**何も送らない**（ノイズ制御の要諦）
```

#### 二段階配信: 管理者による転送実行

```
1. ADMIN が GET /api/v1/teams/B/friend-feed で受信投稿を確認
2. 転送判断: POST /api/v1/teams/B/friend-feed/{postId}/forward
   body: { target: 'MEMBER' or 'MEMBER_AND_SUPPORTER', comment }
3. バックエンドが以下を実行:
   a. friend_content_forwards に INSERT（冪等性チェック: source_post_id × forwarding_team_id で重複確認）
   b. 新たな timeline_posts を生成:
      - scope_type = 'FRIEND_FORWARD'
      - scope_id = forwarding_team_id (= B_id)
      - content = comment + 元投稿の抜粋
      - metadata = { source_post_id, source_team_id, source_team_name, target }
   c. friend_content_forwards.forwarded_post_id に新投稿 ID を紐付け
   d. 通知配信:
      - target='MEMBER': B チームの MEMBER 全員に FRIEND_FORWARD_RECEIVED 通知
      - target='MEMBER_AND_SUPPORTER': MEMBER + SUPPORTER
   e. Valkey Sorted Set（timeline:FRIEND_FORWARD:{B_id}）に ZADD
   f. WebSocket 配信（B チーム内メンバー向け）
4. 201 Created を返却
```

#### 二段階配信: 転送取消

```
1. DELETE /api/v1/teams/B/friend-feed/{postId}/forward/{forwardId}
2. バックエンドが以下を実行:
   a. friend_content_forwards.revoked_at = NOW(), revoked_by = ユーザーID
   b. 対応する timeline_posts（FRIEND_FORWARD 種別）を論理削除
   c. 既送信通知は削除しない（「取り消されました」と表示）
   d. Valkey から ZREM
3. 204 を返却
```

### 重要な判定ロジック

- **フレンド関係のキャッシュ**: Valkey `friend:list:{teamId}` に Set 型でフレンド相手のチーム ID 群を保持。TTL 1時間。フォロー/フォロー解除時に両チーム分を INVALIDATE
- **管理者フィードの取得クエリ**: `team_friends` テーブルで自チームの相手チーム ID 一覧を取得後、`timeline_posts WHERE scope_type='TEAM' AND scope_id IN (相手チーム ID 群) AND deleted_at IS NULL` を UNION で抽出（Phase 1 = クエリ時結合）
- **通知送信時の対象ユーザー解決**: `MANAGE_FRIEND_TEAMS` 権限保持者を `user_roles` + `user_permission_groups` + `permission_group_permissions` の JOIN で解決
- **転送対象範囲の判定**: `friend_content_forwards.target` を `timeline_posts` 表示フィルタで利用。GET /timeline のレスポンス絞り込み時に `scope_type='FRIEND_FORWARD'` の投稿について、閲覧者のロールが target の許可範囲内かチェック
- **自動転送フォルダ（`auto_forward_enabled = TRUE`）**: 現段階では非推奨（情報過多のリスク高）。設計だけ残し、Phase 3 での提供時に詳細フロー確定

### スケーラビリティ

| 観点 | 対策 |
|------|-----|
| フレンド関係検索の高頻度化 | Valkey `friend:list:{teamId}` によるキャッシュ。TTL 1時間 + 書き込み時 INVALIDATE |
| 管理者フィードのフィード生成コスト | Phase 1 はクエリ時結合（コピー不要・整合性自動維持）。Phase 3 で1000 フレンド超えの大規模チームに備え物理コピー方式へ移行可能な設計 |
| 通知の大量発行 | 逆方向通知（フレンドフォルダ全員宛て）は非同期バッチ配信（Kafka or DB テーブルキュー）。同期API は 202 Accepted を返却し `queued_teams_count` のみ応答 |
| 整合性チェック | 日次バッチで `follows` の対称性と `team_friends` レコードの整合を突合。齟齬があれば自動修復 or アラート |
| レート制限 | フォロー操作: 1チーム1時間30回 / 通知送信: 1チーム1日100回 / フォルダ操作: 制限なし（軽量） |

## 8. 通知・プッシュ通知設計

### 新規通知種別（`NotificationType` enum 追加）

| 種別 | トリガー | 受信者 | デフォルト Push |
|------|---------|--------|-----|
| `FRIEND_ESTABLISHED` | 相互フォロー成立 | 両チームの ADMIN / MANAGE_FRIEND_TEAMS 保持 DEPUTY_ADMIN | NORMAL |
| `FRIEND_DISSOLVED` | フレンド解除（片方の解除で即発生） | 両チームの ADMIN / MANAGE_FRIEND_TEAMS 保持 DEPUTY_ADMIN | NORMAL |
| `FRIEND_FEED_POST_RECEIVED` | フレンドチームの新規投稿が管理者フィードに到着 | 受信チームの管理者 | NORMAL（集約配信: 1時間に1回まとめる） |
| `FRIEND_NOTIFICATION_RECEIVED` | フレンドチームから直接通知送信された（POST /friend-notifications/send） | 受信チームの管理者 | HIGH（即時配信） |
| `FRIEND_FORWARD_RECEIVED` | 自チーム ADMIN が転送を実行 | 配信範囲の MEMBER / SUPPORTER | NORMAL |
| `FRIEND_FORWARD_REVOKED` | 転送取消時 | 配信済み通知を受け取った MEMBER / SUPPORTER | LOW |

### 通知受信設定

- `notification_type_preferences` で個別種別の ON/OFF が可能（既存メカニズムを流用）
- `notification_preferences` の `scope_type` に `FRIEND_TEAM` / `FRIEND_FOLDER` を追加し、特定フレンドやフォルダ単位の受信制御を可能にする（Phase 3）
- `FRIEND_NOTIFICATION_RECEIVED` は `priority = HIGH` の場合は種別 OFF を無視して配信（安全側設計）

### 集約配信（rate-limiting digest）

- `FRIEND_FEED_POST_RECEIVED` は1時間に1回のまとめ通知方式がデフォルト（UX 配慮）
- 例: 「過去1時間で3件のフレンド投稿が届きました。管理者フィードをご確認ください」
- `platform_settings.friend_feed_digest_window_minutes` でウィンドウ幅調整可能（デフォルト 60）

## 9. セキュリティ・プライバシー

### チーム非公開コンテンツのフレンド可視性

- フレンドチームの TEAM スコープ投稿は、相手チーム内コンテンツであり**外部非公開が前提**
- 管理者フィードで受信する時点でも、元投稿の `visibility` 属性を尊重する
  - `visibility = PRIVATE` の投稿は**フレンド配信の対象外**
  - `visibility = MEMBERS_ONLY` の投稿も Phase 1 ではフレンド配信対象外（安全側設計）
  - フレンド配信対象となるのは、明示的に `share_with_friends = TRUE` マークされた投稿、または将来拡張の `visibility = FRIENDS_INCLUSIVE` 投稿のみ（Phase 3 で詳細仕様確定）
- Phase 1 のリリース時点では: 「フレンド配信用に明示的に投稿された通知」に限定。既存の TEAM 投稿は自動的にフレンド連携されない

### 転送時のプライバシー確認

- 転送 API 実行時、元投稿に個人を特定できるフィールド（実名・添付画像内の顔）が含まれる場合は、UI で警告を表示（Phase 3 で AI 判定を検討）
- 転送投稿のメタデータに「転送元フレンドチーム名」を必ず含める。匿名化しない（透明性担保）
- 転送元チーム側からは、自分の投稿がどのフレンドチームに転送されたか履歴を確認可能（`friend_content_forwards.source_team_id` からの逆引き API を Phase 3 で提供）

### 悪意ある大量フォローからの保護

- レート制限: 1チーム1時間に30回のフォロー操作（429 レスポンス）
- アンチスパム: 24時間以内に同じチームからのフォロー→解除を3回繰り返した場合、そのチームを**フォロー拒否リスト**に自動追加（F04.5 モデレーション連携）
- SYSTEM_ADMIN は `DELETE /api/v1/admin/team-friends/{teamFriendId}` で強制解除可能（監査ログ `FRIEND_FORCE_DISSOLVED` 記録）
- 新規設立から3日以内のチームはフォロー送信不可（スパム対策）

### 監査ログ

| イベント | audit_logs イベント名 |
|---------|-----|
| フォロー実行 | `TEAM_FOLLOW` |
| フォロー解除 | `TEAM_UNFOLLOW` |
| フレンド成立 | `FRIEND_ESTABLISHED` |
| フレンド解除 | `FRIEND_DISSOLVED` |
| 管理者強制解除 | `FRIEND_FORCE_DISSOLVED` |
| 転送実行 | `FRIEND_CONTENT_FORWARDED` |
| 転送取消 | `FRIEND_CONTENT_FORWARD_REVOKED` |
| フレンド通知送信 | `FRIEND_NOTIFICATION_SENT` |

## 13. セキュリティ考慮事項

- **所有権チェック**: フォロー/フォロー解除 API は `{id}` チームの ADMIN / MANAGE_FRIEND_TEAMS 保持 DEPUTY_ADMIN であることを必ず検証
- **フレンドチームのみ閲覧可**: 管理者フィード・フレンド通知・転送 API は **フレンド関係が成立している場合のみ** 機能する（`team_friends` 存在チェック必須）
- **転送対象範囲の厳密化**: `target = 'MEMBER_AND_SUPPORTER'` で転送された投稿は、SUPPORTER ロールのユーザーも閲覧可。逆に `target = 'MEMBER'` の場合 SUPPORTER は閲覧不可（タイムライン表示時のロールチェック必須）
- **GDPR 相当の同意**: フレンド関係成立時に両チームの ADMIN が「自チームの投稿・通知がフレンドチームに共有される可能性がある」ことを理解している状態で成立させる（規約同意フロー）
- **凍結されたチームはフォロー不可**: `teams.deleted_at IS NOT NULL` または `archived_at IS NOT NULL` のチームはフォロー/フォロー対象から除外
- **通報連携**: `content_reports.target_type` に `TEAM_FRIEND_RELATION` を追加し、不適切なフレンド関係を通報可能とする（Phase 3）

## 14. 懸念事項・残課題

### 設計上の懸念

- **大規模環境でのパフォーマンス**: 1チームあたり1000フレンドを超える運用が想定される組織（地域連合・大規模教育法人等）で、管理者フィードのクエリ時結合がボトルネックにならないか。Phase 3 で負荷試験後に物理コピー方式への移行を検討
- **片方向フォロー状態の UX**: 相手が気づかずに承認しない状態が長期化した場合のフォロー解除フローを明示すべきか（自動タイムアウト機能の要否）
- **フォルダの同期**: A チームが B をフォルダ X に入れた後、フレンド解除→再成立した場合にフォルダ情報を復元するか（現設計では失われる）。復元ポリシーを Phase 2 で確定
- **SUPPORTER の視認範囲拡大**: 転送 `MEMBER_AND_SUPPORTER` 時にサポーターが他団体情報を見られる状態は、プライバシー観点で慎重に扱う。利用規約での明文化が必要

### 将来拡張への布石

- **非対称な「親子関係」・「パートナーシップ」**: 現設計は対等な相互フォローだが、将来的に「上位組織→下位組織」の階層関係（例: 本部→支部）を別機能として追加する可能性。`team_friend_relation_type` ENUM カラムの追加で対応可能な設計
- **組織間フレンド（Org-to-Org）**: 本機能の拡張として、`follows` の `TEAM` に加えて `ORGANIZATION` 値を受け入れれば組織単位フレンドも実現可能。現設計の影響なく拡張可能
- **フレンド限定イベント・マッチング**: F08.1 マッチングとの統合で、`match_requests.visibility` に `FRIEND_TEAMS_ONLY` を追加
