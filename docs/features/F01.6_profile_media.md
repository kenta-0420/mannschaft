# F01.6 プロフィールメディア（アイコン・バナー管理）

---

## ステータス: 設計完了

---

## 1. 概要・背景

YouTube チャンネルページのように、**個人・チーム・組織のダッシュボードにアイコン画像（チャンネルアイコン相当）とバナー画像（チャンネルアート相当）を設定・表示できる**機能。

### 1.1 既存状況

| スコープ | アイコン DB | バナー DB | 備考 |
|---------|------------|----------|------|
| ユーザー（個人） | `users.avatar_url` ✅ 実装済 | ❌ 未追加 | F01.1 §2 |
| チーム | ❌ 未追加 | ❌ 未追加 | F01.2 §3 に計画済・DB 未実装 |
| 組織 | ❌ 未追加 | ❌ 未追加 | F01.2 §4 に計画済・DB 未実装 |

F01.2 設計書が `icon_url` / `banner_url` の追加を計画しているが、  
Flyway マイグレーションおよび API が未実装のため、**本フェーズ（F01.6）で一括実装する**。

---

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| **アイコン** | プロフィールを代表する正方形の小画像（ユーザー: アバター / チーム・組織: ロゴ相当）|
| **バナー** | ダッシュボードヘッダーに表示される横長装飾画像（YouTube チャンネルアート相当）|
| **R2 キー** | Cloudflare R2 オブジェクトの識別子。DB には完全 URL でなくキーを保存する |
| **コミット** | クライアントが R2 への直接アップロード完了後、サーバーに r2Key を通知して DB を更新する操作 |
| **孤立オブジェクト** | コミットされず DB に紐付かない R2 オブジェクト（クライアント障害等で発生）|

---

## 3. 機能要件

### 3.1 アイコン（チャンネルアイコン相当）

- プロフィールページ・ダッシュボードヘッダー・メンバー一覧・通知・チャットなど全体で参照される
- 正方形で表示（40×40 ～ 200×200 px）
- 未設定時はスコープ別デフォルト表示（§9.3 参照）

### 3.2 バナー（チャンネルアート相当）

- 各スコープのダッシュボード / プロフィールページのヘッダーに表示
- 横長（推奨アスペクト比 3:1 / 推奨サイズ 1500×500 px）
- 未設定時はテンプレートカラーのグラデーション背景を表示

### 3.3 トリミング・リサイズ方針

| 処理 | 担当 | 仕様 |
|------|------|------|
| **アイコン** | **フロントエンド** | アップロード前にブラウザ上でユーザーが正方形クロップを確認・調整。確定後に R2 へ送信 |
| **バナー** | **フロントエンド** | 3:1 の推奨比率をガイド表示。クロップは任意（強制しない）。`object-fit: cover` でブラウザ側に表示を委ねる |
| **リサイズ** | **フロントエンド** | アイコンは 400×400 px 以上の場合にダウンスケールして送信（クライアント側 Canvas 処理）|

> サーバー側での画像変換は行わない（実装コスト・Workers コスト削減）

### 3.4 設定可能な操作

| 操作 | 内容 |
|------|------|
| **アップロード** | アップロード URL を発行し、クライアントが R2 に直接 PUT → コミット |
| **削除** | R2 オブジェクト削除 + DB カラムを NULL クリア |
| **差し替え** | アップロード + コミット時に旧オブジェクトを自動削除 |

## 5. アップロードフロー

```
① POST .../profile-media/{role}/upload-url
   → { uploadUrl, r2Key, expiresIn:600 }
   サーバーは r2Key を Valkey に pending_profile_media:{scope}:{scopeId}:{role}:r2key で TTL=700s 保存

② クライアントが R2 に直接 PUT

③ PUT .../profile-media/{role}  { r2Key }
   → DB 更新（旧 r2Key があれば R2 から削除）
   → Valkey の pending キーを削除
```

### 孤立オブジェクト対策

②成功後③が呼ばれなかった場合（クライアント障害等）の対策:

- アップロード URL 発行時（①）に `profile_media_pending` テーブル（または既存の一時テーブル）に  
  `(scope, scopeId, role, r2Key, createdAt)` を INSERT する
- コミット成功時（③）に対応レコードを DELETE
- 日次バッチ `cleanupOrphanProfileMedia()` が `createdAt < NOW() - INTERVAL 12 HOUR` の  
  残存レコードに対し R2 オブジェクトを削除してレコードを物理削除（毎日 02:45 実行）
- `@Scheduled(cron = "0 45 2 * * *")` を `ProfileMediaService` に実装

**Valkey SCAN を使わない理由**: 本番環境でのキースキャンはパフォーマンスリスクがあるため、  
孤立管理は DB テーブルで行う。

### アップロード URL 期限切れ時の再試行

Presigned PUT URL は 600 秒で失効する。クライアントが R2 PUT で 403 を受け取った場合は、  
再度 `POST .../upload-url` を呼び出して新しい URL を取得し、同ファイルを再送する（クライアント側対応）。  
サーバー側では古い pending レコードはバッチで自動削除されるため特段の処理は不要。

## 6. API 設計

### 6.1 エンドポイント一覧

| メソッド | パス | 説明 |
|---------|------|------|
| `POST` | `/api/v1/users/me/profile-media/{role}/upload-url` | ユーザー自身の画像アップロード URL 発行 |
| `PUT` | `/api/v1/users/me/profile-media/{role}` | ユーザー自身の画像コミット（R2 アップロード完了後）|
| `DELETE` | `/api/v1/users/me/profile-media/{role}` | ユーザー自身の画像削除 |
| `POST` | `/api/v1/teams/{teamId}/profile-media/{role}/upload-url` | チーム画像アップロード URL 発行 |
| `PUT` | `/api/v1/teams/{teamId}/profile-media/{role}` | チーム画像コミット |
| `DELETE` | `/api/v1/teams/{teamId}/profile-media/{role}` | チーム画像削除 |
| `POST` | `/api/v1/organizations/{orgId}/profile-media/{role}/upload-url` | 組織画像アップロード URL 発行 |
| `PUT` | `/api/v1/organizations/{orgId}/profile-media/{role}` | 組織画像コミット |
| `DELETE` | `/api/v1/organizations/{orgId}/profile-media/{role}` | 組織画像削除 |

全エンドポイント: `@PreAuthorize("isAuthenticated()")`

`{role}` の有効値: `icon` / `banner`

### 6.2 アップロード URL 発行

**リクエスト** `POST .../profile-media/{role}/upload-url`
```json
{
  "contentType": "image/jpeg",
  "fileSize": 2097152
}
```

| フィールド | 型 | 必須 | 制約 |
|-----------|-----|------|------|
| `contentType` | String | ✅ | `image/jpeg`, `image/png`, `image/webp`, `image/gif`（gif はアイコンのみ）|
| `fileSize` | Long | ✅ | アイコン: 1 ～ 5,242,880 B（5 MB）/ バナー: 1 ～ 10,485,760 B（10 MB）|

**レスポンス** `200 OK`
```json
{
  "uploadUrl": "https://r2.example.com/...",
  "r2Key":     "teams/1/icon/550e8400-e29b-41d4-a716.jpg",
  "expiresIn": 600
}
```

### 6.3 コミット（DB 反映）

**リクエスト** `PUT .../profile-media/{role}`
```json
{
  "r2Key": "teams/1/icon/550e8400-e29b-41d4-a716.jpg"
}
```

**r2Key 検証ルール（セキュリティ）**:  
受信した `r2Key` が `{scope}/{scopeId}/{role}/` で始まること（例: `teams/1/icon/`）を厳密に検証する。  
一致しない場合は 403 を返す。これにより他スコープのオブジェクトを横取りできない。

- 旧キーが存在する場合は R2 から即時削除
- 旧 R2 削除失敗はログ警告のみ（DB 更新は続行）
- Valkey の pending キーを削除

**レスポンス** `200 OK`
```json
{
  "mediaRole": "icon",
  "url": "https://cdn.mannschaft.app/teams/1/icon/550e8400.jpg",
  "expiresAt": "2026-04-17T17:00:00Z"
}
```

> `url` は署名付き URL（有効期限: 3600 秒）をサービス層で生成して返す

### 6.4 削除

**リクエスト** `DELETE .../profile-media/{role}`（ボディなし）

**レスポンス** `204 No Content`

- R2 オブジェクトを削除 → DB カラムを NULL クリア
- 既に NULL の場合は 204 を返す（冪等）

### 6.5 エラーレスポンス

| ケース | HTTP | 説明 |
|--------|------|------|
| スコープ不在（チーム/組織が見つからない）| 404 | TEAM_NOT_FOUND / ORG_NOT_FOUND |
| `role` が `icon` / `banner` 以外 | 400 | INVALID_MEDIA_ROLE |
| MIME タイプ不正 | 400 | UNSUPPORTED_CONTENT_TYPE |
| GIF をバナーに指定 | 400 | GIF_NOT_ALLOWED_FOR_BANNER |
| ファイルサイズ超過 | 400 | FILE_SIZE_EXCEEDED |
| r2Key プレフィックス不一致（コミット時）| 403 | INVALID_R2_KEY |
| 権限なし | 403 | PROFILE_MEDIA_FORBIDDEN |

## 7. R2 ストレージ設計

### 7.1 キーパターン

```
users/{userId}/icon/{uuid}.{ext}              ← ユーザーアイコン（アバター）
users/{userId}/banner/{uuid}.{ext}            ← ユーザーバナー
teams/{teamId}/icon/{uuid}.{ext}              ← チームアイコン
teams/{teamId}/banner/{uuid}.{ext}            ← チームバナー
organizations/{orgId}/icon/{uuid}.{ext}       ← 組織アイコン
organizations/{orgId}/banner/{uuid}.{ext}     ← 組織バナー
```

### 7.2 ファイル制約

| 区分 | MIME タイプ | 最大サイズ | 推奨サイズ | アスペクト比 |
|------|-----------|-----------|-----------|------------|
| アイコン | JPEG / PNG / WebP / GIF | 5 MB | 400×400 px | 1:1 |
| バナー | JPEG / PNG / WebP（GIF 不可）| 10 MB | 1500×500 px | 3:1 |

> バナーへの GIF を禁止する理由: アニメーション GIF は帯域・パフォーマンスに影響が大きく、  
> ダッシュボードの視認性低下を防ぐため。UI 上でファイル選択時に `.gif` を accept から除外する。

### 7.3 Presigned URL TTL

| 用途 | 有効期限 |
|------|---------|
| アップロード用 PUT URL | 600 秒 |
| 表示用 GET URL（レスポンス）| 3600 秒 |

### 7.4 表示用 URL キャッシュ（Valkey）

| キー形式 | TTL | 備考 |
|---------|-----|------|
| `profile:icon:user:{userId}` | 3500 秒 | ユーザーアイコン URL キャッシュ |
| `profile:banner:user:{userId}` | 3500 秒 | ユーザーバナー URL キャッシュ |
| `profile:icon:team:{teamId}` | 3500 秒 | チームアイコン URL キャッシュ |
| `profile:banner:team:{teamId}` | 3500 秒 | チームバナー URL キャッシュ |
| `profile:icon:org:{orgId}` | 3500 秒 | 組織アイコン URL キャッシュ |
| `profile:banner:org:{orgId}` | 3500 秒 | 組織バナー URL キャッシュ |

スコープをすべて lowercase で区別し、`user` / `team` / `org` の衝突を排除する。  
コミット・削除時は対応するキャッシュキーを即時 Evict する。  
Evict は `@CacheEvict`（Spring Cache + Valkey）で実装するため、  
複数インスタンス構成でも全ノードのキャッシュが一致する（Valkey は単一共有 KVS）。

### 7.5 旧ファイル削除ポリシー

- **コミット時**: 旧 r2Key が DB に存在する場合、R2 オブジェクトを即時削除
- **削除 API 呼び出し時**: 即時削除
- **孤立オブジェクト（コミット未到達）**: Valkey の `pending_profile_media:*` キーを日次バッチで走査し、TTL 切れのものを R2 から削除（毎日 02:45 実行）

### 7.6 MultipartUploadService の ALLOWED_PREFIXES 更新

プロフィールメディアは最大 10 MB のため Multipart Upload は使用しない（`R2StorageService.generateUploadUrl()` のみ）。  
ただし将来の拡張に備え、`MultipartUploadService.ALLOWED_PREFIXES` に以下を追加する:

```java
// MultipartUploadService.java の ALLOWED_PREFIXES Set に追加
"users/", "teams/", "organizations/"
```

4. R2 に PUT（進捗バー表示）
5. `PUT .../profile-media/icon` でコミット
6. UI を即時更新（楽観的更新）

## 11. 実装構成

### 11.1 バックエンド

```
app/profile/
  controller/
    UserProfileMediaController.java      (POST/PUT/DELETE /users/me/profile-media/{role})
    TeamProfileMediaController.java      (POST/PUT/DELETE /teams/{id}/profile-media/{role})
    OrgProfileMediaController.java       (POST/PUT/DELETE /organizations/{id}/profile-media/{role})
  service/
    ProfileMediaService.java             (共通ロジック: URL発行・コミット・削除・孤立クリーンアップ)
  dto/
    ProfileMediaUploadUrlRequest.java    (contentType, fileSize)
    ProfileMediaUploadUrlResponse.java   (uploadUrl, r2Key, expiresIn)
    ProfileMediaCommitRequest.java       (r2Key)
    ProfileMediaResponse.java            (mediaRole, url, expiresAt)
  enums/
    ProfileMediaScope.java               (USER / TEAM / ORGANIZATION)
    ProfileMediaRole.java                (ICON / BANNER)
```

`ProfileMediaService` は `R2StorageService` + Valkey（StringRedisTemplate）のみに依存。  
`MultipartUploadService` には依存しない（最大 10 MB のため Presigned PUT URL で十分）。

**ユーザースコープの命名対応**:  
ユーザーのアイコンは API では `role=icon` で受け付けるが、DB カラムは既存の `users.avatar_url` に書き込む。  
`ProfileMediaService` 内でスコープが `USER` かつロールが `ICON` の場合は `avatar_url` カラムを更新すること。

**ProfileMediaScope の再利用確認**:  
実装時に既存の `ScopeType` enum（TEAM / ORGANIZATION 等）が存在するか確認し、  
存在する場合は `ProfileMediaScope` を新設せず `ScopeType` を流用すること。

**コントローラー命名規則**:
- `UserProfileMediaController`
- `TeamProfileMediaController`
- `OrganizationProfileMediaController`（`Org` 略記ではなく `Organization` をフルで使う）

### 11.2 Flyway マイグレーション

```
V9.077__add_banner_url_to_users.sql
V9.078__add_icon_banner_url_to_teams.sql
V9.079__add_icon_banner_url_to_organizations.sql
```
