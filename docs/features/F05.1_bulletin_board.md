# F05.1: 掲示板

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 5
> **最終更新**: 2026-03-13
> **関連ドキュメント**: F05.3 電子印鑑（押印連携なし・将来検討）、F04.3 プッシュ通知（重要度別通知）

---

## 1. 概要

組織・チーム内のメンバーが目的別カテゴリのもとでスレッド形式の情報共有を行う掲示板機能。ネスト返信（ツリー構造）により文脈を保った議論ができ、5段階の重要度設定と既読トラッキングにより、重要な連絡の見落とし防止と確認状況の可視化を実現する。ピン留め・ロック・アーカイブ機能で運用上の情報整理をサポートし、リアクション（絵文字スタンプ）で軽量なフィードバックを可能にする。カテゴリ別の投稿権限制御、下書き自動保存、一括既読、画像サムネイルプレビューなどの利便性機能も備える。

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `bulletin_categories` | カテゴリ管理（連絡事項、相談、雑談等） | あり |
| `bulletin_threads` | スレッド（投稿本文 + メタデータ） | あり |

| `bulletin_read_status` | 既読トラッキング | なし |

| `bulletin_reactions` | スレッド/返信へのリアクション（絵文字スタンプ） | なし |

### テーブル定義

#### `bulletin_categories`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION') | NO | — | スコープ種別 |
| `scope_id` | BIGINT UNSIGNED | NO | — | FK → teams または organizations |
| `name` | VARCHAR(50) | NO | — | カテゴリ名（例: 連絡事項、相談、雑談） |
| `description` | VARCHAR(200) | YES | NULL | カテゴリの説明 |
| `display_order` | INT UNSIGNED | NO | 0 | 表示順序 |
| `color` | VARCHAR(7) | YES | NULL | カテゴリカラー（HEX形式: #FF5733） |
| `post_min_role` | ENUM('ADMIN', 'DEPUTY_ADMIN', 'MEMBER', 'SUPPORTER') | NO | 'MEMBER' | このカテゴリにスレッドを投稿できる最低ロール |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（作成者）。ON DELETE SET NULL（退会時にカテゴリは残す） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_bulletin_categories_scope (scope_type, scope_id, display_order)  -- スコープ別カテゴリ一覧
```

**制約・備考**
- `scope_type` によって `scope_id` の参照先が異なるため DB レベルの FK 制約は設けない（アプリ層で検証）
- カテゴリ名の重複防止は DB UNIQUE 制約ではなく **Service 層で `deleted_at IS NULL` の既存レコードと照合**して制御する（論理削除レコードと同名のカテゴリを再作成可能にするため。DB UNIQUE はソフトデリートと競合する）
- カテゴリ削除時: 配下のスレッドは「未分類」として残す（カテゴリとの紐付けを NULL にセット）
- 初期カテゴリ: チーム/組織作成時に「連絡事項」「相談」「雑談」の3件を自動作成

#### `bulletin_threads`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `category_id` | BIGINT UNSIGNED | YES | NULL | FK → bulletin_categories。ON DELETE SET NULL（カテゴリ削除時に未分類へ移行）。NULL = 未分類 |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION') | NO | — | スコープ種別 |
| `scope_id` | BIGINT UNSIGNED | NO | — | FK → teams または organizations |
| `author_id` | BIGINT UNSIGNED | NO | — | FK → users（投稿者）。ON DELETE RESTRICT |
| `title` | VARCHAR(200) | NO | — | スレッドタイトル |
| `body` | TEXT | NO | — | 本文（Markdown 対応） |
| `priority` | ENUM('CRITICAL', 'IMPORTANT', 'WARNING', 'INFO', 'LOW') | NO | 'INFO' | 重要度（5段階） |
| `read_tracking_mode` | ENUM('NONE', 'COUNT_ONLY', 'SHOW_READERS') | NO | 'COUNT_ONLY' | 既読表示モード |
| `is_pinned` | BOOLEAN | NO | FALSE | ピン留め状態 |
| `is_locked` | BOOLEAN | NO | FALSE | ロック状態（TRUE = 返信不可） |
| `is_archived` | BOOLEAN | NO | FALSE | アーカイブ状態（TRUE = 一覧のデフォルト表示から除外） |
| `reply_count` | INT UNSIGNED | NO | 0 | 返信数（denormalized。返信追加/削除時にアトミック更新） |
| `read_count` | INT UNSIGNED | NO | 0 | 既読者数（denormalized。既読記録時に +1、ユーザー退会時の減算はバッチで補正） |
| `last_replied_at` | DATETIME | YES | NULL | 最終返信日時（スレッド一覧のソート用） |
| `source_type` | VARCHAR(50) | YES | NULL | 自動生成元の機能種別（例: `SAFETY_CHECK`）。手動投稿の場合は NULL |
| `source_id` | BIGINT UNSIGNED | YES | NULL | 自動生成元のレコード ID |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_bulletin_threads_scope_pinned (scope_type, scope_id, is_pinned DESC, last_replied_at DESC)  -- スコープ別スレッド一覧（ピン留め優先 + 新着順）
INDEX idx_bulletin_threads_category (category_id, last_replied_at DESC)                                -- カテゴリ別スレッド一覧
INDEX idx_bulletin_threads_author (author_id)                                                          -- 投稿者別の検索
INDEX idx_bulletin_threads_source (source_type, source_id)                                             -- 自動生成元の逆引き
FULLTEXT INDEX ft_bulletin_threads_title (title)                                                       -- タイトル全文検索
```

**制約・備考**
- `reply_count` は denormalized カウンターキャッシュ。返信の INSERT/DELETE 時に `UPDATE bulletin_threads SET reply_count = reply_count ± 1` でアトミック更新（COUNT クエリ廃止）
- `read_count` は denormalized カウンターキャッシュ。既読記録時に +1。ユーザー退会による既読データ CASCADE 削除時は即時減算せず、90日バッチの実行時に COUNT で補正する（退会は低頻度のため許容）
- `last_replied_at` も返信追加時に更新。返信が論理削除された場合は直前の返信の `created_at` で再計算する（`SELECT MAX(created_at) FROM bulletin_replies WHERE thread_id = ? AND deleted_at IS NULL`）。返信が全て削除された場合は NULL に戻る。スレッド一覧の「最近の活動順」ソートでは `COALESCE(last_replied_at, created_at)` を使用し、返信なしスレッドも正しくソートする
- `source_type` + `source_id`: 安否確認機能からの自動生成スレッドを識別（`safety_checks.bulletin_thread_id` からの逆引きにも対応）
- `FULLTEXT` インデックスは MySQL FULLTEXT を使用し `MATCH(title) AGAINST(? IN BOOLEAN MODE)` で検索。将来的に本文検索が必要になった場合は `FULLTEXT(title, body)` への拡張で対応可能（検索ロジックは Service 層に集約し、インデックス変更のみで移行できる設計とする）
- 論理削除時: 配下の返信・既読データはそのまま保持（スレッド復元時に復活）。ただし既読データは90日バッチで物理削除される
- `category_id` が NULL のスレッドは「未分類」として扱う（カテゴリ削除時のフォールバック）

#### `bulletin_replies`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `thread_id` | BIGINT UNSIGNED | NO | — | FK → bulletin_threads。ON DELETE CASCADE |
| `parent_reply_id` | BIGINT UNSIGNED | YES | NULL | FK → bulletin_replies（自己参照）。ON DELETE SET NULL（親返信が物理削除された場合、子返信はスレッド直下に昇格）。NULL = スレッド直下の返信 |
| `author_id` | BIGINT UNSIGNED | NO | — | FK → users（投稿者）。ON DELETE RESTRICT |
| `body` | TEXT | NO | — | 返信本文（Markdown 対応） |
| `depth` | TINYINT UNSIGNED | NO | 0 | ネストの深さ（0 = スレッド直下、1 = 返信の返信、...）。最大5 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時（SoftDeletableEntity 適用） |

**インデックス**
```sql
INDEX idx_bulletin_replies_thread (thread_id, created_at ASC)           -- スレッド内の返信一覧（時系列）
INDEX idx_bulletin_replies_parent (parent_reply_id, created_at ASC)     -- 特定返信への子返信一覧
INDEX idx_bulletin_replies_author (author_id)                           -- 投稿者別の検索
```

**制約・備考**
- **ツリー構造**: `parent_reply_id` による自己参照で再帰的なネストを実現。`depth` カラムで深さを追跡し、最大5階層に制限（それ以上のネストは 400 エラー）
- **表示順**: ツリー構造の描画はフロントエンド側で `parent_reply_id` を元に再構築。API は `created_at ASC` のフラットリストで返却し、クライアントがネスト表示を組み立てる
- `thread_id` に ON DELETE CASCADE を設定 — スレッドの物理削除時に返信も連動削除（論理削除では CASCADE しない。論理削除は Service 層で個別制御）
- 論理削除された返信は「この返信は削除されました」として表示し、子返信のツリーは保持する（子返信が孤立しない設計）
- 返信追加時に `bulletin_threads.reply_count` を +1、論理削除時に -1（deleted_at をセットした時点でカウントから除外）

#### `bulletin_read_status`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `thread_id` | BIGINT UNSIGNED | NO | — | FK → bulletin_threads。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `read_at` | DATETIME | NO | CURRENT_TIMESTAMP | 既読日時 |

**インデックス**
```sql
UNIQUE KEY uq_bulletin_read_status_thread_user (thread_id, user_id)  -- 同一ユーザーの重複既読防止
INDEX idx_bulletin_read_status_user (user_id, read_at DESC)          -- ユーザー別の既読履歴
```

**制約・備考**
- `read_tracking_mode = 'NONE'` のスレッドでも既読レコードは INSERT する（未読/既読のフィルタリング用。表示モードは「既読UIをどう見せるか」の制御であり、記録自体は常に行う）
- ON DELETE CASCADE: スレッド削除時に既読データも連動削除
- **データ保持期限**: 既読データは投稿（`bulletin_threads.created_at`）から **90日後** にバッチで物理削除する（蓄積量削減）
- ON DELETE CASCADE（users）: ユーザー退会時に既読データも削除（既読カウントの正確性より、退会ユーザーのデータクリーンアップを優先）

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `target_type` | ENUM('THREAD', 'REPLY') | NO | — | 添付先種別 |
| `target_id` | BIGINT UNSIGNED | NO | — | 添付先の PK（bulletin_threads.id or bulletin_replies.id） |
| `file_name` | VARCHAR(255) | NO | — | 元ファイル名 |
| `file_key` | VARCHAR(500) | NO | — | S3 オブジェクトキー |

| `content_type` | VARCHAR(100) | NO | — | MIME タイプ |
| `is_image` | BOOLEAN | NO | FALSE | 画像ファイルかどうか（`content_type` が `image/*` の場合 TRUE。サムネイル表示判定用） |
| `uploaded_by` | BIGINT UNSIGNED | YES | NULL | FK → users（アップロード者）。ON DELETE SET NULL（退会時にファイルは残す） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_bulletin_attachments_target (target_type, target_id)  -- 添付先ごとの取得
INDEX idx_bulletin_attachments_uploaded_by (uploaded_by)         -- ユーザー別の添付一覧
```

**制約・備考**
- 1スレッド/返信あたり最大5ファイル（アプリ層で制御）
- 1ファイル最大6MB（S3 Pre-signed URL のアップロード時にサイズ制限を適用）
- ポリモーフィック参照（`target_type` + `target_id`）のため DB 制約の FK は設けない
- スレッド/返信の論理削除時に添付ファイルはそのまま保持（復元時にファイルが失われないため）。物理削除時に S3 オブジェクトも削除（バッチまたはイベントリスナー）
- `remove_attachment_ids` で個別削除された添付ファイルは `bulletin_attachments` から物理削除し、S3 オブジェクトも非同期で削除する（`@Async` + `S3Client.deleteObject`）
- 大容量ファイルの共有は F05.5 ファイル共有を利用する想定。掲示板の添付は軽量な資料共有（PDF、写真、Officeドキュメント等）用途

#### `bulletin_reactions`

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `target_type` | ENUM('THREAD', 'REPLY') | NO | — | リアクション対象種別 |
| `target_id` | BIGINT UNSIGNED | NO | — | 対象の PK（bulletin_threads.id or bulletin_replies.id） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `emoji` | VARCHAR(10) | NO | — | 絵文字コード（プリセット: 👍👏🙏😊❤️） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_bulletin_reactions_target_user_emoji (target_type, target_id, user_id, emoji)  -- 同一ユーザーの同一絵文字重複防止
INDEX idx_bulletin_reactions_target (target_type, target_id)                                   -- 対象ごとのリアクション集計
INDEX idx_bulletin_reactions_user (user_id)                                                    -- ユーザー別のリアクション一覧
```

**制約・備考**
- プリセット絵文字のみ許可（👍👏🙏😊❤️の5種）。カスタム絵文字は不可（アプリ層でバリデーション）
- ポリモーフィック参照（`target_type` + `target_id`）のため DB 制約の FK は設けない
- 同一ユーザーが同一対象に異なる絵文字を複数付与可能（1つの投稿に👍と❤️の両方を付けられる）
- リアクション追加/削除時に通知は発行しない（ノイズ防止）
- スレッド/返信の論理削除時にリアクションはそのまま保持（復元時に復活）。物理削除時に CASCADE で連動削除

### リクエスト／レスポンス仕様

#### `GET /api/v1/bulletin/categories`

スコープ内のカテゴリ一覧を表示順序で取得する。

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|

## 5. ビジネスロジック

### スレッド投稿フロー

```
1. ユーザーがスレッドを投稿
2. バリデーション:
   a. title: 1〜200文字
   b. body: 1〜10,000文字
   c. category_id: 指定時は同一スコープ（scope_type + scope_id が一致）内に存在するか検証
   d. SUPPORTER はスレッド作成不可（返信のみ）。さらにカテゴリの `post_min_role` でロール制限を検証
   d2. カテゴリ指定時: ユーザーのロールが `post_min_role` 以上であるか検証（例: `post_min_role = 'ADMIN'` のカテゴリには MEMBER は投稿不可）
   e. 添付ファイル: ファイル数 ≤ 5、各サイズ ≤ 6MB、content_type ホワイトリスト + ファイルヘッダ（マジックバイト）の二重検証
3. bulletin_threads へ INSERT
3a. 添付ファイルがある場合: S3 へアップロード → bulletin_attachments へ INSERT
4. 重要度に応じた通知を発行（ApplicationEvent: BulletinThreadCreatedEvent）:
   - CRITICAL: プッシュ通知 + LINE通知 + お知らせ欄 + ポップアップ
   - IMPORTANT: プッシュ通知 + お知らせ欄
   - WARNING: お知らせ欄
   - INFO: お知らせ欄（通常）
   - LOW: 通知なし（掲示板内でのみ閲覧可能）
5. メンション解析 → メンションされたユーザーへ個別通知
```

### 返信投稿フロー

```
1. ユーザーが返信を投稿（スレッド直下 or 既存返信への返信）
2. バリデーション:
   a. body: 1〜5,000文字
   b. スレッドがロック中でないことを確認（is_locked = false）
   c. ネスト返信の場合: 親返信の depth + 1 ≤ 5 を確認
3. bulletin_replies へ INSERT（depth = 親の depth + 1。スレッド直下は 0）
3a. 添付ファイルがある場合: S3 へアップロード → bulletin_attachments へ INSERT
4. bulletin_threads の reply_count を +1、last_replied_at を更新（アトミック UPDATE）
5. 通知対象を収集（スレッド作成者 + 親返信の作成者）し、重複を排除してから通知（ApplicationEvent: BulletinReplyCreatedEvent）。自分自身への通知はスキップ
6. メンション解析 → メンションされたユーザーへ個別通知（ステップ5と重複するユーザーは除外）
```

### 既読記録フロー

```
1. ユーザーがスレッド詳細を開く（GET /threads/{id}）→ サーバー側で暗黙的に既読を記録
   ※ GET の副作用を避けたい場合はフロントエンドから明示的に POST /threads/{id}/read を呼び出す方式も可。
     現設計では GET 詳細時に Service 層で自動記録する方式を採用（ユーザーの手間を省くため）
2. bulletin_read_status へ UPSERT（存在チェック後に INSERT。既存の場合はスキップ）
2a. 新規 INSERT の場合: bulletin_threads.read_count を +1（アトミック UPDATE）
3. read_tracking_mode に関わらず記録する:
   - NONE: 記録するが UI に表示しない
   - COUNT_ONLY: 「既読 N」の数字を表示
   - SHOW_READERS: アイコン一覧 + クリックでアカウント名表示
```

### 安否確認連携フロー

```
1. 安否確認が実行される → SafetyCheckCreatedEvent
2. BulletinService が購読:
   a. 対象スコープの掲示板に専用スレッドを自動生成
      - author_id: 安否確認を実行したユーザー（safety_checks.created_by）
      - タイトル: 「【安否確認】{実行日時 yyyy/MM/dd HH:mm}」
      - 優先度: CRITICAL
      - read_tracking_mode: SHOW_READERS
      - source_type: SAFETY_CHECK, source_id: safety_check.id
      - is_pinned: true（自動ピン留め）
   b. safety_checks.bulletin_thread_id に生成したスレッドの ID を保存
3. メンバーが安否回答 → SafetyCheckResponseEvent
   a. 専用スレッドに自動返信を追加（author_id = 回答したユーザー。本文: 「{ユーザー名}: {ステータス} — {メッセージ}」）
   b. BulletinReplyCreatedEvent を発行 → ポーリング経由でフロントエンドに反映（将来 WebSocket 導入時はリアルタイム配信に切り替え）
4. 安否確認クローズ → スレッドを自動ロック（is_locked = true）
```

### お知らせ掲示板への伝搬フロー（カスケード通知連携）

```
上位組織からのお知らせ（announcement）が掲示板に表示される仕組み:

1. 上位組織の ADMIN がお知らせを発行（announcement_scope を指定）
2. 各チーム/組織の掲示板ロード時にプルモデルで判定:
   a. 自組織の直接スレッド → 表示
   b. 上位組織からのお知らせで announcement_scope が自スコープを含む → 表示
3. 表示判定ルール:
   - SELF: 送信元組織の掲示板のみ
   - ORGANIZATIONS: 送信元 + サブツリー内の全子組織の掲示板
   - TEAMS: 送信元 + サブツリー内の全組織 + 全チームの掲示板

※ 伝搬ロジックの詳細は F01.2 組織・チーム設計書を参照

**パフォーマンス戦略**:
- 祖先組織の取得は再帰 CTE ではなく `group_hierarchy`（クロージャテーブル）の JOIN で O(1) 取得
- 祖先一覧は組織階層の変更頻度が低いため Valkey に TTL 1時間でキャッシュ（組織移動時にキャッシュ無効化）
- お知らせ自体もスコープ別に Valkey キャッシュ（お知らせ作成/削除時に無効化）
- 階層の深さに依存しないクエリ構造のため、大規模組織でも安定したレスポンスタイムを維持
```

### リアルタイム更新戦略

```
■ 初期リリース（Phase 5）: ポーリング方式（Twitter/タイムライン型）
- スレッド一覧: 画面上部に「新しいスレッドがあります」バナーを表示（30秒間隔のポーリング）
- スレッド詳細: 「新しい返信があります」バナーを表示（同上）
- ユーザーがバナーをタップして手動リロード（プルリフレッシュ）
- ポーリングエンドポイント: GET /bulletin/threads/updates?scope_type=TEAM&scope_id=5&since={ISO8601}
  → 指定日時以降の新規スレッド数・更新スレッド数を返却（軽量レスポンス）

■ 将来（WebSocket 導入時）: イベント駆動に切り替え
- 以下の ApplicationEvent を発行する設計は Phase 5 で組み込み済み:
  - BulletinThreadCreatedEvent（新スレッド作成）
  - BulletinReplyCreatedEvent（返信追加）
  - BulletinThreadUpdatedEvent（ピン留め/ロック/重要度変更）
- WebSocket 導入時はこれらのイベントを STOMP + Valkey メッセージブローカー経由で配信するリスナーを追加するだけで移行可能
- フロントエンドはポーリング → WebSocket への切り替えをフィーチャーフラグで制御
```

### コルクボード連携

```
掲示板スレッドはコルクボードの参照カード（ref_type = 'BULLETIN_THREAD'）として保存可能。

1. ユーザーがスレッドをコルクボードに保存
2. corkboard_cards に INSERT（ref_type='BULLETIN_THREAD', ref_id=thread_id, content_snapshot=本文先頭5,000文字）
3. 元スレッドが削除されても content_snapshot で内容を保持
4. GET /bulletin-threads/{id}/context で前後のスレッドコンテキストを取得可能
```

### 既読データのバッチ削除

```
1. 日次バッチ（深夜帯に実行）
2. bulletin_read_status のうち、紐付く bulletin_threads が以下の条件を全て満たすレコードを物理削除:
   a. bulletin_threads.created_at が90日以上前
   b. bulletin_threads.is_pinned = false（ピン留めスレッドは除外）
   ※ ピン留め解除後は通常通り90日カウントダウンが始まる
3. バッチサイズ: 1,000件ずつ DELETE（ロック時間を最小化）
4. 削除対象: 全 read_tracking_mode のスレッドの既読データ（モードに関わらず一律削除）
5. 削除後: 対象スレッドの read_count を COUNT で再計算して補正
```

### 下書き保存（フロントエンド）

```
■ 自動保存（localStorage）
- スレッド作成・編集フォームで10秒間隔のデバウンス付き自動保存
- キー: `bulletin_draft_{scope_type}_{scope_id}_{thread_id || 'new'}`
- 保存内容: { title, body, category_id, priority, files（ファイル名のみ） }
- 復元: フォーム表示時に localStorage を確認し、下書きが存在する場合は復元ダイアログを表示
  - 「復元する」→ フォームに反映
  - 「破棄する」→ localStorage から削除
- 投稿成功時: 対応する下書きを localStorage から削除
- サーバーサイドの DB 保存は不要（コスト0、オフラインでも動作）
```

### リアクション集計

```
1. リアクション追加/削除時:
   a. bulletin_reactions へ INSERT / DELETE
   b. レスポンスに最新の reaction_counts を返却（GROUP BY emoji で集計）
2. スレッド一覧/詳細取得時:
   a. reaction_counts: 対象の全リアクションを GROUP BY emoji で集計（LEFT JOIN + サブクエリ）
   b. my_reactions: リクエストユーザー自身のリアクション絵文字の配列（詳細画面のみ）
3. 通知: リアクションの追加/削除では通知を発行しない（返信のノイズ代替のため、通知まで出すと本末転倒）
```

### 画像サムネイル生成

```
1. 添付ファイルのアップロード時に content_type が image/* の場合:
   a. is_image = true をセット
   b. S3 アップロード後、CloudFront Functions（または Lambda@Edge）で サムネイル（幅200px）を自動生成
   c. サムネイルは S3 の別プレフィックス（`thumbnails/`）に保存
2. 画像添付のレスポンスに thumbnail_url を含める（Pre-signed URL）
3. 非画像ファイルの場合: thumbnail_url = null
4. スレッド一覧では attachment_count + 最初の画像のサムネイルを返却（プレビュー表示用）
```

### 重要な判定ロジック
- **スレッド作成権限**: MEMBER 以上（SUPPORTER はスレッド作成不可、返信のみ可能）。さらにカテゴリの `post_min_role` による制限あり
- **スレッド削除権限**: 投稿者本人 or ADMIN。DEPUTY_ADMIN はコンテンツ削除権限が権限グループで明示的に付与されている場合のみ
- **返信削除権限**: 投稿者本人 or ADMIN（同上）
- **ピン留め・ロック権限**: ADMIN or DEPUTY_ADMIN（MANAGE_CONTENT 権限）
- **重要度変更権限**: 投稿者本人 or ADMIN
- **ロック中の制限**: 返信の投稿・編集が不可。スレッドのタイトル・本文の編集は ADMIN のみ可
- **ネスト深さ制限**: 最大5階層。超過時は 400 エラー
- **カテゴリ削除時**: 配下スレッドの `category_id` を NULL に更新（未分類に移行）。スレッドは削除しない
- **アーカイブ権限**: ADMIN or DEPUTY_ADMIN（MANAGE_CONTENT 権限）。アーカイブされたスレッドは一覧のデフォルト表示から除外（`include_archived=true` で表示可能）。返信・既読記録は引き続き可能
- **ソート**: `last_activity`（デフォルト）= `COALESCE(last_replied_at, created_at) DESC`、`created_at` = `created_at DESC`、`reply_count` = `reply_count DESC`。ピン留めスレッドは全ソートで常に先頭
- **リアクション**: 全ロール（GUEST 除く）がリアクション可能。プリセット絵文字のみ（👍👏🙏😊❤️）。通知なし
- **引用表示**: 返信レスポンスの `parent_reply_id` が非 NULL の場合、フロントエンドで親返信の `body` 先頭200文字を引用ブロックとして表示（DB 変更不要、表示ロジックのみ）
- **カテゴリ別投稿権限**: `post_min_role` でカテゴリごとにスレッド投稿可能な最低ロールを設定。デフォルトは MEMBER。例: 「連絡事項」を ADMIN 専用にする場合は `post_min_role = 'ADMIN'` に設定
- **重要度変更時の通知範囲**: 重要度の引き上げ時のみ通知を発行し、引き下げ時は通知しない

| 変更パターン | 通知範囲 |
|------------|---------|
| → CRITICAL | 全スコープメンバー（既読者含む。安全性に関わる見落とし防止） |
| → IMPORTANT / WARNING | 未読者のみ（既読者への再通知はノイズになるため） |
| → INFO / LOW | 通知なし（重要度を下げる変更で通知は不要） |

## 6. セキュリティ考慮事項

- **認可チェック**: 全 API エンドポイントで `scope_type` + `scope_id` に対するユーザーの所属・ロールを検証
- **XSS 防止**: 本文の Markdown レンダリング時に HTML サニタイズ（`<script>`, `<iframe>`, `onerror` 等を除去）。サーバーサイドでは生の Markdown を保存し、レンダリングはフロントエンドで実施
- **コンテンツ削除の監査**: 他者のスレッド/返信を削除した場合は監査ログに記録（`BULLETIN_THREAD_DELETED`, `BULLETIN_REPLY_DELETED`）
- **レートリミット**: スレッド作成に `Bucket4j` で 1分間5件の制限を適用（スパム防止）。返信は1分間10件。ポーリングエンドポイント（`/threads/updates`）はユーザーあたり1分間20リクエスト（30秒間隔の正常利用の約10倍をバッファ）
- **FULLTEXT 検索の入力制限**: 検索クエリは最大100文字に制限。特殊文字（`+`, `-`, `*`, `~`）をエスケープして MySQL FULLTEXT のインジェクションを防止
- **安否確認スレッドの保護**: `source_type = 'SAFETY_CHECK'` のスレッドは手動削除不可（安否確認機能のクローズ時にのみロック可能）
- **既読データのプライバシー**: `read_tracking_mode = 'COUNT_ONLY'` のスレッドでは readers API が個人情報を返さない（カウントのみ）
- **添付ファイルのバリデーション**: アップロード時に `content_type` をホワイトリスト検証（`image/*`, `application/pdf`, `application/msword`, `application/vnd.openxmlformats-officedocument.*`, `text/plain`, `text/csv`）+ ファイルヘッダ（マジックバイト）の二重検証で `content_type` 偽装を防止。実行可能ファイル（`.exe`, `.sh`, `.bat` 等）は拒否。ファイルサイズは S3 Pre-signed URL の生成時に6MB上限を設定
- **添付ファイルのアクセス制御**: S3 Pre-signed URL の有効期限は15分。URL 生成時にスコープへの所属を検証

```
V5.001__create_bulletin_categories.sql       -- カテゴリテーブル作成
V5.002__create_bulletin_threads.sql          -- スレッドテーブル作成（FULLTEXT インデックス含む）
V5.003__create_bulletin_replies.sql          -- 返信テーブル作成（自己参照 FK）
V5.004__create_bulletin_read_status.sql      -- 既読トラッキングテーブル作成
V5.005__create_bulletin_attachments.sql      -- 添付ファイルテーブル作成
V5.006__create_bulletin_reactions.sql       -- リアクションテーブル作成
```

**マイグレーション上の注意点**
- `bulletin_threads.author_id` は `users` テーブルへの FK → Phase 1 で作成済み
- `bulletin_threads.category_id` は `bulletin_categories.id` への FK → V5.001 が先に実行されること
- `bulletin_replies.thread_id` は `bulletin_threads.id` への FK（ON DELETE CASCADE）→ V5.002 が先に実行されること
- `bulletin_replies.parent_reply_id` は自己参照 FK → 同テーブル内のため制約なし
- `bulletin_read_status.thread_id` は ON DELETE CASCADE → V5.002 が先に実行されること
- FULLTEXT インデックスは InnoDB で対応（MySQL 5.6+ で利用可能、8.0 で推奨）
- `bulletin_attachments` はポリモーフィック参照のため FK 制約なし。V5.002（threads）、V5.003（replies）の後に実行されること
- `bulletin_reactions` はポリモーフィック参照のため FK 制約なし。V5.002（threads）、V5.003（replies）の後に実行されること

## 7.1 お知らせウィジェット連携（F02.6）

掲示板スレッドは、作成・編集画面で「お知らせウィジェットに表示する」チェックを入れることで、ダッシュボード上部の横断お知らせウィジェット（F02.6）に表示できる。

### 表示フラグの扱い

- `bulletin_threads` テーブルには **専用カラムを追加しない**（F02.6 案B採用）
- フロントエンドは編集画面を開いた際に `GET /api/v1/teams/{teamId}/announcements?source_type=BULLETIN_THREAD` で既存登録を確認し、チェック状態を復元
- `is_archived = TRUE` のスレッドはお知らせ化不可（400）。アーカイブ時に `announcement_feeds` も自動削除

### 優先度のマッピング

| bulletin_threads.priority | announcement_feeds.priority |
|---|---|
| `CRITICAL` | `URGENT` |
| `IMPORTANT` | `IMPORTANT` |
| `WARNING` | `NORMAL` |
| `INFO` | `NORMAL` |
| `LOW` | `NORMAL` |

### 連動削除・同期

- スレッドの論理削除時、`BulletinThreadDeletedEvent` を発行 → `AnnouncementSyncEventListener` が `announcement_feeds.source_deleted_at` をセット
- タイトル・優先度変更時に `BulletinThreadUpdatedEvent` を発行 → `title_cache` / `priority` を同期
- カテゴリ削除（未分類化）は announcement_feeds に影響しない

### UI 配置

`BulletinThreadForm.vue` の重要度セレクタの下に `AnnouncementToggle.vue` を配置。

### ソースタイプ自動生成からの除外

`source_type = SAFETY_CHECK`（安否確認から自動生成されたスレッド）はお知らせウィジェット化の**対象外**（F02.6 側で弾く）。安否確認は独自の UI フローで目立たせる責務があるため、お知らせウィジェットとの重複を避ける。
