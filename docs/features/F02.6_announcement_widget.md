## 1. 概要

チーム・組織ダッシュボードの上部に配置する横断型の「お知らせウィジェット」。ブログ・掲示板・タイムライン・回覧板・アンケート/投票など複数機能の投稿を、投稿者が「お知らせウィジェットへ表示する」フラグを付けることでメンバーへの目立つ連絡に格上げして集約表示する。

### 本機能が解決する課題

- 重要な連絡をどの機能（ブログ／掲示板／タイムライン）で出しても、閲覧者はダッシュボード上部の1つの枠で見落とさず確認できる
- 投稿者は「お知らせとして見せたい」を1クリックで意思表示でき、既存の投稿フローを中断しない
- 回覧板・アンケートの告知を「まず気づいてもらう」動線として使える（回覧板は確認締切・アンケートは回答締切を表示）

### 本ドキュメントのスコープ

| 機能 | 説明 |
|------|------|
| 横断お知らせフィード | 複数機能のコンテンツを横断表示するポリモルフィックフィード |
| 表示フラグ機能 | 各投稿フォームの「お知らせに表示する」チェック |
| ピン留め・並び順 | CRITICAL/IMPORTANT/URGENT 優先 + ピン留め + 新着順 |
| 既読管理 | ウィジェット内での既読/未読表示（ユーザーごと） |
| 配信対象制御 | 各コンテンツの既存公開範囲を継承したフィルタリング |

**スコープ外**:
- 運営（プラットフォーム運営側）からのお知らせ → F10.1 `platform_announcements`
- 通知配信（プッシュ/メール/アプリ内バナー） → F04.3
- 確認取得（受信者の確認追跡） → F04.9
- 個人ダッシュボードの NOTICES ウィジェット（個人宛通知サマリー） → F02.2

### F02.2 既存お知らせ系ウィジェットとの関係

| ウィジェット | データソース | 本機能との関係 |
|---|---|---|
| `NOTICES`（個人） | `notifications`（F04.3） | 個人宛の通知サマリー。本機能とは独立 |
| `PLATFORM_ANNOUNCEMENTS` | `platform_announcements`（F10.1） | 運営からのお知らせ。本機能とは独立 |
| `TEAM_NOTICES` / `ORG_NOTICES` | `notifications`（F04.3） | 従来はチーム通知の集計だった。**本機能の導入により `TEAM_NOTICES` / `ORG_NOTICES` のデータソースを本 `announcement_feeds` に置き換える**（F02.2 既存ウィジェット名を流用しつつ実装を差し替え） |
| `ANNOUNCEMENT_WIDGET`（新） | `announcement_feeds`（本機能） | 個人ダッシュボードでの横断お知らせ（所属チーム/組織のお知らせを集約） |

> **注**: `TEAM_NOTICES` / `ORG_NOTICES` の widget_key は維持し、データソースのみ切り替える。フロントエンドコンポーネントはリネーム（`WidgetTeamNotices.vue` → `WidgetAnnouncements.vue` 等）で追従する。

## 3. DB設計

### 設計方針の比較（案A vs 案B）

お知らせ化の実現方式として、以下2案を比較検討した。

#### 案A: 各コンテンツテーブルに `display_in_announcement` BOOLEAN を追加

```sql
ALTER TABLE blog_posts         ADD COLUMN display_in_announcement BOOLEAN NOT NULL DEFAULT FALSE;
ALTER TABLE bulletin_threads   ADD COLUMN display_in_announcement BOOLEAN NOT NULL DEFAULT FALSE;
ALTER TABLE timeline_posts     ADD COLUMN display_in_announcement BOOLEAN NOT NULL DEFAULT FALSE;
ALTER TABLE circulation_documents ADD COLUMN display_in_announcement BOOLEAN NOT NULL DEFAULT FALSE;
ALTER TABLE surveys            ADD COLUMN display_in_announcement BOOLEAN NOT NULL DEFAULT FALSE;
```

| 観点 | 評価 |
|---|---|
| 実装の素朴さ | ◎ シンプル。単純な WHERE 条件 |
| 削除整合性 | ◎ 親コンテンツが物理削除されれば自動消滅 |
| 権限継承 | ◎ 各テーブルの visibility を素直に利用 |
| 横断取得 | △ UNION ALL 5テーブルが必要。ソート/ページング性能が劣化 |
| ピン留め・既読のデノーマライズ | × 各テーブルに `announcement_pinned_at` / 既読テーブル5種を追加が必要 |
| 新機能追加時 | × 追加するたびに DDL 5〜6種類の改修が発生 |
| 拡張性 | × アンケート告知・TODO 告知など将来の拡張が都度 DDL になる |

#### 案B: 独立 `announcement_feeds` テーブル（ポリモルフィック関連）

1つの横断テーブルに `source_type` + `source_id` を持たせ、投稿者が「お知らせ表示」チェックを入れた瞬間に INSERT する。

| 観点 | 評価 |
|---|---|
| 実装の素朴さ | ○ 1 テーブルへの INSERT/DELETE |
| 削除整合性 | △ 親が削除されたら ApplicationEvent で連動削除が必要（実装はシンプル） |
| 権限継承 | △ 表示時に `source_type` ごとに親を JOIN して visibility を確認 |
| 横断取得 | ◎ 単一テーブルの SELECT 1発。ソート/ページングが素直 |
| ピン留め・既読 | ◎ `announcement_feeds` と `announcement_read_status` に集約でき、各機能テーブルを汚さない |
| 新機能追加時 | ◎ `source_type` 値を増やすだけ。DDL 変更なし |
| 拡張性 | ◎ 追加フィールド（フラグ）も本テーブルに集約できる |

#### 推奨案: **案B（独立 `announcement_feeds` テーブル）**

理由:
1. 将来の拡張性が決定的に優れる（新機能が増えるたびに 5〜6 テーブルへの DDL を発生させない）
2. ピン留め・既読・優先度など横断的なメタデータを1か所で扱える
3. お知らせ化フラグを「投稿データと物理的に同じテーブル」に置く必要性がない（概念的にも別レイヤ）
4. 単一テーブルの SELECT で並び順・ページングが素直に書ける
5. 親コンテンツ削除時の連動削除は ApplicationEvent + `@TransactionalEventListener(phase = AFTER_COMMIT)` の既存パターンで解決済み

**ただし案Bの IDOR リスクへの対策は 6 章で厳格に定める**。

### テーブル定義

#### `announcement_feeds`

投稿者が「お知らせウィジェットへ表示する」を ON にしたコンテンツを登録する横断テーブル。`source_type` + `source_id` で元コンテンツを参照するポリモルフィック設計。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | ENUM('TEAM', 'ORGANIZATION') | NO | — | 表示スコープ種別 |
| `scope_id` | BIGINT UNSIGNED | NO | — | 表示スコープの ID（teams.id / organizations.id） |
| `source_type` | VARCHAR(30) | NO | — | 元コンテンツ種別。値: `BLOG_POST`, `BULLETIN_THREAD`, `TIMELINE_POST`, `CIRCULATION`, `SURVEY`（将来拡張用に VARCHAR） |
| `source_id` | BIGINT UNSIGNED | NO | — | 元コンテンツの ID（ポリモルフィック） |
| `author_id` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）表示フラグを付けた操作者（通常は投稿者） |
| `title_cache` | VARCHAR(200) | NO | — | 表示用タイトル（元コンテンツから非正規化。タイトルがない場合は本文先頭30文字） |
| `excerpt_cache` | VARCHAR(300) | YES | NULL | 本文抜粋（非正規化。150文字目安。リスト表示用） |
| `priority` | ENUM('URGENT', 'IMPORTANT', 'NORMAL') | NO | 'NORMAL' | お知らせ優先度。ブログ `CRITICAL`/`IMPORTANT` は `URGENT`/`IMPORTANT` に、掲示板 `CRITICAL`/`IMPORTANT`/`WARNING` は `URGENT`/`IMPORTANT`/`NORMAL` に正規化 |
| `is_pinned` | BOOLEAN | NO | FALSE | ピン留め。ADMIN のみ変更可能。ピン留め件数はスコープごとに最大5件（Service 層制限） |
| `pinned_at` | DATETIME | YES | NULL | ピン留め操作日時（ピン留め中の並び順用） |
| `pinned_by` | BIGINT UNSIGNED | YES | NULL | FK → users（ON DELETE SET NULL）ピン留めした ADMIN |
| `visibility` | VARCHAR(30) | NO | — | 閲覧可能な範囲。元コンテンツから継承（`PUBLIC`/`MEMBERS_ONLY`/`SUPPORTERS_AND_ABOVE`）。元が変わった時は同期更新 |
| `starts_at` | DATETIME | YES | NULL | 表示開始日時（NULL = 即時。予約公開のブログ記事を事前登録する場合に使用） |
| `expires_at` | DATETIME | YES | NULL | 表示終了日時（NULL = 期限なし。アンケート締切・回覧期限で自動失効させる） |
| `source_deleted_at` | DATETIME | YES | NULL | 元コンテンツ削除検出日時（ApplicationEvent 経由で反映。NULL 以外の行はウィジェット一覧から除外） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
-- 1スコープ・1コンテンツあたり1レコードを保証（同一投稿を複数回お知らせ化できない）
UNIQUE KEY uq_af_scope_source (scope_type, scope_id, source_type, source_id)

-- ウィジェット取得の主要クエリ: scope で絞り込み、ピン留め優先 + 優先度 + 新着順
INDEX idx_af_scope_feed (scope_type, scope_id, is_pinned DESC, priority, created_at DESC)

-- 期限切れバッチ用
INDEX idx_af_expires (expires_at)

-- ソース削除バッチ・通報調査用の逆引き
INDEX idx_af_source (source_type, source_id)

-- 著者別（自分のお知らせ一覧 / 退会処理）
INDEX idx_af_author (author_id)
```

**制約・備考**
- `scope_type + scope_id` はポリモルフィック参照。アプリ層で existence 検証
- `source_type + source_id` もポリモルフィック参照。FK 制約は張らず、**削除連動は ApplicationEvent ベース**（6.1 章で詳述）
- 物理削除は以下のタイミング:
  1. 投稿者/ADMIN が「お知らせ表示」チェックを外した（DELETE）
  2. 元コンテンツが物理削除または論理削除された（`source_deleted_at` セットのみ。90日後にバッチで物理削除）
  3. `expires_at` を過ぎた（バッチで物理削除）
- `title_cache` / `excerpt_cache` は元コンテンツ更新時に同期更新（ApplicationEvent 経由）。非同期であるため短時間の乖離は許容（UX 上の問題なし）
- `visibility` は**表示時に毎回元テーブルを JOIN せず**、このカラムで一次フィルタを行う。元の visibility が変更されたときは Event で追従
- `priority` の正規化マッピング:
  - BLOG: `CRITICAL` → `URGENT` / `IMPORTANT` → `IMPORTANT` / `NORMAL` → `NORMAL`
  - 掲示板: `CRITICAL` → `URGENT` / `IMPORTANT` → `IMPORTANT` / `WARNING` → `NORMAL` / `INFO`/`LOW` → `NORMAL`
  - タイムライン: 優先度なし → 常に `NORMAL`
  - 回覧板: `URGENT` → `URGENT` / `NORMAL` → `NORMAL`
  - アンケート: 優先度なし → 常に `NORMAL`（締切が近づくと UI 上ハイライト）
- **1スコープあたりの登録上限**: 1000 件（古いものから FIFO で物理削除）。実運用で 1000 件を超えることはないが DoS 対策として明記
- **1ユーザーの投稿あたりの登録**: 同一ユーザーが同一 scope に5分あたり最大 10 件まで（レートリミット。6章）

#### `announcement_read_status`

ユーザーごとのお知らせ既読記録。ウィジェットの「未読バッジ」「既読後にリストから薄く表示」等の UX で使用。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `announcement_feed_id` | BIGINT UNSIGNED | NO | — | FK → announcement_feeds（ON DELETE CASCADE） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users（ON DELETE CASCADE） |
| `read_at` | DATETIME | NO | CURRENT_TIMESTAMP | 既読日時 |

**インデックス**
```sql
UNIQUE KEY uq_ars_feed_user (announcement_feed_id, user_id)   -- 重複既読防止
INDEX idx_ars_user (user_id, read_at DESC)                      -- ユーザー別既読履歴
```

**制約・備考**
- 既読レコードは親 `announcement_feeds` が削除された時点で CASCADE 削除
- 保持期間: **90日** — `announcement_feeds.created_at` 基準で90日経過した feed の既読はバッチで物理削除（ユーザー数 × お知らせ数の爆発を防ぐ）
- 未読数は `announcement_feeds` - `announcement_read_status`（user_id 該当分）の差集合を Valkey にキャッシュ（TTL 60秒）。既読/未読の切替時は Valkey の未読カウンタを減算

### ER図（テキスト形式）
```
announcement_feeds (1) ──── (N) announcement_read_status
         │
         ├─→ [scope_type + scope_id で teams / organizations を参照]
         ├─→ [source_type + source_id で blog_posts / bulletin_threads / timeline_posts / circulation_documents / surveys を参照]
         └─→ users (author_id, pinned_by)
```

### リクエスト／レスポンス仕様

#### `GET /api/v1/teams/{teamId}/announcements`

**認可**: そのチームのメンバー以上（MEMBER / SUPPORTER / ADMIN / DEPUTY_ADMIN / SYSTEM_ADMIN）。SUPPORTER は `visibility IN ('PUBLIC', 'SUPPORTERS_AND_ABOVE')` のみ。

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `cursor` | Long | No | カーソル（announcement_feed ID）。省略時は最新から |
| `limit` | Integer | No | 取得件数（デフォルト: 10、最大: 50） |
| `include_read` | Boolean | No | 既読も含めるか（デフォルト: `true`。ウィジェットから呼ぶ専用ページで false を使用予定） |
| `source_type` | String | No | 種別で絞り込み（`BLOG_POST` 等） |

**認可**:

## 5. ビジネスロジック

### 5.0 `source_url` の生成ルール

API レスポンスの `source_url` は各 source_type ごとに以下のパターンで生成する（バックエンドで計算）:

| source_type | パス |
|---|---|
| `BLOG_POST` | `/teams/{scope_id}/blog/{source_id}` または `/organizations/{scope_id}/blog/{source_id}` |
| `BULLETIN_THREAD` | `/teams/{scope_id}/bulletin/threads/{source_id}` |
| `TIMELINE_POST` | `/teams/{scope_id}/timeline#post-{source_id}` |
| `CIRCULATION` | `/teams/{scope_id}/circulations/{source_id}` |
| `SURVEY` | `/teams/{scope_id}/surveys/{source_id}` |

- 組織スコープは `/teams/` を `/organizations/` に置換
- 個人ダッシュボードから開く場合も同一 URL を使用（スコープメンバーであることは遷移先ページ側でガード）

### 5.1 コンテンツ作成画面でのフラグ付与フロー

```
1. ユーザーがブログ記事の投稿画面で「お知らせウィジェットに表示する」をチェック
2. POST /blog/posts または PUT /blog/posts/{id} のレスポンス成功後、
   フロントエンドが POST /teams/{teamId}/announcements を呼び出し
   ({ source_type: "BLOG_POST", source_id: 保存された記事ID })
3. バックエンドは
   a. 認可チェック（著者 or ADMIN）
   b. scope 一致チェック
   c. 重複チェック（409）
   d. announcement_feeds に INSERT
   e. title_cache / excerpt_cache / visibility / priority を元記事から計算して保存
   f. ApplicationEvent: AnnouncementFeedCreatedEvent 発行
4. チェックを外した場合は DELETE /teams/{teamId}/announcements/{id}
5. 表示フラグは投稿データ側には保存しない（案B）。次回編集時、フロントは既存 announcement_feeds の有無を確認してチェック状態を復元
```

### 5.2 元コンテンツの削除連動

```
1. ブログ記事が論理削除される（deleted_at セット）
2. ApplicationEvent: BlogPostDeletedEvent 発行
3. AnnouncementSyncEventListener が非同期で受信
4. announcement_feeds の該当レコードの source_deleted_at を NOW() に更新
   (物理削除ではなく source_deleted_at セット → ウィジェット表示から除外)
5. 90日後、バッチ（AM 3:00）で source_deleted_at IS NOT NULL のレコードを物理削除
6. 復元（deleted_at 解除）のイベントが来た場合は source_deleted_at を NULL に戻す
```

### 5.3 元コンテンツ更新時の同期

```
1. ブログ記事のタイトル/本文/公開範囲が変更される
2. ApplicationEvent: BlogPostUpdatedEvent 発行
3. AnnouncementSyncEventListener が非同期で受信
4. announcement_feeds の title_cache / excerpt_cache / visibility / priority を再計算して UPDATE
5. 公開範囲が狭まった（例: MEMBERS_ONLY → PRIVATE）場合、
   a. SUPPORTER 以下には非表示にする → visibility 更新のみで十分
   b. PRIVATE へ変更された場合は announcement_feeds を物理削除（お知らせ化の前提が崩れるため）
```

### 5.4 期限切れバッチ

```
- 5分ごとに @Scheduled で実行
- expires_at IS NOT NULL AND expires_at <= NOW() のレコードを物理削除
- 削除件数が多い場合は 1000 件ずつチャンク処理
```

### 5.5 アンケート・回覧の自動フラグ

- アンケートが `PUBLISHED` に遷移した時、`surveys.post_announcement_on_publish` が TRUE なら `SurveyService` 内で自動 INSERT（既定 FALSE。作成画面のチェックで ON にする）
  - 呼び出しは `AnnouncementFeedService.createFromSource(sourceType, sourceId, scope, author)` に集約（POST API を経由せず Service 層で直接）
  - 認可チェックは呼び出し元 Service が保証（アンケート公開は既に権限検証済みのため）
  - `expires_at` は `surveys.expires_at` をコピー（締切と同時にお知らせも失効）
- 回覧が `IN_PROGRESS` に遷移した時、`circulation_documents.post_announcement_on_start` が TRUE なら `CirculationService` 内で同様に呼び出し
  - `expires_at` は `circulation_documents.due_date + 1日` をコピー（締切翌日に失効）
- タイムライン/ブログ/掲示板は **ユーザーがチェックボックスを明示的に ON** にした場合のみお知らせ化（自動化しない）。これらは作成画面の UI で投稿者の意図を取る設計

### 5.6 通知（F04.3）との連携

お知らせウィジェットへの登録 **自体** は通知を発生させない（ウィジェットへの配置は「目立つ場所に置く」という UI 行為であり、プッシュ通知とは独立）。ただし以下の条件でプッシュ通知を連動させる:

- `priority = URGENT` かつ `source_type IN ('BLOG_POST', 'BULLETIN_THREAD')` → 既存のブログ/掲示板の優先度別通知（F06.1 / F05.1 既存仕様）を維持。お知らせウィジェットはそれを妨げない
- 確認取得が必要なケース（臨時休業・緊急連絡）は F04.9 確認通知を推奨。本機能のお知らせは「ダッシュボード上部への常駐」であり、到達保証のための強制配信は行わない

### 5.7 モデレーション連動（F04.5）

- `content_reports` で通報された元コンテンツが `HIDDEN` / `DELETED` に遷移した場合、関連する `announcement_feeds` を物理削除
- SYSTEM_ADMIN の管理画面から `announcement_feeds` を直接削除する API は設けない（元コンテンツ側の通報フローを使う）
- **お知らせウィジェット UI 上には通報ボタンを配置しない**（元コンテンツページに遷移してから通報する導線に統一）。ユーザーが違和感なく使える導線のため、ウィジェット上のアクションは「クリックで遷移」「既読マーク」「（ADMIN のみ）お知らせ解除」「（ADMIN のみ）ピン留め」のみ

## 6. セキュリティ考慮事項

### 6.1 IDOR（Insecure Direct Object Reference）対策

ポリモルフィック参照の最大リスクは **他スコープ/他チームのコンテンツをお知らせ化される** こと。以下で厳格に防ぐ:

1. **POST 時の二重検証**:
   - パス変数の `teamId` と `source_id` の実スコープが一致することを確認
   - `source_type` ごとに scope 取得用の軽量 Repository メソッドを実装:
     - `BlogPostRepository.findScopeById(id): Optional<Scope>` — `team_id` / `organization_id` / `user_id` の XOR を返す
     - `BulletinThreadRepository.findScopeById(id): Optional<Scope>` — `scope_type + scope_id`
     - `TimelinePostRepository.findScopeById(id): Optional<Scope>`
     - `CirculationDocumentRepository.findScopeById(id): Optional<Scope>`
     - `SurveyRepository.findScopeById(id): Optional<Scope>`
   - **個人ブログ（`user_id IS NOT NULL`）はチーム/組織スコープのお知らせ化対象外**のため、scope 一致検証で弾かれる（400 Bad Request）
   - ソーシャルプロフィール投稿（`social_profile_id IS NOT NULL`）も同様に 400
   - 不一致なら 400 Bad Request
2. **著者一致の検証**:
   - MEMBER の場合、`source_id` の `author_id / user_id / created_by` が JWT の userId と一致
   - ADMIN の場合のみ他者コンテンツを登録可
3. **source_type のホワイトリスト化**:
   - 許可値は Java enum `AnnouncementSourceType` で管理。DB は VARCHAR(30) で保持（将来拡張の柔軟性を確保）
   - 新規 source_type 追加時は必ず対応する Repository の `findScopeById` 実装 + 認可ルール定義とセットで入れる（チェックリストを PR テンプレートに記載）

### 6.2 権限チェック漏れ防止

- `GET /announcements` レスポンス組み立て時、**元コンテンツの最新 visibility を再検証**しない代わりに、`announcement_feeds.visibility` を使う。同期が遅れた場合の許容 window は最大 30 秒（Event キューの SLA）
- SUPPORTER には `visibility = 'MEMBERS_ONLY'` のレコードを返さない
- 検証は Repository 層の `@Query` レベルで WHERE 句に入れる（Service 層の if 文に依存しない）

### 6.3 XSS / SQL Injection 対策

- `title_cache` / `excerpt_cache` は **元コンテンツの plain text から切り出し** て保存（Markdown レンダリングはしない）。フロントエンドは `v-html` を使わず `{{ ... }}` で表示
- Markdown 特殊文字は残してよい（テキストとしてそのまま見せるため）
- SQL は JPA + Repository 経由。動的 SQL は使わない

### 6.4 レートリミット（Bucket4j）

| 操作 | 制限 |
|---|---|
| `POST /announcements`（お知らせ化） | 5分10件/ユーザー |
| `DELETE /announcements/{id}` | 1分10件/ユーザー |
| `PATCH /pin` | 1分20件/ユーザー |
| `POST /read` | 1分100件/ユーザー |
| `POST /read-all` | 1分5件/ユーザー |
| `GET /announcements`（一覧） | 1分60件/ユーザー |
| `GET /announcements/me` | 1分30件/ユーザー |

### 6.5 個人情報の取り扱い

- `author.display_name` / `avatar_url` は `users` の公開カラムのみ露出
- ソーシャルプロフィールで投稿された個人ブログのお知らせ化は**対象外**（チームダッシュボード上のお知らせなので、匿名人格は不要）。バックエンドで `blog_posts.social_profile_id IS NOT NULL` のコンテンツは 400 で拒否

### 6.6 通報・非表示への対応

F04.5 の通報による `HIDDEN` / `DELETED` 遷移 → ApplicationEvent で announcement_feeds を物理削除（5.7 章で既述）

### 6.7 タイムゾーン処理

- DB 格納はすべて UTC（DATETIME 列。JVM の TZ 設定を UTC に固定するプロジェクト規約に準拠）
- API レスポンスは ISO 8601 with offset（`+09:00` 等）
- `starts_at` / `expires_at` の評価は DB 側 `UTC_TIMESTAMP()` を使用（アプリ層で `Instant.now()` で比較）
- クライアント表示はブラウザのローカル TZ で整形

### 6.8 多言語対応（F11.2 / F11.3 連携）

- `title_cache` / `excerpt_cache` は投稿者が入力した言語のまま保存（翻訳しない）。これは元コンテンツ（ブログ記事等）の多言語化が F11.2 で本格対応するまでの挙動
- F11.2 で多言語コンテンツ対応が入った場合、announcement_feeds にも `title_cache_i18n JSON` 等のカラム追加を検討（本設計書の範囲外・将来タスク）
- UI ラベル（「お知らせ」「重要」「締切間近」など）は F11.3 のロケールファイルで完全多言語化（本設計書 12 章）

### 6.9 PWA / オフライン対応（F11.1 連携）

- `GET /announcements` は PWA キャッシュ対象に含める（SWR 戦略）
- 既読マーク（POST /read）はオフライン時、IndexedDB の sync queue に入れて再接続時に送信
- お知らせ化（POST /announcements）はオフライン時は実行不可（投稿フォームのチェックボックスを disabled にし、ツールチップ「オフラインのため設定できません」を表示）。投稿本体はオフライン対応されるが、お知らせフラグは投稿成功後の逐次 API であり、オフラインキューに積まない
- ピン留め操作（PATCH /pin）もオフライン不可（ADMIN 操作のため誤操作の影響が大きく、確実にオンラインで行う）

**インタラクション**:
- クリック → 元コンテンツのページに遷移 + 既読マーク
- モバイル長押し → プレビューモーダル
- カード右端にチェックマーク（既読）/ バツマーク（ADMIN のみ：お知らせから外す）
- アクセシビリティ: `role="list"` + 各カード `role="listitem"` + キーボードで Enter / Space 既読

### 投稿フォームでのチェックボックス配置

```
[タイトル入力欄]
[本文エディタ]
[公開範囲: メンバーのみ ▾]
[ ] お知らせウィジェットに表示する        ← ここに新規追加
    ℹ ダッシュボード上部で目立つ位置に表示されます
    [ ] ピン留めする（ADMIN のみ表示）
```

- チェック ON 時にだけ「表示期限（任意）」の日時ピッカーがスライドダウン
- MEMBER は `priority` を選べず、ADMIN のみ `URGENT`/`IMPORTANT` を選択可
- 投稿の submit 成功後に announcement_feeds への POST を実行（submit 時に並列発行ではなく逐次）

## 11. 決定事項（残存課題ではなく確定したもの）

本設計書は完了ステータスとするため、未解決事項ではなく「**決定事項**」として以下を明記する。

- **案B採用** — 独立 `announcement_feeds` テーブルによるポリモルフィック参照方式を採用。将来拡張性とメタデータ集約性を優先。DDL 変更を毎回発生させない設計
- **source_type は VARCHAR(30)** — 将来 TODO・スケジュール・ファイル共有への拡張を想定して enum ではなく VARCHAR で保持。Java 側は `AnnouncementSourceType` enum で受けるが、DB スキーマは柔軟性を残す
- **既読の保持期間は 90 日** — ユーザー数 × お知らせ数の爆発を回避。90 日を超えた古いお知らせは再び未読扱いになるが、UX 上の問題は実質無視できる
- **ピン留め上限は5件/スコープ** — 過度なピン留めで UI が埋まるのを防ぐ。6件目以降の試行は 409
- **お知らせウィジェットは通知を発生させない** — プッシュ通知は元コンテンツ側（F06.1 / F05.1 / F04.3）の仕組みを維持。本機能は「ダッシュボード上部の目立つ枠」という表示レイヤのみ
- **既存 `TEAM_NOTICES` / `ORG_NOTICES` ウィジェットのデータソースを本 announcement_feeds に切替** — widget_key は維持しつつコンポーネント実装を差し替え（F02.2 に追記）
- **個人ダッシュボードには `ANNOUNCEMENT_WIDGET` を新規追加** — F02.2 の widget_key 一覧に追記（個人スコープで所属全チーム/組織のお知らせを横断集約）
- **確認取得が必要な通知は F04.9 を使う** — 本機能は閲覧/既読まで。確認取得の追跡は F04.9 の責務
- **ソーシャルプロフィール投稿はお知らせ化できない** — チーム/組織スコープのお知らせは実名著者のみ
