# F06: メンバー紹介 + ギャラリー

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 6
> **最終更新**: 2026-03-18

---

## 1. 概要

チーム・組織のメンバー紹介ページをメインページ＋年度別詳細ページの階層構造で管理し、各メンバーのプロフィールを画像・一言・拡張フィールド付きで掲載する機能を提供する。ギャラリーはメンバー向けの**写真・動画アルバム管理機能**で、アルバム単位で閲覧権限を設定し、チーム・組織内のアクティビティ写真・活動動画を体系的に蓄積・共有する。ファイルストレージは Cloudflare R2 に集約し、Cloudflare Workers 経由で署名付き配信を行う。

**インデックス**
```sql
UNIQUE KEY uq_tp_slug_team (team_id, slug, deleted_at)       -- チーム内スラッグ一意性
UNIQUE KEY uq_tp_slug_org (organization_id, slug, deleted_at) -- 組織内スラッグ一意性
UNIQUE KEY uq_tp_year_team (team_id, year, deleted_at)       -- チーム内年度一意性（YEARLY ページ）
UNIQUE KEY uq_tp_year_org (organization_id, year, deleted_at) -- 組織内年度一意性
INDEX idx_tp_team_status (team_id, status, sort_order)        -- チーム別ページ一覧
INDEX idx_tp_org_status (organization_id, status, sort_order) -- 組織別ページ一覧
```

**制約・備考**
- `team_id` と `organization_id` は XOR:
  ```sql
  CONSTRAINT chk_tp_scope
    CHECK (
      (team_id IS NOT NULL AND organization_id IS NULL)
      OR (team_id IS NULL AND organization_id IS NOT NULL)
    )
  ```
- `page_type = 'MAIN'` の場合は `year` は NULL。`page_type = 'YEARLY'` の場合は `year` 必須:
  ```sql
  CONSTRAINT chk_tp_year
    CHECK (
      (page_type = 'MAIN' AND year IS NULL)
      OR (page_type = 'YEARLY' AND year IS NOT NULL)
    )
  ```
- チーム/組織あたり MAIN ページは1つのみ。**Service 層で `page_type = 'MAIN'` かつ同一スコープの未削除レコード存在チェック**を行い、重複時は 409 Conflict を返却する。DB の Generated Column や部分一意インデックスは使用しない（シンプルさ優先）
- 論理削除: `deleted_at DATETIME nullable`（SoftDeletableEntity 適用）

```sql

**制約・備考**
- `user_id` が NULL になった場合でも `display_name` と `photo_s3_key` で掲載を継続
- **member_number**: スポーツチームの背番号、企業の社員番号、マンション管理組合の部屋番号等、チーム/組織ごとにメンバーを一意に識別する番号。同一ページ内では重複不可（アプリ層でバリデーション。NULL は複数可）。活動記録の参加者追加時に番号入力 or コンボボックスで素早くメンバーを選択するために使用。カスタムフィールドではなく専用カラムにすることでインデックス検索・他機能からの参照を実現
- **MEMBER セルフ編集**: `team_pages.allow_self_edit = true` の場合、MEMBER は自分の `member_profiles` レコード（`user_id` = 自分）に対して `bio`, `photo_s3_key`, `custom_field_values` を直接編集可能（即時反映、承認不要）。`display_name`, `position`, `sort_order`, `is_visible` は ADMIN/DEPUTY_ADMIN のみ編集可能（ページの見た目制御は管理者が責任を持つ）。不適切な内容は ADMIN が `is_visible = false` で対処
- **custom_field_values の JSON バリデーション**: Service 層で `member_profile_fields` と照合し、存在しない field_id の値を拒否・field_type に応じた型チェック（NUMBER なら数値文字列か）・is_required フィールドの未入力チェックを実施する。不正な JSON 構造は 400 エラー
- 物理削除（ページ削除時にカスケード削除）

**制約・備考**
- アルバム削除時にカスケード削除（R2 ファイルの削除はアプリケーション層で非同期実行）
- 物理削除
- **写真の後処理**: Cloudflare Images（または Cloudflare Workers）で短辺300px へリサイズしてサムネイル生成し、`thumbnails/` プレフィックス配下に書き出し。`thumbnail_r2_key` に R2 オブジェクトキーを格納。WebP 変換・サイズ別配信は Cloudflare Images の variant（`https://imagedelivery.net/{account_hash}/{image_id}/{variant}`）を活用
- **動画の後処理（ハイブリッド方式）**: 詳細は `docs/db_design_details.md` の「サムネイル生成戦略」セクションを参照
  - `duration_seconds <= 300`（5 分以下）: Cloudflare Workers（ffmpeg.wasm）が非同期に (a) メタデータ抽出（duration / codec / resolution）、(b) 00:01 時点のポスターフレームを JPEG 化して `thumbnails/{元のパス}.jpg` に書き戻し、(c) `thumbnail_r2_key` を UPDATE、(d) `processing_status = READY` に更新
  - `duration_seconds > 300`（5 分超）: クライアント側で `<video>` + Canvas によるフレーム抽出を事前実行 → サムネイルを `thumbnails/{元のパス}.jpg` に Presigned PUT → 続けて動画本体を R2 にアップロード → バックエンドは HeadObject で両方の R2 上の実在を確認してから `processing_status = READY` で INSERT
  - 両方失敗した場合は `processing_status = FAILED` に遷移し、フロントエンドで汎用動画アイコンのプレースホルダー画像を表示。SYSTEM_ADMIN は後述の `POST /api/v1/system-admin/gallery/regenerate-thumbnails` で再生成をリクエスト可能
  - **Cloudflare Stream は採用しない**（R2 と別課金体系で割高になるため）
- `taken_at` は EXIF の `DateTimeOriginal`（画像）または QuickTime メタデータ（動画）から抽出。取得できない場合は NULL
- **アップロード可能な形式**:
  - **PHOTO**: JPEG, PNG, WebP, HEIC（HEIC は JPEG に変換して保存。`content_type` は変換後の MIME タイプを記録）
  - **VIDEO**: MP4 (H.264 + AAC) 推奨、WebM (VP9), MOV（QuickTime）
- **サイズ上限（機能別容量上限は設けない）**:
  - **PHOTO / VIDEO ともに個々のファイルに対する容量課金上限は設けない**。チーム・組織・個人の容量枠（`storage_subscriptions.included_bytes`）の範囲内なら自由にアップロードできる
  - **技術的下限**: 単発 PUT は 100MB まで。100MB 超のファイル（長尺試合動画など）は F05.5 の Multipart Upload フロー（`POST /api/v1/files/multipart/start`、`target_prefix = gallery/{scope_type}/{scope_id}/album-{albumId}/`）を利用してアップロードし、完了後に返却された `file_key` を `POST /api/v1/gallery/albums/{id}/photos` に渡して登録する。**F05.5 Multipart API 呼び出し前に F06.2 側でギャラリー編集権限（アルバムへの `allow_member_upload` 確認 + スコープへのアクセス権）を確認すること（認可責任: 呼び出し元責任 — 案B 確定）**
  - **Multipart 1 オブジェクト上限**: 5TB（R2 ハード上限）
- `cover_photo_id = NULL` かつメディアが0件のアルバムは、フロントエンドでプレースホルダー画像を表示する
- **クォータ計上**: PHOTO / VIDEO ともに `file_size` が当該スコープのストレージクォータに計上される。`thumbnails/` プレフィックス配下のサムネイル・Cloudflare Images の variant はシステム生成物なので非計量

**認可**: MEMBER 以上

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `q` | String | Yes | 検索クエリ（メンバー番号の前方一致 or 表示名の部分一致。最小1文字） |
| `team_page_id` | Long | No | 特定ページのメンバーに限定（未指定: 最新の PUBLISHED ページから検索） |
| `limit` | int | No | 取得件数（デフォルト: 10、最大: 20） |

    {
      "member_profile_id": 42,
      "user_id": 12,
      "display_name": "田中太郎",
      "member_number": "10",
      "position": "FW",
      "photo_s3_key": "profiles/tanaka.jpg"
    },
    {
      "member_profile_id": 43,
      "user_id": 15,
      "display_name": "田中次郎",
      "member_number": "15",
      "position": "MF",
      "photo_s3_key": null

  ]
}
```

**補足**
- 検索は `member_number` の前方一致（`10` → `10`, `100`, `101` がヒット）と `display_name` の部分一致を OR 条件で実行
- `member_number` が完全一致する結果を最優先でソート（番号入力で一発選択しやすくする）
- `q` が数字のみの場合は `member_number` の前方一致を優先表示、それ以外は `display_name` の部分一致を優先表示
- フロントエンド実装: コンボボックス（入力欄 + ドロップダウン候補リスト）。入力文字が変わるたびに debounce（300ms）で API を呼び出し、候補を表示。候補をクリック or Enter で選択すると参加者として追加

**認可**: ADMIN、DEPUTY_ADMIN（MANAGE_CONTENT）、または MEMBER（`allow_member_upload = true` のアルバム）

写真・動画アップロード用の R2 Presigned URL を発行する。クライアントはこの URL に対して直接 PUT でファイルをアップロードし、完了後に `POST /albums/{id}/photos` でメタデータを登録する。

**リクエストボディ**
```json
{
  "media_type": "VIDEO",
  "original_filename": "match_20260301.mp4",
  "content_type": "video/mp4",
  "file_size": 314572800
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `media_type` | String | Yes | `PHOTO` / `VIDEO` |
| `original_filename` | String | Yes | 元ファイル名 |
| `content_type` | String | Yes | MIME タイプ。PHOTO: `image/*` / VIDEO: `video/mp4`, `video/webm`, `video/quicktime` |
| `file_size` | Long | Yes | ファイルサイズ（バイト）。機能別容量上限なし（スコープ容量枠内で自由）。単発 PUT の技術的上限は 100MB。100MB 超は F05.5 Multipart Upload フロー経由 |

**処理内容**
1. 権限チェック + `allow_member_upload` 検証
2. `content_type` と拡張子の整合性チェック
3. `file_size` 技術下限チェック（単発 PUT 100MB 以下。超過時は 400 で F05.5 Multipart Upload フローを案内）
4. `StorageQuotaService.checkQuota()` で空き容量確認（不足時 409 + プラン更新導線）
5. R2 オブジェクトキー生成: `gallery/{scope_type}/{scope_id}/album-{albumId}/{media_type|lowercase}-{uuid}.{ext}`（単一バケット `mannschaft-storage` 内）
6. R2 Presigned PUT URL を発行（有効期限: PHOTO は10分、VIDEO は15分）

**レスポンス（200 OK）**
```json
{
  "data": {
    "upload_url": "https://{accountId}.r2.cloudflarestorage.com/mannschaft-storage/gallery/TEAM/1/album-8/video-abc123.mp4?X-Amz-Signature=...",
    "file_key": "gallery/TEAM/1/album-8/video-abc123.mp4",
    "expires_in": 900,
    "max_size": 104857600
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | `media_type` が不正 / `content_type` が非対応 / 単発 PUT で 100MB 超（F05.5 Multipart Upload フローの利用を案内） |
| 403 | 権限不足（MEMBER で allow_member_upload = false） |
| 404 | アルバムが存在しない |
| 409 | ストレージクォータ不足（プラン更新導線を含む） |

#### `POST /api/v1/gallery/albums/{id}/photos`

**認可**: ADMIN、DEPUTY_ADMIN（MANAGE_CONTENT）、または MEMBER（`allow_member_upload = true` のアルバム）

R2 アップロード完了後にメディア（写真・動画）メタデータを DB に登録する。

**リクエストボディ**
```json
{
  "items": [
    {
      "media_type": "PHOTO",
      "s3_key": "gallery/TEAM/1/album-8/photo-xyz001.jpg",
      "original_filename": "IMG_1234.jpg",
      "file_size": 3145728,
      "content_type": "image/jpeg",
      "caption": "集合写真"
    },
    {
      "media_type": "VIDEO",
      "s3_key": "gallery/TEAM/1/album-8/video-xyz002.mp4",
      "original_filename": "match_highlights.mp4",
      "file_size": 314572800,
      "content_type": "video/mp4",
      "caption": "ハイライト"
    }
  ]
}
```

**処理内容**
1. 各 `s3_key` について R2 HeadObject で実在確認（存在しなければ 400）
2. `StorageQuotaService.checkQuota()` 再チェック（レースコンディション対策）
3. `photos` INSERT。PHOTO は `processing_status = PENDING`（サムネイル生成前）、VIDEO は `processing_status = PENDING`（メタデータ抽出前）
4. `photo_albums.photo_count` アトミック加算
5. `StorageQuotaService.recordUpload()`
6. 後処理を非同期起動（ハイブリッド方式 — 詳細は `docs/db_design_details.md` の「サムネイル生成戦略」参照）:
   - PHOTO: Cloudflare Workers または Cloudflare Images で EXIF 抽出 → GPS 等プライベート情報ストリップ → HEIC → JPEG 変換 → 短辺300px サムネイル生成（`thumbnails/` プレフィックス配下）→ `processing_status = READY`
   - VIDEO（`duration_seconds <= 300`）: Cloudflare Workers（ffmpeg.wasm）がメタデータ抽出（duration / codec / resolution）→ 00:01 時点ポスターフレーム抽出 → `thumbnails/{元のパス}.jpg` に書き戻し → `processing_status = READY`
   - VIDEO（`duration_seconds > 300`）: クライアント側で事前に Canvas フレーム抽出を完了させ、サムネイルを先に R2 にアップロードしてから本体 POST を呼ぶ運用。バックエンドは両方の HeadObject で実在確認 → `processing_status = READY` で即時登録
   - 失敗時は `processing_status = FAILED`、フロントエンドは汎用動画アイコンのプレースホルダー画像を表示

**レスポンス（201 Created）**
```json
{
  "data": {
    "uploaded_count": 2,
    "album_photo_count": 15,
    "items": [
      {
        "id": 101,
        "media_type": "PHOTO",
        "thumbnail_url": null,
        "processing_status": "PENDING"
      },
      {
        "id": 102,
        "media_type": "VIDEO",
        "thumbnail_url": null,
        "processing_status": "PENDING"
      }
    ]
  }
}
```

**備考**
- サムネイルは非同期生成。`processing_status = 'PENDING' → 'PROCESSING' → 'READY'` と遷移し、完了後に `thumbnail_url` が設定される

## 5. ビジネスロジック

### メンバー紹介ページ管理フロー

```
1. ADMIN がメインページ（MAIN）を作成 → status = DRAFT
2. セクションを追加: HEADING, TEXT, IMAGE, MEMBER_LIST を自由に配置
3. メンバープロフィールを登録（個別 or 一括）
4. プレビュー確認後、PATCH /publish で status = PUBLISHED
5. 翌年度:
   a. 新しい YEARLY ページを作成（year = 2027）
   b. POST /team/pages/{id}/copy-members で前年度ページからメンバーを一括コピー
   c. 卒業/退会メンバーを削除、新メンバーを追加
   d. メインページのセクションを更新して新年度ページへのリンクを追加
```

### ギャラリーメディアアップロードフロー

```
1. フロントエンドが動画の duration をブラウザ側で判定
   - duration <= 300秒（5分以下）: 標準フロー（Workers 側でサムネイル生成）
   - duration > 300秒（5分超）: クライアント側サムネイル事前生成フロー

2. 標準フロー（PHOTO / 短尺 VIDEO）:
   a. POST /gallery/albums/{id}/media/upload-url で単発 PUT 用の R2 Presigned URL を取得
      - 100MB 以下の場合のみこのフロー。超過時は F05.5 Multipart Upload フローに切り替え
      - media_type（PHOTO / VIDEO）・content_type・file_size を指定
      - バックエンドが StorageQuotaService.checkQuota() でクォータ事前確認
   b. ブラウザから R2 へ直接アップロード（PUT {upload_url}、単一バケット mannschaft-storage）
   c. POST /gallery/albums/{id}/photos で写真/動画メタデータをバックエンドに登録
      - 各 s3_key を R2 HeadObject で実在確認
      - クォータ再チェック → recordUpload() で加算
      - photos INSERT（processing_status = PENDING）
   d. 後処理（ハイブリッド方式）:
      - PHOTO: Cloudflare Workers / Cloudflare Images が EXIF 抽出・GPS ストリップ・HEIC→JPEG 変換・
               短辺300px サムネイル生成 → thumbnails/{元のパス}.jpg に書き出し →
               photos レコードを UPDATE（thumbnail_r2_key, taken_at, width, height, content_type, processing_status = READY）
      - 短尺 VIDEO: Cloudflare Workers（ffmpeg.wasm）がメタデータ抽出（duration / codec / resolution / 撮影日時）、
                    GPS ストリップ、00:01 時点のポスターフレームを JPEG で抽出し thumbnails/{元のパス}.jpg に書き戻し →
                    photos レコードを UPDATE（thumbnail_r2_key, duration_seconds, video_codec, width, height,
                    taken_at, processing_status = READY）

3. クライアント側サムネイル事前生成フロー（長尺 VIDEO、duration > 300秒）:
   a. ブラウザ内で HTMLVideoElement を使い動画をロード
   b. <video> の currentTime を 1 秒に設定 → Canvas に描画 → toBlob で JPEG Blob を生成
   c. POST /gallery/albums/{id}/media/upload-url を2回呼び、本体 + サムネイル両方の Presigned PUT URL を取得
      - 100MB 超の本体動画は F05.5 の Multipart Upload フロー（POST /api/v1/files/multipart/start、
        target_prefix = "gallery/{scopeType}/{scopeId}/album-{albumId}/"）を使用
   d. サムネイル（JPEG）→ 動画本体 の順で R2 にアップロード
      - サムネイルの R2 キー: thumbnails/{元のオブジェクトキー}.jpg
   e. POST /gallery/albums/{id}/photos で両方の s3_key を渡してメタデータ登録
   f. バックエンドは両方の HeadObject で実在確認 → photos INSERT（processing_status = READY に即時遷移）

4. 失敗時フォールバック:
   - Workers 実行失敗 or クライアントサムネイル欠落 → processing_status = FAILED
   - フロントエンドで汎用動画アイコンのプレースホルダー画像を表示
   - SYSTEM_ADMIN は POST /api/v1/system-admin/gallery/regenerate-thumbnails で再生成リクエスト可能

5. photo_albums.photo_count をアトミック更新
```

### ギャラリーの R2 ファイル削除

```
メディア（写真・動画）削除時:
1. photos レコードを物理削除
2. photo_albums.photo_count をアトミック減算
3. StorageQuotaService.recordDeletion() で使用量減算
4. ApplicationEvent を発行 → 非同期リスナーが R2 ファイルを削除
   - s3_key（本体）
   - thumbnail_r2_key（サムネイル。`thumbnails/` プレフィックス配下）

アルバム論理削除時:
- photos レコードはカスケード削除されない（論理削除のため）
- メディアは引き続き R2 に存在
- クォータも即時減算しない

アルバム物理削除時（バッチ処理）:
- photos レコードをカスケード削除
- 全メディアの R2 ファイルを非同期バッチで削除
- StorageQuotaService.recordDeletion() を一括実行
```

### アクセス制御ロジック

```
メンバー紹介ページ:
1. status = PUBLISHED かつ visibility = PUBLIC → 認証不要
2. status = PUBLISHED かつ visibility = MEMBERS_ONLY → MEMBER 以上
3. status = DRAFT → ADMIN, MANAGE_CONTENT 権限保持者のみ

ギャラリー:
1. ギャラリーモジュールが有効化されていること（選択式モジュール No.6）
2. visibility = ALL_MEMBERS → チーム/組織の MEMBER 以上
3. visibility = SUPPORTERS_AND_ABOVE → SUPPORTER 以上
4. visibility = ADMIN_ONLY → ADMIN, DEPUTY_ADMIN のみ
5. 写真アップロード: ADMIN, DEPUTY_ADMIN, または allow_member_upload = true の場合 MEMBER
6. 写真削除: アップロードした本人、または ADMIN/DEPUTY_ADMIN
7. メディアダウンロード: allow_download = true の場合のみ。閲覧権限を持つ全ユーザー
   - allow_download = false の場合: Cloudflare Workers 経由の署名付き配信 URL に `Content-Disposition: inline` を強制。
     フロントエンドで右クリック保存抑制（完全防止は不可能だがカジュアルコピー抑止として有効）
   - 動画は Workers ルートで HTTP Range Request に対応し、`<video>` タグからシーク・ストリーミング再生が可能
```

### ストレージクォータ

```
- クォータ管理は横断設計「統合ストレージクォータ」（docs/cross-cutting/storage_quota.md）に一元化。
  写真・動画ともに scope 帰属（TEAM / ORGANIZATION）の used_bytes に計上される
- チーム/組織あたりのメディア件数ソフト上限:
  - 合計: 5,000件（写真 + 動画）
- ストレージ容量はスコープの storage_subscriptions.included_bytes に従う
  （Phase 3 の暫定値: 組織 50GB / チーム 5GB。Phase 8 で有料プラン化）
- アップロード時に StorageQuotaService.checkQuota() で空き容量を確認し、超過時は 409 Conflict を返却
- ADMIN は使用量を各スコープの GET /teams/{id}/storage で確認可能
- 動画は 1本 で数百 MB〜数 GB に達することもあるため、少数でも容量を占めやすい点を UI で注意喚起（アップロード前に「予測使用率」をダイアログで表示する UX ガードを実装）
```

### R2 孤立ファイルクリーンアップ

```
対象: メンバープロフィール画像、ページカバー画像、セクション画像

- メンバープロフィール画像の更新時:
  1. 旧 photo_s3_key（R2 オブジェクトキー）を取得
  2. 新しい photo_s3_key で UPDATE
  3. ApplicationEvent を発行 → 非同期リスナーが旧 R2 オブジェクトを削除

- ページカバー画像・セクション画像も同様のフロー
- 削除対象が NULL（画像未設定だった場合）は何もしない

- Presigned PUT URL で発行したが POST /gallery/albums/{id}/photos で
  登録されなかった R2 オブジェクトは、日次バッチ（深夜5:00）で
  プレフィックス `gallery/` 配下を走査し、アップロードから24時間経過 かつ
  DB 未登録のファイルを削除する
```

### 論理削除の物理削除ポリシー

```
- 対象テーブル: team_pages, photo_albums
- 物理削除タイミング: 論理削除（deleted_at）から 90日後
- 実行間隔: 週次バッチ（日曜深夜2:00。F06_cms_blog と同一バッチ）
- 処理:
  1. team_pages: 関連する team_page_sections, member_profiles をカスケード削除
     - member_profiles.photo_s3_key の R2 オブジェクトを非同期削除
  2. photo_albums: 関連する photos をカスケード削除
     - 全メディアの s3_key + thumbnail_r2_key（`thumbnails/` プレフィックス配下）を非同期バッチで R2 削除
     - StorageQuotaService.recordDeletion() で使用量減算（写真・動画のバイト合計。サムネイルは非計量）
  3. 物理 DELETE 実行
- 1回あたりの処理上限: 100件（動画は1本で数百MB になるため特に慎重に）
```

### サムネイル再生成

```
POST /api/v1/system-admin/gallery/regenerate-thumbnails

- 認可: SYSTEM_ADMIN のみ
- 用途: サムネイルサイズ変更、品質変更時に既存サムネイルを再生成
- 処理:
  1. 全 photos レコード（または team_id / organization_id 指定）を対象

- リクエストボディ:

```

## 6. セキュリティ考慮事項

- **認可チェック**: TeamPageService / GalleryService の入り口で所属検証
- **画像・動画バリデーション**: アップロードされたファイルの Content-Type と Magic Bytes を検証（偽装防止）
- **EXIF プライバシー**: **全画像アップロード（ギャラリー写真・メンバープロフィール画像・ページカバー画像・セクション画像）**で GPS 情報を含む EXIF データをストリップする（位置情報漏洩防止）。**動画**も同様に QuickTime メタデータから GPS 情報をストリップ
- **R2 アクセス制御**: ギャラリーメディアは **Cloudflare Workers 経由の署名付き URL** でのみアクセス可能。Workers ルートが (a) 対象アルバムの visibility チェック、(b) `allow_download` の判定、(c) R2 ストリーミング or 短命 Presigned GET URL への 302 を担う。R2 バケットへのパブリック直接アクセスは禁止。`s3_key` → 配信 URL への変換はアプリケーション層（Workers + バックエンド）で実施
- **プレビュートークン**: SecureRandom で64文字のトークンを生成。有効期限24時間。推測不可能なエントロピーを確保
- **レートリミット**: メディアアップロード API（upload-url 発行 + 登録）に `Bucket4j` で 1分間に30回の制限
- **ファイルサイズ制限**: 機能別容量上限は設けない（チーム・組織・個人の容量枠の範囲内なら自由にアップロード可能）。技術的下限として単発 PUT 100MB、100MB 超は F05.5 の Multipart Upload フロー（1 オブジェクト 5TB まで）経由。1アルバムあたり合計500件を件数上限として維持
- **一括登録の保護**: `POST /team/members/bulk` は1リクエストあたり最大100件
- **ダウンロード制限**: `allow_download = false` のアルバムは一括ダウンロード API を 403 で拒否。Workers 経由の配信 URL に `Content-Disposition: inline` を強制し、ブラウザのダウンロードダイアログを抑制。ZIP 一括ダウンロードの一時ファイルは R2 Object Lifecycle で1時間後に自動削除
- **Presigned URL の安全性**: R2 Presigned PUT URL の有効期限は写真10分・動画15分。`file_key` は UUID ベースで推測不可能。バケットは単一 `mannschaft-storage` の `gallery/` プレフィックス配下（機能分離はプレフィックスで実現）

- 番号は V6.016 から開始（V6.001〜V6.009 は F06.1_cms_blog.md、V6.010〜V6.015 は F06.4_activity_records.md で使用済み）

- ギャラリーテーブルは選択式モジュールだが、テーブル自体は常に作成（モジュール有効化チェックはアプリケーション層で実施）

- **`thumbnail_s3_key` → `thumbnail_r2_key` のカラムリネーム**: V6.021 / V6.023 時点で既に `photos.thumbnail_s3_key` として定義されていたカラムは、S3 名残削除のため `thumbnail_r2_key` にリネームする。新規環境では V6.021 を直接 `thumbnail_r2_key` として作成すればよく、既存環境向けの ALTER スクリプトは本設計書のスコープ外（運用時に F06.2 テーブルの存在・非存在を判定して `ALTER TABLE photos RENAME COLUMN thumbnail_s3_key TO thumbnail_r2_key` を適用するカスタムマイグレーションで対応）。F06.1 の `blog_media_uploads.thumbnail_r2_key`（V6.024 追加）と命名を揃える
  > **[実装時判断]** 既存本番環境で `thumbnail_s3_key` カラムが存在する場合、
  > RENAME COLUMN の Flyway スクリプト番号を他の進行中マイグレーションと
  > 衝突しないよう実装フェーズで採番すること。
  > 新規環境は V6.021 で直接 `thumbnail_r2_key` として作成するため不要。
