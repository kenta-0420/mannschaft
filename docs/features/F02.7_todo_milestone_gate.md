# F02.7: TODO マイルストーンゲート（関所機能）

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 15
> **最終更新**: 2026-04-20
> **モジュール種別**: F02.3 TODO管理の拡張機能
> **関連ドキュメント**: [F02.3 TODO管理](F02.3_todo_project.md)、[F02.2 ダッシュボード](F02.2_dashboard.md)、[F04.3 プッシュ通知](F04.3_push_notification.md)、[F10.3 監査ログ](F10.3_audit_logs.md)

---

## 1. 概要

プロジェクトの **マイルストーン（中間目標）** に「関所（ゲート）」の性質を持たせ、**前の関所が未達成のあいだは後続の関所配下の TODO を操作不能（ロック）にする** 機能。イベント準備のように段階的な達成が必要なプロジェクト（企画→手配→リハーサル→本番）で、順序を守らずにタスクが進むことを防ぐ。

マイルストーン達成の判定モードは **AUTO**（紐付く全 TODO 完了で自動達成）と **MANUAL**（管理者が手動で完了操作）の2種類を提供。達成時には後続マイルストーンが自動でアンロックされ、関係者にプッシュ通知を送信する。

本機能は F02.3 で定義済みの `project_milestones` / `todos` テーブルを拡張する形で実装するため、既存データとの互換性を保ったまま段階的に導入可能。

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全組織・チームのゲート状態参照・強制アンロック |
| ADMIN | マイルストーンの完了モード設定、強制アンロック、ゲート状態確認 |
| DEPUTY_ADMIN | `MANAGE_TODO` 権限保有時のみ、ADMIN と同権限 |
| MEMBER | ゲート状態の閲覧。ロック中 TODO に対するステータス変更・編集・削除は **423 Locked で拒否**（コメント追加はロック中も可） |
| SUPPORTER | 公開プロジェクトのゲート進捗ゲージ表示のみ |
| GUEST | 対象外 |

### 対象レベル
- [x] 組織 (Organization) — 組織プロジェクトのゲート
- [x] チーム (Team) — チームプロジェクトのゲート
- [x] 個人 (Personal) — 個人プロジェクトのゲート（ロック機構は使用可能。強制アンロックは作成者本人）

## 3. DB設計

F02.3 で定義済みの既存テーブルに対する **ALTER のみ**。新規テーブル追加なし。

### テーブル変更一覧
| テーブル名 | 変更内容 | 根拠 |
|-----------|---------|------|
| `project_milestones` | ゲート関連8カラム追加（progress_rate, is_locked, locked_by_milestone_id, completion_mode, locked_at, unlocked_at, force_unlocked, version） | ロック状態・完了モード・達成率・競合制御の管理 |
| `todos` | ロック関連2カラム追加（milestone_locked, position） | マイルストーンロック由来のロック状態・表示順 |

### `project_milestones` への追加カラム

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `progress_rate` | DECIMAL(5,2) | NO | 0.00 | 達成率（%。紐付く TODO の完了率から算出して非正規化保存） |
| `is_locked` | BOOLEAN | NO | FALSE | ロック状態フラグ。前マイルストーンが未完了の場合 TRUE |
| `locked_by_milestone_id` | BIGINT UNSIGNED | YES | NULL | ロック原因の前マイルストーン ID（FK → project_milestones。ON DELETE SET NULL）。UI でロック理由を動的表示するため、title 文字列ではなく ID 参照で保持し i18n 対応を容易にする |
| `completion_mode` | VARCHAR(16) | NO | 'AUTO' | 完了判定モード。`AUTO` / `MANUAL` |
| `locked_at` | DATETIME | YES | NULL | ロック開始日時（監査用） |
| `unlocked_at` | DATETIME | YES | NULL | アンロック日時（監査用） |
| `force_unlocked` | BOOLEAN | NO | FALSE | 強制アンロック済みフラグ。TRUE の場合、前マイルストーンが後から未完了に戻っても再ロックしない（冪等性保証） |
| `version` | BIGINT UNSIGNED | NO | 0 | 楽観的ロック用バージョン（JPA `@Version`）。TODO ステータス変更と並行するマイルストーン状態更新の競合制御に使用 |

**インデックス**（追加）
```sql
INDEX idx_pm_locked (project_id, is_locked)                        -- ロック状態での一覧取得
INDEX idx_pm_completion_mode (project_id, completion_mode)          -- AUTO マイルストーンのバッチ判定用
INDEX idx_pm_locked_by (locked_by_milestone_id)                     -- 前マイルストーン参照（FK 貼られるが明示）
```

**制約・備考**
- `completion_mode` は ENUM ではなく VARCHAR(16) で定義（MySQL ENUM は拡張時に ALTER 必要なため。アプリ層で Enum 化。将来の拡張余地あり）
- `progress_rate` は紐付く TODO の `completed / total * 100`（`deleted_at IS NULL` の TODO のみ分母に含む）を算出して非正規化。TODO のステータス変更時に同時 UPDATE。F02.3 準拠で TODO ステータスは `OPEN` / `IN_PROGRESS` / `COMPLETED` の3種のみ
- `locked_at` / `unlocked_at` は監査ログ目的。最新の状態遷移のみ保持（履歴は F10.3 監査ログで別管理）
- **循環参照防止**: `locked_by_milestone_id` は同一プロジェクト内で必ず `sort_order < 自身` のマイルストーンを指す。アプリ層でバリデーション
- `force_unlocked = TRUE` のマイルストーンは、前マイルストーンの完了/再オープンの影響を受けない（冪等）
- `version` は JPA `@Version` で自動更新。アプリは意識しない

### `todos` への追加カラム

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `milestone_locked` | BOOLEAN | NO | FALSE | 所属マイルストーンがロック中の場合 TRUE。ステータス変更 API の事前チェックに使用 |
| `position` | SMALLINT UNSIGNED | NO | 0 | 同一マイルストーン内での表示順。ドラッグ＆ドロップ並び替え用 |

**インデックス**（追加）
```sql
INDEX idx_todo_locked (project_id, milestone_id, milestone_locked)  -- マイルストーン内のロック TODO 数カウント
INDEX idx_todo_position (milestone_id, position)                    -- 表示順ソート
```

**制約・備考**
- `milestone_locked` は `milestone_id IS NOT NULL` かつ当該マイルストーンが `is_locked = TRUE` の場合のみ TRUE。NULL マイルストーン TODO は常に FALSE
- `position` は同一マイルストーン内で重複許容（UI 側で並び替え時のみ再計算）

### ER 図（ゲート関連のみ抜粋）
```
project_milestones (1) ──── (N) project_milestones [locked_by_milestone_id]
                                  └─ 前マイルストーンの完了でアンロック

project_milestones (1) ──── (N) todos [milestone_id]
                                  └─ milestone_locked はマイルストーンの is_locked に同期
```

## 4. API設計

### 新規エンドポイント一覧

| メソッド | パス | 認証 | 権限 | 説明 |
|---------|-----|------|------|------|
| GET | `/api/v1/teams/{teamId}/projects/{id}/gates` | 必要 | スコープメンバー以上 | プロジェクト全体のゲート状態サマリー |
| PATCH | `/api/v1/teams/{teamId}/projects/{id}/milestones/{mid}/completion-mode` | 必要 | ADMIN / DEPUTY_ADMIN | 完了モード変更（AUTO ↔ MANUAL） |
| PATCH | `/api/v1/teams/{teamId}/projects/{id}/milestones/{mid}/force-unlock` | 必要 | ADMIN のみ | 強制アンロック（緊急対応用） |
| PATCH | `/api/v1/teams/{teamId}/projects/{id}/milestones/{mid}/initialize-gate` | 必要 | ADMIN / DEPUTY_ADMIN | 既存プロジェクトでのゲート初期化 |
| PATCH | `/api/v1/teams/{teamId}/projects/{id}/milestones/{mid}/todos/reorder` | 必要 | ADMIN / DEPUTY_ADMIN | マイルストーン内 TODO の並び替え（position 一括更新） |

**個人スコープ**: `/api/v1/users/me/projects/{id}/...` の相対パスで同じ構造を提供（F02.3 と整合）。

### 既存エンドポイントへの追加

以下は F02.3 で定義済みのエンドポイントにフィールドを追加する形で対応。破壊的変更なし。

| パス | 追加フィールド／挙動変更 |
|-----|-----------------------|
| `GET /projects/{id}` レスポンスの `milestones[]` | `progress_rate`, `is_locked`, `locked_by_milestone_id`, `locked_by_milestone_title`, `completion_mode`, `locked_todo_count`, `force_unlocked` |
| `PATCH /todos/{id}/status` | ロック中 TODO は **423 Locked** で拒否（例外なし） |
| `PUT /todos/{id}` | ロック中 TODO の編集は **423 Locked** で拒否（タイトル・説明・期限・優先度・担当者の変更を一律拒否） |
| `PATCH /todos/{id}/assignees` | ロック中 TODO の担当者変更は **423 Locked** で拒否 |
| `DELETE /todos/{id}` | ロック中 TODO の **論理削除は許可**（プロジェクト整理目的。削除されれば progress_rate の分母から除外されるのみで、マイルストーン達成を不当に早めることはない） |
| `POST /todos/bulk-status` | 一括変更対象にロック中 TODO が含まれる場合、該当分のみスキップして結果に `skipped_locked_ids` を返す（全件エラーにはしない） |
| `PATCH /projects/{id}/milestones/{mid}/complete` | 手動完了時、後続マイルストーンのアンロックを自動実行 |
| `PATCH /projects/{id}/milestones/reorder` | sort_order 変更時、影響を受ける全マイルストーンの `locked_by_milestone_id` / `is_locked` を再計算 |
| `DELETE /projects/{id}/milestones/{mid}` | 削除時、`locked_by_milestone_id = 削除対象ID` の後続マイルストーンを再評価（前々関所が完了していればアンロック。未完了なら前々関所を新たな `locked_by_milestone_id` に設定） |

#### `PATCH /api/v1/teams/{teamId}/projects/{id}/milestones/{mid}/completion-mode`

**リクエストボディ**
```json
{
  "completion_mode": "MANUAL"
}
```

**レスポンス（200 OK）**: 更新後のマイルストーン情報（F02.3 `MilestoneResponse` 形式）

**動作**:
- `AUTO → MANUAL` への変更: 即時反映。次回 TODO 完了時に自動達成判定を実行しない
- `MANUAL → AUTO` への変更: 即時反映。現在の TODO 完了状況を即座に評価し、全完了なら即マイルストーン完了 + 後続アンロック

#### `PATCH /api/v1/teams/{teamId}/projects/{id}/milestones/{mid}/force-unlock`

ADMIN のみ実行可能。ロック状態のマイルストーンを前マイルストーンの達成状況によらず強制的にアンロックする。

**リクエストボディ**
```json
{
  "reason": "会場手配は別チームが進行中のため、リハーサル準備を先行"
}
```

**レスポンス（200 OK）**
```json
{
  "data": {
    "milestone_id": 3,
    "unlocked_at": "2026-04-20T14:30:00",
    "forced_by_user_id": 10,
    "reason": "会場手配は別チームが進行中のため、リハーサル準備を先行"
  }
}
```

**動作**:
- 対象マイルストーンの `is_locked = FALSE`、`force_unlocked = TRUE`、紐付く全 TODO の `milestone_locked = FALSE`
- 監査ログに `MILESTONE_FORCE_UNLOCKED` を記録（`reason` 必須。100文字以内。XSS対策としてサーバー側でエスケープ）
- `force_unlocked = TRUE` の場合、前マイルストーンが後から未完了に戻されても **再ロックしない**（冪等性保証）
- `reason` は監査ログ F10.3 の保持期間（デフォルト1年）と同じタイミングで削除される

**権限**（PERSONAL スコープの場合）:
- 個人プロジェクト (`scope_type = 'PERSONAL'`): プロジェクト作成者本人のみ実行可能（ADMIN ロールとは独立した判定）
- チーム/組織プロジェクト: 該当スコープの ADMIN のみ。DEPUTY_ADMIN は不可

#### `PATCH /api/v1/teams/{teamId}/projects/{id}/milestones/{mid}/initialize-gate`

既存プロジェクト（F02.7 導入前から存在するもの）でゲート機能を有効化する移行用 API。

**リクエストボディ**

**レスポンス（200 OK）**: 更新後の TODO 一覧

**バリデーション**:
- `todo_ids` 内の全 TODO が `milestone_id = {mid}` かつ `deleted_at IS NULL` であること

#### エラーレスポンス

| ステータス | 条件 |
|-----------|------|
| 400 | バリデーションエラー（`reason` 未入力、不正な `completion_mode` 値等） |
| 403 | 権限不足（MEMBER が ADMIN 専用 API を叩いた等） |
| 404 | プロジェクト/マイルストーン/TODO が該当スコープに存在しない |
| 409 | マイルストーン状態の競合（`force-unlock` 対象が既にアンロック済み等） |
| 423 | **Locked**（ロック中 TODO へのステータス変更時。`{ "error_code": "MILESTONE_LOCKED", "unlock_condition": "前マイルストーン『会場・備品手配』を完了" }`） |

## 5. ビジネスロジック

### 5.1 マイルストーン状態遷移

```
   ┌─────────────┐       (前マイルストーン未完了)       ┌─────────────┐
   │ 新規マイルストーン │  ──────────────────────>      │  LOCKED     │
   └─────────────┘                                   └─────────────┘
         │                                                │
   (前マイルストーン完了)                              (前マイルストーン完了 or force-unlock)
         │                                                │
         ↓                                                ↓
   ┌─────────────┐    (AUTO: 全TODO完了)            ┌─────────────┐
   │  UNLOCKED   │  ──or (MANUAL: 手動完了)───>    │  COMPLETED  │
   └─────────────┘                                   └─────────────┘
                      ←──── 再オープン（F02.3 既存） ────
```

### 5.2 コアフロー

#### TODO 完了時のゲート評価（MilestoneGateService.evaluateOnTodoStatusChanged）

```
1. TODO ステータスが `COMPLETED` に変化
2. 所属マイルストーン M を取得
3. M の completion_mode を確認
   - AUTO: ステップ4へ
   - MANUAL: progress_rate のみ更新して終了
4. M 配下の TODO 完了率を集計
   - 全 TODO 完了（total > 0 かつ total == completed）→ M.complete()
   - そうでなければ progress_rate 更新のみ
5. M.complete() が実行された場合:
   a. M の後続マイルストーン群 Ns（locked_by_milestone_id = M.id なもの全て。通常は1件だが削除・並び替えの結果複数になる可能性あり）を取得
   b. 各 N について `force_unlocked = TRUE` なら状態保持（スキップ）
   c. それ以外の N は N.isLocked = FALSE、N.unlocked_at = now()、N.locked_by_milestone_id = NULL
   d. アンロックした各 N 配下の全 TODO の milestone_locked = FALSE
   e. 各アンロック N について MilestoneUnlockedEvent を発行（F04.3 プッシュ通知・WebSocket 配信）
   f. 監査ログ MILESTONE_AUTO_COMPLETED（M に対して）/ MILESTONE_UNLOCKED（各 N に対して）を記録
```

**重要な判定ロジック**:
- **M に紐付く TODO が0件の場合、AUTO でも自動完了しない**（空のマイルストーンを無意味に自動完了させない）
- **論理削除済み TODO (`deleted_at IS NOT NULL`) は集計から除外**
- TODO ステータスは F02.3 準拠で `OPEN` / `IN_PROGRESS` / `COMPLETED` の3種。「完了」判定は `status = COMPLETED` のみ

#### マイルストーン手動完了時（既存 API 拡張）

```
1. ADMIN / DEPUTY_ADMIN が PATCH /milestones/{mid}/complete を実行
2. M.isCompleted = TRUE、M.completedAt = now()
3. 後続マイルストーンのアンロック処理（上記 5a 以降と同じ）
```

#### マイルストーン削除時（既存 API 拡張）

```
1. ADMIN / DEPUTY_ADMIN が DELETE /milestones/{mid} を実行
2. 削除前に後続マイルストーン N（locked_by_milestone_id = mid）を取得
3. M を物理削除（F02.3 既存動作。論理削除なしのため ON DELETE SET NULL が発火）
4. 後続マイルストーン N について再評価:
   - mid の前の前マイルストーン P（sort_order - 2）を取得
   - P が存在して COMPLETED ならアンロック（N.isLocked = FALSE）
   - P が未完了または存在しない（M が最初の関所だった）なら locked_by_milestone_id を P.id or NULL に振り替え
5. MilestoneChainRebuiltEvent を発行
```

#### 新規マイルストーン追加時

```
1. ADMIN が POST /milestones で新規作成（M_new）
2. プロジェクト内で sort_order 末尾に配置（既存の最大 + 1）
3. 直前のマイルストーン M_prev（sort_order - 1）が存在する場合:
   - M_prev が未完了なら M_new.isLocked = TRUE、M_new.locked_by_milestone_id = M_prev.id
   - M_prev が完了済みなら M_new.isLocked = FALSE
4. 既存の新規マイルストーン挿入（sort_order を途中に挿入）は F02.3 既存の sort_order 再計算ロジックに従う。この場合、影響を受けるマイルストーン全てのロック状態を再計算
```

#### マイルストーン並び替え時（新規 API: `PATCH /milestones/reorder`）

```
1. ADMIN が並び替え後の milestone_id 配列を送信
2. 各マイルストーンの sort_order を 0, 1, 2, ... に再割当
3. **循環参照検知**: 並び替え後の順序で locked_by_milestone_id が sort_order <= 自身 を指す場合、エラー。アプリ層で sort_order < 自身 のみ許容
4. 各マイルストーン M（sort_order 昇順）について:
   - M.force_unlocked == TRUE なら状態保持（再ロックしない）
   - M が sort_order = 0 ならロック対象外
   - M_prev (sort_order - 1) が完了済み → M.isLocked = FALSE
   - M_prev が未完了 → M.isLocked = TRUE、M.locked_by_milestone_id = M_prev.id
5. M 配下の TODO の milestone_locked を同期
6. MilestoneChainRebuiltEvent を発行
```

#### プロジェクト複製時（F02.3 の該当機能がある場合）

```
1. マイルストーンは複製。各マイルストーンは is_completed = FALSE で複製
2. progress_rate = 0.00、is_locked は sort_order に従って再計算
3. force_unlocked は常に FALSE にリセット（複製先ではクリーンな状態）
4. TODO は milestone_locked を再計算
```

#### プロジェクト作成時（F02.3 既存動作＋追加）

F02.3 既存: プロジェクト作成時にマイルストーンは0件。ゲート初期化は **マイルストーン追加時に自動実行**。

#### ロック中 TODO のステータス変更拒否

```
1. PATCH /todos/{id}/status のエントリーで TODO を取得
2. `todos.milestone_locked == TRUE` なら 423 Locked を返す
3. レスポンスボディに unlock_condition（前マイルストーンのタイトル）を含める
4. MEMBER にはロックバッジを表示して UI 側でも操作を disabled にする（二重防御）
```

**例外**:
- 個人プロジェクトの場合、作成者本人の TODO 操作も通常どおりロック対象（ADMIN 権限判定ではなく「作成者 = PERSONAL スコープ所有者」としての自分のロックに従う）
- **論理削除（DELETE API による `deleted_at` セット）のみロック中も許可**（プロジェクト整理目的。削除されれば分母から外れるだけで、達成判定を不当に早める危険はない）

#### ロック中 TODO への通知抑制

ロック中 TODO は操作できない状態なので、以下の通知を抑制する：

| notification_type | ロック中の挙動 |
|---|---|
| `TODO_DUE_TOMORROW` | **送信しない**（ロック解除されるまで延期されたとみなす） |
| `TODO_OVERDUE` | **送信しない**（アンロック後に改めて判定） |
| `TODO_ASSIGNED` | 送信する（担当の通知自体は重要） |
| `TODO_COMMENT_ADDED` | 送信する（議論は可能） |

通知抑制は F04.3 の期限リマインダーバッチ内で `milestone_locked = FALSE` を条件に追加する形で実装。

### 5.3 既存プロジェクトへの移行戦略

| 既存プロジェクト | 扱い |
|---------------|------|
| マイルストーン0件 | 影響なし |
| マイルストーン1件 | 最初のマイルストーンは常にアンロック。影響なし |
| マイルストーン2件以上 | デフォルトでは **全てアンロック状態**。ADMIN が `initialize-gate` API を手動実行してゲートを有効化 |

**理由**: 既存マイルストーンに対する自動ロックは運用中のプロジェクトに予期せぬ影響を与えるため、必ず ADMIN の明示的操作で有効化させる。UI にも「ゲートを有効化する」ボタンを設置。

### 5.4 競合制御

- マイルストーン状態変更は `@Transactional`（`REPEATABLE_READ`）+ JPA の `@Version`（`project_milestones.version` カラム）で保護
- TODO ステータス変更と並行してマイルストーン状態更新が発生した場合、`OptimisticLockException` を検出したらサービス層で自動リトライ1回
- リトライでも失敗する場合は **409 Conflict** を返し、クライアントに再読み込みを促す（レスポンスに `retry_after_seconds: 2` を含める）
- `MilestoneGateService.evaluateOnTodoStatusChanged()` は TODO ステータス変更トランザクション内で同一 Transaction として実行（原子性保証）

## 6. セキュリティ考慮事項

### 6.1 認可・アクセス制御
- **ゲート関連 API の認可**: 対象プロジェクトの `scope_type` + `scope_id` に所属するユーザーのみアクセス可能。IDOR 防止のため `teamId` / `projectId` / `milestoneId` の三重検証
- **強制アンロック権限**: ADMIN のみ実行可能。DEPUTY_ADMIN は `MANAGE_TODO` 権限があっても **強制アンロックは不可**（運用事故防止）
- **完了モード変更権限**: ADMIN / DEPUTY_ADMIN（`MANAGE_TODO`）。MEMBER は不可

### 6.2 ロックバイパスの防止
- **サーバーサイド二重チェック**: `milestone_locked = TRUE` の TODO に対するステータス変更・担当者変更・期限変更・編集は全て 423 Locked で拒否
- **論理削除（DELETE → `deleted_at` セット）のみロック中も許可**: プロジェクト整理目的。削除された TODO は分母から外れるのみで達成判定を不当に早めない
- **UI 側のクライアント検証は信頼しない**: Vue 側で disabled にしていても API 側で必ず再検証
- **バッチ処理からの変更も例外なし**: 期限超過通知等で TODO ステータスを変更する場合も `milestone_locked` を尊重

### 6.3 監査ログ（F10.3 連携）
| action_type | 記録タイミング | 記録内容 |
|---|---|---|
| `MILESTONE_LOCKED` | 初回ロック時（新規作成 or 初期化 API） | project_id, milestone_id, locked_by_milestone_id |
| `MILESTONE_UNLOCKED` | 前マイルストーン達成による自動アンロック | project_id, milestone_id, triggered_by_milestone_id |
| `MILESTONE_FORCE_UNLOCKED` | 強制アンロック | project_id, milestone_id, forced_by_user_id, reason |
| `MILESTONE_COMPLETION_MODE_CHANGED` | 完了モード変更 | project_id, milestone_id, old_mode, new_mode |
| `MILESTONE_AUTO_COMPLETED` | AUTO モードによる自動完了 | project_id, milestone_id |
| `MILESTONE_CHAIN_REBUILT` | 削除・並び替えによる再計算 | project_id, affected_milestone_ids[] |

全ての監査ログは `actor_user_id` / `occurred_at` / `ip_address` を含む。監査ログの命名は F04.3 プッシュ通知の `notification_type` と統一（`MILESTONE_UNLOCKED` で一致）。

**GDPR/プライバシー対応**:
- `force_unlocked.reason` フィールドは F10.3 の監査ログ保持ポリシー（デフォルト1年）に従って自動削除
- PII（個人情報）を reason に含めないよう、UI 側でバリデーション注意喚起を表示

### 6.4 レート制限
- ゲート API (`GET /gates`): 1ユーザーあたり1分60回（ダッシュボード更新頻度を考慮し緩め）
- 強制アンロック API: 1ユーザーあたり1時間10回（誤操作防止）
- 完了モード変更 API: 1ユーザーあたり1分10回
- ゲート初期化 API: 1プロジェクトあたり10秒1回（重い処理のため絞る）
- TODO 並び替え API: 1ユーザーあたり1分30回

### 6.5 DoS・パフォーマンス対策
- プロジェクト内のマイルストーン上限: **50件**（F02.3 では上限なしだが本機能で追加。大量マイルストーンによるカスケード更新の負荷対策）
- マイルストーン内の TODO 上限: F02.3 既存のプロジェクト内 TODO 上限（設計時 500件）を踏襲
- ゲート初期化 API は最大50件のマイルストーン更新 + 関連 TODO 更新を1トランザクションで実施。それ以上の場合は 400 エラー

### 6.6 データ整合性
- `project_milestones.locked_by_milestone_id` の FK は `ON DELETE SET NULL`。前マイルストーンが削除された場合は自動的に NULL になり、次回の操作で再評価される
- `progress_rate` は非正規化フィールドだが、TODO ステータス変更時の原子的 UPDATE で整合性保証。不整合検知用のデイリーバッチ（`MilestoneReconciliationBatch`）で差異があれば補正（F02.3 の `projects.progress_rate` 補正バッチと同日時に実行）
- force-unlock の reason フィールド: SQL インジェクション対策として Parameterized Query、XSS 対策としてサーバー側で HTML エスケープ保存

## 7. Flywayマイグレーション

```
V15.001__alter_project_milestones_add_gate_columns.sql  -- project_milestones に8カラム追加
V15.002__alter_todos_add_milestone_lock_columns.sql      -- todos に2カラム追加
V15.003__backfill_existing_milestones_progress_rate.sql  -- 既存マイルストーンの progress_rate 初期値を計算
```

### V15.001 の詳細

```sql
ALTER TABLE project_milestones
  ADD COLUMN progress_rate DECIMAL(5,2) NOT NULL DEFAULT 0.00 AFTER is_completed,
  ADD COLUMN is_locked BOOLEAN NOT NULL DEFAULT FALSE AFTER progress_rate,
  ADD COLUMN locked_by_milestone_id BIGINT UNSIGNED NULL AFTER is_locked,
  ADD COLUMN completion_mode VARCHAR(16) NOT NULL DEFAULT 'AUTO' AFTER locked_by_milestone_id,
  ADD COLUMN locked_at DATETIME NULL AFTER completion_mode,
  ADD COLUMN unlocked_at DATETIME NULL AFTER locked_at,
  ADD COLUMN force_unlocked BOOLEAN NOT NULL DEFAULT FALSE AFTER unlocked_at,
  ADD COLUMN version BIGINT UNSIGNED NOT NULL DEFAULT 0 AFTER force_unlocked,
  ADD CONSTRAINT fk_pm_locked_by FOREIGN KEY (locked_by_milestone_id) REFERENCES project_milestones(id) ON DELETE SET NULL,
  ADD INDEX idx_pm_locked (project_id, is_locked),
  ADD INDEX idx_pm_completion_mode (project_id, completion_mode),
  ADD INDEX idx_pm_locked_by (locked_by_milestone_id);
```

### V15.002 の詳細

```sql
ALTER TABLE todos
  ADD COLUMN milestone_locked BOOLEAN NOT NULL DEFAULT FALSE AFTER milestone_id,
  ADD COLUMN position SMALLINT UNSIGNED NOT NULL DEFAULT 0 AFTER milestone_locked,
  ADD INDEX idx_todo_locked (project_id, milestone_id, milestone_locked),
  ADD INDEX idx_todo_position (milestone_id, position);
```

### V15.003 の詳細（既存データのバックフィル）

```sql
-- 既存マイルストーンの progress_rate を TODO 完了率から計算して更新
-- F02.3 準拠で TODO ステータスは OPEN / IN_PROGRESS / COMPLETED の3種。論理削除済みは deleted_at で判定
UPDATE project_milestones pm
JOIN (
  SELECT
    milestone_id,
    COUNT(*) AS total,
    SUM(CASE WHEN status = 'COMPLETED' THEN 1 ELSE 0 END) AS completed
  FROM todos
  WHERE milestone_id IS NOT NULL
    AND deleted_at IS NULL
  GROUP BY milestone_id
) t ON pm.id = t.milestone_id
SET pm.progress_rate = ROUND(t.completed * 100.0 / t.total, 2);

-- position は既存 TODO の作成順を踏襲（created_at 昇順で 0, 1, 2, ...）
-- MySQL 8.0 の ROW_NUMBER を使用
-- Flyway により一度だけ実行される（チェックサム管理）ため冪等性は不要
UPDATE todos t1
JOIN (
  SELECT
    id,
    ROW_NUMBER() OVER (PARTITION BY milestone_id ORDER BY created_at, id) - 1 AS pos
  FROM todos
  WHERE milestone_id IS NOT NULL
    AND deleted_at IS NULL
) t2 ON t1.id = t2.id
SET t1.position = t2.pos;
```

**マイグレーション上の注意点**
- V15.001 の `FOREIGN KEY` 自己参照は MySQL 8.0 でサポート済み
- 既存の `is_locked` / `milestone_locked` の初期値は全て FALSE なので、アプリケーション再起動後も既存プロジェクトは影響を受けない（5.3 の移行戦略に従う）
- V15.003 はオンライン実行可能（大量データでは数分かかる可能性あり。メンテナンス時間帯推奨）
- ロールバック手順: `ALTER TABLE ... DROP COLUMN` で追加カラムを削除すれば V15.001〜V15.002 の影響は除去可能。V15.003 の UPDATE 結果は DROP COLUMN で消滅するためロールバック不要（カラム自体が無くなる）
