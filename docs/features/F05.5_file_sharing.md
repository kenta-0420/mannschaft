# F05.5: ファイル共有

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 5
> **最終更新**: 2026-03-14
> **関連ドキュメント**: F01.3 テンプレート・モジュール（デフォルト機能 #8）、.claudecode.md §26 Pre-signed URL フロー

---

## 1. 概要

チーム・組織内のドキュメント・資料を共有するストレージ機能。階層フォルダ管理、アクセス権限制御（チーム内/グループ内/個人指定）、バージョン管理を備え、**Cloudflare R2 の S3 互換 API で発行する Presigned URL** による R2 直接アップロード/ダウンロードで高速なファイル転送を実現する。チームごとのストレージ使用量をリアルタイムに追跡し、無料枠を超過した場合はストレージプランへのアップグレードを促す。動画・大容量ファイル対応として R2 Multipart Upload を採用する。

---

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全チーム/組織のファイル参照。ストレージプラン管理 |
| ADMIN | フォルダ・ファイルの CRUD・権限設定・ストレージプラン変更 |
| DEPUTY_ADMIN | `MANAGE_FILES` 権限を持つ場合: フォルダ・ファイルの CRUD・権限設定 |
| MEMBER | 権限が付与されたフォルダ・ファイルの閲覧・アップロード・ダウンロード |
| SUPPORTER | 権限が付与されたフォルダ・ファイルの閲覧・ダウンロードのみ（アップロード不可） |
| GUEST | 対象外 |

### 対象レベル
- [x] 組織 (Organization) — 組織共有フォルダ
- [x] チーム (Team) — チーム共有フォルダ
- [x] 個人 (Personal) — 個人用ストレージ（チームのストレージ枠を消費）

### テーブル定義

#### `shared_folders`

フォルダの階層構造。隣接リスト方式（`parent_id`）で管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（TEAM / ORGANIZATION / PERSONAL） |
| `team_id` | BIGINT UNSIGNED | YES | NULL | FK → teams。TEAM / PERSONAL の場合に設定 |
| `organization_id` | BIGINT UNSIGNED | YES | NULL | FK → organizations。ORGANIZATION の場合に設定 |
| `user_id` | BIGINT UNSIGNED | YES | NULL | FK → users。PERSONAL の場合に設定 |
| `parent_id` | BIGINT UNSIGNED | YES | NULL | FK → shared_folders（自己参照）。NULL = ルートフォルダ |
| `name` | VARCHAR(255) | NO | — | フォルダ名 |
| `description` | VARCHAR(500) | YES | NULL | フォルダの説明 |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `version` | BIGINT | NO | 0 | 楽観的ロックバージョン（@Version） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_shared_folders_team (team_id, parent_id, name)               -- チーム別のフォルダ一覧
INDEX idx_shared_folders_org (organization_id, parent_id, name)        -- 組織別のフォルダ一覧
INDEX idx_shared_folders_user (user_id, parent_id, name)               -- 個人別のフォルダ一覧
INDEX idx_shared_folders_parent (parent_id)                             -- 子フォルダ取得
```

**制約・備考**
- `scope_type` のスコープ制約:
  - `TEAM`: `team_id` NOT NULL, `organization_id` NULL, `user_id` NULL
  - `ORGANIZATION`: `team_id` NULL, `organization_id` NOT NULL, `user_id` NULL
  - `PERSONAL`: `team_id` NOT NULL, `organization_id` NULL, `user_id` NOT NULL（個人用だがチームのストレージ枠を消費）
- フォルダ名は同一親フォルダ内で一意（Service 層バリデーション。大文字/小文字を区別しない）
- フォルダ階層の最大深度: 10階層（Service 層で検証。深すぎる階層はナビゲーション UX を損なうため）
- 論理削除時: 配下のファイル・サブフォルダはそのまま保持（フォルダ復元時に復活）。ストレージ使用量は論理削除時に減算しない（完全削除バッチで減算）
- ルートフォルダ（`parent_id = NULL`）はスコープごとに自動作成。ユーザーが明示的に作成するのはサブフォルダ以降

#### `shared_files`

ファイルのメタデータ。実ファイルは **Cloudflare R2** に保存し、`file_key` で参照（カラム名は歴史的経緯で維持）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `folder_id` | BIGINT UNSIGNED | NO | — | FK → shared_folders。ON DELETE CASCADE |
| `name` | VARCHAR(255) | NO | — | ファイル名（拡張子含む） |
| `file_key` | VARCHAR(500) | NO | — | R2 オブジェクトキー（最新バージョン。カラム名は歴史的経緯で `file_key` を維持） |
| `file_size` | BIGINT UNSIGNED | NO | — | ファイルサイズ（バイト。最新バージョン） |
| `content_type` | VARCHAR(100) | NO | — | MIME タイプ |
| `description` | VARCHAR(500) | YES | NULL | ファイルの説明 |
| `current_version` | INT UNSIGNED | NO | 1 | 現在のバージョン番号 |
| `uploaded_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `download_count` | INT UNSIGNED | NO | 0 | ダウンロード数（denormalize） |
| `version` | BIGINT | NO | 0 | 楽観的ロックバージョン（@Version） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_shared_files_folder (folder_id, name)                -- フォルダ内のファイル一覧
INDEX idx_shared_files_uploader (uploaded_by, created_at DESC)  -- アップロード者別
FULLTEXT INDEX ft_shared_files_name (name)                      -- ファイル名検索
```

**制約・備考**
- `file_key`: Cloudflare R2 オブジェクトキー。フォーマット: `files/{scopeType}/{scopeId}/{uuid}.{ext}`（§26 準拠）
- `file_size`: 最新バージョンのサイズ。バージョン更新時に上書き。ストレージ使用量の計算に使用
- **ファイルサイズ上限**:
  - **通常アップロード（1リクエスト PUT）**: 1ファイル **100MB**。テキスト・PDF・画像・Office ドキュメント等の日常的なファイル向け
  - **Multipart Upload**: 1ファイル **最大 5GB**。動画・大容量アーカイブ・バックアップ等。R2 の S3 互換 Multipart Upload API（`CreateMultipartUpload` → 各パートを Presigned PUT → `CompleteMultipartUpload`）を使用し、クライアントが 5〜10MB のパートに分割してアップロード
  - 上限超過時は 413 Payload Too Large を返却
- ファイル名は同一フォルダ内で一意（Service 層バリデーション）。重複時は自動リネーム（`report.pdf` → `report (1).pdf`）
- `download_count`: ダウンロード URL 発行時にアトミック更新

#### `shared_file_versions`

ファイルのバージョン履歴。同一ファイルの過去バージョンを保持し、ロールバックを可能にする。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `file_id` | BIGINT UNSIGNED | NO | — | FK → shared_files。ON DELETE CASCADE |
| `version_number` | INT UNSIGNED | NO | — | バージョン番号（1始まり） |
| `file_key` | VARCHAR(500) | NO | — | R2 オブジェクトキー（このバージョンの実体） |
| `file_size` | BIGINT UNSIGNED | NO | — | ファイルサイズ（バイト） |
| `content_type` | VARCHAR(100) | NO | — | MIME タイプ |
| `uploaded_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `comment` | VARCHAR(500) | YES | NULL | バージョンコメント（例: 「第2稿修正」） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_file_versions (file_id, version_number)           -- 同一ファイルのバージョン番号一意
INDEX idx_file_versions_file (file_id, version_number DESC)     -- バージョン一覧（新しい順）
```

**制約・備考**
- バージョン保持上限: 1ファイルあたり最大20バージョン。超過時は最古バージョンを自動削除（R2 オブジェクトも削除。ストレージ使用量を減算）
- 新バージョンアップロード時: `shared_files.file_key` / `file_size` / `content_type` / `current_version` を更新し、旧バージョンの情報を `shared_file_versions` に INSERT
- 全バージョンのファイルサイズ合計がストレージ使用量に加算される。R2 はバージョン管理機能を使わず、各バージョンを別オブジェクトキーで管理する

#### `file_permissions`

フォルダ・ファイル単位のアクセス権限。ポリモーフィック方式で権限付与先（ユーザー/ロール/グループ）を指定。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `target_type` | VARCHAR(20) | NO | — | 権限対象種別（FOLDER / FILE） |
| `target_id` | BIGINT UNSIGNED | NO | — | 対象の shared_folders.id または shared_files.id |
| `grantee_type` | VARCHAR(20) | NO | — | 権限付与先種別（USER / ROLE / GROUP） |
| `grantee_id` | BIGINT UNSIGNED | NO | — | 付与先の users.id / roles.id / group_hierarchy.id |
| `permission` | VARCHAR(20) | NO | — | 権限レベル（VIEW / DOWNLOAD / UPLOAD / MANAGE） |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_file_permissions (target_type, target_id, grantee_type, grantee_id)  -- 同一権限の重複防止
INDEX idx_file_permissions_target (target_type, target_id)                           -- 対象別の権限一覧
INDEX idx_file_permissions_grantee (grantee_type, grantee_id)                       -- 付与先別の権限一覧
```

**制約・備考**
- `permission` の階層（上位は下位を包含）:
  - `VIEW`: ファイル名・メタデータの閲覧のみ
  - `DOWNLOAD`: VIEW + ファイルのダウンロード
  - `UPLOAD`: DOWNLOAD + ファイルのアップロード・バージョン更新
  - `MANAGE`: UPLOAD + ファイル/フォルダの削除・移動・権限変更
- フォルダの権限は配下のファイル・サブフォルダに継承される（Service 層で再帰的に評価）。ファイル個別に権限を設定した場合はフォルダ権限より優先
- `grantee_type = 'ROLE'` の場合: そのロールを持つ全ユーザーに適用（例: `MEMBER` ロール → チーム全メンバー）
- ADMIN / DEPUTY_ADMIN（MANAGE_FILES）は暗黙的に全フォルダ・ファイルへの MANAGE 権限を持つ
- 個人フォルダ（`scope_type = 'PERSONAL'`）は所有者に暗黙的 MANAGE 権限。他ユーザーへの共有は `file_permissions` で明示設定

#### `shared_file_stars`

ファイルの個人用お気に入り。よく使うファイルに個人的にスターを付けて素早くアクセスする。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `file_id` | BIGINT UNSIGNED | NO | — | FK → shared_files。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_file_stars_user_file (user_id, file_id)        -- 同一ファイルへの重複スター防止
INDEX idx_file_stars_user (user_id, created_at DESC)          -- スター付きファイル一覧
```

**制約・備考**
- 他ユーザーには非公開（本人のみ参照可能）
- ファイルが論理削除されてもスターレコードは保持（ゴミ箱から復元時に復活）。完全削除時に CASCADE で削除

#### `shared_file_comments`

ファイルに紐づくコメントスレッド。ファイルに関するディスカッションをチャットと分離して集約。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `file_id` | BIGINT UNSIGNED | NO | — | FK → shared_files。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `body` | TEXT | NO | — | コメント本文（最大5,000文字） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_file_comments_file (file_id, created_at ASC)        -- ファイル別コメント一覧（古い順）
```

**制約・備考**
- ファイルへの VIEW 以上の権限を持つユーザーがコメント閲覧可能
- コメント投稿は DOWNLOAD 以上の権限が必要（VIEW のみのユーザーはコメント閲覧のみ）
- コメント削除は投稿者本人 or ADMIN / MANAGE_FILES 権限保持者
- コメント投稿時に ApplicationEvent: `FileCommentedEvent` を発行（ファイルのアップロード者 + 他のコメント者に通知）

#### `shared_file_links`

外部公開用の共有リンク。認証不要の期限付き URL でチームメンバー以外（取引先・保護者等）にファイルを共有。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `file_id` | BIGINT UNSIGNED | NO | — | FK → shared_files。ON DELETE CASCADE |
| `token` | VARCHAR(64) | NO | — | 共有トークン（`SecureRandom` + Base64URL で生成。推測不可能） |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `password_hash` | VARCHAR(255) | YES | NULL | パスワード保護（bcrypt ハッシュ。NULL = パスワードなし） |
| `expires_at` | DATETIME | YES | NULL | 有効期限（NULL = 無期限。最大30日を推奨） |
| `download_limit` | INT UNSIGNED | YES | NULL | ダウンロード回数上限（NULL = 無制限） |
| `download_count` | INT UNSIGNED | NO | 0 | 現在のダウンロード回数 |
| `is_active` | BOOLEAN | NO | TRUE | 手動無効化フラグ |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_file_links_token (token)                        -- トークン一意 + 検索用
INDEX idx_file_links_file (file_id)                            -- ファイル別のリンク一覧
```

**制約・備考**
- 共有リンクの URL フォーマット: `/shared/{token}`（フロントエンドの SSR ページ。SEO 不要のため noindex）
- ダウンロード時: `token` で `shared_file_links` を検索 → `is_active` / `expires_at` / `download_limit` を検証 → パスワード設定時はパスワード入力画面を表示 → Pre-signed URL を発行
- MANAGE 権限を持つユーザーのみリンクを作成・無効化可能
- 監査ログ: `FILE_LINK_CREATED` / `FILE_LINK_REVOKED` を記録
- 1ファイルあたりの有効リンク数上限: 10件（Service 層バリデーション）

#### `shared_file_tags`

ファイルのタグ付け。フォルダ構造では分類しきれないファイルをタグで横断的に管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `file_id` | BIGINT UNSIGNED | NO | — | FK → shared_files。ON DELETE CASCADE |
| `tag_name` | VARCHAR(50) | NO | — | タグ名（例: 「契約書」「請求書」「マニュアル」） |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_file_tags_file_tag (file_id, tag_name)          -- 同一ファイルへの同一タグ重複防止
INDEX idx_file_tags_tag (tag_name)                             -- タグ別のファイル検索
INDEX idx_file_tags_file (file_id)                             -- ファイル別のタグ一覧
```

**制約・備考**
- タグ名は正規化（前後の空白除去、全角→半角統一）。大文字/小文字を区別しない
- 1ファイルあたり最大20タグ
- タグ追加は UPLOAD 以上の権限が必要
- タグ名のオートコンプリート: チーム/組織内の既存タグから前方一致で候補表示（`GET /files/tags/suggest?q=契&team_id=5`）

## 4. API設計

### エンドポイント一覧
| メソッド | パス | 認証 | 説明 |
|---------|-----|------|------|
| GET | `/api/v1/files/folders` | 必要 | フォルダ一覧取得 |
| POST | `/api/v1/files/folders` | 必要 | フォルダ作成 |
| GET | `/api/v1/files/folders/{id}` | 必要 | フォルダ詳細取得（配下のファイル・サブフォルダ含む） |
| PUT | `/api/v1/files/folders/{id}` | 必要 | フォルダ更新（リネーム・移動） |
| DELETE | `/api/v1/files/folders/{id}` | 必要 | フォルダ削除（論理削除。配下ごと） |
| GET | `/api/v1/files` | 必要 | ファイル一覧取得（検索・フィルタ） |
| POST | `/api/v1/files/upload-url` | 必要 | アップロード用の R2 Presigned PUT URL 発行（100MB 以下の単発アップロード） |
| POST | `/api/v1/files/multipart/start` | 必要 | R2 Multipart Upload 開始（100MB〜5GB の大容量・動画ファイル向け） |
| POST | `/api/v1/files/multipart/{uploadId}/part-url` | 必要 | Multipart の各パート用 Presigned URL 発行 |
| POST | `/api/v1/files/multipart/{uploadId}/complete` | 必要 | Multipart Upload 完了処理 |
| DELETE | `/api/v1/files/multipart/{uploadId}` | 必要 | Multipart Upload 中断（アップロード済みパート破棄） |
| POST | `/api/v1/files` | 必要 | ファイル登録（R2 アップロード完了後。通常/Multipart 共通） |

#### `POST /api/v1/files/upload-url`

100MB 以下の単発アップロード用に **Cloudflare R2 の Presigned PUT URL** を発行する（`backend/.claudecode.md` §26 準拠、発行元エンドポイントは R2）。

**リクエストボディ**
```json
{
  "folder_id": 1,
  "file_name": "企画書.pdf",
  "content_type": "application/pdf",
  "file_size": 2048000
}
```

- `folder_id`: 必須。アップロード先フォルダ
- `file_size`: 最大 100MB（104,857,600 バイト）。100MB を超えるファイルは `POST /api/v1/files/multipart/start` を使用
- ストレージ枠チェック: チームの `storage_used_bytes + file_size` が上限を超える場合は 409 エラー

**レスポンス（200 OK）**
```json
{
  "data": {
    "upload_url": "https://{accountId}.r2.cloudflarestorage.com/mannschaft-files/files/TEAM/5/abc123.pdf?X-Amz-Signature=...",
    "file_key": "files/TEAM/5/abc123.pdf",
    "expires_in": 600
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | content_type が許可されていない / file_size が上限超過（100MB 超は Multipart 利用を促すメッセージを返却） |
| 403 | フォルダへの UPLOAD 権限がない / SUPPORTER はアップロード不可 |
| 404 | フォルダが存在しない |
| 409 | ストレージ枠不足（プランアップグレードを促すメッセージを含む） |

#### `POST /api/v1/files/multipart/start`

動画・大容量ファイル向けに R2 Multipart Upload を開始する。R2 の S3 互換 `CreateMultipartUpload` API を呼び出し、`upload_id` を発行する。

**リクエストボディ**
```json
{
  "folder_id": 1,
  "file_name": "practice_video.mp4",
  "content_type": "video/mp4",
  "file_size": 2147483648,
  "part_count": 205,
  "part_size": 10485760
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `folder_id` | Long | Yes | アップロード先フォルダ |
| `file_name` | String | Yes | 元ファイル名 |
| `content_type` | String | Yes | MIME タイプ。動画・アーカイブ等のホワイトリスト |
| `file_size` | Long | Yes | 合計ファイルサイズ（バイト）。上限 5GB |
| `part_count` | Integer | Yes | 分割数（1〜10,000。R2 / S3 の上限に合わせる） |
| `part_size` | Long | Yes | 1パートのサイズ（最終パート除く各パート 5MB 以上推奨） |

**処理内容**
1. フォルダへの UPLOAD 権限チェック
2. `file_size` 上限チェック（5GB）
3. `StorageQuotaService.checkQuota()` で空き容量確認
4. R2 オブジェクトキー生成: `files/{scopeType}/{scopeId}/{uuid}.{ext}`
5. R2 `CreateMultipartUpload` 呼び出し → `upload_id` 取得

**レスポンス（200 OK）**
```json
{
  "data": {
    "upload_id": "2~abc123xyz...",
    "file_key": "files/TEAM/5/def456.mp4",
    "part_count": 205,
    "part_size": 10485760
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | content_type が許可されていない / file_size が 5GB 超過 / part_count が範囲外 |
| 403 | UPLOAD 権限なし / SUPPORTER |
| 409 | ストレージ枠不足 |

---

#### `POST /api/v1/files/multipart/{uploadId}/part-url`

Multipart の各パート用の Presigned PUT URL を発行する。クライアントはパート番号を指定してこのエンドポイントを呼び、R2 に直接 PUT する。

**リクエストボディ**
```json
{
  "file_key": "files/TEAM/5/def456.mp4",
  "part_numbers": [1, 2, 3, 4, 5]
}
```

- バックエンドは `part_numbers` 配列の各要素について `UploadPart` 用の Presigned URL を発行（複数をまとめて返すことで効率化）

**レスポンス（200 OK）**
```json
{
  "data": {
    "part_urls": [
      { "part_number": 1, "upload_url": "https://{accountId}.r2.cloudflarestorage.com/...?partNumber=1&uploadId=..." },
      { "part_number": 2, "upload_url": "..." }
    ],
    "expires_in": 600
  }
}
```

- クライアントは各 URL に対して PUT で該当パートをアップロードし、レスポンスの `ETag` ヘッダを記録

#### `POST /api/v1/files/multipart/{uploadId}/complete`

全パートアップロード完了後、R2 に `CompleteMultipartUpload` を呼び出して最終オブジェクトを組み立てる。

**リクエストボディ**
```json
{
  "file_key": "files/TEAM/5/def456.mp4",
  "parts": [
    { "part_number": 1, "etag": "\"abc123\"" },
    { "part_number": 2, "etag": "\"def456\"" }
  ]
}
```

**処理内容**
1. R2 `CompleteMultipartUpload` を呼び出し
2. R2 `HeadObject` で最終的な `file_size` を取得し、リクエスト時の値と照合（差分があれば 400）
3. この時点ではまだ DB に登録しない。クライアントは続けて `POST /api/v1/files` でメタデータ登録

**レスポンス（200 OK）**
```json
{
  "data": {
    "file_key": "files/TEAM/5/def456.mp4",
    "file_size": 2147483648
  }
}
```

#### `POST /api/v1/files`

R2 アップロード完了後（単発 PUT または Multipart Complete 後）にファイルを DB に登録する。

**リクエストボディ**
```json
{
  "folder_id": 1,
  "name": "企画書.pdf",
  "file_key": "files/TEAM/5/abc123.pdf",
```

1. チームの `storage_used_bytes` をアトミックに加算
2. `shared_file_versions` にバージョン1を INSERT

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | バリデーションエラー / file_key が R2 に存在しない |
| 403 | フォルダへの UPLOAD 権限がない |

#### `POST /api/v1/files/{id}/versions`

既存ファイルの新バージョンをアップロードする。

**リクエストボディ**
```json
{
  "file_key": "files/TEAM/5/def456.pdf",
  "file_size": 2560000,
  "content_type": "application/pdf",
  "comment": "第2稿: 価格表を更新",
  "version": 1
}
```

- `version`: 必須。楽観的ロック（同時に別ユーザーがバージョン更新するのを防止）
- バージョン上限（20件）超過時: 最古バージョンを自動削除 + R2 オブジェクト削除 + ストレージ使用量減算

**レスポンス（201 Created）**: 更新後のファイル詳細を返却

**副作用**:
1. `shared_files` の `file_key` / `file_size` / `content_type` / `current_version` を更新
2. 旧バージョン情報を `shared_file_versions` に INSERT
3. ストレージ使用量: 新バージョン分を加算（旧バージョンも保持するため減算なし。上限超過で自動削除時に減算）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | バリデーションエラー / file_key が R2 に存在しない |
| 403 | ファイルへの UPLOAD 権限がない |
| 404 | ファイルが存在しない |
| 409 | 楽観的ロック競合 / ストレージ枠不足 |

#### `POST /api/v1/files/{id}/versions/{versionNumber}/restore`

過去バージョンを最新バージョンとして復元する。

**リクエストボディ**
```json
{
  "version": 3
}
```

- `version`: 必須。楽観的ロック
- 復元処理: 指定バージョンの `file_key` / `file_size` / `content_type` を `shared_files` にコピーし、`current_version` を +1。現在の最新バージョンを `shared_file_versions` に INSERT
- ストレージ使用量: 復元先バージョンのファイルを R2 内で CopyObject（新しい file_key を生成）するため、新バージョン分のサイズが加算される

**レスポンス（200 OK）**: 復元後のファイル詳細を返却

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|

## 5. ビジネスロジック

### ファイルプレビュー

```
1. content_type に基づくインラインプレビュー（フロントエンド実装）:
   a. application/pdf: <iframe src="{Workers 経由の署名付き download URL}"> で表示
   b. image/*: <img src="{Workers 経由の署名付き download URL}" loading="lazy"> で表示
      CSS: max-width: 100%; max-height: 600px; object-fit: contain
   c. その他: ダウンロードボタンのみ表示（プレビュー不可）

2. プレビュー用 URL:
   a. GET /files/{id}/download-url と同一の Workers 署名付き URL を使用
   b. Content-Disposition: inline（プレビュー用）vs attachment（ダウンロード用）の区別は
      クエリパラメータ ?disposition=inline で制御
```

### ゴミ箱

```
1. ゴミ箱一覧:
   a. GET /files?deleted=true / GET /files/folders?deleted=true で取得
   b. レスポンスに remaining_days（完全削除までの残日数 = 30 - (NOW() - deleted_at).days）を付与
   c. スコープ別にフィルタ可能（team_id / organization_id）

2. 復元:
   a. POST /files/{id}/restore: deleted_at = NULL に更新
   b. POST /files/folders/{id}/restore: 配下を再帰的に復元
   c. 元のフォルダが削除済みの場合: ルートフォルダ直下に復元（フロントエンドで通知）
   d. ストレージ使用量: 論理削除時に減算していないため復元時の加算も不要

3. 即時完全削除:
   a. DELETE /files/{id}?permanent=true（ADMIN のみ）
   b. R2 オブジェクト削除 + DB 物理削除 + storage_used_bytes 減算（全バージョン含む）
   c. 監査ログ: FILE_PERMANENTLY_DELETED を記録

4. 自動完全削除:
   a. 既存の完全削除バッチ（日次 04:00 JST）で処理
   b. deleted_at < NOW() - INTERVAL 30 DAY のレコードを対象
```

### ファイルアップロードフロー

#### 通常アップロード（100MB 以下）

```
1. フロントエンド: POST /files/upload-url で R2 Presigned PUT URL を取得
   a. バックエンド: content_type ホワイトリスト検証
   b. バックエンド: 拡張子と content_type の整合性チェック（偽装検知。例: .pdf なのに application/x-executable は拒否）
   c. バックエンド: file_size 上限チェック（100MB）
   d. バックエンド: ストレージ枠チェック（used_bytes + file_size <= plan.included_bytes or 従量課金）
   e. バックエンド: file_key をUUIDベースで生成（files/{scopeType}/{scopeId}/{uuid}.{ext}）
   f. バックエンド: R2 の S3 互換 API で Presigned PUT URL を発行（有効期限 10分）

2. フロントエンド: Cloudflare R2 に直接アップロード（PUT {uploadUrl}）

3. フロントエンド: POST /files でDB登録
   a. バックエンド: file_key の R2 存在確認（HeadObject）
   b. バックエンド: ストレージ枠の再チェック（upload-url 発行後に他ユーザーがアップロードして枠が埋まるレースコンディション対策。不足時は 409 + R2 の孤立ファイルは日次バッチで清掃）
   c. バックエンド: shared_files INSERT + shared_file_versions INSERT（v1）
   d. バックエンド: storage_used_bytes をアトミック加算
   e. ApplicationEvent: FileUploadedEvent（通知連携用）
```

#### Multipart アップロード（100MB〜5GB。動画・大容量ファイル向け）

```
1. フロントエンド: POST /files/multipart/start で Multipart 開始
   a. バックエンド: content_type ホワイトリスト検証
   b. バックエンド: file_size 上限チェック（5GB）
   c. バックエンド: ストレージ枠チェック
   d. バックエンド: R2 の S3 互換 CreateMultipartUpload を呼び出し、upload_id を取得

2. フロントエンド: POST /files/multipart/{uploadId}/part-url で各パート用 Presigned URL を取得
   - 一度に複数 part_numbers をリクエストして効率化可能

3. フロントエンド: 各パートを R2 に並列アップロード（PUT {partUrl}）
   - レスポンスの ETag を記録

4. フロントエンド: POST /files/multipart/{uploadId}/complete で完了処理
   a. バックエンド: R2 CompleteMultipartUpload 呼び出し
   b. バックエンド: HeadObject で最終サイズを確認

5. フロントエンド: POST /files で DB 登録（通常フローと同じ）
   a. バックエンド: HeadObject + ストレージ枠再チェック
   b. shared_files / shared_file_versions INSERT
   c. storage_used_bytes 加算

6. 中断時: DELETE /files/multipart/{uploadId} で R2 AbortMultipartUpload を呼び出し破棄
```

### ストレージ使用量管理

```
1. 使用量の追跡:
   a. チーム: teams.storage_used_bytes で管理（TEAM / PERSONAL スコープのファイル）
   b. 組織: organizations.storage_used_bytes で管理（ORGANIZATION スコープのファイル）
      ※ 組織のストレージ枠はチームとは独立。組織単位でストレージプランを契約
   c. 加算タイミング: ファイル登録（POST /files）、新バージョンアップロード（POST /versions）
   d. 減算タイミング: ファイル完全削除（バッチ）、バージョン自動削除
   ※ 論理削除時は減算しない（復元の可能性があるため）

2. 完全削除バッチ:
   a. @Scheduled で日次実行（深夜 04:00 JST）
   b. 条件: deleted_at < NOW() - INTERVAL 30 DAY（論理削除から30日経過）
   c. 処理順序: ファイル（+ 全バージョンの R2 オブジェクト）を先に物理削除 → その後フォルダを物理削除
     （shared_files.folder_id の FK CASCADE でファイルが先に消えるため、フォルダから削除しても動作するが、
      storage_used_bytes の減算を正確に行うためファイル単位で先に処理する）
   d. storage_used_bytes 減算: 各ファイルの全バージョン合計サイズを減算（1トランザクション内）
   e. バッチサイズ: 100件ずつ処理。処理件数を application log に出力

3. R2 孤立ファイル清掃（§26 準拠）:
   a. upload-url / multipart/start で発行した R2 キーが POST /files で登録されなかった場合の清掃
   b. 日次バッチ（深夜 05:00 JST）で files/ プレフィックス配下を走査
   c. アップロードから24時間経過 かつ DB 未登録のファイルを R2 から削除
   d. 中断された Multipart Upload は R2 ListMultipartUploads で検出し、24時間経過分を AbortMultipartUpload で破棄

4. ストレージ枠超過時の挙動:
   a. アップロード API で 409 エラー（「ストレージ容量が不足しています」）
   b. レスポンスに現在の使用量・上限・推奨プランを含める
   c. 使用量が80%を超えた時点で管理者にプッシュ通知
     （ApplicationEvent: StorageWarningEvent）

5. ストレージ警告チェック:
   a. @Scheduled で日次実行（深夜 06:00 JST）
   b. 全チーム + 全組織の storage_used_bytes / plan.included_bytes を算出
   c. 80%超過 かつ 未通知 → 管理者にプッシュ通知
   d. 通知済みフラグ: teams / organizations テーブルに is_storage_warning_sent BOOLEAN DEFAULT FALSE を追加
      （使用量が80%未満に戻ったらリセット）

6. 使用量整合性チェック:
   a. 日次バッチ（深夜 06:30 JST）で storage_used_bytes と
      SUM(shared_files の全バージョン file_size WHERE deleted_at IS NULL) を照合
   b. 不整合があれば実際の合計値に補正 + WARN ログ出力
```

### 権限評価フロー

```
1. リクエスト時の権限チェック:
   a. ADMIN / DEPUTY_ADMIN（MANAGE_FILES）→ 暗黙的 MANAGE 権限（チェック終了）
   b. 対象ファイル/フォルダの file_permissions を検索
   c. ファイル個別の権限があればそれを適用
   d. なければ親フォルダ → 祖父フォルダ → ... → ルートフォルダの順で再帰的に検索
   e. grantee_type の評価順序:
      - USER: 完全一致
      - GROUP: ユーザーが所属するグループに一致
      - ROLE: ユーザーのロールに一致
   f. 複数マッチした場合は最も高い permission を採用
   g. 全階層でマッチする権限がない場合: **暗黙の拒否**（403 エラー）。デフォルトは「権限なし」であり、明示的に付与されない限りアクセス不可

2. 権限の継承:
   a. フォルダに設定した権限は配下の全ファイル・サブフォルダに自動継承
   b. サブフォルダ/ファイル個別の設定で上書き可能
   c. 上書きは「追加」ではなく「置換」（親の権限を完全にオーバーライド）
```

### フォルダ操作

```
1. フォルダ移動（PUT /folders/{id}）:
   a. 移動先フォルダへの MANAGE 権限を検証
   b. 循環参照チェック: 移動先が自分自身の配下でないことを検証
   c. スコープ変更不可: TEAM → ORGANIZATION 等のスコープ間移動は禁止

2. フォルダ削除（DELETE /folders/{id}）:
   a. 配下の全ファイル・サブフォルダを再帰的に論理削除
   b. 30日後にバッチで完全削除
   c. ルートフォルダは削除不可

3. パンくずリスト:
   a. parent_id を再帰的に辿ってルートまでのパスを構築
   b. MySQL 8.0 の WITH RECURSIVE で1クエリで取得
```

### 権限評価キャッシュ

```
1. 再帰的フォルダ権限評価のコスト削減:
   a. Valkey キャッシュ: mannschaft:cache:file-perm:{userId}:{targetType}:{targetId}
   b. 値: 評価済みの permission レベル（VIEW / DOWNLOAD / UPLOAD / MANAGE / DENIED）
   c. TTL: 5分
   d. キャッシュ無効化タイミング:
      - PUT /files/{id}/permissions 実行時: 対象ファイル/フォルダの全メンバーのキャッシュを削除
      - PUT /files/folders/{id}/permissions 実行時: 対象フォルダ + 配下全階層のキャッシュを削除
        （配下のキーパターン: mannschaft:cache:file-perm:*:FOLDER:{folderId} で SCAN + DEL）
      - チームメンバーの追加/削除時: 該当チームの全キャッシュを削除

2. キャッシュヒット時: DB アクセスなしで即座に権限判定を返却
3. キャッシュミス時: 通常の再帰的権限評価を実行し、結果をキャッシュに保存
```

### ドラッグ&ドロップ一括アップロード（フロントエンド UX ガイド）

```
1. ファイルのドラッグ&ドロップ:
   a. フォルダ詳細画面のファイル一覧エリアにドロップゾーンを設置
   b. 複数ファイルの同時ドロップに対応（並列で upload-url 取得 → R2 アップロード → DB 登録。100MB 超は自動的に Multipart フローに切り替え）
   c. アップロード進捗バーを各ファイルに表示
   d. 1ファイルでも失敗した場合は該当ファイルのみエラー表示（他のファイルは続行）

2. フォルダのドラッグ&ドロップ:
   a. ブラウザの File System Access API（webkitdirectory）でフォルダ構造を取得
   b. フォルダ構造を解析し、POST /files/folders で階層を再現 → 各ファイルを順次アップロード
   c. フォルダ名の重複時は自動リネーム（サーバー側のルールに準拠）
   d. 進捗表示: 「フォルダ: 3/5 完了、ファイル: 12/28 完了」のように全体進捗を表示
```

### 共有リンクフロー

```
1. 共有リンク作成（POST /files/{id}/links）:
   a. MANAGE 権限チェック
   b. token を SecureRandom + Base64URL で生成（64文字）
   c. 1ファイルあたりの有効リンク数上限（10件）チェック
   d. パスワード指定時: bcrypt でハッシュ化して password_hash に保存
   e. 監査ログ: FILE_LINK_CREATED

2. 共有リンクからのダウンロード（GET /shared/{token}）:
   a. token で shared_file_links を検索
   b. バリデーション:
      - is_active = TRUE
      - expires_at が未来（または NULL）
      - download_count < download_limit（または download_limit = NULL）
   c. パスワード設定時: パスワード入力画面を表示 → bcrypt 照合
   d. バリデーション通過: Workers 経由の署名付きダウンロード URL（または短命 R2 Presigned GET URL）を発行
   e. download_count をアトミック更新

3. 共有リンク無効化（DELETE /files/links/{linkId}）:
   a. is_active = FALSE に更新（レコードは保持。監査ログ用）
   b. 監査ログ: FILE_LINK_REVOKED
```

### 重要な判定ロジック

- **ファイル管理権限**: ADMIN or DEPUTY_ADMIN（`MANAGE_FILES` 権限）は全フォルダ・ファイルへの暗黙的 MANAGE 権限
- **アップロード権限**: UPLOAD 以上の権限 + SUPPORTER 以外
- **ダウンロード権限**: DOWNLOAD 以上の権限
- **個人フォルダ**: 所有者に暗黙的 MANAGE 権限。他ユーザーへの共有は明示設定
- **ファイル名重複時**: 自動リネーム（`(N)` サフィックス追加）
- **フォルダ深度上限**: 10階層（Service 層バリデーション）
- **バージョン上限**: 20件/ファイル（最古バージョンを自動削除）
- **ストレージ使用量**: 論理削除時は減算しない、完全削除時に減算
- **スター**: 全ロール（VIEW 以上）が付与可能。個人用のため他ユーザーには非公開
- **コメント閲覧**: VIEW 以上の権限。**コメント投稿**: DOWNLOAD 以上の権限。SUPPORTER は閲覧のみ
- **タグ追加/削除**: UPLOAD 以上の権限
- **共有リンク作成/無効化**: MANAGE 権限のみ。1ファイルあたり有効リンク上限10件

## 6. セキュリティ考慮事項

- **認可チェック**: 全 API エンドポイントで `file_permissions` を評価。権限のないファイル/フォルダへのアクセスは 403
- **R2 Presigned URL の安全性**: URL の有効期限（アップロード 10分、Multipart パート 10分、ダウンロード用の Workers 署名付き URL は 5分、Workers が内部で発行する R2 Presigned GET は同5分以内）を厳守。`file_key` は UUID ベースで推測不可能
- **IDOR 防止**: `PUT /files/{id}` / `DELETE /files/{id}` は権限チェック必須。他チームのファイル ID を URL 直打ちしてもアクセスは 403
- **ファイル内容の検証**: `content_type` のホワイトリスト検証。通常アップロード: `image/jpeg`, `image/png`, `image/webp`, `application/pdf`, `application/vnd.openxmlformats-officedocument.*`, `text/plain`, `text/csv`, `application/zip`。Multipart アップロード: 上記に加え `video/mp4`, `video/webm`, `video/quicktime` を許可。実行可能ファイル（`.exe`, `.sh`, `.bat` 等）は拒否。拡張子と `content_type` の整合性チェックで偽装ファイルを検知
- **監査ログ**: 以下のイベントを監査ログに記録
  - `FILE_UPLOADED`: ファイルアップロード（単発 / Multipart 両方）
  - `FILE_DOWNLOADED`: ファイルダウンロード（ダウンロード URL 発行時に記録）
  - `FILE_DELETED`: ファイル削除
  - `FILE_PERMISSION_CHANGED`: 権限変更
  - `FOLDER_DELETED`: フォルダ削除（配下のファイル数を記録）
  - `FILE_PERMANENTLY_DELETED`: ADMIN による即時完全削除
  - `FILE_RESTORED`: ゴミ箱からの復元
  - `FILE_LINK_CREATED`: 共有リンク作成（有効期限・パスワード有無を記録）
  - `FILE_LINK_REVOKED`: 共有リンク無効化
  - `FILE_LINK_DOWNLOADED`: 共有リンク経由のダウンロード（token・IP アドレスを記録）
- **レートリミット**: `Bucket4j` で以下の制限を適用
  - アップロード URL 発行（`POST /upload-url` / `POST /multipart/start` / `POST /multipart/{uploadId}/part-url`）: 1分間20件（合計カウント）
  - ダウンロード URL 発行（`GET /download-url`）: 1分間60件
  - 共有リンクダウンロード（`GET /shared/{token}`）: IP あたり1分間10件（認証不要 API のため IP ベースで制限）
  - コメント投稿（`POST /files/{id}/comments`）: 1分間20件
- **ストレージ使用量の整合性**: 日次バッチで `storage_used_bytes` と実際の `SUM(file_size)` を照合し、不整合があれば補正 + WARN ログ出力
