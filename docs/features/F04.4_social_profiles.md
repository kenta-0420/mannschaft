# F04.4: ソーシャルプロフィール・フォロー

> **ステータス**: 🟢 設計完了（F01.7 連携・フォロー可視化拡張を追記中 🟡）
> **実装フェーズ**: Phase 4
> **最終更新**: 2026-04-20

---

## 1. 概要

パブリックタイムラインやブログでの匿名投稿を支えるソーシャルプロフィール（匿名人格）と、ユーザー間のフォロー関係を管理する。1ユーザーにつき最大3つのソーシャルプロフィールを作成可能。実名アカウントとは独立した表示名・アバター・ハンドルを持ち、通常時は実名との紐付けを非公開とする。フォロー関係は実名ユーザー・ソーシャルプロフィール間の全組み合わせをサポートし、パブリックタイムラインの `feed=following` フィルタに使用する。

> **段階リリース**: パブリックタイムラインと同時にアップデートで解放。`platform_settings.public_timeline_enabled = false` の間、ソーシャルプロフィール作成・フォロー関連の全 API は 403 を返却。コードと DB は Phase 4 で実装完了しておく。

### 2026-04-20 追記: フォロー可視化拡張

本フィーチャーのフォロー関連 API は当初「パブリックタイムラインのフィード絞り込み」用途を主目的に設計されていた。今回の拡張で以下を追加する:

- **プロフィールのカウント表示**: ユーザープロフィール / チームプロフィールに `followingCount` / `followerCount` / `teamFriendCount` / `supporterCount` を表示
- **フォロー一覧 UI**: 「私がフォローしている一覧」「私をフォローしている一覧」ページの UI 設計を本ドキュメントで確定
- **他ユーザーのフォロー一覧の閲覧**: プロフィール公開設定（`is_follow_list_public`）に基づき、第三者のフォロー一覧を閲覧可能にする（横の広がり）
- **F01.7 カスタム公開範囲テンプレートとの連携**: follow 関係を参照するテンプレートルール（`TEAM_FRIEND_OF` 等）が F01.7 から本機能の follows / team_friends を参照する

> **注意**: ユーザー間の実名フォロー機能は作らない。対象は (1) ソーシャルプロフィール間のフォロー、(2) ユーザー → チーム / 組織のフォロー。詳細は §5 参照。

### フォロー可視化拡張（Phase 2）の UX 目的（2026-04-21 追記）

Phase 2 で達成したい利用者体験を以下に明記する:

- **「横の広がり」の可視化**: 知人の知人を辿るヒューリスティクスを提供し、地域チームやフレンド関係を起点にコミュニティを有機的に発見できるようにする
- **フォロー一覧 UI の提供**: 自分のフォロー / フォロワーを一覧表示する専用ページ（`/profile/following`, `/profile/followers`）を提供
- **他ユーザーのフォロー一覧の閲覧**: プライバシー設定（デフォルト PUBLIC）に従い、第三者のフォロー一覧を閲覧可能にする
- **ADHD傾向ユーザーへの配慮**: 一覧閲覧時の情報量が過剰にならないよう、タブ分け・無限スクロール化で段階的に読み込む設計とする（`MEMORY.md` user_adhd_tendency.md 参照）

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | ソーシャルプロフィールの凍結（is_active 制御）。通報対応時のみ実名逆引き |
| ADMIN | 自分のプロフィール CRUD + フォロー管理 |
| DEPUTY_ADMIN | 自分のプロフィール CRUD + フォロー管理 |
| MEMBER | 自分のプロフィール CRUD + フォロー管理 |
| SUPPORTER | 自分のプロフィール CRUD + フォロー管理 |
| GUEST | 閲覧のみ（プロフィール作成・フォロー不可） |

### 対象レベル
- [x] 個人 (Personal) — 各ユーザーが自分のソーシャルプロフィールとフォロー関係を管理
- [x] チーム (Team) — 拡張: ユーザー → チームフォロー、および teamFriendCount / supporterCount のカウント集計（2026-04-20 追加）

### 2.5 カウント表示拡張（2026-04-20 追加 / 2026-04-21 補強）

プロフィールレスポンスに以下の動的集計値を含める。DB の正規化（カウンターキャッシュ）は今回は行わず、**クエリ時集計** で提供する。高トラフィック化した段階でキャッシュ化を検討。

**集計方針**:
- **Phase 1**（本対応）: 動的 `COUNT(*)` クエリで集計。`ProfileCountAggregator` を Service 層に新設し、1プロフィール取得あたりのカウント系クエリを並列発行する
- **Phase 2 以降**: 実トラフィックでの P95/P99 を計測し、50ms を超える場合はカウンターキャッシュ（`user_counts` / `team_counts` テーブル）への denormalize を検討。フォロー／フレンド変動時のカウンター更新は非同期ジョブ経由を候補とする

> **UserSummaryResponse / TeamSummaryResponse への反映**: `UserSummaryResponse` には `followingCount` / `followerCount` を含め、`TeamSummaryResponse` には `supporterCount` を含める。これは `GET /api/v1/users/{userId}` / `GET /api/v1/teams/{teamId}/summary` 両方のレスポンスで提供する（F01.5 §5 冒頭の `teamFriendCount` と同じ整合方針）。

| カウント名 | 表示箇所 | 算出元 | 備考 |
|----------|----------|-------|------|
| `followingCount` | ユーザープロフィール / ソーシャルプロフィール | `follows` の `follower_type = USER/SOCIAL_PROFILE, follower_id = 自分` | USER→自分の SOCIAL_PROFILE は除外 |
| `followerCount` | ユーザープロフィール / ソーシャルプロフィール | `follows` の `followed_type = USER/SOCIAL_PROFILE, followed_id = 自分` | 自己フォローは除外 |
| `teamFriendCount` | ユーザープロフィール / チームプロフィール | ユーザーの場合: 所属チームすべてのフレンドチーム総数を合算（重複除去） / チームの場合: そのチームのフレンドチーム数 | F01.5 `team_friends` を参照 |
| `supporterCount` | チームプロフィール | そのチームの SUPPORTER ロール保有ユーザー数 | F01.2 `user_roles` を参照 |
| `followedTeamCount` | ユーザープロフィール | `follows` の `followed_type = TEAM, follower_id = 自分` | F01.5 で導入した TEAM type を対象 |

> **フォロー一覧の公開可否**: `users.is_follow_list_public`（BOOLEAN NOT NULL DEFAULT FALSE）を追加する。TRUE の場合のみ第三者が当該ユーザーの follow list を閲覧可能。詳細は §4 `GET /api/v1/users/{userId}/following` 参照。

### リクエスト／レスポンス仕様

#### `POST /api/v1/social-profiles`

**認可**: ログイン済みユーザー（GUEST 除く）

ソーシャルプロフィールを作成する。1ユーザー最大3つ。

**リクエストボディ**
```json
{
  "handle": "coffee_cat",
  "display_name": "コーヒー好きのねこ",
  "bio": "毎朝コーヒーを飲みながら投稿しています"
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `handle` | String | Yes | ハンドル名（英数字+アンダースコア、3〜30文字、グローバル一意） |
| `display_name` | String | Yes | 表示名（1〜50文字） |
| `bio` | String | No | 自己紹介文（最大300文字） |

**レスポンス（201 Created）**
```json
{
  "data": {
    "id": 5,
    "handle": "coffee_cat",
    "display_name": "コーヒー好きのねこ",
    "avatar_url": null,
    "bio": "毎朝コーヒーを飲みながら投稿しています",
    "is_active": true,
    "follower_count": 0,
    "following_count": 0,
    "created_at": "2026-03-14T10:00:00"
  }
}
```

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | handle 形式不正（英数字+アンダースコア以外 / 3〜30文字外） / display_name が空 / 50文字超過 / bio が300文字超過 |
| 403 | GUEST |
| 409 | handle が既に使用されている / プロフィール数が上限（3）に達している |

#### `GET /api/v1/social-profiles/handle/{handle}`

**認可**: ログイン済みユーザー

ハンドル名でソーシャルプロフィールの公開情報を取得する。`user_id` は返却しない（匿名性担保）。

**レスポンス（200 OK）**
```json
{
  "data": {
    "id": 5,
    "handle": "coffee_cat",
    "display_name": "コーヒー好きのねこ",
    "avatar_url": "https://cdn.example.com/sp/5.webp",
    "bio": "毎朝コーヒーを飲みながら投稿しています",
    "is_active": true,
    "follower_count": 42,
    "following_count": 15,
    "is_following": false,
    "created_at": "2026-03-14T10:00:00"
  }
}
```

**補足**
- `is_following`: 現在のログインユーザー（全ソーシャルプロフィール含む）がこのプロフィールをフォローしているか
- `is_active = false` のプロフィールも参照可能だが、投稿機能は使用不可であることを UI で表示

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 404 | ハンドルが存在しない |

#### `PUT /api/v1/social-profiles/{id}`

**認可**: プロフィール所有者本人

プロフィールを編集する。`handle` の変更は30日に1回のみ。

**リクエストボディ**
```json
{
  "handle": "coffee_cat_v2",
  "display_name": "コーヒー猫",
  "bio": "更新しました",
  "avatar_url": "https://cdn.example.com/sp/5_new.webp"
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `handle` | String | Yes | ハンドル名（変更は30日に1回。未変更の場合は現在値を送信） |
| `display_name` | String | Yes | 表示名 |
| `bio` | String | No | 自己紹介文 |
| `avatar_url` | String | No | アバター URL（別途 S3 アップロード API で取得した URL を指定） |

**レスポンス（200 OK）**: 更新後のプロフィールオブジェクト（POST レスポンスと同一構造）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | handle 形式不正 / display_name が空 |
| 403 | プロフィール所有者でない |
| 404 | プロフィール ID が存在しない |
| 409 | handle が既に使用されている / handle 変更が30日以内に実行済み |

---

#### `DELETE /api/v1/social-profiles/{id}`

**認可**: プロフィール所有者本人

ソーシャルプロフィールを削除する。関連する投稿の `social_profile_id` は SET NULL になり「削除済みアカウント」として表示される。

**レスポンス（204 No Content）**

**補足**
- フォロー関係（`follows` テーブルで該当プロフィールが follower/followed のレコード）も削除
- 削除後、当該プロフィールの `handle` は即座に再利用可能になる
- 削除は不可逆。投稿は残るが、投稿者情報は「削除済みアカウント」に置換される

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | プロフィール所有者でない |
| 404 | プロフィール ID が存在しない |

---

#### `PATCH /api/v1/admin/social-profiles/{id}/freeze`

**認可**: SYSTEM_ADMIN のみ

ソーシャルプロフィールの凍結/解除を行う。通報対応のモデレーション機能。

**リクエストボディ**
```json
{
  "is_active": false,
  "reason": "通報対応: 不適切なコンテンツの投稿"
}
```

**レスポンス（200 OK）**
```json
{
  "data": {
    "id": 5,
    "handle": "coffee_cat",
    "is_active": false
  }
}
```

**補足**
- 凍結されたプロフィール（`is_active = false`）は投稿・リプライ・リアクションが不可
- 既存の投稿は閲覧可能（削除されない）
- 凍結/解除は `audit_logs` に記録（SOCIAL_PROFILE_FREEZE / SOCIAL_PROFILE_UNFREEZE）

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | SYSTEM_ADMIN でない |
| 404 | プロフィール ID が存在しない |

#### `POST /api/v1/follows`

**認可**: ログイン済みユーザー（GUEST 除く）

対象をフォローする。

**リクエストボディ**
```json
{
  "follower_type": "SOCIAL_PROFILE",
  "follower_id": 5,
  "followed_type": "SOCIAL_PROFILE",
  "followed_id": 10
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `follower_type` | String | Yes | `USER` / `SOCIAL_PROFILE` |
| `follower_id` | Long | Yes | フォローする側の ID |
| `followed_type` | String | Yes | `USER` / `SOCIAL_PROFILE` |
| `followed_id` | Long | Yes | フォローされる側の ID |

**レスポンス（201 Created）**
```json
{
  "data": {
    "id": 100,
    "follower_type": "SOCIAL_PROFILE",
    "follower_id": 5,
    "followed_type": "SOCIAL_PROFILE",
    "followed_id": 10,
    "created_at": "2026-03-14T10:30:00"
  }
}
```

**補足**
- `follower_type = USER` の場合: `follower_id` は現在のログインユーザーの `users.id` であること
- `follower_type = SOCIAL_PROFILE` の場合: `follower_id` は現在のログインユーザーが所有するソーシャルプロフィールの ID であること
- 自分自身へのフォロー（同一 type + id の組み合わせ）はアプリ層で禁止
- USER が自分の SOCIAL_PROFILE をフォローすること（type が異なる）は許可。ただしフォロワー数表示からは除外

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | follower_type / followed_type が不正 / follower_id が自分のものでない |
| 403 | GUEST |
| 404 | followed_id が存在しない |
| 409 | 既にフォロー済み |

#### `GET /api/v1/follows/following`

**認可**: ログイン済みユーザー

フォロー中の一覧をカーソルベースで取得する。

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `follower_type` | String | Yes | `USER` / `SOCIAL_PROFILE` |
| `follower_id` | Long | Yes | 自分の ID |
| `cursor` | Long | No | カーソル |
| `limit` | Integer | No | 取得件数（デフォルト: 20） |

**レスポンス（200 OK）**
```json
{
  "data": [
    {
      "id": 100,
      "followed_type": "SOCIAL_PROFILE",
      "followed_id": 10,
      "profile": {
        "handle": "morning_runner",
        "display_name": "朝ランナー",
        "avatar_url": "https://cdn.example.com/sp/10.webp"
      },
      "created_at": "2026-03-14T10:30:00"
    }
  ],
  "meta": {
    "next_cursor": 99,
    "limit": 20,
    "has_next": false
  }
}
```

**補足**
- `followed_type = USER` の場合は `profile` の代わりに `user` オブジェクト（`id`, `display_name`, `avatar_url`）を返却
- `follower_id` が自分のものであることをバリデーション（他者のフォロー一覧は参照不可）

## 5. ビジネスロジック

### 主要フロー

#### ソーシャルプロフィールの作成
```
1. POST /api/v1/social-profiles を呼び出し
2. バックエンドが以下を検証:
   a. ユーザーが GUEST でないこと
   b. 既存プロフィール数が3未満であること
   c. handle がグローバル一意であること（UNIQUE KEY で保証）
   d. handle が英数字+アンダースコアのみ、3〜30文字であること
   e. display_name が1〜50文字であること
3. user_social_profiles に INSERT
4. レスポンスを返却（user_id は含めない）
```

#### ソーシャルプロフィールの削除
```
1. DELETE /api/v1/social-profiles/{id} を呼び出し
2. バックエンドが以下を検証:
   a. プロフィールが存在すること
   b. プロフィール所有者であること（user_id 一致）
3. 以下を実行:
   a. follows テーブルから該当プロフィールの全フォロー関係を DELETE
      （follower_type=SOCIAL_PROFILE AND follower_id={id}）
      OR（followed_type=SOCIAL_PROFILE AND followed_id={id}）
   b. user_social_profiles から DELETE
   c. timeline_posts.social_profile_id は ON DELETE SET NULL で自動 NULL 化
   d. Valkey のフォロー ID キャッシュを無効化
```

#### フォロー
```
1. POST /api/v1/follows を呼び出し
2. バックエンドが以下を検証:
   a. follower_id が自分のものであること
   b. followed_id が存在すること
   c. 自己フォロー（同一 type + id）でないこと
   d. 重複フォローでないこと（UNIQUE KEY で保証）
3. follows に INSERT
4. Valkey のフォロー ID キャッシュを無効化（key: following:{userId}）
```

#### ハンドル変更の30日制限
```
1. PUT /api/v1/social-profiles/{id} で handle が変更された場合
2. 前回の handle 変更日時を確認:
   - user_social_profiles.updated_at と現在の handle を比較
   - handle が実際に変更されている場合、前回変更から30日経過しているか確認
   - 実装: handle 変更時に `handle_changed_at` を別途記録するか、
     audit_logs の SOCIAL_PROFILE_HANDLE_CHANGED イベントの最新日時で判定
3. 30日未満の場合は 409 エラー
```

### 重要な判定ロジック

- **プロフィール上限**: 1ユーザー最大3つ。`SELECT COUNT(*) FROM user_social_profiles WHERE user_id = :userId` で確認
- **ハンドル一意性**: `UNIQUE KEY uq_social_profiles_handle` で DB レベル保証。409 は DUPLICATE KEY 例外をキャッチ
- **匿名性**: API レスポンスに `user_id` を含めない（`/me` エンドポイントと管理者凍結 API を除く）
- **凍結プロフィール**: `is_active = false` のプロフィールは投稿・リプライ・リアクション不可。既存投稿は閲覧可能
- **フォロワー数**: `follows` テーブルの COUNT で算出。自己フォロー（USER → 自分の SOCIAL_PROFILE）は除外。将来的にカウンターキャッシュ（denormalize）を検討

### フォロー可視化拡張のロジック（2026-04-20 追加）

#### `teamFriendCount` の集計
- **ユーザー視点**: `SELECT COUNT(DISTINCT CASE WHEN tf.team_a_id = mytId THEN tf.team_b_id ELSE tf.team_a_id END) FROM user_roles ur JOIN team_friends tf ON (tf.team_a_id = ur.team_id OR tf.team_b_id = ur.team_id) WHERE ur.user_id = :userId`
  - 所属チームが複数ある場合、フレンドチームの和集合（重複除去）
- **チーム視点**: `SELECT COUNT(*) FROM team_friends WHERE team_a_id = :teamId OR team_b_id = :teamId`

#### `supporterCount` の集計
- `SELECT COUNT(*) FROM user_roles WHERE team_id = :teamId AND role = 'SUPPORTER' AND status = 'ACTIVE'`

#### `is_follow_list_public` の運用
- 新規カラム `users.is_follow_list_public BOOLEAN NOT NULL DEFAULT FALSE` を追加
- ユーザープロフィール設定 API（F01.6 既存）でトグル変更可能
- FALSE → TRUE に変更した瞬間、既存のフォロー一覧が第三者に即可視化される点をフロントエンドで警告表示

## 6. セキュリティ考慮事項

- **匿名性の担保**: ソーシャルプロフィールの `user_id` は公開 API のレスポンスに一切含めない。通報対応時（`content_reports.identity_disclosed = true`）のみ SYSTEM_ADMIN に開示
- **ハンドル名のバリデーション**: 英数字 + アンダースコアのみ。予約語（admin, system, support, null 等）はブラックリストで禁止
- **レートリミット**: プロフィール作成は Bucket4j で 1時間に5回。フォロー API は1分間に30回。ハンドル参照は1分間に60回
- **所有権チェック**: プロフィール編集・削除は `user_id` の一致を必ず検証。フォロー API は `follower_id` の所有確認を実施
- **XSS 対策**: `display_name` / `bio` はレスポンス時にサニタイズ
- **監査ログ**: SYSTEM_ADMIN による凍結/解除操作は `audit_logs` に記録。通常のプロフィール CRUD は記録しない
- **フォロー一覧の情報開示制御（2026-04-20 追加）**:
  - `is_follow_list_public = FALSE` のユーザーのフォロー一覧は、本人以外は 403 で拒否する
  - IDOR 対策として、存在しない user_id と非公開ユーザーの判別が付かないよう、両者を **403** で統一（404 を返さない）
  - ソーシャルプロフィールのフォロー一覧は既存通り公開（匿名人格のため follow list の秘匿性は低い）
  - レートリミット: `/users/{userId}/following` 系は Bucket4j で 1分間に 30回（スクレイピング対策）

### フォロー一覧公開範囲（2026-04-21 追記）

既存の `users.is_follow_list_public` BOOLEAN カラムは、公開 / 非公開の 2 値制御のみ提供する。今回の拡張で **3 値 enum** `users.follow_list_visibility` を新設し、既存 BOOLEAN は deprecate する（マイグレーションで値を移行）。

| 値 | 閲覧可否 | 用途 |
|------|-------|------|
| `PUBLIC` | 誰でも閲覧可（デフォルト） | 横の広がりを積極活用したい層 |
| `FRIENDS_ONLY` | 自分と相互フォロー関係にあるユーザーのみ閲覧可 | 限定公開したい層 |
| `PRIVATE` | 本人のみ閲覧可 | 非公開徹底したい層 |

**デフォルト**: `PUBLIC`（横の広がり UX を促進するため。ただし ADHD 傾向ユーザーを含む全ユーザーに対し、初回ログイン時にオンボーディングで明示的に告知し、変更 UI を設定画面に目立つ形で配置する。§8 未解決事項参照）

**認可ルール**:
- `PRIVATE` の場合: 本人以外は 403
- `FRIENDS_ONLY` の場合: 相互フォロー関係にあるか（`follows` を双方向でチェック）を検証し、ない場合は 403
- `PUBLIC` の場合: 認証ユーザー全員が閲覧可

**マイグレーション**:
- V15.021 で `users.follow_list_visibility VARCHAR(16) NOT NULL DEFAULT 'PUBLIC'` を追加
- 既存 `is_follow_list_public = TRUE` → `PUBLIC`、`FALSE` → `PRIVATE` に移行
- V15.022 で `users.is_follow_list_public` カラムを DROP（Phase 2 移行完了後）

**PRIVATE ユーザーへの扱い**:
- `follow_list_visibility = PRIVATE` のユーザーの `/api/v1/users/{userId}/following` を第三者が呼ぶと 403 Forbidden を返す
- レスポンスボディに `error_code = FOLLOW_LIST_PRIVATE` を含め、UI 側で「このユーザーはフォロー一覧を非公開にしています」を表示する
