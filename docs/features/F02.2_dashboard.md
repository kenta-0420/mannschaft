## 1. 概要

ログイン後のホーム画面として、**個人・チーム・組織**の3レベルのダッシュボードを提供する。各ダッシュボードはウィジェット構成で、ユーザーが表示/非表示・並び順を自由にカスタマイズできる。

ダッシュボードは各機能の「入口」として、横断的に情報を集約する読み取り専用のハブ。データの実体は各機能（タイムライン、スケジュール、TODO 等）が管理し、ダッシュボードは集約・表示のみを担う。

### 本ドキュメントのスコープ

| 機能 | 説明 |
|------|------|
| 個人ダッシュボード | ログインユーザーの横断的な情報集約（お知らせ・イベント・投稿・未読・TODO・カレンダー） |
| チームダッシュボード | チーム全体の活動状況・お知らせ・イベント・TODO・課金サマリー |
| 組織ダッシュボード | 傘下チーム一覧・組織全体の統計・TODO・課金サマリー |
| ウィジェットカスタマイズ | 表示/非表示トグル・並び替え・モジュール連動 |

**スコープ外**:
- ADMIN 向け管理者ダッシュボード → F10.1
- SYSTEM_ADMIN 向けシステム管理者ダッシュボード → F10.1
- TODO 管理の CRUD（作成・編集・削除・ステータス変更） → F02.3（別ドキュメント）
- 個人カレンダーの Google Calendar 同期 → F03.3

### テーブル定義

#### `dashboard_widget_settings`

ウィジェットの表示/非表示と並び順。ユーザー × スコープ × ウィジェット種別ごとに1レコード。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。設定の所有者 |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`PERSONAL` / `TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | 0 | スコープ ID。`PERSONAL` の場合は `0`（UNIQUE KEY に NULL を含めると MySQL で重複防止が機能しないため） |
| `widget_key` | VARCHAR(50) | NO | — | ウィジェット種別キー（後述のウィジェット一覧参照） |
| `is_visible` | BOOLEAN | NO | TRUE | 表示/非表示 |
| `sort_order` | INT | NO | 0 | 並び順（昇順。小さいほど上位に表示） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_dws_user_scope_widget (user_id, scope_type, scope_id, widget_key)  -- ユーザー×スコープ×ウィジェットの一意制約
INDEX idx_dws_user_scope (user_id, scope_type, scope_id, sort_order)              -- スコープ別のウィジェット一覧取得
```

**制約・備考**
- `scope_id` は `PERSONAL` の場合 `0`（個人ダッシュボードはユーザーに1つのみ）。`TEAM` / `ORGANIZATION` の場合はスコープ実 ID を設定。MySQL の UNIQUE KEY は NULL を含むと重複防止が機能しないため、sentinel value `0` を使用
- レコードが存在しないウィジェットはデフォルト表示（`is_visible = TRUE`）として扱う。初回アクセス時にレコードは作成しない（遅延作成: `PUT /dashboard/widgets` 実行時に UPSERT）
- モジュール無効化時に `dashboard_widget_settings` は**書き換えない**。フロントエンド側の `WIDGET_MODULE_MAP` でフィルタリングし、モジュール再有効化時にウィジェット設定が自動復活する設計（F01.3 で決定済み）

#### `activity_feed`

アクティビティフィード。各機能の Service 層が ApplicationEvent を発行し、`ActivityFeedEventListener` が非同期で INSERT する。ダッシュボードの「最近のアクティビティ」ウィジェットのデータソース。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `actor_id` | BIGINT UNSIGNED | NO | — | FK → users。行動者 |
| `activity_type` | VARCHAR(30) | NO | — | アクティビティ種別（`POST_CREATED` / `EVENT_CREATED` / `MEMBER_JOINED` / `TODO_COMPLETED` / `BULLETIN_CREATED` / `POLL_CREATED` / `FILE_UPLOADED`） |
| `target_type` | VARCHAR(30) | NO | — | 対象エンティティの種別（`TIMELINE_POST` / `SCHEDULE` / `TODO` / `BULLETIN_THREAD` / `POLL` / `FILE`） |
| `target_id` | BIGINT UNSIGNED | NO | — | 対象エンティティの ID |
| `summary` | VARCHAR(200) | NO | — | 表示用サマリー（例: 「新しい投稿を作成しました」） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_af_scope (scope_type, scope_id, created_at DESC)  -- スコープ別フィード取得
INDEX idx_af_actor (actor_id, created_at DESC)               -- 自分の行動除外フィルタ
INDEX idx_af_created (created_at)                             -- 30日超レコードの削除バッチ用
```

**制約・備考**
- ポリモーフィック参照（`target_type + target_id`。FK 制約なし、アプリ層で検証）
- 保持期間: **30日**。30日超のレコードは日次バッチ（AM 3:00）で物理削除
- 書き込み: 各機能の Service 層が `ApplicationEvent` を発行 → `ActivityFeedEventListener` が `@Async` + `@TransactionalEventListener(phase = AFTER_COMMIT)` で INSERT。メインのトランザクションに影響させない
- 読み取り: `GET /dashboard/activity` で所属スコープの `scope_id` を IN 句で指定し、`actor_id != 自分` でフィルタ。レスポンスの `scope_name` / `actor.display_name` / `actor.avatar_url` は `teams` / `organizations` / `users` テーブルとの JOIN で取得
- `audit_logs` との棲み分け: `audit_logs` はセキュリティ監査・法的証跡用（IP・UA・セッション情報含む、immutable）。`activity_feed` はユーザー向け表示用（7種別のみ、30日で物理削除）。両者は独立して書き込まれる
- 削除済みコンテンツの扱い: 対象コンテンツが論理削除された場合、`activity_feed` のレコードは削除しない（30日で自然消滅）。API レスポンス組み立て時に対象の `deleted_at IS NOT NULL` を検出したら該当レコードをスキップし、代替件数分を追加取得する

#### `chat_contact_folders`

チャット・連絡先のカスタムフォルダ。ユーザーが DM やコンタクトを分類するために自由に作成する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。フォルダの所有者 |
| `name` | VARCHAR(50) | NO | — | フォルダ名（例: 「家族」「仕事関係」「趣味仲間」） |
| `icon` | VARCHAR(30) | YES | NULL | フォルダアイコン（emoji or アイコン名。例: `👨‍👩‍👧‍👦`, `briefcase`） |
| `color` | VARCHAR(7) | YES | NULL | フォルダカラー（hex。例: `#3B82F6`）。NULL = デフォルト色 |
| `sort_order` | INT | NO | 0 | 表示順（昇順） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_ccf_user (user_id, sort_order)  -- ユーザー別のフォルダ一覧
```

**制約・備考**
- 1ユーザーあたり最大 **20フォルダ**（フォルダ乱立防止）
- フォルダ名はユーザー内で一意（Service 層バリデーション）
- チーム別のグルーピングはシステムが自動生成するため、`chat_contact_folders` には含めない（チーム = 暗黙のフォルダ）
- フォルダ削除時: 配下の `chat_contact_folder_items` を CASCADE 削除。アイテムは「未分類」に戻る

---

#### `chat_contact_folder_items`

フォルダへのアイテム（DM チャネル / 連絡先）の割り当て。1アイテム = 1フォルダ（Gmail ラベルのような多重所属ではなく、メールフォルダ方式）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `folder_id` | BIGINT UNSIGNED | NO | — | FK → chat_contact_folders。ON DELETE CASCADE |
| `item_type` | VARCHAR(20) | NO | — | アイテム種別（`DM_CHANNEL`: DM チャットチャネル / `CONTACT`: 連絡先ユーザー） |
| `item_id` | BIGINT UNSIGNED | NO | — | `DM_CHANNEL` → chat_channels.id / `CONTACT` → users.id |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_ccfi_item (item_type, item_id)               -- 1アイテム1フォルダの制約（同一アイテムの重複割り当て防止）
INDEX idx_ccfi_folder (folder_id)                            -- フォルダ別のアイテム一覧
```

**制約・備考**
- 1アイテム = 1フォルダ。フォルダ間の移動は旧レコード DELETE + 新レコード INSERT（UPSERT）
- フォルダに割り当てられていないアイテムは「未分類」として扱う（DB にレコードなし）
- `item_type = 'DM_CHANNEL'`: `chat_channels` の `type = 'DM'` であることをアプリ層で検証（グループチャネルは対象外。グループチャネルはチーム配下で自動グルーピングされる）
- `item_type = 'CONTACT'`: DM のないユーザー（連絡先として管理したいがまだ会話していない）も登録可能

#### `PUT /api/v1/dashboard/widgets`

**認可**: MEMBER 以上

ウィジェット設定を一括更新する（UPSERT）。

**リクエストボディ**
```json
{
  "scope_type": "PERSONAL",
  "widgets": [
    { "widget_key": "NOTICES", "is_visible": true, "sort_order": 0 },
    { "widget_key": "UPCOMING_EVENTS", "is_visible": true, "sort_order": 1 },
    { "widget_key": "PERSONAL_TODO", "is_visible": true, "sort_order": 2 },
    { "widget_key": "MY_POSTS", "is_visible": false, "sort_order": 3 },
    { "widget_key": "PERFORMANCE_SUMMARY", "is_visible": true, "sort_order": 4 }
  ]
}
```

**レスポンス（200 OK）**: 更新後のウィジェット設定一覧（`GET /dashboard/widgets` と同形式）

**処理**:
1. `scope_type = PERSONAL` の場合、`scope_id` は省略可（バックエンドが `0` に変換して DB 操作）。`TEAM` / `ORGANIZATION` の場合は `scope_id` 必須
2. リクエストの `widgets` 配列を `dashboard_widget_settings` に UPSERT（`ON DUPLICATE KEY UPDATE`）
3. リクエストに含まれないウィジェットの設定は変更しない（差分更新）
4. 無効なモジュールに依存するウィジェットの `is_visible = true` 設定も許可する（モジュール再有効化時に自動復活するよう保全）

**バリデーション**:
- `widget_key` がスコープに対して有効なキーであること（個人ダッシュボードに `TEAM_TODO` は不可）
- `sort_order` は 0 以上の整数

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 400 | 無効な `widget_key` / バリデーションエラー |
| 403 | スコープメンバーでない |

## 4.5. フロントエンド UI 仕様

### ウィジェットカード

各ウィジェットは統一されたカードコンポーネントで描画する。

```
デザイントークン:
- 角丸: rounded-2xl（border-radius: 1rem）
- 背景: bg-white dark:bg-gray-800
- 影: shadow-sm hover:shadow-md transition-shadow
- パディング: p-4（モバイル） / p-6（PC）
- ボーダー: border border-gray-100 dark:border-gray-700
- 最小高さ: min-h-[160px]（コンテンツがなくてもカードの高さを揃える）
```

**コンポーネント構造**:
```
components/
  dashboard/
    DashboardGrid.vue          -- グリッドレイアウト（レスポンシブ）+ ドラッグ&ドロップ制御
    DashboardWidgetCard.vue    -- 共通カードラッパー（角丸・影・ヘッダー・ドラッグハンドル）
    DashboardWidgetSettings.vue -- ウィジェット設定パネル（表示/非表示トグル・並び替え・リセット）
    DashboardEmptyState.vue    -- データなし時の空状態表示（アイコン + メッセージ）
    widgets/
      WidgetNotices.vue
      WidgetUpcomingEvents.vue
      WidgetPersonalTodo.vue
      WidgetMyPosts.vue
      WidgetUnreadThreads.vue
      WidgetRecentActivity.vue
      WidgetPersonalCalendar.vue
      WidgetPerformanceSummary.vue
      WidgetTeamActivity.vue
      WidgetTeamBilling.vue
      WidgetOrgStats.vue
      ...（ウィジェット種別ごとに1ファイル）
```

### レスポンシブグリッドレイアウト

```
DashboardGrid.vue のグリッド定義:

- モバイル（< 768px）:  grid-cols-2  gap-3   -- 2列並び
- タブレット（768-1023px）: grid-cols-3  gap-4   -- 3列並び
- PC（1024-1439px）:     grid-cols-4  gap-4   -- 4列並び
- ワイドPC（>= 1440px）:  grid-cols-4  gap-6   -- 4列（余白を広げる）

Tailwind クラス:
  grid grid-cols-2 md:grid-cols-3 lg:grid-cols-4 gap-3 md:gap-4 lg:gap-6

カードサイズ:
- 通常ウィジェット: col-span-1（1セル占有）
- カレンダーウィジェット（PERSONAL_CALENDAR）: col-span-2（モバイル: col-span-2 で横幅全体）
- 課金サマリー（TEAM_BILLING / ORG_BILLING）: col-span-2（内訳表示にスペースが必要）
```

### ウィジェットの並び替え

2つの方法で並び替えが可能。どちらの操作でも最終的に `PUT /dashboard/widgets` を呼び出して `sort_order` を永続化する。

```
1. ドラッグ&ドロップ（メイン操作）
   a. ライブラリ: vue-draggable-plus（Vue 3 ネイティブ対応の SortableJS ラッパー。旧 vuedraggable@next より活発にメンテされている）
   b. DashboardGrid.vue 内で <draggable> コンポーネントを使用
   c. ドラッグハンドル: 各 DashboardWidgetCard のヘッダー左端にグリップアイコン（⠿）を表示
   d. ドラッグ中: opacity-50 + ring-2 ring-primary でドラッグ元をハイライト
   e. ドロップ先: ring-2 ring-dashed ring-primary/50 でプレースホルダーを表示
   f. ドロップ完了: sort_order を再計算し、PUT /dashboard/widgets を debounce（500ms）で呼び出し
   g. モバイル対応: タッチイベントで長押し（300ms）→ ドラッグ開始
   h. アクセシビリティ: キーボード操作（Tab でフォーカス → Space で選択 → 矢印キーで移動 → Space で確定）

2. メニューからの並び替え（設定画面）
   a. ダッシュボード右上の「ウィジェット設定」ボタン（歯車アイコン）
   b. モーダル or スライドパネルで全ウィジェット一覧を表示
   c. 各ウィジェット: トグルスイッチ（表示/非表示）+ ドラッグ並び替え
   d. モジュール無効のウィジェット: グレーアウト + ツールチップ
   e. ロール制限のウィジェット: 権限不足ユーザーには一覧に表示しない（API が返却しない）
   f. 「保存」ボタンで PUT /dashboard/widgets を一括送信
   g. 「リセット」ボタン: デフォルトの sort_order + is_visible に復元
```

### チャート・数値データのアニメーション

データを表示するウィジェットでは、**カウントアップアニメーション**と**チャートのグロウアニメーション**を適用し、値が「上がっていく」演出でダッシュボードに動きを持たせる。

```
1. カウントアップアニメーション（数値ウィジェット）
   a. 対象: unread_count, total_members, active_members, overdue_count,
      current_month_total, posts_this_week 等の数値フィールド
   b. 実装: composable useCountUp.ts
      - 0 → 目標値まで ease-out で増加（duration: 800ms）
      - requestAnimationFrame ベース（setInterval は使わない）
      - Intersection Observer と連動: ビューポートに入った時点でアニメーション開始
      - 整数は小数点なしでフォーマット、金額は toLocaleString('ja-JP') でカンマ区切り
   c. 例: 「未読 0 → 5」「今月の課金 ¥0 → ¥15,000」がスッと上がる

2. チャートのグロウアニメーション（統計ウィジェット）
   a. 対象: team_activity（棒グラフ）、org_stats（円グラフ）、
      team_page_views（折れ線グラフ）、billing 内訳（ドーナツチャート）、
      通報統計トレンド 等
   b. チャートライブラリ: Chart.js（vue-chartjs ラッパー）
      - 軽量（tree-shakable）、Tailwind のカラーパレットと統合しやすい
      - バンドルサイズ最適化: 必要なチャートタイプのみ個別登録（BarElement, LineElement, ArcElement, RadialLinearScale のみ。全チャート一括登録の Chart.register(...registerables) は禁止）
      - animation.duration: 1000ms
      - animation.easing: 'easeOutQuart'（急に上がって滑らかに収束）
   c. グロウ演出の種別:
      - 棒グラフ: 高さ 0 → 目標値まで下から伸びる
      - 折れ線グラフ: 左から右にラインが描画される（draw アニメーション）
      - ドーナツ / 円グラフ: 0° → 360° まで扇形が広がる
      - ゲージ: 0% → 目標% までメーターが回転する（active_rate 等）
   d. ダークモード対応:
      - Chart.js のデフォルトカラーを Tailwind の dark: バリアントに連動
      - useAppearance composable の isDark を watch し、チャートの color / gridColor を切替

3. スケルトン → アニメーション → 完成形 の遷移
   a. 第1段階: スケルトン UI（pulse アニメーション付きのプレースホルダー）
   b. 第2段階: データ取得完了 → カウントアップ / チャートグロウが同時に開始
   c. 第3段階: アニメーション完了 → 完成形のチャート・数値が静止表示
   d. 再取得時（WebSocket 更新 / stale-while-revalidate）:
      現在値 → 新しい値へのトランジション（duration: 400ms）。
      スケルトンには戻さない（ちらつき防止）

4. reduced-motion 対応
   a. prefers-reduced-motion: reduce の場合、全アニメーションを無効化
   b. カウントアップ: 即座に最終値を表示
   c. チャート: animation.duration = 0 で即時描画
   d. Tailwind の motion-safe: / motion-reduce: バリアントを活用
```

### ウィジェット別のチャート種別

| widget_key | チャート種別 | 表示内容 |
|-----------|-----------|---------|
| `NOTICES` | カウントアップ | `unread_count` のバッジ表示 |
| `UPCOMING_EVENTS` | — | リスト形式（チャートなし） |
| `PERSONAL_TODO` | カウントアップ + ゲージ | `overdue_count` のバッジ + 完了率ゲージ（`completed / total`） |
| `MY_POSTS` | — | リスト形式（チャートなし） |
| `UNREAD_THREADS` | カウントアップ | `total_unread_bulletin` + `total_unread_chat` のバッジ |
| `RECENT_ACTIVITY` | — | タイムライン形式（チャートなし） |
| `CHAT_HUB` | カウントアップ | `total_unread` バッジ + 各チャネルの `unread_count` バッジ |
| `PERSONAL_CALENDAR` | — | ミニカレンダー UI |

### ウィジェット単位のエラー状態

各ウィジェットのデータ取得が失敗した場合、**そのウィジェットのみ**エラー表示し、他のウィジェットは正常描画を維持する。

```

- レイアウト: カード内中央配置（空状態と同じ位置）

- DashboardWidgetCard.vue に status props を追加:
  'loading' | 'success' | 'error' | 'empty'
- status に応じてスロット切替:
  loading → スケルトン UI
  success → ウィジェットコンテンツ
  error → エラー状態（リトライボタン付き）
  empty → 空状態（DashboardEmptyState.vue）

- GET /dashboard の CompletableFuture で個別ウィジェットの取得が失敗した場合、

```

```
   d. 「フォルダに移動」ボタン → フォルダ選択ドロップダウン

   f. 完了 → トースト「3件をフォルダ「家族」に移動しました」+ 選択モード解除

2. キャンセル
   a. 「キャンセル」ボタンで選択モード解除（チェック全解除）
   b. 画面外タップでも解除

3. 制限
   a. 一度に選択可能な最大件数: 20件（API の bulk 上限に合わせる）
   b. 20件を超えて選択しようとした場合: トースト「一度に移動できるのは20件までです」
```

## 5. ビジネスロジック

### ダッシュボードデータ集約フロー

```
1. ユーザーがダッシュボードにアクセス（GET /dashboard）
   a. JWT からユーザー ID を取得
   b. dashboard_widget_settings から表示設定を取得（レコードなし = デフォルト表示）
   c. 横断取得の対象スコープを決定:
      team_memberships の last_accessed_at 降順で上位20スコープに絞り込み
      （所属数 <= 20 の場合は全スコープ対象）
   d. priority パラメータで取得対象を分岐

2. priority=CRITICAL の場合（第1段階 — フロントエンド初回リクエスト）
   a. NOTICES: notifications テーブルから未読 + 直近20件
   b. UPCOMING_EVENTS: schedules + schedule_attendances（今後7日間）
   c. PERSONAL_TODO: todos + todo_assignees（自分担当、未完了5件 + 期限超過数）
   d. PLATFORM_ANNOUNCEMENTS: is_pinned = TRUE or priority = 'URGENT' のみ
   ※ 上記4つを CompletableFuture で並行取得。目標 P95 < 300ms

3. priority=ALL の場合（デフォルト — 全ウィジェット）
   a. 第1段階のウィジェット + 以下を CompletableFuture で並行取得
   b. PLATFORM_ANNOUNCEMENTS: 全公開中レコード
   c. MY_POSTS: timeline_posts（自分の投稿、最新5件）
   d. UNREAD_THREADS: bulletin_threads + chat_channel_members の未読集計
   e. RECENT_ACTIVITY: activity_feed テーブルから対象スコープの最新10件
      （actor_id != 自分でフィルタ）
   f. PERSONAL_CALENDAR: schedules（個人 + 所属チーム公開イベント）
   g. PERFORMANCE_SUMMARY: パフォーマンスデータ集約
   ※ 目標 P95 < 800ms

4. レスポンス組み立て
   a. 全ウィジェットの結果を集約
   b. widget_settings を添付（フロントエンドのレイアウト復元用）
   c. scope_coverage を添付（total_scopes / displayed_scopes / has_hidden_scopes）
   d. is_visible = FALSE のウィジェットはキー自体をレスポンスに含めない
   e. has_hidden_scopes = TRUE の場合、フロントエンドに
      "他 {N} チームのイベント・未読は各チームのダッシュボードで確認できます" を表示

5. フロントエンドの2段階ロード戦略
   a. 画面アクセス時: スケルトンUIを即時レンダリング
   b. GET /dashboard?priority=CRITICAL を呼び出し、第1段階ウィジェットを描画
   c. 第2段階ウィジェット: Intersection Observer でビューポートに入ったときに
      個別API（GET /dashboard/my-posts 等）で useLazyAsyncData 経由で遅延取得
   d. Nuxt の <Suspense> でウィジェット単位のローディング状態を管理
```

### チーム/組織ダッシュボードのデータ取得

```
1. GET /dashboard/team/{teamId} or GET /dashboard/organization/{orgId}
   a. スコープメンバーシップを検証（非メンバーは 403）
   b. ロール + 権限グループ判定:
      - ADMIN → 課金サマリー + アクセス解析を含む
      - DEPUTY_ADMIN → BILLING_VIEW あり → 課金サマリーを含む
                        ANALYTICS_VIEW あり → アクセス解析を含む
      - MEMBER / SUPPORTER → どちらも含まない
   c. 個人ダッシュボードと同様に CompletableFuture で並行取得

2. 課金サマリーの取得（権限ありの場合のみ）
   a. team_subscriptions から現在のアクティブサブスクリプション
   b. subscription_invoices から当月の請求データ
   c. ストレージ使用量（team_storage_usage 等）
   d. 組織ダッシュボードの場合: org_count_billing_tiers から組織数課金情報
   e. 課金合計（税込） = モジュール課金 + パッケージ課金 + ストレージ課金 + 組織数課金
```

### ウィジェット設定の管理

```
1. 初回アクセス時
   a. dashboard_widget_settings にレコードが存在しない → デフォルト設定で表示
   b. レコードの遅延作成: ユーザーが PUT /dashboard/widgets を実行するまで INSERT しない
   c. デフォルトの sort_order はウィジェット一覧の定義順

2. ウィジェットの並び替え
   a. フロントエンドのドラッグ&ドロップで sort_order を変更
   b. 変更後に PUT /dashboard/widgets で一括 UPSERT

3. モジュール連動（F01.3 で設計済み）
   a. フロントエンド: WIDGET_MODULE_MAP でウィジェット → モジュール slug の対応を定義
   b. 無効モジュールのウィジェット: グレーアウト表示（disabled）、ツールチップ「このモジュールは現在無効です」
   c. dashboard_widget_settings は書き換えない → モジュール再有効化時に設定が自動復活
   d. WebSocket の ModuleToggledEvent で即時更新（Pinia ストア teamModulesStore）

4. ロール制限ウィジェットの除外
   a. TEAM_BILLING / TEAM_PAGE_VIEWS / ORG_BILLING はロール + 権限グループの要件を満たさないユーザーに対して、
      GET /dashboard/widgets の一覧から除外（存在自体を返却しない）
   b. PUT /dashboard/widgets でロール制限ウィジェットの widget_key を送信した場合は無視（エラーにしない）
   c. ロール昇格時（MEMBER → ADMIN 等）: 次回 GET /dashboard/widgets で自動的に一覧に出現（デフォルト設定で表示）
   d. モジュール無効のグレーアウトとの違い:
      - モジュール無効 → グレーアウト表示（機能の存在を認知させる。ADMIN に有効化を依頼する導線）
      - ロール制限 → 一覧から除外（ロールはユーザー自身で変更できないため、認知させるメリットなし）
```

### 課金サマリー・アクセス解析ウィジェットの表示ロジック

```
1. ロール別の返却判定
   a. ADMIN → team_billing / org_billing / team_page_views を常に含む
   b. DEPUTY_ADMIN → 権限グループのパーミッションで判定:
      - BILLING_VIEW パーミッションあり → team_billing / org_billing を含む
      - ANALYTICS_VIEW パーミッションあり → team_page_views を含む
      - どちらもなし → 含まない（MEMBER と同等）
   c. MEMBER / SUPPORTER → 含まない

2. チームダッシュボード
   a. team_billing: 課金なしの場合 { "current_month_total": 0, "message": "今月の課金はありません" }
   b. 内訳: module_billing / package_billing / storage_billing
   c. 次回請求日 + 請求履歴ページへのリンク

3. 組織ダッシュボード
   a. org_billing: チームダッシュボードの内訳 + 組織数課金（org_count_billing）
   b. org_count_detail: 現在のチーム数・無料枠・超過チーム数・超過単価
```

### アクティビティフィードの書き込み

```
1. 各機能の Service 層がコンテンツ操作を実行
   a. 例: TimelinePostService.create() でタイムライン投稿を作成

2. 操作完了後に ApplicationEvent を発行
   a. 例: applicationEventPublisher.publishEvent(
        new ActivityEvent(ActivityType.POST_CREATED, scopeType, scopeId, actorId,
                          TargetType.TIMELINE_POST, postId))

3. ActivityFeedEventListener が非同期で受信
   a. @Async + @TransactionalEventListener(phase = AFTER_COMMIT)
   b. メインのトランザクションに影響させない（失敗しても元操作はロールバックしない）
   c. summary テキストを activity_type に基づいて生成
   d. activity_feed テーブルに INSERT

4. 保持期間管理（日次バッチ AM 3:00）
   a. created_at < NOW() - INTERVAL 30 DAY のレコードを物理削除
   b. 1バッチ 5,000 件ずつ DELETE（ロック時間制限）

5. 対象 activity_type と発行元:
   a. POST_CREATED      → TimelinePostService.create()
   b. EVENT_CREATED      → ScheduleService.create()
   c. MEMBER_JOINED      → TeamMembershipService.join()
   d. TODO_COMPLETED     → TodoService.updateStatus() (status → COMPLETED)
   e. BULLETIN_CREATED   → BulletinThreadService.create()
   f. POLL_CREATED       → PollService.create()
   g. FILE_UPLOADED      → FileService.upload()
```

### スコープ横断取得の絞り込み

```
1. 個人ダッシュボードの横断取得（upcoming_events / unread_threads / recent_activity）
   a. team_memberships から last_accessed_at DESC で上位20スコープを取得
   b. last_accessed_at IS NULL のスコープは末尾に配置（一度もアクセスしていない）
   c. 20件に収まる場合は全スコープが対象（絞り込みなし）

2. last_accessed_at の更新タイミング
   a. GET /dashboard/team/{teamId} へのアクセス時
   b. チームページ（/teams/{teamId}）へのアクセス時
   c. 更新は非同期（@Async）で実行。リクエストのレスポンスタイムに影響させない
   d. 頻繁な更新を避けるため、前回更新から1時間以内は UPDATE をスキップ

3. レスポンスの scope_coverage
   a. total_scopes: ユーザーの全所属スコープ数
   b. displayed_scopes: 今回の横断取得に含まれたスコープ数（最大20）
   c. has_hidden_scopes: total_scopes > displayed_scopes

4. 通知への影響: なし
   a. プッシュ通知（F04.3）は全所属チームから漏れなく配信
   b. 個別のチームダッシュボード（GET /dashboard/team/{id}）も制限なし
   c. 絞り込みは個人ダッシュボードの「横断表示」のみに適用
```

### platform_announcements の統合

```
1. 全ダッシュボード（個人/チーム/組織）に platform_announcements 配列を含める
2. 取得条件:
   a. published_at IS NOT NULL（公開済み）
   b. published_at <= NOW()（公開日時到達済み）
   c. expires_at IS NULL OR expires_at > NOW()（期限内）
   d. deleted_at IS NULL（未削除）
   e. target_scope = 'ALL' OR (target_scope = 'ADMINS_ONLY' AND ユーザーが ADMIN 以上)
3. is_pinned = TRUE のお知らせはレスポンスの先頭に配置
4. URGENT 優先度のお知らせはフロントエンドで赤色バナーとして表示
```

### WebSocket によるリアルタイム更新

```
1. リアルタイム更新対象ウィジェット（既存 WebSocket トピックを購読）
   a. NOTICES: /topic/notifications/{userId} を購読
      → 新着通知受信時に notices.unread_count をインクリメント + アイテムを先頭に追加
   b. UNREAD_THREADS + CHAT_HUB: /topic/chat/{channelId} を購読
      → 新着メッセージ受信時に該当チャネルの unread_count をインクリメント
      → CHAT_HUB のチャネルカードの未読バッジも同時に更新
      → /topic/bulletin/{threadId} を購読し掲示板の未読も同様に更新
   c. PLATFORM_ANNOUNCEMENTS: /topic/platform-announcements を購読
      → URGENT お知らせの即時表示（赤色バナー）

2. リアルタイム更新対象外ウィジェット（stale-while-revalidate で対応）
   a. upcoming_events, personal_todo, my_posts, team_activity,
      personal_calendar, performance_summary, billing 等
   b. フロントエンド: ブラウザタブがアクティブに戻った時（visibilitychange イベント）
      + 最終取得から5分以上経過している場合にバックグラウンドで再取得
   c. Pinia ストア dashboardStore に lastFetchedAt を保持し、
      stale 判定に使用

3. 実装方針
   a. ダッシュボード固有の WebSocket トピックは作らない（既存トピックの再利用）
   b. Pinia ストア dashboardStore が WebSocket イベントを受信し、
      該当ウィジェットのリアクティブデータのみ更新（全体の再取得はしない）
   c. WebSocket 切断時のフォールバック: 30秒間隔のポーリングは行わない。
      再接続時に GET /dashboard?priority=CRITICAL で最新データを取得
```

### 挨拶ヘッダーの生成

```
1. GET /dashboard のレスポンスに greeting オブジェクトを含める
2. 挨拶文の生成（バックエンド）
   a. ユーザーの display_name を取得
   b. サーバー時刻（JST）に応じた挨拶を選択:
      - 5:00-11:59 → 「おはようございます、{display_name}さん」
      - 12:00-17:59 → 「こんにちは、{display_name}さん」
      - 18:00-4:59 → 「こんばんは、{display_name}さん」
3. サマリー文の生成
   a. 当日のイベント件数 + 未読通知数を集約
   b. 「今日のイベントは{N}件、未読は{N}件です」
   c. どちらも0件の場合: 「新しいお知らせはありません」
4. greeting は priority=CRITICAL でも priority=ALL でも常に返却
```

### チーム脱退時のウィジェット設定クリーンアップ

```
1. ユーザーがチームを脱退（TeamMembershipService.leave()）
2. ApplicationEvent: TeamMemberLeftEvent を発行
3. DashboardCleanupEventListener が非同期で受信
   a. @Async + @TransactionalEventListener(phase = AFTER_COMMIT)
   b. dashboard_widget_settings から該当ユーザー × scope_type=TEAM × scope_id=脱退チームID のレコードを DELETE
   c. 失敗しても脱退処理はロールバックしない（非同期・ベストエフォート）
4. 組織脱退時も同様（scope_type=ORGANIZATION × scope_id=脱退組織ID）
5. フォールバック: 月次バッチ（毎月1日 AM 4:00）で team_memberships / organization_memberships に存在しない
   scope_id を持つ dashboard_widget_settings を一括 DELETE（孤児レコード掃除）
```

### activity_feed 書き込みのエラーハンドリング

```
1. ActivityFeedEventListener の共通エラーハンドリング
   a. try-catch で INSERT 失敗をキャッチ
   b. 失敗時: WARN ログを出力（event_type, scope, actor_id, エラーメッセージ）
   c. リトライは行わない（30日で消えるデータのためコストに見合わない）
   d. メトリクス: activity_feed_write_failures カウンターをインクリメント（Micrometer）
      → Prometheus / Grafana で監視。閾値（1時間に100件超）で SYSTEM_ADMIN アラート

2. 実装パターン
   a. ActivityFeedEventListener は abstract ではなく単一クラス
   b. 全 activity_type の INSERT を1メソッドで処理（activity_type ごとにクラスを分けない）
   c. summary テキスト生成は ActivitySummaryGenerator ユーティリティに委譲
```

### 重要な判定ロジック

- **ウィジェットのデフォルト表示**: `dashboard_widget_settings` にレコードがない場合はデフォルト表示（`is_visible = TRUE`）。`BILLING_PERSONAL` のみデフォルト非表示（将来対応のため）
- **課金サマリーのロール・権限制限**: バックエンドで JWT のロール + 権限グループを検証。ADMIN は常に含む。DEPUTY_ADMIN は `BILLING_VIEW` パーミッションがある場合のみ含む。MEMBER / SUPPORTER には含めない（フロントエンド側の表示制御に依存しない）
- **非表示ウィジェットのスキップ**: `GET /dashboard` は `is_visible = FALSE` のウィジェットのデータ取得自体をスキップする（不要な DB クエリを実行しない）
- **スコープ横断の自動絞り込み**: 個人ダッシュボードの横断取得は `team_memberships.last_accessed_at` 降順で**上位20スコープ**に制限。所属数が20以下なら全スコープ対象。通知配信（F04.3）や個別チームダッシュボードには影響しない
- **アクティビティの範囲**: `recent_activity` は `activity_feed` テーブルから取得。自分の行動は含まない（`actor_id != 自分` でフィルタ）。自分の活動は `my_posts` で確認。`audit_logs` は参照しない（セキュリティ監査専用）
- **アクティビティフィードの保持期間**: `activity_feed` は30日で物理削除。日次バッチ（AM 3:00）で5,000件ずつ DELETE
- **2段階ロード**: `GET /dashboard?priority=CRITICAL` で第1段階（notices / upcoming_events / personal_todo）のみ高速返却（目標 P95 < 300ms）。フロントエンドは Intersection Observer で第2段階ウィジェットを遅延取得
- **last_accessed_at の更新制限**: チームダッシュボード/チームページアクセス時に非同期で更新。前回更新から1時間以内は UPDATE をスキップ（書き込み頻度の制限）
- **非表示チームの通知**: `scope_coverage.has_hidden_scopes = TRUE` の場合、フロントエンドに「他 N チームのイベント・未読は各チームのダッシュボードで確認できます」バナーを表示
- **未読数の管理**: Valkey カウンター（`unread:{userId}`）で管理。通知追加で +1、既読で -1。COUNT(*) クエリを廃止
- **ウィジェット設定の差分更新**: `PUT /dashboard/widgets` は送信されたウィジェットのみ更新。リクエストに含まれないウィジェットの設定は維持される
- **WIDGET_MODULE_MAP**: フロントエンド定数（`frontend/constants/widgetModuleMap.ts`）としてウィジェット↔モジュール slug の対応を管理。現時点では全25ウィジェット中 `PERFORMANCE_SUMMARY` → `performance-management` のみ。他は全て `null`（デフォルト機能、常時有効）。選択式モジュール固有のウィジェットは各モジュールの feature doc 設計時に MAP とウィジェット一覧に同時追加。バックエンドは `GET /dashboard/widgets` のレスポンスに `is_module_enabled` を含めてフロントエンドの描画判断を支援
- **チーム/組織ダッシュボードのキャッシュ**: 統計系ウィジェット（`team_activity` / `org_stats`）は Valkey に TTL 5分でキャッシュ。リアルタイム性が必要なウィジェット（`notices` / `unread_threads`）はキャッシュしない
- **チーム/組織ダッシュボードに `priority` パラメータは不要**: 個人ダッシュボードの `priority` は複数チームの横断取得（IN 句）が重いから必要。チーム/組織は単一スコープのクエリで高速なため不要。Phase 11 の負荷テストで問題が出た場合に追加を検討
- **Phase 11 負荷テスト検証指標**: `GET /dashboard?priority=CRITICAL` P95 < 300ms、`GET /dashboard?priority=ALL` P95 < 800ms、`GET /dashboard/team/{id}` P95 < 500ms。超過時は統計系ウィジェットの Valkey TTL を5分→15分に延長で対応
- **Quick Actions の楽観的更新**: TODO 完了・出欠回答・通知既読の Quick Action は楽観的更新（API レスポンスを待たずに UI 更新）。失敗時はロールバック + トースト通知。オフライン時はボタンを disabled 化
- **リンクスルー**: 各ウィジェットのヘッダーに「もっと見る →」リンクを配置。`DashboardWidgetCard.vue` の `linkTo` props で統一管理。リンク先は各機能の詳細ページ
- **挨拶ヘッダー**: `GET /dashboard` のレスポンスに `greeting` オブジェクトを含む。時間帯に応じた挨拶（JST）+ 当日イベント件数 + 未読通知数のサマリー。`priority=CRITICAL` でも `priority=ALL` でも常に返却
- **スコープ切替タブ**: フロントエンドのみの実装（API追加なし）。Pinia `dashboardStore.currentScope` でアクティブスコープを管理。タブ切替時は前タブのデータをメモリキャッシュ（TTL 5分）し、戻ったときに即座表示 + バックグラウンド再取得
- **統計期間切替**: `stats_period` クエリパラメータ（`TODAY` / `WEEK` / `MONTH`）でチーム/組織の統計ウィジェットの集計期間を変更。デフォルト `WEEK`。選択した期間は `dashboardStore` にスコープごとに記憶し次回復元
- **ウィジェット設定のリセット**: `DELETE /dashboard/widgets` でスコープ内の全 `dashboard_widget_settings` を削除。次回 GET 時にデフォルト設定で返却
- **チーム脱退時のクリーンアップ**: `TeamMemberLeftEvent` → `DashboardCleanupEventListener` が非同期で該当スコープの `dashboard_widget_settings` を DELETE。フォールバック: 月次バッチ（AM 4:00）で孤児レコード一括削除
- **activity_feed 書き込み失敗**: WARN ログ出力 + Micrometer カウンター `activity_feed_write_failures` をインクリメント。リトライなし（30日で消えるデータ）。1時間に100件超で SYSTEM_ADMIN アラート
- **ウィジェット単位のエラーハンドリング**: `GET /dashboard` の CompletableFuture で個別ウィジェットの取得が失敗した場合、該当キーを `{ "error": true }` で返却（HTTP 200 部分成功）。フロントエンドは `DashboardWidgetCard.vue` の `status` props で `error` 状態を描画し、リトライボタンで個別 API を再呼び出し
- **CHAT_HUB のチーム絞り込み**: 未読ありチーム（全件）+ 直近アクティブ上位5チーム。`has_more_teams = true` の場合は「他のチームを表示」ボタンで `all_teams=true` パラメータ付きで再取得
- **CHAT_HUB と UNREAD_THREADS の共存**: 両方有効時、未読件数は同一値。表示形式が異なるため共存可。CHAT_HUB 有効時にウィジェット設定画面で UNREAD_THREADS 非表示の推奨注記を表示（自動非表示はしない）
- **メッセージプレビューのプライバシー**: `user_appearance_settings.hide_chat_preview = TRUE` の場合、フロントエンドで `last_message_preview` を「新着メッセージ」に置換。API レスポンスは変更しない。CHAT_HUB ヘッダーの 👁 アイコンで一時的にトグル可能
- **データ鮮度**: 各ウィジェットのフッターに「○分前に更新」表示。10分以上経過で amber 警告色。手動リフレッシュは ↻ アイコンで個別 API 再呼び出し
- **プルトゥリフレッシュ**: モバイル（< 768px）のみ。50px 以上の下引きで `GET /dashboard?priority=ALL` を再呼び出し
- **オンボーディングツアー**: 初回アクセス時に driver.js で6ステップのガイド表示。localStorage で完了フラグ管理。ウィジェット設定画面から再表示可能

## 6. セキュリティ考慮事項

- **スコープメンバーシップ検証**: `/dashboard/team/{teamId}` / `/dashboard/organization/{orgId}` は対象スコープのメンバーであることを検証。非メンバーは 403
- **課金情報のアクセス制御**: 課金サマリーは ADMIN + DEPUTY_ADMIN（`BILLING_VIEW` パーミッション保持者）のみ。アクセス解析は ADMIN + DEPUTY_ADMIN（`ANALYTICS_VIEW` パーミッション保持者）のみ。バックエンドで JWT ロール + 権限グループを検証し、権限不足のレスポンスからキーを除外
- **platform_announcements の target_scope 遵守**: `ADMINS_ONLY` のお知らせは ADMIN 以上のユーザーにのみ返却。MEMBER / SUPPORTER のレスポンスからは除外
- **レートリミット**: `Bucket4j` で以下の制限を適用
  - ダッシュボード一括取得（`GET /dashboard`、`GET /dashboard/team/{id}`、`GET /dashboard/organization/{id}`）: 1分間30件/ユーザー
  - ウィジェット個別取得（`GET /dashboard/notices` 等）: 1分間60件/ユーザー
  - ウィジェット設定更新（`PUT /dashboard/widgets`）: 1分間10件/ユーザー
  - ウィジェット設定リセット（`DELETE /dashboard/widgets`）: 1分間5件/ユーザー
  - フォルダ作成（`POST /chat-folders`）: 1分間10件/ユーザー
  - フォルダアイテム操作（`PUT /chat-folders/{id}/items`）: 1分間30件/ユーザー
  - フォルダアイテム一括操作（`PUT /chat-folders/{id}/items/bulk`）: 1分間10件/ユーザー
- **フォルダの所有者検証**: `/chat-folders/{id}` の PUT / DELETE は `chat_contact_folders.user_id` が JWT のユーザー ID と一致することを検証。他ユーザーのフォルダ操作は 403
- **DM チャネルのメンバーシップ検証**: `PUT /chat-folders/{id}/items` で `item_type=DM_CHANNEL` を指定した場合、`chat_channel_members` にユーザーが含まれることを検証。参加していない DM チャネルをフォルダに追加不可
- **データ漏洩防止**: 各ウィジェットのデータ取得時に、コンテンツのスコープ・公開設定を検証。非公開チームの情報が個人ダッシュボードの `recent_activity` に漏れないよう、可視性フィルタを適用。`activity_feed` の読み取りは**ユーザーの現在の所属スコープ ID のみ** IN 句に含める（脱退済みスコープのアクティビティは返却しない）
