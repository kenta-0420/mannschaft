# F02.5: 行動メモ

> **ステータス**: 🟢 設計確定（軍議に持ち込み可能な状態）
> **実装フェーズ**: Phase 2（追補）
> **最終更新**: 2026-04-09
> **モジュール種別**: デフォルト機能 #5（個人生産性ツール）
> **関連ドキュメント**: F02.3 TODO管理（前日夜の予定入力連携）、F04.1 タイムライン（終業時の非公開投稿先）、F06.1 CMS・ブログ（週次まとめの出力先）、F06.3 タイムラインダイジェスト（自動生成バッチの実装パターン参照元）、F12.3 GDPR個人データ（エクスポート対象）

---

## 1. 概要

ADHD 傾向のあるユーザー向けに「終業時に1日を区切る・翌朝を前夜に決める」という認知パターンを支援する個人専用の行動記録機能。日中はワンショット入力で行動メモを垂れ流し、終業時に当日分をまとめてタイムラインへ非公開投稿、週末には1週間分の行動メモから AI 不要のテンプレートで非公開ブログを自動生成する。

**ADHD 認知パターンとの整合性**:

- **前日夜に翌日を決める** → 朝のスイッチオン摩擦を前日のうちに潰せる（朝は選択肢ゼロで動ける）
- **当日はメモ垂れ流し・タグ付けは後回し可** → 「後で整理可能なフロー」原則に合致
- **終業時に非公開投稿で区切る** → 1日を物理的に閉じる儀式 + 未来の自分への記録
- **デフォルトは徹底してドライ** → 気分入力（mood）等の「内面に意識を向けさせる UI」は **デフォルト OFF**。意識を向けたくない人の集中を奪わないことを優先する

`memory/user_adhd_tendency.md` で明示されている「入力摩擦ゼロ・必須項目最小化・後で整理可能」の3原則に完全準拠する。

**設計方針**:

- **新規テーブルは最小限**（`action_memos` / `action_memo_tags` / `action_memo_tag_links` / `user_action_memo_settings` の4本）
- **既存機構を最大限再利用**: 終業時の投稿は `timeline_posts` の `PostScopeType.PERSONAL` をそのまま使い、週次まとめは `blog_posts` の `Visibility.PRIVATE` に書き込む
- **TODO は予定、行動メモは実績**: 「予定 = TODO」「実績 = 行動メモ」と役割を明確に分離。F02.3 既存 API を流用し新しい予定管理レイヤーは作らない
- **対象は個人スコープのみ**: チーム/組織には展開しない。完全にプライベート（本人以外閲覧不可）
- **AI 不使用**: 週次まとめは TEMPLATE 方式で固定。F06.3 のような AI コスト/フラグ管理は不要
- **ドライモード優先**: mood・タグなどの「装飾的入力」はオプトイン。基本機能は「テキスト入力 → 送信 → 終わり」のみで完結する

## 3. DB設計

### テーブル一覧

| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `action_memos` | 行動メモ本体（日付・本文・関連TODO・終業投稿の参照） | あり（`deleted_at`） |
| `action_memo_tags` | ユーザー所有のタグマスタ（名前・色） | あり（`deleted_at`） |
| `action_memo_tag_links` | メモとタグの中間テーブル | なし |
| `user_action_memo_settings` | ユーザーごとの行動メモ設定（mood の表示 ON/OFF 等） | なし |

### テーブル定義

#### `action_memos`

行動メモ本体。1メモ = 1行動。日中は気軽にワンショット投稿できるよう必須項目を `content` のみに絞る。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE。所有ユーザー |
| `memo_date` | DATE | NO | — | このメモが属する日付（JST 基準。終業時に1日分をまとめる際のキー） |
| `content` | TEXT | NO | — | メモ本文（最大 5,000 文字。タイトルは持たない＝摩擦ゼロのため） |
| `mood` | VARCHAR(20) | YES | NULL | 気分タグ（任意。`GREAT` / `GOOD` / `OK` / `TIRED` / `BAD` のプリセット値。**設定 OFF のユーザーは常に NULL**） |
| `related_todo_id` | BIGINT UNSIGNED | YES | NULL | FK → todos。ON DELETE SET NULL。緩い紐付け（「この TODO に関連する実績」） |
| `timeline_post_id` | BIGINT UNSIGNED | YES | NULL | FK → timeline_posts。ON DELETE SET NULL。終業時に投稿された PERSONAL ポストへの参照 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | 入力日時 |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**

```sql
INDEX idx_am_user_date    (user_id, memo_date, deleted_at, created_at)  -- 日別メモ取得（最頻アクセス）
INDEX idx_am_user_created (user_id, created_at, deleted_at)             -- 全期間時系列取得（週次集計用）
INDEX idx_am_user_mood    (user_id, memo_date, mood, deleted_at)        -- 週次バッチの mood セクション判定用（mood IS NOT NULL の存在チェック）
INDEX idx_am_related_todo (related_todo_id)                              -- TODO 詳細画面での「関連メモ」表示
```

**制約・備考**

- **必須項目は `content` のみ**: タイトル無し、タグ無し、TODO 紐付け無しでも投稿可能。摩擦ゼロを最優先
- **`memo_date` の自動セット**: クライアント側で日付指定なしの場合、サーバー側で `LocalDate.now(ZoneId.of("Asia/Tokyo"))` を適用。深夜0時跨ぎは UTC ではなく JST で判定（生活リズムの「1日」に合わせる）
- **`memo_date` のバックフィル可**: 翌日になってから前日分を追記する場合、`memo_date` を明示指定して過去日付に書き込める。ただし「未来日付」への書き込みは不可（バリデーションエラー）
- **論理削除のみ**: 物理削除はしない。本人があとから振り返れるようにする
- **`mood` は設定で制御**: `user_action_memo_settings.mood_enabled = false` のユーザーがリクエストで mood を送信しても、Service 層で **silent に NULL に置き換える**（バリデーションエラーにはしない）。設定を後から ON にしたとき、過去のメモは NULL のまま残るが、それ以降は値が入る
- **`related_todo_id` のスコープ整合性**: 紐付け先 TODO が同一 `user_id` の `scope_type = 'PERSONAL'` であることをアプリ層で検証。チーム/組織 TODO への紐付けは禁止（プライバシー漏洩防止）
- **`timeline_post_id`**: 終業時の `publish-daily` API が成功した時にだけ埋まる。それ以外は NULL のまま。終業投稿後にメモを編集しても投稿側は再生成しない（過去ログとして固定）
- **1日あたりのメモ数上限**: 1ユーザー × 1日で最大 200件（アプリ層検証。スパム/自動投稿防止）

#### `action_memo_tags`

ユーザー所有のタグマスタ。タグは完全にユーザー個人の名前空間で、他人と共有しない。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `name` | VARCHAR(50) | NO | — | タグ名（例: '勉強', '運動', '読書', '#焦った'） |
| `color` | VARCHAR(7) | YES | NULL | 表示色（HEX。例: '#FF6B6B'）。NULL = デフォルト色 |
| `sort_order` | SMALLINT UNSIGNED | NO | 0 | 並び順（タグ管理画面用） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**

```sql
UNIQUE KEY uq_amt_user_name (user_id, name, deleted_at)  -- 同一ユーザー内のタグ名重複防止
INDEX idx_amt_user_sort     (user_id, sort_order)         -- タグ管理画面の一覧
```

**制約・備考**

- **1ユーザーあたりのタグ上限**: 100件（アプリ層で検証。論理削除済みは含めない）
- **論理削除時の中間テーブル整合性**: タグを論理削除しても `action_memo_tag_links` は残す。過去メモには「削除済みタグ」として元のタグ名で表示する（後述のレスポンス仕様参照）
- **タグの「復活」機能は提供しない**: 一度削除したタグは復元できない。同名のタグを再作成すれば新しい `id` が付与され、過去メモに紐付いていた論理削除済みタグとは別物になる。これは「行動メモは記録性重視でシンプルに保つ」設計判断（§11）
- **削除済みタグの API レスポンス表現**: タグを取得する際、論理削除済みのタグも返却するが `deleted` フラグを `true` にしてクライアントに区別させる
  ```json
  {"id": 12, "name": "運動", "color": "#5DADE2", "deleted": false}
  {"id": 9,  "name": "夜更かし", "color": null, "deleted": true}
  ```
- **タグ作成タイミング**: メモ入力時に「既存タグから選択」または「新規タグを作成」のどちらかを選ぶ。インクリメンタル検索 + 新規作成のサジェスト UI で摩擦を最小化。サジェスト候補には論理削除済みのタグは表示しない

#### `action_memo_tag_links`

メモとタグの中間テーブル。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `memo_id` | BIGINT UNSIGNED | NO | — | FK → action_memos。ON DELETE CASCADE |
| `tag_id` | BIGINT UNSIGNED | NO | — | FK → action_memo_tags。ON DELETE CASCADE |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**

```sql
UNIQUE KEY uq_amtl_memo_tag (memo_id, tag_id)  -- 同一メモへの同一タグ重複防止
INDEX idx_amtl_tag          (tag_id)             -- タグ別メモ一覧（タグ画面）
```

**制約・備考**

- **1メモあたりのタグ数上限**: 10個（アプリ層で検証。タグ付け過多を防ぐ）

#### `user_action_memo_settings`

ユーザーごとの行動メモ設定。レコードが存在しないユーザーは「すべてデフォルト値」として扱う（バックフィル不要）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `user_id` | BIGINT UNSIGNED | NO | — | PK 兼 FK → users。ON DELETE CASCADE |
| `mood_enabled` | BOOLEAN | NO | FALSE | 気分（mood）入力 UI を表示するか。**デフォルト OFF（ドライモード）** |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**

```sql
PRIMARY KEY (user_id)
```

**制約・備考**

- **PK = `user_id`**: 1ユーザー1レコード。AUTO_INCREMENT の id は持たない
- **デフォルトの哲学**: 「気分入力」のような内面に意識を向けさせる UI は集中を奪う可能性があるため、デフォルトは OFF。明示的に ON にしたユーザーだけが恩恵を受けられる
- **レコード未作成のユーザー**: `mood_enabled = false` と等価に扱う。Service 層の `getMoodEnabled(userId)` メソッドが「レコードなし → false を返す」ロジックを内包する
- **設定変更時は UPSERT**: `INSERT ... ON DUPLICATE KEY UPDATE` または JPA の `save()` で UPSERT
- **将来の拡張**: テンプレート選択、終業時刻のリマインド時刻、週次まとめの曜日カスタマイズ等を後で足す前提のテーブル設計

## 4. API設計

### エンドポイント一覧

| メソッド | パス | 認証 | 説明 |
|---------|-----|------|------|
| POST | `/api/v1/action-memos` | 必要 | 行動メモを1件作成（ワンショット） |
| GET | `/api/v1/action-memos` | 必要 | 自分の行動メモ一覧（日付・タグでフィルタ） |
| GET | `/api/v1/action-memos/{id}` | 必要 | 行動メモ1件取得 |
| PATCH | `/api/v1/action-memos/{id}` | 必要 | メモ本文・タグ・関連TODOの編集 |
| DELETE | `/api/v1/action-memos/{id}` | 必要 | 論理削除 |
| POST | `/api/v1/action-memos/{id}/tags` | 必要 | メモにタグを追加（複数可） |
| DELETE | `/api/v1/action-memos/{id}/tags/{tagId}` | 必要 | メモからタグを除去 |
| POST | `/api/v1/action-memos/{id}/link-todo` | 必要 | メモを TODO に紐付け |
| POST | `/api/v1/action-memos/publish-daily` | 必要 | 当日分のメモをまとめてタイムラインに PERSONAL 投稿 |
| GET | `/api/v1/action-memo-tags` | 必要 | 自分のタグ一覧 |
| POST | `/api/v1/action-memo-tags` | 必要 | タグ作成 |
| PATCH | `/api/v1/action-memo-tags/{id}` | 必要 | タグ編集（名前・色） |
| DELETE | `/api/v1/action-memo-tags/{id}` | 必要 | タグ論理削除 |
| GET | `/api/v1/action-memo-settings` | 必要 | 自分の行動メモ設定を取得 |
| PATCH | `/api/v1/action-memo-settings` | 必要 | 行動メモ設定の更新（mood_enabled 等） |

> **共通仕様（重要）**:
> - すべての GET / PATCH / DELETE エンドポイントは **常に認証ユーザー自身のデータのみを操作対象とする**。他ユーザー ID を明示指定するクエリパラメータは存在しない。Service 層の入口で `currentUser.id == resource.userId` を必ず検証し、不一致時は 404 を返す（403 ではなく 404 を使うことで、リソースの存在自体を漏洩させない）
> - **週次まとめブログの取得は本機能の API ではなく、F06.1 CMS・ブログの `GET /api/v1/blog-posts?visibility=PRIVATE` を流用する**。週次まとめは `blog_posts` に `visibility=PRIVATE` で書き込まれるため、ユーザーの「マイブログ（非公開）」一覧から自然に閲覧できる。新規 API は作らない（既存機構の再利用方針）

### リクエスト／レスポンス仕様

#### `POST /api/v1/action-memos`

ワンショット投稿。日中の高頻度入力を想定。

**リクエストボディ**

```json
{
  "content": "朝 30分 散歩した",
  "memo_date": "2026-04-09",
  "mood": "GOOD",
  "related_todo_id": null,
  "tag_ids": [12, 17]
}
```

- `content` のみ必須。それ以外は省略可
- `memo_date` 省略時はサーバー側で JST の今日に自動セット
- `tag_ids` 省略時は空配列扱い
- `mood` は `user_action_memo_settings.mood_enabled = false` の場合、Service 層で **silent に無視**される（NULL として保存。エラーは返さない）

**レスポンス（201 Created）**

```json
{
  "data": {
    "id": 4521,
    "memo_date": "2026-04-09",
    "content": "朝 30分 散歩した",
    "mood": null,
    "related_todo_id": null,
    "timeline_post_id": null,
    "tags": [
      {"id": 12, "name": "運動", "color": "#5DADE2"},
      {"id": 17, "name": "朝", "color": null}
    ],
    "created_at": "2026-04-09T08:32:14"
  }
}
```

> **`mood` が `null` で返る理由**: 上記のレスポンス例は `mood_enabled = false` のユーザーがリクエストに `mood: "GOOD"` を送ったケース。サーバー側で無視されて NULL になっている。これはデータの矛盾ではなく、設定通りの動作。

**エラーレスポンス**

| ステータス | 条件 |
|-----------|------|
| 400 | `content` が空 / 5,000文字超 / `memo_date` が未来日付 / 1日200件超過 |
| 404 | `tag_ids` に存在しないタグ / 他人の TODO を `related_todo_id` に指定（IDOR 対策で 403 ではなく 404） |
| 429 | レートリミット超過（1分60回） |

#### `POST /api/v1/action-memos/publish-daily`

当日分（または指定日分）のメモをまとめて整形し、タイムラインに `PERSONAL` 投稿として公開する。終業時の儀式。

**リクエストボディ**

```json
{
  "memo_date": "2026-04-09",
  "extra_comment": "今日はよく動けた。明日も頑張る"
}
```

- `memo_date` 省略時は JST の今日
- `extra_comment` は本文末尾に追記される任意のひと言
- **本文の組み立てに関するパラメータは持たない**（`include_tags` / `include_mood` 等は廃止）。タグは常に本文に含められる。mood は `mood_enabled = true` のユーザーのみ本文に含められ、それ以外のユーザーでは自動的に省略される。フラグでの上書きを許すと UI と認知負荷が増えるため、サーバー側で挙動を一本化する

**処理内容**

1. 該当日の当該ユーザーの `action_memos` を時系列順に取得（論理削除済みは除外）
2. 本文を `## YYYY-MM-DD の行動ログ\n\n- HH:MM content [タグ]\n...` の形式で組み立て
3. `mood_enabled = true` のユーザーは各行に mood の絵文字を付与
4. `timeline_posts` に `scope_type = PERSONAL`, `scope_id = user_id`, `posted_as_type = USER`, `content = 上記本文` で INSERT
5. 各 `action_memos.timeline_post_id` を作成された投稿 ID で更新
6. 同日に既に投稿が存在する場合は、新規作成ではなく旧投稿を論理削除して再作成（=「上書き再投稿」）

**レスポンス（201 Created）**

```json
{
  "data": {
    "timeline_post_id": 88231,
    "memo_count": 7,
    "memo_date": "2026-04-09"
  }
}
```

**エラーレスポンス**

| ステータス | 条件 |
|-----------|------|
| 400 | 該当日にメモが0件 |
| 429 | 1分間に5回以上の publish-daily 呼び出し（連打防止） |

## 5. ビジネスロジック

### 主要フロー

#### 5.1. 日中のワンショット入力（ドライモード）

```
1. ユーザーが /action-memo を開く
2. テキスト入力欄にメモを書いて Enter（または送信ボタン）
3. POST /api/v1/action-memos が呼ばれる
   - content のみ必須。memo_date は自動で今日
4. 入力欄がクリアされ、上に新しいメモがスタックされる
5. （mood_enabled = false なら気分入力 UI は表示されない）
6. タグ付けは「あとからまとめて」することも、その場で1タップで付けることも可能
```

> **設計意図**: 入力 → 送信 → クリア の最短ループを徹底する。「タイトルを考える」「カテゴリを選ぶ」「気分を選ぶ」など認知負荷の高い操作を一切要求しない。これがデフォルト体験。

#### 5.2. 気分入力モード（オプトイン）

```
1. ユーザーが /action-memo/settings で「気分入力を有効にする」をトグル ON
2. PATCH /api/v1/action-memo-settings で mood_enabled = true
3. 以降のメモ入力画面に「気分」セレクタ（5択 + 絵文字）が表示される
4. 入力時に気分を選ぶ。選ばずに送信した場合は NULL（必須ではない）
5. メモ一覧に気分の絵文字が表示される
6. 嫌になったらいつでも OFF に戻せる（過去メモの mood 値はそのまま保持）
```

> **設計意図**: 「気分を意識すること自体が集中を奪う」という ADHD 的な懸念に配慮。デフォルトはドライ、必要な人だけが ON にする。OFF に戻したときに過去データは失わない。

#### 5.3. 前日夜の予定入力（TODO 連携）

```
1. 21時前後にユーザーが /action-memo/closing を開く
2. 画面下部の「明日やること」入力欄にタスクを書く
3. 内部的に POST /api/v1/todos を呼ぶ
   - dueDate = 翌日（自動）
   - scope_type = PERSONAL
   - scope_id = user_id
4. 翌朝、ユーザーは普段通り /todos/my で TODO 一覧を見て動き出せる
   （新機能側で別画面を作らない＝既存機構の再利用）
```

> **設計意図**: 「翌日の予定」と「TODO」を別物として持たない。1つの真実（TODO テーブル）を共有する。新機能の `closing.vue` 画面はあくまで「夜のリズム導線」であり、データは F02.3 の既存 API を呼ぶ薄いラッパー。

#### 5.4. 終業時の振り返り + タイムライン投稿

```
1. ユーザーが /action-memo/closing を開く（同じ画面）
2. 画面上部に当日メモ一覧 + 完了/未完 TODO 一覧 が表示される
3. 未完 TODO に対して「明日に回す」ボタン → PATCH /api/v1/todos/{id} で due_date を翌日に
4. 「今日を締める」ボタン → POST /api/v1/action-memos/publish-daily
5. タイムラインに PERSONAL スコープで本日分のまとめが投稿される
6. ホーム画面に戻る
```

> **設計意図**: 「今日を物理的に閉じる」儀式を提供することで認知的な区切りをつける。投稿は完全に非公開（PERSONAL）なので、人目を気にせず本音を書ける。

#### 5.5. 週次まとめブログの自動生成

```
@Scheduled(cron = "0 0 21 * * SUN", zone = "Asia/Tokyo")
@SchedulerLock(name = "actionMemoWeeklySummary", lockAtMostFor = "PT30M", lockAtLeastFor = "PT1M")
public void generateWeeklySummaries() {
  // 1. 過去7日間に action_memos を1件以上書いた user_id を抽出
  //    （0件のユーザーは blog_posts レコードを作らない＝スキップ）
  // 2. 各ユーザーについて:
  //    a. 当週のメモを memo_date 順に取得
  //    b. タグごとにグルーピング + 件数集計
  //    c. その週のメモに mood IS NOT NULL が1件以上あれば気分セクションを生成
  //       （※ user_action_memo_settings.mood_enabled の現在値ではなく、
  //          実際の入力データを基準にすることで、OFF→ON→OFF と切り替えた
  //          ユーザーでも矛盾なくセクション表示の有無を決定できる）
  //    d. テンプレートに流し込んで Markdown を生成
  //    e. blog_posts に INSERT
  //       - visibility = PRIVATE
  //       - cross_post_to_timeline = false
  //       - title = "週次ふりかえり: YYYY-MM-DD 〜 YYYY-MM-DD"
  // 3. ログ出力（生成件数 / スキップ件数 / 失敗件数）
}
```

実装は `ErrorReportWeeklySummaryService` をテンプレートとしてコピー。AI は使わず、純粋な統計とテンプレート埋め込みのみ（コストゼロ・常時 ON）。

**バッチ運用ポリシー**:

- **失敗時の自動リトライは非実装**。あるユーザーの生成中に例外が発生した場合は、当該ユーザーをスキップして次のユーザーに進む（1人の失敗で全体が止まらない）
- **catch-up 機構なし**: バッチ実行時刻（日曜 21:00 JST）にサーバーが落ちていた場合は、その週のまとめは生成されない。SYSTEM_ADMIN が手動で再実行できるよう、`POST /api/v1/admin/action-memo/regenerate-weekly-summary?week=YYYY-Www&userId=...` という管理 API を Phase 3 で別途用意する（**ただしユーザー個別のメモは閲覧しない**。生成のみトリガーする）
- **その週にメモが0件のユーザー**: `blog_posts` レコードを作成しない（無音）。Phase 3 で「先週はメモがありませんでした」というリマインド通知を送るかは未定（§12 未解決事項）
- **mood セクションの分岐基準**: `user_action_memo_settings.mood_enabled` の現在値**ではなく**、その週の `action_memos` に `mood IS NOT NULL` のレコードが1件以上存在するかどうかで判定する。これにより mood を OFF→ON→OFF と切り替えたユーザーでも、過去に入力した気分データが正しく反映される

**週次まとめテンプレート（mood_enabled = true の場合のフル版）**:

```markdown
# 週次ふりかえり: 2026-04-06 〜 2026-04-12

## 📊 今週のサマリー
- メモ件数: 32件
- 投稿日数: 6/7日
- 平均気分: 🙂 GOOD

## 📈 気分の推移
- 月: 🙂 GOOD / GOOD / OK
- 火: 🙂 GOOD
- 水: 😐 OK / OK / 😩 TIRED
- 木: 😩 TIRED / TIRED
- 金: 🙂 GOOD / 😄 GREAT
- 土: 😄 GREAT / GREAT / GREAT
- 日: 🙂 GOOD / GOOD

## 🏷️ よく使ったタグ
1. #運動 (8件)
2. #勉強 (6件)
3. #読書 (4件)

## 📝 今週の代表的なメモ（ランダム抽出）
- [4/06] 朝散歩 30分。気持ちよかった
- [4/08] 数学ドリル進めた #勉強
- [4/12] 読書会で発表した #読書

## 🔗 関連 TODO
- 完了: 12件
- 未完: 3件 → 来週へ繰越
```

`mood_enabled = false` の場合は「平均気分」と「気分の推移」セクションを省略する（残りは同じ）。

### 重要な判定ロジック

- **`memo_date` 自動セットの境界**: JST で 0:00 を跨いだ瞬間に新しい日のメモになる。深夜作業中にメモを書いた場合、その日のメモは「翌日扱い」になることを許容する（生活リズム優先）。違和感があれば手動で `memo_date` を前日に書き換え可能
- **`publish-daily` の冪等性**: 同じ `memo_date` で複数回呼ばれた場合、旧投稿を論理削除してから新規投稿を作る（差分パッチではない）。タイムライン上での「投稿が連続する」見た目を防ぐ。**旧投稿への返信は PERSONAL スコープのため理論上発生しない**（他人は見えない）が、将来スコープ仕様が変わった場合は孤立リプライ対策を別途設計する必要がある旨を実装ヘッダーコメントに記載すること
- **既に投稿済みメモの編集**: `timeline_post_id` が埋まっているメモを編集しても、自動で旧投稿は更新しない。ユーザーが明示的に `publish-daily` を再実行した場合のみ新しい投稿が作られる（過去ログとしての固定性を優先）
- **`extra_comment` の扱い**: 投稿本文の末尾に `\n\n---\n{extra_comment}` の形で追記する。`action_memos` 自体には保存しない（投稿の一部として扱う）。**XSS 対策**: 2026-04-09 時点で `TimelinePostService` には HTML サニタイズ実装が存在しない（grep で確認済み）。本機能の Phase 1 着手時に F04.1 タイムラインの実装状況を再確認し、未実装ならば **共通基盤として `org.jsoup.Jsoup.clean(text, Safelist.none())` 相当のサニタイズユーティリティを `common/security/HtmlSanitizer.java` に新規作成し、F02.5 と F04.1 の両方で利用する**。本機能の独自実装にはしない（横断的な責務のため）。Phase 1 までにこの判断を完了させる
- **`mood` の silent ignore**: 設定 OFF のユーザーが mood を送ってきても 400 エラーは返さない。理由: クライアントの状態とサーバー設定にラグがあった場合（例: 設定変更直後・複数デバイス併用）にエラーで弾くと UX が悪化する。silent NULL 化で安全側に倒す
- **1日 200 件上限超過時の UX**: 201 件目の投稿リクエストで 400 エラー + メッセージ「本日の上限（200件）に達しました。明日以降にお書きください」を返す。クライアント側ではダイアログでこの文言を表示し、入力欄をクリアしない（書きかけを失わない）

## 6. セキュリティ考慮事項

- **認可チェック（所有者一致の固定ルール）**: `ActionMemoService` / `ActionMemoTagService` / `ActionMemoSettingsService` のすべての public メソッドの入口で `currentUser.id == resource.userId` を検証する。SYSTEM_ADMIN であっても他人のメモにはアクセス不可（権限ベースの ABAC ではなく、所有者一致の固定ルール）。違反時は **404 Not Found** を返す（403 ではなく 404 にすることで、リソースの存在自体を漏洩させない＝IDOR 対策）
- **`related_todo_id` の検証**: 紐付け対象の TODO が `scope_type = PERSONAL` かつ `scope_id = currentUser.id` であることを必ず確認。チーム/組織 TODO への紐付けを許すと「タグ経由でチームの活動が漏洩する」可能性があるため
- **レートリミット**: `Bucket4j` で以下を適用。**設計意図**: ADHD ユーザーの「思いついた瞬間に書く」摩擦ゼロ原則と矛盾しないよう、いずれも **人間が手動で打ち込む現実的な最大値の数倍** に設定している。これらの制限は **ボット・スクリプト・連打バグの防御** を目的としており、通常利用で到達することは想定しない
  - `POST /api/v1/action-memos`: 1ユーザーあたり 1分60回まで（人間の物理的な手入力上限の約3〜5倍）
  - `POST /api/v1/action-memos/publish-daily`: 1ユーザーあたり 1分5回まで（終業ボタンの誤連打防止）
  - `POST /api/v1/action-memo-tags`: 1ユーザーあたり 1分20回まで（タグ作成スパム防止）
  - `PATCH /api/v1/action-memo-settings`: 1ユーザーあたり 1分10回まで
  - **超過時の UX**: 429 を返すと同時にクライアント側で「投稿が早すぎます。少し待ってから再送してください」と表示し、**入力欄の内容を保持する**（書きかけを失わせない）。`localStorage` への自動保存も併用する
- **GDPR エクスポート連携（Phase 1.5 で必須）**: F12.3 GDPR 個人データエクスポートに **Phase 1 完了直後に組み込む**。具体的には以下を実施:
  1. `ActionMemoEntity` / `ActionMemoTagEntity` / `ActionMemoTagLinkEntity` / `UserActionMemoSettingsEntity` の4エンティティに `@PersonalData(category = "action_memos")` を付与する。これを忘れると `PersonalDataCoverageValidator` の起動時チェックで ERROR ログが出る（F12.3 §3.5 の網羅性チェック機構に乗る）
  2. F12.3 §3.2 のエクスポート対象データ表に「`action_memos` / 行動メモ / `action_memos`, `action_memo_tags`, `action_memo_tag_links`, `user_action_memo_settings` / `action_memos.json` / JSON」の行を追加
  3. F12.3 §3.4 の `PersonalDataCollector.CATEGORY_FILES` に `Map.entry("action_memos", "action_memos.json")` を追加し、`collectByCategory(...)` メソッド内に `case "action_memos": return collectActionMemos(userId);` を追加
  4. `PersonalDataCollectorTest` に行動メモ収集のテストケースを追加
- **退会時のデータ削除**: `users` の物理削除時に `ON DELETE CASCADE` で4テーブルすべてが連鎖削除される。論理退会の場合はメモも論理削除のままで保持。**論理削除済みメモは GDPR エクスポートには含めない**（ユーザーが「削除した」と認識しているデータをエクスポートに含めると意図に反する）
- **ログ出力時の `content` マスキング**: `ActionMemoService` のログ出力では `memo.content` を平文で記録しない。INFO レベルでは `memoId` / `userId` / `memo_date` / `length` のみ、ERROR レベルでも先頭20文字 + `...` で打ち切る。**理由**: 行動メモは「思考の生ログ」を含み、SYSTEM_ADMIN がエラーログ閲覧時にプライベートな思考を覗くリスクを排除する
- **タイムライン投稿経由の漏洩リスク**: `publish-daily` で生成される `timeline_posts` レコードも `scope_type = PERSONAL` のため、F04.1 のフィード取得 API は他人にこれを返さない。**ただし、F04.1 の管理画面・モデレーション機能が PERSONAL を一覧に含めていないことを実装時に必ず確認**（漏れていればチケット起票して修正）
- **暗号化**: 行動メモは「思考の生ログ」としてプライベートな情報を含むが、Phase 1 ではアプリ層暗号化は導入しない。理由: 他のプライベートテーブル（チャット・個人TODO 等）も暗号化していないため、行動メモだけ先行すると整合性が悪い。**全テーブル横断の共通基盤**として将来的に別案件で導入する

### 6.x 運用・監視

- **ログレベル方針**:
  - `INFO`: メモ作成の `memoId` / `userId` / `memo_date` / `content.length`（**content 本文は出力しない**）、`publish-daily` 成功時の `timeline_post_id` / `memo_count`、週次バッチの `生成件数 / スキップ件数 / 失敗件数`
  - `WARN`: レートリミット超過、200件/日の上限到達、`mood_enabled` の頻繁な切り替え（10分以内に5回超）
  - `ERROR`: DB エラー、`PersonalDataCoverageValidator` の起動時チェック失敗、週次バッチで個別ユーザーの生成失敗。**ERROR レベルでも `content` は先頭20文字 + `...` で打ち切る**
- **メトリクス（Micrometer / Prometheus）**:
  - `action_memo_created_total{user_segment}` — メモ作成数のカウンター
  - `action_memo_publish_daily_total{result="success|error"}` — 終業投稿の成否
  - `action_memo_batch_generated_total` — 週次バッチ成功数
  - `action_memo_batch_skipped_total` — 週次バッチで「メモ0件のためスキップ」した件数
  - `action_memo_batch_failed_total` — 週次バッチで個別ユーザーの失敗数
  - `action_memo_mood_enabled_users` — `mood_enabled = true` のユーザー数（gauge。日次更新）
- **アラート条件**:
  - 週次バッチ全体が **2回連続で起動失敗**（@SchedulerLock 取得失敗ではなく、メソッド本体が例外で終了） → SYSTEM_ADMIN にメール通知
  - `action_memo_batch_failed_total` が **1回の実行で全体の 10% を超過** → WARN ログ + Slack 通知（インフラがあれば）
  - `POST /api/v1/action-memos` の 5xx エラー率が **5分間で 1% 超** → WARN
