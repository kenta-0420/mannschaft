# F02.2.1: ダッシュボード ウィジェットのロール別可視性制御

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 2（F02.2 後続）
> **最終更新**: 2026-04-26
> **モジュール種別**: F02.2 マイダッシュボードのサブ機能
> **関連ドキュメント**: F02.2 マイダッシュボード（基盤）、F01.2 組織・チーム・メンバー・ロール（visibility・ロール階層）

---

## 1. 概要

チーム／組織ダッシュボードのウィジェットを、閲覧者ロール（PUBLIC / SUPPORTER / MEMBER）に応じて表示・非表示制御できるようにする。出欠確認状況・アンケート結果など内部運用情報がチーム外の閲覧者（PUBLIC）にも見えてしまっている現状を是正する。

各ウィジェットには「最低必要ロール」（`min_role`）が設定され、閲覧者のロールがそれ以上でない場合は API レスポンスから除外される。デフォルト値はウィジェットの性質ごとにシステム定義として持ち、ADMIN／DEPUTY_ADMIN がチーム／組織単位で上書き設定できる。

> **本ドキュメントとの参照規約**: ウィジェット一覧の正本は F02.2（マイダッシュボード）が保持する。本ドキュメントではウィジェットの「デフォルト `min_role` 値」のみを定義し、ウィジェット仕様（key・名称・データソース）は F02.2 を参照する。新ウィジェットを追加する際は F02.2 のウィジェット一覧と本ドキュメントのデフォルト値表の**両方**を更新すること（CI で整合性チェックあり、後述 §12）。

### 本ドキュメントのスコープ

| 機能 | 説明 |
|------|------|
| ロール別ウィジェット可視性 | チーム／組織ダッシュボードのウィジェット可視性を閲覧者ロールに応じて自動制御 |
| 管理者向け可視性設定UI | ADMIN／DEPUTY_ADMIN がウィジェットごとに最低必要ロールを変更可能 |
| サーバー側フィルタリング | 認可レイヤーでウィジェットデータを返さない（クライアント側非表示だけに頼らない） |
| デフォルト値の標準化 | 各ウィジェットの推奨デフォルト `min_role` を設計時に定義 |

**スコープ外**:
- 個人ダッシュボード（`scope_type = 'PERSONAL'`）は対象外（自分のデータしか出ないため）
- 未認証アクセスの開放（既存 visibility 体系に従い、認証は引き続き必須）
- ADMIN／DEPUTY_ADMIN 専用ウィジェット（課金・解析）の制御 → F02.2 で既に対応済み
- ロール定義そのものの変更 → F01.2 の既存ロール階層を踏襲

## 2. スコープ

### 対象ロール（閲覧者）

| ロール | 略号 | 説明 |
|--------|------|------|
| SYSTEM_ADMIN | — | 全ウィジェット閲覧可能（バイパス） |
| ADMIN / DEPUTY_ADMIN | — | 全ウィジェット閲覧可能（バイパス。F02.2 既存ルールにより BILLING／ANALYTICS は委任権限が必要） |
| MEMBER | M | `min_role <= MEMBER` のウィジェットを閲覧可 |
| SUPPORTER | S | `min_role <= SUPPORTER` のウィジェットを閲覧可 |
| PUBLIC | P | スコープに対して MEMBER／SUPPORTER ロールを持たない認証済みユーザー。`min_role = PUBLIC` のウィジェットのみ閲覧可 |
| 未認証 | — | 対象外（F01.2 visibility ルールに従う。本機能では対応しない） |

### F01.2 既存ロール定義との関係

F01.2 では `SYSTEM_ADMIN | ADMIN | DEPUTY_ADMIN | MEMBER | SUPPORTER | GUEST` の6種類が定義されている。本機能の `PUBLIC` は「F01.2 で定義された GUEST ロール」または「ロール非保有」の認証済みユーザー全般を指す。

実装上は次の方針:
- 現コードベースに `MembershipService` クラスは**存在せず**、同等機能は `AccessControlService` が提供している（`getRoleName` / `isAdminOrAbove` / `hasRoleOrAbove` 等）。本機能では `MembershipService` を新設**せず**、既存 `AccessControlService` を拡張して活用する。
- 本機能では `RoleResolver.resolveViewerRole(userId, scopeType, scopeId): ViewerRole` ラッパーを新設し、内部で `AccessControlService.getRoleName` を呼び出して戻り値を `MEMBER | SUPPORTER | PUBLIC | DEPUTY_ADMIN | ADMIN | SYSTEM_ADMIN` の6値に正規化する。
- `AccessControlService.getRoleName` が `null`（メンバーシップなし）または `GUEST` を返した場合、本機能では一律 `PUBLIC` として扱う。
- これにより F01.2 のロール定義および既存 API には影響を与えない。

### ロール階層（強い順）

```
SYSTEM_ADMIN ＞ ADMIN ＞ DEPUTY_ADMIN ＞ MEMBER ＞ SUPPORTER ＞ PUBLIC
```

**`min_role` の取りうる値**: `PUBLIC` / `SUPPORTER` / `MEMBER` の3値のみ（ADMIN／DEPUTY_ADMIN 限定ウィジェットは F02.2 の既存「表示ロール」フィールドで別途管理し、`min_role` の対象外）。

### 対象レベル

- [x] 組織 (Organization)
- [x] チーム (Team)
- [ ] 個人 (Personal)（対象外）

### 既存設計との整合

- F02.2 の「対象ロール」表で「GUEST: 対象外」となっていた箇所を、本機能で **PUBLIC 閲覧者の対応**として追加する（F02.2 を改訂）。
- F01.2 の `visibility` （PUBLIC / ORGANIZATION_ONLY / PRIVATE）はスコープ全体の公開範囲を決めるもの。本機能の `min_role` は visibility 内でさらにウィジェット粒度の制御を加える。両者は独立して機能する（visibility = PRIVATE のチームは PUBLIC 閲覧者から見えない時点で本機能の判定は走らない）。

### visibility 別の閲覧者ロール判定

F01.2 の visibility と本機能のロール判定の組み合わせ表:

| visibility | 同チーム MEMBER | 同チーム SUPPORTER | **同組織別チームメンバー** | **完全外部の認証済みユーザー** |
|-----------|----------------|-------------------|---------------------------|------------------------------|
| PUBLIC（チーム） | MEMBER | SUPPORTER | PUBLIC | PUBLIC |
| ORGANIZATION_ONLY（チーム） | MEMBER | SUPPORTER | PUBLIC | 403（visibility 段階で拒否） |
| PRIVATE（チーム） | MEMBER | SUPPORTER | 403 | 403 |
| PUBLIC（組織） | MEMBER | SUPPORTER | （該当なし、組織レベル） | PUBLIC |
| PRIVATE（組織） | MEMBER | SUPPORTER | （該当なし、組織レベル） | 403 |

**設計判断**: 「同組織別チームメンバー」は本機能では **PUBLIC 扱い**とする（その対象チームに対しては MEMBER／SUPPORTER ではないため）。これにより「ORGANIZATION_ONLY visibility のチームを同組織の他チームメンバーが見る → PUBLIC 閲覧者として `min_role = PUBLIC` のウィジェットのみ表示」という動作になる。

## 3. DB設計

### テーブル一覧

| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `dashboard_widget_role_visibility` | スコープ × ウィジェットごとの最低必要ロール（管理者設定） | なし |

### テーブル定義

#### `dashboard_widget_role_visibility`

ADMIN／DEPUTY_ADMIN が設定する、各ウィジェットの最低必要ロール（`min_role`）。レコードがないウィジェットはアプリ層で定義したデフォルト `min_role` が適用される（DB に全件 INSERT しない）。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | スコープ ID |
| `widget_key` | VARCHAR(50) | NO | — | ウィジェット種別キー（F02.2 のウィジェット一覧と同一） |
| `min_role` | VARCHAR(20) | NO | — | 最低必要ロール（`PUBLIC` / `SUPPORTER` / `MEMBER`） |
| `updated_by` | BIGINT UNSIGNED | NO | — | FK → users。最終更新者（監査用） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**

```sql
UNIQUE KEY uq_dwrv_scope_widget (scope_type, scope_id, widget_key)  -- スコープ×ウィジェットの一意制約
INDEX idx_dwrv_scope (scope_type, scope_id)                          -- スコープ別の設定一括取得
```

**制約・備考**

- `scope_type` は `TEAM` または `ORGANIZATION` のみ。`PERSONAL` は対象外（CHECK 制約は MySQL 互換性のため Service 層で検証）。
- `min_role` の取りうる値は `PUBLIC` / `SUPPORTER` / `MEMBER` の3値のみ（Service 層で検証、enum 型は使わず VARCHAR で運用）。
- レコードがないウィジェットはアプリ層のデフォルト値（後述）が適用される。すなわち「未設定 = デフォルト推奨値」。
- ADMIN／DEPUTY_ADMIN 限定ウィジェット（`TEAM_BILLING` / `TEAM_PAGE_VIEWS` / `ORG_BILLING`）は本テーブルの管理対象外。F02.2 既存の「表示ロール」列が引き続き有効。
- スコープ削除時: F01.2 のチーム／組織削除に追従。`ON DELETE CASCADE` ではなく、F02.2 既存の `dashboard_widget_settings` と同じ運用方針（スコープ削除時は別途バッチクリーンアップ）。

### widget_key 命名規則と対応マップ

- **DB 永続層（`dashboard_widget_role_visibility.widget_key`）**: F02.2 と同じ `UPPER_SNAKE_CASE`（例: `TEAM_MEMBER_ATTENDANCE`）。既存 `WidgetKey.java` enum の値と一致。
- **フロントエンド composable（`useDashboardWidgets.ts` の `key` フィールド）**: 既存実装は `kebab-case`（例: `attendance-results`）。
- 両者の対応マップは `WidgetKeyMap`（バックエンド・フロントエンド両方に同名定数）で一元管理する。
- API では DB 形式（`UPPER_SNAKE_CASE`）でやり取りする。フロントエンド側で受信時にマップで解決する。

## 4. API設計

### エンドポイント一覧

| メソッド | パス | 認証 | 認可 | 説明 |
|---------|-----|------|------|------|
| GET | `/api/v1/dashboard/team/{teamId}/widget-visibility` | 必要 | MEMBER 以上 | チームのウィジェット可視性設定一覧（管理者向け） |
| GET | `/api/v1/dashboard/organization/{orgId}/widget-visibility` | 必要 | MEMBER 以上 | 組織のウィジェット可視性設定一覧（管理者向け） |
| PUT | `/api/v1/dashboard/team/{teamId}/widget-visibility` | 必要 | ADMIN / DEPUTY_ADMIN | チームのウィジェット可視性設定一括更新 |
| PUT | `/api/v1/dashboard/organization/{orgId}/widget-visibility` | 必要 | ADMIN / DEPUTY_ADMIN | 組織のウィジェット可視性設定一括更新 |

加えて、**既存 F02.2 の `GET /api/v1/dashboard/team/{teamId}` および `GET /api/v1/dashboard/organization/{orgId}` の仕様改訂** を行う。

### リクエスト／レスポンス仕様

#### `GET /api/v1/dashboard/team/{teamId}/widget-visibility`

**認可**: MEMBER 以上（PUBLIC でも自分のチームを含めて見られるが、PUBLIC は他チームの本エンドポイントを呼ぶ用途がないので簡略化のため MEMBER 以上）。

**レスポンス（200 OK）**

```json
{
  "data": {
    "scope_type": "TEAM",
    "scope_id": 123,
    "widgets": [
      {
        "widget_key": "TEAM_NOTICES",
        "min_role": "PUBLIC",
        "is_default": true
      },
      {
        "widget_key": "TEAM_MEMBER_ATTENDANCE",
        "min_role": "MEMBER",
        "is_default": true
      },
      {
        "widget_key": "TEAM_LATEST_POSTS",
        "min_role": "PUBLIC",
        "is_default": false,
        "updated_by": { "id": 10, "display_name": "山田太郎" },
        "updated_at": "2026-04-26T10:00:00+09:00"
      }
    ]
  }
}
```

- `is_default = true`: DB にレコードなし、アプリ層デフォルトを返却
- `is_default = false`: DB に明示設定あり

ADMIN／DEPUTY_ADMIN 限定ウィジェットはレスポンスから除外する（設定画面に出さないため）。

#### `PUT /api/v1/dashboard/team/{teamId}/widget-visibility`

**認可**: ADMIN は無条件で可。DEPUTY_ADMIN は権限グループに `DASHBOARD_WIDGET_VISIBILITY_MANAGE` パーミッションが含まれる場合のみ可（F01.2 の権限グループ仕組みを利用）。それ以外は 403。

**新パーミッション**: `DASHBOARD_WIDGET_VISIBILITY_MANAGE` を F01.2 のパーミッション一覧に追加する。デフォルトの DEPUTY_ADMIN 権限グループには **含めない**（保守的な初期値）。明示的に付与した DEPUTY_ADMIN のみが可視性設定を変更できる。

**リクエスト**

```json
{
  "widgets": [
    { "widget_key": "TEAM_LATEST_POSTS", "min_role": "PUBLIC" },
    { "widget_key": "TEAM_MEMBER_ATTENDANCE", "min_role": "MEMBER" }
  ]
}
```

**動作**

- リクエストに含まれない `widget_key` の既存レコードは**変更しない**（差分更新）。
- リクエストの `min_role` がアプリ層デフォルトと一致する場合、DB レコードを **DELETE** する（テーブルを肥大化させない）。一致しない場合は UPSERT。
- `widget_key` が未知（ALL_WIDGETS に存在しない）の場合は 400。
- `min_role` が `PUBLIC` / `SUPPORTER` / `MEMBER` 以外の場合は 400。
- ADMIN／DEPUTY_ADMIN 限定ウィジェットを更新しようとした場合は 400（"WIDGET_NOT_CONFIGURABLE"）。
- 監査ログ（`audit_logs`）に `DASHBOARD_WIDGET_VISIBILITY_UPDATED` 種別で記録（誰が・いつ・どのスコープを・どう変更したか）。

**レスポンス（200 OK）**: 更新後の全ウィジェット可視性（GET と同じスキーマ）。

#### `GET /api/v1/dashboard/team/{teamId}`（F02.2 既存 API の改訂）

**認可改訂**:
- 既存: MEMBER 以上
- **新**: 認証必須。閲覧者ロールが MEMBER／SUPPORTER／PUBLIC のいずれかとして判定される（F01.2 visibility との組み合わせは別途。例: visibility = PRIVATE のチームに対する PUBLIC 閲覧者は 403、visibility = PUBLIC のチームの PUBLIC 閲覧者は 200 で本エンドポイントを呼べる）。

**動作改訂**:
1. リクエスト処理時点で閲覧者の本スコープでのロールを `MembershipService.resolveRoleForScope(userId, scopeType, scopeId)` で取得（戻り値は `SYSTEM_ADMIN | ADMIN | DEPUTY_ADMIN | MEMBER | SUPPORTER | PUBLIC`）。
2. `WidgetVisibilityResolver.resolve(scopeType, scopeId)` でスコープの可視性マップ（widget_key → min_role）を取得（DB レコード + デフォルト値の合成）。
3. 各ウィジェットについて閲覧者ロール ≧ `min_role` のもののみデータを取得。
4. レスポンスの `widget_settings` フィールドに各ウィジェットの可視性情報を追加: `[{ widget_key, is_visible_to_viewer, min_role, viewer_role }]`。
5. 不可視ウィジェットはデータ自体を含めない（キーごと省略）。

**レスポンス追加例**:

```json
{
  "data": {
    "team_notices": [...],
    "team_upcoming_events": [...],
    "viewer_role": "PUBLIC",
    "widget_visibility": [
      { "widget_key": "TEAM_NOTICES", "min_role": "PUBLIC", "is_visible": true },
      { "widget_key": "TEAM_MEMBER_ATTENDANCE", "min_role": "MEMBER", "is_visible": false }
    ]
  }
}
```

`viewer_role` を含めることで、フロントエンドが「自分はこのスコープでは PUBLIC 扱いだ」と把握できる。

**`widget_visibility` 配列の包含方針**:
- **本機能の対象ウィジェット**（PUBLIC/SUPPORTER/MEMBER で制御するもの）のみを含める。
- **ADMIN／DEPUTY_ADMIN 限定ウィジェット**（`TEAM_BILLING` / `TEAM_PAGE_VIEWS` / `ORG_BILLING`）は本配列に**含めない**。それらの可視性は F02.2 既存仕様に従い、レスポンス本体にデータが含まれる／含まれないで表現される（`viewer_role` から ADMIN/DEPUTY_ADMIN 専用ウィジェットの可視性をフロント側で判断）。
- これにより本配列は「`min_role` の管理対象」と完全に1対1で対応し、フロントの設定 UI ロジックがシンプルになる。

**用語統一**: レスポンスのフィールド名は `is_visible`（boolean）に統一する。「is_visible_to_viewer」のようなバリエーションは使わない。

**エラーレスポンス**:

| ステータス | 条件 |
|-----------|------|
| 401 | 未認証 |
| 403 | スコープの visibility により閲覧不可（F01.2 既存ルール） |
| 404 | チーム／組織が存在しない・論理削除済み |

## 5. ビジネスロジック

### ロール解決（`RoleResolver.resolveViewerRole`）

```
入力: userId, scopeType, scopeId
出力: SYSTEM_ADMIN | ADMIN | DEPUTY_ADMIN | MEMBER | SUPPORTER | PUBLIC

実装は既存 AccessControlService.getRoleName(userId, scopeType, scopeId) をラップする:

1. user.is_system_admin == true → SYSTEM_ADMIN
2. AccessControlService.getRoleName を呼び出し、戻り値を以下にマップ:
   - "ADMIN" → ADMIN
   - "DEPUTY_ADMIN" → DEPUTY_ADMIN
   - "MEMBER" → MEMBER
   - "SUPPORTER" → SUPPORTER
   - "GUEST" または null → PUBLIC（認証済みだがロールなし／GUEST 相当）
```

組織配下チームの継承は本機能では考慮しない（既存 F01.2 の継承ルールを `AccessControlService` 経由で間接的に利用するのみ）。

### 可視性解決（`WidgetVisibilityResolver.resolve`）

```
入力: scopeType, scopeId
出力: Map<widget_key, min_role>

1. ALL_WIDGETS の中で scope_type に該当するウィジェットを列挙
2. 各ウィジェットに対しデフォルト min_role を設定
3. dashboard_widget_role_visibility から DB 設定を取得し、上書き
4. 結果マップを返却
```

### キャッシュ戦略

| キャッシュ対象 | キー | TTL | 無効化タイミング |
|---------------|------|-----|------------------|
| ウィジェット可視性マップ | `dashboard:widget-visibility:{scopeType}:{scopeId}` | 300秒 | `PUT /widget-visibility` 実行時に該当キーを DEL |
| 閲覧者ロール解決 | `dashboard:viewer-role:{userId}:{scopeType}:{scopeId}` | 60秒 | memberships テーブル更新時（F00.5 MembershipChangedEvent 発火時）に該当 user の関連キーを DEL（既存の F01.2 ロール変更フックに乗る） |

キャッシュ無効化は Service 層で同期的に行う（Valkey の `DEL` コマンドは ~1ms）。PUT 時の DEL に失敗してもエラーにはせず WARN ログ + 5分後に TTL で自然解消。

ロール変更時の即時反映: 現コードベースに `MembershipChangedEvent` 等のメンバーシップ変更通知イベントは**存在しない**（Grep 確認済）。本機能 PR で `MembershipChangedEvent` を新規定義し、F01.2 の `RoleService.assignRole` / `changeRole` / `removeMembership` 完了時に `ApplicationEventPublisher` 経由で発火させる。リスナー（本機能側）は `@TransactionalEventListener(phase = AFTER_COMMIT)` で `dashboard:viewer-role:{userId}:*` キーを DEL する。これにより「降格直後にダッシュボードを見ても古いロールでデータが見える」事故を防ぐ。

**新規イベント仕様**:
- パッケージ: `com.mannschaft.app.role.event.MembershipChangedEvent`（F01.2 既存 role パッケージ配下）
- フィールド: `userId`（必須）/ `scopeType`（TEAM / ORGANIZATION）/ `scopeId` / `changeType`（ASSIGNED / CHANGED / REMOVED）
- 発火点: `RoleService` の上記3メソッドの最終 commit 前（`@Transactional` 内で `eventPublisher.publishEvent`）
- 既存テストへの影響: F01.2 既存単体テストには影響なし（イベント発火が増えるだけ。リスナーが0件なら no-op）

**Race Condition の許容範囲**: リスナーは `AFTER_COMMIT` で動作するため、メンバーシップ更新コミット直後 ~100ms 以内にダッシュボードを開いた場合、最大で**60秒（TTL 上限）**まで古いロールが残るタイムウィンドウがある。本機能ではこれを許容する。

**Valkey 障害時のフォールバック**: Valkey に到達不能な場合、Service 層は例外をキャッチし WARN ログ後にキャッシュをスキップして DB 直取得で動作する（degradation mode）。ただし全リクエストごとに DB クエリが走るため、Valkey 復旧までは応答時間 +50ms 程度の劣化を見込む。Valkey 不通アラートは既存の運用基盤（Phase 1 監視）に委ねる。

### 認可フロー（GET /dashboard/team/{teamId}）

```
1. JWT 認証 → currentUserId
2. teams.find(teamId) → 404 / 削除済み確認
3. F01.2 visibility チェック → 403 if 閲覧不可
4. RoleResolver.resolveViewerRole → viewerRole
5. WidgetVisibilityResolver.resolve → visibilityMap
6. 各ウィジェット:
   - viewerRole >= visibilityMap[widget_key] なら data 取得
   - そうでなければスキップ
7. レスポンス組み立て
```

ADMIN／DEPUTY_ADMIN は SYSTEM_ADMIN と同様、可視性チェックをバイパスする（管理者は全データを見られる）。

### 設定保存時の差分処理

```
入力: 一括更新リクエスト
1. 各 widget_key について:
   a. アプリ層デフォルト min_role を取得
   b. リクエストの min_role と比較
   c. 一致 → DB から DELETE
   d. 不一致 → UPSERT（updated_by = currentUserId）
2. audit_logs に DASHBOARD_WIDGET_VISIBILITY_UPDATED で記録
3. Valkey キャッシュ無効化
```

UPSERT SQL イメージ（`created_at` を維持しつつ `updated_by` / `updated_at` を更新）:

```sql
INSERT INTO dashboard_widget_role_visibility
  (scope_type, scope_id, widget_key, min_role, updated_by, created_at, updated_at)
VALUES (?, ?, ?, ?, ?, NOW(), NOW())
ON DUPLICATE KEY UPDATE
  min_role = VALUES(min_role),
  updated_by = VALUES(updated_by),
  updated_at = NOW();
```

`created_at` は INSERT 時のみ設定され、UPDATE では維持される（MySQL の標準挙動）。

### Service 層での再認可

F01.2 のセキュリティ規約に従い、`DashboardWidgetVisibilityService` の全パブリックメソッドの入り口で次を実施する:

1. `currentUser` のスコープ所属を再検証（既存 `AccessControlService.checkMembership` 経由）
2. ADMIN／DEPUTY_ADMIN+`DASHBOARD_WIDGET_VISIBILITY_MANAGE` パーミッションを持つかチェック
3. 不足する場合は `AccessDeniedException` をスロー（コントローラ層で 403 にマップ）

Controller 層の `@PreAuthorize` は **第一防御**であり、Service 層の入り口チェックを **必須**として並行配置する。

## 7. セキュリティ考慮事項

### 認可の二重防御

- **サーバー側**: API でロール判定を行い、不可視ウィジェットのデータは**レスポンスに含めない**。これが第一防御。
- **クライアント側**: API レスポンスの `widget_visibility` を信頼してフィルタ表示。改ざんされても、既にデータが含まれていなければ漏洩しない。

クライアント側のみのフィルタリングは禁止（F1: フロントの非表示制御だけでは情報漏洩の可能性がある）。

### 既存ウィジェット API 個別エンドポイントの認可

ダッシュボード集約 API（`GET /dashboard/team/{teamId}`）でフィルタしても、ウィジェット個別 API（例: `GET /attendance/{eventId}/results`）が別途存在する場合、そちらでも認可を強化する必要がある。本機能の対象は集約 API のみ。個別 API の認可改訂は各機能の責任範囲（F03.8 出欠管理など）。

**実装注意**: 個別 API が既に MEMBER 以上を要求している場合は問題なし。SUPPORTER／PUBLIC 向けの個別 API が存在し、ダッシュボード集約 API では非表示でも個別 API が叩けるケースは脆弱性となる。各個別 API のレビューを実装時に必ず行う（チェックリスト化）。

#### 実装時にレビューすべき個別 API リスト

本機能の `min_role` でデフォルトを `MEMBER` または `SUPPORTER` にしたウィジェットに対応する個別 API が、対応する認可ガード（最低 MEMBER 以上 or 最低 SUPPORTER 以上）を持っていることを実装フェーズで再確認する。

| ウィジェット | 関連個別 API | 期待される認可レベル |
|--------------|-------------|---------------------|
| TEAM_TODO | `GET /api/v1/teams/{teamId}/todos` | MEMBER 以上 |
| TEAM_PROJECT_PROGRESS | `GET /api/v1/teams/{teamId}/projects` | MEMBER 以上 |
| TEAM_UNREAD_THREADS | `GET /api/v1/teams/{teamId}/bulletins/unread` | MEMBER 以上 |
| TEAM_MEMBER_ATTENDANCE | `GET /api/v1/teams/{teamId}/schedules/{id}/attendances` | MEMBER 以上（ウィジェット集約とは別に F03.8 の認可ルールを適用） |
| TEAM_LATEST_POSTS | `GET /api/v1/teams/{teamId}/timeline` | SUPPORTER 以上（タイムライン visibility 連動） |
| ORG_TODO | `GET /api/v1/organizations/{orgId}/todos` | MEMBER 以上 |
| ORG_STATS | `GET /api/v1/organizations/{orgId}/stats` | SUPPORTER 以上 |

**実装手順**: 各 API の `@PreAuthorize` または Service 層認可を確認。不足する場合は当該 feature doc に修正タスクを追加し PR を切る（本機能の PR とは別）。実装着手前にチェックリストとして PR テンプレートに添付する。

### 監査ログ

ADMIN／DEPUTY_ADMIN による可視性設定変更は `audit_logs` に必ず記録。後から「誰がいつ TEAM_MEMBER_ATTENDANCE を PUBLIC に変えたか」を追跡可能にする。

```json
{
  "action_type": "DASHBOARD_WIDGET_VISIBILITY_UPDATED",
  "actor_id": 10,
  "scope_type": "TEAM",
  "scope_id": 123,
  "details": { "changes": [{ "widget_key": "TEAM_LATEST_POSTS", "before": "MEMBER", "after": "PUBLIC" }] }
}
```

### Visibility との関係

F01.2 のスコープ visibility（PUBLIC / ORGANIZATION_ONLY / PRIVATE）は本機能の判定の **前** に行われる。

- `visibility = PRIVATE` のチーム: メンバー以外は 403（ダッシュボード API を呼べない）
- `visibility = PUBLIC` のチーム: PUBLIC 閲覧者でもダッシュボード API を呼べる → ここで本機能の `min_role` が効く

つまり「PRIVATE チームの PUBLIC 閲覧者向け min_role」のような組み合わせは発生しない。Service 層で 403 となる。

### レート制限

| エンドポイント | 対象 | 制限 |
|---------------|------|------|
| `PUT /widget-visibility` | ADMIN / DEPUTY_ADMIN | IP 単位 60req/min（通常） |
| `GET /dashboard/team/{teamId}` | PUBLIC 閲覧者（`viewer_role = PUBLIC`） | IP+user 単位 30req/min（スクレイピング対策の追加制限） |
| `GET /dashboard/team/{teamId}` | MEMBER 以上 | IP 単位 60req/min（通常） |
| `GET /dashboard/organization/{orgId}` | PUBLIC 閲覧者 | IP+user 単位 30req/min |
| `GET /dashboard/organization/{orgId}` | MEMBER 以上 | IP 単位 60req/min（通常） |
| `GET /widget-visibility` | MEMBER 以上 | IP 単位 60req/min（通常） |

PUBLIC 閲覧者向けの追加制限はスクレイピング・ボット対策。Filter 層（既存の `RateLimitFilter`）で `viewer_role` を判定して適用する。

### 改ざんリスク

- リクエストボディで `viewer_role` を偽装しても、サーバー側で必ず再判定するため意味なし。
- `widget_key` に未知のキーを送ると 400。
- `min_role` に enum 外の値（例: `ADMIN`）を送ると 400。

## 8. テスト方針

### 単体テスト（バックエンド）

- `MembershipService.resolveRoleForScope`: 各ロールパターン（SYSTEM_ADMIN / ADMIN / DEPUTY_ADMIN / MEMBER / SUPPORTER / PUBLIC）の戻り値を網羅
- `WidgetVisibilityResolver.resolve`: DB レコードあり／なし、デフォルト値の合成、ADMIN 限定ウィジェットの除外
- `DashboardWidgetVisibilityService.update`: 差分更新、デフォルト一致時の DELETE、未知キー 400、不正 min_role 400、監査ログ書き込み
- `DashboardController.getTeamDashboard`: viewer_role 判定、可視ウィジェットのみデータ含有、不可視ウィジェットの省略、ADMIN バイパス

### 統合テスト（バックエンド）

- visibility 別 × ロール別の組み合わせ:
  - PRIVATE チーム × PUBLIC ユーザー → 403
  - PUBLIC チーム × PUBLIC ユーザー × min_role=MEMBER のウィジェット → 該当ウィジェット非表示
  - PUBLIC チーム × MEMBER ユーザー × min_role=MEMBER のウィジェット → 表示
  - PUBLIC チーム × ADMIN → 全ウィジェット表示
- PUT 後の GET で設定が反映されること
- PUT 後 audit_logs にレコードが追加されること
- PUT で ADMIN 限定ウィジェットを送信 → 400

### E2E（フロントエンド）

- ADMIN ユーザー: 設定ダイアログにロール別タブが表示される
- MEMBER ユーザー: 設定ダイアログにロール別タブが表示されない
- ADMIN がウィジェットの min_role を変更 → 保存 → 別ユーザー（SUPPORTER）でログイン → 該当ウィジェットが非表示に
- PUBLIC 閲覧者（他チームのメンバー）が PUBLIC visibility のチームを閲覧 → 一部ウィジェットのみ表示

### 退行テスト

- 既存の F02.2 の MEMBER 向けダッシュボード表示が壊れていないか
- 個人ダッシュボード（PERSONAL）には影響しない
- ADMIN 限定ウィジェット（BILLING / PAGE_VIEWS）の挙動が変わらない

### テストフィクスチャ方針

`TestDataBuilder` に以下の便宜メソッドを追加:

```java
TestDataBuilder
    .aUser().withRole(SystemRole.USER)
    .joinedTo(team1, TeamRole.MEMBER)        // チーム1ではメンバー
    .joinedTo(team2, TeamRole.SUPPORTER)     // チーム2ではサポーター
    .notJoinedTo(team3)                      // チーム3では非メンバー (PUBLIC 扱い)
    .build();
```

PUBLIC 閲覧者再現は「非メンバー」として `notJoinedTo()` でセットアップ。E2E テストでは Playwright fixture に role 別ユーザーを定義（`fixtures/users.ts`）し、テストごとに切り替える。

CREATE TABLE dashboard_widget_role_visibility (

### `V2.042__seed_dashboard_widget_visibility_permission.sql`

新パーミッション `DASHBOARD_WIDGET_VISIBILITY_MANAGE` を `permissions` テーブルに seed し、`role_permissions` 経由で ADMIN に自動付与・DEPUTY_ADMIN に天井登録する。

**実コードベースの `permissions` テーブルスキーマ**: `id / name / display_name / scope / created_at / updated_at`（設計書初版で書いた `code / description / category` の3カラムは存在しない。実装は実スキーマに合わせる）。

**役割割当の方針**:
- ADMIN: `is_default = 1` で自動付与（無条件可）
- DEPUTY_ADMIN: `is_default = 0` で天井のみ登録（ADMIN が `permission_groups` 経由で明示付与した場合のみ可）
- MEMBER / SUPPORTER / GUEST: `role_permissions` エントリ自体作らない（絶対に付与不可・安全側設計）

```sql
INSERT INTO permissions (name, display_name, scope, created_at, updated_at)
VALUES ('DASHBOARD_WIDGET_VISIBILITY_MANAGE', 'ダッシュボードウィジェット可視性管理', 'TEAM', NOW(), NOW());

INSERT INTO role_permissions (role_id, permission_id, is_default, created_at)
SELECT r.id, p.id, 1, NOW()
FROM roles r CROSS JOIN permissions p
WHERE r.name = 'ADMIN' AND p.name = 'DASHBOARD_WIDGET_VISIBILITY_MANAGE';

INSERT INTO role_permissions (role_id, permission_id, is_default, created_at)
SELECT r.id, p.id, 0, NOW()
FROM roles r CROSS JOIN permissions p
WHERE r.name = 'DEPUTY_ADMIN' AND p.name = 'DASHBOARD_WIDGET_VISIBILITY_MANAGE';
```

参考: `V9.071__add_manage_friend_teams_permission.sql` と同パターン。

### audit_logs への新 action_type 追加

`audit_logs.action_type` は VARCHAR で運用されているため、新値 `DASHBOARD_WIDGET_VISIBILITY_UPDATED` の追加は DDL 不要（アプリ側で書き込み開始するだけ）。

## 12. 保守性

### コードの一元管理

- ウィジェットのデフォルト `min_role` 値は次の2箇所のみ:
  - バックエンド: `WidgetDefaultMinRoleMap` Java 定数クラス
  - フロントエンド: `useDashboardWidgets.ts` の `ALL_WIDGETS` 配列
- 両者を比較する単体テストを追加（バックとフロントで定義がズレないように）

### CI 整合性チェックのテストスケルトン

バックエンドは `WidgetVisibilityConsistencyTest`（`@SpringBootTest`）でフロントエンドの `widget-default-min-role.json`（フロントから自動生成された JSON）と比較:

```java
@Test
void backendDefaultMinRolesMatchFrontendDefinitions() throws IOException {
    Map<String, String> frontendDefaults = objectMapper.readValue(
        Path.of("../frontend/app/composables/widget-default-min-role.json").toFile(),
        new TypeReference<>() {}
    );
    Map<String, MinRole> backendDefaults = WidgetDefaultMinRoleMap.getAll();
    assertThat(backendDefaults.keySet()).containsExactlyInAnyOrderElementsOf(frontendDefaults.keySet());
    backendDefaults.forEach((key, role) ->
        assertThat(role.name()).isEqualTo(frontendDefaults.get(key))
    );
}
```

フロントエンド側は `npm run build` 時に `useDashboardWidgets.ts` から JSON を自動生成（小さなビルドスクリプトを `scripts/extract-widget-defaults.ts` として配置）。生成 JSON は `.gitignore` に追加せず Git 管理し、PR 時に差分が出れば必ず両方更新される運用。

CI ジョブ:
- `backend-tests` で上記単体テスト実行
- フロントエンドは `npm run lint` + `npm run test:unit` で `widget-default-min-role.json` の生成 + 検証

### モニタリングメトリクス

Micrometer で次のメトリクスを公開:

| メトリクス名 | 種類 | ラベル | 説明 |
|-------------|------|--------|------|
| `dashboard_widget_visibility_filter_total` | Counter | `widget_key`, `scope_type`, `viewer_role`, `outcome`（`shown` / `hidden`） | フィルタ判定の発生回数 |
| `dashboard_widget_visibility_update_total` | Counter | `scope_type`, `outcome`（`success` / `error`） | PUT 設定更新の発生回数 |
| `dashboard_widget_visibility_cache_hit_total` | Counter | `cache_key_type`（`visibility-map` / `viewer-role`） | キャッシュヒット数 |
| `dashboard_widget_visibility_cache_miss_total` | Counter | 同上 | キャッシュミス数 |

**カーディナリティ管理**: `widget_key` のラベルは ALL_WIDGETS（30種程度）×`scope_type`（2）×`viewer_role`（6）= 360 系列で許容範囲内。`user_id` などはラベルにしない（爆発防止）。

### アラート

- `dashboard_widget_visibility_update_total{outcome="error"}` の 5分間和が10超 → Slack `#alerts-backend` 通知
- キャッシュヒット率が 30% を下回る状態が 1時間継続 → INFO ログ（即時対応不要だが調整余地あり）

### ロール比較ロジック

- バックエンド: `RoleHierarchy.isAtLeast(userRole, minRole): boolean` を共通ユーティリティとして実装
- フロントエンド: `composables/useRoleHierarchy.ts` で同等のロジック
- 両者で同じ階層定義を持たせる

### 新ウィジェット追加時の手順

1. `ALL_WIDGETS` 配列に追加（フロント＆バック両方）
2. `WidgetDefaultMinRoleMap` にデフォルト `min_role` を追加（チーム／組織それぞれ）
3. テスト追加
4. F02.2 ドキュメントのウィジェット一覧に追記
5. 本ドキュメント（F02.2.1）のデフォルト値表に追記

CI で「ALL_WIDGETS の全キーがデフォルト min_role 表に存在する」アサートを設置（漏れ防止）。
