# F11.3: UI 国際化（i18n）

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 11
> **最終更新**: 2026-04-10（実装完了・設計書更新）
> **モジュール種別**: デフォルト機能（全アカウント対象）
> **関連ドキュメント**: F11.2 多言語コンテンツ（ユーザー投稿コンテンツの翻訳）、F01.1 認証（users.locale フィールド）

---

## 1. 概要

F11.2「多言語コンテンツ」がユーザー投稿コンテンツ（ブログ・お知らせ等）の翻訳を扱うのに対し、本機能はアプリケーション自体の UI テキスト（バリデーションメッセージ、ダイアログ、エラーメッセージ、ラベル、ボタン等）をアカウント登録時に指定した言語で表示する機能を提供する。

**解決する問題**:
- バリデーションエラーが `"required"` や `"must not be blank"` 等の英語で表示される
- エラーダイアログのメッセージが言語不統一
- ユーザーの `locale` 設定がバックエンドに保存されているが、UI に反映されていない
- フロントエンドのラベル・ボタン・プレースホルダーが日本語固定

**対象メッセージ種別**:
- Bean Validation メッセージ（`@NotBlank`, `@Size`, `@Email` 等のエラー文言）
- 業務エラーメッセージ（`BusinessException` のエラー文言）
- API レスポンスの `message` フィールド
- フロントエンドのラベル・ボタン・ダイアログ・プレースホルダー・バリデーション文言

|-----------|--------|:-----------:|:-------------:|------|

| `en` | 英語 | ◎ 必須 | ◎ 必須 | 国際共通語 |

### 未ログイン時の言語決定フロー

```
1. ブラウザの Accept-Language ヘッダー → ベース言語コードを抽出（例: "en-US" → "en"）
2. 抽出したコードが対応言語6種に含まれれば採用
3. 対応外の場合 → ja（日本語）にフォールバック
```

## 3. DB設計

### 新規テーブル：なし

既存の `users.locale`（VARCHAR(5)）をそのまま使用する。

### 既存テーブルの関連フィールド

#### `users`

| カラム名 | 型 | 許容値 | 説明 |
|---------|---|--------|------|
| `locale` | VARCHAR(5) | `ja` / `en` / `zh` / `ko` / `es` / `de` | 言語コード（ISO 639-1） |
| `timezone` | VARCHAR(50) | 例: `Asia/Tokyo` | タイムゾーン |

**`users.locale` の NULL 対応**: NULL の場合はデフォルト `ja` として扱う。
**バリデーション**: `PUT /api/auth/profile` で locale を更新する際、対応6言語以外の値は 400 Bad Request で拒否する。不正な値が DB に保存されるのを防ぐ。

## 4. バックエンド設計

### 4.1 アーキテクチャ概要

```
HTTP リクエスト受信
    ↓
UserLocaleFilter（OncePerRequestFilter）
※ JwtAuthenticationFilter より後、DispatcherServlet より前に実行
    - JWT から userId を取得（JwtAuthenticationException のみキャッチ、他は素通し）
    - users.locale を参照（キャッシュ経由、NULL の場合 "ja"）
    - 未ログイン時は Accept-Language ヘッダーから判定（サブタグ除去・正規化済み）
    - SUPPORTED_LOCALES に含まれない値は "ja" にフォールバック
    - LocaleContextHolder にセット
    ↓
try {
    Controller / Service 処理
    ↓
    バリデーションエラー発生時
        → ValidationMessages_{locale}.properties から文言取得
    業務エラー発生時
        → messages_{locale}.properties から文言取得
    ↓
    ApiErrorResponse の message フィールドに言語別文言をセット
} finally {
    LocaleContextHolder.resetLocaleContext()  ← 必ず実行（スレッドプール汚染防止）
}
```

### 4.2 application.yml の設定（必須）

```yaml
spring:
  messages:
    basename: messages,ValidationMessages
    encoding: UTF-8
    fallback-to-system-locale: false
    use-code-as-default-message: false
```

> **⚠️ 注意**: `encoding: UTF-8` を明示しないと、中国語・韓国語・スペイン語の特殊文字が文字化けする。
> Java のデフォルト properties エンコーディングは ISO-8859-1 のため、Spring Boot のデフォルト設定（UTF-8）に依存しているだけでは不十分な環境がある。

### 4.2.1 `LocalValidatorFactoryBean` の設定（必須・最重要）

**Bean Validation（`@NotBlank`, `@Size` 等）がユーザーの locale でメッセージを返すには、`LocalValidatorFactoryBean` に Spring の `MessageSource` を紐付けることが必須。この設定がないと `ValidationMessages_{locale}.properties` は参照されず、Hibernate Validator のデフォルト英語メッセージで固定される。**

```java
// src/main/java/com/mannschaft/app/config/I18nConfig.java
@Configuration
@RequiredArgsConstructor
public class I18nConfig {

    private final MessageSource messageSource;

    @Bean
    public LocalValidatorFactoryBean validator() {
        LocalValidatorFactoryBean factory = new LocalValidatorFactoryBean();
        // MessageSource を紐付けることで LocaleContextHolder.getLocale() が参照される
        factory.setValidationMessageSource(messageSource);
        return factory;
    }
}
```

> **注意**: Spring Boot は `LocalValidatorFactoryBean` を自動設定するが、`MessageSource` との紐付けは自動では行われない。上記の明示的な Bean 定義が必要。
> `WebMvcConfigurer.getValidator()` のオーバーライドでも同等の設定が可能だが、Bean 定義方式が推奨。

### 4.3 メッセージプロパティファイル構成

```
src/main/resources/
├── messages.properties              ← フォールバック（ja と同じ内容）★同期必須
├── messages_ja.properties           ← 日本語
├── messages_en.properties           ← 英語
├── messages_zh.properties           ← 中国語（簡体）※ファイル自体はUTF-8で保存
├── messages_ko.properties           ← 韓国語  ※同上
├── messages_es.properties           ← スペイン語
├── messages_de.properties           ← ドイツ語
├── ValidationMessages.properties    ← フォールバック（ja と同じ内容）★同期必須
├── ValidationMessages_ja.properties
├── ValidationMessages_en.properties
├── ValidationMessages_zh.properties ※ファイル自体はUTF-8で保存
├── ValidationMessages_ko.properties ※同上
├── ValidationMessages_es.properties
└── ValidationMessages_de.properties
```

> **⚠️ フォールバック動作（重要）**: Spring の `ResourceBundleMessageSource` は、指定 locale のファイルにキーが存在しない場合 `messages.properties`（基底ファイル）にフォールバックする。`messages_ja.properties` にはフォールバックしない。
> 新しいキーを追加する際は **`messages_ja.properties` と `messages.properties` の両方** に追加すること。片方だけ追加すると、対応外言語ユーザーで `NoSuchMessageException` が発生する（`use-code-as-default-message=false` の場合）。

### 4.6 UserLocaleFilter（実装詳細）

```java
// src/main/java/com/mannschaft/app/common/i18n/UserLocaleFilter.java
@Component
@Order(Ordered.LOWEST_PRECEDENCE - 10)
// Spring Security の DelegatingFilterProxy（@Order = -100）より後に実行される。
// → JwtAuthenticationFilter が SecurityContextHolder にセット済みの状態で動作する。
@RequiredArgsConstructor
public class UserLocaleFilter extends OncePerRequestFilter {

    private static final Set<String> SUPPORTED_LOCALES =
        Set.of("ja", "en", "zh", "ko", "es", "de");
    private static final Locale DEFAULT_LOCALE = Locale.JAPANESE;

    // 処理順（重要: finally でのクリアを必ず実装すること）:
    //
    // 1. SecurityContextHolder からログインユーザーの userId を取得
    //    → JwtAuthenticationFilter が先行して SecurityContext にセット済みのため JWT の再パース不要
    //    → Authentication が null または isAuthenticated() == false の場合は未ログインとして扱う
    //
    // 2. userId が取得できた場合（ログイン済み）:
    //    → UserLocaleCache.getLocale(userId) でキャッシュ参照（TTL: 5分）
    //    → キャッシュミス時のみ UserRepository.findLocaleById(userId) を実行
    //    → 返り値が null の場合は "ja" にフォールバック（null チェック必須）
    //    → Locale.forLanguageTag(localeStr).getLanguage() でベース言語コードのみ抽出
    //       例: "en-US" → "en", "zh-CN" → "zh", "zh-TW" → "zh"
    //    → SUPPORTED_LOCALES に含まれない場合は DEFAULT_LOCALE にフォールバック
    //
    // 3. 未ログイン時:
    //    → Accept-Language ヘッダーを Locale.LanguageRange.parse() で解析
    //    → ⚠️ IllegalArgumentException を catch して DEFAULT_LOCALE にフォールバック
    //       （不正ヘッダー例: "invalid;q=abc" でフィルターごとクラッシュするため必須）
    //    → 優先度順に SUPPORTED_LOCALES と照合
    //    → Accept-Language: en-US,en;q=0.9,ja;q=0.8 → "en" を採用
    //    → Accept-Language が null または空の場合 → DEFAULT_LOCALE
    //    → マッチなし → DEFAULT_LOCALE
    //
    // 4. LocaleContextHolder.setLocaleContext(new SimpleLocaleContext(locale), false) でセット
    //    ⚠️ 第2引数 inheritable は必ず false（デフォルト）にすること。
    //       Virtual Threads 環境（spring.threads.virtual.enabled=true）で
    //       inheritable=true を使うと InheritableThreadLocal が予期しないスレッドに伝搬する。
    //
    // 5. [必須] finally ブロックで LocaleContextHolder.resetLocaleContext() を実行
    //    → スレッドプールでスレッドが再利用される際の locale 汚染を防ぐ
    //    → Virtual Threads でもメモリリーク防止のため省略不可
}
```

**⚠️ セキュリティ注意点**:
- `Accept-Language` ヘッダーはユーザーが自由に偽装できる。ただし、言語の決定に使うだけでありアクセス制御に影響しないため、偽装されても実害はない（別言語のメッセージが表示されるだけ）
- `SecurityContextHolder` から userId を取得するため JWT を二重パースしない。アクセス制御の判断は JwtAuthenticationFilter に一元化する

**⚠️ パフォーマンス注意点**:
- locale 取得のための DB クエリをリクエストごとに実行すると高負荷時に DB がボトルネックになる
- **必須**: `UserLocaleCache`（Caffeine Cache または Spring @Cacheable）で TTL 5分のキャッシュを設ける
- locale 変更時はキャッシュを即時 evict する（`@CacheEvict` はサービスメソッド成功後に実行されるため、DB コミット前に evict する心配はない）

### 4.7 ErrorCode との統合（移行戦略）

現在の `ErrorCode` インターフェースは `getMessage()` で日本語文字列をハードコードして返している。

**Phase 11 での対応（方式C: 最小変更）**:
- `GlobalExceptionHandler` に `MessageSource` を注入し、`BusinessException` キャッチ時に多言語メッセージを返す
- `ErrorCode.getCode()` をメッセージキーに変換する（変換ルール: `TEAM_001` → `error.team.001`）
- properties にキーが存在しない場合は `ErrorCode.getMessage()`（日本語）にフォールバック
- 既存の `ErrorCode.getMessage()` は互換性のため残す

**キー命名規則（方式C）**:
```
messages.properties のキー形式:
  error.{ドメイン小文字}.{番号}
  例: TEAM_001 → error.team.001
      AUTH_003 → error.auth.003
      COMMON_001 → error.common.001
```

**`GlobalExceptionHandler` の修正例（最小変更）**:
```java
@RestControllerAdvice
@RequiredArgsConstructor
public class GlobalExceptionHandler {

    private final MessageSource messageSource;  // ← 追加

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ApiErrorResponse> handleBusinessException(BusinessException ex) {
        ErrorCode errorCode = ex.getErrorCode();
        // messages_{locale}.properties からメッセージ取得。キー不在時は日本語にフォールバック
        String messageKey = "error." + errorCode.getCode().toLowerCase().replace("_", ".", 1);
        // 例: TEAM_001 → "error.team.001"
        String message;
        try {
            message = messageSource.getMessage(
                messageKey,
                null,
                LocaleContextHolder.getLocale()
            );
        } catch (NoSuchMessageException e) {
            message = errorCode.getMessage();  // フォールバック（既存の日本語）
        }
        // ... レスポンス生成
    }
}
```

> **注意**: `TEAM_001.toLowerCase()` は `"team_001"` になる。`replace("_", ".", 1)` は Java では `replaceFirst("_", ".")` で 1 箇所だけ置換する。実際のコードでは `"error." + code.substring(0, code.lastIndexOf('_')).toLowerCase() + "." + code.substring(code.lastIndexOf('_') + 1)` のように分割すること。

**Phase 12 での完全移行（方式A）**:
- `ErrorCode` インターフェースに `getMessageKey()` を追加
- 各 ErrorCode Enum に `messageKey` フィールドを追加
- `getMessage()` を deprecated にして段階廃止

### 4.8 @Async・WebSocket での locale 伝搬

`UserLocaleFilter` は HTTP リクエストスレッドに locale をセットするが、`@Async` メソッドや WebSocket ハンドラーは別スレッドで動作するため locale が引き継がれない。

**対応方針**:
- **@Async メソッド**: ビジネスロジック内でユーザー向けメッセージを生成する場合、locale をメソッド引数として明示的に渡す
- **WebSocket（SignageWebSocketPublisher 等）**: サイネージのメッセージは特定ユーザー向けではなく全画面向けのため、locale 対応は不要。将来的にユーザー個別通知が必要になった場合は引数渡しで対応
- **メール・プッシュ通知（@TransactionalEventListener）**: 受信者の locale を DB から取得してメッセージを組み立てる（現行の通知実装パターンに準拠）

## 5. ハイブリッド翻訳戦略

F11.3 の i18n 実装とブラウザ自動翻訳を組み合わせ、ユーザーが翻訳ボタンを押さなくても自動で適切な言語が表示されるようにする。

### 5.1 役割分担

| 種別 | 担当 | 理由 |
|------|------|------|
| APIエラーメッセージ | **F11.3（i18n）** | バックエンドから返る文字列はブラウザが拾えない |
| バリデーション文言 | **F11.3（i18n）** | Vue の状態変化後に動的に表示されるためブラウザ翻訳が追いつかない |
| トースト・モーダル | **F11.3（i18n）** | 動的生成のため同上 |
| 静的ラベル・ボタン | **F11.3（i18n）** | 対応6言語は実装で対応 |
| 6言語以外（アラビア語・タイ語等） | **ブラウザ自動翻訳** | F11.3 の対象外言語はブラウザに委任 |
| F11.3 段階実装中の未翻訳箇所 | **ブラウザ自動翻訳** | 実装が追いつくまでの安全網 |

### 5.2 Chrome自動翻訳を「ボタン不要」で発動させる仕組み

Chrome は `<html lang>` ≠ Chromeの言語設定 のとき自動翻訳を発動する（ユーザー操作不要）。

**`<html lang>` の設定方針**:

| フェーズ | `<html lang>` の値 | Chrome の動作 |
|---------|-------------------|--------------|
| F11.3 段階実装中 | `"ja"` 固定 | 非日本語ユーザー向けに自動翻訳が発動（安全網として機能） |
| F11.3 完全実装後 | `{userLocale}` に動的切替 | ページ言語とブラウザ言語が一致するため Chrome は介入しない |

> **⚠️ 競合リスク（段階実装中）**: F11.3 が英語テキストを表示している状態で `lang="ja"` のままにしておくと、Chrome が「日本語ページ」と誤認して英語→英語翻訳を試みる場合がある。この誤翻訳リスクを抑えるため、後述の `translate="no"` による制御が必須。

### 5.3 F11.3 が担当する箇所を Chrome の二重翻訳から守る

Chrome が翻訳した後に React が DOM を書き換えると表示が壊れる。また、F11.3 が正しい言語で表示済みの箇所を Chrome が再翻訳すると文言が崩れる。

**対策: `translate="no"` の系統的付与**

```vue
<!-- ❌ 忘れた場合: Chrome が二重翻訳して文言が崩れる可能性 -->
<p class="error-message">{{ $t('validation.required') }}</p>

<!-- ✅ 正しい実装: i18n 制御済み要素には必ず translate="no" -->
<p translate="no" class="error-message">
  {{ $t('validation.required') }}
</p>

<!-- ✅ i18n 未対応の静的コンテンツ → Chrome に翻訳させる（属性なし） -->
<footer>© 2026 Mannschaft</footer>
```

**保守負担の軽減策**: `translate="no"` の付け忘れを防ぐため、i18n テキストを表示する共通コンポーネント（`<I18nText>` 等）を作成し、内部で自動付与する。

```vue
<!-- components/I18nText.vue -->
<template>

```
ユーザーがページを開く

### 6.3 locale の取得・切り替えフロー（Nuxt 3）

```
1. 初回アクセス（未ログイン）:
   @nuxtjs/i18n の detectBrowserLanguage が navigator.language を自動検出
   → SUPPORTED_LOCALES に含まれれば採用、それ以外は "ja"（defaultLocale）
   → <html lang> も自動で切り替わる

2. ログイン成功:
   APIレスポンスの locale フィールドを取得
   → useI18n().setLocale(locale) で即時切り替え
   → useAuthStore に locale を保存

3. プロフィール設定で言語変更:
   → PUT /api/auth/profile { locale: "en" }
   → 成功後に setLocale(locale) で即時反映

4. ページリロード時:
   @nuxtjs/i18n が cookie または detectBrowserLanguage で復元
   （locale は useAuthStore 経由でも保持）
```

> **⚠️ Nuxt 3 の SSR 対応**: `@nuxtjs/i18n` は SSR を考慮した設計のため、`localStorage` を直接操作せずに `setLocale()` / `useLocale()` composable を使う。`localStorage` が必要な場合は Nuxt の `useNuxtApp().$i18n` 経由でアクセスし、`process.client` でガードする。

### 6.4 バックエンドとの locale コード統一

バックエンドとフロントエンドで異なる locale コード形式を使うと言語の不一致が発生する。

| NG パターン | OK パターン |
|------------|------------|
| BE: `"zh"` / FE: `"zh-CN"` → 不一致 | BE・FE ともに `"zh"` で統一 |
| BE: `"ko"` / FE: `"ko-KR"` → 不一致 | BE・FE ともに `"ko"` で統一 |

**`@nuxtjs/i18n` での正規化**: `locales` 配列に `code: 'zh'` と定義することで、`zh-CN` は自動的に `zh` にマッピングされる。追加の正規化ユーティリティは不要。ただし composable で手動設定する場合は以下を使う:

```ts
// app/utils/normalizeLocale.ts
const SUPPORTED = ['ja', 'en', 'zh', 'ko', 'es', 'de'] as const
export const normalizeLocale = (lang: string): string => {
  const base = lang.split('-')[0].toLowerCase()
  return (SUPPORTED as readonly string[]).includes(base) ? base : 'ja'
}
```

## 9. セキュリティ・不具合チェックリスト

実装時に必ず確認すること。

### セキュリティ
- [x] `Accept-Language` ヘッダー値は `SUPPORTED_LOCALES` で必ずホワイトリスト検証する（偽装されても言語変更のみで実害なし、ただし検証を怠ると予期しない Locale が設定される）
- [x] `UserLocaleFilter` は `SecurityContextHolder` から userId を取得する（JWT の二重パース禁止。アクセス制御は JwtAuthenticationFilter に一元化）
- [x] `PUT /api/auth/profile` の locale 更新は対応6言語以外を 400 で拒否する（不正値の DB 保存防止）
- [x] locale はメッセージ表示のみに使用し、アクセス制御・認証に一切関与しないことを確認する

### 不具合防止（バックエンド）
- [x] **`I18nConfig.java` で `LocalValidatorFactoryBean` に `MessageSource` を紐付ける**（最重要。この設定がないと Bean Validation が多言語化されない）
- [x] `UserLocaleFilter` の `finally` ブロックで `LocaleContextHolder.resetLocaleContext()` を必ず呼ぶ（スレッドプール汚染防止）
- [x] `LocaleContextHolder.setLocaleContext(ctx, false)` の `inheritable` は `false` 固定（Virtual Threads 環境で `true` にすると予期しない伝搬が発生）
- [x] `Locale.LanguageRange.parse()` は `IllegalArgumentException` を `catch` して `DEFAULT_LOCALE` にフォールバックする（不正な Accept-Language でフィルタークラッシュを防ぐ）
- [x] `users.locale` が `null` の場合は `Locale.forLanguageTag()` に渡す前に `"ja"` に差し替える（NPE 防止）
- [x] `application.yml` に `spring.messages.encoding: UTF-8`、`spring.messages.basename: messages,ValidationMessages` を設定する
- [x] `messages.properties`（基底）と `messages_ja.properties` に同じキーを必ず両方追加する（フォールバックは `messages.properties` のみ参照されるため）
- [x] `Locale.forLanguageTag(localeStr).getLanguage()` でベース言語コードのみ抽出する（`en-US` → `en`、`zh-CN` → `zh`）
- [x] `UserLocaleFilter` のフィルター順序が Spring Security より後になっていることを確認する（`@Order(Ordered.LOWEST_PRECEDENCE - 10)` は Security の `-100` より大きい数値のため後続実行される）
- [x] `GlobalExceptionHandler` に `MessageSource` を注入し、`messages_{locale}.properties` からメッセージを取得する（§4.7 参照）
- [x] `@Async` メソッドでユーザー向けメッセージを生成する場合は locale を引数で明示的に渡す（ThreadLocal は伝搬しない）
- [x] locale キャッシュ（TTL 5分）を実装し、locale 変更時は `@CacheEvict` で即時 evict する

### 不具合防止（フロントエンド）
- [x] BE と FE で locale コードの形式を統一する（どちらも `zh` 等のベースコードのみ）
- [x] フロントエンドの `normalizeLocale()` でブラウザの `navigator.language` を正規化する（`en-US` → `en`）
- [x] `localStorage` へのアクセスは `typeof window !== 'undefined'` でガードする（SSR 環境での `undefined` クラッシュ防止）
- [x] i18n 制御済みの DOM 要素には `translate="no"` を付与する（Chrome 二重翻訳防止）。`settings/language.vue` のタイトル・ラベル・ボタンに付与済み

## 10. 懸念事項

1. **翻訳精度**: 中国語・韓国語・スペイン語・ドイツ語の翻訳文言は AI 生成。Phase 11 では AI 生成で実装し、アプリ規模拡大後にネイティブレビューする。
2. **フロント翻訳ファイルの分量**: 全画面を6言語対応する翻訳ファイルは相当量。優先ページ（ログイン・登録・主要ダッシュボード）から段階的に整備する。
3. **ErrorCode 変更の破壊的影響**: 既存テストが日本語文言をアサートしているため、段階的移行（Phase 11 は方式C）を維持する。
4. **RTL 言語**: 今回の6言語はすべて LTR（左書き）のため RTL 対応は不要。
5. **`zh-TW`（繁体字）ユーザーへの扱い**: `Locale.forLanguageTag("zh-TW").getLanguage()` は `"zh"` を返すため、台湾・香港ユーザー（繁体字使用）が簡体字コンテンツを受け取ることになる。Phase 11 では `zh` = 簡体字のみのサポートとし、繁体字（`zh-TW`）ユーザーへの告知または将来的な `zh-Hant` 対応を検討事項として積み残す。
