# F09.3: 駐車場区画管理

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 9
> **最終更新**: 2026-03-15
> **関連ドキュメント**: F04.3 プッシュ通知（申請結果通知・割り当て変更通知・譲渡希望マッチ通知）、F09.1 住民台帳（居住者管理連携）、F01.3 テンプレート・モジュール（選択式モジュール #10）

---

## 1. 概要

マンション・スポーツ複合施設等において、駐車場区画の登録・管理・利用者割り当て・空き申請・譲渡希望を一元管理する機能。管理者（ADMIN / DEPUTY_ADMIN）が区画番号・種別（屋内 / 屋外 / 身障者用等）・月額料金を区画ごとに登録し、個人ユーザーが車両（車・バイク・自転車を最大3台）を登録する。管理者は車両単位で区画への割り当てを行い、単一指定・複数同時指定・一括指定に対応する。空き区画の状況をリアルタイムに表示し、先着（FIRST_COME）または抽選（LOTTERY）による空き申請・割り当てフローを提供する。現在の利用者が不要な区画の譲渡希望を掲示し、他メンバーが申請 → 管理者承認で譲渡が完了する。ナンバープレート情報は個人情報保護のため暗号化して保存する。空き区画のウォッチリスト通知、利用者間の区画交換、契約期間管理、料金変更履歴、付随区画のサブリース（貸し出し）掲示板を備え、マンション管理組合の実運用に必要な機能を網羅する。

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全組織・チームの区画・割り当て・申請・譲渡希望の参照 |
| ADMIN | 区画 CRUD・個別価格設定・車両割り当て（単一/複数/一括）・空き申請の承認/抽選実行・譲渡承認・ステータス管理（MAINTENANCE 切り替え等） |
| DEPUTY_ADMIN | `MANAGE_PARKING` 権限を持つ場合: 区画 CRUD・割り当て・申請承認・譲渡承認 |
| MEMBER | 車両登録（最大3台）・空き区画の確認・空き申請の提出・自分の割り当て区画の確認・譲渡希望の登録・他者の譲渡希望への申し込み |
| SUPPORTER | 空き区画の確認（閲覧のみ） |
| GUEST | 対象外 |

### テーブル定義

#### `parking_spaces`

駐車場区画の基本情報。区画番号・種別・月額料金・ステータスを管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `scope_type` | VARCHAR(20) | NO | — | スコープ種別（`TEAM` / `ORGANIZATION`） |
| `scope_id` | BIGINT UNSIGNED | NO | — | teams.id または organizations.id |
| `space_number` | VARCHAR(20) | NO | — | 区画番号（例: 'A-101', 'B-205', '屋外-15'） |
| `space_type` | ENUM('INDOOR', 'OUTDOOR', 'ACCESSIBLE', 'MOTORCYCLE', 'BICYCLE', 'VISITOR', 'OTHER') | NO | 'OUTDOOR' | 区画種別（屋内 / 屋外 / 身障者用 / バイク専用 / 自転車専用 / 来場者用 / その他） |
| `space_type_label` | VARCHAR(50) | YES | NULL | 区画種別の補足ラベル（`space_type = OTHER` 時の自由記述。例: '大型車専用', 'EV充電付き'） |
| `price_per_month` | DECIMAL(10,0) | YES | NULL | 月額料金（円）。NULL = 無料 |
| `status` | ENUM('VACANT', 'OCCUPIED', 'MAINTENANCE') | NO | 'VACANT' | 区画ステータス（空き / 使用中 / メンテナンス） |
| `floor` | VARCHAR(10) | YES | NULL | 階数・エリア（例: 'B1', '1F', '屋上'）。表示・フィルタ用 |
| `notes` | VARCHAR(500) | YES | NULL | 管理者メモ（例: '柱あり注意', '高さ制限2.0m'） |
| `application_status` | ENUM('NOT_ACCEPTING', 'ACCEPTING', 'LOTTERY_CLOSED') | NO | 'NOT_ACCEPTING' | 申請受付状態（NOT_ACCEPTING=受付停止 / ACCEPTING=受付中 / LOTTERY_CLOSED=抽選受付終了・実行待ち） |
| `allocation_method` | ENUM('FIRST_COME', 'LOTTERY') | YES | NULL | 割り当て方式（application_status = ACCEPTING 時のみ有効。先着 / 抽選）|
| `application_deadline` | DATETIME | YES | NULL | 申請締切日時（LOTTERY 時は必須。FIRST_COME 時は任意。NULL = 無期限）|
| `created_by` | BIGINT UNSIGNED | NO | — | FK → users（登録者）。ON DELETE RESTRICT |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
UNIQUE KEY uq_ps_scope_number (scope_type, scope_id, space_number, deleted_at) -- 同一スコープ内の区画番号重複防止（論理削除考慮）
INDEX idx_ps_scope_status (scope_type, scope_id, status)                       -- スコープ別の区画一覧・空き検索
INDEX idx_ps_scope_type (scope_type, scope_id, space_type, status)             -- 種別絞り込み
INDEX idx_ps_scope_floor (scope_type, scope_id, floor, status)                 -- 階数・エリア絞り込み
```

**制約・備考**
- `scope_type` + `scope_id` はポリモーフィック参照。FK 制約なし（アプリ層で検証）
- `space_number` は同一スコープ内で一意。論理削除後は同一番号で再作成可能（`deleted_at` を UNIQUE KEY に含めることで対応。MySQL の場合 `deleted_at IS NULL` の行のみ一意制約が効く）
- `status` 遷移: `VACANT` → `OCCUPIED`（割り当て時に自動）→ `VACANT`（割り当て解除時に自動）。`VACANT` / `OCCUPIED` → `MAINTENANCE`（管理者が手動設定）→ `VACANT`（管理者が手動復旧）。`OCCUPIED` → `MAINTENANCE` 時は既存の割り当てを先に解除する必要がある（Service 層で検証、409 エラー）
- VISITOR 区画の空き判定: `parking_spaces.status` は常に `VACANT` のまま維持。時間帯ごとの予約有無は `parking_visitor_reservations` テーブルの重複チェッククエリで動的に判定する（区画レベルの単一ステータスでは時間帯別の空き管理ができないため）
- `price_per_month` は区画ごとに個別設定可能（同一スコープ内で料金が異なるケースに対応）。NULL は無料区画（自転車置き場等）
- `space_type_label`: `space_type = OTHER` 以外でも補足情報として使用可能（例: `INDOOR` + `'EV充電付き'`）
- 区画の物理削除は不可（割り当て履歴の参照先として必要）。論理削除のみ
- `application_status` 遷移: `NOT_ACCEPTING`（デフォルト。区画作成直後）→ `ACCEPTING`（ADMIN が受付開始）→ `LOTTERY_CLOSED`（LOTTERY 時、deadline 超過で自動 or ADMIN 手動）→ `NOT_ACCEPTING`（割り当て完了後に自動復帰）
- `application_deadline` 超過時は期限切れバッチで `application_status` を `LOTTERY_CLOSED` に自動更新
- `allocation_method` は区画レベルで管理（各申請レコードにはコピーしない。区画の `allocation_method` を参照）

#### `registered_vehicles`

ユーザーが登録する車両情報。1ユーザーにつき最大3台まで登録可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `vehicle_type` | ENUM('CAR', 'MOTORCYCLE', 'BICYCLE') | NO | — | 車種区分（車 / バイク / 自転車） |
| `plate_number` | VARBINARY(512) | NO | — | ナンバープレート（AES-256-GCM 暗号化して保存。個人情報保護） |
| `plate_number_hash` | CHAR(64) | NO | — | ナンバープレートの SHA-256 ハッシュ（一意性チェック・検索用） |
| `nickname` | VARCHAR(50) | YES | NULL | 車両のニックネーム（例: '通勤用', '白いプリウス'）。表示用 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_rv_user (user_id, deleted_at)                           -- ユーザーの車両一覧
UNIQUE KEY uq_rv_plate_hash (plate_number_hash, deleted_at)       -- ナンバープレートの重複登録防止（暗号化カラムでは UNIQUE 不可のためハッシュで代用）
```

**制約・備考**
- 1ユーザーあたり最大3台（`deleted_at IS NULL` の件数を Service 層でバリデーション）
- `plate_number`: AES-256-GCM で暗号化して保存。暗号化キーは AWS KMS で管理。復号は Service 層の `VehicleEncryptionService` が担当し、API レスポンスではマスキング表示（例: `"***-1234"`）
- `plate_number_hash`: 暗号化済みカラムでは `UNIQUE` 制約や検索が不可能なため、SHA-256 ハッシュを別途保存。同一ナンバーの重複登録をハッシュで検出。ソルトはアプリケーション定数（`ParkingProperties.plateHashSalt`）で管理
- `vehicle_type = BICYCLE` の場合: `plate_number` は防犯登録番号を保存。暗号化・ハッシュの扱いは同一
- `user_id` ON DELETE CASCADE: ユーザー退会時に車両情報もカスケード削除。割り当て履歴（`parking_assignments`）は `vehicle_id` の FK が残るが ON DELETE SET NULL で対応
- 論理削除: 車両の削除は論理削除。アクティブな割り当て（`parking_assignments.released_at IS NULL`）がある場合は削除不可（Service 層で 409 エラー）

#### `parking_assignments`

車両と区画の紐付け。`released_at IS NULL` が現在の割り当て、`released_at IS NOT NULL` が過去の履歴。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `space_id` | BIGINT UNSIGNED | NO | — | FK → parking_spaces。ON DELETE RESTRICT |
| `vehicle_id` | BIGINT UNSIGNED | YES | NULL | FK → registered_vehicles。ON DELETE SET NULL（退会時に NULL 化。履歴として保持） |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT。車両所有者（`vehicle_id` NULL 化後も利用者を特定するため冗長保持） |
| `assigned_by` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT。割り当て実行者（ADMIN / DEPUTY_ADMIN） |
| `assigned_at` | DATETIME | NO | CURRENT_TIMESTAMP | 割り当て日時 |
| `contract_start_date` | DATE | YES | NULL | 契約開始日（NULL = assigned_at と同日） |
| `contract_end_date` | DATE | YES | NULL | 契約終了予定日（NULL = 無期限） |
| `released_at` | DATETIME | YES | NULL | 割り当て解除日時。NULL = 現在有効な割り当て |
| `released_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL。解除実行者 |
| `release_reason` | VARCHAR(200) | YES | NULL | 解除理由（例: '退去', '譲渡', 'メンテナンス'） |

**インデックス**
```sql
UNIQUE KEY uq_pa_space_active (space_id, released_at)              -- 1区画に同時に1台のみ割り当て（released_at IS NULL の行が最大1件）
INDEX idx_pa_vehicle (vehicle_id, released_at)                     -- 車両の割り当て履歴
INDEX idx_pa_user (user_id, released_at)                           -- ユーザーの割り当て一覧
INDEX idx_pa_space_history (space_id, assigned_at DESC)            -- 区画の割り当て履歴
```

**制約・備考**
- `uq_pa_space_active`: MySQL では `NULL` 値は UNIQUE KEY の重複チェック対象外のため、`released_at IS NULL` の行は `space_id` ごとに最大1件に制約される。`released_at IS NOT NULL` の行は同一 `space_id` に複数存在可能（履歴）
- `space_id` ON DELETE RESTRICT: 区画削除時に割り当て履歴が存在する場合はエラー。論理削除を使用する設計のため、物理削除は想定しない
- `vehicle_id` ON DELETE SET NULL: 車両の物理削除（ユーザー退会時のカスケード）で NULL 化。`user_id` は RESTRICT で保持し、履歴の利用者特定は可能
- `created_at` / `updated_at` が未定義のため追記不要（`assigned_at` / `released_at` が実質的なタイムスタンプとして機能）
- 割り当て時の処理: (1) `parking_assignments` に INSERT (`released_at = NULL`) → (2) `parking_spaces.status` = `OCCUPIED` に更新
- 割り当て解除時の処理: (1) `parking_assignments.released_at` = `NOW()` に更新 → (2) `parking_spaces.status` = `VACANT` に更新
- 一括割り当て（`POST /parking/spaces/bulk-assign`）: 1リクエストで複数の車両-区画ペアを同時に割り当て可能（最大20ペア）。トランザクション内で全件処理
- `contract_end_date` 設定時: 契約終了30日前にリマインダー通知をバッチ送信（F04.3: `PARKING_CONTRACT_EXPIRING`）
- 期限超過しても自動解除しない（ADMIN の判断に委ねる。契約更新の可能性があるため）

#### `parking_applications`

空き区画への利用申請・譲渡希望への申し込み。先着（FIRST_COME）または抽選（LOTTERY）で割り当てを決定する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `space_id` | BIGINT UNSIGNED | NO | — | FK → parking_spaces。ON DELETE CASCADE |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `vehicle_id` | BIGINT UNSIGNED | NO | — | FK → registered_vehicles。ON DELETE CASCADE。申請対象の車両 |
| `source_type` | ENUM('VACANCY', 'LISTING') | NO | 'VACANCY' | 申請種別（空き区画申請 / 譲渡希望への申し込み） |
| `listing_id` | BIGINT UNSIGNED | YES | NULL | FK → parking_listings。source_type = LISTING 時のみ設定 |
| `status` | ENUM('PENDING', 'APPROVED', 'REJECTED', 'CANCELLED', 'LOTTERY_PENDING') | NO | 'PENDING' | 申請ステータス |
| `priority` | SMALLINT UNSIGNED | NO | 0 | 申請優先度（0 = 通常。ADMIN が手動調整可能。身障者用区画等の優先割り当てに使用） |
| `message` | VARCHAR(500) | YES | NULL | 申請者のメッセージ（例: '子供の送迎で必要', '現在の区画が遠い'） |
| `rejection_reason` | VARCHAR(500) | YES | NULL | 拒否理由（ADMIN が REJECTED 時に設定。任意） |
| `lottery_number` | INT UNSIGNED | YES | NULL | 抽選番号（LOTTERY モードで `SecureRandom` により付与。当選判定用） |
| `decided_at` | DATETIME | YES | NULL | 決定日時（APPROVED / REJECTED 時に設定） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE KEY uq_papp_space_user_source (space_id, user_id, source_type, status)  -- 同一区画への同一種別の二重申請防止（PENDING/LOTTERY_PENDING のみ Service 層で制御）
INDEX idx_papp_space_status (space_id, status, created_at)         -- 区画別の申請一覧
INDEX idx_papp_user (user_id, status)                              -- ユーザーの申請一覧
INDEX idx_papp_lottery (space_id, status, lottery_number)           -- 抽選処理用
```

**制約・備考**
- `uq_papp_space_user_source`: Service 層で `status IN ('PENDING', 'LOTTERY_PENDING')` の場合のみ重複チェック。APPROVED / REJECTED / CANCELLED の過去申請がある場合は再申請可能
- `source_type = 'VACANCY'`: 通常の空き区画申請。`listing_id = NULL`
- `source_type = 'LISTING'`: 譲渡希望への申し込み。`listing_id` で対象の掲示を特定。区画の `allocation_method` は使用しない（ADMIN が申し込み一覧から選択して譲渡確定）
- 申請時の割り当て方式は区画の `parking_spaces.allocation_method` を参照する（申請レコードにはコピーしない）
- `status` 遷移:
  - FIRST_COME: `PENDING` → `APPROVED`（ADMIN 承認）/ `REJECTED`（ADMIN 拒否）/ `CANCELLED`（申請者取消）
  - LOTTERY: `LOTTERY_PENDING`（抽選受付中）→ `APPROVED`（当選）/ `REJECTED`（落選）/ `CANCELLED`（申請者取消）
- `lottery_number`: 抽選実行時に `SecureRandom` で1〜10000の範囲の乱数を生成し、`priority` を加味した上で最小番号の申請者を当選とする。同一番号の場合は `created_at` が早い方を優先
- APPROVED 時の処理: (1) `parking_assignments` に INSERT → (2) `parking_spaces.status` = `OCCUPIED` → (3) 同一区画の他の PENDING / LOTTERY_PENDING 申請を一括 `REJECTED` → (4) F04.3 通知
- CANCELLED: 申請者自身が取り消し。PENDING / LOTTERY_PENDING の場合のみ可能

#### `parking_listings`

区画の譲渡希望リスト。現在区画を使用中のユーザーが「この区画を譲りたい」と掲示し、他メンバーが申し込む。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|

```sql

```

**制約・備考**
- 譲渡希望の掲示は現在の区画利用者（`parking_assignments.user_id = currentUser.id` かつ `released_at IS NULL`）のみ可能。Service 層で検証
- `status` 遷移: `OPEN`（掲示中）→ `RESERVED`（ADMIN が譲受人を仮決定）→ `TRANSFERRED`（ADMIN が譲渡完了を確定）。`OPEN` / `RESERVED` → `CANCELLED`（掲示者が取消 or ADMIN が取消）
- `TRANSFERRED` 時の処理:
  1. 旧割り当て（`assignment_id`）の `released_at` = `NOW()`、`release_reason` = `'譲渡'` に更新
  2. `parking_spaces.status` = `OCCUPIED` のまま維持
  3. 新規 `parking_assignments` を INSERT（`vehicle_id` = `transferee_vehicle_id`）
  4. 旧利用者・新利用者の双方にプッシュ通知（F04.3）
- `listed_by` が退会した場合: ON DELETE CASCADE で掲示が物理削除。割り当て自体は別途管理者が対処
- ADMIN は譲渡希望を介さず直接の割り当て変更も可能（通常の割り当て解除 + 新規割り当てで対応）

#### `parking_visitor_reservations`

| `reserved_by` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE。予約したメンバー |
| `visitor_name` | VARCHAR(100) | YES | NULL | 来訪者名（任意。管理用メモ） |
| `visitor_plate_number` | VARCHAR(30) | YES | NULL | 来訪者の車両ナンバー（任意。プレーンテキスト保存。一時利用のため暗号化不要。例: '大阪 300 あ 1234'） |

| `time_from` | TIME | NO | — | 利用開始時刻 |

| `approved_by` | BIGINT UNSIGNED | YES | NULL | FK → users。ON DELETE SET NULL。承認した管理者 |
| `approved_at` | DATETIME | YES | NULL | 承認日時（CONFIRMED 時に設定） |

| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_pvr_space_date (space_id, reserved_date, time_from, time_to)  -- 区画別の予約一覧・重複チェック
INDEX idx_pvr_user (reserved_by, reserved_date)                          -- ユーザーの予約一覧
INDEX idx_pvr_date_status (reserved_date, status)                        -- 日別の予約管理・NO_SHOW検出バッチ
```

**制約・備考**
- `space_type = VISITOR` の区画のみ予約可能（Service 層で検証）。通常区画（INDOOR/OUTDOOR等）は月極割り当て管理のみ
- `time_from` / `time_to` は **30分単位** で指定（例: 09:00, 09:30, 10:00, ...）。30分刻みでない値は 400 エラー（Service 層で `minutes % 30 == 0` を検証）
- 最小予約時間: 30分（`time_from` + 30分 <= `time_to`）
- 時間帯の重複チェック: 同一 `space_id` + `reserved_date` で時間帯が重複する **PENDING_APPROVAL / CONFIRMED / CHECKED_IN** の予約が存在する場合は 409（仮予約でも枠を確保する）
- `visitor_plate_number` はプレーンテキスト保存。来場者は一時利用であり、居住者の `registered_vehicles.plate_number`（暗号化）とは異なる扱い
- `status` 遷移: `PENDING_APPROVAL`（MEMBER が申請）→ `CONFIRMED`（ADMIN が承認）/ `REJECTED`（ADMIN が却下）/ `CANCELLED`（申請者がキャンセル）。`CONFIRMED` → `CHECKED_IN`（来訪者到着。予約者本人 or ADMIN が操作）/ `CANCELLED`（キャンセル）/ `NO_SHOW`（翌日バッチ）。`CHECKED_IN` → `COMPLETED`（利用終了。予約者本人 or ADMIN が操作）
- 1ユーザーあたり同日の予約上限: `parking_settings.max_visitor_reservations_per_day` で制御（デフォルト: 2）
- 予約は `parking_settings.visitor_reservation_max_days_ahead` 日先まで可能（デフォルト: 30）
- NO_SHOW 検出: 翌日0時のバッチで `reserved_date < TODAY AND status = 'CONFIRMED'` を `NO_SHOW` に自動更新（PENDING_APPROVAL のまま期日を過ぎた予約は `CANCELLED` に自動更新）
- ADMIN は `auto_approve: true` で直接 `CONFIRMED` で作成可能。また `parking_settings.visitor_reservation_requires_approval = FALSE` の場合は全ユーザーの予約が自動 CONFIRMED

#### `parking_subleases`

| カラム名 | 型 | NULL | デフォルト | 説明 |

| `price_per_month` | DECIMAL(10,0) | NO | — | 月額貸し出し料金（円） |
| `payment_method` | ENUM('DIRECT', 'STRIPE') | NO | 'DIRECT' | 決済方法（DIRECT=個人間直接やり取り / STRIPE=Stripe 経由の自動引き落とし） |
| `available_from` | DATE | NO | — | 貸し出し開始可能日 |

| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**インデックス**
```sql
INDEX idx_psl_scope (status, created_at DESC)                     -- サブリース掲示一覧
INDEX idx_psl_offered_by (offered_by, status)                     -- 自分の掲示一覧

```

**制約・備考**
- サブリースは区画の「使用権の一時貸し出し」。`parking_assignments` の権利保有者は変わらない（管理組合上の名義は元の割り当て者のまま）
- 成立後、借り手が実際に利用する際は ADMIN が別途 `parking_assignments` を作成するか、サブリース専用の利用記録で管理（提案1: 既存の assignments を共用、提案2: sublease_assignments を新設。本設計では提案1を採用し、`release_reason = 'サブリース終了'` で区別）
- `payment_method` の選択:
  - `DIRECT`: 個人間で直接やり取り（現金手渡し、銀行振込等）。プラットフォームは金額表示のみで決済に関与しない
  - `STRIPE`: Stripe Connect 経由の自動月額引き落とし。借り手のクレジットカードから提供者の口座に自動送金。**Stripe 決済手数料（3.6% + ¥40/件）+ Mannschaft プラットフォーム手数料（F08.2 共通率）が発生**するため、掲示作成時に手数料内訳と受取見込額を表示
- `payment_method = STRIPE` の場合、提供者は Stripe Connect アカウント（Standard）の作成・連携が必要。未連携の場合は `payment_method = STRIPE` での掲示は 422 エラー
- 掲示者は `payment_method` を自由に選択可能。手数料を避けたい場合は `DIRECT` を選択
- `available_from <= available_to` は Service 層でバリデーション
- 掲示は区画の現在の利用者（`parking_assignments.user_id = currentUser.id`）のみ可能
- 譲渡（`parking_listings`）とサブリースの違い: 譲渡 = 恒久的な権利移転（ADMIN 承認で割り当て変更）、サブリース = 一時的な貸し出し（期間終了後に権利が戻る）

#### `parking_sublease_payments`

サブリース（Stripe 決済）の月額支払い記録。`payment_method = STRIPE` のサブリースのみ対象。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `sublease_id` | BIGINT UNSIGNED | NO | — | FK → parking_subleases。ON DELETE RESTRICT |
| `payer_user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT。借り手（支払い者） |
| `payee_user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE RESTRICT。提供者（受取人） |
| `amount` | DECIMAL(10,0) | NO | — | 借り手の支払い金額（円）= `parking_subleases.price_per_month` |
| `stripe_fee` | DECIMAL(10,0) | NO | — | Stripe 決済手数料（円）。`amount * 3.6% + 40` で自動計算 |
| `platform_fee` | DECIMAL(10,0) | NO | — | Mannschaft プラットフォーム手数料（円）。F08.2 と共通の手数料率で計算 |
| `platform_fee_rate` | DECIMAL(5,4) | NO | — | 適用された手数料率（例: 0.0500 = 5%）。請求時点のレートを記録（遡及変更防止） |
| `net_amount` | DECIMAL(10,0) | NO | — | 提供者の受取金額（`amount - stripe_fee - platform_fee`） |
| `billing_month` | CHAR(7) | NO | — | 対象月（例: '2026-05'） |
| `stripe_payment_intent_id` | VARCHAR(100) | YES | NULL | Stripe PaymentIntent ID（Stripe 決済の場合） |
| `stripe_transfer_id` | VARCHAR(100) | YES | NULL | Stripe Transfer ID（提供者への送金記録） |
| `status` | ENUM('PENDING', 'SUCCEEDED', 'FAILED', 'REFUNDED') | NO | 'PENDING' | 決済ステータス |
| `failed_reason` | VARCHAR(500) | YES | NULL | 失敗理由（FAILED 時。Stripe エラーメッセージ） |
| `paid_at` | DATETIME | YES | NULL | 支払い完了日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
INDEX idx_pslp_sublease (sublease_id, billing_month)               -- サブリースごとの支払い履歴
INDEX idx_pslp_payer (payer_user_id, status)                        -- 借り手の支払い一覧
INDEX idx_pslp_payee (payee_user_id, status)                        -- 提供者の受取一覧
UNIQUE KEY uq_pslp_sublease_month (sublease_id, billing_month)     -- 同一月の二重請求防止
INDEX idx_pslp_status (status, created_at)                          -- FAILED の再試行対象検索
```

**制約・備考**
- `payment_method = STRIPE` のサブリースに対してのみ生成。`DIRECT` の場合はレコードを作成しない
- 毎月1日のバッチで、アクティブな Stripe サブリースの当月分 `PENDING` レコードを自動生成 → Stripe API で決済実行
- Stripe Connect の **Destination Charge** パターンを使用。Stripe の `application_fee_amount` パラメータでプラットフォーム手数料を徴収:
  - 借り手が支払う金額: `amount`（= `price_per_month`）
  - Stripe が徴収: `stripe_fee`（3.6% + ¥40）
  - Mannschaft が徴収: `platform_fee`（`amount * platform_fee_rate`）
  - 提供者が受取: `net_amount` = `amount - stripe_fee - platform_fee`
- `platform_fee_rate` は F08.2 の決済設定（`payment_settings.platform_fee_rate`）から取得。チーム/組織の会費決済と**同一の手数料率**を適用し、決済種別による料率差を設けない。請求時点のレートを `platform_fee_rate` カラムに記録して変更の遡及を防止
- Stripe の `application_fee_amount` に `platform_fee` を設定することで、Mannschaft の Stripe アカウントにプラットフォーム手数料が自動入金される
- `FAILED` の場合: 3日後・7日後・14日後に自動リトライ（最大3回）。3回失敗で ADMIN + 双方に通知
- `REFUNDED`: サブリース途中解約時、日割り計算で未使用分を返金（`Stripe Refund API` 使用）
- 提供者の Stripe Connect アカウント情報は F08.2 の `stripe_customers` テーブルに `stripe_connect_account_id` として管理（本機能では参照のみ）

#### `stripe_connect_accounts`

Stripe Connect アカウント連携情報。サブリースの受取人として登録するユーザーの Stripe Connect (Standard) アカウントを管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `user_id` | BIGINT UNSIGNED | NO | — | FK → users。ON DELETE CASCADE |
| `stripe_account_id` | VARCHAR(100) | NO | — | Stripe Connect アカウント ID（例: 'acct_1234567890'） |
| `charges_enabled` | BOOLEAN | NO | FALSE | Stripe 側で決済受付が有効か |
| `payouts_enabled` | BOOLEAN | NO | FALSE | Stripe 側で出金が有効か |
| `onboarding_completed` | BOOLEAN | NO | FALSE | Stripe オンボーディング完了フラグ |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_sca_user (user_id)                    -- 1ユーザー1アカウント
UNIQUE KEY uq_sca_stripe (stripe_account_id)        -- Stripe アカウント ID の一意性
```

**制約・備考**
- Stripe Connect Standard アカウントを使用（Express / Custom ではない。ユーザーが Stripe ダッシュボードで自分の売上・出金を管理できるため）
- オンボーディングフロー: (1) `POST /stripe-connect/onboarding` で Stripe Account Link を生成 → (2) ユーザーが Stripe のオンボーディング画面で本人確認・銀行口座登録 → (3) Webhook で `account.updated` を受信 → `charges_enabled` / `payouts_enabled` / `onboarding_completed` を更新
- `onboarding_completed = FALSE` かつ `charges_enabled = FALSE` の場合、`payment_method = STRIPE` でのサブリース掲示は不可（422 エラー）
- Stripe Connect アカウントの作成は任意。サブリースで Stripe 決済を使いたい提供者のみが作成する

| 403 | 操作権限なし（CONFIRMED + 24h以内に MEMBER がキャンセル試行等） |

## 5. ビジネスロジック

### 区画ステータス遷移

```
                    ┌─────────────┐
          ┌────────→│   VACANT    │←───────────┐
          │         └──────┬──────┘            │
          │                │                   │
          │      assign / approve              │
          │                │                 release
          │                ▼                   │
          │         ┌─────────────┐            │
          │         │  OCCUPIED   │────────────┘
          │         └──────┬──────┘
          │                │ ※ OCCUPIED → MAINTENANCE は不可
          │                │   先に release が必要
          │                │
    maintenance=false      │ ※ VACANT からのみ遷移可
          │                │
          │         ┌──────▼──────┐
          └─────────│ MAINTENANCE │
                    └─────────────┘
```

**遷移ルール**
- `VACANT` → `OCCUPIED`: 割り当て実行時に自動遷移。`parking_assignments` INSERT + ステータス更新をトランザクション内で実行
- `OCCUPIED` → `VACANT`: 割り当て解除時に自動遷移。`parking_assignments.released_at` 更新 + ステータス更新をトランザクション内で実行
- `VACANT` → `MAINTENANCE`: ADMIN が手動設定。割り当て中の区画は先に解除が必要
- `MAINTENANCE` → `VACANT`: ADMIN が手動復旧
- `OCCUPIED` → `MAINTENANCE` は直接遷移不可（既存の利用者への影響を防止。先に解除 → VACANT → MAINTENANCE のステップを踏む）
- VISITOR 区画の空き判定: `parking_spaces.status` は常に `VACANT` のまま維持。時間帯ごとの予約有無は `parking_visitor_reservations` テーブルの重複チェッククエリで動的に判定する（区画レベルの単一ステータスでは時間帯別の空き管理ができないため）

### 申請・割り当てフロー（FIRST_COME）

```
1. ADMIN が PATCH /spaces/{id}/accept-applications で申請受付を開始（allocation_method = FIRST_COME）
2. MEMBER が POST /parking/applications で申請
   ├─ 区画の allocation_method = FIRST_COME を参照
   └─ status = PENDING
3. ADMIN が申請一覧を確認
4. ADMIN が PATCH /applications/{id}/approve で承認
   ├─ parking_assignments に INSERT
   ├─ parking_spaces.status → OCCUPIED
   ├─ 同一区画の他の PENDING 申請 → 一括 REJECTED
   ├─ 承認者にプッシュ通知（PARKING_APPLICATION_APPROVED）
   └─ 拒否された申請者にプッシュ通知（PARKING_APPLICATION_REJECTED）
```

### 申請・割り当てフロー（LOTTERY）

```
1. ADMIN が PATCH /spaces/{id}/accept-applications で抽選受付を開始（allocation_method = LOTTERY、application_deadline 設定）
2. MEMBER が申請（status = LOTTERY_PENDING）
3. application_deadline 超過で application_status → LOTTERY_CLOSED（バッチ or ADMIN 手動）。ADMIN が POST /applications/lottery で抽選実行
   ├─ 各申請に SecureRandom で lottery_number を付与
   ├─ priority DESC → lottery_number ASC → created_at ASC で当選者決定
   ├─ 当選者: APPROVED + 割り当て実行
   ├─ 落選者: REJECTED（rejection_reason = "抽選の結果、落選しました"）
   └─ 全申請者にプッシュ通知
```

### 抽選ロジック詳細

```sql
-- 抽選番号の付与後、当選者の決定
SELECT id, user_id, vehicle_id, priority, lottery_number, created_at
FROM parking_applications
WHERE space_id = :spaceId
  AND status = 'LOTTERY_PENDING'
ORDER BY priority DESC, lottery_number ASC, created_at ASC
LIMIT 1
```

`priority` を加味することで、身障者用区画への優先割り当て等を実現。通常は全員 `priority = 0` で純粋なランダム抽選となる。

### 譲渡フロー

```
1. 現在の利用者が POST /parking/listings で譲渡希望を掲示（status: OPEN）
2. 他の MEMBER が POST /listings/{id}/apply で申し込み
3. ADMIN が申し込み一覧を確認し、譲受人を決定
4. ADMIN が PATCH /listings/{id}/transfer で譲渡確定
   ├─ 旧割り当ての released_at を設定（release_reason = '譲渡'）
   ├─ 新規 parking_assignments を INSERT
   ├─ parking_listings.status → TRANSFERRED
   ├─ 旧利用者にプッシュ通知（PARKING_LISTING_TRANSFERRED）
   └─ 新利用者にプッシュ通知（PARKING_ASSIGNMENT_CREATED）
```

```java
) {}
```

> **設計判断**: `price_per_month` 変更時に既存割り当ての料金は変更しない。新規割り当てから新料金が適用される。料金変更の遡及適用が必要な場合は F08.2 の決済設計で対応する。

### 車両・区画種別の互換性チェック

割り当て・申請時に車両種別（`vehicle_type`）と区画種別（`space_type`）の互換性をソフトバリデーションする。非互換の場合はレスポンスに `warning` を含めるが、割り当て自体はブロックしない（`force: true` で警告スキップ可能）。

```
互換性マッピング（ParkingProperties で定数管理）:
  CAR        → INDOOR, OUTDOOR, ACCESSIBLE, OTHER        = OK
  CAR        → MOTORCYCLE, BICYCLE                        = WARN
  MOTORCYCLE → INDOOR, OUTDOOR, MOTORCYCLE, OTHER         = OK
  MOTORCYCLE → ACCESSIBLE, BICYCLE                        = WARN
  BICYCLE    → BICYCLE, OUTDOOR, OTHER                    = OK
  BICYCLE    → INDOOR, ACCESSIBLE, MOTORCYCLE             = WARN
```

- Service 層で互換性チェック → 非互換の場合はレスポンスに `"warning": "バイク専用区画にCARを割り当てようとしています"` を含める
- ADMIN が `force: true` をリクエストボディに含めれば警告スキップ可能
- `force: true` の操作は監査ログ（F10.3）に記録

### 来場者用区画の予約フロー（承認制・30分単位）

```
1. ADMIN が VISITOR 区画を作成（space_type = VISITOR）
   ※ VISITOR 区画は月極割り当て対象外（assign/application/listing の操作不可）
   ※ VISITOR 区画の status は常に VACANT（空き判定は予約テーブルで動的に行う）
2. MEMBER が POST /visitor-reservations で予約申請（30分単位で時間指定）
   ├─ 時間帯重複チェック（PENDING_APPROVAL / CONFIRMED / CHECKED_IN と重複不可）
   ├─ 30分単位バリデーション（time_from / time_to の分が 00 or 30）
   ├─ 同日予約上限チェック（1ユーザー2件/日）
   ├─ 予約可能期間チェック（最大30日先まで）
   └─ status = PENDING_APPROVAL（仮予約。枠は確保される）
3. ADMIN が予約一覧で仮予約を確認
4. ADMIN が PATCH /approve で承認 → status = CONFIRMED（確定）
   └─ または PATCH /reject で却下 → status = REJECTED（枠解放）
5. 来訪者到着 → PATCH /check-in（CONFIRMED → CHECKED_IN。予約者本人 or ADMIN）
6. 利用終了 → PATCH /complete（CHECKED_IN → COMPLETED。予約者本人 or ADMIN）
7. 翌日バッチ:
   ├─ CONFIRMED のまま予約日を過ぎた予約 → NO_SHOW
   └─ PENDING_APPROVAL のまま予約日を過ぎた予約 → CANCELLED（自動失効）
```

> **設計判断**: 仮予約（PENDING_APPROVAL）の時点で時間枠を確保する（重複チェック対象に含める）。これにより「申請して承認を待っている間に他の人に取られる」ことを防止。ADMIN が却下すれば枠は即座に解放される。ADMIN は `auto_approve: true` で直接確定も可能（緊急時や自身の来客用）。

> **30分単位の設計判断**: 5分/10分/15分単位は選択肢が多すぎて UX が悪く、1時間単位は粒度が粗い。30分単位は「9:00-12:00」「13:30-15:00」等の一般的な利用パターンに合致し、管理画面のタイムライン表示にも適した粒度。

> **設計判断**: VISITOR 区画は通常の月極割り当て（assign/application/listing）とは完全に分離する。VISITOR 区画に対する assign / application / listing API は 422 を返す。これにより、来場者予約と月極管理のロジックが混在しない。

### ウォッチリスト通知

割り当て解除時（`ParkingAssignmentReleasedEvent`）に、空いた区画の条件と合致するウォッチリストを検索して通知:

```sql
SELECT pw.id, pw.user_id
FROM parking_watchlist pw
WHERE pw.scope_type = :scopeType
  AND pw.scope_id = :scopeId
  AND pw.is_active = TRUE
  AND pw.user_id != :releasedUserId
  AND (pw.space_type IS NULL OR pw.space_type = :spaceType)
  AND (pw.floor IS NULL OR pw.floor = :floor)
  AND (pw.max_price IS NULL OR pw.max_price >= :pricePerMonth)
```

合致した `user_id` に `PARKING_WATCHLIST_MATCHED` 通知を送信後、`is_active = FALSE` に更新。

### 区画交換（スワップ）

```
1. ADMIN が POST /spaces/swap で交換リクエスト
2. 両方の parking_assignments を FOR UPDATE でロック（ID 昇順でデッドロック防止）
3. 旧割り当て2件: released_at = NOW(), release_reason = '区画交換'
4. 新割り当て2件: vehicle_id を入れ替えて INSERT
5. parking_spaces.status は変更なし（OCCUPIED → OCCUPIED）
6. 双方の利用者にプッシュ通知
```

### サブリースフロー

```
1. 区画所有者（車未使用）が POST /subleases で貸し出し掲示（status: OPEN）
2. 他の MEMBER が POST /subleases/{id}/apply で申し込み
3. 提供者が申し込み一覧を確認し、希望の借り手を選定
4. ADMIN が PATCH /subleases/{id}/approve で最終承認（管理規約チェック）
   ├─ parking_sublease_applications.status → APPROVED
   ├─ 他の PENDING 申し込み → 一括 REJECTED
   ├─ parking_subleases.status → MATCHED
   ├─ 借り手の車両で parking_assignments を新規作成
   └─ 双方にプッシュ通知
5. 【Stripe の場合】毎月1日のバッチで自動請求:
   ├─ parking_sublease_payments に PENDING レコード作成
   ├─ Stripe Destination Charge で借り手のカードから決済
   ├─ Stripe Transfer で提供者の Connect アカウントに送金
   ├─ 成功 → SUCCEEDED / 失敗 → FAILED（3日後・7日後・14日後にリトライ）
   └─ 3回連続失敗 → ADMIN + 双方に通知
6. サブリース期間終了時:
   ├─ 提供者 or ADMIN が PATCH /subleases/{id}/terminate
   ├─ 借り手の割り当てを解除（released_at 設定）
   ├─ 【Stripe の場合】Subscription キャンセル + 未使用分の日割り返金
   └─ 所有者の割り当ては維持
```

> **設計判断**: サブリース中は1区画に2件のアクティブ `parking_assignments` が存在する（所有者 + 借り手）。`uq_pa_space_active (space_id, released_at)` の UNIQUE KEY 制約により通常は1件だが、サブリース割り当ては `assigned_by` カラムの参照先や `release_reason` で区別する。実装時に `is_sublease BOOLEAN DEFAULT FALSE` カラムを `parking_assignments` に追加して明示的に区別することを推奨。

> **決済方法の選択**: 提供者はサブリース掲示作成時に `payment_method` を選択する。`DIRECT`（手数料なし・個人間で直接やり取り）と `STRIPE`（手数料あり・自動引き落としで手間なし）を任意に選べる。STRIPE 選択時はレスポンスに手数料内訳（Stripe 手数料 + Mannschaft プラットフォーム手数料）と受取見込額を事前提示し、提供者が判断できるようにする。

> **手数料構造（STRIPE 選択時）**:
> ```
> 借り手の支払い: 10,000円（= price_per_month）
>   ├─ Stripe 決済手数料:         400円（3.6% + ¥40）→ Stripe が徴収
>   ├─ Mannschaft プラットフォーム手数料: 500円（5%想定）→ application_fee_amount で Mannschaft に入金
>   └─ 提供者の受取:             9,100円 → Connect アカウント経由で銀行口座に出金
> ```
> プラットフォーム手数料率は F08.2 の `payment_settings.platform_fee_rate` を参照し、チーム/組織の会費決済と同率を適用する。これにより決済種別による料率差がなく、料金体系がシンプルに保たれる。

> **Stripe Connect アーキテクチャ**: Destination Charge パターンを採用。プラットフォームが PaymentIntent を作成し、`transfer_data.destination` に提供者の Connect アカウント、`application_fee_amount` に Mannschaft プラットフォーム手数料を指定。Stripe が決済手数料を差し引き、プラットフォーム手数料を Mannschaft に入金した残りが提供者の Connect アカウントに送金 → 提供者の銀行口座に自動出金（Stripe のデフォルト出金スケジュールに従う）。

### 契約期間管理

毎日0時のバッチで、契約終了30日前の割り当てを検出してリマインダー通知:

```sql
SELECT pa.id, pa.user_id, pa.space_id, pa.contract_end_date
FROM parking_assignments pa
WHERE pa.released_at IS NULL
  AND pa.contract_end_date IS NOT NULL
  AND pa.contract_end_date = DATE_ADD(CURDATE(), INTERVAL 30 DAY)
```

通知内容: 「区画 {space_number} の契約が {contract_end_date} に満了します。更新については管理者にお問い合わせください。」

## 6. セキュリティ考慮事項

- **ナンバープレートの暗号化**: `plate_number` は AES-256-GCM で暗号化して保存。暗号化キーは AWS KMS で管理。平文ナンバーは DB に一切保存しない。API レスポンスでは常にマスキング表示
- **ナンバープレートのハッシュ**: 一意性チェックには SHA-256 ハッシュ（ソルト付き）を使用。暗号化カラムに対する UNIQUE 制約の代替
- **認可チェック**: 区画管理操作（CRUD・割り当て・申請承認）は ADMIN / DEPUTY_ADMIN（`MANAGE_PARKING`）限定。Service 層入り口でスコープメンバーシップ + ロール + パーミッションを検証
- **スコープ分離**: 全 API で `teamId` パスパラメータとリクエストユーザーのスコープメンバーシップを検証。他チームの区画情報へのアクセスを防止
- **車両所有権チェック**: 車両の更新・削除は所有者本人のみ。申請時の `vehicle_id` が `currentUser` の車両であることを検証
- **プライバシー保護（割り当て情報）**: MEMBER / SUPPORTER には区画の `current_assignment`（誰が利用しているか）を非表示。ADMIN / DEPUTY_ADMIN のみ閲覧可能
- **レート制限**: 車両登録 API は `Bucket4j` で 1ユーザーあたり1分間に5回。申請 API は 1ユーザーあたり1分間に10回。抽選実行 API は 1区画あたり1分間に1回
- **一括割り当てのサイズ制限**: `bulk-assign` は最大20ペア。DoS 防止
- **暗号化キーローテーション**: KMS キーのローテーション時は、全 `plate_number` を再暗号化するバッチジョブを実行。`plate_number_hash` は変更不要（ハッシュはキー非依存）
- **VISITOR 区画の操作分離**: `space_type = VISITOR` の区画に対する assign / application / listing API は 422 を返す。来場者予約 API のみ使用可能。逆に、VISITOR 以外の区画に対する visitor-reservations API も 422。これにより月極管理と来場者予約のロジック混在を防止
- **来場者予約の閲覧制御**: ADMIN は全予約を閲覧可能。MEMBER は自分の予約のみ閲覧（他メンバーの来客情報はプライバシー保護のため非公開）
- **監査ログ**: 割り当て操作（`PARKING_ASSIGNED`、`PARKING_RELEASED`）、一括割り当て（`PARKING_BULK_ASSIGNED`）、抽選実行（`PARKING_LOTTERY_EXECUTED`）、譲渡完了（`PARKING_TRANSFERRED`）を `audit_logs` に記録
