# F09.11: 広告主ダッシュボード

> **ステータス**: 🟢 設計完了
> **実装フェーズ**: Phase 10（F09.7 §10 の広告プラットフォーム拡張と同時実装。Phase 9 番号は広告機能系列として付与）
> **最終更新**: 2026-03-28
> **モジュール種別**: デフォルト機能（広告主向け。組織 ADMIN がアクセス）
> **関連ドキュメント**: F09.7 広告表示（広告配信基盤・クリエイティブ管理）、F09.2 プロモーション配信（SegmentEvaluator 共有）、F08.2 会費・決済（Stripe Connect 決済基盤）、F10.4 経営分析ダッシュボード（プラットフォーム側の広告収益分析）

---

## 1. 概要

広告主（組織 ADMIN）向けのセルフサービスダッシュボードを提供する。キャンペーンのパフォーマンス確認・料金シミュレーション・請求明細の閲覧を一箇所で完結させ、広告出稿の意思決定を支援する。

F09.7 で設計済みの広告プラットフォーム拡張（`ads` / `ad_campaigns` / `ad_targeting_rules` / `ad_impressions` / `ad_clicks` / `ad_daily_stats` の6テーブル）を前提とし、その上に広告主向けのレポーティング・シミュレーション・請求機能を追加する。

### 本ドキュメントのスコープ

| 機能 | 説明 |
|------|------|
| キャンペーンパフォーマンス | インプレッション・クリック・CTR・費用の推移をキャンペーン/クリエイティブ別に表示 |
| 地域×テンプレート別レポート | ターゲティング条件ごとの配信実績と費用内訳 |
| 料金シミュレーター | 地域・テンプレート・表示回数・課金モデルを入力して概算費用を算出 |
| 請求明細 | 月ごとの広告費用サマリと明細 |
| クリエイティブ比較 | 同一キャンペーン内のクリエイティブ別パフォーマンス比較（A/B テスト支援） |
| 予算ペーシング＆消化予測 | 現在の消化ペースに基づく予算枯渇日・推奨日次配分の表示 |
| ベンチマーク比較 | 自社 CTR とプラットフォーム平均・同一テンプレート平均の比較 |
| 最適化レコメンデーション | breakdown データに基づく地域・クリエイティブ・トレンドの自動改善提案 |
| パフォーマンスエクスポート＆定期レポート | CSV エクスポート + 週次/月次レポートのメール自動配信 |
| コンバージョントラッキング | 広告クリック後のチーム加入・組織加入等の成果追跡 |
| credit_limit 増額申請 | 広告主からのセルフサービス増額リクエスト + SYSTEM_ADMIN 承認 |
| 広告主オンボーディング | 組織 ADMIN が広告主として利用開始する登録フロー。承認前でもキャンペーン下書き作成可能 |

### F09.7 との関係

- F09.7 §10 で定義された将来拡張テーブル（6テーブル）は本機能の前提。DB 定義は F09.7 側で管理し、本ドキュメントでは参照のみ
- F09.7 が広告の「配信・表示・計測」を担い、本機能は「分析・シミュレーション・請求」を担う
- 広告クリエイティブの入稿・審査フローは F09.7 の拡張として設計（本ドキュメントのスコープ外）
- 本ドキュメントの API パスは `/api/v1/advertiser/**`（広告主専用プレフィックス）

### F10.4 との関係

- F10.4 の `analytics_daily_ads` は**プラットフォーム全体**の広告収益集計（SYSTEM_ADMIN 向け）
- 本機能の `ad_daily_stats` は**キャンペーン/クリエイティブ単位**の詳細集計（広告主向け）
- 両者は同じ `ad_impressions` / `ad_clicks` を元に異なる粒度で集計する

### テーブル定義

#### `advertiser_accounts`

組織が広告主として利用開始する際に作成される。組織と 1:1 の関係。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `organization_id` | BIGINT UNSIGNED | NO | — | organizations.id（1:1） |
| `status` | ENUM('PENDING', 'ACTIVE', 'SUSPENDED') | NO | 'PENDING' | アカウント状態 |
| `company_name` | VARCHAR(200) | NO | — | 広告主としての表示名（組織名と異なる場合あり） |
| `contact_email` | VARCHAR(254) | NO | — | 請求・連絡先メールアドレス |
| `billing_method` | ENUM('STRIPE', 'INVOICE') | NO | 'STRIPE' | 決済方法（Stripe 自動課金 or 請求書払い） |
| `stripe_customer_id` | VARCHAR(50) | YES | NULL | Stripe Customer ID（billing_method=STRIPE の場合） |
| `credit_limit` | DECIMAL(12,2) | NO | 100000 | 月間広告費上限（円）。上限到達で配信自動停止。0 は不可（CHECK 制約） |
| `approved_by` | BIGINT UNSIGNED | YES | NULL | 承認した SYSTEM_ADMIN の user_id |
| `approved_at` | DATETIME | YES | NULL | 承認日時 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**UNIQUE KEY**: `(organization_id)` WHERE `deleted_at IS NULL`（アプリ層で保証。§3 備考参照）
**INDEX**: `idx_status (status, deleted_at)` — ステータス別の検索用
**FK**: `organization_id → organizations(id)`, `approved_by → users(id)`
**CHECK**: `CHECK (credit_limit > 0)` — 0 以下は不可（0 = 即停止ではなく、1円以上の設定を強制）

**備考**:
- `organization_id` の一意性は `deleted_at` を含むため DB レベルの UNIQUE KEY は使用しない（F09.7 `affiliate_configs` と同一パターン）。アプリ層で `SELECT ... WHERE organization_id = ? AND deleted_at IS NULL` でチェック
- `status` の遷移: `PENDING` → `ACTIVE`（SYSTEM_ADMIN 承認）→ `SUSPENDED`（支払い遅延 or 規約違反）→ `ACTIVE`（復帰）。`PENDING` → `ACTIVE` のみ SYSTEM_ADMIN 操作、他は自動 or SYSTEM_ADMIN 操作
- `credit_limit`: デフォルト 100,000円。SYSTEM_ADMIN が個別に引き上げ可能。月間広告費がこの上限に到達した時点でキャンペーンを自動停止し、広告主に通知

#### `ad_rate_cards`

地域×テンプレート×課金モデル別の広告単価。SYSTEM_ADMIN が管理する料金マスタ。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `target_prefecture` | VARCHAR(20) | YES | NULL | 都道府県（NULL = 全国一律） |
| `target_template` | VARCHAR(30) | YES | NULL | チームテンプレート（NULL = 全テンプレート共通） |
| `pricing_model` | ENUM('CPM', 'CPC') | NO | — | 課金モデル |
| `unit_price` | DECIMAL(10,4) | NO | — | 単価（CPM: 1000インプレッションあたり円、CPC: 1クリックあたり円） |
| `min_daily_budget` | DECIMAL(10,2) | NO | 500 | 最低日予算（円） |
| `effective_from` | DATE | NO | — | 適用開始日 |
| `effective_until` | DATE | YES | NULL | 適用終了日（NULL = 無期限） |
| `created_by` | BIGINT UNSIGNED | NO | — | 作成者の user_id（SYSTEM_ADMIN） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | |

**一意性**: `(target_prefecture, target_template, pricing_model, effective_from)` の組み合わせはアプリ層で保証（NULL を含むため DB UNIQUE KEY は使用しない。F09.7 `affiliate_configs` と同一パターン）
**INDEX**: `idx_effective (effective_from, effective_until)` — 有効な料金の検索用
**FK**: `created_by → users(id)`
**CHECK**: `CHECK (unit_price > 0)`, `CHECK (min_daily_budget >= 100)`

**備考**:
- `target_prefecture = NULL AND target_template = NULL` がデフォルト料金（全国・全テンプレート共通）。より具体的な条件がある場合はそちらが優先（マッチング優先順位: 都道府県+テンプレート > 都道府県のみ > テンプレートのみ > 全国共通）
- 料金改定時は新しい `effective_from` のレコードを追加する（既存レコードは変更しない）。過去の料金で発生した請求の整合性を保証
- 単価の例: 東京都×スポーツチーム×CPM = ¥50/1000imp、全国共通×CPC = ¥30/click

#### `ad_invoices`

広告費用の月次請求書。月次バッチで自動生成。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `advertiser_account_id` | BIGINT UNSIGNED | NO | — | advertiser_accounts.id |
| `invoice_number` | VARCHAR(20) | NO | — | 請求書番号（`INV-{YYYYMM}-{連番5桁}`。PDF ヘッダーに表示） |
| `invoice_month` | DATE | NO | — | 請求対象月（月初日） |
| `total_amount` | DECIMAL(12,2) | NO | 0 | 請求合計額（税抜） |
| `tax_rate` | DECIMAL(5,2) | NO | 10.00 | 適用消費税率（%）。請求書発行時点の税率をスナップショット保存 |
| `tax_amount` | DECIMAL(12,2) | NO | 0 | 消費税額（total_amount × tax_rate / 100、端数切り捨て） |
| `total_with_tax` | DECIMAL(12,2) | NO | 0 | 請求合計額（税込） |
| `status` | ENUM('DRAFT', 'ISSUED', 'PAID', 'OVERDUE') | NO | 'DRAFT' | 請求状態 |
| `stripe_invoice_id` | VARCHAR(50) | YES | NULL | Stripe Invoice ID（Stripe 課金の場合） |
| `issued_at` | DATETIME | YES | NULL | 請求書発行日時 |
| `paid_at` | DATETIME | YES | NULL | 支払日時 |
| `due_date` | DATE | YES | NULL | 支払期限（請求書払いの場合） |
| `note` | VARCHAR(500) | YES | NULL | 管理者メモ（入金確認の証跡等） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | |

**UNIQUE KEY**: `(advertiser_account_id, invoice_month)`, `(invoice_number)`
**INDEX**: `idx_status_due (status, due_date)` — OVERDUE バッチの検索用、`idx_account_status (advertiser_account_id, status)` — 広告主別の未払い検索用
**FK**: `advertiser_account_id → advertiser_accounts(id)`
**CHECK**: `CHECK (DAY(invoice_month) = 1)` — 月初日のみ許容

#### `ad_invoice_items`

請求書の明細行。キャンペーン×課金モデル別の内訳。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `invoice_id` | BIGINT UNSIGNED | NO | — | ad_invoices.id |
| `campaign_id` | BIGINT UNSIGNED | NO | — | ad_campaigns.id |
| `campaign_name` | VARCHAR(200) | NO | — | キャンペーン名（スナップショット） |
| `pricing_model` | ENUM('CPM', 'CPC') | NO | — | 課金モデル |
| `impressions` | BIGINT UNSIGNED | NO | 0 | 当月インプレッション数 |
| `clicks` | BIGINT UNSIGNED | NO | 0 | 当月クリック数 |
| `unit_price` | DECIMAL(10,4) | NO | — | 適用された単価 |
| `subtotal` | DECIMAL(12,2) | NO | 0 | 小計（CPM: impressions/1000 × unit_price、CPC: clicks × unit_price） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | |

**INDEX**: `idx_invoice_id (invoice_id)` — 請求書ごとの明細取得用
**FK**: `invoice_id → ad_invoices(id)`, `campaign_id → ad_campaigns(id)`

#### `ad_conversions`

広告クリック後のコンバージョンを追跡する。広告リンクに付与された `tracking_id` をもとに、クリックからコンバージョンまでを紐付ける。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `click_id` | BIGINT UNSIGNED | NO | — | ad_clicks.id |
| `campaign_id` | BIGINT UNSIGNED | NO | — | ad_campaigns.id |
| `ad_id` | BIGINT UNSIGNED | NO | — | ads.id |
| `conversion_type` | ENUM('TEAM_JOIN', 'ORG_JOIN', 'MODULE_ACTIVATE', 'EVENT_REGISTER') | NO | — | コンバージョン種別 |
| `converted_user_id` | BIGINT UNSIGNED | NO | — | コンバージョンしたユーザーの user_id |
| `converted_at` | DATETIME | NO | — | コンバージョン発生日時 |
| `attribution_window_days` | TINYINT UNSIGNED | NO | 7 | アトリビューション期間（クリックから何日以内のアクションを成果とするか） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**UNIQUE KEY**: `(click_id, conversion_type)` — 同一クリック×同一種別の重複コンバージョンを DB レベルで防止（イベント駆動のレース条件対策）
**INDEX**: `idx_campaign_converted (campaign_id, converted_at)` — キャンペーン別コンバージョン集計用、`idx_click_id (click_id)` — クリックとの紐付け用、`idx_ad_converted (ad_id, converted_at)` — クリエイティブ別コンバージョン集計用（conversion_rank 算出）
**FK**: `click_id → ad_clicks(id)`, `campaign_id → ad_campaigns(id)`, `ad_id → ads(id)`, `converted_user_id → users(id)`

**備考**:
- **write-once レコード**: コンバージョンは発生時に1回のみ INSERT し、更新しない設計のため `updated_at` は意図的に省略
- アトリビューションモデル: ラストクリック方式（同一ユーザーが複数広告をクリックした場合、最後のクリックに成果を帰属）
- `attribution_window_days`: デフォルト 7 日。キャンペーン単位で設定可能にするかは将来検討（当面は全キャンペーン共通）
- `converted_user_id` はコンバージョン集計のための内部参照。広告主向け API では返却しない（PII 保護）

#### `ad_report_schedules`

定期レポートのスケジュール設定。広告主が週次/月次のパフォーマンスレポートをメールで受信する設定を管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `advertiser_account_id` | BIGINT UNSIGNED | NO | — | advertiser_accounts.id |
| `frequency` | ENUM('WEEKLY', 'MONTHLY') | NO | — | 配信頻度 |
| `recipients` | JSON | NO | — | 配信先メールアドレス（最大5件） |
| `include_campaigns` | JSON | YES | NULL | 対象キャンペーンID（NULL = 全キャンペーン） |
| `enabled` | BOOLEAN | NO | TRUE | 有効/無効 |
| `last_sent_at` | DATETIME | YES | NULL | 最終配信日時 |
| `created_by` | BIGINT UNSIGNED | NO | — | 作成者の user_id |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除日時 |

**INDEX**: `idx_account_enabled (advertiser_account_id, enabled, deleted_at)` — 有効スケジュールの取得用
**FK**: `advertiser_account_id → advertiser_accounts(id)`, `created_by → users(id)`

#### `ad_credit_limit_requests`

広告主からの credit_limit 増額申請。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `advertiser_account_id` | BIGINT UNSIGNED | NO | — | advertiser_accounts.id |
| `current_limit` | DECIMAL(12,2) | NO | — | 申請時点の credit_limit（スナップショット） |
| `requested_limit` | DECIMAL(12,2) | NO | — | 希望額 |
| `reason` | VARCHAR(500) | NO | — | 申請理由 |
| `status` | ENUM('PENDING', 'APPROVED', 'REJECTED') | NO | 'PENDING' | 申請状態 |
| `reviewed_by` | BIGINT UNSIGNED | YES | NULL | 審査した SYSTEM_ADMIN の user_id |
| `reviewed_at` | DATETIME | YES | NULL | 審査日時 |
| `review_note` | VARCHAR(500) | YES | NULL | 審査メモ（却下理由等） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | |

**INDEX**: `idx_account_status (advertiser_account_id, status)` — 広告主別の未処理申請検索用

### ER図（テキスト表現）

```
organizations (id)
  └─ advertiser_accounts (organization_id) 1:1
       ├─ ad_invoices (advertiser_account_id) 1:N
       │    └─ ad_invoice_items (invoice_id) 1:N → ad_campaigns
       ├─ ad_report_schedules (advertiser_account_id) 1:N
       └─ ad_credit_limit_requests (advertiser_account_id) 1:N

ad_rate_cards (target_prefecture, target_template, pricing_model) — 独立マスタ

ad_conversions (click_id, campaign_id, ad_id) → ad_clicks, ad_campaigns, ads

  ad_campaigns (advertiser_organization_id) → organizations
  ad_daily_stats (campaign_id, ad_id, date) ← パフォーマンス・ベンチマーク・レコメンデーションのデータソース

```

#### `POST /api/v1/advertiser/register`

**認可**: ADMIN（自組織の広告主登録がまだ存在しない場合のみ）

**リクエストボディ**:
```json

  "company_name": "株式会社サンプルスポーツ",
```

**バリデーション**:

- `contact_email`: 有効なメールアドレス形式
- `billing_method`: `STRIPE` / `INVOICE`

**レスポンス（201 Created）**:
```json

    "id": 1,
    "organization_id": 42,
    "status": "PENDING",
    "company_name": "株式会社サンプルスポーツ",
    "contact_email": "ads@sample-sports.co.jp",
    "billing_method": "STRIPE",
    "credit_limit": 100000,

}
```

**承認前の操作可能範囲**: `status = PENDING` の広告主は以下の操作が可能:
- キャンペーンの `DRAFT` 作成（F09.7 のキャンペーン CRUD API）
- クリエイティブの入稿（F09.7 の広告クリエイティブ CRUD API）
- 料金シミュレーターの利用
- 請求書一覧の閲覧（空のリスト）

`PENDING` の広告主が**不可能**な操作:
- キャンペーンの `ACTIVE` 化（`account.status = ACTIVE` が前提条件。F09.7 の `ad_campaigns` ステータス遷移で検証）
- パフォーマンスレポートの閲覧（配信実績がないため）

- `comparison`: 他の地域×テンプレートの料金を最大5件表示（広告主が比較検討できるように）
- 料金テーブルが未設定の条件の場合はデフォルト料金（全国共通）にフォールバック

## 5. ビジネスロジック

### 料金マッチングロジック

```
1. 入力: prefecture, template, pricing_model, date
2. ad_rate_cards を以下の優先順位で検索（effective_from <= date AND (effective_until IS NULL OR effective_until >= date)）:
   a. prefecture = ? AND template = ? AND pricing_model = ?（完全一致）
   b. prefecture = ? AND template IS NULL AND pricing_model = ?（都道府県のみ一致）
   c. prefecture IS NULL AND template = ? AND pricing_model = ?（テンプレートのみ一致）
   d. prefecture IS NULL AND template IS NULL AND pricing_model = ?（デフォルト料金）
3. 最初にマッチした料金を適用
4. 同一優先度で複数マッチした場合は effective_from が最新のものを優先
5. どの条件にもマッチしない場合はエラー（料金未設定。SYSTEM_ADMIN にデフォルト料金の設定を促す）
```

### 月次請求バッチ

```
1. @Scheduled(cron="0 0 5 1 * *", zone="Asia/Tokyo") で毎月1日 AM 5:00 (JST) に実行
2. 排他制御: ShedLock で同時実行を防止
3. 対象: status = ACTIVE の全 advertiser_accounts
4. 各広告主について:
   a. 前月の ad_daily_stats から campaign 別に impressions / clicks を集計
   b. 各 campaign × 各日について、その日時点で有効な ad_rate_cards の単価を料金マッチングロジックで特定（日単位課金。月中の料金改定にも正確に対応）
   c. ad_invoices に DRAFT レコードを INSERT（既存かつ status=DRAFT の場合のみ UPDATE。ISSUED/PAID/OVERDUE は更新しない — レース条件防止）
   d. invoice_number を採番: `INV-{YYYYMM}-{連番5桁}`（月ごとにリセット）。採番は `SELECT MAX(invoice_number) FROM ad_invoices WHERE invoice_month = ? FOR UPDATE` でロック取得後にインクリメント。ShedLock でバッチ自体が排他制御されているため並行採番は発生しない
   e. ad_invoice_items にキャンペーン別明細を INSERT
   f. subtotal 合計 → total_amount、環境変数 AD_TAX_RATE の税率を tax_rate に保存、tax_amount = FLOOR(total_amount × tax_rate / 100)（端数切り捨て）
   g. billing_method に応じた処理:
      - STRIPE: Stripe Invoice API で請求書を作成 → Webhook（invoice.paid / invoice.payment_failed）で結果を非同期受信
        - Stripe API 失敗時: status = DRAFT のまま保持、batch_job_logs に ERROR、SYSTEM_ADMIN に通知。次回手動リトライで対応
      - INVOICE: status = ISSUED、due_date = 翌月末日。メールで請求書 PDF を送付
5. エラー時: batch_job_logs に ERROR + SYSTEM_ADMIN 通知。個別広告主のエラーは他に影響しない
```

### 請求書払い（INVOICE）の支払確認フロー

```
1. 方針: 銀行振込の入金確認は手動。SYSTEM_ADMIN が管理画面から mark-paid する
2. 自動 OVERDUE 化:
   a. @Scheduled(cron="0 0 6 * * *", zone="Asia/Tokyo") で毎日 AM 6:00 (JST) に実行
   b. 排他制御: ShedLock で同時実行を防止
   c. status = ISSUED かつ due_date < TODAY の請求書を検索
   c. status = OVERDUE に更新
   d. 広告主にメール通知（「お支払い期限を過ぎています」）
   e. SYSTEM_ADMIN にもプッシュ通知（未払い一覧リンク付き）
3. OVERDUE 後の対応:
   a. 7日超過: 2回目のリマインドメール送信
   b. 30日超過: SYSTEM_ADMIN に警告通知。advertiser_accounts.status = SUSPENDED の判断を促す
   c. SUSPENDED 時は全キャンペーン自動停止（credit_limit 到達時と同じロジック）
4. 入金確認後: SYSTEM_ADMIN が PATCH /mark-paid で PAID に更新 → 広告主に確認メール送信
```

### credit_limit 監視ロジック

```
1. 日次集計バッチ（F09.7 の ad_daily_stats 集計）完了後にイベント発火
2. 各 ACTIVE 広告主について:
   a. 当月の ad_daily_stats から累計費用を算出
   b. 累計費用 >= credit_limit × 0.8: 広告主に警告通知（「予算の80%に到達」）
   c. 累計費用 >= credit_limit:
      - 当該広告主の全キャンペーンを自動停止（ad_campaigns.status = PAUSED）
      - 広告主に停止通知
      - SYSTEM_ADMIN にも通知
3. SYSTEM_ADMIN が credit_limit を引き上げた場合、広告主が手動でキャンペーンを再開
```

### A/B テスト有意差判定

```
1. 同一キャンペーン内のクリエイティブが2つ以上ある場合に実行
2. 各クリエイティブの CTR を算出（impressions = 0 の場合は CTR = 0.0）
3. 前提条件チェック:
   a. 各クリエイティブのインプレッションが 1000 以上であること
   b. いずれかのクリエイティブで clicks = 0 の場合はカイ二乗検定の期待度数が不定になるため、Fisher の正確確率検定にフォールバック
4. CTR 上位2つについて検定を実施:
   a. 帰無仮説: 2つのクリエイティブの CTR に差はない
   b. 有意水準: p < 0.05
   c. 検定手法: 期待度数が全セル 5 以上の場合はカイ二乗検定、それ以外は Fisher の正確確率検定
5. クリエイティブが3つ以上の場合:
   a. 全ペアのうち CTR 差が最大のペアで検定を実施
   b. 「winner は CTR 上位2つの比較結果のみに基づく」旨をレスポンスに注記（多重比較の完全な補正は行わない）
6. 有意差ありの場合: winner を返却（ad_id, reason, p_value）
7. 有意差なし or サンプル不足の場合: winner = null
```

### ゼロ除算の共通ルール

パフォーマンス系 API の全レスポンスに適用:
- `impressions = 0` の場合: `ctr = 0.0`, `avg_cpm = null`, `avg_cpc = null`
- `clicks = 0` の場合: `avg_cpc = null`
- 対象期間にデータが存在しない場合: summary の各フィールドは `0` or `null`（金額系は `0`、率系は `null`）
- `conversions = 0` の場合: `conversion_rate = 0.0`, `cost_per_conversion = null`

### 予算ペーシング算出ロジック

```
1. 当月の ad_daily_stats から日別費用を取得
2. daily_avg_cost = 当月累計費用 / 経過日数
3. projected_monthly_cost = daily_avg_cost × 当月の日数
4. remaining_budget = credit_limit - 当月累計費用
5. remaining_days_at_current_pace = remaining_budget / daily_avg_cost（daily_avg_cost = 0 の場合は null）
6. budget_exhaustion_date = TODAY + remaining_days_at_current_pace
7. pacing_status の判定:
   - projected_monthly_cost < credit_limit × 0.7: UNDER_PACING
   - projected_monthly_cost <= credit_limit: ON_TRACK
   - projected_monthly_cost > credit_limit: OVER_PACING
8. 経過日数が 0 の場合（月初日でバッチ未完了）: budget_forecast 全体を null で返す（ゼロ除算防止）
9. 経過日数が 1〜2 の場合: 推定精度が低いため pacing_status = null（他のフィールドは算出して返す）
```

### ベンチマーク算出ロジック

```
1. プラットフォーム平均:
   a. 過去30日の全キャンペーンの ad_daily_stats から SUM(clicks) / SUM(impressions) × 100 を算出
   b. 自社 CTR がプラットフォーム全体の何パーセンタイルに位置するかを算出:
      - 全キャンペーンの CTR を降順ソート → 自社の順位 / 全体数 × 100
2. 同一テンプレート平均:
   a. 自社キャンペーンのターゲティングテンプレート（ad_targeting_rules）と同一テンプレートを持つ他キャンペーンを抽出
   b. 過去30日の CTR / CPC 平均を算出
3. プライバシー保護:
   a. 同一テンプレートのキャンペーン数が 5 件未満の場合はベンチマークを返却しない（null）
      — 少数のキャンペーンの実績が推測されるリスクを回避
   b. 個別キャンペーンの数値は返却しない（平均値のみ）
```

### レコメンデーション生成ロジック

```
1. 日次バッチ完了後に各 ACTIVE 広告主について評価（ApplicationEvent）
2. ルール一覧（優先度順）:
   a. BUDGET_ALERT (HIGH): pacing_status = OVER_PACING
   b. CTR_DECLINE (HIGH): 直近7日の CTR が前7日比で 20% 以上低下
   c. CONVERSION_DROP (HIGH): 直近7日のコンバージョン率が前7日比で 30% 以上低下
   d. REGION_OPPORTUNITY (HIGH): breakdown で CTR が全体平均の 1.3 倍以上の地域がある
   e. CREATIVE_WINNER (MEDIUM): A/B テスト判定で有意差が検出された
   f. UNDER_PACING (MEDIUM): pacing_status = UNDER_PACING
   g. LOW_CTR (LOW): 自社 CTR がプラットフォーム平均の 0.5 倍未満
3. 最大5件を返却（HIGH → MEDIUM → LOW の順）
4. 同一キャンペーンに対する同一 type のレコメンデーションは24時間内に1回のみ（スパム防止）
5. レコメンデーション結果はキャッシュに保存（Spring Cache, TTL 1時間）。API リクエストごとの再計算は行わない
6. 日次バッチ完了時にレコメンデーションキャッシュを全エビクトし、次回 API リクエストで最新データに基づき再生成
```

### コンバージョントラッキングロジック

```
1. アトリビューションモデル: ラストクリック
2. トラッキングフロー:
   a. 広告クリック時: ad_clicks にレコード INSERT + クリック ID を Cookie に保存（HttpOnly; Secure; SameSite=None; 有効期限 = attribution_window_days）。SameSite=None + Secure はクロスサイトでの Cookie 送信に必須（広告クリック先 → 別ページ → コンバージョンの動線）
   b. コンバージョンイベント発生時（チーム加入・組織加入等）:
      - ApplicationEvent で ConversiontTrackingListener が受信
      - Cookie からクリック ID を取得
      - クリック ID が有効期限内かを検証（ad_clicks.occurred_at + attribution_window_days >= NOW()）
      - 有効な場合: ad_conversions に INSERT
3. コンバージョン種別:
   - TEAM_JOIN: チームメンバーシップ作成時（F01.2 の TeamMemberService のイベント）
   - ORG_JOIN: 組織メンバーシップ作成時（F01.2）
   - MODULE_ACTIVATE: モジュール有効化時（F01.3）
   - EVENT_REGISTER: イベント参加登録時（F03.8）
4. 重複排除: 同一 click_id × conversion_type の組み合わせは1回のみ記録（アプリ層で検証）
5. 集計: performance API はリアルタイムで ad_conversions を COUNT する（日次集計ではなく、件数が少ないためオンデマンドで十分）
```

## 6. セキュリティ考慮事項

- **マルチテナント分離**: 全広告主向け API で `advertiser_accounts.organization_id = currentUser の所属組織` を Service 層入り口で必ず検証。キャンペーン・請求書は `ad_campaigns.advertiser_organization_id` 経由で間接的に組織紐付けを検証。他組織のデータは 403（存在有無も返さない）
- **認可チェック**: 広告主向け API は `ADMIN` + `DEPUTY_ADMIN（MANAGE_ADS）`。料金シミュレーターのみ広告主未登録でもアクセス可（出稿検討目的）。SYSTEM_ADMIN 向け API は SYSTEM_ADMIN のみ
- **レートリミット**: `Bucket4j` で以下の制限を適用
  - 広告主 API（`/advertiser/**`）: 1分間30件
  - 料金シミュレーター: 1分間60件（未登録ユーザーの連打を考慮）
  - 請求書 PDF ダウンロード: 1時間20件
  - 広告主登録（`/advertiser/register`）: 1日1件/組織（スパム防止）
  - CSV エクスポート: 1時間10件
  - credit_limit 増額申請: 1日1件/組織
  - 定期レポートスケジュール作成: 1時間5件
- **請求書のアクセス制御**: PDF は S3 Pre-signed URL で返却（有効期限5分）。URL の直接共有による第三者アクセスを時間制限で抑制
- **Stripe Customer ID の保護**: `stripe_customer_id` は API レスポンスに含めない（内部管理情報）
- **監査ログ**: 以下のイベントを `audit_logs` に記録
  - `ADVERTISER_REGISTERED`: 広告主登録申請
  - `ADVERTISER_APPROVED`: SYSTEM_ADMIN による承認
  - `ADVERTISER_SUSPENDED`: アカウント停止（reason を含む）
  - `ADVERTISER_REACTIVATED`: SUSPENDED → ACTIVE への復帰
  - `AD_RATE_CARD_CREATED`: 料金テーブル追加
  - `AD_RATE_CARD_UPDATED`: 料金テーブル更新
  - `AD_INVOICE_ISSUED`: 請求書発行
  - `AD_INVOICE_OVERDUE`: 支払期限超過による自動 OVERDUE 化
  - `AD_INVOICE_MARK_PAID`: 手動入金確認（operator_user_id・invoice_id・paid_at・note を記録）
  - `AD_INVOICE_PDF_DOWNLOADED`: 請求書 PDF ダウンロード
  - `AD_CAMPAIGN_AUTO_PAUSED`: credit_limit 到達による自動停止
  - `AD_CREDIT_LIMIT_CHANGED`: credit_limit 変更（変更前後の値・operator_user_id を記録）
  - `AD_ACCOUNT_UPDATED`: 広告主アカウント情報更新（変更フィールドを記録）
  - `AD_RATE_CARD_DELETED`: 料金テーブル削除
  - `AD_CREDIT_LIMIT_REQUEST_APPROVED`: 増額申請承認
  - `AD_CREDIT_LIMIT_REQUEST_REJECTED`: 増額申請却下
  - `AD_REPORT_SCHEDULE_CREATED`: 定期レポートスケジュール作成
  - `AD_REPORT_SCHEDULE_DELETED`: 定期レポートスケジュール削除
  - `AD_PERFORMANCE_EXPORTED`: パフォーマンスデータ CSV エクスポート
- **コンバージョンデータの PII 保護**: `ad_conversions.converted_user_id` は API レスポンスに含めない。広告主には件数・率のみ返却。個人を特定できる情報の漏洩を防止
- **Cookie トラッキングの同意**: コンバージョントラッキング Cookie の設置前にユーザー同意を取得する（F09.7 §10 のプライバシー対応と連動。Cookie バナーで広告パフォーマンス計測目的を明示）
- **ベンチマークデータの保護**: 同一テンプレートのキャンペーン数が 5 件未満の場合はベンチマークを返却しない（個別キャンペーンの推測を防止）
- **mark-paid の冪等性**: `status = PAID` の請求書に対する再呼び出しは 409 Conflict を返す。二重入金確認を防止
- **料金テーブルの改竄防止**: `ad_rate_cards` の更新 API（PUT）は提供しない。料金改定は常に POST（新レコード追加）で行う。過去レコード（`effective_from <= TODAY`）の削除は不可（400 で拒否）。未来日のレコードのみ DELETE 可能
- **PDF の S3 キー**: `invoices/{advertiser_account_id}/{uuid}.pdf` 形式で保存。UUID を含めることで S3 キーの推測を防止。Pre-signed URL の有効期限は 5 分。ステータス変更時（mark-paid 等）は S3 の既存 PDF を削除し、次回ダウンロード時に再生成
- **note フィールドのサニタイズ**: mark-paid の `note` は HTML タグを除去（strip_tags）して保存。管理画面での表示時は必ず HTML エスケープ。許容文字: Unicode 文字・数字・基本的な記号。制御文字は除去
- **消費税計算の正確性**: 税率は環境変数 `AD_TAX_RATE`（デフォルト: 10.00）で管理し、請求書発行時に `ad_invoices.tax_rate` にスナップショット保存する。税額の端数は切り捨て（`FLOOR`）。税率変更時は環境変数を更新するだけで、過去の請求書の税率は発行時の値が保持される。インボイス制度対応として、PDF 請求書にプラットフォームの適格請求書発行事業者登録番号を記載する
- **Stripe API キー管理**: F08.2 と同一方針。Stripe Secret Key / Webhook Secret は環境変数 or AWS Secrets Manager で管理。DB・ログへの出力禁止
- **Stripe Webhook 処理**: 月次バッチで Stripe Invoice を作成した場合、以下の Webhook イベントを処理:
  - `invoice.paid`: `ad_invoices.status = PAID`、`paid_at` を更新
  - `invoice.payment_failed`: `ad_invoices.status = OVERDUE` に更新、広告主にメール通知、SYSTEM_ADMIN にプッシュ通知
  - Webhook 署名検証は `Stripe-Signature` ヘッダーで必ず実施（F08.2 と同一パターン）

## 8. パフォーマンス考慮

- パフォーマンス API は `ad_daily_stats`（F09.7 の日次集計テーブル）を参照するため、生ログ（`ad_impressions` / `ad_clicks`）へのクエリは発生しない
- 料金シミュレーターは `ad_rate_cards`（レコード数は最大 47都道府県 × テンプレート数 × 2課金モデル × 数世代 = 数百件）を参照するため軽量
- 請求書 PDF は初回生成後に S3 にキャッシュ。2回目以降は Pre-signed URL を返すのみ
- breakdown API（地域×テンプレート別）は最大 47 × テンプレート数 のセルを返すため、日次集計テーブルの `campaign_id` + `date` INDEX を活用

### API レスポンスキャッシュ

| エンドポイント | キャッシュ | 理由 |
|--------------|----------|------|
| パフォーマンス系（performance, creatives, breakdown, overview） | `Cache-Control: private, max-age=300`（5分） | 日次集計ベースのため頻繁な更新なし |
| 料金シミュレーター | `Cache-Control: private, max-age=3600`（1時間） | 料金テーブルの変更頻度は低い |
