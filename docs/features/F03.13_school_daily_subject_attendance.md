## 1. 概要

### 目的

学校の日常運用に欠かせない「**毎日の出欠点呼**」と「**教科ごとの出席確認**」をデジタル化し、以下を自動化する:

- 朝の出欠点呼（学級担任が一括登録）
- 各教科の授業ごとの出欠確認（教科担任が登録）
- **「前の授業にいたのに今いない」自動検知** → 学級担任 + 保護者へ即時通知
- 出欠状況の集計・出席統計（月次・学期別）
- 保護者への「お子さまが学校に到着しました」「○限を欠席しました」通知

### 想定ユースケース

| シーン | 流れ |
|--------|------|
| 朝の出欠点呼 | 担任がクラス画面で全員を点呼 → 欠席者の保護者に「○○くんが本日欠席となっています」確認通知 |
| 教科ごとの出席確認 | 教科担任が教室で授業開始前に出欠登録 → 前授業まで出席だったが今欠席 → 担任 + 保護者に即時通知 |
| 体調不良で早退 | 養護教諭が「早退」を登録 → 担任・教科担任・保護者に通知 + 帰宅後の到着確認通知 |
| 欠席連絡（家庭→学校） | 保護者がアプリから当日の欠席・遅刻連絡 → 担任に通知 → 自動的に出欠登録に反映 |

### 前提条件

- F03.9 時間割管理が実装済み（`timetable_slots` で日次の教科コマが定義済み）
- F03.12 ケア対象者見守り通知が実装済み（保護者リンク = `user_care_links`）
- F03.1 スケジュール・出欠管理が実装済み（`schedule_attendances` を流用）
- F04.3 プッシュ通知基盤

## 2. ドメイン定義

### 出欠の単位

| 単位 | 説明 |
|------|------|
| **DAILY**（日次出欠） | 1日に1レコード。朝の点呼で記録される「その日の登校状態」 |
| **PERIOD**（時限別出欠） | 時限ごとに1レコード。教科担任が記録する「その時限の出席状態」 |

DAILY と PERIOD は併存し、相互参照する。DAILY=PRESENT でも PERIOD=ABSENT が連続すれば「前の授業にいたのに今いない」検知の対象になる。

### 出欠ステータス

`schedule_attendances.status` の enum をベースに拡張:

| 値 | 説明 |
|----|------|
| `PRESENT` | 出席 |
| `LATE` | 遅刻 |
| `EARLY_LEAVE` | 早退 |
| `ABSENT` | 欠席 |
| `OFFICIAL_ABSENCE` | 公欠（忌引・公的活動など、出席日数換算外） |
| `SCHOOL_ACTIVITY` | 学校行事による不在（出席日数に含む） |
| `UNDECIDED` | 未確認（点呼前のデフォルト） |

### 欠席理由

| 値 | 説明 |
|----|------|
| `SICK` | 体調不良 |
| `INJURY` | 怪我 |
| `FAMILY_REASON` | 家庭の事情 |
| `BEREAVEMENT` | 忌引 |
| `INFECTIOUS_DISEASE` | 出席停止（インフルエンザ等） |
| `MENTAL_HEALTH` | 心身の不調 |
| `OFFICIAL_BUSINESS` | 公的活動 |
| `OTHER` | その他（コメント必須） |

## 4. サービス設計

### 4.1 SchoolAttendanceService

```
// 日次出欠
+ submitDailyRollCall(teamId, attendanceDate, entries[], operatorUserId): RollCallSummary
  - 担任画面の点呼結果を一括登録
  - 各 entry: { studentUserId, status, absenceReason?, arrivalTime?, comment? }
  - status=ABSENT/LATE の場合、保護者に「お子さまが本日欠席/遅刻となっています」確認通知（F03.12連携）
  - status=PRESENT の場合、保護者に「お子さまが学校に到着しました」通知

+ getDailyAttendance(teamId, attendanceDate): DailyAttendanceListResponse
+ getStudentAttendanceHistory(studentUserId, dateRange): AttendanceHistoryResponse
+ updateDailyRecord(recordId, dto): DailyAttendanceResponse

// 時限別出欠
+ submitPeriodAttendance(teamId, attendanceDate, periodNumber, entries[], operatorUserId): PeriodAttendanceSummary
  - 教科担任の出欠登録
  - 内部で AttendanceTransitionDetectionService を呼び出し「前にいたのに今いない」検知

+ getPeriodAttendance(teamId, attendanceDate, periodNumber): PeriodAttendanceResponse
+ getStudentDailyTimeline(studentUserId, attendanceDate): StudentTimelineResponse
  - その日の全時限のステータスを横並び表示
```

### 4.2 AttendanceTransitionDetectionService

「前にいたのに今いない」検知の中核。

```
+ detectTransition(teamId, studentUserId, attendanceDate, currentPeriod, currentStatus): TransitionAlert?
  - period_attendance_records から当該生徒の同日内の前時限を取得
  - 直前まで PRESENT/LATE だったが、今 ABSENT になった場合 → アラート生成
  - 既に attendance_transition_alerts に未解決アラートがあればスキップ（重複防止）
  - alert_level の判定:
      * 同日の早退連絡(EARLY_LEAVE)が family_attendance_notices にあれば NORMAL かつ自動解決
      * 連絡なしで2時限連続 ABSENT → URGENT
  - 通知対象:
      * 学級担任 + 副担任（class_homerooms から解決）
      * 主保護者（user_care_links から解決）
      * URGENT の場合は教科担任にも通知（次の授業に来ない可能性があるため）

+ resolveAlert(alertId, resolverUserId, note): void
  - 教員 or 保護者が「保健室に行ってた」「家に帰ってた」等の説明を記録して解決
```

### 4.3 FamilyAttendanceNoticeService

```
+ submitNotice(studentUserId, dto, submitterUserId): FamilyNoticeResponse
  - 保護者が家庭から欠席・遅刻連絡を送信
  - 担任 + 副担任にプッシュ通知
  - 当日朝の点呼前であれば daily_attendance_records に自動反映オプション

+ acknowledgeNotice(noticeId, acknowledgerUserId): void
+ applyToAttendanceRecord(noticeId): DailyAttendanceResponse
  - 連絡内容を daily_attendance_records へ反映
```

## 6. 通知フロー

### 6.1 朝の点呼後の保護者通知

```
担任が朝の点呼を提出
    ↓
SchoolAttendanceService.submitDailyRollCall()
    ↓ 各生徒について
status=PRESENT → 保護者に「お子さまが学校に到着しました」通知 (任意設定でOFF可)
status=ABSENT  → 保護者に「お子さまが本日欠席となっています。連絡がない場合はご確認ください」通知
status=LATE    → 保護者に「お子さまが遅刻として記録されました」通知
status=UNDECIDED → 通知なし
    ↓
F03.12 CareEventNotificationService.notify*() を経由
```

### 6.2 「前にいたのに今いない」検知フロー

```
教科担任が3限の出欠を提出（ある生徒を ABSENT に登録）
    ↓
SchoolAttendanceService.submitPeriodAttendance()
    ↓
AttendanceTransitionDetectionService.detectTransition() を呼び出し
    ↓
2限の period_attendance_records を確認
    ↓
2限=PRESENT && 3限=ABSENT && 早退連絡なし
    ↓
attendance_transition_alerts レコード作成
    ↓
通知:
  - 学級担任 + 副担任にプッシュ通知 (priority=HIGH)
    「⚠️ 太郎くんが2限まで出席でしたが、3限から不在です。所在を確認してください」
  - 主保護者にプッシュ通知
    「⚠️ 太郎くんが学校で前の授業まで出席していましたが、現在の授業に不在です。学校に確認しています」
  - alert_level=URGENT (2時限連続不在) なら4限の教科担任にも事前通知
    ↓
担任が「保健室にいた」「早退連絡が遅れていた」等を確認後、resolveAlert() を呼び出してアラート解決
    ↓
保護者に「解決: ○○の理由でした。ご安心ください」フォローアップ通知
```

### 6.3 保護者からの欠席連絡フロー

```
保護者が朝、家から「太郎が発熱で本日欠席」連絡を送信
    ↓
FamilyAttendanceNoticeService.submitNotice()
    ↓
担任 + 副担任にプッシュ通知（音 + バッジ）
「○○の保護者から本日欠席連絡: 発熱」
    ↓
担任が出勤後、画面で「了解」をタップ
    ↓
acknowledgeNotice() 呼び出し
    ↓
点呼前なら自動で daily_attendance_records.status=ABSENT に反映可能
点呼後なら担任が手動で applyToAttendanceRecord() を実行
    ↓
保護者に「欠席連絡を確認しました」フォローアップ通知
```

## 8. セキュリティ・プライバシー

### 8.1 アクセス制御

| 操作 | 必要な権限 |
|------|----------|
| 日次点呼登録 | 学級担任、副担任 |
| 時限別出欠登録 | 教科担任（その時限の `teacher_user_id` 一致 or 学級担任） |
| 出欠閲覧（クラス全体） | 学級担任、副担任、ADMIN |
| 出欠閲覧（自分・自分の被ケア対象） | 本人、保護者 |
| 家庭連絡送信 | 保護者（ケアリンクで結ばれた被ケア対象向けのみ） |

| 移動検知アラート解決 | 学級担任、保護者 |
| 学級担任設定変更 | チームADMIN |

### 8.2 個人情報・健康情報の保護

- `family_attendance_notices.reason_detail` は **AES-256-GCM 暗号化**（病名・診断書情報は要配慮個人情報）
- `attached_file_keys` の S3 オブジェクトは署名付きURL + 短期有効（5分）
- 出欠データのアクセスログは F10.3 監査ログに記録（誰がいつ誰の出欠を見たか）
- 卒業後の生徒データは設定された保管期間後に物理削除（既存 GDPR ポリシー F12.3 準拠）

### 8.3 通知頻度の配慮

- 「朝の到着通知」は保護者の `user_care_links.notify_on_checkin = TRUE` の場合のみ送信（毎日通知は鬱陶しいと感じる保護者も）
- ただし「欠席通知」「移動検知アラート」「家庭連絡確認」は通知設定にかかわらず常に送信（安全配慮の最低限）

## 9. エッジケース・エラーハンドリング

| ケース | 対応 |
|--------|------|
| 担任が点呼前に保護者連絡が届いた | 該当生徒を「📩 連絡あり」でハイライト + 1タップで反映 |
| 連絡なしで欠席 → 保護者に通知 → 後から保護者連絡が届いた | 通知済みフラグはそのまま、家庭連絡を後付けで `daily_attendance_records.family_notice_id` に紐付け |
| 教科担任が時間割変更を見落として旧コマで登録 | `timetable_changes` を確認し、変更がある場合は警告表示「本日この時限は変更されています」 |
| 移動検知で「実は早退連絡があった」と判明 | resolveAlert() でアラートを解決、`resolution_note` に記録、保護者にフォロー通知 |
| 「前の授業」が休講・自習だった場合 | period_attendance_records が NULL → 移動検知の起点にならない（PRESENT履歴が必要） |
| 保健室登校・別室登校 | 出席ステータスは PRESENT、`comment` に「保健室登校」と記録（カスタム拡張は v2 で `attendance_location` 列を検討） |
| 1人の生徒が2つのクラスに所属（兼籍） | 各クラスごとに独立した `daily_attendance_records` を持つ。担任が複数いる場合は `class_homerooms` 各レコードへ通知 |
| 学校行事（遠足・修学旅行）で全員 SCHOOL_ACTIVITY | 担任が「全員一括登録」ボタンで一発設定。F03.10 年間行事計画と連動可能 |
| 生徒が転校・退学 | 過去の出欠レコードは保持（指導要録のため）。新規登録は不可（class_homerooms.effective_until で在籍管理） |

## 10. パフォーマンス考慮

- `daily_attendance_records` / `period_attendance_records` は1学期で数十万件規模 → date + team_id の複合インデックスで高速化
- 移動検知ロジックは教科担任の登録イベントごとに1人1クエリ → クラス40人で 3msec 以内目標
- 月次統計は集計結果を `attendance_monthly_summaries` キャッシュテーブルに日次バッチで保存（オプション、Phase 3 で検討）
- アラートの未解決一覧は Valkey に 30秒キャッシュ（教員の頻繁なポーリング対策）

## 14. 学期出席要件管理

### 14.1 背景・目的

学校教育では「出席日数」「教科ごとの出席率」が進級・卒業・単位認定に直結する。担任・教科担任は **「あと何日休めるか」「保健室登校は出席に含むか」「○○の理由は公欠か」** を常時把握する必要がある。

本機能は F03.13 の日次・時限出欠を学期単位で集計し、出席要件（規程ベース）を管理・警告する。

### 14.2 ドメイン定義

#### 出席ロケーション (attendance_location)

「物理的にどこにいたか」を独立して管理。`status=PRESENT` のままでも保健室登校・別室登校・オンライン登校を区別できる。

| 値 | 説明 | 出席日数換算 |
|----|------|-------------|
| `CLASSROOM` | 通常教室登校 | カウント |
| `SICK_BAY` | 保健室登校 | 学校規程による（既定: カウント） |
| `SEPARATE_ROOM` | 別室登校（不登校支援） | 学校規程による（既定: カウント） |
| `LIBRARY` | 図書室登校 | 学校規程による（既定: カウント） |
| `ONLINE` | オンライン登校（リモート授業） | 学校規程による（既定: カウント） |
| `HOME_LEARNING` | 家庭学習（教育支援センター連携など） | 学校規程による（既定: 公欠相当） |
| `OUT_OF_SCHOOL` | 校外活動・実習 | カウント |
| `NOT_APPLICABLE` | 出欠が PRESENT 以外（ABSENT等） | — |

#### 出席要件区分 (requirement_category)

進級・単位認定の基準として組織/学年で定義する規程。

| 値 | 説明 |
|----|------|
| `GRADE_PROMOTION` | 進級要件（年度内出席率） |
| `GRADUATION` | 卒業要件 |
| `SUBJECT_CREDIT` | 教科単位認定要件（高校に多い） |
| `PERFECT_ATTENDANCE` | 皆勤賞 |
| `CUSTOM` | 自由設定（部活動皆勤・委員会等） |

### 14.3 DB設計

  ADD COLUMN location_changed_during_day BOOLEAN NOT NULL DEFAULT FALSE COMMENT '同日内でロケーションが変化したか（朝CLASSROOM→途中SICK_BAY等）';

### 14.5 通知フロー

#### 出席要件警告フロー（教育的配慮を重視した非公開設計）

> **設計方針**: 「あと何日休めるか」「要件未達まで残何日か」といった残数情報を **生徒本人へ自動通知することは行わない**。
> 残数を直接伝えると「あと3日休める」と消費的に解釈されたり、達成プレッシャーで心身に悪影響を与えるリスクがある。
> アラートは **担任・教科担任が把握 → 生徒の状況を理解した上で必要な範囲・タイミング・表現で開示判断** する形にする。

```
日次バッチ runDailyEvaluation() が evaluations を更新
    ↓
ある生徒の評価が OK→WARNING に変化（出席率が80%を下回った等）
    ↓
自動通知配信（教員側のみ）:
  - 学級担任: 「○○さんが進級要件のWARNING水準に到達」
  - 副担任: 同上
  - 該当教科の教科担任（SUBJECT_CREDIT規程の場合）: 「○○さんの△△教科出席率がWARNING水準」
  - 教務（RISK以上のみ）
    ↓
生徒本人・保護者には自動通知しない（後述§14.5.1の開示判断フローへ）
    ↓
WARNING→RISK に変化した場合:
  - 担任画面に赤色アラートバナー表示
  - 担任が「面談を設定」「保護者に連絡」を1タップで起動できるアクションボタンを提示
    ↓
RISK→VIOLATION に確定した場合:
  - 担任 + 教務に確定通知（教員側のみ）
  - 担任画面に補習・特例措置のフロー誘導カード表示
  - 保護者・本人への通知は担任の明示的判断を経て発信
```

#### 14.5.1 担任による開示判断フロー（生徒・保護者への手動開示）

担任・教科担任は評価結果を見て、**生徒の状態（精神面・家庭環境・学業意欲）を踏まえた上で開示の要否・範囲を判断** する。

```
担任が AtRiskStudentList で個別生徒を選択
    ↓
StudentRequirementDetailScreen に評価詳細が表示される
    ↓
担任が「開示する」を選択した場合:
  「開示モード」を選んで送信
    1. 数値あり開示: 出席率・残り日数・規程名を含む
    2. 数値伏せ開示: 「要件達成にご注意ください」のみ伝える（残数は記載しない）
    3. 個別面談予約: 通知ではなく面談設定だけ送る
    ↓
DisclosureService.disclose(evaluationId, mode, recipients[], message) を呼び出し
    ↓
attendance_disclosure_records に開示記録を残す
    ↓
保護者と本人（必要な範囲）に通知配信
    ↓
担任が開示しない場合:
  attendance_disclosure_records.decision='WITHHELD' + 理由を記録
  （自分の指導記録としてあとで参照可能）
```

#### 14.5.2 緊急例外: VIOLATION 確定時の自動通知範囲

VIOLATION（要件未達確定）の場合のみ、教務側で「自動で保護者にも通知する」設定を組織レベルで有効化できる（任意）。デフォルトは **OFF**（担任判断）。これは進級・卒業に影響する重大事項であり、担任が忘れた場合の事故防止策。

- 組織設定 `attendance_requirement_rules.notify_violation_to_guardian_automatically BOOLEAN DEFAULT FALSE` を追加
- 有効時のみ VIOLATION で自動配信、それ以外は §14.5.1 の手動開示フローを必ず通る

### 14.8 セキュリティ・プライバシー

| 対象 | 考慮事項 |
|------|---------|
| 保健室登校情報 | 健康・心身配慮の機微情報。閲覧範囲は本人/保護者/担任/養護教諭/学年主任に限定。一般教員には「PRESENT」のみ表示 |
| 別室登校情報 | 不登校支援の文脈。閲覧範囲は本人/保護者/担任/学年主任/スクールカウンセラーに限定 |
| 出席要件評価結果（残数・要件状態） | **教員（担任・教科担任・教務）のみ自動アクセス可**。生徒本人・保護者は担任の明示的な開示判断（DisclosureService）を経た情報のみ閲覧可能。教育的配慮を制度化 |
| 開示判断記録 | attendance_disclosure_records.message / withhold_reason は AES-256-GCM 暗号化（指導記録の機微情報） |
| ロケーション変化理由 | `attendance_location_changes.reason / note` は AES-256-GCM 暗号化（健康情報配慮） |
| 要件違反通知の文面 | 直接的な「未達」表現は本人・保護者・担任のみ。他のメンバーには伏せる |
| 監査ログ | 規程の作成・変更・違反解消は F10.3 監査ログに必須記録 |
| 規程の遡及変更 | 既存の評価結果に影響する変更時は警告 + 影響範囲レポートを表示 |

### 14.9 エッジケース・エラーハンドリング

| ケース | 対応 |
|--------|------|
| 同日内で教室→保健室→教室と複数回ロケーション変化 | attendance_location_changes に複数レコード、UI でタイムライン表示 |
| 保健室登校を出席に含めない学校設定 | attendance_requirement_rules.count_sick_bay_as_present=FALSE に設定 → サマリで sick_bay_days はカウントするが present_days には含めない |
| 学期途中で規程が変更された | 規程の effective_from/effective_until で適用範囲を制限。過去の評価は当時の規程で再計算可能（evaluated_at で凍結） |
| 転入生（学期途中入学） | summary.period_from を入学日に設定。total_school_days は入学日以降のみカウント |
| 長期入院（30日超のSICK等） | OFFICIAL_ABSENCE 扱い + 担任が attendance_requirement_evaluations をマニュアル resolve（特例承認） |
| 集計バッチが失敗 | summary.last_recalculated_at が更新されない → 翌日のバッチで再試行。手動再計算APIも提供 |
| 教科別出席率（subject_breakdown）の集計コスト | 日次バッチで JSON カラムに事前集計。リアルタイム再計算は要請時のみ |
| 規程の競合（組織規程 vs チーム規程） | チーム規程を優先。両方存在する場合、フロントで「チーム個別設定が適用中」と明示 |
| 過去年度のデータ参照 | summary は年度ごとに保持されているため自然に対応。ただし2年以上前のデータは集計コスト軽減のため UI で別タブに分離 |

### 14.10 パフォーマンス考慮

- `student_attendance_summaries` は事前集計テーブル。日次バッチ + 出欠変更時の差分更新で常時新鮮さを維持
- 全校生徒1000人 × 200日 = 20万レコード/年でも summary は 1000レコード/学期 で完結（O(N) ではなく O(students) に削減）
- `attendance_requirement_evaluations` の status 変化判定は前回値との比較のみ（フルスキャン不要）
- 月次クラス集計は Valkey に 1時間キャッシュ
- summary 再計算の手動 API はレート制限（1ユーザー/1時間に5回）

## 15. 精査結果（学期出席要件管理セクション §14）

§14 追加後、不備・セキュリティ・ユーザビリティ・見落とし・保守性の観点で **2回精査** を実施。修正済みの項目を以下に列挙する。

### 15.1 1回目精査での修正

| 観点 | 指摘 | 修正 |
|------|------|------|
| 不備 | 規程変更時に既存の summary・evaluation が古い基準のまま残る | §14.9 に「規程の遡及変更」エッジケースを明記、`evaluated_at` で凍結する仕様を追加 |
| 不備 | 教科別出席率が summary に反映される仕組みが曖昧 | `subject_breakdown JSON` カラムを student_attendance_summaries に追加、日次バッチで集計と明記 |
| セキュリティ | 保健室・別室登校情報が一般教員に露見するリスク | §14.8 に閲覧範囲をロール別に明記、一般教員には「PRESENT」のみ表示 |
| セキュリティ | 違反通知の文面が他生徒に見えるとイジメ等の誘発リスク | 通知文面の閲覧範囲を「本人・保護者・担任のみ」に限定、他には伏せる旨を明記 |
| ユーザビリティ | 「あと何日休めるか」が一目で分からないと意味がない | StudentAttendanceSummaryCard.vue に `remaining_allowed` 横棒バー表示を仕様化 |
| 保守 | summary を全件再集計するとコスト過大 | 差分更新方式を採用、`recalculateOnAttendanceChange` フックを設計 |

### 15.2 2回目精査での修正

| 観点 | 指摘 | 修正 |
|------|------|------|
| 見落とし | 転入・転出時の集計起点が曖昧 | `student_attendance_summaries.period_from / period_to` を採用し、入学日以降をカウントする仕様を §14.9 に明記 |
| 見落とし | 遅刻N回で欠席1日換算する規程対応 | `attendance_requirement_rules.count_late_as_absence_threshold` を追加（高校等での慣習対応） |
| 見落とし | 違反確定後の救済措置（補習・特例承認）の記録手段がない | `AttendanceRequirementEvaluationService.resolveViolation()` API + `RequirementResolutionModal.vue` を追加 |

### 15.3 未解決問題の解消

§12 の未解決問題リストから以下を §14 で解消:

- ✅ 「教科を跨ぐ『保健室登校』専用ステータス」 → §14.2 attendance_location ENUM で解決
- ✅ 「出席停止（インフルエンザ等）の自動延長」 → §14.9 「長期入院」エッジケースで担任マニュアル管理 + 公欠扱いで対応
- ✅ 「通知表（指導要録）への自動転記」の前提となる出席集計データ → §14.3 student_attendance_summaries で提供。転記機能自体は v3 で別タスク化（既知の制限として残す）
