## 1. 概要

同じスポーツ・業種のチームが「実際に登録・消費している備品」を匿名集計し、「他のチームでよく使われている備品ランキング」として備品管理画面に表示する機能。各ランクアイテムには Amazon アフィリエイトリンクを付与し、チームの補充購入がそのままプラットフォームの収益（アフィリエイト報酬）につながるよう設計する。

### なぜ今作るか

- **ADMIN の課題**: 備品管理担当者は「同種のチームが何を使っているか」を知る手段がなく、備品選定に時間がかかる
- **プラットフォームの課題**: F09.7 の広告バナーはクリック率が低く（一般的な CTR: 0.1〜0.3%）、収益が限定的
- **解決策**: 備品補充というコンテキストで自然に生まれる「これ買いたい」という意欲を、集合知（ランキング）とアフィリエイトで収益化する

### 設計上の最重要原則

1. **プライバシーファースト**: 個別チームのデータは絶対に露出しない。必ずN件以上の集計でのみ表示
2. **opt-out 保証**: データ提供を望まないチームは設定で除外できる
3. **正直な UI**: 「他のチームのランキングデータを元に表示しています」と明示。ステルスマーケティングにならない
4. **付加価値が先**: アフィリエイト収益は副産物。ADMIN にとって「参考になる情報」であることが第一

## 3. 用語定義

| 用語 | 定義 |
|------|------|
| 同類チーム | 同じ `teams.template`（例: `SPORTS`）を持つチーム群 |
| 備品スコア | チームへの登録数 × 重み + 消費回数 × 重み で算出する人気指標 |
| 最小表示閾値 | ランキングに表示するために必要な最小チーム数（デフォルト: 5チーム）。これ未満のアイテムは非表示（逆算攻撃防止） |
| アフィリエイトリンク | `equipment_items.amazon_asin` + `affiliate_configs.tag_id` から生成される Amazon 商品直リンク |
| opt-out チーム | データ提供を拒否したチーム。集計対象から除外される |

#### `equipment_rankings`

日次バッチで再計算されたランキング結果を保持する。正規化はせず、高速 SELECT のためにデータを非正規化して保持する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_template` | VARCHAR(50) | NO | — | チームテンプレート（`teams.template` の値。例: `SPORTS`、`CORPORATE`）|
| `category` | VARCHAR(100) | NO | `'__ALL__'` | 備品カテゴリ（`'__ALL__'` = 全カテゴリ横断集計。NULL は使わない。MySQL の UNIQUE INDEX が NULL 同士を等しいと扱わないため sentinel 値を使用） |
| `rank` | SMALLINT UNSIGNED | NO | — | 同一 template × category 内の順位（1始まり） |
| `item_name` | VARCHAR(200) | NO | — | 備品名（集計時点の最頻出名称。完全一致ではなく代表値） |
| `normalized_name` | VARCHAR(200) | NO | — | 名寄せ後のキー（大文字統一・括弧除去・全半角統一後の値。集計グループ化に使用） |
| `amazon_asin` | VARCHAR(10) | YES | NULL | ASIN（同一 normalized_name で最多出現の ASIN。`asin_confidence >= 50` の場合のみ設定。50%未満は NULL） |
| `asin_confidence` | TINYINT UNSIGNED | YES | NULL | ASIN 信頼度（最多出現 ASIN の占有率 %。例: 3チーム中2チームが同じ ASIN → 67）|
| `team_count` | INT UNSIGNED | NO | — | この備品を登録しているチーム数（公開値。最小表示閾値チェックに使用） |
| `total_quantity` | INT UNSIGNED | NO | — | 登録されている総在庫数（参考値） |
| `consume_event_count` | INT UNSIGNED | NO | 0 | 過去90日の消費イベント数（消耗品の需要の高さを反映） |
| `score` | DECIMAL(10,2) | NO | — | 総合スコア（`team_count × 3.0 + consume_event_count × 1.0`）|
| `calculated_at` | DATETIME | NO | — | 集計日時 |

**インデックス**
```sql
INDEX idx_er_template_category_rank (team_template, category, rank)  -- 一覧取得の主クエリ
INDEX idx_er_template_rank (team_template, rank)                     -- カテゴリなし全体ランキング用
UNIQUE INDEX uq_er_template_category_name (team_template, category, normalized_name)  -- 重複防止
```

**制約・備考**
- このテーブルは「ビューの代替」として機能する。正規化せず、バッチが `DELETE FROM` → `INSERT` で再構築する（`TRUNCATE` は MySQL DDL でありトランザクション内でロールバック不可のため使用しない）
- `team_count < 最小表示閾値` のレコードも INSERT するが、API 層でフィルタして返却しない（将来の閾値変更に対応するため DB には保持）
- `category = '__ALL__'` のレコードは全カテゴリを横断したランキング（「今週の全備品トップ10」等に使用）。NULL ではなく sentinel 値を使うのは、MySQL の UNIQUE INDEX が `NULL != NULL` と評価するため同一 `(template, NULL, name)` の重複挿入を防げないから
- `amazon_asin` は最多出現 ASIN のシェアが50%以上の場合のみ設定。50%未満の場合は NULL とし、フロントエンドで Amazon リンクを非表示にする（分散した ASIN で特定商品を推薦するのはミスリーディング）

#### `equipment_ranking_exclusions`

チームの opt-out 設定と、SYSTEM_ADMIN による特定アイテムの除外設定を管理する。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `exclusion_type` | ENUM('TEAM_OPT_OUT', 'ITEM_EXCLUSION') | NO | — | 除外種別 |
| `team_id` | BIGINT UNSIGNED | YES | NULL | FK → teams（`TEAM_OPT_OUT` 時に設定） |
| `normalized_name` | VARCHAR(200) | YES | NULL | 除外アイテム名（`ITEM_EXCLUSION` 時に設定。品質問題・法的問題のある商品を除外） |
| `reason` | VARCHAR(300) | YES | NULL | 除外理由（SYSTEM_ADMIN のメモ用） |
| `excluded_by_user_id` | BIGINT UNSIGNED | YES | NULL | FK → users（ITEM_EXCLUSION の場合、操作した SYSTEM_ADMIN） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |

**インデックス**
```sql
UNIQUE INDEX uq_ere_team_opt_out (team_id, exclusion_type)  -- 同一チームの二重 opt-out 防止（DB レベル保証）
UNIQUE INDEX uq_ere_item_name (normalized_name, exclusion_type)  -- 同一アイテムの二重除外防止
```

**制約**
- `TEAM_OPT_OUT` の場合: `team_id NOT NULL`, `normalized_name IS NULL`
- `ITEM_EXCLUSION` の場合: `team_id IS NULL`, `normalized_name NOT NULL`
```sql
CONSTRAINT chk_ere_scope
  CHECK (
    (exclusion_type = 'TEAM_OPT_OUT' AND team_id IS NOT NULL AND normalized_name IS NULL)
    OR (exclusion_type = 'ITEM_EXCLUSION' AND team_id IS NULL AND normalized_name IS NOT NULL)
  )
```

### 名寄せロジック（normalized_name の生成）

同じ備品が「ビブス」「ビブス赤」「ビブス（赤）」と異なる名称で登録される場合がある。厳密なマスター管理はしないが、基本的な名寄せを行う。

```
正規化手順:
1. 全角英数字 → 半角変換
2. 大文字 → 小文字変換（英字）
3. 括弧内の補足（例: 「（赤）」「（会議室用）」）を除去
4. 先頭・末尾・連続スペースをトリム
5. 記号（・、#、.等）を除去

例:
  「ビブス赤 #3」→ 「ビブス赤 3」（色は別アイテムとして保持。番号の '#' のみ除去）
  「プロジェクター（EPSON）」→ 「プロジェクター」（括弧内メーカー名を除去）
  「Ｗｈｉｔｅ Ｂｏａｒｄ」→ 「white board」（全角→半角、大→小文字変換）
  「テーピング・50mm」→ 「テーピング50mm」（記号除去。サイズは残す = 別アイテム）
```

> **設計判断**: 「ビブス」と「ビブス赤」は**別アイテム**として扱う。色・サイズは商品の区別要因であり、同じアイテムに名寄せすると誤った ASIN にリンクされるリスクがある。除去するのは括弧内の補足（メーカー名・用途説明）のみ。
>
> **現フェーズの割り切り**: 完全な名寄せは自然言語処理が必要で複雑。Phase 12 では上記の基本正規化のみ実装し、精度は「ある程度合う」レベルで許容する。品名マスター（SYSTEM_ADMIN 承認制）は Phase 13+ で検討。

#### `GET /api/v1/teams/{teamId}/equipment/trending`

**認可**: ADMIN / DEPUTY_ADMIN（`MANAGE_EQUIPMENT`）

**クエリパラメータ**
| パラメータ | 型 | 必須 | 説明 |
|-----------|---|------|------|
| `category` | String | No | カテゴリでフィルタ。未指定: 全カテゴリ横断（`category = '__ALL__'` のレコードを返却） |
| `limit` | Integer | No | 件数（デフォルト 10、最大 20） |
| `linked_only` | Boolean | No | `true`: ASIN あり（Amazon リンクあり）のアイテムのみ返却。デフォルト: `false`（全アイテム表示。Amazon リンクなしでも参考情報として有用） |

**レスポンス（200 OK）**
```json
{
  "data": {
    "team_template": "SPORTS",
    "category": null,
    "opt_out": false,
    "ranking": [
      {
        "rank": 1,
        "item_name": "ビブス",
        "category": "ユニフォーム",
        "team_count": 42,
        "consume_event_count": 4,
        "amazon_asin": "B01234567X",
        "replenish_url": "https://www.amazon.co.jp/dp/B01234567X?tag=mannschaft-22"
      },
      {
        "rank": 2,
        "item_name": "テーピング",
        "category": "医療・応急",
        "team_count": 38,
        "consume_event_count": 15,
        "amazon_asin": null,
        "replenish_url": null
      }
    ],
    "total_templates_teams": 120,
    "calculated_at": "2026-04-12T03:00:00"
  }
}
```

**フィールド補足**
| フィールド | 説明 |
|-----------|------|
| `opt_out` | 当該チームが opt-out している場合 `true`。opt-out 中は自チームのデータが集計に含まれない（ランキング閲覧自体は可能）|
| `team_count` | このアイテムを登録しているチーム数。最小表示閾値（5チーム）未満のアイテムは含まれない |
| `consume_event_count` | 過去90日間の消費イベント数。消耗品の需要頻度を直感的に示す（`score` は内部ソート用で非公開）|
| `replenish_url` | `amazon_asin` があり、かつ `affiliate_configs` に有効な AMAZON 設定がある場合のみ生成。それ以外は `null` |
| `total_templates_teams` | 同一テンプレートに属するチーム総数（opt-out 除外後）。「120チーム中42チームが使用」の表示に使用 |
| `calculated_at` | 最後にランキングが集計された日時（データ鮮度の表示に使用） |

> `score`（内部ソート用スコア）は API レスポンスに含めない。重みの逆算が可能になり、ランキング操作（スコアを意図的に上げる行為）のリスクがある。ADMIN には `team_count`（42チームが使用）と `consume_event_count`（過去90日で15回補充）という直感的な値を表示する。

**エラーレスポンス**
| ステータス | 条件 |
|-----------|------|
| 403 | 権限不足 |
| 404 | `teamId` が存在しない |
| 503 | ランキングデータが未集計（`equipment_rankings` が空）。初回バッチ実行前は503 |

## 6. バッチ処理設計

### 日次ランキング集計バッチ

**実行タイミング**: 毎日 03:00（深夜3時。ユーザーアクティビティが最も低い時間帯）

**処理フロー**:
```
1. Valkey 分散ロック取得（キー: mannschaft:lock:equipment-ranking-batch、TTL: 10分）
   - 取得失敗 → ログ出力して終了（既にバッチ実行中）
2. opt-out チームの team_id 一覧を equipment_ranking_exclusions から取得
3. equipment_items テーブルを集計:
   - team_id IS NOT NULL のみ（Phase 12 はチーム向けのみ。organization_id のみの備品は除外）
   - opt-out チームを除外
   - deleted_at IS NULL のみ（論理削除済みの備品は除外）
   - equipment_items JOIN teams ON equipment_items.team_id = teams.id で teams.template を取得
   - teams.template でグループ化
   - normalized_name（名寄せ済み名称）でさらにグループ化
   - 集計値を計算:
     a. team_count: DISTINCT team_id の数
     b. total_quantity: SUM(quantity) (opt-outチーム除外後)
     c. amazon_asin: 最多出現 ASIN（複数ある場合は COUNT DESC LIMIT 1）
     d. asin_confidence: 最多出現 ASIN の占有率（%）
        - 占有率 >= 50% → amazon_asin を採用
        - 占有率 < 50% → amazon_asin = NULL, asin_confidence = NULL
4. equipment_assignments テーブルを集計（過去90日の消費イベント）:
   - assigned_at >= NOW() - INTERVAL 90 DAY かつ assigned_at = returned_at（消費記録）
   - opt-out チームを除外
   - normalized_name でグループ化 → consume_event_count を取得
5. スコア計算: team_count × 3.0 + consume_event_count × 1.0
6. equipment_ranking_exclusions(ITEM_EXCLUSION) に含まれる normalized_name を除外
7. @Transactional 内で:
   a. DELETE FROM equipment_rankings（全件削除）
   b. 新しい集計結果を一括 INSERT
   c. コミット（失敗時はロールバック → 旧データ消失なし）
8. Valkey キャッシュを無効化（キー: mannschaft:cache:equipment:trending:*）
9. Valkey 分散ロックを解放
10. 集計結果のサマリーをログ出力（総アイテム数、template 別件数、所要時間）
```

> **TRUNCATE を使わない理由**: MySQL の `TRUNCATE TABLE` は DDL 操作であり、暗黙的コミットが発生する。`@Transactional` 内でロールバック不可のため、`DELETE FROM` + `INSERT` をトランザクション内で実行し、途中失敗時の安全性を確保する。

**処理性能の考慮**:
- `equipment_items` のレコード数が多い場合でも JOIN を最小化（非正規化集計）
- バッチ実行時間の見積もり: チーム数1万・備品10万件で約2〜5分
- `DELETE FROM` は `TRUNCATE` より低速だが、レコード数は `top-n-per-category × template数 × category数` で最大数千件程度のため問題ない
- バッチ失敗時: トランザクションがロールバックされ、前回の集計データをそのまま使用し続ける（古くても表示するほうがよい）

**Scheduler 設定** (`application.yml`):
```yaml
equipment:
  ranking:
    cron: "0 0 3 * * *"    # 毎日 03:00
    min-team-count: 5       # 最小表示閾値
    score-weight-team: 3.0  # team_count の重み
    score-weight-consume: 1.0  # consume_event_count の重み
    top-n-per-category: 20  # カテゴリ別上位保存件数
```

## 7. キャッシュ戦略

```
■ Valkey キャッシュ設計:
  - キー: mannschaft:cache:equipment:trending:{template}:{category|__ALL__}
  - 値: top-20 の全ランキングデータ（JSON シリアライズ）
  - TTL: 1時間（日次バッチで更新されるデータのため、バッチ後に無効化で十分）
  - 無効化: 日次バッチ完了後に全 trending キーを削除
  - `limit` パラメータはキャッシュキーに含めない。top-20 をキャッシュしアプリ層でスライスする（limit=5 も limit=10 も同一キャッシュから応答。キャッシュ効率最大化）

■ キャッシュのメリット:
  - 同じ template の多数チームが同時にランキングを参照しても DB 負荷なし
  - ADMIN が備品管理ページを開くたびに JOIN クエリが走らない
  - limit 違いでキャッシュが分散しない（1テンプレート×1カテゴリ = 1エントリ）

■ グレースフルデグラデーション:
  - Valkey 接続エラー時: equipment_rankings テーブルから直接 SELECT（TTL なしで毎回 DB アクセス）
  - equipment_rankings テーブルが空（初回バッチ未実行）: 503 を返却し、フロントエンドは「準備中」メッセージを表示
```

## 9. ビジネスロジック

### スコア計算の根拠

| 指標 | 重み | 理由 |
|------|------|------|
| `team_count × 3.0` | 高 | 「どれだけ多くのチームが採用しているか」= 汎用性・定番感の指標。最も信頼性が高い |
| `consume_event_count × 1.0` | 中 | 消耗品の需要頻度を反映。消耗品でない備品は 0 になるため、重みを低めに設定 |

> スコアの重みは `application.yml` で設定可能。運用データが蓄積した段階で調整する。

### opt-out の影響範囲

```
opt-out 前にカウントされたデータ → 次回バッチで除外（即時反映ではない）
opt-out 中のチームがランキングを閲覧 → 可能（閲覧制限はしない）
opt-out 中のチームの備品作成 → 可能（ただし集計には含まれない）
opt-out 解除後 → 翌朝のバッチから再びカウントに含まれる
```

### 最小表示閾値の役割

`team_count >= 5` チームのアイテムのみ API が返却する理由:
- 3チームしかいないランキングは「特定チームを容易に逆算できる」リスクがある
- 例: サッカーチームが3チームしかない地域で「地元チームが借りているものが丸わかり」になってしまう
- 5チームという閾値は厳しすぎず（少量データでも機能）、かつ個別特定を困難にするバランスを取った値

### テンプレート別の初期データ不足問題

サービス初期段階では、同じ template のチームが5チーム未満の可能性がある。その場合:
- `equipment_rankings` にはレコードが存在するが、API フィルタ後に0件になる
- フロントエンドは「まだランキングデータが不足しています。より多くのチームが参加すると表示されます」メッセージを表示
- SYSTEM_ADMIN が閾値を下げることで一時的に表示可能（`application.yml` の `min-team-count` を変更）

## 10. プライバシー・法的考慮事項

### 個人情報保護

- 集計には `equipment_items` の **チーム単位** データのみ使用。ユーザー個人への紐付けは一切なし
- `equipment_assignments` は消費回数の**匿名集計**（`COUNT GROUP BY normalized_name`）のみに使用。個別の貸出先（`assigned_to_user_id`）は参照しない。集計結果は `consume_event_count`（全チーム合算の数値）としてのみ公開され、特定チーム・特定ユーザーの消費履歴を逆算することは不可能
- ランキング API のレスポンスには個別チームの情報は含まない（`team_count` のみ）

### 利用規約への記載（実装前に必須）

  備品ランキング機能（集計・匿名化済み）に利用される場合があります。
  この機能を利用しない場合は、備品管理設定よりオプトアウトできます。」

## 11. セキュリティ考慮事項

- **認可チェック**: `EquipmentRankingService` 入り口で teamId とユーザーの所属・ロールを検証
- **データ推測攻撃**: `team_count` が最小表示閾値（5）を下回るアイテムは返却しない（前述）
- **スコア非公開**: `score`（内部ソートスコア）を API レスポンスに含めない。重みの逆算によるランキング操作（特定商品の順位を意図的に上げる行為）を防止
- **ASIN 信頼度フィルタ**: ASIN の占有率50%未満のアイテムには Amazon リンクを付けない。分散した ASIN でユーザーを誤った商品に誘導するリスクを排除
- **SYSTEM_ADMIN の除外権限**: 不適切な商品名（違法・公序良俗違反）は SYSTEM_ADMIN が `equipment_ranking_exclusions` テーブルから除外可能
- **レートリミット**: `/equipment/trending` は Bucket4j で 1分間に30回の制限（scraping 対策）
- **バッチ排他制御**: `/recalculate` API と日次バッチは Valkey 分散ロック（`mannschaft:lock:equipment-ranking-batch`、TTL: 10分）で排他。同時実行によるデータ不整合を防止
- **監査ログ**: `EQUIPMENT_RANKING_OPT_OUT` / `EQUIPMENT_RANKING_OPT_OUT_REMOVED` / `EQUIPMENT_RANKING_RECALCULATED` / `EQUIPMENT_RANKING_ITEM_EXCLUDED` / `EQUIPMENT_RANKING_ITEM_EXCLUSION_REMOVED` を記録

## 14. 設計判断の記録

### 解決済み

- [x] **名寄せの精度向上**: 「ビブス」と「ビブス赤」は**別アイテム**。色・サイズは商品の区別要因であり、同じ ASIN にリンクされないよう別として扱う。括弧内のメーカー名・用途説明のみ除去する。品名マスター（SYSTEM_ADMIN 承認制）は Phase 13+ で検討
- [x] **組織レベルのスコープ**: Phase 12 は**チーム向けのみ**実装。組織には `teams.template` に相当する統一分類がなく、同類判定ロジックの設計が別途必要。Phase 13+ で `organizations.category` カラム導入を検討してから対応
- [x] **楽天対応**: Phase 12 は **AMAZON のみ**。楽天 URL の構造が複雑（店舗ID + 商品ID の組み合わせ）で、`rakuten_item_code` の普及率も現段階では未知数。Amazon 単独でも十分な収益が見込めるため、楽天は Phase 13+ に延期
- [x] **ASIN の正確性**: 最多出現 ASIN の**占有率50%以上の場合のみ採用**。50%未満は `amazon_asin = NULL`（リンクなし）。`asin_confidence` カラムで占有率を記録し、API フィルタで制御。30%:30%:40% のような分散状態で特定商品をリンクするのはミスリーディングなため
- [x] **スコアの透明性**: `team_count` と `consume_event_count` の**内訳を API レスポンスに含める**。`score` は内部ソート用に非公開（重みの逆算によるランキング操作を防止）。フロントエンドでは「42チームが使用 / 過去90日で15回補充」のような直感的な値を表示
