> **ステータス**: 🟢 設計完了

> **関連ドキュメント**: F03.1 スケジュール・出欠管理、F01.3 テンプレート・モジュール（選択式モジュール #15）、F03.10 年間行事計画（学期連携）、**F03.13 学校日次・教科別出欠管理（時間割を出欠基盤として参照）**、**F03.15 個人時間割（チーム時間割と連動する個人スコープ機能）**

---

## 1. 概要

曜日×時限の週間時間割パターンを定義・管理する機能。学校のクラス・塾・習い事教室など、週単位で繰り返す固定スケジュールを持つ組織に向けた選択式モジュール。学期ごとに時間割を切り替え可能で、臨時の時間割変更（授業変更・休講・補講・日単位の一括休講）にも対応する。スケジュール機能（F03.1）が「個別イベント」を管理するのに対し、本機能は「毎週繰り返す固定パターン」を管理する点で役割が異なる。

## 2. スコープ

### 対象ロール
| ロール | 操作可能な範囲 |
|--------|--------------|
| SYSTEM_ADMIN | 全チーム・組織の時間割参照 |
| ADMIN | 時限定義・学期定義・時間割の CRUD。臨時変更の登録・更新・削除。PDF出力 |
| DEPUTY_ADMIN | `MANAGE_TIMETABLE` 権限を持つ場合: ADMIN と同等 |
| MEMBER | 時間割の閲覧。臨時変更の通知受信 |
| SUPPORTER | `visibility = 'PUBLIC'` の時間割のみ閲覧可。臨時変更の通知受信（公開時間割のみ） |
| GUEST | 対象外 |

### 対象レベル
- [x] 組織 (Organization) — 時限定義のデフォルトテンプレート管理・学期テンプレート
- [x] チーム (Team) — 時間割の作成・管理（クラス＝チーム）
- [ ] 個人 (Personal)

### パーミッション追加
本モジュールの実装時に、F01.2 パーミッション定義に `MANAGE_TIMETABLE` を追加する。
- 対象: DEPUTY_ADMIN の委譲可能権限として登録
- 権限範囲: 時間割・学期定義・臨時変更の CRUD。時限テンプレート管理は ADMIN 専用のため対象外

## 3. DB設計

### テーブル一覧
| テーブル名 | 役割 | 論理削除 |
|-----------|------|---------|
| `timetable_period_templates` | 組織レベルの時限定義テンプレート（1限: 8:30-9:20 等） | なし |
| `timetable_terms` | 学期・期間定義（1学期: 4/1〜7/20 等） | なし |
| `timetables` | 時間割マスター（チーム×学期で1つ） | あり |
| `timetable_slots` | 時間割の各コマ（曜日×時限→教科/担当/教室） | なし |
| `timetable_changes` | 特定日の臨時変更（授業変更・休講・補講・日単位一括休講） | なし |

### テーブル定義

#### `timetable_period_templates`

組織レベルで時限（何限目が何時から何時まで）を定義するテンプレート。配下のチームはこれを継承してチーム独自の時限定義をオーバーライドすることも可能。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `organization_id` | BIGINT UNSIGNED | NO | — | FK → organizations |
| `period_number` | TINYINT UNSIGNED | NO | — | 時限番号（1〜15）。表示順もこの値で決定 |
| `label` | VARCHAR(50) | NO | — | 表示名（例: "1限目", "朝の会", "昼休み"） |
| `start_time` | TIME | NO | — | 開始時刻 |
| `end_time` | TIME | NO | — | 終了時刻 |
| `is_break` | BOOLEAN | NO | FALSE | 休憩枠フラグ（TRUE = 昼休み等。時間割コマの割当対象外だが表示上は表示） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_tpt_org_period (organization_id, period_number)
INDEX idx_tpt_org (organization_id)
```

**制約・備考**
- 1組織あたり最大15枠（アプリ層でバリデーション）
- `start_time < end_time` の CHECK 制約を適用
- `is_break = TRUE` の枠は `timetable_slots` に割当不可（アプリ層でバリデーション）
- 表示順は `period_number` の昇順で固定（別途 `sort_order` は持たない。時限番号が表示順序を兼ねる）
- **組織に属さない単独チーム** は組織テンプレートを継承できないため、`timetables.period_override`（後述）が必須となる。`period_override = NULL` かつ親組織なしの場合は時間割作成時に 400 エラーを返す

---

#### `timetable_terms`

学期・期間の定義。年度をまたぐ場合は `academic_year` で管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | YES | NULL | FK → teams（チーム固有の学期。NULL = 組織レベルの共通定義） |
| `organization_id` | BIGINT UNSIGNED | YES | NULL | FK → organizations（組織レベルの学期テンプレート。NULL = チーム固有） |
| `academic_year` | SMALLINT | NO | — | 年度（例: 2026） |
| `name` | VARCHAR(100) | NO | — | 学期名（例: "1学期", "前期", "春学期"） |
| `start_date` | DATE | NO | — | 開始日 |
| `end_date` | DATE | NO | — | 終了日 |
| `sort_order` | TINYINT UNSIGNED | NO | 0 | 年度内の表示順 |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
INDEX idx_tt_team_year (team_id, academic_year)
INDEX idx_tt_org_year (organization_id, academic_year)
```

**制約・備考**
- `team_id` と `organization_id` は XOR（いずれか1つのみ非NULL）
  ```sql
  CONSTRAINT chk_term_scope CHECK (
    (team_id IS NOT NULL AND organization_id IS NULL) OR
    (team_id IS NULL AND organization_id IS NOT NULL)
  )
  ```
- `start_date < end_date` の CHECK 制約を適用
- 同一スコープ内で期間の重複はアプリ層でバリデーション
- **同一スコープ × 年度 × 名前の重複防止**: MySQL の UNIQUE KEY は NULL カラムを含む場合に重複を防げないため（`NULL != NULL`）、アプリ層で `team_id/organization_id + academic_year + name` の一意性を INSERT/UPDATE 前に検証する
- チームが学期を未定義の場合、親組織の学期テンプレートを継承。`timetables.term_id` は組織レベルの `timetable_terms` を参照可能
- **F03.10 連携**: 本テーブルの学期定義は、年間行事ビュー（F03.10）でのセクション分け（学期別行事グループ）にも利用可能

#### `timetables`

時間割マスター。チーム×学期で1つの時間割を持つ。学期途中で時間割が変わる場合は `effective_from` で切り替え。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `team_id` | BIGINT UNSIGNED | NO | — | FK → teams |
| `term_id` | BIGINT UNSIGNED | NO | — | FK → timetable_terms |
| `name` | VARCHAR(200) | NO | — | 時間割名（例: "2026年度 1学期 通常時間割"） |
| `status` | ENUM('DRAFT', 'ACTIVE', 'ARCHIVED') | NO | 'DRAFT' | 公開状態 |
| `visibility` | ENUM('MEMBERS_ONLY', 'PUBLIC') | NO | 'MEMBERS_ONLY' | 公開範囲。`PUBLIC`: SUPPORTER も閲覧可。`MEMBERS_ONLY`: MEMBER 以上のみ |
| `effective_from` | DATE | NO | — | この時間割の適用開始日 |
| `effective_until` | DATE | YES | NULL | 適用終了日（NULL = 学期終了まで有効） |
| `period_override` | JSON | YES | NULL | チーム独自の時限定義（NULL = 組織テンプレートを継承）。形式は下記参照 |
| `week_pattern_enabled` | BOOLEAN | NO | FALSE | A/B週パターンの有効化フラグ。TRUE の場合、`timetable_slots` で `week_pattern = 'A'` / `'B'` のコマを登録可能 |
| `week_pattern_base_date` | DATE | YES | NULL | A週の起点日（この日を含む週がA週。以降は交互にB→A→B…）。`week_pattern_enabled = TRUE` の場合は必須 |
| `notes` | TEXT | YES | NULL | 管理者向けメモ |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（SET NULL on delete） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |
| `deleted_at` | DATETIME | YES | NULL | 論理削除 |

**インデックス**
```sql
INDEX idx_tm_team_term (team_id, term_id)
INDEX idx_tm_team_status (team_id, status)             -- ACTIVE な時間割の取得用
INDEX idx_tm_effective (team_id, effective_from, effective_until)  -- 日付で有効な時間割を特定
```

**`period_override` JSON 形式**
```json
[
  { "period_number": 1, "label": "1限目", "start_time": "08:45", "end_time": "09:35", "is_break": false },
  { "period_number": 2, "label": "2限目", "start_time": "09:45", "end_time": "10:35", "is_break": false },
  { "period_number": 3, "label": "中休み", "start_time": "10:35", "end_time": "10:50", "is_break": true },
  { "period_number": 4, "label": "3限目", "start_time": "10:50", "end_time": "11:40", "is_break": false }
]
```

**`period_override` の解決ルール**
- `period_override` は**全置換**方式。指定する場合はチームで使用する全時限を網羅する必要がある（組織テンプレートとのマージは行わない）
- `period_override = NULL`: 組織テンプレート（`timetable_period_templates`）をそのまま継承
- `period_override ≠ NULL`: 組織テンプレートを完全に無視し、JSON の定義のみを使用
- 組織未所属チームは `period_override` が必須（NULL 不可）

**ステータス遷移ルール**
| 遷移 | 可否 | 条件・挙動 |
|------|:----:|-----------|
| `DRAFT → ACTIVE` | ○ | 同一 team_id + term_id で適用期間が重複する既存 ACTIVE は自動 ARCHIVED。全メンバーへプッシュ通知（ApplicationEvent: TimetableActivatedEvent） |
| `ACTIVE → ARCHIVED` | ○ | 手動アーカイブ。臨時変更は保持される |
| `ARCHIVED → DRAFT` | ○ | 再編集用。ACTIVE 化前に内容を修正可能 |
| `ACTIVE → DRAFT` | × | 公開中の時間割を直接 DRAFT に戻すことはできない。先にアーカイブしてから DRAFT に戻す |
| `DRAFT → ARCHIVED` | × | 未公開の時間割を直接アーカイブすることはできない |

**コマの編集制約**
- `DRAFT` 状態: コマ（`timetable_slots`）の自由な編集が可能
- `ACTIVE` 状態: コマの直接編集は不可。当日の変更は臨時変更（`timetable_changes`）で対応。恒久的な変更が必要な場合は `duplicate` で複製してから新版を ACTIVE 化する
- `ARCHIVED` 状態: コマの編集は不可（読み取り専用）。DRAFT に戻してから編集する

**制約・備考**
- 論理削除: `deleted_at DATETIME nullable`
- **組織未所属チーム**: `period_override = NULL` かつチームが組織に属さない場合、時間割作成時に 400 エラー。`period_override` で時限定義を直接指定する必要がある
- **`effective_from` / `effective_until` と学期範囲の整合性**: `effective_from` は紐づく `timetable_terms` の `start_date` 以降、`effective_until`（非 NULL 時）は `end_date` 以前であることをアプリ層でバリデーション。範囲外の場合は 400 エラー
- **DRAFT の可視性**: MEMBER / SUPPORTER には `status IN ('ACTIVE', 'ARCHIVED')` の時間割のみ返す。DRAFT は ADMIN / DEPUTY_ADMIN（`MANAGE_TIMETABLE` 権限保持者）にのみ表示

#### `timetable_slots`

時間割の各コマ。1つの `timetable_id` に対して曜日×時限の組み合わせで教科・担当・教室を定義。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `timetable_id` | BIGINT UNSIGNED | NO | — | FK → timetables（ON DELETE CASCADE） |
| `day_of_week` | ENUM('MON', 'TUE', 'WED', 'THU', 'FRI', 'SAT', 'SUN') | NO | — | 曜日 |
| `period_number` | TINYINT UNSIGNED | NO | — | 時限番号（`timetable_period_templates.period_number` に対応） |
| `week_pattern` | ENUM('EVERY', 'A', 'B') | NO | 'EVERY' | 週パターン。`EVERY`: 毎週共通、`A`: A週のみ、`B`: B週のみ。`timetables.week_pattern_enabled = FALSE` の場合は常に `EVERY` |
| `subject_name` | VARCHAR(100) | NO | — | 教科名（例: "国語", "数学", "英語"） |
| `teacher_name` | VARCHAR(100) | YES | NULL | 担当者名 |
| `room_name` | VARCHAR(100) | YES | NULL | 教室・場所（例: "3-A教室", "体育館", "音楽室"） |
| `color` | VARCHAR(7) | YES | NULL | 表示色（HEXカラーコード。例: "#FF6B6B"）。NULL = フロントエンドのデフォルト配色 |
| `notes` | VARCHAR(300) | YES | NULL | 備考（例: "体操服持参", "教科書p.120〜"） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_ts_timetable_day_period_week (timetable_id, day_of_week, period_number, week_pattern)
INDEX idx_ts_timetable_day (timetable_id, day_of_week)   -- 曜日別取得用
```

**制約・備考**
- 同一時間割内で曜日×時限×週パターンの組み合わせはユニーク。`week_pattern = 'EVERY'` のコマがある場合、同じ曜日×時限に `'A'` / `'B'` のコマは登録不可（アプリ層でバリデーション）
- `is_break = TRUE` の時限番号は割当不可（アプリ層でバリデーション）
- `subject_name` のサジェスト用に、同一チーム・組織で過去に使用された教科名の一覧取得 API を提供
- コマの編集は `timetables.status = 'DRAFT'` の場合のみ許可（ACTIVE / ARCHIVED 時は 409 エラー）

#### `timetable_changes`

特定の日付に対する臨時変更。授業変更・休講・補講・日単位一括休講を管理。

| カラム名 | 型 | NULL | デフォルト | 説明 |
|---------|---|------|-----------|------|
| `id` | BIGINT UNSIGNED | NO | AUTO_INCREMENT | PK |
| `timetable_id` | BIGINT UNSIGNED | NO | — | FK → timetables（ON DELETE CASCADE） |
| `target_date` | DATE | NO | — | 変更対象日 |
| `period_number` | TINYINT UNSIGNED | YES | NULL | 変更対象の時限番号。`NULL` = 全コマ対象（`DAY_OFF` 時） |
| `change_type` | ENUM('REPLACE', 'CANCEL', 'ADD', 'DAY_OFF') | NO | — | 変更種別 |
| `subject_name` | VARCHAR(100) | YES | NULL | 変更後の教科名（REPLACE / ADD 時に必須。CANCEL / DAY_OFF 時は NULL） |
| `teacher_name` | VARCHAR(100) | YES | NULL | 変更後の担当者名 |
| `room_name` | VARCHAR(100) | YES | NULL | 変更後の教室 |
| `reason` | VARCHAR(300) | YES | NULL | 変更理由（例: "先生出張のため", "学校行事", "台風接近のため臨時休校"） |
| `notify_members` | BOOLEAN | NO | TRUE | メンバーへの通知要否 |
| `created_by` | BIGINT UNSIGNED | YES | NULL | FK → users（SET NULL on delete） |
| `created_at` | DATETIME | NO | CURRENT_TIMESTAMP | |
| `updated_at` | DATETIME | NO | CURRENT_TIMESTAMP ON UPDATE | |

**インデックス**
```sql
UNIQUE KEY uq_tc_timetable_date_period (timetable_id, target_date, period_number)
INDEX idx_tc_timetable_date (timetable_id, target_date)   -- 日別の臨時変更取得用
```

**`change_type` の挙動**
| 値 | 説明 | `period_number` | `subject_name` |
|----|------|:---------------:|---------------|
| `REPLACE` | 通常コマを別の教科に差し替え | 必須 | 必須（変更後の教科） |
| `CANCEL` | 休講（そのコマは空きになる） | 必須 | NULL |
| `ADD` | 通常は授業がない枠に補講を追加 | 必須 | 必須（追加する教科） |
| `DAY_OFF` | 日単位の一括休講（臨時休校・学校行事等。その日の全コマをキャンセル） | NULL | NULL |

**制約・備考**
- **臨時変更は `status = 'ACTIVE'` の時間割のみ登録可能**（DRAFT はコマ直接編集で対応、ARCHIVED は読み取り専用）。ACTIVE 以外に対する POST は 409 エラー
- `target_date` は当該時間割の `effective_from`〜`effective_until`（または学期終了日）の範囲内（アプリ層でバリデーション）
- `notify_members = TRUE` の場合、INSERT / UPDATE 後に ApplicationEvent を発行してプッシュ通知
- 過去日の変更も許可（記録目的）
- `DAY_OFF` が登録された日は、同日の個別コマ変更（REPLACE / CANCEL / ADD）より優先される。`DAY_OFF` と個別変更が同日に共存する場合、weekly ビューでは `DAY_OFF` を優先表示
- **UNIQUE KEY の `period_number = NULL`（DAY_OFF）**: MySQL では `NULL` 値同士は UNIQUE 制約の対象外のため、同一 `timetable_id + target_date` に複数の `DAY_OFF` が INSERT されうる。アプリ層で同一日の DAY_OFF 重複を検証する

## 5. ビジネスロジック

### 主要フロー

#### 時間割作成〜公開フロー
```
1. ADMIN が学期を定義（または組織テンプレートを継承）
2. ADMIN が時間割を DRAFT で作成
3. ADMIN がコマを PUT で一括登録（曜日ごとに編集可）
4. ADMIN が activate で DRAFT → ACTIVE 化
5. 全メンバーへプッシュ通知（ApplicationEvent: TimetableActivatedEvent）
6. MEMBER はダッシュボードで今日の時間割を閲覧
```

#### 臨時変更フロー
```
1. ADMIN が臨時変更を POST（REPLACE / CANCEL / ADD / DAY_OFF）
2. notify_members = true の場合、ApplicationEvent を発行
3. プッシュ通知サービスがメンバーに通知
   - コマ単位: 「4/13(月) 2限目: 数学→体育に変更」
   - DAY_OFF: 「4/14(火): 台風接近のため臨時休校」
4. weekly ビューは臨時変更を自動反映
```

#### 学期切り替え・時間割複製フロー
```
1. 新学期開始前に ADMIN が duplicate で既存時間割をコピー
2. target_term_id と effective_from を指定
3. 複製された DRAFT 時間割でコマの差分を編集
4. activate で公開（旧時間割は自動 ARCHIVED）
```

#### スケジュール ↔ 時間割の自動連携フロー
```
【スケジュール → 時間割】
1. スケジュール（F03.1）で all_day = true のイベントが作成される
2. event_category が「休業日」系カテゴリの場合、ScheduleCreatedEvent を発行
3. TimetableAutoLinkListener がイベントを受信
4. 当該チームに ACTIVE な時間割が存在し、対象日が適用期間内の場合:
   - DAY_OFF を自動登録（notify_members = false: 通知はスケジュール側で済んでいるため）
   - ADMIN にダッシュボード通知「スケジュール『入学式』に連動して 4/7 を休講に設定しました」
5. ACTIVE な時間割がない場合は何もしない（サイレントスキップ）

【時間割 → スケジュール】
1. ADMIN が DAY_OFF を登録し、連動スケジュール作成を希望（リクエストに create_schedule: true を指定）
2. TimetableChangeCreatedEvent を発行
3. ScheduleAutoLinkListener がイベントを受信
4. 当該チームに all_day + 同日 + title が reason と一致するスケジュールが未存在の場合:
   - schedules に all_day = true のイベントを自動作成（title = reason, event_type = 'EVENT'）
5. 既に同日に同名スケジュールがある場合は重複作成しない
```

> 自動連携はいずれも **提案型**（作成後に ADMIN へ通知）であり、ADMIN が不要と判断した場合は手動で削除可能。自動連携を無効にするチーム設定（`auto_link_timetable_schedule BOOLEAN DEFAULT TRUE`）を `teams` テーブルに追加する。

#### 週間変更ダイジェスト通知
```
1. @Scheduled バッチが毎週日曜 18:00 に起動
2. 各チームの ACTIVE な時間割について、翌週（月〜日）の timetable_changes を集約
3. 変更がある場合のみ、チームの全メンバー（+ visibility = 'PUBLIC' なら SUPPORTER も）にプッシュ通知
4. 通知内容例:
   「来週の時間割変更:
    ・4/14(火): 臨時休校（台風接近のため）
    ・4/16(木) 2限目: 数学→体育に変更」
5. 変更がない週は通知しない
```

> 既存の通知基盤（F04.3 プッシュ通知）を利用。ダイジェスト通知の受信設定はユーザー通知設定で ON/OFF 可能。

### ダッシュボード連携（F02.2）

本モジュール有効時に、ダッシュボードウィジェットとして以下を追加する:

**チームダッシュボード:**
- 「今日の時間割」ウィジェット: ACTIVE な時間割の今日のコマ一覧を表示（臨時変更・DAY_OFF 反映済み、A/B 週自動判定）。データソースは `GET /timetables/{id}/slots/today`

**個人ダッシュボード:**
- 「今日の時間割」ウィジェット: 所属する全チームのうち時間割モジュールが有効なチームの今日のコマと、**F03.15 個人時間割の今日のコマ** を統合してマージ表示。チーム名ラベル付き。データソースは `GET /api/v1/me/dashboard/timetable-today`（F03.15 で定義）
- 「今日のメモ」ウィジェット: F03.15 個人メモ機能で登録したメモを集約表示（予習・復習・持参物）。データソースは `GET /api/v1/me/timetable-slot-notes/today`

> ウィジェットの実装詳細は F02.2 ダッシュボード設計に追記する。

### F03.15 個人時間割との連携 (2026-05-03 追加)

本機能は F03.15（個人時間割）から以下の連携点を持つ:

1. **個人時間割コマからのリンク受け側**: `personal_timetable_slots.linked_timetable_id` / `linked_slot_id` で参照される。リンクされたチーム時間割の臨時変更（`timetable_changes`）は F03.15 のリスナーが個人スケジュール（F03.2）に自動反映する
2. **コマ共通メモ更新通知**: `timetable_slots.notes` を MANAGE_TIMETABLE 権限保持者が更新した時、`TimetableSlotNoteUpdatedEvent` を発行する（**未実装：本機能拡張として F03.15 Phase 4 と同梱予定**）
3. **個人メモ機能の対応**: チーム時間割のコマに対して、ユーザーが個人的な予習・復習・持参物・自由メモを残せる（`timetable_slot_user_notes` テーブル、F03.15 が管理）。**チーム共通メモ（slots.notes）と個人メモ（user_notes）は別物**で、共通メモは MANAGE_TIMETABLE 権限保持者がチーム全員に向けて編集、個人メモは本人のみ閲覧

### 重要な判定ロジック
- **有効な時間割の特定**: `status = 'ACTIVE'` かつ `effective_from <= today` かつ (`effective_until IS NULL` OR `effective_until >= today`)
- **時限定義の解決順序**: `timetables.period_override` > `timetable_period_templates`（組織テンプレート）。組織未所属チームは `period_override` が必須
- **A/B 週の判定**: `week_pattern_base_date` からの経過週数を算出。偶数週 = A週、奇数週 = B週。`week_pattern_enabled = FALSE` の場合は常に `EVERY` のコマのみ表示
- **臨時変更の反映**: weekly ビュー取得時に `timetable_changes` を JOIN し、`is_changed` フラグと `change` オブジェクトを付与。`DAY_OFF` は `is_day_off = true` として日単位で表現
- **DAY_OFF と個別変更の優先度**: 同日に `DAY_OFF` と個別の `REPLACE`/`ADD` が共存する場合、`DAY_OFF` が優先。個別変更は無視される

- `timetable_period_templates` → `timetable_terms` → `timetables` → `timetable_slots` / `timetable_changes` の順で作成（FK依存）
- マイグレーション番号は V11.001〜V11.005 で確定済み
